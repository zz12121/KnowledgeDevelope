# 视图解析与渲染

## 视图解析机制

SpringMVC 的视图解析遵循策略模式：控制器方法返回逻辑视图名 → `DispatcherServlet` 遍历所有 `ViewResolver`（按 `order` 从小到大）→ 第一个能解析的 `ViewResolver` 返回 `View` 对象 → 调用 `View.render()` 渲染模型数据 → 生成响应。

如果方法加了 `@ResponseBody`，会绕过视图解析，直接由 `HttpMessageConverter` 序列化返回值到响应体。

---

## 主流视图技术

| 视图技术 | ViewResolver 实现 | 适用场景 |
|----------|------------------|---------|
| JSP | `InternalResourceViewResolver` | 传统 Java Web，前后端不分离 |
| Thymeleaf | `ThymeleafViewResolver` | 现代 HTML5，天然支持前后端分离 |
| FreeMarker | `FreeMarkerViewResolver` | 高性能模板渲染，代码生成 |
| JSON | `MappingJackson2JsonView` | RESTful API，前后端分离 |

`InternalResourceViewResolver` 配置最简单，把逻辑视图名加前缀和后缀变成物理路径，通过 `RequestDispatcher.forward()` 转发到对应 JSP：

```java
@Bean
public InternalResourceViewResolver viewResolver() {
    InternalResourceViewResolver resolver = new InternalResourceViewResolver();
    resolver.setPrefix("/WEB-INF/views/");
    resolver.setSuffix(".jsp");
    return resolver;
}
// "user/list" 会解析为 /WEB-INF/views/user/list.jsp
```

## 模型数据传递

控制器向视图传递数据有几种方式，底层都使用 `BindingAwareModelMap` 实现：

**`Model` 接口**（最推荐）：类型安全，简洁：
```java
@GetMapping("/users")
public String listUsers(Model model) {
    model.addAttribute("users", userService.findAll());
    return "user/list";
}
```

**`ModelMap`**：需要 Map 操作时使用（`putAll`/`containsKey` 等）。

**`Map`**：参数类型用 `Map<String, Object>` 也可以，效果一样。

**`ModelAndView`**：同时封装视图名和模型数据，适合需要精细控制的场景：
```java
@GetMapping("/user/{id}")
public ModelAndView getUser(@PathVariable Long id) {
    ModelAndView mav = new ModelAndView("user/detail");
    mav.addObject("user", userService.findById(id));
    mav.setStatus(HttpStatus.OK);
    return mav;
}
```

## 重定向与转发

**`forward`（请求转发）**：服务器内部跳转，URL 不变，共享 Request 属性，只有一次请求。用法：`return "forward:/other/path"`。

**`redirect`（重定向）**：浏览器重新发请求，URL 变化，两次请求，Request 属性不共享。用法：`return "redirect:/other/path"`。

| 特性 | forward | redirect |
|------|---------|---------|
| 请求次数 | 1次 | 2次 |
| URL 是否变化 | 不变 | 变化 |
| Request 共享 | 共享 | 不共享 |
| 适用场景 | 页面内跳转 | 防止重复提交（PRG 模式） |

---

## RedirectAttributes

重定向后无法共享 Request 属性，`RedirectAttributes` 解决了在重定向间传递数据的问题：

```java
@PostMapping("/users")
public String createUser(UserDTO user, RedirectAttributes redirectAttrs) {
    userService.create(user);
    // addFlashAttribute：存入 Session，重定向后自动取出并清除（一次性）
    redirectAttrs.addFlashAttribute("message", "用户创建成功");
    // addAttribute：拼接到 URL 查询参数
    redirectAttrs.addAttribute("id", user.getId());
    return "redirect:/users/{id}";
}
```

`addFlashAttribute()` 内部通过 `FlashMapManager` 把数据临时存入 Session，重定向请求处理完后自动清除，用于传递提示消息非常方便。

## 内容协商（Content Negotiation）

同一个接口根据请求的 `Accept` 头返回不同格式（JSON/XML/HTML）的内容，由 `ContentNegotiationManager` 管理。

支持三种协商策略：
1. `Accept` 请求头（标准方式）
2. URL 参数（`?format=json`）
3. URL 扩展名（`.json`/`.xml`，Spring Boot 默认禁用）

生产环境建议只启用 `Accept` 头策略，其他方式可能带来安全问题。
