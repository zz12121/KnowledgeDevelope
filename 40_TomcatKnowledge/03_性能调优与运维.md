# Tomcat 性能调优与运维

## 一、JVM 调优

### 1.1 基础 JVM 参数

```bash
# bin/setenv.sh（不存在时创建）
export JAVA_OPTS="\
  -server \
  -Xms2g \
  -Xmx2g \
  -XX:MetaspaceSize=256m \
  -XX:MaxMetaspaceSize=512m \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -XX:G1HeapRegionSize=16m \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/app/logs/heapdump.hprof \
  -Xlog:gc*:file=/app/logs/gc.log:time,uptime:filecount=5,filesize=20m \
  -Djava.security.egd=file:/dev/./urandom \
  -Dfile.encoding=UTF-8 \
  -Djava.awt.headless=true"
```

### 1.2 GC 选型

| GC 算法 | 适用场景 | 关键参数 |
|--------|---------|---------|
| G1GC（推荐）| 堆 4G+ 低延迟要求 | `-XX:MaxGCPauseMillis=200` |
| ZGC | 堆 16G+ 极低延迟（<10ms）| `-XX:+UseZGC` (JDK 15+) |
| ParallelGC | 批处理，吞吐量优先 | `-XX:+UseParallelGC` |
| CMS（已废弃）| 不推荐 | — |

## 二、Connector 调优

### 2.1 生产建议配置

```xml
<Executor name="tomcatThreadPool"
          namePrefix="catalina-exec-"
          maxThreads="400"
          minSpareThreads="50"
          maxQueueSize="200" />

<Connector executor="tomcatThreadPool"
           port="8080"
           protocol="org.apache.coyote.http11.Http11NioProtocol"
           maxConnections="10000"
           acceptCount="200"
           connectionTimeout="5000"
           keepAliveTimeout="15000"
           maxKeepAliveRequests="200"
           compression="on"
           compressionMinSize="1024"
           compressableMimeType="text/html,text/xml,text/css,
                                  application/json,application/javascript"
           URIEncoding="UTF-8"
           server=" " />  <!-- 隐藏 Server 响应头 -->
```

### 2.2 调优参考数据

| 应用类型 | maxThreads 建议 | 备注 |
|---------|----------------|------|
| IO 密集型（DB/Redis/HTTP 调用）| 300~500 | 线程大量时间等待，多线程提升吞吐 |
| CPU 密集型（计算/图像处理）| CPU核数 × 2 | 过多线程反而切换开销大 |
| 混合型（典型 Web 应用）| 200~400 | 从200开始，压测后调整 |

## 三、安全加固清单

```bash
# 1. 删除默认应用
rm -rf $CATALINA_HOME/webapps/{ROOT,docs,examples,host-manager}
# 保留 manager 但加固访问控制

# 2. 修改端口
# server.xml: port="8080" → 非标准端口
# Server shutdown port: 8005 → 随机

# 3. conf/web.xml 关闭目录列表
# DefaultServlet: listings = false

# 4. 禁用危险 HTTP 方法（conf/web.xml）
# security-constraint + empty auth-constraint

# 5. manager 应用访问控制（conf/tomcat-users.xml + RemoteAddrValve）
```

```xml
<!-- conf/tomcat-users.xml -->
<tomcat-users>
    <role rolename="manager-gui"/>
    <role rolename="manager-script"/>
    <user username="admin" password="STRONG_PASSWORD_HERE"
          roles="manager-gui,manager-script"/>
</tomcat-users>

<!-- webapps/manager/META-INF/context.xml：只允许本机访问 -->
<Valve className="org.apache.catalina.valves.RemoteAddrValve"
       allow="127\.\d+\.\d+\.\d+|::1|0:0:0:0:0:0:0:1" />
```

## 四、监控与故障排查

### 4.1 关键监控指标

```bash
# JMX 监控（通过 jconsole 或 Prometheus JMX Exporter）
Catalina:type=ThreadPool,name="http-nio-8080"
  # currentThreadsBusy：当前活跃线程数（接近 maxThreads 时告警）
  # connectionCount：当前连接数
  # maxThreads：最大线程数

# 常用运维命令
tail -f logs/catalina.out           # 实时查看日志
tail -f logs/localhost_access_log.*.txt  # 实时访问日志

# 快速统计慢请求（%D 列 > 1000ms）
awk '{if($NF>1000) print}' access.log | wc -l
awk '{if($NF>1000) print $7}' access.log | sort | uniq -c | sort -rn | head
```

### 4.2 常见故障排查

**线程池满（HTTP 503 / Connection refused）**：
```bash
# 确认线程池状态
# 通过 Manager App: http://localhost:8080/manager/status
# 看 maxThreads 和 currentThreadsBusy

# jstack 分析线程栈
jstack $(pgrep java) > /tmp/thread-dump.txt
grep -c "in Object.wait" /tmp/thread-dump.txt  # 等待中的线程数
grep -c "BLOCKED" /tmp/thread-dump.txt          # 阻塞的线程数
```

**内存泄漏（Full GC 频繁）**：
```bash
jstat -gcutil $(pgrep java) 2000 10   # 每2秒观察一次，共10次
# O（老年代）列持续上升 → 泄漏嫌疑

jmap -histo $(pgrep java) | head -20  # 快速看哪些类实例最多
```

**卡住的请求（响应时间异常长）**：
```bash
# 找出运行超过60秒的线程（需要配置 StuckThreadDetectionValve）
grep "stuck" logs/catalina.out | tail -20

# 配置 Valve
# <Valve className="org.apache.catalina.valves.StuckThreadDetectionValve" threshold="60" />
```

### 4.3 Prometheus + Grafana 监控集成

```xml
<!-- pom.xml（Spring Boot + Micrometer）-->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

```yaml
# application.yml
management:
  endpoints:
    web:
      exposure:
        include: prometheus, health, info
  metrics:
    tags:
      application: ${spring.application.name}
```

关键 Tomcat 指标（通过 `/actuator/prometheus` 暴露）：
- `tomcat_threads_busy_threads`：当前忙线程数
- `tomcat_connections_current_connections`：当前连接数
- `tomcat_global_request_count_total`：请求总数
- `tomcat_global_request_max_time_seconds`：最大请求时间

## 五、高可用架构

### 5.1 Nginx + Tomcat 集群

```nginx
# nginx.conf
upstream tomcat_cluster {
    least_conn;   # 最少连接数负载均衡
    server tomcat1:8080 max_fails=3 fail_timeout=30s;
    server tomcat2:8080 max_fails=3 fail_timeout=30s;
    server tomcat3:8080 backup;   # 备用节点
    
    keepalive 32;   # 与 Tomcat 保持长连接
}

server {
    listen 80;
    
    location / {
        proxy_pass http://tomcat_cluster;
        proxy_http_version 1.1;
        proxy_set_header Connection "";   # 支持 keepalive
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
    }
}
```

### 5.2 Session 共享方案对比

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|---------|
| Sticky Session（Nginx ip_hash）| 简单，无需改代码 | 宕机丢 Session，负载不均匀 | 小型项目 |
| Tomcat 集群复制（DeltaManager）| 任一节点宕机无感知 | 网络开销大，不适合大集群 | 中型，Session 数据小 |
| Redis（Spring Session）| 高可用，水平扩展 | 需要 Redis | **生产推荐** |

```xml
<!-- Spring Session + Redis 集成 -->
<!-- pom.xml -->
<!-- spring-boot-starter-data-redis + spring-session-data-redis -->
```
```yaml
# application.yml
spring:
  session:
    store-type: redis
    timeout: 30m
  redis:
    host: redis.company.com
    port: 6379
```
