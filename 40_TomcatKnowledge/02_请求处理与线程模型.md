# Tomcat 请求处理与线程模型

## 一、NIO 线程模型

Tomcat NIO（默认模式）采用经典的 **Reactor 模式**，包含三类线程：

```
┌─────────────────────────────────────────────────────────┐
│  外部客户端请求                                           │
│       ↓ TCP 连接                                         │
│  ┌──────────────┐                                       │
│  │   Acceptor   │  1个线程，ServerSocketChannel.accept() │
│  │   Thread     │  → SocketChannel 注册到 Poller         │
│  └──────┬───────┘                                       │
│         ↓ 注册 OP_READ                                   │
│  ┌──────────────────────────────────────────────────┐   │
│  │  Poller Threads（数量 = min(2, CPU核数)）         │   │
│  │  Selector.select() 轮询就绪事件                    │   │
│  │  有数据可读 → 封装 SocketProcessor → 提交到线程池  │   │
│  └──────────────────────┬───────────────────────────┘   │
│                          ↓ 提交任务                       │
│  ┌──────────────────────────────────────────────────┐   │
│  │  Worker Thread Pool（默认 maxThreads=200）        │   │
│  │  Http11Processor → 解析 HTTP → 调用 CoyoteAdapter │   │
│  │  → Engine → Host → Context → Wrapper → Servlet   │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 关键设计点

**BIO vs NIO 的核心差别**：
- BIO（Tomcat 8 之前）：每个 TCP 连接对应一个 Worker 线程，1万连接需要1万线程，不可扩展
- NIO：Acceptor 接受连接后交给 Poller 管理，只有真正有数据时才占用 Worker 线程，1万连接可能只需200个 Worker 线程

**Keep-Alive 的优化**：HTTP/1.1 默认 Keep-Alive，连接处理完请求后不立即关闭，而是重新注册到 Poller 等待下一个请求，这样连接复用不需要反复三次握手，大幅提升高频短请求的性能。

## 二、请求处理完整链路

```
TCP 建立连接
    ↓
Acceptor.run()
  ServerSocketChannel.accept() → SocketChannel（非阻塞）
  → 封装成 NioChannel → 注册到 Poller（OP_READ）
    ↓
Poller.run()
  Selector.select(timeout) → 获取就绪的 SocketChannel
  → 创建 SocketProcessor → 提交到 Executor（线程池）
    ↓
SocketProcessor.run()（Worker 线程）
  → Http11Processor.process()
    → 读取 Socket 数据，解析 HTTP 请求行、请求头
    → 构造 Coyote Request / Response 对象
    → CoyoteAdapter.service()
        ↓
CoyoteAdapter.service()
  → 将 Coyote Request → Catalina HttpServletRequest（包装适配）
  → connector.getService().getContainer().getPipeline().getFirst().invoke(request, response)
        ↓
Engine Pipeline（StandardEngineValve）
  → 根据 Host 名选择 Host，调用 Host Pipeline
        ↓
Host Pipeline（AccessLogValve → StandardHostValve）
  → 根据 Context Path 选择 Context，调用 Context Pipeline
        ↓
Context Pipeline（StandardContextValve）
  → 根据 Servlet 路径找到 Wrapper，调用 Wrapper Pipeline
        ↓
Wrapper Pipeline（StandardWrapperValve）
  → 从 Servlet 实例池获取 Servlet（或创建新实例）
  → 构建 FilterChain（按 web.xml filter-mapping 顺序）
  → FilterChain.doFilter(request, response)
        ↓
Filter1 → Filter2 → ... → FilterN → Servlet.service()
  （Spring MVC: DispatcherServlet.service()）
        ↓
响应写回，Keep-Alive 则 SocketChannel 重新注册到 Poller
```

## 三、线程池配置与调优

### 3.1 使用独立 Executor（推荐大流量场景）

```xml
<!-- 独立的 Executor 比 Connector 内置线程池更灵活 -->
<Executor name="tomcatThreadPool"
          namePrefix="catalina-exec-"
          maxThreads="400"
          minSpareThreads="50"
          maxQueueSize="200"
          threadPriority="5"
          daemon="true" />

<Connector executor="tomcatThreadPool"
           port="8080" protocol="org.apache.coyote.http11.Http11NioProtocol"
           connectionTimeout="20000"
           maxConnections="10000"
           acceptCount="200" />
```

### 3.2 各参数含义

```
maxThreads=400       Worker 线程最大数
                     → IO 密集型（等 DB/Redis/HTTP）：200~500
                     → CPU 密集型（计算）：CPU核数 × 2
                     
minSpareThreads=50   最小空闲线程（防止突发请求创建线程的延迟）

maxConnections=10000 NIO 下同时维护的最大 TCP 连接数
                     → 超过这个数，新连接进入 acceptCount 队列
                     
acceptCount=200      OS 内核级别的连接等待队列（backlog）
                     → 超过 maxConnections+acceptCount，直接拒绝

connectionTimeout=20000  等待完整 HTTP 请求头的超时（ms）
                         → 防止慢客户端占用连接

keepAliveTimeout=15000   Keep-Alive 连接空闲超时（ms）

maxKeepAliveRequests=100 单连接最多复用次数（防止单客户端独占）
```

### 3.3 Spring Boot 嵌入式 Tomcat 配置

```yaml
server:
  tomcat:
    threads:
      max: 400              # maxThreads
      min-spare: 50         # minSpareThreads
    max-connections: 10000  # maxConnections
    accept-count: 200       # acceptCount
    connection-timeout: 20s
    keep-alive-timeout: 15s
    max-keep-alive-requests: 100
```

## 四、异步 Servlet 支持

```java
// 异步 Servlet（解决长轮询/Comet 场景下的线程占用问题）
@WebServlet(urlPatterns = "/async", asyncSupported = true)
public class AsyncServlet extends HttpServlet {
    
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        // 开启异步处理，立即释放 Worker 线程
        AsyncContext ctx = req.startAsync();
        ctx.setTimeout(30000);   // 30秒超时
        
        // 异步任务（在其他线程/事件中完成响应）
        ctx.start(() -> {
            try {
                // 模拟长时间操作
                Thread.sleep(5000);
                resp.getWriter().write("Async response");
                ctx.complete();   // 完成异步处理
            } catch (Exception e) {
                ctx.complete();
            }
        });
        // doGet 方法直接返回，Worker 线程已被释放
    }
}
```

**应用场景**：SSE（Server-Sent Events）、长轮询、Webhook 回调等需要长时间保持连接但大部分时间空闲的场景。Spring MVC 的 `DeferredResult`、`ResponseBodyEmitter` 底层就是用 AsyncContext 实现的。

## 六、源码深度：Servlet 容器核心实现

### 6.1 Servlet 接口与生命周期

Servlet 是 Java Web 开发的基石，Tomcat 作为 Servlet 容器的实现，完全遵循 Servlet 规范。Servlet 接口定义了五个核心方法：

```java
public interface Servlet {
    // 初始化 Servlet，由容器调用
    void init(ServletConfig config) throws ServletException;
    
    // 处理请求，每个请求都会调用
    void service(ServletRequest req, ServletResponse res) throws ServletException, IOException;
    
    // 销毁 Servlet，释放资源
    void destroy();
    
    ServletConfig getServletConfig();
    String getServletInfo();
}
```

**Servlet 生命周期**（Single Instance + Multi-Thread）：

```
┌──────────────────────────────────────────────────────────────┐
│                     Servlet 生命周期                          │
├──────────────────────────────────────────────────────────────┤
│  1. 加载（Load）                                              │
│     ↓ 客户端首次访问时加载                                     │
│  2. 实例化（Instantiate）                                     │
│     ↓ new Servlet() — 单例模式，只创建一次                     │
│  3. 初始化（Init）                                            │
│     ↓ init() — 只调用一次                                      │
│  4. 服务（Service）                                           │
│     ↓ doGet/doPost — 每次请求都调用，多线程并发                │
│  5. 销毁（Destroy）                                           │
│     ↓ Tomcat 关闭时调用，释放资源                              │
└──────────────────────────────────────────────────────────────┘
```

**load-on-startup 配置**：

```xml
<!-- web.xml 中配置，值越小优先级越高 -->
<servlet>
    <servlet-name>DispatcherServlet</servlet-name>
    <servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
    <load-on-startup>1</load-on-startup>  <!-- 启动时加载 -->
</servlet>
```

- 正整数/0：容器启动时加载，值越小越先加载
- 负数/未配置：首次请求时延迟加载

**Servlet 多线程安全问题**：

```java
// ❌ 危险：成员变量会被多线程并发修改
public class UnsafeServlet extends HttpServlet {
    private int count = 0;  // 共享状态，非线程安全
    
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        count++;  // 多线程并发时会出现竞态条件
        resp.getWriter().write("Count: " + count);
    }
}

// ✅ 安全做法 1：使用局部变量
public class SafeServlet1 extends HttpServlet {
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        int count = 0;  // 局部变量，每个请求独立
        // ...
    }
}

// ✅ 安全做法 2：同步控制
public class SafeServlet2 extends HttpServlet {
    private int count = 0;
    private final Object lock = new Object();
    
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        synchronized(lock) {
            count++;
        }
        // ...
    }
}
```

### 6.2 HttpServlet 请求分发机制

HttpServlet 是我们最常用的 Servlet 抽象类，它将 ServletRequest 转换为 HttpServletRequest，并根据 HTTP 方法分发到对应的 doXXX() 方法：

```java
public abstract class HttpServlet extends GenericServlet {
    
    // 入口：将通用 Request 转换为 HTTP Request
    public void service(ServletRequest req, ServletResponse res) {
        HttpServletRequest request = (HttpServletRequest) req;
        HttpServletResponse response = (HttpServletResponse) res;
        service(request, response);  // 委托给重载方法
    }
    
    // HTTP 方法分发
    protected void service(HttpServletRequest req, HttpServletResponse resp) {
        String method = req.getMethod();
        
        if (method.equals("GET")) {
            long lastModified = getLastModified(req);
            if (lastModified == -1L) {
                doGet(req, resp);  // 常规 GET
            } else {
                // 缓存相关：If-Modified-Since
                long ifModifiedSince = req.getDateHeader("If-Modified-Since");
                if (ifModifiedSince < (lastModified / 1000L) * 1000L) {
                    maybeSetLastModified(resp, lastModified);
                    doGet(req, resp);
                } else {
                    resp.setStatus(304);  // Not Modified
                }
            }
        } else if (method.equals("POST")) {
            doPost(req, resp);
        } else if (method.equals("HEAD")) {
            doHead(req, resp);  // 只返回响应头，不返回正文
        } else if (method.equals("PUT")) {
            doPut(req, resp);
        } else if (method.equals("DELETE")) {
            doDelete(req, resp);
        } else if (method.equals("OPTIONS")) {
            doOptions(req, resp);  // 返回支持的 HTTP 方法
        } else if (method.equals("TRACE")) {
            doTrace(req, resp);  // 回显请求，用于调试
        } else {
            // 405 Method Not Allowed
            resp.sendError(501);
        }
    }
    
    // 各方法默认实现：返回 405/400 错误
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        String protocol = req.getProtocol();
        String msg = "HTTP method GET is not supported";
        if (protocol.endsWith("1.1"))
            resp.sendError(405, msg);
        else
            resp.sendError(400, msg);
    }
}
```

**Spring MVC 的 DispatcherServlet** 继承关系：

```
HttpServlet
    ↓
FrameworkServlet
    ↓
DispatcherServlet
    ↓
protected void service(HttpServletRequest request, HttpServletResponse response)
    → processRequest(request response)
        → doService(request, response)
            → doDispatch(request, response)  // 核心：handler 映射与分发
```

### 6.3 ServletContainerInitializer (SCI)

Servlet 3.0 引入的 SPI 机制，允许在容器启动时执行自定义初始化逻辑，Spring Boot 嵌入式 Tomcat 就依赖此机制启动 Spring 容器：

```java
// 1. 定义初始化器（SPI 实现）
@HandlesTypes({Application.class, WebApplicationInitializer.class})
public class SpringServletContainerInitializer implements ServletContainerInitializer {
    
    @Override
    public void onStartup(Set<Class<?>> webAppInitializerClasses, 
                         ServletContext servletContext) throws ServletException {
        
        List<WebApplicationInitializer> initializers = new LinkedList<>();
        
        // 2. 实例化所有 WebApplicationInitializer
        for (Class<?> waiClass : webAppInitializerClasses) {
            if (!waiClass.isInterface() && !Modifier.isAbstract(waiClass.getModifiers())) {
                try {
                    initializers.add((WebApplicationInitializer) 
                        waiClass.getDeclaredConstructor().newInstance());
                } catch (Throwable t) {
                    // 忽略
                }
            }
        }
        
        // 3. 按优先级排序并调用 onStartup
        Collections.sort(initializers);
        for (WebApplicationInitializer initializer : initializers) {
            initializer.onStartup(servletContext);
        }
    }
}
```

**配置 SPI**（Tomcat 自动发现）：

```properties
# META-INF/services/javax.servlet.ServletContainerInitializer
# 文件内容：
org.springframework.web.SpringServletContainerInitializer
```

**SpringBoot 启动流程**：

```
SpringServletContainerInitializer.onStartup()
    ↓
SpringWebAppInitializer.onStartup()
    ↓
AbstractDispatcherServletInitializer.onStartup()
    ↓
DispatcherServlet.init()  // 创建 Spring 容器
```

### 6.4 Tomcat 类加载器架构（WebappClassLoader 打破双亲委派）

Tomcat 的类加载器设计是为了实现应用隔离，每个 Web 应用有独立的类加载器：

```
┌─────────────────────────────────────────────────────────────┐
│                    Tomcat 类加载器层次                        │
├─────────────────────────────────────────────────────────────┤
│  Bootstrap ClassLoader（ JVM 内置，/lib）                    │
│      ↑                                                        │
│  Extension ClassLoader（$JAVA_HOME/jre/lib/ext/）             │
│      ↑                                                        │
│  System ClassLoader（CLASSPATH）                             │
│      ↑                                                        │
│  CommonClassLoader（$CATALINA_HOME/lib/）                    │
│      ↑────────┬──────────────────────────┐                   │
│              │                           │                   │
│  CatalinaClassLoader            SharedClassLoader           │
│  (Tomcat 私有类)                 (应用共享库)                 │
│                                      ↑                       │
│                              WebappClassLoader × N           │
│                              (每个 Context 独立)             │
│                              加载：                          │
│                              WEB-INF/classes/               │
│                              WEB-INF/lib/*.jar              │
└─────────────────────────────────────────────────────────────┘
```

**WebappClassLoader 打破双亲委派**（关键设计）：

```java
public class WebappClassLoader extends WebappClassLoaderBase {
    
    @Override
    public Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        
        // 1. 先检查 JVM 缓存
        Class<?> clazz = findLoadedClass(name);
        if (clazz != null) {
            return clazz;
        }
        
        // 2. java.* 核心类必须委托 Bootstrap（不可打破）
        if (name.startsWith("java.") || name.startsWith("javax.")) {
            return super.loadClass(name, resolve);
        }
        
        // 3. 优先从 WEB-INF/classes 和 WEB-INF/lib 加载（打破双亲委派！）
        clazz = findClass(name);  // 自己加载
        if (clazz != null) {
            return clazz;
        }
        
        // 4. 再委托给父加载器（CommonClassLoader）
        return super.loadClass(name, resolve);
    }
}
```

**打破双亲委派的原因**：
- 应用隔离：不同应用可以用不同版本的同一第三方库
- Servlet 规范要求：应用的类优先于容器类加载
- 热部署支持：重新加载时清空已加载类

**Tomcat 8+ 类加载顺序**：

```java
// Tomcat 8.5+ 默认顺序（可通过 delegate 属性配置）
// delegate="false" 时：
// 1. Bootstrap → Extension → System → Common（标准双亲委派）
// 2. WebappClassLoader 优先加载自己的类

// delegate="true" 时（完全使用双亲委派）：
// 1. 先委托父加载器，父加载不了才自己加载
// 2. 不打破双亲委派，应用隔离性差
```

### 6.5 请求处理完整流程（源码视角）

```
┌──────────────────────────────────────────────────────────────┐
│                   Tomcat 请求处理源码流程                      │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. NIO 通道层                                               │
│     Acceptor → SocketChannel → Poller（注册 OP_READ）        │
│                                                               │
│  2. SocketProcessor（Worker 线程）                            │
│     Http11Processor.parseRequest() → 解析 HTTP                │
│     → 创建 org.apache.coyote.Request/Response                │
│                                                               │
│  3. CoyoteAdapter（适配层）                                   │
│     request = (Request) req.getAttribute(Request.class);     │
│     connector.getService().getContainer()                    │
│         .getPipeline().getFirst().invoke(request, response); │
│                                                               │
│  4. 容器 Pipeline 链                                          │
│     StandardEngineValve → StandardHostValve                  │
│     → StandardContextValve → StandardWrapperValve            │
│                                                               │
│  5. StandardWrapperValve（Servlet 调度核心）                  │
│     servlet = wrapper.allocate();  // 获取 Servlet 实例       │
│     filterChain = ApplicationFilterFactory.createFilterChain();│
│     filterChain.doFilter(request, response);                 │
│                                                               │
│  6. FilterChain 链式调用                                      │
│     Filter1.doFilter() → Filter2.doFilter() → ...            │
│     → Servlet.service()                                      │
│     → HttpServlet.service() → doGet/doPost()                 │
│                                                               │
│  7. 响应写回，Keep-Alive 则重新注册到 Poller                   │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

**StandardWrapperValve 核心代码**：

```java
public final class StandardWrapperValve extends ValveBase {
    
    @Override
    public void invoke(Request request, Response response) {
        
        // 1. 从 Wrapper 获取 Servlet 实例
        StandardWrapper wrapper = (StandardWrapper) getContainer();
        Servlet servlet = null;
        try {
            servlet = wrapper.allocate();  // 单例，获取或创建 Servlet
            
            // 2. 创建 FilterChain
            ApplicationFilterChain filterChain = 
                ApplicationFilterFactory.createFilterChain(request, servlet);
            
            // 3. 依次调用 Filter，最后调用 Servlet.service()
            filterChain.doFilter(request.getRequest(), response.getResponse());
            
        } finally {
            // 4. 释放资源
        }
    }
}
```

**FilterChain 核心代码**：

```java
public final class ApplicationFilterChain implements FilterChain {
    
    private ApplicationFilterConfig[] filters;  // Filter 数组
    private Servlet servlet;                     // 目标 Servlet
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response) 
            throws IOException, ServletException {
        
        internalDoFilter(request, response);  // 递归调用
    }
    
    private void internalDoFilter(ServletRequest request, ServletResponse response) {
        
        // 1. 按顺序执行所有 Filter
        if (pos < n) {
            ApplicationFilterConfig filterConfig = filters[pos++];
            Filter filter = filterConfig.getFilter();
            filter.doFilter(request, response, this);  // 传入 this 实现链式调用
            return;
        }
        
        // 2. 所有 Filter 执行完毕，调用 Servlet
        servlet.service(request, response);
    }
}
```
