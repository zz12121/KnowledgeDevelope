# 生产者与消费者工作原理

## 一、生产者工作流程

Kafka 生产者采用**主线程-累加器-Sender 线程**的异步批处理模型，核心设计目标是高吞吐：

```
应用线程（主线程）                     后台 Sender 线程
    ↓
1. ProducerRecord 创建
2. 拦截器链处理（ProducerInterceptor）
3. 序列化（Key/Value → bytes）
4. 分区器选择目标分区（Partitioner）
5. 追加到 RecordAccumulator（内存缓冲区）
                                    ← 批次满 or linger.ms 超时
                                    6. 拉取就绪批次（ProducerBatch）
                                    7. 按目标 Broker 分组打包
                                    8. NetworkClient 发送
                                    9. Broker 处理并响应
                                    10. 触发回调（Callback）
```

### 1.1 核心代码

```java
Properties props = new Properties();
props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());
props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());
// 可靠性配置
props.put(ProducerConfig.ACKS_CONFIG, "all");
props.put(ProducerConfig.RETRIES_CONFIG, Integer.MAX_VALUE);
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, "true");
// 性能配置
props.put(ProducerConfig.BATCH_SIZE_CONFIG, 16384);         // 16KB
props.put(ProducerConfig.LINGER_MS_CONFIG, 5);               // 等待5ms填充批次
props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "lz4");   // 批量压缩

KafkaProducer<String, String> producer = new KafkaProducer<>(props);

// 异步发送（推荐）—— 通过 Callback 感知结果
ProducerRecord<String, String> record = new ProducerRecord<>(
    "OrderTopic",    // topic
    "order-1001",    // key（决定分区）
    "订单内容..."     // value
);
producer.send(record, (metadata, exception) -> {
    if (exception != null) {
        log.error("消息发送失败，进行补偿处理", exception);
    } else {
        log.info("发送成功: topic={}, partition={}, offset={}",
                metadata.topic(), metadata.partition(), metadata.offset());
    }
});

// 同步发送（低吞吐，慎用）
RecordMetadata metadata = producer.send(record).get();
```

## 二、分区策略

分区器（Partitioner）决定消息路由到哪个分区，优先级如下：

1. **显式指定分区**：构建 `ProducerRecord` 时直接传入 `partition` 参数，绝对优先
2. **基于 Key 哈希**：有 Key 时，使用 Murmur2 哈希：`hash(key) % numPartitions`，保证相同 Key 的消息落在同一分区（顺序性保证）
3. **粘性分区（Sticky Partitioning）**：无 Key 时（Kafka 2.4+），随机选一个分区，在当前批次期间保持粘性，比轮询更利于批次积累，提升吞吐量

**自定义分区器**：
```java
public class BusinessPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();
        if (keyBytes == null) {
            // 无 Key：粘性分配或轮询
            return ThreadLocalRandom.current().nextInt(numPartitions);
        }
        String keyStr = (String) key;
        // VIP 订单固定发到最后一个分区（优先消费）
        if (keyStr.startsWith("VIP_")) {
            return numPartitions - 1;
        }
        return Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;
    }

    @Override public void close() {}
    @Override public void configure(Map<String, ?> configs) {}
}
```

```java
props.put(ProducerConfig.PARTITIONER_CLASS_CONFIG, BusinessPartitioner.class.getName());
```

## 三、acks 参数与可靠性

`acks` 是控制消息可靠性的最关键配置：

| acks 值 | 含义 | 可靠性 | 吞吐量 | 适用场景 |
|--------|------|--------|--------|---------|
| `0` | 不等待任何确认 | 最低（可能丢消息） | 最高 | 日志收集，极高吞吐，允许丢失 |
| `1`（默认） | 等待 Leader 写入本地日志后确认 | 中等（Leader 写后即刻宕机可能丢失） | 中等 | 大部分业务日志 |
| `all`（或 `-1`） | 等待 Leader + 所有 ISR 副本都写入后确认 | 最高 | 最低 | 金融交易、计费系统 |

**关联配置**（`acks=all` 时必须配合）：
```properties
# Broker/Topic 级别配置
min.insync.replicas=2    # ISR 最少需要 2 个副本确认
replication.factor=3     # 3 副本，容忍 1 个 Broker 宕机
```

> 当 ISR 副本数小于 `min.insync.replicas` 时，生产者会收到 `NotEnoughReplicasException`，写入拒绝——用可用性换数据安全。

## 四、生产者幂等性与事务

### 4.1 单分区幂等性（Exactly-Once within Partition）

开启幂等性后，Kafka 为每个生产者实例分配唯一 **PID**，为每个 `<Topic, Partition>` 维护单调递增的 **Sequence Number**，Broker 通过序列号去重：

```java
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, "true");
// 开启后自动设置 acks=all, retries=MAX, max.in.flight.requests=5
```

**Broker 去重逻辑**：
- 序列号 = 期望值（上次 +1）→ 正常写入
- 序列号 ≤ 期望值 → 重复消息，直接丢弃
- 序列号 远大于期望值 → 抛出 `OutOfOrderSequenceException`（有消息丢失）

### 4.2 跨分区事务（Exactly-Once across Partitions）

事务保证向多个 Topic/Partition 发送消息的原子性（要么全部成功，要么全部失败）：

```java
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, "true");
props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "order-producer-txn-1"); // 唯一事务 ID

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions(); // 初始化（向 Transaction Coordinator 注册）

try {
    producer.beginTransaction();
    
    // 原子发送到多个 Topic
    producer.send(new ProducerRecord<>("OrderTopic", "order-1001", "下单"));
    producer.send(new ProducerRecord<>("InventoryTopic", "sku-001", "扣库存"));
    producer.send(new ProducerRecord<>("PointsTopic", "user-001", "加积分"));
    
    producer.commitTransaction(); // 向所有涉及分区写入 COMMIT 标记
} catch (Exception e) {
    producer.abortTransaction();  // 向所有涉及分区写入 ABORT 标记
    throw e;
}
```

**消费者侧**（只消费已提交事务的消息）：
```java
props.put(ConsumerConfig.ISOLATION_LEVEL_CONFIG, "read_committed");
// 默认 read_uncommitted，会读到未提交事务的消息
```

## 五、批处理性能参数

| 参数 | 默认值 | 调优方向 |
|------|--------|---------|
| `batch.size` | 16KB | 增大 → 更大批次，提升吞吐，增加内存和延迟 |
| `linger.ms` | 0（不等待） | 设置 5-100ms → 等待填满批次，显著提升吞吐量 |
| `buffer.memory` | 32MB | 增大 → 缓冲更多消息，应对流量峰值 |
| `compression.type` | none | lz4/snappy → 减少网络和磁盘，用 CPU 换 I/O |
| `max.block.ms` | 60000ms | 缓冲区满时阻塞时间，超时抛 TimeoutException |

**生产环境推荐配置**：
```properties
# 高吞吐场景
batch.size=65536         # 64KB
linger.ms=10             # 等待10ms
compression.type=lz4
buffer.memory=67108864   # 64MB

# 低延迟场景
batch.size=16384         # 16KB
linger.ms=0              # 不等待
compression.type=none
```

## 六、消费者工作流程

Kafka 消费者采用**单线程事件循环 + 后台心跳线程**模型：

```
1. new KafkaConsumer() → 初始化 Metadata、NetworkClient、Coordinator、Fetcher
2. subscribe(topics)   → 声明订阅意图（更新 SubscriptionState）
3. poll(Duration)      → 触发加入组流程（第一次 poll）
   ├── 寻找 GroupCoordinator（FindCoordinatorRequest）
   ├── 加入组（JoinGroupRequest）
   ├── Leader 消费者执行分区分配
   ├── 同步分配结果（SyncGroupRequest）
   └── 开始拉取消息（FetchRequest）
4. 处理 ConsumerRecords（业务逻辑）
5. 提交 Offset（commitSync / commitAsync）
```

### 6.1 核心代码

```java
Properties props = new Properties();
props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
props.put(ConsumerConfig.GROUP_ID_CONFIG, "order-consumer-group");
props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());
props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());
props.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "earliest"); // 无历史 offset 时从最早开始
props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, "false");   // 关闭自动提交

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("OrderTopic"));

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
        for (ConsumerRecord<String, String> record : records) {
            System.out.printf("partition=%d, offset=%d, key=%s, value=%s%n",
                    record.partition(), record.offset(), record.key(), record.value());
            processOrder(record.value()); // 业务处理
        }
        // 业务处理完成后手动提交 offset（避免丢消息）
        consumer.commitSync();
    }
} finally {
    consumer.close();
}
```

## 七、消费位点（Offset）管理

### 7.1 自动提交 vs 手动提交

| 方式 | 配置 | 特点 | 风险 |
|------|------|------|------|
| **自动提交** | `enable.auto.commit=true`，`auto.commit.interval.ms=5000` | 简单，定期后台提交 | 消息处理中途宕机会丢消息（还未提交） |
| **同步手动提交** | `commitSync()` | 提交确认后再处理下一批，不丢消息 | 阻塞，降低吞吐量 |
| **异步手动提交** | `commitAsync(callback)` | 不阻塞，吞吐量高 | 失败不自动重试，需在 callback 中处理 |

**生产环境推荐**：关闭自动提交 + 同步手动提交（保证 At Least Once）：
```java
props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, "false");

// 处理完一批消息后提交
for (ConsumerRecord<String, String> record : records) {
    process(record);
}
consumer.commitSync(); // 确认消息已处理才提交位点
```

### 7.2 手动指定消费位点

```java
// 从最新位置开始消费
consumer.seekToEnd(consumer.assignment());

// 从最早位置开始（用于重放历史数据）
consumer.seekToBeginning(consumer.assignment());

// 从特定 offset 开始（精准回溯）
consumer.seek(new TopicPartition("OrderTopic", 0), 1000);

// 按时间戳定位 offset（3.0+ 支持）
Map<TopicPartition, Long> timestamps = Collections.singletonMap(
    new TopicPartition("OrderTopic", 0),
    System.currentTimeMillis() - 24 * 3600 * 1000L // 昨天此刻
);
Map<TopicPartition, OffsetAndTimestamp> offsetMap = consumer.offsetsForTimes(timestamps);
offsetMap.forEach((tp, offsetTs) -> consumer.seek(tp, offsetTs.offset()));
```

## 八、消费者 Rebalance

Rebalance 触发条件：
- 消费者加入组（实例新增）
- 消费者离开组（宕机或 `close()`）
- 消费者长时间未调用 `poll()`（超过 `max.poll.interval.ms`）
- Topic 分区数变化
- 正则订阅时有新匹配的 Topic 出现

**Rebalance 的副作用**：Stop-the-World，整个消费者组暂停消费，可能导致消息堆积。

**减少不必要 Rebalance 的配置**：

```properties
# 心跳超时（适当调大，避免网络抖动触发误判）
session.timeout.ms=30000
# 心跳间隔（< session.timeout.ms / 3）
heartbeat.interval.ms=10000
# 最大消息处理间隔（必须大于单批最大处理时间！）
max.poll.interval.ms=300000
# 静态成员资格（Kafka 2.3+），重启时不触发 Rebalance
group.instance.id=consumer-instance-1
# 粘性分区分配策略，减少分区迁移
partition.assignment.strategy=org.apache.kafka.clients.consumer.StickyAssignor
```

## 九、消息传递语义

| 语义 | 含义 | 实现方式 |
|------|------|---------|
| **At Most Once** | 至多一次，可能丢消息 | `acks=0` 或自动提交（处理前提交） |
| **At Least Once** | 至少一次，可能重复 | `acks=all` + 手动提交（处理后提交） |
| **Exactly Once** | 精确一次，不丢不重 | 幂等生产者 + 事务 + `read_committed` + 业务幂等性 |

**Exactly-Once 完整配置**：
```java
// 生产者
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, "true");
props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "unique-txn-id");

// 消费者
props.put(ConsumerConfig.ISOLATION_LEVEL_CONFIG, "read_committed");
props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, "false");
```
