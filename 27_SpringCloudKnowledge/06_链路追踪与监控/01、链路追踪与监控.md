# 链路追踪与监控

## 一、为什么需要链路追踪

在微服务架构中，一个请求可能跨越多个服务。当出现问题时：

- **故障定位难**：传统日志分散在各服务，手工关联效率极低
- **性能瓶颈难找**：不知道哪个服务最慢
- **依赖关系不清晰**：服务依赖图难以维护

**分布式链路追踪**通过唯一的 TraceID 将分散在不同服务中的日志串联起来，形成完整调用链视图。

---

## 二、核心概念

- **Trace（链路）**：代表一次完整的请求过程，唯一 TraceID。如一次下单请求，从网关 → 认证 → 订单 → 库存，整体是一个 Trace。
- **Span（跨度）**：Trace 中的一个工作单元，代表一个服务内的具体操作。包含 SpanID、ParentSpanID、开始时间、耗时、Tags 等。
- **Annotation（注解）**：记录特定时间点的事件：
  - `cs`（Client Send）：客户端发送请求
  - `sr`（Server Receive）：服务端接收请求
  - `ss`（Server Send）：服务端发送响应
  - `cr`（Client Receive）：客户端接收响应

## 三、Spring Cloud Sleuth

### 什么是 Sleuth

Sleuth 是 Spring Cloud 提供的**分布式链路追踪解决方案**，负责生成和传播追踪数据。无侵入式设计，引入依赖即可自动集成。

### 工作原理

1. **生成 TraceID**：请求进入系统时，Sleuth 检查是否有追踪信息，没有则生成新的 TraceID 和 Root Span
2. **传播追踪信息**：服务间调用时，Sleuth 自动将追踪信息注入 HTTP 请求头（B3 格式）：
   ```http
   X-B3-TraceId: 0af7651916cd43dd8448eb211c80319c
   X-B3-SpanId: 0af7651916cd43dd
   X-B3-ParentSpanId: 0000000000000000
   ```
3. **MDC 注入**：TraceID 自动注入到 SLF4J MDC 中，日志里自动带上 TraceID，方便问题排查
4. **数据上报**：Span 数据异步批量发送给 Zipkin

### 基本配置

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-zipkin</artifactId>
</dependency>
```

```yaml
spring:
  zipkin:
    base-url: http://localhost:9411
  sleuth:
    sampler:
      probability: 1.0  # 采样率，生产环境一般用 0.1（10%）
```

## 四、Zipkin

### 什么是 Zipkin

Zipkin 是 Twitter 开源的**分布式追踪系统**，负责收集、存储和可视化追踪数据。基于 Google Dapper 论文实现。

### 架构组件

- **Collector（收集器）**：接收 Sleuth 发送的追踪数据（支持 HTTP 和 Kafka）
- **Storage（存储层）**：可插拔存储后端——In-Memory（测试用）/ MySQL（中小规模）/ Elasticsearch（生产推荐）
- **Query Service（查询服务）**：提供 REST API 查询追踪数据
- **Web UI**：可视化界面，展示调用链瀑布图、服务依赖图等

## 五、采样策略

全量采集会产生大量数据，影响系统性能，因此需要采样策略：

- **恒定采样**：按固定比例（如 `probability: 0.1` 表示 10%）。简单，但不适应流量波动。
- **速率限制采样**：限制每秒采样数量，如每秒最多采样 100 个 Trace。
- **自适应采样**：根据系统负载动态调整采样率。
- **针对性采样**：对特定路径或异常请求全量采样。

**生产建议**：正常流量 5%-10% 采样，异常请求 100% 采样。

## 六、SkyWalking vs Zipkin

| 特性 | Zipkin + Sleuth | SkyWalking |
|------|----------------|-----------|
| 接入方式 | SDK 集成（有少量代码侵入） | Java Agent 字节码增强（零侵入） |
| 功能范围 | 专注链路追踪 | 追踪 + 指标 + 日志，完整 APM |
| UI 功能 | 简单，专注链路展示 | 功能丰富，拓扑图、性能矩阵等 |
| 社区活跃度 | 成熟稳定，更新较慢 | Apache 顶级项目，快速迭代 |
| 适用场景 | Spring Cloud 基础链路追踪 | 需要完整 APM 能力，对侵入性敏感 |

**选型建议**：Spring Cloud 项目基础需求用 Zipkin + Sleuth 够用；需要完整可观测性平台、零侵入或大规模分布式系统，用 SkyWalking。

## 七、监控体系

### Spring Boot Actuator

Actuator 提供生产就绪的监控端点：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus  # 暴露的端点
  endpoint:
    health:
      show-details: always
```

常用端点：
- `/actuator/health`：应用健康状态
- `/actuator/metrics`：应用指标（JVM、线程、请求数等）
- `/actuator/prometheus`：Prometheus 格式指标（配合 Grafana 使用）
- `/actuator/info`：应用信息

### Prometheus + Grafana

完整的可观测性监控栈：

```
应用（Actuator /prometheus）→ Prometheus（抓取指标）→ Grafana（可视化）
                                                      ↓
                                              告警（AlertManager）
```

### 监控的四个黄金指标

- **延迟（Latency）**：请求处理时间
- **流量（Traffic）**：QPS、TPS
- **错误率（Errors）**：失败请求比例
- **饱和度（Saturation）**：资源使用率（CPU、内存、连接池）
