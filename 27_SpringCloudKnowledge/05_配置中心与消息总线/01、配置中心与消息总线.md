# 配置中心与消息总线

## 一、为什么需要配置中心

微服务架构下，配置分散在每个服务中，带来以下痛点：

- **修改麻烦**：改一个公共配置要逐个服务修改、重启
- **环境不隔离**：dev/test/prod 配置容易混乱
- **无法动态生效**：修改配置必须重启服务
- **缺乏审计**：谁改了什么配置无从追溯

**配置中心**集中管理所有服务配置，解决以上问题：

- 集中存储、统一管理
- 支持动态推送，无需重启
- 多环境隔离（namespace）
- 变更历史记录与版本回滚

## 二、Spring Cloud Config

### 基本架构

Spring Cloud Config 采用 **客户端-服务器架构**：

- **Config Server**：连接配置存储库（通常是 Git），向客户端提供配置
- **Config Client**：集成在各微服务中，启动时从 Config Server 拉取配置

```yaml
# Config Server 配置
spring:
  cloud:
    config:
      server:
        git:
          uri: https://github.com/yourorg/config-repo
          search-paths: '{application}'
          default-label: main

# 启动类
@SpringBootApplication
@EnableConfigServer
public class ConfigServerApplication { ... }
```

```yaml
# Config Client 配置（bootstrap.yml）
spring:
  application:
    name: user-service
  cloud:
    config:
      uri: http://config-server:8888
      profile: dev
      label: main
```

### 配置文件命名规则

Config Server 按以下规则查找配置文件：

```
/{application}/{profile}[/{label}]
/{application}-{profile}.yml
/{label}/{application}-{profile}.yml
```

例如 `user-service-dev.yml`、`user-service-prod.yml`。

### 核心局限性

- **不支持原生实时推送**：配置变更后，客户端不会自动感知，需要结合 Spring Cloud Bus + `@RefreshScope` 手动触发刷新
- **无可视化界面**：不如 Apollo、Nacos 直观
- **功能相对简单**：适合中小型项目

## 三、Nacos 配置中心

Nacos 是目前国内微服务配置中心的主流选择，相比 Spring Cloud Config 优势明显：

- **原生支持动态推送**：基于长连接，配置变更后实时推送到客户端
- **可视化控制台**：Web 界面直接管理配置
- **版本管理**：内置配置变更历史
- **三层隔离**：命名空间 + 分组 + 服务名，多环境管理很清晰
- **一体化**：注册中心 + 配置中心合二为一

### 基本使用

```yaml
# bootstrap.yml
spring:
  application:
    name: user-service
  cloud:
    nacos:
      config:
        server-addr: localhost:8848
        file-extension: yaml
        namespace: dev  # 命名空间
        group: DEFAULT_GROUP
```

### 动态刷新

在 Bean 上加 `@RefreshScope`，配置变更后该 Bean 会自动重建：

```java
@RestController
@RefreshScope  // 支持动态刷新
public class ConfigController {
    @Value("${app.timeout}")
    private int timeout;
}
```

## 四、Apollo

携程开源的企业级配置中心，功能最全面：

- 完善的权限管理（不同角色只能修改特定配置）
- 灰度发布（只推送给部分实例）
- 配置审计（完整变更记录）
- 多集群、多环境支持

适合中大型企业对配置管理有严格要求的场景。

## 五、Spring Cloud Bus（消息总线）

### 解决的问题

Spring Cloud Config 本身不支持配置实时推送。当 Git 仓库配置发生变更时，需要手动通知各服务刷新。Spring Cloud Bus 通过**消息队列**（RabbitMQ 或 Kafka）广播刷新通知。

### 工作原理

```
运维修改 Git 配置
    ↓
手动调用 Config Server 的 /bus/refresh 端点
    ↓
Config Server 发布消息到消息总线（MQ）
    ↓
所有 Config Client 收到消息 → 触发 @RefreshScope Bean 重建 → 配置生效
```

### 配置示例

```yaml
# 引入 Bus 依赖后，Config Client 增加配置
spring:
  rabbitmq:
    host: localhost
    port: 5672
management:
  endpoints:
    web:
      exposure:
        include: bus-refresh  # 暴露刷新端点
```

```bash
# 触发全量刷新
curl -X POST http://config-server:8888/actuator/bus-refresh

# 只刷新特定服务
curl -X POST http://config-server:8888/actuator/bus-refresh/user-service
```

## 六、配置中心选型对比

| 特性 | Spring Cloud Config | Nacos | Apollo |
|------|---------------------|-------|--------|
| 存储后端 | Git/SVN/文件系统 | 数据库（Derby/MySQL） | MySQL |
| 实时推送 | 需要 Bus 配合 | 原生支持 | 原生支持 |
| 可视化界面 | 无 | 有 | 有（功能最全） |
| 多环境 | Profile + Label | Namespace | Environment |
| 权限控制 | 基本 | 基本 | 完善 |
| 灰度发布 | 无 | 有 | 有 |
| 适用场景 | 小型 Spring 项目 | 国内主流微服务 | 企业级、权限要求高 |
