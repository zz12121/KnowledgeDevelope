# 服务注册与发现

## 一、核心概念

**服务注册与发现**解决微服务实例动态变化带来的寻址问题：

- **服务注册**：实例启动时向注册中心上报自身元数据（IP、端口、服务名等），通过心跳机制维持租约
- **服务发现**：消费者不直接写死地址，而是查询注册中心获取可用实例列表，结合负载均衡算法选择实例

### 发现模式对比

**客户端发现模式（Spring Cloud 默认）**
- 消费者从注册中心获取实例列表，在本地做负载均衡
- 代表：Eureka + Ribbon
- 优点：灵活、去中心化；缺点：客户端复杂度高

**服务端发现模式**
- 消费者通过负载均衡器（API 网关）发请求，网关查注册中心并转发
- 代表：Kubernetes Service
- 优点：对客户端透明；缺点：需维护中间件

## 二、Eureka

### 基本原理

Eureka 是 Netflix 开源的服务发现框架，遵循 **AP** 模型（高可用 + 分区容错）。

**工作流程**：
1. **注册**：Client 启动时向 Server 发 POST 请求注册自身元数据
2. **续约（心跳）**：Client 默认每 **30 秒**发一次心跳，Server 收到后重置租约
3. **拉取列表**：Client 每 **30 秒**从 Server 拉取服务注册表缓存到本地
4. **调用**：消费者用本地缓存 + Ribbon 负载均衡选实例
5. **下线**：Client 关闭时发 DELETE 请求，Server 移除实例

```java
// Eureka Server 启动
@SpringBootApplication
@EnableEurekaServer
public class EurekaServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}
```

```yaml
# Eureka Client 配置
eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka
  instance:
    lease-renewal-interval-in-seconds: 30    # 心跳间隔
    lease-expiration-duration-in-seconds: 90 # 超时时间（90s未收到心跳则剔除）
```

### 三级缓存架构

Eureka Server 用三级缓存平衡性能与一致性：

1. **ReadWriteCache**（读写缓存）：Guava Cache 实现，响应客户端查询请求
2. **ReadOnlyCache**（只读缓存）：定时与读写缓存同步（默认 30s）
3. **Client 本地缓存**：客户端内存缓存，减少网络调用

由于缓存机制，服务状态更新存在最多 **90 秒**延迟。

### 自我保护机制

**触发条件**：15 分钟内心跳丢失比例超过 **15%**（即只有不到 85% 的实例正常心跳）

**工作机制**：
- 进入保护模式后，**不再剔除**因长时间未收到心跳的服务实例
- 继续接受新注册，但不同步到其他节点
- 网络恢复后自动退出保护模式

**设计目的**：防止网络分区时误删服务实例，宁可保留不健康实例，也不大规模剔除。

```yaml
# 生产环境一般保持默认（开启自我保护）
eureka:
  server:
    enable-self-preservation: true
```

### 高可用集群

Eureka Server 节点间**互相注册**，形成去中心化对等集群：

```yaml
# 节点1 配置
eureka:
  client:
    service-url:
      defaultZone: http://eureka-node2:8762/eureka,http://eureka-node3:8763/eureka
```

## 三、Nacos

### 为什么需要 Nacos

Nacos = **Eureka + Config + Bus**，一站式解决服务发现、配置管理、服务治理：

- Netflix 组件（Eureka、Config 等）已进入维护模式
- Nacos 功能更强大：AP/CP 可切换、支持配置动态推送、持久化更可靠
- 阿里在亿级流量场景下打磨，国内微服务首选

### Eureka vs Nacos 核心差异

| 维度 | Eureka | Nacos |
|------|--------|-------|
| 定位 | 专注服务发现 | 服务发现 + 配置管理一体 |
| CAP 模型 | 仅 AP | AP/CP 可切换 |
| 健康检查 | 客户端心跳（30s） | 临时实例心跳（5s）+ 非临时实例服务端探测（20s） |
| 服务发现 | 客户端定时拉取（30s 延迟） | 拉取 + 服务端主动推送，变更感知更及时 |
| 配置管理 | 需配合 Spring Cloud Config | 原生集成 |
| 持久化 | 内存 | 支持 MySQL 持久化 |

### Nacos 实例类型

**临时实例（默认）**：
- 客户端心跳上报（默认 5s 一次）
- 15s 未收到心跳 → 标记不健康
- 30s 未收到心跳 → **剔除**实例
- 适合弹性伸缩的无状态服务

**非临时实例**（配置 `ephemeral=false`）：
- 服务端主动探测（默认 20s）
- 探测失败 → 标记不健康，**但不剔除**
- 实例信息永久保留，适合核心服务

### Nacos 一致性协议

Nacos 没有单一选择，根据场景同时使用两种协议：

- **Distro（AP）**：用于临时实例注册。阿里自研最终一致性协议，每节点负责一部分数据，异步 Gossip 同步，写入延迟低、可用性极高。
- **Raft（CP）**：用于非临时实例和配置管理。强一致性，Leader 处理写请求并复制到多数派节点后才返回成功。

### Nacos 三层隔离模型

- **命名空间（Namespace）**：最顶层环境隔离，如 dev/test/prod，不同命名空间服务完全隔离
- **分组（Group）**：同一命名空间内的逻辑分组，如按项目模块分组
- **集群（Cluster）**：同一服务的不同部署集合，如按机房分集群，消费者优先调用同集群实例

关系：`命名空间 > 分组 > 服务 > 集群 > 实例`

### Nacos 服务注册流程

```java
// Spring Cloud 自动配置流程（简化）
// 1. NacosDiscoveryAutoConfiguration 创建 NacosServiceRegistry
// 2. NacosServiceRegistry.register() 封装 Instance 对象
// 3. NamingProxy 向 /nacos/v1/ns/instance 发 POST 请求
// 4. Server 端 InstanceController 处理，ServiceManager 写入内存注册表
// 5. 临时实例：客户端启动 BeatReactor 心跳线程
//    非临时实例：服务端启动主动健康检查任务
```

### 源码深度：Nacos 服务注册全流程

#### 1. 自动装配入口 — NacosDiscoveryAutoConfiguration

Spring Boot 通过 `spring.factories` 加载 `NacosDiscoveryAutoConfiguration`：

```java
// spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  com.alibaba.boot.nacos.discovery.autoconfigure.NacosDiscoveryAutoConfiguration

@ConditionalOnProperty(name = NacosDiscoveryConstants.ENABLED, matchIfMissing = true)
@EnableNacosDiscovery            // 核心注解，触发 Bean 注册
@EnableConfigurationProperties(value = NacosDiscoveryProperties.class)
public class NacosDiscoveryAutoConfiguration {
    @Bean
    public NacosDiscoveryAutoRegister discoveryAutoRegister() {
        return new NacosDiscoveryAutoRegister();  // 监听 WebServerInitializedEvent
    }
}
```

`@EnableNacosDiscovery` 通过 `@Import(NacosDiscoveryBeanDefinitionRegistrar.class)` 触发 Bean 注册流程。

#### 2. Bean 注册流程 — NacosDiscoveryBeanDefinitionRegistrar

```java
public class NacosDiscoveryBeanDefinitionRegistrar
        implements ImportBeanDefinitionRegistrar, EnvironmentAware {

    @Override
    public void registerBeanDefinitions(AnnotationMetadata importingClassMetadata,
            BeanDefinitionRegistry registry) {
        // Step 1: 读取 @EnableNacosDiscovery 注解属性 → 解析占位符 → 注册为全局 Properties Bean
        registerGlobalNacosProperties(attributes, registry, environment,
                DISCOVERY_GLOBAL_NACOS_PROPERTIES_BEAN_NAME);

        // Step 2: 注册公共 Bean（NacosApplicationContextHolder、AnnotationNacosInjectedBeanPostProcessor）
        registerNacosCommonBeans(registry);

        // Step 3: 注册 Discovery 相关 Bean（NacosServiceRegistry 等）
        registerNacosDiscoveryBeans(registry);
    }
}
```

**注册基础设施 Bean 的通用方法**：

```java
public static void registerInfrastructureBean(BeanDefinitionRegistry registry,
        String beanName, Class<?> beanClass, Object... constructorArgs) {
    BeanDefinitionBuilder builder = BeanDefinitionBuilder.rootBeanDefinition(beanClass);
    for (Object arg : constructorArgs) {
        builder.addConstructorArgValue(arg);
    }
    builder.setRole(BeanDefinition.ROLE_INFRASTRUCTURE); // 标记为基础设施 Bean，不暴露给用户
    registry.registerBeanDefinition(beanName, builder.getBeanDefinition());
}
```

#### 3. 触发注册 — NacosDiscoveryAutoRegister

`NacosDiscoveryAutoRegister` 实现 `ApplicationListener<WebServerInitializedEvent>`，在 Web 服务器完成初始化（端口绑定完成）后触发注册：

```java
@Override
public void onApplicationEvent(WebServerInitializedEvent event) {
    if (!discoveryProperties.isAutoRegister()) {
        return;  // 可通过配置关闭自动注册
    }
    Register register = discoveryProperties.getRegister();

    // 自动填充 IP（若未配置）
    if (StringUtils.isEmpty(register.getIp())) {
        register.setIp(NetUtils.localIP());
    }
    // 自动填充 Port（从 Web 容器获取，解决动态端口问题）
    if (register.getPort() == 0) {
        register.setPort(event.getWebServer().getPort());
    }
    register.getMetadata().put("preserved.register.source", "SPRING_BOOT");

    // 调用 NamingService 执行注册
    namingService.registerInstance(serviceName, register.getGroupName(), register);
}
```

**关键点**：监听 `WebServerInitializedEvent` 而非 `ApplicationStartedEvent` 是因为只有 Web 服务器启动后端口才确定，此时才能正确注册 IP+Port。

#### 4. 客户端注册 — NamingService.registerInstance

```java
@Override
public void registerInstance(String serviceName, String groupName, Instance instance) throws NacosException {
    // 只有临时实例才需要心跳
    if (instance.isEphemeral()) {
        BeatInfo beatInfo = new BeatInfo();
        beatInfo.setServiceName(NamingUtils.getGroupedName(serviceName, groupName));
        beatInfo.setIp(instance.getIp());
        beatInfo.setPort(instance.getPort());
        beatInfo.setPeriod(instance.getInstanceHeartBeatInterval()); // 默认 5000ms
        // 注册心跳定时任务
        beatReactor.addBeatInfo(NamingUtils.getGroupedName(serviceName, groupName), beatInfo);
    }
    // 向 Nacos Server 发送注册请求
    serverProxy.registerService(NamingUtils.getGroupedName(serviceName, groupName), groupName, instance);
}
```

#### 5. 心跳机制 — BeatReactor

```java
public void addBeatInfo(String serviceName, BeatInfo beatInfo) {
    String key = buildKey(serviceName, beatInfo.getIp(), beatInfo.getPort());
    // 若已有同 key 的心跳任务，先停止旧任务（防止重复注册）
    BeatInfo existBeat = dom2Beat.remove(key);
    if (existBeat != null) {
        existBeat.setStopped(true);
    }
    dom2Beat.put(key, beatInfo);
    // 创建定时任务，period 默认 5s
    executorService.schedule(new BeatTask(beatInfo), beatInfo.getPeriod(), TimeUnit.MILLISECONDS);
}
```

**BeatTask 执行逻辑**：

```java
@Override
public void run() {
    // 1. 向 /nacos/v1/ns/instance/beat 发送心跳
    JSONObject result = serverProxy.sendBeat(beatInfo, BeatReactor.this.lightBeatEnabled);

    // 2. 检查响应 code
    int code = result.get(CommonParams.CODE).asInt();
    if (code == NamingResponseCode.RESOURCE_NOT_FOUND) {
        // 3. 若 Server 找不到此实例（如 Server 重启），则重新注册
        serverProxy.registerService(beatInfo.getServiceName(), groupName, instance);
    }
    // 4. finally 中重新调度（间隔从 Server 返回值中动态获取）
    executorService.schedule(new BeatTask(beatInfo), nextTime, TimeUnit.MILLISECONDS);
}
```

**学习点**：BeatTask 不是 ScheduledExecutorService 的固定周期任务，而是在 `finally` 中手动 `schedule` 下一次执行，这样可以根据 Server 返回的动态 interval 调整心跳频率。

// 用随机起始索引，避免所有客户端都打到第一个 Nacos 节点
Random random = new Random(System.currentTimeMillis());
int index = random.nextInt(servers.size());
for (int i = 0; i < servers.size(); i++) {
    String server = servers.get(index);
    try {

        // 一台失败，轮询到下一台
        index = (index + 1) % servers.size();
    }
}

#### 7. 服务端处理 — ServiceManager

服务端 `InstanceController` 接收注册请求，委托 `ServiceManager` 处理：

```java
public void registerInstance(String namespaceId, String serviceName, Instance instance) throws NacosException {
    // Step 1: 确保 Service 对象存在（不存在则创建空服务）
    createEmptyService(namespaceId, serviceName, instance.isEphemeral());

    Service service = getService(namespaceId, serviceName);
    // Step 2: 将 Instance 加入 Service 的实例列表
    addInstance(namespaceId, serviceName, instance.isEphemeral(), instance);
}
```

**服务存储结构**：

```java
// 双层 ConcurrentHashMap：namespaceId → serviceName → Service
private Map<String, Map<String, Service>> serviceMap = new ConcurrentHashMap<>();

public void putService(Service service) {
    // 双重检查锁确保 namespace 级别 Map 的初始化
    if (!serviceMap.containsKey(service.getNamespaceId())) {
        synchronized (putServiceLock) {
            if (!serviceMap.containsKey(service.getNamespaceId())) {
                serviceMap.put(service.getNamespaceId(), new ConcurrentHashMap<>(16));
            }
        }
    }
    serviceMap.get(service.getNamespaceId()).put(service.getName(), service);
}
```

**服务初始化** — `putServiceAndInit`：

```java
private void putServiceAndInit(Service service) throws NacosException {
    putService(service);
    service.init();  // 启动 ClientBeatCheckTask（临时实例健康检查）
    // 注册一致性监听器，临时实例用 Distro，非临时用 Raft
    consistencyService.listen(KeyBuilder.buildInstanceListKey(namespaceId, serviceName, true), service);
    consistencyService.listen(KeyBuilder.buildInstanceListKey(namespaceId, serviceName, false), service);
}
```

#### 8. 完整注册时序图

```
客户端启动
    ↓
WebServerInitializedEvent 触发
    ↓
NacosDiscoveryAutoRegister.onApplicationEvent()
    ↓
NamingService.registerInstance()
    ├── BeatReactor.addBeatInfo()  → 启动 5s 心跳定时任务（BeatTask）
    └── NamingProxy.registerService()
            ↓
        POST /nacos/v1/ns/instance
            ↓
        InstanceController.register()
            ↓
        ServiceManager.registerInstance()
            ├── createEmptyService() → putService() → service.init()
            └── addInstance() → consistencyService.put(key, instances)
                                    ↓
                              Distro 协议异步同步到其他节点（临时实例）
                              Raft  协议多数派确认后返回（非临时实例）
```

#### 9. 健康检查 — 服务端 Beat 接口

```
客户端 BeatTask 每 5s 发送: PUT /nacos/v1/ns/instance/beat
    ↓
Server 端: InstanceController.beat()
    ↓
ServiceManager.getInstance() → 找到实例 → 更新最后心跳时间
    ↓
若 RESOURCE_NOT_FOUND（code=20404）→ 客户端感知后重新注册
```

**关键参数**：
- 心跳间隔：5s（`preserved.heart.beat.interval`）
- 健康超时：15s（`preserved.heart.beat.timeout`）
- 删除超时：30s（`preserved.ip.delete.timeout`）

## 四、Consul

Consul 是 HashiCorp 开源的服务网格解决方案，遵循 **CP** 模型。

特点：
- 强一致性（基于 Raft 协议）
- 原生支持多数据中心
- 内置 KV 存储，可用于配置管理
- 支持 DNS 和 HTTP 两种服务发现方式
- 与 Kubernetes 集成良好

适合对一致性要求高的场景，如金融行业核心服务注册。
