# 服务注册与发现

## 一、核心概念

**服务注册与发现**解决微服务实例动态变化带来的寻址问题：

- **服务注册**：实例启动时向注册中心上报自身元数据（IP、端口、服务名等），通过心跳机制维持租约
- **服务发现**：消费者不直接写死地址，而是查询注册中心获取可用实例列表，结合负载均衡算法选择实例

### 发现模式对比

**客户端发现模式（Spring Cloud 默认）**
- 消费者从注册中心获取实例列表，在本地做负载均衡
- 代表：Eureka + Ribbon
- 优点：灵活、去中心化；缺点：客户端复杂度高

**服务端发现模式**
- 消费者通过负载均衡器（API 网关）发请求，网关查注册中心并转发
- 代表：Kubernetes Service
- 优点：对客户端透明；缺点：需维护中间件

## 二、Eureka

### 基本原理

Eureka 是 Netflix 开源的服务发现框架，遵循 **AP** 模型（高可用 + 分区容错）。

**工作流程**：
1. **注册**：Client 启动时向 Server 发 POST 请求注册自身元数据
2. **续约（心跳）**：Client 默认每 **30 秒**发一次心跳，Server 收到后重置租约
3. **拉取列表**：Client 每 **30 秒**从 Server 拉取服务注册表缓存到本地
4. **调用**：消费者用本地缓存 + Ribbon 负载均衡选实例
5. **下线**：Client 关闭时发 DELETE 请求，Server 移除实例

```java
// Eureka Server 启动
@SpringBootApplication
@EnableEurekaServer
public class EurekaServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}
```

```yaml
# Eureka Client 配置
eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka
  instance:
    lease-renewal-interval-in-seconds: 30    # 心跳间隔
    lease-expiration-duration-in-seconds: 90 # 超时时间（90s未收到心跳则剔除）
```

### 三级缓存架构

Eureka Server 用三级缓存平衡性能与一致性：

1. **ReadWriteCache**（读写缓存）：Guava Cache 实现，响应客户端查询请求
2. **ReadOnlyCache**（只读缓存）：定时与读写缓存同步（默认 30s）
3. **Client 本地缓存**：客户端内存缓存，减少网络调用

由于缓存机制，服务状态更新存在最多 **90 秒**延迟。

### 自我保护机制

**触发条件**：15 分钟内心跳丢失比例超过 **15%**（即只有不到 85% 的实例正常心跳）

**工作机制**：
- 进入保护模式后，**不再剔除**因长时间未收到心跳的服务实例
- 继续接受新注册，但不同步到其他节点
- 网络恢复后自动退出保护模式

**设计目的**：防止网络分区时误删服务实例，宁可保留不健康实例，也不大规模剔除。

```yaml
# 生产环境一般保持默认（开启自我保护）
eureka:
  server:
    enable-self-preservation: true
```

### 高可用集群

Eureka Server 节点间**互相注册**，形成去中心化对等集群：

```yaml
# 节点1 配置
eureka:
  client:
    service-url:
      defaultZone: http://eureka-node2:8762/eureka,http://eureka-node3:8763/eureka
```

## 三、Nacos

### 为什么需要 Nacos

Nacos = **Eureka + Config + Bus**，一站式解决服务发现、配置管理、服务治理：

- Netflix 组件（Eureka、Config 等）已进入维护模式
- Nacos 功能更强大：AP/CP 可切换、支持配置动态推送、持久化更可靠
- 阿里在亿级流量场景下打磨，国内微服务首选

### Eureka vs Nacos 核心差异

| 维度 | Eureka | Nacos |
|------|--------|-------|
| 定位 | 专注服务发现 | 服务发现 + 配置管理一体 |
| CAP 模型 | 仅 AP | AP/CP 可切换 |
| 健康检查 | 客户端心跳（30s） | 临时实例心跳（5s）+ 非临时实例服务端探测（20s） |
| 服务发现 | 客户端定时拉取（30s 延迟） | 拉取 + 服务端主动推送，变更感知更及时 |
| 配置管理 | 需配合 Spring Cloud Config | 原生集成 |
| 持久化 | 内存 | 支持 MySQL 持久化 |

### Nacos 实例类型

**临时实例（默认）**：
- 客户端心跳上报（默认 5s 一次）
- 15s 未收到心跳 → 标记不健康
- 30s 未收到心跳 → **剔除**实例
- 适合弹性伸缩的无状态服务

**非临时实例**（配置 `ephemeral=false`）：
- 服务端主动探测（默认 20s）
- 探测失败 → 标记不健康，**但不剔除**
- 实例信息永久保留，适合核心服务

### Nacos 一致性协议

Nacos 没有单一选择，根据场景同时使用两种协议：

- **Distro（AP）**：用于临时实例注册。阿里自研最终一致性协议，每节点负责一部分数据，异步 Gossip 同步，写入延迟低、可用性极高。
- **Raft（CP）**：用于非临时实例和配置管理。强一致性，Leader 处理写请求并复制到多数派节点后才返回成功。

### Nacos 三层隔离模型

- **命名空间（Namespace）**：最顶层环境隔离，如 dev/test/prod，不同命名空间服务完全隔离
- **分组（Group）**：同一命名空间内的逻辑分组，如按项目模块分组
- **集群（Cluster）**：同一服务的不同部署集合，如按机房分集群，消费者优先调用同集群实例

关系：`命名空间 > 分组 > 服务 > 集群 > 实例`

### Nacos 服务注册流程

## 四、Consul

Consul 是 HashiCorp 开源的服务网格解决方案，遵循 **CP** 模型。

特点：
- 强一致性（基于 Raft 协议）
- 原生支持多数据中心
- 内置 KV 存储，可用于配置管理
- 支持 DNS 和 HTTP 两种服务发现方式
- 与 Kubernetes 集成良好

适合对一致性要求高的场景，如金融行业核心服务注册。
