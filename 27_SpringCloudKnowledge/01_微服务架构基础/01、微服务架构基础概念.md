# 微服务架构基础概念

## 一、什么是微服务

微服务（Microservices）是一种软件架构风格，将一个应用程序拆分为一组小型、独立的服务，每个服务：

- **围绕具体业务能力**构建，单一职责
- **独立运行**，有自己的进程
- **通过轻量级机制通信**（通常是 HTTP REST 或消息队列）
- **独立部署和扩展**，互不影响

### 架构演进对比

| 架构 | 粒度 | 数据管理 | 通信方式 | 部署 |
|------|------|----------|----------|------|
| 单体 | 粗粒度 | 共享数据库 | 进程内调用 | 整体部署 |
| SOA | 中粒度 | 可能共享 | ESB 总线 | 服务独立部署 |
| 微服务 | 细粒度 | 每服务独立 | REST/gRPC | 完全独立 |

### 核心优缺点

**优点**
- 技术多样性：每个服务可以用最合适的技术栈
- 独立部署：改一个服务只需重新部署该服务
- 精确扩展：只对瓶颈服务扩容
- 容错隔离：服务故障不会蔓延到整个系统

**缺点**
- 分布式复杂性：网络延迟、分布式事务等挑战
- 运维要求高：大量服务实例需要容器化管理
- 数据一致性难：跨服务数据同步需要最终一致性方案

## 二、CAP 理论

CAP 是分布式系统设计的核心理论，三者只能同时满足其二：

- **C（Consistency 一致性）**：所有节点在同一时间看到相同数据
- **A（Availability 可用性）**：每个请求都能获得响应（不保证最新数据）
- **P（Partition tolerance 分区容错）**：网络分区时系统仍能运行

**在微服务中的应用**：分区容错（P）几乎是必选项，因此实际上是在 CP 和 AP 之间做选择。

- **Eureka**：选择 AP，保证高可用，允许数据短暂不一致
- **Zookeeper/Consul**：选择 CP，保证一致性，分区时可能不可用
- **Nacos**：支持 AP/CP 切换，根据实例类型灵活选择

## 三、领域驱动设计（DDD）

DDD 是微服务拆分的重要方法论，核心思想是通过领域模型指导复杂系统设计。

### 核心概念

- **限界上下文（Bounded Context）**：定义领域模型的边界，是服务拆分的自然边界。订单上下文、用户上下文、库存上下文各自独立。
- **聚合根（Aggregate Root）**：数据修改的入口点，维护业务一致性。`Order` 作为聚合根，包含 `OrderItem`。
- **领域事件（Domain Event）**：表示领域中发生的重要事件，用于服务间通信。`OrderCreatedEvent` 触发库存扣减。
- **实体与值对象**：实体有唯一标识（如用户 ID），值对象描述特征（如地址）。

### 服务拆分原则

1. **单一职责**：每个服务专注一个明确的业务领域
2. **松耦合高内聚**：服务间依赖最小，内部功能高度相关
3. **数据独立**：每个服务拥有自己的数据源，避免共享数据库
4. **参考康威定律**：系统架构往往反映组织的沟通结构，按团队边界划分服务

## 四、微服务设计原则

### 基础设计原则

- **服务自治**：服务独立开发、测试、部署和扩展
- **轻量级通信**：采用 REST、gRPC 等跨语言协议
- **接口版本化**：定义稳定、向后兼容的 API 契约
- **隔离与容错**：通过超时、熔断、降级避免级联故障
- **可观测性**：集成日志、监控和链路追踪

### 粒度把握

服务粒度没有固定标准，实用参考：
- 一个服务由 **2-5 人小团队**维护
- 代码可在 **2 小时内**完全理解
- API 接口数量**不超过 10 个**

## 五、数据一致性方案

微服务采用**最终一致性**而非强一致性：

### Saga 模式

长事务分解为多个本地事务，通过补偿事务回滚：

```
订单创建 → 库存扣减 → 支付
         ↓ 失败
补偿：取消订单 ← 恢复库存
```

### TCC 模式

- **Try**：预留资源（冻结库存）
- **Confirm**：确认操作（实际扣减）
- **Cancel**：释放资源（解冻库存）

### CQRS

命令端处理写操作，查询端提供读模型，两者可以异步同步，适合读多写少场景。

### 事件溯源

存储状态变化事件序列而非当前状态，通过重放事件重建状态，适合审计追踪场景。

## 六、服务拆分策略

**绞杀者模式**：逐步从单体中剥离功能，而非一次性重写。

```
单体应用 → 剥离用户服务 → 剥离订单服务 → ... → 完全微服务化
```

典型演进路径：
1. 单体阶段：快速验证业务
2. 垂直拆分：按功能模块分离（前后端分离）
3. 服务化探索：将稳定功能抽为独立服务
4. 领域驱动拆分：按业务领域深入拆分
5. 微服务成熟：完善监控、治理等基础设施
