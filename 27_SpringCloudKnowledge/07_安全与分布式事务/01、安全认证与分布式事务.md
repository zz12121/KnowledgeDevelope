# 安全认证与分布式事务

## 一、微服务安全问题

微服务架构的安全挑战比单体应用更复杂：

- **攻击面扩大**：服务拆分后内部 API 数量激增
- **服务间通信安全**：东西向流量（服务间）可能被窃听
- **统一认证难**：每个服务都要做认证，重复且不一致
- **密钥管理**：配置中心的密钥、数据库密码等敏感信息

**核心防护思路**：
- API 网关做南北向统一认证（外部请求）
- 服务间 mTLS（双向 TLS）加密（内部请求）
- OAuth 2.0 + JWT 实现无状态认证
- 配置加密保护敏感配置

## 二、OAuth 2.0

### 是什么

OAuth 2.0 是**行业标准授权协议（RFC 6749）**，解决"第三方应用在不需要用户密码的情况下，获得对用户资源的有限访问权限"的问题。

四个核心角色：
- **Resource Owner（资源所有者）**：通常是终端用户
- **Client（客户端）**：希望访问用户资源的第三方应用
- **Authorization Server（授权服务器）**：验证用户身份并颁发令牌
- **Resource Server（资源服务器）**：存放用户资源的服务器

### 四种授权模式

**授权码模式（最常用、最安全）**

适用于有后端的 Web 应用。先通过前端获取授权码，再通过后端用授权码换 Token，Token 不暴露给浏览器。

```
用户 → 客户端 → 授权服务器（登录授权）→ 返回 code
客户端后端 → 用 code 换 Access Token → 访问资源服务器
```

**客户端凭证模式**

适用于机器对机器（M2M）认证，没有用户参与。客户端用自己的 Client ID + Secret 直接申请 Token。微服务间的信任调用常用此模式。

**密码模式**

用户直接把用户名密码交给客户端，仅适用于高度信任的内部应用，不推荐在对外场景使用。

**隐式模式**

已被淘汰，不推荐使用。

## 三、JWT

### 是什么

JWT（JSON Web Token，RFC 7519）是一种**自包含的令牌格式**，用于在各方之间安全地传输信息。

**核心价值**：实现无状态认证。服务器不需要保存 Session，JWT 本身包含了用户信息，验证签名即可。

### 结构

JWT 由三部分组成，用 `.` 分隔：`Header.Payload.Signature`

- **Header**：令牌类型（JWT）+ 签名算法（如 HS256、RS256）
- **Payload**：用户信息（Claims），如 `sub`（用户 ID）、`exp`（过期时间）、`iss`（签发者）等
- **Signature**：对 Header + Payload 的签名，防止篡改

### 工作流程

```
1. 用户登录 → 认证服务器验证通过 → 签发 JWT 返回客户端
2. 客户端后续请求在 Header 中携带 JWT：Authorization: Bearer <token>
3. 资源服务器收到请求 → 验证签名 → 检查过期时间 → 从 Payload 解析用户信息
4. 无需查库，JWT 自包含所有需要的信息
```

```java
// 验证 JWT（java-jwt 库）
Algorithm algorithm = Algorithm.HMAC256("your-secret");
JWTVerifier verifier = JWT.require(algorithm)
    .withIssuer("your-app")
    .build();
DecodedJWT jwt = verifier.verify(token);
String userId = jwt.getSubject();
```

### JWT vs Session

| 维度 | Session | JWT |
|------|---------|-----|
| 存储 | 服务端 | 客户端 |
| 扩展性 | 分布式需要共享 Session（Redis） | 天然无状态，水平扩展简单 |
| 安全性 | Session ID 被盗用风险 | 签名保证不可篡改，但无法主动失效 |
| 适用场景 | 传统 Web 应用 | 微服务、分布式、移动端 |

**JWT 的主要缺点**：签发后无法主动注销（除非维护黑名单）。

## 四、分布式事务

### 为什么难

微服务架构下，一次业务操作往往涉及多个服务（订单服务、库存服务、支付服务），各服务有各自的数据库。**跨服务、跨数据库的数据一致性问题**就是分布式事务的核心挑战。

### BASE 理论

分布式事务通常放弃强一致性（ACID），转而追求**最终一致性**：

- **Basically Available（基本可用）**：故障时允许损失部分可用性
- **Soft State（软状态）**：允许系统存在中间状态
- **Eventually Consistent（最终一致性）**：经过一段时间后数据最终一致

## 五、分布式事务解决方案

### 两阶段提交（2PC）

**强一致性方案**，基于 XA 协议：

- **准备阶段**：协调者向所有参与者发 Prepare，参与者执行但不提交，返回 Ready
- **提交阶段**：所有参与者都 Ready → 协调者发 Commit；有任何失败 → 发 Rollback

**缺点**：协调者宕机会导致参与者长时间阻塞（等锁），性能差，不适合高并发场景。

### TCC 模式

**业务层面的最终一致性方案**：

- **Try**：预留资源（冻结库存 100 件）
- **Confirm**：确认操作（实际扣减库存）
- **Cancel**：释放资源（解冻库存）

```java
// TCC 接口定义
@LocalTCC
public interface OrderTccAction {
    @TwoPhaseBusinessAction(name = "createOrder", commitMethod = "confirm", rollbackMethod = "cancel")
    boolean tryCreateOrder(BusinessActionContext ctx, @BusinessActionContextParameter("orderId") Long orderId);
    
    boolean confirm(BusinessActionContext ctx);
    boolean cancel(BusinessActionContext ctx);
}
```

**优点**：性能较好，业务可以灵活控制。**缺点**：需要编写 3 套业务逻辑，侵入性强。

### Saga 模式

将长事务分解为多个本地事务，每个本地事务成功后发布事件触发下一步；失败则执行补偿事务（反向操作）。

```
订单创建 → 库存扣减 → 支付扣款
              ↓ 失败
支付退款 ← 库存恢复 ← 订单取消  （补偿事务）
```

**优点**：完全异步，无锁，高可用。**缺点**：只能保证最终一致，中间状态对外可见。

### Seata

阿里开源的**分布式事务框架**，支持多种模式：

- **AT 模式（自动）**：无侵入，框架自动生成 undo log，支持自动回滚。适合大多数场景。
- **TCC 模式**：业务手动控制，性能更好。
- **XA 模式**：基于数据库 XA 协议，强一致性。
- **Saga 模式**：长事务场景。

```java
// Seata AT 模式，只需在入口方法加注解
@GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
public void createOrder(Order order) {
    orderService.save(order);      // 操作订单库
    inventoryService.reduce(...);  // 操作库存库
    paymentService.charge(...);    // 操作支付库
    // 任何步骤失败，Seata 自动回滚所有操作
}
```

## 六、本地消息表

适用于对一致性要求不高、可以接受短暂延迟的场景：

```
业务操作 + 写消息表（同一个本地事务）
    ↓
定时任务扫描消息表 → 发送到 MQ → 消费方处理 → 更新消息状态
```

**特点**：实现简单，可靠性高（消息不会丢），但有延迟，不适合实时要求强的场景。

## 七、服务监控与治理

### 服务监控指标

- **健康状态**：服务是否存活（Spring Actuator `/health`）
- **QPS/TPS**：每秒请求量
- **P99 延迟**：99% 请求的响应时间
- **错误率**：失败请求比例
- **线程池/连接池使用率**：资源饱和度

### 服务治理最佳实践

- **幂等性设计**：所有对外接口做幂等，避免重试导致重复操作
- **超时与重试**：设置合理的超时时间，重要接口配置重试（幂等前提下）
- **熔断与降级**：下游故障时快速失败，避免资源耗尽
- **限流**：保护服务不被超出容量的流量压垮
- **灰度发布**：新版本先对小部分用户开放，验证无误后全量
- **服务依赖治理**：定期梳理服务依赖图，避免循环依赖
