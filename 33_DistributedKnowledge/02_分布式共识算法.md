## Paxos 算法

### 核心思想

Paxos 是第一个被证明正确的分布式共识算法，由 Leslie Lamport 在 1989 年提出。它保证在异步网络（消息可能延迟、丢失、重排序）和少数节点故障的情况下，多个节点对**同一个值**达成共识。

### 三种角色

| 角色 | 职责 |
|---|---|
| **Proposer（提议者）** | 提出值，驱动共识过程 |
| **Acceptor（接受者）** | 投票接受或拒绝提议，存储已接受的值 |
| **Learner（学习者）** | 学习最终确定的值（通常是应用层角色） |

### 两阶段流程

**Phase 1：Prepare（准备阶段）**
1. Proposer 选取一个全局递增的编号 `n`，向所有 Acceptor 广播 `Prepare(n)` 请求；
2. Acceptor 收到 `Prepare(n)` 后：
   - 如果 `n` > 自己见过的最大编号，承诺不再接受编号 < `n` 的提议，并返回自己已接受的最高编号提议（如果有）；
   - 否则，拒绝。

**Phase 2：Accept（接受阶段）**
1. Proposer 收到多数 Acceptor 的 Promise 响应后：
   - 如果有 Acceptor 返回了已接受的值，选取其中编号最高的值（**必须尊重历史决定，防止覆盖已达成的共识**）；
   - 否则，可以使用自己想提议的值；
   - 向多数 Acceptor 广播 `Accept(n, v)`；
2. Acceptor 收到 `Accept(n, v)` 后，若 `n` >= 自己承诺的最大编号，则接受该提议。

### Paxos 的局限性

- **活锁问题**：两个 Proposer 交替提高编号，互相阻止对方 Accept，导致系统永远无法达成共识；
- **实现复杂**：单值 Paxos 扩展到 Multi-Paxos（日志复制场景）需要大量工程化改造，细节极多；
- **性能差**：每个提议需要两轮网络往返。

## Raft 算法

Raft 是 2013 年提出的共识算法，目标是"可理解性"（Understandability），用来替代工程实现复杂的 Paxos。

### 三大子问题

1. **Leader 选举（Leader Election）**
2. **日志复制（Log Replication）**
3. **安全性（Safety）**：确保已提交的日志不会被覆盖

### Leader 选举

Raft 集群中每个节点处于三种状态之一：**Follower / Candidate / Leader**。

```
正常状态：
  所有节点 = Follower，接收 Leader 的心跳

Leader 宕机：
  某个 Follower 超时（election timeout，150~300ms 随机），转为 Candidate
  向所有节点发送 RequestVote RPC
  收到多数（>N/2）的赞成票 → 成为 Leader
  立即向所有 Follower 发送心跳，阻止新选举

平票/超时：
  再次随机等待后重新选举（随机 timeout 降低平票概率）
```

**选票条件**：节点只会投票给日志"至少和自己一样新"的 Candidate（比较 last log term，再比较 last log index），确保 Leader 拥有所有已提交的日志。

### 日志复制

```
1. 客户端请求 → Leader 将日志条目追加到本地日志（uncommitted）
2. Leader 并行发送 AppendEntries RPC 给所有 Follower
3. 多数节点（包括 Leader 自身）写入成功 → Leader 提交（committed），应用到状态机
4. Leader 在下次心跳中通知 Follower 提交，Follower 也应用到状态机
5. Leader 返回结果给客户端
```

### Raft vs Paxos 对比

| 维度 | Raft | Paxos |
|---|---|---|
| **设计目标** | 可理解性、可工程实现 | 理论正确性证明 |
| **Leader 机制** | 强 Leader（所有写操作通过 Leader） | 无固定 Leader（Multi-Paxos 可有） |
| **日志复制** | 严格顺序，简单直接 | 可并行提交，但顺序复杂 |
| **Leader 选举** | 随机超时，简单有效 | 需额外机制防活锁 |
| **工程实现难度** | 相对简单 | 极复杂 |
| **代表实现** | etcd、TiKV、Apache Ratis | ZooKeeper（类 Paxos，即 ZAB） |

### Java 实现：Apache Ratis

Apache Ratis 是 Apache 提供的 Raft 协议 Java 实现，TiKV 的 Java SDK 和 HDFS 的部分组件使用。

```java
// Ratis 简单使用示例
RaftGroup group = RaftGroup.valueOf(
    RaftGroupId.randomId(),
    Arrays.asList(peer1, peer2, peer3)
);

RaftServer server = RaftServer.newBuilder()
    .setGroup(group)
    .setStateMachine(new MyStateMachine())  // 业务状态机
    .setProperties(new RaftProperties())
    .build();
server.start();

// 客户端写入
RaftClient client = RaftClient.newBuilder()
    .setProperties(new RaftProperties())
    .setRaftGroup(group)
    .build();
RaftClientReply reply = client.io().send(Message.valueOf("key=value"));
```

## ZAB 协议（ZooKeeper Atomic Broadcast）

ZAB 是 ZooKeeper 专用的原子广播协议，与 Raft 思路相近但细节不同，分为**崩溃恢复**和**消息广播**两大模式。

### 核心概念：ZXID

ZXID（ZooKeeper Transaction ID）是 64 位整数：
- 高 32 位：**epoch（纪元）**，每次 Leader 选举后递增，标识当前 Leader 任期
- 低 32 位：**计数器**，每个事务递增

ZXID 的设计保证了全局有序：新 Leader 的 epoch 更大，可以识别并丢弃旧 Leader 的未提交事务。

### 四阶段协议

| 阶段 | 名称 | 主要工作 |
|---|---|---|
| **Phase 0** | 选举（Leader Election） | Fast Leader Election，选出 ZXID 最大的节点为 Leader |
| **Phase 1** | 发现（Discovery） | Followers 将自己的 epoch 和日志发给 Leader，Leader 确定最新 epoch |
| **Phase 2** | 同步（Synchronization） | Leader 将自己的日志同步给所有 Follower，确保一致性基线 |
| **Phase 3** | 广播（Broadcast） | 正常运行阶段，Leader 广播事务给 Follower，类似 2PC |

### 广播阶段（类 2PC）

```
客户端写请求 → Leader
Leader 生成事务 T(ZXID) → 广播 PROPOSAL 给所有 Follower
Follower 写入磁盘 → 返回 ACK
Leader 收到多数 ACK → 发送 COMMIT
Follower 提交事务 → 应用到内存树
Leader 返回响应给客户端
```

**与标准 2PC 的区别**：ZAB 的 Leader 在收到多数 ACK 后直接提交，不等待所有 Follower，少数 Follower 滞后不影响整体，性能更好。

### 脑裂（Split-Brain）防护

Raft 和 ZAB 都通过 **Quorum 机制（多数派）** 防止脑裂：
- 在网络分区时，只有包含多数节点（> N/2）的分区才能选出新 Leader 并提交事务；
- 少数派分区无法选出 Leader，只能等待网络恢复，不会形成两个并发 Leader；
- 这也是为什么 Raft/ZooKeeper 集群节点数推荐奇数（3/5/7），偶数节点无法提升容灾能力但增加了所需 Quorum 大小。

## 分布式选主模式对比

| 机制 | 原理 | 代表系统 |
|---|---|---|
| **Bully 算法** | 进程 ID 最大的节点为 Leader，简单但通信开销大 | 早期系统 |
| **Raft 随机超时** | 随机化超时降低冲突概率，简单有效 | etcd、TiKV |
| **ZAB Fast Leader Election** | 选 ZXID 最大（日志最新）的节点 | ZooKeeper |
| **基于租约（Lease）** | Leader 持有带过期时间的租约，续约维持 Leader 身份 | etcd lease、Kubernetes |
