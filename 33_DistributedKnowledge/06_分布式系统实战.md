## 秒杀系统设计

### 设计目标

- **系统稳定**：超高并发下不崩溃
- **防超卖**：库存扣减绝对准确
- **保证公平**：防黄牛刷单

### 分层过滤架构

```
前端层
  ├─ 活动页面/商品详情 → CDN 静态化（不打到后端）
  ├─ 倒计时结束后才亮按钮，点击后立即置灰（防重复提交）
  └─ 验证码/滑块（分散峰值，增加机器刷单难度）
      ↓ 过滤 ~95% 流量

网关层
  ├─ 限流（基于用户ID/IP，令牌桶算法）
  ├─ 黑名单（封禁异常用户）
  └─ 频次控制（同一用户每秒最多 N 次请求）
      ↓ 过滤 ~90% 剩余流量

服务层（核心）
  ├─ Redis 预扣库存（Lua 脚本原子操作，防超卖）
  ├─ 幂等性检查（防重复下单）
  └─ 成功 → 发消息到 MQ
      ↓ 只有成功扣减库存的请求进入 MQ

异步处理层
  └─ 消费 MQ，创建订单 + 扣减 DB 库存 + 通知用户
```

### 防超卖核心：Redis Lua 脚本

```lua
-- 秒杀扣库存 Lua 脚本（原子执行）
local stock_key = KEYS[1]     -- "seckill:stock:{productId}"
local bought_key = KEYS[2]    -- "seckill:bought:{productId}"（已购用户集合）
local user_id = ARGV[1]
local quantity = tonumber(ARGV[2])

-- 1. 防重复购买
if redis.call('SISMEMBER', bought_key, user_id) == 1 then
    return -1  -- 已购买
end

-- 2. 检查库存
local stock = tonumber(redis.call('GET', stock_key))
if not stock or stock < quantity then
    return 0   -- 库存不足
end

-- 3. 原子扣减
redis.call('INCRBY', stock_key, -quantity)
redis.call('SADD', bought_key, user_id)
return 1  -- 成功
```

### 缓存预热

```java
@Component
public class SeckillCacheWarmer implements ApplicationRunner {
    @Override
    public void run(ApplicationArguments args) {
        // 系统启动时将今日秒杀商品库存加载到 Redis
        List<SeckillProduct> products = seckillProductService.getTodayProducts();
        for (SeckillProduct p : products) {
            String stockKey = "seckill:stock:" + p.getProductId();
            redis.opsForValue().set(stockKey, String.valueOf(p.getStock()));
            // 设置到活动结束时间 + buffer 后过期
            redis.expireAt(stockKey, p.getEndTime().plusHours(1));
        }
        log.info("秒杀缓存预热完成，共 {} 个商品", products.size());
    }
}
```

## 幂等性设计

### 幂等性三要素

1. **唯一标识**：每个请求携带全局唯一 ID（clientOrderId / messageId）
2. **状态检查**：执行前检查该 ID 是否已处理
3. **原子操作**：检查和执行必须是原子的（防并发重复执行）

### 各场景解决方案

**场景1：HTTP API 防重复提交**

```java
// 客户端生成 idempotencyKey，服务端 Redis 去重
@PostMapping("/orders")
public Result createOrder(@RequestHeader("Idempotency-Key") String idempotencyKey,
                          @RequestBody OrderRequest request) {
    String cacheKey = "idempotent:order:" + idempotencyKey;
    
    // 原子检查+写入（Redis SET NX）
    Boolean isNew = redis.opsForValue()
        .setIfAbsent(cacheKey, "processing", Duration.ofMinutes(10));
    
    if (Boolean.FALSE.equals(isNew)) {
        // 已处理，直接返回缓存的结果
        String cachedResult = redis.opsForValue().get(cacheKey + ":result");
        return cachedResult != null ? Result.success(cachedResult) 
                                    : Result.success("处理中，请稍候查询");
    }
    
    try {
        Order order = orderService.createOrder(request);
        redis.opsForValue().set(cacheKey + ":result", 
                                order.getOrderId(), Duration.ofMinutes(10));
        redis.opsForValue().set(cacheKey, "done", Duration.ofMinutes(10));
        return Result.success(order);
    } catch (Exception e) {
        redis.delete(cacheKey);  // 失败时清除，允许重试
        throw e;
    }
}
```

**场景2：消息队列消费幂等**

```java
@KafkaListener(topics = "payment-events")
public void handlePaymentEvent(PaymentEvent event) {
    String key = "mq:consumed:" + event.getMessageId();
    
    // SET NX 去重（过期时间根据消息保留时间设置）
    Boolean isFirst = redis.opsForValue()
        .setIfAbsent(key, "1", Duration.ofDays(7));
    
    if (Boolean.FALSE.equals(isFirst)) {
        log.info("消息 {} 已消费，跳过重复处理", event.getMessageId());
        return;
    }
    
    // 执行业务逻辑
    processPayment(event);
}
```

**场景3：数据库乐观锁（状态机幂等）**

```java
// 订单支付：只能从 PENDING → PAID，天然幂等
int updated = orderRepository.updateStatus(
    orderId, "PENDING", "PAID", version
);
// UPDATE orders SET status='PAID', version=version+1
// WHERE order_id=? AND status='PENDING' AND version=?
if (updated == 0) {
    // 0 行更新：要么已支付（幂等），要么并发冲突
    Order order = orderRepository.findById(orderId);
    if ("PAID".equals(order.getStatus())) {
        return;  // 已支付，幂等返回成功
    }
    throw new OptimisticLockException("并发冲突，请重试");
}
```

## 高可用分布式锁服务设计

### RedLock 算法（Redis 多节点锁）

针对单 Redis 节点锁的高可用问题，Redis 官方提出了 RedLock 算法：

```
部署 5 个独立 Redis 节点（非主从，非集群）
获取锁：
  1. 记录当前时间 T1
  2. 按顺序向 5 个节点尝试获取锁（每个节点超时时间很短，如 50ms）
  3. 计算总耗时 ΔT = now - T1
  4. 如果在 3 个或以上节点成功获取锁，且 ΔT < 锁有效时间，则获取成功
  5. 实际持锁时间 = 锁有效时间 - ΔT

释放锁：向所有 5 个节点发送释放命令（无论是否成功获取了锁）
```

**RedLock 争议**：Martin Kleppmann 指出 RedLock 在时钟漂移场景下存在安全问题。Redisson 通过在业务代码中使用**fencing token（栅栏令牌）** 来进一步保护。大多数业务场景下，Redisson 的单 Redis + 看门狗方案已经足够。

## 异地多活实践

### 用户分区路由策略

```java
// 根据用户 ID 确定主场机房
public String getHomeDC(long userId) {
    // 用户 ID 按 Shard 数取模，奇数 → DC1，偶数 → DC2
    int shard = (int)(userId % 2);
    return shard == 0 ? "DC_SHANGHAI" : "DC_BEIJING";
}

// 请求路由
public Response handleRequest(long userId, Request request) {
    String homeDC = getHomeDC(userId);
    String currentDC = getCurrentDC();
    
    if (homeDC.equals(currentDC)) {
        // 在主场机房，直接处理（读写都可以）
        return localService.handle(request);
    } else {
        // 不在主场，写操作需要路由到主场（避免数据冲突）
        if (request.isWrite()) {
            return rpcClient.forwardTo(homeDC, request);
        } else {
            // 读操作可以用本机房数据（接受短暂延迟）
            return localService.handle(request);
        }
    }
}
```

### 数据同步：Binlog + MQ

```
主场 DC1（MySQL 主）
    ↓ Canal 监听 Binlog
    ↓ 发送变更消息到 MQ（含 DC 标识，防止回环）
MQ（跨机房 Kafka，异步复制）
    ↓
从场 DC2（消费 MQ，写入本地 MySQL 从库）
    ↓
DC2 本地 MySQL（数据延迟约 100~500ms）
```

**防回环**：消息携带 `sourceDC` 字段，消费者检查如果 `sourceDC == currentDC` 则跳过，防止数据无限循环同步。

## 分布式系统设计面试速查

| 场景 | 关键设计点 |
|---|---|
| **下单防重复** | 幂等 Key（客户端生成 UUID）+ Redis NX + DB 唯一索引兜底 |
| **支付防超扣** | 数据库乐观锁（version 字段）+ 状态机（PENDING→PAID 单向流转） |
| **缓存与 DB 一致** | Cache Aside 模式；更新 DB 后删缓存（而非更新）；Binlog 监听兜底 |
| **服务间通信失败** | 熔断 + 降级 + 重试（指数退避 + 抖动）+ 超时控制 |
| **大促流量洪峰** | 限流（令牌桶）+ 消息队列削峰 + 弹性扩容（HPA）+ Redis 预扣库存 |
| **数据库扩容** | 分库分表（ShardingSphere）+ 双写迁移（逐步切读流量）+ 最终下线旧库 |
| **跨服务事务** | 优先考虑消息最终一致性（本地消息表/事务消息）；必须强一致则 Seata AT |
