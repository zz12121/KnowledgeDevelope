## CAP 理论

### 三个属性

- **C（Consistency，强一致性）**：所有节点在同一时刻看到相同的数据。任意节点返回的数据都是最新的写入结果。
- **A（Availability，可用性）**：每个请求都能在有限时间内得到响应（不保证返回的是最新数据）。
- **P（Partition Tolerance，分区容忍性）**：系统能够容忍网络分区（节点间通信失败）仍然正常运行。

### P 为什么不能放弃？

**网络分区在分布式系统中是必然发生的**——网络延迟、丢包、机器宕机随时可能发生。如果放弃 P，意味着一旦发生网络分区就停止服务，这对生产系统是不可接受的。

因此，实际工程中 CAP 的选择是：**P 必须保留，在 C 和 A 之间权衡**。

### CA 的工程含义

| 选择 | 代表系统 | 行为描述 |
|---|---|---|
| **CP** | ZooKeeper、etcd、HBase | 网络分区时，拒绝服务（返回错误），保证数据一致性 |
| **AP** | Eureka、Cassandra、CouchDB | 网络分区时，仍然提供服务，可能返回旧数据 |

**ZooKeeper 为什么是 CP？**
ZooKeeper 基于 ZAB 协议，写操作需要超过半数节点确认才能提交。如果发生网络分区，少数节点组无法达到 Quorum，写操作被拒绝，从而牺牲了可用性保证了一致性。

**Eureka 为什么是 AP？**
Eureka 的设计哲学是"宁可返回旧数据，也不拒绝服务"。网络分区时，各个 Eureka Server 继续对外服务，并保留自己的注册表（自我保护模式：不再剔除注册信息），即使注册表可能已过期。

```java
// Eureka 自我保护模式配置
# 当心跳续约比例低于阈值（默认85%），开启自我保护，停止剔除实例
eureka.server.enable-self-preservation=true
eureka.server.renewal-percent-threshold=0.85
```

## BASE 理论

BASE 是 CAP 中 **AP 方案**的工程延伸，是对强一致性的妥协，但并非放弃一致性。

| 原则 | 含义 |
|---|---|
| **BA（Basically Available，基本可用）** | 系统出现故障时，允许损失部分可用性（如响应时间延长、部分功能降级），但核心功能仍可用 |
| **S（Soft State，软状态）** | 允许系统中存在中间状态（如订单"支付中"），中间状态不影响系统整体可用性 |
| **E（Eventually Consistent，最终一致性）** | 系统中所有副本的数据，经过一段时间的同步，**最终会达到一致**的状态 |

### 最终一致性的几种形式

| 形式 | 说明 | 典型场景 |
|---|---|---|
| **因果一致性** | 有因果关系的操作，观察顺序与因果顺序一致 | 评论系统（先有帖子才能评论） |
| **读自己写一致性** | 用户能读到自己刚写入的数据 | 用户修改个人资料后立刻看到更新 |
| **会话一致性** | 同一会话内保证读自己写一致性 | 同一用户的操作保证自身一致 |
| **单调读一致性** | 多次读取不会读到比上次更旧的数据 | 防止时间倒退 |
| **最终一致性** | 无其他更新时，最终所有副本达到一致 | 商品库存跨机房同步 |

## PACELC 模型

CAP 只描述了**网络分区（P）**发生时的取舍，但忽略了正常运行时（无分区）的延迟 vs 一致性权衡。PACELC 是对 CAP 的扩展：

```
If P (partition) then:
    C (consistency) or A (availability)
ELC (Else, when no partition):
    L (latency) or C (consistency)
```

**三类系统分类**

| 类型 | 分区时 | 正常时 | 代表系统 |
|---|---|---|---|
| **PA / EL** | 选 A（可用性） | 选 L（低延迟） | DynamoDB、Cassandra、Riak |
| **PC / EC** | 选 C（一致性） | 选 C（强一致） | HBase、BigTable、ZooKeeper |
| **PA / EC** | 选 A（可用性） | 选 C（一致性） | MySQL（默认同步，但主从切换短暂 AP） |

PACELC 帮助我们在系统设计时更全面地评估一致性与性能的取舍，而不仅仅在极端故障场景下。

## 逻辑时钟与向量时钟

分布式系统中，各节点没有全局时钟，无法通过物理时间判断事件先后顺序。逻辑时钟用来建立**事件的偏序关系（happened-before）**。

### Lamport 逻辑时钟

**规则**：
1. 每个进程维护一个本地计数器 `C`；
2. 进程每发生一个内部事件，`C++`；
3. 进程发送消息时，将当前 `C` 附在消息中；
4. 进程接收消息时，`C = max(本地C, 消息C) + 1`。

**局限**：Lamport 时钟只能判断"事件 A 可能在事件 B 之前发生"，但无法区分**并发事件**（两个互不影响的事件）。

### 向量时钟（Vector Clock）

每个进程维护一个向量 `V[1..N]`（N 为进程数），精确刻画各进程的"认知状态"。

```
进程 A 发送消息，携带自己的向量 [A:2, B:0, C:0]
进程 B 接收后更新为 [A:2, B:1, C:0]（逐维取 max，然后自增）
```

**比较规则**：
- V1 < V2（V1 happened-before V2）：V1 每一维都 ≤ V2 且至少一维 <
- V1 和 V2 **并发**：V1 某维 > V2 对应维，同时 V1 某维 < V2 对应维

**应用**：Amazon DynamoDB 使用向量时钟检测冲突写入，当检测到两个版本并发时，将冲突数据都返回给客户端，由应用层或用户决定合并策略（"购物车商品合并"是经典案例）。

```java
// 向量时钟简单实现示意
public class VectorClock {
    private final Map<String, Integer> clock = new HashMap<>();

    public void increment(String nodeId) {
        clock.merge(nodeId, 1, Integer::sum);
    }

    public void update(Map<String, Integer> received) {
        received.forEach((node, time) ->
            clock.merge(node, time, Math::max)
        );
    }

    public boolean isConcurrentWith(VectorClock other) {
        boolean thisGreater = false, otherGreater = false;
        Set<String> allNodes = new HashSet<>(clock.keySet());
        allNodes.addAll(other.clock.keySet());
        for (String node : allNodes) {
            int t = clock.getOrDefault(node, 0);
            int o = other.clock.getOrDefault(node, 0);
            if (t > o) thisGreater = true;
            if (o > t) otherGreater = true;
        }
        return thisGreater && otherGreater; // 两者都有对方没有的更新 = 并发
    }
}
```

## 一致性模型层次

从强到弱，常见一致性模型：

```
线性一致性（Linearizability）  ← 最强
    所有操作看起来是原子的、按全局时间顺序发生
    代表：ZooKeeper 读（sync + read）、etcd

顺序一致性（Sequential Consistency）
    所有进程看到的操作顺序相同，但不要求与实时时钟对齐
    代表：Java volatile（单变量语义）

因果一致性（Causal Consistency）
    有因果关系的操作保序，并发操作可不同顺序

最终一致性（Eventual Consistency）  ← 最弱
    无冲突时最终所有副本达到一致
    代表：DNS、Cassandra、Eureka
```

工程实践中，大多数分布式数据库（Cassandra、DynamoDB）通过**可调一致性（Tunable Consistency）**在强弱之间灵活选择，如 Cassandra 的 `QUORUM`（多数派读写）。
