## 服务注册与发现

### 核心流程

```
服务提供者启动 → 向注册中心注册（IP + 端口 + 服务名 + 元数据）
                ↓ 定期心跳续约（保持活跃）
服务消费者     → 订阅/查询注册中心获取实例列表
                ↓ 本地缓存实例列表
                ↓ 负载均衡选择实例
                ↓ 发起调用
注册中心       → 监控心跳超时，剔除失效实例 → 通知订阅者更新
```

### Eureka vs Nacos vs ZooKeeper

| 维度 | Eureka | Nacos | ZooKeeper |
|---|---|---|---|
| **一致性模型** | AP（最终一致，自我保护模式） | 支持 AP（默认）和 CP 两种模式 | CP（ZAB 协议，强一致） |
| **健康检查** | 客户端心跳 | 客户端心跳 + 服务端主动探测 | 会话机制（临时节点自动删除） |
| **推送机制** | 客户端轮询 + 增量同步 | UDP 推送 + 长轮询 | Watch 机制（变更时推送） |
| **配置管理** | 不支持 | 支持（一站式） | 可用，但不是主要功能 |
| **维护状态** | 停止维护（2019） | 活跃维护，Spring Cloud Alibaba | 活跃维护 |

## 负载均衡

### 服务端负载均衡 vs 客户端负载均衡

| | 服务端负载均衡 | 客户端负载均衡 |
|---|---|---|
| **架构** | 独立的负载均衡器（Nginx/LVS）位于客户端和服务端之间 | 客户端内置负载均衡逻辑，直接选择目标实例 |
| **优点** | 对客户端透明，集中管理 | 减少一次网络跳转，延迟低 |
| **缺点** | 负载均衡器是单点（需高可用），需维护额外组件 | 客户端需集成 SDK，升级困难 |
| **代表** | Nginx、LVS、F5、云 SLB/ELB | Ribbon、Spring Cloud LoadBalancer |

### LVS 三种工作模式

**LVS（Linux Virtual Server）**是四层（传输层）负载均衡，工作在内核空间，性能极高（百万 QPS）。

| 模式 | 原理 | 特点 |
|---|---|---|
| **NAT（网络地址转换）** | LVS 同时修改请求和响应的 IP/端口，流量经过 LVS | 部署简单，但 LVS 是带宽瓶颈 |
| **TUN（IP 隧道）** | 请求经 LVS，响应直接从 RS 返回客户端（LVS 和 RS 可在不同网段） | RS 可跨地域，响应不经 LVS | 
| **DR（直接路由）** | 请求经 LVS 转发（修改 MAC 地址），响应直接从 RS 返回（LVS 和 RS 在同一局域网） | 性能最好，最常用；LVS 和 RS 必须在同一网段 |

**LVS + Keepalived** 是经典高可用方案：两台 LVS 通过 Keepalived 实现主备，VIP 漂移保证高可用。

### Nginx 七层负载均衡策略

```nginx
upstream backend {
    # 1. 轮询（默认）
    server 192.168.1.1:8080;
    server 192.168.1.2:8080;
    
    # 2. 加权轮询
    server 192.168.1.1:8080 weight=3;
    server 192.168.1.2:8080 weight=1;
    
    # 3. IP Hash（同一 IP 始终路由到同一后端，解决 Session 问题）
    ip_hash;
    
    # 4. 最少连接（least_conn）
    least_conn;
    
    # 5. 健康检查（upstream_check_module 模块）
    check interval=3000 rise=2 fall=3 timeout=1000 type=http;
}
```

## 分布式锁

### 三种实现对比

| 实现 | 性能 | 可靠性 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| **Redis（SET NX PX）** | 高 | 中（单 Redis 存在极端脑裂风险） | 低 | 高性能，容忍极少概率的锁失效 |
| **ZooKeeper（临时顺序节点）** | 中 | 高（强一致性） | 中 | 强一致性要求，如分布式选主 |
| **数据库（唯一索引）** | 低 | 高 | 低 | 低频操作，已有 DB 环境 |

### Redis 分布式锁

**正确实现要点**：
1. SET NX PX 原子操作（不能分两步）；
2. Value 存 UUID，释放锁时验证是否是自己的锁（Lua 脚本保证原子性）；
3. Redisson 看门狗机制自动续期，防止持有锁的线程业务未完成锁就过期。

```java
// Redisson 可重入锁（推荐生产使用）
RLock lock = redissonClient.getLock("order:lock:" + orderId);
try {
    // 尝试获取锁，最多等待 3s，持有 30s（看门狗会自动续期）
    if (lock.tryLock(3, 30, TimeUnit.SECONDS)) {
        try {
            // 业务逻辑
            processOrder(orderId);
        } finally {
            lock.unlock();  // 释放锁
        }
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
```

**Redisson 看门狗原理**：持有锁的线程启动一个后台 Watchdog 定时任务（默认每 10s 执行一次），检测业务是否还在运行，如果是则将锁的过期时间延长（默认 30s），直到业务完成或线程结束。

**Redisson 可重入锁原理**：底层使用 Redis Hash 结构，field 为 `threadId`，value 为重入次数。同一线程多次获取锁时，重入计数 +1；释放时 -1，计数为 0 时才真正删除锁。

### ZooKeeper 分布式锁

```java
// 基本思路：临时顺序节点 + Watch
// 1. 在 /locks/my-lock/ 下创建临时顺序节点：/locks/my-lock/lock-0000000001
// 2. 获取该目录下所有子节点，排序
// 3. 如果自己是最小节点，获取锁
// 4. 否则，Watch 自己前一个节点（不是最小节点），避免羊群效应
// 5. 当前一个节点被删除时（持锁者释放或宕机），收到 Watch 通知，重新判断
```

**避免羊群效应**：每个等待节点只 Watch 自己的前一个节点，而不是 Watch 所有节点。这样每次只有一个等待者被唤醒，而不是所有等待者同时被唤醒后竞争。

## 限流算法

### 四种算法详解

**1. 固定窗口（计数器）**

```java
// 每分钟最多 100 个请求
String key = "ratelimit:" + userId + ":" + (currentTimeMillis() / 60000);
long count = redis.incr(key);
if (count == 1) redis.expire(key, 60);
if (count > 100) throw new RateLimitException("请求过于频繁");
```

**缺陷**：窗口临界点突刺——假设限制每分钟 100 个，用户在 00:59 发送 100 个，在 01:00 再发 100 个，在两秒内达到 200 个请求，超过限制。

**2. 滑动窗口**

```java
// 使用 Redis ZSet 实现精准滑动窗口
String key = "sliding:" + userId;
long now = System.currentTimeMillis();
long windowStart = now - 60000; // 1分钟窗口

// Lua 脚本保证原子性
String script = """
    redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', ARGV[1])  -- 清除窗口外数据
    local count = redis.call('ZCARD', KEYS[1])                 -- 当前窗口请求数
    if count < tonumber(ARGV[3]) then
        redis.call('ZADD', KEYS[1], ARGV[2], ARGV[2])          -- 记录本次请求
        redis.call('EXPIRE', KEYS[1], 60)
        return 1  -- 放行
    end
    return 0  -- 限流
    """;
Object result = redis.eval(script, List.of(key), List.of(
    String.valueOf(windowStart), String.valueOf(now), "100"));
```

**3. 令牌桶**

令牌以固定速率放入桶中（如 100 个/秒），请求来时消耗令牌，无令牌则限流。支持**突发流量**（桶中积累的令牌可以消耗）。Java 实现：`RateLimiter`（Guava）。

**4. 漏桶**

请求以任意速率流入桶，以固定速率流出（处理）。桶满则丢弃请求。**严格限制处理速率**，不允许突发，适合保护下游。Nginx 的 `limit_req` 模块使用漏桶算法。

**对比总结**

| | 固定窗口 | 滑动窗口 | 令牌桶 | 漏桶 |
|---|---|---|---|---|
| **突发流量** | 部分支持（临界点问题） | 不支持 | 支持 | 不支持 |
| **实现复杂度** | 低 | 中 | 中 | 中 |
| **精准度** | 低（临界点问题） | 高 | 高 | 高 |
| **适用场景** | 粗粒度限流 | 精准 API 限流 | 允许突发的场景 | 保护下游速率 |

## 熔断器

### 三种状态

```
CLOSED（关闭）─── 失败率达阈值 ──→ OPEN（打开）
     ↑                                    ↓
     │                              等待复位时间
     │                                    ↓
     └─ 探测请求成功 ─── HALF-OPEN（半开）
```

- **CLOSED（关闭）**：正常状态，请求正常通过，统计失败率
- **OPEN（打开）**：熔断触发，直接拒绝请求，执行降级逻辑（快速失败）
- **HALF-OPEN（半开）**：等待复位时间后，放行少量探测请求，若成功则关闭，若失败则重新打开

### Sentinel vs Hystrix

| 维度 | Hystrix | Sentinel |
|---|---|---|
| **维护状态** | 停止维护（2018） | 活跃维护（阿里开源） |
| **限流算法** | 信号量/线程池隔离 | 多种算法（QPS/并发数/系统负载） |
| **熔断策略** | 异常比例 | 异常比例 + 异常数 + 慢调用比例 |
| **控制台** | Hystrix Dashboard（功能弱） | Sentinel Dashboard（实时规则推送） |
| **扩展性** | 插件有限 | 丰富的扩展 SPI |
| **选型建议** | 遗留系统维护 | 新项目首选 |
