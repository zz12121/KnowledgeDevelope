## 一致性哈希

### 普通哈希分片的问题

```java
// 普通取模分片
int node = hash(key) % N;  // N 为节点数
```

当节点数 N 变化时（增加或删除节点），几乎所有的 key 都需要重新分配，导致**大量数据迁移**，对正在运行的服务影响极大。

### 一致性哈希原理

1. 将整个哈希空间抽象为一个首尾相连的圆环（0 ~ 2³²-1）；
2. 将**节点**和**数据**都映射到同一个哈希环上；
3. 查找数据所在节点：从数据的哈希值出发，**顺时针方向找到的第一个节点**就是负责存储该数据的节点。

```
哈希环示意：

                  Node A (hash=100)
                /
    ─── data_x → Node A
   |                    \
 Node C (hash=800)    data_y → Node B
   |                    /
    ─── Node B (hash=500)
```

**节点增减时只影响相邻区间**：增加一个节点，只有该节点与其前驱节点之间的数据需要迁移（约 1/N 的数据），而不是全量重哈希。

### 虚拟节点解决数据倾斜

如果物理节点少，节点在哈希环上分布不均匀，会造成数据倾斜（某些节点数据量远多于其他节点）。

**解决方案**：每个物理节点虚拟出多个**虚拟节点**（如 150 个），均匀分布在哈希环上：

```java
// 一致性哈希 + 虚拟节点实现
public class ConsistentHash {
    private final TreeMap<Integer, String> ring = new TreeMap<>();
    private final int virtualNodes;

    public void addNode(String node) {
        for (int i = 0; i < virtualNodes; i++) {
            String virtualKey = node + "#VN" + i;
            ring.put(hash(virtualKey), node);
        }
    }

    public String getNode(String key) {
        if (ring.isEmpty()) return null;
        int hash = hash(key);
        // 顺时针找到第一个节点
        Map.Entry<Integer, String> entry = ring.ceilingEntry(hash);
        if (entry == null) {
            entry = ring.firstEntry(); // 环形，超出最大值则回到第一个
        }
        return entry.getValue();
    }

    private int hash(String key) {
        // 使用 MurmurHash 等分布均匀的哈希函数
        return Math.abs(key.hashCode());
    }
}
```

### Redis Cluster 哈希槽

Redis Cluster 没有使用严格的一致性哈希，而是使用固定的 **16384 个哈希槽**：
```
slot = CRC16(key) % 16384
```
每个 Master 节点负责一定范围的槽（如 0~5460、5461~10922、10923~16383）。扩容时只需将部分槽从旧节点迁移到新节点，比一致性哈希更简单直接，属于一致性哈希的简化变体。

## 分库分表

### 四种拆分方式

| 方式 | 定义 | 场景 |
|---|---|---|
| **垂直分库** | 按业务模块拆分，每个库只包含一个业务的表 | 单体服务演进为微服务，每个服务独立数据库 |
| **水平分库** | 同一张表的数据按规则分散到多个库 | 单库存储量/连接数到达瓶颈 |
| **垂直分表** | 将宽表按列拆分，常用列一张表，大字段/不常用列另一张表 | 表列数过多、存在大字段影响查询性能 |
| **水平分表** | 同一张表的数据按规则分散到多个表（同库或跨库） | 单表数据量过大（通常 > 2000 万行） |

> **记忆口诀**：垂直是"瘦身"（减少列/表），水平是"分身"（减少行）。

### ShardingSphere 五步处理流程

```
SQL 解析（Parse）
    ↓ 将 SQL 解析为抽象语法树（AST），提取 table/condition/orderBy 等信息
SQL 路由（Route）
    ↓ 根据分片键值，计算目标库和目标表（如 user_id % 4 → db_1.user_2）
SQL 改写（Rewrite）
    ↓ 将逻辑 SQL（user 表）改写为物理 SQL（user_2 表），处理分页/聚合函数
SQL 执行（Execute）
    ↓ 并行向各目标库发送改写后的 SQL
结果归并（Merge）
    ↓ 收集各库结果，做内存排序/聚合/分页，返回给应用
```

### 跨分片查询的挑战

1. **跨分片排序**：`ORDER BY` 需要各分片分别排序，再在内存中归并（归并排序）；
2. **跨分片分页**：`LIMIT 10 OFFSET 1000` 需要从所有分片各取 1010 条，归并排序后再取第 1001~1010 条，性能随页码增大急剧下降；
3. **跨分片 JOIN**：应尽量避免，通过业务层拼装（先查一侧数据，再 IN 查另一侧）或使用绑定表（同规则分片的关联表）；
4. **跨分片事务**：需要分布式事务（Seata/XA），性能较差，尽量通过业务设计规避。

## 分布式 ID

分库分表后，自增主键在各库各自递增，会产生 ID 冲突，需要全局唯一 ID。

### 五种方案对比

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **UUID** | 随机生成 128 位 | 简单，无依赖 | 无序（影响 B+ 树插入性能），存储长（36字符） |
| **数据库号段** | 批量从 DB 获取一段 ID（如每次取 1000 个），内存中顺序发放 | 有序，依赖低 | DB 是单点，需要高可用方案 |
| **Redis INCR** | 利用 Redis 原子自增 | 简单，性能高 | Redis 持久化配置不当可能重复 |
| **雪花算法（Snowflake）** | 64 位整数，本地生成，毫秒级有序 | 高性能，有序，分布式 | 时钟回拨问题，强依赖机器时钟 |
| **美团 Leaf** | 双号段 + 雪花的混合实现 | 高可用，解决时钟回拨 | 依赖 ZooKeeper 分配机器 ID |

### 雪花算法详解

**64 位结构（1 + 41 + 5 + 5 + 12 = 64）**：

```
[0][0000000000 0000000000 0000000000 0000000000 0][00000][00000][000000000000]
 ↑                          ↑                       ↑       ↑          ↑
符号位(1)            毫秒时间戳(41位)          数据中心(5) 机器(5)  序列号(12)
```

- **时间戳（41位）**：毫秒级，可以使用约 69 年（2^41 / 1000 / 3600 / 24 / 365 ≈ 69）
- **数据中心 ID（5位）**：最多 32 个数据中心
- **机器 ID（5位）**：每个数据中心最多 32 台机器
- **序列号（12位）**：同一毫秒内最多生成 4096 个 ID

```java
public class SnowflakeIdGenerator {
    private static final long DATA_CENTER_ID_BITS = 5L;
    private static final long MACHINE_ID_BITS = 5L;
    private static final long SEQUENCE_BITS = 12L;
    private static final long MAX_SEQUENCE = ~(-1L << SEQUENCE_BITS);  // 4095

    private final long dataCenterId;
    private final long machineId;
    private long lastTimestamp = -1L;
    private long sequence = 0L;

    public synchronized long nextId() {
        long timestamp = currentTimeMillis();
        if (timestamp < lastTimestamp) {
            // 时钟回拨处理：等待时钟追上，超出阈值则抛异常
            long offset = lastTimestamp - timestamp;
            if (offset <= 5) {  // 5ms 内等待
                try { Thread.sleep(offset << 1); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
                timestamp = currentTimeMillis();
                if (timestamp < lastTimestamp) throw new RuntimeException("时钟回拨超出容忍范围");
            } else {
                throw new RuntimeException("时钟回拨 " + offset + "ms，无法生成 ID");
            }
        }
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & MAX_SEQUENCE;
            if (sequence == 0) timestamp = waitNextMillis(lastTimestamp); // 序列号溢出，等待下一毫秒
        } else {
            sequence = 0L;
        }
        lastTimestamp = timestamp;
        return ((timestamp - 1609459200000L) << (DATA_CENTER_ID_BITS + MACHINE_ID_BITS + SEQUENCE_BITS))
             | (dataCenterId << (MACHINE_ID_BITS + SEQUENCE_BITS))
             | (machineId << SEQUENCE_BITS)
             | sequence;
    }
}
```

## 分布式事务

### 四种模式对比

| 模式 | 原理 | 侵入性 | 性能 | 适用场景 |
|---|---|---|---|---|
| **2PC（XA）** | 协调者统一 Prepare/Commit，强一致 | 低 | 差（锁持有时间长） | 强一致要求，如银行转账 |
| **TCC** | Try-Confirm-Cancel，业务自实现 | 高（需实现3个方法） | 好 | 强一致 + 高性能，如支付 |
| **Saga** | 一系列本地事务 + 补偿操作，最终一致 | 中 | 好 | 长流程，如订单履约 |
| **本地消息表** | 本地 DB 存消息 + MQ 异步通知，最终一致 | 中 | 好 | 跨服务异步通知 |

### Seata 四种模式

**AT 模式（Auto Transaction，零侵入推荐）**

```
1. 业务 SQL 执行前，框架解析 SQL，生成 undo_log（记录数据前镜像）
2. 执行业务 SQL（如 UPDATE stock SET count = count - 1）
3. 同时写入 undo_log（记录数据后镜像）
4. 全局事务提交：TC 通知所有 RM 清除 undo_log
5. 全局事务回滚：TC 通知所有 RM 通过 undo_log 反向补偿
```

AT 模式通过**数据行锁 + undo_log 自动补偿**实现分布式事务，对业务代码零侵入，只需加 `@GlobalTransactional` 注解。

```java
@Service
public class OrderService {
    @GlobalTransactional  // Seata AT 模式，全局事务注解
    public void createOrder(OrderRequest request) {
        // 1. 扣减库存（调用库存服务）
        stockService.deductStock(request.getProductId(), request.getQuantity());
        // 2. 创建订单
        orderRepository.save(new Order(request));
        // 3. 扣减账户余额（调用账户服务）
        accountService.deductBalance(request.getUserId(), request.getAmount());
        // 如果任何一步失败，Seata 会通过 undo_log 自动回滚所有步骤
    }
}
```

**TCC 三大坑**

| 问题 | 描述 | 解决方案 |
|---|---|---|
| **空回滚** | Try 未执行（网络超时），Cancel 被调用 | Cancel 检查业务数据，若 Try 记录不存在，直接返回成功 |
| **幂等** | Confirm/Cancel 被重复调用 | 通过唯一索引（xid + branch_id）保证幂等 |
| **悬挂** | Cancel 先于 Try 执行（网络乱序） | Try 时检查如果 Cancel 已执行，直接返回失败，不再执行 Try |
