## 可观测性三支柱

现代分布式系统的可观测性（Observability）建立在三个支柱上，缺一不可：

| 支柱 | 作用 | 主流工具 |
|---|---|---|
| **Metrics（指标）** | 量化系统状态，回答"是否有问题" | Prometheus + Grafana |
| **Logs（日志）** | 记录事件详情，回答"发生了什么" | ELK Stack / Loki |
| **Traces（链路追踪）** | 还原请求路径，回答"哪里慢/出错" | SkyWalking / Jaeger / Zipkin |

**三者联动**：通过在日志中注入 TraceId，可以从监控告警一键跳转到对应请求的链路追踪，再从链路追踪跳转到具体服务的错误日志，实现"一条线贯穿问题排查全流程"。

## 链路追踪

### 数据模型

```
一次分布式请求（TraceId = T1）：
  
  Span S1 (ROOT)         Span S2                Span S3
  ├─ 网关处理             ├─ 订单服务处理          ├─ 库存服务处理
  ├─ traceId: T1         ├─ traceId: T1          ├─ traceId: T1
  ├─ spanId: S1          ├─ spanId: S2           ├─ spanId: S3
  ├─ parentId: null      ├─ parentId: S1         ├─ parentId: S2
  ├─ startTime: 0ms      ├─ startTime: 5ms       ├─ startTime: 8ms
  └─ duration: 50ms      └─ duration: 40ms       └─ duration: 20ms
  
  调用树（按 ParentId 还原）：
  S1 → S2 → S3
  瀑布图（按时间轴展示各 Span 耗时分布）
```

### 采样策略

全量采集成本极高，生产环境必须采样：

```
策略1：固定比例（头部采样）
  - 在请求入口决定是否采样（如 1%），并通过 Header 传递给下游
  - 一条链路要么全部采集，要么全部不采集
  - 配置：spring.sleuth.sampler.probability=0.01

策略2：错误强制采样
  - 正常请求 1% 采样，错误请求 100% 采样
  - 确保所有异常链路都能被追踪到

策略3：尾部采样（Tail Sampling）
  - 先暂存所有 Span，完成后根据结果（慢/错误）决定是否保留
  - 最准确，但内存消耗高
  - OTel Collector 支持此策略
```

### OpenTelemetry 标准化

```
应用（OTel SDK 埋点）
    ↓ OTLP 协议
OTel Collector（接收、过滤、转换、导出）
    ↓ 导出到
 ┌──────────────────────┐
 │  Jaeger（追踪）       │
 │  Prometheus（指标）   │
 │  Loki（日志）         │
 └──────────────────────┘
```

OTel 实现"一次埋点，多后端导出"，未来更换后端无需改业务代码。

## 指标监控

### Prometheus 数据模型

Prometheus 使用**多维时间序列**，每个时间序列由指标名称 + 标签（Labels）唯一标识：

```
# HELP http_requests_total The total number of HTTP requests.
# TYPE http_requests_total counter
http_requests_total{method="POST", handler="/api/orders", status="200"} 1027
http_requests_total{method="GET",  handler="/api/orders", status="200"} 5832
http_requests_total{method="POST", handler="/api/orders", status="500"} 23
```

**四种指标类型**：
- **Counter（计数器）**：单调递增，如请求总数、错误总数；`rate()` 函数计算速率
- **Gauge（仪表盘）**：可增可减的瞬时值，如当前连接数、内存使用量
- **Histogram（直方图）**：观测值分布，如请求延迟；支持计算分位数（`histogram_quantile`）
- **Summary（摘要）**：类似 Histogram，但分位数在客户端计算

### 常用 PromQL

```promql
# 5分钟内 HTTP 5xx 错误率
rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])

# P99 响应时间（需要 Histogram 类型的指标）
histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service))

# JVM 堆内存使用率
jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} * 100

# 数据库连接池使用率
hikaricp_connections_active / hikaricp_connections_max
```

### Spring Boot 集成 Prometheus

```xml
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

```yaml
management:
  endpoints:
    web:
      exposure:
        include: prometheus, health
  metrics:
    tags:
      application: ${spring.application.name}   # 所有指标自动加 application 标签
```

## 日志体系

### ELK Stack 架构

```
应用日志文件
    ↓ Filebeat（轻量采集）
Logstash（解析/过滤/丰富）
    ↓ 
Elasticsearch（存储/索引/搜索）
    ↓
Kibana（可视化/告警）
```

**结构化日志（JSON 格式）** 是现代最佳实践，减少 Logstash 解析工作：

```java
// Logback JSON 配置
<appender name="JSON" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <!-- 自动包含 traceId（从 MDC） -->
    </encoder>
</appender>
```

```json
{
  "timestamp": "2026-03-15T10:30:00.000Z",
  "level": "ERROR",
  "logger": "com.example.OrderService",
  "message": "订单创建失败",
  "traceId": "abc123def456",    // 通过 MDC 自动注入
  "spanId": "def789",
  "userId": "user-001",
  "orderId": "order-001"
}
```

### Loki（轻量日志方案）

Loki 是 Grafana Labs 的日志系统，设计理念是"像 Prometheus 一样处理日志"：

```
只对标签（Labels）建索引 ← 低成本
日志内容压缩存储（块文件）
使用 LogQL 查询（语法类似 PromQL）

# LogQL 示例：查询 order-service 的 ERROR 日志
{app="order-service", env="prod"} |= "ERROR" | json | traceId != ""
```

**Grafana + Loki + Prometheus 统一观测平台**：
- 在 Grafana 中同时配置 Prometheus 和 Loki 数据源
- 通过 TraceId/SpanId 关联指标、日志、链路追踪
- 实现从告警 → 指标图表 → 错误日志 → 调用链的全链路排查

## 告警设计

### 告警规则最佳实践

```yaml
# Prometheus 告警规则（alerting_rules.yml）
groups:
- name: service.rules
  rules:
  # 接口错误率告警
  - alert: HighErrorRate
    expr: |
      (
        rate(http_requests_total{status=~"5.."}[5m]) /
        rate(http_requests_total[5m])
      ) > 0.05
    for: 2m           # 持续 2 分钟才触发（过滤毛刺）
    labels:
      severity: critical
    annotations:
      summary: "{{ $labels.service }} 错误率过高"
      description: "当前错误率 {{ $value | humanizePercentage }}，超过 5% 阈值"

  # P99 响应时间告警
  - alert: SlowResponseTime
    expr: |
      histogram_quantile(0.99,
        sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service)
      ) > 2
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "{{ $labels.service }} P99 响应时间超过 2s"
```

### Alertmanager 去重分组配置

```yaml
route:
  group_by: ['alertname', 'cluster', 'service']   # 按这些标签分组
  group_wait: 30s        # 同一组内等待 30s 收集更多告警再发送
  group_interval: 5m     # 同一组告警发送间隔
  repeat_interval: 4h    # 未解决的告警每 4h 重复通知一次
  
  routes:
  - match:
      severity: critical
    receiver: oncall-phone    # Critical 走电话/短信
  - match:
      severity: warning
    receiver: dingtalk-group  # Warning 走钉钉群

inhibit_rules:
- source_match:
    alertname: NodeDown      # 节点宕机时
  target_match:
    severity: warning        # 抑制该节点上所有 warning 级别告警
  equal: ['instance']
```
