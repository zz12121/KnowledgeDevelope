# 高可用架构

## 1. 主从复制原理

主从复制是 Redis 高可用的基础，实现**数据冗余、故障恢复和读写分离**。复制过程分三个阶段。

### 第一阶段：连接建立

1. 从节点通过配置 `replicaof <masterip> <masterport>` 或命令设置主节点
2. 从节点创建专用文件事件处理器，向主节点发起 Socket 连接
3. 连接成功后发送 `PING` 命令检查连接可用性
4. 若主节点设置了密码（`requirepass`），从节点发送 `AUTH` 命令验证

### 第二阶段：数据同步（PSYNC）

Redis 2.8+ 使用 `PSYNC` 命令，支持**全量复制**和**部分复制**。

**全量复制**（初次连接或偏移量失效时）：
1. 从节点发送 `PSYNC ? -1`
2. 主节点执行 `BGSAVE`，生成 RDB 快照
3. 发送 RDB 文件给从节点，从节点清空旧数据后加载
4. 期间新写命令缓存在**复制客户端缓冲区**
5. RDB 加载完成后，主节点将缓冲区积累的命令发给从节点

**部分复制**（断线重连时）——依赖三个核心概念：

| 概念 | 说明 |
|------|------|
| **复制偏移量（offset）** | 主从各自维护，标识已同步的数据量，对比偏移量判断是否一致 |
| **复制积压缓冲区（backlog）** | 主节点维护的固定大小环形队列（默认 1MB），存储最近传播的写命令 |
| **服务器运行 ID（Run ID）** | 每个 Redis 节点的唯一标识，重连时携带，用于判断是否需要全量复制 |

重连时从节点发送 `PSYNC <runid> <offset>`，若偏移量之后的数据仍在积压缓冲区中，则执行部分复制（仅补发缺失数据）；否则触发全量复制。

### 第三阶段：命令传播

同步完成后，主节点每执行一个写命令，**异步**广播给所有从节点。这是**最终一致性**模型。

**WAIT 命令**：`WAIT <numreplicas> <timeout>` 阻塞客户端直到指定数量从节点确认复制，提供弱一致性保证（会牺牲可用性）。

## 2. 哨兵模式（Sentinel）

哨兵模式是 Redis 的**高可用解决方案**，通过监控主从节点并在主节点故障时自动执行故障转移。

### 工作流程

1. **监控**：每个 Sentinel 以每秒一次的频率向所有被监控节点发送 PING 命令

2. **主观下线（SDOWN）**：若节点在 `down-after-milliseconds` 毫秒内未有效回复，该 Sentinel 将其标记为主观下线

3. **客观下线（ODOWN）**：将主节点标记为主观下线的 Sentinel 询问其他 Sentinel，若达到 **Quorum（法定人数）** 的 Sentinel 都报告主观下线，则标记为**客观下线**，触发故障转移

4. **Sentinel 选举**：通过 **Raft 算法**选出一个 Sentinel 领导者负责执行故障转移

5. **故障转移**：
   - 从从节点中选出最优者（考虑优先级、复制偏移量、Run ID）
   - 发送 `SLAVEOF NO ONE` 将其提升为新主节点
   - 向其他从节点发送 `SLAVEOF` 命令跟随新主节点
   - 更新配置，通知客户端新主节点地址

### 脑裂问题与解决

**脑裂（Split-Brain）**：网络分区时，集群中可能选举出多个主节点，客户端向旧主节点写入的数据会在网络恢复后丢失。

**解决方案**：
```properties
# 主节点至少有1个从节点连接且延迟在10秒内，否则拒绝写请求
min-replicas-to-write 1
min-replicas-max-lag 10
```

网络分区时，被隔离的旧主节点会因为没有从节点连接而拒绝写请求，减少数据不一致窗口。

## 3. 集群模式（Cluster）

Redis Cluster 是**去中心化的分片集群**方案，通过数据分片解决海量数据和高并发问题。

### 数据分片（哈希槽）

- 集群将数据划分为 **16384 个哈希槽（Hash Slot）**
- 分片算法：`HASH_SLOT = CRC16(key) % 16384`
- 每个主节点负责一部分槽（如 3 主节点各负责约 5461 个槽）
- 数据与节点解耦：扩缩容只需迁移部分槽，无需重新计算所有哈希值

**哈希标签（Hash Tag）**：用 `{}` 标记计算槽的部分，如 `user:{1000}:name` 和 `user:{1000}:email` 的槽由 `1000` 决定，确保落在同一个节点。

### 请求路由

1. 客户端向节点发送命令，节点计算 key 所属槽
2. 若该槽在当前节点：直接执行
3. 若不在当前节点：返回 **MOVED 重定向**，告知正确节点地址
4. 智能客户端（如 JedisCluster）缓存槽-节点映射，后续直连正确节点

数据迁移期间，访问正在迁移的键可能收到 **ASK 重定向**（临时性，不同于 MOVED 的永久性）。

### 节点通信（Gossip 协议）

所有节点通过 Gossip 协议互联，每个节点都维护完整的集群元数据（各节点状态、负责的槽范围等），实现去中心化。

### 故障转移

1. 节点间通过 PING 检测故障，未响应节点被标记为**疑似下线**
2. 多数主节点确认后标记为**已下线**
3. 故障节点的从节点通过投票（基于配置纪元，需 N/2+1 票）选出新主节点
4. 新主节点接管哈希槽，通过 Gossip 协议广播集群更新

### 集群限制

- 不支持跨节点的多键操作（MSET、事务等），除非使用哈希标签确保 key 落在同一节点
- 不支持多数据库（只能使用 DB 0）
- 扩容/缩容需要手动执行槽迁移

```bash
# 扩容：添加新节点
redis-cli --cluster add-node <new_ip:port> <existing_ip:port>
# 槽重分配
redis-cli --cluster reshard <existing_ip:port>
# 缩容：先迁移槽，再移除节点
redis-cli --cluster del-node <existing_ip:port> <node_id>
```

## 4. 读写分离

**实现方式**：
1. **客户端直连**：配置读写两个连接池，写操作发往主节点，读操作发往从节点
2. **代理中间件**：Twemproxy、Codis 等代理根据操作类型路由，对客户端透明
3. **框架支持**：Spring Data Redis 配置 Lettuce 主从模式，自动路由读写

**注意**：主从复制是异步的，从节点数据可能稍旧，读写分离只适用于对一致性要求不高的场景。
