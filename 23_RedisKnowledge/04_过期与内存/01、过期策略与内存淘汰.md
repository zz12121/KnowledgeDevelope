# 过期策略与内存淘汰

## 1. 过期键删除策略

Redis 采用**惰性删除 + 定期删除**相结合的策略，在 CPU 合理使用和避免内存浪费之间取得平衡。

> 注意：Redis **不使用**定时删除（设置键过期时同时创建定时器），因为大量定时器会严重消耗 CPU。

### 惰性删除（Lazy Expiration）

**触发时机**：客户端访问某个键时，才检查并删除。

**源码实现**（`db.c` 中的 `expireIfNeeded` 函数）：
```c
int expireIfNeeded(redisDb *db, robj *key) {
    if (!keyIsExpired(db, key)) return 0; // 未过期，直接返回
    
    server.stat_expiredkeys++;
    // 根据 lazyfree-lazy-expire 配置决定同步还是异步删除
    return server.lazyfree_lazy_expire 
        ? dbAsyncDelete(db, key) 
        : dbSyncDelete(db, key);
}
```

所有读写命令（GET、HGET、SET 等）执行前都会调用此函数。

- **优点**：CPU 友好，只在必要时触发
- **缺点**：内存不友好，过期但从不被访问的键会一直占用内存（内存泄漏）

### 定期删除（Active Expiration）

**触发时机**：Redis 的周期性任务函数 `serverCron` 调用 `activeExpireCycle`，默认每秒执行 10 次（由 `hz` 参数控制）。

**执行流程**：
1. 遍历最多 16 个数据库（`REDIS_DBCRON_DBS_PER_CALL`）
2. 从每个数据库的过期字典（`expires`）中**随机抽取 20 个键**
3. 删除其中已过期的键
4. 若本轮过期键比例 > **25%**，重复步骤 2-3，直到比例降低或达到时间上限
   - **正常模式**：每次执行不超过约 25ms
   - **快速模式**：执行时间更短，用于紧急清理

**自适应设计**：过期键密度越高，定期删除越积极清理，反之越保守，避免过度消耗 CPU。

## 2. 内存淘汰策略

当 Redis 使用内存达到 `maxmemory` 上限时，触发内存淘汰（Eviction）。

**配置方式**：
```bash
maxmemory 4gb
maxmemory-policy allkeys-lru
# 或运行时动态配置
CONFIG SET maxmemory-policy volatile-lfu
```

### 八种淘汰策略

| 策略 | 应用范围 | 机制 | 适用场景 |
|------|---------|------|---------|
| `noeviction` | 所有键 | **默认**。写操作报 OOM 错误，读/删正常 | 保证数据不丢失，需确保内存充足 |
| `allkeys-lru` | 所有键 | 淘汰**最近最久未使用**的键 | 热点分布均匀，无法预测访问模式 |
| `volatile-lru` | 带过期时间的键 | 淘汰带过期时间的键中**最近最久未使用**的 | 需要淘汰数据，但保留永久键 |
| `allkeys-random` | 所有键 | **随机**淘汰任意键 | 访问概率相等，无热点 |
| `volatile-random` | 带过期时间的键 | 随机淘汰带过期时间的键 | 同 volatile-lru，但不需要 LRU 精度 |
| `volatile-ttl` | 带过期时间的键 | 淘汰**剩余生存时间最短**的键 | 通过过期时间层级管理数据优先级 |
| `allkeys-lfu` | 所有键 | 淘汰**访问频率最低**的键 | 存在明确热点数据，希望长期保留高频键 |
| `volatile-lfu` | 带过期时间的键 | 带过期时间的键中淘汰**访问频率最低**的 | 需要淘汰数据且存在热点，希望保留永久键 |

## 3. LRU 与 LFU 算法对比

| 特征 | LRU（最近最少使用） | LFU（最不经常使用） |
|------|-------------------|-------------------|
| **核心思想** | 淘汰**最久没被访问**的键 | 淘汰**访问频率最低**的键 |
| **关注点** | 访问的**时间距离**（何时访问） | 访问的**频率次数**（访问多少次） |
| **Redis 实现** | 每个键记录最近一次访问的时间戳（近似算法，采样 5 个键取最旧的） | 每个键记录访问计数器，会随时间**衰减** |
| **适用场景** | 近期访问行为更能预测未来（如新闻首页） | 存在长期稳定热点数据（如商品热门榜单） |

**Redis 近似 LRU/LFU**：Redis 不维护全局链表，而是在淘汰时随机采样若干键，选出最符合条件的键删除。通过 `maxmemory-samples`（默认 5）控制采样数量，值越大越精准但消耗越多 CPU。

**LFU 衰减机制**：LFU 计数器会随时间衰减（`lfu-decay-time` 配置，默认 1 分钟），避免历史热点数据长期占据高频位置。

## 4. 过期时间设置命令

| 命令 | 语法 | 单位 | 说明 |
|------|------|------|------|
| `EXPIRE` | `EXPIRE key seconds` | 秒 | 设置相对过期时间 |
| `PEXPIRE` | `PEXPIRE key milliseconds` | 毫秒 | 高精度版本 |
| `EXPIREAT` | `EXPIREAT key unix-timestamp` | 秒级时间戳 | 设置绝对过期时间点 |
| `PEXPIREAT` | `PEXPIREAT key unix-ms-timestamp` | 毫秒级时间戳 | 高精度版本 |
| `SETEX` | `SETEX key seconds value` | 秒 | 设置值同时设置过期时间（原子操作） |

**查看和管理过期时间**：
- `TTL key`：剩余时间（秒），`-2` 表示键不存在，`-1` 表示无过期时间
- `PTTL key`：剩余时间（毫秒）
- `PERSIST key`：移除过期时间，变为永久键

## 5. 内存监控与诊断

```bash
# 全面的内存信息
INFO memory

# 关键指标含义
used_memory          # Redis 实际使用内存（字节）
used_memory_rss      # 进程从 OS 角度占用内存（含碎片）
used_memory_peak     # 历史内存峰值
mem_fragmentation_ratio  # 内存碎片率 = used_memory_rss / used_memory
                         # > 1 表示有碎片；< 1 表示使用了 Swap

# 碎片率 > 1.5 时需要处理
# 启用自动碎片整理（Redis 4.0+）
CONFIG SET activedefrag yes
```
