# 安全与运维

## 1. 密码认证

### 1.1 设置密码

**方式一：配置文件（持久化生效，需重启）**

```bash
# /etc/redis/redis.conf
requirepass YourStrongPassword123!
```

**方式二：命令行临时设置（立即生效，重启后失效）**

```bash
redis-cli
127.0.0.1:6379> CONFIG SET requirepass "YourStrongPassword123!"
```

**验证方式**：
```bash
redis-cli -a YourPassword          # 连接时直接带密码
redis-cli → AUTH YourPassword      # 连接后再认证
```

密码强度建议至少 16 位，包含大小写字母、数字和特殊字符。在主从架构中，还需要在从节点配置 `masterauth`，否则从节点无法连上主节点。

### 1.2 禁用危险命令

生产环境必须做的一件事就是禁用或重命名高危命令：

```bash
# 禁用（设置为空字符串）
rename-command FLUSHALL ""
rename-command FLUSHDB ""
rename-command DEBUG ""

# 重命名为不可猜测的名称
rename-command KEYS "REDIS_KEYS_DO_NOT_USE"
rename-command CONFIG "REDIS_CONFIG_INTERNAL"
rename-command EVAL ""  # 若业务不用 Lua 可以直接禁掉
```

需要注意，`rename-command` 改了之后，应用里所有用到该命令的地方也要对应修改。另外在 Sentinel/Cluster 模式下，某些内部命令（如 `CONFIG`）被哨兵用来监控，随意禁用可能导致哨兵工作异常，需要谨慎。

### 1.3 网络访问控制

```bash
# 绑定到指定网卡，不对外暴露
bind 127.0.0.1 192.168.1.100

# 开启保护模式（没设密码且 bind 包含 0.0.0.0 时，保护模式阻止非本地访问）
protected-mode yes

# 配置防火墙，只允许应用服务器 IP 访问 6379
iptables -A INPUT -p tcp --dport 6379 -s 192.168.1.0/24 -j ACCEPT
iptables -A INPUT -p tcp --dport 6379 -j DROP
```

**Redis 不应该直接暴露在公网**，这是最基础的安全要求。

## 2. 备份与恢复

### 2.1 备份策略

Redis 的备份依赖持久化文件（RDB 或 AOF）。

**RDB 快照备份**：配置自动触发条件，并定期拷贝 `dump.rdb` 到备份存储。

```bash
# redis.conf 自动 RDB 配置
save 900 1       # 900 秒内有至少 1 个 key 变更
save 300 10
save 60 10000
dbfilename dump.rdb
dir /var/lib/redis
```

**AOF 备份**：AOF 文件记录了所有写命令，更安全但文件较大。混合持久化（`aof-use-rdb-preamble yes`）是目前推荐的方案，恢复速度快且数据损失少。

**企业级定时备份脚本**：

```bash
#!/bin/bash
# 每小时备份 RDB，保留 48 小时
CUR_DATE=$(date +%Y%m%d%H)
BACKUP_DIR="/backup/redis/$CUR_DATE"
mkdir -p $BACKUP_DIR

# 触发 BGSAVE 后等待完成，再拷贝
redis-cli BGSAVE
sleep 5
cp /var/lib/redis/dump.rdb $BACKUP_DIR/dump.rdb

# 清理 48 小时前的备份
find /backup/redis -type d -mtime +2 -exec rm -rf {} \;
```

### 2.2 恢复流程

1. **停止 Redis 服务**：`systemctl stop redis-server`
2. **替换数据文件**：将备份的 `dump.rdb`（或 AOF 文件）拷贝到 `dir` 配置的目录
3. **启动服务**：`systemctl start redis-server`
4. **验证数据**：`redis-cli DBSIZE`，抽查关键 key 是否存在

如果 RDB 和 AOF 同时存在，Redis 优先加载 AOF 文件（因为 AOF 数据通常比 RDB 更新），记住这个细节。

## 3. 监控

### 3.1 内置监控命令

**INFO 命令** 是最常用的监控入口，分模块查看运行状态：

```bash
redis-cli info all           # 所有信息
redis-cli info memory        # 内存使用
redis-cli info stats         # 命令统计
redis-cli info replication   # 主从同步状态
redis-cli info clients       # 连接信息
redis-cli info keyspace      # 各 DB 的 key 数量
```

**关键监控指标**：
- `used_memory_human`：实际数据占用内存
- `mem_fragmentation_ratio`：内存碎片率，正常应在 1.0-1.5 之间
- `connected_clients`：当前连接数，超过阈值说明连接池有问题
- `keyspace_hits / (keyspace_hits + keyspace_misses)`：缓存命中率
- `master_replication_offset - slave_replication_offset`：主从复制延迟

**MONITOR 命令**：实时输出所有执行的命令，排查问题很好用，但会显著降低 Redis 性能（官方数据约降低 50%），**只在调试时短暂使用，绝对不能用在生产环境长期运行**。

```bash
redis-cli monitor  # Ctrl+C 停止
```

### 3.2 外部监控方案

生产环境推荐 **Prometheus + Redis-Exporter + Grafana** 的组合：

```bash
# 启动 redis_exporter
./redis_exporter --redis.addr redis://localhost:6379

# Prometheus 配置抓取（prometheus.yml）
scrape_configs:
  - job_name: redis
    static_configs:
      - targets: ['localhost:9121']
```

Grafana 导入 Redis 官方 Dashboard（ID: 763），可以看到内存、命中率、命令 QPS、连接数等所有关键指标的时序曲线，设置告警规则后异常自动通知。

### 3.3 实时性能诊断

```bash
redis-cli --latency          # 实时延迟（毫秒）
redis-cli --latency-history  # 每 15 秒输出一次延迟历史
redis-cli --stat             # 实时统计（内存/key数/命中率/秒操作数）
redis-cli --bigkeys          # 扫描 Big Key（在低峰期执行）
redis-cli --hotkeys          # 热 Key 分析（需 maxmemory-policy 非 none）
```

## 4. 内存碎片整理

### 4.1 碎片产生原因

Redis 使用 jemalloc 做内存分配。频繁的 key 增删改（尤其是大量 key 过期）会导致分配的内存块大小参差不齐，形成**外部碎片**（分配了内存但没被数据填满的空隙）。

`mem_fragmentation_ratio = used_memory_rss / used_memory`：

- **1.0-1.5**：正常范围，轻微碎片
- **> 1.5**：碎片严重，需要整理
- **< 1.0**：发生了内存交换（Swap），Redis 性能会严重下降

### 4.2 处理方案

**方案一：Redis 4.0+ 自动碎片整理（推荐）**

```bash
CONFIG SET activedefrag yes                    # 开启自动整理
CONFIG SET active-defrag-ignore-bytes 100mb    # 碎片超过 100MB 才触发
CONFIG SET active-defrag-threshold-lower 10    # 碎片率超过 10% 才触发
CONFIG SET active-defrag-threshold-upper 100   # 碎片率超 100% 时最大力度整理
CONFIG SET active-defrag-cycle-min 1           # 整理占用 CPU 的最小比例
CONFIG SET active-defrag-cycle-max 25          # 整理占用 CPU 的最大比例
```

自动碎片整理会在后台渐进式地将数据从碎片内存区域迁移到连续内存，对性能影响极小。

**方案二：手动清理**

```bash
redis-cli memory purge  # 触发 jemalloc 内存归还
```

**方案三：重启（核心数据不能丢时慎用）**

重启后 Redis 重新申请内存，碎片自然消失，但这是最后手段，生产上一般不这么干。

## 5. 版本升级

### 5.1 升级前准备

在升级前需要做三件事：
1. **兼容性检查**：查阅 Release Notes，确认是否有命令废弃或行为变更，检查客户端库版本是否支持新协议（如 RESP3）
2. **模块兼容性**：RedisJSON、RediSearch 等模块需要确认支持新版本
3. **数据备份**：执行 `BGSAVE` 并确认 RDB 文件生成成功

### 5.2 单机升级步骤

```bash
# 下载并编译新版本
wget https://download.redis.io/releases/redis-7.2.tar.gz
tar xzf redis-7.2.tar.gz && cd redis-7.2
make && make install

# 平滑重启（会加载 dump.rdb 恢复数据）
redis-cli SHUTDOWN SAVE  # 触发最后一次 RDB 再关闭
redis-server /etc/redis/redis.conf
```

### 5.3 集群滚动升级

1. 先逐个升级所有**从节点**（从节点升级不影响读写）
2. 对每个主节点执行手动故障转移（`CLUSTER FAILOVER`），将已升级的从节点提升为主
3. 原主节点降为从节点后升级，完成一个分片
4. 重复直到所有节点升级完毕

**回滚方案**：保留旧版本二进制文件，回滚时用旧版本 `redis-server` + 之前备份的 RDB 重启即可。

## 6. 故障排查

### 6.1 常见故障分类

**内存不足**：`INFO memory` 查看 `used_memory`，对比 `maxmemory` 配置。处理方法：优化淘汰策略、清理不必要的 key、升级内存或扩容。

**连接数超标**：`INFO clients` 看 `connected_clients`。通常原因是连接池配置过大、连接泄漏（用完没 close）。调整 `maxclients` 只是治标，排查连接泄漏才是治本。

**持久化失败**：`INFO persistence` 查看 `rdb_last_bgsave_status` 或 `aof_last_write_status`。常见原因是磁盘空间不足或 `/var/lib/redis` 目录权限问题。

**主从同步异常**：`INFO replication` 查看 `master_link_status`。检查网络连通性、密码配置（`masterauth`）、防火墙规则。

### 6.2 系统性排查流程

```bash
# 1. 查看服务状态
systemctl status redis-server

# 2. 实时查看日志
tail -f /var/log/redis/redis-server.log

# 3. 网络延迟诊断
redis-cli --latency

# 4. 实时统计
redis-cli --stat

# 5. 慢查询分析
redis-cli slowlog get 20
redis-cli slowlog len

# 6. 查看当前所有客户端
redis-cli client list

# 7. 内存详情
redis-cli info memory
redis-cli memory doctor  # 给出内存健康建议（Redis 4.0+）
```

`memory doctor` 这个命令非常实用，它会分析内存使用状况，给出人类可读的诊断建议，比如 "Your aof_buf is too big" 之类的，不需要手动去 INFO 里找指标。

## 7. 重要配置参数速查

**安全相关**：
```bash
requirepass "your_password"     # 访问密码
rename-command FLUSHALL ""      # 禁用危险命令
bind 127.0.0.1                  # 限制监听 IP
protected-mode yes              # 保护模式
```

**性能与内存**：
```bash
maxmemory 4gb                   # 最大内存限制
maxmemory-policy allkeys-lru    # 内存淘汰策略
maxclients 10000                # 最大客户端连接数
tcp-keepalive 300               # TCP 保活间隔（秒）
```

**持久化**：
```bash
save 900 1                      # RDB 自动触发
appendonly yes                  # 开启 AOF
appendfsync everysec            # AOF 同步策略
aof-use-rdb-preamble yes        # 混合持久化
```

**内存碎片整理**：
```bash
activedefrag yes                          # 开启自动碎片整理
active-defrag-ignore-bytes 100mb          # 触发阈值
active-defrag-threshold-lower 10          # 碎片率下限
```
