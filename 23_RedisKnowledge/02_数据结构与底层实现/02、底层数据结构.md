# 底层数据结构

## 1. SDS（Simple Dynamic String）

SDS 是 Redis 设计的**二进制安全动态字符串**，解决了 C 字符串的多个固有缺陷。

**核心结构**：多种 header 类型（sdshdr8/16/32/64）+ 柔性字节数组，根据字符串长度自动选择最小的 header 类型。

```c
struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len;        // 已使用长度
    uint8_t alloc;      // 总分配长度（不含header和\0）
    unsigned char flags;// 低3位标识header类型
    char buf[];         // 实际数据（柔性数组）
};
```

**相比 C 字符串的优势**：

| 特性 | C 字符串 | SDS |
|------|---------|-----|
| 获取长度 | O(N)，遍历至 `\0` | O(1)，读 `len` 字段 |
| 缓冲区溢出 | 可能，需手动检查 | 自动检查并扩容 |
| 内存重分配 | 每次修改都需要 | 预分配 + 惰性释放策略 |
| 二进制安全 | 否（`\0` 截断） | 是，可存任意二进制数据 |
| 兼容 C 函数 | 是 | 是（自动追加 `\0`） |

**空间预分配策略**：
- SDS 长度 < 1MB：预分配双倍空间（`newlen * 2`）
- SDS 长度 ≥ 1MB：每次多分配 1MB

**惰性空间释放**：字符串缩短时保留多余空间，`alloc - len` 部分留作将来增长使用。

## 2. 跳跃表（Skip List）

跳跃表是**有序数据结构**，通过多级索引实现 O(logN) 的查找效率，是 ZSet 的核心底层结构。

**结构示意**：
```
Level 4: [Head] -----------------> [Node 50] -> [Tail]
Level 3: [Head] -----> [Node 20] -> [Node 50] -> [Tail]
Level 2: [Head] -> [Node 10] -> [Node 20] -> [Node 50] -> [Tail]
Level 1: [Head] -> [Node 10] -> [Node 20] -> [Node 30] -> [Node 50] -> [Tail]
```

**节点结构**：
```c
typedef struct zskiplistNode {
    sds ele;                          // 成员对象
    double score;                     // 分值
    struct zskiplistNode *backward;   // 后退指针（用于逆序遍历）
    struct zskiplistLevel {
        struct zskiplistNode *forward;// 前进指针
        unsigned long span;           // 跨度（用于计算排名）
    } level[];                        // 层级数组
} zskiplistNode;
```

**核心特性**：
- **随机层级**：每个节点的层数通过 `zslRandomLevel()` 随机生成（每层概率 25%，最大 32 层）
- **时间复杂度**：查找/插入/删除平均 O(logN)，最坏 O(N)
- **对比平衡树**：实现更简单，范围操作（ZRANGE）效率相当，且无旋转操作

**为什么用跳跃表而不是红黑树**：实现简单、支持高效的顺序遍历和范围查询。

## 3. 压缩列表（ziplist）

ziplist 是**内存紧凑型顺序结构**，为小规模数据提供高内存密度存储，被 Hash、List、ZSet 作为小数据时的编码。

**整体结构**：
```
zlbytes | zltail | zllen | entry1 | entry2 | ... | zlend
  4字节    4字节   2字节    变长      变长              1字节(0xFF)
```

**entry 结构**：
```
prevlen | encoding | entry-data
```
- `prevlen`：前驱节点长度，用于反向遍历（< 254 字节用 1 字节，否则 5 字节）
- `encoding`：数据编码类型
- `entry-data`：实际数据

**连锁更新问题（Cascade Update）**：当插入或扩大某个节点时，后续节点的 `prevlen` 可能需要从 1 字节扩展到 5 字节，进而触发后续节点也需要更新，最坏 O(N²)。这是 ziplist 只适用于小规模数据的根本原因。

## 4. listpack

listpack 是 Redis 7.0 用于替代 ziplist 的新结构，核心改进是**每个 entry 只记录自身长度**，彻底解决连锁更新问题。

**entry 结构**：
```
encoding | entry-data | backlen
```
- `backlen`：当前 entry 的总长度，用于反向遍历（只记录自己，不记录前驱）

由于不存储前驱节点长度，修改任意节点不会影响其他节点，杜绝了连锁更新。

---

## 5. quicklist

quicklist 是**listpack（或 ziplist）的双向链表**，作为 List 类型的底层实现。

**节点结构**：
```c
typedef struct quicklistNode {
    struct quicklistNode *prev;
    struct quicklistNode *next;
    unsigned char *entry;     // 指向 listpack/ziplist
    size_t sz;                // entry 字节数
    unsigned int count;       // entry 中的元素数量
    // ...
} quicklistNode;
```

**控制参数**：
- `list-max-listpack-size`（旧名 `list-max-ziplist-size`）：控制每个节点的最大容量
  - 正数：最大元素数量（如 128）
  - 负数：-1（4KB）、-2（8KB，默认）、-3（16KB）、-4（32KB）、-5（64KB）
- `list-compress-depth`：两端不压缩的节点数（0 = 不压缩，默认；1 = 两端各 1 个不压缩，其余用 LZF 压缩）

## 6. 字典（dict）

Redis 字典采用哈希表实现，支持渐进式 rehash。

**核心结构**：
```c
typedef struct dict {
    dictType *type;
    dictht ht[2];     // 两个哈希表，用于渐进式 rehash
    long rehashidx;   // rehash 进度，-1 表示未在 rehash
    // ...
} dict;
```

**渐进式 rehash 过程**：
1. 当负载因子（used/size）超过阈值时，为 `ht[1]` 分配更大空间，开始 rehash
2. 每次对字典进行增删查改时，顺带将 `ht[0]` 的 `rehashidx` 处的桶迁移到 `ht[1]`
3. rehash 期间，查找操作同时检查两个哈希表；新插入的数据直接进入 `ht[1]`
4. 所有桶迁移完毕后，释放 `ht[0]`，将 `ht[1]` 赋给 `ht[0]`，`rehashidx` 重置为 -1

**优点**：避免单次 rehash 的大量计算阻塞主线程，保证在高负载下也能稳定响应。

## 7. intset（整数集合）

intset 是**纯整数集合的高效存储结构**，用于 Set 类型所有元素均为整数时。

**结构**：
```c
typedef struct intset {
    uint32_t encoding;  // 编码方式：INT16/INT32/INT64
    uint32_t length;    // 元素数量
    int8_t contents[];  // 实际整数数组（有序存储）
} intset;
```

**编码升级**：添加超出当前编码范围的整数时，自动升级（16→32→64 位），升级不可逆（即使删除触发升级的元素）。

**查找**：二分查找，O(logN)，元素有序存储。
