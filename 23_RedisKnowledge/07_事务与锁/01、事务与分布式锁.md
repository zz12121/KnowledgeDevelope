# 事务与分布式锁

## 1. Redis 事务原理

Redis 事务通过**命令队列 + 原子执行**机制实现，包含三个阶段：

### 三个阶段

**① MULTI（开始事务）**：设置客户端状态为 `CLIENT_MULTI`，后续命令不立即执行。

**② 命令入队**：非 MULTI/EXEC/DISCARD/WATCH 的命令存入事务队列（`client->mstate.commands`，FIFO）。

**③ EXEC（执行事务）**：遍历队列依次执行所有命令，一次性返回所有结果，执行期间不会被其他客户端命令打断。

**DISCARD**：清空命令队列，放弃事务。

### 错误处理

- **EXEC 前发现错误**（语法错误、命令不存在）：整个事务被拒绝，所有命令都不执行
- **EXEC 后发现错误**（如对字符串执行哈希操作）：**错误命令报错，但其他命令继续执行**，已执行的命令不回滚

### 为什么不支持回滚

Redis 认为事务错误通常是**编程错误**，应在开发阶段修复，而不是依赖运行时回滚。不支持回滚简化了内核实现，提升了性能。这与关系型数据库的事务概念有本质区别。

## 2. WATCH 乐观锁

`WATCH` 实现**乐观锁**，在 MULTI 前监控指定键，若事务执行前键被修改则放弃事务。

### 源码实现

1. `WATCH key`：将客户端添加到 `db->watched_keys` 字典中该 key 的监控列表
2. 任何修改键的命令成功后，调用 `signalModifiedKey()` → `touchWatchedKey()`，将监控该键的所有客户端标记为 `CLIENT_DIRTY_CAS`
3. `EXEC` 时检查客户端是否有 `CLIENT_DIRTY_CAS` 标志，若有则拒绝执行事务，返回 `nil`

### 使用示例（Jedis）

```java
jedis.watch("balance");
Transaction tx = jedis.multi();
tx.decrBy("balance", 100);
tx.incrBy("other_account", 100);
List<Object> result = tx.exec(); // 若balance被修改，返回null
if (result == null) {
    // 事务失败，重试
}
```

## 3. 分布式锁

### 演进过程

**版本1：SETNX + EXPIRE（有原子性问题）**
```java
// 问题：两个命令非原子，中间宕机会死锁
Boolean lock = jedis.setnx("lock_key", "1");
if (lock) {
    jedis.expire("lock_key", 30); // 若这里宕机，锁永远不释放
}
```

**版本2：原子 SET 命令**
```java
// 使用 SET NX PX 保证加锁和设置过期时间的原子性
String result = jedis.set("lock_key", "random_value", "NX", "PX", 30000);
if ("OK".equals(result)) {
    // 获取锁成功
}
```

**版本3：安全锁（验证锁值 + Lua 脚本原子解锁）**
```java
// 加锁：使用唯一值（UUID）避免误删他人的锁
String uniqueId = UUID.randomUUID().toString();
String result = jedis.set("lock_key", uniqueId, "NX", "PX", 30000);

// 解锁：Lua 脚本保证"判断 + 删除"原子性
String luaScript = 
    "if redis.call('get', KEYS[1]) == ARGV[1] then " +
    "   return redis.call('del', KEYS[1]) " +
    "else " +
    "   return 0 " +
    "end";
jedis.eval(luaScript, Collections.singletonList("lock_key"), Collections.singletonList(uniqueId));
```

### 分布式锁的五个核心问题

| 问题 | 描述 | 解决方案 |
|------|------|---------|
| **死锁** | 客户端崩溃，锁无法释放 | 设置过期时间（PX 选项） |
| **误删锁** | A 的锁超期后 B 获锁，A 完成后误删 B 的锁 | 锁值使用唯一标识（UUID），删除前验证 |
| **原子性** | SETNX + EXPIRE 非原子 | 使用 `SET NX PX` 或 Lua 脚本 |
| **锁续期** | 业务执行时间超过锁过期时间 | 看门狗机制（Redisson 自动续期）|
| **集群故障** | 主从切换导致锁在新主节点上不存在 | RedLock 算法（有争议） |

### 可重入锁（Redisson 方案）

使用 Hash 结构 + Lua 脚本实现：
- Hash `field` 存储客户端唯一标识（UUID + 线程ID），`value` 存储重入次数
- 加锁：若 Hash 不存在则创建（计数=1）；若存在且是当前持有者则 `HINCRBY`（计数+1）
- 解锁：`HINCRBY`（计数-1），计数降为 0 时 `DEL` 整个 Hash

```lua
-- 加锁 Lua 脚本
if (redis.call('exists', KEYS[1]) == 0) then
    redis.call('hset', KEYS[1], ARGV[1], 1);
    redis.call('pexpire', KEYS[1], ARGV[2]);
    return 1;
end;
if (redis.call('hexists', KEYS[1], ARGV[1]) == 1) then
    redis.call('hincrby', KEYS[1], ARGV[1], 1);
    redis.call('pexpire', KEYS[1], ARGV[2]);
    return 1;
end;
return 0;
```

### Redlock 算法

适用于 Redis 集群主从切换导致锁丢失的场景（有争议）：

1. 向 N 个独立 Redis 节点（通常 5 个）依次请求加锁
2. 超过半数（N/2+1）成功且总耗时 < 锁有效期，认为加锁成功
3. 加锁失败则向所有节点发起解锁请求

**争议点**：依赖系统时钟同步，在进程 GC Pause 等极端场景下仍可能出现问题。

### Redis 锁 vs ZooKeeper 锁

| 特性 | Redis 分布式锁 | ZooKeeper 分布式锁 |
|------|--------------|------------------|
| **一致性模型** | AP（优先可用性，异步复制可能丢锁） | CP（强一致性） |
| **性能** | 高（内存操作） | 较低（节点间同步） |
| **锁释放** | 依赖过期时间 | 临时节点，会话断开自动释放 |
| **等待机制** | 客户端自旋轮询 | Watch 机制监听，顺序排队 |
| **实现复杂度** | 简单，但锁续期、集群故障处理较复杂 | 临时顺序节点天然支持，实现更可靠 |

**选择建议**：追求高性能高并发且能容忍极低概率锁失效 → Redis；要求高可靠强一致 → ZooKeeper。
