# 消息队列与发布订阅

## 1. 发布订阅（Pub/Sub）

### 工作原理

Redis Pub/Sub 包含三个角色：发布者（Publisher）、频道（Channel）、订阅者（Subscriber）。

**核心数据结构**：
- `redisServer.pubsub_channels`：字典，键为频道名，值为订阅该频道的客户端链表
- `redisServer.pubsub_patterns`：链表，存储模式订阅关系（如 `news.*`）

**主要命令**：
```bash
SUBSCRIBE channel1 channel2     # 订阅指定频道
PSUBSCRIBE news.*               # 模式订阅（支持通配符）
PUBLISH channel message         # 向频道发布消息
UNSUBSCRIBE channel1            # 退订频道
PUNSUBSCRIBE news.*             # 退订模式
```

### Pub/Sub 的局限性

| 缺陷 | 说明 |
|------|------|
| **消息无持久化** | 没有订阅者在线时消息丢失，重启后消息不复存在 |
| **无消息确认（ACK）** | 发布者无法知晓订阅者是否成功接收 |
| **无历史消息** | 只能接收订阅后发布的消息，无法回溯 |
| **广播模式** | 消息发给所有订阅者，无法负载均衡 |
| **客户端缓冲区溢出** | 消费慢时缓冲区堆积，超过限制（默认 8MB）会被强制断连 |

**适用场景**：消息可靠性要求不高的实时通知（如 Chat、实时推送），不适用于金融交易、订单处理等。

## 2. 基于 List 的消息队列

**原理**：生产者 `LPUSH`，消费者 `BRPOP`（阻塞等待）。

```java
// 生产者
jedis.lpush("queue", "message1");

// 消费者（阻塞版，超时0表示无限等待）
List<String> result = jedis.brpop(0, "queue");
String message = result.get(1); // result.get(0)是key名
processMessage(message);
```

**可靠消费（RPOPLPUSH）**：先将消息从队列移到处理中列表，处理完成后从处理中列表删除，崩溃恢复时重新处理。

**List 消息队列的问题**：
1. 无 ACK 机制，`BRPOP` 取出后若崩溃消息永久丢失
2. 一条消息只能被一个消费者消费，不支持多消费者组
3. 消息堆积无自动限制，会耗尽内存
4. 缺乏延迟队列、优先级队列等高级特性

**适用**：消息量小、允许少量丢失、业务逻辑简单的场景。

## 3. 基于 Sorted Set 的延迟队列

**原理**：将消息内容作为成员，**执行时间戳**作为分数（score）存入 ZSet，消费者轮询获取当前时间前的消息。

```java
// 生产者：延迟10秒执行
long executeTime = System.currentTimeMillis() + 10_000;
jedis.zadd("delay_queue", executeTime, "message_content");

// 消费者
while (!Thread.interrupted()) {
    Set<String> messages = jedis.zrangeByScore("delay_queue", 0, System.currentTimeMillis(), 0, 1);
    if (messages.isEmpty()) {
        Thread.sleep(500); // 避免空轮询
        continue;
    }
    String message = messages.iterator().next();
    // 用事务或Lua确保原子性，避免多客户端重复消费
    Transaction tx = jedis.multi();
    tx.zrem("delay_queue", message);
    if (tx.exec() != null) {
        processMessage(message);
    }
}
```

**缺点**：轮询有延迟误差，ZRANGE + ZREM 非原子需要事务/Lua 保护。

## 4. Redis Stream（推荐）

Redis 5.0 引入，是专为消息队列设计的数据结构，解决了 Pub/Sub 和 List 的局限性。

### 核心概念

| 概念 | 说明 |
|------|------|
| **消息 ID** | `timestamp-sequence`，如 `1640995200000-0`，唯一且有序 |
| **消费者组（Consumer Group）** | 组内消费者共享进度，每条消息仅被组内一个消费者处理，实现负载均衡 |
| **Last Delivered ID** | 组记录最后分发的消息 ID，崩溃恢复后从正确位置继续 |
| **PEL（Pending Entries List）** | 已分发但未 ACK 的消息，用于重试 |

### 基本操作

```java
// 生产者：添加消息（* 表示自动生成ID）
StreamEntryID id = jedis.xadd("mystream", StreamEntryID.NEW_ENTRY, 
    Map.of("field1", "value1", "field2", "value2"));

// 创建消费者组（从最新消息开始）
jedis.xgroupCreate("mystream", "group1", StreamEntryID.LAST_ENTRY, true);

// 消费者：读取消息（> 表示未读消息）
List<Map.Entry<String, List<StreamEntry>>> responses = jedis.xreadGroup(
    "group1", "consumer1", 1, 200, false,
    Map.of("mystream", StreamEntryID.UNRECEIVED_ENTRY));

// 处理并确认
for (StreamEntry entry : responses.get(0).getValue()) {
    try {
        processMessage(entry.getFields());
        jedis.xack("mystream", "group1", entry.getID()); // 确认
    } catch (Exception e) {
        // 不ACK，消息留在PEL中等待重试
    }
}

// 控制Stream最大长度，避免无限增长
jedis.xadd("mystream", 
    new XAddParams().maxLen(10000).approximateTrimming(),
    StreamEntryID.NEW_ENTRY, messageFields);
```

### 常用监控命令

```bash
XLEN mystream                       # 消息数量
XPENDING mystream group1 - + 10     # 查看PEL中未确认的消息
XCLAIM mystream group1 consumer2 60000 entry-id  # 转移超时未处理的消息
XINFO STREAM mystream               # Stream详细信息
XINFO GROUPS mystream               # 消费者组信息
```

## 5. Redis Stream vs Kafka

| 特性 | Redis Stream | Apache Kafka |
|------|-------------|-------------|
| **数据存储** | 基于内存（可持久化），容量受 RAM 限制 | 基于磁盘，支持 TB 级长期存储 |
| **延迟** | 亚毫秒级，实时性极高 | 毫秒级，吞吐量更高 |
| **集群** | Redis Cluster 分片，跨槽操作需重定向 | 原生分区副本，设计成熟 |
| **消息保留** | 基于最大长度或 TTL 淘汰 | 灵活的时间/大小保留策略 |
| **生态** | 轻量，快速部署 | Kafka Connect、Streams 等丰富生态 |
| **适用场景** | 低延迟实时系统，数据量可控 | 大数据管道，日志聚合，高吞吐长期存储 |

**选择建议**：
- 需要**低延迟、快速部署**，数据量可控 → Redis Stream
- 需要**高吞吐、长期存储、复杂流处理** → Kafka
