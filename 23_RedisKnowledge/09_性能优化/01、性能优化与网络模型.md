# 性能优化

## 1. 核心优化方向

Redis 性能优化是系统工程，需从架构、配置、命令、数据模型多维度入手。核心原则是**避免任何阻塞主线程的操作**——Redis 的命令执行是单线程，任何慢操作都会影响整体服务。

---

## 2. 网络模型

### 单线程 Reactor 模型（6.0 前）

Redis 的核心网络模型是**单线程 Reactor**：
1. **I/O 多路复用器**：封装 `epoll`（Linux）/`kqueue`（macOS），高效监听大量 Socket 事件
2. **事件分发器**：将就绪的事件分发给对应处理器
3. **事件处理器**：连接应答、命令请求、命令回复处理器

**工作流程**（`ae.c` 源码）：
- `aeMain` → 循环调用 `aeProcessEvents`
- `aeProcessEvents` → `epoll_wait` 等待 Socket 事件
- 新连接：`acceptTcpHandler` 创建客户端状态
- 命令请求：`readQueryFromClient` 读取解析，`processCommand` 执行
- 命令回复：将结果写入客户端缓冲区，`sendReplyToClient` 发送

### Redis 6.0 多线程 I/O

在单线程处理中，主线程在发送大量数据给客户端时会成为瓶颈，因此 6.0 引入多线程 I/O：
- **多线程范围**：仅用于**读取请求**（`readQueryFromClient`）和**发送响应**（`sendReplyToClient`）
- **命令执行**：仍然是单线程，保持原子性
- **配置**：默认关闭，需设置 `io-threads-do-reads yes` 和 `io-threads 4`（建议 CPU 核心数的 3/4）

## 3. Pipeline（管道）

Pipeline 将多个无依赖的命令打包一次发送，减少网络往返（RTT）。

**对比**：
```
普通模式：Request1 → Response1 → Request2 → Response2（N次RTT）
Pipeline：(Request1 + Request2 + Request3) → (Response1 + Response2 + Response3)（1次RTT）
```

**Java 示例（Jedis）**：
```java
Pipeline p = jedis.pipelined();
p.set("key1", "value1");
p.incr("counter");
Response<String> value = p.get("key1"); // 此时命令未发送
p.sync();                               // 批量发送并执行
String result = value.get();            // 安全获取结果
```

**注意**：Pipeline 中的命令**无原子性保证**，需要原子性请用 `MULTI/EXEC` 或 Lua 脚本。

**与批量命令的选择**：
- **原生批量命令**（`MSET/MGET/HMGET` 等）：操作同一数据结构的多个字段，有原子性保证
- **Pipeline**：批量执行不同命令或不同 Key，更灵活，无原子性

## 4. 大 Key 问题

**大 Key 定义**：Value 体积过大（字符串 > 10KB）或集合元素数量过多（> 5000）。

**危害**：
1. **阻塞主线程**：`DEL` 大 Key、`SMEMBERS/HGETALL` 大集合消耗大量 CPU
2. **网络拥堵**：传输占用带宽，影响其他请求
3. **内存不均**：集群模式下大 Key 导致节点内存使用不均衡
4. **持久化影响**：fork 时内存拷贝压力大，可能导致长时间 STW

**处理方案**：
- **拆分**：大 Hash 按字段哈希取模拆分为多个小 Hash（`user:info:{uid%10}`）
- **压缩**：Value 是 JSON/文本时，客户端压缩（GZIP）后存储
- **合适的数据结构**：用 HyperLogLog 替代 Set 做基数统计
- **异步删除**：使用 `UNLINK`（非阻塞）替代 `DEL`，或配置 `lazyfree-lazy-expire yes`

**大 Key 发现**：
```bash
redis-cli --bigkeys          # 扫描各类型最大的Key
redis-cli --memkeys          # 按内存占用扫描
OBJECT ENCODING key          # 查看编码
OBJECT IDLETIME key          # 查看空闲时间
DEBUG OBJECT key             # 查看详细信息（开发环境）
```

## 5. 热 Key 问题

**热 Key**：访问频率远高于其他 Key 的特定 Key（如顶流明星微博、秒杀商品信息）。

**危害**：集中在单个 Redis 节点，导致该节点 CPU/网络负载过高，影响整体服务。

**处理方案**：
1. **本地缓存**：Guava Cache/Caffeine 做 L1 缓存，大幅降低 Redis 访问量
2. **热 Key 分片**：将热 Key 复制成多个副本（`hotkey:0`、`hotkey:1`……），客户端随机选择
3. **热 Key 发现**：`redis-cli --hotkeys`（需 LFU 淘汰策略），或自定义监控工具
4. **集群代理**：代理层对热 Key 做读写分离

## 6. Lazy Free（惰性删除）

Redis 4.0 引入，将删除操作从主线程同步阻塞改为**后台线程异步回收内存**。

**触发方式**：
- **`UNLINK key`**：逻辑删除 key（立即从全局字典移除），内存回收异步进行
- **`FLUSHDB ASYNC` / `FLUSHALL ASYNC`**：异步清空数据库
- **配置触发**：让内存淘汰、过期键删除等也使用 Lazy Free

```properties
lazyfree-lazy-eviction yes      # 内存不足淘汰时异步删除
lazyfree-lazy-expire yes        # 过期键删除时异步
lazyfree-lazy-server-del yes    # 服务端隐式删除时异步（如rename/set覆盖）
replica-lazy-flush yes          # 从节点全量复制前清空数据时异步
```

## 7. 避免阻塞操作

| 危险操作 | 替代方案 |
|---------|---------|
| `KEYS *` | `SCAN cursor MATCH pattern COUNT 100`（游标分批扫描） |
| `SMEMBERS`（大集合） | `SSCAN cursor MATCH pattern COUNT 100` |
| `HGETALL`（大哈希） | `HSCAN cursor` |
| `LRANGE 0 -1`（大列表） | `LRANGE 0 99`（分页） |
| 长耗时 Lua 脚本 | 确保脚本轻量，无长时间循环 |
| `SORT`（大集合排序） | 客户端排序或使用 ZSet 预排序 |
| 大集合 `SUNION/SINTER` | 异步在客户端计算 |

**持久化影响**：AOF 的 `always` 策略每写必刷盘，性能最差；生产环境用 `everysec`。控制 RDB/AOF 重写不在业务高峰触发。

## 8. 慢查询日志

```bash
# 配置（阈值单位：微秒，1ms = 1000μs）
CONFIG SET slowlog-log-slower-than 5000  # 超过5ms记录
CONFIG SET slowlog-max-len 1000          # 最多保留1000条
CONFIG REWRITE                           # 持久化到配置文件

# 查看
SLOWLOG GET 10     # 最近10条慢查询
SLOWLOG LEN        # 当前慢查询数量
SLOWLOG RESET      # 清空慢查询日志
```

**注意**：慢查询记录的是命令在 Redis **服务端实际执行时间**，不含网络传输和排队时间。

**性能分析工具**：
```bash
redis-benchmark -h 127.0.0.1 -p 6379 -c 50 -n 10000  # 压测
redis-cli --latency          # 实时延迟监测
redis-cli --latency-history  # 延迟历史（每15秒一次）
redis-cli --stat             # 实时统计信息
```
