# 缓存应用模式

## 1. 缓存三大问题

### 缓存穿透（Cache Penetration）

**问题**：查询**缓存和数据库中都不存在**的数据，每次请求直接访问数据库，无法利用缓存。通常由恶意构造不存在 key 的请求引发。

**解决方案**：

**方案一：缓存空对象**（简单，推荐）
```java
Object data = getFromCache(key);
if (data == null) {
    data = getFromDB(key);
    if (data == null) {
        // 缓存空值，设置短过期时间（5分钟）
        stringRedisTemplate.opsForValue().set(key, "NULL", Duration.ofMinutes(5));
        return null;
    }
    setToCache(key, data, Duration.ofMinutes(30));
}
return data;
```

**方案二：布隆过滤器**（内存极省，适合大规模）
```java
// 系统启动时将所有合法key预热到布隆过滤器
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    1_000_000,  // 预期元素数量
    0.01        // 可接受误判率
);
// 请求进来时先过滤
if (!bloomFilter.mightContain(key)) return null; // 直接拦截
```

**方案三：接口层校验**：对请求参数合法性校验（ID 是否为负数、格式是否合法），拦截明显非法请求。

---

### 缓存击穿（Cache Breakdown / Hot Key Expiration）

**问题**：一个**热点 Key 突然失效**，大量并发请求同时穿透到数据库，数据库压力激增。

**解决方案**：

**方案一：互斥锁**（一致性好，有等待延迟）
```java
if (cacheValue == null) {
    if (tryLock(lockKey)) {
        try {
            // 双重检查，防止其他线程已重建
            cacheValue = getFromCache(key);
            if (cacheValue == null) {
                cacheValue = getFromDB(key);
                setToCache(key, cacheValue, ...);
            }
        } finally {
            releaseLock(lockKey);
        }
    } else {
        Thread.sleep(50);
        return queryById(id); // 重试
    }
}
```

**方案二：逻辑过期**（性能好，可能读到旧数据）
- 不设置物理过期时间，在 Value 中存储逻辑过期时间
- 发现逻辑过期后，返回旧数据，**异步**启动线程刷新缓存

**方案三：热点数据永不过期 + 后台异步更新**：对极热点数据设置永不过期，通过定时任务更新。

---

### 缓存雪崩（Cache Avalanche）

**问题**：**大量缓存数据同时失效**，或 Redis 服务宕机，所有请求直接访问数据库。

**解决方案**：

1. **随机过期时间**：避免大量 Key 同时失效
```java
int expireTime = BASE_TTL + new Random().nextInt(600); // 基础30分钟 + 0~10分钟随机
redisTemplate.opsForValue().set(key, value, expireTime, TimeUnit.MINUTES);
```

2. **多级缓存架构**：本地缓存（Caffeine）+ Redis，Redis 宕机时本地缓存兜底

3. **Redis 高可用**：主从复制 + 哨兵或集群，避免单点故障

4. **熔断降级**：引入 Sentinel/Hystrix，数据库压力过大时快速失败或返回兜底数据

## 2. 缓存更新策略

### Cache Aside Pattern（旁路缓存，推荐）

**读操作**：先读缓存 → 命中则返回 → 未命中则读 DB → 写入缓存并返回

**写操作**：先更新 DB → 再**删除**缓存（注意：是删除而不是更新）

**为什么是删除缓存而不是更新缓存？**
- 更新缓存可能导致并发时缓存值错乱（A 先更新缓存，B 后更新，但 B 的 DB 写入更晚，缓存与 DB 不一致）
- 删除缓存后，下次读时从 DB 加载最新值，更安全

**优先推荐**：简单有效，避免并发写的缓存数据不一致。

### Read/Write Through（穿透读写）

应用将**缓存作为主要数据源**：
- **读穿透**：缓存未命中时，由缓存组件从 DB 加载
- **写穿透**：应用更新缓存，缓存组件同步更新 DB

**优点**：代码简洁，缓存组件封装了 DB 操作。**缺点**：需要缓存组件支持此模式。

### Write Behind / Write Back（异步写回）

**写操作**先更新缓存，由后台异步批量更新 DB。

**优点**：写入性能极高，适合写多读少场景。**缺点**：数据一致性弱，宕机可能丢数据。

---

## 3. 缓存一致性

### 核心问题

Cache Aside 的**先更新 DB、后删除缓存**方案在极小概率下仍有并发问题：
- 线程 A 读缓存未命中，读 DB（旧值）
- 线程 B 更新 DB，删除缓存
- 线程 A 将旧值写入缓存

这种情况要求线程 A 的**读 DB**发生在线程 B 的**写 DB**之前，且线程 A 的**写缓存**发生在线程 B 的**删缓存**之后，概率极低。

### 延迟双删

```java
// 更新DB
updateDB(data);
// 删除缓存
deleteCache(key);
// 延迟几百毫秒再次删除（清除 A 刚写入的旧缓存）
Thread.sleep(500);
deleteCache(key);
```

### 基于 Binlog 的异步同步

通过 Canal 等工具监听 MySQL Binlog，异步更新缓存。与业务代码解耦，是更优雅的方案。

### 版本号机制

缓存数据中增加版本号，只允许更新版本号更高的数据覆盖旧版本。

## 4. 布隆过滤器详解

**原理**：使用一个 BitSet 和多个哈希函数：
- 添加元素：对元素用多个哈希函数计算多个位置，全部置 1
- 查询元素：检查所有对应位是否都为 1，若有 0 则**一定不存在**，若全为 1 则**可能存在**（假阳性）

**特点**：
- 不存在 → 一定返回 false（无假阴性）
- 存在 → 可能返回 true（有假阳性，误判率由 BitSet 大小和哈希函数数量决定）
- 不支持删除（删除会影响其他元素的判断）

**分布式环境**：可使用 RedisBloom 模块，在 Redis 中维护布隆过滤器，支持分布式共享。

## 5. 热点数据处理

**热点数据**：访问频率极高的 Key（如顶流明星的微博）。

**危害**：
- 集中在单个 Redis 节点，导致该节点 CPU/网络负载过高
- 过期瞬间大量并发直接访问 DB（即缓存击穿）

**处理策略**：

1. **本地缓存**：应用层 Guava Cache 或 Caffeine 做二级缓存，减少 Redis 网络请求
2. **逻辑分片**：将热 Key 复制成多个副本，如 `hotkey:1`、`hotkey:2`……，客户端随机选一个
3. **热点发现**：`redis-cli --hotkeys`（需要 LFU 淘汰策略），或自定义监控统计访问频率
4. **永不过期 + 后台更新**：热点数据不设过期时间，通过定时任务保持数据新鲜
