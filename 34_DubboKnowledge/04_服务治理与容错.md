## 一、负载均衡

### 1.1 算法对比

| 算法 | 实现类 | 特点 | 适用场景 |
|---|---|---|---|
| **Random**（默认）| `RandomLoadBalance` | 按权重随机，权重相同则纯随机 | 服务器性能相似，大多数场景 |
| **RoundRobin** | `RoundRobinLoadBalance` | 轮询，权重加权 | 均匀流量分发 |
| **LeastActive** | `LeastActiveLoadBalance` | 选当前活跃请求数最少的实例 | 响应时间差异较大时（自动加权）|
| **ConsistentHash** | `ConsistentHashLoadBalance` | 一致性哈希（虚拟节点），相同参数路由同一实例 | 有状态服务、缓存亲和 |
| **ShortestResponse** | `ShortestResponseLoadBalance` | 选最近响应时间最短的实例（滑动窗口统计）| 响应时间敏感的服务 |

**LeastActive 的精妙之处**：活跃请求数少不一定说明服务器轻松，也可能说明这台服务器处理速度快。LeastActive 本质上是把请求发给"最有能力接新请求的实例"。

### 1.2 权重与预热

Dubbo 支持给每个 Provider 设置权重（默认 100），权重越高，接收到的流量比例越大。

**预热（Warmup）机制**：新上线的实例需要 JVM JIT 编译预热，此时处理能力弱。Dubbo 通过预热机制让新实例的权重随时间缓慢增长到配置值：

```
有效权重 = min(配置权重, 配置权重 × (运行时间 / warmup时间))
```

配置：
```yaml
dubbo.provider.warmup: 600000  # 预热 10 分钟
dubbo.provider.weight: 100
```

这样新实例上线后，10 分钟内流量从 0 缓慢增加到正常水平，避免冷启动压垮。

### 1.3 一致性哈希实现

`ConsistentHashLoadBalance` 使用虚拟节点解决节点分布不均的问题：
- 默认每个 Provider 创建 160 个虚拟节点（`hash.nodes=160`）
- 虚拟节点均匀分布在 Hash 环上（0 到 2^32-1）
- 请求的 hash key 默认取第一个参数（可配置 `hash.arguments`）
- 在环上顺时针找到最近的虚拟节点，路由到对应的真实 Provider

## 二、集群容错

### 2.1 策略对比

| 策略 | 实现类 | 行为 | 适用场景 |
|---|---|---|---|
| **Failover**（默认）| `FailoverClusterInvoker` | 失败后重试其他节点，`retries=2`（共3次）| 读操作，幂等操作 |
| **Failfast** | `FailfastClusterInvoker` | 失败立即抛异常，不重试 | 核心写操作（创建订单等）|
| **Failsafe** | `FailsafeClusterInvoker` | 失败记录日志，返回空结果 | 非核心辅助操作（如审计日志）|
| **Failback** | `FailbackClusterInvoker` | 失败后异步定时重试（默认 5 秒/次）| 消息通知等最终一致性操作 |
| **Forking** | `ForkingClusterInvoker` | 同时调用多个节点，最快的一个返回 | 实时性要求极高，可接受多倍资源开销 |
| **Broadcast** | `BroadcastClusterInvoker` | 广播所有节点，任意一个失败则失败 | 缓存刷新、配置推送等 |

**选型原则**：
- 读服务 → `Failover`（允许重试，保证成功率）
- 写服务 → `Failfast`（不重试，防止重复写入）
- 非关键 → `Failsafe`（失败不影响主流程）

### 2.2 Failover 重试的坑

**非幂等写操作绝对不能用 Failover！**

场景：创建订单接口，第一次调用超时，Failover 触发重试到另一台，若第一次请求实际已成功（只是响应超时），就会创建两笔订单。

解决方案：
```java
@DubboReference(retries = 0)  // 写操作必须设置 retries=0
private OrderService orderService;
```

## 三、Filter 链（过滤器）

### 3.1 Filter 机制

Filter 是 Dubbo 的 AOP 机制，通过**责任链 + 装饰器模式**在调用前后插入逻辑。

```java
@SPI
public interface Filter {
    // 核心拦截方法
    Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException;
    
    // 可选：响应结果回调（Dubbo 3.x）
    interface Listener {
        void onResponse(Result appResponse, Invoker<?> invoker, Invocation invocation);
        void onError(Throwable t, Invoker<?> invoker, Invocation invocation);
    }
}
```

### 3.2 Filter 链的构建

Filter 链在服务暴露/引用时由 `ProtocolFilterWrapper.buildInvokerChain()` 构建：

```java
// 伪代码：展示链式构建逻辑（实际是逆序构建，使调用顺序与配置顺序一致）
private static <T> Invoker<T> buildInvokerChain(Invoker<T> invoker, 
                                                  String key, String group) {
    List<Filter> filters = ExtensionLoader.getActivateExtension(invoker.getUrl(), key, group);
    // 从最后一个 Filter 开始，逆序包装（装饰器模式）
    // 最终形成：Filter1 → Filter2 → ... → FilterN → 真正的 Invoker
    Invoker<T> last = invoker;
    for (int i = filters.size() - 1; i >= 0; i--) {
        final Filter filter = filters.get(i);
        final Invoker<T> next = last;
        last = new FilterNode<>(invoker, next, filter);  // 每层包装
    }
    return last;
}
```

**调用过程**（以 3 个 Filter 为例）：
```
请求 → Filter1.invoke() 
         → Filter2.invoke() 
              → Filter3.invoke() 
                   → 真正的 Invoker
              ← Filter3.invoke() 返回
         ← Filter2.invoke() 返回
    ← Filter1.invoke() 返回
```

### 3.3 内置关键 Filter

**Provider 端**：
- `EchoFilter`：回声测试（`$echo` 方法）
- `ClassLoaderFilter`：设置正确的 ClassLoader
- `ExecuteLimitFilter`：并发数控制（`executes` 参数）
- `AccessLogFilter`：访问日志
- `MonitorFilter`：调用统计（上报到监控中心）
- `TimeoutFilter`：Provider 端超时检测

**Consumer 端**：
- `FutureFilter`：异步回调处理
- `MonitorFilter`：调用统计
- `ActiveLimitFilter`：Consumer 端并发控制（`actives` 参数）
- `ConsumerContextFilter`：设置 RpcContext

### 3.4 自定义 Filter

```java
// 自定义日志 Filter（Provider 端自动激活）
@Activate(group = PROVIDER, order = 100)
public class RequestLogFilter implements Filter {
    
    private static final Logger log = LoggerFactory.getLogger(RequestLogFilter.class);
    
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        long start = System.currentTimeMillis();
        String service = invoker.getInterface().getSimpleName();
        String method = invocation.getMethodName();
        
        try {
            Result result = invoker.invoke(invocation);
            long elapsed = System.currentTimeMillis() - start;
            log.info("[DUBBO] {}.{} success, elapsed={}ms", service, method, elapsed);
            return result;
        } catch (RpcException e) {
            long elapsed = System.currentTimeMillis() - start;
            log.error("[DUBBO] {}.{} failed, elapsed={}ms, error={}", 
                      service, method, elapsed, e.getMessage());
            throw e;
        }
    }
}
```

**注册**（`META-INF/dubbo/org.apache.dubbo.rpc.Filter`）：
```
requestLog=com.example.RequestLogFilter
```

### 3.5 Filter vs Spring AOP

| 对比 | Dubbo Filter | Spring AOP |
|---|---|---|
| **作用范围** | 所有 Dubbo 调用（含跨进程）| Spring 管理的 Bean 方法调用 |
| **触发时机** | RPC 调用层，含网络传输前后 | 本地方法调用 |
| **上下文** | 访问 `RpcContext`（附件、TraceID、调用方信息）| 访问 Spring `ApplicationContext` |
| **组装方式** | `@Activate` 按条件自动加入 Filter 链 | `@Aspect` + 切点表达式 |
| **适用场景** | RPC 级别的治理（限流、鉴权、监控）| 业务级别的切面（事务、日志、权限）|

**选择原则**：需要访问 RPC 上下文（调用方 IP、TraceID、附件）或控制 RPC 调用本身（拦截请求、修改响应）→ Filter；纯业务切面（事务、本地方法增强）→ AOP。

## 四、动态配置与路由

### 4.1 动态配置

Dubbo 支持运行时修改服务配置，无需重启。配置优先级（高到低）：

```
方法级配置 > 接口级配置 > 全局配置
Consumer 端 > Provider 端（超时等参数）
```

通过配置中心（Nacos/ZK）动态下发 Override 规则：

```yaml
# 在 Nacos 配置中心修改 User 服务的超时时间（实时生效）
configVersion: v2.7
scope: service
key: com.example.UserService
enabled: true
configs:
- addresses: ["0.0.0.0"]
  side: provider
  parameters:
    timeout: "5000"
    weight: "200"
```

### 4.2 路由规则

Dubbo 支持多种路由规则，用于将请求路由到特定的 Provider 子集：

**条件路由**（`ConditionRouter`）：
```
# 将 VIP 用户路由到高性能机器
method=getUser&arguments[0]=VIP => host=192.168.1.1,192.168.1.2
```

**标签路由**（`TagRouter`）：
```yaml
# Provider 打标签
dubbo.provider.tag: gray

# 路由规则：带 gray tag 的请求只路由到打了 gray 标签的 Provider
configVersion: v3.0
force: true
tags:
- name: gray
  addresses: ["192.168.1.100:20880"]
```

标签路由常用于**灰度发布**：给新版本 Provider 打 `gray` 标签，将少量流量（通过网关注入标签）路由到灰度机器。

## 五、安全与认证

### 5.1 Token 验证（防绕过注册中心直连）

**问题**：Consumer 可以不经过注册中心，直接连接 Provider（配置 `url` 参数）。这样会绕过所有注册中心层面的权限控制。

**Token 验证**：Provider 在注册时生成 Token，写入注册中心。Consumer 必须从注册中心获取 Token，并在每次调用时附带。直连时无法获取 Token，调用会被拒绝。

```yaml
dubbo.provider.token: true  # 自动生成 UUID 作为 Token
# 或
dubbo.provider.token: "固定密钥"  # 静态 Token
```

### 5.2 AK/SK 签名鉴权

适用于**防篡改 + 防重放**的高安全场景：

```java
@Activate(group = {CONSUMER}, order = -1000)
public class SignatureFilter implements Filter {
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) {
        String ak = "your-access-key";
        String sk = "your-secret-key";
        long timestamp = System.currentTimeMillis();
        String nonce = UUID.randomUUID().toString();
        
        // 签名内容 = 接口名 + 方法名 + 时间戳 + nonce + SK
        String content = invoker.getInterface().getName() + ":" + 
                         invocation.getMethodName() + ":" + timestamp + ":" + nonce;
        String signature = HmacSHA256(content, sk);
        
        // 将认证信息放入 Attachment，跟随请求传递
        invocation.setAttachment("ak", ak);
        invocation.setAttachment("timestamp", String.valueOf(timestamp));
        invocation.setAttachment("nonce", nonce);
        invocation.setAttachment("signature", signature);
        
        return invoker.invoke(invocation);
    }
}
```

Provider 端对应 Filter 验证：从 Attachment 取出 ak/timestamp/nonce/signature，重新计算签名对比；检查 timestamp 是否在有效期内（如 5 分钟），防止重放攻击。

### 5.3 Token vs AK/SK 选型

| 对比维度 | Token 验证 | AK/SK 签名 |
|---|---|---|
| **防护目标** | 防止绕过注册中心直连 | 防篡改 + 防重放 |
| **安全强度** | 中（Token 泄露即失效）| 高（SK 不在网络传输）|
| **实现复杂度** | 简单，Dubbo 内置 | 需要自定义 Filter |
| **适用场景** | 内部服务之间的简单隔离 | 对外 API、高安全业务 |
| **动态管理** | 重新发布服务才能更换 Token | 可通过配置中心动态更新密钥 |

### 5.4 SSL/TLS 传输加密

```xml
<!-- Provider 端 -->
<dubbo:protocol name="dubbo" port="20880">
    <dubbo:parameter key="ssl-enabled" value="true"/>
    <dubbo:parameter key="server-key-cert-chain-path" value="/path/to/server.crt"/>
    <dubbo:parameter key="server-private-key-path" value="/path/to/server.key"/>
</dubbo:protocol>

<!-- Consumer 端 -->
<dubbo:consumer>
    <dubbo:parameter key="ssl-enabled" value="true"/>
    <dubbo:parameter key="client-trust-cert-collection-path" value="/path/to/ca.crt"/>
</dubbo:consumer>
```

底层由 Netty 的 `SslHandler` 实现，在 `NettyServer`/`NettyClient` 初始化时加入 Pipeline。

## 六、并发控制与连接控制

### 6.1 并发控制（executes）

`ExecuteLimitFilter`：通过 **Semaphore（信号量）**限制 Provider 端单个方法的并发执行数。

```yaml
dubbo.service."com.example.UserService".executes: 100        # 服务级
dubbo.service."com.example.UserService".methods.getUser.executes: 50  # 方法级（优先）
```

超过限制时，立即抛出 `RpcException`（快速失败，不排队），防止慢方法积压导致线程池满。

### 6.2 连接控制（accepts）

限制 Provider 接受的最大 TCP 连接数（在 TCP 连接建立阶段控制，比请求处理更早）：

```yaml
dubbo.protocol.accepts: 1000  # 最多接受 1000 个连接
```

超过限制时，直接关闭新连接，消费者端收到连接被拒绝的异常。
