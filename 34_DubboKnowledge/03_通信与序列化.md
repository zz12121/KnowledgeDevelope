## 一、Dubbo 协议（默认协议）

### 1.1 协议格式

Dubbo 协议使用**定长头（16字节）+ 变长体**的格式：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Header（16字节）                               │
├──────────┬──────────┬──────────────────────────────────────────────┤
│ Magic    │ Flag     │ Status │ Request ID（8字节）│ Data Length（4字节）│
│ 2字节    │ 1字节    │ 1字节  │ Long               │ Int                │
│ 0xdabb   │ 请求/响应 │ 状态码 │ 唯一请求标识        │ Body 数据长度       │
└──────────┴──────────┴────────┴────────────────────┴────────────────┘
│                        Body（变长）                                   │
│  序列化后的 RpcInvocation（请求端）或 AppResponse（响应端）               │
└─────────────────────────────────────────────────────────────────────┘
```

**Flag 字段（1字节）含义**：
- Bit 7：是否是请求（1=请求，0=响应）
- Bit 6：是否需要响应（1=双向，0=单向/心跳）
- Bit 5：是否是心跳事件
- Bit 0-4：序列化类型 ID（决定用哪种序列化方式）

**源码**：`org.apache.dubbo.remoting.exchange.codec.ExchangeCodec` 实现 Header 编解码；`DubboCodec`（子类）负责 Body 的序列化/反序列化。

### 1.2 单连接多路复用

**核心设计**：Dubbo 协议中，一个 Consumer 进程对一个 Provider IP:Port 只建立 **1 条 TCP 长连接**（`connections=1`），所有方法调用共享这条连接。

```
Consumer 进程  ────→  Provider A (IP:20880)
               1个 TCP 连接
               所有并发请求都在这条连接上发送（用 Request ID 标识）
```

**多路复用原理**：
1. 每个请求分配唯一的 64 位 `Request ID`
2. 请求发送后，`DefaultFuture` 以 Request ID 为 key 缓存等待的 Future
3. 响应到达时，从 Header 中读取 Request ID，找到对应的 `DefaultFuture` 唤醒
4. 不同请求的响应可以乱序到达，但通过 ID 能正确对应

**为什么一般不需要多连接**：Netty 的异步 IO 使单条连接可以高效处理大量并发请求（每个请求异步等待，不阻塞连接）。只有在极高延迟场景（跨机房）单连接出现队头阻塞时，才考虑增加 `connections`。

### 1.3 心跳机制

Dubbo 使用**双向心跳**保活长连接：

- **发送频率**：默认 60 秒发一次心跳（`heartbeat=60000`）
- **超时关闭**：若在 `heartbeat * 3` 时间内没有收到任何数据，认为连接已死，主动关闭并重连
- **实现**：`HeartbeatTimerTask`（消费者端定时发心跳）和 `ReconnectTimerTask`（检测死连接并重连）

## 二、Triple 协议（云原生协议）

### 2.1 设计理念

Triple 协议是 Dubbo 3.x 的默认 RPC 协议，基于 **HTTP/2 + Protocol Buffers**，完全兼容 gRPC 规范。

**为什么要推出 Triple？**

| 问题（Dubbo协议）| Triple 的解决方案 |
|---|---|
| 私有协议，网关/Sidecar 无法解析 | 基于 HTTP/2 标准，所有 HTTP/2 工具都能处理 |
| 不支持流式通信 | 原生支持单向/双向流 |
| 多语言互通困难 | 基于 Protobuf IDL，语言无关的接口定义 |
| 与 gRPC 生态不兼容 | 完全兼容 gRPC，可与 gRPC 服务直接互通 |

### 2.2 协议格式

Triple 复用 HTTP/2 的帧格式：

```
HTTP/2 Header Frame（元数据）：
  :method: POST
  :path: /org.apache.dubbo.demo.DemoService/sayHello
  content-type: application/grpc+proto   # 兼容 gRPC
  te: trailers
  grpc-timeout: 3S
  tri-service-version: 1.0.0             # Dubbo 扩展头：版本
  tri-service-group: test                # Dubbo 扩展头：分组
  tri-trace-rpcid: ...                   # TraceID（通过 HTTP Header 传递）

HTTP/2 Data Frame（Protobuf 序列化的请求体）：
  [5字节压缩标志 + 消息长度] + [Protobuf 二进制数据]
```

Dubbo 特有的元数据（attachment：版本、分组、超时等）通过 HTTP/2 Header 传递，`TripleCodec` 负责与 Dubbo 内部 `Attachments` 的互相转换。

### 2.3 流式通信

Triple 协议支持 4 种通信模式：

```java
public interface DemoService {
    // 1. 一元调用（Unary）：请求-响应
    HelloReply sayHello(HelloRequest request);

    // 2. 服务端流（Server Streaming）：一个请求，多个响应
    default void sayHelloServerStream(HelloRequest request, StreamObserver<HelloReply> responseObserver) {}

    // 3. 客户端流（Client Streaming）：多个请求，一个响应
    default StreamObserver<HelloRequest> sayHelloClientStream(StreamObserver<HelloReply> responseObserver) {
        return null;
    }

    // 4. 双向流（Bidirectional Streaming）：多个请求，多个响应
    default StreamObserver<HelloRequest> sayHelloBiStream(StreamObserver<HelloReply> responseObserver) {
        return null;
    }
}
```

## 三、Netty 线程模型

### 3.1 三层线程模型

Dubbo 基于 Netty，采用经典的 **Reactor 模式 + 业务线程池**三层结构：

```
网络层（Netty）              Dubbo 层                   业务层
┌──────────────┐           ┌────────────────┐          ┌──────────────────┐
│ Boss 线程组   │ accept    │                │          │                  │
│ (1-2 个线程) │ ────────→ │  Worker 线程组  │ dispatch │  业务线程池       │
│              │           │ (CPU核数×2线程) │ ────────→│  (Fixed/Eager 等)│
└──────────────┘           │                │          │                  │
                           │ 负责 IO读写     │          │ 负责业务逻辑执行  │
                           │ 编解码         │          │ 不能阻塞！        │
                           └────────────────┘          └──────────────────┘
```

- **Boss 线程组**：负责 Accept 新的 TCP 连接，将连接注册到 Worker
- **Worker 线程组**：负责 IO 读写、编解码（`DubboCodec`），不能阻塞（否则影响所有连接）
- **业务线程池**：执行实际的业务逻辑，允许阻塞

### 3.2 Dispatcher（派发策略）

Dispatcher 决定**哪些消息交给业务线程池处理，哪些直接在 Worker 线程处理**：

| Dispatcher | 说明 | 适用场景 |
|---|---|---|
| `message`（**默认**）| 只将请求/响应消息派发到业务线程池；连接/断开/心跳由 Worker 直接处理 | 大多数场景 |
| `all` | 所有消息（包括连接、断开、心跳）都派发到业务线程池 | 连接/断开处理有耗时逻辑时 |
| `direct` | 所有消息直接在 Worker 线程处理（不派发）| 业务逻辑极轻量，无 IO 操作 |
| `execution` | 只将请求派发到业务线程池；其他消息（响应、心跳）在 Worker 处理 | Consumer 端（只有响应消息）|
| `connection` | 连接/断开消息派发到业务线程池，其他在 Worker 处理 | 连接管理复杂时 |

**配置**：
```yaml
dubbo.protocol.dispatcher: message  # 默认值，一般无需修改
```

### 3.3 业务线程池策略

| 线程池 | 特点 | 适用场景 |
|---|---|---|
| `fixed`（**默认**）| 固定线程数 + 无界队列，线程预创建好 | 稳定流量，避免线程创建开销 |
| `cached` | 弹性线程数，空闲 60s 回收，不排队（SynchronousQueue）| 突发短流量 |
| `limited` | 有上限的 cached，只增不减（避免频繁创建/销毁）| 流量增长型服务 |
| `eager` | 优先创建线程（而非排队），达到 max 才排队 | 突发高流量，不希望请求排队等待 |

记忆口诀：**Fixed 稳、Cached 弹、Limited 限弹、Eager 急**

**配置**：
```yaml
dubbo.protocol.threads: 200         # 线程数
dubbo.protocol.threadpool: eager    # 线程池策略
```

### 3.4 业务线程池隔离

**问题**：如果所有服务共用同一个业务线程池，一个慢方法会占满线程池，影响同一 Provider 上的所有服务。

**方案一：多端口隔离（协议级隔离）**

```xml
<!-- 普通服务 -->
<dubbo:protocol name="dubbo" port="20880" threads="200" id="dubbo1"/>
<!-- 核心服务，独立端口和线程池 -->
<dubbo:protocol name="dubbo" port="20881" threads="100" id="dubbo2"/>

<dubbo:service interface="NormalService"  protocol="dubbo1"/>
<dubbo:service interface="CoreService"   protocol="dubbo2"/>
```

**方案二：自定义 Filter 动态切换线程池**

```java
@Activate(group = PROVIDER, order = -100)
public class IsolatedThreadPoolFilter implements Filter {
    // 为核心服务预留独立的 Executor
    private final ExecutorService coreExecutor = 
        new ThreadPoolExecutor(50, 50, 0L, TimeUnit.MILLISECONDS, 
                               new LinkedBlockingQueue<>(500));
    
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        if (isCoreService(invoker.getInterface())) {
            // 提交到独立线程池，返回 AsyncRpcResult
            CompletableFuture<AppResponse> future = CompletableFuture.supplyAsync(() -> {
                try {
                    return (AppResponse) invoker.invoke(invocation);
                } catch (RpcException e) {
                    throw new RuntimeException(e);
                }
            }, coreExecutor);
            return AsyncRpcResult.newDefaultAsyncResult(future, invocation);
        }
        return invoker.invoke(invocation);
    }
}
```

**方案三：Dubbo 3.x 服务级 Executor 配置**

```yaml
dubbo.service:
  "com.example.CoreService":
    executor-service-component: CoreServiceExecutor  # 绑定独立 Executor
```

## 四、序列化方案对比

### 4.1 各方案对比

| 序列化 | 性能 | 跨语言 | 是否需要 IDL | 特点 |
|---|---|---|---|---|
| **Hessian2**（默认）| 中等 | 有限 | 否 | 兼容性好，无需预注册类 |
| **Kryo** | 极高 | 否 | 否 | 速度最快，需要预注册类型 |
| **FST** | 高 | 否 | 否 | 接近 Kryo，更易用 |
| **Protobuf** | 高 | 是 | 是（.proto 文件）| 跨语言首选，Triple 协议内置 |
| **JSON（Fastjson/Jackson）**| 低 | 是 | 否 | 调试友好，性能较差 |
| **Java 原生** | 最低 | 否 | 否 | 仅用于兼容遗留系统，强烈不推荐 |

### 4.2 Kryo 使用注意

Kryo 的高性能依赖于**类型注册**（为每个类分配固定 ID），使用时必须预注册所有传输类：

```java
public class MyKryoOptimizer implements SerializationOptimizer {
    @Override
    public Collection<Class> getSerializableClasses() {
        return Arrays.asList(
            UserDTO.class, OrderDTO.class, PageResult.class, ...
        );
    }
}
```

```yaml
dubbo.protocol.serialization: kryo
dubbo.protocol.optimizer: com.example.MyKryoOptimizer
```

未注册的类会降级为按类名序列化，性能下降；更严重的是，如果类注册顺序不一致（Consumer 和 Provider 不同），会导致反序列化错误。

### 4.3 序列化安全

**漏洞背景**：Java 反序列化漏洞（如 Shiro 反序列化漏洞）的根源是攻击者可以构造恶意的序列化数据，触发类加载和方法执行。

**Dubbo 的防护**：
1. 避免使用 Java 原生序列化（`Serializable` + `ObjectInputStream`）
2. 使用 Hessian2/Kryo/Protobuf 等现代序列化框架，不依赖 Java 类加载机制
3. Dubbo 3.x 中对允许反序列化的类有白名单机制（可配置）

```yaml
# Dubbo 3.x 序列化安全配置
dubbo:
  application:
    serialize-check-status: WARN  # WARN: 打警告日志；STRICT: 严格白名单
```

## 五、粘包与拆包处理

TCP 是流式协议，不保证消息边界，需要应用层处理粘包/拆包。

Dubbo 使用**定长头**解决这个问题：

```java
// ExchangeCodec.decode() 核心逻辑（简化）
public Object decode(Channel channel, ChannelBuffer buffer) {
    int readable = buffer.readableBytes();
    if (readable < HEADER_LENGTH) {
        return DecodeResult.NEED_MORE_INPUT;  // 头还没收完，等待
    }
    
    // 读取 Header，获取 Body 长度
    byte[] header = new byte[HEADER_LENGTH];
    buffer.readBytes(header);
    int len = Bytes.bytes2int(header, 12);  // Header 中的 Data Length 字段
    
    if (readable < HEADER_LENGTH + len) {
        return DecodeResult.NEED_MORE_INPUT;  // Body 还没收完，等待
    }
    
    // Header + Body 都到齐了，解码
    byte[] data = new byte[len];
    buffer.readBytes(data);
    return decodeBody(channel, header, data);
}
```

通过"先读 Header 获取 Body 长度，再读 Body"的方式，完美解决了 TCP 粘包/拆包问题。
