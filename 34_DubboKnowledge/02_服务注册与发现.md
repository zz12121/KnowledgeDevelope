## 一、注册中心核心功能

注册中心在 Dubbo 中承担三个职责：
1. **服务注册**：Provider 启动时将服务地址和元数据写入注册中心
2. **服务发现**：Consumer 从注册中心查找并订阅 Provider 地址列表
3. **变更推送**：Provider 上下线时，注册中心通知所有订阅的 Consumer

### 1.1 Registry SPI 接口

```java
public interface Registry extends RegistryService {
    URL getUrl();
    boolean isAvailable();
    void destroy();
}

public interface RegistryService {
    void register(URL url);
    void unregister(URL url);
    void subscribe(URL url, NotifyListener listener);  // 监听变化
    void unsubscribe(URL url, NotifyListener listener);
    List<URL> lookup(URL url);                          // 主动查询
}
```

所有注册中心实现（ZK/Nacos/Etcd/K8s）都基于此接口，通过 SPI 按配置加载。

## 二、主流注册中心对比

| 对比维度 | ZooKeeper | Nacos（默认 AP）| Nacos（CP 模式）|
|---|---|---|---|
| **一致性模型** | CP（ZAB 协议） | AP（最终一致） | CP（Raft 协议）|
| **服务发现** | 临时节点 + Watcher | 心跳续约 + 长轮询 | 同左 |
| **配置中心** | 支持（节点数据）| 原生内建，功能丰富 | 同左 |
| **注册数据量** | 较小（节点存元数据）| 支持大量实例 | 同左 |
| **脑裂处理** | 停止服务保证一致 | 允许临时不一致，保高可用 | 同 ZK |
| **动态配置** | 需要自行封装 | 内建 Config API，支持灰度 | 同左 |
| **健康检查** | 依赖临时节点 + Session | 主动心跳 + 探测 | 同左 |
| **生态** | 成熟，被大量中间件使用 | Spring Cloud Alibaba 首选 | 同左 |

**选型建议**：
- **新项目**：首选 Nacos（AP 模式），服务发现 + 配置中心二合一，与 Dubbo + Spring Cloud Alibaba 生态集成最完整
- **强一致要求**（如分布式锁场景）：ZooKeeper 或 Nacos CP 模式
- **云原生 K8s 环境**：直接使用 K8s API Server 作为注册中心（Dubbo 3.x 支持）

### 2.1 ZooKeeper 注册中心原理

ZK 用**临时节点**存储服务地址，利用 Session 机制实现健康检查：

```
/dubbo
  /com.example.UserService
    /providers
      /dubbo://192.168.1.1:20880/...    # Provider A 的临时节点
      /dubbo://192.168.1.2:20880/...    # Provider B 的临时节点
    /consumers
      /consumer://192.168.1.3/...       # Consumer 的临时节点
    /configurators                       # 动态配置（Dubbo Admin 下发）
    /routers                             # 路由规则
```

- Provider 注册：`ZookeeperRegistry.doRegister()` 创建临时节点（URL 编码为路径）
- Consumer 订阅：`ZookeeperRegistry.doSubscribe()` 在 `/providers` 路径上添加 Watcher
- Provider 下线：Session 超时后，ZK 自动删除临时节点，触发 Watcher 通知所有 Consumer

**应用级服务发现路径（Dubbo 3.x）**：
```
/services
  /user-service
    /192.168.1.1:20880    # 实例级别，不含接口信息
    /192.168.1.2:20880
```

### 2.2 Nacos 注册中心原理

Nacos 的服务模型：**Namespace > Group > Service > Cluster > Instance**

- Dubbo 的 `application` → Nacos `ServiceName`
- 实例上报：Provider 通过心跳（默认 5s）维持注册，30s 无心跳则被标记为不健康，60s 后删除
- Consumer 订阅：Nacos 基于长轮询推送变更，比 ZK Watcher 更稳定（不受 Session 影响）

## 三、接口级 vs 应用级服务发现

这是 Dubbo 2.x → 3.x 最核心的变化，**必须深刻理解**。

### 3.1 接口级服务发现（Dubbo 2.x）

每个接口独立注册，服务唯一标识 = 接口名 + 分组 + 版本号：

```
注册路径：/dubbo/com.example.UserService/providers/dubbo://192.168.1.1:20880?...
```

**问题**：
- 假设 1 个应用 50 个接口 × 10 个实例 = 500 条记录，且每条重复包含应用名/IP/端口等相同信息
- 1 个实例下线需要推送 50 次删除通知（推送风暴）
- Consumer 需要为每个接口建立独立的 Watch（50 个 Watch）
- 大规模集群（1000+ 实例，100+ 接口）时注册中心压力极大

### 3.2 应用级服务发现（Dubbo 3.x）

以应用为粒度注册，注册中心只存实例信息：

```
注册路径：/services/user-service/192.168.1.1:20880?application=user-service&protocol=dubbo
```

接口元数据通过**服务自省（Service Introspection）**单独获取：

**两阶段发现机制**：

```
阶段1 - 地址发现：
Consumer 从注册中心获取 → user-service 的实例列表（10个IP:Port）

阶段2 - 服务自省：
Consumer 从目标实例的 MetadataService 获取 → 该实例提供的接口列表
（Local 模式：直接通过 RPC 调用实例内置的 MetadataService 接口）
```

**效果**：
- 注册中心数据量：500 条 → 10 条（降低 98%）
- Consumer Watch 数：50 → 1
- 下线推送次数：50 → 1

### 3.3 服务自省（Service Introspection）

Consumer 需要从 Provider 获取接口元数据，有两种模式：

**Remote 模式**：
- Provider 启动时，将接口元数据发布到远程元数据中心（ZK/Nacos 的特定路径）
- Consumer 从元数据中心查询，无需直接调用 Provider

**Local 模式（默认）**：
- Provider 暴露内置的 `MetadataService` RPC 接口
- Consumer 直接通过 RPC 调用 Provider 的 `MetadataService.getServiceDefinition()` 获取接口详情
- 点对点，避免了元数据中心的依赖

源码关键类：
- `MetadataServiceExporter`：Provider 端，收集并暴露元数据
- `MetadataUtils.getMetadataServiceProxy()`：Consumer 端，获取目标实例的 MetadataService 代理

## 四、注册中心宕机时的可用性保障

**注册中心宕机，服务调用不会中断**。

Dubbo 在消费者端维护了**本地服务列表缓存**：

1. Consumer 启动时，从注册中心订阅 Provider 列表，缓存到 `RegistryDirectory.methodInvokerMap`
2. 注册中心宕机后，`RegistryDirectory` 中的 `Invoker` 列表不变，Consumer 继续使用缓存的地址调用
3. Dubbo 还将地址信息持久化到本地文件（`~/.dubbo/dubbo-registry-{application}-{registryAddress}.cache`），即使 Consumer 重启，也能从文件恢复上次的地址列表

**限制**：宕机期间，新 Provider 上线或旧 Provider 下线的变更无法同步到 Consumer。

## 五、多注册中心与服务分组

### 5.1 多注册中心场景

```xml
<dubbo:registry id="zk-cn" address="zookeeper://zk-cn:2181"/>
<dubbo:registry id="nacos-us" address="nacos://nacos-us:8848"/>

<!-- 向两个注册中心同时注册 -->
<dubbo:service interface="com.example.GlobalService" registry="zk-cn,nacos-us"/>
```

**实现**：`ServiceConfig.doExportUrlsFor1Protocol()` 遍历注册中心列表，为每个注册中心调用一次 `RegistryProtocol.export()`，在多个注册中心各创建一份注册节点。

**消费者合并**：从多个注册中心获取的 `Invoker` 列表，由 `MergeableCluster` 合并为一个统一的集群 Invoker。

### 5.2 服务分组（Group）

服务三元组唯一标识：**接口名 + 分组（group）+ 版本号（version）**

```java
@DubboService(group = "payment", version = "1.0.0")
public class PaymentUserServiceImpl implements UserService {}

@DubboService(group = "logistics", version = "1.0.0")
public class LogisticsUserServiceImpl implements UserService {}
```

**分组聚合**：
```java
// 同时调用多个分组，合并结果
@DubboReference(group = "*", merger = "list")
private UserService userService;
```

`MergeableClusterInvoker` 并发调用各分组，通过 `ListMerger`/`MapMerger` 合并返回值。

## 六、版本迁移：2.x → 3.x 的双注册/双订阅

> 详细迁移步骤参见：`10_DevelopLanguage/009_Dubbo/01_DubboSubject/15、版本升级与兼容性.md`

**核心配置**：

```yaml
# Provider 端
dubbo.application.register-mode: all  # 同时写接口级和应用级（双注册）

# Consumer 端
dubbo.application.service-discovery.migration: APPLICATION_FIRST  # 优先应用级，失败降级接口级（双订阅）
```

**三态状态机**：

```
FORCE_INTERFACE  →  APPLICATION_FIRST  →  FORCE_APPLICATION
  （兼容态）            （过渡态）             （完成态）
```

状态迁移通过配置中心动态下发 `MigrationRule`，无需重启任何服务，实现流量的平滑切换。
