## 一、监控与链路追踪

### 1.1 Dubbo 内置监控能力

**MonitorFilter**：内置 Filter，统计每个服务/方法的调用次数、失败次数、最大/最小/平均耗时，上报到 MonitorCenter（通常是 Dubbo Admin 或自研监控平台）。

**QoS（Quality of Service）**：提供在线运维命令接口。

```bash
# 通过 telnet 连接 QoS 端口
telnet localhost 22222

# 常用命令
ls                    # 列出所有服务
ls -l                 # 详细信息（含提供者地址）
ps                    # 查看 Provider 列表
count UserService      # 统计某服务的调用次数和成功率
trace UserService 10   # 追踪 10 次调用的耗时分布
online                # 手动上线服务（优雅发布）
offline               # 手动下线服务（优雅停机）
```

也可通过 HTTP 访问（默认 22222 端口）：
```bash
curl http://localhost:22222/ls
curl http://localhost:22222/ready   # 健康检查（与 K8s 探针对接）
```

Dubbo 3.x 内置 Metrics 支持：

```yaml

    aggregation:
```

### 1.3 SkyWalking 链路追踪（推荐）

**无侵入 Agent 方式**（推荐）：

```bash
# JVM 启动参数添加 Agent
-javaagent:/path/to/skywalking-agent.jar
-Dskywalking.agent.service_name=user-service
-Dskywalking.collector.backend_service=oap-server:11800
```

SkyWalking Agent 通过字节码增强（Java Agent + Byte Buddy），自动拦截 Dubbo 调用，注入/提取 TraceID，无需修改任何业务代码。

**TraceID 传递原理**：

```
网关（生成 TraceID=abc123）
  → Dubbo Consumer（通过 SkyWalking Agent 从 ThreadLocal 取 TraceID）
    → RpcContext.getServerAttachments().put("sw8", "abc123-span1")  // 放入 Attachment
      → Dubbo Provider（SkyWalking Agent 从 Attachment 取 TraceID）
        → 继续传递给下游服务
```

**MDC 日志关联**：SkyWalking 会自动将 TraceID 注入到 SLF4J 的 MDC 中，配置 logback/log4j2 后，所有日志自动包含 TraceID：

```xml
<!-- logback.xml -->
<pattern>%d{yyyy-MM-dd HH:mm:ss} [%X{SW_CTX}] [%thread] %-5level %logger - %msg%n</pattern>
```

这样，一次请求的所有日志（跨多个服务）都带有相同的 TraceID，在 ELK 中用 TraceID 过滤，就能还原完整的请求链路。

### 1.4 Zipkin 集成（显式 Filter 方式）

```java
@Activate(group = {CONSUMER, PROVIDER})
public class ZipkinTracingFilter implements Filter {
    private final Tracing tracing;
    private final Tracer tracer;
    
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        // Consumer 端：从上下文取父 Span 或创建新 Span
        Span span = tracer.nextSpan()
            .name(invocation.getMethodName())
            .tag("service", invoker.getInterface().getName())
            .start();
        
        // 通过 Attachment 传递 TraceID（B3 格式）
        try (Tracer.SpanInScope ws = tracer.withSpanInScope(span)) {
            TraceContext ctx = span.context();
            invocation.setAttachment("X-B3-TraceId", ctx.traceIdString());
            invocation.setAttachment("X-B3-SpanId", ctx.spanIdString());
            invocation.setAttachment("X-B3-Sampled", "1");
            
            return invoker.invoke(invocation);
        } catch (Throwable e) {
            span.error(e);
            throw e;
        } finally {
            span.finish();
        }
    }
}
```

### 1.5 TraceID 在异步线程中如何传递

**问题**：SkyWalking Agent 的 TraceID 存储在 `ThreadLocal` 中，当业务代码使用线程池执行异步任务时，新线程无法获取父线程的 TraceID，链路断裂。

**方案一：手动包装（显式控制）**

```java
// 在 Filter 中手动捕获快照，异步任务中恢复
AbstractSpan parentSpan = ContextManager.capture();

CompletableFuture.runAsync(() -> {
    ContextManager.continued(parentSpan);  // 恢复 TraceID
    try {
        // 业务逻辑
    } finally {
        ContextManager.stopSpan();         // 清理
    }
}, executor);
```

**方案二：TTL（TransmittableThreadLocal）自动传递（推荐）**

阿里开源的 `transmittable-thread-local`（TTL）：

```xml
<dependency>
    <groupId>com.alibaba</groupId>
    <artifactId>transmittable-thread-local</artifactId>
    <version>2.14.2</version>
</dependency>
```

```bash
# JVM 启动参数加入 TTL Agent
-javaagent:transmittable-thread-local-2.14.2-jar-with-dependencies.jar
```

TTL Agent 会自动增强线程池的 `execute`/`submit` 方法，在提交任务时自动捕获所有 `TransmittableThreadLocal` 的值，并在任务执行时恢复，**完全透明，无需修改业务代码**。

**方案三：SkyWalking Agent 内置传递**

SkyWalking 7.0+ 版本的 Agent 可以自动处理 `ThreadPoolExecutor`/`CompletableFuture` 等场景，只需配置 Agent 即可，无需额外操作。

## 二、Kubernetes 部署最佳实践

### 2.1 IP 注册问题（容器化核心问题）

容器内有多个网络接口，Dubbo 必须注册 **Pod IP** 到注册中心，而不是宿主机 IP 或回环地址。

**推荐解决方案**（通过 K8s Downward API 注入 Pod IP）：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
spec:
  template:
    spec:
      containers:
      - name: user-service
        image: user-service:latest
        env:
        # K8s 注入 Pod IP
        - name: POD_IP
          valueFrom:
            fieldRef:
              fieldPath: status.podIP
        # Dubbo 使用 Pod IP 注册
        - name: DUBBO_IP_TO_REGISTRY
          value: $(POD_IP)
        - name: DUBBO_IP_TO_BIND
          value: "0.0.0.0"
        ports:
        - containerPort: 20880  # Dubbo 协议端口
        - containerPort: 22222  # QoS 端口（用于健康检查）
        livenessProbe:
          httpGet:
            path: /live
            port: 22222
        readinessProbe:
          httpGet:
            path: /ready
            port: 22222
        resources:
          requests:
            cpu: "500m"
            memory: "1Gi"
          limits:
            cpu: "2"
            memory: "4Gi"
```

**IP 获取优先级**（`NetUtils.getLocalHost()`）：
1. 环境变量 `DUBBO_IP_TO_REGISTRY`
2. JVM 参数 `-Ddubbo.ip.to.registry`
3. 配置 `dubbo.network.interface.prefer`（指定网卡名）
4. 自动探测（容器内可能不准确）

强烈建议通过环境变量显式注入，避免依赖自动探测。

### 2.2 优雅上下线

**启动阶段**：
- 配置 `delay=-1`（等 Spring 初始化完成后暴露服务）
- K8s `readinessProbe` 检查 `/ready` 端点，仅当 Provider 就绪后才将流量路由到该 Pod

**停机阶段（非常重要）**：

```yaml
# 配置 preStop hook，给 Dubbo 优雅停机的时间
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 15"]  # 等待 K8s 从 Service 摘除流量

# Pod 终止等待时间，必须 > preStop 时间 + 请求处理超时时间
terminationGracePeriodSeconds: 60
```

**Dubbo 停机流程**：
1. 调用 `dubbo:application qos offline`（或 Spring 触发 `@PreDestroy`）
2. 向注册中心注销服务，Consumer 停止路由流量到该 Provider
3. 等待正在处理的请求完成（`shutdownTimeout` 默认 10 秒）
4. 关闭 Netty Server，释放端口
5. 关闭注册中心连接，释放资源

### 2.3 使用 Kubernetes 作为注册中心

```yaml
dubbo:
  registry:
    address: kubernetes://
    parameters:
      registry-type: service   # 使用 K8s Service API
  protocol:
    name: tri
    port: 50051
```

需要为服务账号配置 RBAC 权限：

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: dubbo-service-discovery
rules:
- apiGroups: [""]
  resources: ["pods", "endpoints", "services"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["update", "patch"]   # 写入 Pod Annotation
```

## 三、Service Mesh 集成

> 详细原理参见：`10_DevelopLanguage/009_Dubbo/01_DubboSubject/16、云原生与容器化.md`

### 3.1 双模 Mesh 对比

| 维度 | Sidecar Mesh | Proxyless Mesh |
|---|---|---|
| **架构** | App + Envoy Sidecar | App（内置 xDS 客户端）|
| **性能** | 额外 ~1-2ms 延迟 | 无额外延迟 |
| **语言支持** | 语言无关 | 目前主要 Java |
| **运维复杂度** | Sidecar 注入/升级需管理 | 仅管理 SDK 版本 |
| **功能完整性** | 全功能（mTLS/负载均衡/熔断）| 逐步完善中 |
| **适用场景** | 多语言混合/遗留系统 | Java 为主/性能敏感 |

### 3.2 Proxyless Mesh 关键配置

```yaml
# pom.xml 添加依赖
# dubbo-mesh-istio

# application.yml
dubbo:
  application:
    name: user-service
  registry:
    address: mesh://istiod.istio-system.svc:15012?type=proxyless
```

xDS 资源订阅流程：
1. `XdsClient` 与 Istiod 建立 gRPC 长连接
2. 订阅 `EDS`（服务实例列表）→ 更新 `RegistryDirectory`
3. 订阅 `RDS/LDS`（路由和负载均衡规则）→ 更新 `RouterChain` 和 `LoadBalance`
4. Istiod 下发的 `VirtualService`/`DestinationRule` 由 `MeshRouter` 翻译为 Dubbo 内部路由逻辑

## 四、性能调优全攻略

### 4.1 调优方法论

```
建立监控基线（Prometheus + Grafana）
  ↓
全链路压测（JMeter + 真实场景）
  ↓
找到瓶颈（CPU/内存/IO/线程）
  ↓
针对性优化（一次只改一个变量）
  ↓
验证效果（重新压测）
  ↓
循环迭代
```

### 4.2 关键调优参数

```yaml
dubbo:
  protocol:
    name: dubbo
    port: 20880
    threads: 300           # 业务线程池大小（IO密集型可以更大）
    threadpool: eager      # 突发流量下优先创建线程
    accepts: 2000          # 最大连接数
    serialization: kryo    # 高性能序列化
    payload: 10485760      # 最大数据包大小（10MB，默认88MB较大）
    heartbeat: 60000       # 心跳间隔
    
  provider:
    timeout: 3000
    delay: -1              # Spring 初始化完成后再暴露
    retries: 0             # Provider 端不重试
    executes: 200          # 服务级并发数限制
    
  consumer:
    timeout: 5000          # 消费者超时 > 提供者超时
    retries: 2             # 读操作可重试；写操作在方法级设置 retries=0
    loadbalance: leastactive  # 响应时间差异大时用 LeastActive
    check: false           # 启动不检查依赖，避免启动顺序问题
```

### 4.3 JVM 调优要点

```
# 生产环境推荐 G1 GC 配置
-Xms4g -Xmx4g                    # 堆大小固定，避免动态扩缩引起 Full GC
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200          # GC 目标停顿时间（避免超时）
-XX:G1HeapRegionSize=16m
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=512m
-XX:+HeapDumpOnOutOfMemoryError   # OOM 时自动 Dump
-XX:HeapDumpPath=/app/logs/
```

Full GC 停顿 > 配置的超时时间 → Consumer 端超时，即使 Provider 实际没有问题。GC 调优是避免"假超时"的重要手段。

### 4.4 压测关键指标

| 指标 | 含义 | 告警阈值 |
|---|---|---|
| **P99 响应时间** | 99% 的请求在 Xms 内完成 | > 目标 SLA 的 2 倍 |
| **错误率** | 调用失败（超时/异常）的比例 | > 0.1% |
| **线程池活跃度** | 活跃线程数 / 最大线程数 | > 90% |
| **GC 停顿频率** | Full GC 次数和停顿时间 | > 1次/分钟 或 > 500ms |
| **网络带宽** | 进出流量 | > 70% 带宽上限 |

## 五、高可用架构设计

### 5.1 注册中心高可用

- ZooKeeper：3/5 节点集群（需要 Quorum = N/2 + 1 节点存活）
- Nacos：至少 3 节点集群，使用嵌入式 Raft 选主

**注册中心宕机时的保障**：Consumer 本地缓存 + 本地文件缓存，短时间宕机不影响已有调用。

### 5.2 Provider 高可用

- 至少 3 个实例，跨可用区（AZ）部署
- 新实例上线：通过 `delay + warmup` 缓慢接收流量
- 实例下线：先注销（等 Consumer 感知到），再停止处理，再关闭端口

### 5.3 Consumer 高可用

- Failover 策略 + 合理的 `retries` 配置
- 本地 Mock 降级：为非核心依赖提前准备降级逻辑
- 配合 Sentinel 熔断：下游故障时快速熔断，防止雪崩

### 5.4 灰度发布方案

**标签路由灰度**：

```
流程：
1. 新版本 Provider 打标签（dubbo.provider.tag=gray）
2. 在 Dubbo Admin 配置标签路由规则
3. 通过网关在特定请求中注入标签（RpcContext.getServerAttachments().put("dubbo.tag", "gray")）
4. 携带 gray 标签的请求只路由到灰度 Provider
5. 验证稳定后，全量发布，删除路由规则
```

**版本号灰度**：

```java
// 同时部署 v1.0.0 和 v2.0.0
@DubboService(version = "2.0.0")
public class UserServiceV2Impl implements UserService {}

// 通过动态配置控制路由权重（90% v1, 10% v2）
```
