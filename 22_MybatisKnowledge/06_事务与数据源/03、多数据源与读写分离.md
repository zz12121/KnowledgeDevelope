# 多数据源与读写分离

## 1. 静态多数据源 vs 动态数据源

**静态多数据源**：每个 Mapper 在项目启动时就已绑定到固定的 DataSource，通过 `@MapperScan` + `sqlSessionFactoryRef` 实现。适用于数据库间完全隔离的场景（如主数据库 + 日志数据库）。详见 [[02、数据源与连接池#5. 多数据源静态配置|静态多数据源配置]]。

**动态数据源**：单个 `SqlSessionFactory` 背后对接多个物理 DataSource，在**运行时**根据业务逻辑动态切换。核心技术是 Spring 的 `AbstractRoutingDataSource`。

---

## 2. AbstractRoutingDataSource 动态切换

`AbstractRoutingDataSource` 是 Spring 提供的抽象类，其内部维护一个 `targetDataSources` Map（key → DataSource），每次获取连接时调用 `determineCurrentLookupKey()` 返回当前要使用的 key，从 Map 中找到对应的 DataSource。

### 实现步骤

**① 定义 ThreadLocal 上下文持有器**

```java
public class DataSourceContextHolder {
    private static final ThreadLocal<String> CONTEXT_HOLDER = new ThreadLocal<>();

    public static void setDataSourceType(String type) {
        CONTEXT_HOLDER.set(type);
    }

    public static String getDataSourceType() {
        return CONTEXT_HOLDER.get();
    }

    public static void clearDataSourceType() {
        CONTEXT_HOLDER.remove();  // 防止内存泄漏，用完必须清理
    }
}
```

**② 继承 AbstractRoutingDataSource，实现路由逻辑**

```java
public class DynamicDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return DataSourceContextHolder.getDataSourceType();
    }
}
```

**③ 配置动态数据源 Bean**

```java
@Bean
public DataSource dynamicDataSource(
        @Qualifier("primaryDataSource") DataSource primary,
        @Qualifier("secondaryDataSource") DataSource secondary) {
    Map<Object, Object> targetDataSources = new HashMap<>();
    targetDataSources.put("primary", primary);
    targetDataSources.put("secondary", secondary);

    DynamicDataSource dynamicDataSource = new DynamicDataSource();
    dynamicDataSource.setTargetDataSources(targetDataSources);
    dynamicDataSource.setDefaultTargetDataSource(primary); // 默认数据源
    return dynamicDataSource;
}
```

**④ AOP 切面 + 自定义注解优雅切换**

```java
// 自定义注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface DataSource {
    String value() default "primary";
}

// AOP 切面
@Aspect
@Component
public class DataSourceAspect {
    @Before("@annotation(ds)")
    public void before(JoinPoint joinPoint, DataSource ds) {
        DataSourceContextHolder.setDataSourceType(ds.value());
    }

    @After("@annotation(ds)")
    public void after(JoinPoint joinPoint, DataSource ds) {
        DataSourceContextHolder.clearDataSourceType();
    }
}

// 使用示例
@Service
public class UserService {
    @DataSource("secondary")
    public User getUserFromSecondaryDB(Long id) {
        return userMapper.selectById(id);
    }
}
```

### 注意事项

- **事务与数据源切换顺序**：动态数据源切换必须发生在事务开启**之前**。因为 Spring 事务开始时会通过 `DataSourceTransactionManager` 从 DataSource 获取连接并绑定到 ThreadLocal，此后连接已固定，再修改 ThreadLocal 中的 key 也来不及了。
- **ThreadLocal 内存泄漏**：切面的 `@After` 必须调用 `clearDataSourceType()`，避免线程复用时使用错误的数据源。

## 3. 读写分离

读写分离是多数据源的特例：**Master（主库）** 负责写操作，**Slave（从库）** 负责读操作，利用数据库主从同步架构分摊读压力。

### 实现方案

架构与动态数据源相同，路由逻辑更简单：

```java
public class ReadWriteDataSourceRouter extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        // 利用 Spring 事务的 readOnly 属性自动判断读写
        boolean isReadOnly = TransactionSynchronizationManager.isCurrentTransactionReadOnly();
        return isReadOnly ? "slave" : "master";
    }
}
```

### 与 Spring 事务集成

在 Service 方法上标注 `@Transactional(readOnly = true)` 即可自动路由到从库：

```java
@Service
public class UserService {

    // 写操作 → 路由到 master
    @Transactional
    public void createUser(User user) {
        userMapper.insert(user);
    }

    // 读操作 → 路由到 slave
    @Transactional(readOnly = true)
    public User getUserById(Long id) {
        return userMapper.selectById(id);
    }
}
```

### 多从库负载均衡

当有多个从库时，`determineCurrentLookupKey()` 中可加入轮询或随机逻辑：

```java
private final List<String> slaveKeys = Arrays.asList("slave1", "slave2", "slave3");
private final AtomicInteger counter = new AtomicInteger(0);

@Override
protected Object determineCurrentLookupKey() {
    if (TransactionSynchronizationManager.isCurrentTransactionReadOnly()) {
        int index = counter.getAndIncrement() % slaveKeys.size();
        return slaveKeys.get(index);
    }
    return "master";
}
```

### 注意事项

- **主从延迟**：从库存在复制延迟（通常毫秒级），刚写入 master 的数据可能在 slave 还未同步。对一致性要求高的读操作（如付款后立刻读取余额）应走主库。
- **强一致读**：通过注解或上下文强制当前请求走主库，避免读到脏数据。

## 4. 方案对比

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| 静态多数据源 | 数据库完全隔离 | 配置简单，不会误切 | 灵活性差 |
| AbstractRoutingDataSource | 运行时切换，读写分离 | 对业务代码透明，灵活 | 需注意事务顺序 |
| Sharding-JDBC | 分库分表 | 功能全面，支持复杂分片规则 | 引入额外复杂度 |
