# 事务管理与传播行为

## 一、MyBatis 的事务管理机制

MyBatis 通过 `Transaction` 接口和 `SqlSession` 生命周期管理事务，根据使用场景分为两种模式。

**JdbcTransaction**：用于独立使用 MyBatis（不整合 Spring）的场景。`commit()` / `rollback()` 直接调用底层 JDBC Connection 的对应方法，事务完全由开发者手动控制。

**ManagedTransaction**：将事务管理权交给外部容器（Spring 或应用服务器），`commit()` 和 `rollback()` 都是空实现，事务由容器控制。

**独立使用 MyBatis 时的标准写法**：
```java
SqlSession session = sqlSessionFactory.openSession(); // autoCommit=false，开启事务
try {
    userMapper.insert(user);
    logMapper.insertLog("CREATE");
    session.commit();  // 手动提交
} catch (Exception e) {
    session.rollback();  // 回滚
} finally {
    session.close();
}
```

## 二、Spring 集成后的事务管理

与 Spring 集成后，事务主导权完全交给 Spring 的 `DataSourceTransactionManager`，MyBatis 的 Transaction 接口退出舞台。

**整合核心机制**：
1. `DataSourceTransactionManager` 开启事务，将 Connection 包装为 `ConnectionHolder` 绑定到当前线程的 ThreadLocal（`TransactionSynchronizationManager`）
2. `SqlSessionTemplate` 每次操作前，通过 `TransactionSynchronizationManager.getResource()` 取出当前事务绑定的 SqlSession，确保同一事务内所有操作使用同一连接
3. 方法结束，Spring 的 `TransactionInterceptor` 根据是否有异常决定 commit 或 rollback

**声明式事务使用**：
```java
@Service
public class UserService {
    @Transactional
    public void createUserWithLog(User user) {
        userMapper.insert(user);          // 操作1
        logMapper.insertLog(user.getId()); // 操作2，同一事务
        // 方法正常结束自动 commit，抛出 RuntimeException 自动 rollback
    }
}
```

## 三、事务传播行为

传播行为定义了**事务方法之间调用时事务如何传播**，是 Spring 框架的概念，通过 `@Transactional(propagation = ...)` 配置。

**最常用的三种**：

**REQUIRED（默认）**：有事务就加入，没有就新建。绝大多数场景用这个就够了。

**REQUIRES_NEW**：无论如何都新建一个独立事务，并挂起当前事务。两个事务完全独立，互不影响。典型场景：操作日志记录——即使主业务回滚，也要确保日志写入成功：
```java
@Transactional(propagation = Propagation.REQUIRED)
public void placeOrder(Order order) {
    // 主业务
    logActivity(order.getId());  // 日志用独立事务，主业务回滚不影响日志
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void logActivity(Long orderId) {
    logMapper.insert(new Log(orderId, "PLACED"));
}
```

**NESTED**：嵌套事务，是外部事务的一部分（通过 Savepoint 实现），嵌套事务可以独立回滚而不影响外部事务，但提交依赖外部事务。与 REQUIRES_NEW 的区别：REQUIRES_NEW 完全独立，NESTED 只是外部事务的一个保存点。

## 四、事务回滚规则

**Spring 默认回滚规则**：只有 `RuntimeException`（非受检异常）和 `Error` 时自动回滚，受检异常（`Exception` 及其子类）默认**不回滚**。

这是常见陷阱，建议统一配置：
```java
// 生产环境推荐：对所有 Exception 都回滚
@Transactional(rollbackFor = Exception.class)
public void updateUser(User user) throws BusinessException {
    userMapper.update(user);
}
```

**手动标记回滚**（不抛异常但需要回滚）：
```java
@Transactional
public void process() {
    // 发现业务异常不想抛出，但需要回滚
    TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
}
```
