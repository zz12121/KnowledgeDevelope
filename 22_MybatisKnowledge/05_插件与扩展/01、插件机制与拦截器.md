# 插件机制与拦截器

## 一、插件机制原理

MyBatis 插件机制的本质是 **JDK 动态代理 + 责任链模式**。它允许在不修改源码的情况下，对 MyBatis 四大核心组件的特定方法进行拦截和增强。

**可拦截的四大组件**：
- **Executor**：SQL 执行器，可拦截 `update`、`query`、`commit`、`rollback` 等方法。典型应用：分页插件、数据权限、读写分离
- **StatementHandler**：SQL 语句构建器，可拦截 `prepare`、`parameterize`、`query` 等方法。典型应用：**SQL 重写**（如分页 SQL 改造）、性能监控
- **ParameterHandler**：参数处理器，可拦截 `setParameters`。典型应用：参数加密、参数验证
- **ResultSetHandler**：结果集处理器，可拦截 `handleResultSets`。典型应用：结果解密、数据脱敏

## 二、自定义插件步骤

**1. 实现 Interceptor 接口**：
```java
@Intercepts({
    @Signature(type = StatementHandler.class, 
               method = "prepare",
               args = {Connection.class, Integer.class})
})
public class MyPlugin implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 在这里写增强逻辑
        // invocation.getTarget() 获取目标对象
        // invocation.getMethod() 获取被拦截的方法
        // invocation.getArgs() 获取方法参数
        return invocation.proceed();  // 继续执行原方法
    }

    @Override
    public Object plugin(Object target) {
        return Plugin.wrap(target, this);  // 固定写法，生成代理对象
    }

    @Override
    public void setProperties(Properties properties) {
        // 接收 mybatis-config.xml 中配置的属性
    }
}
```

**2. 注册插件**：
```xml
<plugins>
    <plugin interceptor="com.example.MyPlugin">
        <property name="someProperty" value="someValue"/>
    </plugin>
</plugins>
```

**源码加载过程**：
1. `XMLConfigBuilder` 解析 `<plugins>` 标签，将拦截器实例添加到 `Configuration` 的 `InterceptorChain`
2. 创建四大组件时（如 `Configuration.newExecutor()`），调用 `interceptorChain.pluginAll(target)`，遍历所有拦截器对目标对象层层包装，生成代理链
3. 调用代理方法时，`Plugin.invoke()` 检查是否命中 `@Signature` 声明的拦截点，命中则调用 `interceptor.intercept()`

## 三、SQL 性能监控插件

通过拦截 `Executor.query` 和 `Executor.update` 实现慢 SQL 告警：

```java
@Intercepts({
    @Signature(type = Executor.class, method = "query", 
               args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class}),
    @Signature(type = Executor.class, method = "update", 
               args = {MappedStatement.class, Object.class})
})
public class SqlPerformanceInterceptor implements Interceptor {
    private static final long SLOW_SQL_THRESHOLD = 1000;

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return invocation.proceed();
        } finally {
            long cost = System.currentTimeMillis() - start;
            MappedStatement ms = (MappedStatement) invocation.getArgs()[0];
            if (cost > SLOW_SQL_THRESHOLD) {
                log.warn("慢SQL: {} 耗时: {}ms", ms.getId(), cost);
                // 可上报到 SkyWalking、Prometheus 等 APM 系统
            }
        }
    }
}
```

## 四、分页插件（PageHelper）

PageHelper 是 MyBatis 最广泛使用的分页插件，拦截 `Executor.query` 实现物理分页。

**工作流程**：
1. 调用 `PageHelper.startPage(pageNum, pageSize)` 将分页参数存入 **ThreadLocal**（保证线程安全）
2. 执行 Mapper 查询时，拦截器从 ThreadLocal 取出分页参数
3. 拦截器先执行 `COUNT(*)` 查询获取总记录数，再改写原始 SQL 添加 `LIMIT offset, size`
4. 执行改写后的分页 SQL，将结果和总数封装到 `PageInfo` 对象返回
5. 清理 ThreadLocal 中的分页参数

**物理分页 vs 逻辑分页**：物理分页在数据库层执行，只返回当前页数据，网络传输和内存开销小，**生产环境必须用物理分页**。逻辑分页（RowBounds）是查出所有数据到内存后再截取，数据量大时会造成 OOM，只适合测试或极少量数据的场景。
