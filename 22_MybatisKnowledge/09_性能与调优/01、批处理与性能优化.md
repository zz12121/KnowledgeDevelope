# 批处理与性能优化

## 1. 批量插入的三种方式

### 方式一：`<foreach>` 单条 SQL（最高效）

将多条数据合并为一条 INSERT 语句，**显著减少网络往返次数**：

```xml
<insert id="batchInsert" parameterType="java.util.List">
    INSERT INTO user(name, age, email) VALUES
    <foreach collection="list" item="item" separator=",">
        (#{item.name}, #{item.age}, #{item.email})
    </foreach>
</insert>
```

**源码角度**：`DynamicContext` 解析 `<foreach>` 标签，将多个 VALUES 组拼接成完整 SQL。

**注意**：单条 SQL 过长会增加数据库解析压力，建议每批 500~1000 条，超过则分批调用。

### 方式二：`ExecutorType.BATCH`（JDBC 级别批处理）

基于 JDBC 的 `addBatch()` 机制，将多条 SQL 缓存后一次性提交：

```java
// 获取 BATCH 模式的 SqlSession（Spring 项目用 SqlSessionTemplate）
SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH);
UserMapper mapper = sqlSession.getMapper(UserMapper.class);

try {
    for (int i = 0; i < userList.size(); i++) {
        mapper.insert(userList.get(i));   // 不立即执行，加入批处理队列
        if ((i + 1) % 1000 == 0) {
            sqlSession.flushStatements(); // 每 1000 条提交一次，避免内存溢出
        }
    }
    sqlSession.commit();
} catch (Exception e) {
    sqlSession.rollback();
} finally {
    sqlSession.close();
}
```

**源码机制**：`BatchExecutor.doUpdate()` 调用 `PreparedStatement.addBatch()`，`flushStatements()` 触发 `executeBatch()`。

**性能提升关键**：在 MySQL 连接 URL 中加 `rewriteBatchedStatements=true`，MySQL JDBC 驱动会将 `executeBatch()` 重写为单条多值 INSERT，性能提升几倍甚至十几倍：

```
jdbc:mysql://localhost:3306/my_db?rewriteBatchedStatements=true
```

### 方式三：循环单条插入（不推荐）

N 次 INSERT 语句，N 次网络往返，性能最差。只在数据量极小或需要逐条处理业务逻辑时使用。

### 获取批量插入的主键

```xml
<insert id="batchInsert" useGeneratedKeys="true" keyProperty="id">
    INSERT INTO user(name, email) VALUES
    <foreach collection="list" item="item" separator=",">
        (#{item.name}, #{item.email})
    </foreach>
</insert>
```

执行后 `list` 中每个对象的 `id` 属性会被自动填充。

## 2. 三种执行器对比

MyBatis 提供三种 `Executor` 实现，`SqlSession` 创建时指定：

| 执行器 | 实现类 | 特点 | 适用场景 |
|--------|--------|------|----------|
| **SIMPLE**（默认） | `SimpleExecutor` | 每次执行为 SQL 创建新的 `PreparedStatement` | 通用场景，小数据量 |
| **REUSE** | `ReuseExecutor` | 复用相同 SQL 的 `PreparedStatement`（Map 缓存） | 相同 SQL 频繁执行，减少预编译开销 |
| **BATCH** | `BatchExecutor` | 缓存多个 `addBatch()`，延迟执行 | 大数据量批量插入/更新 |

全局配置（谨慎，影响所有操作）：
```java
config.setDefaultExecutorType(ExecutorType.BATCH);
```

## 3. 大数据量查询优化

### 流式查询（ResultHandler）

避免将大量数据一次性加载到 JVM 堆内存：

```java
sqlSession.select("com.example.mapper.UserMapper.streamingQuery",
    param,
    new ResultHandler<User>() {
        @Override
        public void handleResult(ResultContext<? extends User> context) {
            User user = context.getResultObject();
            // 逐条处理，处理完即可被 GC
            processUser(user);
        }
    });
```

**源码**：`DefaultResultSetHandler` 通过设置 `ResultSet.TYPE_FORWARD_ONLY` 和 `fetchSize` 实现流式获取，JDBC 驱动按批从数据库拉取数据而不是一次性全部加载。

### 游标查询（Cursor）

MyBatis 3.4.6+ 支持，更优雅的流式处理方式：

```java
try (Cursor<User> cursor = userMapper.scanAll()) {
    for (User user : cursor) {
        processUser(user);
    }
}
```

### 分页查询（避免深分页）

传统 `LIMIT offset, size` 在 offset 很大时性能很差（MySQL 需要扫描 offset + size 行再丢弃前 offset 行）。推荐使用**基于索引的游标分页**：

```xml
<select id="pagingQuery" resultType="User">
    SELECT * FROM user
    WHERE id > #{lastId}            <!-- 上一页最后一条记录的 ID -->
    ORDER BY id
    LIMIT #{pageSize}
</select>
```

## 4. N+1 查询问题

查询主表 N 条记录时，对每条记录触发一次关联查询，共执行 1 + N 次 SQL，是一种常见的性能陷阱。

**典型场景**：

```xml
<!-- 嵌套查询方式，会触发 N+1 -->
<resultMap id="userResultMap" type="User">
    <collection property="orders" ofType="Order"
                select="selectOrdersByUserId" column="id"/>
</resultMap>
```

查询 100 个用户 → 执行 1 次用户查询 + 100 次订单查询 = 101 次 SQL。

**解决方案**：

**方案一：JOIN 查询一次获取（推荐，数据量不大时）**

```xml
<resultMap id="userWithOrdersMap" type="User">
    <id property="id" column="user_id"/>
    <collection property="orders" ofType="Order">
        <id property="id" column="order_id"/>
        <result property="amount" column="order_amount"/>
    </collection>
</resultMap>

<select id="selectUserWithOrders" resultMap="userWithOrdersMap">
    SELECT u.id as user_id, u.name,
           o.id as order_id, o.amount as order_amount
    FROM user u
    LEFT JOIN orders o ON u.id = o.user_id
    WHERE u.id = #{id}
</select>
```

**方案二：先批量查主表 ID，再 IN 查关联表**

```sql
-- 第一步：查用户
SELECT * FROM user WHERE ...

-- 第二步：一次性批量查订单
SELECT * FROM orders WHERE user_id IN (1, 2, 3, ..., 100)
```

**方案三：延迟加载（适合关联数据不常用场景）**

```xml
<settings>
    <setting name="lazyLoadingEnabled" value="true"/>
    <setting name="aggressiveLazyLoading" value="false"/>
</settings>
```

延迟加载本质上也是 N+1，只是将查询时机推迟到属性被真正访问时，并不能减少 SQL 次数。对于列表查询，JOIN 或批量 IN 是更好的选择。

## 5. SQL 调优要点

**MyBatis 层面**：
- 使用 `<sql>/<include>` 复用公共 SQL 片段，避免 `SELECT *`
- 动态 SQL 中加 `LIMIT` 限制结果集大小，防止意外全表扫描
- 对查询频繁且数据变化少的场景，合理配置二级缓存

**MyBatis 配置优化**：

```yaml
mybatis:
  configuration:
    cache-enabled: true                    # 开启二级缓存
    lazy-loading-enabled: true             # 开启延迟加载
    aggressive-lazy-loading: false         # 按需加载（不要激进）
    map-underscore-to-camel-case: true     # 自动驼峰转换
    default-fetch-size: 100                # 流式查询默认批量大小
    default-statement-timeout: 30          # 默认语句超时（秒）
```

**连接池优化**：

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20    # 根据数据库连接数上限和应用实例数计算
      minimum-idle: 5          # 空闲连接保持数，减少频繁创建连接开销
      connection-timeout: 30000
      idle-timeout: 600000
```

**监控与诊断**：
- 开发环境：配置 `log-impl: StdOutImpl`，打印完整 SQL
- 生产环境：结合 Druid 监控或 MyBatis 性能插件，记录慢 SQL（阈值一般设 500ms）
- 定期使用 `EXPLAIN` 分析执行计划，确保关键查询使用了索引

## 6. Executor 源码深度解析

### 6.1 Executor 接口设计

`Executor` 是 MyBatis 的核心接口，`SqlSession` 接口的所有数据库操作都委托给 `Executor` 完成：

```java
public interface Executor {
  // 执行 update / insert / delete 三类 SQL
  int update(MappedStatement ms, Object parameter) throws SQLException;

  // 执行 select，返回结果列表（重载：有/无 CacheKey 版本）
  <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds,
                    ResultHandler resultHandler, CacheKey cacheKey, BoundSql boundSql) throws SQLException;
  <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds,
                    ResultHandler resultHandler) throws SQLException;

  // 游标形式返回结果（流式）
  <E> Cursor<E> queryCursor(MappedStatement ms, Object parameter, RowBounds rowBounds) throws SQLException;

  // 刷新批处理语句（BatchExecutor 专用）
  List<BatchResult> flushStatements() throws SQLException;

  // 事务相关
  void commit(boolean required) throws SQLException;
  void rollback(boolean required) throws SQLException;

  // 一级缓存相关
  CacheKey createCacheKey(MappedStatement ms, Object parameterObject,
                          RowBounds rowBounds, BoundSql boundSql);
  boolean isCached(MappedStatement ms, CacheKey key);
  void clearLocalCache();

  // 延迟加载
  void deferLoad(MappedStatement ms, MetaObject resultObject, String property,
                 CacheKey key, Class<?> targetType);

  Transaction getTransaction();
  void close(boolean forceRollback);
  boolean isClosed();
}
```

Executor 的类继承关系：

```
Executor（接口）
 ├── BaseExecutor（抽象类，模板方法模式，实现一级缓存和事务管理）
 │    ├── SimpleExecutor   ——  每次新建 PreparedStatement
 │    ├── ReuseExecutor    ——  按 SQL 缓存复用 Statement
 │    └── BatchExecutor    ——  addBatch 积累后 executeBatch
 └── CachingExecutor（装饰器，负责二级缓存，包装上面三种之一）
```

### 6.2 BaseExecutor：模板方法 + 一级缓存

#### 6.2.1 核心字段

```java
public abstract class BaseExecutor implements Executor {
  protected Transaction transaction;           // 事务对象
  protected Executor wrapper;                  // 被 CachingExecutor 包装时指向外层

  protected ConcurrentLinkedQueue<DeferredLoad> deferredLoads; // 延迟加载队列
  protected PerpetualCache localCache;                         // 一级缓存（结果对象）
  protected PerpetualCache localOutputParameterCache;          // 一级缓存（存储过程 OUT 参数）
  protected Configuration configuration;

  protected int queryStack;   // 嵌套查询层数计数
  private boolean closed;
}
```

#### 6.2.2 query() 模板方法 —— 一级缓存查询流程

```java
@Override
public <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds,
                          ResultHandler resultHandler) throws SQLException {
  // 第一步：获取绑定 SQL 和参数信息
  BoundSql boundSql = ms.getBoundSql(parameter);
  // 第二步：组装 CacheKey（MappedStatement.id + offset + limit + SQL + 实参）
  CacheKey key = createCacheKey(ms, parameter, rowBounds, boundSql);
  // 第三步：转到重载方法执行真正查询逻辑
  return query(ms, parameter, rowBounds, resultHandler, key, boundSql);
}

@Override
public <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds,
                          ResultHandler resultHandler, CacheKey key, BoundSql boundSql) {
  if (queryStack == 0 && ms.isFlushCacheRequired()) {
    clearLocalCache();              // 非嵌套且 flushCache=true → 清空一级缓存
  }
  List<E> list;
  try {
    queryStack++;
    // 从一级缓存查找
    list = resultHandler == null ? (List<E>) localCache.getObject(key) : null;
    if (list != null) {
      handleLocallyCachedOutputParameters(ms, key, parameter, boundSql); // 存储过程特殊处理
    } else {
      // 缓存未命中 → 查数据库（调用 doQuery 抽象方法，由子类实现）
      list = queryFromDatabase(ms, parameter, rowBounds, resultHandler, key, boundSql);
    }
  } finally {
    queryStack--;
  }
  if (queryStack == 0) {
    // 顶层查询完成 → 触发所有延迟加载
    for (DeferredLoad deferredLoad : deferredLoads) {
      deferredLoad.load();
    }
    deferredLoads.clear();
    // localCacheScope=STATEMENT 时每次查询后清缓存
    if (configuration.getLocalCacheScope() == LocalCacheScope.STATEMENT) {
      clearLocalCache();
    }
  }
  return list;
}
```

#### 6.2.3 CacheKey 的组成

CacheKey 唯一标识一次查询，由以下 5 个维度组成：

```java
@Override
public CacheKey createCacheKey(MappedStatement ms, Object parameterObject,
                                RowBounds rowBounds, BoundSql boundSql) {
  CacheKey cacheKey = new CacheKey();
  cacheKey.update(ms.getId());              // 1. MappedStatement 的 id（Mapper.方法名）
  cacheKey.update(rowBounds.getOffset());   // 2. 分页 offset
  cacheKey.update(rowBounds.getLimit());    // 3. 分页 limit
  cacheKey.update(boundSql.getSql());       // 4. SQL 语句（含 ? 占位符）
  // 5. 所有实参值（过滤 OUT 类型参数）
  for (ParameterMapping parameterMapping : boundSql.getParameterMappings()) {
    if (parameterMapping.getMode() != ParameterMode.OUT) {
      Object value = resolveParamValue(parameterObject, parameterMapping, boundSql);
      cacheKey.update(value);
    }
  }
  if (configuration.getEnvironment() != null) {
    cacheKey.update(configuration.getEnvironment().getId()); // 6. 环境 ID（多数据源场景）
  }
  return cacheKey;
}
```

> **重点**：CacheKey 内部通过 `hashcode + checksum + count + updateList` 四个字段组合判等，避免哈希冲突。只有 6 个维度完全相同，才视为同一查询命中缓存。

#### 6.2.4 update() —— 写操作强制清缓存

```java
@Override
public int update(MappedStatement ms, Object parameter) throws SQLException {
  // 任何写操作（insert/update/delete）都会清空一级缓存，防止脏读
  clearLocalCache();
  // 调用子类的 doUpdate() 方法
  return doUpdate(ms, parameter);
}
```

#### 6.2.5 事务提交/回滚 —— flushStatements 时机

```java
@Override
public void commit(boolean required) throws SQLException {
  clearLocalCache();          // 清一级缓存
  flushStatements();          // 执行 BatchExecutor 中积压的 SQL
  if (required) {
    transaction.commit();     // 提交事务
  }
}

@Override
public void rollback(boolean required) throws SQLException {
  if (!closed) {
    clearLocalCache();
    flushStatements(true);    // isRollback=true，BatchExecutor 直接丢弃积压的 SQL
    if (required) {
      transaction.rollback();
    }
  }
}
```

### 6.3 SimpleExecutor：最简实现

`SimpleExecutor` 是 MyBatis 默认执行器，每次执行 SQL 都创建新的 `PreparedStatement`，执行完立即关闭：

```java
@Override
public <E> List<E> doQuery(MappedStatement ms, Object parameter, RowBounds rowBounds,
                            ResultHandler resultHandler, BoundSql boundSql) throws SQLException {
  Statement stmt = null;
  try {
    Configuration configuration = ms.getConfiguration();
    // 1. 创建 StatementHandler（PreparedStatementHandler）
    StatementHandler handler = configuration.newStatementHandler(
        wrapper, ms, parameter, rowBounds, resultHandler, boundSql);
    // 2. prepare（创建 PreparedStatement）+ parameterize（绑定 ? 参数）
    stmt = prepareStatement(handler, ms.getStatementLog());
    // 3. 执行 SQL + ResultSetHandler 映射结果
    return handler.<E>query(stmt, resultHandler);
  } finally {
    closeStatement(stmt);  // 每次执行完立即关闭，不复用
  }
}

private Statement prepareStatement(StatementHandler handler, Log statementLog) throws SQLException {
  Connection connection = getConnection(statementLog);
  Statement stmt = handler.prepare(connection, transaction.getTimeout());
  handler.parameterize(stmt);   // 处理 ? 占位符
  return stmt;
}

@Override
public List<BatchResult> doFlushStatements(boolean isRollback) {
  return Collections.emptyList();  // SimpleExecutor 不支持批处理，直接返回空
}
```

### 6.4 ReuseExecutor：Statement 复用

`ReuseExecutor` 通过 `statementMap`（SQL → Statement）缓存已编译的 `PreparedStatement`，对相同 SQL 不重复调用 JDBC 的 `prepare()`，减少数据库预编译开销：

```java
// 核心：以 SQL 文本为 key 缓存 Statement 对象
private final Map<String, Statement> statementMap = new HashMap<>();

private Statement prepareStatement(StatementHandler handler, Log statementLog) throws SQLException {
  Statement stmt;
  String sql = handler.getBoundSql().getSql();
  if (hasStatementFor(sql)) {
    // 命中缓存：复用 Statement，只需重新绑定参数
    stmt = getStatement(sql);
    applyTransactionTimeout(stmt);
  } else {
    // 未命中：创建新 Statement，并放入缓存
    Connection connection = getConnection(statementLog);
    stmt = handler.prepare(connection, transaction.getTimeout());
    putStatement(sql, stmt);
  }
  handler.parameterize(stmt);   // 每次都要重新绑定实参
  return stmt;
}

// 事务提交/回滚/连接关闭时，必须关闭所有缓存的 Statement
@Override
public List<BatchResult> doFlushStatements(boolean isRollback) throws SQLException {
  for (Statement stmt : statementMap.values()) {
    closeStatement(stmt);   // 关闭所有缓存 Statement
  }
  statementMap.clear();
  return Collections.emptyList();
}
```

> **注意**：`statementMap` 的 key 是 SQL 字符串（含 `?`），value 是 Statement 对象。事务结束时必须关闭，否则会造成连接泄漏。`doFlushStatements()` 被 `commit()` / `rollback()` / `close()` 调用，确保及时清理。

### 6.5 BatchExecutor：JDBC 批处理

`BatchExecutor` 实现了真正的 JDBC 批处理，核心思路：将多条 DML 语句通过 `addBatch()` 积累在 `PreparedStatement` 中，等到 `flushStatements()` 时一次性调用 `executeBatch()` 发送给数据库：

```java
public class BatchExecutor extends BaseExecutor {
  // 缓存多个 Statement，每个 Statement 对应一种 SQL 结构
  private final List<Statement> statementList = new ArrayList<>();
  // 每个 Statement 对应的批处理结果（记录影响行数）
  private final List<BatchResult> batchResultList = new ArrayList<>();
  // 上一次处理的 SQL 文本（用于判断是否可以复用 Statement）
  private String currentSql;
  // 上一次处理的 MappedStatement
  private MappedStatement currentStatement;

  @Override
  public int doUpdate(MappedStatement ms, Object parameterObject) throws SQLException {
    final StatementHandler handler = configuration.newStatementHandler(
        this, ms, parameterObject, RowBounds.DEFAULT, null, null);
    final String sql = handler.getBoundSql().getSql();
    final Statement stmt;
    // 关键判断：当前 SQL 结构 + MappedStatement 与上一条相同 → 复用 Statement
    if (sql.equals(currentSql) && ms.equals(currentStatement)) {
      // 复用 statementList 中最后一个 Statement
      int last = statementList.size() - 1;
      stmt = statementList.get(last);
      applyTransactionTimeout(stmt);
      handler.parameterize(stmt);   // 绑定本次参数
      // 在 BatchResult 中追加本次参数对象
      batchResultList.get(last).addParameterObject(parameterObject);
    } else {
      // 不同 SQL → 创建新 Statement，并加入缓存
      Connection connection = getConnection(ms.getStatementLog());
      stmt = handler.prepare(connection, transaction.getTimeout());
      handler.parameterize(stmt);
      currentSql = sql;
      currentStatement = ms;
      statementList.add(stmt);
      batchResultList.add(new BatchResult(ms, sql, parameterObject));
    }
    // 核心：调用 JDBC 的 addBatch()，不立即执行，积累到 Statement 内部
    handler.batch(stmt);
    return BATCH_UPDATE_RETURN_VALUE;  // 返回魔法值，表示批处理模式
  }

  @Override
  public List<BatchResult> doFlushStatements(boolean isRollback) throws SQLException {
    try {
      List<BatchResult> results = new ArrayList<>();
      if (isRollback) {
        return Collections.emptyList();   // 回滚时直接丢弃
      }
      for (int i = 0; i < statementList.size(); i++) {
        Statement stmt = statementList.get(i);
        applyTransactionTimeout(stmt);
        BatchResult batchResult = batchResultList.get(i);
        // 核心：调用 JDBC 的 executeBatch()，批量发送 SQL 到数据库
        batchResult.setUpdateCounts(stmt.executeBatch());
        // 处理自动生成的主键（Jdbc3KeyGenerator）
        KeyGenerator keyGenerator = batchResult.getMappedStatement().getKeyGenerator();
        if (Jdbc3KeyGenerator.class.equals(keyGenerator.getClass())) {
          ((Jdbc3KeyGenerator) keyGenerator).processBatch(
              batchResult.getMappedStatement(), stmt, batchResult.getParameterObjects());
        }
        closeStatement(stmt);
        results.add(batchResult);
      }
      return results;
    } finally {
      // 清理所有状态
      currentSql = null;
      statementList.clear();
      batchResultList.clear();
    }
  }
}
```

**BatchExecutor 的 SQL 合并策略**：

```
addBatch 顺序：
  UPDATE user SET age=? WHERE id=?  [id=1, age=20]  → Statement-A.addBatch()
  UPDATE user SET age=? WHERE id=?  [id=2, age=25]  → Statement-A.addBatch()（相同 SQL，复用）
  INSERT INTO user(name) VALUES(?)  [name='张三']    → Statement-B.addBatch()（新 SQL，新建）
  UPDATE user SET age=? WHERE id=?  [id=3, age=30]  → Statement-C.addBatch()（SQL 相同但中间插入了其他，新建）

executeBatch 时：
  Statement-A.executeBatch()  → 影响 2 行
  Statement-B.executeBatch()  → 影响 1 行
  Statement-C.executeBatch()  → 影响 1 行
```

> **坑点**：连续相同 SQL 才会被合并到同一个 Statement。如果批量 update 和 insert 交替出现，每次切换都会创建新 Statement，反而增加开销。推荐：同类型 SQL 连续执行，再切换。

### 6.6 Executor 选择指南

| 场景 | 推荐 Executor | 原因 |
|------|--------------|------|
| 通用 CRUD，数据量小 | `SIMPLE`（默认） | 简单可靠，无额外复杂度 |
| 同一事务内相同 SQL 高频执行 | `REUSE` | 复用 PreparedStatement，减少编译开销 |
| 大批量插入/更新（千条以上） | `BATCH` | addBatch + executeBatch 减少网络往返 |
| 有二级缓存需求 | `CachingExecutor` 包装 | 自动处理二级缓存读写 |

**Spring 中局部使用 BATCH 模式**（推荐做法，避免全局污染）：

```java
@Service
public class UserBatchService {
  @Autowired
  private SqlSessionFactory sqlSessionFactory;

  @Transactional
  public void batchInsertUsers(List<User> users) {
    try (SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH)) {
      UserMapper mapper = sqlSession.getMapper(UserMapper.class);
      for (int i = 0; i < users.size(); i++) {
        mapper.insert(users.get(i));
        if ((i + 1) % 1000 == 0) {
          sqlSession.flushStatements();  // 每 1000 条刷一次，控制内存
        }
      }
      sqlSession.flushStatements();     // 最后一批
      // 注意：这里不能调用 sqlSession.commit()，事务由 @Transactional 管理
    }
  }
}
```
