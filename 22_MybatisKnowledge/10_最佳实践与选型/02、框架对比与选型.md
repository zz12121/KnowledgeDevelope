# 框架对比与选型

## 1. MyBatis vs JPA（Hibernate）

### 设计哲学

**MyBatis** 是**半自动化 ORM**，SQL 由开发者掌控，框架只做参数映射和结果集转换。核心理念：开发者最了解业务，应该直接控制 SQL。

**JPA（Hibernate）** 是**全自动 ORM**，通过对象与数据库表的映射（`@Entity`、`@Column`），开发者操作对象，框架自动生成 SQL。核心理念：屏蔽 SQL 细节，让开发者专注领域模型。

### 执行机制对比

- **MyBatis**：`SqlSessionTemplate` → `MappedStatement`（描述 SQL）→ `Executor`（执行 SQL + 处理缓存）。开发者编写的 SQL 直接执行，调用链清晰。
- **JPA**：`EntityManager` → `SessionImpl`（管理实体状态）→ JPQL 解析为 AST → 转换为数据库方言 SQL。中间经过多层转换，调试复杂 SQL 时不直观。

### 详细对比

| 对比维度 | MyBatis | JPA（Hibernate） |
|----------|---------|-----------------|
| SQL 控制 | 完全掌控，手动编写与优化 | 有限控制，自动生成 |
| 开发效率 | 中等，简单 CRUD 也需写 SQL | 高，零代码 CRUD |
| 性能调优 | 直接精准，每条 SQL 可优化 | 间接，需理解生成 SQL 和缓存机制 |
| 学习曲线 | 平缓，会 SQL 即可上手 | 陡峭，需掌握实体状态、懒加载、一级/二级缓存 |
| 数据库兼容性 | 需手动处理方言（如分页） | 优秀，方言自动适配 |
| 复杂查询 | 天然擅长，可写任意 SQL | 复杂场景需 Native Query，繁琐 |
| 对象关联 | 手动处理（association/collection） | 自动管理（@OneToMany、@ManyToMany） |

### 选择建议

- **选 MyBatis**：
  - 业务有大量复杂 SQL（多表关联、聚合、窗口函数、报表）
  - 对 SQL 性能有极致要求（高并发交易系统）
  - 遗留系统改造，需要与现有存储过程集成
  - 团队 SQL 能力强，希望直接控制数据库交互

- **选 JPA**：
  - 快速原型开发，CRUD 为主
  - DDD（领域驱动设计）项目，领域对象建模为核心
  - 需要多数据库兼容

- **混合使用**：核心复杂业务用 MyBatis，简单模块用 JPA，但需注意两者事务管理的一致性（使用同一个 `DataSourceTransactionManager`）。

## 2. MyBatis vs Spring Data JDBC

**Spring Data JDBC** 是 Spring Data 家族成员，本质是 `JdbcTemplate` 的封装，提供 Repository 抽象，**故意不支持懒加载、脏检查等复杂 ORM 特性**，遵循"简单至上"原则。

| 特性 | MyBatis | Spring Data JDBC |
|------|---------|-----------------|
| 复杂结果映射 | 支持嵌套 resultMap、关联查询 | 简单一对一映射 |
| 动态 SQL | 完整的动态 SQL 标签体系 | 基本 CRUD，复杂查询用 `@Query` |
| 缓存 | 一级 + 二级缓存 | 无内置缓存 |
| 懒加载 | 支持 | 不支持（故意的） |
| 启动速度 | 中等 | 快（无需解析复杂映射） |
| 学习成本 | 中等 | 低 |

**选择场景**：
- **MyBatis**：企业级应用，有复杂映射、动态 SQL 需求
- **Spring Data JDBC**：微服务架构中简单的数据访问层，追求极简和快速启动

## 3. MyBatis vs MyBatis-Plus

两者不是竞争关系，MyBatis-Plus 是 MyBatis 的增强套件，详见 [[08_MyBatis-Plus/01、MyBatis-Plus核心功能|MyBatis-Plus核心功能]]。

**何时引入 MyBatis-Plus**：
- 项目有大量单表 CRUD，希望减少重复代码
- 需要快速开发，利用代码生成器
- 需要逻辑删除、乐观锁、自动填充等通用功能

**何时保持原生 MyBatis**：
- 复杂多表关联查询为主
- 对框架依赖有严格限制
- 已有成熟的 BaseMapper 封装

## 4. MyBatis 优缺点总结

### 核心优势

1. **SQL 灵活性**：复杂查询、存储过程、数据库特定函数，完全掌控
2. **性能优化空间大**：接近 JDBC 性能，每条 SQL 可单独调优，支持批量操作
3. **调试友好**：SQL 日志清晰，问题定位直接
4. **学习成本低**：会 SQL 即可上手，与 Spring 生态无缝集成

### 主要局限性

1. **开发效率相对低**：简单 CRUD 也需要编写 SQL 和映射，代码量大
2. **数据库移植性差**：SQL 中使用数据库特定语法（如分页、特殊函数）降低可移植性
3. **对象关联处理弱**：复杂多层对象关系需手动维护，不如 JPA 自动管理方便
4. **XML 文件冗长**：大型项目中 XML 映射文件可能变得难以维护

### 适用场景总结

| 场景 | 推荐 | 理由 |
|------|------|------|
| 金融/报表系统 | ✅ MyBatis | 复杂 SQL，性能敏感 |
| 高并发交易系统 | ✅ MyBatis | 精准 SQL 调优 |
| 快速原型/内部工具 | ✅ JPA | 开发效率优先 |
| 微服务简单 CRUD | ✅ Spring Data JDBC | 轻量，快速启动 |
| 标准企业 CRUD | ✅ MyBatis-Plus | 兼顾效率和 SQL 控制 |

## 附录：MyBatis 工具三剑客

### 一、MyBatis-Generator（代码生成器）

> MyBatis-generator 是一个能快速生成 XML、DAO 接口、实体类、注解类的代码生成器。

**官网地址**：https://mybatis.org/generator/index.html

**Maven 配置**：
```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.mybatis.generator</groupId>
            <artifactId>mybatis-generator-maven-plugin</artifactId>
            <version>1.4.2</version>
            <configuration>
                <overwrite>true</overwrite>
            </configuration>
        </plugin>
    </plugins>
</build>
```

**配置文件 generatorConfig.xml**：
```xml
<!DOCTYPE generatorConfiguration PUBLIC "-//mybatis.org//DTD MyBatis Generator Configuration 1.0//EN"
 "http://mybatis.org/dtd/mybatis-generator-config_1_0.dtd">
<generatorConfiguration>
  <context id="DB2Tables" targetRuntime="MyBatis3">
    <jdbcConnection driverClass="com.mysql.cj.jdbc.Driver" 
                    connectionURL="jdbc:mysql://ip:port/db" 
                    userId="root" password="root"/>
    <javaModelGenerator targetPackage="com.muse.po" targetProject="src/main/java"/>
    <sqlMapGenerator targetPackage="mapper" targetProject="src/main/resources"/>
    <javaClientGenerator type="XMLMAPPER" targetPackage="com.muse.mapper" targetProject="src/main/java"/>
    <table tableName="tb_user" domainObjectName="UserPO" mapperName="UserMapper"/>
  </context>
</generatorConfiguration>
```

### 二、PageHelper（分页插件）

> 是一个开源的分页插件，原理是通过 Spring 的 AOP 来实现的，在执行 SQL 的时候把分页语句再执行一次。

**官网地址**：https://mybatis.io

**Spring Boot 引入依赖**：
```xml
<dependency>
    <groupId>com.github.pagehelper</groupId>
    <artifactId>pagehelper-spring-boot-starter</artifactId>
    <version>2.1.0</version>
</dependency>
```

**普通 Maven 项目**：
```xml
<dependency>
    <groupId>com.github.pagehelper</groupId>
    <artifactId>pagehelper</artifactId>
    <version>6.1.0</version>
</dependency>
```

**使用方式**：
```java
// 只需要调用 PageHelper 的 startPage 方法，传入分页参数即可
PageHelper.startPage(pageNum, pageSize);
List<User> users = userMapper.selectAll();
// 返回的结果自动包含分页信息
PageInfo<User> pageInfo = new PageInfo<>(users);
```

> 注意：配置分页插件后启动时会输出 banner，如果输出多次 banner 说明配置了多次分页插件，需要排查。

### 三、MyBatis-Plus（增强工具）

> MyBatis-Plus（简称 MP）是一个 MyBatis 的增强工具，在 MyBatis 的基础上只做增强不做改变，为简化开发、提高效率而生。

**官网地址**：https://baomidou.com

**引入依赖**：
```xml
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-spring-boot3-starter</artifactId>
    <version>3.5.5</version>
</dependency>
```

**核心使用**：
```java
// Mapper 继承 BaseMapper
public interface UserMapper extends BaseMapper<UserPO> {
}

// 查询示例
List<UserPO> users = userMapper.selectList(null); // 查询所有

// 条件查询
LambdaQueryWrapper<UserPO> queryWrapper = new LambdaQueryWrapper<>();
queryWrapper.eq(UserPO::getName, "张三");
UserPO user = userMapper.selectOne(queryWrapper);

// 插入（自动回填主键）
userMapper.insert(user);

// 更新
UpdateWrapper<UserPO> updateWrapper = new UpdateWrapper<>();
updateWrapper.eq("id", 1).set("name", "新名字");
userMapper.update(null, updateWrapper);

// 删除
QueryWrapper<UserPO> queryWrapper = new QueryWrapper<>();
queryWrapper.eq("name", "张三");
userMapper.delete(queryWrapper);
```
