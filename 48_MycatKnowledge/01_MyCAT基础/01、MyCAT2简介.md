# MyCAT2 简介

## 1.1 分布式数据库特点

**分布式数据库**是指数据在物理上分布而在逻辑上集中管理的数据库系统。

### 核心特点

| 特点 | 说明 |
|------|------|
| **透明性** | 用户不必关心数据的逻辑分区和物理位置分布的细节，也不必关心重复副本的一致性问题 |
| **数据冗余性** | 通过冗余实现系统的可靠性、可用性，改善性能；热点数据就近分布减少网络通信消耗 |
| **易于扩展性** | 可通过水平扩展提高整体性能，也能通过垂直扩展提高性能，扩展不需要修改程序 |
| **自治性** | 各节点上的数据由本地DBMS管理，具有自治处理能力 |

> MyCAT 作为中间件，是分布式数据库与应用程序之间的桥梁。

### MyCAT2 原理

MyCAT 作为中间件，最主要的动作是：
1. **拦截 SQL 语句** - 接收客户端发送的 SQL
2. **解析分析** - 对 SQL 进行语法解析、分析路由
3. **路由到真实数据库** - 根据配置将 SQL 转发到实际数据库
4. **结果归并** - 将多个数据库的结果合并返回给客户端

> MyCAT2 可以支持处理**单表单月30亿级别**的数据量。

## 1.3 为什么需要读写分离

### 性能提升
在高并发的情况下，数据库的读操作往往远多于写操作。通过读写分离，可以将读请求分散到多个从数据库上，只将写请求发送到主数据库，这样可以显著减轻主数据库的压力，提升整个数据库系统的处理能力。

### 可扩展性
随着业务量的增加，数据库的负载会逐渐增大。读写分离允许动态地添加更多的从数据库来应对读负载的增加，从而实现数据库的水平扩展。

### 高可用性
在 MyCAT 的读写分离模式下，如果主数据库发生故障，可以迅速切换到从数据库进行读操作，保证业务的连续性。同时，也可以通过配置多个从数据库来提高系统的容错能力。

### 资源优化
读写分离可以根据数据库的负载情况，优化硬件资源的使用。例如，读操作通常不需要太高的CPU性能，但需要更多的I/O资源和网络带宽；可以为读服务器配置更优的I/O设备和更宽的带宽，而为写服务器配置更强的CPU。

## 1.4 为什么需要分库分表

当数据量达到千万级别时，单库单表的性能会显著下降。分库分表的主要目的：

1. **突破单库容量限制** - 单个数据库的存储空间和连接数有限
2. **提高查询性能** - 大表查询变慢，分表后减少数据量
3. **提高写入性能** - 并行写入多个分片
4. **解决事务问题** - 大事务成本高，分片后小事务更快
