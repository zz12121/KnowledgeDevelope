# 主从复制延迟问题

## 1. 如何查看同步延迟状态？

### 1.1 查看 Seconds_Behind_Master

```sql
SHOW REPLICA STATUS\G;
```

查看 `Seconds_Behind_Master` 字段：
- `0`：表示无延迟
- `NULL`：表示 IO 线程或 SQL 线程没有运行
- `大于0`：表示从库比主库延迟的秒数

### 1.2 查看 Relay_Log 位置

```sql
-- 查看主库 Binlog 位置
SHOW MASTER STATUS;
-- File: mysql-bin.000001
-- Position: 1234

-- 查看从库 Relay Log 位置
SHOW REPLICA STATUS\G
-- Relay_Log_File: relay-bin.000001
-- Relay_Log_Pos: 1234
-- Read_Master_Log_Pos: 1234
```

计算延迟：`Read_Master_Log_Pos - Relay_Log_Pos`

### 1.3 使用延迟判断

```sql
-- 使用 mysqldump 备份时记录 GTID
-- 在从库检查
SELECT 
    SUBSTRING_INDEX(LAST_EXECUTED_EVENT, ':', -1) - 
    SUBSTRING_INDEX(LAST_APPLIED_EVENT, ':', -1) AS delay;
```

## 2. 主从复制延迟原因

| 原因 | 说明 |
|------|------|
| **网络延迟** | 主从之间的网络传输延迟 |
| **从库负载** | 从库上执行大量查询或备份任务 |
| **大事务** | 主库上的大事务会导致 Binlog 传输延迟 |
| **单线程复制** | 旧版本 MySQL 从库 SQL 线程是单线程 |
| **Binlog 格式** | Statement 格式可能比 Row 格式慢 |
| **硬件差异** | 从库硬件性能低于主库 |

## 3. 解决方案

### 3.1 架构方面

1. **采用链式主从架构**
   ```
   Master → Slave1 → Slave2
   ```
   读写分离的压力分摊到多个从库

2. **使用 MySQL 5.6+ 的并行复制**
   - 5.6 版本支持基于库的并行复制
   - 5.7 版本支持基于 Group Commit 的并行复制
   - 8.0 版本支持基于 Write Set 的并行复制

3. **读写分离**
   - 写操作走主库
   - 读操作走从库

### 3.2 配置方面

1. **调整 sync_binlog 参数**
   ```ini
   sync_binlog = 1  # 每次事务提交都同步到磁盘
   ```

2. **使用 GTID 模式**
   ```ini
   gtid_mode = ON
   enforce_gtid_consistency = ON
   ```

3. **从库配置**
   ```ini
   read_only = ON
   relay_log_recovery = ON
   ```

4. **优化 Binlog 格式**
   ```ini
   binlog_format = ROW  # 使用行格式，更安全但日志量大
   ```

## 4. 根本解决方案

### 4.1 MySQL 5.6 并行复制策略

**基于库的并行复制**：
- 原理：不同库的数据可以并行复制
- 限制：单库情况下无法并行

```ini
slave_parallel_workers = 4
slave_parallel_type = LOGICAL_CLOCK
```

### 4.2 MySQL 5.7 并行复制策略

**基于 Group Commit 的并行复制**：
- 原理：同一组提交的事务可以并行执行
- 条件：`binary log group commit`

配置：
```ini
slave_parallel_workers = 8
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_db_threads = 4
```

### 4.3 MySQL 8.0 并行复制

**基于 Write Set 的并行复制**：
- 原理：基于事务的 Write Set 检测冲突
- 优点：即使单库也可以并行

```ini
slave_parallel_workers = 16
slave_parallel_type = LOGICAL_CLOCK
transaction_write_set_extraction = XXHASH64
binlog_transaction_dependency_tracking = WRITESET
```

## 5. 最佳实践

1. **监控延迟**
   - 部署监控脚本定期检查 `Seconds_Behind_Master`
   - 设置告警阈值

2. **避免大事务**
   - 批量操作分批执行
   - 使用延迟删除/更新

3. **硬件优化**
   - 从库使用 SSD
   - 从库配置更高内存

4. **业务优化**
   - 读写分离 + 延迟感知
   - 关键读操作走主库
