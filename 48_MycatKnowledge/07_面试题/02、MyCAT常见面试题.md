# MyCAT 常见面试题

## 1. MyCAT 基础

### Q1：什么是 MyCAT？

**答**：MyCAT 是一个开源的分布式数据库中间件，基于阿里开源的 Cobar 产品而开发，MyCAT 是 Cobar 的替代品。MyCAT 支持分布式 SQL 查询，兼容 MySQL 通信协议，可以理解为 MySQL 的集群或分布式解决方案。

### Q2：MyCAT 和 ShardingSphere 的区别？

| 对比项 | MyCAT | ShardingSphere |
|--------|-------|----------------|
| 架构 | 独立中间件 | 客户端/代理模式 |
| 部署 | 需要额外部署 | 可作为 Sidecar |
| 配置 | XML 配置 | Java/配置文件 |
| 社区 | 相对较小 | Apache 顶级项目 |
| 更新 | 较慢 | 活跃 |

### Q3：MyCAT 支持哪些功能？

**答**：
- 读写分离
- 分库分表
- 数据库集群
- 数据迁移
- 分布式事务（XA）
- SQL 拦截
- 高可用

## 2. 分库分表

### Q4：为什么要分库分表？

**答**：
1. 单库数据量过大会导致查询变慢
2. 单库连接数有限，无法支撑高并发
3. 单库写入性能瓶颈
4. 大事务成本高

### Q5：分片键如何选择？

**答**：选择分片键的原则：
1. 尽量选择查询频率最高的字段
2. 避免跨分片查询
3. 避免数据倾斜
4. 常见选择：用户ID、订单ID、时间

### Q6：跨分片查询如何处理？

**答**：
1. **广播表**：每个分片都存储全量数据
2. **ER 表**：子表的分片键与父表一致
3. **多次查询**：先查询分片列表，再分别查询
4. **冗余存储**：在多个分片中冗余数据

## 3. 事务问题

### Q7：MyCAT 如何处理分布式事务？

**答**：MyCAT 支持两种事务模式：
1. **本地事务（proxy）**：默认模式，跨库事务在 commit 阶段失败可能导致不一致，但兼容性最好
2. **XA 事务**：标准的两阶段提交，需要后端数据库支持 XA

```sql
-- 设置事务类型
SET transaction_policy = 'xa';  -- XA 事务
SET transaction_policy = 'proxy'; -- 本地事务
```

### Q8：分布式事务有哪些问题？

**答**：
1. **数据一致性**：分布式环境下的数据同步问题
2. **事务隔离级别**：各节点可能隔离级别不同
3. **性能开销**：两阶段提交需要多次网络通信

## 4. 性能优化

### Q9：如何优化 MyCAT 性能？

**答**：
1. **合理选择分片键**：避免跨分片查询
2. **使用连接池**：减少连接创建开销
3. **读写分离**：分摊查询压力
4. **减少事务范围**：尽量在单个分片内完成
5. **批量操作**：使用批量 SQL 减少网络开销

### Q10：MyCAT 分页查询优化？

**答**：
1. **避免深度分页**：使用游标分页或延迟关联
2. **利用索引**：确保分片键有索引
3. **结果缓存**：对热点数据缓存

```sql
-- 低效
SELECT * FROM table LIMIT 1000000, 10;

-- 高效（基于ID）
SELECT * FROM table WHERE id > 1000000 LIMIT 10;
```

## 5. 高可用

### Q11：MyCAT 如何实现高可用？

**答**：
1. **配置集群**：使用 MyCAT 集群
2. **主从切换**：配置心跳检测，自动切换
3. **负载均衡**：多个 MyCAT 节点
4. **后端高可用**：使用双主或 MGR 集群

### Q12：MyCAT 心跳检测机制？

**答**：
- `heartbeatTimeout`：心跳超时时间
- `maxRetry`：最大重试次数
- `minSwitchTimeInterval`：最小切换间隔
- `slaveThreshold`：从节点延迟阈值

## 6. 实战问题

### Q13：分库分表后唯一ID如何生成？

**答**：
1. **UUID**：简单但无序
2. **数据库自增**：需要独立的 ID 生成服务
3. **雪花算法**：推荐，性能好且有序
4. **Redis INCR**：需要 Redis 支持

### Q14：分库分表后如何JOIN？

**答**：
1. **同分片键 JOIN**：直接执行
2. **ER 表**：自动在同分片进行
3. **广播表**：利用冗余数据
4. **多次查询**：程序内合并结果
5. **全局视图**：借助外部计算引擎

### Q15：如何选择分片数量？

**答**：
1. 根据数据量预估
2. 一般选择 2 的幂次方（便于位运算）
3. 考虑扩容，预留余量
4. 避免过多分片（增加管理复杂度）

## 7. MyCAT2 新特性

### Q16：MyCAT2 相比 MyCAT1 有哪些改进？

**答**：
1. **性能提升**：使用 Vertx 异步框架
2. **配置简化**：支持 SQL 注释动态配置
3. **SQL 支持**：更好的 SQL 兼容性
4. **事务增强**：更好的 XA 支持

### Q17：MyCAT2 为什么使用 Vertx？

**答**：
1. 高性能非阻塞 IO
2. 支撑高并发连接
3. 简化线程模型
4. 更好的资源利用
