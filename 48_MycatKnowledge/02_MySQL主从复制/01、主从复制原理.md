# MySQL 主从复制原理

## 1. MySQL 主从复制概念

**主从复制**是指将主数据库（Master）中的 DDL 和 DML 操作通过二进制日志传输到从数据库（Slave）上，然后将这些日志重新执行（重做），从而使得从数据库的数据与主数据库保持一致。

MySQL 支持**单向、异步复制**，复制过程中一个服务器充当主服务器，而一个或多个其它服务器充当从服务器。

### 主从复制的作用

1. **高可用性** - 当主数据库出现问题时，可以切换到从数据库
2. **读写分离** - 可以进行数据库层面的读写分离，实现负载均衡
3. **数据备份** - 可以在从数据库上进行实时数据备份

## 2. 主从复制原理

### 复制原理图

```
┌─────────────────┐                         ┌─────────────────┐
│    Master       │                         │    Slave        │
│                 │                         │                 │
│ ┌─────────────┐ │   Binlog (日志传输)    │ ┌─────────────┐ │
│ │ Binlog      │ │ ─────────────────────► │ │ Relay Log   │ │
│ │ (二进制日志) │ │                         │ │ (中继日志)   │ │
│ └─────────────┘ │                         │ └─────────────┘ │
│       │         │                         │       │         │
│       │         │                         │       ▼         │
│       ▼         │                         │ ┌─────────────┐ │
│  DDL/DML 操作   │                         │ │ 应用 SQL    │ │
│  记录到 Binlog │                         │ └─────────────┘ │
└─────────────────┘                         └─────────────────┘
```

### 复制过程

1. **主服务器**上的数据变更操作（如 INSERT、UPDATE、DELETE 等）会被记录到**二进制日志（Binary Log）**中

2. **从服务器**连接到主服务器，并读取主服务器的二进制日志，将其写入到自己的**中继日志（Relay Log）**中

3. 从服务器上的 **I/O 线程**将二进制日志写入中继日志后，**SQL 线程**会读取中继日志并将日志中的操作应用到从服务器的数据库中

### 核心线程

| 线程 | 说明 |
|------|------|
| **Binlog Dump 线程** | 主服务器上负责发送 Binlog 给从服务器 |
| **I/O 线程** | 从服务器上负责读取主服务器的 Binlog 并写入 Relay Log |
| **SQL 线程** | 从服务器上负责读取 Relay Log 并执行 SQL |

## 3. 主从复制配置步骤

### 3.1 主服务器配置

1. 修改 `my.cnf` 配置文件：
```ini
[mysqld]
server-id=1
log-bin=mysql-bin
binlog-do-db=需要复制的数据库
```

2. 创建复制用户：
```sql
CREATE USER 'replication_user'@'%' IDENTIFIED WITH mysql_native_password BY 'root';
GRANT REPLICATION SLAVE ON *.* TO 'replication_user'@'%';
FLUSH PRIVILEGES;
```

3. 查看主服务器状态：
```sql
SHOW MASTER STATUS;
```

### 3.2 从服务器配置

1. 修改 `my.cnf` 配置文件：
```ini
[mysqld]
server-id=2
relay-log=relay-bin
relay-log-index=relay-bin.index
```

2. 配置主从复制：
```sql
CHANGE REPLICATION SOURCE TO 
    SOURCE_HOST='主服务器IP',
    SOURCE_USER='replication_user',
    SOURCE_PASSWORD='root',
    SOURCE_LOG_FILE='binlog.000001',
    SOURCE_LOG_POS=123;
```

3. 启动复制：
```sql
START REPLICA;
```

4. 查看从服务器状态：
```sql
SHOW REPLICA STATUS\G
```

## 4. 主从复制延迟问题

### 4.1 查看同步延迟状态

```sql
-- 方法1：查看 Seconds_Behind_Master
SHOW REPLICA STATUS\G
-- Seconds_Behind_Master 表示从库比主库延迟的秒数

-- 方法2：查看Relay_Log位置
-- Relay_Log_Pos 与 Read_Master_Log_Pos 的差值
```

### 4.2 延迟原因

| 原因 | 说明 |
|------|------|
| **网络延迟** | 主从之间的网络传输延迟 |
| **从库负载** | 从库上执行大量查询或备份任务 |
| **大事务** | 主库上的大事务会导致 Binlog 传输延迟 |
| **单线程复制** | 旧版本 MySQL 从库 SQL 线程是单线程 |

### 4.3 解决方案

**架构方面**：
- 采用链式主从架构（Master → Slave1 → Slave2）
- 使用 MySQL 5.6+ 的并行复制

**配置方面**：
- 调整 `sync_binlog` 参数
- 使用 `gtid` 模式复制
- 从库开启 `read_only`

**根本解决**：
- MySQL 5.6 版本的并行复制策略
- MySQL 5.7 的并行复制策略（基于 Group Commit）
