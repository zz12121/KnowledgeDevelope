# Spring 配置与注解体系

## 1 三种配置方式

Spring 配置方式随版本演进：

**XML 配置**（传统）：通过 XML 文件定义 Bean 和依赖，`BeanDefinitionReader` 解析后注册 BeanDefinition：

```xml
<beans>
    <bean id="userService" class="com.example.UserService">
        <property name="userDao" ref="userDao"/>
    </bean>
</beans>
```

适合遗留系统，或需要在不重新编译的情况下修改配置。

**注解配置**（Spring 2.5+）：`@Component`/`@Service`/`@Autowired` 等注解自动扫描注册，代码即配置：

```java
@Service
public class UserService {
    @Autowired
    private UserDao userDao;
}
```

**Java 配置类**（Spring 3.0+）：`@Configuration` + `@Bean` 完全替代 XML，类型安全，IDE 支持重构：

```java
@Configuration
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService(userDao());
    }
}
```

**混合配置**：实际项目中常组合使用，通过 `@ImportResource` 引入 XML，或通过 `<context:component-scan>` 扫描注解组件。

**选择原则**：现代项目以注解 + Java 配置为主，XML 只用于遗留集成场景。

## 2 @Configuration 的 CGLIB 增强

`@Configuration` 标记的类会被 **CGLIB 动态代理增强**，这是它与 `@Component` 的关键区别：

```java
@Configuration
public class AppConfig {
    @Bean
    public UserService userService() {
        return new UserService(userDao());  // 多次调用 userDao() 返回同一个实例
    }

    @Bean
    public UserDao userDao() {
        return new UserDaoImpl();  // 实际只被创建一次
    }
}
```

没有 CGLIB 增强时，`userDao()` 每次调用都会 `new UserDaoImpl()`；有了 CGLIB 增强，Spring 拦截这些方法调用，确保 `@Bean` 方法始终返回容器中同一个 Bean 实例，维护 singleton 语义。

源码：`ConfigurationClassPostProcessor` 检查到 `@Configuration` 后，用 `ConfigurationClassEnhancer` 进行 CGLIB 增强。

## 3 @Bean 注解

`@Bean` 方法的返回值会被注册为 Spring Bean：

```java
@Configuration
public class DataConfig {
    @Bean(name = {"primaryDS", "mainDataSource"})  // 可以指定多个名称
    @Primary
    @Scope("singleton")
    public DataSource primaryDataSource() {
        return new HikariDataSource(config);
    }

    // 生命周期控制
    @Bean(initMethod = "init", destroyMethod = "close")
    public ExpensiveResource expensiveResource() {
        return new ExpensiveResource();
    }
}
```

源码：`ConfigurationClassParser` 将 `@Bean` 方法解析为 `BeanDefinition`，其 `beanClass` 为工厂方法类型，Spring 通过调用该工厂方法创建 Bean 实例。

## 4 @ComponentScan 组件扫描

`@ComponentScan` 替代 XML 中的 `<context:component-scan>`，自动扫描和注册组件：

```java
@Configuration
@ComponentScan(
    basePackages = "com.example",
    includeFilters = @Filter(type = FilterType.ANNOTATION, classes = Service.class),
    excludeFilters = @Filter(type = FilterType.REGEX, pattern = ".*Test.*")
)
public class AppConfig { }
```

底层：`ClassPathScanningCandidateComponentProvider` 扫描指定包，找到带有 `@Component` 或其派生注解的类，注册为 BeanDefinition。

## 5 @Import 与 ImportSelector

`@Import` 用于模块化配置，导入其他配置类：

```java
@Configuration
@Import({DatabaseConfig.class, SecurityConfig.class})
public class AppConfig { }
```

配合 `ImportSelector` 可以实现**条件化动态导入**，这是 Spring Boot 自动配置的核心机制：

```java
public class ProfileBasedImportSelector implements ImportSelector {
    @Override
    public String[] selectImports(AnnotationMetadata metadata) {
        // 根据环境条件动态决定导入哪些配置类
        return new String[] { "com.example.config.DevConfig" };
    }
}
```

## 6 @Conditional 条件化配置

`@Conditional` 系列注解实现**条件化 Bean 注册**，是 Spring Boot 自动配置的基础：

- `@ConditionalOnClass`：类路径下存在指定类时生效
- `@ConditionalOnMissingBean`：容器中不存在某 Bean 时生效（防止用户自定义配置被覆盖）
- `@ConditionalOnProperty`：配置属性满足条件时生效
- `@ConditionalOnWebApplication`：Web 应用环境时生效

```java
@Bean
@ConditionalOnClass(RedisTemplate.class)
@ConditionalOnProperty(name = "cache.enabled", havingValue = "true", matchIfMissing = false)
public CacheManager redisCacheManager() {
    return new RedisCacheManager();
}
```

每个 `@Conditional` 注解对应一个实现 `Condition` 接口的类，`ConfigurationClassParser` 解析时调用 `matches()` 方法进行条件判断。

## 7 @Profile 环境切换

`@Profile` 实现基于环境的配置切换，典型场景是开发/测试/生产环境数据源不同：

```java
@Configuration
@Profile("dev")
public class DevConfig {
    @Bean
    public DataSource devDataSource() {
        return new EmbeddedDatabaseBuilder().setType(EmbeddedDatabaseType.H2).build();
    }
}

@Configuration
@Profile("prod")
public class ProdConfig {
    @Bean
    public DataSource prodDataSource() {
        return new DriverManagerDataSource("jdbc:mysql://prod:3306/db");
    }
}
```

激活方式：`spring.profiles.active=dev`（配置文件）或 JVM 参数 `-Dspring.profiles.active=prod`。

`@Profile` 本质上是 `@Conditional(ProfileCondition.class)` 的特化版本。

## 8 @Value 属性注入

`@Value` 支持属性文件注入和 SpEL 表达式：

```java
@Component
public class AppConfig {
    @Value("${app.name:DefaultApp}")       // 从配置文件注入，带默认值
    private String appName;

    @Value("${server.port}")
    private int serverPort;

    @Value("#{systemProperties['java.version']}")  // SpEL 表达式
    private String javaVersion;

    @Value("#{T(Math).random() * 100}")    // SpEL 调用静态方法
    private double randomValue;
}
```

## 9 @PropertySource 加载属性文件

```java
@Configuration
@PropertySource("classpath:app.properties")
@PropertySource(value = "classpath:${ENV:dev}/config.properties", ignoreResourceNotFound = true)
public class AppConfig {
    @Autowired
    private Environment env;

    public void setup() {
        String url = env.getProperty("database.url");
    }
}
```

`PropertySourceProcessor` 将属性文件内容加载到 `Environment` 的 `PropertySource` 列表中，供 `@Value` 解析使用。

## 10 @Qualifier 和 @Primary 解决装配歧义

当容器中有多个同类型 Bean 时，需要明确指定注入哪一个：

**@Primary**：标记"首选" Bean，当有多个同类型 Bean 时自动优先选择：

```java
@Bean
@Primary
public DataSource primaryDataSource() { return new PrimaryDataSource(); }
```

**@Qualifier**：精确指定 Bean 名称，优先级高于 `@Primary`：

```java
@Autowired
@Qualifier("primaryDataSource")
private DataSource dataSource;
```

**选择原则**：只有一个首选 Bean 时用 `@Primary`；需要明确指定某个特定 Bean 时用 `@Qualifier`。

## 11 @PostConstruct 和 @PreDestroy

这两个是 JSR-250 标准注解，管理 Bean 的生命周期回调：

```java
@Service
public class DatabaseConnection {
    private Connection connection;

    @PostConstruct
    public void init() {
        // Bean 属性注入完成后执行，相当于 init-method
        // 执行时机：BeanPostProcessor 前置处理之后、初始化完成之前
        this.connection = createConnection();
    }

    @PreDestroy
    public void cleanup() {
        // Bean 销毁前执行，相当于 destroy-method
        if (connection != null) connection.close();
    }
}
```

与 `InitializingBean.afterPropertiesSet()` 的区别：功能相同，但 `@PostConstruct` 是标准 Java 注解，不依赖 Spring 接口，代码更"干净"。推荐用注解方式。
