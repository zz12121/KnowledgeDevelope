# 任务调度与异步执行

Spring 提供了两套非常实用的机制：一套是定时任务（@Scheduled），另一套是异步执行（@Async）。这两个功能看起来很简单，但背后的坑也不少，生产环境必须认真配置。

---

## 一、定时任务：@Scheduled

### 基本使用

启用定时任务只需两步：在入口类或配置类上加 `@EnableScheduling`，然后在具体方法上加 `@Scheduled`。

```java
@SpringBootApplication
@EnableScheduling
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

@Component
public class ReportScheduler {
    
    // 每 5 秒执行一次（不管上次任务是否完成）
    @Scheduled(fixedRate = 5000)
    public void generateHealthReport() { }
    
    // 上次执行完毕后，等 3 秒再执行下一次
    @Scheduled(fixedDelay = 3000)
    public void processQueueMessage() { }
    
    // 每天中午 12 点执行
    @Scheduled(cron = "0 0 12 * * ?")
    public void generateDailyReport() { }
    
    // 启动后延迟 10 秒再开始，之后每 5 秒执行一次
    @Scheduled(initialDelay = 10000, fixedRate = 5000)
    public void delayedTask() { }
}
```

**`fixedRate` vs `fixedDelay` 的区别：**

- `fixedRate`：以任务开始时间为基准，固定频率触发，不关心上次是否执行完。如果任务执行时间超过间隔，会产生堆积。
- `fixedDelay`：等上次任务彻底完成后，再等指定时间才触发下次。更安全，适合顺序性强的任务。

---

### Cron 表达式

Spring 使用 Quartz Cron 的变体，格式是 **秒 分 时 日 月 周 [年]**，共 6-7 个字段。

**常用特殊字符：**

- `*`：任意值
- `?`：放弃指定（仅用于日和周字段，两者互斥）
- `-`：范围，比如 `MON-FRI`
- `/`：步长，比如 `0/30` 表示从 0 开始每 30 个单位
- `,`：枚举多个值

**常用示例：**

```
0 0 * * * ?          每整点执行一次
0 0/30 * * * ?       每 30 分钟执行一次
0 0 9 * * ?          每天上午 9 点执行
0 0 12 * * MON-FRI   周一到周五中午 12 点执行
0 0 0 1 * ?          每月 1 号 0 点执行
0 0 0 L * ?          每月最后一天 0 点执行
```

---

### 原理机制

`@EnableScheduling` 通过 `@Import` 导入 `SchedulingConfiguration`，注册了 `ScheduledAnnotationBeanPostProcessor`。这个后处理器会在 Bean 初始化完成后扫描所有 `@Scheduled` 方法，把它们封装成 Task 对象注册到 `ScheduledTaskRegistrar`。

实际调度用的是 JDK 的 `ScheduledThreadPoolExecutor`，默认只有一个线程，这是最常见的坑。

---

### 单线程陷阱 & 配置线程池

**默认是单线程的！** 如果你有多个定时任务，任何一个耗时过长，都会阻塞其他任务。生产环境必须配置线程池：

```java
@Configuration
@EnableScheduling
public class SchedulerConfig implements SchedulingConfigurer {
    
    @Override
    public void configureTasks(ScheduledTaskRegistrar registrar) {
        ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
        scheduler.setPoolSize(10);
        scheduler.setThreadNamePrefix("Scheduled-");
        scheduler.setAwaitTerminationSeconds(30);
        scheduler.setWaitForTasksToCompleteOnShutdown(true);
        scheduler.initialize();
        registrar.setTaskScheduler(scheduler);
    }
}
```

---

### 分布式环境注意事项

集群部署时，多个实例都会执行定时任务，通常需要保证只有一个实例执行。解决方案：

1. **分布式锁**（推荐）：用 Redis 或 Zookeeper 实现，任务执行前抢锁，执行完释放
2. **数据库锁**：在数据库层面控制唯一执行
3. **专门的调度框架**：Quartz 集群模式（基于数据库），XXL-JOB、Elastic-Job 等

## 二、异步执行：@Async

### 基本使用

启用异步同样两步：`@EnableAsync` 开启能力，`@Async` 标记方法：

```java
@Configuration
@EnableAsync
public class AsyncConfig { }

@Service
public class EmailService {
    
    // 无返回值：调用后立即返回，方法在新线程中执行
    @Async
    public void sendWelcomeEmail(String email) {
        // 发送邮件逻辑，不影响主线程
    }
    
    // 有返回值：通过 Future 获取结果
    @Async
    public Future<String> processDataAsync(String data) {
        // 耗时处理...
        return new AsyncResult<>("处理完成");
    }
    
    // 指定使用特定线程池
    @Async("batchJobExecutor")
    public Future<String> executeBatchJob(String jobId) {
        return new AsyncResult<>("Job " + jobId + " completed");
    }
}
```

---

### 原理机制

`@EnableAsync` 注册了 `AsyncAnnotationBeanPostProcessor`，为包含 `@Async` 方法的 Bean 创建 AOP 代理。调用代理方法时，`AsyncExecutionInterceptor` 会拦截调用，把方法包装成任务提交给线程池执行。

默认如果找不到自定义的 `TaskExecutor`，会用 `SimpleAsyncTaskExecutor`，每次调用都创建新线程，**绝对不能用于生产环境**！

---

### 配置线程池（必须）

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {
    
    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);                  // 核心线程数
        executor.setMaxPoolSize(20);                  // 最大线程数
        executor.setQueueCapacity(100);               // 队列容量
        executor.setThreadNamePrefix("Async-");
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy()); // 拒绝策略：调用方自己执行
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(60);
        executor.initialize();
        return executor;
    }
    
    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) -> {
            log.error("异步方法 {} 执行异常: {}", method.getName(), ex.getMessage(), ex);
        };
    }
}
```

**多线程池配置**，按任务类型分开（IO 密集型 vs CPU 密集型）：

```java
@Configuration
@EnableAsync
public class MultiExecutorConfig {
    
    @Bean("ioBoundExecutor")
    public Executor ioBoundExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(20); // IO 任务可以多配一些
        executor.setThreadNamePrefix("IO-Async-");
        executor.initialize();
        return executor;
    }
    
    @Bean("cpuBoundExecutor")
    public Executor cpuBoundExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        // CPU 密集型：线程数 ≈ CPU 核心数
        executor.setCorePoolSize(Runtime.getRuntime().availableProcessors());
        executor.setThreadNamePrefix("CPU-Async-");
        executor.initialize();
        return executor;
    }
}

@Service
public class SpecializedService {
    
    @Async("ioBoundExecutor")
    public void dataExportTask() { /* 文件导出/网络请求 */ }
    
    @Async("cpuBoundExecutor")
    public Future<ComplexResult> heavyComputation() { /* 计算密集任务 */ }
}
```

---

### 常见坑点

**坑一：自调用不生效**

`@Async` 基于 AOP 代理，同类内部调用不走代理，注解白加。解法同 `@Transactional`：把异步方法提取到单独的 Bean 里。

**坑二：只能修饰 public 方法**

非 public 方法无法被 AOP 代理，加了也不生效。

**坑三：异常不会传播给调用方**

异步方法里的异常不会抛给调用者，必须通过 `AsyncUncaughtExceptionHandler` 来捕获处理，否则异常会被静默吞掉，很难排查。

**坑四：Spring 事务不跨线程**

异步方法运行在独立线程，原有的事务上下文不会传播过来。如果异步方法里需要事务，需要在方法上单独加 `@Transactional`。

## 三、线程池参数调优建议

| 参数 | IO 密集型 | CPU 密集型 | 说明 |
|------|-----------|------------|------|
| 核心线程数 | 20-50 | CPU 核心数 | 常驻线程数 |
| 最大线程数 | 100-200 | CPU 核心数 + 1 | 峰值扩容上限 |
| 队列容量 | 100-500 | 50-100 | 排队缓冲区 |
| 拒绝策略 | CallerRunsPolicy | AbortPolicy | 前者不丢任务，后者直接抛异常 |

一个常见的口诀：**IO 密集型线程数 = CPU 核数 × 2；CPU 密集型线程数 = CPU 核数 + 1**。
