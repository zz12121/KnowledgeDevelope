# 死信队列与延迟队列

## 一、死信队列（DLQ）

### 什么是死信队列
死信队列（Dead Letter Queue）存放无法被正常消费的消息。消息变为死信后，会被重新路由到**死信交换器（DLX）**，最终进入死信队列。死信队列本身就是普通队列，只是专门用于接收死信消息。

### 消息成为死信的三种情况
1. **消息被消费者拒绝且不重新入队**：`basicReject(tag, false)` 或 `basicNack(tag, false, false)`
2. **消息 TTL 过期**：队列级别 TTL 或消息级别 TTL 到期
3. **队列达到最大长度**：超过 `x-max-length` 限制

### 完整配置示例

```java
@Configuration
public class DeadLetterConfig {
    
    // 业务队列：绑定死信交换器
    @Bean
    public Queue businessQueue() {
        Map<String, Object> args = new HashMap<>();
        args.put("x-dead-letter-exchange", "dlx.exchange");     // 死信交换器
        args.put("x-dead-letter-routing-key", "dlx.key");       // 死信路由键
        args.put("x-message-ttl", 60000);                        // 消息 TTL（可选）
        args.put("x-max-length", 1000);                          // 最大长度（可选）
        return new Queue("business.queue", true, false, false, args);
    }

    // 死信交换器
    @Bean
    public DirectExchange dlxExchange() {
        return new DirectExchange("dlx.exchange", true, false);
    }

    // 死信队列
    @Bean
    public Queue dlxQueue() {
        return new Queue("dlx.queue", true);
    }

    @Bean
    public Binding dlxBinding() {
        return BindingBuilder.bind(dlxQueue()).to(dlxExchange()).with("dlx.key");
    }
}

// 死信消费者
@RabbitListener(queues = "dlx.queue")
public void handleDlxMessage(Message message) {
    log.error("收到死信消息: {}", new String(message.getBody()));
    // 人工干预、告警或重新处理
}
```

### 使用场景
- **异常消息诊断**：处理失败的消息转入 DLQ，集中分析原因
- **延迟重试**：TTL 过期 → 死信 → DLQ，实现延迟重试
- **流量控制**：超出处理能力的消息进入 DLQ 等待

## 二、延迟队列

### 方案一：TTL + 死信队列（原生，无需插件）

**实现原理**：消息先发送到带 TTL 的中间队列，等待过期后成为死信，被路由到真正的处理队列。

```
生产者 → [TTL 中间队列（不挂消费者）] --TTL到期--> [DLX] → [延迟处理队列] → 消费者
```

```java
// 生产者发送延迟消息（消息级 TTL）
public void sendDelayMessage(String message, int delayMs) {
    MessagePostProcessor processor = msg -> {
        msg.getMessageProperties().setExpiration(String.valueOf(delayMs));
        return msg;
    };
    rabbitTemplate.convertAndSend("business.exchange", "business.key", message, processor);
}

// 延迟消费者监听 DLQ
@RabbitListener(queues = "dlx.queue")
public void handleDelayMessage(String message) {
    log.info("收到延迟消息: {}", message);
    // 执行延迟业务逻辑（如订单超时取消）
}
```

**优点**：无需插件，兼容性好  
**缺点**：精度不高，大规模延迟消息性能较差

### 方案二：rabbitmq_delayed_message_exchange 插件

**原理**：消息在 Exchange 端延迟（持久化到 Mnesia 数据库），到期后路由到目标队列。精度更高。

```java
@Bean
public CustomExchange delayedExchange() {
    Map<String, Object> args = new HashMap<>();
    args.put("x-delayed-type", "direct");
    return new CustomExchange("delayed.exchange", "x-delayed-message", true, false, args);
}

// 发送延迟消息
public void sendDelayedMessage(String message, int delayMs) {
    rabbitTemplate.convertAndSend("delayed.exchange", "delayed.key", message,
        msg -> {
            msg.getMessageProperties().setHeader("x-delay", delayMs);
            return msg;
        });
}
```

**优点**：精确延迟，支持任意延迟时间  
**缺点**：需要安装插件，集群环境配置略复杂

### 方案三：外部调度器 + MQ

调度器（如 Quartz、分布式任务框架）到时间后再发消息到 MQ 处理。精度最高，但系统复杂度也最高。适合大规模、高精度的延迟需求。

## 三、消息 TTL

### 队列级别 TTL
```java
Map<String, Object> args = new HashMap<>();
args.put("x-message-ttl", 60000);  // 60 秒，队列内所有消息共享
Queue queue = new Queue("ttl.queue", true, false, false, args);
```

### 消息级别 TTL
```java
public void sendWithTTL(String message, int ttl) {
    rabbitTemplate.convertAndSend("exchange", "key", message,
        msg -> {
            msg.getMessageProperties().setExpiration(String.valueOf(ttl));
            return msg;
        });
}
```

**两者区别：**
- 优先级：消息 TTL 优先级高。两者共存时取**较小值**生效
- 粒度：队列 TTL 统一管理（开销小），消息 TTL 按条控制（开销大）
- 注意：过期消息不会立即删除，等待下次队列扫描时处理，TTL=0 也可能有微小延迟
