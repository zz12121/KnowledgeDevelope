# 消息可靠性与持久化

## 一、三层持久化配置

RabbitMQ 重启后要保证消息不丢失，必须同时配置三层持久化（木桶原理，缺一不可）：

### 1. 交换器持久化（`durable=true`）
```java
channel.exchangeDeclare(
    "persistent_exchange",
    "direct",
    true  // durable=true，持久化
);
```
非持久化交换器在 Broker 重启后会被删除，消息无法路由。

### 2. 队列持久化（`durable=true`）
```java
channel.queueDeclare(
    "persistent_queue",
    true,   // durable=true，持久化
    false,  // 非独占（持久化队列不能独占）
    false,  // 非自动删除
    null
);
```

### 3. 消息持久化（`deliveryMode=2`）
```java
// 方式一：使用内置属性常量（推荐）
channel.basicPublish(
    "persistent_exchange", "routing.key",
    MessageProperties.PERSISTENT_TEXT_PLAIN,  // deliveryMode=2
    message.getBytes()
);

// 方式二：手动构造
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .deliveryMode(2)  // 1=非持久化，2=持久化
    .contentType("text/plain")
    .build();
channel.basicPublish("exchange", "key", props, message.getBytes());
```

**底层实现**：RabbitMQ 内部通过 `msg_store` 模块追加写入消息内容，通过 `queue_index` 模块记录消息在队列中的位置。

**注意事项**：
- 持久化有约 10% 的延迟开销，需根据业务权衡
- 持久化不是实时刷盘，存在极短的时间窗口

## 二、生产者确认机制（Publisher Confirm）

Publisher Confirm 是 RabbitMQ 提供的**异步确认机制**，确保消息已成功到达 Broker。

**工作原理**：
1. 生产者调用 `channel.confirmSelect()` 进入 confirm 模式
2. 每条消息分配唯一 DeliveryTag
3. Broker 处理后异步返回 Ack / Nack
4. 生产者通过回调处理结果

**Spring Boot 配置：**
```yaml
spring:
  rabbitmq:
    publisher-confirm-type: correlated  # 开启关联确认
    publisher-returns: true             # 开启 Return 机制
```

```java
@Bean
public RabbitTemplate rabbitTemplate(ConnectionFactory connectionFactory) {
    RabbitTemplate template = new RabbitTemplate(connectionFactory);
    template.setMandatory(true);  // 开启 mandatory
    
    // 确认回调：消息是否到达 Exchange
    template.setConfirmCallback((correlationData, ack, cause) -> {
        if (ack) {
            log.info("消息到达 Exchange，ID: {}", correlationData.getId());
        } else {
            log.error("消息未到达 Exchange，ID: {}，原因: {}", 
                     correlationData.getId(), cause);
            // 重发或记录告警
        }
    });
    
    // Return 回调：消息是否从 Exchange 路由到 Queue
    template.setReturnsCallback(returned -> {
        log.error("消息无法路由！Exchange: {}，RoutingKey: {}，回复码: {}",
                 returned.getExchange(), returned.getRoutingKey(), 
                 returned.getReplyCode());
    });
    
    return template;
}
```

**发送时携带关联数据：**
```java
CorrelationData correlationData = new CorrelationData(UUID.randomUUID().toString());
rabbitTemplate.convertAndSend("exchange", "routing.key", message, correlationData);
```

## 三、消费者确认机制（Consumer ACK）

**三种 ACK 模式：**

- **自动确认（NONE）**：消息发出即确认，消费者崩溃时消息丢失。仅用于可容忍丢失的非重要消息
- **手动确认（MANUAL）**：业务处理完成后手动确认，**生产环境推荐**
- **不确认**：特殊场景使用

**手动确认代码：**
```java
@RabbitListener(queues = "order.queue")
public void handleOrderMessage(Message message, Channel channel, 
                             @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag) {
    try {
        processBusiness(message);
        
        // 处理成功，手动 Ack（false = 只确认当前消息）
        channel.basicAck(deliveryTag, false);
        
    } catch (BusinessException e) {
        // 业务异常：拒绝且不重新入队（避免无限重试）
        channel.basicReject(deliveryTag, false);
        
    } catch (Exception e) {
        // 系统异常：拒绝并重新入队重试
        channel.basicNack(deliveryTag, false, true);
    }
}
```

**自动 vs 手动 ACK 的核心区别：**
- 自动 ACK：消息发出立即确认，若消费者在处理中崩溃，消息永久丢失
- 手动 ACK：消息处理期间保持 Unacked 状态，崩溃后自动重新投递

## 四、mandatory 参数与不可路由消息处理

`mandatory=true` 时，当消息无法路由到任何队列，Broker 通过 `ReturnCallback` 将消息返回给生产者，而不是直接丢弃（默认行为）。

```java
template.setMandatory(true);
template.setReturnsCallback(returned -> {
    log.warn("消息无法路由: {}({})->{}，回复码: {}",
            returned.getExchange(), returned.getRoutingKey(),
            returned.getReplyCode(), returned.getReplyText());
    // 转入死信处理或告警
});
```

**不可路由消息的完整处理流程：**
1. **实时监控**：ReturnCallback 立即发现路由问题
2. **自动归档**：存入审计数据库
3. **告警通知**：通知开发人员检查绑定配置
4. **备用交换器（Alternate Exchange）**：声明主交换器时指定备用交换器，接收所有无法路由的消息

```java
Map<String, Object> exchangeArgs = new HashMap<>();
exchangeArgs.put("alternate-exchange", "ae.exchange");
channel.exchangeDeclare("main.exchange", "direct", true, false, exchangeArgs);
```

## 五、事务机制 vs Publisher Confirm

**事务机制：**
```java
try {
    channel.txSelect();   // 开启事务
    channel.basicPublish("exchange1", "key1", null, message1.getBytes());
    channel.basicPublish("exchange2", "key2", null, message2.getBytes());
    channel.txCommit();   // 提交
} catch (Exception e) {
    channel.txRollback(); // 回滚
}
```

**两者对比：**

| 特性 | 事务机制 | Publisher Confirm |
|------|----------|-------------------|
| 机制类型 | 同步阻塞 | 异步非阻塞 |
| 性能影响 | 比 Confirm 慢 10-20 倍 | 轻微 |
| 确认粒度 | 批量（事务级） | 单条或批量 |
| 使用场景 | 强原子性且可接受性能损耗 | 绝大多数生产场景 |

**结论：绝大多数场景选 Publisher Confirm，仅在需要严格原子性时才用事务机制。**
