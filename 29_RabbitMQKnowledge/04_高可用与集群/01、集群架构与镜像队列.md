# 集群架构与高可用

## 一、集群节点类型

RabbitMQ 集群中有两种节点类型：

**磁盘节点（Disk Node）**
- 元数据（交换器、队列定义、绑定关系、用户权限等）存储到磁盘
- 重启后不丢失配置信息，可靠性高
- **生产环境必须至少有一个磁盘节点**

**内存节点（RAM Node）**
- 元数据只存储在内存中，重启后配置丢失
- 读写速度更快，可提升吞吐量
- 不能将全部节点都设为内存节点（否则全部崩溃后无法恢复）

**典型架构**：1-2 个磁盘节点 + 多个内存节点（混合架构）

```bash
# 将节点加入集群并指定为内存节点
rabbitmqctl stop_app
rabbitmqctl join_cluster --ram rabbit@disk-node1
rabbitmqctl start_app
```

## 二、普通集群（默认模式）

**特点**：节点间只同步元数据，**消息数据仅存储在创建队列的节点上**。

当消费者连接到非队列所在节点时，该节点会从队列主节点**临时转发**消息。

**局限性**：
- 若队列所在节点宕机，即使消息已持久化，该队列也无法使用（非真正高可用）
- 所有请求最终汇聚到主节点，存在性能瓶颈

**适用场景**：主要用于提升吞吐量（多节点分摊压力），不用于高可用需求。

---

## 三、镜像队列（传统高可用方案）

**特点**：队列数据在多个节点间**完整复制**，形成主从结构。主节点（Master）接收写请求，同步到所有镜像从节点（Slave）。

**配置方式：**
```bash
# 镜像到所有节点
rabbitmqctl set_policy ha-all "^ha\." '{"ha-mode":"all"}'

# 精确数量模式：镜像到 2 个节点
rabbitmqctl set_policy ha-two "^ha\." '{"ha-mode":"exactly","ha-params":2,"ha-sync-mode":"automatic"}'

# 特定节点
rabbitmqctl set_policy ha-nodes "^ha\." '{"ha-mode":"nodes","ha-params":["rabbit@node1","rabbit@node2"]}'
```

**Java 客户端：**
```java
Map<String, Object> queueArgs = new HashMap<>();
queueArgs.put("x-ha-policy", "all");
channel.queueDeclare("ha.queue", true, false, false, queueArgs);
```

**同步模式：**
- `ha-sync-mode: manual`：新节点加入时不自动同步历史消息
- `ha-sync-mode: automatic`：新节点加入时自动同步（推荐）

**镜像节点数量权衡：**
- 镜像越多，数据越安全，但性能和网络开销越大
- 大型集群避免使用 `all` 模式（过度复制）

## 四、仲裁队列（Quorum Queue）— 推荐方案

**RabbitMQ 3.8.0+ 推荐使用**，基于 **Raft 共识算法**实现，替代传统镜像队列。

**核心优势：**
- 强一致性保证，Leader 故障时自动从多数派中选举新 Leader
- 简化配置，无需复杂的策略设置
- 更好的故障处理，自动应对网络分区和节点故障

```java
// 创建仲裁队列
Map<String, Object> args = new HashMap<>();
args.put("x-queue-type", "quorum");
channel.queueDeclare("quorum.queue", true, false, false, args);
```

**性能权衡：** 仲裁队列吞吐量比普通集群低约 40-50%，延迟高 40-50%，这是强一致性的代价。对于重要业务消息，值得付出这个代价。

## 五、Java 客户端高可用连接

```java
public Connection createHAConnection() throws Exception {
    ConnectionFactory factory = new ConnectionFactory();
    
    // 多节点地址
    Address[] addresses = {
        new Address("rabbitmq-node1", 5672),
        new Address("rabbitmq-node2", 5672), 
        new Address("rabbitmq-node3", 5672)
    };
    
    factory.setUsername("guest");
    factory.setPassword("guest");
    factory.setAutomaticRecoveryEnabled(true);  // 自动重连
    factory.setNetworkRecoveryInterval(5000);   // 5 秒后尝试恢复
    
    return factory.newConnection(addresses);
}
```

## 六、网络分区（Network Partition）

**什么是网络分区**：集群节点间因网络问题无法互相通信，形成多个独立子集群（"脑裂"）。后果是数据不一致，服务不可用。

**自动处理策略配置（`rabbitmq.conf`）：**
```bash
# 暂停少数派节点（推荐生产使用）
cluster_partition_handling = pause_minority

# 自动恢复模式
cluster_partition_handling = autoheal

# 忽略（默认，不推荐生产）
cluster_partition_handling = ignore
```

**排查命令：**
```bash
rabbitmqctl cluster_status          # 查看集群状态
tail -f /var/log/rabbitmq/rabbitmq.log | grep partition  # 监控分区日志
```

## 七、集群监控关键指标

```bash
rabbitmqctl cluster_status                                          # 集群状态
rabbitmqctl list_queues name messages_ready messages_unacknowledged # 队列积压
rabbitmqctl status | grep memory                                    # 内存使用
```

**告警阈值建议：**
- 内存使用率 > 80%
- 磁盘剩余 < 20%
- 消息堆积数 > 10,000
- 节点间网络延迟 > 100ms

**Prometheus 监控：**
```yaml
scrape_configs:
  - job_name: 'rabbitmq'
    static_configs:
      - targets: ['rabbitmq-node1:15692', 'rabbitmq-node2:15692']
```
