# 高级特性与最佳实践

## 一、消息幂等性与重复消费

消息重复是"至少一次"语义下的必然，防止重复消费的核心是**幂等性设计**。

### 重复消费的根本原因
- 生产者未收到 ACK，触发重试 → 消息被发送多次
- 消费者处理成功后、提交 Offset / ACK 前崩溃 → 消息被重新投递

### 幂等方案

**唯一 ID + Redis 去重**
```java
@RabbitListener(queues = "order.queue")
public void handleOrder(OrderMessage order, Channel channel, 
                        @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag) {
    String messageId = order.getMessageId();
    
    // 使用 SETNX：只有首次才能设置成功
    Boolean isFirstProcess = redisTemplate.opsForValue()
        .setIfAbsent("msg:processed:" + messageId, "1", 24, TimeUnit.HOURS);
    
    if (Boolean.TRUE.equals(isFirstProcess)) {
        // 首次处理
        orderService.process(order);
        channel.basicAck(deliveryTag, false);
    } else {
        // 重复消息，直接确认跳过
        log.warn("重复消息，已跳过: {}", messageId);
        channel.basicAck(deliveryTag, false);
    }
}
```

**数据库唯一约束**
```sql
CREATE TABLE order_records (
    id BIGINT PRIMARY KEY,
    message_id VARCHAR(64) UNIQUE,  -- 唯一索引天然防重
    order_data TEXT,
    created_at DATETIME
);
```
消费时捕获 `DataIntegrityViolationException` 即认为是重复消息，直接 ACK 跳过。

**状态机机制**
有状态流转的业务（如订单）：处理前先查当前状态，仅当处于预期状态时才执行，配合乐观锁（version 字段）防并发重复。

## 二、优先级队列

RabbitMQ 原生支持优先级队列，内部使用**最大堆**维护，高优先级消息总在堆顶。

> 注意：只有队列有积压时优先级才有效，消费者空闲时消息到达即被消费，优先级无法体现。

```java
// 创建支持优先级的队列（最高优先级为 10）
Map<String, Object> args = new HashMap<>();
args.put("x-max-priority", 10);
channel.queueDeclare("priority_queue", true, false, false, args);

// 发送不同优先级的消息
for (int i = 0; i < 100; i++) {
    AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
        .priority(i % 10)  // 优先级 0-9
        .build();
    channel.basicPublish("", "priority_queue", props, ("Priority " + (i % 10)).getBytes());
}
```

## 三、预取（QoS）配置

`basicQos` 限制 Broker 一次推送给消费者的最大未确认消息数，防止消费者过载。

```java
// 消费者每次最多预取 10 条，处理完才接收新消息
channel.basicQos(10);
channel.basicConsume("order.queue", false, deliverCallback, consumerTag -> {});
```

合理设置 QoS 可以：
- 防止单个消费者积压大量 Unacked 消息
- 在多消费者场景下实现更均衡的负载分配
- 结合批量 ACK 进一步提升吞吐量

## 四、性能优化策略

**生产者优化：**
- 使用 Publisher Confirm 异步确认，不用事务机制
- 批量发送：在应用层累积消息后批量投递
- 合理设置 `connectionPool` 连接池大小

**消费者优化：**
- 合理设置 `basicQos` 预取数量（建议 10-100）
- 批量 ACK：`channel.basicAck(lastDeliveryTag, true)` 一次确认多条
- 消费逻辑轻量化，避免耗时的同步 RPC 调用

**Broker 优化：**
- 队列长度限制（`x-max-length`）防止无限堆积
- 合理规划 vhost，避免单 vhost 队列数过多
- 定期清理过期队列和未使用的绑定

## 五、安全与权限管理

```bash
# 创建用户
rabbitmqctl add_user app_user app_password

# 设置 vhost 权限（配置/写/读）
rabbitmqctl set_permissions -p /production app_user ".*" ".*" ".*"

# 设置角色（管理员）
rabbitmqctl set_user_tags app_user management

# 删除用户
rabbitmqctl delete_user app_user
```

**最小权限原则：**
- 生产者：只需对特定 Exchange 有**写权限**
- 消费者：只需对特定 Queue 有**读权限**
- 运维账号：管理员权限（management 标签）

## 六、Spring Boot 集成最佳实践

```yaml
spring:
  rabbitmq:
    host: localhost
    port: 5672
    username: guest
    password: guest
    virtual-host: /production
    listener:
      simple:
        acknowledge-mode: manual       # 手动确认
        prefetch: 10                   # QoS 预取
        concurrency: 3                 # 消费者线程数
        max-concurrency: 10
    publisher-confirm-type: correlated # 开启 Confirm
    publisher-returns: true            # 开启 Return
```

```java
// 消费者配置模板
@RabbitListener(queues = "order.queue")
public void consume(OrderMessage order, Channel channel, 
                    @Header(AmqpHeaders.DELIVERY_TAG) long tag) throws IOException {
    try {
        // 幂等性检查
        if (isDuplicate(order.getId())) {
            channel.basicAck(tag, false);
            return;
        }
        
        // 业务处理
        orderService.process(order);
        
        // 成功确认
        channel.basicAck(tag, false);
    } catch (BusinessException e) {
        // 业务异常，不重试（进死信）
        channel.basicReject(tag, false);
    } catch (Exception e) {
        // 系统异常，重新入队
        channel.basicNack(tag, false, true);
    }
}
```
