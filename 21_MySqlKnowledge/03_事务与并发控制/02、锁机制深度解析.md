# 锁机制深度解析

> MySQL 的锁机制是并发控制的基础。理解各种锁的类型、范围、加锁规则，才能分析和解决死锁、锁等待等生产问题。

---

## 一、锁的分类体系

```
MySQL 锁体系
├── 按粒度
│   ├── 全局锁（Global Lock）
│   ├── 表级锁（Table-Level Lock）
│   │   ├── 表读写锁（LOCK TABLES）
│   │   ├── 元数据锁（MDL, Metadata Lock）
│   │   ├── 意向锁（Intention Lock）
│   │   └── 自增锁（AUTO-INC Lock）
│   └── 行级锁（Row-Level Lock）—— InnoDB 独有
│       ├── 记录锁（Record Lock）
│       ├── 间隙锁（Gap Lock）
│       ├── 临键锁（Next-Key Lock）
│       └── 插入意向锁（Insert Intention Lock）
├── 按属性
│   ├── 共享锁（S Lock / 读锁）
│   └── 排他锁（X Lock / 写锁）
└── 按思想
    ├── 乐观锁
    └── 悲观锁
```

## 二、共享锁与排他锁

### 2.1 基本规则

| | 已持有 S 锁 | 已持有 X 锁 |
|--|------------|------------|
| **申请 S 锁** | ✅ 兼容 | ❌ 不兼容（等待）|
| **申请 X 锁** | ❌ 不兼容（等待）| ❌ 不兼容（等待）|

**核心原则**：
- **读读共享**：多个事务可以同时加 S 锁
- **读写互斥**：有 S 锁时不能加 X 锁，反之亦然
- **写写互斥**：同一时间只有一个事务可以持有 X 锁

### 2.2 显式加锁语法

```sql
-- 加共享锁（S 锁）
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;    -- MySQL 5.x
SELECT * FROM users WHERE id = 1 FOR SHARE;              -- MySQL 8.0+

-- 加排他锁（X 锁）
SELECT * FROM users WHERE id = 1 FOR UPDATE;

-- MySQL 8.0 新增锁选项
SELECT * FROM users WHERE id = 1 FOR UPDATE NOWAIT;       -- 无法加锁立即报错（不等待）
SELECT * FROM users WHERE id = 1 FOR UPDATE SKIP LOCKED;  -- 跳过已锁定的行
```

**隐式加锁**：
- `INSERT`、`UPDATE`、`DELETE` 操作自动对涉及的行加 X 锁

## 三、行级锁详解

**行级锁只在 InnoDB 中存在**，且**行锁是加在索引上的，而不是直接加在数据行上**。这是一个非常重要的特性，有以下推论：
- 如果查询条件没有走索引，行锁会**退化为表锁**
- 不同索引可能加不同范围的锁，需要结合具体索引类型理解

### 3.1 记录锁（Record Lock）

**锁定索引上的单个记录**（包括该记录本身）。

```sql
-- 假设 idx_id 是主键
SELECT * FROM users WHERE id = 5 FOR UPDATE;
-- 在 id=5 的索引记录上加 X 型记录锁
-- 其他事务无法修改 id=5 的行，但可以修改其他行

-- 如果是普通索引，还会在对应的主键索引上加记录锁（回表路径上的锁）
```

### 3.2 间隙锁（Gap Lock）

**锁定索引记录之间的间隙（不包含记录本身）**，目的是防止其他事务在这个区间内插入数据（防止幻读）。

```
假设 users 表 id 值为：1, 5, 10, 20

间隙区间：(-∞, 1), (1, 5), (5, 10), (10, 20), (20, +∞)

-- 查询 id > 5 AND id < 10（空查询）
SELECT * FROM users WHERE id > 5 AND id < 10 FOR UPDATE;
-- Gap Lock 锁住 (5, 10) 这个区间
-- 其他事务无法在这个区间插入 id=6, 7, 8, 9 的行
```

**间隙锁的特性**：
- 间隙锁是**共享的**：多个事务可以同时持有同一个间隙的 Gap Lock，互不阻塞
- 间隙锁只阻塞**插入操作**，不阻塞其他间隙锁
- 在 **RC 隔离级别**下，间隙锁**自动失效**（只在 RR 和 Serializable 下生效）

### 3.3 临键锁（Next-Key Lock）

**Next-Key Lock = Record Lock + Gap Lock**，锁定一个**左开右闭的区间 (a, b]**。

这是 **InnoDB 在 RR 隔离级别下默认使用的行锁算法**。

```
假设 users 表 id 值为：1, 5, 10, 20

InnoDB 的临键锁区间（左开右闭）：
(-∞, 1], (1, 5], (5, 10], (10, 20], (20, +∞)

-- 查询 id >= 5 AND id <= 10 FOR UPDATE
SELECT * FROM users WHERE id >= 5 AND id <= 10 FOR UPDATE;
-- Next-Key Lock 锁住：
-- (1, 5]：锁住 id=5 的记录 + (1,5) 间隙
-- (5, 10]：锁住 id=10 的记录 + (5,10) 间隙
-- 防止在 (1, 10] 范围内插入任何新行
```

### 3.4 插入意向锁（Insert Intention Lock）

**INSERT 操作在真正写入记录前，先尝试获取的一种特殊间隙锁**。

多个事务想要在同一个间隙的不同位置插入记录时，插入意向锁之间**互相兼容**（不会互相阻塞），提高了插入的并发性。

但是，插入意向锁和间隙锁（Gap Lock / Next-Key Lock）**不兼容**：

```
场景：
- 事务A：SELECT FOR UPDATE 获得了 (5, 10] 的 Next-Key Lock
- 事务B：INSERT id=7，需要获取 (5, 10) 的插入意向锁
- 结果：事务B等待，因为插入意向锁与事务A的间隙锁冲突
```

## 四、表级锁

### 4.1 意向锁（Intention Lock）

意向锁是 InnoDB **自动管理**的表级锁，用于告知其他事务"某行已被锁定"，从而帮助快速判断是否可以加表锁。

```
事务A要对某行加 S 锁：
  → 先自动在表上加 IS（意向共享锁）
  → 再在行上加 S 锁

事务A要对某行加 X 锁：
  → 先自动在表上加 IX（意向排他锁）
  → 再在行上加 X 锁

意向锁的意义：
  当事务B需要给整个表加 X 锁时：
  - 不需要逐行检查是否有行锁
  - 只需检查表上是否有 IS 或 IX
  - 如果有 IX，则等待（因为表有行被锁定）
  - 极大减少了表锁申请的检查开销
```

**意向锁兼容矩阵**：

| | IS | IX | S（表锁）| X（表锁）|
|--|----|----|---------|---------|
| **IS** | ✅ | ✅ | ✅ | ❌ |
| **IX** | ✅ | ✅ | ❌ | ❌ |
| **S（表锁）** | ✅ | ❌ | ✅ | ❌ |
| **X（表锁）** | ❌ | ❌ | ❌ | ❌ |

### 4.2 全局锁（FTWRL）

```sql
-- 对整个数据库加读锁（通常用于全库备份）
FLUSH TABLES WITH READ LOCK;

-- 此时：所有写操作（DML、DDL）都被阻塞

-- 释放全局锁
UNLOCK TABLES;

-- 更好的备份方式（InnoDB 表）
mysqldump --single-transaction ...
-- 使用事务快照备份，无需全局锁
```

### 4.3 元数据锁（MDL）

MDL 在**用户执行 SQL 时自动加锁**，保护表的结构（元数据）不被并发的 DDL 修改：
- 执行 DML（SELECT/INSERT/UPDATE/DELETE）→ 自动加 MDL 读锁
- 执行 DDL（ALTER TABLE）→ 自动加 MDL 写锁，写锁申请需等待所有读锁释放

**MDL 锁是生产事故的常见来源**：

```
场景：Online DDL 阻塞
1. 业务连接 A：BEGIN; SELECT * FROM users; （长事务，持有 MDL 读锁，未提交）
2. DBA 执行：ALTER TABLE users ADD COLUMN phone VARCHAR(20);
   → 需要 MDL 写锁，等待连接 A 释放
3. 后续业务连接 B, C, D...：所有新的 DML 也在等待（MDL 写锁请求阻塞了后续读锁申请）
4. 数据库连接队列爆满，服务不可用！

解决：
- ALTER TABLE 前先检查是否有长事务（SHOW PROCESSLIST）
- 优先使用 pt-online-schema-change 或 gh-ost 进行无锁 DDL
```

## 五、乐观锁与悲观锁

### 5.1 悲观锁

**假设冲突必然发生，先加锁再操作**。

```sql
-- 使用数据库锁实现（行锁）
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- 加 X 锁
-- 检查业务逻辑
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

**适用场景**：写操作频繁、冲突概率高的场景（如库存扣减、账户余额）。

### 5.2 乐观锁

**假设冲突很少发生，提交时才检查是否被修改**。通常通过**版本号（version）** 或**时间戳**实现：

```sql
-- 表中添加 version 字段
ALTER TABLE products ADD COLUMN version INT NOT NULL DEFAULT 0;

-- Java 应用中的使用流程：
-- 1. 查询时获取 version
SELECT id, stock, version FROM products WHERE id = 1;
-- 假设返回：id=1, stock=100, version=5

-- 2. 更新时带上 version 条件
UPDATE products
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 5;
-- 如果 affected rows = 0，说明有人在我们查询后更新了这行
-- 可以重试或抛出异常

-- 3. 检查 affected rows
-- affected rows = 1 → 更新成功
-- affected rows = 0 → 乐观锁冲突，需要重试
```

**适用场景**：读多写少、冲突概率低的场景（如文章点赞、积分累加）。

## 六、死锁分析与处理

### 6.1 死锁形成条件

```
两个事务互相等待对方的锁：

事务A：锁住 id=1，等待 id=2 的锁
事务B：锁住 id=2，等待 id=1 的锁

→ 互相等待，永远无法推进
```

**InnoDB 的死锁处理**：
- 自动检测死锁（等待图，Wait-For Graph）
- 选择"代价最小"的事务（回滚 undo log 量少的）进行回滚
- 另一个事务获得锁继续执行

### 6.2 常见死锁场景

```sql
-- 场景1：加锁顺序不一致
-- 事务A：UPDATE table SET x WHERE id=1; UPDATE table SET x WHERE id=2;
-- 事务B：UPDATE table SET x WHERE id=2; UPDATE table SET x WHERE id=1;
-- 解决：约定相同的加锁顺序（总是 id 从小到大）

-- 场景2：间隙锁死锁
-- 事务A：SELECT FOR UPDATE WHERE id BETWEEN 1 AND 10（获得 Gap Lock）
-- 事务B：INSERT id=5（等待 A 的 Gap Lock）
-- 事务A：INSERT id=7（等待 B 的插入意向锁）
-- 解决：尽量避免长事务，减少间隙锁持有时间
```

### 6.3 查看和预防死锁

```sql
-- 查看最近一次死锁信息
SHOW ENGINE INNODB STATUS\G
-- 找到 "LATEST DETECTED DEADLOCK" 部分

-- 查看当前锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 预防措施
-- 1. 约定相同的资源加锁顺序
-- 2. 缩短事务，快速提交，减少锁持有时间
-- 3. 合理设置锁等待超时
SET GLOBAL innodb_lock_wait_timeout = 10;  -- 等待超过10秒自动回滚
-- 4. 降低隔离级别（RC 下无间隙锁，死锁减少）
-- 5. 批量操作时对数据排序（避免随机顺序加锁）
```

### 6.4 行锁退化为表锁

```sql
-- 当查询条件不走索引，行锁会退化为表锁！
-- 假设 users 表的 name 列没有索引
UPDATE users SET status = 1 WHERE name = 'Alice';
-- 全表扫描 → 锁定全表所有行（等效表锁）
-- 并发更新其他用户名的操作都会被阻塞！

-- 解决：确保 WHERE 条件走索引
CREATE INDEX idx_name ON users(name);
```

## 七、面试要点速查

| 问题 | 核心答案 |
|------|---------|
| 行锁加在哪里？ | 加在索引上，不走索引则退化为表锁 |
| 间隙锁的目的？ | 防止其他事务在范围内插入新行，解决幻读 |
| Next-Key Lock 是什么？ | 记录锁 + 间隙锁的组合，锁左开右闭区间 (a, b] |
| 意向锁的作用？ | 表级锁，表示行已被锁，辅助快速判断是否可以加表锁 |
| 乐观锁和悲观锁的区别？ | 悲观锁：先加锁防冲突；乐观锁：提交时通过版本号检测冲突 |
| 如何避免死锁？ | 统一加锁顺序；缩短事务；设置锁等待超时 |
| RC 和 RR 在锁上的主要区别？ | RC 下无间隙锁，锁范围更小，并发更高但无法防幻读 |
| MDL 锁是什么？如何避免 MDL 阻塞？| MDL 保护表结构；DDL 前检查长事务；使用 gh-ost 无锁 DDL |
