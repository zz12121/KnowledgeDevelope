# 隔离级别与并发问题

> 隔离级别是数据库在**并发一致性**与**系统性能**之间做出的权衡取舍。理解隔离级别，本质上是理解：不同的隔离策略下，哪些并发异常被允许存在，MySQL 又是如何从实现层面控制这些异常的。

---

## 1. 四大并发异常

### 1.1 脏读（Dirty Read）

**定义**：事务 A 读取了事务 B **尚未提交**的数据，若事务 B 随后回滚，则事务 A 读到了从未真实存在的数据。

```
时间线：
T1: 事务B BEGIN
T2: 事务B: UPDATE account SET balance = 50 WHERE id=1;  -- 原值 100，未提交
T3: 事务A: SELECT balance FROM account WHERE id=1;       -- 读到 50 ← 脏读！
T4: 事务B: ROLLBACK                                       -- 50 这个值从未生效
T5: 事务A 拿着 50 去做业务逻辑 → 数据错误
```

**危害**：基于"幻影数据"做决策，可能导致业务逻辑严重出错（如超卖、重复支付）。

### 1.2 不可重复读（Non-Repeatable Read）

**定义**：事务 A 在同一事务内对**同一行数据**执行两次读取，期间事务 B **提交了修改**，导致两次结果不一致。

```
时间线：
T1: 事务A BEGIN
T2: 事务A: SELECT balance FROM account WHERE id=1;  -- 读到 100
T3: 事务B: UPDATE account SET balance=80 WHERE id=1; COMMIT
T4: 事务A: SELECT balance FROM account WHERE id=1;  -- 读到 80 ← 不可重复读！
T5: 事务A COMMIT
```

**与脏读的区别**：读到的是**已提交**的数据，所以数据是"真实的"，但对同一事务来说读取不稳定。

### 1.3 幻读（Phantom Read）

**定义**：事务 A 在同一事务内执行两次**范围查询**，期间事务 B **插入或删除了符合条件的行**并提交，导致两次结果集的**行数**不同。

```
时间线：
T1: 事务A BEGIN
T2: 事务A: SELECT COUNT(*) FROM orders WHERE status='pending';  -- 返回 10
T3: 事务B: INSERT INTO orders(status) VALUES('pending'); COMMIT
T4: 事务A: SELECT COUNT(*) FROM orders WHERE status='pending';  -- 返回 11 ← 幻读！
T5: 事务A COMMIT
```

**与不可重复读的区别**：
- 不可重复读：同一**行**数据前后值不同（UPDATE/DELETE 导致）
- 幻读：前后查询到的**行数**不同（INSERT 导致，出现"幻象行"）

### 1.4 丢失更新（Lost Update）

**定义**：两个事务同时读取同一行并基于读取值进行更新，其中一个事务的更新被另一个覆盖。

```
T1: 事务A 读取 stock=10
T2: 事务B 读取 stock=10
T3: 事务A: UPDATE SET stock=9  COMMIT  -- 减1
T4: 事务B: UPDATE SET stock=9  COMMIT  -- 也减1，但基于旧值，实际只扣了一次
```

> 丢失更新需通过**乐观锁**或**SELECT ... FOR UPDATE** 解决，标准 SQL 隔离级别不直接解决此问题。

## 2. 四种隔离级别

SQL 标准（ISO/IEC 9075）定义的四种隔离级别，从低到高：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | MySQL 实现机制 |
|----------|------|-----------|------|---------------|
| **READ UNCOMMITTED** | ❌ 存在 | ❌ 存在 | ❌ 存在 | 直接读最新数据，不用任何保护 |
| **READ COMMITTED** | ✅ 避免 | ❌ 存在 | ❌ 存在 | 每次读生成新 Read View |
| **REPEATABLE READ** | ✅ 避免 | ✅ 避免 | ⚠️ 快照读避免，当前读仍可能 | Read View 复用 + Next-Key Lock |
| **SERIALIZABLE** | ✅ 避免 | ✅ 避免 | ✅ 避免 | 所有读加共享锁，强制串行化 |

**MySQL 默认隔离级别：REPEATABLE READ（RR）**

---

## 3. MySQL 各隔离级别的底层实现

### 3.1 READ UNCOMMITTED（读未提交）

```
实现：不使用 MVCC，直接读最新版本（无论是否提交）
锁策略：写操作加行锁，读操作不加锁
性能：最高（几乎无开销）
安全：最低（允许脏读）
```

实际应用极少，只在对一致性无要求的统计场景（如近似计数）中使用。

### 3.2 READ COMMITTED（读已提交）

```
实现：使用 MVCC，每次快照读生成新的 Read View
原理：新 Read View 的 m_ids 已不包含已提交事务，所以能看到最新提交
解决：脏读（Read View 过滤未提交版本）
未解决：不可重复读（每次 Read View 不同）、幻读
```

RC 是 **Oracle/SQL Server 的默认级别**，也是互联网业务中常用的折中选择（比 RR 并发更高，无间隙锁导致的死锁问题）。

### 3.3 REPEATABLE READ（可重复读）

```
实现：MVCC（快照读） + Next-Key Lock（当前读）
快照读：第一次读时生成 Read View，整个事务复用
当前读：加 Next-Key Lock（Record Lock + Gap Lock）
解决：脏读、不可重复读、快照读幻读
未完全解决：当前读幻读（需 Next-Key Lock 辅助）
```

**MySQL InnoDB 在 RR 下对幻读的处理**：

| 读取方式 | 幻读处理机制 | 说明 |
|----------|-------------|------|
| 快照读（`SELECT`） | MVCC Read View 复用 | 新插入行的 trx_id ≥ max_trx_id，不可见 |
| 当前读（`SELECT...FOR UPDATE`、`UPDATE`、`DELETE`） | **Next-Key Lock** | 锁定扫描范围，阻止其他事务在范围内插入 |

> ⚠️ **RR 仍存在幻读的情况（边界场景）**：
> 在同一事务内，先执行快照读（不加锁），再执行当前读（加锁），由于当前读会读最新版本，可能看到快照读时"不存在"的行：
> ```sql
> -- 事务A（RR级别）
> SELECT * FROM t WHERE id > 10;    -- 快照读，读到空
> -- 此时事务B INSERT INTO t VALUES(15); COMMIT
> SELECT * FROM t WHERE id > 10 FOR UPDATE;  -- 当前读，读到 id=15
> -- 前后不一致 = 幻读（特殊场景）
> ```
> 解决：从事务开始就用 `FOR UPDATE` 或 `LOCK IN SHARE MODE` 一致使用当前读。

### 3.4 SERIALIZABLE（串行化）

```
实现：所有 SELECT 自动加共享锁（等效于 SELECT ... LOCK IN SHARE MODE）
效果：完全串行执行，消除所有并发异常
代价：性能最差，高并发场景吞吐量骤降
```

InnoDB 的 SERIALIZABLE 实质上将快照读升级为当前读（加锁），与 MVCC 机制不同。

## 4. 查看与设置隔离级别

```sql
-- 查看当前会话隔离级别（MySQL 8.0+）
SELECT @@transaction_isolation;

-- 查看全局隔离级别
SELECT @@global.transaction_isolation;

-- 设置当前会话隔离级别（仅影响当前连接）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置全局隔离级别（影响新连接，重启后失效）
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 持久化配置（写入 my.cnf）
# [mysqld]
# transaction_isolation = REPEATABLE-READ
```

**Spring/Java 中设置隔离级别**：

```java
// 方式1：@Transactional 注解
@Transactional(isolation = Isolation.READ_COMMITTED)
public void queryReport() {
    // 业务逻辑
}

// 方式2：编程式事务
DefaultTransactionDefinition def = new DefaultTransactionDefinition();
def.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
TransactionStatus status = txManager.getTransaction(def);
```

## 5. 各业务场景的隔离级别选择

| 业务场景 | 推荐隔离级别 | 原因 |
|----------|-------------|------|
| **电商订单、支付核心** | RR（MySQL默认） | 保证可重复读，结合 Next-Key Lock 防幻读 |
| **报表统计、数据分析** | RC 或 RR | 允许看到最新提交，RC 并发更高 |
| **库存扣减（高并发）** | RR + `SELECT...FOR UPDATE` | 悲观锁防止并发超卖 |
| **余额类账务系统** | RR 或 SERIALIZABLE | 要求强一致性 |
| **日志写入、数据采集** | RC | 并发高，一致性要求低 |
| **跨行转账（分布式）** | 应用层分布式事务（Seata/TCC） | 数据库隔离级别跨节点无效 |

### RC 还是 RR 的争议

互联网大厂（如阿里、美团）生产环境有时将 MySQL 全局隔离级别设置为 **RC**，原因如下：

| 维度 | RR | RC |
|------|----|----|
| 间隙锁 | 有（可能导致死锁） | 无 |
| 并发性 | 较低 | 更高 |
| binlog 格式要求 | Statement/Row 均可 | **必须使用 Row 格式**（Statement 在 RC 下主从可能不一致） |
| 幻读保护 | 有（快照读 + Next-Key Lock） | 无 |
| 适合场景 | 一致性要求高的OLTP | 高并发读写，业务层处理一致性 |

## 6. 实战：常见并发问题的解决方案

### 6.1 防止超卖（库存并发扣减）

```sql
-- 方案1：悲观锁（SELECT ... FOR UPDATE）
BEGIN;
SELECT stock FROM products WHERE id=1 FOR UPDATE;  -- 当前读，加排他锁
-- 应用层判断 stock > 0
UPDATE products SET stock = stock - 1 WHERE id = 1;
COMMIT;

-- 方案2：乐观锁（版本号/条件更新）
UPDATE products 
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND stock > 0 AND version = #{version};
-- 检查 affected rows = 1，否则重试或提示失败
```

### 6.2 防止重复提交（幂等控制）

```sql
-- 利用唯一索引
CREATE UNIQUE INDEX idx_order_dedup ON pay_records(order_id, biz_type);

-- 插入时捕获唯一键冲突
INSERT IGNORE INTO pay_records(order_id, biz_type, amount) VALUES(?, ?, ?);
-- 或者
INSERT INTO pay_records(...) VALUES(...)
ON DUPLICATE KEY UPDATE updated_at = NOW();
```

### 6.3 避免长事务

```sql
-- 查找当前长事务（运行超过30秒）
SELECT trx_id, trx_started, trx_state,
       TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS running_seconds,
       trx_query
FROM information_schema.INNODB_TRX
WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 30
ORDER BY trx_started;

-- kill 对应连接
KILL CONNECTION {trx_mysql_thread_id};
```

**长事务的危害**：
1. 持有锁时间长，阻塞其他事务
2. Read View 存活时间长，Undo Log 无法清理（表空间膨胀）
3. binlog 文件堆积，影响主从复制

## 7. 并发问题总结对比

| 并发问题 | 本质 | 触发条件 | 标准解决级别 | MySQL 具体机制 |
|----------|------|---------|-------------|---------------|
| 脏读 | 读未提交数据 | 读取未提交事务的写 | ≥ RC | MVCC Read View 过滤 |
| 不可重复读 | 同行读取不稳定 | 两次读取间有事务提交 UPDATE | ≥ RR | Read View 复用 |
| 快照读幻读 | 结果集行数变化 | 两次读取间有事务提交 INSERT | ≥ RR（MySQL） | Read View 复用（trx_id ≥ max 不可见）|
| 当前读幻读 | FOR UPDATE 读到新行 | 两次当前读间有事务提交 INSERT | SERIALIZABLE | **Next-Key Lock** |
| 丢失更新 | 覆盖写 | 两事务基于旧值更新 | 应用层处理 | FOR UPDATE / 乐观锁 |
