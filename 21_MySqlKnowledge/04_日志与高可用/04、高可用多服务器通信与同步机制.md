# 高可用多服务器通信与同步机制

> 高可用数据库架构的核心在于多台服务器之间的**通信协调**与**数据同步**。从传统的主从复制到现代的 MGR、PXC 集群，不同的架构采用不同的通信协议和一致性模型。深入理解这些底层机制，是设计、部署和维护高可用数据库系统的前提。

## 2. 主从复制通信原理

### 2.1 异步复制的通信机制

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      异步复制通信流程                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  主库（Master）                    从库（Slave）                         │
│  ┌─────────────┐               ┌─────────────┐                         │
│  │   事务执行   │               │             │                         │
│  │  1. 写 Buffer Pool           │             │                         │
│  │  2. 写 Redo Log              │             │                         │
│  │  3. 写 Binlog                │             │                         │
│  │     (fsync)                  │             │                         │
│  │  4. 返回"提交成功"           │             │                         │
│  └─────────────┘               │             │                         │
│        │                       │             │                         │
│        │  Binlog Dump 线程      │             │                         │
│        │  (推送模式)             │             │                         │
│        │                       │             │                         │
│        │  网络传输               │             │                         │
│        │───────────────────────▶│             │                         │
│        │  Binlog Events         │             │                         │
│        │                       │             │                         │
│        │                       │  I/O 线程    │                         │
│        │                       │  接收并写入  │                         │
│        │                       │  Relay Log  │                         │
│        │                       └──────┬──────┘                         │
│        │                              │                                 │
│        │                              │  SQL 线程                      │
│        │                              │  回放 SQL                       │
│        │                              ▼                                 │
│        │                       ┌─────────────┐                         │
│        │                       │  应用变更   │                         │
│        │                       │  到数据文件 │                         │
│        │                       └─────────────┘                         │
│                                                                         │
│  时序：T1                 T2        T3         T4                        │
│  ────┼─────────────────┼──────────┼──────────┼──────────▶ 时间         │
│      │ 事务提交         │          │          │                          │
│      │ (主库已提交)      │          │          │                          │
│      │                 │          │          │                          │
│      │                 │ Binlog   │          │                          │
│      │                 │ 已发送到 │          │                          │
│      │                 │ 从库     │          │                          │
│      │                 │          │          │                          │
│      │                 │          │ 从库 SQL │                          │
│      │                 │          │ 执行完成 │                          │
│      │                 │          │          │                          │
│                                                                         │
│  问题：T1~T4 时间窗口内，如果主库宕机                                     │
│       → 主库已提交的数据可能未同步到从库                                   │
│       → 切换后数据丢失                                                   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 半同步复制的通信增强

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      半同步复制通信流程                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  主库（Master）                    从库（Slave）                         │
│  ┌─────────────┐               ┌─────────────┐                         │
│  │   事务执行   │               │             │                         │
│  │  1. 写 Buffer Pool           │             │                         │
│  │  2. 写 Redo Log              │             │                         │
│  │  3. 写 Binlog                │             │                         │
│  │     (fsync)                  │             │                         │
│  │  4. 等待从库 ACK             │             │                         │
│  └─────────────┘               │             │                         │
│        │                       │             │                         │
│        │  Binlog Events        │             │                         │
│        │───────────────────────▶│             │                         │
│        │                       │             │                         │
│        │                       │  I/O 线程    │                         │
│        │                       │  写入 Relay │                         │
│        │                       │  返回 ACK   │                         │
│        │◀──────────────────────│             │                         │
│        │  (Semi-Sync ACK)      │             │                         │
│        │                       │             │                         │
│        │  5. 提交事务           │             │                         │
│        │  6. 返回"提交成功"     │             │                         │
│        │                       │             │                         │
│        │  等待超时降级          │             │                         │
│        │  (rpl_semi_sync_master_timeout)     │                         │
│        │                       │             │                         │
│                                                                         │
│  无损半同步（After Sync）模式：                                            │
│                                                                         │
│  普通半同步：InnoDB Commit → 等待 ACK → 返回成功                          │
│              问题：Commit 和 等待 ACK 之间崩溃                             │
│                     事务已提交但从库未收到                                  │
│                                                                         │
│  无损半同步：写 Binlog → 等待 ACK → InnoDB Commit → 返回成功             │
│              优势：主库 Commit 成功 = 从库已收到                            │
│                    保证 RPO=0                                             │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.3 主从复制的核心通信参数

```sql
-- 1. 半同步复制配置
-- 主库安装半同步插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';

-- 从库安装半同步插件
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

-- 启用半同步（主从都需要）
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_slave_enabled = 1;

-- 超时时间（默认 10 秒，超时降级为异步）
SET GLOBAL rpl_semi_sync_master_timeout = 10000;

-- 等待从库数量（1=至少1个，N=至少N个）
SET GLOBAL rpl_semi_sync_master_wait_for_slave_count = 1;

-- 2. 复制相关状态查看
-- 从库查看复制状态
SHOW SLAVE STATUS\G

-- 关键状态字段：
-- Slave_IO_Running: I/O 线程是否运行
-- Slave_SQL_Running: SQL 线程是否运行
-- Seconds_Behind_Master: 复制延迟（秒）
-- Last_IO_Error: I/O 线程最新错误
-- Last_SQL_Error: SQL 线程最新错误

-- 3. 监控主从通信延迟
-- 在主库创建心跳表
CREATE TABLE heartbeat (
    id INT AUTO_INCREMENT PRIMARY KEY,
    ts TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    server_id INT DEFAULT @@server_id
) ENGINE=InnoDB;

-- 主库定期更新
UPDATE heartbeat SET ts=NOW() WHERE id=1;

-- 从库计算延迟
SELECT TIMESTAMPDIFF(SECOND, ts, NOW()) AS replication_delay
FROM heartbeat;

-- 4. GTID 模式简化通信
-- 启用 GTID（主从都需要）
SET GLOBAL gtid_mode = ON;
SET GLOBAL enforce_gtid_consistency = ON;

-- 基于 GTID 配置复制（无需指定文件和位置）
CHANGE MASTER TO
    MASTER_HOST='192.168.1.100',
    MASTER_USER='repl',
    MASTER_PASSWORD='replpass',
    MASTER_AUTO_POSITION=1;
```

### 3.2 Paxos 协议在 MGR 中的应用

```
Paxos 协议核心原理：

┌─────────────────────────────────────────────────────────────────────────┐
│                    Paxos 协议基本流程                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  角色：                                                                 │
│  - Proposer：提议者（发起事务的节点）                                     │
│  - Acceptor：接受者（组内所有节点）                                        │
│  - Learner：学习者（同步最终结果）                                         │
│                                                                         │
│  MGR 中的 Paxos 流程（单主模式）：                                         │
│                                                                         │
│  1. 提议阶段（Prepare）                                                   │
│     ┌──────────┐                                                        │
│     │Proposer  │                                                        │
│     │ (Node1)  │                                                        │
│     └────┬─────┘                                                        │
│          │  Prepare(N)    N = 提案编号（递增）                           │
│          ├─────────────────────────────────────────▶┌──────────┐       │
│          │                                          │Acceptor  │       │
│          │                                          │ (Node2)  │       │
│          │                                          └──────────┘       │
│          │                                          ┌──────────┐       │
│          ├─────────────────────────────────────────▶│Acceptor  │       │
│          │                                          │ (Node3)  │       │
│          │                                          └──────────┘       │
│          │                                                                  │
│          │  Accept(N, V)    V = 提议值（事务内容）                        │
│          │  ←────────────────────────────────────────┤                 │
│          │  Promise(N)       Promise = 保证不再接受                         │
│          │                   编号小于 N 的提议                             │
│          │                                                                  │
│  2. 接受阶段（Accept）                                                   │
│          │  Accept(N, V)    如果 majority promise，                        │
│          ├─────────────────────────────────────────▶┌──────────┐       │
│          │                则发送 Accept 请求       │Acceptor  │       │
│          │                                          └──────────┘       │
│          │                                          ┌──────────┐       │
│          ├─────────────────────────────────────────▶│Acceptor  │       │
│          │                                          │ (Node3)  │       │
│          │                                          └──────────┘       │
│          │                                                                  │
│          │  Accepted(N, V)                                               │
│          │  ←────────────────────────────────────────┤                 │
│          │                                                                  │
│  3. 提交阶段（Commit）                                                   │
│          │  一旦 majority Accepted，提交事务                               │
│          │  所有节点按相同顺序执行事务                                      │
│                                                                         │
│  MGR 中的优化：                                                           │
│  - 使用 ViewId 代替 Paxos 的提案编号                                      │
│  - MGR 内部使用 XCOM（基于 Paxos 的通信库）                               │
│  - 支持三种模式：单主、多主、容错模式                                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 MGR 故障检测与恢复机制

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    MGR 故障检测与恢复                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 故障检测（Failure Detection）                                        │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  组成员健康检查                                                    │   │
│  │                                                                  │   │
│  │  ┌────────┐    PING    ┌────────┐    PING    ┌────────┐        │   │
│  │  │ Node1  │◀──────────▶│ Node2  │◀──────────▶│ Node3  │        │   │
│  │  │        │            │        │            │        │        │   │
│  │  │  OK    │            │  OK    │            │  OK    │        │   │
│  │  └────────┘            └────────┘            └────────┘        │   │
│  │                                                                  │   │
│  │  检测机制：                                                       │   │
│  │  - 每个节点周期性向其他节点发送心跳                               │   │
│  │  - 默认 1 秒一次                                                 │   │
│  │  - 连续 5 次无响应（默认）判定为不可达                            │   │
│  │  - 触发组成员变更                                                │   │
│  │                                                                  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  2. 组成员变更（Membership Change）                                      │
│                                                                         │
│  场景：Node3 故障，组成员变更                                             │
│                                                                         │
│  变更前：                    变更后：                                      │
│  ┌────────┐                 ┌────────┐                                   │
│  │ Node1  │                 │ Node1  │                                   │
│  │   ✓    │                 │   ✓    │                                   │
│  └────────┘                 └────────┘                                   │
│       │                          │                                       │
│       │    ┌────────┐            │      ┌────────┐                      │
│       ├──▶│ Node2  │◀─────────── ├──────▶│ Node2  │                      │
│       │    │   ✓    │            │       │   ✓    │                      │
│       │    └────────┘            │       └────────┘                      │
│       │    ┌────────┐            │                                       │
│       ├──▶│ Node3  │ (不可达)     │                                       │
│            │   ✗    │            │                                       │
│            └────────┘            │                                       │
│                                  ▼                                       │
│  新 View: {Node1, Node2}        新 View: {Node1, Node2}                  │
│  视图变更通过 Paxos 共识，确保一致性                                       │
│                                                                         │
│  3. 故障恢复（Recovery）                                                 │
│                                                                         │
│  当故障节点重新加入：                                                     │
│  - 节点向组内其他节点请求缺失的事务                                       │
│  - 通过分布式恢复机制同步数据                                             │
│  - 追赶完成后正式加入组                                                   │
│  - 应用透明切换，不影响业务                                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.4 MGR 配置与监控

```sql
-- 1. MGR 基本配置
[mysqld]
# 基础配置
server-id = 1
gtid-mode = ON
enforce-gtid-consistency = ON
log-bin = ON

## 4. PXC（Percona XtraDB Cluster）通信机制

### 4.1 Galera 协议原理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Galera 同步复制原理                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Galera 是一种**同步多主**复制协议：                                        │
│  - 所有节点同时接受事务                                                   │
│  - 在所有节点上执行验证（Certification）                                   │
│  - 验证通过后提交                                                       │
│                                                                         │
│  事务提交流程：                                                          │
│                                                                         │
│  1. 本地执行阶段                                                         │
│     ┌──────────┐                                                        │
│     │  Client  │                                                        │
│     └────┬─────┘                                                        │
│          │ 写请求                                                        │
│          ▼                                                              │
│     ┌──────────┐                                                        │
│     │  Node1   │                                                        │
│     │ (本地)   │                                                        │
│     │ 执行事务 │                                                        │
│     │ 生成     │                                                        │
│     │ WriteSet│                                                        │
│     └────┬─────┘                                                        │
│          │                                                              │
│  2. 复制阶段（WriteSet Replication）                                      │
│          │  Broadcast WriteSet                                          │
│          ├─────────────────────────────────────────▶┌──────────┐        │
│          │                                        │  Node2   │        │
│          │                                        │  验证   │        │
│          │                                        └────┬─────┘        │
│          │                                        ┌────┴─────┐        │
│          │                                        │  验证   │        │
│          │                                        │ 通过   │        │
│          │                                        └────┬─────┘        │
│          │                                              │               │
│          │  Broadcast WriteSet                         │               │
│          ├─────────────────────────────────────────────┼─────────────▶┌┤
│          │                                        ┌────▼─────┐        │
│          │                                        │  Node3   │        │
│          │                                        │  验证   │        │
│          │                                        └──────────┘        │
│          │                                                                  │
│  3. 提交阶段（Certification + Commit）                                   │
│          │  Certification 检查（全局排序）                                  │
│          │  ←────────────────────────────────────────                   │
│          │  Certification 通过                                             │
│          │  提交本地事务                                                   │
│          │                                                              │
│          │  Certification 失败                                            │
│          │  回滚本地事务                                                   │
│          │                                                              │
│  关键概念：                                                               │
│  - WriteSet：事务修改的行集合（包含主键和列）                               │
│  - Certification：验证 WriteSet 是否与已提交事务冲突                      │
│  - 全局事务序号（GTS）：保证所有节点以相同顺序提交                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.2 Galera 与 MGR 对比

| 维度 | Galera (PXC) | MGR |
|------|-------------|-----|
| **一致性模型** | 同步复制，强一致 | 异步/同步混合，最终一致 |
| **多主支持** | 原生多主 | 需要多主模式 |
| **冲突检测** | WriteSet Certification | Paxos 协议 + WriteSet |
| **写入吞吐** | 受最慢节点限制 | 多数派确认即可 |
| **网络要求** | 低延迟（< 5ms 推荐） | 低延迟（< 5ms 推荐） |
| **节点数** | 建议 3~5 个 | 建议 3/5/7 个 |
| **生态** | Percona/Red Hat | MySQL 官方 |

## 5. 故障检测与自动failover

### 5.1 故障检测的核心挑战

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    故障检测的经典难题                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 什么是"故障"？                                                       │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  场景：Node2 和 Node3 通信正常，但都收不到 Node1 的消息           │    │
│  │                                                                  │    │
│  │       Node1 ──?──▶ Node2   Node1 ──?──▶ Node3                   │    │
│  │                                                                  │    │
│  │  可能情况：                                                      │    │
│  │  - Node1 真的宕机了                                              │    │
│  │  - Node1 正常，但 Node1 ↔ Node2 网络断了                         │    │
│  │  - Node1 正常，但 Node1 ↔ Node3 网络断了                         │    │
│  │  - Node1 正常，但 Node2 ↔ Node3 之间的网络断了                    │    │
│  │                                                                  │    │
│  │  问题：分布式系统无法区分"节点故障"和"网络分区"                    │    │
│  │                                                                  │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                         │
│  2. 脑裂（Split Brain）问题                                              │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  问题：如果检测故障不准确，会导致多主同时写入                       │    │
│  │                                                                  │    │
│  │  正常状态：                                                      │    │
│  │       App ──▶ VIP ──▶ Node1 (主) ── 复制 ──▶ Node2 (从)          │    │
│  │                                                                  │    │
│  │  网络分区后（Node1 和 Node2 失联）：                              │    │
│  │                                                                  │    │
│  │  错误判断：两边都认为对方故障，各自选主                            │    │
│  │       App1 ──▶ VIP1 ──▶ Node1 (主)                              │    │
│  │       App2 ──▶ VIP2 ──▶ Node2 (主)  ← 数据分叉！                 │    │
│  │                                                                  │    │
│  │  解决方案：                                                      │    │
│  │  - 多数派机制：需要超过半数节点同意才能选主                        │    │
│  │  - 法定人数：确保最多只有一个多数派                                │    │
│  │  - Fencing：选主后隔离旧主                                        │    │
│  │                                                                  │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.2 多数派与法定人数

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    多数派（Quorum）机制                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  为什么需要多数派？                                                       │
│                                                                         │
│  假设 3 节点集群，网络分区成 2 个：                                       │
│                                                                         │
│  ┌──────────────┐              ┌──────────────┐                        │
│  │  Partition A │              │  Partition B │                        │
│  │              │              │              │                        │
│  │   Node1 (1)  │     网络      │   Node2 (1)  │                        │
│  │              │    断开       │   Node3 (1)  │                        │
│  │              │              │              │                        │
│  │  节点数=1    │              │  节点数=2    │                        │
│  └──────────────┘              └──────────────┘                        │
│                                                                         │
│  多数派计算：                                                             │
│  - 总节点数 N = 3                                                         │
│  - 多数派 = floor(N/2) + 1 = 2                                            │
│  - Partition A: 1 节点 < 2 ❌ 无法选主                                    │
│  - Partition B: 2 节点 >= 2 ✅ 可以选主                                   │
│                                                                         │
│  结果：只有 Partition B 能继续提供服务                                    │
│        避免脑裂！                                                         │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│  奇数节点 vs 偶数节点                                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  3 节点集群：                                                            │
│  - 容忍 1 个节点故障                                                      │
│  - 需要 2/3 节点同意才能形成多数派                                        │
│  - 可用性：66.7%                                                          │
│                                                                         │
│  4 节点集群：                                                            │
│  - 容忍 1 个节点故障                                                      │
│  - 需要 3/4 节点同意才能形成多数派                                        │
│  - 可用性：75%（但故障时更脆弱）                                          │
│                                                                         │
│  结论：生产环境优先选择**奇数节点**                                        │
│       3 节点 > 2 节点（相同故障容忍能力，更高可用性）                       │
│       5 节点 > 4 节点（相同故障容忍能力，更高可用性）                       │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│  MGR 的多数派计算                                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  MGR 使用 ViewId 作为版本号，每个新的视图变更递增                │    │
│  │                                                                  │    │
│  │  节点状态 + ViewId：                                            │    │
│  │                                                                  │    │
│  │  ViewId=5: {Node1, Node2, Node3}                                 │    │
│  │                                                                  │    │
│  │  Node3 故障后：                                                 │    │
│  │  ViewId=6: {Node1, Node2}  ← 多数派确认的新视图                  │    │
│  │                                                                  │    │
│  │  Node3 重新加入：                                               │    │
│  │  ViewId=7: {Node1, Node2, Node3}                               │    │
│  │                                                                  │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.3 MHA 故障切换流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    MHA 故障切换详细流程                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  初始状态：                                                              │
│  ┌─────────┐      Binlog       ┌─────────┐      ┌─────────┐            │

│  └─────────┘                  └─────────┘      └─────────┘            │
│       │                                                                 │
│       │  MHA Manager 监控                                                │
│       ▼                                                                 │
│  1. 故障检测                                                             │
│     - MHA Manager 每秒 ping Master                                      │
│     - 连续 3 次（默认）ping 不通，判定为故障                              │

│     │  比较各从库的 relay log 位置：                                  │ │
│     │                                                                 │ │
│     │  Slave1: Master_Log_File=mysql-bin.000100                      │ │
│     │           Read_Master_Log_Pos=12345678                         │ │
│     │                                                                 │ │
│     │  Slave2: Master_Log_File=mysql-bin.000100                      │ │
│     │           Read_Master_Log_Pos=12456789  ← 最新，选它！          │ │
│     │                                                                 │ │
│     │  如果使用 GTID：比较 Executed_Gtid_Set                           │ │
│     └────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  3. 差异补偿（Binlog Server）                                            │
│     ┌────────────────────────────────────────────────────────────────┐ │
│     │  如果候选主库的 relay log 比主库少：                            │ │
│     │                                                                 │ │
│     │  方式一（推荐）：SSH 到原主库，拉取缺失的 Binlog                  │ │
│     │    scp mysql-bin.000101 user@slave1:/var/lib/mysql/            │ │
│     │                                                                 │ │
│     │  方式二：从其他从库获取 relay log                                 │ │
│     │                                                                 │ │
│     │  如果原主库完全不可达，无法获取差异                               │ │

│     │                                                                 │ │
│     └────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  4. 提升候选主库                                                         │
│     - 在候选主库执行：STOP SLAVE;                                        │
│     - 执行：RESET SLAVE ALL;                                            │
│     - 执行：RESET MASTER;                                                │

│                                                                         │
│  5. 重新配置其他从库                                                      │
│     - 其他从库执行 CHANGE MASTER TO 指向新主库                            │
│     - 使用 GTID 自动定位                                                 │
│                                                                         │
│  6. VIP 切换（或通知配置中心）                                           │
│     - 执行 master_ip_failover_script                                     │
│     - 将 VIP 漂移到新主库                                                │
│     - 或通知注册中心更新主库地址                                          │
│                                                                         │
│  总耗时：约 20~60 秒（取决于网络和数据差异）                              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 6. 数据一致性保证

### 6.1 主从数据一致性

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    主从一致性保证机制                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 复制模式选择                                                         │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  模式          │  一致性   │  性能   │  数据丢失风险             │    │
│  │─────────────────────────────────────────────────────────────────│    │
│  │  异步复制       │  低      │  高    │  主库故障可能丢失数据     │    │
│  │  半同步复制     │  中      │  中    │  极端情况可能丢失 1 事务   │    │
│  │  全同步复制     │  高      │  低    │  无（所有从库确认）        │    │
│  │  MGR           │  高      │  中    │  无（Paxos 保证）          │    │
│  │  PXC           │  高      │  低    │  无（同步复制）            │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                         │
│  2. GTID 保证不丢数据                                                   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  GTID = source_id:transaction_id                                 │    │
│  │       ↑ 主库 UUID        ↑ 事务序号                                │    │
│  │                                                                  │    │
│  │  例子：3E11FA47-71CA-11E1-9E33-C80AA9429562:1-100               │    │
│  │                                                                  │    │
│  │  GTID 自动定位：                                                  │    │
│  │  - 从库记录已执行的事务 GTID 集合                                  │    │
│  │  - 主库记录所有事务的 GTID 集合                                    │    │
│  │  - 复制时自动跳过已执行的事务，不重复不遗漏                         │    │
│  │                                                                  │    │
│  │  切换时自动对齐：                                                  │    │
│  │  - CHANGE MASTER TO MASTER_AUTO_POSITION=1                      │    │
│  │  - 自动找到正确的 Binlog 位置                                      │    │
│  │                                                                  │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                         │
│  3. 定期数据校验                                                         │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  pt-table-checksum：在线校验主从数据一致性                        │    │
│  │                                                                  │    │
│  │  工作原理：                                                      │    │
│  │  1. 在主库计算每个表的校验和（基于数据内容）                       │    │
│  │  2. 通过复制在从库计算相同的数据校验和                              │    │
│  │  3. 比较主从校验和，检测不一致                                      │    │
│  │                                                                  │    │
│  │  使用示例：                                                      │    │
│  │  pt-table-checksum --user=root --ask-pass \                     │    │
│  │      --databases=myapp --tables=orders,users \                  │    │
│  │      h=master_host                                              │    │
│  │                                                                  │    │
│  │  结果解读：                                                      │    │
│  │  - DIFFS=0：一致                                                 │    │
│  │  - DIFFS=1：不一致                                               │    │
│  │                                                                  │    │
│  │  修复工具：pt-table-sync                                         │    │
│  │  pt-table-sync --print --sync-to-master \                       │    │
│  │      h=slave_host,D=myapp,t=orders                              │    │
│  │                                                                  │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.2 多活架构的一致性挑战

│       │    应用同时写入两个主库      │                                   │
│       ▼                              ▼                                   │
│  ┌─────────────────────────────────────────────────────────────┐       │
│  │  冲突场景：                                                   │       │
│  │                                                              │       │
│  │  T1 时刻：DC1 的用户 A 转账 100 元（余额 1000 → 900）         │       │
│  │  T2 时刻：DC2 的用户 A 转账 50 元（余额 1000 → 950）          │       │
│  │                                                              │       │
│  │  同步后：                                                     │       │
│  │  - DC1 收到 DC2 的更新：1000 - 50 = 950                       │       │
│  │  - DC2 收到 DC1 的更新：1000 - 100 = 900                      │       │
│  │                                                              │       │
│  │  结果：DC1 余额 900，DC2 余额 950，不一致！                    │       │
│  │                                                              │       │
│  └─────────────────────────────────────────────────────────────┘       │
│                                                                         │
│  解决方案：                                                              │
│                                                                         │
│  1. 应用层解决（推荐）                                                   │
│     - 按用户/地域划分写入权限                                             │
│     - 同一用户只允许在一个 DC 写入                                         │
│                                                                         │
│  2. 冲突检测与处理                                                       │
│     - Last Write Wins（LWW）：以时间戳为准，保留最新                       │
│     - 应用自定义冲突解决策略                                             │
│                                                                         │
│  3. 分布式事务                                                           │
│     - 使用 XA 分布式事务（两阶段提交）                                    │
│     - 使用 3PC 或其他分布式一致性协议                                     │
│                                                                         │
│  4. 最终一致性方案                                                       │
│     - 接受短暂不一致                                                     │
│     - 通过异步消息纠正                                                   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 8. 面试要点速查

| 问题 | 核心答案 |
|------|---------|
| 主从复制有哪几种模式？ | 异步复制（可能丢数据）、半同步复制（至少一个从库确认）、全同步复制（所有从库确认） |
| 半同步复制和无损半同步的区别？ | 普通半同步在 InnoDB Commit 后等待 ACK；无损半同步在写 Binlog 后等待 ACK，保证主库提交时从库已收到 |
| MGR 使用什么协议保证一致性？ | Paxos 分布式共识协议，通过多数派机制保证数据一致 |
| 什么是脑裂问题？ | 多主同时写入导致数据分叉，通过多数派机制避免 |
| 为什么节点数通常选择奇数？ | 奇数节点能用更少的节点形成多数派，可用性更高 |
| 网络分区时怎么处理？ | 少数派自动停止写入，等待多数派恢复，避免脑裂 |
| MGR 和 PXC 有什么区别？ | MGR 是 MySQL 官方方案，基于 Paxos；PXC 基于 Galera 协议，原生支持多主 |
| 如何保证主从数据一致性？ | 使用 GTID 避免位置错乱、定期用 pt-table-checksum 校验、使用半同步复制 |
