# 主从复制原理

> 主从复制是 MySQL 高可用和水平扩展的基础设施。理解其原理，有助于排查延迟、数据不一致、主库切换等生产问题，也是理解读写分离、MGR 等高级架构的前提。

---

## 1. 主从复制的核心价值

| 目标 | 说明 |
|------|------|
| **高可用（HA）** | 主库故障时，快速切换到从库，降低 MTTR（平均故障恢复时间） |
| **读写分离** | 主库处理写请求，从库分担读请求，提升系统读并发能力 |
| **数据备份** | 在从库执行备份，不影响主库性能 |
| **异地容灾** | 跨机房部署从库，实现异地数据冗余 |

---

## 2. 复制核心组件

### 2.1 三大线程

```
主库                              从库
 │                                 │
 │  Binlog Dump 线程               │  I/O 线程
 │  ┌──────────────────┐           │  ┌──────────────┐
 │  │ 读取 Binlog 文件  │──────────▶│  │ 接收 Binlog  │
 │  │ 发送给从库 I/O 线程│           │  │ 写入 Relay Log│
 │  └──────────────────┘           │  └──────────────┘
 │  （每个从库连接对应一个）            │        │
                                   │        ▼
                                   │  ┌──────────────┐
                                   │  │ Relay Log    │
                                   │  │（中继日志缓冲）│
                                   │  └──────────────┘
                                   │        │
                                   │  SQL 线程（或并行复制线程）
                                   │  ┌──────────────┐
                                   │  │ 读取 Relay Log│
                                   │  │ 回放 SQL 事件 │
                                   │  └──────────────┘
```

| 线程 | 所在 | 职责 |
|------|------|------|
| **Binlog Dump 线程** | 主库 | 每当从库连接，主库创建此线程，持续读取并推送 Binlog 事件 |
| **I/O 线程** | 从库 | 连接主库，接收 Binlog 事件，写入本地 Relay Log |
| **SQL 线程** | 从库 | 读取 Relay Log，解析并回放 SQL，更新从库数据 |

### 2.2 Relay Log（中继日志）

- 格式与 Binlog 完全相同，位于从库磁盘
- 解耦了"拉取 Binlog"和"回放 SQL"两个操作，I/O 线程和 SQL 线程可独立工作
- 回放完成后自动清理（`relay_log_purge=ON`）

## 3. 复制完整流程

```
① 主库执行事务，写入 Binlog（两阶段提交保证 Redo Log 和 Binlog 一致）

② 从库 I/O 线程连接主库，请求从某个 Binlog 位置（File + Position 或 GTID）开始的事件

③ 主库 Binlog Dump 线程读取 Binlog，发送事件给从库 I/O 线程

④ 从库 I/O 线程将接收到的事件追加写入 Relay Log，并更新 master.info（记录同步位置）

⑤ 从库 SQL 线程读取 Relay Log，回放事件，更新数据，记录位置到 relay-log.info

⑥ Relay Log 被 SQL 线程消费完后，Purge 线程自动清理
```

## 4. 三种复制模式（按数据安全强度）

| 模式 | 主库何时返回成功 | 数据安全 | 性能 | 适用场景 |
|------|----------------|---------|------|---------|
| **异步复制** | 主库写完 Binlog 即返回 | 低（主库崩溃可能丢数据） | 最高 | 对一致性要求低的场景 |
| **半同步复制** | 等待**至少一个从库**写入 Relay Log 后返回 | 较高（至少一份副本） | 中 | 常见业务场景 |
| **全同步复制** | 等待**所有从库**执行完成后返回 | 最高 | 最低 | 极少使用，性能不可接受 |

### 无损半同步复制（MySQL 5.7+）

传统半同步在主库 **存储引擎提交后** 才等待从库确认，如果主库在等待期间宕机，事务已提交但从库可能没收到。

**无损半同步**改为在主库**存储引擎提交前**等待从库确认（after_sync 模式）：

```
普通半同步：
  InnoDB Commit → 等待从库ACK → 返回客户端成功
  问题：Commit 和 等待ACK 之间崩溃 → 数据已提交但从库未收到

无损半同步（After Sync）：
  写 Binlog → 等待从库ACK → InnoDB Commit → 返回客户端成功
  保证：只要主库提交，从库必定已收到
```

```sql
-- 启用半同步复制
-- 主库
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_master_timeout = 10000;  -- 超时降级为异步（毫秒）

-- 从库
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
```

## 5. 基于 GTID 的复制

### 5.1 什么是 GTID

**GTID（Global Transaction Identifier）** 是每个事务的全局唯一标识：

```
GTID 格式：source_id:transaction_id
示例：3E11FA47-71CA-11E1-9E33-C80AA9429562:1-100
       ↑ 主库 UUID                            ↑ 事务序号范围
```

### 5.2 GTID 复制的优势

| 传统复制（File + Position） | GTID 复制 |
|---------------------------|-----------|
| 切换主库时需手动找到新主库的 Binlog 位置 | 自动定位，无需手动指定位置 |
| 运维复杂，容易出错 | 简化故障转移，自动跳过已执行事务 |
| 多源复制较难处理 | 天然支持多源复制 |

### 5.3 GTID 配置

```sql
-- my.cnf 配置（主从均需）
[mysqld]
gtid_mode = ON
enforce_gtid_consistency = ON
log_bin = ON
server_id = 1   # 主从不同

-- 查看已执行的 GTID 集合
SHOW MASTER STATUS\G  -- Executed_Gtid_Set 字段

-- 基于 GTID 的从库配置
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',
  MASTER_USER='repl',
  MASTER_PASSWORD='replpass',
  MASTER_AUTO_POSITION=1;   -- 自动使用 GTID 定位，无需指定文件和位置
```

## 6. 主从延迟问题

### 6.1 主从延迟的来源

```
主库写入速度 > 从库 SQL 线程回放速度 → 延迟累积

主要原因：
1. 从库硬件弱于主库（CPU/磁盘 I/O 差距）
2. 大事务（主库执行100秒的大事务，从库也要执行100秒）
3. DDL 操作（ALTER TABLE 会导致长时间表锁，从库串行执行）
4. 单线程回放（MySQL 5.6 之前 SQL 线程是单线程）
5. 网络延迟（跨机房、带宽不足）
```

### 6.2 监控主从延迟

```sql
-- 查看从库状态（核心指标）
SHOW SLAVE STATUS\G

-- 关注字段：
-- Seconds_Behind_Master: 从库落后主库的秒数（近似值，不完全准确）
-- Master_Log_File / Read_Master_Log_Pos: 从库 I/O 线程读到的位置
-- Relay_Master_Log_File / Exec_Master_Log_Pos: 从库 SQL 线程执行到的位置
-- Last_Error: 最近的错误信息
```

> ⚠️ `Seconds_Behind_Master=0` 不代表无延迟，它是 SQL 线程基于 Relay Log 中事件时间戳计算的，可能有误差。使用 **`pt-heartbeat`** 更精确：

```bash
# 主库持续写心跳
pt-heartbeat --user=root --ask-pass --create-table \
  -D test --update --daemonize

# 从库检测延迟
pt-heartbeat --user=root --ask-pass -D test --monitor --host=slave_host
```

### 6.3 解决主从延迟

**方案1：并行复制（MTS，Multi-Threaded Slave）**

```sql
-- MySQL 5.7+（推荐配置）
-- 在从库 my.cnf 中配置
slave_parallel_workers = 8          # 并行 SQL 线程数
slave_parallel_type = LOGICAL_CLOCK # 基于逻辑时钟（组提交中的事务可并行）
slave_preserve_commit_order = ON    # 保证提交顺序（配合 LOGICAL_CLOCK 使用）

-- MySQL 8.0+ WRITESET 模式（更高并行度）
slave_parallel_type = WRITESET      # 基于行级写集合，无依赖的事务并行执行
```

| 并行类型 | MySQL 版本 | 原理 | 并行粒度 |
|---------|-----------|------|---------|
| `DATABASE` | 5.6 | 不同库的事务并行 | 粗（单库无法并行） |
| `LOGICAL_CLOCK` | 5.7 | 同一 Binlog 组内事务并行 | 中 |
| `WRITESET` | 8.0 | 写集合无冲突的事务并行 | 细（推荐） |

**方案2：拆分大事务**

```sql
-- 原：一次更新100万行（主从都慢）
UPDATE orders SET status=2 WHERE created_at < '2024-01-01';

-- 优化：分批处理，每批1万行
SET @batch_size = 10000;
REPEAT
  UPDATE orders SET status=2 
  WHERE created_at < '2024-01-01' AND status=1
  LIMIT @batch_size;
UNTIL ROW_COUNT() = 0 END REPEAT;
```

**方案3：从库强制读主库（业务层处理）**

```java
// Spring AbstractRoutingDataSource 实现：
// 写操作后短时间内（如2秒）强制读主库
@Transactional
public Order createOrder(OrderDTO dto) {
    Order order = orderRepository.save(dto.toEntity());
    // 设置标记：接下来2秒内读主库
    DataSourceContextHolder.forceMaster(2, TimeUnit.SECONDS);
    return order;
}
```

## 7. 保证主从数据一致性

### 7.1 常规手段

1. **从库设为只读**（防止误操作）：
   ```sql
   SET GLOBAL read_only = ON;              -- 防普通用户写入
   SET GLOBAL super_read_only = ON;        -- 连 SUPER 权限用户也禁止写入（MySQL 5.7.8+）
   ```

2. **使用 GTID 复制**：GTID 确保每个事务只执行一次，避免位置错乱导致的重复或遗漏

3. **定期校验**：使用 `pt-table-checksum` 定期核对主从数据一致性

4. **不一致修复**：使用 `pt-table-sync` 将从库数据修复为与主库一致

### 7.2 pt-table-checksum 使用

```bash
# 在主库执行校验（对从库无侵入性影响）
pt-table-checksum \
  --user=root --ask-pass \
  --databases=my_db \
  --tables=orders,users \
  --recursion-method=hosts \
  h=master_host

# 结果解读：
# DIFFS=0 → 一致   DIFFS=1 → 有差异，需修复
```

## 8. 读写分离的延迟问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 写后立即读，从库未同步 | 主从复制延迟 | 写后短期强制读主库；或使用半同步复制降低延迟窗口 |
| 同一事务内的查询走了从库 | 中间件/AOP 路由不当 | 事务内所有操作（含只读）走主库 |
| 从库负载不均 | 请求分发策略单一 | 轮询/加权轮询/最少连接等负载均衡策略 |
| 从库宕机影响读服务 | 无健康检查 | 中间件（ShardingSphere/MyCat）自动摘除故障从库 |

## 9. 主从复制常见问题排查

### 9.1 复制报错（Duplicate entry）

```sql
-- 查看错误详情
SHOW SLAVE STATUS\G  -- Last_SQL_Error 字段

-- 常见原因：在从库直接写入数据，导致主库同步过来时主键冲突
-- 临时跳过错误（谨慎使用！）
STOP SLAVE;
SET GLOBAL SQL_SLAVE_SKIP_COUNTER = 1;  -- 跳过1个事件
START SLAVE;

-- 从库只读 + GTID 是根本解决方案
```

### 9.2 延迟报警处理

```
排查步骤：
1. SHOW SLAVE STATUS\G → 确认延迟值和错误信息
2. SHOW PROCESSLIST → 查看 SQL 线程在执行什么
3. 找到慢 SQL → EXPLAIN 分析执行计划
4. 检查是否有大事务/DDL 占用
5. 考虑开启并行复制或硬件升级
```
