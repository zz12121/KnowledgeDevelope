# 分库分表

> 分库分表是应对单机数据库容量和性能瓶颈的最终手段，但同时也是复杂度最高的架构决策之一。在采用分库分表之前，应优先穷尽索引优化、缓存、读写分离等方案。

---

## 1. 什么时候考虑分库分表

```
单机数据库性能优化路径（优先级从高到低）：

1. 索引优化         → 消除慢查询（成本最低）
2. SQL 重写         → 避免全表扫描
3. 缓存层（Redis）  → 热点数据前置
4. 读写分离         → 解决读并发压力
5. 垂直分表         → 宽表拆分，优化 I/O
6. 历史数据归档     → 冷热分离，缩减在线数据量
7. 分区表           → 单机分区，适合时序数据
8. 分库分表         ← 最后手段，复杂度大幅上升
```

**触发分库分表的信号**：
- 单表行数 > 2000万，且持续增长，EXPLAIN 中 rows 巨大
- 单库并发写入导致锁等待时间居高不下
- 单机磁盘容量接近极限（TB 级以上）
- 读写分离已经到极限，主库写入依然是瓶颈

## 2. 四种拆分方式

### 2.1 垂直分库（Vertical Sharding by Database）

**按业务模块**将不同的表拆分到不同的数据库实例。

```
拆分前：
single_db
├── users
├── products
├── orders
├── payments
└── messages

拆分后：
user_db        product_db      order_db       payment_db
└── users       └── products    ├── orders     └── payments
                                └── order_items
```

**效果**：
- 不同业务的数据库实例相互隔离，某个业务的高负载不影响其他业务
- 符合微服务拆分思路，各服务独占自己的数据库
- **缺点**：单库内表数据量依然可能很大，未解决单表大数据量问题

### 2.2 水平分库（Horizontal Sharding by Database）

**将同一张表的数据**按分片键分散到多个数据库实例。

```
orders 表（1000万行）按 user_id % 4 水平分库：

order_db_0: user_id % 4 = 0 的订单
order_db_1: user_id % 4 = 1 的订单
order_db_2: user_id % 4 = 2 的订单
order_db_3: user_id % 4 = 3 的订单
```

**效果**：每个库数据量降低为原来的 1/N，写入并发压力分散。

### 2.3 垂直分表（Vertical Splitting by Column）

**将一张宽表按字段使用频率**拆分为主表 + 扩展表。

```
user（宽表，20个字段）：
id | name | phone | email | avatar_url | bio | address | birthday | ...

拆分为：
user_basic（高频字段）：    user_profile（低频字段）：
id | name | phone | email    user_id | avatar_url | bio | address | birthday
```

**目的**：减少主表行宽，使单数据页能容纳更多行，提升缓存命中率和全表扫描效率。

### 2.4 水平分表（Horizontal Splitting by Row）

**将单张大表的行**按规则拆分到多张结构相同的表。

```
orders_2024  orders_2025  orders_2026
（按年分表）

或：
orders_0  orders_1  orders_2  orders_3
（按 order_id % 4 分表）
```

## 3. 分片策略选型

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **范围分片** | 按分片键的连续区间（时间/ID范围） | 扩容只加新分片，无需迁移 | 热点问题（最新分片写入集中） | 时序数据（日志、订单）|
| **哈希取模** | `hash(key) % N` | 数据均匀分布 | 扩容时需全量重新分布（成本极高） | 无顺序要求的随机读写 |
| **一致性哈希** | 哈希环，节点增减只影响相邻数据 | 扩容时数据迁移量小（~1/N） | 实现复杂，需虚拟节点保证均匀 | 分布式架构，扩展性优先 |
| **地理分片** | 按用户地区分配 | 就近部署降低延迟，数据合规 | 各地区数据量可能不均 | 本地生活、出行服务 |

### 分片键选型原则

1. **数据均匀**：分片键的基数要足够大，取值要分散（不能用性别、状态等低基数字段）
2. **支持高频查询**：大部分查询条件中都包含分片键，避免跨分片扫描
3. **不可变性**：分片键的值不能修改（否则需要跨分片迁移数据）
4. **业务相关性**：
   - 电商订单系统：以 `buyer_id` 分片（支持买家查单）
   - 同时支持卖家查单：通过数据双写（`seller_id` 分片的副本）或 ES 异构索引

## 4. 分库分表后的核心挑战

### 4.1 分布式 ID 生成

单机自增 ID 在多库多表场景下会重复，必须用分布式 ID：

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| UUID | 简单，本地生成 | 无序（B+树页分裂频繁），存储大（36字符） | ❌ 不推荐 |
| 数据库序列表 | ID 有序 | 单点故障，写入性能瓶颈 | ⚠️ 小规模可用 |
| **雪花算法（Snowflake）** | 高性能，趋势递增，64位Long | 时钟回拨问题需处理 | ✅ 首选 |
| 美团 Leaf-Segment | 数据库号段预分配，高可用 | 依赖 DB | ✅ 推荐 |
| 百度 UidGenerator | 基于雪花算法，解决时钟回拨 | 实现复杂 | ✅ 推荐 |

**雪花算法（64位）结构**：

```
┌────────────┬──────────────────┬────────────────┬────────────────┐
│  1位符号位 │  41位时间戳（ms） │ 10位工作机器ID  │ 12位序列号     │
│  固定为0   │（约69年）         │（最多1024节点） │（每ms最多4096个）│
└────────────┴──────────────────┴────────────────┴────────────────┘
```

```java
// 雪花算法核心实现
public class SnowflakeIdGenerator {
    private final long workerId;
    private final long datacenterId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;
    
    // 位移常量
    private static final long SEQUENCE_BITS = 12L;
    private static final long WORKER_ID_BITS = 5L;
    private static final long DATACENTER_ID_BITS = 5L;
    private static final long MAX_SEQUENCE = ~(-1L << SEQUENCE_BITS);  // 4095
    
    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        
        // 时钟回拨处理
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("Clock moved backwards!");
        }
        
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & MAX_SEQUENCE;
            if (sequence == 0) {
                timestamp = waitNextMillis(lastTimestamp);  // 等到下一毫秒
            }
        } else {
            sequence = 0L;
        }
        
        lastTimestamp = timestamp;
        // 组装64位 ID
        return ((timestamp - EPOCH) << 22) | (datacenterId << 17) 
             | (workerId << 12) | sequence;
    }
}
```

### 4.2 跨分片查询与分页

**问题**：`SELECT * FROM orders ORDER BY created_at DESC LIMIT 0, 10` 在4个分片上如何执行？

```
方案1：全局视野法（性能差）
  → 每个分片各取 LIMIT 10
  → 内存汇总排序取前10
  → 偏移量越大，性能越差（LIMIT 1000000, 10 需从每个分片取 1000010 条）

方案2：游标分页（推荐，但限制跳页）
  → 记录上一页最后一条记录的 created_at 和 id
  → 下一页：WHERE created_at < ? OR (created_at = ? AND id < ?)
  → 只能"下一页"，不能跳页

方案3：异构索引（大厂常用）
  → 将排序、检索字段同步到 ES
  → 在 ES 中完成分页查询，获取主键列表
  → 再按主键回 MySQL 分片查询完整数据
```

### 4.3 跨分片 JOIN

**原则：分库分表后应尽量避免 JOIN，通过以下策略规避**：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **全局广播表** | 字典表在每个分片冗余一份 | 地区、类目等小型字典表 |
| **字段冗余** | 订单表冗余商品名称，避免关联商品表 | 查询时需要的关联字段不多 |
| **应用层 JOIN** | 先查A，再用结果查B，内存拼接 | 关联数据量不大 |
| **数据异构** | 双写到 ES，在 ES 中完成关联 | 复杂检索场景 |
| **表分组** | 关联紧密的表用相同分片键，保证在同一分片 | 订单表 + 订单明细表 |

### 4.4 分布式事务

分库分表后，跨分片的事务不能再用单机事务保证，需要引入分布式事务方案：

| 方案 | 原理 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **XA 事务** | 数据库原生 2PC | 强一致 | 性能差，锁时间长 | 小并发，强一致需求 |
| **TCC** | Try-Confirm-Cancel 业务补偿 | 性能好，可控 | 业务侵入性强 | 金融类高并发场景 |
| **Seata AT** | 基于 Undo Log 的自动补偿 | 对业务代码侵入小 | 额外开销，需 TC 服务 | 通用 OLTP |
| **消息事务** | 本地事务 + 消息最终一致 | 高可用，高并发 | 最终一致，有延迟 | 异步业务场景 |
| **SAGA** | 长事务拆分为一系列本地事务 + 补偿 | 适合长业务流程 | 补偿逻辑复杂 | 工作流、订单系统 |

## 5. 常用分库分表中间件

| 中间件 | 模式 | 特点 |
|--------|------|------|
| **ShardingSphere-JDBC** | Client（Jar包嵌入） | 轻量，高性能，Java 生态首选，无额外部署 |
| **ShardingSphere-Proxy** | Proxy（独立服务） | 语言无关，对应用透明，支持多语言 |
| **MyCat** | Proxy | 功能完善，社区曾活跃，近年维护减少 |
| **TDDL** | Client | 阿里开源，但社区版资料少，依赖自身配置系统 |

### ShardingSphere-JDBC 配置示例

```yaml
# application.yml（Spring Boot）
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://localhost:3306/order_db_0
      ds1:
        type: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://localhost:3306/order_db_1
    rules:
      sharding:
        tables:
          orders:
            actual-data-nodes: ds${0..1}.orders_${0..3}
            database-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: db-inline
            table-strategy:
              standard:
                sharding-column: order_id
                sharding-algorithm-name: table-inline
        sharding-algorithms:
          db-inline:
            type: INLINE
            props:
              algorithm-expression: ds${user_id % 2}
          table-inline:
            type: INLINE
            props:
              algorithm-expression: orders_${order_id % 4}
```

## 6. 数据迁移策略（双写方案）

将线上数据从单库迁移到分库分表，必须零停机：

```
迁移步骤（双写过渡）：

1. 全量同步：将旧库历史数据按分片规则写入新分库（可用 Canal + 自定义路由）

2. 开启双写：新写入操作同时写旧库和新分库（灰度流量先切5%）

3. 校验一致性：对比旧库和新分库数据，修复不一致

4. 切读流量：将查询流量逐步从旧库切到新分库（10% → 50% → 100%）

5. 停止双写：全部流量切到新分库后，停止向旧库写入

6. 旧库保留观察：保留旧库作为数据回滚的后路，观察稳定后下线
```
