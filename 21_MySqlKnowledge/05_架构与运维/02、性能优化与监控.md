# 性能优化与监控

> MySQL 性能优化是一个系统工程，需要从硬件、配置、架构、SQL 四个层次分层治理。监控是优化的前提——没有监控就无法发现问题，没有数据就无法验证优化效果。

---

## 1. 分层优化体系

```
MySQL 性能优化层次（由底向上）：

┌─────────────────────────────────────────┐
│  应用层优化                               │ ← 连接池配置、事务管理、ORM优化
├─────────────────────────────────────────┤
│  SQL 与索引层                             │ ← 覆盖索引、避免全表扫描（收益最大）
├─────────────────────────────────────────┤
│  架构层                                  │ ← 读写分离、分库分表、缓存
├─────────────────────────────────────────┤
│  MySQL 配置层                             │ ← Buffer Pool、日志刷盘策略
├─────────────────────────────────────────┤
│  硬件与 OS 层                             │ ← SSD、内存、NUMA、文件系统
└─────────────────────────────────────────┘
```

## 2. 关键配置参数调优

### 2.1 内存相关（影响最大）

```ini
# ============ InnoDB Buffer Pool ============
# 最重要的参数，专用服务器设为物理内存的 70%-80%
innodb_buffer_pool_size = 8G

# 多实例减少内存分配锁竞争（BP > 1GB 时设置，建议每实例 ≥ 1GB）
innodb_buffer_pool_instances = 8

# 启动时自动加载上次关闭前的热页列表（快速预热）
innodb_buffer_pool_load_at_startup = ON
innodb_buffer_pool_dump_at_shutdown = ON
innodb_buffer_pool_dump_pct = 25  # 保存最热的 25% 数据

# ============ 连接级缓冲（每连接独立分配，不要设太大！）============
sort_buffer_size = 2M
join_buffer_size = 2M
read_buffer_size = 1M
read_rnd_buffer_size = 1M
```

### 2.2 I/O 相关

```ini
# ============ Redo Log ============
# 日志文件大小（默认 48MB，生产建议 1-4GB）
innodb_log_file_size = 1G
innodb_log_files_in_group = 2

# 事务提交刷盘策略
# 1=每次提交 fsync（最安全） 2=写 OS 缓存（每秒 fsync） 0=性能最高但有风险
innodb_flush_log_at_trx_commit = 1  # 核心业务
# innodb_flush_log_at_trx_commit = 2  # 可接受1秒丢失的场景

# I/O 能力（告诉 InnoDB 磁盘的 IOPS 能力）
innodb_io_capacity = 4000       # SSD
# innodb_io_capacity = 400      # HDD
innodb_io_capacity_max = 20000  # 突发 IOPS 上限

# 每次刷多少脏页（与 io_capacity 配合）
innodb_lru_scan_depth = 2048

# ============ Binlog ============
# 每次提交 fsync（与 innodb 2PC 配合）
sync_binlog = 1
# 允许 Row 格式 Binlog 只记录变更列（减少日志量）
binlog_row_image = MINIMAL
```

### 2.3 连接与并发

```ini
# 最大连接数（根据应用并发需求设置，注意每个连接消耗内存）
max_connections = 1000

# 线程缓存（避免频繁创建销毁线程）
thread_cache_size = 100  # 建议 max_connections 的 10%

# 空闲连接超时（防止大量 Sleep 连接积压）
wait_timeout = 300
interactive_timeout = 300

# 行锁等待超时（默认 50 秒太长）
innodb_lock_wait_timeout = 10

# ============ 表缓存 ============
# 打开表的缓存数量
table_open_cache = 4000
table_definition_cache = 2000
```

### 2.4 Buffer Pool 深度调优

Buffer Pool 使用改进的 LRU 算法（分为 New/Old 两个区域）：

```
Buffer Pool LRU 链表：

New Sublist (63%)              Old Sublist (37%)
│←──────────────────────│──────────────────────→│
Head（最热）                                  Tail（淘汰端）
                              ↑
                        midpoint（新数据页首次加载到此处）

晋升条件：
  首次加载 → 插入 Old 区头部（midpoint）
  在 innodb_old_blocks_time（默认1秒）内再次被访问 → 晋升 New 区头部
  
设计目的：
  防止一次性全表扫描/备份操作将真正的热页从 New 区驱逐
```

```sql
-- 查看 Buffer Pool 命中率（目标 > 99%）
SELECT 
  (1 - SUM(CASE WHEN event_name = 'innodb_buffer_pool_reads' 
                THEN sum_number_of_bytes_read ELSE 0 END) /
   SUM(CASE WHEN event_name = 'innodb_buffer_pool_read_requests'
            THEN count_read ELSE 0 END)) * 100 AS bp_hit_rate
FROM performance_schema.file_summary_by_event_name
WHERE event_name LIKE 'wait/io/file/innodb%';

-- 更简单的方法
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read%';
-- Innodb_buffer_pool_reads / Innodb_buffer_pool_read_requests < 1%（物理读比例）

-- 查看各 BP 实例状态
SELECT POOL_ID, DATABASE_PAGES, OLD_DATABASE_PAGES, 
       MODIFIED_DATABASE_PAGES, PAGES_MADE_YOUNG
FROM information_schema.INNODB_BUFFER_POOL_STATS;
```

## 3. 慢查询定位与分析

### 3.1 开启慢查询日志

```sql
-- 动态开启（不重启）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;       -- 超过1秒（生产可设 0.5 或更低）
SET GLOBAL log_queries_not_using_indexes = ON;  -- 未使用索引的也记录
SET GLOBAL log_throttle_queries_not_using_indexes = 10;  -- 每分钟最多记录10条（避免日志爆炸）
SET GLOBAL min_examined_row_limit = 1000;  -- 扫描行数 < 1000 的不记录

-- 持久化配置（写 my.cnf）
[mysqld]
slow_query_log = ON
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
log_queries_not_using_indexes = ON
```

### 3.2 分析慢日志

```bash
# mysqldumpslow（内置，简单统计）
mysqldumpslow -s t -t 20 /var/log/mysql/slow.log    # 按总耗时排序，前20

# pt-query-digest（Percona Toolkit，专业分析）
pt-query-digest /var/log/mysql/slow.log > /tmp/slow_report.txt

# 输出解读（以 pt-query-digest 为例）：
# Rank: SQL 排名
# Response time: 总耗时及占比
# Calls: 执行次数
# R/Call: 平均响应时间（越小越好）
# V/M: 响应时间变化系数（越大说明越不稳定）
# Item: SQL 指纹（模板化的 SQL）
```

### 3.3 EXPLAIN 执行计划分析

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status = 'pending';

-- 关键字段解读：
-- type（访问类型，越靠前越好）：
--   system > const > eq_ref > ref > range > index > ALL
--   ref：索引范围扫描（多行）
--   range：范围扫描（BETWEEN, >, <）
--   index：全索引扫描（不读数据行）
--   ALL：全表扫描（最差）

-- key：实际使用的索引（NULL = 没用索引，需要优化！）
-- rows：预估扫描行数（越少越好）
-- Extra 重要值：
--   Using index：覆盖索引，无需回表（最佳）
--   Using where：WHERE 条件过滤
--   Using filesort：内存/磁盘排序（需要优化 ORDER BY 索引）
--   Using temporary：创建临时表（需要优化 GROUP BY/DISTINCT）
--   Using index condition：ICP 索引条件下推（好）

-- MySQL 8.0.18+ 可获取实际执行数据（而非预估）
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 100;
```

**key_len 计算**（判断联合索引使用了哪些列）：

```
字段类型         key_len 计算
INT NOT NULL   → 4字节
INT NULL       → 4+1=5字节（额外1字节标记NULL）
BIGINT         → 8字节
VARCHAR(n) UTF8MB4 NOT NULL → n*4 + 2 字节（2字节记录变长长度）
VARCHAR(n) UTF8MB4 NULL     → n*4 + 2 + 1 字节
```

## 4. 性能监控体系

### 4.1 核心监控指标

| 监控维度 | 核心指标 | 健康标准 | 查询命令 |
|----------|---------|---------|---------|
| **吞吐量** | QPS、TPS | 稳定，无骤降 | `SHOW GLOBAL STATUS LIKE 'Queries'` |
| **响应时间** | P95/P99 延迟 | P99 < 100ms | APM 工具（SkyWalking/Prometheus） |
| **缓冲池** | BP 命中率 | > 99% | `SHOW ENGINE INNODB STATUS` |
| **锁竞争** | 行锁等待次数 | 尽量趋0 | `SHOW GLOBAL STATUS LIKE 'Innodb_row_lock%'` |
| **连接** | 当前连接数 / 最大连接数 | < 80% | `SHOW GLOBAL STATUS LIKE 'Threads_connected'` |
| **I/O** | 磁盘读写 IOPS、延迟 | await < 10ms | iostat -x |
| **复制** | 主从延迟 | < 1秒 | `SHOW SLAVE STATUS` |

### 4.2 InnoDB Status 解读

```sql
SHOW ENGINE INNODB STATUS\G

-- 关键段落：
-- SEMAPHORES：锁信号量等待（持续大量等待说明并发争用严重）
-- LATEST DETECTED DEADLOCK：最近死锁信息（SQL、事务、锁链）
-- TRANSACTIONS：当前活跃事务列表
-- FILE I/O：I/O 线程状态
-- BUFFER POOL AND MEMORY：
--   Buffer pool size: 总页数
--   Free buffers: 空闲页数
--   Database pages: 数据页数
--   Modified db pages: 脏页数（持续偏高说明刷盘跟不上写入）
--   Buffer pool hit rate: 命中率（目标 1000/1000）
-- LOG：
--   Log sequence number: 最新 LSN
--   Log flushed up to: 已刷盘 LSN
--   Last checkpoint at: 最后 Checkpoint 位置
--   差值大 = 待刷新数据多 = I/O 压力大
-- ROW OPERATIONS：
--   Number of rows inserted/updated/deleted/read
```

### 4.3 Performance Schema（高级监控）

```sql
-- 查找最耗时的 SQL 语句（Top 10）
SELECT DIGEST_TEXT, COUNT_STAR AS calls,
       ROUND(SUM_TIMER_WAIT / 1e12, 3) AS total_seconds,
       ROUND(AVG_TIMER_WAIT / 1e9, 3) AS avg_ms,
       SUM_ROWS_EXAMINED AS rows_examined
FROM performance_schema.events_statements_summary_by_digest
WHERE SCHEMA_NAME = 'my_db'
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;

-- 查找锁等待最多的表
SELECT OBJECT_NAME, COUNT_READ, SUM_TIMER_READ,
       COUNT_WRITE, SUM_TIMER_WRITE
FROM performance_schema.table_lock_waits_summary_by_table
WHERE OBJECT_SCHEMA = 'my_db'
ORDER BY SUM_TIMER_WRITE DESC
LIMIT 10;

-- 查找未使用的索引
SELECT OBJECT_SCHEMA, OBJECT_NAME, INDEX_NAME
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE OBJECT_SCHEMA = 'my_db'
  AND INDEX_NAME != 'PRIMARY'
  AND COUNT_STAR = 0
ORDER BY OBJECT_NAME;

-- 查找当前线程在等什么（实时诊断）
SELECT t.THREAD_ID, t.PROCESSLIST_STATE,
       ew.EVENT_NAME, ew.TIMER_WAIT/1e9 wait_ms
FROM performance_schema.events_waits_current ew
JOIN performance_schema.threads t ON ew.THREAD_ID = t.THREAD_ID
WHERE t.PROCESSLIST_STATE IS NOT NULL
ORDER BY ew.TIMER_WAIT DESC
LIMIT 10;
```

### 4.4 五步性能瓶颈定位法

```
1. 确认现象
   → 监控告警：QPS下降、P99 延迟升高、连接数飙升
   → 用户反馈：页面加载变慢、超时

2. 全局资源检查
   → top: CPU 使用率（%us/%sy）
   → iostat -x 1: I/O 等待（await）和磁盘利用率（%util）
   → free -h: 内存使用，是否有 Swap
   → netstat -an: 网络连接数

3. 数据库内部定位
   → SHOW PROCESSLIST：看当前连接在做什么（是否大量 Locked/Sending data）
   → SHOW ENGINE INNODB STATUS：看死锁、脏页、锁等待
   → 慢查询日志：找到热门慢 SQL

4. 执行计划分析
   → EXPLAIN 可疑 SQL：看 type、key、rows、Extra
   → 是否全表扫描？是否有 filesort/temporary？

5. 深挖等待事件
   → performance_schema.events_waits_current：线程当前等待的资源
   → 定位到是锁等待、I/O 等待还是 CPU 计算瓶颈
```

## 5. 连接池优化

```
连接池选型：
  HikariCP（Spring Boot 默认）→ 性能最佳，代码简洁
  Druid（阿里开源）→ 功能丰富，监控完善，支持 SQL 防火墙
```

```yaml
# HikariCP 关键配置（application.yml）
spring:
  datasource:
    hikari:
      # 核心配置
      maximum-pool-size: 20      # 最大连接数（建议 CPU核数 * 2 ~ 4，IO密集可适当增加）
      minimum-idle: 5            # 最小空闲连接
      connection-timeout: 3000   # 获取连接超时（ms）
      idle-timeout: 600000       # 空闲连接最长存活时间（10分钟）
      max-lifetime: 1800000      # 连接最长存活时间（30分钟，需 < wait_timeout）
      
      # 连接有效性检测
      connection-test-query: SELECT 1  # 连接健康检查 SQL
      keepalive-time: 60000      # 每60秒发送一次 keepalive（防止防火墙超时断连）
      
      # 泄漏检测（开发/测试环境开启）
      leak-detection-threshold: 30000  # 30秒未归还则告警
```

## 6. 减少磁盘 I/O 的实战技巧

| 技巧 | 说明 | 效果 |
|------|------|------|
| **覆盖索引** | 索引包含查询所需全部列，无需回表 | 消除随机 I/O |
| **增大 Buffer Pool** | 让工作集常驻内存 | 读操作变内存访问 |
| **增大 Redo Log 文件** | 减少 Checkpoint 频率 | 减少刷脏页 I/O |
| **SSD 替换 HDD** | IOPS 提升 10-100 倍 | 所有 I/O 操作受益 |
| **`innodb_flush_method=O_DIRECT`** | 绕过 OS 缓存直接写（避免双重缓存） | 减少内存压力 |
| **`innodb_file_per_table=ON`** | 每表独立 .ibd 文件，便于管理和回收 | 防止 ibdata1 膨胀 |
| **分区表** | 将历史数据按时间分区，查询只扫描相关分区 | 大幅减少扫描范围 |
