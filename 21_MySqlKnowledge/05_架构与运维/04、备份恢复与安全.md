# 备份恢复与安全

> 数据是企业最重要的资产。备份策略的核心目标是满足业务的 RTO（恢复时间目标）和 RPO（恢复点目标），而安全的核心是最小权限原则和纵深防御。

---

## 1. 备份策略规划

### 1.1 核心目标

| 指标 | 说明 | 典型要求 |
|------|------|---------|
| **RPO**（Recovery Point Objective） | 最多丢失多少数据 | 核心业务：0 ~ 5分钟 |
| **RTO**（Recovery Time Objective） | 最长恢复多少时间 | 核心业务：< 30分钟 |
| **备份窗口** | 备份操作占用多少时间 | 应避开业务高峰 |

### 1.2 三种备份量类型

| 类型 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **全量备份** | 某时间点的完整数据快照 | 恢复简单，直接还原 | 时间长，占用空间大 |
| **增量备份** | 自上次备份以来变化的数据（Binlog） | 快速，空间小 | 恢复时需逐步累加，链式依赖，任一损坏无法恢复 |
| **差异备份** | 自上次全备以来的所有变化 | 恢复比增量简单（全备 + 一次差异备份） | 随时间推移，差异包越来越大 |

**推荐策略（企业常用）**：
```
每周日凌晨 2:00 → 全量备份（mysqldump 或 XtraBackup）
每天凌晨 2:00   → 差异备份或增量备份
实时              → Binlog 流式备份到远程存储（保留至少 7 天）
```

## 2. 逻辑备份（mysqldump）

### 2.1 基本用法

```bash
# 备份单个数据库（InnoDB 热备份：--single-transaction）
mysqldump -u root -p \
  --single-transaction \           # InnoDB 一致性快照（不锁表，热备关键）
  --quick \                        # 逐行输出，防止大表 OOM
  --master-data=2 \                # 记录 Binlog 位置（注释形式，用于 PITR）
  --routines \                     # 备份存储过程和函数
  --triggers \                     # 备份触发器
  --events \                       # 备份事件
  my_database > /backup/my_database_$(date +%Y%m%d_%H%M%S).sql

# 压缩备份（节省存储空间）
mysqldump -u root -p --single-transaction my_database | \
  gzip > /backup/my_database_$(date +%Y%m%d_%H%M%S).sql.gz

# 备份所有数据库
mysqldump -u root -p --single-transaction --all-databases > /backup/full_$(date +%Y%m%d).sql

# 仅备份表结构（不含数据）
mysqldump -u root -p --no-data my_database > /backup/schema_only.sql
```

### 2.2 恢复

```bash
# 从 mysqldump 备份恢复
mysql -u root -p my_database < /backup/my_database_20260314.sql

# 恢复压缩备份
gunzip < /backup/my_database_20260314.sql.gz | mysql -u root -p my_database

# 恢复时跳过外键检查（加速恢复，恢复后自动恢复）
mysql -u root -p my_database <<EOF
SET foreign_key_checks = 0;
SOURCE /backup/my_database_20260314.sql;
SET foreign_key_checks = 1;
EOF
```

## 3. 物理备份（Percona XtraBackup）

### 3.1 为什么用 XtraBackup

| 对比维度 | mysqldump | XtraBackup |
|----------|----------|------------|
| 备份原理 | SQL 层全量查询输出 | 直接拷贝 InnoDB 数据文件 + 持续追踪 Redo Log |
| 备份速度 | 慢（行扫描） | **快**（块级拷贝） |
| 恢复速度 | 慢（逐行执行 SQL） | **快**（直接替换数据文件） |
| 对业务影响 | 较大（CPU/内存压力） | 小（I/O 压力，锁持有时间极短） |
| 适用规模 | < 100GB | **TB 级别大数据库** |
| 工具来源 | MySQL 官方内置 | Percona 开源（免费） |

### 3.2 备份流程

```bash
# 1. 全量备份
xtrabackup --backup \
  --user=backup_user \
  --password=backup_pass \
  --target-dir=/backup/full_$(date +%Y%m%d) \
  --compress \            # 备份时压缩（需 --compress-threads=4）
  --compress-threads=4

# 2. 准备备份（应用 Redo Log，使备份达到一致状态）
xtrabackup --prepare \
  --target-dir=/backup/full_20260314

# 3. 恢复（需先停止 MySQL，清空 data 目录）
systemctl stop mysql
rm -rf /var/lib/mysql/*

xtrabackup --copy-back \
  --target-dir=/backup/full_20260314

# 修复文件权限
chown -R mysql:mysql /var/lib/mysql

# 启动 MySQL
systemctl start mysql
```

### 3.3 增量备份（XtraBackup）

```bash
# 首先进行一次全量备份
xtrabackup --backup --target-dir=/backup/full_sunday

# 基于全量进行增量备份（每天）
xtrabackup --backup \
  --incremental-basedir=/backup/full_sunday \
  --target-dir=/backup/inc_monday

# 连续增量（周二基于周一）
xtrabackup --backup \
  --incremental-basedir=/backup/inc_monday \
  --target-dir=/backup/inc_tuesday

# 恢复时需按顺序合并：全量 → inc_monday → inc_tuesday
xtrabackup --prepare --apply-log-only --target-dir=/backup/full_sunday
xtrabackup --prepare --apply-log-only \
  --target-dir=/backup/full_sunday \
  --incremental-dir=/backup/inc_monday
xtrabackup --prepare \
  --target-dir=/backup/full_sunday \
  --incremental-dir=/backup/inc_tuesday
# 最后执行 copy-back
```

## 4. 时间点恢复（PITR）

**场景**：误执行了 `DROP TABLE orders` 或 `UPDATE orders SET amount=0 WHERE 1=1`，需要恢复到误操作之前的状态。

### 4.1 前提条件

- **Binlog 必须开启**（`log_bin = ON`，生产环境应永远开启）
- Binlog 文件保存完整（建议备份到对象存储，如腾讯云 COS）
- 记录全量备份时对应的 Binlog 位置（`--master-data` 参数）

### 4.2 PITR 操作步骤

```bash
# 第1步：恢复最近的全量备份
mysql -u root -p < /backup/full_20260314.sql

# 第2步：查看全备对应的 Binlog 位置
head -30 /backup/full_20260314.sql | grep "CHANGE MASTER"
# 输出示例：-- CHANGE MASTER TO MASTER_LOG_FILE='mysql-bin.000234', MASTER_LOG_POS=154;

# 第3步：找到误操作的时间点或位置
# 在 Binlog 中搜索误操作 SQL
mysqlbinlog --base64-output=DECODE-ROWS -v mysql-bin.000234 | grep -i "drop table"
# 找到误操作的 Position（假设是 9527）

# 第4步：应用全备结束位置到误操作之前的 Binlog
mysqlbinlog \
  --start-position=154 \
  --stop-position=9527 \        # 误操作之前的位置
  mysql-bin.000234 \
  mysql-bin.000235 | \          # 可以指定多个 Binlog 文件
  mysql -u root -p

# 或者按时间点恢复（更常用，不需要找 Position）
mysqlbinlog \
  --start-datetime="2026-03-14 20:00:00" \
  --stop-datetime="2026-03-14 21:50:00" \  # 误操作发生在 21:55
  mysql-bin.000234 | \
  mysql -u root -p

echo "数据已恢复到 2026-03-14 21:50:00"
```

### 4.3 误操作防范

```sql
-- 开启 sql_safe_updates（限制无 WHERE 的 DELETE/UPDATE）
SET GLOBAL sql_safe_updates = ON;
SET GLOBAL sql_select_limit = 10000;  -- 限制 SELECT 返回行数

-- 为重要表添加延迟删除列（逻辑删除代替物理删除）
-- deleted_at DATETIME NULL DEFAULT NULL

-- DML 操作使用事务，不直接提交
BEGIN;
UPDATE orders SET status = 2 WHERE id = 12345;
-- 先查看影响行数，确认无误后再 COMMIT
SELECT ROW_COUNT();  -- 查看受影响行数
ROLLBACK;  -- 发现不对，回滚！
-- COMMIT;  -- 确认无误，提交
```

## 5. SQL 注入防防

### 5.1 预编译语句（根本解决方案）

```java
// 错误示范：字符串拼接（SQL 注入漏洞！）
String sql = "SELECT * FROM users WHERE username = '" + username + "'";
// 攻击者输入：' OR '1'='1
// 实际 SQL：SELECT * FROM users WHERE username = '' OR '1'='1'

// 正确做法：PreparedStatement
String sql = "SELECT * FROM users WHERE username = ? AND password = ?";
try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
    pstmt.setString(1, username);   // 参数化，不会被解释为 SQL
    pstmt.setString(2, password);
    ResultSet rs = pstmt.executeQuery();
}
```

### 5.2 MyBatis 的 #{} vs ${}

```xml
<!-- 安全：#{} 使用预编译参数 -->
<select id="getUser">
    SELECT * FROM users WHERE username = #{username}
</select>

<!-- 危险：${} 直接字符串拼接，存在注入风险！ -->
<select id="getUser">
    SELECT * FROM users WHERE username = '${username}'
</select>

<!-- ${} 的合理使用场景（动态表名/列名，只能用可信来源） -->
<select id="getTableData">
    SELECT * FROM ${tableName} LIMIT 10
    -- tableName 必须是程序内部白名单，绝不能是用户输入！
</select>
```

### 5.3 输入校验（纵深防御）

```java
// 参数类型校验
public User getUserById(long id) {
    if (id <= 0) throw new IllegalArgumentException("Invalid user id");
    // ...
}

// 字符串格式校验
if (!username.matches("[a-zA-Z0-9_]{3,20}")) {
    throw new IllegalArgumentException("Username contains invalid characters");
}

// 使用白名单控制动态排序字段（防止通过 ORDER BY 注入）
Set<String> ALLOWED_SORT_COLUMNS = Set.of("id", "name", "created_at");
if (!ALLOWED_SORT_COLUMNS.contains(sortColumn)) {
    throw new IllegalArgumentException("Invalid sort column");
}
```

## 6. 权限管理

### 6.1 最小权限原则

```sql
-- 创建专用账户（按角色区分权限）
-- 应用程序账户（只需增删改查）
CREATE USER 'app_user'@'192.168.1.%' IDENTIFIED BY 'StrongPassword123!';
GRANT SELECT, INSERT, UPDATE, DELETE ON my_db.* TO 'app_user'@'192.168.1.%';

-- 只读账户（用于读写分离的从库账户）
CREATE USER 'readonly_user'@'%' IDENTIFIED BY 'ReadPassword456!';
GRANT SELECT ON my_db.* TO 'readonly_user'@'%';

-- 备份账户（最小权限备份）
CREATE USER 'backup_user'@'localhost' IDENTIFIED BY 'BackupPass789!';
GRANT SELECT, RELOAD, LOCK TABLES, REPLICATION CLIENT ON *.* TO 'backup_user'@'localhost';

-- 从库复制账户
CREATE USER 'repl'@'192.168.1.%' IDENTIFIED BY 'ReplPass!';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'192.168.1.%';

FLUSH PRIVILEGES;
```

### 6.2 MySQL 8.0 的角色管理

```sql
-- 创建角色（将一组权限打包）
CREATE ROLE 'app_read', 'app_write';

GRANT SELECT ON my_db.* TO 'app_read';
GRANT INSERT, UPDATE, DELETE ON my_db.* TO 'app_write';

-- 将角色赋予用户
GRANT 'app_read', 'app_write' TO 'app_user'@'%';

-- 默认激活角色（用户登录自动获得角色权限）
ALTER USER 'app_user'@'%' DEFAULT ROLE 'app_read', 'app_write';

-- 查看用户权限
SHOW GRANTS FOR 'app_user'@'%';
```

### 6.3 权限审计

```sql
-- 查看所有用户及来源（定期审查，清理无用账户）
SELECT user, host, authentication_string != '' as has_password,
       password_expired, account_locked
FROM mysql.user
ORDER BY user;

-- 查找有所有权限的高危账户
SELECT user, host FROM mysql.user
WHERE Super_priv = 'Y' OR Grant_priv = 'Y';

-- 修改 root 只允许本机访问（禁止远程 root 登录）
UPDATE mysql.user SET host = 'localhost' WHERE user = 'root' AND host = '%';
FLUSH PRIVILEGES;
```

## 7. 数据加密

### 7.1 传输加密（TLS/SSL）

```sql
-- 查看 SSL 状态
SHOW VARIABLES LIKE 'have_ssl';
SHOW VARIABLES LIKE 'ssl_%';

-- 强制某个账户必须使用 SSL 连接
ALTER USER 'sensitive_user'@'%' REQUIRE SSL;

-- Java JDBC 连接字符串启用 SSL
# jdbc:mysql://localhost:3306/mydb?useSSL=true&requireSSL=true&verifyServerCertificate=true
```

### 7.2 静态数据加密（InnoDB 表空间加密）

```sql
-- MySQL 5.7.11+ 支持 InnoDB 表空间加密（需配置 keyring 插件）
-- 为单张表启用加密
CREATE TABLE sensitive_data (
    id INT PRIMARY KEY,
    ssn VARCHAR(20),
    salary DECIMAL(10,2)
) ENCRYPTION = 'Y';

-- 修改已有表为加密（会重建表）
ALTER TABLE sensitive_data ENCRYPTION = 'Y';

-- 查看哪些表已加密
SELECT name, encryption FROM information_schema.innodb_tablespaces
WHERE encryption = 'Y';
```

### 7.3 应用层字段加密（最高安全级别）

```java
// 敏感字段（手机号、身份证）在应用层加密后再存入数据库
@Component
public class AESEncryptor {
    private final String key = "16-byte-secret!!";  // 实际应从配置中心获取
    
    public String encrypt(String plaintext) throws Exception {
        Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");
        cipher.init(Cipher.ENCRYPT_MODE, new SecretKeySpec(key.getBytes(), "AES"));
        return Base64.getEncoder().encodeToString(cipher.doFinal(plaintext.getBytes()));
    }
    
    public String decrypt(String ciphertext) throws Exception {
        Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");
        cipher.init(Cipher.DECRYPT_MODE, new SecretKeySpec(key.getBytes(), "AES"));
        return new String(cipher.doFinal(Base64.getDecoder().decode(ciphertext)));
    }
}
```

> **应用层加密的缺点**：加密后的字段无法用普通 LIKE 查询，需使用确定性加密（加密后结果一致）配合等值查询，或在 ES 中维护明文索引（注意安全风险）。

## 8. 备份最佳实践

```
备份策略 Checklist：
✅ 备份文件存储在独立介质（不与数据库同一磁盘）
✅ 备份文件定期同步到对象存储（腾讯云 COS、异地容灾）
✅ 定期执行恢复演练（备份没有经过验证就等于没有备份）
✅ Binlog 实时备份（binlog2s3 等工具）
✅ 备份加密（备份文件本身也是敏感数据）
✅ 监控备份任务状态（备份失败立即告警）
✅ 记录备份文件清单和对应的 Binlog 位置
✅ 保留至少 30 天的备份历史（满足监管合规要求）
```
