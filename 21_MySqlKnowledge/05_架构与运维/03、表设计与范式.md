# 表设计与范式

> 好的表设计是系统稳健运行的基础，影响索引效率、查询性能、扩展能力和维护成本。表设计没有银弹，核心是在规范性、查询性能和可扩展性三者间做出合理权衡。

---

## 1. 数据库范式

### 1.1 三大范式的核心要求

| 范式 | 核心要求 | 解决的问题 | 违反示例 |
|------|---------|-----------|---------|
| **1NF（第一范式）** | 字段是**原子性**的，不可再分 | 确保数据基本完整性 | `contact_info` 字段同时存电话和地址 |
| **2NF（第二范式）** | 满足1NF + **非主属性完全依赖整个主键**（消除部分依赖） | 解决联合主键下的数据冗余 | 选课表 `(学号, 课程号, 成绩, 课程学分)` 中，`课程学分` 只依赖 `课程号` |
| **3NF（第三范式）** | 满足2NF + **非主属性不传递依赖主键**（消除传递依赖） | 消除间接冗余 | `员工(员工ID, 部门ID, 部门名称)`，`部门名称` 依赖 `部门ID`，而非直接依赖 `员工ID` |

### 1.2 传递依赖示例（3NF 违反）

```
违反3NF：
employee(emp_id, dept_id, dept_name, dept_location)
         主键      ↓         ↓            ↓
                dept_id → dept_name → dept_location  （传递依赖）

问题：
- 更新部门名称时，要更新所有该部门员工的记录（更新异常）
- 删除最后一个员工时，部门信息一并丢失（删除异常）
- 插入员工时必须关联存在的部门（插入异常）

符合3NF（拆分）：
employee(emp_id, dept_id)          -- 员工表，dept_id 是外键
department(dept_id, dept_name, dept_location)  -- 部门表
```

### 1.3 反范式化

**反范式化**是有意引入数据冗余，以空间换时间，提升特定查询性能的设计策略。

**什么时候反范式化**：
1. 频繁 JOIN 导致查询性能问题，且 JOIN 的表数据相对稳定
2. 需要避免跨分库 JOIN（分布式场景）
3. 统计类查询需要实时聚合，引入汇总表

**常见反范式化技术**：

| 技术 | 示例 | 注意事项 |
|------|------|---------|
| **冗余字段** | 订单表冗余 `product_name`，避免每次关联商品表 | 商品改名时需同步更新冗余字段 |
| **汇总表** | 用户积分汇总表（`user_points`），避免每次聚合流水 | 需要业务层保证与流水表一致 |
| **派生列** | 订单表中存储 `total_amount`（= 各明细 amount 之和） | 更新明细时需同步更新派生列 |
| **合并 1:1 关系** | 将用户基本信息和设置信息合并（低频字段用垂直分表代替） | 表变宽，影响扫描效率 |

## 2. 字段类型选型

### 2.1 整数类型

| 类型 | 存储 | 有符号范围 | 无符号范围 | 推荐场景 |
|------|------|-----------|-----------|---------|
| `TINYINT` | 1字节 | -128 ~ 127 | 0 ~ 255 | 状态码、枚举值 |
| `SMALLINT` | 2字节 | -32768 ~ 32767 | 0 ~ 65535 | 年龄、评分 |
| `INT` | 4字节 | -21亿 ~ 21亿 | 0 ~ 42亿 | 常规数量、普通 ID |
| `BIGINT` | 8字节 | -9.2×10¹⁸ ~ 9.2×10¹⁸ | 0 ~ 1.8×10¹⁹ | **主键、雪花 ID、金额（分）** |

> **金额存储**：**绝对不能用 FLOAT/DOUBLE**（浮点精度问题），用 `DECIMAL(精度, 标度)` 或 `BIGINT`（以分为单位，100分=1元）。

### 2.2 字符串类型

| 类型 | 存储方式 | 最大长度 | 适用场景 |
|------|---------|---------|---------|
| `CHAR(n)` | 固定 n 字节（不足填充空格） | 255 字符 | 长度固定（MD5哈希32位、手机号11位） |
| `VARCHAR(n)` | 实际长度 + 1~2 字节前缀 | 65535 字节 | 长度变化（姓名、地址、描述） |
| `TEXT` | 独立存储区（超出768字节用溢出页） | 65535 字节 | 文章内容、评论 |
| `MEDIUMTEXT` | 独立存储区 | 16MB | 大文本内容 |
| `LONGTEXT` | 独立存储区 | 4GB | 极大文本（慎用于 MySQL） |

**VARCHAR vs CHAR 选型**：
- 固定长度（如 MD5、手机号、身份证号）→ **CHAR**（访问速度更快，无长度计算开销）
- 变长内容 → **VARCHAR**（节省存储空间）
- 频繁更新且经常变长 → **VARCHAR**（避免 CHAR 更新时页内频繁移动）

### 2.3 时间类型

| 类型 | 存储 | 时间范围 | 时区 | 推荐场景 |
|------|------|---------|------|---------|
| `DATETIME` | 5字节（MySQL 5.6.4+） | 1000-01-01 ~ 9999-12-31 | **无时区转换** | 业务时间（如预约时间） |
| `TIMESTAMP` | 4字节 | 1970-01-01 ~ 2038-01-19 | **自动转 UTC 存储，读取时转当前时区** | 记录创建/更新时间 |
| `DATE` | 3字节 | 1000-01-01 ~ 9999-12-31 | - | 只需日期（生日） |

**`created_at` 字段的最佳实践**：
```sql
-- 推荐（DATETIME，业务时间语义明确，不受时区影响）
created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

-- 注意：TIMESTAMP 在 MySQL 8.0.28+ 同样支持 ON UPDATE
```

> **TIMESTAMP 的 2038 年问题**：TIMESTAMP 使用 4 字节存储 Unix 时间戳，最大值为 2038-01-19 03:14:07 UTC。新系统应优先使用 DATETIME。

### 2.4 NULL vs 默认值

```sql
-- 不推荐（允许 NULL）
name VARCHAR(100),
status TINYINT,

-- 推荐（NOT NULL + 默认值）
name VARCHAR(100) NOT NULL DEFAULT '',
status TINYINT NOT NULL DEFAULT 0,
```

**为什么尽量避免 NULL**：
- NULL 参与的比较（`!= NULL`）始终为 NULL，逻辑复杂
- NULL 值使索引统计信息不准确，影响优化器选择
- 联合索引中含 NULL 列的计算更复杂
- 某些 ORM 框架对 NULL 处理有陷阱（如 MyBatis 条件判断）

## 3. 主键选型

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **AUTO_INCREMENT BIGINT** | 单体应用，无分库分表 | 顺序递增，索引效率最高，存储小 | 全局不唯一，分布式场景需处理冲突 |
| **雪花算法（Snowflake）** | 分布式系统 | 全局唯一，趋势递增，64位数字 | 依赖机器时钟，需处理时钟回拨 |
| **UUID** | 任何场景 | 全局唯一，本地生成 | 无序（页分裂严重），36字符字符串，存储和索引效率差 |
| **UUID_v7 / ULID** | 新项目 | 全局唯一 + 有序（时间前缀） | MySQL 8.0 需自定义函数 |

> **UUID 为什么不适合做主键**：UUID 完全随机，每次 INSERT 都会在 B+树的随机位置插入，导致**页分裂频繁**（随机 I/O），写入性能比自增 ID 差 10 倍以上，且 36 字节字符串比 8 字节 BIGINT 占用更多空间和索引体积。

## 4. 外键的使用建议

**互联网大厂的通用做法**：数据库层面**不使用外键约束**，在应用层保证数据一致性。

**原因**：
1. 外键会在写操作时产生额外的跨表锁，成为高并发下的锁热点
2. 分库分表后外键无法跨库存在
3. DDL 操作（如 ALTER TABLE）在有外键时更复杂
4. 微服务架构中，不同服务数据库隔离，天然无法用外键

**应用层替代方案**：
- 业务代码中先校验关联记录存在性再写入
- 事务保证多表操作的原子性
- 定期数据一致性校验任务（发现孤儿记录）

## 5. 表设计最佳实践

### 5.1 必备字段规范

```sql
CREATE TABLE orders (
    -- 主键（分布式场景用雪花ID，单体用自增BIGINT）
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    
    -- 业务字段...
    user_id BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
    amount DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '订单金额（元）',
    status TINYINT NOT NULL DEFAULT 0 COMMENT '状态 0-待支付 1-已支付 2-已取消',
    
    -- 时间字段（必备）
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP 
               ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    
    -- 逻辑删除（大多数场景优于物理删除）
    is_deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除 0-正常 1-已删除',
    
    PRIMARY KEY (id),
    INDEX idx_user_id (user_id),         -- 用户ID查询
    INDEX idx_status_created (status, created_at)  -- 按状态+时间查询
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='订单表';
```

### 5.2 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 表名 | 小写下划线，有业务含义 | `user_profile`、`order_detail` |
| 字段名 | 小写下划线 | `user_id`、`created_at` |
| 索引名 | `idx_列名` | `idx_user_id`、`idx_status_created` |
| 唯一索引 | `uk_列名` | `uk_email`、`uk_order_no` |
| 主键 | `PRIMARY KEY (id)` | - |
| 避免 | 数据库保留字（`order`、`group`） | 用 `orders`、`user_group` |

### 5.3 大表优化策略

**垂直分表**（冷热字段分离）：

```sql
-- 原始宽表（20+字段，部分字段很少查询）
user(id, name, phone, email, avatar_url, bio, address, birthday, settings_json, ...)

-- 优化后
user_basic(id, name, phone, email)           -- 核心高频字段
user_profile(user_id, avatar_url, bio, address, birthday, settings_json, ...)  -- 低频字段
```

**分区表**（时序数据归档）：

```sql
-- 按年分区（适合日志、订单等时序数据）
CREATE TABLE access_logs (
    id BIGINT NOT NULL AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    created_at DATETIME NOT NULL,
    log_data TEXT,
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (YEAR(created_at)) (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- 查询时只扫描相关分区（分区裁剪）
SELECT * FROM access_logs WHERE created_at BETWEEN '2025-01-01' AND '2025-12-31';
-- EXPLAIN 可见 partitions: p2025

-- 归档旧数据（直接删除整个分区，比 DELETE 快数倍）
ALTER TABLE access_logs DROP PARTITION p2023;
```

### 5.4 字段长度原则

- **按需分配**：`VARCHAR(100)` 不代表会分配 100 字节，但会影响 `memory` 存储引擎和 `filesort` 时的临时缓冲区分配
- **不要过度预留**：`VARCHAR(255)` 不比 `VARCHAR(50)` 在磁盘上多占空间，但在内存中（filesort/GROUP BY）会多占
- **JSON 字段**：MySQL 5.7.8+ 支持原生 JSON 类型，提供路径查询和自动校验，但 JSON 列上无法建常规索引（需用生成列）

```sql
-- JSON 字段 + 生成列索引
ALTER TABLE products 
ADD COLUMN category_id INT AS (JSON_UNQUOTE(attributes->'$.category_id')) STORED;
ADD INDEX idx_category_id (category_id);
```

## 6. 视图与存储过程的使用建议

| 对象 | 优点 | 生产建议 |
|------|------|---------|
| **视图（View）** | 隐藏复杂查询，简化接口，权限控制 | 适合只读报表视图；避免在视图上再嵌套视图（性能难以预测） |
| **存储过程（Stored Procedure）** | 减少网络传输，复用逻辑 | 互联网场景不推荐（调试困难，版本管理麻烦，扩展性差）；数仓/报表场景可用 |
| **触发器（Trigger）** | 自动维护一致性 | 慎用（隐式操作，难以调试，DML 性能开销，复制场景可能问题） |
