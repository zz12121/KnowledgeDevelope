# 索引原理与数据结构

> 索引是 MySQL 性能调优最核心的手段。理解 B+树的结构、InnoDB 聚簇索引的设计，是掌握一切索引相关知识的基础。

---

## 一、为什么需要索引

没有索引时，查询一条数据需要**全表扫描**（Full Table Scan）：
- 从第一行开始逐行比较，直到找到目标行
- 时间复杂度 O(n)
- 100万行数据，最坏情况读取 100 万次 I/O

有索引后：
- B+树结构，树高约为 3~4 层
- 每次查询最多 3~4 次 I/O
- 时间复杂度 O(log n)

**代价权衡**：
- 占用额外磁盘空间（每个索引是独立的 B+树）
- 写操作需要维护索引（INSERT/UPDATE/DELETE 都需要更新相关索引）
- 因此索引不是越多越好，需要有针对性地设计

---

## 二、B+树索引原理

### 2.1 B+树结构特点

InnoDB 使用 **B+树（B+ Tree）** 作为索引数据结构。理解 B+树要从其设计目标出发：**专为减少磁盘 I/O 而设计的平衡多路查找树**。

```
                    ┌─────────────────┐
                    │  根节点 Root     │
                    │  [15] [30]       │  ← 非叶子节点只存 key（不存 row 数据）
                    └──┬──────┬──────-┘
                       │      │
           ┌───────────┘      └───────────┐
    ┌──────▼──────┐             ┌─────────▼──────┐
    │ [5][10][15] │             │ [20][25][30]    │  ← 内部节点
    └──┬──┬──┬───┘             └───┬──┬──┬──────┘
       │  │  │                     │  │  │
    ┌──▼─┐┌──▼─┐┌──▼─┐        ┌──▼─┐┌──▼─┐┌──▼─┐
    │ 1,3││5,8 ││10,13│        │17,18││22  ││27,29│  ← 叶子节点，存完整数据（聚簇）
    │    ││    ││     │  ←→    │     ││    ││     │     或主键值（二级索引）
    └────┘└────┘└─────┘  双向链表  └─────┘└────┘└─────┘
```

**B+树的核心特性**：

1. **非叶子节点只存 Key 和指针，不存行数据**：使得每个节点能容纳更多 Key（更高的"扇出率"），同样数据量树高更低
2. **所有数据存储在叶子节点**：查询任何数据都必须到达叶子节点，查询性能稳定
3. **叶子节点通过双向链表连接**：支持高效的**范围查询**和**顺序遍历**

### 2.2 为什么不用 B 树？

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| **数据存储位置** | 所有节点都存数据 | 只有叶子节点存数据 |
| **范围查询** | 需要中序遍历，效率低 | 叶子链表，O(k) 顺序扫描 |
| **每个节点容量** | 存数据导致 key 数量少，树更高 | 只存 key，扇出率高，树更矮 |
| **查询稳定性** | 非叶子节点可能命中，深度不固定 | 必须到叶子节点，深度固定 |

**为什么不用哈希表？**

| 操作 | 哈希表 | B+ 树 |
|------|-------|-------|
| 等值查询 | O(1)（极快）| O(log n) |
| 范围查询 | **不支持** | O(log n + k) |
| 排序 | **不支持** | 天然有序 |
| 内存占用 | 大（需要哈希表）| 可按需加载 |

> InnoDB 提供了**自适应哈希索引（Adaptive Hash Index）**，针对频繁访问的 B+树 Page 自动在内存中建立哈希表，作为 B+树的补充加速点查。这是 InnoDB 的自动优化，无需手动干预。

### 2.3 B+树的高度与 I/O 次数

每个 B+树节点的大小默认是 **16KB**（InnoDB page size）。

**非叶子节点能存多少个 Key？**
- 设 Key 大小为 8 字节（BIGINT），子节点指针 6 字节
- 每个节点可存约：16KB / (8 + 6) ≈ **1170 个 Key**
- 即每层扇出率约为 1170

**叶子节点能存多少行数据？**
- 设每行数据平均 1KB
- 每个叶子节点可存约 16 行

**三层 B+树的容量**：
- 第 1、2 层（非叶子）：1170 × 1170 ≈ **136 万个叶子节点**
- 总行数：136万 × 16 ≈ **2200 万行**

**结论**：B+树通常只有 2~3 层，查找任意数据最多 **3 次磁盘 I/O**（根节点常驻内存则只需 2 次）

## 三、聚簇索引与二级索引

### 3.1 聚簇索引（Clustered Index）

InnoDB 中，**主键索引就是聚簇索引**。其特点是：

> **表数据文件本身就是按主键组织的 B+树**，叶子节点存储的是完整的数据行。

```
聚簇索引 B+树（主键 = id BIGINT）：

                 [ 10 | 30 | 50 ]
                /     |     \
        [1|2|10]  [10|20|30]  [30|40|50]    ← 内部节点
          ↓           ↓           ↓
  ┌─────────────┐  ┌──────────────┐  ...    ← 叶子节点
  │id=1, row_data│  │id=10,row_data│
  │id=2, row_data│  │id=20,row_data│
  │id=10,row_data│  │id=30,row_data│
  └─────────────┘  └──────────────┘
       ↔                ↔              ← 双向链表
```

**如果表没有主键**，InnoDB 会按以下顺序选择聚簇索引：
1. 首选第一个定义的非空唯一索引
2. 若无，自动生成一个 6 字节的隐藏行 ID（`DB_ROW_ID`）作为聚簇索引

因此**每张 InnoDB 表都有且仅有一个聚簇索引**。

### 3.2 二级索引（Secondary Index / Non-Clustered Index）

除主键之外的所有索引都是二级索引（也叫辅助索引）。二级索引的叶子节点存储的是：**索引列的值 + 对应行的主键值**（不是完整行数据）。

```
二级索引 B+树（索引列 = email）：

叶子节点：
┌──────────────────────────────────────────┐
│  email='alice@x.com'  → 主键 id=3        │
│  email='bob@x.com'    → 主键 id=1        │
│  email='charlie@x.com'→ 主键 id=7        │
└──────────────────────────────────────────┘
         ↓
    回表（Bookmark Lookup）
         ↓
    用 id 去聚簇索引查找完整行数据
```

### 3.3 回表（Bookmark Lookup）

当查询需要的字段不在二级索引中时，需要：
1. 先用二级索引找到主键值
2. 再用主键值去聚簇索引查找完整行数据

**回表代价**：
- 每次回表 = 一次聚簇索引的 B+树查找 ≈ 2~3 次磁盘 I/O
- 如果结果集有 N 行，回表最多 N 次额外 I/O

**减少回表的方案：覆盖索引**（见下章）

## 四、覆盖索引（Covering Index）

### 4.1 什么是覆盖索引

**覆盖索引**：查询需要的所有字段都包含在索引中，无需回表。

```sql
-- 表结构
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),
    age INT,
    email VARCHAR(100),
    address TEXT
);

-- 创建联合索引
CREATE INDEX idx_name_age ON users(name, age);

-- 覆盖索引查询（SELECT 的字段全在索引中）
SELECT name, age FROM users WHERE name = 'Alice';
-- ✅ EXPLAIN 的 Extra 显示 "Using index"，无回表

-- 非覆盖索引查询（SELECT 包含索引外的字段）
SELECT name, age, email FROM users WHERE name = 'Alice';
-- ❌ 需要回表获取 email 字段
-- EXPLAIN 的 Extra 显示 "Using index condition"，有回表
```

### 4.2 覆盖索引的实际应用

```sql
-- 场景：分页查询，获取用户列表（只需要 id, name）
-- 如果 idx_name_age 包含了 name，则无需回表
SELECT id, name FROM users WHERE name LIKE 'A%' ORDER BY name LIMIT 100;

-- 场景：统计查询
SELECT COUNT(*) FROM users WHERE name = 'Alice';
-- 只需要 name 列，二级索引上就能完成，无需回表

-- 延迟关联（先用覆盖索引缩小范围，再回表）
-- 适用于大偏移量分页
SELECT u.* FROM users u
INNER JOIN (
    SELECT id FROM users WHERE age > 18 ORDER BY name LIMIT 10000, 20
) t ON u.id = t.id;
```

## 五、页分裂与有序插入

### 5.1 页分裂的问题

B+树中每个叶子节点（Page）默认 16KB，当一个 Page 满了，需要分裂：

```
插入 id=25 到满页时：
原 Page：[10][15][20][22][24]（满）
分裂：
  Page1：[10][15][20]
  Page2：[22][24][25]
  父节点更新指针
```

页分裂的代价：
1. 分配新 Page
2. 移动一半数据
3. 更新父节点
4. 产生碎片（Page 利用率从 100% 降到 50%）

### 5.2 为什么自增主键对写入友好

使用自增主键（AUTO_INCREMENT）时，每次插入都在 B+树的最右侧叶子节点追加：
- 不触发页分裂（直到当前 Page 满了才在末尾追加一个新 Page）
- **顺序写入**，磁盘 I/O 效率高
- 索引紧凑，碎片少

使用 UUID（随机）主键时：
- 每次插入位置随机，频繁触发页分裂
- 索引碎片化严重，Buffer Pool 利用率低
- 写入性能随数据量增大明显下降

## 六、面试要点速查

| 问题 | 核心答案 |
|------|---------|
| B+树有哪些特点？ | 非叶节点只存Key/指针，叶节点存数据，叶节点双向链表，树高固定 |
| 为什么用 B+树不用 B 树？ | 范围查询用链表效率更高；非叶节点不存数据扇出率更高，树更矮 |
| 为什么用 B+树不用哈希？ | 哈希不支持范围查询和排序 |
| 聚簇索引是什么？ | 数据文件按主键组织的 B+树，叶节点存完整行数据 |
| 二级索引叶节点存什么？ | 索引列的值 + 主键值（不存完整行，需回表） |
| 什么是覆盖索引？ | 查询所需字段全在索引中，无需回表，EXPLAIN Extra 显示 "Using index" |
| 为什么自增主键比 UUID 写入快？ | 自增顺序追加无页分裂；UUID 随机插入频繁触发页分裂 |
| InnoDB 三层 B+树能存多少数据？ | 约 2000 万行，查询最多 3 次磁盘 I/O |
