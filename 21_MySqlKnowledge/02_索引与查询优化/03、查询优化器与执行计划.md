# 查询优化器与执行计划

> 优化器是 MySQL 的大脑，理解它的工作原理，才能知道为什么某条 SQL 慢，以及如何让它快起来。

---

## 一、查询优化器工作原理

### 1.1 什么是基于成本的优化（CBO）

MySQL 使用**基于成本的优化器（Cost-Based Optimizer, CBO）**，其核心逻辑是：

> 为一条 SQL 枚举所有可能的执行方案，计算每种方案的代价，选择代价最小的方案执行。

"代价"主要包括两部分：
- **I/O 代价**：读取数据页的次数（磁盘 I/O 是最主要的成本）
- **CPU 代价**：比较记录、排序等 CPU 计算消耗

**统计信息的重要性**：CBO 的决策依赖表的统计信息（行数、每列的区分度等）。统计信息过时会导致优化器选错方案。

```sql
-- 手动更新统计信息（数据变化很大时执行）
ANALYZE TABLE orders;

-- 查看表统计信息
SHOW TABLE STATUS LIKE 'orders';
SELECT * FROM information_schema.TABLES WHERE TABLE_NAME = 'orders';
```

### 1.2 优化器的主要优化策略

**① 条件化简**：
```sql
-- 原始 SQL
WHERE 1 = 1 AND a = 1        → WHERE a = 1
WHERE a > 1 AND a > 3        → WHERE a > 3（取更严格的条件）
```

**② 常量替换**：
```sql
WHERE a = 1 AND b = a + 1    → WHERE a = 1 AND b = 2
```

**③ 子查询转换**（Semi-Join Optimization）：
```sql
-- 优化器自动将以下 IN 子查询转为半连接
WHERE id IN (SELECT user_id FROM orders WHERE amount > 100)
-- 内部转换为类似：
JOIN orders ON users.id = orders.user_id AND orders.amount > 100
```

**④ JOIN 顺序优化**：
- 自动选择最优的表连接顺序（当表数 ≤ `optimizer_search_depth` 时）
- 一般原则：选择过滤后**结果集最小**的表作为驱动表

**⑤ 索引选择**：
- 计算使用每个索引的扫描代价（预估扫描的 page 数）
- 选择代价最小的索引
- 如果全表扫描代价低于索引扫描，会选全表扫描

## 二、EXPLAIN 执行计划深度解读

### 2.1 基本用法

```sql
EXPLAIN SELECT u.name, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.age > 18
GROUP BY u.id
ORDER BY order_count DESC
LIMIT 10;
```

输出示例：
```
+----+-------------+-------+--------+----------------+---------+---------+------------------+------+----------------------------------------------+
| id | select_type | table | type   | possible_keys  | key     | key_len | ref              | rows | Extra                                        |
+----+-------------+-------+--------+----------------+---------+---------+------------------+------+----------------------------------------------+
|  1 | SIMPLE      | u     | range  | idx_age        | idx_age | 4       | NULL             |  150 | Using where; Using temporary; Using filesort |
|  1 | SIMPLE      | o     | ref    | idx_user_id    | idx_user_id | 8   | mydb.u.id        |    3 | NULL                                         |
+----+-------------+-------+--------+----------------+---------+---------+------------------+------+----------------------------------------------+
```

### 2.2 关键字段详解

#### `type`（访问类型）—— 最重要的优化指标

从好到坏排序：

| type | 含义 | 典型场景 | 是否需要优化 |
|------|------|---------|------------|
| `system` | 表只有一行 | 系统表 | 无需 |
| `const` | 主键或唯一索引等值查询 | `WHERE id = 1` | 无需 |
| `eq_ref` | 关联中的唯一索引扫描 | JOIN 时被驱动表用主键关联 | 无需 |
| `ref` | 非唯一索引等值查询 | `WHERE status = 1`（有索引）| 良好 |
| `ref_or_null` | 类似 ref，但允许 NULL | `WHERE col = 1 OR col IS NULL` | 尚可 |
| `range` | 索引范围扫描 | `WHERE id BETWEEN 1 AND 100` | 尚可 |
| `index` | 全索引扫描（比 ALL 好一点） | 覆盖索引但全扫 | 需关注 |
| `ALL` | 全表扫描 | 无索引或选错索引 | **必须优化** |

**判断标准**：生产查询中如果出现 `ALL`，几乎一定需要优化。`index` 视情况而定。

#### `Extra`（额外信息）—— 诊断关键细节

| Extra 值 | 含义 | 是好事还是坏事？ |
|---------|------|----------------|
| `Using index` | **覆盖索引**，无需回表 | ✅ 很好 |
| `Using where` | Server 层做了额外过滤 | 中性（可能是正常的） |
| `Using index condition` | 使用了 ICP（索引条件下推）| ✅ 好 |
| `Using temporary` | 使用了临时表（GROUP BY / ORDER BY）| ⚠️ 需关注，视场景优化 |
| `Using filesort` | 使用了文件排序（无法利用索引顺序）| ⚠️ 需关注，可能很慢 |
| `Using join buffer` | 使用了 join_buffer（被驱动表无索引）| ⚠️ 被驱动表缺索引 |
| `Select tables optimized away` | 优化器直接计算出结果 | ✅ 很好 |
| `Impossible WHERE` | WHERE 条件永远为 false | ⚠️ SQL 逻辑问题 |

#### `key_len` 计算

`key_len` 表示使用了索引的前多少字节，可以判断联合索引是否被充分利用：

```
BIGINT NOT NULL：8 字节
INT NOT NULL：4 字节
VARCHAR(50) utf8mb4 NOT NULL：50 × 4 + 2（长度信息）= 202 字节
VARCHAR(50) utf8mb4 NULL：50 × 4 + 2 + 1（NULL标志）= 203 字节

联合索引 (a INT, b VARCHAR(20)) 的完整 key_len = 4 + (20×4+2) = 86 字节
如果 key_len = 4，说明只用了 a 列
如果 key_len = 86，说明 a, b 都用上了
```

### 2.3 EXPLAIN FORMAT=JSON（更详细的信息）

```sql
EXPLAIN FORMAT=JSON SELECT * FROM users WHERE age > 18;
```

JSON 格式包含成本估算（`cost_info`），可以看到优化器预估的具体代价：
```json
{
  "query_block": {
    "cost_info": {
      "query_cost": "12.36"
    },
    "table": {
      "cost_info": {
        "read_cost": "8.40",
        "eval_cost": "1.50",
        "prefix_cost": "9.90",
        "data_read_per_join": "2K"
      }
    }
  }
}
```

### 2.4 EXPLAIN ANALYZE（MySQL 8.0.18+）

`EXPLAIN ANALYZE` 会**实际执行 SQL** 并输出预估值 vs 实际值，是最准确的分析工具：

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE age > 18 ORDER BY name LIMIT 10;
```

输出包含：
- `actual time=X..Y`：实际第一行到最后一行的时间（毫秒）
- `rows=N`：实际返回行数
- `loops=N`：循环次数

通过对比 `rows（预估）` 和 `actual rows（实际）`，可以发现统计信息不准确的情况。

## 三、常见执行计划问题与优化

### 3.1 全表扫描（type = ALL）

```sql
-- 问题 SQL（users 表有1000万行，无 address 索引）
EXPLAIN SELECT * FROM users WHERE address LIKE '%Beijing%';
-- type=ALL, rows=10000000 → 必须优化

-- 方案1：使用全文索引（适合全文搜索）
CREATE FULLTEXT INDEX ft_address ON users(address);
SELECT * FROM users WHERE MATCH(address) AGAINST('Beijing');

-- 方案2：迁移到 Elasticsearch（大规模文本搜索的最佳方案）
```

### 3.2 Using filesort

```sql
-- 问题：ORDER BY 使用了不在索引中的列
SELECT * FROM users WHERE city = 'Beijing' ORDER BY created_at DESC LIMIT 10;

-- 如果 idx_city 只包含 city，则 created_at 排序会触发 filesort
-- 解决：建 (city, created_at) 联合索引
CREATE INDEX idx_city_time ON users(city, created_at DESC);
```

**filesort 两种排序算法**：
- **单路排序**：一次性把查询所有字段读入 sort_buffer，排序后直接返回
- **双路排序**（旧）：先读 (id, 排序字段)，排序后再回表读数据，需要两次 I/O

MySQL 会根据 `max_length_for_sort_data` 自动选择，一般情况下单路排序更快。

### 3.3 Using temporary（临时表）

```sql
-- 问题：GROUP BY 的列不在索引中
SELECT city, COUNT(*) FROM users GROUP BY city;

-- 优化：在 city 列上建索引
CREATE INDEX idx_city ON users(city);
-- 使用索引排序的 GROUP BY 就不需要临时表了
```

### 3.4 优化器选错索引

```sql
-- 现象：EXPLAIN 显示用了错误的索引（或不用索引）
-- 可能原因：统计信息不准确，或某个索引对这条查询确实更差

-- 方案1：更新统计信息
ANALYZE TABLE users;

-- 方案2：强制使用指定索引（临时方案）
SELECT * FROM users FORCE INDEX(idx_age) WHERE age > 18;

-- 方案3：通过优化 SQL 让优化器自然选择（治本）
```

## 四、面试要点速查

| 问题 | 核心答案 |
|------|---------|
| MySQL 查询优化器的工作原理？ | CBO：枚举执行方案，计算 I/O + CPU 代价，选择成本最低的方案 |
| EXPLAIN 中最重要的字段？ | `type`（访问类型）和 `Extra`（额外信息） |
| type=ALL 意味着什么？ | 全表扫描，几乎必须优化（加索引或改写 SQL） |
| Extra 中的 "Using filesort" 意味着什么？ | 无法利用索引排序，需要额外排序操作，可能影响性能 |
| Extra 中的 "Using index" 意味着什么？ | 覆盖索引，无需回表，非常好 |
| 统计信息为什么重要？ | CBO 依赖统计信息做决策，统计信息过时会导致选错执行计划 |
| key_len 的作用？ | 判断联合索引有没有被完整利用 |
