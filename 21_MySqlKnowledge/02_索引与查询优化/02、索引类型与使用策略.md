# 索引类型与使用策略

> 知道索引的原理之后，还要掌握各种索引类型的适用场景、联合索引的设计规则，以及导致索引失效的各种陷阱。

---

## 一、索引类型全览

### 1.1 按用途分类

| 索引类型 | 创建语法 | 核心特点 | 适用场景 |
|---------|---------|---------|---------|
| **主键索引** | `PRIMARY KEY(col)` | 唯一非空，每表仅一个，InnoDB中是聚簇索引 | 每表必备，业务主键或自增ID |
| **唯一索引** | `UNIQUE INDEX(col)` | 保证字段值唯一，允许一个NULL | 邮箱、手机号、身份证等业务唯一标识 |
| **普通索引** | `INDEX(col)` | 无约束，允许重复和NULL | 频繁作为 WHERE/JOIN 条件的字段 |
| **联合索引** | `INDEX(col1, col2, col3)` | 多列复合，遵循最左前缀原则 | 多字段联合查询，覆盖索引 |
| **全文索引** | `FULLTEXT INDEX(col)` | 支持自然语言全文搜索 | 文章/评论内容搜索（大型应用推荐 ES） |
| **前缀索引** | `INDEX(col(n))` | 只对字段前 n 字节建索引 | 长字符串字段，减少索引空间 |

### 1.2 主键索引的特殊性

```sql
-- InnoDB 主键选择规则（优先级依次降低）：
-- 1. 显式定义的 PRIMARY KEY
-- 2. 第一个定义的非空唯一索引
-- 3. 自动生成的 6 字节 DB_ROW_ID（隐藏列）

-- 推荐做法：
CREATE TABLE orders (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,  -- 显式自增主键
    order_no VARCHAR(32) NOT NULL UNIQUE,           -- 业务唯一键单独建唯一索引
    user_id BIGINT UNSIGNED NOT NULL,
    amount DECIMAL(12, 2) NOT NULL,
    INDEX idx_user_id (user_id)
);
```

## 二、联合索引与最左前缀原则

### 2.1 联合索引的存储结构

联合索引 `(a, b, c)` 在 B+树中按照 **a → b → c** 的顺序排序：
- 先按 a 排序
- a 相同的，再按 b 排序
- a、b 都相同的，再按 c 排序

```
联合索引 (last_name, first_name, age) 叶子节点：
┌───────────────────────────────────────────────────────┐
│ ('Li', 'Ming', 25) → id=1                            │
│ ('Li', 'Ming', 28) → id=5                            │
│ ('Li', 'Wei', 22)  → id=3                            │
│ ('Wang', 'Fang', 30) → id=2                          │
│ ('Wang', 'Tao', 24)  → id=4                          │
└───────────────────────────────────────────────────────┘
```

### 2.2 最左前缀原则

**规则**：必须从联合索引的**最左边的列**开始，且不能跳过中间列。

设有联合索引 `idx_abc (a, b, c)`：

| 查询条件 | 使用情况 | 原因 |
|---------|---------|------|
| `WHERE a = 1` | ✅ 使用 a | 从最左列开始 |
| `WHERE a = 1 AND b = 2` | ✅ 使用 a, b | 连续使用 |
| `WHERE a = 1 AND b = 2 AND c = 3` | ✅ 使用 a, b, c | 完整使用 |
| `WHERE b = 2` | ❌ 不用索引 | 未从 a 开始 |
| `WHERE a = 1 AND c = 3` | ⚠️ 仅用 a | 跳过了 b，c 无法利用 |
| `WHERE a = 1 AND b > 2 AND c = 3` | ⚠️ 用 a, b（范围之后停止）| b 范围查询后，c 无法利用 |
| `WHERE a = 1 ORDER BY b` | ✅ 用 a（+b 用于排序）| 排序可利用索引顺序 |

**关键规律**：
- **等值条件（=）** 可以充分利用对应列的索引
- **范围条件（>, <, BETWEEN, LIKE 'prefix%'）** 之后的列无法利用索引

### 2.3 联合索引设计原则

**①"区分度高的列"放前面**

区分度（selectivity）= 不重复值数量 / 总行数，范围 (0, 1]，越接近 1 越好。

```sql
-- 查询各列的区分度
SELECT
    COUNT(DISTINCT gender) / COUNT(*) AS gender_sel,        -- 极低，不适合单独建索引
    COUNT(DISTINCT city) / COUNT(*) AS city_sel,            -- 中等
    COUNT(DISTINCT email) / COUNT(*) AS email_sel           -- 极高，适合单独唯一索引
FROM users;
```

**② 考虑覆盖索引**

把查询频繁使用的字段加入联合索引，实现覆盖索引，避免回表：

```sql
-- 查询用户列表：WHERE city ORDER BY name LIMIT
-- 如果建 (city, name)，则可以覆盖排序，避免 filesort
CREATE INDEX idx_city_name ON users(city, name);

-- 如果还需要查询 age，建 (city, name, age) 可覆盖这个查询
SELECT name, age FROM users WHERE city = 'Beijing' ORDER BY name LIMIT 10;
```

**③ 等值列在前，范围列在后**

```sql
-- 查询：WHERE status = 1 AND created_at BETWEEN ? AND ?
-- 正确：(status, created_at)  → status 等值在前，created_at 范围在后
CREATE INDEX idx_status_time ON orders(status, created_at);

-- 错误：(created_at, status) → created_at 范围之后 status 无法利用
```

## 三、索引失效场景详解

### 3.1 完整失效场景

```sql
-- 假设：CREATE INDEX idx_name ON users(name);
--       CREATE INDEX idx_age_name ON users(age, name);

-- ❌ 1. 对索引列使用函数
WHERE YEAR(created_at) = 2024          -- 改为：WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'
WHERE UPPER(name) = 'ALICE'            -- 改为：使用正确大小写，或改排序规则

-- ❌ 2. 对索引列进行计算/运算
WHERE id + 1 = 5                       -- 改为：WHERE id = 4
WHERE salary * 1.1 > 10000             -- 改为：WHERE salary > 10000 / 1.1

-- ❌ 3. 隐式类型转换（索引列和查询值类型不匹配）
WHERE phone = 13800138000              -- phone 是 VARCHAR，但用了数字
-- MySQL 会在 phone 上执行 CAST(phone AS DECIMAL)，导致函数失效

-- ❌ 4. LIKE 以 % 开头
WHERE name LIKE '%Alice'               -- 无法利用 B+树的前缀排序
-- ✅ 以 % 结尾可以用索引：
WHERE name LIKE 'Alice%'

-- ❌ 5. OR 连接包含非索引列
WHERE name = 'Alice' OR address = 'Beijing'
-- address 没有索引，整个条件退化为全表扫描
-- 改为 UNION ALL:
SELECT * FROM users WHERE name = 'Alice'
UNION ALL
SELECT * FROM users WHERE address = 'Beijing';

-- ❌ 6. 违反最左前缀（联合索引）
WHERE name = 'Alice'                   -- 联合索引是 (age, name)，age 未在条件中

-- ❌ 7. NOT IN / NOT EXISTS / != / <>
-- 这些负向条件通常无法使用索引（数据量少时优化器可能选择使用）
WHERE status != 1
WHERE id NOT IN (1, 2, 3)

-- ❌ 8. 使用 IS NULL 或 IS NOT NULL（视情况而定）
-- IS NULL 在某些情况下可以走索引，但行为因版本和优化器而异
-- 建议：字段设置 NOT NULL DEFAULT，避免 NULL
```

### 3.2 优化器可能不走索引的情况

即使查询条件可以使用索引，优化器也可能选择**全表扫描**：

```sql
-- 当索引选择性很低时（如 status 字段只有 0/1 两个值）
-- 如果大量数据 status = 1，走索引可能比全表扫描还慢
WHERE status = 1  -- 如果 90% 数据都是 status=1，优化器可能选择全表扫描

-- 强制使用索引（慎用，通常优化器比你更了解）
SELECT * FROM orders FORCE INDEX(idx_status) WHERE status = 1;

-- 提示优化器（更柔和的方式）
SELECT * FROM orders USE INDEX(idx_status) WHERE status = 1;
```

## 四、前缀索引

### 4.1 什么时候用前缀索引

当字段是长字符串（如 URL、邮箱、地址），直接建索引会占用大量空间。前缀索引只对前 n 个字节建索引：

```sql
-- 查询前缀索引的合适长度（平衡区分度和空间）
SELECT
    COUNT(DISTINCT LEFT(email, 5)) / COUNT(*) AS sel_5,
    COUNT(DISTINCT LEFT(email, 8)) / COUNT(*) AS sel_8,
    COUNT(DISTINCT LEFT(email, 10)) / COUNT(*) AS sel_10,
    COUNT(DISTINCT email) / COUNT(*) AS sel_full
FROM users;

-- 假设 sel_10 ≈ sel_full，则取前缀长度 10 即可
CREATE INDEX idx_email ON users(email(10));
```

### 4.2 前缀索引的局限

- **无法用于覆盖索引**：因为索引中只存了前缀，不能代替完整字段值
- **无法用于 ORDER BY**：排序需要完整值
- 只适合 `WHERE email = ?` 或 `WHERE email LIKE 'prefix%'`

## 五、索引条件下推（ICP）

**Index Condition Pushdown（ICP）**是 MySQL 5.6 引入的优化，把 WHERE 条件的过滤"下推"到存储引擎层（而不是在 Server 层过滤）：

```
没有 ICP 时（MySQL 5.6 之前）：
存储引擎 → 根据 name 找到主键 → 回表 → 返回行给 Server → Server 过滤 age > 20
（即使 age 不满足，也先回表再过滤，浪费 I/O）

有 ICP 时（MySQL 5.6+）：
存储引擎 → 根据 name 找到索引行 → 在存储引擎层过滤 age > 20 → 满足才回表
（不满足 age 条件的直接跳过，减少回表次数）
```

**验证 ICP 是否生效**：`EXPLAIN` 的 `Extra` 字段显示 `Using index condition`。

## 六、全文索引

### 6.1 适用场景与限制

```sql
-- 创建全文索引
CREATE FULLTEXT INDEX ft_content ON articles(title, body);

-- 使用全文索引搜索
SELECT * FROM articles
WHERE MATCH(title, body) AGAINST('MySQL database performance' IN BOOLEAN MODE);

-- 布尔模式语法
WHERE MATCH(body) AGAINST('+MySQL -Oracle' IN BOOLEAN MODE);
-- + 表示必须包含，- 表示必须排除
```

**局限性**：
- 只支持 `CHAR`、`VARCHAR`、`TEXT` 类型
- InnoDB 全文索引有停用词（Stop Words）限制
- 不支持中文（需要分词插件或改用 Elasticsearch）
- 大规模全文搜索场景，**推荐使用 Elasticsearch / OpenSearch**

## 七、索引设计黄金法则

```
✅ 正确使用索引的核心原则：

1. 高频查询的 WHERE 条件列必须有索引
2. JOIN 的关联列必须有索引
3. ORDER BY / GROUP BY 列考虑加入联合索引
4. 优先建联合索引，覆盖多个查询需求
5. 联合索引：等值条件在前，范围条件在后，SELECT 字段尽量在索引中
6. 控制单表索引数量（建议不超过 5~6 个），写多读少的表更要少建
7. 选择性低的列（如性别、状态）单独建索引意义不大
8. 字符串长字段使用前缀索引，但注意不能覆盖索引
```

## 八、面试要点速查

| 问题 | 核心答案 |
|------|---------|
| 最左前缀原则？ | 联合索引必须从最左列开始匹配，不能跳列，范围查询之后的列失效 |
| 索引失效的常见原因？ | 函数/计算/类型转换/左模糊/OR含非索引列/违反最左前缀 |
| 覆盖索引的好处？ | 无需回表，直接从索引获取所有需要的字段，EXPLAIN显示"Using index" |
| 什么情况下建索引反而更慢？ | 选择性极低的列（如性别）；小表全表扫描更快；频繁写少读的表 |
| ICP 是什么？ | 索引条件下推，将 WHERE 过滤下推到存储引擎层，减少回表次数 |
| 前缀索引的缺点？ | 无法覆盖索引，无法 ORDER BY |
