# 数据类型与字符集

> 选择正确的数据类型是表设计的基础。错误的数据类型不仅浪费存储空间，还会影响索引效率和查询性能。

---

## 一、数值类型

### 1.1 整数类型

| 类型 | 字节数 | 有符号范围 | 无符号范围 | 典型用途 |
|------|--------|-----------|-----------|---------|
| `TINYINT` | 1 | -128 ~ 127 | 0 ~ 255 | 布尔值（0/1）、状态码 |
| `SMALLINT` | 2 | -32768 ~ 32767 | 0 ~ 65535 | 年份、小编号 |
| `MEDIUMINT` | 3 | -8M ~ 8M | 0 ~ 16M | 中等规模计数 |
| `INT` | 4 | -2.1B ~ 2.1B | 0 ~ 4.2B | 通用主键（单库单表） |
| `BIGINT` | 8 | -9.2×10¹⁸ ~ 9.2×10¹⁸ | 0 ~ 1.8×10¹⁹ | 分布式主键、雪花ID |

**选型原则**：能用小类型就不用大类型，每个字节都影响内存和磁盘效率。

**UNSIGNED 的作用**：`INT UNSIGNED` 的正整数范围是 `INT` 的两倍，适合不会出现负数的主键字段。

### 1.2 精确小数类型（金融必用）

```sql
DECIMAL(M, D)
-- M：总有效位数（1~65）
-- D：小数点后位数（0~30）
-- 例：DECIMAL(10, 2) 表示最多 10 位数字，其中 2 位在小数点后
-- 存储范围：-99999999.99 ~ 99999999.99
```

**为什么金融场景必须用 DECIMAL 而非 FLOAT/DOUBLE？**

```sql
-- FLOAT 精度损失示例
SELECT 0.1 + 0.2;   -- 输出：0.30000000000000004（近似值）

-- DECIMAL 精确计算
SELECT CAST(0.1 AS DECIMAL(10,1)) + CAST(0.2 AS DECIMAL(10,1));  -- 输出：0.3
```

`FLOAT`（4字节）和 `DOUBLE`（8字节）使用 IEEE 754 浮点数表示，有精度损失，**绝对不能用于金额、利率、汇率**等精确计算场景。

## 二、字符串类型

### 2.1 CHAR vs VARCHAR

| 特性 | CHAR(N) | VARCHAR(N) |
|------|---------|-----------|
| **存储方式** | 定长，不足 N 字节用空格补齐 | 变长，实际长度 + 1~2 字节长度信息 |
| **存储空间** | 固定 N 字节（浪费空间） | 按实际长度存储（节省空间） |
| **查询速度** | 稍快（无需读取长度） | 稍慢 |
| **适用场景** | 固定长度：MD5（32位）、手机号、身份证 | 变长字段：用户名、邮箱、地址 |
| **最大长度** | 255 字节 | 65535 字节（行总长度限制） |
| **尾部空格** | 存储时去除，检索时不保留 | 保留 |

**关键知识点**：`VARCHAR(N)` 的 N 是**字符数**而非字节数。`VARCHAR(255)` 在 UTF-8 字符集下最多占用 255×3=765 字节。

### 2.2 TEXT 和 BLOB 家族

| 类型 | 最大长度 | 用途 |
|------|---------|------|
| `TINYTEXT` / `TINYBLOB` | 255 字节 | 很短的文本/二进制 |
| `TEXT` / `BLOB` | 64KB | 文章正文、图片缩略图 |
| `MEDIUMTEXT` / `MEDIUMBLOB` | 16MB | 中等大小文件 |
| `LONGTEXT` / `LONGBLOB` | 4GB | 大文件（不推荐存数据库） |

**TEXT/BLOB 的存储机制**：
- 小值（< `innodb_page_size / 2`，通常 8KB）直接内嵌存储在行记录中
- 大值存储在独立的"溢出页"中，行记录只保存 20 字节指针
- **尽量避免在频繁查询的表中存储大 TEXT/BLOB 字段**，会导致：
  1. 行记录变大，Buffer Pool 中同样大小能缓存的行数变少
  2. 全表扫描时 I/O 放大
  3. 建议将大字段拆分到独立扩展表，用主键关联

## 三、日期时间类型

| 类型 | 字节 | 格式 | 范围 | 时区 |
|------|------|------|------|------|
| `DATE` | 3 | YYYY-MM-DD | 1000-01-01 ~ 9999-12-31 | 无 |
| `TIME` | 3 | HH:MM:SS | -838:59:59 ~ 838:59:59 | 无 |
| `DATETIME` | 8 | YYYY-MM-DD HH:MM:SS | 1000-01-01 ~ 9999-12-31 | **不受时区影响** |
| `TIMESTAMP` | 4 | YYYY-MM-DD HH:MM:SS | 1970-01-01 ~ **2038-01-19** | **受时区影响** |
| `YEAR` | 1 | YYYY | 1901 ~ 2155 | 无 |

### DATETIME vs TIMESTAMP 深度对比

**TIMESTAMP 的时区行为**：
```sql
-- TIMESTAMP 存储的是 UTC 时间戳（整数）
-- 查询时自动转换为 session 时区的本地时间
SET time_zone = '+08:00';
INSERT INTO t (ts) VALUES ('2024-01-01 12:00:00');  -- 存储为 UTC 04:00:00

SET time_zone = '+00:00';
SELECT ts FROM t;  -- 返回 2024-01-01 04:00:00
```

**选型建议**：
- 用 `DATETIME`：存储创建时间、更新时间等，不受时区影响，范围更大，**推荐**
- 用 `TIMESTAMP`：需要利用自动更新特性（`DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`），且确认不会超过 2038 年
- **避免用 INT 存时间戳**：可读性差，范围问题一样存在，无任何优势

**Java 映射关系**：

| MySQL 类型 | Java 类型 |
|-----------|-----------|
| `DATETIME` | `java.time.LocalDateTime` |
| `TIMESTAMP` | `java.time.Instant` 或 `java.time.ZonedDateTime` |
| `DATE` | `java.time.LocalDate` |
| `TIME` | `java.time.LocalTime` |

## 四、JSON 类型（MySQL 5.7+）

```sql
CREATE TABLE config (
    id INT PRIMARY KEY,
    settings JSON
);

-- 插入
INSERT INTO config VALUES (1, '{"theme": "dark", "fontSize": 14}');

-- 查询（提取 JSON 字段）
SELECT settings->'$.theme' FROM config WHERE id = 1;     -- 返回 "dark"（含引号）
SELECT settings->>'$.theme' FROM config WHERE id = 1;    -- 返回 dark（去引号）

-- 生成虚拟列并创建索引
ALTER TABLE config ADD COLUMN theme VARCHAR(20)
    GENERATED ALWAYS AS (settings->>'$.theme') VIRTUAL;
CREATE INDEX idx_theme ON config(theme);
```

**JSON 类型的优缺点**：
- ✅ 灵活存储非结构化数据，内置 JSON 函数支持字段级操作
- ✅ 比 VARCHAR 存储 JSON 更紧凑（二进制格式）
- ❌ 整列 JSON 不能直接索引，需要借助**虚拟生成列**
- ❌ 不适合频繁更新的字段（更新需重写整个 JSON 值）

## 五、字符集与排序规则

### 5.1 字符集

MySQL 常用字符集：

| 字符集 | 说明 | 每字符最大字节 |
|--------|------|--------------|
| `latin1` | 西欧字符，不支持中文 | 1 |
| `utf8` | MySQL 的"假 UTF-8"，**不完整**，最多 3 字节/字符，不支持 emoji | 3 |
| `utf8mb4` | 真正的 UTF-8，支持所有 Unicode 字符（包括 emoji 🎉） | 4 |

**生产环境必须使用 `utf8mb4`**，原因：
1. `utf8` 是 MySQL 的历史遗留问题，最大只支持 3 字节，无法存储 4 字节的 emoji（如 😊）
2. 存储 emoji 会报错：`Incorrect string value`
3. `utf8mb4` 完全兼容 `utf8`

```sql
-- 推荐的全局配置
[mysqld]
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci
```

### 5.2 排序规则（Collation）

排序规则决定了字符串的**比较和排序方式**：

| 排序规则 | 特点 |
|---------|------|
| `utf8mb4_general_ci` | 速度快，但少数情况下不符合语言规范（ci = case insensitive） |
| `utf8mb4_unicode_ci` | 准确，符合 Unicode 标准，**推荐** |
| `utf8mb4_bin` | 区分大小写，按字节值比较，精确但严格 |
| `utf8mb4_0900_ai_ci` | MySQL 8.0 新增，基于 Unicode 9.0，性能更好，**MySQL 8.0 推荐** |

**大小写敏感问题**：
```sql
-- ci（case insensitive）：不区分大小写
SELECT * FROM users WHERE name = 'alice';  -- 也能匹配 'Alice', 'ALICE'

-- bin 或 cs：区分大小写
ALTER TABLE users MODIFY name VARCHAR(50) CHARACTER SET utf8mb4 COLLATE utf8mb4_bin;
SELECT * FROM users WHERE name = 'alice';  -- 只匹配 'alice'
```

## 六、数据类型选型最佳实践

### 6.1 主键选型

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 单体应用/单库单表 | `BIGINT AUTO_INCREMENT` | 自增有序，B+树写入性能最好 |
| 微服务/分库分表 | 雪花算法（`BIGINT`） | 全局唯一 + 有序 + 数字类型 |
| 避免 | `VARCHAR UUID` | 无序随机写入，B+树频繁页分裂 |

**为什么自增主键对 InnoDB 性能更好**：
- InnoDB 的聚簇索引按主键顺序存储数据
- 自增 ID 每次都在末尾插入，无需移动现有数据
- UUID 随机插入会导致频繁**页分裂（Page Split）**，碎片化严重

### 6.2 避免 NULL 的理由

```sql
-- NULL 使索引统计复杂化
-- NULL 需要特殊比较语法（IS NULL / IS NOT NULL）
-- NULL 值在聚合函数中被忽略，容易出错

-- 推荐：为字段设置 DEFAULT 值
CREATE TABLE users (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL DEFAULT '',
    age TINYINT UNSIGNED NOT NULL DEFAULT 0,
    email VARCHAR(100) NOT NULL DEFAULT '',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## 七、面试要点速查

| 问题 | 核心答案 |
|------|---------|
| 为什么不用 FLOAT 存金额？ | 浮点数有精度损失，应用 DECIMAL(M,D) |
| DATETIME vs TIMESTAMP？ | DATETIME 不受时区影响范围大；TIMESTAMP 自动更新但范围到 2038 年 |
| utf8 和 utf8mb4 的区别？ | utf8 最多 3 字节不支持 emoji；utf8mb4 完整 UTF-8 编码，生产必用 |
| 为什么推荐自增主键？ | 有序递增，InnoDB 聚簇索引写入在末尾，无页分裂 |
| VARCHAR(N) 的 N 是字节还是字符？ | 字符数。UTF-8 下每字符最多 3~4 字节 |
