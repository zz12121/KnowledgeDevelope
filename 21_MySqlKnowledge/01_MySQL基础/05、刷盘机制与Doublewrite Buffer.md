# 刷盘机制与 Doublewrite Buffer

> 数据从内存写入磁盘是数据库最核心的操作之一。InnoDB 通过精妙的刷盘机制，在保证数据持久性的同时最大化 I/O 性能。Doublewrite Buffer 则是解决"部分页写入"问题的关键设计。

---

## 1. 刷盘全景概览

### 1.1 为什么要刷盘

```
MySQL 中的数据流向：

┌─────────────────────────────────────────────────────────────────────────┐
│                              内存层                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │ Buffer Pool  │  │ Log Buffer   │  │ Change Buffer│  │ AHI          │ │
│  │ （数据页）    │  │ （Redo Log） │  │ （变更缓冲） │  │ （哈希索引） │ │
│  └──────┬───────┘  └──────┬───────┘  └──────────────┘  └──────────────┘ │
│         │                 │                                             │
│         │ 刷盘（Flush）    │ 刷盘（Flush）                               │
│         ▼                 ▼                                             │
└─────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────┐
│                              磁盘层                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │ .ibd 文件    │  │ ib_logfile*  │  │ ibdata1      │  │ Doublewrite  │ │
│  │ （表数据）    │  │ （Redo Log） │  │ （系统表空间）│  │ Buffer       │ │
│  └──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘ │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

刷盘的核心矛盾：
- 内存是易失的（断电丢失）
- 磁盘是持久的（断电不丢）
- 但磁盘 I/O 比内存慢 1000 倍以上

解决方案：WAL（Write-Ahead Logging）+ 异步批量刷盘
```

### 1.2 三种刷盘场景

| 刷盘类型 | 刷什么 | 触发条件 | 性能影响 |
|---------|--------|---------|---------|
| **Redo Log 刷盘** | 事务日志 | 事务提交时 | 关键路径，需 fsync |
| **Binlog 刷盘** | 逻辑日志 | 事务提交时 | 关键路径，需 fsync |
| **数据页刷盘** | Buffer Pool 脏页 | 异步批量 | 后台线程，不阻塞事务 |

## 2. Redo Log 刷盘机制详解

### 2.1 Redo Log 的物理结构

```
Redo Log 文件组（循环写入）：

┌─────────────────────────────────────────────────────────────────────────┐
│                        Redo Log 文件组                                   │
│                    （默认 2 个文件，每个 48MB）                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  ib_logfile0                          ib_logfile1                        │
│  ┌─────────────────────────┐          ┌─────────────────────────┐       │
│  │ Block 0 │ Block 1 │ ... │ ───────▶ │ ... │ Block N │ Block 0 │       │
│  │ 512B    │ 512B    │     │          │     │ 512B    │ 512B    │       │
│  └─────────────────────────┘          └─────────────────────────┘       │
│         │                                      │                        │
│         ▼                                      ▼                        │
│    write pos（当前写入位置）              checkpoint（已刷盘位置）        │
│                                                                          │
│  空白区域 = 可写入的新日志                                                │
│  填充区域 = 已写入但未刷脏页的日志（崩溃恢复时需要）                       │
│                                                                          │
│  当 write pos 追上 checkpoint 时，必须先推进 checkpoint（刷脏页）         │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

Redo Log Block 结构（512 字节）：
┌─────────────────────────────────────────────────────────────────────────┐
│  Log Block Header（12字节）                                              │
│  ├── Block Number（4字节）：块编号                                        │
│  ├── Data Length（2字节）：实际数据长度                                   │
│  ├── First Record Offset（2字节）：第一个日志记录在块内的偏移              │
│  └── Checkpoint Number（4字节）： checkpoint 序号                         │
├─────────────────────────────────────────────────────────────────────────┤
│  Log Block Body（约500字节）                                             │
│  └── 实际的 Redo Log 记录（MLOG_* 类型）                                  │
├─────────────────────────────────────────────────────────────────────────┤
│  Log Block Trailer（4字节）                                              │
│  └── Checksum（CRC32校验和）                                             │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 Redo Log 刷盘策略

```sql
-- 核心参数：innodb_flush_log_at_trx_commit
-- 控制事务提交时 Redo Log 的刷盘行为

┌─────────────────────────────────────────────────────────────────────────┐
│  设置值 = 1（默认，最高安全性）                                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  事务提交流程：                                                           │
│  1. 事务执行，生成 Redo Log 记录                                          │
│  2. 写入 Log Buffer（用户线程）                                           │
│  3. 事务提交                                                              │
│  4. fsync() 将 Log Buffer 刷到磁盘（强制同步）                            │
│  5. 返回客户端"提交成功"                                                  │
│                                                                          │
│  特点：每次提交都 fsync，最安全，性能最低                                  │
│  适用：金融、支付等强一致性场景                                            │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│  设置值 = 2（平衡模式）                                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  事务提交流程：                                                           │
│  1. 事务执行，生成 Redo Log 记录                                          │
│  2. 写入 Log Buffer                                                       │
│  3. 事务提交                                                              │
│  4. write() 写入 OS Page Cache（不 fsync）                                │
│  5. 返回客户端"提交成功"                                                  │
│  6. 每秒后台线程 fsync 一次                                               │
│                                                                          │
│  特点：MySQL 崩溃不丢数据，OS 崩溃丢 1 秒数据                              │
│  适用：大多数业务场景（推荐）                                              │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│  设置值 = 0（最高性能）                                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  事务提交流程：                                                           │
│  1. 事务执行，生成 Redo Log 记录                                          │
│  2. 写入 Log Buffer                                                       │
│  3. 事务提交（不刷盘！）                                                   │
│  4. 每秒后台线程 write + fsync 一次                                       │
│                                                                          │
│  特点：MySQL 崩溃可能丢 1 秒数据                                           │
│  适用：日志、统计等对一致性要求不高的场景                                  │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.3 Redo Log 刷盘性能对比

| 设置值 | 安全性 | 性能（TPS） | 适用场景 |
|--------|--------|------------|---------|
| **1** | 最高（零丢失） | 基准（100%） | 金融交易、支付系统 |
| **2** | 较高（OS崩溃丢1秒） | ~300% | 大多数业务系统（推荐） |
| **0** | 低（MySQL崩溃丢1秒） | ~400% | 日志、监控、非核心数据 |

## 3. 数据页刷盘机制详解

### 3.1 脏页的产生

```
UPDATE 操作的数据流向：

┌─────────────────────────────────────────────────────────────────────────┐
│  1. 执行 UPDATE users SET name='Alice' WHERE id=1;                      │
│                                                                          │
│  2. 读取数据页（Page 100）到 Buffer Pool（如果不在内存）                  │
│     ┌───────────────────────────────────────────────────────────────┐   │
│     │  Page 100（内存）                                              │   │
│     │  ┌─────────────────────────────────────────────────────────┐  │   │
│     │  │ id=1 │ name=Bob │ ...                                   │  │   │
│     │  └─────────────────────────────────────────────────────────┘  │   │
│     └───────────────────────────────────────────────────────────────┘   │
│                                                                          │
│  3. 在 Buffer Pool 中修改数据                                             │
│     ┌───────────────────────────────────────────────────────────────┐   │
│     │  Page 100（内存，已修改 = 脏页）                                │   │
│     │  ┌─────────────────────────────────────────────────────────┐  │   │
│     │  │ id=1 │ name=Alice │ ...  ← 修改了 name 字段              │  │   │
│     │  └─────────────────────────────────────────────────────────┘  │   │
│     │  Dirty Bit = 1                                                │   │
│     └───────────────────────────────────────────────────────────────┘   │
│                                                                          │
│  4. 生成 Redo Log 记录（保证崩溃可恢复）                                  │
│     MLOG_REC_UPDATE: 表空间ID=5, 页号=100, 偏移=128, 旧值=Bob, 新值=Alice │
│                                                                          │
│  5. 事务提交，Redo Log 刷盘                                               │
│                                                                          │
│  6. 数据页（脏页）不立即刷盘！等待后台线程异步刷盘                          │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

关键设计：WAL（Write-Ahead Logging）
- 先写日志（Redo Log），再刷数据页
- 日志是顺序写，速度快
- 数据页是随机写，速度慢但可批量异步执行
```

### 3.2 脏页刷盘的触发条件

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       脏页刷盘触发条件                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  1. LRU 链表淘汰                                                          │
│     └── 当 Buffer Pool 满，需要加载新页时                                 │
│         └── 从 LRU 尾部淘汰页，如果是脏页则先刷盘                          │
│                                                                          │
│  2. Redo Log 空间不足                                                     │
│     └── write pos 接近 checkpoint                                         │
│         └── 必须推进 checkpoint → 刷脏页                                   │
│                                                                          │
│  3. 定时刷新                                                              │
│     └── 每秒刷新一次（默认）                                              │
│         └── 保证脏页不会停留太久                                          │
│                                                                          │
│  4. 脏页比例过高                                                          │
│     └── 脏页占 Buffer Pool 比例 > innodb_max_dirty_pages_pct（默认75%）   │
│         └── 加快刷盘速度                                                  │
│                                                                          │
│  5. 系统空闲时                                                            │
│     └── MySQL 检测到负载较低                                              │
│         └── 主动刷盘，为高峰期预留空间                                     │
│                                                                          │
│  6. 正常关闭                                                              │
│     └── 关闭前将所有脏页刷盘                                              │
│         └── 保证干净关闭（fast shutdown = 0）                              │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 Flush List 与刷盘顺序

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       Flush List 结构                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  Flush List 是按**最早修改时间（oldest_modification）**排序的链表          │
│                                                                          │
│  头部（最早修改）                                            尾部        │
│     │                                                         │          │
│     ▼                                                         ▼          │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  Page A → Page B → Page C → Page D → Page E → ...               │    │
│  │  LSN=100   LSN=150  LSN=200  LSN=300  LSN=500                   │    │
│  │  （最早）                                                    （最晚）│    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                          │
│  Checkpoint 推进：                                                        │
│  - 刷盘 Page A（LSN=100）→ Checkpoint 推进到 100                          │
│  - 刷盘 Page B（LSN=150）→ Checkpoint 推进到 150                          │
│  - 刷盘 Page C（LSN=200）→ Checkpoint 推进到 200                          │
│                                                                          │
│  为什么按 LSN 排序？                                                       │
│  - 保证 Checkpoint 单调递增                                               │
│  - 崩溃恢复时从 Checkpoint 开始扫描 Redo Log                              │
│  - 如果 Page C 比 Page B 先刷盘，Checkpoint 推进到 200                    │
│    但 Page B（LSN=150）还未刷盘，崩溃后会丢失 Page B 的修改！               │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.4 刷盘的关键参数

```sql
-- Page Cleaner 线程数（负责刷脏页）
-- 默认 4，SSD 可增大到 8~16
SET GLOBAL innodb_page_cleaners = 8;

-- 脏页占 Buffer Pool 比例达到多少开始刷盘（默认 75%）
SET GLOBAL innodb_max_dirty_pages_pct = 75.0;

-- 脏页达到此比例强制刷盘（默认 90%）
SET GLOBAL innodb_max_dirty_pages_pct_lwm = 10.0;

-- 每秒 I/O 操作数上限（默认 200，SSD 可设 2000+）
SET GLOBAL innodb_io_capacity = 2000;

-- 紧急情况下每秒 I/O 操作数上限（默认 2000）
SET GLOBAL innodb_io_capacity_max = 4000;

-- 自适应刷盘（根据脏页产生速度动态调整）
SET GLOBAL innodb_adaptive_flushing = ON;

-- 自适应刷盘的脏页比例上限
SET GLOBAL innodb_adaptive_flushing_lwm = 10;

-- 查看刷盘状态
SHOW ENGINE INNODB STATUS\G
-- 关注：
-- - Modified db pages：当前脏页数
-- - Pending flushes (fsync) log：待刷盘的日志
-- - Pages flushed up to：已刷盘的 LSN
```

## 4. Doublewrite Buffer 详解

### 4.1 什么是"部分页写入"

```
问题场景：16KB 数据页写入磁盘时系统崩溃

┌─────────────────────────────────────────────────────────────────────────┐
│  正常写入流程：                                                           │
│                                                                          │
│  Buffer Pool 中的脏页（16KB）                                             │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  Page Data（完整的16KB）                                         │    │
│  │  ┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐│    │
│  │  │ 2KB    │ 2KB    │ 2KB    │ 2KB    │ 2KB    │ 2KB    │ 2KB    ││    │
│  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │  │    │
│  │  └────┴───┴────┴───┴────┴───┴────┴───┴────┴───┴────┴───┴────┴───┘│    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                              │                                           │
│                              ▼ write() 系统调用                          │
│  磁盘上的数据页（16KB）                                                    │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  Page Data（完整的16KB）                                         │    │
│  │  ┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐│    │
│  │  │ 2KB    │ 2KB    │ 2KB    │ 2KB    │ 2KB    │ 2KB    │ 2KB    ││    │
│  │  └────────┴────────┴────────┴────────┴────────┴────────┴────────┘│    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│  部分页写入问题（写入过程中崩溃）：                                         │
│                                                                          │
│  Buffer Pool 中的脏页（16KB）                                             │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  Page Data（完整的16KB）                                         │    │
│  │  ┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐│    │
│  │  │ 新数据 │ 新数据 │ 新数据 │ 新数据 │ 新数据 │ 新数据 │ 新数据 ││    │
│  │  └────────┴────────┴────────┴────────┴────────┴────────┴────────┘│    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                              │                                           │
│                              ▼ 写入过程中崩溃！                            │
│  磁盘上的数据页（损坏！）                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │  Page Data（损坏的16KB）                                         │    │
│  │  ┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐│    │
│  │  │ 新数据 │ 新数据 │ 新数据 │ 旧数据 │ 旧数据 │ 旧数据 │ 旧数据 ││    │
│  │  │ （已写）      │ （已写）      │ （已写）      │ （未写）              │    │
│  │  └────────┴────────┴────────┴────────┴────────┴────────┴────────┘│    │
│  │                                                                  │    │
│  │  结果：页数据一半新一半旧，完全损坏！                                   │    │
│  │  问题：Redo Log 恢复需要旧的完整页作为基础，但页已损坏！                  │    │
│  │  Redo Log 记录的是"在旧页基础上如何修改"，不是完整页内容！                │    │
│  └─────────────────────────────────────────────────────────────────┘    │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.2 Doublewrite Buffer 的工作原理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Doublewrite Buffer 工作流程                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  Doublewrite Buffer 是系统表空间（ibdata1）中的一块连续区域                 │
│  大小：2MB（128 个页）                                                    │
│  结构：分为两个 1MB 的块（Doublewrite Block 1 和 Block 2）                 │
│                                                                          │
│  刷盘流程：                                                               │
│                                                                          │
│  1. 收集脏页（每次最多 128 个页）                                          │
│     ┌─────────┐ ┌─────────┐ ┌─────────┐         ┌─────────┐             │
│     │ Dirty 1 │ │ Dirty 2 │ │ Dirty 3 │  ...    │ Dirty N │             │
│     └────┬────┘ └────┬────┘ └────┬────┘         └────┬────┘             │
│          │           │           │                   │                  │
│          └───────────┴───────────┴───────────────────┘                  │
│                              │                                           │
│                              ▼                                           │
│  2. 第一步：顺序写入 Doublewrite Buffer（2MB 连续空间）                     │
│     ┌─────────────────────────────────────────────────────────────────┐ │
│     │  Doublewrite Buffer（ibdata1 中）                                │ │
│     │  ┌────────────────────────────────────────────────────────────┐ │ │
│     │  │ Page 1 │ Page 2 │ Page 3 │ ... │ Page N │ 空闲 │ 空闲 │...│ │ │
│     │  └────────────────────────────────────────────────────────────┘ │ │
│     │                                                                  │ │
│     │  特点：顺序写，速度快，原子性（2MB 写入是原子的）                      │ │
│     └─────────────────────────────────────────────────────────────────┘ │
│                              │                                           │
│                              ▼ fsync()                                   │
│  3. 等待 Doublewrite Buffer 刷盘完成                                       │
│                                                                          │
│                              │                                           │
│                              ▼                                           │
│  4. 第二步：离散写入实际表空间位置（.ibd 文件）                              │
│     ┌─────────┐      ┌─────────┐      ┌─────────┐                       │
│     │ .ibd 1  │      │ .ibd 2  │      │ .ibd 3  │                       │
│     │ ┌─────┐ │      │ ┌─────┐ │      │ ┌─────┐ │                       │
│     │ │Page1│ │      │ │Page2│ │      │ │Page3│ │                       │
│     │ └─────┘ │      │ └─────┘ │      │ └─────┘ │                       │
│     └─────────┘      └─────────┘      └─────────┘                       │
│                                                                          │
│     特点：随机写，速度慢，可能部分写入                                       │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

崩溃恢复时的处理：

情况1：Doublewrite Buffer 写入完成，.ibd 写入未完成（部分页写入）
  → 从 Doublewrite Buffer 中读取完整页，覆盖 .ibd 中的损坏页
  → 然后应用 Redo Log 恢复

情况2：Doublewrite Buffer 写入未完成
  → Doublewrite Buffer 中的页不完整，但 .ibd 中的页还是旧的完整页
  → 直接用 Redo Log 恢复 .ibd 中的页即可

无论哪种情况，始终有一个完整的页可用！
```

### 4.3 Doublewrite Buffer 的性能影响

```
性能开销分析：

┌─────────────────────────────────────────────────────────────────────────┐
│  写放大（Write Amplification）                                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  无 Doublewrite Buffer：                                                  │
│    脏页 → 直接写入 .ibd（1 次随机 I/O）                                    │
│    总写入量：16KB                                                        │
│                                                                          │
│  有 Doublewrite Buffer：                                                  │
│    脏页 → Doublewrite Buffer（顺序写）                                    │
│         → .ibd 实际位置（随机写）                                         │
│    总写入量：16KB × 2 = 32KB（2 倍写放大）                                 │
│                                                                          │
│  但实际影响：                                                             │
│  - Doublewrite Buffer 是顺序写，速度快（接近内存速度）                       │
│  - 主要开销是额外的 fsync 调用                                             │
│  - SSD 场景下影响较小（SSD 顺序写和随机写差距不大）                           │
│  - 通常性能损失 < 5%，但保证了数据安全                                       │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

禁用 Doublewrite Buffer（仅限特定场景）：

-- 如果底层存储支持原子写（如 Fusion-io），可以禁用
SET GLOBAL innodb_doublewrite = OFF;

-- 或者使用 O_DIRECT 绕过 OS 缓存，配合特定硬件
[mysqld]
innodb_flush_method = O_DIRECT
innodb_doublewrite = OFF  -- 仅当存储支持 16KB 原子写时
```

### 4.4 Doublewrite Buffer 相关参数

```sql
-- 是否启用 Doublewrite Buffer（默认 ON）
SET GLOBAL innodb_doublewrite = ON;

-- 查看 Doublewrite Buffer 使用情况
SHOW GLOBAL STATUS LIKE 'Innodb_dblwr%';

-- 关键指标：
-- Innodb_dblwr_pages_written：写入 Doublewrite Buffer 的页数
-- Innodb_dblwr_writes：Doublewrite Buffer 写次数
-- Innodb_dblwr_page_writes：从 Doublewrite Buffer 恢复的页数（崩溃后）

-- 计算 Doublewrite Buffer 效率
-- 如果 Innodb_dblwr_writes 很大但 Innodb_dblwr_pages_written 很小
-- 说明有很多小批量写入，可以优化刷盘策略
```

## 5. 崩溃恢复（Crash Recovery）

### 5.1 崩溃恢复的完整流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     MySQL 崩溃恢复流程                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  1. 启动时检测                                                           │
│     └── 检查 ibdata1 中的 checkpoint 信息                                 │
│         └── 获取上次刷盘的位置（checkpoint LSN）                          │
│                                                                          │
│  2. 扫描 Redo Log                                                        │
│     └── 从 checkpoint LSN 开始扫描 Redo Log                               │
│         └── 收集所有已提交事务的修改记录                                   │
│                                                                          │
│  3. 检查数据页完整性                                                      │
│     └── 对每个需要恢复的页：                                               │
│         ├── 读取 .ibd 文件中的页                                           │
│         ├── 检查页的 Checksum                                              │
│         ├── Checksum 失败 → 从 Doublewrite Buffer 恢复                    │
│         └── Checksum 成功 → 继续下一步                                     │
│                                                                          │
│  4. 应用 Redo Log（前滚）                                                 │
│     └── 对每个页应用 Redo Log 记录                                         │
│         └── 页 LSN < Redo Log LSN → 应用该记录                             │
│                                                                          │
│  5. 处理 Prepare 状态事务（两阶段提交）                                     │
│     └── 扫描 Undo Log，找到处于 Prepare 状态的事务                          │
│         ├── 检查 Binlog 中是否有该事务的完整记录                            │
│         ├── Binlog 有 → 提交该事务（Commit）                               │
│         └── Binlog 无 → 回滚该事务（Rollback）                             │
│                                                                          │
│  6. 清理 Undo Log                                                        │
│     └── 删除已提交事务的 Undo Log                                          │
│                                                                          │
│  7. 恢复完成，接受连接                                                    │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.2 Redo Log 恢复的核心逻辑

```
Redo Log 记录格式：

┌─────────────────────────────────────────────────────────────────────────┐
│  MLOG_REC_UPDATE（数据页更新记录）                                        │
├─────────────────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────────────────────┐│
│  │  Type        │ 1字节  │ 记录类型（MLOG_REC_UPDATE = 0x1D）          ││
│  │  Space ID    │ 4字节  │ 表空间 ID                                  ││
│  │  Page Number │ 4字节  │ 页号                                       ││
│  │  Offset      │ 2字节  │ 页内偏移                                   ││
│  │  Old Data    │ 变长   │ 旧数据（用于生成 Undo）                     ││
│  │  New Data    │ 变长   │ 新数据                                     ││
│  └─────────────────────────────────────────────────────────────────────┘│
│                                                                          │
│  恢复时的判断逻辑：                                                        │
│                                                                          │
│  1. 读取数据页（从 .ibd 或 Doublewrite Buffer）                            │
│  2. 获取页头中的 Page LSN（该页最后刷盘时的 LSN）                          │
│  3. 比较 Page LSN 和 Redo Log LSN：                                        │
│                                                                          │
│     if (Page LSN >= Redo Log LSN)                                         │
│         该修改已在页中，跳过                                               │
│     else                                                                  │
│         应用该 Redo Log 记录到页中                                         │
│                                                                          │
│  4. 更新 Page LSN 为 Redo Log LSN                                          │
│  5. 继续下一条记录                                                         │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.3 恢复时间优化

```sql
-- 查看上次崩溃恢复时间
-- 从错误日志中搜索 "InnoDB: Database was not shutdown normally"
-- 到 "InnoDB: Ready for connections" 的时间差

-- 优化恢复时间的参数

-- Redo Log 文件大小（默认 48MB，建议 1~4GB）
-- 太小：频繁 checkpoint，影响性能
-- 太大：崩溃恢复时间长
[mysqld]
innodb_log_file_size = 1G
innodb_log_files_in_group = 4

-- 优化恢复过程的并发度
innodb_read_io_threads = 16    -- 读取线程
innodb_write_io_threads = 16   -- 写入线程

-- 快速关闭（不刷脏页，依赖 Redo Log 恢复）
-- 适合需要快速重启的场景
SET GLOBAL innodb_fast_shutdown = 1;  -- 默认 1
-- 0 = 完全关闭（刷所有脏页，慢但启动快）
-- 1 = 快速关闭（不刷脏页，依赖恢复）
-- 2 = 强制关闭（类似 kill -9，需要完整恢复）
```

## 6. 刷盘机制的监控与调优

### 6.1 核心监控指标

```sql
-- 1. Redo Log 状态
SHOW ENGINE INNODB STATUS\G
-- 关注：
-- - Log sequence number：当前 LSN
-- - Log flushed up to：已刷盘的 LSN
-- - Pages flushed up to：数据页已刷盘的 LSN
-- - Last checkpoint at：上次 checkpoint LSN

-- 计算未刷盘的日志量（过大说明刷盘跟不上）
-- Log sequence number - Last checkpoint at

-- 2. 脏页状态
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_pages_%';
-- - Innodb_buffer_pool_pages_dirty：当前脏页数
-- - Innodb_buffer_pool_pages_total：总页数
-- - 脏页比例 = dirty / total

-- 3. 刷盘统计
SHOW GLOBAL STATUS LIKE 'Innodb_pages_%';
-- - Innodb_pages_written：已写入的页数
-- - Innodb_pages_read：已读取的页数

-- 4. Doublewrite Buffer 统计
SHOW GLOBAL STATUS LIKE 'Innodb_dblwr%';
```

### 6.2 常见问题诊断

| 问题 | 诊断方法 | 解决方案 |
|------|---------|---------|
| **刷盘跟不上** | checkpoint age 接近 max | 增大 innodb_io_capacity，增加 page_cleaners |
| **Redo Log 满** | write pos 接近 checkpoint | 增大 innodb_log_file_size |
| **脏页比例高** | dirty pages > 75% | 调整 innodb_max_dirty_pages_pct |
| **恢复时间长** | 错误日志显示恢复耗时 | 减小 innodb_log_file_size，或优化 io_threads |
| **部分页写入** | Innodb_dblwr_page_writes > 0 | 正常现象，说明 Doublewrite 在工作 |

### 6.3 生产环境配置建议

```ini
[mysqld]
# Redo Log 配置（SSD 环境）
innodb_log_file_size = 2G
innodb_log_files_in_group = 4
innodb_log_buffer_size = 64M

# 刷盘策略（平衡性能和安全）
innodb_flush_log_at_trx_commit = 2
innodb_flush_method = O_DIRECT  # 绕过 OS 缓存，减少双重缓存

# 脏页刷盘（SSD 优化）
innodb_page_cleaners = 16
innodb_io_capacity = 5000
innodb_io_capacity_max = 10000
innodb_max_dirty_pages_pct = 75
innodb_max_dirty_pages_pct_lwm = 10
innodb_adaptive_flushing = ON

# Doublewrite Buffer（SSD 可关闭，如果支持原子写）
# innodb_doublewrite = OFF  # 谨慎使用！

# I/O 线程
innodb_read_io_threads = 16
innodb_write_io_threads = 16
```

## 7. 面试要点速查

| 问题 | 核心答案 |
|------|---------|
| 什么是部分页写入？ | 16KB 页写入磁盘时崩溃，导致页数据一半新一半旧，完全损坏 |
| Doublewrite Buffer 解决什么问题？ | 解决部分页写入问题，保证崩溃恢复时有完整的页可用 |
| Doublewrite Buffer 的工作原理？ | 脏页先顺序写入 Doublewrite Buffer，再随机写入 .ibd；崩溃时从 DWB 恢复 |
| innodb_flush_log_at_trx_commit 三个值的区别？ | 1=每次提交 fsync（最安全）；2=写 OS 缓存每秒 fsync（平衡）；0=每秒后台刷盘（最快） |
| 脏页什么时候刷盘？ | LRU 淘汰、Redo Log 满、定时、脏页比例高、系统空闲、正常关闭 |
| 为什么 Redo Log 刷盘是顺序写？ | Redo Log 是追加写入，天然顺序；数据页是随机位置修改，是随机写 |
| 崩溃恢复时怎么处理 Prepare 状态的事务？ | 检查 Binlog，有则提交，无则回滚 |
