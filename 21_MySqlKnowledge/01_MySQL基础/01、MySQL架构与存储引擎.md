# MySQL 架构与存储引擎

> 理解 MySQL 的整体架构和存储引擎机制，是掌握所有高级特性的前提。本章从顶层视角建立 MySQL 的"立体模型"。

---

## 一、MySQL 整体架构

### 1.1 三层架构总览

MySQL 采用经典的**客户端-服务器（C/S）架构**，服务端内部分为三个核心层次：

```
┌─────────────────────────────────────────────────────────┐
│                      Client Layer                        │
│        (MySQL CLI / JDBC / Python mysql-connector)       │
└──────────────────────┬──────────────────────────────────┘
                       │ TCP/IP (Protocol: MySQL/X)
┌──────────────────────▼──────────────────────────────────┐
│                   Connection Layer                        │
│  • 连接管理（线程池/连接池）                               │
│  • 身份认证（用户名+密码+Host）                            │
│  • 权限验证（GRANT 表）                                   │
│  • 连接限流（max_connections）                            │
└──────────────────────┬──────────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────────┐
│                    SQL Layer                             │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────────┐  │
│  │  Parser  │→│ PreProc  │→│Optimizer │→│ Executor  │  │
│  │ 词法/语法 │ │ 语义检查  │ │ 执行计划  │ │  调用引擎  │  │
│  └──────────┘ └──────────┘ └──────────┘ └───────────┘  │
│  • Query Cache（MySQL 8.0 已移除）                       │
└──────────────────────┬──────────────────────────────────┘
                       │  Handler API
┌──────────────────────▼──────────────────────────────────┐
│                Storage Engine Layer                      │
│   ┌─────────┐  ┌─────────┐  ┌────────┐  ┌──────────┐   │
│   │ InnoDB  │  │ MyISAM  │  │ Memory │  │ Archive  │   │
│   └─────────┘  └─────────┘  └────────┘  └──────────┘   │
└──────────────────────┬──────────────────────────────────┘
                       │
               ┌───────▼───────┐
               │  File System  │
               │  (OS / Disk)  │
               └───────────────┘
```

**关键设计特点**：Storage Engine Layer 通过**可插拔架构（Pluggable Storage Engine）** 与 SQL 层解耦，上层 SQL 处理完全不感知底层存储细节，只通过统一的 Handler API 交互。这意味着不同的表可以使用不同的存储引擎。

### 1.2 SQL 执行全流程

一条 `SELECT` 语句从客户端发出到返回结果，在 MySQL 内部经历了以下步骤：

**① 建立连接**
- 客户端（JDBC）发起 TCP 连接到 MySQL Server
- Connection Layer 进行**身份验证**（用户名、密码、来源 IP 校验）
- 验证通过后，从线程池取出一个工作线程处理这条连接

**② Parser（解析器）**
- **词法分析**：把 SQL 字符串拆成 Token（关键字、表名、字段名、运算符等）
- **语法分析**：验证 Token 序列是否符合 SQL 语法规范，生成**抽象语法树（AST）**
- 语法错误在此阶段被捕获，返回 `You have an error in your SQL syntax`

**③ Pre-Processor（预处理器）**
- 检查 AST 中引用的表、字段是否实际存在（语义检查）
- 把 `*` 展开成实际的字段列表
- 校验权限（是否有 SELECT 权限）

**④ Optimizer（查询优化器）**
- 核心任务：为 SQL 生成一个**代价最低**的执行计划
- 基于成本模型（CBO, Cost-Based Optimizer）评估各种可能方案
- 决定：使用哪个索引、多表 JOIN 的顺序、是否使用临时表/排序等
- 输出：执行计划（可通过 `EXPLAIN` 查看）

**⑤ Executor（执行器）**
- 按照执行计划，通过 **Handler API** 调用存储引擎接口
- 每次调用可能返回一行或一批行数据
- 对每行数据进行 WHERE 过滤（Server 层过滤）、聚合、排序等操作
- 组装结果集返回客户端

## 二、存储引擎详解

### 2.1 存储引擎对比

| 特性 | InnoDB | MyISAM | Memory | Archive |
|------|--------|--------|--------|---------|
| **事务支持** | ✅ ACID | ❌ | ❌ | ❌ |
| **锁粒度** | 行级锁 | 表级锁 | 表级锁 | 行级锁 |
| **外键** | ✅ | ❌ | ❌ | ❌ |
| **MVCC** | ✅ | ❌ | ❌ | ❌ |
| **全文索引** | ✅（5.6+） | ✅ | ❌ | ❌ |
| **COUNT(*)** | 需实时统计 | 直接返回 | 需实时统计 | - |
| **崩溃恢复** | ✅ 强（redo log） | ❌ 弱 | ❌（数据丢失） | - |
| **数据存储** | 磁盘持久化 | 磁盘持久化 | 纯内存 | 磁盘（高压缩） |
| **适用场景** | 绝大多数 OLTP | 读多写少 | 缓存/临时表 | 归档日志 |

**自 MySQL 5.5 起，InnoDB 是默认存储引擎**，新项目无特殊需求应始终选择 InnoDB。

### 2.2 InnoDB 深度解析

#### 内存结构

```
┌──────────────────────── InnoDB 内存结构 ────────────────────────┐
│                                                                  │
│  ┌─────────────────────── Buffer Pool ─────────────────────┐   │
│  │  Data Pages │ Index Pages │ Insert Buffer │ Undo Pages  │   │
│  │                                                          │   │
│  │  • LRU算法管理（Young + Old 区）                          │   │
│  │  • 默认128MB，生产建议 50%~80% 物理内存                   │   │
│  │  • 脏页由 Flush Thread 异步写回磁盘                        │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌──── Log Buffer ────┐   ┌──── Change Buffer ─────────────┐   │
│  │ redo log 的内存缓冲 │   │ 非唯一辅助索引的变更暂存         │   │
│  │ 事务提交时 fsync    │   │ 读页时合并，减少随机 I/O        │   │
│  └────────────────────┘   └────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

**Buffer Pool 是 InnoDB 性能的核心**：
- 所有数据读写都先经过 Buffer Pool
- 读操作：先查 Buffer Pool，命中则直接返回（缓存命中），未命中则从磁盘加载页
- 写操作：修改 Buffer Pool 中的页（产生脏页），**不立即写磁盘**，由后台线程批量刷盘
- 通过 WAL（先写日志）保证崩溃安全：脏页未刷盘时 redo log 已持久化

**Buffer Pool 的 LRU 改进**：
- 传统 LRU 有"缓存污染"问题（如全表扫描会把热数据挤出）
- InnoDB 将 LRU 链表分为 **Young（5/8）** 和 **Old（3/8）** 两个区
- 新加载的页先进入 Old 区头部
- 只有在 Old 区停留超过 1 秒后再次被访问，才会晋升到 Young 区
- 全表扫描的页在 Old 区快速淘汰，不影响热数据

#### 磁盘结构

```
InnoDB 磁盘文件：
├── ibdata1              ← 系统表空间（数据字典、Doublewrite Buffer）
├── ib_logfile0          ← Redo Log 文件（循环写入）
├── ib_logfile1          ← Redo Log 文件
└── user_db/
    ├── table1.ibd       ← 独立表空间（数据 + 索引）
    └── table2.ibd
```

**表空间（Tablespace）层级结构**：
```
Tablespace
  └─ Segment（段）：数据段 / 索引段 / 回滚段
       └─ Extent（区）：64个连续Page，1MB
            └─ Page（页）：默认 16KB，最小 I/O 单元
                 └─ Row（行）：实际的数据记录
```

**Doublewrite Buffer（双写缓冲）**：
- 解决"部分页写入"问题（16KB 页在写磁盘时如果宕机，可能写了一半）
- 写数据页时先写一份到 Doublewrite Buffer 区域，再写到实际位置
- 崩溃恢复时先检查 Doublewrite Buffer 中的页是否完整

### 2.3 InnoDB vs MyISAM 核心差异

从设计理念来看：

**InnoDB** 的核心哲学是"**安全优先，兼顾性能**"：
- 所有写操作通过 WAL 先写 redo log，保证 crash-safe
- 行级锁 + MVCC 让读写互不阻塞，高并发场景表现出色
- 聚簇索引把数据和主键索引存在一起，主键查询无需回表

**MyISAM** 的核心哲学是"**简单快速，读场景优先**"：
- 数据文件（.MYD）和索引文件（.MYI）分离存储
- 没有事务机制，没有崩溃恢复，数据易损坏
- 表级锁在高并发写入时成为严重瓶颈
- 维护了精确的行数计数，`COUNT(*)` 极快（InnoDB 无法做到，因为 MVCC 下不同事务看到的行数不同）

## 三、连接管理

### 3.1 连接生命周期

```
客户端发起 TCP 连接
    ↓
MySQL 三次握手 + 身份认证（challenge-response）
    ↓
连接建立，服务端创建/分配线程
    ↓
执行 SQL（可以多次）
    ↓
连接关闭，线程归还线程池
```

### 3.2 关键连接参数

| 参数 | 说明 | 典型值 |
|------|------|--------|
| `max_connections` | 最大并发连接数 | 500~2000 |
| `wait_timeout` | 空闲连接超时（秒） | 3600 |
| `interactive_timeout` | 交互式连接超时 | 3600 |
| `thread_cache_size` | 线程缓存大小 | 50~200 |

**连接池的重要性**：每次建立 MySQL 连接代价较高（TCP 握手 + 认证 + 分配线程）。Java 应用必须使用连接池（HikariCP / Druid），避免频繁创建销毁连接。HikariCP 是目前性能最好的 Java 连接池，也是 Spring Boot 默认选择。

## 四、面试要点速查

| 问题 | 核心答案 |
|------|---------|
| MySQL 架构分几层？ | 三层：连接层、SQL层（解析/优化/执行）、存储引擎层 |
| SQL 执行流程？ | 连接认证 → 解析 → 预处理 → 优化 → 执行 → 返回结果 |
| InnoDB 和 MyISAM 最核心的区别？ | 事务/行锁/外键/MVCC/崩溃恢复 InnoDB 全都支持，MyISAM 都不支持 |
| Buffer Pool 的作用？ | 缓存数据和索引页，减少磁盘 I/O，是 InnoDB 性能的核心 |
| 为什么 InnoDB 的 COUNT(*) 比 MyISAM 慢？ | InnoDB 因 MVCC 无法直接返回精确行数，需要扫描；MyISAM 维护了精确计数器 |
| 可插拔存储引擎的意义？ | 不同表可用不同引擎，通过 Handler API 统一接口，上层 SQL 无感知 |
