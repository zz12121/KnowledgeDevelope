# 消息队列核心概念与架构

## 一、消息队列的定义与核心价值

消息队列（Message Queue，MQ）是一种**异步通信机制**，在分布式系统的不同组件或服务之间传递消息，充当临时的消息存储容器（缓冲区）。

**三大核心价值：**
- **应用解耦**：生产者和消费者通过消息间接通信，双方无需直接感知对方的存在或状态，可以独立开发、部署和扩展。
- **异步处理**：生产者发送消息后立即返回，无需等待消费者处理完成，提升系统响应速度和整体吞吐量。
- **流量削峰（削峰填谷）**：面对突发高并发流量（如秒杀），消息队列作为缓冲区，让下游系统按自身最大处理能力平稳消费，避免洪峰冲垮系统。

**生产者-消费者-Broker 三角关系：**
- **生产者（Producer）**：消息的发起方，负责创建消息并投递到指定目的地（Topic / Queue）。只管发，不管谁消费。
- **消费者（Consumer）**：消息的处理方，从 Broker 获取消息并执行业务逻辑。支持点对点（队列）和发布/订阅两种模式。
- **Broker**：消息队列服务器本身，负责接收、存储、路由消息，并保证消息的持久化和高可用。

## 二、消息传递语义

消息传递语义描述消息在生产、存储、消费整个生命周期中的可靠性保证：

**三种语义等级：**

**1. 至多一次（At Most Once）**
消息最多被传递一次，可能丢失，但绝不重复。发送后不等确认，或自动 ACK 后处理业务。丢失可以接受、但不能重复的日志场景适用。

**2. 至少一次（At Least Once）**
消息至少被传递一次，绝不丢失，但可能重复。实际中最常用，通过重试机制保证：
- 生产者：同步发送并等待 Broker ACK，失败则重试
- 消费者：业务处理**成功后**再手动提交 Offset / ACK

**3. 恰好一次（Exactly Once）**
消息有且仅被传递一次。实现最复杂、性能开销最大，需结合**幂等性**或**分布式事务**。Kafka 生产者通过 `enable.idempotence=true` 实现单会话幂等；Kafka 事务消息提供完整的 Exactly Once。

> 实践原则：业务侧尽量实现"至少一次" + **幂等消费**，这是可靠性与性能的最佳平衡点。

## 三、推拉模式（Push vs Pull）

**推模式（Push）**
Broker 主动将消息推送给消费者，代表：RabbitMQ。
- 优点：实时性高，消息到达即推送，延迟低
- 缺点：Broker 难以估计消费者处理能力，推送过快可能压垮消费者

**拉模式（Pull）**
消费者主动轮询 Broker 拉取消息，代表：Kafka。
- 优点：消费者自主控制速率，不会过载，天然支持批量处理
- 缺点：轮询间隔带来延迟，无消息时会产生大量无效请求

**长轮询（Long Polling）**
RocketMQ 等采用的折中方案：消费者发起 Pull 请求，若队列无消息不立即返回，而是挂起一段时间。有新消息到达时立即返回。兼顾实时性与消费者自主控制。

## 四、消息可靠性保障

消息不丢失需要在**生产者、Broker、消费者**三端同时保障：

**生产者端：**
- **ACK 确认机制**：发送消息后等待 Broker 确认，超时或失败则重试
- **事务消息**：保证消息发送与本地事务的原子性
- **同步发送**：关键业务使用同步发送，确认写入成功

**Broker 端：**
- **消息持久化**：将消息写入磁盘，防止重启丢失。RabbitMQ 中创建队列时设 `durable=true`，发送消息设 `deliveryMode=2`；Kafka 通过分区副本和日志持久化保障
- **高可用集群**：多副本机制，节点宕机时自动切换

**消费者端：**
- **手动 ACK**：关闭自动确认，业务逻辑**成功执行后**才向 Broker 发送确认信号
- RabbitMQ：`channel.basicAck(deliveryTag, false)`
- Kafka：`consumer.commitSync()` 手动提交 Offset

## 五、消息幂等性设计

消息重复是"至少一次"语义下的必然，幂等性是对抗重复的核心手段。

**幂等性**：同一操作执行一次和执行多次，最终结果完全相同。

**常见实现方案：**

**唯一 ID 去重**
```java
// 生产者为每条消息分配全局唯一ID（如UUID）
// 消费者使用Redis SETNX或数据库唯一索引校验是否已处理
String msgId = UUID.randomUUID().toString();
if (redisTemplate.opsForValue().setIfAbsent("msg:" + msgId, "1", 24, TimeUnit.HOURS)) {
    // 首次处理
    businessProcess(message);
}
// 否则跳过，避免重复
```

**数据库唯一约束**
利用唯一索引天然防重：订单表设置订单号唯一索引，重复插入时捕获 `DataIntegrityViolationException` 即可。

**状态机机制**
有状态流转的业务（如订单从"待支付"→"已支付"），处理前先查当前状态，仅当符合预期状态时才执行，通常结合乐观锁（version 字段）防并发。

## 六、消息顺序性保证

**RabbitMQ**：将需要保证顺序的消息发送到**同一个队列**，由**单个消费者**处理。多消费者并行消费会打乱顺序。

**Kafka**：在单个 Partition 内保证顺序。通过为相关消息设置相同的 **Key**，使其哈希路由到同一分区，实现局部顺序。

**业务层补偿**：消费者端有时需要通过状态机或分布式锁，在业务逻辑层面额外保证操作的顺序性。

---

## 七、死信队列（DLQ）

**死信队列**（Dead-Letter Queue）用于存放无法被正常消费的消息，是异常处理和延迟重试的重要机制。

**消息成为死信的三种情况：**
1. 消息被消费者否定确认（NACK）且设置不重新入队，或超过最大重试次数
2. 消息在队列中的存活时间（TTL）过期
3. 队列长度达到上限

**核心用途：**
- **异常处理与诊断**：失败消息进入 DLQ，后续集中分析原因（逻辑错误、数据格式问题等）
- **延迟重试**：消息先发送到带 TTL 的中间队列，过期后路由到 DLQ，再由 DLQ 绑定到真正的处理队列，实现延迟重试

---

## 八、消息堆积处理

消息积压的根本原因是**消费速度持续低于生产速度**。

**应急处理：**
- **扩容消费者**：增加消费者实例数量，提升并行消费能力。Kafka 中消费者数不应超过分区数
- **临时队列中转**：创建更大磁盘空间的临时 Topic，先用不处理业务的"搬运程序"快速转移积压消息，再启多个消费者并发处理

**根本优化：**
- **优化消费逻辑**：排查慢查询、Full GC 等性能瓶颈，同步改异步，使用 `CompletableFuture` 并行处理
- **批量消费**：配置消费者批量拉取，减少网络交互
- **队列拆分**：按优先级或业务类型拆分队列，隔离处理

## 九、常见中间件选型

| 中间件 | 语言 | 定位 | 适用场景 |
|--------|------|------|----------|
| **Kafka** | Scala/Java | 高吞吐流数据平台 | 大数据日志收集、实时流处理、数据管道 |
| **RocketMQ** | Java | 金融级业务消息 | 电商交易、分布式事务、顺序消息 |
| **RabbitMQ** | Erlang | 复杂路由企业消息 | 中小型企业应用、复杂路由需求 |
| **ActiveMQ** | Java | JMS 标准实现 | 传统 Java EE 企业集成 |

**选型口诀：**
- 大数据 / 流处理 → **Kafka**
- 金融 / 电商 / 事务 → **RocketMQ**
- 复杂路由 / 企业集成 → **RabbitMQ**
