# 消息可靠性、序列化与高级特性

## 一、分布式事务消息

分布式事务消息用于解决跨服务操作的数据一致性问题——本地数据库事务和消息发送这两件事如何做到"要么都成功，要么都失败"。

### 本地消息表方案（可靠性最高）
```sql
-- 业务库内创建消息表
CREATE TABLE local_message (
    id BIGINT PRIMARY KEY,
    business_id VARCHAR(64),
    content TEXT,
    status TINYINT DEFAULT 0,  -- 0:待发送 1:已发送
    create_time DATETIME
);
```
执行步骤：
1. 同一数据库事务内，同时写入业务数据 + 消息记录
2. 定时任务轮询 `status=0` 的消息，发送到 MQ，成功则更新状态为 `1`
3. 即使进程宕机，重启后定时任务继续补偿发送

**优点**：实现简单，100% 保证不丢失  
**缺点**：依赖数据库表，高并发下轮询有一定延迟

### RocketMQ 事务消息（半消息机制）
```java
TransactionMQProducer producer = new TransactionMQProducer("group");
producer.setTransactionListener(new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            orderService.createOrder(arg); // 执行本地事务
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }
    
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // Broker 回查：本地事务是否成功
        return orderService.isOrderExist(msg.getTransactionId()) 
               ? LocalTransactionState.COMMIT_MESSAGE 
               : LocalTransactionState.ROLLBACK_MESSAGE;
    }
});
```

流程：
1. 生产者向 Broker 发送**半消息**（对消费者不可见）
2. 执行**本地事务**
3. 根据结果向 Broker 发送 Commit / Rollback
4. Broker 长时间未收到确认，主动**回查**生产者事务状态

## 二、消息序列化方案

序列化决定消息在网络中的格式和效率：

### 主流方案对比

**JSON（Jackson / Gson）**
```java
ObjectMapper mapper = new ObjectMapper();
byte[] bytes = mapper.writeValueAsBytes(userObject); // 序列化
User user = mapper.readValue(bytes, User.class);     // 反序列化
```
- 优点：可读性好、跨语言、调试方便
- 缺点：体积大（字段名明文存储），性能相对最差
- 适用：业务系统、对可读性有要求的场景

**Protocol Buffers（Protobuf）**
```java
// 根据 .proto 文件生成的 Java 类
UserProto.User user = UserProto.User.newBuilder()
    .setId(123).setName("John").build();
byte[] bytes = user.toByteArray();   // 序列化（比JSON快2-3倍）
UserProto.User parsed = UserProto.User.parseFrom(bytes); // 反序列化
```
- 优点：高性能（序列化比 JSON 快 2-3 倍）、高压缩率（比 JSON 小 30%-80%）、强类型 Schema、向前向后兼容
- 缺点：需预定义 `.proto` 文件，二进制不可读
- 适用：微服务 RPC、对性能和带宽敏感的场景

**Apache Avro**
- 体积极小（序列化时不含字段名），Schema 演进能力极强
- 需要 Schema Registry 等外围组件
- 适用：大数据领域（Kafka + Confluent Schema Registry）

**性能排名（从高到低）：**
- 速度：Protobuf > Avro > MessagePack > JSON
- 体积：Avro ≈ Protobuf < MessagePack < JSON

### 向后兼容设计规则（以 Protobuf 为例）
1. **永不修改已有字段编号**：字段标签一旦分配，不能复用或更改
2. **新字段设为可选**：避免引入必填字段，旧版本无法提供该值
3. **废弃字段不删除**：标记 `deprecated=true`，绝不重用已废弃编号

```protobuf
message User {
  int64 user_id = 1;
  string email = 2;
  optional string phone_number = 3; // 新增用新编号，设为可选
}
```

### 敏感数据处理原则
- **最小化**：消息中只传递不敏感的唯一标识符，消费者按需查询
- **字段级加密**：序列化前对敏感字段（身份证、手机号）用 AES 加密
- **选择性序列化**：用 `@JsonIgnore` 等注解排除敏感字段

## 三、高可用与集群机制

各主流 MQ 的高可用实现对比：

**Kafka：分区副本（Replica）+ ISR 机制**
- 每个 Partition 有多个副本，分布在不同 Broker 上
- Leader 处理所有读写，Follower 异步拉取同步
- ISR 列表维护与 Leader 保持同步的副本集合
- `acks=all` 要求 ISR 内所有副本确认才算写入成功

**RabbitMQ：镜像队列 / Quorum 队列**
- 镜像队列：数据复制到多个节点，形成主从结构
- Quorum 队列（推荐）：基于 Raft 共识算法，强一致性，替代镜像队列

**RocketMQ：主从复制（Master-Slave）**
- 同步复制（SYNC_MASTER）：主节点等从节点确认，数据零丢失，性能略低
- 异步复制（ASYNC_MASTER）：主节点写入即返回，吞吐更高但极端情况可能丢少量数据

## 四、消息队列监控要点

**最关键指标：消息堆积量（Lag）**
表示消费者当前消费 Offset 与分区最新 Offset 的差值。Lag 持续增长 → 消费能力不足，需立即处理。

**其他核心指标：**
- **生产/消费吞吐量（TPS）**：速率异常下跌可能是 Broker 或客户端故障
- **Broker CPU / 内存 / 磁盘 I/O**：磁盘空间不足是最高优先级告警
- **请求处理延迟（P99、P999）**：延迟飙升是性能瓶颈信号

**监控工具：**
- Kafka：JMX Exporter + Prometheus + Grafana
- RocketMQ：自带管理控制台 + `mqadmin` 命令 + JMX
- RabbitMQ：内置 Web 管理界面（端口 15672）

## 五、容量规划

**磁盘需求估算：**
```
磁盘(GB) ≈ 日均消息量 × 每条消息大小 × 保留天数 × 副本数 × 1.5
```

举例：日均 1 亿条消息，每条 2KB，保留 3 天，3 副本：
```
100,000,000 × 0.002MB × 3 × 3 × 1.5 ≈ 2.7TB
```

**关键规划维度：**
- 消息大小与 TPS 峰值
- 数据保留策略（实时消费 vs 离线审计）
- 副本因子（通常 3 副本）
- 分区数（Kafka：影响并发度，但非越多越好，文件句柄开销会增加）

## 六、安全保障体系

**认证（Authentication）**：解决"你是谁"
- SSL/TLS 证书双向认证
- SASL/SCRAM：基于挑战-响应，密码加盐哈希存储
- SASL/GSSAPI：Kerberos，大型企业内网常用

**授权（Authorization）**：解决"你能做什么"
- ACL（访问控制列表）：精细到"Principal X 对 Topic Y 有 READ 权限"
- RBAC 模型：权限分配到角色，用户归属角色

**传输加密：**
- TLS 加密所有通信：`security.protocol=SASL_SSL`

**网络隔离：**
- 安全组 / 防火墙限制 Broker 端口（9092）仅对应用服务器 IP 段开放
- VPC 内部部署，不直接暴露公网
