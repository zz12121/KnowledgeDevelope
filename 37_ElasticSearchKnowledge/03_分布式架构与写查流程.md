## 1. 节点类型与职责

**ES 集群由多种角色的节点组成，生产环境建议角色分离。**

| 节点角色 | 配置 | 职责 |
|---------|------|------|
| **Master** | `node.roles: [master]` | 管理集群元数据（索引创建/删除）、分片分配、节点加入/离开 |
| **Data** | `node.roles: [data]` | 存储分片数据，执行 CRUD、搜索、聚合 |
| **Ingest** | `node.roles: [ingest]` | 执行 Ingest Pipeline（数据预处理） |
| **Coordinate** | `node.roles: []` | 路由请求到目标分片，汇总结果。**任意节点都可临时充当协调节点** |
| **ML** | `node.roles: [ml]` | 执行机器学习任务 |
| **Remote Cluster Client** | `node.roles: [remote_cluster_client]` | 跨集群搜索 |

**生产环境推荐架构（7节点）：**

```
3个专用 Master 节点（不存数据）
4个专用 Data 节点
2个协调节点（可选，用于负载均衡入口）
```

**查看节点角色：**

```bash
GET /_cat/nodes?v&h=name,node.role,heap.percent,cpu,load_1m
```

## 2. 文档路由算法

**ES 通过路由算法确定文档存储在哪个主分片上：**

```
shard_num = hash(_routing) % number_of_primary_shards
```

- `_routing` 默认是文档的 `_id`
- **这就是为什么主分片数创建后不可修改**——一旦修改，所有文档的路由结果都会变

**自定义路由（将相关数据聚集在同一分片，减少查询扇出）：**

```json
// 写入时指定路由
PUT /orders/_doc/1?routing=user_123
{"user_id": "user_123", "amount": 99.9}

// 查询时指定路由（只查特定分片，性能提升 N 倍）
GET /orders/_search?routing=user_123
{
  "query": {"term": {"user_id": "user_123"}}
}
```

**注意**：使用自定义路由要避免数据倾斜（routing key 要分布均匀）。

## 3. 数据写入流程（五阶段详解）

### 3.1 写入请求路由

```
客户端请求
    ↓
协调节点（任意节点）
    ↓ 计算 shard_num = hash(_id) % primary_shards
    ↓
目标主分片所在节点
    ↓ 写入主分片
    ↓ 并行同步到所有副本分片
    ↓
协调节点返回响应
```

### 3.2 Lucene 写入机制（核心）

```
Step 1: 写入内存 Buffer（IndexWriter 缓冲区）
        + 写入 Translog（事务日志，追加写，持久化到磁盘）
        ↓ 此时文档不可搜索

Step 2: refresh（默认 1 秒触发）
        Buffer → 内存中的 Segment（不可变的数据块）
        ↓ 文档可被搜索（near real-time）
        ↓ 但 Segment 未 fsync 到磁盘（断电仍靠 Translog 恢复）

Step 3: flush（Translog 超大/定期触发）
        Segment fsync 到磁盘（OS Cache → 磁盘）
        清空 Translog，生成新 Translog
        ↓ 数据持久化完成

Step 4: Segment 合并（后台自动）
        多个小 Segment → 合并为大 Segment
        物理删除被标记为删除的文档
        释放磁盘空间，提升查询性能

Step 5: 副本同步（与主分片并行）
        主分片写入成功后，并行向所有副本发送写入请求
        wait_for_active_shards 控制等待几个副本确认后返回
```

**Translog 的作用：**
- 防止 Segment 未 fsync 时节点宕机导致数据丢失
- 节点重启时读取 Translog 中未 flush 的操作，重放到 Segment
- 类似 MySQL 的 redo log

### 3.3 写入参数调优

```json
// 写入时的一致性控制
PUT /products/_doc/1?wait_for_active_shards=2
// wait_for_active_shards: 1(仅主分片) / 2(主+1副本) / all

// 写入后是否立即 refresh（让文档立即可搜索）
PUT /products/_doc/1?refresh=true       // 立即 refresh（影响性能）
PUT /products/_doc/1?refresh=wait_for  // 等待下次自动 refresh
PUT /products/_doc/1?refresh=false     // 默认，不触发 refresh
```

## 4. 数据查询流程（两阶段详解）

### 4.1 Query Phase（查询阶段）

```
协调节点收到搜索请求
    ↓
广播到索引的所有主分片或副本分片（轮询负载均衡）
    ↓ 每个分片：
    ↓   1. 解析查询
    ↓   2. 在本地 Lucene 执行搜索
    ↓   3. 返回 Top K 文档的 { _id, _score, sort_values }（只返回 ID，不返回 _source）
    ↓
协调节点：
    收集所有分片的结果，全局排序（归并排序）
    取最终 Top K 的文档 ID 列表
```

### 4.2 Fetch Phase（获取阶段）

```
协调节点：
    根据 Query Phase 得到的文档 ID 列表
    精确定位每个文档所在的分片
    ↓ 向对应分片发送 Multi GET 请求
    ↓
每个分片：
    根据 _id 读取完整的 _source（序列化为 JSON）
    ↓
协调节点：
    汇总所有文档，按评分排序
    返回最终结果给客户端
```

### 4.3 为什么要两阶段？

**性能考量**：如果第一阶段就返回完整 `_source`，每个分片返回 10010 条完整文档（深分页场景），网络传输和内存开销巨大。两阶段设计让第一阶段只传递轻量的文档 ID + 分数，大大减少数据传输量。

### 4.4 DFS Query Phase（分布式词频统计）

默认情况下，BM25 评分基于分片本地的词频统计，可能导致**相同词在不同分片评分不一致**。

```json
// 使用 DFS 模式先收集全局词频统计
GET /products/_search?search_type=dfs_query_then_fetch
{
  "query": {"match": {"title": "手机"}}
}
```

**注意**：DFS 需要额外的网络往返，数据量大时不推荐，通常数据量够大时分片间词频差异可忽略。

## 5. 更新与删除机制

### 5.1 更新文档

**ES 中的更新是"读-改-写"操作，非原地更新：**

```
1. 读取旧文档（从主分片）
2. 应用更新（局部更新则合并字段）
3. 将新版本的文档重新写入（生成新文档，版本号 +1）
4. 旧文档标记为"已删除"（逻辑删除，占用磁盘直到 Segment 合并时物理删除）
```

**并发更新控制（乐观锁）：**

```json
POST /products/_update/1?if_seq_no=5&if_primary_term=1
{"doc": {"price": 4999}}
// 如果 seq_no 不匹配（被其他请求修改过），返回 409 Conflict
// 应用层 catch 409 后重新读取再更新
```

### 5.2 删除文档

**ES 的删除是逻辑删除（Soft Delete）：**

```
DELETE /products/_doc/1
→ 在 Segment 中将该文档标记为已删除（.del 文件记录）
→ 文档占用的磁盘空间不会立即释放
→ 查询时会过滤掉已删除文档
→ Segment 合并时才物理删除，释放空间
```

**强制合并释放空间：**

```json
POST /products/_forcemerge?max_num_segments=1&only_expunge_deletes=true
```

## 6. 脑裂（Split Brain）与 Master 选举

### 6.1 脑裂问题

**脑裂**：网络分区导致集群分裂，两个子集群各自选出一个 Master，形成两个"大脑"，造成数据不一致。

```
正常：  Node1(M) ← → Node2 ← → Node3

网络分区后：
子集群1: Node1(M) 认为自己是 Master
子集群2: Node2(M) 自选为 Master
两边各自处理写入 → 数据冲突
```

### 6.2 防脑裂机制

**ES 7.x 之前：minimum_master_nodes 设置（法定多数）**

```yaml
# 设置为 (master节点数/2) + 1
# 3个节点时设置为 2，5个节点时设置为 3
discovery.zen.minimum_master_nodes: 2
```

**ES 7.x 之后：自动管理（基于 Raft 改进的 Quorum 机制）**

- ES 自动计算 Quorum，无需手动设置
- 只有获得超过半数 Master-eligible 节点投票的候选人才能成为 Master
- **这就是为什么 Master 节点数推荐奇数（3或5），偶数节点数在网络分区时容易各占一半，无法选出 Master**

### 6.3 Master 选举流程（ES 7.x+）

```
1. 节点发现：通过 discovery.seed_hosts 找到种子节点
2. 推举候选：每个 Master-eligible 节点自荐，优先推举 ClusterStateVersion 最新的
3. 投票选举：候选人获得超过 (voting_config_size/2 + 1) 票即当选
4. 发布集群状态：新 Master 向所有节点广播最新的集群状态
5. 集群就绪：所有节点确认收到集群状态后，集群恢复服务
```

**查看当前 Master：**

```bash
GET /_cat/master?v
GET /_cluster/state/master_node
```

## 7. 数据一致性保证

### 7.1 写一致性（write_concern）

```json
// wait_for_active_shards 控制写入确认
PUT /products/_doc/1?wait_for_active_shards=all  // 等所有副本确认（强一致，影响可用性）
PUT /products/_doc/1?wait_for_active_shards=1    // 只等主分片确认（默认，最终一致）
```

### 7.2 读一致性问题

**问题**：写完主分片后立即从副本读，可能读到旧数据（副本同步有延迟）。

**解决方案：**
1. **写完后用主分片读**：`GET /products/_doc/1?preference=_primary`
2. **写完后等 refresh**：`PUT /products/_doc/1?refresh=wait_for`
3. **业务容忍最终一致**：大多数读场景 1 秒内副本都会同步完成

### 7.3 并发写控制

```json
// 乐观并发控制（推荐）
POST /products/_update/1?if_seq_no=10&if_primary_term=1
{"doc": {"stock": 99}}

// 外部版本控制（配合 DB 同步）
PUT /products/_doc/1?version=5&version_type=external
{"title": "iPhone 15"}
```

## 8. Translog 详解

**Translog 是 ES 的 WAL（Write-Ahead Log），保证未 flush 数据的持久性。**

```yaml
# Translog 配置
index.translog.durability: request     # 每次请求 fsync（默认，安全但慢）
index.translog.durability: async       # 异步 fsync（高性能，允许丢少量数据）
index.translog.sync_interval: 5s      # async 模式的 fsync 间隔
index.translog.flush_threshold_size: 512mb  # 超过此大小触发 flush
```

**Translog 与 flush 的关系：**

```
写操作 → Translog 追加写（持久化）→ Buffer 写入（内存）
refresh → Buffer → Segment（内存）
flush  → Segment fsync（磁盘）→ 清空 Translog

节点宕机重启：
  读取 Translog → 重放未 flush 的写操作 → 恢复数据
```

## 9. Segment 合并机制

**Segment 是 Lucene 的不可变数据文件，每次 refresh 产生一个新 Segment。**

**合并的必要性：**
- Segment 数量增多（每秒 1 个，1 小时就有 3600 个）
- 查询需要扫描所有 Segment，Segment 越多查询越慢
- 已删除文档物理占用磁盘，需合并时清理

**合并策略（TieredMergePolicy）：**

```json
PUT /my-index/_settings
{
  "index.merge.policy.max_merge_at_once": 10,      // 每次最多合并几个 Segment
  "index.merge.policy.segments_per_tier": 10,      // 每层目标 Segment 数
  "index.merge.scheduler.max_thread_count": 1       // 合并线程数（机械硬盘设 1，SSD 可以多）
}
```

**手动强制合并（低峰期执行）：**

```json
// 合并为 1 个 Segment（适合只读的历史索引）
POST /my-index/_forcemerge?max_num_segments=1

// 只清除已删除文档，不大量合并（影响小）
POST /my-index/_forcemerge?only_expunge_deletes=true
```

**注意**：forcemerge 会消耗大量磁盘 I/O 和 CPU，只在低峰期对只读索引执行。
