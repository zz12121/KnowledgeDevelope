## 1. 集群健康监控

### 1.1 集群状态三色

| 颜色 | 含义 | 紧急程度 |
|------|------|---------|
| 🟢 Green | 所有主分片和副本分片都正常 | 正常 |
| 🟡 Yellow | 所有主分片正常，**部分副本分片未分配** | 需关注，集群可读写但降低了容灾能力 |
| 🔴 Red | **部分主分片未分配**，对应数据不可读写 | 紧急，立即处理 |

```bash
# 集群整体健康
GET /_cluster/health?pretty

# 索引级别健康
GET /_cluster/health/my-index

# 等待集群变绿（运维脚本中常用）
GET /_cluster/health?wait_for_status=green&timeout=30s
```

### 1.2 关键监控指标

**JVM 指标（最重要）：**

```bash
GET /_nodes/stats/jvm?filter_path=nodes.*.jvm.mem,nodes.*.jvm.gc

# 关注指标：
# jvm.mem.heap_used_percent > 75% → 告警
# jvm.gc.collectors.old.collection_time_in_millis 增长 → GC 压力大
```

**磁盘指标：**

```bash
GET /_cat/allocation?v&h=node,disk.used_percent,disk.avail,disk.total

# 关注：
# disk.used_percent > 80% → 告警（默认高水位 85%）
# disk.used_percent > 90% → 紧急（停止分配新分片）
# disk.used_percent > 95% → 索引变只读
```

**性能指标：**

```bash
GET /_cat/nodes?v&h=name,heap.percent,cpu,load_1m,search.query_current,search.query_time,indexing.index_current

GET /_cat/thread_pool?v&h=name,active,queue,rejected&s=rejected:desc
# rejected > 0 → 需要扩容或限流
```

**分片指标：**

```bash
GET /_cat/shards?v&h=index,shard,prirep,state,docs,store,node&s=state
# UNASSIGNED 状态的分片需要立即排查
```

### 1.3 Kibana 监控面板

**Stack Monitoring（X-Pack 自带）：**
- 路径：Kibana → Stack Monitoring
- 功能：节点 JVM/CPU/磁盘实时图表、搜索/写入 QPS 趋势、集群健康历史

## 2. 索引生命周期管理（ILM）

### 2.1 ILM 四个阶段

| 阶段 | 触发条件 | 典型操作 |
|------|---------|---------|
| **Hot** | 立即 | rollover（按大小/时间/文档数滚动）、写入优化 |
| **Warm** | 触发后 N 天 | forcemerge（减少 Segment）、shrink（减少分片数）、freeze（减少内存） |
| **Cold** | 触发后 N 天 | searchable_snapshot（挂载快照，几乎不占资源）、readonly |
| **Delete** | 触发后 N 天 | delete（删除索引） |

### 2.2 ILM 策略配置

```json
PUT /_ilm/policy/logs-30day-policy
{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_size": "50gb",
            "max_age": "1d",
            "max_docs": 100000000
          },
          "set_priority": {"priority": 100}
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "forcemerge": {"max_num_segments": 1},
          "shrink": {"number_of_shards": 1},
          "allocate": {
            "require": {"data": "warm"}
          },
          "set_priority": {"priority": 50}
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "freeze": {},
          "allocate": {
            "require": {"data": "cold"}
          },
          "set_priority": {"priority": 0}
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": {"delete": {}}
      }
    }
  }
}

// 索引模板中绑定 ILM 策略
PUT /_index_template/logs-template
{
  "index_patterns": ["logs-*"],
  "data_stream": {},
  "template": {
    "settings": {
      "index.lifecycle.name": "logs-30day-policy",
      "index.lifecycle.rollover_alias": "logs-write"
    }
  }
}
```

**查看 ILM 执行状态：**

```bash
GET /logs-app-000001/_ilm/explain
GET /_ilm/stats
```

## 3. 快照备份与恢复

### 3.1 配置快照仓库

```json
// 文件系统快照仓库
PUT /_snapshot/my-backup
{
  "type": "fs",
  "settings": {
    "location": "/mnt/elasticsearch/backups",
    "compress": true,
    "max_snapshot_bytes_per_sec": "100mb",
    "max_restore_bytes_per_sec": "100mb"
  }
}

// S3/OSS 快照仓库（需要对应插件）
PUT /_snapshot/s3-backup
{
  "type": "s3",
  "settings": {
    "bucket": "my-es-backups",
    "region": "us-east-1",
    "base_path": "elasticsearch/"
  }
}
```

### 3.2 手动快照

```json
// 创建快照
PUT /_snapshot/my-backup/snapshot-2024-03-15
{
  "indices": "orders-*,products",
  "ignore_unavailable": true,
  "include_global_state": false
}

// 查看快照状态
GET /_snapshot/my-backup/snapshot-2024-03-15

// 查看所有快照
GET /_snapshot/my-backup/_all?pretty
```

### 3.3 快照生命周期管理（SLM）

```json
PUT /_slm/policy/daily-backup
{
  "schedule": "0 30 2 * * ?",  // 每天凌晨 2:30
  "name": "<daily-snap-{now/d}>",
  "repository": "my-backup",
  "config": {
    "indices": ["data-*", "products"],
    "include_global_state": false,
    "ignore_unavailable": true
  },
  "retention": {
    "expire_after": "30d",
    "min_count": 5,
    "max_count": 50
  }
}

// 立即执行一次
PUT /_slm/policy/daily-backup/_execute

// 查看执行历史
GET /_slm/stats
```

### 3.4 恢复快照

```json
// 恢复整个快照（所有索引）
POST /_snapshot/my-backup/snapshot-2024-03-15/_restore
{
  "indices": "orders-2024-03",
  "ignore_unavailable": true,
  "rename_pattern": "(.+)",
  "rename_replacement": "restored-$1",
  "index_settings": {
    "index.number_of_replicas": 0
  },
  "include_aliases": false
}

// 恢复完成后恢复副本
PUT /restored-orders-2024-03/_settings
{"index.number_of_replicas": 1}
```

## 4. 索引重建（Reindex）

### 4.1 使用场景

- Mapping 字段类型需要修改（如 `text` 改 `keyword`）
- 分片数需要调整
- 分析器配置需要更改
- 数据迁移（从旧集群到新集群）

### 4.2 Reindex 操作

```json
// 基本 Reindex（同集群）
POST /_reindex
{
  "source": {
    "index": "products-old",
    "query": {"term": {"is_active": true}},
    "size": 1000
  },
  "dest": {
    "index": "products-new",
    "op_type": "create"  // create：已存在则跳过；index：覆盖
  },
  "script": {
    "source": "ctx._source.price = ctx._source.price * 1.1"  // 迁移时修改数据
  }
}

// 跨集群 Reindex
POST /_reindex
{
  "source": {
    "remote": {
      "host": "http://old-cluster:9200",
      "username": "elastic",
      "password": "oldpassword"
    },
    "index": "products"
  },
  "dest": {"index": "products"}
}
```

### 4.3 零停机 Reindex（别名切换）

```
1. 创建新索引 products_v2（新 Mapping）
2. Reindex: products_v1 → products_v2
3. 等待 Reindex 完成
4. 增量同步：Reindex 新增数据（带时间过滤）
5. 切换别名：
   POST /_aliases
   {
     "actions": [
       {"remove": {"index": "products_v1", "alias": "products"}},
       {"add":    {"index": "products_v2", "alias": "products"}}
     ]
   }
6. 验证，删除旧索引
```

## 5. 滚动重启（Rolling Restart）

**滚动重启**：逐个节点重启，不停止整个集群服务，适合配置变更、版本升级。

```bash
# Step 1：关闭分片自动分配（防止重启时触发大量数据迁移）
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "primaries"
    # 或 "none"（完全禁止） 或 "primaries"（只允许主分片）
  }
}

# Step 2：等待集群同步（确保内存数据 flush 到磁盘）
POST /_nodes/_local/flush/synced

# Step 3：重启目标节点
sudo systemctl restart elasticsearch

# Step 4：等待节点加入集群
GET /_cat/nodes?v

# Step 5：等待集群恢复 Green
GET /_cluster/health?wait_for_status=green&timeout=60s

# Step 6：重新开启分片分配
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "all"
  }
}

# Step 7：对下一个节点重复 Step 3~6
```

## 6. 容量规划

### 6.1 存储容量

```
原始数据大小 × 1.45（索引开销约 45%）× 副本系数 / 磁盘利用率上限（80%）

示例：100GB 原始数据，1个副本：
  100GB × 1.45 × 2 / 0.8 ≈ 362GB 磁盘
```

**索引开销来源：**
- 倒排索引（词典 + 倒排列表）：~30%
- DocValues（正排，用于聚合排序）：~10%
- 存储压缩（best_compression 可减少 30-40%）

### 6.2 内存容量

```
JVM 堆 = min(物理内存 × 50%, 30GB)  // 建议不超过 30GB
文件系统缓存 ≥ JVM 堆大小             // Lucene 大量使用 OS 文件缓存

最低推荐：64GB 物理内存（堆 32GB + 文件缓存 32GB）
```

### 6.3 节点规模估算

```
分片总数 = 主分片数 × (1 + 副本数)
每节点分片数 = 分片总数 / 节点数
建议每节点分片数 ≤ 200 个

示例：1TB 数据，主分片 20 个，1 个副本：
  分片总数 = 40
  如果每节点最多 20 个分片 → 至少需要 2 个节点
  建议 3 节点（+ 3 个 master-eligible 节点）
```

## 7. 版本升级

### 7.1 升级前准备

```bash
# 1. 阅读 Breaking Changes 文档（必须）
# 2. 在测试环境验证

# 3. 备份快照
PUT /_snapshot/pre-upgrade-backup/before-upgrade
{"indices": "*", "include_global_state": true}

# 4. 检查弃用 API 使用（升级前修复）
GET /_migration/deprecations

# 5. 检查集群健康（必须 Green）
GET /_cluster/health
```

### 7.2 滚动升级（ES 支持相邻主版本间滚动升级）

```
支持：7.x → 8.x（单版本跨越）
不支持：6.x → 8.x（需先升到 7.x）

步骤：
1. 关闭分片分配
2. 逐节点停止 → 安装新版本 → 启动
3. 等待节点加入集群，分片恢复
4. 重开分片分配
5. 重复，直到所有节点升级完成
```

## 8. Watcher 告警

```json
// 配置告警：集群变黄时发邮件
PUT /_watcher/watch/cluster-health-alert
{
  "trigger": {"schedule": {"interval": "5m"}},
  "input": {
    "http": {
      "request": {
        "host": "localhost",
        "port": 9200,
        "path": "/_cluster/health",
        "auth": {"basic": {"username": "elastic", "password": "changeme"}}
      }
    }
  },
  "condition": {
    "compare": {
      "ctx.payload.status": {"not_eq": "green"}
    }
  },
  "actions": {
    "send_email": {
      "email": {
        "to": "ops-team@example.com",
        "subject": "【ES告警】集群状态异常：{{ctx.payload.status}}",
        "body": {
          "text": "集群状态：{{ctx.payload.status}}\n未分配分片数：{{ctx.payload.unassigned_shards}}\n时间：{{ctx.execution_time}}"
        }
      }
    },
    "log_to_index": {
      "index": {
        "index": "cluster-alerts",
        "doc_type": "_doc"
      }
    }
  }
}

// 查看告警执行历史
GET /_watcher/watch/cluster-health-alert/_history

// 手动触发告警（测试）
POST /_watcher/watch/cluster-health-alert/_execute
```
