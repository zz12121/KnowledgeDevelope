## 核心知识点速记

### 倒排索引三要素
```
Term Dictionary（词项字典）
    ↓ 二分查找 O(log N)
Term Index（前缀树，在内存中）
    ↓ 快速定位
Posting List（文档 ID 列表，Frame of Reference 压缩）
```

### 写入五阶段
```
1. 写 Buffer（内存缓冲区）+ Translog（持久化日志）
2. refresh（1s，Buffer → Segment，可被搜索，但未 fsync）
3. flush（Translog 达阈值，Segment fsync 到磁盘，清空 Translog）
4. merge（后台合并小 Segment，删除标记文档）
5. 副本同步（主分片 → 副本分片，并行同步）
```

### 查询两阶段（Scatter-Gather）
```
Query Phase：协调节点 → 所有分片 → 各分片返回 Top K 文档ID+分数
Fetch Phase：协调节点 → 目标分片（精准拉取） → 返回完整文档
```

### 节点类型速记
```
Master     → 集群元数据管理（索引创建/删除/分片分配）
Data       → 存储数据，执行 CRUD 和搜索
Ingest     → 执行 Pipeline 预处理
Coordinate → 路由请求，汇总结果（任意节点都可充当）
ML         → 机器学习任务
```

### BM25 评分公式核心
```
score(d,q) = Σ IDF(qi) × TF_norm(qi,d)

IDF：词越罕见，IDF 越大（log((N-n+0.5)/(n+0.5)+1)）
TF：词频越高，TF 越大，但有饱和上限（k1 参数控制）
字段长度归一化：文档越短，相同 TF 评分越高
```

### 分片数量经验法则
```
单个分片大小：10GB ~ 50GB（日志可到 50GB，搜索建议 20GB 以内）
分片数 = 总数据量 / 单分片目标大小
节点数 = 分片数 / 每节点承载分片数（建议每节点 <200 分片）

注意：分片一旦创建数量不可变（只能 Split/Shrink）
```

### 脑裂防护
```
ES 7.x 以前：discovery.zen.minimum_master_nodes = (master节点数/2) + 1
ES 7.x 以后：自动管理（基于 Raft 改进），无需手动设置
```

### 性能优化速查
```
写入优化：
  ✓ refresh_interval: 30s（批量导入时设 -1）
  ✓ Bulk size: 5~15MB / 1000~5000 条
  ✓ 初始导入：replicas: 0，导完再恢复
  ✓ translog.durability: async（允许丢少量数据换高吞吐）

查询优化：
  ✓ 过滤用 filter（可缓存），精确匹配不要用 match
  ✓ 避免深分页（>10000），用 search_after 游标
  ✓ 聚合字段用 keyword，不要在 text 上聚合（Fielddata）
  ✓ 控制返回字段（_source filtering），减少传输量
  ✓ 路由（routing key）减少扫描分片数

JVM 调优：
  ✓ Heap: 物理内存 50%，不超过 32GB
  ✓ Xms == Xmx（固定堆大小）
  ✓ GC: G1GC（ES 7.x 默认）
  ✓ 开启 Circuit Breaker 防止 OOM
```

### 常用 API 速查
```bash
# 集群健康
GET /_cluster/health?pretty

# 节点状态
GET /_cat/nodes?v&h=name,heap.percent,cpu,disk.used_percent

# 查看分片分配
GET /_cat/shards?v&s=state

# 诊断未分配原因
GET /_cluster/allocation/explain

# 热线程
GET /_nodes/hot_threads

# 线程池积压
GET /_cat/thread_pool?v&h=name,active,queue,rejected&s=queue:desc

# 强制合并（减少 Segment 数量，释放被删除文档空间）
POST /my-index/_forcemerge?max_num_segments=1

# 刷新索引（使数据立即可搜索）
POST /my-index/_refresh

# 清除缓存
POST /my-index/_cache/clear
```
