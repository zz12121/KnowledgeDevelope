## 一、ZooKeeper 性能特征

| 维度 | 读 | 写 |
|------|----|----|
| 数据来源 | 内存（全量数据在内存） | 需要 WAL + ZAB 广播 |
| 扩展方式 | 增加 Observer 线性扩展 | 无法水平扩展（单 Leader） |
| 典型 QPS | 10w+（加 Observer 后更高） | 5000~15000 |
| 主要瓶颈 | 几乎没有 | 磁盘 fsync + ZAB 两阶段 + Leader 单点 |

## 二、性能优化清单

### 2.1 磁盘优化（最重要）

```properties
# 事务日志单独挂 SSD（写多，fsync 是瓶颈）
dataLogDir=/ssd1/zookeeper/log

# 快照单独磁盘（读多）
dataDir=/ssd2/zookeeper/data
```

> 事务日志的 fsync 延迟直接决定写延迟：HDD ~5ms/次，SSD ~0.1ms/次，差 50 倍。

### 2.2 扩展读能力

```properties
# 添加 Observer 节点（不参与投票，不影响写吞吐）
server.4=obs1:2888:3888:observer
server.5=obs2:2888:3888:observer
```

### 2.3 JVM 调优

```bash
# 推荐 JVM 参数（4~8GB 内存，G1GC 控制停顿）
export SERVER_JVMFLAGS="\
  -Xmx4g -Xms4g \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/data/zookeeper/logs/"
```

### 2.4 自动清理（必须配置）

```properties
autopurge.snapRetainCount=3    # 至少保留3个，用于回滚
autopurge.purgeInterval=1      # 每小时清理一次
```

### 2.5 限制客户端连接数

```properties
maxClientCnxns=60   # 防止单个 IP 占满所有连接
```

## 三、四字命令速查

```bash
# 通用格式
echo <command> | nc <host> <port>

# 示例
echo ruok | nc zoo1 2181  # 存活检查，返回 imok
echo stat | nc zoo1 2181  # 集群状态
echo mntr | nc zoo1 2181  # 监控指标（KV格式）
echo cons | nc zoo1 2181  # 当前连接详情
echo wchs | nc zoo1 2181  # Watcher 汇总
echo dump | nc zoo1 2181  # Session 和临时节点
echo isro | nc zoo1 2181  # 是否只读模式（rw/ro）
```

**必须在 zoo.cfg 中白名单开启：**
```properties
4lw.commands.whitelist=stat,ruok,conf,mntr,isro,cons,wchs
```

## 四、关键监控指标（mntr 命令）

```bash
echo mntr | nc zoo1 2181 | grep -E \
  "zk_(avg_latency|max_latency|outstanding_requests|open_file_descriptor|server_state|znode_count|watch_count|followers)"
```

| 指标 | 健康参考值 | 告警阈值 |
|------|---------|---------|
| `zk_avg_latency` | < 5ms | > 10ms |
| `zk_max_latency` | < 50ms | > 100ms |
| `zk_outstanding_requests` | 0~10 | > 100 |
| `zk_znode_count` | < 500k | > 1M |
| `zk_watch_count` | < 100k | > 500k |
| `zk_open_file_descriptor_count` | < 系统限制60% | > 80% |
| `zk_server_state` | 集群只有1个 leader | 出现0个或多个 |

## 六、常见故障排查

### 6.1 集群状态快速检查

```bash
#!/bin/bash
# 检查所有节点状态和角色
for host in zoo1 zoo2 zoo3; do
    echo -n "$host ruok: "; echo ruok | nc $host 2181; echo
    echo -n "$host role: "; echo stat | nc $host 2181 | grep Mode; echo
    echo -n "$host zxid: "; echo stat | nc $host 2181 | grep Zxid; echo
done
```

### 6.2 故障现象与处理

| 现象 | 可能原因 | 排查命令 |
|------|---------|---------|
| `ruok` 不返回 | 进程崩溃 / 端口占用 | `ps aux \| grep zookeeper`；查 `zookeeper.out` |
| 一直处于 LOOKING | 3888 端口不通 / 多数节点宕机 | `telnet zoo1 3888`；`echo stat \| nc` 检查各节点 |
| 写操作超时 | 磁盘 I/O 慢 / Follower 同步慢 | `iostat -x 1`；`echo mntr \| nc` 看 `outstanding` |
| 大量 Session Expired | sessionTimeout 太短 / GC 停顿 | 增大 sessionTimeout；`jstat -gcutil` 查 GC |
| 磁盘告警 | 未配置 autopurge | 立刻执行 `PurgeTxnLog`，配置 autopurge |

### 6.3 Session Expired 处理

```java
// 使用 Curator 优雅处理 Session 状态
client.getConnectionStateListenable().addListener((c, state) -> {
    switch (state) {
        case SUSPENDED:
            // 连接暂时断开，暂停业务（锁可能还有效）
            businessService.pause();
            break;
        case RECONNECTED:
            // 重连成功，恢复业务
            businessService.resume();
            break;
        case LOST:
            // Session 超时！临时节点全没了！
            // 必须重新注册服务、重新抢锁等
            businessService.reinitialize();
            break;
    }
});
```

## 七、羊群效应 & 惊群问题

**羊群效应（Herd Effect）：**
所有等待者监听同一个锁节点，锁释放时大家同时被唤醒，只有一个能成功。

**解决：临时顺序节点 + 只监听前一个节点（Curator 内置）**

**惊群（Thundering Herd）：**
大量客户端监听同一父路径，子节点频繁变化时同时触发大量 Watcher。

**缓解方案：**
1. 随机延迟响应（收到通知后随机等 100~500ms 再处理）
2. 推拉结合（Watcher 标记"有变化"，定时任务批量拉取）
3. Curator Cache 类自动合并多个通知

## 八、安全加固清单

```properties
# ① 开启 Digest 认证（必须）
# 所有创建的节点设置非 world:anyone 的 ACL

# ② 开启 SSL/TLS（3.5.5+）
secureClientPort=2281
ssl.keyStore.location=/path/to/keystore.jks
ssl.trustStore.location=/path/to/truststore.jks

# ③ 限制四字命令（不要全开放）
4lw.commands.whitelist=stat,ruok,conf,mntr,isro

# ④ 禁止 world 权限（应用层）
# 所有 ZNode 至少配置 digest 或 ip ACL

# ⑤ 不要暴露 2181 端口到公网
# 使用防火墙/安全组限制访问来源
```
