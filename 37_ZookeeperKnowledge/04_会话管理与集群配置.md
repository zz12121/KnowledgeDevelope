## 一、Session 生命周期

```
[初始] NOT_CONNECTED
    ↓ 发起连接
[连接中] CONNECTING
    ↓ 连接建立，Session 创建成功
[正常] CONNECTED ←─────────────────────────────┐
    ↓ 网络断开                                  │ 重连成功（Session未超时）
[重连中] CONNECTING ─────────────────────────────┘
    ↓ 未在 sessionTimeout 内重连
[关闭] NOT_CONNECTED（Session Expired 状态）
```

**Session Expired 的严重性：**
- 该 Session 创建的**所有临时节点被删除**
- 分布式锁自动释放
- 服务注册临时节点消失，其他节点认为服务下线
- 客户端必须重建 ZooKeeper 实例，重新初始化所有 ZK 资源

---

## 二、Session 关键属性

| 属性 | 说明 |
|------|------|
| **Session ID** | 64位全局唯一标识，高位含 myid 防冲突，由 Leader 分配 |
| **Session Password** | 16字节随机密码，重连时携带，防止其他客户端伪造 Session |
| **Session Timeout** | 客户端设置，服务端在范围内协商（min=2×tickTime, max=20×tickTime） |

---

## 三、心跳机制

```
pingInterval = sessionTimeout / 3   （每 1/3 超时时间发一次 PING）
readTimeout  = sessionTimeout × 2/3 （超过 2/3 超时时间无响应则重连）
```

客户端 `SendThread` 负责定时发送 PING；服务端 `SessionTracker` 收到任何请求（包括 PING）都刷新 Session 过期时间。

---

## 四、Session 超时时间设置建议

| 场景 | 建议值 | 原因 |
|------|--------|------|
| 常规生产 | 30s | 平衡故障感知速度与网络抖动容忍 |
| 高可用服务发现 | 10~15s | 快速感知服务下线 |
| 跨机房/网络质量差 | 60~90s | 容忍跨机房网络抖动 |

**配置要求：**
- `sessionTimeout >= 2 × tickTime`（最小限制）
- Curator 建议：`sessionTimeout >= 3 × connectionTimeout`

## 五、zoo.cfg 核心配置速查

```properties
# =============== 基础配置 ===============
tickTime=2000               # 基本时间单位（ms），心跳间隔
dataDir=/data/zk/data       # 快照文件目录
dataLogDir=/data/zk/log     # 事务日志目录（★ 必须与 dataDir 分离！）
clientPort=2181             # 客户端连接端口

# =============== 集群超时配置 ===============
initLimit=10    # Follower 初始连接 Leader 超时：10×2000ms=20s
syncLimit=5     # Follower 与 Leader 同步超时：5×2000ms=10s

# =============== 集群成员 ===============
server.1=zoo1:2888:3888         # host:数据端口:选举端口
server.2=zoo2:2888:3888
server.3=zoo3:2888:3888
server.4=zoo4:2888:3888:observer  # Observer 节点

# =============== 性能优化 ===============
leaderServes=yes              # Leader 是否接受客户端请求（建议 no，让 Leader 专注协调）
maxClientCnxns=60             # 每个 IP 最大连接数
autopurge.snapRetainCount=3   # 保留最新 N 个快照（★ 必须配置！）
autopurge.purgeInterval=1     # 每小时自动清理（★ 必须配置！）
snapCount=100000              # 每多少次写操作触发快照

# =============== 安全配置 ===============
4lw.commands.whitelist=stat,ruok,conf,mntr,isro  # 四字命令白名单

# =============== 动态配置（3.5+）===============
reconfigEnabled=true          # 开启动态重配置
```

## 六、myid 文件配置

每台服务器的 `dataDir` 目录下必须有 `myid` 文件：

```bash
# zoo1 服务器
echo "1" > /data/zk/data/myid

# zoo2 服务器
echo "2" > /data/zk/data/myid

# zoo3 服务器
echo "3" > /data/zk/data/myid
```

## 七、Quorum（法定人数）

```
Quorum = floor(N/2) + 1  （超过半数）
```

| 节点数 | Quorum | 允许宕机 |
|-------|--------|---------|
| 3 | 2 | 1 |
| 5 | 3 | 2 |
| 7 | 4 | 3 |

Quorum 在三处用到：
1. Leader 选举：需要 ≥ Quorum 票
2. 写事务提交：需要 ≥ Quorum 个 ACK
3. 集群可用性：存活节点 ≥ Quorum 才对外服务

## 八、动态扩缩容（3.5+）

```bash
# 添加节点（先以 Observer 加入，数据同步后提升为 Participant）
zkCli.sh -server zoo1:2181
reconfig -add server.4=zoo4:2888:3888:participant

# 删除节点
reconfig -remove 4

# 查看当前配置
config
```

**扩容最佳实践：**
1. 新节点先配置为 Observer，等数据同步完成
2. 再通过 reconfig 提升为 Participant
3. 每次只增减一台，逐步进行
