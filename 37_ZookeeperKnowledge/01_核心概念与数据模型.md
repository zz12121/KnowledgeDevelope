## 一、ZooKeeper 是什么

ZooKeeper 是一个**分布式协调服务**，基于树形数据模型提供分布式系统所需的协调能力：
- 存储少量协调元数据（不是通用 KV 存储）
- 通过 Watcher 机制主动推送变化通知
- 基于 ZAB 协议保证强一致性（CP 模型）

核心应用场景：服务注册发现 / 配置中心 / 分布式锁 / Leader 选举 / 集群管理

---

## 二、ZNode 树形数据模型

```
/
├── dubbo
│   └── com.example.OrderService
│       ├── providers       # 服务提供者临时节点
│       └── consumers       # 服务消费者
├── config
│   └── db-url              # 持久节点，存配置值
└── locks
    └── order-lock-0001     # 临时顺序节点，分布式锁
```

**路径规则：**
- 绝对路径，以 `/` 开头
- 路径唯一标识 ZNode
- 每个 ZNode 既可存数据（最大 1MB），也可有子节点

## 三、ZNode 四种类型

| 类型 | 持久？ | 有序？ | 典型用途 |
|------|--------|--------|---------|
| PERSISTENT | ✅ | ❌ | 配置存储、命名空间 |
| PERSISTENT_SEQUENTIAL | ✅ | ✅ | 全局唯一 ID 生成 |
| EPHEMERAL | ❌ | ❌ | 服务注册、Leader 选举 |
| EPHEMERAL_SEQUENTIAL | ❌ | ✅ | **分布式锁**（最常用） |

> **新增类型（ZK 3.6+）：** Container（子节点全删后自动删除），TTL 节点（超时自动删除）

## 四、ZNode stat 元数据

| 字段 | 含义 |
|------|------|
| `czxid` | 节点创建时的事务 ID |
| `mzxid` | 节点最后修改的事务 ID |
| `ctime` / `mtime` | 创建/修改时间戳 |
| `version` | **数据版本号**（每次 setData +1） |
| `cversion` | 子节点版本号 |
| `aversion` | ACL 版本号 |
| `ephemeralOwner` | 临时节点的 Session ID（持久节点为 0） |
| `dataLength` | 数据长度（字节） |
| `numChildren` | 子节点数量 |

**version 实现乐观锁：**
```java
Stat stat = new Stat();
zk.getData("/config/db", false, stat);
// 指定 version，ZK 做 CAS 校验，不匹配则抛 BadVersionException
zk.setData("/config/db", newData, stat.getVersion());
```

## 五、ZXID（全局事务 ID）

```
ZXID（64 bits）= epoch（高32位，Leader纪元） | counter（低32位，事务序号）
```

**作用：**
1. 全局有序：比较 ZXID 大小即知事务先后
2. 选举时判断数据新鲜度：ZXID 越大，数据越新
3. 防止脑裂：旧 Leader（epoch 更小）的事务被拒绝
4. epoch 在每次 Leader 选举后 +1，counter 重置为 0

## 六、ACL 权限控制

**格式：** `schema:id:permissions`

**schema 类型：**
- `world:anyone` — 所有人（默认，危险！）
- `digest:user:sha1-password` — 用户名密码
- `ip:192.168.1.0/24` — IP 白名单
- `auth` — 当前已认证用户

**permissions 位掩码：**
- `c`(CREATE) `r`(READ) `w`(WRITE) `d`(DELETE) `a`(ADMIN)

```java
// 生产环境推荐：只允许认证用户访问
List<ACL> acls = Arrays.asList(
    new ACL(ZooDefs.Perms.ALL,
        new Id("digest", DigestAuthenticationProvider.generateDigest("admin:secret")))
);
```

> **注意：** ACL 不递归，每个节点独立设置，子节点不继承父节点的 ACL。

## 七、ZNode 数量与容量规划

| 规模 | ZNode 数量 | 内存估算 | 备注 |
|------|-----------|---------|------|
| 小 | < 10万 | < 200MB | 开发/测试 |
| 中 | 10万~100万 | 200MB~2GB | 一般生产 |
| 大 | > 100万 | > 2GB | 需专项优化 |

**最佳实践：**
- ZNode 只存元数据（路径/地址/版本号），不存业务数据
- 单个节点数据建议 < 1KB，绝对不超过 1MB
- 超大数据存 OSS/DB，ZNode 只存引用
