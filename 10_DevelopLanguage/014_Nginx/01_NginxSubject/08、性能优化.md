### 八、性能优化类

---

###### 1. 如何优化 Nginx 的性能？

Nginx 性能优化从多个层面入手：

**① 进程与连接层：**
```nginx
worker_processes auto;          # Worker 数 = CPU 核数
worker_cpu_affinity auto;       # CPU 亲和性绑定，减少 CPU 缓存失效
worker_rlimit_nofile 65535;     # 最大文件描述符数
worker_priority -10;            # 提升 Worker 进程优先级

events {
    worker_connections 10240;
    use epoll;
    multi_accept on;            # 一次 accept 尽量多的连接
    accept_mutex off;           # 连接数高时关闭互斥锁（减少竞争）
}
```

**② 传输与协议层：**
```nginx
http {
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    keepalive_requests 1000;    # 每个长连接最多处理1000个请求
    
    # 开启 HTTP/2（减少连接数，多路复用）
    listen 443 ssl http2;
    
    # 压缩
    gzip on;
    gzip_comp_level 4;          # 生产建议 2-4，避免过高压缩耗 CPU
}
```

**③ 缓存层：**
```nginx
open_file_cache max=10000 inactive=20s;
open_file_cache_valid 30s;
proxy_cache_path ...;           # 启用代理缓存
```

**④ 系统层（Linux 内核参数）：**
```bash
# /etc/sysctl.conf
net.core.somaxconn = 65535          # 半连接队列大小
net.ipv4.tcp_max_syn_backlog = 65535
net.core.netdev_max_backlog = 65535
net.ipv4.tcp_fin_timeout = 10       # 减少 TIME_WAIT 时间
net.ipv4.tcp_tw_reuse = 1           # 复用 TIME_WAIT 连接
```

###### 2. keepalive 连接对性能有什么影响？

HTTP keepalive 允许同一个 TCP 连接处理多个 HTTP 请求，避免频繁三次握手/四次挥手：

**对客户端的 keepalive（Nginx ↔ 浏览器）：**
```nginx
keepalive_timeout 75s;       # 长连接保持时间（客户端侧）
keepalive_requests 1000;     # 单连接最多请求数
keepalive_disable msie6;     # 对 IE6 禁用（兼容性问题）
```

**对上游后端的 keepalive（Nginx ↔ 后端服务）：**
```nginx
upstream backend {
    server 192.168.1.10:8080;
    keepalive 32;            # 保持 32 个空闲长连接
    keepalive_requests 100;
    keepalive_timeout 60s;
}

location /api/ {
    proxy_pass http://backend;
    proxy_http_version 1.1;          # 必须指定 HTTP/1.1
    proxy_set_header Connection "";  # 清除 Connection: close 头
}
```

**性能影响：**
- 没有 keepalive：每个请求都要 TCP 三次握手 → 增加约 100-300ms 延迟
- 开启 keepalive：连接复用，高并发下减少 70% 以上的 TCP 连接建立开销
- 典型场景：Nginx → Spring Boot 的 `upstream keepalive 32` 可将 API 响应时间从 50ms 降至 15ms

###### 3. 如何调整 Nginx 的缓冲区大小？

缓冲区影响 Nginx 的内存使用和响应速度：

```nginx
http {
    # 客户端请求缓冲区
    client_header_buffer_size    4k;       # 读取请求头的缓冲区（大部分请求够用）
    large_client_header_buffers  4 16k;   # 大请求头（长 Cookie、JWT Token）
    client_body_buffer_size      128k;    # 请求体缓冲（超过会写临时文件）
    client_max_body_size         50m;     # 请求体最大值（上传文件限制）
    
    # 代理缓冲区（影响响应是否在内存中完成）
    proxy_buffering on;                    # 开启代理缓冲（默认 on）
    proxy_buffer_size 8k;                 # 读取上游响应头的缓冲区
    proxy_buffers 8 64k;                  # 读取上游响应体的缓冲区数量和大小
    proxy_busy_buffers_size 128k;         # 正在发送给客户端的缓冲区大小
    proxy_temp_file_write_size 256k;      # 临时文件写入块大小
    
    # 当 proxy_buffering off 时：实时转发（适合 SSE、WebSocket）
    # 当 proxy_buffering on 时：完整缓冲后再发给客户端（适合普通 API）
}
```

**调优方向：**
- 若后端响应大（如下载文件）：增大 `proxy_buffers`，减少临时文件写入
- 若上传文件多：增大 `client_body_buffer_size`，减少磁盘 I/O
- 若请求头很大（JWT token 很长）：增大 `large_client_header_buffers`

###### 4. 如何优化 Nginx 的文件描述符限制？

文件描述符（FD）是高并发场景的常见瓶颈，每个连接需要消耗 FD：

```nginx
# nginx.conf 层面
worker_rlimit_nofile 65535;    # 每个 Worker 进程的 FD 上限
```

```bash
# 操作系统层面（/etc/security/limits.conf）
nginx   soft    nofile  65535
nginx   hard    nofile  65535
root    soft    nofile  65535
root    hard    nofile  65535
# * 表示所有用户：
*       soft    nofile  65535
*       hard    nofile  65535

# 系统全局 FD 上限（/etc/sysctl.conf）
fs.file-max = 2097152
# 执行生效：
sysctl -p
```

**验证配置是否生效：**
```bash
# 查看 Nginx Worker 进程的 FD 限制
cat /proc/$(pgrep -f "nginx: worker")/limits | grep "open files"
# 查看当前打开的 FD 数量
ls /proc/$(pgrep -f "nginx: worker")/fd | wc -l
```

**经验公式：** FD 上限 ≥ `worker_connections` × 2（每个连接需要客户端FD + 上游FD，另加日志文件等）

###### 5. 如何使用 Nginx 的 HTTP/2 协议？

```nginx
server {
    listen 443 ssl http2;    # 在 listen 指令中添加 http2
    server_name www.example.com;
    
    ssl_certificate     /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    
    # HTTP/2 推送（Server Push，主动将关联资源推给客户端）
    location / {
        http2_push /style.css;       # 推送 CSS
        http2_push /app.js;          # 推送 JS
        proxy_pass http://backend;
    }
    
    # HTTP/2 推送预加载（从 Link 响应头读取）
    http2_push_preload on;
    # 后端在响应头中添加：Link: </style.css>; rel=preload; as=style
}
```

**HTTP/2 的性能提升：**
- **多路复用**：一个 TCP 连接并发处理多个请求，彻底解决 HTTP/1.1 的队头阻塞
- **Header 压缩（HPACK）**：头部压缩率约 70-80%，减少传输量
- **Server Push**：提前推送资源，减少往返次数
- **二进制分帧**：比 HTTP/1.1 的文本格式解析更高效

**注意：** HTTP/2 在 Nginx 中只支持 HTTPS（浏览器要求），HTTP 明文的 h2c 浏览器不支持。

###### 6. 如何配置 Nginx 的多核 CPU 优化？

```nginx
# 绑定 Worker 进程到特定 CPU 核心（减少 CPU 缓存污染）
worker_processes 4;
worker_cpu_affinity 0001 0010 0100 1000;  # 4核，每个 Worker 绑定一个核心
# 8核配置：
# worker_cpu_affinity 00000001 00000010 00000100 00001000 00010000 00100000 01000000 10000000;

# 自动绑定（推荐）
worker_processes auto;
worker_cpu_affinity auto;   # Nginx 1.9.10+ 支持
```

**其他 CPU 相关优化：**
```nginx
# 提升 Worker 进程优先级（-20最高，19最低）
worker_priority -10;

# 关闭互斥锁（高并发下减少 accept_mutex 带来的延迟，推荐并发数高时设为 off）
events {
    accept_mutex off;
    accept_mutex_delay 500ms;  # accept_mutex on 时的等待时间
}
```

###### 追问：Nginx 的 worker_processes 设置为 CPU 核数就够了吗？什么时候需要设置更多？

**通常情况：`worker_processes auto`（= CPU 核数）已经最优**。

但在以下场景，设置为 CPU 核数 × 2 可能有收益：
1. **I/O 密集型且有阻塞系统调用**：如使用了某些阻塞模块，Worker 卡在等待时其他 Worker 可以顶上
2. **NUMA 架构服务器**：多个 CPU 插槽，可设置为总核数 × 2，充分利用 NUMA 节点

**反而不能设太多的原因：**
- 每个 Worker 独立维护连接状态、缓存等数据结构，进程过多反而增加内存占用和上下文切换
- Nginx 事件驱动模型下，单核单 Worker 已能接近内核极限吞吐量
- 超过 CPU 核数后，多 Worker 竞争 CPU 时间片，性能反而下降

**压测是王道：** 用 `wrk` 或 `ab` 实测，找到最优的 Worker 数。
