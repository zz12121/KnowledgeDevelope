### 三、反向代理类

---

###### 1. 什么是反向代理？Nginx 如何实现反向代理？

**反向代理**是指代理服务器代替**后端服务器**接收来自客户端的请求，客户端不知道真实的后端服务器地址。与之相对的是**正向代理**（代替客户端访问外部服务器）。

```
客户端 → [反向代理 Nginx] → 后端服务器（Tomcat/Spring Boot/Node.js）
```

**反向代理 vs 正向代理：**
- 正向代理：帮客户端"翻墙"，服务端不知道真实客户端
- 反向代理：帮服务端接客，客户端不知道真实服务端

**Nginx 实现反向代理核心配置：**
```nginx
server {
    listen 80;
    server_name api.example.com;
    
    location /api/ {
        proxy_pass http://127.0.0.1:8080;       # 代理到后端
        proxy_set_header Host $host;             # 传递 Host 头
        proxy_set_header X-Real-IP $remote_addr; # 传递真实客户端IP
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**反向代理的价值：**
- 隐藏后端服务器真实 IP，提升安全性
- 统一入口，方便做限流、鉴权、日志
- SSL 卸载：前端 HTTPS，后端 HTTP，减轻后端压力
- 横向扩展：增减后端节点，客户端无感知

###### 2. proxy_pass 指令的作用是什么？如何使用？

`proxy_pass` 是 Nginx 反向代理的核心指令，决定请求被转发到哪里。

**关键细节：proxy_pass 末尾是否带 `/` 效果完全不同！**

```nginx
# 情况一：proxy_pass 不带 URI（不带 /）
location /api/ {
    proxy_pass http://backend;          # 保留 /api/ 前缀
    # 请求 /api/user → 转发到 http://backend/api/user
}

# 情况二：proxy_pass 带 URI（带 /）
location /api/ {
    proxy_pass http://backend/;         # 去掉 /api/ 前缀
    # 请求 /api/user → 转发到 http://backend/user
}

# 情况三：proxy_pass 带具体路径
location /api/ {
    proxy_pass http://backend/v2/;      # 替换为 /v2/
    # 请求 /api/user → 转发到 http://backend/v2/user
}
```

**完整的反向代理头部配置（最佳实践）：**
```nginx
location / {
    proxy_pass http://backend;
    proxy_http_version 1.1;                      # 启用 HTTP/1.1（支持 keepalive）
    proxy_set_header Connection "";              # 清除 Connection 头（启用长连接）
    proxy_set_header Host              $host;
    proxy_set_header X-Real-IP         $remote_addr;
    proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_connect_timeout 10s;
    proxy_read_timeout    60s;
    proxy_send_timeout    60s;
}
```

###### 3. 如何在 Nginx 中配置代理缓存？

代理缓存可以将后端响应缓存在 Nginx 本地，相同请求直接命中缓存，大幅降低后端压力：

```nginx
http {
    # 定义缓存存储区域
    proxy_cache_path /var/cache/nginx 
                     levels=1:2              # 两级目录结构，避免单目录文件过多
                     keys_zone=my_cache:10m  # 共享内存区名称和大小（存储 key 的索引）
                     max_size=10g            # 缓存文件最大总大小
                     inactive=60m            # 60分钟未访问则清除
                     use_temp_path=off;      # 直接写入缓存目录，避免二次拷贝
    
    server {
        location /api/ {
            proxy_cache my_cache;
            proxy_cache_key "$scheme$request_method$host$request_uri";  # 缓存 key
            proxy_cache_valid 200 302 10m;  # 200/302 响应缓存 10 分钟
            proxy_cache_valid 404      1m;  # 404 响应缓存 1 分钟
            proxy_cache_use_stale error timeout updating;  # 后端出错时返回过期缓存
            proxy_cache_lock on;            # 同一 key 只有一个请求穿透到后端（防雪崩）
            
            add_header X-Cache-Status $upstream_cache_status;  # 响应头显示缓存命中情况
            proxy_pass http://backend;
        }
        
        # 绕过缓存（如管理后台）
        location /api/admin/ {
            proxy_cache_bypass 1;
            proxy_no_cache     1;
            proxy_pass http://backend;
        }
    }
}
```

`$upstream_cache_status` 取值：`HIT`（命中）/ `MISS`（未命中）/ `EXPIRED`（过期）/ `BYPASS`（绕过）。

###### 4. 如何在 Nginx 中设置代理超时？

```nginx
location /api/ {
    proxy_pass http://backend;
    
    proxy_connect_timeout 5s;     # 与后端建立 TCP 连接的超时（建议 3-10s）
    proxy_send_timeout    30s;    # Nginx 向后端发送请求的超时
    proxy_read_timeout    60s;    # Nginx 等待后端响应的超时（最关键）
    
    # 超时后的重试（谨慎配置，避免重复提交）
    proxy_next_upstream error timeout http_500 http_502 http_503;
    proxy_next_upstream_tries 2;  # 最多重试到 2 台后端
    proxy_next_upstream_timeout 10s;
}
```

**生产建议：**
- `proxy_connect_timeout` 超过 10s 基本说明后端宕机，不要设太大
- `proxy_read_timeout` 要根据业务最慢接口设定（如文件上传可设 300s）
- `proxy_next_upstream` 的 `non_idempotent` 标记要慎用，POST 请求默认不重试

###### 5. 如何在 Nginx 中传递客户端真实 IP？

默认情况下后端拿到的是 Nginx 的 IP，需要通过 Header 透传真实 IP：

```nginx
location / {
    proxy_pass http://backend;
    
    # 单层代理：直接传递客户端 IP
    proxy_set_header X-Real-IP $remote_addr;
    
    # 多层代理：追加 IP 链（格式：客户端IP, 上层代理IP, ...）
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    
    proxy_set_header X-Forwarded-Proto $scheme;  # 透传协议（http/https）
    proxy_set_header Host $host;
}
```

**后端 Java（Spring Boot）获取真实 IP：**
```java
public String getClientIp(HttpServletRequest request) {
    String ip = request.getHeader("X-Real-IP");
    if (ip == null || ip.isEmpty()) {
        ip = request.getHeader("X-Forwarded-For");
        if (ip != null && ip.contains(",")) {
            ip = ip.split(",")[0].trim();  // 取第一个（最原始的客户端IP）
        }
    }
    if (ip == null || ip.isEmpty()) {
        ip = request.getRemoteAddr();
    }
    return ip;
}
```

**安全注意**：`X-Forwarded-For` 可以被客户端伪造，需要在 Nginx 最外层用 `real_ip_module` 进行可信 IP 校验：
```nginx
set_real_ip_from 10.0.0.0/8;      # 信任内网代理
real_ip_header X-Forwarded-For;
real_ip_recursive on;
```

###### 6. upstream 模块是如何工作的？

`upstream` 定义一组后端服务器，是负载均衡的基础：

```nginx
upstream backend {
    # 负载均衡策略（默认轮询）
    # least_conn;       # 最少连接数策略
    # ip_hash;          # 基于客户端 IP 哈希（会话保持）
    # hash $request_uri consistent;  # 基于 URI 一致性哈希
    
    server 192.168.1.10:8080 weight=3;          # 权重为3
    server 192.168.1.11:8080 weight=1;          # 权重为1
    server 192.168.1.12:8080 backup;            # 备用服务器
    server 192.168.1.13:8080 down;              # 标记为下线
    
    # 故障转移配置
    server 192.168.1.10:8080 max_fails=3 fail_timeout=30s;
    # 30秒内失败3次，则将该节点剔除30秒
    
    keepalive 32;           # 与每台后端保持 32 个长连接（减少 TCP 握手）
    keepalive_requests 100; # 每个长连接最多处理 100 个请求
    keepalive_timeout 60s;  # 长连接空闲超时
}
```

**工作机制：**
1. Nginx 根据负载均衡算法选择一台后端服务器
2. 若选中的服务器不可用（连接失败/超时），触发 `proxy_next_upstream` 重试
3. 若某服务器在 `fail_timeout` 内失败次数达到 `max_fails`，临时剔除该服务器
4. `fail_timeout` 后再次尝试，若成功则恢复该服务器

###### 追问：proxy_pass 到 upstream 时，如何实现请求级别的超时重试而不影响幂等性？

核心在于区分**幂等请求**和**非幂等请求**：

```nginx
upstream backend {
    server 192.168.1.10:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:8080 max_fails=3 fail_timeout=30s;
}

location /api/ {
    proxy_pass http://backend;
    
    # 只在以下情况重试（错误/超时/特定状态码）
    # 注意：不加 non_idempotent，则 POST/PATCH 等非幂等方法不会重试
    proxy_next_upstream error timeout http_502 http_503;
    proxy_next_upstream_tries 2;    # 最多换一台重试
    proxy_next_upstream_timeout 5s; # 重试总时间不超过5s
}

# 对于幂等接口（GET/查询），可更激进地重试
location /api/query/ {
    proxy_pass http://backend;
    proxy_next_upstream error timeout http_500 http_502 http_503;
    proxy_next_upstream_tries 3;
}
```

对于 POST 等非幂等请求，建议在应用层实现幂等性（如订单号去重），而非依赖 Nginx 重试。
