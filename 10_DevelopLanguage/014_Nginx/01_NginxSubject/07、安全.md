###### 1. 如何隐藏 Nginx 的版本号？

默认情况下 Nginx 会在响应头和错误页面中暴露版本号（如 `Server: nginx/1.24.0`），这给攻击者提供了可利用的信息：

```nginx
http {
    server_tokens off;  # 关闭版本号显示（响应头只显示 "nginx"）
    
    # 进一步：完全修改 Server 头（需要 headers_more 模块）
    # more_set_headers 'Server: webserver';
}
```

**同时建议隐藏的信息：**
```nginx
# 自定义错误页面，避免暴露 Nginx 默认错误页
error_page 403 404 500 502 503 504 /error.html;
location = /error.html {
    root /var/www/errors;
    internal;  # 只允许内部重定向，外部不能直接访问
}
```

###### 2. 如何在 Nginx 中防止 DDoS 攻击？

```nginx
http {
    # 限制请求速率
    limit_req_zone $binary_remote_addr zone=req_limit:10m rate=10r/s;
    
    # 限制并发连接数
    limit_conn_zone $binary_remote_addr zone=conn_limit:10m;
    
    server {
        # 应用速率限制
        limit_req zone=req_limit burst=20 nodelay;
        limit_conn conn_limit 10;
        
        # 限制请求体大小（防止大文件攻击）
        client_max_body_size 10m;
        
        # 限制慢连接攻击（Slowloris）
        client_header_timeout 10s;
        client_body_timeout   10s;
        send_timeout          10s;
        
        # 禁止空 User-Agent 和异常 User-Agent
        if ($http_user_agent = "") {
            return 444;  # 444：Nginx 特有，直接关闭连接不返回任何响应
        }
        
        # 只允许常用 HTTP 方法
        if ($request_method !~ ^(GET|POST|PUT|DELETE|HEAD|OPTIONS)$) {
            return 405;
        }
    }
}
```

**深度防御层次：**
1. **网络层**：云厂商 DDoS 高防（腾讯云/阿里云 BGP 清洗）
2. **传输层**：防火墙限制单 IP 连接速率、SYN Cookie
3. **应用层**：Nginx 限流 + WAF（Modsecurity/NAXSI）+ 验证码
4. **CDN**：将攻击流量分散到全球边缘节点

###### 3. 如何限制 Nginx 的并发连接数？

```nginx
http {
    # 定义共享内存区域，存储各 IP 的连接数
    limit_conn_zone $binary_remote_addr zone=perip:10m;   # 按 IP 限制
    limit_conn_zone $server_name zone=perserver:10m;      # 按虚拟主机限制
    
    server {
        # 单 IP 最多 10 个并发连接
        limit_conn perip 10;
        
        # 整个虚拟主机最多 100 个并发连接
        limit_conn perserver 100;
        
        # 超过限制时返回 503（默认 503）
        limit_conn_status 503;
        
        # 记录限制日志
        limit_conn_log_level warn;
        
        # 不同 location 可以有不同限制
        location /api/ {
            limit_conn perip 5;   # API 接口更严格
        }
        
        location /download/ {
            limit_conn perip 2;   # 下载接口限制更严
            limit_rate 1m;        # 同时限制下载速度为 1MB/s
        }
    }
}
```

###### 4. limit_req 和 limit_conn 模块的作用是什么？

**`limit_conn`（连接数限制）：** 控制同一时刻某个 key（如 IP）的并发连接数。

**`limit_req`（请求速率限制）：** 控制请求的**速率**（每秒请求数），基于**漏桶算法**实现：

```nginx
http {
    # 定义速率限制区域
    # $binary_remote_addr 比 $remote_addr 节省内存（4字节 vs 最多46字节）
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=100r/s;  # 100次/秒
    limit_req_zone $binary_remote_addr zone=login_limit:10m rate=1r/s;  # 1次/秒（防暴力破解）
    
    server {
        location /api/ {
            limit_req zone=api_limit burst=200 nodelay;
            # burst=200：允许短时间突发200个请求（令牌桶效果）
            # nodelay：突发请求立即处理（不排队等待），超出 burst 返回 503
        }
        
        location /login {
            limit_req zone=login_limit burst=5;
            # 不加 nodelay：超出速率的请求排队等待，队列满了才返回 503
            # 适合需要"缓冲"的场景
        }
    }
}
```

**两者对比：**

| 维度 | limit_conn | limit_req |
|------|-----------|----------|
| 控制维度 | 并发连接数（同时） | 请求速率（每秒） |
| 算法 | 简单计数 | 漏桶算法 |
| 适用场景 | 防连接耗尽 | 防接口刷刷刷 |
| 是否支持突发 | 不支持 | 支持（burst 参数） |

###### 5. 如何配置 Nginx 的基本认证（Basic Auth）？

```bash
# 1. 生成密码文件（使用 htpasswd 工具，openssl 或 apache2-utils 包）
htpasswd -c /etc/nginx/.htpasswd admin       # 创建文件并添加 admin 用户
htpasswd /etc/nginx/.htpasswd developer      # 追加 developer 用户
# 输入密码后，密码以 bcrypt 哈希存储
```

```nginx
# 2. Nginx 配置
server {
    location /admin/ {
        auth_basic "Admin Area";                    # 弹出的认证提示
        auth_basic_user_file /etc/nginx/.htpasswd;  # 密码文件路径
        proxy_pass http://admin-backend;
    }
    
    # 需要认证的特定文件
    location /api/internal/ {
        auth_basic "Internal API";
        auth_basic_user_file /etc/nginx/.htpasswd;
        
        # 同时限制 IP
        allow 10.0.0.0/8;
        deny  all;
    }
}
```

**生产注意事项：**
- Basic Auth 密码是 Base64 编码传输，**必须配合 HTTPS 使用**，否则明文可见
- 更安全的替代方案：JWT Token、OAuth2、`auth_request` 模块（调用外部认证服务）

###### 6. 如何在 Nginx 中配置 SSL/TLS 安全策略？

```nginx
server {
    listen 443 ssl http2;
    
    ssl_certificate     /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;
    
    # 仅允许 TLS 1.2 和 1.3（禁用老旧不安全的 SSL 3.0/TLS 1.0/1.1）
    ssl_protocols TLSv1.2 TLSv1.3;
    
    # Mozilla Intermediate 推荐加密套件
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;  # TLS 1.3 下由客户端决定加密套件
    
    # OCSP Stapling（在线证书状态装订，减少客户端验证延迟）
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/nginx/ssl/chain.pem;
    resolver 8.8.8.8 8.8.4.4 valid=300s;
    resolver_timeout 5s;
    
    # DH 参数（防止 Logjam 攻击）
    ssl_dhparam /etc/nginx/ssl/dhparam.pem;  # openssl dhparam -out dhparam.pem 4096
    
    # 安全响应头
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
    add_header X-Frame-Options "SAMEORIGIN" always;          # 防点击劫持
    add_header X-Content-Type-Options "nosniff" always;      # 防 MIME 嗅探
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Content-Security-Policy "default-src 'self'" always;
}
```

###### 7. 如何防止 SQL 注入和 XSS 攻击？

Nginx 本身不是 WAF，但可以配合规则做基础过滤：

```nginx
server {
    # 阻断包含 SQL 注入特征的请求
    location / {
        # 检查 URL 中的 SQL 注入关键词
        if ($query_string ~* "(union|select|insert|update|delete|drop|truncate|exec|eval)") {
            return 403;
        }
        
        # 检查 XSS 特征
        if ($query_string ~* "(<script|javascript:|vbscript:|onload=|onerror=)") {
            return 403;
        }
        
        # 阻断 URL 编码的注入尝试
        if ($request_uri ~* "%3c|%3e|%27|%22|%0a|%0d") {
            return 403;
        }
    }
}
```

**更专业的 WAF 方案：**
1. **ModSecurity**：开源 WAF 模块，规则库完善（OWASP CRS）
2. **NAXSI**（Nginx Anti XSS & SQL Injection）：轻量级 Nginx WAF 模块
3. **OpenResty + Lua**：灵活自定义安全逻辑
4. **云 WAF**：腾讯云 WAF、阿里云 WAF（推荐生产使用，无需自维护规则）

###### 追问：Nginx 的 444 状态码是什么？什么时候用？

`444` 是 Nginx **私有扩展状态码**，不属于 HTTP 标准，代表"**直接关闭 TCP 连接，不返回任何响应**"：

```nginx
# 使用场景
if ($http_user_agent = "") {
    return 444;  # 无 UA 的爬虫/扫描器，直接断连
}

if ($http_host = "") {
    return 444;  # 没有 Host 头的扫描请求，直接断连
}

# 阻断已知恶意 IP
geo $bad_ip {
    default 0;
    1.2.3.4 1;
    5.6.7.8 1;
}
if ($bad_ip) {
    return 444;
}
```

与 `return 403` 相比，`return 444` 不返回任何数据，攻击者/扫描器收到的是连接重置，无法通过响应内容判断服务器类型，且节省带宽。适用于对付扫描器、爬虫、DDoS 攻击流量。
