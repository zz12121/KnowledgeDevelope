### 十四、对比类

---

###### 1. Nginx 和 Apache 在架构上有什么区别？

这是面试最高频的对比题，核心差异在于并发模型：

**Apache（多进程/多线程模型）：**
- `prefork MPM`：每个请求一个进程，稳定但内存消耗极大
- `worker MPM`：每个请求一个线程，内存优于 prefork，但高并发下线程切换开销仍很大
- `event MPM`（Apache 2.4）：类 Nginx 的事件驱动，但历史包袱较重

**Nginx（事件驱动 + 异步非阻塞）：**
- Master-Worker 多进程，Worker 单线程 + epoll
- 10000 并发连接对 Nginx 来说轻松应对，内存仅需几十 MB

| 对比维度 | Nginx | Apache |
|---------|-------|--------|
| 并发模型 | 事件驱动，异步非阻塞 | 多进程/多线程（同步阻塞） |
| 高并发内存 | 极低（~数十 MB 处理万级并发） | 较高（千级并发即需数 GB） |
| 静态资源性能 | 极强，sendfile 零拷贝 | 一般 |
| 动态内容 | 需反向代理到后端 | 可内嵌 PHP/Python（mod_php） |
| 配置灵活性 | 较严格（正确即可） | 支持 .htaccess，目录级配置灵活 |
| 模块丰富度 | 较少（但 OpenResty 弥补） | 极丰富（mod_rewrite/mod_security 等） |
| Windows 支持 | 弱 | 好 |

**选择原则：** 新项目几乎全选 Nginx；老项目依赖 .htaccess 或 mod_php 的选 Apache；高并发静态资源/反向代理首选 Nginx。

###### 2. Nginx 和 HAProxy 的区别是什么？

两者都可以做负载均衡，但定位不同：

| 对比维度 | Nginx | HAProxy |
|---------|-------|---------|
| 主要定位 | Web 服务器 + HTTP 反向代理 | 专业负载均衡器 |
| 支持协议 | HTTP/HTTPS/TCP（1.9+）/UDP（1.15+） | HTTP/HTTPS/TCP/UDP（更全面） |
| 负载均衡算法 | 5种（Round Robin/Weight/ip_hash/least_conn/hash） | 8种（含 Source/URI/随机/最少响应时间） |
| 主动健康检查 | 需要商业版或第三方模块 | 原生支持，功能完善 |
| 会话保持 | 基本支持（需模块或 Lua） | 原生 Cookie Stick，功能强大 |
| 连接级监控 | stub_status（基础） | 内置强大 Stats 页面（实时监控） |
| SSL 终止 | ✅ 完善 | ✅ 完善 |
| 静态资源服务 | ✅ 强 | ❌ 不支持 |
| 动态配置 | ❌（需 reload） | ✅（Runtime API 动态增删节点） |

**选择原则：**
- 需要 **静态资源服务 + 反向代理 + 负载均衡** → **Nginx**（全能选手）
- 纯**四层/七层负载均衡**，需要精细的健康检查和监控 → **HAProxy**（专业选手）
- 国内大厂通常 HAProxy 做核心 LB，Nginx 做接入层

###### 3. Nginx 和 LVS 的区别是什么？

LVS（Linux Virtual Server）是 Linux 内核级别的负载均衡，和 Nginx 工作在不同的网络层：

| 对比维度 | Nginx | LVS |
|---------|-------|-----|
| 工作层级 | 七层（应用层，HTTP） | 四层（传输层，TCP/UDP） |
| 工作位置 | 用户空间 | 内核空间（IPVS 模块） |
| 性能 | 较高（数万并发） | 极高（百万级并发，接近硬件转发速度） |
| 功能 | 内容感知，可按 URL/Header 路由 | 纯流量分发，不感知内容 |
| SSL 卸载 | ✅ 支持 | ❌ 不支持 |
| 会话保持 | ✅ 多种方式 | 基于 IP 的简单保持 |
| 健康检查 | 被动（主动需模块） | 需配合 Keepalived |
| 配置复杂度 | 简单，配置文件直观 | 复杂，需要网络层配置 |

**典型架构：LVS + Keepalived（高可用） + Nginx（七层处理）+ 后端集群**
```
客户端 → LVS（四层分发，100万QPS） → Nginx集群（七层处理，SSL/限流/缓存） → 后端服务
```
LVS 负责流量分摊，Nginx 负责业务逻辑处理，各司其职，性能和功能兼顾。

###### 4. Nginx 和 Tengine 的区别是什么？

Tengine 是淘宝/阿里巴巴基于 Nginx 开发的 Web 服务器，于 2011 年开源：

| 对比维度 | Nginx | Tengine |
|---------|-------|---------|
| 主动健康检查 | 需要第三方模块 | 内置（`check` 指令） |
| 请求合并（PIPE） | 不支持 | 支持（将多个小请求合并到后端） |
| 动态 upstream | 需要商业版 | 支持（lua 模块集成） |
| SSL 优化 | 标准 | 有阿里云场景优化 |
| 流量控制 | limit_req/limit_conn | 更丰富的流控指令 |
| Lua 集成 | 需要单独安装 OpenResty | 内置 lua 支持 |
| 社区活跃度 | 极活跃（官方维护） | 较低（阿里内部需求主导，更新节奏慢） |
| 跟进官方版本 | 最新版 | 可能落后 Nginx 官方几个版本 |

**选择建议：**
- 一般项目：**标准 Nginx + OpenResty**，社区活跃、文档丰富、版本最新
- 已有 Tengine 的历史项目：维持现状，但新功能开发应注意版本落后问题
- 阿里云用户：可以直接使用阿里云 SLB（底层 Tengine），无需自维护

###### 5. 什么场景下选择 Nginx，什么场景下选择 Apache？

**选 Nginx 的场景：**

1. **高并发静态资源服务**：CDN 节点、图片服务器、前端打包文件分发
2. **反向代理 + 负载均衡**：Java/Node.js/Python 等后端的接入层
3. **API 网关**：配合 OpenResty/Lua 实现限流、鉴权、路由
4. **HTTPS 终止**：统一处理 SSL/TLS，后端无需关心加密
5. **微服务架构接入层**：Nginx 作为所有服务的统一入口
6. **资源有限的 VPS**：内存消耗极低，512MB 内存的服务器也能流畅运行

**选 Apache 的场景：**

1. **PHP 应用（非 PHP-FPM 模式）**：`mod_php` 内嵌 PHP 处理器，配置简单
2. **需要 `.htaccess` 的应用**：WordPress、Drupal 等 CMS 系统依赖 `.htaccess` 做目录级配置
3. **复杂 CGI/旧有应用**：遗留 Perl、CGI 脚本
4. **需要 `mod_rewrite` 高度灵活性**：某些特殊 URL 重写规则只有 Apache 能优雅实现

**一句话总结：**
- **Nginx**：现代 Web 架构的标配，高并发、低内存、反向代理首选
- **Apache**：遗留 PHP 系统、需要 .htaccess 的 CMS 场景的首选

###### 追问：Nginx、HAProxy、LVS 三者如何组合成企业级高可用架构？

```
互联网流量
     │
     ▼
[DNS 负载均衡] ← 将流量分配到多个接入点（GSLB 全局负载均衡）
     │
     ▼
[LVS + Keepalived] × 2   ← 四层 VIP 高可用（Active/Standby，Keepalived 故障切换）
     │                       内核空间转发，百万QPS，不解包
     ▼
[Nginx 集群] × N         ← 七层处理层（SSL 卸载/限流/缓存/路由）
     │                       部署 WAF、日志收集、金丝雀发布逻辑
     ▼
[后端应用集群]           ← Spring Boot / Go / Node.js 等

高可用保障：
- LVS 层：Keepalived VRRP 协议，故障切换 < 1s
- Nginx 层：LVS 对 Nginx 做主动健康检查，故障节点自动摘除
- 后端层：Nginx upstream max_fails 被动检查 + 健康检查接口主动检查
```

这是国内互联网公司（BAT 级别）的标准架构，每一层各司其职，兼顾了性能（LVS 百万QPS）和功能（Nginx 七层处理）。
