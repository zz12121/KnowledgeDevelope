### 一、基础概念类

---

###### 1. 什么是 Nginx？它的主要用途是什么？

Nginx（发音 "engine-x"）是一款高性能的开源 **HTTP 服务器、反向代理服务器和邮件代理服务器**，由俄罗斯工程师 Igor Sysoev 于 2004 年发布，以高并发、低内存占用著称。

**主要用途：**

1. **静态资源服务器**：直接提供 HTML、CSS、JS、图片等静态文件，性能远超 Apache。
2. **反向代理**：接收客户端请求，转发给后端应用服务器（Tomcat、Node.js、Spring Boot 等），隐藏后端真实地址。
3. **负载均衡**：在多台后端服务器之间分发请求，支持轮询、加权轮询、ip_hash、least_conn 等策略。
4. **HTTP 缓存**：缓存后端响应，减少后端压力，加速响应。
5. **SSL/TLS 终止**：统一处理 HTTPS 加密解密，后端服务可专注业务逻辑。
6. **API 网关**：限流、鉴权、路由，常与 OpenResty/Lua 配合实现。

---

###### 2. Nginx 的架构是怎样的？

Nginx 采用 **Master-Worker 多进程架构**，核心设计如下：

```
+------------------+
|   Master Process  |  ← 管理进程（读配置、管理Worker）
+------------------+
        |
   fork多个Worker
        |
+-------+-------+
| Worker | Worker |  ← 实际处理请求（单线程+异步事件驱动）
+-------+-------+
```

- **Master 进程**：读取并验证配置文件；管理 Worker 进程的生命周期（启动/停止/热重载）；接收信号（HUP/QUIT/USR1 等）。
- **Worker 进程**：每个 Worker 独立运行，单线程但基于 epoll/kqueue 事件驱动，可同时处理数千个连接；Worker 数量通常设置为 CPU 核数（`worker_processes auto`）。
- **Cache Manager / Cache Loader**：可选进程，负责缓存文件的管理和加载。

这种架构的优点：
- Worker 进程崩溃不影响 Master，Master 可快速拉起新 Worker
- 无线程上下文切换开销，内存占用极低
- 热重载（`nginx -s reload`）不中断现有连接

---

###### 3. Nginx 的 master 和 worker 进程分别是做什么的？

**Master 进程（管理者）：**
- 以 root 权限启动，绑定需要特权的端口（如 80、443）
- 读取、解析、验证配置文件
- 创建并监控 Worker 进程，Worker 异常退出时自动重启
- 处理系统信号：`HUP`（重载配置）、`QUIT`（优雅停止）、`USR1`（重开日志文件）、`USR2`（升级二进制）、`WINCH`（优雅停止 Worker）
- 自身**不处理任何客户端请求**

**Worker 进程（工作者）：**
- 从 Master fork 出来，以低权限用户运行（通常是 `nginx` 或 `nobody`）
- 每个 Worker 运行一个**单线程事件循环**，基于 epoll（Linux）/kqueue（BSD/macOS）
- 独立处理网络连接、读取请求、调用上游、返回响应
- Worker 之间通过**共享内存**交换数据（如限流计数器、缓存元数据）
- 每个 Worker 可并发处理 `worker_connections` 个连接

---

###### 4. 什么是 Nginx 的事件驱动模型？

Nginx 的事件驱动模型是其高并发能力的核心，本质是 **I/O 多路复用 + 非阻塞 I/O + 回调机制**。

**核心流程：**
1. Worker 进程通过 `epoll_create` 创建事件监听器
2. 当有新连接或数据到来时，内核将就绪的文件描述符通知给 Nginx（`epoll_wait` 返回）
3. Nginx 遍历就绪事件，调用对应的**回调函数**处理（读请求头、发送响应等）
4. 所有 I/O 操作均为**非阻塞**：如果数据未就绪就立即返回，不阻塞整个进程
5. 处理完一批事件后继续 `epoll_wait`，形成事件循环

**与多线程模型的对比：**

| 维度 | 多线程（Apache prefork） | 事件驱动（Nginx） |
|------|------------------------|----------------|
| 并发1万连接 | 1万线程，内存 > 10GB | 10个Worker，内存 < 100MB |
| 上下文切换 | 频繁，开销大 | 几乎没有 |
| CPU利用率 | 低（大量线程等待I/O） | 高（只处理就绪事件） |

---

###### 5. Nginx 支持哪些操作系统？

Nginx 跨平台支持广泛：

- **Linux**：主要生产环境，支持 epoll，推荐 CentOS/Ubuntu/Debian
- **FreeBSD / macOS**：使用 kqueue 事件模型
- **Windows**：功能受限（不支持 `worker_processes > 1`、`select` 事件模型），不推荐生产使用
- **Solaris / AIX / HP-UX**：使用 `/dev/poll` 或 `eventport`

生产环境**强烈推荐 Linux**，epoll 是目前性能最好的事件模型。

---

###### 追问：Nginx 为什么不用多线程而是单线程事件驱动？

根本原因是 **I/O 密集型场景下，线程的价值远低于其开销**：

1. **线程切换代价高**：创建线程需要分配 8MB 栈空间，上下文切换需保存/恢复 CPU 寄存器，千级并发时切换开销超过业务逻辑本身
2. **锁竞争复杂**：多线程共享内存需要加锁，极易出现死锁和竞态条件，维护成本高
3. **内存压力大**：1万并发 = 1万线程 ≈ 80GB 内存，完全不可行
4. **I/O 大部分时间在等待**：网络 I/O 99% 的时间在等数据包，线程在这期间全部阻塞浪费 CPU

单线程事件驱动的代价：CPU 密集型任务（如复杂 Lua 计算）会阻塞 Worker 事件循环，但 Nginx 定位是网络代理层，不应承担重计算，这个取舍完全合理。
