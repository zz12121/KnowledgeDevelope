### 十一、故障排查类

---

###### 1. Nginx 常见的错误状态码有哪些？如何排查？

| 状态码 | 含义 | 常见原因 |
|--------|------|---------|
| 400 | Bad Request | 请求格式错误，请求头过大（超过 large_client_header_buffers） |
| 403 | Forbidden | 文件权限不足，allow/deny 规则，目录没有 index 文件 |
| 404 | Not Found | 文件不存在，root/alias 路径错误，location 规则不匹配 |
| 413 | Request Entity Too Large | 上传文件超过 client_max_body_size |
| 499 | Client Closed Request | 客户端主动断开（超时等待后端响应），说明后端响应太慢 |
| 500 | Internal Server Error | 后端服务异常，Lua 脚本错误 |
| 502 | Bad Gateway | 上游无响应或连接被拒绝（后端宕机/端口未监听） |
| 503 | Service Unavailable | 所有上游节点不可用，或 limit_req/limit_conn 触发限流 |
| 504 | Gateway Timeout | 上游响应超时，超过了 proxy_read_timeout |

**排查流程：** 看错误日志 → 看访问日志 → 检查后端应用日志 → curl 直接访问后端 → 检查网络/防火墙。

###### 2. 如何排查 Nginx 的 502、504 错误？

**502 Bad Gateway 排查步骤：**
```bash
# 1. 检查后端服务是否正在运行
systemctl status tomcat
netstat -tlnp | grep 8080     # 确认端口在监听

# 2. 直接访问后端（绕过 Nginx）
curl -v http://127.0.0.1:8080/api/test

# 3. 检查 Nginx 错误日志
tail -100 /var/log/nginx/error.log
# 常见错误：
# connect() failed (111: Connection refused)  ← 后端端口未监听
# no live upstreams while connecting to upstream ← 所有节点都不可用

# 4. 检查防火墙（Nginx → 后端的内网端口是否放通）
iptables -L -n | grep 8080
```

**504 Gateway Timeout 排查步骤：**
```bash
# 1. 检查后端响应时间
time curl http://127.0.0.1:8080/api/slow-endpoint

# 2. 查看 proxy_read_timeout 设置
nginx -T | grep proxy_read_timeout

# 3. 检查后端是否有慢 SQL、GC 停顿等
# 查看 Java 应用 GC 日志
# 查看数据库慢查询日志

# 4. 临时增大超时（应急）
proxy_read_timeout 300s;   # 增大到5分钟
nginx -s reload
```

###### 3. 如何检查 Nginx 配置文件是否正确？

```bash
# 检查语法（不会重启服务）
nginx -t
# 输出：nginx: configuration file /etc/nginx/nginx.conf test is successful

# 显示完整配置（含 include 的所有配置文件展开）
nginx -T

# 检查指定配置文件
nginx -t -c /etc/nginx/nginx.conf

# 常见报错示例及含义：
# nginx: [emerg] unknown directive "xxx"     ← 拼写错误或模块未编译
# nginx: [emerg] invalid number of arguments in "proxy_pass" directive  ← 参数错误
# nginx: [emerg] "}" expected near end of file  ← 花括号未闭合
# nginx: [warn] conflicting server name "xxx"  ← 多个 server 有相同 server_name
```

###### 4. 如何平滑重启 Nginx？

```bash
# 方式一：nginx -s reload（最常用）
nginx -s reload
# 先检查配置，配置无误才热重载；Master 通知 Worker 优雅停止（处理完现有请求），
# 然后以新配置启动新 Worker 接管连接。全程零停机。

# 方式二：发送 HUP 信号（与 reload 等价）
kill -HUP $(cat /var/run/nginx.pid)

# 方式三：systemctl reload（现代 Linux 推荐）
systemctl reload nginx

# 完全停止和启动（有停机时间）
nginx -s stop     # 立即停止（等同于 TERM 信号）
nginx -s quit     # 优雅停止（等同于 QUIT 信号，等待现有请求完成）

# 重新打开日志文件（日志切割时用）
nginx -s reopen
kill -USR1 $(cat /var/run/nginx.pid)
```

**各命令对比：**
- `reload` = 热重载配置，零停机 ✅
- `stop` = 立即终止，现有连接中断 ⚠️
- `quit` = 优雅停止，等待所有连接处理完毕 ✅

###### 5. 如何查看 Nginx 的进程状态？

```bash
# 查看 Nginx 进程（Master + Worker）
ps aux | grep nginx
# 输出示例：
# root     12345  0.0  0.0  nginx: master process /usr/sbin/nginx
# nginx    12346  0.0  0.1  nginx: worker process
# nginx    12347  0.0  0.1  nginx: worker process

# 查看 Worker 数量（等于 CPU 核数）
ps aux | grep "nginx: worker" | grep -v grep | wc -l

# 查看每个 Worker 的连接数
cat /proc/$(pgrep -f "nginx: worker" | head -1)/net/sockstat

# 查看 stub_status（需要先配置）
curl http://localhost/nginx_status

# 实时查看连接状态
watch -n 1 'ss -s'

# 查看 Nginx 监听的端口
ss -tlnp | grep nginx
netstat -tlnp | grep nginx
```

###### 6. 如何调试 Nginx 配置问题？

```bash
# 方法一：开启 debug 日志（临时，性能影响极大，排查完立即关闭）
error_log /var/log/nginx/debug.log debug;
# 或只对特定 IP 开启 debug
events {
    debug_connection 192.168.1.100;   # 只对该 IP 开启 debug 日志
}

# 方法二：用 add_header 输出 Nginx 内部变量（调试请求匹配）
location /test {
    add_header X-Uri $uri;
    add_header X-Host $host;
    add_header X-Real-IP $remote_addr;
    add_header X-Request-Filename $request_filename;
    return 200 "debug info in headers";
}

# 方法三：return 200 快速测试 location 匹配
location /api/user/ {
    return 200 "matched: /api/user/";
}

# 方法四：nginx -T 查看最终生效的完整配置
nginx -T | grep -A 10 "server_name api.example.com"

# 方法五：strace 跟踪系统调用（极端情况）
strace -p $(pgrep -f "nginx: worker" | head -1) -e trace=network 2>&1 | head -50
```

###### 追问：Nginx 遇到 "upstream timed out (110: Connection timed out)" 如何处理？

这是最常见的生产故障之一，排查路径：

```bash
# 1. 确认是哪个 upstream 超时，哪个接口
grep "upstream timed out" /var/log/nginx/error.log | awk '{print $NF}' | sort | uniq -c | sort -rn

# 2. 直接测试后端响应时间
time curl -o /dev/null http://后端IP:端口/具体路径

# 3. 检查是否是特定时间段（定时任务/GC/流量高峰）
grep "upstream timed out" /var/log/nginx/error.log | awk '{print $4}' | cut -c1-17 | uniq -c

# 4. 查看后端应用的线程池状态（是否线程耗尽导致排队）
# Spring Boot Actuator：/actuator/metrics/tomcat.threads.busy

# 5. 处理方案
# - 短期：适当增大 proxy_read_timeout
# - 中期：优化慢接口（SQL优化/缓存/异步化）
# - 长期：接口超时熔断（Sentinel/Hystrix），避免慢接口拖垮整个系统
```

核心原则：**超时本身不是问题，是慢接口/后端过载的结果**，要治根本。
