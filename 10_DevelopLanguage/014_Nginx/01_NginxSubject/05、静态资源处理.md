### 五、静态资源处理类

---

###### 1. Nginx 如何处理静态资源？

Nginx 是处理静态资源的**最佳选手**，核心在于它绕过了应用程序层，直接由操作系统内核处理文件读取和网络发送。

```nginx
server {
    listen 80;
    server_name static.example.com;
    
    root /var/www/static;    # 静态资源根目录
    index index.html;        # 默认首页
    
    location / {
        try_files $uri $uri/ =404;  # 尝试文件 → 目录 → 404
    }
    
    # 图片、CSS、JS 单独配置缓存和压缩
    location ~* \.(css|js|jpg|jpeg|png|gif|ico|svg|woff|woff2)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
        access_log off;     # 静态资源不记录访问日志，减少 I/O
    }
}
```

**Nginx 处理静态文件的优化路径：**
1. `sendfile on`：使用零拷贝技术，内核直接将文件传输到网络缓冲区
2. `tcp_nopush on`：配合 sendfile，将响应头和文件内容合并发送（减少 TCP 包数量）
3. `gzip on`：压缩文本资源，减少传输体积
4. `open_file_cache`：缓存文件描述符，避免频繁 open/stat 系统调用

###### 2. 如何配置 Nginx 的 Gzip 压缩？

```nginx
http {
    gzip on;                        # 开启 Gzip
    gzip_comp_level 6;              # 压缩级别 1-9，6 是性能与压缩率的平衡点
    gzip_min_length 1024;           # 小于 1KB 的响应不压缩（压缩反而变大）
    gzip_buffers 4 32k;             # 压缩缓冲区
    gzip_http_version 1.1;          # 仅对 HTTP/1.1 客户端压缩
    gzip_types 
        text/plain
        text/css
        text/javascript
        application/javascript
        application/json
        application/xml
        image/svg+xml
        font/woff
        font/woff2;                 # 压缩的 MIME 类型（text/html 默认开启）
    gzip_vary on;                   # 添加 Vary: Accept-Encoding 响应头
    gzip_disable "MSIE [1-6]\.";    # IE6 不压缩（老浏览器兼容）
    gzip_proxied any;               # 对代理请求也压缩
    
    # 预压缩静态文件（推荐，避免实时压缩 CPU 开销）
    # 配合 gunzip 模块，先用 gzip 命令压缩好 .gz 文件，Nginx 直接返回
    # gzip_static on;               # 需要编译时加入 --with-http_gzip_static_module
}
```

**压缩效果参考：**
- JavaScript/CSS：压缩率 60-80%
- JSON API 响应：压缩率 70-90%
- 图片（JPEG/PNG/WebP）：不需要 Gzip，已经是压缩格式，Gzip 几乎无效

###### 3. 如何在 Nginx 中设置浏览器缓存？

浏览器缓存通过 HTTP 响应头控制，让浏览器将资源缓存在本地：

```nginx
server {
    # 策略一：强缓存（Cache-Control + Expires）
    # 浏览器在有效期内不发请求直接使用本地缓存
    location ~* \.(css|js|jpg|png|gif|ico|woff2)$ {
        expires 30d;                                    # 等价于 Cache-Control: max-age=2592000
        add_header Cache-Control "public, max-age=2592000, immutable";
        # immutable：有效期内浏览器不验证（Chrome 支持）
    }
    
    # 策略二：协商缓存（ETag + Last-Modified）
    # 浏览器发请求询问资源是否更新，未更新返回 304
    location ~* \.(html)$ {
        add_header Cache-Control "no-cache";        # 每次都要协商（不要用 no-store 除非敏感数据）
        etag on;                                    # 开启 ETag（默认开启）
    }
    
    # 策略三：不缓存（API 接口、动态页面）
    location /api/ {
        add_header Cache-Control "no-store, no-cache, must-revalidate";
        add_header Pragma no-cache;
        proxy_pass http://backend;
    }
}
```

**现代前端最佳实践：**
- HTML 文件：`no-cache`（每次协商，保证更新）
- CSS/JS：文件名带 Hash（`app.abc123.js`）→ `immutable` 强缓存（内容变则文件名变）
- 图片：长期强缓存 `max-age=31536000`

###### 4. 如何配置 Nginx 的防盗链？

防盗链阻止其他网站直接引用本站的图片、视频等资源：

```nginx
server {
    location ~* \.(jpg|jpeg|png|gif|mp4|mp3|pdf)$ {
        # valid_referers 白名单
        valid_referers none blocked 
                        server_names 
                        *.example.com 
                        ~\.google\. 
                        ~\.baidu\.;
        # none：允许直接访问（无 Referer 头）
        # blocked：允许无 scheme 的 Referer（防火墙/代理删除的）
        # server_names：允许本站域名
        
        if ($invalid_referer) {
            # 方案一：返回 403 拒绝
            return 403;
            
            # 方案二：返回防盗链图片
            # rewrite ^ /images/theft.png break;
        }
    }
}
```

**注意事项：**
- `valid_referers none` 表示允许直接在浏览器地址栏输入访问（Referer 为空）
- 防盗链只是基础防护，专业盗链者可以伪造 Referer
- 更安全的方案：使用**签名 URL**（时间戳+密钥签名，一次性或限时有效）

###### 5. sendfile 指令的作用是什么？

`sendfile` 指令开启后，Nginx 使用操作系统的**零拷贝**技术传输文件，大幅提升静态资源服务性能。

**传统文件传输（4次拷贝）：**
```
磁盘 → 内核缓冲区 → 用户空间（Nginx进程）→ 内核Socket缓冲区 → 网卡
```

**sendfile 零拷贝（2次拷贝，减少用户态/内核态切换）：**
```
磁盘 → 内核缓冲区 → 内核Socket缓冲区 → 网卡
# 数据不经过用户空间，直接在内核内部完成传输
```

```nginx
http {
    sendfile on;           # 开启零拷贝（默认 off，强烈建议开启）
    sendfile_max_chunk 1m; # 每次 sendfile 调用最大传输量，防止阻塞 Worker
}
```

**何时不要开 sendfile：** 当使用 Gzip 实时压缩时，数据必须经过用户空间处理，`sendfile` 无效（但 `gzip_static` 预压缩文件可以使用 sendfile）。

###### 6. tcp_nopush 和 tcp_nodelay 的区别是什么？

这两个指令控制 TCP 包的发送策略，看似矛盾但可以配合使用：

**`tcp_nopush on`（需要 sendfile 开启）：**
- 等到积累了足够数据才发送 TCP 包（Nagle 算法的 Nginx 版本）
- **目标：减少 TCP 包数量**，将响应头和文件内容合并在一个 TCP 包发出
- 适合**大文件传输**，减少网络包的数量

**`tcp_nodelay on`：**
- 禁用 Nagle 算法，有数据立即发送，不等待
- **目标：减少延迟**，让每个数据包立即到达客户端
- 适合**实时性要求高的场景**（WebSocket、API 接口）

**两者同时开启的行为（Nginx 推荐配置）：**
```nginx
sendfile    on;
tcp_nopush  on;     # 先批量发送（合并响应头和文件数据）
tcp_nodelay on;     # 连接进入 keepalive 状态后立即发送剩余数据
```
两者并不冲突：`tcp_nopush` 在 sendfile 发送文件时生效，`tcp_nodelay` 在 keepalive 连接复用阶段生效。

###### 追问：Nginx 处理静态资源时，open_file_cache 有什么作用？如何配置？

`open_file_cache` 缓存已打开文件的**文件描述符和元数据**，避免每次请求都调用 `open()/stat()` 系统调用：

```nginx
http {
    open_file_cache max=10000 inactive=20s;  # 最多缓存10000个fd，20s未访问则淘汰
    open_file_cache_valid    30s;             # 每30s重新验证缓存的文件是否发生变化
    open_file_cache_min_uses 2;              # 20s内至少被访问2次才加入缓存
    open_file_cache_errors   on;             # 同时缓存文件不存在的错误（避免重复 stat）
}
```

**缓存的内容：**
- 文件描述符（file descriptor）
- 文件大小和最后修改时间（`stat` 结果）
- 目录查找结果
- 文件未找到或权限不足等错误信息

**适用场景：** 静态资源服务器文件数量多、访问频繁时效果显著，可减少数十倍的系统调用。频繁更新的文件需要设置合理的 `open_file_cache_valid` 避免缓存旧内容。
