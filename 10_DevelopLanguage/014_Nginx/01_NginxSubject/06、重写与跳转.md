### 六、重写与跳转类

---

###### 1. Nginx 的 rewrite 指令如何使用？

`rewrite` 指令用于修改请求 URI，可以在 `server`、`location`、`if` 块中使用：

```nginx
# 语法：rewrite regex replacement [flag];

server {
    # 基本重写：将旧路径重写到新路径
    rewrite ^/old/(.*)$ /new/$1 last;
    
    # 带参数的重写
    rewrite ^/article/(\d+)$ /article.php?id=$1 last;
    
    # 移动端重定向
    if ($http_user_agent ~* "(mobile|android|iphone)") {
        rewrite ^/(.*)$ https://m.example.com/$1 redirect;
    }
    
    # 多条 rewrite 规则
    rewrite ^/api/v1/(.*)$ /api/v2/$1 last;     # 版本升级
    rewrite ^/download/(.*)$ /files/$1 break;    # 文件下载路径映射
}
```

**`rewrite` 的处理流程：**
1. Nginx 对 URI 匹配正则
2. 命中后将 URI 替换为 `replacement`
3. 根据 `flag` 决定是继续处理还是跳转

---

###### 2. rewrite 的 flag 标记有哪些？分别是什么作用？

| Flag | 作用 |
|------|------|
| `last` | 重写 URI 后，**重新从头匹配所有 location**（类似重发内部请求）|
| `break` | 重写 URI 后，**停止处理当前 rewrite 规则集**，在当前 location 中继续处理 |
| `redirect` | 返回 **302 临时重定向**，客户端重新请求新 URL（浏览器地址栏改变）|
| `permanent` | 返回 **301 永久重定向**，浏览器会缓存，下次直接请求新 URL |

**实际对比示例：**
```nginx
location /a/ {
    rewrite ^/a/(.*)$ /b/$1 last;
    # last：Nginx 内部重新匹配 /b/...，如果有 location /b/ 会命中并继续处理
}

location /c/ {
    rewrite ^/c/(.*)$ /d/$1 break;
    # break：在当前 location 内处理 /d/...，不再匹配其他 location
    root /var/www;  # 直接在此 root 下找 /d/... 文件
}

# 301 vs 302：
rewrite ^/old$ /new redirect;   # 302：临时，不缓存，SEO不传递
rewrite ^/old$ /new permanent;  # 301：永久，浏览器缓存，SEO权重传递
```

**记忆技巧：** `last` = 重头来过；`break` = 就地处理；`redirect` = 告诉浏览器去那边；`permanent` = 永久告诉浏览器。

###### 3. 如何实现 HTTP 到 HTTPS 的重定向？

**方案一：return 301（推荐，简单高效）**
```nginx
server {
    listen 80;
    server_name www.example.com example.com;
    return 301 https://$host$request_uri;
    # $host 自动匹配 server_name，$request_uri 保留原始路径和参数
}

server {
    listen 443 ssl;
    server_name www.example.com example.com;
    # ... HTTPS 配置
}
```

**方案二：rewrite（不推荐，多一步正则匹配）**
```nginx
server {
    listen 80;
    server_name www.example.com;
    rewrite ^(.*)$ https://$host$1 permanent;
}
```

**HSTS 加强版（建议生产使用）：**
```nginx
server {
    listen 443 ssl;
    # 告知浏览器未来一年内全部用 HTTPS（浏览器级别强制，无法绕过）
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
}
```

**性能注意：** 301 跳转有额外 RTT，前端可通过 `preload` 提交到 HSTS Preload List，彻底消除首次 HTTP 请求。

###### 4. return 和 rewrite 的区别是什么？

| 维度 | `return` | `rewrite` |
|------|----------|-----------|
| 本质 | 直接返回 HTTP 状态码和响应 | 修改 URI 后继续处理 |
| 性能 | 更快（不需要正则匹配） | 稍慢（需要编译并执行正则） |
| 用途 | 重定向、返回错误码、返回固定响应 | URI 变换、条件重写 |
| 后续处理 | 立即结束，返回响应给客户端 | 根据 flag 决定是否继续处理 |

**`return` 的用法：**
```nginx
# 返回重定向
return 301 https://$host$request_uri;
return 302 https://other.example.com;

# 返回错误
return 403 "Access Denied";
return 404;

# 返回简单响应（无需后端）
location /health {
    return 200 "ok\n";
    add_header Content-Type text/plain;
}
```

**什么时候用 return，什么时候用 rewrite：**
- **简单跳转、固定返回** → `return`（性能更好，配置更清晰）
- **需要正则捕获组、复杂 URI 变换** → `rewrite`
- **对外跳转（浏览器地址栏变化）** → `return 301/302` 或 `rewrite ... redirect/permanent`
- **内部路径映射（后端无感知）** → `rewrite ... last/break`

###### 5. 如何在 Nginx 中配置域名跳转？

```nginx
# 场景一：裸域名跳转到 www（SEO 规范化）
server {
    listen 80;
    server_name example.com;
    return 301 https://www.example.com$request_uri;
}

# 场景二：www 跳转到裸域名
server {
    listen 443 ssl;
    server_name www.example.com;
    ssl_certificate     /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;
    return 301 https://example.com$request_uri;
}

# 场景三：多个旧域名统一跳转到新域名
server {
    listen 80;
    server_name old-site.com www.old-site.com legacy.com;
    return 301 https://new-site.com$request_uri;
}

# 场景四：按路径跳转（应用迁移）
server {
    listen 80;
    server_name api.example.com;
    
    location /v1/ {
        return 301 https://api.example.com/v2$request_uri;
    }
    
    location /v2/ {
        proxy_pass http://backend;
    }
}
```

###### 追问：Nginx if 指令有哪些坑？为什么说"if is evil"？

Nginx 的 `if` 指令在 `location` 块内有**诸多非直觉的行为**，官方文档明确警告慎用：

**坑1：if 块内只有部分指令生效**
```nginx
location /api/ {
    if ($request_method = POST) {
        proxy_pass http://post-backend;  # ✅ 可以用
        # proxy_set_header Host $host;   # ⚠️ 可能不生效
    }
    proxy_pass http://get-backend;       # 这行有时会"穿透"到 if 块内
}
```

**坑2：if 不能嵌套**
```nginx
if ($a) {
    if ($b) {  # ❌ 语法错误
    }
}
```

**坑3：多个 if 的行为反直觉**
```nginx
location / {
    if ($condition1) {
        add_header X-A 1;
    }
    if ($condition2) {
        add_header X-B 2;  # 这条会覆盖上面那条！
    }
}
```

**推荐替代方案：**
- 简单条件跳转 → `return` + `map`
- 复杂条件路由 → 拆分多个 `location`
- 高级逻辑 → Lua 脚本（OpenResty）
