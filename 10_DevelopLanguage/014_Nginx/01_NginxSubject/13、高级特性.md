### 十三、高级特性类

---

###### 1. Nginx 的异步非阻塞模型是如何工作的？

Nginx 的异步非阻塞模型由三个核心机制共同实现：

**① 非阻塞 I/O（Non-blocking I/O）：**
Nginx 将所有 socket 设置为非阻塞模式。当 `read()`/`write()` 系统调用数据未就绪时，**立即返回 `EAGAIN`** 而不是挂起进程，Worker 可以继续处理其他事件。

**② I/O 多路复用（epoll）：**
```
Worker 进程
    │
    ├─ epoll_wait()  ← 阻塞等待就绪事件（可监听数万个fd）
    │       │
    │  内核通知就绪fd列表
    │       │
    ├─ 遍历就绪事件，分别调用回调函数处理
    │       ├─ 处理连接A的请求头读取
    │       ├─ 处理连接B的响应发送
    │       └─ 处理连接C的上游响应接收
    │
    └─ 循环回到 epoll_wait()
```

**③ 事件驱动回调（Event-Driven Callbacks）：**
每个 I/O 操作完成时触发对应回调，Nginx 内部维护一个**状态机**，连接处理分解为多个事件和回调阶段（接收请求头 → 读取请求体 → 连接上游 → 接收响应 → 发送响应）。

**本质：** 单线程以时间片方式交错处理 N 个连接，CPU 始终在处理就绪的 I/O，永不空等。

###### 2. Nginx 如何处理高并发请求？

Nginx 高并发能力来自多层设计的协同：

**① 进程模型：** `worker_processes = CPU 核数`，每个 Worker 独立运行，避免锁竞争。

**② epoll 事件驱动：** 单个 Worker 可维护 10000+ 并发连接，内核通过 epoll 的红黑树 + 就绪链表管理，`epoll_wait` 只返回**真正就绪**的 fd，无需遍历所有连接。

**③ 内存管理：** Nginx 为每个请求分配内存池（`ngx_pool_t`），请求结束一次性释放，避免 malloc/free 碎片和开销。

**④ 零拷贝：** `sendfile` 直接从内核页缓存发送文件，不经用户态，减少 CPU 和内存开销。

**⑤ 连接复用：** upstream `keepalive` 复用到后端的 TCP 连接，减少握手开销。

**理论并发上限计算：**
```
最大并发 = worker_processes × worker_connections
每个连接消耗内存约 20KB（连接 + 缓冲区）
8核 × 10240连接 × 20KB ≈ 1.6GB 内存 → 支持约 8 万并发连接
```

###### 3. Nginx 的 epoll 模型是什么？

epoll（Event Poll）是 Linux 2.6+ 内核提供的 I/O 多路复用机制，是 Nginx 高并发的基础：

**select/poll 的问题（老方案）：**
- `select` 最多监听 1024 个 fd（FD_SETSIZE 限制）
- `poll` 无数量限制，但每次调用都需要**遍历所有 fd** 检查就绪状态，O(n) 复杂度
- 内核态/用户态数据拷贝开销大

**epoll 的解决方案：**

```c
// 1. 创建 epoll 实例（内核红黑树）
int epfd = epoll_create1(0);

// 2. 注册感兴趣的 fd 和事件
struct epoll_event ev = {.events = EPOLLIN | EPOLLET, .data.fd = sockfd};
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev);

// 3. 等待就绪事件（只返回已就绪的 fd，O(1)）
int n = epoll_wait(epfd, events, MAX_EVENTS, -1);
for (int i = 0; i < n; i++) {
    handle(events[i].data.fd);  // 只处理就绪的 fd
}
```

**epoll 的三个关键优势：**
1. 监听 fd 数量无限制（受内存限制）
2. 每次返回**只有就绪的 fd**，O(1) 复杂度（红黑树管理注册 fd，就绪链表存储就绪 fd）
3. 支持 `ET`（边缘触发）模式：只在状态变化时通知一次，减少唤醒次数

###### 4. Nginx 如何实现零拷贝（Zero Copy）？

Nginx 通过 `sendfile()` 系统调用实现零拷贝，原理是**让数据在内核内部流转，不经过用户空间**：

**传统文件发送（4次数据拷贝 + 4次上下文切换）：**
```
read():  磁盘 → 内核缓冲区 → 用户空间（Nginx进程内存）
write(): 用户空间 → 内核Socket缓冲区 → 网卡DMA
```

**sendfile 零拷贝（2次数据拷贝 + 2次上下文切换）：**
```
sendfile(): 磁盘 → 内核Page Cache → Socket缓冲区 → 网卡DMA
// 数据全程在内核内部流转，CPU全程不参与数据搬运
```

**Linux 2.4+ 的 scatter/gather sendfile（真正零拷贝）：**
```
磁盘 → 内核Page Cache
              ↓（只传递文件描述符和偏移量，不拷贝数据）
         Socket缓冲区 → 网卡DMA直接读Page Cache
```

```nginx
http {
    sendfile on;
    sendfile_max_chunk 512k;   # 防止单次 sendfile 阻塞 Worker 太久
    tcp_nopush on;             # 配合 sendfile，合并数据包
}
```

**限制：** Gzip 实时压缩需要数据经过用户空间，无法使用 sendfile；使用 `gzip_static` 预压缩的 .gz 文件可以用 sendfile。

###### 5. Nginx 的内存池机制是怎样的？

Nginx 的内存池（`ngx_pool_t`）是其高性能的重要保障，为每个**连接/请求**分配独立内存池：

```c
// 内存池结构（简化）
typedef struct {
    ngx_pool_data_t  d;        // 当前内存块的数据区
    size_t           max;      // 小块内存的最大值
    ngx_pool_t      *current;  // 当前可用内存块
    ngx_chain_t     *chain;    // 链表
    ngx_pool_large_t *large;   // 大块内存链表（> max 的分配）
    ngx_pool_cleanup_t *cleanup;  // 清理回调链表
} ngx_pool_t;
```

**内存池的工作模式：**
1. **请求开始**：为该请求创建内存池（初始 4KB 左右）
2. **请求处理中**：所有内存分配（请求解析、Header 存储、缓冲区等）都从池中分配，**O(1) 分配，无锁**
3. **小块内存**（< pool->max）：在预分配的内存块中顺序分配，没有碎片
4. **大块内存**（>= pool->max）：调用 `malloc`，挂在 large 链表上
5. **请求结束**：**一次性销毁整个内存池**，所有内存块统一释放，无需逐个 free

**优势：**
- 避免频繁 `malloc`/`free` 的系统调用开销
- 消除内存碎片（池内顺序分配）
- 连接/请求结束时一次性回收，简化内存管理逻辑

###### 6. Nginx 如何处理惊群问题？

**惊群问题（Thundering Herd）：** 多个 Worker 进程同时监听同一端口，一个新连接到来时，内核会唤醒**所有**等待的 Worker，但只有一个能成功 `accept()`，其他白白被唤醒造成性能浪费。

**Nginx 的解决方案演进：**

**早期方案：accept_mutex（互斥锁）**
```nginx
events {
    accept_mutex on;       # 开启 accept 互斥锁（老版本默认 on）
    accept_mutex_delay 500ms;  # 竞争锁失败的等待时间
}
```
同一时刻只有持有锁的 Worker 监听新连接，避免惊群。**缺点：** 引入锁竞争，`500ms` 延迟可能影响连接接受速度。

**Linux 3.9+ 新方案：SO_REUSEPORT**
```nginx
server {
    listen 80 reuseport;    # 每个 Worker 独立绑定同一端口
    # 内核在 accept 队列级别就做了负载均衡，每次只唤醒一个 Worker
}
```
`SO_REUSEPORT` 让内核为每个 Worker 维护独立的 accept 队列，连接到来时内核直接选择一个队列，**真正解决惊群**，且性能更好。Nginx 1.9.1+ 支持。

###### 7. Nginx 的连接池是如何工作的？

Nginx 与上游后端之间的**keepalive 连接池**（不是连接池框架意义上的连接池，而是空闲连接缓存）：

```nginx
upstream backend {
    server 192.168.1.10:8080;
    keepalive 32;              # 每个 Worker 对该 upstream 保持 32 个空闲长连接
    keepalive_requests 1000;   # 每个连接最多处理 1000 个请求后关闭重建
    keepalive_timeout 60s;     # 空闲连接超过 60s 关闭
}
```

**工作流程：**
1. 请求到来时，先从**空闲连接池**取一个已建立的 TCP 连接
2. 若池中有空闲连接，直接发送请求，**跳过 TCP 三次握手（节省约 100ms）**
3. 若池为空，新建 TCP 连接处理请求
4. 请求完成后，连接放回池中（而非关闭），等待下一个请求使用
5. 空闲超过 `keepalive_timeout` 后关闭，释放资源

**配置要求（缺一不可）：**
```nginx
proxy_http_version 1.1;          # 必须 HTTP/1.1 才支持 keepalive
proxy_set_header Connection "";  # 必须清除 Connection 头，否则默认发 close
```

###### 追问：epoll 的 LT 模式和 ET 模式有什么区别？Nginx 用的是哪种？

| 模式 | 全称 | 触发时机 | 特点 |
|------|------|---------|------|
| LT | Level Trigger（水平触发） | 只要 fd 处于就绪状态（缓冲区有数据），每次 epoll_wait 都会返回 | 编程简单，数据不会遗漏，多次通知 |
| ET | Edge Trigger（边缘触发） | 仅在 fd **状态发生变化**时通知一次（从不可读变为可读） | 高效，但必须一次性读完所有数据，否则不会再次通知 |

**Nginx 使用 ET 模式 + 非阻塞 I/O：**
- ET 模式减少 epoll_wait 的返回次数，减少系统调用开销
- 每次 I/O 事件触发时，Nginx 用 while 循环持续读/写直到 `EAGAIN`（数据读完），保证数据完整性
- 配合非阻塞 I/O，`EAGAIN` 立即返回不阻塞，而不是挂起 Worker 进程

这就是为什么 Nginx 在处理大量短连接时，比基于 LT 模式的实现有更低的系统调用开销。
