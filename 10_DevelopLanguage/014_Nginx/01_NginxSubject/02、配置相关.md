###### 1. Nginx 的配置文件结构是怎样的？

Nginx 配置文件（默认 `/etc/nginx/nginx.conf`）采用**层级块结构**，从外到内依次为：

```nginx
# 全局块：影响 Nginx 整体运行
worker_processes  auto;
error_log  /var/log/nginx/error.log warn;
pid        /var/run/nginx.pid;

events {
    # 事件块：网络连接相关配置
    worker_connections  1024;
    use epoll;
    multi_accept on;
}

http {
    # HTTP 块：所有 HTTP 服务配置
    include       mime.types;
    default_type  application/octet-stream;
    
    upstream backend {
        # upstream 块：定义后端服务器组
        server 127.0.0.1:8080;
    }
    
    server {
        # server 块：虚拟主机配置
        listen       80;
        server_name  example.com;
        
        location / {
            # location 块：具体 URI 匹配规则
            proxy_pass http://backend;
        }
    }
}
```

**层级说明：**
- `全局块` → `events块` → `http块` → `server块` → `location块`
- 外层配置可被内层覆盖（就近原则）
- 可用 `include` 拆分配置：`include /etc/nginx/conf.d/*.conf;`

###### 2. 如何在 Nginx 中配置虚拟主机？

虚拟主机（Virtual Host）允许一台 Nginx 服务多个域名/网站，分三种方式：

**① 基于域名（最常用）：**
```nginx
server {
    listen 80;
    server_name www.site-a.com site-a.com;
    root /var/www/site-a;
    index index.html;
}

server {
    listen 80;
    server_name www.site-b.com;
    root /var/www/site-b;
    index index.html;
}
```

**② 基于端口：**
```nginx
server {
    listen 8080;
    server_name localhost;
    root /var/www/app1;
}
server {
    listen 9090;
    server_name localhost;
    root /var/www/app2;
}
```

**③ 基于 IP：**
```nginx
server {
    listen 192.168.1.10:80;
    server_name _;
    root /var/www/site1;
}
```

Nginx 匹配 server 的顺序：精确域名 → 通配符（`*.example.com`）→ 正则 → `default_server`。

###### 3. location 指令的匹配规则有哪些？优先级是什么？

`location` 支持 5 种匹配修饰符，**优先级从高到低**：

| 优先级 | 语法 | 说明 |
|--------|------|------|
| 1（最高） | `location = /uri` | 精确匹配，命中立即停止 |
| 2 | `location ^~ /prefix` | 前缀匹配，命中后不再检查正则 |
| 3 | `location ~ /regex` | 正则匹配（区分大小写），按配置顺序匹配 |
| 3 | `location ~* /regex` | 正则匹配（不区分大小写） |
| 4（最低） | `location /prefix` | 普通前缀匹配，选最长前缀后继续检查正则 |

**示例：**
```nginx
location = /api/health { return 200 "ok"; }    # 精确匹配
location ^~ /static/   { root /var/www; }      # 前缀，阻断正则
location ~* \.(jpg|png|gif)$ { expires 30d; }  # 正则，图片缓存
location /api/         { proxy_pass http://backend; }  # 普通前缀
location /             { return 404; }          # 兜底
```

**完整匹配流程：**
1. 先查 `=` 精确匹配，命中立即用
2. 扫描所有前缀匹配，记录最长命中的 location
3. 若最长命中是 `^~`，直接使用，不检查正则
4. 按配置文件顺序检查所有正则，第一个命中的正则 location 生效
5. 若无正则命中，使用步骤2记录的最长前缀 location

###### 4. 如何配置 Nginx 的日志格式？

```nginx
http {
    # 自定义日志格式
    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer" '
                      '"$http_user_agent" "$http_x_forwarded_for" '
                      '$request_time $upstream_response_time';
    
    # JSON 格式（推荐，方便 ELK 收集）
    log_format  json_log  '{"time":"$time_iso8601",'
                           '"ip":"$remote_addr",'
                           '"method":"$request_method",'
                           '"uri":"$request_uri",'
                           '"status":$status,'
                           '"rt":"$request_time",'
                           '"upstream_rt":"$upstream_response_time",'
                           '"body_size":$body_bytes_sent}';
    
    access_log  /var/log/nginx/access.log  main;     # 使用 main 格式
    error_log   /var/log/nginx/error.log   warn;     # 错误日志级别
}

server {
    access_log  /var/log/nginx/api.access.log  json_log;  # server级别单独配置
}
```

常用内置变量：`$remote_addr`（客户端IP）、`$request_time`（请求总耗时）、`$upstream_response_time`（后端响应时间）、`$status`（HTTP状态码）。

###### 5. worker_processes 和 worker_connections 参数如何设置？

**`worker_processes`**（Worker 进程数）：
```nginx
worker_processes auto;  # 推荐：自动等于 CPU 核数
# worker_processes 4;   # 或手动指定
```
- 推荐设为 `auto` 或 CPU 核数
- I/O 密集型（如反向代理）可设为 CPU*2，但收益边际递减
- 过多 Worker 会增加进程切换和内存开销

**`worker_connections`**（每个 Worker 最大连接数）：
```nginx
events {
    worker_connections 10240;
    use epoll;
    multi_accept on;  # 一次接受尽可能多的连接
}
```
- 默认 1024，生产环境建议 10240～65535
- **理论最大并发连接 = worker_processes × worker_connections**
- 受系统文件描述符限制，需同步调整：
```nginx
worker_rlimit_nofile 65535;  # nginx 配置
# /etc/security/limits.conf
# nginx  soft  nofile  65535
# nginx  hard  nofile  65535
```

###### 6. 如何配置 Nginx 的超时时间？

```nginx
http {
    # 客户端请求相关超时
    client_header_timeout  15s;    # 读取请求头超时（默认60s）
    client_body_timeout    60s;    # 读取请求体超时（默认60s）
    send_timeout           60s;    # 发送响应给客户端超时（默认60s）
    keepalive_timeout      75s;    # Keep-Alive 连接保持时间（默认75s）
    
    # 代理/上游相关超时
    proxy_connect_timeout  10s;    # 连接上游超时（不能太大）
    proxy_send_timeout     60s;    # 向上游发送请求超时
    proxy_read_timeout     60s;    # 从上游读取响应超时（最常调的）
    
    # upstream keepalive
    upstream backend {
        server 127.0.0.1:8080;
        keepalive 32;              # 保持 32 个长连接到后端
    }
}
```

**实践建议：**
- `proxy_connect_timeout` 不要超过 10s，否则会长时间占用 Worker
- `proxy_read_timeout` 根据业务接口最长耗时设置，避免提前超时断开
- 接口超时优先在应用层控制，Nginx 超时作为兜底保障

###### 7. 如何在 Nginx 中配置 HTTPS？

```nginx
server {
    listen 443 ssl http2;
    server_name www.example.com;
    
    # 证书文件（推荐 Let's Encrypt 或商业 CA 颁发）
    ssl_certificate     /etc/nginx/ssl/fullchain.pem;   # 包含证书链
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;
    
    # 协议与加密套件（Mozilla 推荐配置）
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;  # TLS1.3 时 off 更安全
    
    # 性能优化
    ssl_session_cache   shared:SSL:10m;  # 10MB 会话缓存
    ssl_session_timeout 1d;
    ssl_session_tickets off;             # 关闭 session tickets（提升前向安全）
    
    # HSTS（强制 HTTPS，一年有效期）
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    
    location / {
        proxy_pass http://backend;
    }
}

# HTTP 强制跳转 HTTPS
server {
    listen 80;
    server_name www.example.com;
    return 301 https://$host$request_uri;
}
```

###### 8. 如何配置 Nginx 的访问控制（allow/deny）？

`allow/deny` 指令基于 IP 地址进行访问控制，**从上到下匹配，第一条命中规则生效**：

```nginx
# 全局配置：只允许内网访问
http {
    # 白名单模式：允许特定 IP，拒绝其他
    allow 192.168.1.0/24;
    allow 10.0.0.0/8;
    allow 127.0.0.1;
    deny  all;
}

server {
    # 保护敏感接口（如 /admin）
    location /admin {
        allow 192.168.1.100;
        allow 10.10.10.0/24;
        deny  all;
        proxy_pass http://backend;
    }
    
    # 保护 Nginx 状态页
    location /nginx_status {
        stub_status;
        allow 127.0.0.1;
        deny  all;
    }
}
```

更复杂的访问控制（基于 Header、Cookie、JWT 等）需要配合 `ngx_http_auth_request_module` 或 Lua 脚本实现。

###### 追问：Nginx 的 server_name 是如何匹配的？精确匹配和通配符的优先级？

Nginx 处理一个请求时，先根据 `listen` 端口找到候选 server 块，再在候选中按以下优先级匹配 `server_name`：

1. **精确名称**：`server_name www.example.com;` — 最高优先级
2. **以 `*` 开头的通配符**：`server_name *.example.com;` — 匹配所有子域名
3. **以 `*` 结尾的通配符**：`server_name www.example.*;` — 匹配所有 TLD
4. **正则表达式**：`server_name ~^www\d+\.example\.com$;` — 按配置文件顺序匹配，最后考虑
5. **`default_server`**：所有规则都不匹配时的兜底 server

如果没有任何 server 匹配，Nginx 会用该端口的**第一个 server 块**处理请求（所以建议显式设置 `default_server`）。
