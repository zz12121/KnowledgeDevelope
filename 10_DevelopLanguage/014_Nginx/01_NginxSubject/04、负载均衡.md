### 四、负载均衡类

---

###### 1. Nginx 支持哪些负载均衡策略？

Nginx 内置 5 种负载均衡策略：

| 策略 | 指令 | 特点 | 适用场景 |
|------|------|------|---------|
| 轮询 | 默认 | 依次分发，均等权重 | 无状态应用，服务器性能相近 |
| 加权轮询 | `weight=N` | 按权重分配请求比例 | 服务器性能差异明显 |
| IP 哈希 | `ip_hash` | 同一 IP 固定到同一后端 | 需要会话保持（Session） |
| 最少连接 | `least_conn` | 优先分配给活跃连接数最少的节点 | 请求处理时间差异大 |
| 一致性哈希 | `hash $key consistent` | 基于自定义 key 的一致性哈希 | 缓存服务器（减少缓存失效） |

Nginx Plus（商业版）额外支持：
- **Random**：随机选择（可指定选择两台中最小连接数的）
- **Least Time**：基于响应时间选择（生产环境更智能）

---

###### 2. 轮询、加权轮询、ip_hash、least_conn 的区别是什么？

**轮询（Round Robin）：**
```nginx
upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    server 192.168.1.12:8080;
    # 请求依次分发给3台服务器：1→2→3→1→2→3→...
}
```
优点：实现简单，分配均匀。缺点：不考虑后端实际负载，可能把慢请求堆积到某台服务器。

**加权轮询（Weighted Round Robin）：**
```nginx
upstream backend {
    server 192.168.1.10:8080 weight=3;  # 处理3/5的请求
    server 192.168.1.11:8080 weight=2;  # 处理2/5的请求
}
```
按权重比例分配，适合服务器配置不均的场景（如新旧机器混用）。

**IP 哈希（ip_hash）：**
```nginx
upstream backend {
    ip_hash;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    # 同一客户端 IP 始终访问同一后端（基于 IP 前3段哈希）
}
```
缺点：如果大量用户在同一 NAT 后面，会导致负载不均。某节点下线时哈希重分布，Session 失效。

**最少连接（least_conn）：**
```nginx
upstream backend {
    least_conn;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    # 每次将新请求分给当前活跃连接数最少的节点
}
```
优点：动态感知后端负载，避免慢请求堆积。缺点：需要跟踪每台服务器的连接数，有少量开销。

###### 3. 如何实现 Nginx 的会话保持（session sticky）？

**方案一：ip_hash（简单但有缺陷）**
```nginx
upstream backend {
    ip_hash;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```
问题：NAT 场景下大量用户映射到同一 IP，负载不均。

**方案二：基于 Cookie 的 sticky（Nginx Plus 原生支持，开源版需模块）**
```nginx
# Nginx Plus 语法
upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    sticky cookie srv_id expires=1h domain=.example.com path=/;
}
```

**方案三：hash $cookie_session（开源版替代方案）**
```nginx
upstream backend {
    hash $cookie_JSESSIONID consistent;  # 基于 Session Cookie 哈希路由
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```

**方案四（推荐）：Session 集中存储，彻底解决问题**
将 Session 存入 Redis，后端变成无状态服务，无需 Nginx 层会话保持。这才是现代微服务架构的正确做法。

###### 4. 如何配置 Nginx 的健康检查？

**被动健康检查（内置，默认启用）：**
```nginx
upstream backend {
    server 192.168.1.10:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:8080 max_fails=3 fail_timeout=30s;
    # max_fails=3：30秒内连续失败3次，则标记该节点 30 秒不可用
    # fail_timeout=30s：不可用持续时间，过后再尝试
}
```
被动检查的缺点：需要真实请求失败才能发现节点故障，会有少量请求失败。

**主动健康检查（需要 nginx_upstream_check_module 或 Nginx Plus）：**

使用 `nginx_upstream_check_module` 第三方模块：
```nginx
upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    
    # 每隔 3 秒发送 HTTP 探测，超时 1s，连续失败 3 次下线，成功 2 次上线
    check interval=3000 rise=2 fall=3 timeout=1000 type=http;
    check_http_send "HEAD /health HTTP/1.0\r\n\r\n";
    check_http_expect_alive http_2xx;
}
```

**Nginx Plus 原生主动检查：**
```nginx
upstream backend {
    server 192.168.1.10:8080;
    health_check interval=5s fails=3 passes=2 uri=/health match=status_ok;
}
match status_ok {
    status 200;
    body ~ "ok";
}
```

###### 5. max_fails 和 fail_timeout 参数的作用是什么？

这两个参数共同控制 Nginx 对**失败后端节点的被动隔离**：

```nginx
upstream backend {
    server 192.168.1.10:8080 max_fails=3 fail_timeout=30s;
}
```

- **`max_fails=3`**：在 `fail_timeout` 时间窗口内，与该服务器通信失败次数达到 3 次，则临时将其标记为不可用
- **`fail_timeout=30s`**：
  - 计数失败次数的时间窗口（3次失败需在30秒内发生）
  - 节点被标记不可用后，等待的恢复时间（30秒后再次探测）

**什么算"失败"：** 连接超时、代理错误（由 `proxy_next_upstream` 定义）。

**默认值：** `max_fails=1`，`fail_timeout=10s`，生产建议适当放宽避免误判：`max_fails=3 fail_timeout=30s`。

**一个容易踩的坑：** 如果 upstream 只有 1 台服务器，`max_fails` 无效，Nginx 不会将唯一节点标记为不可用（否则所有请求都没法处理）。

###### 6. backup 和 down 标记在负载均衡中的作用是什么？

```nginx
upstream backend {
    server 192.168.1.10:8080;               # 正常节点
    server 192.168.1.11:8080;               # 正常节点
    server 192.168.1.12:8080 backup;        # 备用节点
    server 192.168.1.13:8080 down;          # 永久下线
}
```

**`backup`（热备）：**
- 正常情况下不参与负载均衡，流量全走其他节点
- 当所有正常节点都不可用时，backup 节点自动顶上接收流量
- 常用于**灰度发布**和**容灾备份**

**`down`（下线）：**
- 永久标记该节点为不可用，Nginx 不会向其发送任何请求
- 常用于**手动下线节点**做维护（不删配置，维护完删掉 `down` 标记即可）
- 与 `max_fails` 达到上限的临时不可用不同，`down` 是永久的，直到手动删除并 reload

**运维场景：**
```bash
# 下线某节点维护（修改配置后热重载，无需重启）
# 1. 修改配置添加 down 标记
# 2. nginx -s reload
# 3. 等待现有连接处理完毕（graceful）
# 4. 维护完成后删除 down 标记，再次 reload
```

###### 追问：Nginx 开源版和商业版（Nginx Plus）在负载均衡能力上的核心差距是什么？

| 功能 | Nginx 开源版 | Nginx Plus |
|------|------------|-----------|
| 被动健康检查 | ✅ 支持 | ✅ 支持 |
| 主动健康检查 | ❌ 需第三方模块 | ✅ 原生支持 |
| 会话保持 Cookie Sticky | ❌ 需第三方模块 | ✅ 原生支持 |
| 负载均衡算法 | 5种 | 5种 + Random + Least Time |
| 动态上游（不 reload 增删节点） | ❌ | ✅ API 动态配置 |
| 实时监控 Dashboard | ❌ 仅 stub_status | ✅ 完整 Dashboard + API |
| DNS 动态解析 upstream | ❌ | ✅ |

开源方案可通过 **OpenResty（Nginx + LuaJIT）+ lua-resty-upstream-healthcheck** 实现主动健康检查，是国内互联网公司的常见选择。
