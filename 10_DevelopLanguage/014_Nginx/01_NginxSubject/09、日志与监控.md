### 九、日志与监控类

---

###### 1. Nginx 的访问日志和错误日志分别记录什么？

**访问日志（access_log）：** 记录每一个成功处理的客户端请求。

默认格式的每行示例：
```
192.168.1.1 - - [15/Mar/2026:10:30:00 +0800] "GET /api/user HTTP/1.1" 200 1234 "https://example.com" "Mozilla/5.0..." "10.0.0.1"
```

字段含义：`客户端IP` - `认证用户` - `时间` `"请求行"` `状态码` `响应大小` `"Referer"` `"User-Agent"` `"X-Forwarded-For"`

**错误日志（error_log）：** 记录 Nginx 运行过程中的错误和警告。

日志级别从低到高：`debug` → `info` → `notice` → `warn` → `error` → `crit` → `alert` → `emerg`

```nginx
error_log /var/log/nginx/error.log warn;   # 记录 warn 及以上级别
```

错误日志常见内容：
- 配置语法错误
- 上游连接失败（`upstream timed out`）
- 文件权限问题（`Permission denied`）
- Worker 进程崩溃

###### 2. 如何自定义 Nginx 的日志格式？

```nginx
http {
    # 标准格式（带请求耗时和上游耗时）
    log_format detailed '$remote_addr - $remote_user [$time_local] '
                        '"$request" $status $body_bytes_sent '
                        '"$http_referer" "$http_user_agent" '
                        'rt=$request_time urt=$upstream_response_time '
                        'ua=$upstream_addr us=$upstream_status';
    
    # JSON 格式（推荐 ELK/Loki 收集时使用）
    log_format json escape=json
        '{"timestamp":"$time_iso8601",'
        '"client_ip":"$remote_addr",'
        '"method":"$request_method",'
        '"uri":"$request_uri",'
        '"status":$status,'
        '"bytes_sent":$body_bytes_sent,'
        '"request_time":$request_time,'
        '"upstream_time":"$upstream_response_time",'
        '"upstream_addr":"$upstream_addr",'
        '"user_agent":"$http_user_agent",'
        '"x_forwarded_for":"$http_x_forwarded_for",'
        '"host":"$host"}';
    
    access_log /var/log/nginx/access.log detailed;
    
    server {
        # 关闭某些路径的日志（减少 I/O，如健康检查）
        location /health {
            access_log off;
            return 200 "ok";
        }
        
        # API 接口使用 JSON 格式（方便 ELK 解析）
        location /api/ {
            access_log /var/log/nginx/api.log json;
            proxy_pass http://backend;
        }
    }
}
```

###### 3. 如何实现 Nginx 日志的切割？

**方案一：logrotate（最常用，系统自带）**

```bash
# /etc/logrotate.d/nginx
/var/log/nginx/*.log {
    daily               # 每天切割
    missingok           # 文件不存在时不报错
    rotate 52           # 保留 52 份（约一年）
    compress            # 压缩旧日志（.gz）
    delaycompress       # 延迟压缩（下次切割时才压缩上次的日志）
    notifempty          # 空文件不切割
    create 640 nginx adm  # 新建日志文件的权限和所有者
    sharedscripts       # 多个文件匹配时脚本只执行一次
    postrotate
        # 发送 USR1 信号给 Nginx，重新打开日志文件（不中断服务）
        if [ -f /var/run/nginx.pid ]; then
            kill -USR1 `cat /var/run/nginx.pid`
        fi
    endscript
}
```

**方案二：Nginx 原生定时切割**

Nginx 本身不支持按时间自动切割，但可以配合 cron 定期发送 `USR1` 信号：
```bash
# crontab -e
0 0 * * * /bin/mv /var/log/nginx/access.log /var/log/nginx/access.$(date +%F).log && kill -USR1 $(cat /var/run/nginx.pid)
```

**`USR1` 信号的作用：** 告诉 Nginx Master 进程重新打开所有日志文件，Worker 进程继续写入新文件，实现无损切割。

###### 4. stub_status 模块如何使用？能获取哪些信息？

`stub_status` 是 Nginx 内置的基础监控模块，暴露实时连接统计：

```nginx
server {
    listen 80;
    server_name localhost;
    
    location /nginx_status {
        stub_status;
        allow 127.0.0.1;
        allow 10.0.0.0/8;
        deny  all;
        access_log off;
    }
}
```

访问 `http://localhost/nginx_status` 输出示例：
```
Active connections: 291        ← 当前活跃连接数（重要指标）
server accepts handled requests
 16630948 16630948 31070465   ← 总接受连接数 / 已处理连接数 / 总请求数
Reading: 6 Writing: 179 Waiting: 106  ← 读请求中/写响应中/keepalive等待中
```

**指标解读：**
- `Active connections`：当前同时连接数，与 `worker_connections` 上限对比，判断是否接近瓶颈
- `accepts == handled`：若不等则说明有连接被丢弃（FD 耗尽或 backlog 满）
- `Waiting`：keepalive 空闲连接，过高说明 `keepalive_timeout` 设置过大
- `Reading`：正在读取请求，过高说明客户端发送速度慢
- `Writing`：正在发送响应，过高说明客户端下载速度慢或后端响应慢

###### 5. 如何使用第三方工具监控 Nginx？

**方案一：Prometheus + nginx-prometheus-exporter（推荐生产）**

```bash
# 安装 exporter，读取 stub_status 并暴露 Prometheus 指标
./nginx-prometheus-exporter -nginx.scrape-uri=http://localhost/nginx_status

# Prometheus 采集配置
scrape_configs:
  - job_name: 'nginx'
    static_configs:
      - targets: ['localhost:9113']
```

Grafana 可直接导入 Nginx 官方 Dashboard（ID: 12708）。

**方案二：ELK 日志分析**

```bash
# Filebeat 收集 Nginx 日志
filebeat.inputs:
  - type: filestream
    paths: ["/var/log/nginx/*.log"]
    parsers:
      - ndjson:
          target: ""

output.elasticsearch:
  hosts: ["elasticsearch:9200"]
  index: "nginx-%{+YYYY.MM.dd}"
```

Kibana 可用于分析：TOP URL、错误率、响应时间分布、地理分布。

**方案三：GoAccess（轻量实时分析）**

```bash
# 实时分析访问日志（终端 Dashboard）
goaccess /var/log/nginx/access.log -c --log-format=COMBINED
# 生成 HTML 报告
goaccess /var/log/nginx/access.log -o /var/www/report.html --log-format=COMBINED --real-time-html
```

###### 追问：如何通过 Nginx 日志快速定位慢接口和高错误率接口？

**方法一：通过 `awk` 分析日志**

```bash
# 找出平均响应时间最长的 TOP 10 接口
awk '{print $7, $NF}' /var/log/nginx/access.log | \
  awk '{url=$1; time=$2; sum[url]+=time; count[url]++} END {for(url in sum) print sum[url]/count[url], url}' | \
  sort -rn | head -10

# 统计各状态码的请求数量
awk '{print $9}' /var/log/nginx/access.log | sort | uniq -c | sort -rn

# 找出 5xx 错误最多的接口
awk '$9 ~ /^5/ {print $7}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -20
```

**方法二：开启 Nginx 慢查询日志（在 log_format 中记录请求时间后用 awk 过滤）**

```bash
# 找出响应时间超过 2 秒的请求
awk '$NF > 2 {print $0}' /var/log/nginx/access.log | tail -100
```

**方法三：Kibana / Grafana 可视化**

在 ELK 中对 `request_time` 字段做 Percentile 聚合（P99/P95/P50），比平均值更能反映真实用户体验。
