# 三、架构与原理

> 双链：[[38_ElasticSearchKnowledge/03_分布式架构与写查流程]]

---

###### 1. 什么是分片（Shard）和副本（Replica）？它们的作用是什么？

**分片（Shard）**：
- ES 把一个索引的数据水平拆分成多个**主分片（Primary Shard）**，每个主分片是一个独立的 Lucene 索引
- 默认 1 个主分片（7.x 之前默认 5 个），创建索引后主分片数**不可修改**（需要 Reindex）
- 作用：**水平扩展**，数据分布在多个节点，实现并行读写

**副本（Replica）**：
- 每个主分片可以有若干个**副本分片（Replica Shard）**，副本是主分片的完整拷贝
- 默认 1 个副本，副本数可以随时修改
- 作用双重：**高可用**（主分片挂了副本提升为主）+ **提升读性能**（搜索请求可以打到副本）

**核心约束**：
- 主分片和其副本**不会分配在同一节点**上，否则一台机器挂了就数据全丢了
- 单节点时副本无法分配，集群会是黄色（Yellow）状态（主分片都在，但副本未分配）

```
索引（3主分片 × 1副本）在3节点上的分布：
Node-1: P0, R1
Node-2: P1, R2
Node-3: P2, R0
```

###### 2. 什么是文档路由？路由算法是怎样的？

当你写入或查询一个文档时，ES 需要知道这个文档在哪个分片上，这就是路由（Routing）。

**默认路由算法：**
```
分片编号 = hash(routing) % 主分片数
```
其中 `routing` 默认是文档的 `_id`。

**这也是为什么主分片数不可改变**：如果主分片数变了，`hash(_id) % 新分片数` 得到的分片编号就变了，再也找不到原来的文档了。（这也是为什么 Reindex 需要重新写一份数据）

**自定义路由**：
```bash
# 写入时指定 routing
PUT /logs/_doc/1?routing=user_123
{ "user_id": "user_123", "content": "..." }

# 查询时必须带相同 routing，否则会查所有分片
GET /logs/_doc/1?routing=user_123
```

自定义路由可以让同一用户的数据都落在同一分片，聚合查询时只需查一个分片，性能极好。但要小心**热点分片问题**：如果某个 routing 值数据量极大，那个分片会成为性能瓶颈。

###### 3. Elasticsearch 中有哪些节点类型？各自的作用是什么？

ES 7.x 及以上支持以下角色，一个节点可以同时扮演多个角色：

| 角色 | 配置参数 | 作用 |
|------|---------|------|
| **主节点（Master）** | `node.master: true` | 管理集群元数据（索引创建/删除/分片分配），轻量级，不存数据 |
| **候选主节点（Master-eligible）** | 同上，但未被选中 | 参与选举，随时准备接替当前主节点 |
| **数据节点（Data）** | `node.data: true` | 存储分片数据，处理增删改查和聚合，CPU/内存/磁盘消耗大 |
| **协调节点（Coordinating）** | 三个都设为 false | 接收请求，分发到相关分片，汇总结果返回客户端，无角色节点 |
| **摄取节点（Ingest）** | `node.ingest: true` | 执行 Ingest Pipeline（数据预处理），写入前对文档做转换 |
| **冷热节点（Hot/Warm/Cold）** | 自定义属性 | 分层存储：热节点用 SSD 存最新数据，冷节点用 HDD 存历史数据 |
| **机器学习节点（ML）** | `node.ml: true` | 专门运行 ML 作业（X-Pack 功能） |

**生产建议**：
- 专用主节点（3 台）+ 数据节点（N 台）+ 协调节点（可选），职责分离
- 协调节点部署在负载均衡后面，分担数据节点的协调压力

###### 4. 什么是协调节点？它在查询中起什么作用？

协调节点（Coordinating Node）是负责**接收客户端请求、分发任务、汇总结果**的节点，本身不存储数据。

任何节点都可以充当协调节点（接收请求时就是协调节点），但生产环境推荐专门设置无角色节点做协调。

**在搜索中的工作流程：**
```
1. 客户端请求打到协调节点
2. 协调节点解析请求，确定需要查哪些分片
3. 广播查询请求给所有相关分片（Query 阶段）
4. 各分片返回 [文档ID + 排序值] 的轻量结果
5. 协调节点做全局排序，取 Top N
6. 向对应分片发 Multi GET，拉取完整文档（Fetch 阶段）
7. 合并最终结果返回给客户端
```

**在写入中的工作流程：**
```
1. 协调节点根据路由算法确定目标主分片
2. 将请求转发给持有该主分片的数据节点
3. 主分片写入成功后，同步给副本分片
4. 所有副本确认后（根据 wait_for_active_shards 配置），返回客户端成功
```

###### 5. Elasticsearch 的数据写入流程是怎样的？

ES 写入流程分两个层次：**分布式层**（分片路由）和 **Lucene 层**（数据持久化）。

**分布式层写入流程：**
```
1. 协调节点：路由计算 → 确定目标主分片
2. 主分片节点：执行写入 → 本地 Lucene + Translog
3. 主分片：并行转发给所有副本分片
4. 副本写入确认
5. 返回客户端（根据 consistency 配置决定等待几个副本）
```

**Lucene 层写入流程（核心！）：**
```
文档写入
  ↓
Index Buffer（内存缓冲区）
  ↓ refresh（默认1s）
Segment（内存中，可搜索）← 这里实现"近实时"
  ↓ flush（Translog 超过阈值 或 定时）
Segment 写入磁盘（.lucene 文件）+ Translog 清空
  ↓ merge（后台）
小 Segment 合并成大 Segment
```

**Translog 的作用**：在 Segment flush 到磁盘之前，Translog 提供故障恢复保障。节点重启时重放 Translog 恢复未持久化的数据。

###### 6. Elasticsearch 的数据查询流程是怎样的？

见第9题的两阶段搜索流程描述，这里补充一些细节：

**Query 阶段细节：**
- 默认情况下，协调节点会查询**所有**相关分片（主分片 or 副本轮询，默认轮询副本分担读压力）
- 每个分片本地执行查询，返回 `from + size` 条文档的 ID 和评分（不是最终 size 条）
- **深分页的根因**：10 个分片，每个分片返回 10000 + 10 条，协调节点要排序 100010 条，然后只取 10 条

**Fetch 阶段细节：**
- 协调节点知道每个 Top 文档在哪个分片，向对应分片发 Multi GET
- 分片从 _source 字段读取完整文档（如果禁用了 `_source`，则无法 Fetch）

**GET vs Search 的区别：**
- `GET /index/_doc/id`：直接路由到具体分片，实时读（Real-time GET），延迟极低
- `POST /index/_search`：两阶段搜索，受 refresh 间隔影响，近实时（NRT）

###### 7. 详细描述一下 Elasticsearch 索引文档的过程？

```
Step 1: 客户端发送 PUT/POST 请求到任意节点（该节点成为协调节点）

Step 2: 协调节点计算路由
  shard_num = hash(_id) % primary_shards_count

Step 3: 转发请求到持有目标主分片的节点

Step 4: 主分片处理写入
  4.1 文档写入内存 Buffer（In-memory indexing buffer）
  4.2 同时追加到 Translog（写入磁盘，保证可靠性）
  4.3 refresh 操作（默认1s）：Buffer → 内存 Segment（此时可搜索）
  4.4 flush 操作（定时或 Translog 达到阈值）：内存 Segment → 磁盘 Segment，清空 Translog

Step 5: 主分片并行转发请求给所有副本分片
  副本执行相同的写入操作

Step 6: 所有副本确认写入成功

Step 7: 协调节点返回客户端成功响应（包含 _id、_version、result 等）
```

**版本控制**：每次写入操作都会递增 `_version`，用于乐观并发控制（通过 `if_seq_no` + `if_primary_term` 实现）。

###### 8. 详细描述一下 Elasticsearch 更新和删除文档的过程

**ES 的文档是不可变的（Immutable）**，更新和删除本质上都是"标记 + 新写"。

**更新流程：**
```
POST /index/_update/id
  ↓
1. 读取当前文档（从主分片）
2. 合并修改内容（原文档 + 更新字段）
3. 删除旧文档（在 .del 文件中标记为已删除）
4. 写入新版本文档（生成新 Segment）
5. 递增 _version
```

**删除流程：**
```
DELETE /index/_doc/id
  ↓
1. 在 .del 文件（或位图）中标记该文档为已删除
2. 文档在磁盘上还存在，但搜索时会被过滤掉
3. 等待 Segment 合并时，已标记文档才会被物理删除
```

**关键点**：
- 频繁更新同一文档会产生大量"僵尸文档"（已删除未物理清除），占用磁盘和影响查询性能
- Segment merge 时才真正清理这些已删除文档
- 可以手动触发合并：`POST /index/_forcemerge?max_num_segments=1`（谨慎在生产使用，IO 压力大）

###### 9. 什么是脑裂问题？如何避免脑裂？

**脑裂（Split Brain）**：网络分区时，集群被切成两个部分，每部分都选出了自己的 Master，形成两个独立集群，各自接受写入，导致数据不一致。

**避免脑裂的核心机制 — Quorum（多数派）**：

ES 7.0 之前通过 `discovery.zen.minimum_master_nodes` 参数配置：
```yaml
# 公式：(候选主节点数 / 2) + 1
# 3个候选主节点：minimum_master_nodes = 2
discovery.zen.minimum_master_nodes: 2
```

只有获得超过半数候选主节点认可，才能当选 Master。这样网络分区时，少数派（不足半数）那边无法选出新 Master，保证全局只有一个 Master。

**ES 7.0+ 的改进**：
- 引入了新的选举机制（基于 Raft 简化版），不再需要手动配置 `minimum_master_nodes`
- 通过 `cluster.initial_master_nodes` 配置初始候选节点，自动计算多数派阈值
- 去除了对 ZooKeeper 的依赖，选举机制内置化

---

###### 10. Elasticsearch 是如何实现 Master 选举的？

**ES 7.0 之前（基于 Zen Discovery）：**
1. 每个候选主节点向其他节点广播自己的存在
2. 收集其他节点发来的投票
3. 选出 ID 最小（或 clusterStateVersion 最大）的节点作为 Master
4. 获得 `minimum_master_nodes` 票数的节点当选

**ES 7.0+（基于新选举机制）：**
1. 候选节点（master-eligible nodes）检测到 Master 不可用时，启动选举
2. 发起选举投票请求，目标是成为 Leader
3. 节点投票给 clusterStateVersion 最高的候选节点（数据最新的优先）
4. 获得多数派（>N/2）投票后，当选新 Master
5. 新 Master 发布集群状态到所有节点，完成领导权确认

**与 ZooKeeper / Raft 的区别**：
- ES 的选举是简化的分布式共识，不是完整 Raft 实现
- 但核心思想一致：多数派投票 + 最新数据优先

###### 11. 集群的最小主节点数应该如何设置？

ES 7.0 之前需要手动设置，公式：
```
minimum_master_nodes = floor(候选主节点数 / 2) + 1
```

| 候选主节点数 | minimum_master_nodes | 说明 |
|------------|---------------------|------|
| 1 | 1 | 单节点，无法高可用 |
| 2 | 2 | 任意一台挂了就无法选主，不推荐 |
| **3** | **2** | **推荐最小配置**，容忍1台故障 |
| 5 | 3 | 容忍2台故障 |

**为什么推荐奇数个候选主节点？**
- 偶数个节点（如4个），容错能力和3个一样（都是最多挂1台）
- 但多花了1台机器，不经济
- 奇数节点数 = 最优的容错性/成本比

ES 7.0+ 这些自动计算，`cluster.initial_master_nodes` 设置初始节点列表即可。

###### 12. 如何设计一个高可用的 Elasticsearch 集群？

**硬件层面：**
- 3 个专用主节点（不存数据，低配即可）
- N 个数据节点（根据数据量规划，SSD 优先）
- 可选：协调节点（统一入口，分担数据节点协调压力）
- 多节点部署在不同物理机/机架，避免单点故障

**配置层面：**
```yaml
# elasticsearch.yml
cluster.name: prod-es-cluster
node.master: true    # 专用主节点：data设为false
node.data: false
discovery.seed_hosts: ["master-1", "master-2", "master-3"]
cluster.initial_master_nodes: ["master-1", "master-2", "master-3"]
```

**索引设计层面：**
- 合理设置主分片数（按数据量规划，单分片建议不超过 50GB）
- 至少 1 个副本（数据安全 + 读性能）
- 生产环境建议 2+ 副本

**感知层面（Rack Awareness）**：
```yaml
node.attr.rack: rack1   # 节点打标签
cluster.routing.allocation.awareness.attributes: rack
```
保证同一分片的主副本分布在不同机架。

###### 13. 什么是 Translog？它的作用是什么？

Translog（Transaction Log）是 ES 的**预写日志（WAL, Write-Ahead Log）**，作用是防止数据丢失。

**为什么需要 Translog？**
- 文档写入后先在内存 Buffer 中，`refresh` 后变成内存 Segment（可搜索）
- 内存 Segment 要等 `flush` 操作才写入磁盘（默认 30 分钟或 Translog 达到 512MB）
- 这段时间内，如果节点宕机，内存数据全丢了
- Translog 写入磁盘，宕机重启时重放 Translog，恢复这段时间的数据

**Translog 写入时机**：
- 每次写入操作都会追加到 Translog
- 默认 `index.translog.durability: request`：每个请求都 fsync Translog（数据安全，但写性能略低）
- 可改为 `async`：异步 fsync，性能更高但可能丢失最近5秒数据

**flush 后 Translog 清空**：Segment 成功持久化到磁盘后，对应的 Translog 条目就可以删除了。

###### 14. 什么是 Segment？Segment 合并的机制是什么？

**Segment（段）**：
- ES 底层基于 Lucene，Lucene 把索引拆分成多个不可变的 Segment
- 每个 Segment 是一个完整的倒排索引，包含词典、倒排列表、_source 数据等
- **不可变**：Segment 写入后就不再修改，更新/删除通过 `.del` 标记文件实现

**为什么要合并 Segment？**
- 每次 `refresh` 都产生一个新 Segment，Segment 越来越多
- 搜索时要查所有 Segment 再合并结果，Segment 越多查询越慢
- 小 Segment 太多还浪费文件描述符和内存

**合并机制（Merge Policy）：**
- 后台线程自动合并小 Segment（TieredMergePolicy，ES 默认）
- 合并时物理删除 `.del` 文件中标记的文档，释放磁盘空间
- 合并完成后，新的大 Segment 替换旧的小 Segment，旧 Segment 删除

**合并的代价**：
- 合并是 IO 密集型操作，会占用大量磁盘带宽
- 生产环境高峰期会影响写入和查询性能
- 可以通过 `indices.merge.scheduler.max_thread_count` 控制合并线程数

###### 15. 在并发情况下，Elasticsearch 如何保证读写一致？

ES 使用**乐观并发控制（Optimistic Concurrency Control）**，不用锁，通过版本号保证一致性。

**核心机制：**

每个文档都有两个控制字段：
- `_seq_no`：操作序号，全局单调递增
- `_primary_term`：主分片任期（主分片切换时递增）

**乐观并发控制示例：**
```bash
# 先读取文档，获取 _seq_no 和 _primary_term
GET /products/_doc/1
# 返回：{ "_seq_no": 10, "_primary_term": 1, ... }

# 更新时携带版本号，如果版本不匹配（被别人改过了），更新失败
PUT /products/_doc/1?if_seq_no=10&if_primary_term=1
{ "price": 4999 }
# 如果此时 _seq_no 已经变成 11，会返回 409 Conflict
```

**主副本一致性**：
- 写入请求先到主分片，主分片写成功后同步给副本
- 副本确认后，写入完成
- 读请求默认可以打到副本（可能读到稍旧数据），可以通过 `preference=_primary` 强制读主分片（强一致但影响性能）

> **本章追问：ES 的"近实时"（NRT）是怎么实现的？默认延迟是多少？**
>
> **答**：
> 近实时的核心是 **refresh 机制**。
>
> 文档写入后先在 IndexWriter 的内存 Buffer 中，此时还**不可搜索**。
> `refresh` 操作会把 Buffer 中的数据生成新的 Segment 放入内存，此时就**可以被搜索到**了。
>
> - 默认 `refresh_interval = 1s`，所以写入后最慢 1 秒可搜索（NRT）
> - 如果需要实时搜索，可以调用 `POST /index/_refresh` 手动触发
> - 写入时设置 `?refresh=true` 强制立即 refresh（影响写入性能）
> - 批量导入数据时，可以设置 `refresh_interval: -1` 禁用自动 refresh，导完再手动 refresh，性能提升明显
>
> **注意**：refresh ≠ flush，refresh 只是内存 Buffer → 内存 Segment（可搜索），不保证持久化；flush 才是把 Segment 写到磁盘。
