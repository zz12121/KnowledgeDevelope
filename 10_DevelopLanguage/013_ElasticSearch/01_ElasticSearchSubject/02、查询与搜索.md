> 双链：[[38_ElasticSearchKnowledge/02_倒排索引与查询机制]]

###### 1. Query DSL 是什么？它有哪些主要查询类型？

Query DSL（Domain Specific Language）是 ES 的查询语言，用 JSON 描述查询逻辑，通过 `_search` 接口发送。

**顶层结构：**
```json
POST /products/_search
{
  "query": { ... },       // 查询条件
  "from": 0,              // 分页起始
  "size": 10,             // 每页条数
  "sort": [...],          // 排序
  "_source": ["name"],    // 返回字段
  "aggs": { ... }         // 聚合
}
```

**主要查询类型分两大类：**

**全文查询（Full-text queries）**：会经过分析器处理，用于 `text` 字段
- `match`：标准全文搜索（最常用）
- `match_phrase`：短语搜索，词序必须一致
- `match_phrase_prefix`：前缀短语搜索（自动补全）
- `multi_match`：多字段匹配
- `query_string`：支持 Lucene 语法的查询字符串

**精确查询（Term-level queries）**：不分词，用于 `keyword`/数值/日期字段
- `term`：精确值匹配
- `terms`：多值精确匹配（相当于 IN）
- `range`：范围查询（gt/gte/lt/lte）
- `exists`：字段存在性查询
- `ids`：按文档 ID 查询
- `prefix/wildcard/regexp/fuzzy`：前缀/通配符/正则/模糊查询

**复合查询：**
- `bool`：组合多个查询（must/should/must_not/filter）
- `function_score`：自定义评分
- `dis_max`：多查询取最高分

###### 2. term 查询和 match 查询有什么区别？

这是最高频的 ES 面试题之一，核心区别在于**是否经过分析器处理**。

**term 查询**：
- 不对查询词分析，原样精确匹配
- 适合 `keyword`、数值、布尔、日期等字段
- 如果对 `text` 字段用 `term`，可能查不到（因为 text 存的是分词后的词项，比如 "iPhone 15" 会被分成 "iphone" 和 "15"，用 `term` 查 "iPhone 15" 查不到）

```json
{ "term": { "status": "published" } }
```

**match 查询**：
- 先对查询词用字段的分析器处理，再去倒排索引查
- 适合 `text` 字段全文搜索
- 查询 "iPhone 15" → 先分词成 ["iphone", "15"] → 分别查倒排索引 → 取并集（默认 OR，可改为 AND）

```json
{ "match": { "name": "iPhone 15" } }
```

**一句话总结**：`keyword` 字段用 `term`，`text` 字段用 `match`，用错类型要么查不到，要么性能差。

###### 3. bool 查询的 must、should、must_not、filter 有什么区别？

bool 查询是 ES 中最强大的复合查询，四个子句各有分工：

| 子句 | 含义 | 影响评分 | 缓存 |
|------|------|---------|------|
| `must` | 必须匹配（AND），影响相关性评分 | ✅ | ❌ |
| `should` | 应该匹配（OR），匹配越多分越高 | ✅ | ❌ |
| `must_not` | 必须不匹配，排除文档 | ❌（不计分） | ✅ |
| `filter` | 必须匹配（AND），但不影响评分 | ❌ | ✅ |

**关键区别 — must vs filter：**
- 两者都是"必须满足"，但 `filter` 不计算相关性评分，走的是 filter 上下文
- `filter` 结果会被 **bitset 缓存**，再次执行相同 filter 直接用缓存，性能更好
- **原则**：不需要评分的条件（比如 status=published、price < 1000）用 `filter`，需要相关性排序的用 `must`

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "手机" } }       // 全文搜索，影响评分
      ],
      "filter": [
        { "term":  { "status": "published" } }, // 精确过滤，不计分，有缓存
        { "range": { "price": { "lte": 5000 } } }
      ],
      "must_not": [
        { "term": { "brand": "山寨" } }         // 排除
      ],
      "should": [
        { "term": { "brand": "苹果" } }         // 加分项
      ],
      "minimum_should_match": 0                  // should 至少匹配数量
    }
  }
}
```

###### 4. 什么是全文搜索？Elasticsearch 如何实现全文搜索？

全文搜索就是在大量文本中找出包含查询关键词的文档，核心在于处理自然语言的分词、同义词、语义匹配等问题。

**ES 实现全文搜索的链路：**

```
写入阶段：
  原始文本 → 分析器（分词+过滤） → 词项列表 → 写入倒排索引

查询阶段：
  查询词 → 同一分析器处理 → 词项列表 → 倒排索引查找 → 候选文档集 → 相关性评分排序 → 返回结果
```

**全文搜索增强手段：**

1. **同义词**：配置 synonym filter，"手机" 和 "移动电话" 等价
2. **拼写纠错**：`fuzzy` 查询，允许编辑距离差异（"iphoe" 能匹配 "iphone"）
3. **搜索建议**：`suggest` API，拼写建议（term suggester）或自动补全（completion suggester）
4. **相关性调优**：`boost` 字段权重，`function_score` 自定义评分
5. **高亮**：`highlight` 返回命中片段并标记

###### 5. 什么是相关性评分？如何调整评分算法？

相关性评分（Relevance Score）决定搜索结果的排序，分越高排越前面。ES 默认用 **BM25** 算法（ES 5.0 之前用 TF-IDF）。

**BM25 核心思想（简化版）：**
- **词频（TF）**：查询词在文档中出现越多，相关性越高，但有饱和效应（词频很多时边际效益递减）
- **逆文档频率（IDF）**：查询词在越少文档中出现，说明越有区分度，权重越高（"的"出现在所有文档里，权重接近0）
- **字段长度归一化**：同样出现1次，在短标题里比在长文章里更重要

**调整评分的方式：**

1. **boost**：提升特定字段或查询的权重
```json
{
  "multi_match": {
    "query": "手机",
    "fields": ["title^3", "description^1"]  // title 权重是 description 的3倍
  }
}
```

2. **function_score**：完全自定义评分函数
```json
{
  "function_score": {
    "query": { "match": { "title": "手机" } },
    "functions": [
      { "field_value_factor": { "field": "sales", "modifier": "log1p", "factor": 0.1 } }
    ],
    "boost_mode": "sum"
  }
}
```

3. **script_score**：用 Painless 脚本完全自定义
4. **修改相似度算法**：在 Mapping 中指定 `"similarity": "classic"` 换回 TF-IDF

###### 6. 什么是聚合（Aggregation）？有哪些常用的聚合类型？

聚合相当于 SQL 的 `GROUP BY` + 统计函数，可以对搜索结果进行分析汇总。

**三大类聚合：**

**1. 桶聚合（Bucket Aggregations）**：按条件分组，每组是一个"桶"
```json
"aggs": {
  "by_brand": {
    "terms": { "field": "brand.keyword", "size": 10 }  // 按品牌分组，取前10
  },
  "by_price_range": {
    "range": {
      "field": "price",
      "ranges": [{"to":1000}, {"from":1000,"to":5000}, {"from":5000}]
    }
  },
  "by_date": {
    "date_histogram": { "field": "created", "calendar_interval": "month" }
  }
}
```

**2. 指标聚合（Metric Aggregations）**：计算数值统计
```json
"aggs": {
  "avg_price": { "avg": { "field": "price" } },
  "max_price": { "max": { "field": "price" } },
  "stats_price": { "stats": { "field": "price" } },  // 一次返回 min/max/avg/sum/count
  "price_percentiles": { "percentiles": { "field": "price", "percents": [50, 95, 99] } }
}
```

**3. 管道聚合（Pipeline Aggregations）**：对其他聚合的结果再聚合
```json
"aggs": {
  "monthly_sales": {
    "date_histogram": { "field": "date", "calendar_interval": "month" },
    "aggs": {
      "total": { "sum": { "field": "amount" } }
    }
  },
  "max_monthly": {
    "max_bucket": { "buckets_path": "monthly_sales>total" }  // 找销售额最高的月份
  }
}
```

###### 7. 什么是深分页问题？如何解决？

**深分页问题**：`from + size` 分页时，如果 `from` 很大（比如第10000页），ES 需要在每个分片上取出 `from + size` 条数据汇总排序，数据量爆炸，内存和 CPU 消耗极高。

默认 `from + size` 上限是 10000，超过会报错。

**三种解决方案：**

**1. 调大 max_result_window（不推荐）**：治标不治本，只是放宽限制
```json
PUT /index/_settings
{ "index.max_result_window": 50000 }
```

**2. Scroll API（适合离线数据导出）**：
- 创建一个快照，每次通过 scroll_id 翻页，不排序，顺序遍历
- 适合后台全量导出，不适合用户实时翻页
```json
POST /index/_search?scroll=5m
{ "size": 1000, "query": {...} }
// 返回 scroll_id，下次用：
POST /_search/scroll
{ "scroll": "5m", "scroll_id": "..." }
```

**3. search_after（推荐，实时翻页）**：
- 用上一页最后一条文档的排序值作为游标，无状态，适合实时翻页
- 必须有唯一排序字段（通常加 `_id` 作为 tiebreaker）
```json
// 第一页
{ "size": 10, "sort": [{"price": "desc"}, {"_id": "asc"}] }
// 第二页，传入上一页最后一条的 sort 值
{ "size": 10, "sort": [...], "search_after": [5999, "doc_id_123"] }
```

###### 8. 什么是滚动查询？适用于什么场景？

滚动查询（Scroll Query）就是上面说的 Scroll API，本质是对索引在某一时刻**创建一个快照**，然后持续翻页遍历，不受 `max_result_window` 限制。

**工作原理：**
1. 首次请求时创建快照，服务端保存上下文（scroll context），返回 `scroll_id`
2. 后续请求携带 `scroll_id`，服务端根据上下文返回下一批数据
3. 超过 scroll 存活时间（如 `5m`）后，上下文自动清除

**适用场景：**
- ✅ 后台任务：全量数据导出到文件、数据迁移、重建索引前的数据读取
- ✅ 离线批处理：不在乎数据实时性，只需要遍历所有数据
- ❌ 不适合用户实时翻页（用 search_after 代替）
- ❌ 不适合高并发（每个 scroll 上下文占用服务器资源）

> 注意：用完 scroll 要手动删除上下文，否则占用内存：`DELETE /_search/scroll/<scroll_id>`

###### 9. 描述一下 Elasticsearch 搜索的过程

ES 搜索分两个阶段：**Query 阶段** 和 **Fetch 阶段**。

```
客户端请求
    ↓
协调节点（Coordinating Node）接收请求
    ↓
━━━━━━━━ Query 阶段 ━━━━━━━━
协调节点将查询请求广播到所有相关分片（主分片或副本轮询）
    ↓
每个分片在本地执行查询，返回 [文档ID + 评分] 的轻量列表（不返回 _source）
    ↓
协调节点收集所有分片的结果，做全局排序，取出 Top N 的文档ID列表
    ↓
━━━━━━━━ Fetch 阶段 ━━━━━━━━
协调节点根据文档ID，向对应分片发起 Multi GET 请求，拉取完整文档 _source
    ↓
协调节点合并结果，返回给客户端
```

**为什么分两阶段？**
- Query 阶段只传文档 ID + 分数，数据量小，网络开销低
- Fetch 阶段只拉取最终需要的少量文档，不会把所有文档的完整内容都传输一遍
- 如果一步到位，每个分片返回完整文档，网络和内存开销会爆炸

###### 10. Elasticsearch 对于大数据量（上亿量级）的聚合如何实现？

大数据量聚合是 ES 的难点，核心手段：

**1. 路由优化**：把同类数据路由到同一分片，聚合只查部分分片
```json
PUT /logs/_doc/1?routing=user_123
{ "user_id": "user_123", "action": "buy" }
```

**2. 预计算聚合（Rollup / Transform）**：
- 用 Data Transform 把原始数据提前聚合成汇总索引，查询时查汇总索引
- 类似于数据仓库的 OLAP 物化视图

**3. 调整精度换性能**：
- `terms` 聚合的 `execution_hint` 使用 `map`（精确但慢）或 `global_ordinals`（近似但快）
- `cardinality` 聚合（精确去重）改用 `HyperLogLog++` 近似算法（误差<1%，性能极高）

**4. 缓存利用**：
- 分片级别的聚合结果会缓存，相同过滤条件的聚合直接用缓存
- 在 filter 上下文中的聚合缓存效果更好

**5. 合理的 Mapping**：
- 聚合字段用 `keyword`，`doc_values` 默认开启（列式存储，聚合专用）
- 不参与聚合的 text 字段关闭 `fielddata`（避免内存爆炸）

###### 11. 什么是高亮显示？如何实现搜索结果高亮？

高亮就是在搜索结果中把命中关键词用特殊标签包裹起来，让用户快速看到为什么这条结果匹配了搜索词。

```json
POST /articles/_search
{
  "query": { "match": { "content": "Elasticsearch 倒排索引" } },
  "highlight": {
    "fields": {
      "content": {
        "fragment_size": 150,         // 每个高亮片段的字符数
        "number_of_fragments": 3,     // 最多返回几个片段
        "pre_tags":  ["<em class='highlight'>"],   // 高亮前缀标签
        "post_tags": ["</em>"]                     // 高亮后缀标签
      }
    }
  }
}
```

**三种高亮器：**
- `unified`（默认）：使用 BM25 算法选最相关片段，支持大多数查询
- `plain`：标准 Lucene 高亮器，精确但慢（需要重新分析文本）
- `fvh`（Fast Vector Highlighter）：使用词向量存储，速度最快，需要 Mapping 中开启 `term_vector`

###### 12. 什么是模糊查询？如何使用通配符和正则表达式？

**模糊查询（fuzzy）**：允许查询词和文档词项之间存在编辑距离差异（增删改字符）
```json
{
  "fuzzy": {
    "name": {
      "value": "iphoe",      // 拼错了
      "fuzziness": "AUTO",   // 自动根据词长决定允许的编辑距离（长词允许2）
      "prefix_length": 2     // 前2个字符必须精确匹配（提升性能）
    }
  }
}
```

**通配符查询（wildcard）**：
- `?`：匹配单个字符
- `*`：匹配任意多个字符
```json
{ "wildcard": { "name.keyword": "iPhone*" } }   // 匹配所有 iPhone 开头的
```
> 注意：通配符查询性能很差，尤其是 `*xxx` 开头的查询（相当于全索引扫描），慎用。

**正则表达式查询（regexp）**：
```json
{ "regexp": { "name.keyword": "iPhone [0-9]+" } }
```
> 正则也很慢，只在必要时用，且只对 `keyword` 字段有意义。

###### 13. 什么是查询上下文和过滤上下文？它们有什么区别？

这是 ES 性能优化的核心概念：

**查询上下文（Query Context）**：
- 回答"这个文档有多匹配？"
- 计算相关性评分（`_score`），结果按分数排序
- **不缓存**，每次都重新计算
- 用于：全文搜索、需要按相关性排序的场景

**过滤上下文（Filter Context）**：
- 回答"这个文档匹配还是不匹配？"（是/否问题）
- **不计算评分**，只做布尔判断
- 结果会被 **bitset 缓存**，重复查询直接命中缓存
- 用于：精确匹配、范围过滤、结构化数据筛选

```json
{
  "query": {                          // 查询上下文
    "bool": {
      "must": [
        { "match": { "title": "手机" } }   // 查询上下文：计算评分
      ],
      "filter": [                           // 过滤上下文：不计评分，有缓存
        { "term":  { "status": "active" } },
        { "range": { "price": { "lte": 5000 } } }
      ]
    }
  }
}
```

**性能原则**：凡是不需要相关性排序的条件，一律放 `filter`，享受缓存红利。

###### 14. 如何实现多字段搜索？

**multi_match**：最常用的多字段全文搜索方式
```json
{
  "multi_match": {
    "query": "苹果手机",
    "fields": ["title^3", "description", "brand^2"],  // 字段权重
    "type": "best_fields"   // 取最高分字段的分数（默认）
  }
}
```

**multi_match 的 type 参数：**
- `best_fields`（默认）：取得分最高字段的分数，适合搜索内容集中在单个字段
- `most_fields`：多个字段的分数相加，适合同一内容在多个字段有不同分词
- `cross_fields`：把多个字段当成一个大字段处理，适合人名（first_name + last_name）
- `phrase`：短语匹配模式

**bool + multi_match 组合：**
```json
{
  "bool": {
    "should": [
      { "match": { "title": { "query": "苹果手机", "boost": 3 } } },
      { "match": { "description": "苹果手机" } },
      { "match": { "brand": { "query": "苹果手机", "boost": 2 } } }
    ]
  }
}
```

###### 15. 什么是搜索建议（Suggestion）？有哪些类型？

搜索建议用于实现"你是不是想搜…"和输入框自动补全功能。

**四种 Suggester：**

**1. Term Suggester（词条建议）**：拼写纠错
```json
"suggest": {
  "spell-check": {
    "text": "iphoe",
    "term": { "field": "name", "suggest_mode": "missing" }
  }
}
// 返回：iphone
```

**2. Phrase Suggester（短语建议）**：对整个短语纠错
```json
"suggest": {
  "phrase-suggest": {
    "text": "苹果 手 机",
    "phrase": { "field": "title.trigram" }
  }
}
```

**3. Completion Suggester（自动补全）**：输入框实时补全，性能最好（基于 FST 内存数据结构）
```json
// Mapping 中声明 completion 类型
"suggest_field": { "type": "completion" }

// 查询
"suggest": {
  "product-suggest": {
    "prefix": "苹果",
    "completion": { "field": "suggest_field", "size": 5 }
  }
}
```

**4. Context Suggester**：带上下文过滤的自动补全（比如只补全某个品类下的商品）

> 生产中最常用的是 **Completion Suggester**，因为它是基于内存的 FST 数据结构，响应时间可以做到毫秒级。

---

> **本章追问：`match_phrase` 和 `match` 有什么区别？间隔查询（slop）是什么？**
>
> **答**：
> - `match`：分词后取并集（默认 OR）或交集（operator: AND），不关心词的顺序和位置
> - `match_phrase`：分词后要求词项**顺序一致且相邻**（默认 `slop=0`）
>
> 比如查 "快速 棕色 狐狸"：
> - `match`：文档中有"快速"或"棕色"或"狐狸"任意一个就能匹配
> - `match_phrase`：三个词必须按顺序出现且相邻
>
> **slop** 参数允许词之间有间隔：
> ```json
