###### 1. 如何监控 Elasticsearch 集群的健康状态？

**集群健康三色系统：**

```bash
GET /_cluster/health
```
```json
{
  "status": "green",           // green/yellow/red
  "number_of_nodes": 5,
  "number_of_data_nodes": 3,
  "active_primary_shards": 20,
  "active_shards": 40,
  "relocating_shards": 0,      // 正在迁移的分片
  "initializing_shards": 0,    // 正在初始化的分片
  "unassigned_shards": 0       // 未分配分片数（非0就要关注）
}
```

| 状态 | 含义 |
|------|------|
| 🟢 Green | 所有主分片和副本分片都正常分配 |
| 🟡 Yellow | 所有主分片正常，部分副本未分配（数据安全但冗余不足） |
| 🔴 Red | 部分主分片未分配（有数据不可用！） |

**监控命令速查：**
```bash
GET /_cluster/health?wait_for_status=green&timeout=30s  # 等待健康
GET /_cluster/stats          # 集群统计
GET /_nodes/stats            # 节点统计（JVM/堆/GC/磁盘/索引）
GET /_cat/nodes?v            # 节点列表（可读格式）
GET /_cat/shards?v           # 分片分布
GET /_cat/indices?v&s=index  # 索引列表
```

###### 2. 监控 Elasticsearch 集群需要关注哪些关键指标？

**JVM & 内存：**
- `jvm.mem.heap_used_percent`：heap 使用率，超过 75% 需关注，超过 85% 告警
- `jvm.gc.collectors.old.collection_time_in_millis`：Old GC 耗时，增长快说明 heap 压力大
- `jvm.gc.collectors.young.collection_count`：Young GC 频率

**索引性能：**
- `indices.indexing.index_time_in_millis / index_total`：平均写入延迟
- `indices.search.query_time_in_millis / query_total`：平均查询延迟
- `indices.merges.current`：当前正在合并的 segment 数（过多说明写入压力大）

**集群状态：**
- `cluster.routing.allocation.unassigned_shards`：未分配分片数
- `nodes.count`：节点数（变少了可能有节点挂了）

**磁盘：**
- `fs.total.available_in_bytes`：可用磁盘空间，低于 15% ES 会拒绝写入
- 磁盘水位线：`85%` 停止分配新分片，`90%` 迁移已有分片，`95%` 索引变只读

**CPU & 负载：**
- `os.cpu.percent`：CPU 使用率
- `thread_pool.search.rejected`：搜索线程池拒绝数（说明搜索队列满了）
- `thread_pool.write.rejected`：写入线程池拒绝数

###### 3. Elasticsearch 故障排查的方法和流程是什么？

**标准排查流程（从全局到局部）：**

```
Step 1: 看集群健康状态
GET /_cluster/health

Step 2: 看未分配分片原因（红/黄状态必看）
GET /_cluster/allocation/explain

Step 3: 看节点状态
GET /_nodes/stats?human=true
GET /_cat/nodes?v&h=name,heap.percent,cpu,load_1m,disk.used_percent

Step 4: 看热线程（找出 CPU 高的线程）
GET /_nodes/hot_threads

Step 5: 看慢日志
# ES 日志目录下的 *_index_indexing_slowlog.log 和 *_index_search_slowlog.log

Step 6: 看集群状态（分片分配情况）
GET /_cluster/state/routing_table

Step 7: 查看任务队列（有无堆积）
GET /_tasks?actions=*search*&detailed=true
```

**常见故障速查：**
- 集群 Red → 看 `allocation/explain`，通常是磁盘满/节点挂了/分片损坏
- OOM → 看 `jvm.mem` + fielddata 使用量
- 写入拒绝 → 看 `thread_pool.write.rejected` + 磁盘水位
- 查询超时 → 看慢日志 + Profile API

###### 4. 如何进行 Elasticsearch 的备份和恢复？

ES 备份通过**快照（Snapshot）** 机制实现，支持增量备份。

**配置快照仓库（以 S3 为例）：**
```json
PUT /_snapshot/my_s3_repo
{
  "type": "s3",
  "settings": {
    "bucket": "es-backup-bucket",
    "region": "cn-hangzhou",
    "base_path": "es-snapshots"
  }
}
```

**创建快照（全量/增量）：**
```json
PUT /_snapshot/my_s3_repo/snapshot_20260315
{
  "indices": "logs-*,products",      // 指定索引，默认所有
  "ignore_unavailable": true,        // 忽略不存在的索引
  "include_global_state": false      // 是否包含集群全局状态
}
```

**恢复快照：**
```json
POST /_snapshot/my_s3_repo/snapshot_20260315/_restore
{
  "indices": "products",
  "rename_pattern": "(.+)",
  "rename_replacement": "restored_$1"  // 恢复为新索引名，避免冲突
}
```

**查看快照状态：**
```bash
GET /_snapshot/my_s3_repo/_all        # 所有快照列表
GET /_snapshot/my_s3_repo/snapshot_20260315  # 单个快照详情
GET /_snapshot/my_s3_repo/snapshot_20260315/_status  # 快照进度
```

**增量原理**：快照只备份自上次快照以来变化的 Segment 文件，第一次全量，后续增量，速度快、存储省。

###### 5. 什么是索引生命周期管理？有哪些策略？

ILM（Index Lifecycle Management）用于自动管理索引在不同阶段的策略，特别适合时序日志场景。

**四个阶段：**

```
Hot（热阶段）→ Warm（暖阶段）→ Cold（冷阶段）→ Delete（删除阶段）
```

**ILM 策略示例（日志系统）：**
```json
PUT /_ilm/policy/logs_policy
{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_size": "50gb",       // 超过50GB就滚动到新索引
            "max_age": "1d"           // 或超过1天
          },
          "set_priority": { "priority": 100 }
        }
      },
      "warm": {
        "min_age": "7d",              // 7天后进入暖阶段
        "actions": {
          "shrink": { "number_of_shards": 1 },   // 缩减分片（节省内存）
          "forcemerge": { "max_num_segments": 1 }, // 合并段（提升搜索）
          "readonly": {},                          // 设为只读
          "allocate": { "require": { "box_type": "warm" } }  // 迁移到暖节点
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "allocate": { "require": { "box_type": "cold" } },  // 迁移到冷节点
          "freeze": {}    // 冻结索引（减少内存占用）
        }
      },
      "delete": {
        "min_age": "90d",             // 90天后删除
        "actions": { "delete": {} }
      }
    }
  }
}
```

###### 6. 如何处理 Elasticsearch 节点故障和恢复？

**节点故障处理流程：**

```
1. 检测到节点失联（默认 60 秒后确认宕机）
2. Master 将该节点的主分片标记为 unassigned
3. 将该节点其他节点上的副本提升为主分片（无数据丢失）
4. 重新分配副本分片给其他健康节点
5. 副本同步数据，集群恢复 Green
```

**手动干预恢复：**
```bash
# 查看未分配原因
GET /_cluster/allocation/explain

# 手动触发分片重新分配
POST /_cluster/reroute?retry_failed=true

# 强制迁移分片（谨慎使用）
POST /_cluster/reroute
{
  "commands": [
    {
      "move": {
        "index": "my_index",
        "shard": 0,
        "from_node": "node-1",
        "to_node": "node-2"
      }
    }
  ]
}
```

**预防措施：**
- 节点下线前先通过 ILM 或 allocation filtering 排空分片
- 重要数据保持 2+ 副本
- 跨机架部署（shard allocation awareness）

###### 7. 如何进行 Elasticsearch 集群的容量规划？

**容量规划四要素：**

**1. 数据量预估：**
```
原始数据大小 × 1.45（索引开销）× (1 + 副本数) = 总存储需求
例：100GB 原始 × 1.45 × 2（1副本）= 290GB 磁盘需求（预留20%缓冲 → 350GB）
```

**2. 分片规划：**
```
单分片建议 20~50GB
主分片数 = 总数据量 / 50GB（保守估算）
数据节点数 = 总分片数（主+副）/ 每节点建议分片数（约1000/GB heap）
```

**3. 内存规划：**
```
ES heap = 物理内存 × 50%（max 30GB）
OS cache = 物理内存 × 50%（给 Lucene 文件缓存）
建议：32GB 物理内存（16GB heap + 16GB OS cache）
```

**4. 写入/搜索 QPS 估算：**
- 单数据节点写入：5000~10000 文档/秒（SSD）
- 单分片查询：200~500 QPS（复杂查询）
- 根据目标 QPS 规划节点数和分片数

**容量规划工具**：可以用 ES 的 `_cluster/stats` + 历史数据增长率推算未来6个月需求。

###### 8. Elasticsearch 版本升级需要注意哪些事项？

**升级原则：**
1. **逐版本升级**：7.x → 8.x，不能跨多大版本（6 → 8 不行，必须 6 → 7 → 8）
2. **同版本补丁升级**：7.15 → 7.17 可以直接升
3. **先升 Kibana 再升 ES？不对**：应该先升 ES 节点，再升 Kibana

**升级前检查：**
```bash
GET /_migration/deprecations  # 查看当前版本的废弃 API/配置
```

**滚动升级流程（Rolling Upgrade，零停机）：**
```
1. 禁用分片自动分配（避免频繁迁移）
POST /_cluster/settings
{ "persistent": { "cluster.routing.allocation.enable": "primaries" } }

2. 对一个节点执行：
   - 停止该节点上的 Elasticsearch
   - 升级 ES 软件包
   - 更新 elasticsearch.yml（处理废弃配置）
   - 启动节点
   - 等待节点加入集群并分片分配完成

3. 恢复自动分配
POST /_cluster/settings
{ "persistent": { "cluster.routing.allocation.enable": null } }

4. 对其他节点重复步骤 2-3
```

**注意事项：**
- 升级前**全量快照备份**（无论如何）
- 检查客户端 SDK 版本兼容性
- 废弃 API（如 `_type`）提前迁移

###### 9. 如何处理集群黄色或红色状态？

**Yellow（副本未分配）排查：**
```bash
# 最常见原因：单节点集群（副本无法分配在不同节点）
GET /_cluster/allocation/explain

# 解法1：增加节点
# 解法2：临时将副本数设为0（单节点测试环境）
PUT /my_index/_settings { "number_of_replicas": 0 }
```

**Red（主分片未分配）排查：**
```bash
# Step1: 定位哪个索引的哪个分片有问题
GET /_cat/shards?v&h=index,shard,prirep,state,unassigned.reason

# Step2: 详细原因
GET /_cluster/allocation/explain
{
  "index": "my_index",
  "shard": 0,
  "primary": true
}
```

**常见 Red 状态原因和解法：**

| 原因 | 解法 |
|------|------|
| 磁盘满（watermark） | 清理磁盘，降低水位线，迁移数据 |
| 节点丢失（数据不可恢复） | 从快照恢复，或强制分配（数据丢失风险！） |
| 分片损坏 | `_cluster/reroute` + `allocate_stale_primary` |
| 节点内存不足 | 扩容节点内存或减少分片数 |

###### 10. 什么是滚动重启？如何执行滚动重启？

滚动重启（Rolling Restart）是不停服地逐个重启节点的操作，用于配置更新、版本升级等场景。

**完整滚动重启步骤：**

```bash
# 1. 禁用分片自动重分配（避免节点重启时触发大量分片迁移）
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "primaries"
  }
}

# 2. 执行同步刷新（让内存数据持久化，加快重启后恢复）
POST /_flush/synced

# 3. 停止并重启单个节点
systemctl stop elasticsearch
# ... 修改配置 ...
systemctl start elasticsearch

# 4. 等待节点加入集群
GET /_cat/nodes?v

# 5. 恢复自动分配，等待集群变 Green
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": null
  }
}
GET /_cluster/health?wait_for_status=green&timeout=60s

# 6. 重复步骤 1-5 处理下一个节点
```

**关键点**：步骤1禁用自动分配是关键，否则节点重启时 ES 会立刻开始迁移该节点的分片，重启完成后又迁回来，白白消耗大量 IO。

###### 11. 如何配置和使用 Elasticsearch 的快照仓库？

快照仓库（Snapshot Repository）是快照数据的存储位置，支持多种类型：

**仓库类型：**
- `fs`：本地/网络文件系统（NFS）
- `s3`：AWS S3（需要 repository-s3 插件）
- `azure`：Azure Blob Storage
- `gcs`：Google Cloud Storage
- `hdfs`：Hadoop HDFS
- 阿里云 OSS 等（通过第三方插件）

**本地文件系统仓库配置：**
```yaml
# elasticsearch.yml（所有节点都需要配置）
path.repo: ["/mount/backups"]
```
```json
PUT /_snapshot/local_backup
{
  "type": "fs",
  "settings": {
    "location": "/mount/backups/es-snapshots",
    "compress": true    // 快照数据压缩
  }
}
```

**自动快照策略（SLM - Snapshot Lifecycle Management）：**
```json
PUT /_slm/policy/daily_snapshot
{
  "schedule": "0 0 2 * * ?",        // 每天凌晨2点
  "name": "<daily-snap-{now/d}>",   // 快照名带日期
  "repository": "my_s3_repo",
  "config": {
    "indices": ["*"],
    "ignore_unavailable": true,
    "include_global_state": false
  },
  "retention": {
    "expire_after": "30d",          // 保留30天
    "min_count": 5,                 // 至少保留5个
    "max_count": 50                 // 最多保留50个
  }
}
```

###### 12. 如何进行索引的重建（Reindex）？

Reindex 是把一个索引的数据复制到另一个索引，用于：修改主分片数、修改不可变 Mapping、数据清洗等。

**基本 Reindex：**
```json
POST /_reindex
{
  "source": {
    "index": "products_v1",
    "query": { "match_all": {} }    // 可以只 reindex 部分数据
  },
  "dest": {
    "index": "products_v2",
    "version_type": "external"      // 用源文档版本号，避免覆盖新写入
  }
}
```

**异步 Reindex（大数据量）：**
```bash
POST /_reindex?wait_for_completion=false
# 返回 task_id，用 GET /_tasks/<task_id> 跟踪进度
```

**Reindex + 数据转换（Pipeline）：**
```json
POST /_reindex
{
  "source": { "index": "old_index" },
  "dest": { "index": "new_index", "pipeline": "my_pipeline" }
}
```

**零停机 Reindex 方案（别名切换）：**
```
1. 新建 products_v2，定义新的 Mapping
2. 执行 Reindex（期间 products_v1 正常服务）
3. Reindex 完成后同步 v1 写入期间的增量数据（根据 updated_at 字段）
4. 原子切换别名：products → products_v2
5. 删除 products_v1
```

###### 13. 如何处理磁盘空间不足的问题？

ES 有三个磁盘水位线（watermark），超过就会限制操作：

| 水位线 | 默认值 | 触发动作 |
|--------|--------|---------|
| `low` | 85% | 停止向该节点分配新分片 |
| `high` | 90% | 尝试将该节点的分片迁移走 |
| `flood_stage` | 95% | 所有该节点的索引变为**只读** |

**紧急处理（索引变只读后）：**
```bash
# Step1: 清理磁盘（删除旧索引/快照）
DELETE /old-logs-2025-*

# Step2: 临时调整水位线（争取操作时间）
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.watermark.flood_stage": "99%"
  }
}

# Step3: 解除索引只读状态
PUT /affected_index/_settings
{ "index.blocks.read_only_allow_delete": null }

# Step4: 扩容磁盘，恢复正常水位配置
```

**预防措施：**
- 配置 `index.lifecycle.rollover` + ILM，自动删除过期数据
- 设置磁盘使用率 80% 告警（在 flood_stage 之前有时间处理）
- 定期执行快照 + 删除历史索引

###### 14. 如何配置和使用 X-Pack 监控功能？

X-Pack 是 ES 的商业功能套件，基础监控功能在 Basic 许可证（免费）中包含。

**开启监控（ES 8.x 默认已开启）：**
```yaml
# elasticsearch.yml
xpack.monitoring.collection.enabled: true
```

**Kibana 中查看监控**：
- Stack Monitoring → 集群概览、节点性能、索引性能、JVM 状态

**使用 Metricbeat 替代内置监控（推荐）：**
```yaml
# metricbeat.yml
metricbeat.modules:
  - module: elasticsearch
    hosts: ["http://localhost:9200"]
    period: 10s
    xpack.enabled: true
```

**监控数据查询（内置监控）：**
```bash
GET /.monitoring-es-*/_search    # 查询 ES 监控数据
GET /.monitoring-kibana-*/_search  # 查询 Kibana 监控数据
```

###### 15. 如何设置告警和通知机制？

ES 的告警功能在 **Kibana Alerting**（8.x）或 **X-Pack Watcher**（经典方式）中实现。

**X-Pack Watcher 示例（监控集群 Red 状态并发邮件）：**
```json
PUT /_watcher/watch/cluster_health_red
{
  "trigger": {
    "schedule": { "interval": "1m" }    // 每分钟检查一次
  },
  "input": {
    "http": {
      "request": { "url": "http://localhost:9200/_cluster/health" }
    }
  },
  "condition": {
    "compare": { "ctx.payload.status": { "eq": "red" } }
  },
  "actions": {
    "send_email": {
      "email": {
        "to": ["ops@company.com"],
        "subject": "ES 集群变为 Red 状态！",
        "body": "集群健康状态：{{ctx.payload.status}}，请立即处理！"
      }
    },
    "notify_webhook": {
      "webhook": {
        "method": "POST",
        "url": "https://hooks.slack.com/...",
        "body": "{\"text\": \"ES集群Red告警！\"}"
      }
    }
  }
}
```

**Kibana Alerting（现代方式）**：
- 在 Kibana → Alerts and Actions 中图形化配置
- 支持 Elasticsearch Query、Threshold、Anomaly Detection 等多种告警规则
- 通知方式支持邮件、Slack、PagerDuty、Webhook 等

> **本章追问：ES 集群磁盘突然满了，索引变成只读，如何快速恢复服务？**
>
> **答**：标准应急流程：
>
> 1. **立即确认受影响的索引**：
> ```bash
> GET /_cat/indices?v&h=index,store.size,status
> ```
>
> 2. **临时调高磁盘水位线**（争取操作窗口）：
> ```json
> PUT /_cluster/settings
> { "transient": { "cluster.routing.allocation.disk.watermark.flood_stage": "99%" } }
> ```
>
> 3. **删除不需要的数据释放空间**（历史索引/快照）：
> ```bash
> DELETE /logs-2025-01,logs-2025-02
> ```
>
> 4. **解除只读状态**：
> ```json
> PUT /*/_settings { "index.blocks.read_only_allow_delete": null }
> ```
>
