# 四、性能优化

> 双链：[[38_ElasticSearchKnowledge/04_性能优化与Mapping设计]]

---

###### 1. 如何优化 Elasticsearch 的写入性能？

写入性能优化从四个维度入手：**批量写入、减少刷新、调整副本、系统配置**。

**1. 批量写入（Bulk API）**：
```json
POST /_bulk
{ "index": { "_index": "logs", "_id": "1" } }
{ "message": "error occurred", "timestamp": "2026-03-15" }
{ "index": { "_index": "logs", "_id": "2" } }
{ "message": "user login", "timestamp": "2026-03-15" }
```
- 单条写入有大量网络开销，批量写入摊薄开销
- 单批建议 5~15MB，具体看 bulk 请求体大小（不要超过 100MB）
- 多线程并行发 bulk，充分利用各分片

**2. 调整 refresh_interval**：
```json
PUT /my_index/_settings
{ "refresh_interval": "30s" }   // 批量导入时甚至可设为 -1（禁用）
```

**3. 初始导入时关闭副本**：
```json
PUT /my_index/_settings
{
  "number_of_replicas": 0,     // 先关副本
  "refresh_interval": "-1"
}
// 导完后恢复
PUT /my_index/_settings
{
  "number_of_replicas": 1,
  "refresh_interval": "1s"
}
```

**4. 其他优化**：
- 使用 SSD 磁盘，Translog 写 IO 是瓶颈之一
- `index.translog.durability: async`：异步刷 translog（可能丢5秒数据，换性能）
- 合理设置主分片数，写入可以并行到多个分片

---

###### 2. 批量操作有哪些最佳实践？

**Bulk API 使用要点：**

1. **控制单批大小**：推荐 5~15MB 的请求体，太大会导致 GC 压力，太小网络开销高
2. **错误处理**：Bulk 是部分成功的，响应中每个 item 有自己的状态，需要检查 `errors` 字段
```json
// 响应示例
{
  "errors": true,
  "items": [
    { "index": { "result": "created", "status": 201 } },
    { "index": { "error": { "type": "mapper_exception" }, "status": 400 } }
  ]
}
```
3. **多线程并行**：单线程发 bulk 吞吐量有限，可以开多个线程/goroutine 并行发送
4. **路由优化**：如果使用自定义路由，同一路由的文档放在同一 bulk 请求里
5. **重试机制**：对于 429（Too Many Requests）和 5xx 错误需要退避重试
6. **不要混合大 document**：超大文档（>1MB）单独处理，不要混在 bulk 里拖慢整批

---

###### 3. 如何优化索引的刷新间隔？

`refresh_interval` 控制内存 Buffer → 内存 Segment 的频率，直接影响写入性能和搜索延迟。

```json
// 场景1：实时搜索（日志监控）
PUT /realtime_logs/_settings
{ "refresh_interval": "1s" }    // 默认，1秒内可搜索

// 场景2：准实时（允许5秒延迟）
PUT /products/_settings
{ "refresh_interval": "5s" }    // 写入性能提升约30%

// 场景3：批量导入（不需要实时搜索）
PUT /batch_import/_settings
{ "refresh_interval": "-1" }    // 禁用自动 refresh，导完后手动 refresh
POST /batch_import/_refresh

// 场景4：索引建设期（完全不需要搜索）
// 关闭 refresh + 关闭副本，导完后再开启
```

**refresh 的代价**：每次 refresh 创建新 Segment，小 Segment 太多会影响查询性能（需要查更多 Segment）。后台 merge 会合并这些小 Segment，但 merge 本身也有 IO 开销。

###### 4. 如何优化 Elasticsearch 的查询性能？

**1. Mapping 优化**：
- 不需要搜索的字段关掉索引：`"index": false`
- 不需要聚合/排序的 text 字段关掉 `fielddata`（默认就是关的）
- 不返回给客户端的字段关掉存储：`"store": false`（`_source` 包含所有字段，这个一般不用改）

**2. 查询写法优化**：
- 过滤条件用 `filter` 代替 `must`（有缓存）
- 避免 `wildcard` 前缀通配符（`*xxx`）和大量 `script` 查询
- 聚合字段确保 `doc_values` 开启（默认开启，手动关掉了要恢复）
- 深分页用 `search_after` 替换 `from + size`

**3. 分片策略**：
- 单分片不超过 50GB，搜索时分片越多并行度越高
- 但分片太多也会增加协调节点合并压力，找到平衡点

**4. 缓存利用**：
- `filter` 上下文的结果会被 **shard request cache** 缓存（分片级别的完整结果缓存）
- 固定时间范围的查询（如 `now/d`）比 `now` 更容易命中缓存（`now` 每秒都在变）

**5. 路由优化**：
- 自定义 routing 让查询只打特定分片，减少协调和合并开销

###### 5. Elasticsearch 有哪些缓存机制？如何利用缓存提升性能？

ES 有三层缓存：

**1. Node Query Cache（节点级）**：
- 缓存 `filter` 上下文的查询结果（bitset 格式）
- 默认占用 heap 的 10%（`indices.queries.cache.size: 10%`）
- Key 是查询条件，相同 filter 直接命中缓存
- **优化**：把所有不影响评分的条件都放 `filter`

**2. Shard Request Cache（分片请求级）**：
- 缓存**聚合结果**和**命中数 total**，不缓存 _source 内容
- 只缓存 `size=0` 的聚合请求（即不需要返回文档内容的纯聚合）
- 索引 refresh 后自动失效（数据变了缓存就不准了）
- **优化**：监控仪表盘等重复聚合场景，设置 `size: 0` 享受缓存

**3. Fielddata Cache（字段数据缓存）**：
- `text` 字段做聚合/排序时需要加载 fielddata 到内存（很昂贵！）
- 默认不限制大小，可能导致 OOM
- **优化**：text 字段不要做聚合/排序，用 `keyword` sub-field

```yaml
# elasticsearch.yml
indices.fielddata.cache.size: 20%   # 限制 fielddata 缓存大小
```

###### 6. 如何合理设置分片数量？需要考虑哪些因素？

**经验原则：**
- **单个分片大小**：建议 20~50GB（写入密集型可适当大，搜索密集型可适当小）
- **分片数与节点数**：分片数 = 数据节点数 × 1~3，让每个节点承担相近数量的分片
- **主分片数一旦确定不可修改**（需要 Reindex），所以要预留增长空间

**计算示例：**
```
数据量：500GB，预估增长2倍 → 按1TB规划
单分片大小：50GB
主分片数：1TB / 50GB = 20个主分片
数据节点：5台
每台承担：20 / 5 = 4个主分片，合理
```

**分片过多的问题：**
- 每个分片都是一个 Lucene 实例，占用内存（heap 中的元数据）
- 协调节点需要合并更多分片的结果，延迟增加
- 建议每个节点的分片数不超过 **1000/1GB heap**（即 30GB heap → 最多 30000 个分片，但实际远不要这么多）

**分片过少的问题：**
- 无法利用多节点并行能力
- 单分片过大，segment 合并耗时增加

###### 7. 什么是热点分片问题？如何处理？

**热点分片**：集群中某个分片（Primary 或 Replica）承担了不成比例的大量请求或数据，成为瓶颈。

**产生原因：**
1. **自定义路由不均匀**：某个 routing 值的文档数量远多于其他 routing，造成该分片数据量过大
2. **写入热点**：时序数据按时间路由，最新的分片一直被写入（老分片没有写入）
3. **查询热点**：某些热门关键词的倒排列表非常长，查询该词时对应分片 CPU 飙高

**解决方案：**

1. **路由分散**：增加路由键的多样性，避免单一 routing 值
```json
// 原来按 user_id 路由，改为 user_id + 日期，分散写入
"routing": "user_123_20260315"
```

2. **索引拆分（Split）**：把热点分片拆成多个
```json
POST /hot-index/_split/new-index
{ "settings": { "index.number_of_shards": 10 } }  // 原来5个，拆成10个
```

3. **冷热数据分层**：最新数据放热节点（SSD），历史数据迁移到冷节点（HDD），减少热节点写入压力

4. **副本均衡**：对热点索引增加副本，让读请求分散到多个副本

###### 8. 如何优化段合并策略？

段合并（Segment Merge）是写入性能的隐形杀手，策略不合理会拖慢写入甚至查询。

**调整合并线程数：**
```yaml
# SSD 可以设置更大（SSD 并发 IO 好），机械盘建议设1
index.merge.scheduler.max_thread_count: 1
```

**调整合并触发阈值（TieredMergePolicy）：**
```json
PUT /my_index/_settings
{
  "index.merge.policy.segments_per_tier": 10,     // 每层允许的最大段数，越小合并越积极
  "index.merge.policy.max_merged_segment": "5gb", // 单个段合并后的最大大小
  "index.merge.policy.floor_segment": "2mb"       // 小于此大小的段会被优先合并
}
```

**Force Merge（手动合并）**：
```json
// 将索引合并到1个段，查询性能最好，适合只读的历史索引
POST /old-logs-2026-01/_forcemerge?max_num_segments=1
// 警告：正在写入的索引不要强制合并，IO 消耗极大！
```

**最佳实践**：时序日志场景，历史月份的索引（不再写入）在业务低峰期执行 force merge，大幅提升查询性能。

###### 9. 如何进行 Elasticsearch 的 JVM 堆内存调优？

**黄金法则：**
1. **heap 不超过 31GB**：超过 31GB JVM 会切换到大地址指针（non-compressed oops），GC 效率下降。保守设置 26~30GB 确保 compressed oops 生效
2. **heap = 物理内存的 50%，另 50% 留给 OS cache**：Lucene 大量使用文件系统缓存（mmap），需要足够的 OS cache
3. **heap 大小固定**：`-Xms` = `-Xmx`，避免动态调整堆大小触发 GC

```bash
# jvm.options
-Xms26g
-Xmx26g
```

**GC 选择**：
- ES 8.x 默认使用 G1GC（在大堆下表现更稳定）
- 避免 CMS（已废弃）
- 超大内存（>64GB）可以考虑 ZGC

**OOM 排查**：
- `indices.fielddata.cache.size` 过大导致 fielddata OOM → 限制 fielddata 缓存
- 大量聚合操作 → 增加堆大小或降低聚合精度
- 大量 segment 元数据 → 减少分片数（每个分片占用一定 heap）

###### 10. 说说 Elasticsearch 常用的调优手段？

综合调优清单：

**写入优化：**
- ✅ 使用 Bulk API 批量写入
- ✅ 调大 `refresh_interval`（5s~30s）
- ✅ 批量导入时关闭副本 + 禁用 refresh
- ✅ Translog 改为 async 模式（允许少量数据丢失的场景）

**查询优化：**
- ✅ 过滤条件全部用 `filter`（享受缓存）
- ✅ 禁用不需要的 `_source` 字段（减少网络传输）
- ✅ `search_after` 替代深分页
- ✅ `routing` 减少查询分片数
- ✅ 不用 `wildcard` + `*` 前缀

**Mapping 优化：**
- ✅ 不搜索的字段：`"index": false`
- ✅ text 字段加 keyword sub-field（搜索+聚合两用）
- ✅ 不需要的字段关掉 `_source` 中的存储（或精简 `_source` 内容）
- ✅ 数值类型选够用的最小类型（integer vs long，byte vs integer）

**集群优化：**
- ✅ 主节点专用（不存数据）
- ✅ SSD 磁盘（至少给热数据节点）
- ✅ 冷热分层存储
- ✅ 合理的分片数（单分片 20-50GB）

###### 11. Elasticsearch 在部署时，对 Linux 的设置有哪些优化方法？

**必须做的 Linux 系统优化：**

```bash
# 1. 禁用 Swap（ES 绝对不能用 swap，会导致性能断崖式下降）
swapoff -a
# /etc/fstab 中注释掉 swap 行

# 或者让 ES 锁定内存（elasticsearch.yml）
bootstrap.memory_lock: true

# 2. 调大虚拟内存（Lucene 使用 mmap 大量映射文件）
echo "vm.max_map_count=262144" >> /etc/sysctl.conf
sysctl -p

# 3. 调大文件描述符限制（每个 shard/segment 都占用 fd）
# /etc/security/limits.conf
elasticsearch soft nofile 65536
elasticsearch hard nofile 65536

# 4. 调大最大线程数
elasticsearch soft nproc  4096
elasticsearch hard nproc  4096

# 5. 磁盘 I/O 调度策略（SSD 用 noop/none，机械盘用 deadline）
echo "none" > /sys/block/sda/queue/scheduler
```

**验证 ES 系统配置：**
```json
GET /_nodes?filter_path=nodes.*.process,nodes.*.os
```

###### 12. GC 方面，在使用 Elasticsearch 时要注意什么？

**核心原则：GC 停顿是 ES 的大敌，STW（Stop-The-World）期间集群对外无响应。**

**常见 GC 问题及解法：**

1. **大量 fielddata 占用 heap**：
   - 原因：对 `text` 字段做聚合，fielddata 加载到 heap
   - 解法：用 `keyword` 字段替代；设置 `indices.fielddata.cache.size: 20%`

2. **大对象（Humongous Objects）**：
   - G1GC 中大于 region 1/2 的对象直接进 Humongous 区，无法被 Minor GC 回收
   - 解法：减少大聚合请求；大 bulk 请求体拆小

3. **JVM 配置要点**：
```bash
# jvm.options
-Xms26g -Xmx26g        # heap 固定大小
-XX:+UseG1GC           # 使用 G1
-XX:MaxGCPauseMillis=200  # 目标最大停顿时间
-XX:G1HeapRegionSize=16m  # 根据堆大小调整
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/heap.hprof
```

4. **监控 GC**：
   - `GET /_nodes/stats/jvm` 查看 GC 次数和时间
   - 如果 old GC 频繁（每分钟多次），说明 heap 使用率过高

###### 13. 如何优化磁盘 I/O 性能？

1. **使用 SSD**：Translog fsync、Segment 合并、快照都是 IO 密集型，SSD 效果最明显
2. **dataDir 和 translog 分离到不同磁盘**：
```yaml
path.data: /data/es-data
path.logs: /var/log/elasticsearch
```
3. **多数据路径**（ES 7.13 已废弃多 path.data，改用多节点代替）
4. **RAID 选择**：
   - RAID 0：高 IO，但无冗余（靠 ES 副本保证数据安全）
   - RAID 5/6：有校验开销，写入性能差于 RAID 0
   - **推荐 RAID 0 + ES 副本**
5. **调整 IO 调度**：SSD 使用 `noop` 或 `none` 调度，减少内核调度开销
6. **关闭磁盘预读**（顺序读多的场景）：Lucene 随机 IO 多，预读帮助有限

---

###### 14. 如何优化网络性能？

1. **降低跨分片流量**：
   - `routing` 优化，让查询只打少数分片
   - `_source filtering` 只返回需要的字段

2. **压缩传输**：
```yaml
http.compression: true           # HTTP 响应压缩
transport.compress: true         # 节点间 transport 传输压缩
```

3. **调大网络缓冲区**：
```yaml
transport.tcp.send_buffer_size: 512kb
transport.tcp.receive_buffer_size: 512kb
```

4. **协调节点独立部署**：数据节点之间的内部通信（transport）和客户端 HTTP 通信分离，减少互相干扰

5. **批量请求**：Bulk / MGET / msearch 替代逐条请求，减少网络往返（RTT）

###### 15. 什么是慢查询日志？如何利用它优化性能？

慢查询日志（Slow Log）记录超过阈值的查询，是性能优化的重要诊断工具。

**配置慢查询日志：**
```json
PUT /my_index/_settings
{
  "index.search.slowlog.threshold.query.warn":  "5s",    // 超过5秒记录 WARN
  "index.search.slowlog.threshold.query.info":  "2s",    // 超过2秒记录 INFO
  "index.search.slowlog.threshold.query.debug": "1s",
  "index.search.slowlog.threshold.fetch.warn":  "1s",    // Fetch 阶段慢日志
  "index.search.slowlog.level": "info"
}
```

也可以配置写入慢日志：
```json
{
  "index.indexing.slowlog.threshold.index.warn": "5s",
  "index.indexing.slowlog.threshold.index.info": "2s"
}
```

**分析慢日志步骤：**
1. 找到慢查询的 DSL
2. 用 `Profile API` 深入分析每个查询阶段耗时
3. 常见问题：大量 should 子句、深分页、脚本查询、fielddata 加载

###### 16. 如何处理大字段存储问题？

大字段（如长文章正文、Base64 编码的图片）存储在 `_source` 中会导致磁盘占用大、读取传输慢。

**解决方案：**

1. **禁用 `_source` 全量存储，只存需要的字段**：
```json
PUT /articles
{
  "mappings": {
    "_source": {
      "excludes": ["raw_html", "base64_image"]  // 排除大字段
    }
  }
}
```

2. **大字段截断**：超过一定长度的字段不参与索引
```json
"content": { "type": "text", "ignore_above": 10000 }
```
注意：`ignore_above` 对 keyword 有效，text 字段用 `index_options` 控制

3. **外部存储大对象**：大字段（原文、图片）存 OSS/MinIO，ES 只存 URL

4. **压缩存储**：ES 默认对 `_source` 做 LZ4 压缩，也可以改用更高压缩比：
```yaml
index.codec: best_compression   # 使用 DEFLATE，压缩率更高但读取略慢
```

###### 17. 如何优化索引的 Mapping 设计？

Mapping 设计是性能优化的基础，一旦上线就很难改（需要 Reindex），要在初期设计好。

**核心原则：**

1. **text vs keyword 按需选择**：全文搜索 → `text`，精确匹配/聚合/排序 → `keyword`
2. **关闭无用索引**：不需要搜索的字段 `"index": false`（节省存储和 CPU）
3. **关闭无用 doc_values**：不需要聚合/排序的 text 字段 `"doc_values": false`（节省磁盘）
4. **禁用 `_all` 字段**（ES 6.x 已默认禁用）
5. **合理使用 nested**：object 扁平化存储，nested 独立存储，按查询需求选择
6. **数值类型精准**：能用 `integer` 就不用 `long`，能用 `float` 就不用 `double`
7. **避免 mapping 爆炸**：动态映射会导致字段数爆炸，生产环境设置 `dynamic: strict` 拒绝未定义字段

```json
PUT /products
{
  "mappings": {
    "dynamic": "strict",              // 拒绝未定义字段
    "properties": {
      "id":       { "type": "keyword" },
      "name":     { "type": "text", "analyzer": "ik_max_word",
                    "fields": { "keyword": { "type": "keyword" } } },
      "price":    { "type": "float" },
      "status":   { "type": "keyword", "doc_values": true },
      "tags":     { "type": "keyword" },
      "content":  { "type": "text", "index": true, "doc_values": false },
      "raw_data": { "type": "text",  "index": false, "doc_values": false }
    }
  }
}
```

###### 18. 如何使用 Profile API 分析查询性能？

Profile API 能看到查询的每个执行阶段的耗时，是定位慢查询瓶颈的利器。

```json
POST /products/_search
{
  "profile": true,    // 开启 Profile
  "query": {
    "bool": {
      "must": [ { "match": { "name": "手机" } } ],
      "filter": [ { "term": { "status": "active" } } ]
    }
  }
}
```

**Profile 返回结构：**
```json
{
  "profile": {
    "shards": [
      {
        "id": "[node-1][products][0]",    // 分片 ID
        "searches": [
          {
            "query": [
              {
                "type": "BooleanQuery",
                "time_in_nanos": 1200000,   // 总耗时
                "breakdown": {
                  "score": 300000,          // 评分耗时
                  "create_weight": 50000,   // 构建权重耗时
                  "next_doc": 700000        // 遍历文档耗时
                },
                "children": [...]           // 子查询的耗时明细
              }
            ],
            "collector": [...],             // Collector（排序/聚合）耗时
            "rewrite_time": 5000            // 查询重写耗时
          }
        ]
      }
    ]
  }
}
```

**常见瓶颈识别：**
- `score` 耗时高 → 评分计算复杂，考虑 `function_score` 优化或缓存
- `next_doc` 耗时高 → 文档太多需要遍历，考虑过滤条件更精确
- 某个子查询耗时占大头 → 针对该子查询优化

> **本章追问：ES 写入性能和搜索性能如何取得平衡？**
>
> **答**：两者本质是在同一套基础设施上竞争资源，优化一个往往影响另一个。
>
> | 参数 | 偏向写入 | 偏向搜索 |
> |------|---------|---------|
> | `refresh_interval` | 30s+ 或 -1 | 1s |
> | 副本数 | 0（批量导入期） | 1~2（分担读压力） |
> | `translog.durability` | async | request |
> | Segment 数 | 多（少合并） | 少（多合并） |
>
> **实际做法**：
> - **冷数据**（历史日志）：批量导入时优先写入性能，导完后 force merge + 恢复副本，进入只读搜索模式
> - **热数据**（实时日志/商品）：写入和搜索并发，用 ILM 管理 hot-warm-cold 分层，热层优化写入，暖层优化搜索
