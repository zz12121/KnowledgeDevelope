###### 1. 如何诊断和解决集群性能下降问题？

**排查路径：表象 → 根因 → 解决方案**

**Step 1：检查集群整体健康**

```bash
# 集群健康状态
GET /_cluster/health?pretty

# 节点统计（重点看 JVM、线程池、索引速度）
GET /_nodes/stats?pretty

# 热线程（找出 CPU 占用高的线程）
GET /_nodes/hot_threads
```

**Step 2：检查关键指标**

```bash
# 索引统计（查询耗时、写入速度）
GET /_stats?pretty

# 线程池队列积压
GET /_cat/thread_pool?v&h=name,active,queue,rejected&s=queue:desc

# 查询慢日志
GET /my-index/_settings
PUT /my-index/_settings
{
  "index.search.slowlog.threshold.query.warn": "5s",
  "index.search.slowlog.threshold.query.info": "1s"
}
```

**常见原因及解决：**

| 现象 | 可能原因 | 解决方案 |
|------|---------|---------|
| 查询慢 | 深分页/脚本查询/无 filter 缓存 | 改 search_after，增加 filter，避免脚本 |
| 写入慢 | refresh_interval 太短/bulk size 太小 | 调大 refresh_interval，优化 bulk size |
| JVM GC 频繁 | 堆内存不足/fielddata 爆满 | 增加堆内存，限制 fielddata，开启 circuit breaker |
| 线程池 rejected | 并发请求太多 | 扩容节点，增加队列 size，限流 |
| Segment 数量多 | 合并速度跟不上 | 调整 merge policy，低峰期强制 forcemerge |

###### 2. 如何处理索引无法创建或删除的问题？

**原因排查：**

**① 磁盘水位超限（最常见）**

```bash
# 检查磁盘使用
GET /_cat/allocation?v&h=node,disk.used_percent,disk.avail

# 磁盘超过 85%（high watermark），ES 停止分配新分片
# 磁盘超过 90%（flood stage），索引变只读
```

**解决：**

```json
// 临时调整水位（紧急情况）
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.watermark.low": "90%",
    "cluster.routing.allocation.disk.watermark.high": "95%",
    "cluster.routing.allocation.disk.watermark.flood_stage": "97%"
  }
}

// 解除只读状态
PUT /my-index/_settings
{"index.blocks.read_only_allow_delete": null}
```

**② 集群 Block**

```bash
GET /_cluster/settings
# 检查是否有 cluster.blocks.read_only: true
PUT /_cluster/settings
{"transient": {"cluster.blocks.read_only": false}}
```

**③ 索引数量限制**

```json
// 集群默认最多 1000 个分片/节点
PUT /_cluster/settings
{
  "persistent": {
    "cluster.max_shards_per_node": 2000
  }
}
```

###### 3. 如何解决分片分配失败的问题？

**分片 unassigned 是集群变黄/红的主要原因。**

**查看未分配原因：**

```bash
# 查看未分配分片
GET /_cat/shards?v&h=index,shard,prirep,state,unassigned.reason&s=state

# 详细诊断原因（最有用的命令）
GET /_cluster/allocation/explain
{
  "index": "my-index",
  "shard": 0,
  "primary": true
}
```

**常见原因及解决：**

**① 节点数量不足（副本分片无处分配）**

```json
// 临时减少副本数
PUT /my-index/_settings
{"index.number_of_replicas": 0}
```

**② 磁盘空间不足**（见上题）

**③ 分片分配过滤**

```json
// 检查是否有 allocation filter 导致分片无法分配
GET /my-index/_settings?filter_path=*.settings.index.routing.allocation
// 清除限制
PUT /my-index/_settings
{"index.routing.allocation.require._name": null}
```

**④ 数据损坏，主分片丢失**

```json
// 高风险操作：强制分配空分片（会丢数据）
POST /_cluster/reroute
{
  "commands": [{
    "allocate_empty_primary": {
      "index": "my-index",
      "shard": 0,
      "node": "node-1",
      "accept_data_loss": true
    }
  }]
}
```

###### 4. 如何处理查询超时问题？

**方案一：设置查询超时（防止拖垮集群）**

```json
GET /my-index/_search?timeout=5s
{
  "query": {"match_all": {}}
}
```

设置超时后，ES 会返回已收集到的部分结果，并附上 `"timed_out": true`。

**方案二：排查慢查询**

```bash
# 开启慢查询日志
PUT /my-index/_settings
{
  "index.search.slowlog.threshold.query.warn": "3s",
  "index.search.slowlog.threshold.query.info": "1s",
  "index.search.slowlog.threshold.fetch.warn": "1s"
}

# 查看慢查询日志
tail -f /var/log/elasticsearch/my-cluster_index_search_slowlog.json
```

**方案三：用 Profile API 定位慢查询原因**

```json
GET /my-index/_search
{
  "profile": true,
  "query": {
    "bool": {
      "must": [{"match": {"title": "手机"}}],
      "filter": [{"range": {"price": {"gte": 1000}}}]
    }
  }
}
```

**常见慢查询原因：**
- `wildcard` / `regexp` 查询（不走倒排索引，全扫描）→ 改用 `keyword` + 前缀查询
- 深分页（`from: 10000`）→ 改用 `search_after`
- Fielddata 加载慢（`text` 字段聚合）→ 改用 `keyword` 子字段
- 复杂脚本评分 → 减少脚本使用，改用 `function_score` 内置函数

###### 5. 如何解决内存溢出（OOM）问题？

**ES JVM 内存溢出通常由以下几种原因触发：**

**① Fielddata 爆内存（最常见）**

```bash
# 查看 fielddata 使用
GET /_cat/fielddata?v

# 限制 fielddata 大小
PUT /_cluster/settings
{
  "persistent": {
    "indices.fielddata.cache.size": "20%"
  }
}

# 清除 fielddata 缓存
POST /_cache/clear?fielddata=true
```

根本解决：`text` 类型字段避免聚合，改用 `keyword` 子字段。

**② 堆内存设置不合理**

```bash
# JVM heap 设置（jvm.options）
# 规则：不超过物理内存 50%，不超过 32GB（超过 32GB 关闭 Compressed OOPs，反而更耗内存）
-Xms16g
-Xmx16g  # 必须 Xms == Xmx，防止堆扩缩带来的 GC
```

**③ 大量聚合请求导致 Circuit Breaker 触发**

```bash
# 查看 Circuit Breaker 状态
GET /_nodes/stats/breaker

# 调整熔断器限制
PUT /_cluster/settings
{
  "persistent": {
    "indices.breaker.request.limit": "60%",
    "indices.breaker.fielddata.limit": "40%",
    "indices.breaker.total.limit": "95%"
  }
}
```

**④ GC 频繁导致节点假死**

```bash
# 分析 GC 日志
grep "GC overhead" /var/log/elasticsearch/gc.log | tail -20

# 优化措施
# 1. 升级到 G1GC（ES 7.x+ 默认）
# 2. 减少不必要的 Fielddata 加载
# 3. 增加节点数量，分散内存压力
```

###### 6. 如何处理数据丢失或损坏问题？

**预防优先于恢复！**

**预防手段：**
- 副本数 ≥ 1，跨 AZ 部署
- 定期快照备份（SLM 自动快照策略）
- 不要在生产环境使用 `number_of_replicas: 0`

**数据恢复步骤：**

**① 从快照恢复**

```json
// 查看可用快照
GET /_snapshot/my-backup/_all

// 恢复整个索引
POST /_snapshot/my-backup/snapshot-2024-03-15/_restore
{
  "indices": "orders-2024-03",
  "rename_pattern": "(.+)",
  "rename_replacement": "restored-$1",
  "index_settings": {
    "index.number_of_replicas": 1
  }
}
```

**② 强制分配已有数据的分片（分片未分配但数据存在）**

```json
POST /_cluster/reroute
{
  "commands": [{
    "allocate_stale_primary": {
      "index": "orders",
      "shard": 0,
      "node": "node-1",
      "accept_data_loss": false
    }
  }]
}
```

**③ 从数据库重新同步（Canal/Logstash Reindex）**

如果有数据库作为 Source of Truth，直接重新全量导入比手动恢复更可靠。

###### 7. 如何解决集群不同步问题？

**"集群不同步"通常指副本分片数据落后于主分片，或节点间状态不一致。**

**查看同步状态：**

```bash
# 查看分片同步状态
GET /_cat/shards?v&h=index,shard,prirep,state,docs,store,node

# 查看集群状态（pending tasks）
GET /_cluster/pending_tasks

# 查看节点差异
GET /_nodes/stats/indices?filter_path=nodes.*.indices.docs
```

**副本数据落后（recovery 中）：**
- 正常现象：节点重启或新增节点时副本会从主分片同步数据
- 查看恢复进度：`GET /_cat/recovery?v&active_only=true`
- 耐心等待，确保网络带宽充足，不要频繁重启节点

**Master 节点频繁切换导致状态混乱：**

```bash
# 查看 Master 选举历史
GET /_cat/master?v

# 排查网络分区（GC Pause 也会导致节点被踢出集群）
GET /_nodes/stats/jvm?filter_path=nodes.*.jvm.gc
```

**重置集群状态（极端情况，谨慎操作）：**

```bash
# 单个节点无法加入集群，清除本地集群状态重新加入
# 停止节点，删除 data 目录下的 _state 目录（不是全部数据），再启动
rm -rf /var/lib/elasticsearch/nodes/0/cluster_state/
```

###### 8. 如何处理索引只读状态？

**索引只读是 ES 的自我保护机制**，通常由磁盘空间不足触发（flood stage watermark，默认 95%）。

**判断是否只读：**

```bash
GET /my-index/_settings?filter_path=*.settings.index.blocks
```

如果返回 `"read_only_allow_delete": "true"`，说明索引已被设为只读。

**解除只读的完整步骤：**

```bash
# Step 1：清理磁盘（删除旧索引/快照/临时文件）
DELETE /old-logs-2023-*

# Step 2：查看磁盘使用率
GET /_cat/allocation?v

# Step 3：恢复水位到正常值（或临时调高）
PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.watermark.flood_stage": "97%"
  }
}

# Step 4：解除索引只读状态
PUT /my-index/_settings
{"index.blocks.read_only_allow_delete": null}

# Step 5：解除全局只读（如有）
PUT /_cluster/settings
{"transient": {"cluster.blocks.read_only": false}}
```

**记忆口诀**：磁盘超 95% → 自动只读 → 清磁盘 → 调水位 → 解 block → 恢复写入。

###### 9. 如何解决节点无法加入集群的问题？

**排查步骤：**

**① 检查 elasticsearch.log**

```bash
tail -100 /var/log/elasticsearch/my-cluster.log | grep -i "error\|warn\|failed"
```

**② 检查集群配置一致性**

```yaml
# 所有节点的 cluster.name 必须相同
cluster.name: production-cluster

# discovery.seed_hosts 必须包含正确的种子节点地址
discovery.seed_hosts: ["192.168.1.10:9300", "192.168.1.11:9300", "192.168.1.12:9300"]

# initial_master_nodes 只在第一次启动集群时设置，集群形成后注释掉
# cluster.initial_master_nodes: ["node-1", "node-2", "node-3"]
```

**③ 检查网络连通性**

```bash
# 验证 9300 端口（节点间通信）可达
nc -zv 192.168.1.10 9300
telnet 192.168.1.10 9300

# 检查防火墙
iptables -L -n | grep 9300
```

**④ TLS 证书问题（8.x 默认开启 transport TLS）**

```bash
# 检查证书是否过期
openssl x509 -in /etc/elasticsearch/certs/elastic-certificates.p12 -noout -dates

# 检查证书 SAN 是否包含节点 IP/hostname
openssl x509 -in cert.pem -noout -text | grep "Subject Alternative Name" -A 2
```

**⑤ 版本不兼容**：ES 不支持跨大版本节点混合运行，滚动升级时只允许相邻版本。

###### 10. 如何排查慢查询和慢写入问题？

**慢查询排查：**

**① 开启慢查询日志**

```json
PUT /my-index/_settings
{
  "index.search.slowlog.threshold.query.warn": "5s",
  "index.search.slowlog.threshold.query.info": "2s",
  "index.search.slowlog.threshold.query.debug": "500ms",
  "index.search.slowlog.level": "info"
}
```

慢查询日志位置：`/var/log/elasticsearch/<cluster>_index_search_slowlog.json`

**② Profile API 精确定位**

```json
GET /my-index/_search
{
  "profile": true,
  "query": {
    "bool": {
      "must": [{"match": {"content": "elasticsearch"}}],
      "filter": [{"term": {"status": "active"}}]
    }
  }
}
// 关注 profile.shards[].searches[].query[].time_in_nanos
```

**③ 常见慢查询原因**

| 场景 | 原因 | 优化方案 |
|------|------|---------|
| `wildcard` / `regexp` | 暴力扫描，不走索引 | 用 `prefix` 或拆分 `keyword` |
| 深分页 `from+size` | 每个分片都要排序前 N 条 | 改 `search_after` 游标翻页 |
| `text` 字段聚合 | 需要加载 Fielddata | 改用 `keyword` 子字段 |
| 大量脚本评分 | 每个文档执行脚本 | 用 `function_score` 内置函数 |
| 返回 `_source` 超大 | 大字段传输慢 | 用 `_source: ["field1","field2"]` 过滤 |

**慢写入排查：**

**① 开启写入慢日志**

```json
PUT /my-index/_settings
{
  "index.indexing.slowlog.threshold.index.warn": "10s",
  "index.indexing.slowlog.threshold.index.info": "5s"
}
```

**② 常见慢写入原因及解决**

| 原因 | 解决方案 |
|------|---------|
| `refresh_interval` 太短（默认 1s） | 批量写入时设 `-1`，写完后手动 refresh |
| Bulk size 太小（每次几条） | 调整为 5-15MB / 1000-5000 条 |
| 副本数多（每条需同步到所有副本） | 初始导入时临时设 `replicas: 0`，导完恢复 |
| Translog 同步策略（`fsync` 每请求） | 改为 `async`（有丢数据风险，权衡使用） |
| Mapping 有大量 `nested` 字段 | 嵌套文档写入实际是多个 Lucene 文档，控制 nested 数量 |

> **本章追问：生产环境 Elasticsearch 日常巡检应该检查哪些项目？**
>
> **每日巡检清单：**
> - `GET /_cluster/health` → 确认 green，yellow 需立即处理
> - `GET /_cat/nodes?v&h=name,heap.percent,ram.percent,cpu,disk.used_percent` → JVM 堆 <75%，磁盘 <80%
> - `GET /_cat/shards?v&s=state&h=index,shard,prirep,state` → 无 UNASSIGNED 分片
> - 检查慢查询日志 → 关注 warn 级别的查询
> - 检查 rejected 线程数 → `GET /_cat/thread_pool?v&h=name,rejected`
>
> **每周巡检：**
> - 检查快照是否正常执行（`GET /_slm/stats`）
> - 检查 ILM 策略执行状态（`GET /_ilm/explain`）
> - 评估索引分片大小是否合理（单个分片建议 10-50GB）
> - 检查各节点负载均衡情况（`GET /_cat/allocation?v`）
>
> **记忆口诀**：颜色（green）→ 堆（<75%）→ 磁盘（<80%）→ 分片（无 unassigned）→ 队列（无 rejected）→ 快照（正常执行）。
