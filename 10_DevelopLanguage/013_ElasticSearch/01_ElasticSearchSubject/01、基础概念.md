# 一、基础概念

> 双链：[[38_ElasticSearchKnowledge/01_核心概念与数据模型]]

---

###### 1. 为什么要使用 Elasticsearch？它有哪些核心特性？

用 ES 的根本原因是：**关系型数据库做全文搜索太慢**，MySQL 的 `LIKE '%keyword%'` 走不了索引，百万级数据就扛不住了。ES 基于 Lucene 构建，天生为搜索而生。

**核心特性：**

1. **倒排索引**：把"文档→词"的关系反转成"词→文档列表"，查关键词就是 O(1) 级别的哈希查找
2. **分布式**：数据自动分片，横向扩容，单集群可以处理 PB 级数据
3. **近实时（NRT）**：写入后默认 1 秒内可搜索（refresh 机制），不是严格实时但够用
4. **RESTful API**：所有操作都是 HTTP JSON，上手门槛低
5. **丰富的查询 DSL**：全文搜索/聚合/地理位置/向量搜索，一站式解决
6. **高可用**：副本机制，主分片挂了副本自动提升

**适用场景口诀**：搜索 + 日志 + 分析，这三件事找 ES 准没错。

---

###### 2. Elasticsearch 中的索引、文档、字段分别是什么概念？

直接用类比理解：

| ES 概念 | 类比 MySQL |
|---------|-----------|
| 索引（Index） | 数据库表（Table） |
| 文档（Document） | 一行记录（Row） |
| 字段（Field） | 列（Column） |

**细化说明：**

- **索引**：一类数据的集合，比如 `products`、`logs-2026-03`，有自己的 Mapping 和 Settings
- **文档**：JSON 格式，是 ES 中数据存储的最小单位，每个文档有唯一的 `_id`
- **字段**：文档中的键值对，字段类型由 Mapping 决定（text/keyword/integer/date 等）

> 注意：ES 7.x 开始废弃了 `_type`（类型），8.x 彻底移除。现在一个索引就对应一种文档类型。

---

###### 3. ElasticSearch 中的集群、节点、索引、文档、类型是什么？

这是 ES 的五层概念体系：

```
集群（Cluster）
  └── 节点（Node）× N
        └── 索引（Index）× N
              └── 分片（Shard）× N
                    └── 文档（Document）× N
                          └── 字段（Field）× N
```

- **集群**：多个节点组成的整体，通过集群名（`cluster.name`）来识别归属，共同对外提供服务
- **节点**：一个 ES 进程就是一个节点，每个节点有唯一名字和 UUID，有不同角色（主节点/数据节点/协调节点）
- **索引**：文档的逻辑容器，对应实际数据存储在各个分片上
- **类型（Type）**：历史遗留概念，7.x 废弃，8.x 删除。早期允许一个索引下有多种类型，现在统一用 `_doc`
- **文档**：JSON 对象，带有 `_index`、`_id`、`_version`、`_source` 等元数据字段

###### 4. Elasticsearch 中的倒排索引是什么？

倒排索引是 ES 快速全文搜索的核心数据结构，跟正向索引的方向完全相反。

**正向索引**（传统）：
```
文档1 → ["苹果", "手机", "华为"]
文档2 → ["手机", "三星", "屏幕"]
```
查"手机"要遍历所有文档，慢。

**倒排索引**：
```
"苹果" → [文档1]
"手机" → [文档1, 文档2]
"华为" → [文档1]
"三星" → [文档2]
```
查"手机"直接命中 → [文档1, 文档2]，O(1) 级别。

**倒排索引的组成：**
1. **词项字典（Term Dictionary）**：所有词项的排序列表，支持二分查找
2. **倒排列表（Posting List）**：每个词项对应的文档ID列表，还包含词频（TF）、位置、偏移量
3. **词项索引（Term Index）**：词典的索引（FST 结构），放在内存里加速查找

**文本分析流程：**
```
原始文本 → 分析器（Analyzer） → [字符过滤器 → 分词器 → 词项过滤器] → 词项列表 → 写入倒排索引
```

---

###### 5. Elasticsearch 中的分析器是什么？

分析器（Analyzer）是把原始文本转成索引词项的处理链，写入和搜索时都会用到。

**三个组成部分：**

```
原始文本
   ↓
字符过滤器（Character Filter）  ← 比如：去掉 HTML 标签
   ↓
分词器（Tokenizer）              ← 核心：把文本切分成词项（Token）
   ↓
词项过滤器（Token Filter）       ← 比如：小写化、去停用词、同义词扩展
   ↓
词项列表（Terms）
```

**常用分析器：**

| 分析器 | 适用场景 |
|--------|---------|
| `standard` | 默认，按 Unicode 分词，适合英文 |
| `whitespace` | 按空格分词，不小写化 |
| `simple` | 按非字母切分，全部小写 |
| `ik_max_word` | 中文分词（需安装 IK 插件），最细粒度 |
| `ik_smart` | 中文分词，最粗粒度（智能分词） |

> 中文搜索必须装 IK 分词器，不然 `standard` 会把汉字一个字一个字地切开。

###### 6. 什么是映射（Mapping）？静态映射和动态映射有什么区别？

Mapping 就是索引的**表结构定义**，决定每个字段的数据类型和索引方式。

**动态映射（Dynamic Mapping）**：
- ES 默认行为，第一次写入数据时自动推断字段类型
- 比如写入 `{"age": 25}` → 自动映射为 `integer`
- 坑：`"2026-03-15"` 会被推断为 `date`，`"1234"` 可能被推断为 `long`，类型推断可能不符合预期
- 生产环境**不推荐**完全依赖动态映射

**静态映射（Explicit Mapping）**：
- 手动定义每个字段的类型和参数，精确控制
- 最佳实践：建索引时先定义好 Mapping，避免动态映射带来的意外类型

```json
PUT /products
{
  "mappings": {
    "properties": {
      "name":    { "type": "text",    "analyzer": "ik_max_word" },
      "price":   { "type": "float" },
      "status":  { "type": "keyword" },
      "created": { "type": "date",    "format": "yyyy-MM-dd" }
    }
  }
}
```

> 关键区分：`text` 会被分词，用于全文搜索；`keyword` 不分词，用于精确匹配、排序、聚合。

###### 7. Elasticsearch 支持哪些数据类型？

按大类分：

**核心类型：**
- `text`：全文检索，会分词
- `keyword`：精确值，不分词（品牌、状态、标签等）
- `integer/long/float/double`：数值
- `boolean`：布尔
- `date`：日期，支持多种 format

**复杂类型：**
- `object`：嵌套 JSON 对象（扁平化存储，跨对象查询可能出现笛卡尔积问题）
- `nested`：嵌套对象（独立存储，支持精准的对象内部查询）
- `join`：父子文档（同一索引内定义父子关系）

**特殊类型：**
- `geo_point`：地理坐标（纬度+经度），支持地理距离查询
- `geo_shape`：多边形等复杂地理形状
- `dense_vector`：稠密向量，用于向量相似度搜索（ANN）
- `ip`：IP 地址
- `completion`：搜索建议（自动补全专用）
- `percolator`：反向搜索（把查询存成文档）

> 记忆口诀：**文本分 text/keyword，数值看精度，复杂对象用 nested，向量用 dense_vector。**

###### 8. 什么是 Elasticsearch 的 RESTful API？

ES 所有操作都通过 HTTP 请求完成，格式统一：

```
HTTP方法  /索引名/_操作  请求体(JSON)
```

**常用操作速查：**

```bash
# 创建/覆盖文档（指定 ID）
PUT /products/_doc/1
{ "name": "iPhone 15", "price": 5999 }

# 自动生成 ID
POST /products/_doc
{ "name": "小米手机", "price": 2999 }

# 查询单个文档
GET /products/_doc/1

# 更新文档（部分更新）
POST /products/_update/1
{ "doc": { "price": 5499 } }

# 删除文档
DELETE /products/_doc/1

# 搜索
POST /products/_search
{ "query": { "match": { "name": "手机" } } }

# 批量操作
POST /_bulk
{ "index": { "_index": "products", "_id": "2" } }
{ "name": "华为P60", "price": 4999 }
```

###### 9. Elasticsearch 与传统关系型数据库的区别是什么？

| 维度 | Elasticsearch | MySQL |
|------|--------------|-------|
| 数据模型 | JSON 文档，Schema 灵活 | 表结构，Schema 严格 |
| 查询语言 | Query DSL（JSON over HTTP） | SQL |
| 全文搜索 | 原生支持，倒排索引 | 需要 FULLTEXT 索引，功能弱 |
| 事务 | 不支持（文档级原子操作） | 完整 ACID 事务 |
| 关联查询 | 不支持 JOIN（用 nested/join type 模拟） | 原生 JOIN |
| 一致性 | 最终一致性（NRT） | 强一致性 |
| 适用场景 | 搜索、日志、分析 | 业务数据、事务性操作 |
| 扩展性 | 原生分布式，水平扩展 | 垂直扩展为主，分库分表复杂 |

**结论**：ES 和 MySQL 是互补关系，不是替代关系。业务数据存 MySQL，搜索/日志/分析用 ES，MySQL → ES 的数据同步用 Canal / Logstash / 自研同步服务。

###### 10. Elasticsearch 的版本兼容性如何？

**主要版本里程碑：**

- **5.x**：引入 Query DSL 改进，废弃 `_all` 字段
- **6.x**：一个索引只能有一种类型（单 type 限制）
- **7.x**：废弃 `_type`（统一用 `_doc`），引入新的选举机制（去掉 ZooKeeper 依赖），默认使用 BKD Tree 优化数字范围查询
- **8.x**：彻底删除 `_type`，默认开启安全认证（TLS + 身份验证），向量搜索能力大幅增强

**兼容性注意事项：**
1. 跨大版本升级需要逐版本跳（6→7→8，不能跳跃）
2. 客户端版本要与服务端版本匹配（Java High Level REST Client 在 7.15+ 已标记废弃，推荐用 Elasticsearch Java API Client）
3. Kibana 版本必须与 ES 版本一致

> **本章追问：ES 的 `text` 和 `keyword` 类型分别在什么情况下使用？能混用吗？**
>
> **答**：这是高频考点。
> - `text`：用于需要**全文搜索**的字段，比如商品名称、文章正文。写入时会被分析器分词，支持模糊搜索。**不支持聚合、排序**（因为分词后词项太多）。
> - `keyword`：用于需要**精确匹配、聚合、排序**的字段，比如品牌、状态码、标签。写入时原样保存，不分词。
> - **混用**：可以，用 `fields` 多字段映射，同一字段同时有 `text` 和 `keyword` 两种索引：
> ```json
> "title": {
>   "type": "text",
>   "analyzer": "ik_max_word",
>   "fields": {
>     "keyword": { "type": "keyword", "ignore_above": 256 }
>   }
> }
> ```
> 搜索用 `title`（text），聚合/排序用 `title.keyword`（keyword）。
