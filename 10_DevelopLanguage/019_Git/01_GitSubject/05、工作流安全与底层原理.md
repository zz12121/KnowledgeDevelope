# 05、工作流、安全与底层原理

## Q1. GitFlow vs GitHub Flow vs Trunk-Based 对比

**GitFlow**（适合有版本计划的软件产品）：
```
main       ← 生产版本（每次合并都打 tag）
develop    ← 集成分支
feature/*  ← 从 develop 分，完成后合回 develop
release/*  ← 发版准备（从 develop 分，完成后合 main+develop）
hotfix/*   ← 紧急修复（从 main 分，合 main+develop）
```
✅ 适合：版本化交付（lib/SDK/需要维护多版本）  
❌ 缺点：流程复杂，分支多，合并频繁

**GitHub Flow**（适合 SaaS 持续部署）：
```
main ← 永远可部署
feature/* ← 任意功能分支，PR 合并后自动部署
```
✅ 适合：Web 应用，一天多次部署  
❌ 缺点：没有明确的版本管理，不适合需要维护旧版本的产品

**Trunk-Based Development**（大型互联网，高频发版）：
```
main（trunk）← 所有人提交（或短命特性分支）
Feature Flag  ← 控制功能可见性，代码可以提交但功能未上线
```
✅ 适合：Google/Facebook 规模，每天几百次 CI 流水线  
❌ 缺点：需要强 CI 基础设施和 Feature Flag 框架支撑

## Q2. Pull Request（PR）/ Merge Request（MR）是什么？

PR 是代码合并请求 + 代码审查的工作流：

```
1. 开发者在功能分支完成开发
   git push origin feature/login

2. 在 GitHub/GitLab 上创建 PR
   from: feature/login → into: main

3. Code Review
   - 审查者评论、建议修改
   - CI 自动运行测试、代码扫描
   - 开发者根据反馈更新代码（push 新 commit 到同一分支，PR 自动更新）

4. 审批通过 → 合并
   - Merge commit（保留分支历史）
   - Squash and merge（压缩为一个 commit，推荐）
   - Rebase and merge（线性历史）

5. 合并后自动删除功能分支（GitHub 有此选项）
```

**PR 最佳实践**：
- 每个 PR 只做一件事（Single Responsibility）
- PR 太大（>500行）需要拆分，方便 Review
- 写清楚 PR 描述：做了什么、为什么、如何测试
- 测试覆盖率不能下降

## Q3. 如何保护敏感信息不被提交？

**预防措施**：
```bash
# 1. 配置 .gitignore（最重要）
application-prod.yml
*.env
secrets.properties
.env.local

# 2. 使用环境变量替代硬编码
# 不好：
spring.datasource.password=mysecretpassword

# 好的：
spring.datasource.password=${DB_PASSWORD}

# 3. 使用 git-secrets 工具（检测并阻止包含密码的提交）
git secrets --install
git secrets --register-aws    # 内置 AWS 密钥检测规则
```

**如果已经提交了敏感信息**：
```bash
# 方案1：使用 git filter-repo（推荐，替代已废弃的 filter-branch）
pip install git-filter-repo
git filter-repo --path secrets.properties --invert-paths

# 方案2：BFG Repo Cleaner（更快速）
java -jar bfg.jar --delete-files secrets.properties
java -jar bfg.jar --replace-text passwords.txt   # 替换文件中的密码字符串

# 操作后必须：
git push --force-with-lease --all   # 强推覆盖所有分支
# 立即轮换泄露的密钥！（历史记录虽然删了，但可能已被 fork 或爬取）
```

## Q4. SSH 密钥如何配置？

```bash
# 生成 SSH 密钥对（推荐 Ed25519，比 RSA 更安全更快）
ssh-keygen -t ed25519 -C "your.email@example.com"
# 私钥：~/.ssh/id_ed25519
# 公钥：~/.ssh/id_ed25519.pub

# 将公钥添加到 GitHub/GitLab（Settings → SSH Keys）
cat ~/.ssh/id_ed25519.pub

# 测试连接
ssh -T git@github.com

# 多账号配置（如个人 GitHub + 公司 GitLab）
# ~/.ssh/config
Host github.com
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_github

Host gitlab.company.com
  HostName gitlab.company.com
  User git
  IdentityFile ~/.ssh/id_ed25519_company
```

## Q5. Git 的底层数据结构是什么？

Git 本质上是一个**内容寻址的文件系统**，一切都存储为对象，用 SHA-1 哈希值（40 位十六进制）标识：

**四种 Git 对象**：
```
blob（文件内容）
  → 存储文件内容，SHA-1 = hash(内容)

tree（目录结构）
  → 包含 blob 和 tree 的引用（文件名+权限+SHA-1）

commit（提交）
  → 包含：tree SHA-1, parent commit SHA-1(s), 作者/提交者信息, 时间戳, commit message

tag（附注标签）
  → 包含：target SHA-1, 标签信息, 打标签者信息
```

```
一次提交的对象树：

commit abc1234
  ├─ tree def5678（根目录）
  │    ├─ blob ghi9012（README.md 内容）
  │    ├─ tree jkl3456（src/目录）
  │    │    └─ blob mno7890（Main.java 内容）
  │    └─ ...
  ├─ parent pqr2345（上一次 commit）
  └─ author/message/timestamp
```

**为什么用 SHA-1**：
- 内容一样的文件 → 同一个 blob（自动去重）
- 任何数据改动都会产生不同 SHA-1（完整性保证）
- 哈希寻址，查找极快

> **追问：Git 如何存储文件修改历史？是差量存储还是全量存储？**
> 松散对象（.git/objects/）是全量存储，每次 commit 都保存完整文件内容的 blob。但 Git 会定期做 GC（`git gc`），将松散对象打包成 packfile，packfile 中会使用**delta 压缩**（对相似内容做差量存储），所以最终存储效率比较高。这也是为什么 `git clone` 的仓库比想象中小的原因。

## Q6. Git 三路合并算法是什么？

三路合并（Three-Way Merge）是 Git 自动合并的基础算法：

```
      Base（共同祖先）
      /              \
Branch A            Branch B
```

```
文件的某一行：
  Base:     int x = 10;
  Branch A: int x = 20;    ← A 修改了
  Branch B: int x = 10;    ← B 没改

三路合并结果：int x = 20;  （A 的修改被采用，因为 B 没有改变它）
```

```
冲突情况：
  Base:     int x = 10;
  Branch A: int x = 20;    ← A 修改了
  Branch B: int x = 30;    ← B 也修改了，且改的不一样

三路合并：CONFLICT！无法自动解决，需要手动处理
```

**关键点**：三路合并使用**共同祖先（Base）**作为参照，比仅比较两个版本（二路合并）更智能。只要两方中有一方没有修改某处，就不会产生冲突，即使两个分支都有大量改动，只有真正"针对同一基础做了不同修改"才报冲突。

## Q7. 常见 Git 故障排查

```bash
# "detached HEAD" 状态
# 出现原因：checkout 了一个 commit 或 tag（不是分支名）
git log --oneline             # 确认当前位置
git checkout -b new-branch    # 创建新分支保存当前状态
git switch main               # 或切回正常分支

# "refusing to merge unrelated histories"
# 出现原因：两个仓库没有共同祖先（如本地 init 后关联已有远程仓库）
git pull origin main --allow-unrelated-histories

# 推送被 rejected
# 原因1：远程有本地没有的提交
git pull --rebase             # 先拉取，再推送
# 原因2：强制推送保护分支（应使用 PR 而非直接 push）
# 原因3：需要 force push（有意识的操作）
git push --force-with-lease   # 比 --force 安全（检查远程没有新提交才允许）

# 大文件导致 push 失败（GitHub 单文件 100MB 限制）
git rm --cached largefile.zip
# 加入 .gitignore 或改用 Git LFS

# 查看某个操作前后 HEAD 的状态
git reflog | head -20
```

> **追问：git config --global credential.helper store 有什么风险？**
> `credential.helper store` 将用户名和密码以**明文**存储在 `~/.git-credentials` 文件中（如 `https://user:password@github.com`），任何能读该文件的人（或进程）都能看到你的密码。正确做法是：①优先使用 SSH 密钥认证（不需要密码）；②如果必须用 HTTPS，用 `credential.helper osxkeychain`（macOS 系统钥匙串）或 `credential.helper manager`（Windows Credential Manager）加密存储。
