# 01、基础概念与工作区

## Q1. Git 和 SVN 有什么区别？

**核心区别：分布式 vs 集中式**

| 维度 | Git（分布式）| SVN（集中式）|
|------|------------|------------|
| **仓库存储** | 每个开发者本地都有完整仓库 | 只有中央服务器有完整历史 |
| **提交操作** | 先提交到本地，再 push 到远程 | 直接提交到中央服务器 |
| **离线工作** | ✅ 可以提交、查历史、切分支 | ❌ 大部分操作需要联网 |
| **分支创建** | 极快（只是创建指针，毫秒级）| 慢（需要复制文件到分支目录）|
| **合并能力** | 强，三路合并，变基 | 较弱 |
| **数据完整性** | SHA-1 校验每个对象 | 版本号 |

Git 最大的优势是**分支极其轻量**，鼓励多分支开发模式（功能分支/代码审查/CI），这是 SVN 很难做到的。

## Q2. Git 的工作区、暂存区和版本库是什么？

```
工作区（Working Directory）
  ↓ git add
暂存区（Staging Area / Index）  ← .git/index 文件
  ↓ git commit
版本库（Repository）             ← .git/ 目录
  ↓ git push
远程仓库（Remote Repository）
```

**三个区域的作用**：
- **工作区**：你直接编辑的文件目录，看得见摸得着
- **暂存区（Index）**：`git add` 后文件进入这里，是下次 commit 的准备区
  - 好处：可以精细控制"哪些改动"放入一次 commit，实现原子性提交
- **版本库**：永久保存的提交历史，每次 `git commit` 将暂存区内容写入版本库

```bash
# 查看各区域状态
git status                    # 工作区和暂存区状态
git diff                      # 工作区 vs 暂存区
git diff --cached             # 暂存区 vs 最新 commit（HEAD）
git diff HEAD                 # 工作区 vs 最新 commit
```

## Q3. 如何初始化 Git 仓库？

```bash
# 初始化本地仓库（在现有目录）
git init
git init my-project           # 创建新目录并初始化

# 配置用户信息（必须配置，否则无法提交）
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# 查看配置
git config --list
git config user.name

# 配置文件位置
# --global → ~/.gitconfig（当前用户所有仓库）
# --local  → .git/config（当前仓库，优先级更高）
# --system → /etc/gitconfig（系统所有用户）

# 初始化后的 .git 目录
.git/
├── HEAD          # 指向当前分支
├── config        # 仓库配置
├── index         # 暂存区
├── objects/      # 所有 Git 对象（blob/tree/commit/tag）
└── refs/         # 分支和标签引用
    ├── heads/    # 本地分支
    └── remotes/  # 远程跟踪分支
```

## Q4. git add 和 git stage 的区别？

`git stage` 就是 `git add` 的别名，两者完全等价，只是提供了更符合"暂存"语义的命名。

```bash
git add file.txt              # 添加指定文件到暂存区
git add .                     # 添加当前目录所有变更（新增+修改，不含删除）
git add -A                    # 添加所有变更（包含删除）
git add -u                    # 只添加已跟踪文件的变更（修改+删除，不含新文件）
git add -p                    # 交互式选择代码块添加（patch 模式，精细控制）

# git add -p 很有用：
# 修改了一个文件的多处，只想提交其中一部分变更
# 交互时：y=添加, n=跳过, s=分割, e=手动编辑
```

## Q5. 如何写一个好的 commit message？

遵循 **Conventional Commits** 规范（Angular 团队提出，业界广泛采用）：

```
<type>(<scope>): <subject>

[可选正文]

[可选页脚]
```

**type 类型**：
- `feat`：新功能（feature）
- `fix`：Bug 修复
- `docs`：文档变更
- `style`：代码格式（不影响功能，如空格/分号）
- `refactor`：代码重构（非 bug 修复非新功能）
- `test`：添加/修改测试
- `chore`：构建/辅助工具变更
- `perf`：性能优化

**示例**：
```
feat(user): 新增用户登录接口

- 支持账号密码登录
- 支持手机验证码登录
- 添加登录失败次数限制（5次锁定30分钟）

Closes #123
```

**好的 commit message 的特征**：
- 主题行 < 50 字符，动词开头，简明扼要
- 正文解释"为什么"，而不是"做了什么"（代码已经说明了做了什么）
- 一个 commit 只做一件事（原子性）

## Q6. git status 和 git diff 的区别？

```bash
# git status — 文件级别的状态概览
git status                    # 详细版
git status -s                 # 简短版（?? 未跟踪, M 修改, A 暂存, D 删除）

# git diff — 行级别的内容差异
git diff                      # 工作区 vs 暂存区（还未 git add 的变更）
git diff --cached             # 暂存区 vs HEAD（git add 后待 commit 的变更）
git diff HEAD                 # 工作区 vs HEAD（所有未提交变更）
git diff HEAD~1               # 工作区 vs 上一次提交
git diff commit1 commit2      # 两次 commit 之间的差异
git diff branch1 branch2      # 两个分支之间的差异

# 查看某个文件的变更
git diff -- file.txt
git diff HEAD -- src/main/java/UserService.java

# 只看文件名，不看具体内容
git diff --name-only
git diff --stat               # 简单统计
```

> **追问：如何查看某个文件的所有修改历史？**
> ```bash
> git log --follow -p -- file.txt  # 完整历史含重命名前的记录
> git log --oneline -- file.txt    # 只看提交列表
> git blame file.txt               # 每行最后被谁修改（排查问题时很有用）
> ```
