# 04、高级操作与实用技巧

## Q1. git stash 是什么？如何使用？

`git stash` 将当前工作区和暂存区的变更**临时储藏**起来，让工作区恢复到干净状态，方便切换分支处理紧急任务：

```bash
# 基础用法
git stash                          # 储藏当前修改（含暂存区）
git stash push -m "登录功能半成品"  # 带描述的储藏
git stash -u                       # 包含未跟踪文件（untracked）

# 查看储藏列表
git stash list
# stash@{0}: On feature/login: 登录功能半成品
# stash@{1}: WIP on develop: 8a2b3c4 上一次提交

# 恢复储藏
git stash pop                      # 恢复最近一条并删除（常用）
git stash apply stash@{1}          # 恢复指定条但不删除
git stash drop stash@{0}           # 删除指定条
git stash clear                    # 清空所有

# 查看 stash 内容
git stash show -p stash@{0}        # 详细差异

# 从 stash 创建分支（解决 stash 和当前分支冲突时）
git stash branch new-branch stash@{0}
```

**典型场景**：
```bash
# 正在做功能A，突然要修复紧急 bug
git stash push -m "功能A半成品"
git checkout main
git checkout -b hotfix/critical-bug
# ...修复 bug...
git checkout feature/A
git stash pop
```

## Q2. git cherry-pick 如何使用？

cherry-pick 将**特定 commit** 的变更"摘取"并应用到当前分支，不用合并整个分支：

```bash
# 摘取单个 commit
git cherry-pick abc1234

# 摘取多个 commit（不连续）
git cherry-pick abc1234 def5678

# 摘取一个范围（不含开始 commit）
git cherry-pick abc1234..ghi9012

# 只应用变更，不自动创建 commit
git cherry-pick -n abc1234   # --no-commit
git add .
git commit -m "自定义message"

# 遇到冲突
git cherry-pick --continue   # 解决冲突后继续
git cherry-pick --abort      # 放弃
```

**典型场景**：
- 某个 bug 在多个发布分支都需要修复，从 develop 摘取到 release-1.x、release-2.x
- 误提交到错误的分支，cherry-pick 到正确分支，再删除原来的

## Q3. .gitignore 如何配置？

```bash
# .gitignore 语法
target/         # 忽略 target 目录
*.class         # 忽略所有 .class 文件
!important.txt  # 取消忽略（例外）
/build          # 只忽略根目录下的 build 目录
**/logs/        # 任意层级的 logs 目录
doc/*.txt       # doc/ 下的 .txt 但不递归子目录
```

**Java 项目常用 .gitignore**：
```
# Maven
target/
!.mvn/wrapper/maven-wrapper.jar
*.jar
*.war

# IntelliJ IDEA
.idea/
*.iml
*.iws

# Spring Boot
application-*.properties（包含密码的配置文件）
application-*.yml

# 日志
logs/
*.log

# OS
.DS_Store      # macOS
Thumbs.db      # Windows
```

**已被跟踪的文件如何忽略**：
```bash
# 文件已在版本库中，现在加入 .gitignore 不生效
# 需要先从版本库中移除跟踪（但保留本地文件）
git rm --cached file.txt          # 移除单个文件的跟踪
git rm --cached -r target/        # 递归移除目录

git commit -m "chore: 停止跟踪 xxx 文件"
```

## Q4. git hooks 是什么？如何使用？

Git hooks 是在 Git 操作时自动执行的脚本，存放在 `.git/hooks/` 目录：

```bash
# 常用 hooks
pre-commit      # commit 前执行（代码格式检查、lint）
commit-msg      # 检查 commit message 格式
pre-push        # push 前执行（运行测试）
pre-rebase      # rebase 前
post-merge      # merge 后（如更新依赖）
post-checkout   # checkout 后（如更新 npm 包）
```

**示例：pre-commit 检查 commit message 格式**：
```bash
#!/bin/sh
# .git/hooks/commit-msg
COMMIT_MSG=$(cat $1)
PATTERN="^(feat|fix|docs|style|refactor|test|chore|perf)(\(.+\))?: .{1,72}$"

if ! echo "$COMMIT_MSG" | grep -qE "$PATTERN"; then
    echo "❌ commit message 格式不符合 Conventional Commits 规范"
    echo "格式: <type>(<scope>): <subject>"
    echo "例如: feat(user): 新增用户登录接口"
    exit 1
fi
```

```bash
# 设置可执行权限
chmod +x .git/hooks/commit-msg
```

**Husky（推荐，跨团队共享 hooks）**：
```json
// package.json（前端项目或多语言项目）
{
  "husky": {
    "hooks": {
      "pre-commit": "lint-staged",
      "commit-msg": "commitlint -E HUSKY_GIT_PARAMS"
    }
  }
}
```

## Q5. Git 标签（Tag）如何管理？

```bash
# 轻量标签（只是一个指向 commit 的指针）
git tag v1.0.0
git tag v1.0.0 abc1234      # 给指定 commit 打标签

# 附注标签（完整对象，含打标签者信息和消息，推荐）
git tag -a v1.0.0 -m "Release version 1.0.0"
git tag -a v1.0.0 abc1234 -m "Release 1.0.0"

# 查看标签
git tag                      # 列出所有标签
git tag -l "v1.*"            # 模式过滤
git show v1.0.0              # 查看标签详情

# 删除标签
git tag -d v1.0.0            # 删除本地标签
git push origin --delete v1.0.0  # 删除远程标签

# 推送标签到远程
git push origin v1.0.0       # 推送单个标签
git push origin --tags       # 推送所有标签

# 从标签创建分支（如从发版标签拉 hotfix 分支）
git checkout -b hotfix/1.0.1 v1.0.0
```

## Q6. 什么是 Git LFS？什么时候用？

Git LFS（Large File Storage）将大文件（图片/音视频/数据集）的内容存储在独立的服务器上，Git 仓库中只保留一个小的"指针"文件。

```bash
# 安装 LFS
git lfs install

# 跟踪特定类型的文件
git lfs track "*.psd"
git lfs track "*.zip"
git lfs track "assets/**/*.png"

# 查看被 LFS 管理的文件
git lfs ls-files

# 之后正常 git add/commit/push 即可
```

**使用场景**：
- 设计稿（PSD/Sketch 文件）
- 测试数据集（大 CSV/JSON 文件）
- 编译好的二进制文件（但尽量不要把构建产物放 Git）
- 媒体文件

**不用 LFS 的后果**：仓库越来越大，`git clone` 越来越慢（因为历史中的所有版本都要下载）。

> **追问：什么是 shallow clone？什么时候用？**
> ```bash
> git clone --depth 1 https://github.com/xxx/repo.git  # 只下载最近1次commit
> git clone --depth 50 ...                               # 只下载最近50次
> ```
> CI/CD 环境中非常常用：构建只需要最新代码，不需要完整历史，浅克隆可以把克隆时间从几分钟缩短到几秒。Jenkins/GitLab CI 默认就用 shallow clone。
