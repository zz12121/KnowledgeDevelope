# 03、冲突解决与版本回退

## Q1. 什么情况下会产生代码冲突？如何解决？

**产生冲突的根本原因**：两个分支修改了**同一文件的同一部分**，Git 无法自动决定保留哪一个。

```bash
# 场景：merge 或 rebase 时产生冲突
git merge feature/login
# CONFLICT (content): Merge conflict in src/UserService.java
# Automatic merge failed; fix conflicts and then commit the result.

# 查看有冲突的文件
git status
git diff --name-only --diff-filter=U   # 只显示未解决的冲突文件
```

**冲突标记格式**：
```java
<<<<<<< HEAD（当前分支）
    public void login() {
        // 当前分支的代码
        validateByPassword();
    }
=======
    public void login() {
        // 另一分支的代码
        validateByPhone();
    }
>>>>>>> feature/login（要合入的分支）
```

**解决步骤**：
```bash
# 1. 手动编辑冲突文件（删除冲突标记，保留正确代码）
# 2. 标记为已解决
git add src/UserService.java
# 3. 完成合并
git commit    # 弹出编辑器写 merge commit message
# 或
git commit -m "Merge feature/login: 保留双因素验证逻辑"
```

**放弃合并**：
```bash
git merge --abort    # 放弃 merge，恢复到合并前状态
git rebase --abort   # 放弃 rebase
```

## Q2. git reset、git revert 和 git checkout 的区别？

**`git reset`** — 移动 HEAD（和分支指针），修改历史：
```bash
# 三种模式
git reset --soft HEAD~1   # 只移动 HEAD，commit 撤销但改动保留在暂存区
git reset --mixed HEAD~1  # 移动 HEAD + 清空暂存区，改动在工作区（默认模式）
git reset --hard HEAD~1   # 移动 HEAD + 清暂存区 + 丢弃工作区改动（危险！）

# 回退到指定 commit
git reset --hard abc1234
```

**`git revert`** — 创建一个"反向 commit"，不修改历史：
```bash
git revert HEAD           # 创建一个新 commit，抵消最新 commit 的改动
git revert abc1234        # 撤销指定 commit 的改动
git revert HEAD~3..HEAD   # 撤销最近3个 commit
```

**`git checkout`** — 切换分支/恢复文件：
```bash
git checkout main              # 切换分支
git checkout -- file.txt       # 放弃工作区修改（恢复到暂存区/HEAD状态）
git checkout HEAD -- file.txt  # 从 HEAD 恢复指定文件
```

**使用场景**：
- `reset`：撤销**本地未推送**的提交（不要对已推送的提交 reset --hard）
- `revert`：撤销**已推送**的提交（安全，不影响他人历史）
- `checkout file`：丢弃工作区单文件的修改

## Q3. git reset 三种模式速查

```
场景：有 3 个 commit：A → B → C（HEAD）
      现在想撤销 C

--soft HEAD~1：
  HEAD → B，暂存区和工作区保持 C 的内容
  → 如果想重新写一个 commit message，用 soft

--mixed HEAD~1（默认）：
  HEAD → B，暂存区清空（回到 B），工作区保持 C 的内容
  → 如果想重新 git add 选择部分内容再 commit，用 mixed

--hard HEAD~1：
  HEAD → B，暂存区和工作区都回到 B 状态，C 的改动全丢失
  → 完全不需要 C 的内容时用，危险操作要谨慎
```

## Q4. 如何恢复误删的分支或误 reset 的提交？

`reflog` 是救命工具，记录所有 HEAD 的移动历史：

```bash
# 查看 reflog
git reflog                     # 所有 HEAD 移动记录（默认90天）
git reflog show feature/login  # 指定分支的记录

# 输出示例
abc1234 HEAD@{0}: reset: moving to HEAD~1
def5678 HEAD@{1}: commit: feat: 添加用户登录
ghi9012 HEAD@{2}: checkout: moving from main to feature/login

# 恢复（找到误删前的 commit SHA-1）
git checkout -b feature/login def5678  # 恢复误删的分支
git reset --hard def5678               # 恢复误 reset 的提交
```

**只要没有运行 `git gc`（垃圾回收），unreachable 的 commit 对象还在 `.git/objects/` 中，reflog 找到 SHA-1 就能恢复。**

## Q5. 如何修改最后一次提交？

```bash
# 修改最后一次 commit message
git commit --amend -m "新的提交信息"

# 往最后一次 commit 加文件（不改 message）
git add forgotten_file.txt
git commit --amend --no-edit

# 注意：amend 会改变 commit 的 SHA-1
# 如果已推送，需要 git push --force-with-lease（比 --force 更安全）
git push --force-with-lease origin feature/login
```

## Q6. git pull 和 git fetch 的区别？

```bash
git fetch origin              # 只下载远程变更，不合并到本地分支
git pull origin main          # = fetch + merge（或 rebase，取决于配置）
```

**`git pull` 的两种模式**：
```bash
git pull                      # 默认 merge 模式（可能产生 merge commit）
git pull --rebase             # 用 rebase 替代 merge（线性历史，推荐）

# 全局配置使用 rebase
git config --global pull.rebase true
```

**推荐做法**：
```bash
# 先 fetch，看看有什么变更
git fetch origin
git log HEAD..origin/main --oneline   # 查看远程有哪些新提交

# 然后决定怎么合并
git merge origin/main         # 或
git rebase origin/main
```

> **追问：每次 git pull 都提示配置 pull.rebase 是什么意思？**
> Git 2.27+ 开始，如果没有明确配置 `pull.rebase`，拉代码时会给出警告，建议你明确指定策略。推荐配置 `git config --global pull.rebase true`（rebase 模式，产生更干净的线性历史），团队中如果习惯 merge 的则配置 `pull.rebase false`。
