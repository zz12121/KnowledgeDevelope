# 02、分支管理与合并

## Q1. Git 分支的本质是什么？

Git 分支极其轻量，本质上只是一个**指向某个 commit 的可移动指针**（一个 40 字节的 SHA-1 哈希存在 `.git/refs/heads/` 目录下）。

```bash
# 创建分支（只是创建一个指针，毫秒级）
git branch feature/login
git checkout -b feature/login    # 创建并切换（传统方式）
git switch -c feature/login      # 创建并切换（新语法，推荐）

# 切换分支
git checkout main
git switch main

# 查看分支
git branch                       # 本地分支（* 表示当前）
git branch -r                    # 远程分支
git branch -a                    # 所有分支
git branch -v                    # 显示各分支最新 commit

# 删除分支
git branch -d feature/login      # 已合并才能删
git branch -D feature/login      # 强制删除

# 查看已合并/未合并到当前分支的分支
git branch --merged              # 已合并（可以安全删除）
git branch --no-merged           # 未合并
```

**HEAD 指针**：`.git/HEAD` 文件，指向当前所在分支，切换分支时改变 HEAD 指向。

## Q2. 你们项目使用什么分支策略？

**GitFlow（适合有明确发版周期的项目）**：
```
main/master    ← 只有发版才合并，永远保持可发布状态
develop        ← 开发分支，功能从这里分出，合并回这里
feature/*      ← 功能分支（从 develop 分出）
release/*      ← 发版分支（从 develop 分出，发版后合并回 main 和 develop）
hotfix/*       ← 紧急修复（从 main 分出，修完合并回 main 和 develop）
```

**GitHub Flow（适合持续部署的项目，如 SaaS）**：
```
main         ← 永远可部署
feature/*    ← 功能分支，PR 合并到 main，合并即部署
```

**Trunk-Based Development（高频发版，大公司互联网项目）**：
```
main（trunk）← 所有人直接提交或短生命周期分支（1-2天）
              Feature Flag 控制功能可见性
```

**我们的实践**：GitHub Flow + 语义化版本。功能分支 → PR → Code Review → merge main → 自动 CI/CD 发布。紧急 bug 直接在 main 上修复后打 hotfix tag。

## Q3. git merge 和 git rebase 的区别？

**merge（合并）**：
```bash
# 将 feature 合并到 main
git checkout main
git merge feature/login

# 保留合并记录（--no-ff 禁用快进合并）
git merge --no-ff feature/login -m "Merge feature/login into main"
```
- 保留完整历史（包括分叉和合并节点）
- 产生一个 merge commit（`--no-ff` 时）
- 历史可追溯性好，但提交图比较复杂

**rebase（变基）**：
```bash
# 将 feature 的 base 从旧的 main 变到最新的 main
git checkout feature/login
git rebase main              # feature 的所有 commit 重新应用到最新 main 之后

# 交互式变基（整理提交历史）
git rebase -i HEAD~3         # 整理最近3个 commit
# 在编辑器中：pick/squash/fixup/reword/drop
```
- 产生线性历史（没有分叉，更干净）
- 每个 commit 的 SHA-1 会变（重写历史）
- ❌ 不能对已推送到远程的共享分支 rebase（会导致他人混乱）

**选择原则**：
- 功能分支合并到 main/develop → 用 `merge --no-ff`（保留分支合并记录）
- 同步最新 main 到功能分支（功能开发中）→ 用 `rebase`（保持线性历史）
- 整理提交记录（推送前）→ 用 `rebase -i`（squash 多个临时 commit 为一个）
- 公共分支（main/develop）→ 禁止 rebase（`--force-push` 是危险操作）

## Q4. 什么是快进合并（Fast-forward）？

当 main 没有新的提交，feature 分支是在 main 的基础上继续提交时，merge 不需要创建新的 merge commit，只需将 main 指针"快进"到 feature 所在位置：

```
Before:  main → A → B
               ↑
         feature → C → D

After (fast-forward): main/feature → A → B → C → D
```

```bash
git merge feature          # 自动快进（如果可以）
git merge --no-ff feature  # 禁止快进，强制创建 merge commit（团队推荐）
```

`--no-ff` 的好处：即使可以快进，也创建一个 merge commit，这样从 main 历史上能看到哪些 commit 是同一个功能分支的工作，方便 revert 整个功能。

## Q5. git rebase -i 如何整理提交历史？

```bash
# 整理最近3个 commit
git rebase -i HEAD~3

# 编辑器打开（从上到下是从旧到新）
pick abc1234 feat: 用户登录接口
pick def5678 fix: 修改一个typo
pick ghi9012 fix: 忘记加空格了

# 改成：
pick abc1234 feat: 用户登录接口
squash def5678 fix: 修改一个typo     # 合并到上一个commit
squash ghi9012 fix: 忘记加空格了     # 合并到上一个commit

# 保存退出后，会提示写新的 commit message
# 最终变成一个干净的 commit：feat: 用户登录接口
```

**常用操作符**：
- `pick`：保留这个 commit（默认）
- `squash`（s）：合并到上一个 commit（合并 message）
- `fixup`（f）：合并到上一个 commit（丢弃 message，只保留上一个）
- `reword`（r）：保留 commit，但修改 message
- `drop`（d）：删除这个 commit
- `edit`（e）：暂停在这个 commit，允许修改

## Q6. 如何管理远程分支？

```bash
# 查看远程分支
git remote -v                             # 查看远程仓库
git branch -r                             # 查看远程分支
git branch -a                             # 本地+远程分支

# 将本地分支推送到远程
git push origin feature/login            # 推送到同名远程分支
git push origin local-name:remote-name   # 推送到不同名的远程分支

# 跟踪远程分支（建立本地分支与远程分支的关联）
git branch -u origin/feature/login       # 为当前分支设置上游
git checkout -b feature/login origin/feature/login  # 创建本地分支并跟踪

# 删除远程分支
git push origin --delete feature/login
git push origin :feature/login           # 等价（冒号前空）

# 同步远程分支状态（删除本地存在但远程已删除的跟踪引用）
git remote prune origin
git fetch --prune                         # fetch 同时清理
```

> **追问：如何防止误推到 main 分支？**
> ①在 GitHub/GitLab 上设置 Branch Protection Rules：main 分支需要 PR + Code Review 才能合并；②本地 pre-push hook：检查当前分支，如果是 main/master 则阻止直接 push；③团队约定：直接 push 到 main 是禁止操作，所有变更走 PR 流程。
