# 03、性能调优与配置

## Q1. Tomcat 连接数和线程数如何调优？

**核心参数（conf/server.xml Connector 节点）**：
```xml
<Connector port="8080" protocol="org.apache.coyote.http11.Http11NioProtocol"
    <!-- 线程池 -->
    maxThreads="400"          <!-- 最大工作线程（默认200，IO密集可调高）-->
    minSpareThreads="50"      <!-- 最小空闲线程（预热线程池）-->
    maxQueueSize="100"        <!-- 任务队列（满了则拒绝）-->
    
    <!-- 连接控制 -->
    maxConnections="10000"    <!-- 最大并发连接（NIO默认10000）-->
    acceptCount="200"         <!-- OS级别等待队列（backlog）-->
    
    <!-- 超时 -->
    connectionTimeout="5000"  <!-- 等待完整HTTP请求头的超时（ms）-->
    keepAliveTimeout="15000"  <!-- 空闲连接保持时间（ms）-->
    maxKeepAliveRequests="100" <!-- 单连接最多复用次数（-1无限制）-->
    
    <!-- 性能 -->
    compression="on"          <!-- 开启 Gzip 压缩 -->
    compressionMinSize="2048" <!-- 2KB 以上才压缩 -->
    compressableMimeType="text/html,text/xml,text/css,application/json" />
```

**调优建议**：
- **IO 密集型应用**（DB/外部 API 调用多）：`maxThreads` 可以设大（200~500），线程大量时间等待 IO
- **CPU 密集型应用**（计算密集）：`maxThreads` 设为 CPU 核数 × 2 即可，太多反而因线程切换性能下降
- **高并发长连接**：增大 `maxConnections`（NIO 的优势，不需要大量线程）

## Q2. Tomcat JVM 参数如何调优？

```bash
# bin/setenv.sh（不存在则创建）
export JAVA_OPTS="\
  -server \
  -Xms2g -Xmx2g \
  -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/app/logs/heapdump.hprof \
  -Xlog:gc*:file=/app/logs/gc.log:time,uptime:filecount=5,filesize=20m \
  -Djava.security.egd=file:/dev/./urandom \
  -Dfile.encoding=UTF-8"
```

**关键配置说明**：
- `-Xms` = `-Xmx`：避免堆动态扩缩容，减少 GC 触发
- `-XX:+UseG1GC`：G1 GC 是 Java 9+ 默认，适合大堆（>6G）低延迟场景
- `-Djava.security.egd=file:/dev/./urandom`：解决 Linux 上随机数生成慢（`/dev/random` 会阻塞）导致 Tomcat 启动慢的问题

## Q3. 如何开启 HTTP/2 支持？

```xml
<!-- 需要 Tomcat 8.5+ 且配置 SSL -->
<Connector port="8443" protocol="org.apache.coyote.http11.Http11NioProtocol"
           SSLEnabled="true">
    <!-- 开启 HTTP/2（需要 ALPN 协议协商）-->
    <UpgradeProtocol className="org.apache.coyote.http2.Http2Protocol"
                     keepAliveTimeout="60000" />
    <SSLHostConfig>
        <Certificate certificateFile="conf/cert.pem"
                     certificateKeyFile="conf/key.pem" />
    </SSLHostConfig>
</Connector>
```

**注意**：HTTP/2 要求 HTTPS（浏览器只支持加密的 HTTP/2），且 Java 11+ 内置 ALPN 支持（无需额外 jar）；Java 8 需要通过 APR 或引入 alpn-boot 库。

**Spring Boot 配置**：
```yaml
server:
  http2:
    enabled: true
  ssl:
    enabled: true
    # ...
```

## Q4. 如何配置 Gzip 压缩？

```xml
<!-- server.xml Connector 配置 -->
<Connector port="8080" protocol="HTTP/1.1"
    compression="on"
    compressionMinSize="1024"    <!-- 1KB 以上才压缩 -->
    noCompressionUserAgents="gozilla, traviata"
    compressableMimeType="text/html,text/xml,text/css,text/javascript,
                          application/json,application/javascript" />
```

```yaml
# Spring Boot 嵌入式 Tomcat
server:
  compression:
    enabled: true
    min-response-size: 1024
    mime-types: text/html,text/xml,text/plain,text/css,application/json,application/javascript
```

**注意**：当 Tomcat 前面有 Nginx 时，通常在 Nginx 层统一做压缩，Tomcat 不用再配（避免重复压缩浪费 CPU）。

## Q5. Tomcat 的会话管理与 Session 集群

**Session 管理器类型**：
- `StandardManager`：默认，Session 存内存，Tomcat 关闭时序列化到文件
- `PersistentManager + FileStore`：序列化到文件（不常用）
- `PersistentManager + JDBCStore`：序列化到数据库
- `ClusterManager（DeltaManager/BackupManager）`：集群会话复制

**集群 Session 配置（适合小集群）**：
```xml
<!-- server.xml Engine 下 -->
<Cluster className="org.apache.catalina.ha.tcp.SimpleTcpCluster"
         channelSendOptions="8">
    <Manager className="org.apache.catalina.ha.session.DeltaManager"
             expireSessionsOnShutdown="false"
             notifyListenersOnReplication="true"/>
    <Channel className="org.apache.catalina.tribes.group.GroupChannel">
        <Membership className="org.apache.catalina.tribes.membership.McastService"
                    address="228.0.0.4" port="45564" frequency="500" dropTime="3000"/>
        <Receiver className="org.apache.catalina.tribes.transport.nio.NioReceiver"
                  address="auto" port="4000" autoBind="100" selectorTimeout="5000" maxThreads="6"/>
        <Sender className="org.apache.catalina.tribes.transport.ReplicationTransmitter">
            <Transport className="org.apache.catalina.tribes.transport.nio.PooledParallelSender"/>
        </Sender>
    </Channel>
</Cluster>
```

**生产推荐方案（Session 存 Redis）**：
```xml
<!-- 使用 Redisson/Spring Session 替代 Tomcat 原生集群复制 -->
<!-- 1. Spring Session + Redis（推荐）-->
<!-- @EnableRedisHttpSession -->
<!-- 2. Tomcat + Redisson Session Manager -->
```

> **追问：粘性会话（Sticky Session）和 Session 复制的区别？**
> 粘性会话：Nginx 用 `ip_hash` 或 Cookie 确保同一用户的请求始终打到同一台 Tomcat，Session 只在本机，简单高效，但那台机器宕机时该用户 Session 丢失。Session 复制：所有 Tomcat 实例实时同步 Session，任何一台宕机用户无感知，但同步有网络开销，Session 数据量大时影响性能。生产上最佳实践是用 **Redis 集中存储 Session**（Spring Session），既不需要 Sticky 也不需要复制。
