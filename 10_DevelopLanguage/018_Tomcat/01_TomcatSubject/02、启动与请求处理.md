# 02、启动与请求处理

## Q1. Tomcat 的启动流程是怎样的？

```
bin/startup.sh
  → Bootstrap.main()
    → 初始化 CommonClassLoader/CatalinaClassLoader/SharedClassLoader
    → 反射创建 Catalina 实例（通过 CatalinaClassLoader 加载）
    → Catalina.load()
       → 解析 conf/server.xml → 构建 Server 组件树
       → Server.init()（级联初始化所有子组件）
    → Catalina.start()
       → Server.start()（级联启动所有子组件）
          → Service.start()
            → Connector.start()（启动 Acceptor/Poller 线程，监听端口）
            → Engine.start()
               → Host.start()（扫描 webapps/ 部署 Web 应用）
                 → Context.start()
                    → 加载 web.xml / 实例化 Listener / 初始化 Filter / 加载启动 Servlet
```

**Bootstrap vs Catalina 的区别**：
- `Bootstrap`：负责类加载器的初始化，是 JVM 入口点，尽量不依赖其他类
- `Catalina`：负责 Tomcat 核心启动逻辑（解析 server.xml、启动 Server），通过 `Bootstrap` 反射创建，使用 `CatalinaClassLoader` 加载

## Q2. Tomcat 如何处理一个 HTTP 请求？完整流程是什么？

```
1. 操作系统三次握手建立 TCP 连接

2. Acceptor 线程（1个）接受连接
   → 将 SocketChannel 注册到某个 Poller

3. Poller 线程（CPU核数）用 Selector 轮询就绪的 Channel
   → 有数据可读 → 从线程池取线程处理

4. Worker 线程（线程池中）
   → Http11Processor 读取并解析 HTTP 请求行、请求头
   → 构造 Coyote Request/Response 对象
   → 调用 CoyoteAdapter.service()

5. CoyoteAdapter
   → 将 Coyote Request 转换为 Catalina HttpServletRequest
   → 调用 Connector.getService().getContainer().getPipeline().getFirst().invoke()

6. Engine → Host → Context → Wrapper Pipeline 链调用

7. StandardWrapperValve.invoke()
   → 创建 FilterChain（根据 web.xml 的 filter-mapping）
   → FilterChain.doFilter()（所有 Filter 链式调用）
   → Servlet.service()（DispatcherServlet.doService()）

8. Spring MVC 处理请求 → 返回结果

9. Response 写回，沿原路返回（若 Keep-Alive 则连接保持）
```

## Q3. Tomcat 的线程模型和 NIO/NIO2/APR 区别？

**Tomcat Connector 三种 I/O 模式**：

| 模式 | 协议处理器 | 特点 |
|------|-----------|------|
| **NIO**（默认）| Http11NioProtocol | Java NIO，非阻塞，Acceptor+Poller+Worker，高并发推荐 |
| **NIO2** | Http11Nio2Protocol | Java AIO，异步 I/O，理论最优但实际提升有限 |
| **APR** | Http11AprProtocol | 本地库（OpenSSL），SSL 性能最佳，需安装 apr/tomcat-native |

**NIO 线程模型**：
```
Acceptor 线程（1个）    → 接受 TCP 连接
Poller 线程（CPU核数）  → 用 Selector 监听 Socket 事件（读就绪）
Worker 线程池（可配置） → 实际处理 HTTP 请求
```

关键区分：BIO（旧版 Tomcat 8 以下）每个连接占用一个线程；NIO 接受连接和处理请求分离，少数线程可以管理大量连接。

## Q4. Tomcat 的关键参数：maxThreads、acceptCount、maxConnections

```xml
<Connector port="8080" protocol="HTTP/1.1"
           maxThreads="200"       <!-- 工作线程池最大线程数（默认200）-->
           minSpareThreads="10"   <!-- 线程池最小空闲线程数 -->
           acceptCount="100"      <!-- 连接队列（backlog）长度（默认100）-->
           maxConnections="10000" <!-- 最大并发连接数（NIO默认10000）-->
           connectionTimeout="20000" />
```

**三个参数的关系**：
```
10001 个并发连接进来时：
  前 10000 个 → maxConnections 接收，排队等待处理
  第 10001 个 → OS 内核 acceptCount 队列（再容纳100个）
  超过 10100 个 → 直接拒绝连接（Connection refused）

maxThreads（200）：同时处理请求的线程数
  连接数可以多，但同时处理的只有200个
  其他连接等待线程空闲
```

**调优建议**：
- `maxThreads`：根据 CPU 密集/IO 密集调整，Java 应用一般 200~500（不要太大，线程切换开销）
- `maxConnections`：NIO 下不影响线程数，可以设置较大值（默认 10000 够用）
- `acceptCount`：突发流量缓冲，100~200 够用

## Q5. 如何配置 HTTPS/SSL？

```xml
<!-- conf/server.xml -->
<!-- 方式1：使用 JKS 密钥库 -->
<Connector port="8443" protocol="org.apache.coyote.http11.Http11NioProtocol"
           SSLEnabled="true"
           maxThreads="150" scheme="https" secure="true">
    <SSLHostConfig>
        <Certificate certificateKeystoreFile="conf/server.keystore"
                     certificateKeystorePassword="changeit"
                     type="RSA" />
    </SSLHostConfig>
</Connector>

<!-- 方式2：使用 PEM 格式证书（Tomcat 9+ 推荐）-->
<Connector port="8443" protocol="org.apache.coyote.http11.Http11NioProtocol"
           SSLEnabled="true">
    <SSLHostConfig protocols="TLSv1.2+TLSv1.3">
        <Certificate certificateFile="conf/cert.pem"
                     certificateKeyFile="conf/key.pem"
                     certificateChainFile="conf/chain.pem"
                     type="RSA" />
    </SSLHostConfig>
</Connector>
```

**Spring Boot 嵌入式 Tomcat 配置 HTTPS**：
```yaml
server:
  port: 8443
  ssl:
    enabled: true
    key-store: classpath:keystore.p12
    key-store-password: changeit
    key-store-type: PKCS12
    key-alias: tomcat
    protocol: TLS
    enabled-protocols: TLSv1.2,TLSv1.3
```

## Q6. Tomcat 如何实现优雅停机？

```bash
# 发送 SHUTDOWN 命令（默认监听 8005 端口）
bin/shutdown.sh
# 等价
echo "SHUTDOWN" | nc localhost 8005

# Linux 信号
kill -SIGTERM $(cat /var/run/tomcat.pid)   # 优雅停机
kill -9 pid                                 # 强制（不推荐）
```

**优雅停机流程**：
1. Connector 停止接受新连接
2. 等待当前请求处理完（有超时时间）
3. 执行 Web 应用的 `contextDestroyed` 监听器（资源清理）
4. 卸载 Web 应用
5. 关闭线程池

**Spring Boot 嵌入式 Tomcat 优雅关闭**：
```yaml
server:
  shutdown: graceful          # 优雅关闭（等待进行中的请求完成）
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 最多等待30秒
```
收到 SIGTERM 后，Spring Boot 不再接受新请求，等待当前请求处理完后再退出。

> **追问：Tomcat 的 Bootstrap、Catalina、Server 是什么关系？**
> Bootstrap 是 JVM 程序入口（`bin/startup.sh` 执行的），负责初始化类加载器体系（Common/Catalina/Shared 三层类加载器），然后通过反射用 CatalinaClassLoader 实例化 Catalina 并调用其方法。这样做是为了隔离类加载，Tomcat 框架代码和 Web 应用代码用不同类加载器，避免相互干扰。Catalina 是真正的"总指挥"，负责解析 server.xml 构建 Server 组件树并管理整个生命周期。Server 就是 server.xml 根节点代表的组件，管理 Service 列表。
