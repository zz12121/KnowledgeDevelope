# 01、架构与组件

## Q1. Tomcat 的架构组成是什么？

Tomcat 是一个 Servlet 容器，整体架构是**组件树**模型，每个组件实现 `Lifecycle` 接口，统一管理生命周期：

```
Server（服务器）
└── Service（服务，默认名 Catalina）
    ├── Connector（连接器，处理协议通信）
    │   └── ProtocolHandler（NIO/NIO2/APR）
    │       ├── Acceptor（接受连接）
    │       ├── Poller（NIO 事件选择器）
    │       └── Executor（线程池）
    └── Container（容器，处理请求）
        └── Engine（引擎，名字 Catalina）
            └── Host（虚拟主机，名字 localhost）
                └── Context（Web 应用，对应一个 war）
                    └── Wrapper（Servlet 包装器）
```

**两大核心模块**：
- **Connector（连接器）**：负责网络通信，接收请求、解析协议（HTTP/AJP）、将数据封装成 Request/Response 交给 Container
- **Container（容器）**：负责业务逻辑处理，通过 Pipeline+Valve 链调用 Servlet 处理请求

## Q2. Server、Service、Engine、Host、Context 的层级关系？

```xml
<!-- server.xml 配置层级 -->
<Server port="8005" shutdown="SHUTDOWN">
  <!-- Service 将 Connector 和 Container 关联起来 -->
  <Service name="Catalina">
    <!-- HTTP Connector，监听 8080 端口 -->
    <Connector port="8080" protocol="HTTP/1.1" />
    <!-- AJP Connector，与 Apache httpd 配合 -->
    <!-- <Connector port="8009" protocol="AJP/1.3" /> -->

    <!-- Engine：处理所有 Connector 来的请求 -->
    <Engine name="Catalina" defaultHost="localhost">
      <!-- Host：虚拟主机，对应域名 -->
      <Host name="localhost" appBase="webapps">
        <!-- Context：一个 Web 应用 -->
        <Context path="/myapp" docBase="myapp" />
      </Host>
    </Engine>
  </Service>
</Server>
```

- **Server**：整个 Tomcat 进程（一个 Server 只有一个）
- **Service**：将一组 Connector 和一个 Engine 绑定（一个 Server 可有多个 Service）
- **Engine**：处理所有 HTTP 请求的入口，根据 Host 名路由
- **Host**：虚拟主机，管理多个 Context
- **Context**：一个 Web 应用（一个 war/目录），对应一个域名下的一个路径

## Q3. Tomcat 的 Valve 和 Pipeline 是什么？

Pipeline（管道）是一种**责任链模式**，Container 中的每一层都有一个 Pipeline，请求在 Pipeline 中经过一系列 Valve（阀门）处理后最终到达 Servlet：

```
Engine Pipeline:  StandardEngineValve
  → Host Pipeline:  AccessLogValve → StandardHostValve  
    → Context Pipeline: StandardContextValve
      → Wrapper Pipeline: StandardWrapperValve（最终调用 Servlet）
```

**内置 Valve 示例**：
```xml
<!-- 访问日志 Valve -->
<Valve className="org.apache.catalina.valves.AccessLogValve"
       directory="logs" prefix="localhost_access_log"
       suffix=".txt"
       pattern="%h %l %u %t &quot;%r&quot; %s %b %D" />

<!-- IP 访问限制 Valve -->
<Valve className="org.apache.catalina.valves.RemoteAddrValve"
       allow="192\.168\.1\..*"
       deny="" />

<!-- 限流 Valve -->
<Valve className="org.apache.catalina.valves.StuckThreadDetectionValve"
       threshold="60" />
```

**自定义 Valve**：继承 `ValveBase`，重写 `invoke()` 方法，可以在请求前/后执行自定义逻辑（类似 Filter，但更低层）。

## Q4. Connector 与 Container 的关系是什么？

Connector 和 Container 通过 `CoyoteAdapter` 解耦：

```
外部请求
  ↓
Connector（接收 TCP 连接，解析 HTTP 协议）
  ↓ 将 Coyote Request/Response 包装成 Catalina Request/Response
CoyoteAdapter（适配器，Connector ↔ Container 的桥梁）
  ↓
Engine.Pipeline → Host.Pipeline → Context.Pipeline → Wrapper.Pipeline
  ↓
Servlet.service()
```

**关键点**：Connector 负责 I/O（网络通信），Container 负责逻辑（Servlet 调用），两者通过 Adapter 解耦，这样 Connector 可以换成不同协议（HTTP/AJP/HTTP2）而 Container 不用改。

## Q5. Tomcat 的 Loader 组件的作用是什么？

每个 Context（Web 应用）都有一个 Loader，负责该应用的类加载：

```
Loader → WebappClassLoader（每个Context独立）
  ↑ 加载
/WEB-INF/classes/      Web 应用自己的类
/WEB-INF/lib/*.jar     Web 应用自己的 jar
```

**设计目的**：应用隔离。A 应用和 B 应用可以用不同版本的同一个库（如 log4j 1.x vs 2.x），互不干扰。这是通过每个 Context 独立的 WebappClassLoader 实现的，打破了双亲委派模型（自己的类优先加载，不先委托父加载器）。

> **追问：Tomcat 为什么要打破双亲委派模型？**
> 两个原因：①应用隔离，同一个 Tomcat 可以跑多个使用不同版本库的 Web 应用；②Servlet 规范要求，`WEB-INF/classes` 和 `WEB-INF/lib` 中的类应该优先于应用服务器类路径中的类。这是 Java SPI 加载（`Thread.currentThread().getContextClassLoader()`）场景，必须用子类加载器加载实现类。
