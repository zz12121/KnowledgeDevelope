# 04、类加载与部署

## Q1. Tomcat 的类加载器层次结构是什么？

Tomcat 有自己的类加载器体系，在 JVM 标准三层类加载器之上增加了额外层次：

```
Bootstrap ClassLoader（JVM 内置，加载 rt.jar 等 JDK 核心类）
    ↑
Extension ClassLoader（加载 jre/lib/ext/）
    ↑
Application ClassLoader（加载 CLASSPATH）
    ↑
CommonClassLoader（$CATALINA_HOME/lib/ 下所有 jar）
    ↑ 父加载器
CatalinaClassLoader  SharedClassLoader（通常和 Common 同一个）
（Tomcat 内部类）       （所有 Web 应用共享的类）
                           ↑ 父加载器
                     WebappClassLoader（每个 Context 独立，打破双亲委派）
                       加载 WEB-INF/classes/ 和 WEB-INF/lib/*.jar
```

**WebappClassLoader 加载顺序**（打破双亲委派）：
1. JVM 核心类（Bootstrap 加载的）→ 委托父加载器
2. WEB-INF/classes/（应用类，优先自己加载！）
3. WEB-INF/lib/*.jar（应用 jar）
4. Common 类库（父加载器）
5. 找不到 → ClassNotFoundException

## Q2. 热部署、热加载、自动部署的区别？

```xml
<!-- server.xml Host 配置 -->
<Host name="localhost" appBase="webapps"
      autoDeploy="true"     <!-- 自动部署：扫描 webapps/ 目录，有新 war 就部署 -->
      deployXML="true">

    <!-- conf/Catalina/localhost/ 下的 Context XML 文件也会自动部署 -->
</Host>
```

| 特性 | 热部署（Hot Deploy）| 热加载（Hot Reload）| 自动部署（Auto Deploy）|
|------|----------------|----------------|-----------------|
| 触发方式 | 替换 war 包 | 修改 WEB-INF/classes 等 | webapps/ 下出现新 war |
| 效果 | 整个应用重新加载（销毁旧 WebappClassLoader，创建新的）| 只重新加载修改的类 | 自动扫描并部署 |
| 生产使用 | ✅ 可用 | ❌ 不推荐（内存泄漏风险）| ❌ 关闭（安全考虑）|

**生产建议**：关闭自动部署（`autoDeploy="false"`），通过运维脚本手动部署；开发环境可以开启热加载（IDEA 的热重启更好用）。

## Q3. Tomcat Web 应用部署方式有哪些？

```bash
# ==================== 方式1：将 war 放到 webapps/ ====================
cp myapp.war $CATALINA_HOME/webapps/
# Tomcat 自动解压并部署为 /myapp 路径（如果 autoDeploy=true）

# ==================== 方式2：Context XML 配置 ====================
# conf/Catalina/localhost/ROOT.xml （对应根路径 /）
# conf/Catalina/localhost/myapp.xml （对应 /myapp）
```
```xml
<!-- myapp.xml -->
<Context docBase="/data/myapp" path="/myapp" reloadable="false">
    <!-- 可以在这里定义 JNDI 资源 -->
    <Resource name="jdbc/MyDB" auth="Container" type="javax.sql.DataSource" ... />
</Context>
```

```bash
# ==================== 方式3：Tomcat Manager App ====================
# 通过 Web 界面或 API 部署（需配置 manager 权限）
curl -u admin:password \
  "http://localhost:8080/manager/text/deploy?path=/myapp&war=file:/data/myapp.war"

# ==================== 方式4：Maven Tomcat 插件 ====================
mvn tomcat7:deploy -Dtomcat.url=http://localhost:8080/manager/text
```

## Q4. Tomcat 如何实现应用隔离？

每个 Web 应用（Context）有独立的 `WebappClassLoader`：

```
应用 A 的 WebappClassLoader ← 加载 log4j-1.2.jar
应用 B 的 WebappClassLoader ← 加载 log4j-2.19.jar
```

两个应用各自拥有 log4j 类的独立实例，互不干扰。这是 Tomcat 打破双亲委派模型的核心原因。

**类隔离带来的问题**：
- `ClassCastException`：对象明明是同一个类，但用 `instanceof` 返回 false
  - 原因：两个 WebappClassLoader 各自加载了同一个类，加载器不同导致 Class 对象不同
  - 解决：共享类放到 Tomcat 的 `lib/` 目录（Common 类加载器），所有应用共享同一个

> **追问：为什么 Tomcat 的 WebappClassLoader 要先加载自己的类（打破双亲委派）？**
> Java EE 规范要求 Web 应用能用自己的 jar 覆盖应用服务器提供的版本。举例：应用自带了 commons-logging 2.0，服务器 lib/ 里有 commons-logging 1.0，如果按双亲委派，服务器版本会先被加载，应用自己的包就失效了。打破双亲委派允许每个应用用自己打包的版本，实现真正的应用隔离。但 JVM 核心类（`java.*`、`javax.*`、`sun.*`）无论如何都由 Bootstrap 加载，这是安全红线。
