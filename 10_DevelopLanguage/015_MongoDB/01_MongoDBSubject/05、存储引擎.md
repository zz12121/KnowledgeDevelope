### 五、存储引擎类

---

###### 1. 说说 MongoDB 的存储引擎，有什么优势？

MongoDB 3.2+ 默认使用 **WiredTiger** 存储引擎（替代了老的 MMAPv1）。

**为什么 WiredTiger 更好**：

| 特性 | MMAPv1（废弃） | WiredTiger（当前默认） |
|------|--------------|----------------------|
| 并发控制 | 集合级锁（写多读少时性能差） | **文档级锁**（MVCC，读写几乎不阻塞） |
| 数据压缩 | 不支持 | 支持（Snappy/zlib/zstd，默认 Snappy） |
| 内存效率 | 全部依赖 OS 文件系统缓存 | 自管 Cache（默认内存的 50%） |
| 磁盘占用 | 大 | 小（压缩后通常减少 60-80%） |
| 崩溃恢复 | 较慢（Journal 较简单） | 快（Checkpoint + Journal，秒级恢复） |

---

###### 2. WiredTiger 存储引擎的特点是什么？

**核心特性**：

1. **MVCC（多版本并发控制）**：读操作不阻塞写操作，写操作不阻塞读操作。每个事务看到一致性快照，通过版本链管理并发。

2. **文档级锁**：大幅提升写并发能力。老的 MMAPv1 是集合级锁，多线程写同一集合会排队；WiredTiger 不同文档可并发写入。

3. **B-Tree 索引**：WiredTiger 的索引使用 B-Tree，根节点常驻内存，叶子节点按需加载。

4. **内存 Cache**：WiredTiger 管理自己的内存缓存（默认为 `max(50% of RAM - 1GB, 256MB)`），存储最近访问的"热"数据页，减少磁盘 I/O。

5. **WAL（Write-Ahead Log）**：每次写操作先写 Journal（WAL）日志，再写数据文件。Journal 每 100ms fsync 一次，崩溃后可通过 Journal 恢复到最新 Checkpoint 之后的操作。

---

###### 3. MongoDB 的压缩机制是什么？

WiredTiger 支持**块级压缩**（Block Compression）：

| 压缩算法 | 速度 | 压缩率 | 默认 |
|---------|------|--------|------|
| Snappy | 极快（解压接近内存带宽） | 中等（约 60%） | **数据集合默认** |
| zlib | 慢 | 高（约 80%） | - |
| zstd（4.2+） | 快，比 Snappy 压缩率高 | 高（约 75%） | 日志集合推荐 |
| none | - | 不压缩 | 索引默认（索引本身已是紧凑格式） |

**配置**：
```javascript
// 创建集合时指定压缩算法
db.createCollection("logs", {
    storageEngine: {
        wiredTiger: { configString: "block_compressor=zstd" }
    }
})
```

---

###### 4. 什么是 Journal 日志？有什么作用？

Journal 是 WiredTiger 的**预写日志（WAL）**，位于 `<dbpath>/journal/` 目录。

**作用**：
- **崩溃恢复**：服务器宕机后，未写入数据文件的操作可以通过 Journal 重放，确保数据不丢失
- **持久性保证**：`writeConcern: {j: true}` 确保写操作已写入 Journal 再返回成功

**工作流程**：
```
写操作 → WiredTiger Cache（内存）→ Journal（磁盘，每100ms fsync）
                                        ↕
                               Checkpoint（每60s写一次 data files）
```

**崩溃恢复流程**：
1. 找到最近的 Checkpoint（完整快照点）
2. 从 Checkpoint 之后回放 Journal 中的操作
3. 数据库恢复到崩溃前的状态

**关闭 Journal 的风险**：可能丢失最近 100ms 的写操作，生产环境不推荐关闭。

---

###### 追问：WiredTiger Cache 满了怎么办？

WiredTiger 默认使用 50% - 1GB 的内存作为 Cache。当 Cache 压力大时：

1. **触发 Eviction（逐出）**：后台线程将"脏页"（已修改但未写入磁盘的数据页）写回数据文件，然后将这些页从 Cache 中移除

2. **Eviction 触发阈值**（可配置）：
   - 内存使用 > 80%：后台 eviction 开始工作
   - 内存使用 > 95%：前台应用线程也参与 eviction（性能急剧下降！）

3. **监控与优化**：
   ```javascript
   db.serverStatus().wiredTiger.cache  // 查看 Cache 使用情况
   // "bytes currently in the cache"：当前 cache 使用量
   // "pages evicted by application threads"：若此值高，说明 Cache 严重不足
   ```

4. **解决方案**：增大 `wiredTigerCacheSizeGB`，或增加物理内存，确保工作集（热数据）能放入 Cache。
