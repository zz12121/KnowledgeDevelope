### 四、数据模型与设计类

---

###### 1. 说说 MongoDB 的数据模型，如何选择？

MongoDB 有两种基本的数据关系建模方式：**嵌入式文档** 和 **引用文档**。

**决策树**：
```
数据关系是一对一/一对少量？
    └── 关系数据会一起访问？
            ├── 是 → 嵌入式（减少查询次数）
            └── 否 → 引用（节省空间）

数据关系是一对多（多方较小）？
    ├── 多方数量 < 100 → 嵌入数组（如文章标签）
    └── 多方数量 > 100 → 引用（如用户订单）

数据关系是一对超多（thousands+）？
    └── 引用（避免文档膨胀超过 16MB）

多对多关系？
    └── 通常在双方文档中保存对方 _id 数组（视读写频率决定）
```

---

###### 2. MongoDB 中嵌入式文档和引用文档的区别？

**嵌入式（Embedded）**：将相关数据直接嵌套在父文档中
```javascript
// 用户和地址 - 嵌入式（一对一/一对少）
{
    _id: ObjectId("..."),
    name: "张三",
    address: {          // 嵌套文档
        city: "北京",
        district: "朝阳区"
    }
}
```

**引用式（Reference）**：存储关联文档的 `_id`，查询时用 `$lookup`
```javascript
// 用户（存储 orderId 引用）
{ _id: ObjectId("user001"), name: "张三", orderIds: [ObjectId("o1"), ObjectId("o2")] }
// 订单（单独集合）
{ _id: ObjectId("o1"), userId: ObjectId("user001"), amount: 100 }
```

| 维度 | 嵌入式 | 引用式 |
|------|--------|--------|
| 读取性能 | ✅ 一次查询获取所有数据 | ❌ 需要 $lookup 或多次查询 |
| 数据一致性 | ✅ 天然原子更新 | ❌ 需额外保证一致性 |
| 文档大小 | ❌ 容易膨胀超 16MB | ✅ 文档保持小巧 |
| 数据复用 | ❌ 数据重复（如品牌信息在每个商品中重复） | ✅ 单处更新，全局生效 |
| 适用场景 | 一对一、一对少量、数据强内聚 | 一对多（多方数据大）、多对多、频繁独立访问 |

###### 3. 如何设计 MongoDB 的 Schema？

**核心原则**：**以查询模式驱动 Schema 设计**（与关系型数据库"以数据范式驱动"相反）

**实践指南**：
1. **先问"最频繁的查询是什么"**，据此决定是嵌入还是引用
2. **一起读的数据放一起**（减少 $lookup）
3. **预计算常用聚合值**：如 `commentCount` 字段存在文章文档中，避免每次 `COUNT`
4. **避免无限增长的数组**：评论不要都嵌套在文章中，单独一个集合
5. **使用部分索引**节省空间：只对 `status = "active"` 的文档建索引

###### 4. MongoDB 支持事务吗？如何使用？

**版本历史**：
- 4.0（2018）：支持**副本集**内的多文档 ACID 事务
- 4.2（2019）：支持**分片集群**内的多文档事务

```javascript
// 多文档事务（Node.js 驱动）
const session = client.startSession();

try {
    session.startTransaction({
        readConcern: { level: "snapshot" },
        writeConcern: { w: "majority" }
    });
    
    // 从账户 A 扣款
    await accounts.updateOne(
        { _id: "accountA" },
        { $inc: { balance: -100 } },
        { session }
    );
    
    // 向账户 B 加款
    await accounts.updateOne(
        { _id: "accountB" },
        { $inc: { balance: 100 } },
        { session }
    );
    
    await session.commitTransaction();
    
} catch (error) {
    await session.abortTransaction();
    throw error;
} finally {
    session.endSession();
}
```

**事务限制与注意事项**：
- 事务不能超过 **60 秒**（默认），超时自动回滚
- 事务会带来额外开销，仅在必要时使用（能用单文档原子操作的优先用单文档）
- 单文档操作天然原子，不需要事务

###### 5. MongoDB 的原子性操作有哪些？

MongoDB **单文档操作天然原子**，以下操作保证原子性：

```javascript
// findOneAndUpdate（查找并更新，原子操作）
db.inventory.findOneAndUpdate(
    { item: "apple", qty: { $gte: 10 } },  // 条件
    { $inc: { qty: -10 } },                 // 减库存
    { returnDocument: "after" }             // 返回更新后的文档
)

// 原子更新操作符
$set        // 设置字段值
$inc        // 原子递增/递减
$push       // 向数组追加元素
$addToSet   // 向数组添加唯一元素
$pull       // 从数组移除元素
$pop        // 从数组首/尾移除元素

// 条件更新（乐观锁）
db.orders.updateOne(
    { _id: orderId, status: "pending" },  // 检查当前状态
    { $set: { status: "processing" } }    // 仅在 pending 时更新
)
// 如果 matched: 0，说明被其他操作抢先修改了
```

###### 追问：MongoDB 如何实现乐观锁？

MongoDB 没有内置乐观锁，但可以用**版本号字段**实现：

```javascript
// 1. 读取时记录版本号
const doc = db.products.findOne({ _id: productId });
const version = doc.version;

// 2. 更新时带上版本号条件
const result = db.products.updateOne(
    { _id: productId, version: version },  // 版本号必须匹配
    { 
        $set: { stock: doc.stock - 1 },
        $inc: { version: 1 }               // 更新时递增版本号
    }
);

// 3. 检查是否更新成功
if (result.matchedCount === 0) {
    // 版本不匹配，被其他操作修改了 → 重试
    throw new Error("Optimistic lock conflict, please retry");
}
```
