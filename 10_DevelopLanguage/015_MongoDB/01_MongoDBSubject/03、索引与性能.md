### 三、索引与性能类

---

###### 1. 说说 MongoDB 的索引

MongoDB 索引和关系型数据库一样，本质是用空间换时间的数据结构（B-Tree），用于加速查询。

**核心原则**：
- 没有索引 → MongoDB 全集合扫描（COLLSCAN），百万数据毫秒变秒级
- 有合适索引 → 直接通过索引树定位（IXSCAN），效率提升数十倍
- 索引代价：占用额外磁盘和内存，写入时要同步维护索引（写性能略降）

**索引的底层数据结构**：B-Tree（平衡树），支持范围查询；地理索引使用 R-Tree；文本索引使用倒排索引。

---

###### 2. MongoDB 有哪些索引类型？

| 索引类型 | 创建方式 | 特点 |
|---------|---------|------|
| 单字段索引 | `db.col.createIndex({field: 1})` | 最基础，升序(1)/降序(-1) |
| 复合索引 | `db.col.createIndex({a:1, b:1})` | 遵循最左前缀原则 |
| 多键索引（Multikey） | 对数组字段自动创建 | 为数组中每个元素创建索引条目 |
| 文本索引 | `db.col.createIndex({title:"text"})` | 支持全文搜索（`$text`）；每个集合只能有一个文本索引 |
| 地理空间索引 | `db.col.createIndex({loc:"2dsphere"})` | 支持 `$near`、`$geoWithin` 等地理查询 |
| 哈希索引 | `db.col.createIndex({field:"hashed"})` | 等值查询快，不支持范围查询；分片键常用 |
| TTL 索引 | `db.col.createIndex({date:1},{expireAfterSeconds:3600})` | 自动删除过期文档（日志、会话） |
| 稀疏索引 | `db.col.createIndex({field:1},{sparse:true})` | 只索引包含该字段的文档（节省空间） |
| 部分索引 | `db.col.createIndex({field:1},{partialFilterExpression:{status:"active"}})` | 只索引满足条件的文档 |
| 唯一索引 | `db.col.createIndex({email:1},{unique:true})` | 强制字段值唯一 |

---

###### 3. 什么是复合索引？如何创建？

复合索引是对**多个字段组合**建立的索引：

```javascript
// 创建复合索引（状态升序 + 创建时间降序）
db.orders.createIndex({ status: 1, createdAt: -1 })

// 查询时可以高效使用
db.orders.find({ status: "pending" }).sort({ createdAt: -1 })
```

**最左前缀原则**（复合索引的核心规则）：
```
索引：{ a: 1, b: 1, c: 1 }

能用到索引：
  {a: 1}         ✅  使用前缀 a
  {a: 1, b: 1}   ✅  使用前缀 a+b
  {a: 1, b: 1, c: 1}  ✅  完整使用
  {a: 1, c: 1}   ✅  使用 a，c 无索引（跳过 b）

用不到索引：
  {b: 1}         ❌  没有 a 前缀
  {b: 1, c: 1}   ❌  没有 a 前缀
```

**ESR 规则**（复合索引字段顺序最佳实践）：
- **E**（Equality）等值字段放最前
- **S**（Sort）排序字段放中间
- **R**（Range）范围字段放最后

###### 4. 如何查看 MongoDB 的查询执行计划？

```javascript
// 基础 explain
db.orders.find({ status: "pending" }).explain()

// 详细执行计划（推荐）
db.orders.find({ status: "pending" }).explain("executionStats")

// 完整计划（包含被拒绝的候选索引）
db.orders.find({ status: "pending" }).explain("allPlansExecution")
```

**关键字段解读**：
```json
{
    "queryPlanner": {
        "winningPlan": {
            "stage": "IXSCAN",      // IXSCAN=索引扫描(好); COLLSCAN=全表扫描(差)
            "indexName": "status_1",
            "direction": "forward"
        }
    },
    "executionStats": {
        "nReturned": 100,           // 返回文档数
        "totalKeysExamined": 100,   // 扫描索引键数（越接近 nReturned 越好）
        "totalDocsExamined": 100,   // 扫描文档数（越接近 nReturned 越好）
        "executionTimeMillis": 2    // 执行时间(ms)
    }
}
```

**判断索引是否有效**：`totalDocsExamined / nReturned ≈ 1` 说明索引效率高；比值越大说明扫描了大量无效文档。

###### 5. MongoDB 如何进行性能优化？

**索引优化**：
- 为常用查询条件建立复合索引，遵循 ESR 规则
- 避免不必要的索引（每个索引都有维护成本）
- 定期用 `db.collection.getIndexes()` 检查，删除冗余索引
- 使用 `hint()` 强制指定索引（当优化器选错时）

**查询优化**：
- 使用**投影**（Projection）只返回需要的字段：`find({}, {name:1, age:1, _id:0})`
- 避免 `$where`（JavaScript 执行），使用 MQL 原生查询
- 避免大 skip()（深翻页），用游标/`_id` 范围查询替代
- `$regex` 查询加前缀锚点 `^` 才能用索引：`{name: /^张/}` vs `{name: /张/}`（后者全扫描）

**Schema 优化**：
- 频繁一起查询的数据嵌套存储（减少 $lookup）
- 预计算聚合结果存为字段（如总评论数直接存在文章文档里）
- 避免单文档过大（> 1MB 时考虑分离）

**硬件和配置**：
- 确保 WiredTiger cache（默认内存的 50%）能放下工作集（热数据）
- 使用 SSD 存储，减少 I/O 延迟

---

###### 6. 什么是覆盖查询（Covered Query）？

覆盖查询是指**查询结果所需的所有字段都包含在索引中**，MongoDB 不需要回文档（回表）就能直接从索引返回结果，性能极高。

```javascript
// 索引：{ status: 1, name: 1 }
// 查询：只需要 status 和 name，且过滤也是 status
db.users.find(
    { status: "active" },     // 查询条件在索引里
    { status: 1, name: 1, _id: 0 }  // 投影字段都在索引里（必须排除 _id！）
)
// → 覆盖查询，只走索引，不读磁盘文档

// explain() 验证：
// "inputStage": { "stage": "IXSCAN" }（只有 IXSCAN，没有 FETCH 阶段）
```

**关键**：必须在 Projection 中**排除 `_id`**（用 `_id: 0`），否则 MongoDB 仍会去文档里取 `_id`，破坏覆盖查询。

###### 7. MongoDB 的慢查询如何排查和优化？

```javascript
// 1. 开启慢查询日志（记录超过 100ms 的查询）
db.setProfilingLevel(1, { slowms: 100 })

// 2. 查询慢操作记录
db.system.profile.find().sort({ millis: -1 }).limit(10)

// 3. 用 explain 分析具体慢查询
db.collection.find(slowQuery).explain("executionStats")

// 4. 查看当前正在执行的操作
db.currentOp({ "active": true, "secs_running": { $gt: 1 } })

// 5. 终止慢查询
db.killOp(opid)

// 6. 查看所有索引使用情况（未使用的索引候选删除）
db.collection.aggregate([{ $indexStats: {} }])
```

**常见慢查询原因**：
- 缺少索引（COLLSCAN）→ 建索引
- 索引字段顺序不当 → 按 ESR 规则重建
- 大结果集未分页 → 加 `limit()`
- 聚合管道早期阶段未过滤数据 → `$match` 放最前，`$project` 尽早减少字段
- 内存排序超 100MB 限制 → 在排序字段上建索引

###### 追问：MongoDB 的索引如何选择升序(1)还是降序(-1)？

**单字段索引**：升序和降序无差别，MongoDB 可以双向遍历索引。

**复合索引**：排序方向非常重要！当复合排序（多字段 sort）时，索引中每个字段的方向必须和 sort 中**完全一致或完全相反**才能用上索引：

```javascript
// 索引：{ a: 1, b: -1 }

// ✅ 完全一致
db.col.find().sort({ a: 1, b: -1 })

// ✅ 完全相反（可以反向遍历索引）
db.col.find().sort({ a: -1, b: 1 })

// ❌ 方向不一致（无法用索引排序，内存排序）
db.col.find().sort({ a: 1, b: 1 })
```
