### 一、基础概念类

---

###### 1. MongoDB 是什么？

MongoDB 是一款开源的**文档型 NoSQL 数据库**，由 MongoDB Inc. 于 2009 年发布。它以 **BSON（Binary JSON）格式**存储数据，每条记录是一个灵活的 JSON 文档，不需要预先定义 Schema。

**核心特点**：
- **无 Schema**：每个文档字段可以不同，适合动态变化的数据结构
- **水平扩展**：内置分片（Sharding）支持，轻松应对 TB 级数据
- **高可用**：副本集（Replica Set）自动故障转移，RPO 接近 0
- **丰富查询**：支持嵌套文档查询、正则、地理空间、全文搜索、聚合管道
- **多文档事务**：4.0+ 版本支持，与关系型数据库相当

---

###### 2. 什么是 NoSQL 数据库？

NoSQL（Not Only SQL）是对非关系型数据库的统称，主要解决传统关系型数据库在**高并发、大数据量、灵活 Schema** 场景下的局限。

**NoSQL 四大类型**：

| 类型 | 代表产品 | 数据模型 | 适用场景 |
|------|---------|---------|---------|
| 文档型 | MongoDB、CouchDB | JSON/BSON 文档 | 内容管理、电商、用户画像 |
| 键值型 | Redis、DynamoDB | Key-Value | 缓存、会话、排行榜 |
| 列族型 | HBase、Cassandra | 稀疏列 | 时序数据、日志、IoT |
| 图型 | Neo4j | 节点+边 | 社交网络、知识图谱 |

---

###### 3. MongoDB 的存储结构是什么？

```
MongoDB 存储层次：
  数据库（Database）
    └── 集合（Collection）─ 类似关系型的"表"
          └── 文档（Document）─ 类似关系型的"行"，BSON 格式
                └── 字段（Field）─ 类似关系型的"列"，但每个文档可不同
```

**物理存储**（WiredTiger 引擎）：
- 每个数据库对应一个目录（`/data/db/dbname/`）
- 每个集合对应一个 `.wt` 文件（WiredTiger 格式）
- 另有 `.wt` 索引文件和 `WiredTiger.wt` 元数据文件

---

###### 4. 什么是 BSON 格式，与 JSON 的区别，有什么优势？

BSON（Binary JSON）是 MongoDB 用于存储和传输数据的二进制序列化格式。

**BSON vs JSON 对比**：

| 维度 | JSON | BSON |
|------|------|------|
| 编码 | 文本（UTF-8） | 二进制 |
| 大小 | 较小（纯文本无额外开销） | 通常稍大（包含类型信息和长度前缀） |
| 解析速度 | 较慢（文本解析） | **极快**（二进制，无需扫描分隔符） |
| 数据类型 | 少（只有 string/number/bool/null/array/object） | 丰富（Date/ObjectId/Binary/Decimal128/Int32/Int64） |
| 可遍历 | 需全部解析 | **支持快速跳过字段**（有长度前缀） |

**BSON 的核心优势**：
1. **更丰富的类型**：支持 `Date`（精确时间戳）、`ObjectId`、`Binary`、`Decimal128`（精确小数）
2. **快速扫描**：每个字段有长度前缀，可直接跳到下一个字段，无需全文本扫描
3. **高效存储和传输**：二进制比文本解析快 10 倍以上

---

###### 5. MongoDB 中的集合（Collection）和文档（Document）是什么？

**集合（Collection）**：
- 相当于关系型数据库的"表"，但**无固定 Schema**
- 同一集合中的文档可以有不同的字段
- 集合名区分大小写，不能包含 `$` 或空字符
- 建议同类数据放同一集合（方便索引和查询）

**文档（Document）**：
- 相当于关系型数据库的"行"
- 以 BSON 格式存储，最大 **16MB**（超出用 GridFS）
- 必须有 `_id` 字段（自动生成 ObjectId，或手动指定）
- 支持嵌套文档和数组

```javascript
// 文档示例
{
    "_id": ObjectId("65a1b2c3d4e5f6789012"),
    "name": "张三",
    "age": 28,
    "address": {              // 嵌套文档
        "city": "北京",
        "district": "朝阳区"
    },
    "skills": ["Java", "MongoDB"],  // 数组
    "createdAt": ISODate("2024-01-01T00:00:00Z")
}
```

---

###### 6. MongoDB 中的 _id 字段有什么作用？

`_id` 是 MongoDB 文档的**主键**，每个集合中唯一：
- 如果插入文档时未指定 `_id`，MongoDB 自动生成一个 `ObjectId`
- 可以手动指定任何唯一值（String、Number、UUID 等）
- 默认会在 `_id` 上创建**唯一升序索引**

---

###### 7. 什么是 ObjectId？它的组成结构是什么？

ObjectId 是 MongoDB 默认的主键类型，**12 字节**二进制，用 24 位十六进制字符串表示：

```
ObjectId("65a1b2c3 d4e5f6 789012")
          ─────────────────────────
          4字节    3字节   3字节  2字节
          Unix时间戳  机器标识  进程ID  随机计数器
```

**特点**：
- **全局唯一**：时间戳 + 机器 + 进程 + 计数器，理论上不会重复
- **有序性**：前 4 字节是时间戳，ObjectId 按生成时间单调递增（同一秒内随机）
- **包含时间信息**：`ObjectId.getTimestamp()` 可提取创建时间，不需要额外的 `createdAt` 字段

---

###### 追问：MongoDB 的文档大小限制是多少？超出怎么办？

单个文档最大 **16MB**，这是为了防止文档过大影响性能（每次读取/传输都要加载整个文档）。

**超出 16MB 的解决方案**：使用 **GridFS**
- GridFS 将大文件分割成 255KB 的 chunks（块），存入两个集合：`fs.files`（元数据）和 `fs.chunks`（数据块）
- 适合存储图片、视频、音频、PDF 等大文件
- 支持断点续传和流式读取
