### 八、场景应用类

---

###### 1. 说说 MongoDB 适用哪些场景？如何选择？

**MongoDB 最擅长的场景**：

1. **内容管理**：博客、新闻、CMS — 不同类型内容字段各异，无固定 Schema
2. **用户画像/配置**：用户偏好、行为标签 — 字段动态增减，嵌套结构
3. **产品目录/电商**：不同品类商品属性差异大（手机有内存/屏幕，衣服有尺码/颜色）
4. **事件/日志记录**：高吞吐写入，不需要强事务
5. **实时分析**：聚合管道 + 时序数据
6. **地理空间应用**：附近商家、位置轨迹 — 原生支持 GeoJSON 和地理索引
7. **物联网（IoT）**：设备状态/传感器数据 — 高写入低延迟

**不适合 MongoDB 的场景**：
- 强 ACID 事务（转账、支付核心链路）→ 选 MySQL/PostgreSQL
- 复杂多表 JOIN 报表 → 选 MySQL 或专用 BI 数据仓库
- 数据严格规范化，需要外键约束 → 选关系型数据库

---

###### 2. MongoDB 在什么场景下不适合使用？

1. **金融核心业务**：资金转账、对账 — 对强一致性要求极高，MongoDB 多文档事务虽然支持，但复杂度和性能不如 MySQL
2. **复杂关联查询**：几十张表 JOIN，需要 SQL 复杂分析 — MongoDB `$lookup` 性能远不如 SQL JOIN
3. **需要外键约束**：MongoDB 没有外键，引用完整性需要应用层保证
4. **传统 ERP/MES 系统**：高度结构化，业务规则复杂，关系型更合适
5. **存储超大文件**：单文档 16MB 限制，虽然有 GridFS，但不如专用对象存储（OSS/S3）

---

###### 3. MongoDB 如何处理大数据量的存储？

**单节点优化**：
- 压缩（zstd 压缩率高达 70-80%，减少磁盘占用）
- TTL 索引自动删除过期数据（日志、会话、历史记录）
- 分离冷热数据：热数据正常存储，冷数据压缩后归档

**水平扩展**：
- 分片集群（Sharding）将数据分布到多台服务器
- 选择合适的分片键（高基数、写入均匀）
- 区域分片（Zone Sharding）实现数据本地化

**读扩展**：
- 副本集的 Secondary 节点提供读服务（`readPreference: secondaryPreferred`）
- 多个 Secondary 分摊读压力

---

###### 4. MongoDB 如何实现数据的冷热分离？

**方案一：TTL + 定期归档**
```javascript
// 热数据集合（添加 TTL 索引，30天后过期）
db.createCollection("hot_logs")
db.hot_logs.createIndex({ createdAt: 1 }, { expireAfterSeconds: 2592000 })

// 定时任务：将即将过期的数据迁移到冷数据集合
const now = new Date();
const coldDate = new Date(now.getTime() - 25 * 24 * 3600 * 1000);  // 25天前
const coldData = db.hot_logs.find({ createdAt: { $lt: coldDate } }).toArray();
db.cold_logs.insertMany(coldData);
db.hot_logs.deleteMany({ createdAt: { $lt: coldDate } });
```

**方案二：Zone Sharding（地理/时间分区）**
```javascript
// 按时间分片（最近3个月在 SSD Shard，历史数据在 HDD Shard）
sh.addTagRange("mydb.logs", 
    { createdAt: ISODate("2024-01-01") }, 
    { createdAt: MaxKey }, 
    "hot"  // 标签
)
sh.addShardTag("shard01", "hot")  // SSD shard
sh.addShardTag("shard02", "cold") // HDD shard
```

### 九、安全与备份类

###### 1. MongoDB 的安全机制有哪些？

1. **认证（Authentication）**：SCRAM-SHA-256（默认）、X.509 证书、LDAP
2. **授权（Authorization）**：基于角色的访问控制（RBAC），内置角色（read/readWrite/dbAdmin/clusterAdmin 等）
3. **传输加密（TLS/SSL）**：客户端-服务器、服务器间通信加密
4. **静态加密**：Encrypted Storage Engine（Queryable Encryption，6.0+）
5. **审计日志**：记录所有认证、授权、DDL/DML 操作（企业版）
6. **网络隔离**：`bindIp` 限制监听地址，防火墙只开放必要端口（27017）

###### 3. MongoDB 的备份和恢复方式有哪些？

| 方式 | 工具 | 特点 |
|------|------|------|
| 逻辑备份 | `mongodump` / `mongorestore` | 导出 BSON 文件；简单，跨版本兼容；数据量大时慢 |
| 文件系统快照 | `LVM snapshot` / 云盘快照 | 速度快（分钟级）；需要锁数据库或使用一致性快照 |
| 持续备份 | 基于 Oplog 的增量备份 | 可以恢复到任意时间点（Point-in-Time Recovery） |

###### 4. mongodump 和 mongorestore 的使用场景是什么？

```bash
# 备份整个实例
mongodump --uri="mongodb://user:password@localhost:27017" --out=/backup/$(date +%Y%m%d)

# 备份单个数据库
mongodump --db=mydb --out=/backup/

# 备份单个集合
mongodump --db=mydb --collection=orders --out=/backup/

# 压缩备份（节省空间）
mongodump --db=mydb --gzip --out=/backup/

# 恢复
mongorestore --uri="mongodb://user:password@localhost:27017" --dir=/backup/20240101/

# 恢复单个集合（--drop 先删除再恢复）
mongorestore --db=mydb --collection=orders --drop /backup/mydb/orders.bson
```

**适用场景**：小数据量（< 100GB）的备份恢复；跨版本迁移；定期全量备份。

**不适用场景**：TB 级数据的备份（太慢）→ 改用文件系统快照。

---

###### 5. MongoDB 如何实现增量备份？

基于 **Oplog（操作日志）** 实现增量备份：

```bash
# 1. 先做一次全量备份，记录 Oplog 时间点
mongodump --oplog --out=/backup/full/

# 2. 之后每次增量备份只复制新的 Oplog 条目
mongodump --oplog --oplogLimit="<timestamp:i>" --out=/backup/incr/

# 3. 恢复全量 + 增量（oplogReplay 回放 Oplog）
mongorestore --oplogReplay /backup/full/
mongorestore --oplogReplay --oplogLimit="<恢复点时间戳>" /backup/incr/
```

---

###### 6. MongoDB 的 Oplog 是什么？有什么作用？

Oplog（Operations Log）是 MongoDB 副本集的**核心复制机制**，存储在 `local.oplog.rs` 的固定大小集合中：

**作用**：
1. **数据复制**：Secondary 节点不断从 Primary 的 Oplog 中读取操作，在本地重放，保持数据同步
2. **故障恢复**：配合快照实现 Point-in-Time Recovery（PITR）
3. **CDC（变更数据捕获）**：Change Streams 的底层是 Oplog，可实现实时数据同步到其他系统

**Oplog 特性**：
- 固定大小（Capped Collection），默认占磁盘的 5%，最小 1GB
- 操作以**幂等**形式记录（如 `$inc: {count: 1}` 记录为 `$set: {count: 5}`）
- Secondary 落后 Primary 过多，且 Oplog 被覆盖 → Secondary 进入 RECOVERING 状态，需要全量同步

```javascript
mongotop --uri "mongodb://..." 1               // 每秒显示各集合读写时间
```

###### 2. MongoDB 常用的监控指标有哪些？

| 指标分类 | 关键指标 | 健康参考值 |
|---------|---------|-----------|
| 连接数 | `connections.current / available` | 使用率 < 80% |
| 内存 | `wiredTiger.cache.bytes currently in cache` | < cache 上限 80% |
| 操作延迟 | `opcounters.*（insert/query/update/delete QPS）` | 与压测基线对比 |
| 复制延迟 | `replicationInfo.timeLag`（Secondary 落后秒数） | < 5s |
| 锁等待 | `globalLock.currentQueue.total` | 接近 0 |
| 慢查询 | `system.profile` 中超过阈值的查询 | 无高频慢查询 |
| 磁盘 I/O | WiredTiger 磁盘读写速率 | 与磁盘能力匹配 |

###### 3. 如何排查 MongoDB 的内存占用问题？

```javascript
// 1. 检查 WiredTiger Cache 使用情况
db.serverStatus().wiredTiger.cache

// 2. 检查各集合索引大小
db.collection.stats({ indexDetails: true })

// 3. 找出占用内存最多的集合
db.adminCommand({ listDatabases: 1 }).databases.forEach(function(d) {
    db = db.getSiblingDB(d.name);
    db.getCollectionNames().forEach(function(c) {
        const stats = db[c].stats();
        print(`${d.name}.${c}: ${stats.size} bytes`);
    });
})

// 常见内存问题：
// - 索引过多：每个索引都占内存，定期清理无用索引
// - WiredTiger cache 不足：工作集无法全部放入缓存，频繁磁盘读取
// - 连接数过多：每个连接约占 1MB 内存
```

###### 4. MongoDB 的锁机制是怎样的？

WiredTiger 使用 **文档级锁（Document-Level Locking）** + **乐观并发控制（MVCC）**：

- **读操作**：使用 MVCC 快照，读不阻塞写
- **写操作**：文档级锁，不同文档的写操作可并发
- **集合级操作**（如 `createIndex`、`drop`）：会取集合级写锁，短暂阻塞其他操作
- **意向锁**：全局/数据库/集合级别有意向锁，协调并发

```javascript
// 查看当前锁状态
db.serverStatus().globalLock
// currentQueue.total > 0 说明有操作在等待锁
```

###### 5. 如何处理 MongoDB 的连接数过多问题？

```javascript
// 1. 查看当前连接数
db.serverStatus().connections
// { current: 120, available: 880, totalCreated: 5000 }

// 解决方案：
// a) 使用连接池（应用端配置）
MongoClient.connect(uri, {
    maxPoolSize: 100,     // 最大连接池大小（默认 100）
    minPoolSize: 5,       // 最小保持连接数
    maxIdleTimeMS: 60000  // 空闲连接 60s 后关闭
})

// b) 关闭空闲连接
// 检查是否有未关闭连接（应用 Bug）
db.currentOp().inprog.forEach(function(op) {
    if (op.secs_running > 300) print(op.opid, op.client);  // 查看长时间连接
})

// c) 调整 MongoDB 最大连接数（一般不需要）
// net.maxIncomingConnections: 1000000 （默认很大）
```

###### 1. MongoDB 的 Change Streams 是什么？

Change Streams（变更流）是 MongoDB 3.6+ 的功能，允许应用程序**实时监听集合、数据库或整个实例的数据变更**。

```javascript
// 监听订单集合的变更（Node.js）
const changeStream = db.collection('orders').watch([
    { $match: { operationType: { $in: ["insert", "update"] } } }
]);

changeStream.on("change", (change) => {
    console.log("变更事件:", change.operationType, change.fullDocument);
    // 可以触发：发送通知、同步到其他系统、清除缓存等
});
```

**应用场景**：
- 数据库 → 消息队列（实时数据同步）
- 数据库 → ElasticSearch（搜索索引实时更新）
- 库存变更 → 实时通知用户
- 审计日志（变更追踪）

###### 2. MongoDB 的 GridFS 是什么？适用于什么场景？

GridFS 是 MongoDB 存储超过 16MB 大文件的规范，将文件分割成 **255KB** 的 chunks：

```
GridFS 两个集合：
  fs.files：文件元数据（文件名/大小/上传时间/MD5/自定义元数据）
  fs.chunks：文件数据块（每块 255KB，按 files_id + chunkIndex 关联）
```

```python
# Python 使用 GridFS
from gridfs import GridFS
fs = GridFS(db)

# 上传文件
with open("video.mp4", "rb") as f:
    file_id = fs.put(f, filename="video.mp4", content_type="video/mp4")

# 下载文件
grid_out = fs.get(file_id)
with open("output.mp4", "wb") as f:
    f.write(grid_out.read())
```

**适用场景**：
- 需要存储大于 16MB 的文件（视频、PDF、备份文件）
- 需要和 MongoDB 文档数据在同一系统中管理文件
- 需要对文件进行范围读取（如视频分片传输）

**不适用场景**：高并发大规模文件访问 → 推荐 OSS/S3 对象存储。

###### 4. MongoDB 如何支持时间序列数据？

MongoDB 5.0+ 提供了专用的**时序集合（Time Series Collection）**：

```javascript
// 创建时序集合
db.createCollection("sensor_data", {
    timeseries: {
        timeField: "timestamp",    // 时间字段
        metaField: "metadata",     // 元数据字段（设备 ID 等）
        granularity: "minutes"     // 时间粒度（seconds/minutes/hours）
    },
    expireAfterSeconds: 7776000    // 90 天后自动过期
})

// 插入时序数据
db.sensor_data.insertMany([
    { timestamp: new Date(), metadata: { sensorId: "S001", location: "北京" }, temperature: 25.6, humidity: 60 },
    { timestamp: new Date(), metadata: { sensorId: "S002", location: "上海" }, temperature: 28.1, humidity: 75 }
])
```

**时序集合优势**：自动优化存储格式（相邻时间数据合并存储），写入和查询性能比普通集合高 5-10 倍，且存储空间减少 60-90%。

###### 5. MongoDB 的 TTL 索引是什么？

TTL（Time-To-Live）索引是一种特殊的单字段索引，MongoDB 后台进程每 60 秒扫描一次，**自动删除过期文档**：

```javascript
// 文档在 createdAt 后 3600 秒（1小时）自动删除
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 })

// 用 expireAt 字段指定精确过期时间（expireAfterSeconds 设为 0）
db.scheduled_jobs.createIndex({ expireAt: 1 }, { expireAfterSeconds: 0 })
db.scheduled_jobs.insertOne({ task: "清理临时文件", expireAt: new Date("2024-12-31") })
```

**适用场景**：会话（Session）、验证码、日志、临时缓存、优惠券有效期。

**注意**：TTL 索引字段必须是 `Date` 类型，且字段值不能是数组。删除操作是后台进行，不是精确到秒的实时删除。

---

###### 6. MongoDB 的部分索引（Partial Index）是什么？

只对**满足特定条件的文档**建立索引，节省存储空间和内存：

```javascript
// 只对状态为 "active" 的订单建索引（假设 99% 的查询都在 active 状态）
db.orders.createIndex(
    { userId: 1, createdAt: -1 },
    { partialFilterExpression: { status: "active" } }
)

// 查询时，条件中必须包含过滤条件才能用到部分索引
db.orders.find({ userId: "u001", status: "active" })   // ✅ 使用部分索引
db.orders.find({ userId: "u001" })                      // ❌ 不使用（没有 status 条件）
```

---

###### 7. MongoDB 的稀疏索引（Sparse Index）是什么？

稀疏索引**只索引包含该字段的文档**，不存在该字段的文档不出现在索引中：

```javascript
// 普通索引：所有文档都在索引中（字段不存在时索引值为 null）
db.users.createIndex({ email: 1 })

// 稀疏索引：只有 email 字段存在的文档才进入索引
db.users.createIndex({ email: 1 }, { sparse: true })

// 适用场景：可选字段 + 唯一约束
// （普通唯一索引会将两个都没有 email 的文档判定为重复，稀疏索引不会）
db.users.createIndex({ email: 1 }, { unique: true, sparse: true })
```

### 十二、实战问题类

---

###### 1. 在生产环境中遇到 MongoDB 性能瓶颈如何处理？

**诊断流程**：
```
1. mongostat / mongotop → 确认是读瓶颈还是写瓶颈
2. db.currentOp() → 找到正在执行的慢操作
3. db.system.profile → 分析历史慢查询
4. explain("executionStats") → 确认具体查询是否走索引
5. db.serverStatus().wiredTiger.cache → 确认 Cache 是否充足
```

**常见瓶颈及解决**：
- **慢查询**：添加缺失索引；优化聚合管道（$match 前置）
- **Cache 不足**：增大 `wiredTigerCacheSizeGB`；增加 RAM；读副本集分担
- **写入瓶颈**：批量写入 `insertMany`；关闭不必要索引；调整 `writeConcern`
- **连接数耗尽**：优化连接池配置；检查应用连接泄漏

###### 2. MongoDB 数据迁移有哪些方案？

| 场景 | 方案 |
|------|------|
| 同版本迁移（小量） | `mongodump / mongorestore` |
| 同版本迁移（大量） | `mongomirror`（在线实时同步，无停机） |
| 跨版本升级 | 滚动升级副本集成员（先升 Secondary，最后 Primary）|
| MySQL → MongoDB | ETL 工具（Kettle / 自研脚本）+ 数据映射转换 |
| MongoDB → Elasticsearch | Change Streams / Logstash MongoDB Input |

###### 3. 如何从 MySQL 迁移到 MongoDB？

**迁移步骤**：
1. **数据模型转换**：重新设计 MongoDB Schema（以查询为导向，考虑嵌套 vs 引用）
2. **ETL 数据迁移**：导出 MySQL 数据（CSV/JSON），转换后导入 MongoDB
3. **双写验证**：新功能同时写入 MySQL 和 MongoDB，对比数据一致性
4. **灰度切换**：逐步将读请求切到 MongoDB，验证无误后迁移写请求
5. **下线 MySQL**

---

###### 4. MongoDB 如何进行容量规划？

| 维度 | 估算方法 |
|------|---------|
| 存储 | 原始数据大小 × (1 + 索引比例) × (1 - 压缩率) × 副本数 |
| 内存 | 工作集大小（热数据）≤ WiredTiger Cache（RAM × 50%） |
| CPU | 通常不是瓶颈；聚合/索引建立时 CPU 使用率高 |
| 网络 | 副本集同步带宽 + 应用读写带宽 |

**经验参考**：Snappy 压缩后存储约为原始 40%；工作集 = 最近 30 天活跃数据（根据业务定义）。

###### 5. 遇到 MongoDB 数据不一致问题如何排查？

1. **读写一致性问题**：检查 `readConcern` 设置（默认 `local`，可能读到未复制的数据）→ 改用 `majority` 确保读到多数派确认的数据
2. **副本集延迟**：`rs.printSlaveReplicationInfo()` 查看 Secondary 落后时间
3. **应用层 Bug**：排查并发写操作是否存在竞态条件 → 用原子操作或事务
4. **网络分区**：检查副本集成员连通性（`rs.status()` 中的 `health` 字段）

---

###### 6. MongoDB 在微服务架构中如何应用？

- **每个微服务一个 MongoDB 数据库**（Database per Service 模式），服务间不共享数据库
- **跨服务数据关联**：通过 API 调用或消息队列（而不是数据库 JOIN）
- **Change Streams 实现事件驱动**：一个服务的数据变更，通过 Change Streams 发布到消息队列，其他服务订阅

---

###### 7. MongoDB 如何与 Elasticsearch 集成？

双向集成方案：
1. **MongoDB → ES（单向同步）**：Logstash（jdbc/mongodb input plugin）定时拉取；Change Streams 实时推送到 Kafka，ES 消费
2. **典型架构**：MongoDB 作为主数据库（CRUD），Elasticsearch 作为搜索引擎（全文搜索/复杂分析）
3. **数据一致性**：双写保证一致（写 MongoDB 成功后写 ES，ES 写失败则重试）；或 Canal-like 工具监听 Oplog 同步

###### 追问：MongoDB 的 Change Streams 和 Oplog 的关系是什么？

Change Streams 是建立在 Oplog 上的**高级抽象 API**：

- **底层**：Change Streams 从 `local.oplog.rs` 读取操作记录
- **封装**：提供了更友好的接口（支持过滤、恢复令牌），屏蔽了 Oplog 的底层格式
- **区别**：直接读 Oplog 需要特权（访问 local 数据库），Change Streams 普通用户就可以使用
- **恢复令牌（Resume Token）**：Change Streams 提供断点续传，应用重启后从上次处理的位置继续
