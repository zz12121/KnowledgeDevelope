### 二、对比分析类

---

###### 1. MongoDB 与 MySQL 的区别？

| 对比维度 | MongoDB | MySQL |
|---------|---------|-------|
| 数据模型 | 文档型（BSON，无 Schema） | 关系型（表、行、列，固定 Schema） |
| 查询语言 | MongoDB Query Language（MQL） | SQL |
| Schema | 动态灵活，可随时增加字段 | 需预先定义，修改结构代价高 |
| 事务 | 4.0+ 支持多文档事务，略重 | ACID 事务，成熟稳定 |
| 关联查询 | $lookup（聚合阶段），不如 SQL JOIN 灵活 | JOIN 强大，多表关联高效 |
| 水平扩展 | **原生分片**，天然支持水平扩展 | 需要中间件（ShardingSphere 等） |
| 索引 | 支持单字段/复合/地理/文本/稀疏/TTL | 支持 B+树/哈希/全文索引 |
| 写入性能 | 极高（默认异步刷盘，可关闭 Journal） | 高（WAL+Buffer Pool） |
| 适合场景 | 动态 Schema、嵌套数据、高写入、水平扩展 | 强一致性、复杂关联、金融业务 |

**一句话总结**：MongoDB 以灵活性和扩展性见长，MySQL 以一致性和关联查询见长；不是替代关系，而是互补。

---

###### 2. MongoDB 与 Redis 的区别和使用场景？

| 对比维度 | MongoDB | Redis |
|---------|---------|-------|
| 存储位置 | 磁盘（内存缓存 + 持久化） | **内存**（可选持久化） |
| 数据结构 | BSON 文档（任意复杂结构） | String/Hash/List/Set/ZSet/Bitmap/Stream |
| 查询能力 | **强**（聚合管道、索引、全文搜索） | 弱（只能按 Key 或数据结构特性查询） |
| 数据量 | TB 级，磁盘存储 | GB 级，受内存限制 |
| 持久化 | 可靠（Journal + WiredTiger checkpoint） | 可选（RDB/AOF，可能丢数据） |
| 事务 | 支持多文档 ACID 事务 | 支持 Lua 脚本原子操作，无完整事务 |
| 使用场景 | 主数据存储（内容/用户/订单） | 缓存、会话、排行榜、分布式锁、消息队列 |

**典型组合**：MongoDB 作为主数据库存储业务数据，Redis 作为缓存层加速查询，二者互补。

---

###### 3. MongoDB 与关系型数据库的术语对应关系？

| 关系型数据库 | MongoDB | 说明 |
|------------|---------|------|
| Database | Database | 数据库，完全一致 |
| Table（表） | Collection（集合） | MongoDB 无固定 Schema |
| Row（行） | Document（文档） | BSON 格式，最大 16MB |
| Column（列） | Field（字段） | 每个文档可以有不同字段 |
| Primary Key | _id | 自动生成 ObjectId 或手动指定 |
| Foreign Key | 引用（DBRef 或手动存 _id） | 没有外键约束，需应用层保证一致性 |
| Index | Index | 相似，但支持更多类型（地理/文本/TTL） |
| JOIN | $lookup（聚合） | 不如 SQL JOIN 高效，建议嵌套设计 |
| Transaction | Transaction（4.0+） | 多文档 ACID 事务 |
| View | View（3.4+） | 只读视图，基于聚合管道 |
| Stored Procedure | JavaScript 函数 + $function | 支持但不推荐 |

###### 追问：什么场景该选 MongoDB，什么场景该选 MySQL？

**选 MongoDB 的场景**：
1. **数据结构经常变化**：产品早期快速迭代，Schema 变更频繁
2. **嵌套/层次结构数据**：文章+评论、用户+地址+订单一体化存储
3. **高写入低延迟**：日志、IoT 传感器数据、用户行为数据
4. **需要水平扩展**：数据量预期 TB 级，需要分片
5. **地理空间查询**：附近的商家、轨迹数据
6. **内容管理系统**：不同类型内容字段各异

**选 MySQL 的场景**：
1. **金融/支付业务**：强 ACID 事务，不允许任何数据不一致
2. **复杂多表关联查询**：报表系统、BI 分析
3. **数据规范化需求高**：需要外键约束保证引用完整性
4. **DBA 团队熟悉 SQL**：运维成本更低
