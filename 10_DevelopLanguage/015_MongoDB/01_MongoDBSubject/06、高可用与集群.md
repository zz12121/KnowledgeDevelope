### 六、高可用与集群类

---

###### 1. 如何保证 MongoDB 高可用？

MongoDB 高可用通过**副本集（Replica Set）**实现，生产环境标准配置：

```
Primary（主节点）← 处理所有写操作，向 Secondary 同步 Oplog
    │
    ├── Secondary（从节点1）← 异步复制，可配置为只读
    └── Secondary（从节点2）← 异步复制，可参与选举
```

**最少 3 个节点**（形成多数派投票，防脑裂），或 2 节点 + 1 Arbiter（仲裁节点）。

---

###### 2. 说说 MongoDB 的主从复制

MongoDB 早期的主从复制已**废弃**（MongoDB 4.0 之后彻底移除），现在统一用副本集（Replica Set）实现数据同步，原理类似但更健壮。

**旧主从 vs 副本集**：
- 旧主从：主节点宕机后需要**手动**切换
- 副本集：自动故障检测和 Leader 选举，切换时间通常 **< 10s**

---

###### 3. 说说 MongoDB 的复制集（Replica Set）

副本集是 MongoDB 高可用和数据冗余的核心机制：

**成员类型**：
- **Primary（主节点）**：唯一接受写操作，将操作记录到 Oplog
- **Secondary（从节点）**：异步复制 Primary 的 Oplog，可配置为只读（`readPreference: secondary`）
- **Arbiter（仲裁节点）**：不存储数据，只参与选举投票（节省资源）
- **Hidden Member**：不对外提供服务，用于备份和特殊分析
- **Delayed Member**：延迟一段时间同步（如延迟 1 小时，防误操作）

**Oplog（Operations Log）**：
- 存储在 `local.oplog.rs` 集合中，固定大小（capped collection）
- 记录所有写操作（幂等形式），Secondary 循环读取并应用
- Oplog 大小决定了允许 Secondary 落后 Primary 多久（落后过多会被踢出副本集）

```javascript
// 查看副本集状态
rs.status()

// 查看当前节点角色
rs.isMaster()

// 查看 Oplog 状态（时间窗口）
rs.printReplicationInfo()
```

---

###### 4. 复制集中的选举机制是怎样的？

当 Primary 不可用时，副本集自动触发选举：

**触发条件**：
- Primary 心跳超时（默认 10s 内没有收到心跳）
- Primary 主动降级（维护操作）

**选举流程（Raft 变体）**：
1. 检测到 Primary 失联的 Secondary 等待随机时间后发起选举
2. 候选节点向其他成员请求投票
3. 投票规则：优先选 **Oplog 最新** 的节点（`lastOptime`）；其次考虑 `priority`（优先级）
4. 获得**多数派投票**（超过半数成员）的候选节点成为新 Primary
5. 新 Primary 开始接受写操作，其余节点从新 Primary 同步数据

**选举时间**：通常 10-30 秒内完成（期间写操作不可用！）

**防脑裂**：需要 `n/2 + 1` 节点同意才能完成选举，少于半数节点存活时无法完成选举（宁可不可用，也不产生数据冲突）

---

###### 5. 什么是副本集的心跳机制？

副本集成员间每隔 **2 秒**（默认）互相发送心跳（ping），用于检测成员存活状态。

- 连续 **3 次心跳超时**（即约 6 秒没有响应）→ 标记该成员为不可达
- Primary 不可达，且其他成员达到多数派 → 触发选举

心跳还携带 Oplog 进度、节点状态等信息，Secondary 根据此信息决定是否追赶 Primary。

---

###### 6. 副本集中的 Primary、Secondary、Arbiter 分别是什么？

| 角色 | 功能 | 存储数据 | 参与选举 | 接受读写 |
|------|------|---------|---------|---------|
| Primary | 处理所有写操作，同步 Oplog 给 Secondary | ✅ | ✅（可投票） | 读写均可 |
| Secondary | 复制 Primary 数据，可提供读服务 | ✅ | ✅（可投票） | 只读（需配置） |
| Arbiter | 只参与选举，不存储数据 | ❌ | ✅（只投票） | ❌ |

**Arbiter 的使用场景**：当只有 2 个数据节点时，加入 Arbiter 凑成 3 个投票成员，确保多数派选举能进行，同时节省一台服务器的存储成本。

**注意**：Arbiter 不存储数据，不能替代 Secondary 作为备份！

###### 7. 说说 MongoDB 的分片集群（Sharding）

分片集群将数据分散到多台服务器，实现水平扩展：

```
┌─────────────────────────────────────────┐
│  应用程序                                │
└───────────────────┬─────────────────────┘
                    │ 连接
          ┌─────────▼──────────┐
          │    mongos 路由进程  │  ← 查询路由，对应用透明
          └─────────┬──────────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ Shard 1  │ │ Shard 2  │ │ Shard 3  │  ← 每个 Shard 是一个副本集
  │（副本集） │ │（副本集） │ │（副本集） │
  └──────────┘ └──────────┘ └──────────┘
        ▲
  ┌─────┴─────┐
  │Config Server│  ← 存储分片元数据（哪个 chunk 在哪个 shard）
  │（副本集）   │
  └────────────┘
```

---

###### 8. MongoDB 分片的策略有哪些？

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 范围分片（Range） | 按片键值范围分配 chunk | 支持范围查询 | 可能数据热点（如时间戳递增导致所有写都在最后一个 shard） |
| 哈希分片（Hash） | 对片键哈希后均匀分布 | 写入均匀，无热点 | 范围查询需要广播到所有 shard |
| 区域分片（Zone） | 根据地理/业务区域分片 | 数据本地化（中国数据在中国 Shard） | 需要手动配置 |

---

###### 9. 什么是片键（Shard Key）？如何选择合适的片键？

片键（Shard Key）是决定文档分布在哪个 Shard 的字段（一旦确定无法修改！）

**好的片键标准**：
1. **高基数（High Cardinality）**：不同值尽量多（如 userId），避免过多文档在同一 chunk
2. **写入均匀**：不要用时间戳作哈希片键（但时间戳作范围片键会有热点问题）
3. **查询局部性**：片键应尽量包含在查询条件中，避免广播查询（Scatter Gather）

**常见片键选择**：
- `userId`（哈希）：用户数据按 userId 均匀分布
- `{region: 1, userId: 1}`（复合）：按地区+用户均衡分布，支持地区维度聚合

---

###### 10. MongoDB 的 Config Server 和 mongos 分别是什么？

**Config Server（配置服务器）**：
- 存储分片集群的**元数据**：chunk 的分布（哪个 chunk 在哪个 Shard）、片键范围、均衡状态
- 必须是副本集（生产环境），确保元数据高可用
- 如果 Config Server 不可用，mongos 无法路由新请求（但已缓存的路由还能工作一段时间）

**mongos（查询路由进程）**：
- 应用程序连接的入口，屏蔽了分片细节（应用无需知道数据在哪个 Shard）
- 从 Config Server 获取并缓存路由元数据
- 根据查询条件：定向路由（精确到单 Shard）或广播查询（所有 Shard）
- 聚合各 Shard 的结果返回给应用
- 无状态，可以部署多个 mongos 实现负载均衡

---

###### 11. 分片集群如何进行数据均衡？

**自动均衡（Balancer）**：
MongoDB 的 Balancer 进程（运行在 Config Server 上）负责监控各 Shard 上 chunk 数量，当某个 Shard 的 chunk 数量超过阈值时，自动将 chunk 迁移到较少的 Shard。

```
Shard1: [chunk1][chunk2][chunk3][chunk4][chunk5]  ← 5 chunks
Shard2: [chunk6]                                  ← 1 chunk
                   ↓ Balancer 触发迁移
Shard1: [chunk1][chunk2][chunk3]
Shard2: [chunk6][chunk4][chunk5]
```

**注意**：Chunk 迁移会消耗网络和 I/O 资源，建议在低峰期进行。可以配置均衡时间窗口：
```javascript
// 设置均衡只在凌晨 2-6 点进行
db.settings.updateOne(
    { _id: "balancer" },
    { $set: { activeWindow: { start: "02:00", stop: "06:00" } } },
    { upsert: true }
)
```

###### 追问：分片集群下的事务有什么限制？

MongoDB 4.2+ 支持跨分片的多文档事务，但有以下限制：
- **性能开销更大**：需要通过两阶段提交（2PC）协调跨 Shard 的操作
- **不支持 DDL**：事务中不能创建集合或索引
- **超时限制**：事务最大执行时间 60 秒
- **建议**：尽量通过 Schema 设计避免跨分片事务（将相关数据设计到同一 Shard）
