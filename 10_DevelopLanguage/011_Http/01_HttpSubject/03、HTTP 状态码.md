# 三、HTTP 状态码

###### 1. 1xx 状态码有哪些？分别代表什么含义？

1xx 是**信息性状态码**，表示请求已被接收，服务器正在继续处理。这类状态码在实际业务中很少直接接触到。

**100 Continue**：服务器告诉客户端"可以继续发请求体"。常见于上传大文件时，客户端先发一个包含 `Expect: 100-continue` 的请求头，服务器确认没问题再让客户端发 Body，避免白白传了大量数据才发现被拒绝。

**101 Switching Protocols**：服务器同意客户端的协议切换请求。最典型的就是 WebSocket 握手——客户端发 HTTP Upgrade 请求，服务器返回 101，之后连接就升级为 WebSocket 协议了。

**102 Processing**（WebDAV）：服务器已接受请求但还在处理中，防止客户端超时，很少见。

**103 Early Hints**：在最终响应发送前，先发一些 Link 头，让浏览器提前预加载资源（HTTP/2+ 的新特性）。

---

###### 2. 2xx 状态码有哪些？分别代表什么含义？

2xx 是**成功状态码**，表示请求被成功处理了。

**200 OK**：最常见的成功状态码，请求成功，响应体里有数据。

**201 Created**：资源创建成功。POST 创建新资源后通常返回 201，响应头 `Location` 会指向新创建的资源 URL。

**202 Accepted**：请求已接受，但还没处理完。适合异步操作——比如提交了一个耗时任务，服务器先回 202，任务在后台跑。

**204 No Content**：请求成功，但没有响应体。常见于 DELETE、PUT 操作，或 CORS 预检请求的响应。

**206 Partial Content**：分块传输成功，配合 `Range` 请求头使用。断点续传、视频分段加载就是靠这个。

记忆口诀：**200成功 / 201创建 / 202接受待处理 / 204成功无内容 / 206分块成功**。

---

###### 3. 3xx 状态码有哪些？分别代表什么含义？

3xx 是**重定向状态码**，表示客户端需要去其他 URL 获取资源。

**301 Moved Permanently**：永久重定向。浏览器会缓存新 URL，下次直接去新地址，不再请求旧地址。SEO 影响：权重转移到新 URL。

**302 Found**：临时重定向。每次都会先请求旧地址，然后跳转。SEO 权重不转移。

**303 See Other**：通常用于 POST 请求之后的重定向——告诉客户端"去 GET 这个 URL"。解决表单提交后刷新页面重复提交的问题（Post/Redirect/Get 模式）。

**304 Not Modified**：协商缓存命中，资源没有修改，客户端用本地缓存就行，服务器不返回响应体。

**307 Temporary Redirect**：临时重定向，严格保持原请求方法（POST 重定向后还是 POST，不能变 GET）。

**308 Permanent Redirect**：永久重定向，同样严格保持原请求方法。

---

###### 4. 4xx 状态码有哪些？分别代表什么含义？

4xx 是**客户端错误状态码**，表示请求有问题，是客户端的锅。

**400 Bad Request**：请求格式有问题，比如参数缺失、JSON 格式错误、参数校验失败。

**401 Unauthorized**：未认证——没有提供有效的身份凭证（没登录，或 Token 过期）。注意名字叫 Unauthorized，实际语义是"Unauthenticated"（未认证）。

**403 Forbidden**：已认证但无权限——知道你是谁，但就是不让你访问这个资源。

**404 Not Found**：资源不存在。也经常被用来隐藏"未授权"的资源（不想暴露资源存在与否）。

**405 Method Not Allowed**：请求方法不被允许，比如对只支持 GET 的接口发了 POST。响应头里 `Allow` 字段会列出支持的方法。

**408 Request Timeout**：客户端请求超时。

**409 Conflict**：请求冲突，比如注册时用户名已存在、版本并发冲突（乐观锁失败）。

**413 Content Too Large**：请求体太大，超过服务器限制（上传文件过大时常见）。

**415 Unsupported Media Type**：Content-Type 不支持，比如服务器只接受 JSON，你发了 XML。

**422 Unprocessable Entity**：格式正确但语义有误，常用于参数校验失败的细化表达。

**429 Too Many Requests**：请求频率超限，触发了限流（Rate Limiting）。

---

###### 5. 5xx 状态码有哪些？分别代表什么含义？

5xx 是**服务器错误状态码**，表示服务器处理出了问题，是服务端的锅。

**500 Internal Server Error**：通用服务器错误，程序抛异常了但没有具体指明原因。

**501 Not Implemented**：服务器不支持请求中使用的方法（和 405 的区别：405 是该路由不支持，501 是服务器压根没实现这个方法）。

**502 Bad Gateway**：网关或代理收到了上游服务器的无效响应。常见于 Nginx 反向代理时，后端服务挂掉或返回非法响应。

**503 Service Unavailable**：服务暂时不可用，可能是过载或正在维护。通常是暂时性的，响应头 `Retry-After` 会告诉客户端何时重试。

**504 Gateway Timeout**：网关或代理等待上游服务器响应超时。比如 Nginx 转发请求给后端，后端处理太慢，超过了 Nginx 配置的超时时间。

**508 Loop Detected**：检测到无限重定向循环。

记忆分层：**5xx = 服务端的锅；502/503/504 是代理/网关相关的**。

---

###### 6. 301 和 302 状态码有什么区别？

**301 永久重定向**：
- 表示资源已经**永久**搬到新地址了
- 浏览器会**缓存**这个重定向，下次直接去新地址，不再请求旧地址
- 搜索引擎会将旧 URL 的权重转移到新 URL
- 适用场景：域名迁移、HTTP 强制跳转 HTTPS、URL 规范化（`example.com` → `www.example.com`）

**302 临时重定向**：
- 表示资源**暂时**在新地址，旧地址还有效
- 浏览器**不缓存**这个重定向，每次都要先请求旧地址
- 搜索引擎不转移权重，旧 URL 继续保留价值
- 适用场景：A/B 测试、活动页面临时跳转、未登录跳到登录页

**方法保持问题**（经典陷阱）：
- 规范上，301/302 可以把 POST 请求变成 GET 重定向（浏览器的历史实现）
- 如果要严格保持 POST → POST，应该用 **307/308**
- 307 = 临时重定向 + 保持方法；308 = 永久重定向 + 保持方法

---

###### 7. 303、307 和 308 状态码分别是什么？

这三个状态码都是为了弥补 301/302 在方法保持上的历史遗留问题。

**303 See Other**：
- 告诉客户端：去 GET 这个新 URL 来获取响应
- **主动要求**把方法变成 GET，不管之前是什么方法
- 典型场景：Post/Redirect/Get 模式，防止表单重复提交

**307 Temporary Redirect**：
- 临时重定向，但**严格要求保持原始请求方法**
- POST 就重定向为 POST，GET 就重定向为 GET
- 适合 API 重定向时不想改变语义

**308 Permanent Redirect**：
- 永久重定向，同样**严格保持原始请求方法**
- 是 307 的永久版，也是 301 的"保持方法"升级版

简单对比：

| 状态码 | 永久/临时 | 方法是否改变 |
|--------|-----------|-------------|
| 301 | 永久 | 可能变为 GET（历史行为） |
| 302 | 临时 | 可能变为 GET（历史行为） |
| 303 | 临时 | 强制变为 GET |
| 307 | 临时 | 保持原方法 |
| 308 | 永久 | 保持原方法 |

---

###### 8. 401 和 403 状态码有什么区别？

这两个经常被混淆，区别在于：

**401 Unauthorized（未认证）**：
- 本质是"我不知道你是谁"——没有提供有效的认证凭证
- 场景：没有携带 Token、Token 过期、Token 无效
- 响应头会带 `WWW-Authenticate` 字段，告诉客户端应该用什么认证方式
- 客户端应该去登录，拿到凭证后重试

**403 Forbidden（无权限）**：
- 本质是"我知道你是谁，但你没有权限"——认证通过了，但权限不足
- 场景：普通用户访问管理员接口、跨租户访问他人数据
- 重新登录也没用，需要管理员给你授权

记忆口诀：**401 = 没带门卡，403 = 门卡没开权限**。

注意：有些接口会故意返回 404 而不是 403，来隐藏资源是否存在（安全策略）。

---

###### 9. 502、503 和 504 状态码有什么区别？

这三个状态码都跟代理/网关有关，是后端服务故障时最常见的错误码。

**502 Bad Gateway**：
- 网关/代理收到了上游服务器的**无效响应**（格式错误或无法解析的响应）
- 常见原因：后端服务崩溃/重启中、返回了错误格式的响应
- 关注点：**上游服务有没有正常运行**

**503 Service Unavailable**：
- 服务器当前**不可用**，暂时无法处理请求
- 常见原因：服务过载、正在维护、限流策略触发
- 通常是临时的，响应头 `Retry-After` 告知重试时间

**504 Gateway Timeout**：
- 网关/代理等待上游服务器响应**超时**
- 常见原因：后端接口处理时间过长、慢 SQL、下游服务卡住
- 关注点：**上游响应时间是否超限**

运维排查思路：
- 502 → 检查后端进程是否正常，查后端日志
- 503 → 检查服务是否过载，查资源使用率
- 504 → 检查后端处理耗时，定位慢接口/慢查询

---

###### 10. 【追问】接口设计中如何正确选择 HTTP 状态码？

很多项目存在"万能 200"的坏习惯——不管成功失败都返回 200，用业务码区分。这其实丢失了 HTTP 语义。

**规范化设计原则**：

| 场景 | 推荐状态码 |
|------|-----------|
| 查询成功，有数据 | 200 |
| 创建资源成功 | 201（响应头带 Location） |
| 操作成功，无响应体 | 204 |
| 参数校验失败 | 400 |
| 未登录/Token 过期 | 401 |
| 已登录但无权限 | 403 |
| 资源不存在 | 404 |
| 方法不支持 | 405 |
| 并发冲突（乐观锁失败） | 409 |
| 限流 | 429 |
| 服务器内部错误 | 500 |
| 服务不可用 | 503 |

**业务码 + HTTP 状态码双层设计**（推荐）：

```json
// HTTP Status: 400
{
  "code": "USER_NOT_FOUND",
  "message": "用户不存在",
  "traceId": "abc-123"
}
```

HTTP 状态码给网关、监控、负载均衡用（它们能理解 HTTP 语义），业务码给前端业务逻辑用。
