# 一、HTTP 基础概念

###### 1. HTTP 协议的特点有哪些？

HTTP（HyperText Transfer Protocol，超文本传输协议）是应用层协议，主要有以下几个核心特点：

**无状态**：服务器不保存客户端的任何状态信息，每次请求都是独立的。好处是服务器设计简单、易于扩展；坏处是需要 Cookie/Session/Token 来维持会话。

**无连接**（早期）：HTTP/1.0 每次请求都要新建 TCP 连接，响应完就断掉。HTTP/1.1 引入了持久连接（Keep-Alive），一个 TCP 连接可以复用。

**基于请求-响应模型**：客户端主动发请求，服务器被动响应，通信是单向触发的。

**简单、灵活**：报文格式文本化，可读性强；支持传输任意类型的数据（MIME 类型），不限于 HTML。

**明文传输（HTTP）**：数据不加密，容易被窃听。HTTPS 在此基础上加了 TLS 层来解决安全问题。

记忆口诀：**无状态、无连接（早期）、请求响应、明文传输**。

---

###### 2. 什么是无状态协议？HTTP 为什么是无状态的？

**无状态**是指：服务器处理完一个请求后，不会记住这次请求的任何信息。下次来一个请求，服务器完全不知道这个客户端之前做过什么。

HTTP 被设计成无状态的原因主要有两个：
- **简化服务器设计**：不需要维护大量客户端状态，降低服务器内存开销
- **天然支持水平扩展**：任意一台服务器都可以处理任意请求，负载均衡不需要"会话粘滞"

但无状态带来的问题是无法追踪用户行为（比如登录状态）。解决方案就是在应用层引入状态管理机制：
- **Cookie + Session**：服务端保存会话，客户端保存 SessionID
- **JWT Token**：无服务端存储，把状态放进 Token 里
- **URL 重写**：把 SessionID 附加到 URL 参数里（很少用了）

###### 3. HTTP 协议位于 OSI 模型的哪一层？

HTTP 位于 **OSI 七层模型的第七层——应用层**。

完整的层次关系：

```
应用层（HTTP/FTP/DNS）       ← HTTP 在这里
表示层（数据格式、加密解密）
会话层（建立/维护会话）
传输层（TCP/UDP）            ← HTTP 依赖 TCP
网络层（IP）
数据链路层（以太网）
物理层（物理信号）
```

实际开发中常用的简化版是 **TCP/IP 四层模型**：
- 应用层（HTTP/HTTPS/DNS）
- 传输层（TCP/UDP）
- 网络层（IP）
- 网络接口层

HTTP 底层依赖 TCP 提供可靠传输，而 HTTP/3 改用了 UDP（通过 QUIC 协议实现可靠性）。

###### 4. HTTP 默认使用哪个端口？HTTPS 默认使用哪个端口？

- **HTTP 默认端口：80**
- **HTTPS 默认端口：443**

这是约定俗成的"知名端口"（Well-Known Port），端口范围是 0~1023，由 IANA 统一分配。

实际上端口只是一个标识，你完全可以把 HTTP 服务跑在 8080 或其他端口——很多开发环境就是这么做的（比如 Spring Boot 默认 8080）。但如果用了默认端口，URL 里可以省略端口号，浏览器会自动补全。

其他常见知名端口：FTP(21)、SSH(22)、Telnet(23)、SMTP(25)、DNS(53)、MySQL(3306)、Redis(6379)。

###### 5. 什么是 HTTP 报文？HTTP 报文的组成部分有哪些？

HTTP 报文就是 HTTP 通信中传递的数据块，分两种：**请求报文**和**响应报文**。

**请求报文结构**：

```
POST /api/login HTTP/1.1           ← 请求行（方法 + 路径 + 协议版本）
Host: example.com                  ← 请求头（Headers）
Content-Type: application/json
Content-Length: 38
                                   ← 空行（分隔头部和正文）
{"username":"admin","password":"123"}  ← 请求体（Body）
```

**响应报文结构**：

```
HTTP/1.1 200 OK                    ← 状态行（协议版本 + 状态码 + 状态描述）
Content-Type: application/json     ← 响应头（Headers）
Content-Length: 26
                                   ← 空行
{"code":0,"msg":"success"}         ← 响应体（Body）
```

三部分核心记忆：**起始行（请求行/状态行）+ 头部（Headers）+ 正文（Body）**，三者之间用空行分隔。

###### 6. 【追问】HTTP 的无状态性在微服务架构中带来了哪些挑战，如何解决？

在微服务架构下，无状态性既是优势也带来挑战：

**挑战**：
- **认证信息传递**：每个微服务都需要验证请求合法性，不能依赖单机 Session
- **分布式 Session**：用户登录后的会话数据需要跨服务共享
- **请求追踪**：无状态导致链路追踪困难，无法天然关联同一用户的多次请求

**解决方案**：
- **JWT Token**：把用户信息、权限直接编码进 Token，每个微服务都能独立验证，无需中心化 Session 存储
- **网关统一认证**：在 API Gateway（Spring Cloud Gateway/Kong）层做 Token 验证，验证通过后把用户信息透传给下游（通过 Header 或 ThreadLocal）
- **分布式 Session**：用 Redis 存 Session，所有服务共享（Spring Session + Redis）
- **链路追踪**：引入 TraceId，在 Header 中传递，配合 SkyWalking/Zipkin 做全链路追踪
