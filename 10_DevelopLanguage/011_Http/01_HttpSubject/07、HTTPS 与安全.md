# 七、HTTPS 与安全

> 关联知识：[[35_NettyKnowledge/05_协议与实战应用]]（TLS/SslHandler 实现）

###### 1. HTTPS 使用的是哪种加密方式？

HTTPS 不是单一加密方式，而是**混合加密**——把非对称加密和对称加密结合起来用。

**为什么混合**：
- 非对称加密（RSA/ECDSA）：安全，但慢（计算量大）
- 对称加密（AES/ChaCha20）：快，但密钥分发是难题（如何安全地把密钥给对方？）

**混合方案**：
1. 用**非对称加密**安全地交换密钥（TLS 握手阶段）
2. 用协商好的**对称密钥**加密实际传输数据（数据传输阶段）

既解决了密钥安全传输问题，又保证了数据传输的性能。

**TLS 1.3 的改进**：使用 ECDHE（椭圆曲线 Diffie-Hellman 临时密钥交换），即使服务器私钥泄露，历史通信内容也是安全的（前向保密）。

---

###### 2. 什么是 SSL/TLS 协议？

**SSL（Secure Sockets Layer）** 是早期的安全层协议，SSL 3.0 之后演进为 **TLS（Transport Layer Security）**，两个名字经常混用，但现在说的"SSL"通常就是指 TLS。

TLS 工作在**传输层和应用层之间**，对 HTTP 数据进行加密后传输，就成了 HTTPS。

**版本演进**：
- SSL 2.0 / 3.0：已废弃，存在严重漏洞（POODLE 等）
- **TLS 1.0 / 1.1**：已废弃（2020 年 RFC 8996 标准废弃），不再推荐
- **TLS 1.2**：目前最广泛部署的版本，支持 AEAD 加密（如 AES-GCM）
- **TLS 1.3**：2018 年发布，握手速度更快（1-RTT，支持 0-RTT 恢复），去掉了大量不安全算法，是未来主流

**TLS 提供的三大保障**：
- **机密性**：数据加密传输，第三方无法解读
- **完整性**：MAC 消息认证码，防止数据被篡改
- **认证性**：数字证书验证服务器身份，防止连接到假服务器

---

###### 3. HTTPS 的握手过程是怎样的？

以 TLS 1.2 为例（4次握手，2个 RTT）：

**第1步：Client Hello**（客户端发起）
- 支持的 TLS 版本列表
- 支持的加密套件列表（如 TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256）
- 随机数 ClientRandom

**第2步：Server Hello + Certificate + ServerHelloDone**（服务端响应）
- 选定的 TLS 版本和加密套件
- 随机数 ServerRandom
- 服务器的数字证书（含公钥）

**第3步：客户端验证证书，发送 Pre-Master Secret**
- 验证证书：证书链 → CA 签名 → 域名是否匹配 → 是否过期
- 生成 Pre-Master Secret（随机数），用服务器公钥加密后发送
- 双方根据 ClientRandom + ServerRandom + Pre-Master Secret 推导出对称加密密钥
- 发送 ChangeCipherSpec（告知接下来用加密通信）
- 发送 Finished（加密的握手验证）

**第4步：服务端确认**
- 同样发送 ChangeCipherSpec 和 Finished
- 握手完成，开始加密通信

**TLS 1.3 优化**：把握手压缩到 1-RTT（把 ServerHello 和证书等合并），支持 0-RTT 会话恢复（第二次连接可以直接发数据），速度更快。

###### 4. 什么是数字证书？数字证书的作用是什么？

数字证书是由 **CA（证书颁发机构）** 签发的"身份证明文件"，证明"这个公钥确实是这个网站的"。

**证书包含的核心信息**：
- 持有者信息（域名、组织名）
- 持有者的公钥
- 有效期
- 签发 CA 的名称
- CA 的数字签名（用 CA 的私钥对上述信息做签名）

**作用**：解决中间人攻击问题——防止攻击者伪装成合法服务器：

没有证书的情况：
```
客户端 → 我要和 bank.com 通信
攻击者 → （拦截）我是 bank.com，这是我的公钥（实际上是攻击者的公钥）
客户端 → 好的，用这个"假公钥"加密数据...（已被劫持）
```

有证书的情况：
```
客户端 → 我要和 bank.com 通信
攻击者 → （拦截）我是 bank.com，这是我的公钥
客户端 → 这个公钥没有可信 CA 签名，拒绝！
```

**证书链**：证书不是孤立的，形成信任链：
```
根 CA 证书（预装在操作系统/浏览器中）
  └── 中间 CA 证书（根 CA 签名）
        └── 服务器证书（中间 CA 签名）
```

---

###### 5. 什么是 CA（证书颁发机构）？

CA（Certificate Authority，证书颁发机构）是被广泛信任的第三方机构，负责审核身份并签发数字证书。

**CA 的工作流程**：
1. 申请者（网站运营者）生成密钥对，把公钥和身份信息提交给 CA
2. CA 审核申请者的身份（域名所有权验证、企业资质审核等）
3. CA 用自己的私钥对申请信息签名，生成证书颁发给申请者
4. 申请者把证书配置到服务器上

**CA 分级**：
- **根 CA**：最顶层，证书预装在 OS/浏览器，如 DigiCert、Let's Encrypt（免费）
- **中间 CA**：根 CA 委托，实际负责签发
- **叶子证书**：最终颁发给网站的证书

**证书类型**：
- **DV（域名验证）**：只验证域名所有权，几分钟搞定，Let's Encrypt 免费颁发
- **OV（组织验证）**：验证域名 + 企业信息，需要几天
- **EV（扩展验证）**：最严格，浏览器地址栏会显示绿色组织名，银行常用

**证书吊销**：
- CRL（证书吊销列表）：定期发布的被吊销证书列表
- OCSP（在线证书状态协议）：实时查询证书是否有效

###### 6. HTTPS 如何防止中间人攻击？

中间人攻击（MITM）是攻击者插入客户端和服务器之间，伪装成双方的攻击方式。

HTTPS 通过**证书验证机制**防御：

**防御流程**：
1. 浏览器连接服务器时，服务器返回证书
2. 浏览器验证证书的合法性：
   - 证书是否由可信 CA 签发（证书链验证）
   - 证书是否在有效期内
   - 证书上的域名是否和当前访问的域名匹配
   - 证书是否已被吊销（OCSP/CRL 查询）
3. 任何一项验证失败，浏览器弹出警告，拒绝连接

**攻击者为什么绕不过去**：
- 攻击者没有合法 CA 签名的证书（无法伪造合法证书）
- 即使攻击者申请了同名证书，CT（证书透明度）日志会记录所有签发的证书，异常证书会被发现
- 证书固定（Certificate Pinning）：App 可以把证书 Hash 写死，只信任特定证书（防止 CA 被攻破）

**注意**：如果用户在企业内网、手机抓包工具，主动安装了"根证书"，那相当于主动信任了中间人，HTTPS 就没有防护意义了——这正是 Charles、Fiddler 的工作原理。

###### 7. 什么是 HSTS（HTTP 严格传输安全）？

HSTS（HTTP Strict Transport Security）是服务器告诉浏览器："我们只说 HTTPS，以后别用 HTTP 访问我了"。

**响应头**：
```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

- `max-age=31536000`：在接下来的 1 年内，浏览器强制使用 HTTPS 访问该域名
- `includeSubDomains`：子域名也强制 HTTPS
- `preload`：申请加入浏览器内置的 HSTS 预加载列表（浏览器安装时就内置，首次访问也走 HTTPS）

**解决的问题**：

没有 HSTS 时，即使网站支持 HTTPS，用户第一次输入 `http://bank.com` 时，还是会先走 HTTP，中间人可以在这一步插入攻击（SSL Stripping）。

有了 HSTS，浏览器看到 http 会**自动内部重定向为 https**（不发 HTTP 请求），直接发 HTTPS，SSL Stripping 无法生效。

**注意**：HSTS 一旦设置，`max-age` 内无法访问 HTTP，如果证书有问题也进不来。所以设置前要确认证书配置正确，建议先用短 max-age 测试。

###### 8. 为什么 HTTPS 比 HTTP 慢？如何优化 HTTPS 性能？

**慢在哪里**：

1. **TLS 握手开销**：TLS 1.2 需要 2-RTT（多两次网络往返），TLS 1.3 降到了 1-RTT
2. **加解密计算**：每个数据包都要加密/解密，虽然现代硬件 AES-NI 指令集加速后开销很小（<1%），但还是有
3. **证书验证**：浏览器需要验证证书链，可能需要 OCSP 请求（额外 DNS + TCP + HTTP 往返）

**实际影响**：在现代网络和硬件上，HTTPS 的开销相比 HTTP 大约 1-5%，远没有早期那么显著。主要影响还是首次握手的延迟。

**优化手段**：

- **升级 TLS 1.3**：1-RTT 握手，支持 0-RTT 会话恢复（速度进一步提升）
- **会话恢复（Session Resumption）**：TLS Session ID 或 Session Ticket，复用之前的握手结果，跳过完整握手
- **OCSP Stapling**：服务器预先获取 OCSP 响应并在握手中提供，省去浏览器单独 OCSP 查询
- **HSTS + Preload**：消除 HTTP→HTTPS 重定向的一次往返
- **HTTP/2 或 HTTP/3**：在 TLS 之上用多路复用，减少连接数，摊薄 TLS 握手成本
- **硬件加速**：启用 CPU 的 AES-NI 指令集，OpenSSL 默认支持

###### 9. 什么是双向认证？

普通 HTTPS 是**单向认证**：只有客户端验证服务器身份（服务器出示证书），服务器不验证客户端。

**双向认证（mTLS，Mutual TLS）**：客户端和服务器**互相验证**对方的证书。

**流程**：
1. 服务器出示证书，客户端验证（同单向）
2. 服务器要求客户端也出示证书（`CertificateRequest`）
3. 客户端出示自己的证书
4. 服务器验证客户端证书（是否由受信任 CA 签发、是否有效）
5. 双方握手成功，建立加密连接

**应用场景**：
- **微服务间通信安全**（Service Mesh，如 Istio 的 mTLS）：每个服务都有证书，相互认证
- **API 接口授权**：只有持有客户端证书的调用方才能访问接口
- **企业内网安全**：VPN 连接、员工身份认证
- **银行/金融系统**：U 盾里存的就是客户端证书

**优缺点**：
- 优点：无法伪造身份，安全性极高
- 缺点：客户端证书管理复杂（分发、更新、吊销），实现和运维成本高

###### 10. 【追问】TLS 握手失败的常见原因有哪些，如何排查？

**常见原因**：

1. **证书过期**：最常见，`openssl s_client -connect domain:443` 可以查看证书有效期
2. **域名不匹配**：证书里的 CN/SAN 和实际访问域名不一致（如证书是 `example.com` 但访问 `www.example.com`）
3. **证书链不完整**：中间证书没有配置，浏览器无法验证到根 CA
4. **不支持的 TLS 版本或加密套件**：客户端和服务端协商失败（如老客户端只支持 TLS 1.0，服务端要求 TLS 1.2+）
5. **时间不同步**：服务器时间偏差过大，证书时间范围校验失败
6. **私钥不匹配**：证书和私钥不对应，握手中报 `bad certificate` 错误

**排查工具**：

```bash
# 查看证书信息和握手过程
openssl s_client -connect example.com:443 -tls1_2

# 查看证书链
openssl s_client -connect example.com:443 -showcerts

# 验证证书是否匹配私钥
openssl x509 -noout -modulus -in cert.pem | md5sum
openssl rsa -noout -modulus -in key.pem | md5sum
# 两个 MD5 要相同

# 在线工具
# SSL Labs: https://www.ssllabs.com/ssltest/
```
