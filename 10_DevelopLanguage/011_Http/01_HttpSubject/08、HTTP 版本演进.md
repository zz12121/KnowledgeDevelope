# 八、HTTP 版本演进

###### 1. HTTP/0.9 有哪些特点？

HTTP/0.9 是 1991 年提出的最原始版本，极度简单：

- **只支持 GET 方法**，没有 POST/PUT/DELETE
- **只能传输 HTML 文本**，没有 MIME 类型，也没有响应头
- **没有状态码**：请求成功就返回 HTML，请求失败就关闭连接
- **每次请求完立即关闭连接**（短连接）
- **请求极其简单**：一行搞定 `GET /index.html`

这个版本现在只有历史意义，没有任何实际应用场景。

---

###### 2. HTTP/1.0 相比 HTTP/0.9 有哪些改进？

HTTP/1.0（1996 年，RFC 1945）是一个里程碑版本，把 HTTP 从"只能传 HTML"升级为通用传输协议：

- **引入请求头和响应头**：`Content-Type`、`Content-Length`、`Authorization` 等
- **引入状态码**：有了 200/301/404/500 等完整的状态码体系
- **支持多种 HTTP 方法**：新增 POST、HEAD
- **支持 MIME 类型**：不再局限于 HTML，可以传图片、文件、JSON 等任意格式
- **支持非 ASCII 字符集**：通过 `Content-Encoding` 和 `charset`

**还没解决的问题**：
- 默认短连接（每次请求建立、断开 TCP）
- 不支持虚拟主机（没有 `Host` 头，一台服务器只能跑一个网站）
- 不支持持久连接（`Connection: keep-alive` 是非标准扩展）

---

###### 3. HTTP/1.1 相比 HTTP/1.0 有哪些改进？

HTTP/1.1（1997 年，RFC 2616）是统治互联网 20 多年的版本，核心改进：

**连接管理**：
- 默认持久连接（Keep-Alive），连接可以复用，大幅减少握手开销
- 支持管道化（Pipelining）——连续发多个请求（但响应必须顺序返回）

**虚拟主机支持**：
- 新增必须的 `Host` 头，一台服务器可以托管多个域名的网站

**更多方法**：新增 PUT、DELETE、OPTIONS、TRACE、CONNECT

**分块传输**：
- `Transfer-Encoding: chunked`，边生成边传输，不需要提前知道 Content-Length

**缓存控制增强**：
- 新增 `Cache-Control` 头（比 HTTP/1.0 的 `Expires` 更强大）
- 引入协商缓存：`ETag` / `If-None-Match`

**内容协商**：Accept 系列头（Accept/Accept-Encoding/Accept-Language）

**条件请求**：`If-Modified-Since`、`If-None-Match`、`Range`（断点续传）

---

###### 4. HTTP/1.1 存在哪些性能问题？

尽管 HTTP/1.1 带来了持久连接，但仍有几个根本性的性能瓶颈：

**队头阻塞（Head-of-Line Blocking）**：

持久连接里的请求是串行的——第一个响应没回来，后面的请求只能等。一个慢请求会阻塞整条连接上的所有后续请求。

**连接数限制**：

为了绕过队头阻塞，浏览器对同一域名开多个 TCP 连接（通常 6 个）。但连接数受限，且每个连接都要 TCP 握手、TLS 握手，有开销。

**Header 冗余**：

HTTP/1.1 是文本协议，每次请求都要带大量 Headers（User-Agent、Cookie、Host 等），而且这些 Headers 往往大量重复，浪费带宽。

**服务器推送缺失**：

只能客户端主动请求，服务器无法主动推送资源（即使服务器知道客户端需要）。

**明文传输**：

HTTP/1.1 本身不加密（需要配合 TLS 变成 HTTPS）。

这些问题直接推动了 HTTP/2 的诞生。

---

###### 5. 什么是队头阻塞（Head-of-Line Blocking）？

队头阻塞是指：**队列头部的请求没有处理完，后面的请求就只能干等**。

**HTTP/1.1 的应用层队头阻塞**：

持久连接里请求是串行的，请求 A 响应慢，请求 B 和 C 排队等待，即使它们本来可以并发处理。

```
──── 连接 1 ────
req1(慢) → req1 resp → req2 → req2 resp → req3 ...
```

**HTTP/2 的 TCP 层队头阻塞**：

HTTP/2 解决了应用层队头阻塞（多路复用，不同 Stream 独立），但 HTTP/2 跑在 TCP 上，TCP 是有序的字节流——如果一个 TCP 包丢失，后面所有的包都要等重传，所有 Stream 都受影响。

**HTTP/3 的彻底解决**：

HTTP/3 改用 UDP + QUIC，QUIC 在 UDP 上实现了可靠传输，但以 Stream 为单位，一个 Stream 的丢包不影响其他 Stream，彻底消除了队头阻塞。

---

###### 6. HTTP/2 有哪些新特性？

HTTP/2（2015 年，RFC 7540）基于 Google 的 SPDY 协议演化而来，核心改进：

1. **二进制分帧**：从文本协议变为二进制协议，更高效、更紧凑（详见第7题）
2. **多路复用**：一条 TCP 连接上并发处理多个请求/响应，彻底解决 HTTP/1.1 应用层队头阻塞（详见第8题）
3. **头部压缩（HPACK）**：压缩 Header，减少冗余数据传输（详见第10题）
4. **服务器推送（Server Push）**：服务器主动向客户端推送资源（详见第9题）
5. **优先级和依赖**：客户端可以设置请求的优先级，服务器按优先级处理
6. **流量控制**：基于 Stream 和 Connection 两个级别的流量控制
7. **默认加密**：主流浏览器要求 HTTP/2 必须配合 TLS（虽然规范本身没强制）

---

###### 7. 什么是 HTTP/2 的二进制分帧？

HTTP/1.x 是文本协议，Headers 和 Body 是字符串格式，解析时需要大量字符串处理。

HTTP/2 把所有通信分割成**二进制帧（Frame）**来传输：

```
+-------+------+--------+
|Length |Type  |Flags   |
+-------+------+--------+
|Stream Identifier      |
+-----------------------+
|Frame Payload          |
+-----------------------+
```

**帧类型**：
- `HEADERS` 帧：传输 HTTP 头部（HPACK 压缩后）
- `DATA` 帧：传输 HTTP 正文
- `SETTINGS` 帧：传输连接配置
- `PUSH_PROMISE` 帧：服务器推送声明
- `PING` 帧：心跳检测
- `WINDOW_UPDATE` 帧：流量控制

**核心概念**：
- **Stream（流）**：虚拟的双向通信通道，每个请求/响应对占用一个 Stream，用 Stream ID 标识
- **Message（消息）**：一个完整的请求或响应，由多个帧组成
- **Frame（帧）**：通信的最小单位

每个 HTTP 请求/响应都在独立的 Stream 中传输，不同 Stream 的帧可以**交错发送**，接收端根据 Stream ID 重新组装，这就是多路复用的基础。

---

###### 8. 什么是 HTTP/2 的多路复用？

多路复用是 HTTP/2 最核心的特性：**在同一条 TCP 连接上，并发发送多个请求，并发接收多个响应，互不干扰**。

**HTTP/1.1 的串行问题**：
```
[TCP 连接]
→ req1 → resp1 → req2 → resp2 → req3 → resp3
```
（慢的请求会阻塞后面所有请求）

**HTTP/2 的多路复用**：
```
[TCP 连接 - 一条就够了]
→ req1(Stream1) → req2(Stream2) → req3(Stream3)  // 并发发出
← resp2(Stream2) ← resp1(Stream1) ← resp3(Stream3)  // 乱序返回，按 Stream ID 组装
```

**优势**：
- 一条 TCP 连接搞定所有请求，不需要建多个连接
- 响应可以乱序返回，慢请求不阻塞快请求
- 节省 TCP 握手和 TLS 握手的开销
- 更好地利用 TCP 拥塞控制（单条连接比多条连接更公平）

**一个常被问到的细节**：HTTP/2 的多路复用解决了应用层队头阻塞，但 TCP 层的队头阻塞（丢包重传）还在，要彻底解决需要 HTTP/3 + QUIC。

---

###### 9. 什么是 HTTP/2 的服务器推送？

服务器推送（Server Push）让服务器能够**主动向客户端发送资源**，而不必等客户端请求。

**场景**：客户端请求 `index.html`，服务器知道客户端接下来一定还会请求 `style.css` 和 `app.js`，于是主动推送这两个文件，客户端不需要单独请求。

```
客户端: GET /index.html
服务器: 
  ← PUSH_PROMISE: /style.css   // 提前声明要推送
  ← PUSH_PROMISE: /app.js
  ← HEADERS + DATA: index.html // 返回请求的资源
  ← HEADERS + DATA: style.css  // 主动推送，无需客户端请求
  ← HEADERS + DATA: app.js
```

**客户端控制**：客户端可以发 `RST_STREAM` 帧拒绝不需要的推送（比如本地已有缓存）。

**现实中的问题**：服务器推送实现复杂，难以判断"该推什么"，有时反而推了客户端不需要的资源，浪费带宽。HTTP/3 草案中已经去掉了 Server Push。

实践中，`<link rel="preload">` 加资源预加载提示已经基本取代了 Server Push 的场景。

---

###### 10. 什么是 HTTP/2 的头部压缩（HPACK）？

HTTP/1.x 里 Header 是明文文本，每次请求都要重复传 User-Agent、Cookie、Host 等几百字节的 Header，浪费严重。

HPACK 是 HTTP/2 专用的头部压缩算法：

**两大核心机制**：

**1. 静态表（Static Table）**：内置了 61 个常见的 Header 字段和值，用 1-2 字节的索引表示。比如索引 2 就代表 `method: GET`，索引 8 就代表 `status: 200`。

**2. 动态表（Dynamic Table）**：客户端和服务端各维护一份，存储本次会话中出现过的 Header 键值对。后续请求如果出现相同的 Header，只发索引号就够了，大大压缩传输量。

**Huffman 编码**：对 Header 值使用 Huffman 编码进一步压缩。

**效果**：在实际场景中，HPACK 可以将 Header 压缩 85-95%，对 Cookie 密集的请求效果尤其明显。

注意：HPACK 维护有状态的动态表，这是 HTTP/2 必须有序交付的原因之一，也带来了 TCP 丢包时的全连接阻塞问题。HTTP/3 的 QPACK 改进了这一点，允许乱序压缩。

---

###### 11. HTTP/2 相比 HTTP/1.1 有哪些优势？

核心优势汇总：

| 特性 | HTTP/1.1 | HTTP/2 |
|------|----------|--------|
| 协议格式 | 文本协议 | 二进制分帧，解析更高效 |
| 请求并发 | 多条 TCP 连接（每域名 6 条） | 单条 TCP 连接多路复用 |
| 队头阻塞 | 应用层存在 | 解决应用层队头阻塞（TCP 层仍存在） |
| Header 传输 | 每次全量重复发送 | HPACK 压缩，增量更新 |
| 服务器推送 | 不支持 | 支持（但实践中少用） |
| 优先级 | 不支持 | 支持 Stream 优先级 |
| 加密 | 可选 | 主流浏览器要求 TLS |

**实际性能提升**：页面加载速度提升 15%-50%（取决于资源数量和网络环境），资源越多、Header 越大，收益越明显。

---

###### 12. HTTP/3 有哪些新特性？

HTTP/3（2022 年，RFC 9114）最大的改变是底层协议：**从 TCP 换成了 QUIC（基于 UDP）**。

核心新特性：

**彻底消除队头阻塞**：QUIC 以 Stream 为单位做丢包重传，一个 Stream 的丢包不影响其他 Stream，HTTP/2 的 TCP 队头阻塞问题彻底解决。

**0-RTT 连接建立**：首次连接 1-RTT，重连可以 0-RTT（在握手包里直接带数据），比 TCP+TLS 的 2-RTT 快很多。

**连接迁移（Connection Migration）**：QUIC 用"连接 ID"而不是 IP:端口标识连接。切换网络（Wi-Fi → 4G）时，连接不会断，无缝迁移——手机场景体验极佳。

**内置加密**：QUIC 原生内置 TLS 1.3，加密不可关闭，比 TCP+TLS 集成更深、握手更快。

**改进的拥塞控制**：QUIC 在用户态实现拥塞控制，可以快速迭代更新，不受内核限制。

---

###### 13. HTTP/3 为什么使用 QUIC 协议？

根本原因：TCP 协议栈在操作系统内核中，要改动非常慢（需要更新操作系统），而 QUIC 跑在用户态，迭代灵活。

**TCP 的根本性缺陷**：
- TCP 的有序交付特性导致队头阻塞（不可绕过）
- TCP 握手至少 1-RTT，加上 TLS 握手总共 3-RTT 才能开始发数据（TLS 1.3 是 2-RTT）
- IP 地址 + 端口号组合标识连接，网络切换必然断连
- 头部信息可以被中间节点（路由器/防火墙）修改，难以加密

**QUIC 在 UDP 上重建了 TCP 的优点并解决了其缺陷**：
- 可靠传输（但以 Stream 为单位，互不阻塞）
- 拥塞控制和流量控制（在用户态实现，可快速迭代）
- 原生 TLS 1.3 加密（1-RTT 握手）
- 连接 ID 机制（支持连接迁移）

---

###### 14. QUIC 协议有哪些特点？

QUIC（Quick UDP Internet Connections）是 Google 在 2012 年开始研发、后被 IETF 标准化的传输层协议：

**核心特点**：

**基于 UDP**：UDP 没有建连、排序、重传等开销，QUIC 在其之上自行实现这些功能，同时保持灵活性。

**内置 TLS 1.3**：握手和加密深度集成，首次连接只需 1-RTT（TCP+TLS 需要 3-RTT），重连支持 0-RTT。

**多路复用无队头阻塞**：Stream 独立传输，一个 Stream 的丢包只影响自己，其他 Stream 不受影响。

**连接迁移**：使用 64 位连接 ID 标识连接，IP 地址或端口变化后连接不断。

**用户态实现**：不依赖操作系统内核，可以快速迭代更新（Google 每年都在更新 QUIC 的拥塞算法）。

**前向纠错（可选）**：通过冗余包检测，部分丢包可以不用重传就恢复（减少重传开销）。

**当前状况**：HTTP/3 已被主流浏览器（Chrome、Firefox）和大型网站（Google、Facebook/Meta、Cloudflare）广泛支持，占全球 HTTP 流量约 25%（2024 年数据）。

---

###### 15. 【追问】HTTP/1.1、HTTP/2、HTTP/3 如何选择？

**HTTP/1.1**：
- 不支持 HTTP/2+ 的老服务器/客户端的兜底
- 内网小流量服务，改造成本不值得

**HTTP/2**：
- 生产环境的主流选择，浏览器 + Nginx/CDN 都有成熟支持
- 资源多的页面（API、静态资源）收益显著
- 配置简单：Nginx 开启 `listen 443 ssl http2;` 即可

**HTTP/3**：
- 移动端、弱网场景（丢包率高、网络切换频繁）收益最大
- CDN 边缘节点（Cloudflare、阿里云）已广泛支持
- 自建服务接入 HTTP/3 还需要 QUIC 库支持（如 `nginx-quic`、`quiche`）
- 企业防火墙可能封锁 UDP，需要 fallback 到 TCP

**实践建议**：
- 面向用户的 Web 服务 → 用 CDN（Cloudflare/阿里云），CDN 自动支持 HTTP/2+HTTP/3
- 自建 Nginx → 最低开 HTTP/2，有条件编译 quiche 支持 HTTP/3
- 微服务间通信 → HTTP/2（gRPC 基于 HTTP/2），比 HTTP/1.1 节省连接数
