# 2、性能与并发类

> 这类题考的是你对系统性能的量化感知和优化实战经验。有具体数字比泛泛而谈强百倍。

###### 1. 你们这个 XXX 项目并发高吗？

**答法策略：客观描述规模，引出技术细节**

> "整体来说属于中等并发。平时日活大概 X 万用户，日订单量 X 万笔，核心接口（商品详情页）QPS 在 2000-3000 左右，大促期间能到 8000-10000。我们通过多级缓存（Caffeine + Redis）把大部分读请求拦截在数据库之前，DB 实际的 QPS 控制在 500 以内。"

**不同规模的描述参考：**

- 创业公司/小项目：日活万级，核心接口 QPS 百级 → 诚实说，描述做了哪些预防性设计
- 中型公司：日活十万级，QPS 千到万级 → 重点说缓存/限流方案
- 大厂：日活百万+，QPS 万到十万级 → 说分布式架构/分库分表方案

**面试官想听什么：** 不一定要高并发，关键是你对系统的指标有清晰认知，做了哪些与并发量匹配的技术决策。

###### 2. 你的接口 QPS 是多少？

**关键接口 QPS 参考值（面试用，根据项目实际说）：**

| 接口 | 典型QPS | 特点 |
|------|---------|------|
| 商品详情页 | 1000-5000 | 读多，适合缓存 |
| 下单接口 | 100-500 | 写操作，涉及分布式事务 |
| 支付接口 | 50-200 | 调外部，对超时敏感 |
| 搜索接口 | 500-2000 | ES 支撑 |
| 用户登录 | 100-500 | 有鉴权开销 |

**如果被追问"具体怎么支撑这个 QPS"：**

以商品详情页为例（QPS 3000）：

1. 本地缓存（Caffeine）：拦截 70% 的请求（热门商品命中本地缓存，纳秒级）
2. Redis 缓存：拦截 25% 的请求（未命中本地缓存的，Redis 毫秒级响应）
3. 穿透到 DB：只有 5% 的请求到数据库（实际 DB QPS 约 150）

###### 3. 2W 的 TPS，线上参数如何设置？

**首先确认是哪类服务（I/O密集 vs CPU密集）**

2万 TPS 的 HTTP 接口服务（典型 I/O 密集，大部分时间在等 DB/Redis 响应）：

**线程池（Tomcat）：**

```yaml
server:
  tomcat:
    threads:
      max: 800          # 最大线程数：线程在等 I/O，需要更多线程
      min-spare: 100    # 最小空闲线程
    max-connections: 8192  # 最大连接数（NIO，连接数 >> 线程数）
    accept-count: 200   # 排队等待的连接数
```

估算：2万 TPS，假设平均响应时间 50ms，则同时需要处理的请求数 = 20000 × 0.05 = 1000。设置 threads.max=800（留 200 的余量应对尖峰）。

**数据库连接池（HikariCP）：**

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 100     # 2万 TPS，DB 操作占比假设10%，DB QPS=2000
      minimum-idle: 20
      connection-timeout: 3000
      idle-timeout: 600000
      max-lifetime: 1800000
```

**JVM 参数（4核8G 服务器）：**

```bash
-Xms4g -Xmx4g                    # 堆内存固定，避免动态扩容 STW
-XX:+UseG1GC                      # G1 垃圾回收
-XX:MaxGCPauseMillis=100          # 目标最大停顿100ms
-XX:G1HeapRegionSize=8m           # 堆8G时推荐8MB Region
-XX:InitiatingHeapOccupancyPercent=35  # 堆占35%触发并发标记
-XX:+HeapDumpOnOutOfMemoryError
```

**Redis 连接池：**

```yaml
spring:
  data:
    redis:
      lettuce:
        pool:
          max-active: 200  # 最大连接数
          max-idle: 50
          min-idle: 10
```

###### 4. 4 核 8G 的服务器，线上 JVM 怎么设置合理？

**基本原则：堆内存不超过物理内存的 70%（留给 OS 和 Native 内存）**

```bash
# 4核8G，推荐 JVM 参数
JAVA_OPTS="
  -server
  -Xms4g -Xmx4g                      # 堆内存：4G（8G × 50%，留出余量）
  -XX:MetaspaceSize=256m             # 元空间初始值
  -XX:MaxMetaspaceSize=512m          # 元空间上限
  -XX:MaxDirectMemorySize=1g         # 直接内存（NIO用）
  
  -XX:+UseG1GC                       # G1 GC
  -XX:MaxGCPauseMillis=200           # 目标停顿200ms
  -XX:G1HeapRegionSize=4m            # 4G堆推荐4MB Region
  -XX:InitiatingHeapOccupancyPercent=45
  
  -XX:+HeapDumpOnOutOfMemoryError    # OOM时自动dump
  -XX:HeapDumpPath=/logs/heap.hprof
  
  -Xlog:gc*:file=/logs/gc.log:time,uptime:filecount=5,filesize=20m
  
  -Djava.security.egd=file:/dev/./urandom  # 加快随机数生成（Tomcat SSL用）
"
```

**常见误区：**

- `-Xmx` 不要超过 6G（8G服务器）：OS、Metaspace、堆外内存也需要占用空间
- `-Xms` = `-Xmx`：避免堆动态扩容导致的 STW
- 不要用 CMS（JDK 14 已删除）

###### 5. 你在项目中做了哪些优化？有没有什么效果？

**按优化层次分类回答（有数字支撑）：**

**① SQL 优化**

> "发现商品列表接口有 N+1 查询问题，原来循环查商品详情，改成批量 IN 查询后，DB 请求次数从每次接口请求的 50+ 次降到 2 次，接口响应时间从平均 400ms 降到 80ms。"

**② 缓存引入**

> "商品详情接口是读多写少，我们给它加了 Redis 缓存，设置了1小时 TTL。上线后数据库 QPS 从 3000 降到 300，Redis 缓存命中率稳定在 92%。"

**③ 异步化**

> "发短信、发邮件原来是同步调用，下单时要等待通知接口返回，增加了 200-300ms 延迟。改成 MQ 异步发送后，下单接口 P99 从 600ms 降到 200ms 以内。"

**④ JVM 调优**

> "监控到 Full GC 每小时发生 3-4 次，每次停顿约 2 秒。分析 Heap Dump 发现是 Session 缓存对象积累导致老年代快速填满。优化 Session 生命周期策略后，Full GC 频率降到每天 1-2 次，停顿降到 200ms 以内。"

**⑤ 索引优化**

> "慢查询日志发现订单搜索 SQL 执行时间 3-5 秒。EXPLAIN 分析后发现走了全表扫描（1000万行）。加了 (user_id, status, create_time) 联合索引后，查询时间降到 5ms 以内。"

###### 6. 如何进行性能测试和压力测试？

**工具：JMeter / k6 / Wrk / Gatling**

```bash
# Wrk 简单压测
wrk -t8 -c200 -d30s --script=post_order.lua http://api:8080/order/create

# JMeter（图形界面）：可以模拟复杂场景（登录→浏览→下单流程）
# k6（JavaScript，适合 CI 集成）
k6 run script.js --vus 200 --duration 60s
```

**压测流程：**

1. **环境准备**：压测环境要尽量接近生产（相同配置、相同数据量）；生产数据脱敏导入
2. **基准测试**：单接口从低并发开始（10→50→100→500），找到性能拐点
3. **场景测试**：模拟真实用户行为（浏览:下单:支付 = 10:2:1 的流量比例）
4. **容量测试**：把并发拉到预估峰值的 2 倍，看系统能否撑住
5. **稳定性测试**：在 80% 负载下跑 2-4 小时，观察内存是否持续增长（泄漏）

**关注指标：**
- TPS / QPS
- 响应时间 P50/P95/P99
- 错误率
- CPU/内存利用率
- GC 频率和停顿时间

###### 7. 项目中如何保证高可用？

**从架构层面系统性回答：**

**① 无单点（核心）**

- 应用：多实例 + Nginx 负载均衡
- MySQL：主从复制 + MHA 自动故障切换（或 MGR）
- Redis：Sentinel 3节点 或 Cluster 6节点
- Nacos：3节点集群

**② 熔断降级（故障隔离）**

Sentinel 对下游服务配置熔断规则，下游慢或不可用时自动熔断，快速返回降级结果，不让故障级联传播。

**③ 限流保护（防过载）**

网关层限制总 QPS，服务层按接口细粒度限流，保证核心接口在任何情况下都能响应。

**④ 超时控制（不无限等待）**

所有 HTTP 调用、DB 查询都设置超时时间，快速失败，释放线程。

**⑤ K8s 健康检查 + 自动重启**

容器异常退出或探针失败时，K8s 自动重启，保证服务持续可用。

**⑥ 异地多活（顶级高可用，如有则提）**

两个机房各部署一套服务，流量各承担一半，任一机房故障时 DNS 快速切换。

###### 8. 遇到过系统瓶颈吗？如何定位和解决的？

**STAR 格式回答：**

> **情景：** "有一次大促活动，流量比平时高了5倍，下单接口开始出现大量超时（P99 飙到 10 秒），告警系统立刻报警。"
> 
> **任务：** "我负责排查这次故障，需要在最短时间内恢复服务。"
> 
> **行动：** 
> "先看监控大盘——CPU 利用率正常，但 DB 连接池等待数很高，说明瓶颈在数据库。
> 
> 进一步查 MySQL slow log，发现有几条 JOIN 查询的执行时间超过了 3 秒。EXPLAIN 后发现索引没有覆盖过滤条件——大促新增了活动价格字段，但索引没有更新，导致回表扫描了几十万行。
> 
> 紧急添加索引（Online DDL），同时临时把查询结果缓存到 Redis（5秒 TTL），先让系统恢复正常响应。"
> 
> **结果：** "加索引后查询时间从 3 秒降到 10ms，P99 响应时间恢复到 200ms 以内，整个过程约 15 分钟。事后总结：大促前必须进行压测，新需求涉及的 SQL 要提前做 EXPLAIN 分析。"

###### 9. 如何进行系统容量评估？

**容量评估方法论：**

**① 流量预测**

- 历史同期数据 × 增长系数
- 大促活动：历史大促 QPS × 预估增长（保守2倍，积极3倍）
- 新系统：参考行业标杆，按业务预期估算

**② 单机压测**

```bash
# 在测试环境对单实例进行压测，找到性能拐点
wrk -t8 -c200 -d60s http://app:8080/api/order/create
# 记录：单实例最大稳定 QPS = 500，P99 < 200ms
```

**③ 计算实例数**

```
所需实例数 = 峰值 QPS / 单实例 QPS × 冗余系数(1.5~2)
           = 5000 / 500 × 2 = 20 实例
```

**④ 依赖资源评估**

- DB：估算 TPS（写请求×操作数）、连接数（实例数 × 连接池）
- Redis：估算命令数/s、内存（热点数据量 × 平均大小 × 1.5倍 overhead）
- MQ：估算消息数/s、存储（消息量 × 保留时间 × 消息大小）

**⑤ 预留弹性**

正常运行在 60% 负载（留 40% 应对突发），配合 K8s HPA（超70%时自动扩容）。

大促前提前扩容（提前 1 小时），大促结束后缩容，避免资源浪费。
