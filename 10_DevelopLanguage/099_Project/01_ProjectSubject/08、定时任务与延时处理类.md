# 定时任务与延时处理类

---

###### 1. 自动关闭订单的方式有哪些？

**答：**

订单超时自动关闭是个高频场景（用户下单后30分钟未支付自动取消）。有以下几种实现方式：

**① 定时任务扫表（最简单）**

每隔 5 分钟用 XXL-Job 扫描待支付订单表，找出超时的订单批量关闭。

优点：实现简单，不依赖额外中间件。
缺点：关闭不及时（最多延迟 5 分钟），大表扫描有性能问题，容易漏扫或重复扫。

适合：数据量小、时效性要求不高的场景。

**② RocketMQ 延迟消息（推荐）**

下单时发一条延迟消息（延迟 30 分钟），消息消费时检查订单状态，仍为待支付则关闭：
```java
Message message = new Message("order-close-topic", orderId.getBytes());
message.setDelayTimeLevel(16); // RocketMQ 内置延迟级别，level16 = 30分钟
producer.send(message);
```
优点：及时性好，实现简单，依赖现有 MQ 组件。
缺点：RocketMQ 内置延迟只有固定级别（4.x），自定义精确延迟需要 5.x 版本或其他方案。

**③ Redis ZSet 延迟队列**

下单时把订单 ID 放到 Redis ZSet，score 为超时时间戳；定时任务每秒扫描 score <= now 的成员，触发关单逻辑：
```java
// 下单时
redisTemplate.opsForZSet().add("order:timeout", orderId, expireTimestamp);
// 定时任务
Set<String> expiredOrders = redisTemplate.opsForZSet().rangeByScore("order:timeout", 0, System.currentTimeMillis());
```
优点：精度高（秒级），Redis 扫描性能好。
缺点：Redis 重启后 ZSet 数据需要持久化，且消费逻辑要保证幂等。

**④ 时间轮（HashedWheelTimer）**

内存时间轮，适合单机大量短期定时任务（秒级精度），但服务重启会丢失所有任务，需要持久化配合。

**综合推荐**：30分钟延迟 + 可靠性要求高 → RocketMQ 延迟消息；秒级精度 + 灵活性 → Redis ZSet；量小不追求实时 → 定时扫表。

###### 2. 项目中如何实现定时任务？

**答：**

我们用 XXL-Job 做分布式定时任务，生产环境定时任务不能用单机的 `@Scheduled`，原因是多实例部署时会重复执行。

**XXL-Job 基本架构**：调度中心（xxl-job-admin）负责触发、监控；执行器（我们的 Spring Boot 服务）注册到调度中心，收到调度指令后执行任务。

**使用方式**：
```java
@XxlJob("orderCloseHandler")
public void orderCloseHandler() {
    // 查询超时订单，批量关闭
    List<Long> expiredOrders = orderMapper.queryExpiredOrders();
    for (Long orderId : expiredOrders) {
        orderService.closeOrder(orderId);
    }
}
```
在 XXL-Job 控制台配置 Cron 表达式（如 `0 */5 * * * ?` 每5分钟执行），选择执行器、路由策略（分片、故障转移）。

**关键特性**：
- **分片执行**：大批量任务可以拆分给多个执行器并行处理（`XxlJobHelper.getShardIndex()` 获取分片序号）
- **失败重试**：任务失败后自动重试指定次数
- **告警**：任务失败自动邮件/钉钉告警
- **日志**：调度日志和执行日志集中在控制台，方便排查

**简单场景**：如果只是单机服务，且同一时间只有一个实例在运行，`@Scheduled` + `@SchedulerLock`（ShedLock 库）也可以保证不重复执行。

###### 3. 如何实现延时队列？

**答：**

延时队列是指消息在指定时间后才被消费，常见场景：订单超时关闭、优惠券到期通知、任务延迟重试。

**方案一：RocketMQ 延迟消息（推荐）**

```java
// 4.x：固定延迟级别（1s/5s/10s/30s/1m/2m/3m/4m/5m/6m/7m/8m/9m/10m/20m/30m/1h/2h）
message.setDelayTimeLevel(16); // 30分钟

// 5.x：自定义延迟时间
message.setDeliveryTimestamp(System.currentTimeMillis() + 30 * 60 * 1000);
```
生产级推荐，可靠性高，重启不丢任务。

**方案二：Redis ZSet**

```java
// 加入延迟队列
zsetOps.add("delay:queue", taskId, executeTimestamp);

// 定时扫描（每秒执行）
Set<String> tasks = zsetOps.rangeByScore("delay:queue", 0, System.currentTimeMillis());
for (String taskId : tasks) {
    // 处理任务
    zsetOps.remove("delay:queue", taskId);
}
```
精度高，实现灵活，但需要持久化兜底（AOF），且扫描消费要保证原子性（Lua 脚本或分布式锁）。

**方案三：HashedWheelTimer（单机）**

Netty 的时间轮，内存操作，延迟精度高（毫秒级），适合单机大量延迟任务：
```java
HashedWheelTimer timer = new HashedWheelTimer();
timer.newTimeout(timeout -> closeOrder(orderId), 30, TimeUnit.MINUTES);
```
缺点：服务重启任务丢失，需要从 DB 恢复。

**方案四：RabbitMQ 死信队列 + TTL**

消息设置 TTL，过期后进死信队列，消费死信队列实现延迟效果。实现有点绕，不如 RocketMQ 直接。

###### 4. 定时任务如何保证不重复执行？

**答：**

多实例部署场景下，多个实例同时触发定时任务会导致重复执行，危害很大（比如重复发短信、重复计费）。

**方案一：分布式调度框架（推荐）**

XXL-Job、Elastic-Job 本身就解决了这个问题。调度中心只触发一次，根据路由策略（轮询、故障转移、分片）分配给某个执行器实例，不会多实例同时执行。

**方案二：数据库乐观锁（ShedLock）**

ShedLock 在 DB 或 Redis 里维护一张锁表，任务执行前获取锁，执行完释放：
```java
@Scheduled(cron = "0 0 1 * * ?")
@SchedulerLock(name = "sendDailyReport", lockAtMostFor = "PT10M", lockAtLeastFor = "PT5M")
public void sendDailyReport() {
    // 只有拿到锁的实例会执行
}
```
优点：不依赖 XXL-Job 等框架，在现有 `@Scheduled` 基础上加一个注解即可。

**方案三：Redis SETNX 抢锁**

任务执行前用 Redis `SET NX EX` 抢一把锁，抢到的执行，抢不到的跳过：
```java
Boolean acquired = redis.opsForValue().setIfAbsent("job:lock:sendReport", "1", 10, TimeUnit.MINUTES);
if (!acquired) return; // 没抢到，其他实例已在执行
// 执行任务...
```

**最佳实践**：生产级推荐 XXL-Job 统一管理；如果项目里已有，ShedLock 是轻量好用的补充方案。

###### 5. 大量定时任务如何管理和调度？

**答：**

系统里定时任务一多（几十上百个），就需要统一管理，否则维护成本极高。

**统一调度平台：XXL-Job**

- 所有定时任务在 XXL-Job 控制台集中配置、管理
- 可视化：任务列表、执行日志、调度轨迹一目了然
- 动态调整：不重启服务直接在控制台修改 Cron 表达式、启停任务
- 告警：任务失败、超时自动告警
- 分片：大任务拆分给多个执行器并行处理，提高效率

**任务分类与优先级**：

- 核心任务（对账、结算、库存同步）：单独执行器，有充足资源，配置监控告警
- 普通任务（数据统计、报表生成）：共享执行器
- 轻量任务（清理临时数据、刷新缓存）：低优先级

**避免任务堆积**：

- 设置任务执行超时时间，超时自动中断
- 长任务支持分片执行（按时间段或 ID 范围拆分），每个分片独立执行不互相影响
- 任务执行时间超过触发间隔时，允许配置"上次未完成则本次跳过"的策略

**监控**：

- 任务成功率、平均执行时长接入 Prometheus + Grafana 监控面板
- 任务失败立即告警，告警消息包含任务名称、失败原因、上次成功时间
