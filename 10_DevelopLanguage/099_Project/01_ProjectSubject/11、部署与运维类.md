# 部署与运维类

---

###### 1. 项目是如何部署的？

**答：**

我们项目采用容器化 + Kubernetes 部署，整体流程是 CI/CD 自动化的：

**部署架构**：

- **开发/测试环境**：开发者推代码到 GitLab，GitLab CI 自动触发 Pipeline，构建 Docker 镜像，推送到 Harbor 私有镜像仓库，自动部署到测试环境 K8s 集群。

- **生产环境**：测试通过、代码 Review 后，由 Jenkins Pipeline 执行生产部署。生产部署需要人工确认，避免误操作。K8s 滚动更新（Rolling Update）逐步替换实例，不停机。

**Kubernetes 部署配置**：
```yaml
spec:
  replicas: 3  # 最少 3 副本，保证高可用
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0      # 更新期间最少可用实例数
      maxSurge: 1            # 更新期间最多额外实例数
  template:
    spec:
      containers:
        - name: order-service
          image: harbor.company.com/order-service:v1.2.3
          resources:
            requests:
              cpu: "500m"
              memory: "1Gi"
            limits:
              cpu: "2"
              memory: "4Gi"
          readinessProbe:       # 就绪探针，通过后才接收流量
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
          livenessProbe:        # 存活探针，失败则重启容器
            httpGet:
              path: /actuator/health/liveness
              port: 8080
```

**多环境管理**：Nacos 命名空间区分 dev/test/staging/prod，K8s namespace 也按环境隔离。

###### 2. 有没有用到容器化技术？

**答：**

用了，整个项目都是容器化部署，基于 Docker + Kubernetes。

**Docker 使用**：

每个微服务写一个 Dockerfile，基于 `eclipse-temurin:17-jre-alpine` 基础镜像（轻量，约 200MB，比完整 JDK 镜像小很多）：
```dockerfile
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY target/order-service.jar app.jar
ENV JAVA_OPTS="-Xms512m -Xmx1g -XX:+UseG1GC"
EXPOSE 8080
ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
```

镜像分层优化：COPY 依赖 jar 和应用 jar 分两层，修改代码不影响依赖层，构建速度快。

**Kubernetes 使用**：

- **Deployment**：管理服务副本，支持滚动更新和回滚
- **Service**：内部 DNS 服务发现（`order-service.default.svc.cluster.local`），ClusterIP 或 LoadBalancer
- **ConfigMap/Secret**：配置和密钥注入
- **HPA（水平自动扩缩容）**：根据 CPU 使用率自动扩容 Pod 数量，大促时自动扩，流量退去后缩容
- **PodDisruptionBudget**：保证滚动更新期间最少可用副本数，防止服务不可用

###### 3. 如何实现灰度发布？

**答：**

灰度发布是指先把新版本推给一小部分用户，验证功能和稳定性后再全量推送，降低上线风险。

**方案一：网关层流量切分（我们项目的做法）**

在 Spring Cloud Gateway 里，根据请求头（`X-Gray-User: true`）或用户 ID 百分比，把一定比例的流量路由到新版本服务：
```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service-gray
          uri: lb://order-service-gray    # 新版本的 Service
          predicates:
            - Header=X-Gray-User, true    # 带灰度标签的用户走新版本
          order: 1
        - id: order-service
          uri: lb://order-service         # 稳定版本
          order: 10
```
后端新旧版本同时存在，通过 K8s 不同 Deployment 区分，同一个 K8s Service 可以用 Label Selector 切流量比例。

**方案二：Feature Flag（功能开关）**

在代码里加开关：
```java
if (featureFlag.isEnabled("new-order-flow", userId)) {
    return newOrderService.process(request);
} else {
    return oldOrderService.process(request);
}
```
开关配置在 Nacos，可以动态调整开放比例（1% → 5% → 20% → 100%），出问题立即关闭，回滚时间 < 1 秒。

**灰度指标观察**：新版本服务上线后，观察错误率、TP99、业务指标（下单成功率）是否与旧版本一致，稳定 24 小时后全量。

###### 4. 如何实现蓝绿部署？

**答：**

蓝绿部署是保持两套完全独立的生产环境（蓝环境 = 当前生产版本，绿环境 = 新版本），通过切换流量入口来实现零停机上线。

**实现方式**：

**Kubernetes 蓝绿部署**：

蓝绿两套 Deployment 同时存在，通过修改 Service 的 selector label 切换流量：
```yaml
# 蓝色版本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service-blue
  labels:
    version: blue
---
# 绿色版本（新版本）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service-green
  labels:
    version: green
---
# Service：修改 selector 切换流量
kind: Service
spec:
  selector:
    app: order-service
    version: green   # 切换到绿色版本
```

**切换流程**：
1. 部署绿色版本，做冒烟测试
2. 测试通过后，修改 Service selector 切换到绿色，蓝色停止接收流量
3. 观察一段时间（1~4小时），确认无异常
4. 删除蓝色版本，释放资源

**蓝绿 vs 灰度**：
- 蓝绿：切换是一次性的，风险集中；适合变更较大的版本
- 灰度：流量逐步迁移，风险分散；适合持续迭代

**注意**：蓝绿部署需要双倍资源（两套环境同时运行），有一定成本。

###### 5. CI/CD 流程是怎样的？

**答：**

我们的 CI/CD 基于 GitLab CI + Jenkins，整个流程如下：

**CI（持续集成）流程**：

每次 Push/MR 触发 GitLab CI Pipeline：
1. 代码静态检查（SonarQube，质量门禁不通过则阻断）
2. 单元测试（`mvn test`，覆盖率低于 70% 则失败）
3. Maven 构建（`mvn package -DskipTests`）
4. Docker 镜像构建 + 推送 Harbor

**CD（持续部署）流程**：

- **自动部署到测试环境**：CI 通过后自动触发 Jenkins Job，把镜像部署到测试 K8s 集群
- **手动触发部署到生产**：测试验收通过后，在 Jenkins 手动点击"发布生产"，需要审批（2 人确认）
- Jenkins Pipeline 执行生产部署：滚动更新 K8s Deployment，更新后自动做冒烟测试，失败则自动回滚

**Pipeline 核心节点**：
```
Checkout → 代码质量检查 → 单元测试 → 构建 → 镜像打包 → 推送镜像 
  → 部署测试环境 → 集成测试 → [人工审批] → 部署生产 → 冒烟测试 → 完成
```

**好处**：从提交代码到测试环境可用，全程自动化，约 10 分钟；生产发布标准化，有审批、有记录、有回滚，质量可控。

###### 6. 如何保证服务平滑上线？

**答：**

平滑上线的核心是：新版本实例完全就绪后才接收流量，旧版本实例优雅处理完已有请求后再退出，整个过程用户无感知。

**关键配置**：

**① Kubernetes 就绪探针**

新 Pod 启动后，Kubernetes 会轮询就绪探针，只有就绪探针返回 200 后，才把 Pod 加入 Service 的 Endpoints，开始接收流量。防止应用还在 Spring 容器初始化中就接收到请求：
```yaml
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 30   # Spring Boot 启动时间
  periodSeconds: 5
  failureThreshold: 3
```

**② Spring Boot 优雅停机**

收到 SIGTERM 信号时，Spring Boot 不立即停止，而是：
- 停止接受新请求（从注册中心注销，Nacos 心跳停止）
- 等待正在处理的请求完成（默认最多等 30 秒）
- 然后再关闭 JVM

```yaml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

**③ K8s preStop Hook**

```yaml
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 15"]  # 等待 15 秒，让注册中心感知实例下线，停止路由流量
```

**④ 滚动更新策略**

`maxUnavailable: 0` + `maxSurge: 1`，保证更新期间不降低可用实例数。

###### 7. 项目的备份和恢复策略是什么？

**答：**

备份是系统安全的最后一道防线，必须有完整的策略，而且要定期演练，确认备份可用。

**数据库备份**：

- **全量备份**：每天凌晨 2 点用 XtraBackup（物理备份，比 mysqldump 快 5~10 倍）做全量备份，压缩后上传 OSS，保留 30 天
- **Binlog 备份**：每小时把 binlog 同步到 OSS，支持 Point-in-Time Recovery（可以恢复到任意秒）
- **备份验证**：每周自动在测试环境执行恢复演练，验证备份文件完整可用

**恢复 RTO/RPO 目标**：
- RTO（恢复时间）：< 2 小时（全量恢复）
- RPO（数据丢失窗口）：< 1 小时（依赖 binlog 频率）

**Redis 备份**：

- AOF + RDB 混合持久化，定期同步到 OSS
- Redis Cluster 本身有主从复制，主节点宕机 30 秒内自动 failover

**代码和配置备份**：

- 代码在 GitLab，有多副本存储
- K8s 配置（YAML）存在 GitLab，基础设施即代码（IaC）
- Nacos 配置导出备份，每周执行

**恢复演练**：每季度做一次全流程恢复演练（从 OSS 拉备份 → 恢复数据库 → 验证数据完整性），确保真正需要恢复时不慌。
