# 1、项目基础类

> 这类题考察的是你对自己项目的熟悉程度和表达能力。关键原则：**说真话，说具体，用数字说话**。

###### 1. 你这个 XXX 是个什么样的项目？

**回答模板（STAR 结构）：**

先说背景（Business Context），再说规模，再说技术挑战：

> "我们做的是一个 XX 行业的 B2C 电商平台，核心业务是商品展示、购物车、订单、支付和物流跟踪。日活用户大概 X 万，峰值 QPS 在 XX 左右，数据库主要是 MySQL，缓存用 Redis，消息队列用 RocketMQ，后端是 Spring Boot 微服务架构，部署在 K8s 上。"

**核心要素：**

- **业务定位**：to B / to C？什么行业？核心功能是什么？
- **规模数据**：日活用户、订单量、数据量（让面试官有直观感受）
- **技术栈**：主要框架/中间件
- **你的角色**：独立负责/团队协作，负责哪块

**避免的答法：**

- "就是一个普通的电商项目..."（太模糊）
- "做了很多功能"（没有重点）

---

###### 2. 介绍一下你在项目中的职责和负责模块

**好的回答突出「深度」而非「广度」：**

> "我主要负责订单和支付模块，从需求分析、数据库设计、接口设计到代码实现都是我来做的。技术上比较有挑战的部分是支付模块的分布式事务处理——下单要同时操作订单库、库存库和优惠券库，我们用的是 RocketMQ 事务消息来保证最终一致性。另外我也负责了系统的整体性能优化，把核心下单接口的 P99 从 800ms 优化到了 150ms 以内。"

**强调：**

- 具体负责哪些模块（不要说"参与了整个系统"）
- 解决了什么有技术含量的问题
- 有可以量化的成果（降低了多少延迟/提升了多少性能）

###### 3. 画一下你们整个项目架构？

这道题在白板/面试中让你画图，核心是展示**架构思维和技术全貌**。

**标准微服务架构图（分层描述）：**

```
用户层
   ├── Web 浏览器
   └── APP（iOS / Android）
          ↓ HTTPS
接入层
   ├── CDN（静态资源）
   └── API 网关（Spring Cloud Gateway）
          ├── 鉴权（JWT 验证）
          ├── 限流（Sentinel）
          └── 路由
          ↓
服务层（微服务）
   ├── 用户服务
   ├── 商品服务
   ├── 订单服务
   ├── 支付服务
   └── 通知服务
          ↓（RPC / HTTP）
基础设施
   ├── 注册中心：Nacos
   ├── 配置中心：Nacos Config
   ├── 消息队列：RocketMQ
   ├── 数据库：MySQL（主从）
   ├── 缓存：Redis Cluster
   └── 搜索：Elasticsearch
可观测性
   ├── 链路追踪：SkyWalking
   ├── 监控：Prometheus + Grafana
   └── 日志：ELK
```

**回答时的注意事项：**

从外到内、从上到下地描述，重点说清楚：流量怎么进来、服务间怎么通信、数据存在哪里。不需要面面俱到，抓住核心链路即可。

###### 4. 项目采用什么技术栈？为什么选择这些技术？

**回答要点：说 "选了什么" 更要说 "为什么选"**

```
后端框架：Spring Boot 3.x + Spring Cloud Alibaba
  为什么：生态成熟，团队熟悉，Nacos/Sentinel/Seata 配套完善

服务注册/配置：Nacos
  为什么：同时支持服务发现和配置中心，一套搞定两件事；Eureka 已停更

数据库：MySQL 8.0（InnoDB）
  为什么：业务场景以事务操作为主，MySQL 是最成熟的 OLTP 数据库

缓存：Redis 6.0 Cluster
  为什么：高并发读场景多，Redis 单节点万级 QPS，集群水平扩展

消息队列：RocketMQ
  为什么：有事务消息（分布式事务用），有延迟消息（订单超时取消），国内支持好

链路追踪：SkyWalking
  为什么：Java 无侵入（Agent 方式），不需要改业务代码

部署：K8s + Docker
  为什么：容器化弹性伸缩，结合 HPA 自动扩容
```

面试时不需要全部列出，抓住几个核心技术，说清楚选择原因，比罗列一大堆更有价值。

###### 5. 项目的业务流程是怎样的？

以电商为例，核心业务流程：

```
注册/登录 → 浏览商品 → 加购物车 → 填写收货地址 → 选择优惠券/积分
    ↓
确认下单（预占库存 + 冻结优惠券 + 生成订单）
    ↓
发起支付 → 调第三方支付 → 异步回调 → 支付成功
    ↓
商家接单 → 发货（生成快递单）→ 物流跟踪
    ↓
确认收货（自动或手动）→ 释放积分/返现 → 评价
```

**回答技巧：** 选择你最熟悉的业务流程，说清楚主流程中的关键节点，特别是有技术难点的地方（如下单时的分布式事务、支付的幂等处理）。

###### 6. 项目有多少个微服务？如何划分的？

**典型中型电商项目微服务划分（10-15个服务）：**

```
基础服务（Platform）：用户、认证、通知（短信/推送/邮件）、文件存储
商品域（Product）：商品、类目、品牌、搜索
交易域（Trade）：购物车、订单、支付、退款
物流域（Logistics）：快递单、物流轨迹
运营域（Operation）：优惠券、积分、活动
```

**划分原则（面试时要能说出来）：**

1. **按业务域划分**（Domain-Driven Design）：每个微服务对应一个业务子域，内聚高，与其他服务耦合低
2. **单一职责**：每个服务只做一件事，职责边界清晰
3. **独立部署**：服务可以独立扩缩容，订单服务压力大时单独扩容，不影响其他服务
4. **数据自治**：每个服务有自己的数据库，不共享 DB（跨服务数据通过 API 获取）

**服务数量建议：** 不是越多越好。团队规模决定服务数量——"一块披萨能喂饱的团队（5-8人）负责一个服务"是业界常见经验。微服务太多会增加运维复杂度。
