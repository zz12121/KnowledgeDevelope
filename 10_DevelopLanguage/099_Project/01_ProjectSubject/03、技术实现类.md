###### 1. 接口是怎么定义的？采用什么数据格式？如何实现？

**答：**

我们的接口基本都遵循 RESTful 风格，用 HTTP 方法区分语义：GET 查数据、POST 新建或复杂查询、PUT/PATCH 更新、DELETE 删除。数据格式统一用 JSON，Content-Type 设为 `application/json`。

对外（前端/第三方）和对内（服务间调用）略有区别：

- **对外接口**：Spring Boot + Spring MVC，`@RestController` + `@RequestMapping`，返回统一响应体：
  ```json
  {
    "code": 0,
    "message": "success",
    "data": { ... },
    "traceId": "xxx"
  }
  ```
  非 0 的 code 代表业务错误，HTTP 状态码统一返回 200，只在网关层或 Spring 异常处理器映射业务错误码。
  traceId 会透传到前端，方便排查问题。

- **服务间调用**：我们用 OpenFeign，接口定义放在公共 SDK 里，消费方引入依赖直接调用，和本地方法一样。Feign 底层默认 HTTP（也可换成 HTTP/2 + OkHttp），Ribbon/Spring Cloud LoadBalancer 做客户端负载均衡。

接口文档用 Springdoc（OpenAPI 3），加了 `@Operation`、`@Schema` 注解，启动后自动生成 Swagger UI，前后端对齐就靠这个。

**面试加分点**：统一响应体里带 traceId，一旦出问题前端日志直接对应后端链路，排查效率大幅提升。

###### 2. 实现异步编程的有几种方式？

**答：**

在 Java 项目里，常用的异步方式大概有这几种，从简单到复杂：

**① `@Async` + 线程池（最常用）**

Spring 提供的最简单方式，方法上加 `@Async`，Spring 会把方法调用提交到线程池异步执行。有几个注意事项：
- 要配置自定义线程池，别用默认的 `SimpleAsyncTaskExecutor`（每次创建新线程，没有池化）。
- 必须从 Spring 容器外部调用才能生效（内部直接调用 `this.method()` 无法被代理拦截）。
- 异常要通过 `AsyncUncaughtExceptionHandler` 捕获，否则会被静默吞掉。

**② `CompletableFuture`（推荐，灵活）**

适合需要组合多个异步任务的场景：
```java
CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(() -> userService.getById(id), executor);
CompletableFuture<Order> orderFuture = CompletableFuture.supplyAsync(() -> orderService.getByUserId(id), executor);
CompletableFuture.allOf(userFuture, orderFuture).join();
```
支持 `thenApply`、`thenCompose`、`thenCombine`、`exceptionally` 等链式操作，可以串行、并行、组合，比较适合有复杂依赖关系的异步逻辑。

**③ 消息队列（解耦 + 异步）**

对于耗时的非关键路径操作（发短信、发邮件、更新统计），发到 MQ 异步处理，主流程直接返回。这种方式不是纯粹的"线程异步"，而是通过中间件实现的异步，好处是天然解耦、可重试、可限流。

**④ 线程池 + Future / `ExecutorService.submit`**

更底层的方式，适合需要精细控制线程数的场景，比如批量处理任务分批提交到线程池，然后用 `Future.get()` 等待结果。

**⑤ 响应式编程（WebFlux + Project Reactor）**

如果整个项目用的是 Spring WebFlux，可以用 `Mono`/`Flux` 实现非阻塞异步。适合高并发 IO 密集型场景，但学习曲线较陡，我们项目里没有大规模使用。

**面试加分点**：`@Async` 内部调用不生效是个高频坑，要说清楚原因（Spring AOP 代理只拦截容器外部调用）。

###### 3. 项目中如何实现分布式事务？

**答：**

我们项目中分布式事务主要出现在以下场景：跨服务操作（比如下单同时扣库存）、跨数据库操作、本地操作 + 远程调用（比如写 DB + 发消息）。

针对不同场景，我们用了不同方案：

**方案一：本地消息表（最常用，适合"写DB + 发消息"场景）**

在同一个数据库里建一张消息表，业务操作和消息记录在一个本地事务里完成，然后异步轮询消息表发送 MQ 消息：
```
BEGIN 本地事务
  1. 业务操作（扣库存）
  2. 插入消息记录（状态=待发送）
COMMIT

异步任务：扫描待发送消息 → 发 MQ → 更新状态=已发送
消费者消费失败：MQ 重试，消费者做幂等处理
```
实现简单，只依赖本地数据库，适合大多数场景。

**方案二：RocketMQ 事务消息（强一致场景）**

RocketMQ 支持半消息（Half Message）机制，生产者先发半消息到 Broker，Broker 不会投递给消费者；本地事务执行成功后 Commit，Broker 才投递；本地事务失败则 Rollback，Broker 删除消息；如果生产者宕机，Broker 会定期回查本地事务状态。

**方案三：Seata AT 模式（跨服务数据一致性）**

对于跨服务调用必须保证强一致的场景，我们用 Seata AT 模式。AT 模式无侵入，基于 undolog 实现两阶段提交，Spring Boot 项目引入依赖，在全局事务入口加 `@GlobalTransactional` 注解，Seata 会自动拦截 SQL 生成前后镜像，出问题自动回滚。

**原则**：优先用最终一致性（本地消息表/MQ事务消息），能避免强一致就避免，强一致方案性能代价大。

###### 4. 项目中如何实现幂等性？

**答：**

幂等性的核心是：同一个请求，不管执行多少次，结果都一样。项目里有几个典型场景，每个场景的解法略有不同：

**① 数据库唯一索引（最简单）**

对于"不能重复插入"的场景，直接在数据库加唯一索引，重复插入会抛异常，捕获后返回成功即可。比如优惠券领取表，用 `(user_id, coupon_template_id)` 联合唯一索引防止重复领取。

**② 幂等 Token（通用方案）**

前端请求前先调接口获取一个唯一 token（UUID），存到 Redis。正式请求携带这个 token，后端用 Redis `SET NX` 原子检查：存在说明已处理过，不存在就处理并删除 token。
```java
String key = "idempotent:" + token;
Boolean isNew = redisTemplate.opsForValue().setIfAbsent(key, "1", 10, TimeUnit.MINUTES);
if (!isNew) return "重复请求";
// 业务处理...
```

**③ 状态机 + 乐观锁（状态流转场景）**

订单支付、审批流等场景，状态只能从"待支付"变"已支付"，SQL 里加 `WHERE status = 'PENDING'`，用乐观锁（`version` 字段）配合，更新条数为 0 说明已经被其他请求处理过了。

**④ MQ 消费幂等**

消费者先查本地是否已消费过（Redis set 或 DB 记录），没有再处理，处理后记录消费状态。消息里带唯一业务ID（如 orderId），作为去重 key。

**面试加分点**：幂等 token 方案里，`SET NX` + `DELETE` 这两步要注意，如果先查再删，高并发下有并发漏洞，要用 Lua 脚本保证原子性。

###### 5. 如何保证接口的安全性？

**答：**

接口安全是多层防护，我们从以下几个维度来做：

**① 认证与授权**

所有接口通过网关统一鉴权，用 JWT 做无状态认证。JWT 里包含用户ID、角色、权限列表，网关拦截请求验签，通过后把用户信息透传到下游服务。权限控制用 RBAC 模型，Service 层关键接口加 `@PreAuthorize` 注解。

**② 防重放攻击**

敏感接口（支付、转账）加时间戳 + nonce 签名验证：
- 请求头带 `timestamp`（10分钟内有效）+ `nonce`（一次性随机字符串）+ `sign`（HMAC-SHA256 签名）
- 后端验证 timestamp 时效、nonce 是否用过（Redis SET NX），最后验签

**③ 输入校验**

所有入参用 Bean Validation 校验（`@NotNull`、`@Size`、`@Pattern` 等），防止非法输入。SQL 操作用 MyBatis 预编译防注入，绝不拼字符串。XSS 防御用 Jackson 全局序列化时 HTML 转义，或者 Spring Security 的 HTML Sanitizer。

**④ 限流与防刷**

网关层用 Spring Cloud Gateway + Redis 令牌桶对单个 IP 和用户做限流。高风险接口（登录、注册、发短信）额外加行为验证码和频率限制。

**⑤ HTTPS 传输加密**

所有接口强制 HTTPS，HTTP 请求重定向到 HTTPS，开启 HSTS。

###### 6. 项目中如何实现限流、降级、熔断？

**答：**

我们用 Sentinel 做限流降级熔断，这是阿里开源的，和 Spring Cloud Alibaba 生态配合很好，比 Hystrix 配置更灵活，控制台可视化，支持动态规则推送。

**限流（Flow Control）**

```java
@SentinelResource(value = "queryProduct", blockHandler = "queryProductBlock")
public ProductVO queryProduct(Long id) {
    return productMapper.selectById(id);
}
public ProductVO queryProductBlock(Long id, BlockException ex) {
    return ProductVO.ofDefault(); // 返回默认值或降级数据
}
```
规则配置：QPS 超过阈值触发限流，支持按来源、按链路限流。规则存在 Nacos，动态生效不重启。

**熔断（Circuit Breaker）**

Sentinel 支持三种熔断策略：
- 慢调用比例：RT 超过阈值的请求占比超过设定值，触发熔断
- 异常比例：异常率超过阈值触发
- 异常数：异常数超过阈值触发

熔断打开后，后续请求直接走降级逻辑，一段时间后进入半开状态，放一个探测请求，成功则关闭熔断器。

**降级**

降级是主动的，比如大促时主动关闭推荐系统（非核心功能），保证核心下单链路的资源。我们用 Feature Flag（Nacos 配置开关）控制，一行配置关掉某个非核心功能。

**面试加分点**：Sentinel 和 Hystrix 的核心区别——Sentinel 用信号量 + 滑动时间窗口统计，Hystrix 用线程池隔离，Sentinel 开销更小，更适合高并发场景。

###### 7. 微服务之间如何通信？为什么选择这种方式？

**答：**

我们项目服务间通信分两类：

**① 同步调用：OpenFeign + HTTP**

大部分场景用 OpenFeign 做服务间 RPC 调用。Feign 把 HTTP 调用封装成接口，Consumer 引入 API 包，像调本地方法一样调远程服务：
```java
@FeignClient(name = "order-service", fallback = OrderServiceFallback.class)
public interface OrderServiceClient {
    @GetMapping("/orders/{id}")
    OrderVO getOrder(@PathVariable Long id);
}
```
为什么选 Feign 而不是 Dubbo？主要是我们整个技术栈是 Spring Cloud 生态，HTTP 接口更通用，运维也简单，不依赖额外的注册中心协议。如果对延迟要求极高（比如 TP99 要在 5ms 以内），可以考虑换成 Dubbo gRPC，但我们的场景 HTTP 完全够用。

**② 异步通信：消息队列**

跨服务解耦、无需立即返回结果的场景用 MQ，比如下单成功后通知库存扣减、积分增加、发短信等，都是异步发消息，各服务独立消费。

**选型考虑**：
- 需要同步返回结果 → Feign
- 需要解耦/异步/广播 → MQ
- 对延迟极敏感、内网高频调用 → 考虑 Dubbo/gRPC
- 需要流式处理 → gRPC streaming

###### 8. 项目中如何实现服务发现和注册？

**答：**

我们用 Nacos 做服务注册与发现。服务启动时会自动向 Nacos Server 注册自己的 IP、端口、服务名，同时定期发心跳（默认 5 秒）证明自己还活着；如果 15 秒没收到心跳，Nacos 把该实例标为不健康；30 秒还没有，剔除实例。

服务消费方（比如 Feign）通过 Nacos 客户端拉取服务列表，然后由 Spring Cloud LoadBalancer 做客户端负载均衡（轮询/随机/权重）选一个实例发请求。

**实例变更推送**：Nacos 采用"推拉结合"——服务实例上下线时，Nacos Server 主动通过长轮询推送变更给客户端，客户端不需要频繁主动拉取，实时性好。客户端本地也有缓存，Nacos 挂掉时还能用缓存继续调用，不会立刻不可用。

**Nacos vs Eureka**：Eureka 是纯 AP（强调可用性），网络分区时可能读到旧数据；Nacos 支持 AP/CP 切换，CP 模式下用 Raft 协议保证数据一致性。Nacos 还支持配置中心、健康检查（TCP/HTTP/MySQL）等功能，一个组件顶多个。

###### 9. 项目中如何实现配置中心？

**答：**

我们用 Nacos Config 做配置中心，同时管理应用配置和动态开关。

基本用法：把需要动态变更的配置（数据库连接、限流阈值、Feature Flag 等）放到 Nacos，Spring Boot 引入 `spring-cloud-starter-alibaba-nacos-config`，在 `bootstrap.yml` 里指定 `dataId` 和命名空间。

**动态刷新**：需要热更新的 Bean 加 `@RefreshScope`（方法级别可以用 `@Value` + `@RefreshScope`），Nacos 配置变更后自动推送，Bean 刷新，无需重启服务。对于数据源等无法热刷新的配置，仍然走静态配置文件。

**命名空间隔离**：我们按环境（dev/test/staging/prod）创建不同命名空间，配置完全隔离，不会串环境。

**配置安全**：敏感配置（数据库密码、密钥）用 Jasypt 加密，配置中心存加密串，应用启动时解密。或者接入 KMS（云厂商密钥管理服务），启动时动态拉取。

**版本管理**：Nacos 控制台支持配置历史版本查看和回滚，发布配置前可以 Diff 对比差异，出问题一键回滚到上一版本。

###### 10. 如何实现分布式 ID 生成？

**答：**

分布式 ID 要满足：全局唯一、趋势递增（利于 B+ 树索引）、高性能、高可用。我们根据不同场景用了不同方案：

**主要方案：雪花算法（Snowflake）**

订单ID、用户ID 等核心业务 ID 用雪花算法，64位 Long 类型：
```
1bit(符号) + 41bit(时间戳毫秒) + 10bit(机器ID) + 12bit(序列号)
```
每毫秒最多生成 4096 个 ID，按时间递增，满足数据库索引友好性。

实际项目里用 Hutool 的 `Snowflake` 工具类，机器 ID 从 Redis 分配（或用 K8s Pod 序号），避免多实例冲突。

**时钟回拨问题处理**：
- 如果回拨小于 5ms，等待时钟追上后再生成
- 如果回拨大于 5ms，抛出异常或切换备用机器 ID 段，保证 ID 唯一性

**备选方案：号段模式（适合超高并发）**

美团 Leaf 的号段模式，每次从 DB 申请一段 ID（如 1000~2000），内存里自增分发，快耗尽时提前异步申请下一段（双 buffer），QPS 可达百万级，DB 压力极小。

**不推荐 UUID**：虽然唯一，但无序、36位字符，作为 MySQL 主键会造成 B+ 树页分裂，性能很差。

**面试加分点**：能说清楚雪花算法的 bit 结构和时钟回拨处理方案，加分不少。
