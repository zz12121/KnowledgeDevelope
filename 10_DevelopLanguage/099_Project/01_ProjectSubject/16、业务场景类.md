# 业务场景类

---

###### 1. 支付流程是如何实现的？

**答：**

支付流程是核心业务，设计时要重点考虑三个问题：不超付、不多付、不丢单。

**整体流程**：

```
用户发起支付 → 创建支付单（幂等，同一订单只能有一个待支付的支付单）
    → 调用第三方支付（支付宝/微信）获取支付链接
    → 用户完成支付 → 第三方回调我方接口（通知支付结果）
    → 验签 → 幂等判断（防重复回调）→ 更新订单状态 → 触发后续业务
```

**关键设计点**：

**① 支付单幂等**：同一个订单，无论调用几次发起支付，只生成一个支付单（数据库 `order_id` 唯一索引）。

**② 支付回调幂等**：第三方支付回调可能重复，用支付单 `status` 状态机保证：只有 `status=PENDING` 时才处理，`PAID` 状态再来回调直接返回成功。

**③ 主动查单（兜底）**：用户支付后页面卡住，可能没收到回调。定时任务每隔几分钟主动调第三方查单接口（`queryOrder`），确认支付结果，防止"回调丢失导致订单一直待支付"。

**④ 超时取消**：支付单创建 30 分钟未支付，自动关闭支付单并释放库存（RocketMQ 延迟消息）。

**⑤ 退款幂等**：退款接口用 `refund_no`（我方退款单号）作为幂等 Key，防止重复退款。

###### 2. 订单系统如何设计？

**答：**

订单系统是电商的核心，设计时要关注：数据完整性、状态一致性、性能。

**订单数据模型**：

- **orders 主表**：订单基础信息（user_id、status、total_amount、create_time 等），注意 amount 用 `DECIMAL(10,2)` 不用 `DOUBLE`（避免精度丢失）
- **order_items 明细表**：每个商品一行（product_id、quantity、price），冗余商品快照（name、image）——不存商品ID关联查询，因为商品信息可能变更，订单快照必须是下单时的数据
- **order_address 地址表**：冗余收货地址信息，同样是快照，不关联用户地址库

**订单状态机**：

```
待支付 → 已支付 → 已发货 → 已收货 → 已完成
   ↓            ↓
已取消       退款中 → 已退款
```

状态流转用代码枚举 + 校验，防止非法状态变更（比如"已完成"不能变回"待支付"）。

**分库分表**：

订单表按 `user_id` 哈希分 16 张表（后期可以再分库），查询走 user_id，商户侧查询维护额外的 `merchant_order_index` 表做关联。

**超时关单**：RocketMQ 延迟消息（下单时发 30 分钟延迟消息）。

###### 3. 库存系统如何设计？

**答：**

库存系统的核心挑战是：并发安全（防超卖）、高性能（秒杀场景）、数据一致性。

**两阶段库存模型（预占 + 实扣）**：

```
下单时：预占库存（available_stock -= quantity，reserved_stock += quantity）
支付完成：实扣库存（reserved_stock -= quantity，sold_stock += quantity）
取消订单：释放预占（reserved_stock -= quantity，available_stock += quantity）
```

这样避免了用户下单后不付款导致库存被长期占用。

**防超卖方案**：

- **MySQL 乐观锁（常规场景）**：
  ```sql
  UPDATE stock SET available_stock = available_stock - ? 
  WHERE sku_id = ? AND available_stock >= ?
  ```
- **Redis 预扣（大促/秒杀）**：提前把库存加载到 Redis，用 Lua 脚本原子扣减，成功后发 MQ 异步扣 DB。DB 层加乐观锁兜底双保险。

**库存预热**：大促前提前把热门商品库存从 DB 加载到 Redis，避免活动开始瞬间 Redis miss 打 DB。

**超卖兜底**：即使设计很严密，生产环境也要有对账任务（定时比对 Redis 库存和 DB 库存），及时发现不一致。

###### 4. 优惠券系统如何设计？

**答：**

优惠券系统核心是：防超量领取、精确计算优惠金额、防刷。

**数据模型**：

- **coupon_template（优惠券模板）**：类型（满减/折扣/立减）、面值、使用条件、总发行量、已领取量、有效期
- **user_coupon（用户优惠券实例）**：user_id、template_id、status（未使用/已使用/已过期）、使用时间、绑定订单号

**防超量领取（Redis 原子操作）**：

```lua
-- Lua 脚本原子检查 + 扣减发行量
local stock = tonumber(redis.call('get', KEYS[1]))
if stock == nil or stock <= 0 then return -1 end
redis.call('decr', KEYS[1])
return 1
```
Redis 扣减成功后，异步发 MQ 写入 DB，DB 也加 `total_count` 上限兜底。

**防重复领取**：`user_coupon` 表加 `(user_id, template_id)` 唯一索引，同一用户同一模板只能领一张（如果允许多张则去掉）。

**优惠计算**：策略模式，不同优惠类型实现 `DiscountStrategy` 接口，组合使用时按一定规则叠加（满减不与折扣叠加等）。

**防刷**：同一设备/IP 每天领取上限，行为验证码，异常行为（批量账号领取）风控拦截。

###### 5. 积分系统如何设计？

**答：**

积分系统要保证：不超扣、流水可追溯、支持过期（先进先出）。

**核心设计**：

**① 积分账户表**

```sql
CREATE TABLE point_account (
    user_id BIGINT PRIMARY KEY,
    balance BIGINT NOT NULL DEFAULT 0,   -- 可用积分
    frozen BIGINT NOT NULL DEFAULT 0,    -- 冻结积分（待确认）
    version INT NOT NULL DEFAULT 0       -- 乐观锁
);
```

扣减用乐观锁（`WHERE version = ?`），更新失败重试，防止并发超扣：
```sql
UPDATE point_account SET balance = balance - ?, version = version + 1
WHERE user_id = ? AND balance >= ? AND version = ?
```

**② 积分流水表**

每次变动记录流水（type: 消费/充值/过期/退回，amount，order_id，create_time），不可删改，用于对账和客服查询。

**③ 积分过期（先进先出）**

积分不是简单数字，要记录每批积分的过期时间。扣减时按最近要过期的先扣：
- 额外维护 `point_ledger` 表，每次获得积分记一条（amount, expire_time）
- 扣减时按 `expire_time ASC` 查询可用明细，逐条扣减，记录流水

**④ 冻结机制**

用积分支付前，先冻结（available - frozen），支付确认后正式扣减，取消则解冻。防止用积分的同时又在别的订单使用。

###### 6. 推荐系统如何实现？

**答：**

推荐系统的架构是：召回（Recall）→ 过滤 → 粗排 → 精排 → 结果。

**离线召回（每小时/每天刷新）**：

- **协同过滤**：基于用户行为（浏览、购买、收藏）计算用户相似度，相似用户喜欢的我可能也喜欢（User-CF）；或者基于物品相似度（Item-CF）：你买了 A，和买了 A 的人通常也买 B。
- **内容推荐**：根据用户历史偏好标签（价格区间、品类）匹配商品。
- **向量召回**：用双塔模型（用户 Embedding + 商品 Embedding），向量近似搜索（Milvus/Faiss），实时性好。

**在线排序**：

召回候选集通常几百~几千个，在线精排用机器学习模型（LR/GBDT/深度学习）给每个候选打分，考虑 CTR（点击率预估）+ 用户个性化权重。

**Java 服务职责**：

ML 模型通常用 Python 训练，Java 服务负责：召回层查 Redis/Milvus、过滤（已购买/已加购）、调用排序微服务（Python Flask/FastAPI）、结果缓存、多路召回结果融合（RRF 重排序）。

**简单场景**：规则推荐（热销榜 + 品类偏好），不需要 ML 模型，Redis ZSet + 用户标签匹配就够了。

###### 7. 搜索功能如何实现？

**答：**

搜索功能基于 Elasticsearch 实现，核心链路：数据同步 → 索引设计 → 查询实现。

**数据同步（Canal + MQ → ES）**：

Canal 监听 MySQL 商品表 binlog，变更事件发到 Kafka，ES 消费者更新索引。延迟秒级，商品上架/下架/价格变更几秒内体现在搜索结果中。

**索引设计**：

```json
{
  "mappings": {
    "properties": {
      "name": { "type": "text", "analyzer": "ik_max_word" },   // 中文全文搜索
      "brand": { "type": "keyword" },                           // 精确过滤
      "category_id": { "type": "keyword" },
      "price": { "type": "scaled_float", "scaling_factor": 100 },
      "sale_count": { "type": "integer" },
      "suggest": { "type": "completion" }                       // 搜索建议
    }
  }
}
```

**查询实现**：

```java
BoolQueryBuilder query = QueryBuilders.boolQuery()
    .must(QueryBuilders.matchQuery("name", keyword))           // 全文匹配
    .filter(QueryBuilders.termQuery("category_id", categoryId)) // 品类过滤
    .filter(QueryBuilders.rangeQuery("price").gte(minPrice).lte(maxPrice)); // 价格区间
```

聚合：用 ES Bucket 聚合返回品牌、价格区间等可过滤选项（小红书/淘宝左侧筛选栏）。

相关性优化：`function_score` 加入销量、评分权重，提升高质量商品排名。

###### 8. 如何实现实时通知？

**答：**

实时通知场景：订单状态更新、客服消息、系统通知。

**方案一：WebSocket（全双工，双向通信）**

适合实时性要求高、交互频繁的场景（IM、在线客服）：

```java
@ServerEndpoint("/ws/{userId}")
@Component
public class WebSocketServer {
    private static Map<String, Session> sessions = new ConcurrentHashMap<>();

    @OnOpen
    public void onOpen(Session session, @PathParam("userId") String userId) {
        sessions.put(userId, session);
    }

    // 后端主动推送
    public static void sendMessage(String userId, String msg) {
        Session session = sessions.get(userId);
        if (session != null && session.isOpen()) {
            session.getAsyncRemote().sendText(msg);
        }
    }
}
```

多节点部署时，WebSocket 连接在某一台服务器，消息推送需要 Redis Pub/Sub 广播到所有节点：A 节点收到推送消息 → 发布到 Redis Channel → 所有节点订阅并推送给各自的 WebSocket 连接。

**方案二：SSE（Server-Sent Events，服务端单向推送）**

适合只需要服务端推送客户端的场景（订单状态更新、AI 流式输出），比 WebSocket 轻量：

```java
@GetMapping(value = "/notify/{userId}", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter subscribe(@PathVariable String userId) {
    SseEmitter emitter = new SseEmitter(Long.MAX_VALUE);
    sseEmitterMap.put(userId, emitter);
    return emitter;
}
```

**方案三：短轮询（最简单，兜底）**

前端每隔 5~30 秒发一次请求查询有没有新通知，适合实时性要求不高的场景，实现最简单。
