# 数据处理类

###### 1. 大数据量导出如何处理？

**答：**

大数据量导出（比如导出百万行 Excel）如果处理不当，会 OOM、超时、影响主业务。

**方案：异步 + 流式处理 + 分片**

**① 异步导出（推荐）**

用户点击导出后，不同步等待，而是：
1. 后台创建一个导出任务（记录到 DB），立即返回"任务已创建"
2. 消费者异步处理：分批从 DB 查数据（每批 1000 行，用游标分页），流式写入 Excel
3. 导出完成后上传到 OSS，把下载链接发给用户（邮件/站内通知）

这样主线程不阻塞，用户体验好，服务器内存也不会打满。

**② 流式写入 Excel（避免 OOM）**

用 EasyExcel（阿里开源）的流式写法，分批 write，不一次性加载所有数据到内存：
```java
ExcelWriter writer = EasyExcel.write(ossStream, OrderExportVO.class).build();
WriteSheet sheet = EasyExcel.writerSheet(0, "订单数据").build();

int pageSize = 1000;
long lastId = 0;
while (true) {
    List<OrderExportVO> batch = orderMapper.queryExport(lastId, pageSize);
    if (batch.isEmpty()) break;
    writer.write(batch, sheet);
    lastId = batch.get(batch.size() - 1).getId(); // 游标分页，不用 OFFSET
}
writer.finish();
```

**③ 分片并行导出（超大数据量）**

数据量超过千万时，把数据按 ID 范围分成多个分片，多个 Worker 并行导出为多个文件，最后压缩成 ZIP 提供下载。

**防护**：导出任务限速（同一用户同时只能有 1 个导出任务），防止恶意刷大量导出请求。

###### 2. 大文件上传如何实现？

**答：**

大文件（超过 100MB）直接上传到后端服务有很多问题：占用服务器带宽、容易超时、不支持断点续传。

**方案：客户端直传 OSS + 分片上传**

**① 客户端直传（推荐）**

前端直接上传到 OSS（阿里云 OSS/腾讯云 COS），后端不做中转，节省带宽：
1. 前端向后端申请 OSS 临时上传凭证（STS Token）
2. 前端用凭证直接上传到 OSS 的指定目录
3. 上传成功后，前端把 OSS 文件 Key 发给后端，后端记录文件元数据

**② 分片上传 + 断点续传**

大文件分割成 5~20MB 的分片，逐片上传，支持断点续传（失败重传失败的分片）：

```javascript
// 前端分片逻辑（伪代码）
const CHUNK_SIZE = 10 * 1024 * 1024; // 10MB
const chunks = Math.ceil(file.size / CHUNK_SIZE);
const uploadId = await initMultipartUpload(); // 初始化分片上传

// 并行上传分片（限制并发数，如3个同时上传）
const uploadPromises = Array.from({length: chunks}, (_, i) => {
    const start = i * CHUNK_SIZE;
    const chunk = file.slice(start, start + CHUNK_SIZE);
    return uploadPart(uploadId, i + 1, chunk);
});
await Promise.all(uploadPromises);

// 合并分片
await completeMultipartUpload(uploadId);
```

**③ 秒传（MD5 秒传）**

上传前计算文件 MD5，向后端查询该 MD5 是否已存在，存在则直接返回已有文件的 URL，无需重传（文件相同内容共享存储）。

###### 3. 如何实现数据迁移？

**答：**

数据迁移是一个高风险操作，核心要求：数据完整、不停业务、可验证、可回滚。

**分阶段迁移策略**：

**阶段一：双写（准备期）**

代码改造：写操作同时写新库和旧库，读操作走旧库。确保新库有全量增量数据。

**阶段二：全量迁移**

把旧库存量数据迁移到新库：
- 用 `INSERT INTO new_table SELECT ... FROM old_table WHERE id BETWEEN ? AND ?` 分批迁移（每批 1000~5000 行，避免大事务）
- 或者用数据迁移工具（AWS DMS、阿里云 DTS、Canal）

**阶段三：验证**

对比新旧库数据的一致性：抽样比对关键字段，或者全量 COUNT + 关键业务指标（订单总金额等）对比。

**阶段四：切换读**

读操作切换到新库（先切部分，灰度验证）。

**阶段五：关闭双写**

验证无误后，停止写旧库，完成迁移。

**注意事项**：
- 迁移期间有数据变更，要用 Canal/binlog 捕获增量，追加到新库
- 迁移完成后保留旧库一段时间（1~4周），方便回滚

###### 4. 历史数据如何归档？

**答：**

历史数据归档的目的是：主库只保留热数据（近 6 个月~1 年），老数据迁移到低成本存储，保持主库表行数在可控范围内，索引效率不降级。

**归档流程**：

**① 确定归档策略**

按时间：`create_time < NOW() - INTERVAL 1 YEAR` 的数据为冷数据。
按状态：已完结的订单（status = 'CLOSED'）且超过 6 个月。

**② 使用 pt-archiver 在线归档**

```bash
pt-archiver \
  --source h=主库host,D=db,t=orders \
  --dest h=归档库host,D=archive_db,t=orders_archive \
  --where "status='CLOSED' AND create_time < '2025-01-01'" \
  --limit 1000 \       # 每批 1000 行
  --sleep 0.01 \       # 每批之间休眠 10ms，减少主库压力
  --no-check-charset \
  --purge              # 归档后删除源数据
```
`pt-archiver` 分批处理，对主库影响极小，支持并发控制。

**③ 归档库的查询**

归档库可以是：
- 独立 MySQL 实例（支持 SQL 查询，但成本高）
- TiDB（HTAP，同时支持 TP 和 AP，适合既需要查询也需要分析的场景）
- 导出到 OSS 存 Parquet 格式，用 Presto/Athena 查询（成本最低）

**④ 归档校验**

归档完成后，对比源表和目标表的行数、关键字段 SUM，确保数据无丢失后再清理源数据。

###### 5. 如何实现数据同步？

**答：**

数据同步场景很多：MySQL → ES（搜索场景）、MySQL → Redis（缓存场景）、跨数据库同步、实时报表同步。

**方案一：Canal + MQ（推荐，实时性好）**

Canal 监听 MySQL binlog，把数据变更（INSERT/UPDATE/DELETE）发到 MQ，消费者处理变更事件同步到目标。

优点：解耦、无侵入业务代码、实时性好（秒级）、可靠（MQ 保证消息不丢）。
适合：MySQL → ES（搜索索引同步）、MySQL → Redis（缓存失效通知）。

**方案二：定时轮询（简单场景）**

定时任务扫描 `update_time > 上次同步时间` 的记录，同步到目标。
缺点：有延迟（同步频率决定延迟），删除操作无法感知（数据已删，轮询查不到），只适合允许延迟、无删除的场景。

**方案三：双写（强一致场景）**

业务代码写 DB 的同时也写目标（如 Redis），用事务或本地消息表保证两边一致。
缺点：侵入业务代码，维护成本高。

**方案四：Flink CDC（全量 + 增量一体化）**

Flink CDC 可以先全量同步存量数据，再无缝切换到增量 binlog 同步，适合数据仓库、大数据场景。

###### 6. 报表系统如何设计？

**答：**

报表系统的核心挑战是：查询复杂（多表关联、大数据量聚合）、实时性要求各异（有的需要实时，有的 T+1 够了）。

**架构分层**：

**① 在线实时报表（近实时，延迟 < 1分钟）**

适合简单指标，直接查 Redis（ZSet 计数、HyperLogLog UV 估算）或预聚合表（每分钟更新一次的汇总表）。

**② T+1 离线报表（次日刷新）**

业务数据晚上通过 ETL 同步到数仓（ClickHouse 或大数据平台），报表查数仓。ClickHouse 列存储，亿级数据聚合查询秒出，适合运营报表、财务对账。

**③ 报表查询优化**

- 预计算：高频查询结果提前计算，存中间结果表
- 索引：报表查询的过滤字段（时间、渠道、区域）建复合索引
- 分区：ClickHouse 按日期分区，查询时 partition pruning，不全表扫描
- 缓存：相同参数的报表结果 Redis 缓存 5~30 分钟

**④ 导出支持**

参考大数据量导出方案：异步任务 + EasyExcel 流式写 + OSS 存储 + 用户下载链接通知。
