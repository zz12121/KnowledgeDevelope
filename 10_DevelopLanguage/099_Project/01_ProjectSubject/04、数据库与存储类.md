# 数据库与存储类

---

###### 1. 项目中数据最多的一张表是什么表？多大？

**答：**

这道题是让你结合项目实际说，不要直接说"不知道"或者"我们没统计过"，要给出具体数字和处理思路。

**参考口径（根据项目类型选一个）：**

**电商类项目**：数据量最大的通常是订单表或订单明细表。以日活 100 万的中型电商为例，日均订单量约 50 万，三年累计约 5 亿行，单表约 300~500GB。订单明细表因为每个订单有多个商品，数据量是订单表的 3~5 倍。

**内容类项目**：消息记录表、浏览记录表。日活 50 万的 IM 系统，日均消息量约 2000 万，一年约 70 亿行。

**说完数量，要接着说处理方案**，这才是面试官真正想听的：
- 超过 5000 万行：分区表（按时间范围分区）
- 超过 1 亿行：分库分表（ShardingSphere，按 user_id 哈希）
- 历史数据：定期归档到冷库（OSS + Presto 查询，或者单独的历史库）
- 归档后主库只保留近半年数据，保持表在 3000 万行以内，索引效率高

###### 2. 数据库如何进行分库分表？

**答：**

我们用 Apache ShardingSphere（shardingsphere-jdbc）做分库分表，对业务代码无侵入，只需配置。

**什么时候分库分表**：单表超过 5000 万行、MySQL 单机 IOPS 达到上限、主从复制延迟严重，这时候才考虑分。不要过早优化，分库分表引入的复杂度（跨库事务、分页、排序、聚合）相当大。

**分片策略**：
- **垂直分库**：按业务模块拆库，比如用户库、订单库、商品库独立部署，减少单库连接数，也提高故障隔离性
- **水平分表**：大表按某个字段（通常是 user_id 或 order_id）哈希分片，分到多张表，每张表结构相同但数据不同

**分片键选择**：选择查询频率最高的字段，且数据分布要均匀。订单表通常按 user_id 分片（用户查自己的订单用的最多）；但商户侧需要查某个商户的所有订单，这时可以额外维护一张关系映射表或者建宽表冗余。

**遇到的坑**：
- 跨分片分页：`LIMIT 10 OFFSET 100` 在分库分表场景下性能极差，要改成游标分页（基于 ID 范围）
- 跨库 JOIN：禁止，改成应用层聚合或者引入冗余字段
- 分布式事务：配合 Seata 或本地消息表处理

###### 3. 如何保证数据库的高可用？

**答：**

数据库高可用的核心是：无单点、快速切换、数据不丢失。

**① 主从复制 + 读写分离**

MySQL 配置一主两从（半同步复制），主库写，从库读。半同步复制保证主库提交事务时至少一个从库收到了 binlog，主库宕机时数据不丢失。ShardingSphere 或 MyBatis-Plus 的读写分离插件自动路由读请求到从库。

**② 自动主从切换（MHA / 云 RDS）**

生产环境我们用云厂商的 RDS（如腾讯云 CDB 或阿里云 RDS），主库故障时自动切换到备库，RTO（恢复时间目标）在 30 秒以内。

自建 MySQL 可以用 MHA（Master High Availability Manager），检测到主库不可用时自动提升从库为主库，修改 VIP 漂移，应用层感知最小。

**③ 数据备份**

- 每天全量备份（mysqldump 或 xtrabackup），备份文件上传 OSS
- binlog 实时同步到备份机，支持 Point-in-Time Recovery（时间点恢复）
- 定期演练恢复，确保备份可用

**④ 连接池配置**

HikariCP 配置 `connectionTimeout`、`idleTimeout`，防止连接耗尽。主库切换后，旧连接会报错，HikariCP 的健康检查会自动剔除无效连接并重连。

###### 4. 数据库索引如何设计和优化？

**答：**

索引设计遵循几个原则，结合实际项目经验来说：

**① 建索引的原则**

- **高选择性字段优先**：user_id、order_id 这种取值离散的字段，索引效果好；status 这种只有几个值的字段，单独建索引意义不大（除非结合其他字段做复合索引）
- **复合索引遵循最左前缀**：把查询频率高、过滤效果强的字段放左边。比如订单表经常 `WHERE user_id = ? AND create_time > ?`，建 `(user_id, create_time)` 复合索引
- **覆盖索引减少回表**：SELECT 的字段能被索引覆盖时，不需要回表查数据，效率大幅提升
- **索引不宜太多**：每个额外索引都增加写入开销（更新时要维护 B+ 树），建议单表不超过 5 个索引

**② 实际优化经验**

在我们项目里，遇到一个慢查询：
```sql
SELECT * FROM orders WHERE create_time > '2024-01-01' AND status = 1 ORDER BY create_time DESC
```
原来只有 `(create_time)` 索引，explain 看到 `Using filesort`，QPS 一高就慢。优化方案：
1. 改成 `(status, create_time)` 复合索引，status 放左边（高频过滤条件）
2. SELECT 改成只查需要的字段，让索引覆盖，消除回表
3. 优化后 TP99 从 200ms 降到 20ms

**③ 定期维护**

定期检查未使用的索引（`sys.schema_unused_indexes`），清理冗余索引；大表重建索引用 `pt-online-schema-change` 在线操作，不影响生产。

###### 5. 遇到过慢 SQL 吗？如何优化的？

**答：**

遇到过，而且不止一次。说一个最典型的案例（STAR 格式）：

**背景**：商品搜索接口在大促期间 TP99 突然飙升到 3 秒，正常时候只有 100ms。

**排查过程**：
1. 先看 Skywalking 链路追踪，定位到是 MySQL 查询慢，具体 SQL 是按分类 + 价格范围 + 销量排序的商品列表查询
2. 用 `EXPLAIN` 看执行计划，发现 `type=ALL`（全表扫描），`rows=200万`，原因是查询条件涉及多列，现有索引没覆盖
3. 用 `SHOW PROFILE` 确认瓶颈在 `Sending data`（扫描行数太多）

**优化方案**：
1. 重建索引：新建 `(category_id, price, sale_count)` 复合索引，满足最左前缀
2. 深分页优化：原来的 `LIMIT 100000, 20` 改成子查询覆盖索引 + 回表的方式
3. 引入 ES：商品搜索天然适合 ES，把查询迁移到 ES，MySQL 只做数据源，Canal 同步数据

**效果**：迁移到 ES 后 TP99 降到 30ms，大促期间稳定，没有再出现超时。

**常见慢 SQL 原因**：索引失效（隐式类型转换、函数操作索引列）、深分页、JOIN 没走索引、SELECT *、大事务持锁时间长。

###### 6. 如何处理数据库主从延迟问题？

**答：**

主从延迟（Replication Lag）是读写分离架构绕不开的问题，刚写完的数据去从库读可能还没同步过来。

**产生原因**：MySQL 主从是异步复制（普通异步模式），主库提交事务后，binlog 异步发给从库，从库 IO 线程接收，SQL 线程回放，这中间有延迟，大事务、DDL 操作尤其明显。

**处理方案**：

**① 写后读一致性（强制走主库）**

对于"写完立刻读"的场景，直接路由到主库读。ShardingSphere 支持 `HintManager.setMasterRouteOnly()` 强制走主库，或者用 `@DS("master")` 注解。

**② 延迟检测后读主库**

从库提供延迟时间接口（`SHOW SLAVE STATUS` 中的 `Seconds_Behind_Master`），延迟超过阈值时自动降级到主库读。

**③ 接受最终一致性**

非强一致性场景（比如展示类数据、排行榜），允许短暂延迟，不处理。用户修改完个人信息后，列表页可能 1 秒后才更新，大多数用户是接受的。

**④ 减少延迟本身**

- 避免大事务（拆分成小事务）
- 避免高峰期 DDL（用 pt-osc 在线操作）
- 增大从库 `innodb_buffer_pool_size`，提高回放速度
- 云 RDS 的半同步复制 + 并行回放，延迟通常在 ms 级

###### 7. 项目中有没有用到 NoSQL？为什么用？

**答：**

用到了，主要用了 Redis，部分场景用了 ES。

**Redis 的使用场景**：

- **缓存**：商品详情、用户信息等热点数据缓存，减少 MySQL 压力，响应时间从 20ms 降到 2ms
- **分布式锁**：秒杀、抢红包、防重复提交场景，用 Redisson 实现可重入分布式锁
- **计数器/排行榜**：用 ZSet 实现实时销量排行榜，ZINCRBY 原子更新，ZREVRANGE 查 TopN
- **会话存储**：分布式 Session，用 Spring Session + Redis，多实例间 Session 共享
- **限流**：基于 Redis 令牌桶/滑动窗口实现接口限流

**为什么选 Redis 而不是只用 MySQL**：Redis 是内存数据库，读写延迟在微秒级，MySQL 再快也要毫秒级；Redis 的数据结构（ZSet、Bitmap、HyperLogLog）天然适合某些场景，用 MySQL 实现既麻烦性能又差。

**ES 的使用场景**：商品搜索，支持全文检索、拼音搜索、聚合过滤（按价格区间、品牌过滤），这些用 MySQL LIKE 实现不了，用 ES 的倒排索引效率极高。

###### 8. 如何保证数据一致性？

**答：**

数据一致性在分布式环境下是个难题，根据不同场景有不同的保证方式：

**① 单服务内：数据库事务 + 锁**

同一个服务内，Spring `@Transactional` 保证同一数据库内的多操作原子性。并发更新用悲观锁（`SELECT FOR UPDATE`）或乐观锁（`version` 字段 + `WHERE version = ?`）防止覆盖写。

**② 跨服务：最终一致性**

跨服务场景绝大多数不需要强一致性，用最终一致性更合适：
- 本地消息表：业务操作和消息记录同库同事务，异步发 MQ，消费方幂等处理
- RocketMQ 事务消息：半消息确认后投递，生产者本地事务失败则 Rollback

**③ 缓存与数据库一致性**

Cache Aside 模式：写操作先更新 DB，再删 Redis 缓存（不是更新缓存，避免并发写竞态）。延迟双删：`删缓存 → 更新DB → 延迟500ms再删缓存`，防止短暂读到旧缓存。Canal + binlog 监听 DB 变更，异步更新缓存，彻底解耦。

**④ 对账兜底**

再健壮的系统也可能出现数据不一致（网络超时、服务宕机）。我们每天跑对账任务，比对订单表和支付流水、库存流水是否一致，发现差异自动告警 + 人工介入处理。
