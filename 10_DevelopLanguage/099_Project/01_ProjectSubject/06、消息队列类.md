# 消息队列类

---

###### 1. 项目中为什么使用消息队列？

**答：**

引入 MQ 主要解决三类问题：异步解耦、削峰填谷、可靠通信。

**① 异步解耦**

下单成功后，需要通知库存扣减、积分增加、发短信通知、更新用户画像等多个后续操作。如果同步调用，主流程要等所有操作完成才返回，延迟极高，且任何一个下游服务故障都会影响下单主流程。

用 MQ 后，下单服务只负责写 DB、发消息，其他服务异步消费，主流程响应时间从 500ms 降到 50ms，服务间彻底解耦。

**② 削峰填谷**

秒杀、大促场景，瞬间流量是平时的几十倍。如果请求直接打数据库，分分钟被打垮。用 MQ 做缓冲，生产者按实际流量发消息，消费者按 DB 能承受的速率平稳消费，把洪峰流量"削"成平稳的处理流。

**③ 可靠通信**

跨服务调用如果对方暂时不可用（维护、重启），同步 RPC 直接报错；MQ 可以把消息存起来，等对方恢复后继续消费，天然支持重试，可靠性更高。

**什么时候不用 MQ**：如果调用方需要同步等待结果（比如查用户余额），或者操作链路非常短、延迟要求极低（<10ms），用 Feign 直接调即可，引入 MQ 反而增加复杂度和延迟。

###### 2. 如何保证消息不丢失？

**答：**

消息从生产到消费要经过三段：生产者→Broker→消费者，每段都可能丢，每段都要有保障。

**生产者端**

- RabbitMQ：开启 Publisher Confirm + Publisher Return，confirm 收到 ACK 后才算发送成功，NACK 时重发
- RocketMQ：同步发送（`send()` 而非 `sendOneway()`），收到 SendResult 确认 Broker 持久化成功
- Kafka：`acks=all`，所有 ISR 副本确认后才算成功

**Broker 端**

- 消息持久化：队列/Topic 设置持久化，消息写入磁盘
- 高可用：RabbitMQ Quorum Queue（Raft多数派确认）；Kafka 多副本+`min.insync.replicas=2`；RocketMQ 同步刷盘（`SYNC_FLUSH`）

**消费者端**

- 关闭自动 ACK，改手动 ACK：消费完成后再 ACK，处理失败时 NACK 或不 ACK，Broker 会重新投递
- 幂等消费：消息可能被重复投递（网络超时 + 重试），消费者必须保证幂等（去重 key + 状态检查）

**补偿兜底**

生产者发送前先写本地消息表（与业务操作同一个事务），异步定时任务扫描未确认消息并重发。

###### 3. 如何保证消息不被重复消费？

**答：**

MQ 的"至少一次"语义决定了消息可能重复投递，所以消费端必须自己实现幂等。

**核心思路：给消息打上唯一业务标识，消费前判断是否已消费过。**

**方案一：Redis SET NX 去重**

消费前用消息的唯一 ID（业务ID 或 MQ 消息 ID）做 Redis `SET NX`，设置成功说明是第一次消费，失败说明重复直接丢弃：
```java
String key = "consumed:msg:" + message.getMsgId();
Boolean isFirst = redisTemplate.opsForValue().setIfAbsent(key, "1", 1, TimeUnit.DAYS);
if (!isFirst) return; // 已消费，幂等处理
// 正常业务逻辑...
```

**方案二：数据库唯一索引**

消费时向消费记录表插入 `(msg_id, consumer_id)` 唯一索引记录，插入失败（重复键）说明已消费过，捕获异常按成功处理。

**方案三：状态机**

消息触发的操作有明确状态流转（如订单状态机），消费时带 `WHERE status = 'PENDING'` 条件更新，受影响行为 0 说明已处理过，直接返回成功。

**注意**：去重 key 的 TTL 要根据消息可能重复的最长时间设置，通常 1~7 天，超过这个时间的重复已经不太可能发生。

###### 4. 如何保证消息的顺序性？

**答：**

顺序消息分两种：全局顺序（所有消息有序）和局部顺序（同一个业务实体的消息有序）。全局顺序吞吐量极低，几乎不用；实际中只需要局部顺序，比如同一个订单的创建→付款→发货消息必须有序。

**Kafka 保证局部顺序**：
- 同一个 Partition 内消息有序，把需要有序的消息发到同一个 Partition（按 key 哈希，如 `orderId`）
- 消费时单线程消费同一个 Partition，保证消费顺序
- 注意：消费者手动处理异常重试时要特别小心，避免异步重试打乱顺序

**RocketMQ 保证局部顺序**：
- 发送时用 `MessageQueueSelector` 把同一个 orderId 的消息发到同一个 Queue
- 消费时用 `MessageListenerOrderly`，RocketMQ 对同一个 Queue 加分布式锁，单线程顺序消费

**RabbitMQ 保证局部顺序**：
- 一个 Queue 只能有一个消费者（`prefetch=1` + 单消费者），或者按业务路由到不同 Queue

**消费端注意事项**：
- 消费失败重试时不要并发重试，否则会乱序
- 幂等处理：消息可能重发，要能识别重复并跳过

###### 5. 消息队列如何选型？为什么选择当前的 MQ？

**答：**

我们项目用的是 RocketMQ（如果是其他 MQ 根据实际替换）。选型时主要对比了 RabbitMQ、RocketMQ、Kafka，从以下维度评估：

| 维度 | RabbitMQ | RocketMQ | Kafka |
|---|---|---|---|
| 吞吐量 | 万级 | 十万级 | 百万级 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 延迟消息 | 插件支持 | 原生支持 | 不支持 |
| 事务消息 | 不支持 | 原生支持 | 支持 |
| 顺序消息 | 一定程度支持 | 原生支持 | Partition内有序 |
| 生态 | 广泛 | Spring Cloud Alibaba | 大数据生态 |

**选 RocketMQ 的理由**：
1. 我们是 Spring Cloud Alibaba 技术栈，RocketMQ 集成最简单
2. 需要延迟消息（订单超时取消）和事务消息（分布式事务），RocketMQ 原生支持
3. 吞吐量十万级完全满足我们的业务规模
4. 阿里多年双十一验证，稳定可靠

**如果是 Kafka**：适合日志收集、大数据流处理场景，吞吐量最高，但不适合复杂消息特性（延迟消息、事务消息、死信队列）。

###### 6. 如何处理消息积压？

**答：**

消息积压说明消费速度跟不上生产速度，处理思路：先止血（扩消费能力），再排查根因。

**紧急处理**：

**① 扩消费者实例**

Kubernetes 环境直接扩容消费者 Pod 数量，消费者组自动 Rebalance，分摊 Partition/Queue。这是最快的止血手段。

**② 扩 Partition/Queue**

如果消费者数量已经多于 Partition 数，多余的消费者是空闲的。此时需要扩 Partition（Kafka `kafka-topics.sh --alter --partitions`），然后再扩消费者。注意：扩 Partition 会触发 Rebalance，有短暂停顿。

**③ 批量消费 + 并行处理**

消费者内部把消息批量拉取后，用线程池并行处理，提高单个消费者实例的吞吐量：
```java
@KafkaListener(topics = "order", batch = "true")
public void consume(List<ConsumerRecord<String, String>> records) {
    records.parallelStream().forEach(record -> process(record.value()));
}
```

**根因排查**：

- 消费者代码有慢操作（调第三方接口、慢 SQL）？→ 异步化、优化慢操作
- 消费者频繁异常 → 检查报错日志，修复后重新消费
- 生产者突发流量 → 上游限流，避免下次积压

**极端情况**（积压消息已过期或无法快速消费）：临时把消息导到新 Topic，由更多消费者快速处理，不影响正常消息流。

###### 7. 死信队列如何处理？

**答：**

死信队列（Dead Letter Queue，DLQ）是消息处理失败后的兜底机制，通常在以下情况消息会进死信队列：
- 消费者重试次数达到上限仍失败（如 RabbitMQ 默认重试次数耗尽）
- 消息过期（TTL 超时未消费）
- 队列长度超限被丢弃

**死信消息的处理策略**：

**① 告警 + 人工介入**

死信队列里的消息都是"问题消息"，要接入监控告警：死信队列消息数 > 0 时立即告警，通知相关开发处理。

**② 自动重试（有条件）**

如果失败原因是临时性的（网络抖动、下游服务暂时不可用），可以有一个定时任务定期把死信队列里的消息重新放回原队列消费。注意：消费者必须保证幂等，避免重复消费造成业务问题。

**③ 记录到 DB + 手动处理**

把死信消息入库，记录完整的消息体、失败原因、失败时间，运营或开发可以从后台查看，手动触发重处理或者标记为忽略。

**④ 根因修复**

死信是症状，根本是消费者代码有 Bug 或下游依赖不稳定，要修复根因，不能只靠重试掩盖问题。

**配置示例（RabbitMQ）**：
```yaml
# 开启死信队列
spring.rabbitmq.listener.simple.acknowledge-mode: manual
# 声明 DLQ 时绑定到原队列
x-dead-letter-exchange: dlx.exchange
x-dead-letter-routing-key: dlx.key
x-message-ttl: 60000  # 60秒未消费进死信
```
