# 缓存相关类

---

###### 1. 让你来设计一个分布式缓存系统，你会如何设计？

**答：**

设计一个分布式缓存系统，要考虑以下几个核心维度：

**① 整体架构：多级缓存**

```
客户端请求 → 本地缓存（Caffeine，L1）→ 分布式缓存（Redis，L2）→ 数据库（L3）
```

L1 本地缓存（Caffeine）：容量小（通常几百~几千条），不走网络，延迟在微秒级，适合热点数据、字典数据、配置数据。

L2 分布式缓存（Redis Cluster）：容量大，所有实例共享，一致性好，延迟在毫秒级。

**② Redis 集群方案**

生产环境用 Redis Cluster（3主3从），每个主节点负责一段哈希槽（16384个槽，每个主节点约5461个），天然水平扩展。客户端用 Lettuce（Spring Data Redis 默认），支持异步、响应式，性能好。

**③ 缓存设计细节**

- **Key 设计**：`业务名:实体名:ID`，如 `product:detail:12345`，有前缀区分业务，TTL 设置随机抖动（如 30分钟 ± 5分钟）避免大量 Key 同时失效
- **Value 序列化**：用 JSON（可读性好，方便排查），或 Protobuf（高性能场景）
- **容量规划**：根据热点数据量、命中率目标（期望 99%+）计算内存需求

**④ 可靠性**

- Redis 持久化：AOF + RDB 混合持久化
- 主从复制：每个主节点一个从节点，主节点宕机自动 failover
- 本地缓存降级：Redis 不可用时，部分场景可以直接从 L1 返回旧数据，或者降级走 DB

**⑤ 一致性维护**

Cache Aside 模式：写操作更新 DB 后删缓存；Canal 监听 binlog 异步更新缓存（解耦方案）。

###### 2. 项目中如何使用缓存？缓存策略是什么？

**答：**

缓存策略的选择要根据数据的读写频率和一致性要求来决定，我们项目里用了以下几种：

**① Cache Aside（旁路缓存）—— 最常用**

读：先读缓存，miss 再读 DB，把数据写回缓存。
写：先写 DB，再删缓存（不是更新）。

删而不是更新的原因：更新缓存在并发写场景下容易出现竞态条件（A写DB完、B写DB完、B更新缓存、A更新缓存 → 缓存里是A的旧值但DB里是B的新值）；删缓存后下次读自然加载最新值。

**② Read Through / Write Through（透写）**

由缓存组件自动管理 DB 读写，对业务透明。适合用 ORM 自带缓存（如 MyBatis 二级缓存）的场景，但灵活性较差，我们用得少。

**③ Write Behind（异步写）**

写请求只更新缓存，异步批量刷到 DB。适合写多读多且允许短暂不一致的场景（如计数器、访问量统计），性能最好但有数据丢失风险。

**我们项目的具体实践**：

- 商品详情、分类树：Cache Aside，TTL 30分钟，变更时主动删缓存
- 用户 Session：Write Through，Spring Session + Redis，TTL 2小时
- 活动配置、字典数据：本地缓存（Caffeine），Nacos 配置变更时广播刷新
- 实时排行榜：Redis ZSet 直接当存储，不再经过 DB

###### 3. 如何解决缓存穿透、缓存击穿、缓存雪崩？

**答：**

这三个是缓存经典三问，分清楚定义再说解法：

**缓存穿透**：查询不存在的 Key，每次都打到 DB。

解法：
- 布隆过滤器（BloomFilter）：启动时把所有合法 ID 加载到 BloomFilter，请求进来先判断，不在里面直接拒绝
- 空值缓存：DB 查不到也缓存一个空值（TTL 设短一点，如 60 秒），防止同一个不存在的 Key 频繁打 DB
- 接口层参数校验：ID 格式不合法直接拦截，连 Redis 都不查

**缓存击穿**：热点 Key 突然过期，大量并发请求同时涌入 DB。

解法：
- 互斥锁（Mutex Lock）：缓存 miss 时只让一个请求去加载 DB，其他请求等待。用 Redis `SET NX` 实现分布式锁，加载完后释放锁，其他请求重新读缓存
- 逻辑过期（不设 TTL）：Key 永不过期，Value 里存一个逻辑过期时间，读到过期数据时异步更新缓存，主流程直接返回旧数据（牺牲一点一致性，换取不击穿）

**缓存雪崩**：大量 Key 同时过期，或 Redis 实例挂掉，导致流量涌入 DB。

解法：
- TTL 加随机抖动：`TTL = 基础时间 + random(0, 5分钟)`，避免集中过期
- Redis 高可用：主从 + 哨兵或 Redis Cluster，主节点挂了自动切换从节点
- 本地缓存兜底：Redis 不可用时，L1 本地缓存继续提供服务
- 熔断降级：Redis 连接超时超过阈值，触发 Sentinel 熔断，直接返回默认值或降级数据

###### 4. 缓存和数据库的数据一致性如何保证？

**答：**

缓存一致性是个经典难题，完美方案不存在，核心是"在可接受的一致性窗口内做到最终一致"。

**常规方案：Cache Aside + 删缓存**

写 DB 成功后立即删缓存，下次读时从 DB 加载最新值。大多数场景这就够了，不一致窗口极短（毫秒级）。

**进阶：延迟双删**

防止"先删缓存、后写DB"方案中的并发读写竞态：
```
删缓存 → 写 DB → 延迟 500ms → 再删一次缓存
```
延迟时间要大于一次读操作 + 写缓存的时间，保证在 DB 更新期间进缓存的旧值能被清掉。但这方案稍显"workaround"味道，延迟时间不好把握。

**更强方案：Canal + binlog 异步更新**

Canal 监听 MySQL binlog，捕获数据变更后发 MQ，消费者更新或删除对应缓存。优点：彻底解耦，不依赖业务代码主动删缓存，即使代码忘了删，Canal 也能兜底。缺点：引入额外组件，有一定延迟（通常百毫秒级）。

**不一致的兜底**：设置合理的 TTL（如 30 分钟），超时自然失效，保证最终一致。

**原则**：根据业务容忍度选方案，普通查询接口用 Cache Aside 就好，核心数据（账户余额、库存）写操作之后强制走 DB，不读缓存。

###### 5. Redis 的数据结构在项目中如何应用？

**答：**

Redis 五种基础数据结构在我们项目里都有用到，每种有特定场景：

**String**：
- 缓存：商品详情 JSON 序列化后存 String，`GET/SET` 操作简单高效
- 计数：`INCR` 原子自增，用于请求计数、限流计数
- 分布式锁：`SET key value NX EX 30`

**Hash**：
- 用户 Session：`HSET session:userId field value`，一个 Hash 存用户的多个属性，方便单字段更新
- 购物车：`HSET cart:userId productId quantity`，`HINCRBY` 修改数量，`HGETALL` 获取全部

**List**：
- 消息队列（简单场景）：`LPUSH` 生产，`BRPOP` 阻塞消费
- 最新消息列表：`LPUSH` + `LTRIM` 保持固定长度

**Set**：
- 共同好友、共同关注：`SINTER` 取交集
- 标签去重：用户标签存 Set，天然去重

**ZSet（Sorted Set）**：
- 排行榜：商品销量榜、用户积分榜，`ZINCRBY` 更新分数，`ZREVRANGE` 查 Top N
- 延迟队列：score 存执行时间戳，`ZRANGEBYSCORE 0 now` 扫描到期任务
- 限流：滑动窗口，score 存请求时间戳，`ZREMRANGEBYSCORE` 移除窗口外的请求

**高级数据结构**：
- **Bitmap**：签到记录（按位存储，`SETBIT userId dayOffset 1`），`BITCOUNT` 统计签到天数
- **HyperLogLog**：UV 统计，`PFADD` 添加，`PFCOUNT` 估算（误差 0.81%），内存只用 12KB
- **GEO**：附近的人/商户，底层是 ZSet，`GEOASD` 添加坐标，`GEODIST` 计算距离，`GEORADIUS` 查范围内

###### 6. 如何设置缓存过期时间？

**答：**

缓存 TTL 设置没有统一答案，要根据数据特性来定：

**按数据变更频率分类**：

- **几乎不变的数据**（地区字典、分类树）：TTL 设 1天~7天，或者永不过期配合主动失效（变更时主动删）
- **低频变更数据**（商品详情、用户信息）：TTL 30分钟~2小时，更新时删缓存
- **高频变更数据**（库存、价格）：TTL 5分钟以内，或者不缓存，直接读 DB + 读写分离
- **实时数据**（余额、积分）：不缓存，或者只在当前请求内缓存（请求级别的缓存）

**避免集中过期（缓存雪崩）**：

不要给同类 Key 设置一样的 TTL，加随机抖动：
```java
int ttl = 30 * 60 + new Random().nextInt(5 * 60); // 30分钟 ± 5分钟
redisTemplate.expire(key, ttl, TimeUnit.SECONDS);
```

**不同业务区分 TTL**：

我们在缓存 Key 的命名空间里隐含了 TTL 策略，比如 `hot:product:xxx` 用短 TTL，`config:dict:xxx` 用长 TTL，代码里统一管理。

**不要设过长的 TTL**：TTL 越长，不一致窗口越大。除非是纯配置型数据，否则不建议设置超过 24 小时。

###### 7. 热点数据如何处理？

**答：**

热点数据（热 Key）是缓存的一大挑战：某个 Key 被高频访问，所有请求打到 Redis 同一个节点，甚至单个 Redis 节点 CPU 打满。

**识别热点 Key**：

- Redis `--hotkeys` 参数（Redis 4.0+ 支持，需要 `maxmemory-policy` 设为 LFU）
- 监控 `MONITOR` 命令（生产谨慎使用，影响性能）
- Proxy 层统计（如 Twemproxy、Codis 的统计）
- 业务埋点：对敏感接口在应用层统计 Key 访问频次

**处理方案**：

**① 本地缓存（L1）兜底热点**

热点 Key 同时在本地缓存（Caffeine）存一份，TTL 设短（5~30秒），绝大多数请求在本地缓存命中，不走 Redis 网络。变更时通过 MQ 或 Redis Pub/Sub 广播通知各实例清本地缓存。

**② 热点 Key 分散（Key 拆分）**

把一个热 Key 复制成多份：`hot_product_12345_0`、`hot_product_12345_1`、...、`hot_product_12345_9`，读时随机选一个，写时全部更新。10个副本，流量分散到 10 个节点。

**③ Redis Cluster 避免数据倾斜**

合理分配哈希槽，确保热点数据不都落在同一个节点；用 `{tag}` 控制 Key 的哈希槽分配。

**④ 大促预热**

大促前把热点商品、活动配置提前加载到本地缓存和 Redis，并通知各实例预热，避免冷启动时击穿 DB。

**面试加分点**：本地缓存是处理热 Key 最有效的手段，比 Redis 集群扩容快得多，而且延迟更低（微秒 vs 毫秒）。
