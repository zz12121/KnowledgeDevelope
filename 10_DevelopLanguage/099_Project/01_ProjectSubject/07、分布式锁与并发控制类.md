# 分布式锁与并发控制类

---

###### 1. 假设一个外卖单要发布，现在有 200 名骑手都想接这单，如何保证只有一个骑手接到此单？

**答：**

这是一个典型的并发抢占问题，本质是保证同一时间只有一个请求能"抢单成功"。

**方案一：数据库乐观锁（推荐，简单可靠）**

订单表加一个 `status` 字段（0=待接单，1=已接单）和 `version` 字段，接单时：
```sql
UPDATE orders 
SET status = 1, driver_id = ?, version = version + 1
WHERE order_id = ? AND status = 0 AND version = ?
```
200 个骑手并发执行，数据库的行锁保证只有一个能更新成功（受影响行数 = 1），其他人受影响行数为 0，返回"已被接单"。

这个方案最简单，不引入额外组件，利用数据库本身的并发控制能力。

**方案二：Redis 分布式锁（高并发场景）**

如果并发量极高（比如 10000 人同时抢），数据库锁会成为瓶颈。改用 Redis：
```java
// 用 Redisson
RLock lock = redisson.getLock("order:lock:" + orderId);
boolean locked = lock.tryLock(0, 10, TimeUnit.SECONDS); // 不等待，抢不到直接返回
if (!locked) return "已被接单";
try {
    // 查订单状态，更新
} finally {
    lock.unlock();
}
```

**方案三：消息队列串行化（彻底避免并发）**

把所有接单请求发到 MQ 的同一个 Queue/Partition，消费者单线程顺序处理，天然串行，第一个成功的就是"接单者"，后续直接返回已接单。

**推荐方案**：中等并发（200人）用数据库乐观锁就够，简单可靠，不需要 Redis。高并发抢单（上万人）用 Redis 分布式锁 + Lua 脚本原子操作。

###### 2. 项目中如何实现分布式锁？

**答：**

我们项目里用 Redisson 实现 Redis 分布式锁，基本上是生产级的标准方案。

**基本用法**：
```java
RLock lock = redisson.getLock("lock:order:" + orderId);
try {
    // tryLock(等待时间, 锁超时时间, 时间单位)
    boolean locked = lock.tryLock(3, 30, TimeUnit.SECONDS);
    if (!locked) {
        throw new BusinessException("操作频繁，请稍后重试");
    }
    // 业务逻辑
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

**Redisson 的核心特性**：

- **自动续期（看门狗 Watchdog）**：默认锁超时 30 秒，Watchdog 每 10 秒检查一次，如果持锁线程还活着就续期 30 秒，防止业务未执行完锁就超时释放
- **可重入**：同一线程可以多次获取同一把锁，内部用 Hash 记录重入次数
- **Lua 脚本原子操作**：加锁、解锁都是 Lua 脚本执行，保证原子性

**注意事项**：
- `unlock()` 要放在 `finally` 块，防止异常导致锁不释放
- 解锁前检查 `isHeldByCurrentThread()`，防止 A 的锁被 B 解掉
- 锁 Key 要有业务含义，避免锁粒度太粗（一把锁锁全局）

###### 3. Redis 分布式锁的实现原理和注意事项？

**答：**

**实现原理**：

最基础的 Redis 分布式锁用 `SET NX EX` 命令：
```lua
-- 加锁（Lua 原子操作）
SET lock_key unique_value NX EX 30
-- 解锁（Lua 原子操作，先比较再删）
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```
- `NX`：Key 不存在才设置（原子性，保证只有一个客户端能获锁）
- `EX`：设置超时时间，防止客户端宕机后锁永久不释放
- `unique_value`：每个客户端生成唯一 UUID，解锁时先验证是否是自己的锁，防止误删他人锁

**核心注意事项**：

**① 锁超时时间设置**：业务执行时间 < 锁超时时间，否则业务还没完成锁就超时，导致并发问题。用 Redisson 的 Watchdog 自动续期可以解决这个问题。

**② 解锁必须原子**：先 GET 再 DEL 这两步必须用 Lua 脚本保证原子性，否则两步之间锁可能被别人覆盖，导致误删。

**③ 锁粒度**：锁 Key 要足够细粒度，比如 `lock:order:12345` 而不是 `lock:order`，避免不同订单的操作互相阻塞。

**④ 主从切换问题**：主节点 SET 后宕机，从节点切换为主节点时没有这个 Key，可能导致多客户端同时持锁。要求极高时考虑 RedLock（多节点多数派），但复杂度高，大多数场景单节点 Redis 够用。

###### 4. Zookeeper 分布式锁的实现原理？

**答：**

ZooKeeper 分布式锁基于**临时顺序节点**实现，天然避免死锁（客户端宕机，Session 超时后临时节点自动删除）。

**实现原理**：

1. 所有争锁的客户端在锁路径下创建临时顺序节点，如：
   ```
   /lock/lock-0000000001
   /lock/lock-0000000002  ← 当前客户端
   /lock/lock-0000000003
   ```

2. 每个客户端创建完自己的节点后，获取 `/lock/` 下所有子节点列表，如果自己是序号最小的，获锁成功。

3. 如果不是最小，Watch 自己前一个节点（而不是最小节点），等待通知。

4. 当持锁客户端完成后删除节点，序号第二小的客户端收到 Watch 通知，再次检查自己是否最小，如果是则获锁。

**Watch 前一个节点而不是最小节点的原因**：避免"羊群效应"——如果所有客户端都 Watch 最小节点，最小节点删除时所有客户端同时收到通知、同时发起查询，造成大量无效请求。Watch 前一个节点，每次只有一个客户端被通知，O(1) 唤醒。

**ZK 锁 vs Redis 锁**：

- ZK 锁：不需要设置超时时间（Session 超时自动释放），无死锁风险；但性能差（每次加解锁涉及 ZK 写操作，强一致性代价）
- Redis 锁：性能高（单次操作微秒级），但需要处理锁超时问题；ZK 锁实现更复杂，用 Curator `InterProcessMutex` 封装好了

###### 5. 如何防止超卖问题？

**答：**

超卖是电商高并发场景的核心问题：多个请求并发扣减库存，最终扣多了。

**方案一：数据库乐观锁（常规场景）**

扣减库存时带版本号或者检查库存不小于扣减量：
```sql
UPDATE stock SET quantity = quantity - ? 
WHERE product_id = ? AND quantity >= ?
```
如果更新成功（受影响行 = 1）就是扣减成功，否则库存不足。MySQL 行锁保证并发时只有一个事务能成功扣减。

缺点：高并发时大量请求在 DB 层竞争锁，TPS 有瓶颈。

**方案二：Redis 原子操作（高并发场景）**

用 Redis Lua 脚本原子扣减库存，替代直接操作数据库：
```lua
local stock = tonumber(redis.call('get', KEYS[1]))
if stock <= 0 then return -1 end
if stock < tonumber(ARGV[1]) then return -1 end
redis.call('decrby', KEYS[1], ARGV[1])
return 1
```
扣减成功后发 MQ 异步同步到 DB。大促前把库存预热到 Redis，99% 的请求在 Redis 层解决，DB 只接收异步 MQ。

**方案三：分布式锁（极简场景）**

粒度精确到 SKU，每个 SKU 一把锁，一次只允许一个扣减请求，但这样吞吐量极低，不适合高并发。

**推荐组合**：正常电商用乐观锁；大促秒杀用 Redis Lua 预扣 + DB 乐观锁兜底（双保险）。

###### 6. 如何实现秒杀系统？

**答：**

秒杀的核心挑战是：1秒内大量并发请求（可能上万），竞争有限库存（可能只有100件），要保证数据正确、系统不崩、用户体验好。

**分层漏斗架构**（逐层过滤，越往后流量越少）：

**第一层：前端 + CDN**
- 静态页面（商品图片、详情）走 CDN，不打服务器
- 前端按钮防连点（提交后禁用，3秒内不可再点）
- 参数校验在前端做一遍，减少非法请求

**第二层：网关限流**
- Sentinel 对秒杀接口设置 QPS 上限（如1万 QPS），超出直接返回"活动火爆，请稍后重试"
- 对同一用户 ID 限流（每秒最多 1 次），防止刷接口

**第三层：Redis 预扣库存**
```lua
-- Lua 原子操作，防并发
local stock = tonumber(redis.call('get', KEYS[1]))
if stock <= 0 then return -1 end
redis.call('decr', KEYS[1])
return 1
```
Redis 层扣减成功后，把秒杀请求发到 MQ，返回用户"排队中"。

**第四层：MQ 异步下单**
消费者从 MQ 消费消息，执行数据库下单流程（扣库存 + 创建订单），速率受 DB 能力控制，平稳处理。

**第五层：DB 乐观锁兜底**
```sql
UPDATE stock SET quantity = quantity - 1 WHERE product_id = ? AND quantity > 0
```
保证即使 Redis 出问题，DB 层也不会超卖。

**大促前准备**：
- 提前预热库存到 Redis
- 压测演练（wrk/k6 模拟并发）
- 预案：限流阈值可动态调整，库存可实时查看

###### 7. 分布式锁如果出现死锁怎么办？

**答：**

分布式锁死锁通常有两种情况：

**情况一：锁没有过期时间，持锁进程崩溃**

- Redis 分布式锁：设置了 `EX` 过期时间，即使进程崩溃，锁也会在超时后自动释放
- ZooKeeper 分布式锁：临时节点，Session 超时（通常 30-60 秒）后自动删除，锁自然释放

**情况二：嵌套加锁导致死锁**（两个进程互相等待对方的锁）

- 死锁发生：进程 A 持有 Lock1 等 Lock2，进程 B 持有 Lock2 等 Lock1
- 预防：固定加锁顺序（所有代码按同一个顺序获取多把锁），消除循环等待
- 检测：超时机制，`tryLock(timeout)` 等待超时后放弃并释放已持有的锁，然后随机等待后重试（指数退避 + 抖动，避免活锁）

**排查死锁**：

- 检查是否有多个锁被嵌套获取（代码 Review 时关注）
- 监控锁等待时间，超长等待说明可能有死锁
- 分布式锁的 Key 可以记录加锁时间 + 持锁线程/实例 ID，死锁时方便定位是谁持锁

**根本预防**：
1. 锁的超时时间一定要设置合理
2. 尽量避免同时持有多把锁；如果必须，固定加锁顺序
3. 锁粒度要细，持锁时间要短，减少冲突概率
4. 用 Redisson 等成熟库，不要自己手写分布式锁
