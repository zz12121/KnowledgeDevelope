# 问题排查与解决类

---

###### 1. 项目中，有没有遇到什么难的问题，最后是怎么解决的？

**答：**

这道题是开放题，面试官想考察你的技术深度和解决问题的思路，要用 STAR 格式回答（情景→任务→行动→结果），结合真实或典型案例。

**参考案例（选一个与自己最匹配的）**：

**案例：大促当天订单超时关闭任务积压，导致超时订单没有及时释放库存**

**情景**：某次大促下午 2 点，监控告警显示订单表中有大量"超时待支付"订单没有被关闭，Redis 库存数值已扣减但实际库存未释放，导致后续用户无法下单。

**任务**：紧急排查根因，尽快恢复库存，保证大促正常进行。

**行动**：
1. 立即看 XXL-Job 控制台，发现订单关闭任务执行时间从平时的 3 秒变成了 10 分钟，已经超时
2. 看 Skywalking 链路，发现慢在批量更新订单状态的 SQL
3. `EXPLAIN` 分析，`WHERE expire_time < now() AND status = 'PENDING'` 走了全表扫描（`expire_time` 没有索引）
4. 大促期间订单量是平时的 10 倍，全表扫描从 0.5 秒变成 8 秒，任务执行超时，下次触发时前次任务还没完成，任务堆积
5. 紧急操作：在线添加 `expire_time` 索引（使用 `pt-online-schema-change` 不锁表），同时在控制台临时降低任务触发频率，让当前积压任务先消化完
6. 添加索引后查询从 8 秒降到 0.1 秒，任务恢复正常

**结果**：30 分钟内恢复正常，大促未受影响。事后补充了分批处理逻辑（每批 1000 条），并在 code review checklist 加了"where 条件字段必须有索引"检查项。

###### 2. 遇到过线上故障吗？如何处理的？

**答：**

遇到过，线上故障处理最重要的原则是：**先止血恢复服务，再排查根因**，不能边排查边让用户受影响。

**一次典型案例（Redis 内存打满导致服务雪崩）**：

**情景**：某天凌晨 3 点，监控告警：Redis 内存使用率 100%，大量接口开始 5XX。

**处理过程**：

**立即止血（5分钟内）**：
1. 触发降级开关，非核心接口（推荐、活动展示）走降级逻辑，减少 Redis 读写
2. 扩容 Redis 内存（云厂商控制台直接扩容）

**根因排查**：
1. `redis-cli INFO memory` 看内存详情，`DEBUG JMAP` 查大 Key
2. 发现有一个 List 类型 Key，存的是 "最近浏览记录"，数据量撑破了：某个 Bug 导致 List 无限追加、没有 LTRIM，积累了几个亿条记录，占了 20GB
3. 逐步清理大 Key（用 SCAN 分批删除，不用 DEL 命令，避免阻塞主线程）

**事后改进**：
1. 代码修复 LTRIM 逻辑
2. 加大 Key 内存监控告警（单 Key > 10MB 就告警）
3. 推广 `redis-cli --bigkeys` 定期扫描
4. 制定大 Key 清理 SOP，写进 Runbook

###### 3. 如何排查内存泄漏问题？

**答：**

内存泄漏的典型表现：堆内存持续增长，Full GC 频率越来越高但内存不下降，最终 OOM。

**排查步骤**：

**第一步：确认是内存泄漏还是内存不足**

用 `jstat -gcutil <pid> 5000` 每 5 秒打印 GC 情况：
- Full GC 后 Old Gen 占用率持续上升，且增长不可逆 → 内存泄漏
- Full GC 后回收正常，只是堆太小 → 配置问题，扩堆

**第二步：生成堆转储文件**

```bash
jmap -dump:format=b,file=heap.hprof <pid>
# 或者 OOM 时自动 dump，JVM 参数加：
# -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heap.hprof
```

**第三步：用 MAT（Memory Analyzer Tool）分析 heap dump**

- 打开 MAT，看 "Leak Suspects Report"，MAT 会自动指出最可能的泄漏点
- 找 Retained Heap 最大的对象，分析 GC Root 引用链
- 常见原因：静态集合（Map/List）不断增长不清除、ThreadLocal 没有 remove、缓存没有设置上限、WebappClassLoader 泄漏（各框架加载导致 ClassLoader 被强引用）

**第四步：定位代码**

根据 MAT 的引用链找到持有大量对象的代码位置，看为什么这些对象没有被 GC。

**常见泄漏场景**：
- `Cache<K,V>` 忘了设置 `maximumSize`，无限增长
- `ThreadLocal` 用完忘了 `remove()`，Pool 的线程复用导致 value 堆积
- 事件监听器注册后忘了注销

###### 4. 如何排查 CPU 飙高问题？

**答：**

CPU 飙高的常见原因：死循环、频繁 Full GC（GC 本身消耗 CPU）、正则表达式回溯、大量线程上下文切换。

**排查步骤**：

**第一步：找到高 CPU 进程和线程**
```bash
top       # 找到 CPU 最高的 Java 进程 PID
top -H -p <pid>   # 列出该进程所有线程的 CPU 占用，找出高 CPU 的线程 TID
printf "%x\n" <tid>   # 把 TID 转成十六进制
```

**第二步：jstack 拿到线程堆栈**
```bash
jstack <pid> > thread_dump.txt
# 搜索十六进制 TID（如 0x3f2a）找到对应线程
grep -A 20 "0x3f2a" thread_dump.txt
```
看线程当前在做什么：死循环（RUNNABLE + 同一行代码反复出现）、GC 线程（`GC task thread`）、正则（`java.util.regex`）

**第三步：用 Arthas 在线诊断**
```bash
# 找出 CPU 占用最高的线程
thread -n 3
# 看具体方法耗时
trace com.example.OrderService processOrder
```

**常见原因和对应解法**：
- 死循环 → 找代码逻辑 Bug，通常是 while/for 条件写错
- Full GC 导致 GC 线程 CPU 高 → 排查内存泄漏，调整堆大小和 GC 参数
- 正则回溯 → 检查正则表达式是否有回溯陷阱（嵌套量词），改用非回溯写法
- 大量线程竞争锁 → 看线程状态是否大量 BLOCKED，找热点锁

###### 5. 遇到过死锁吗？如何排查和解决？

**答：**

遇到过。Java 代码级别的死锁表现是：多个线程互相等待对方持有的锁，所有线程都阻塞，CPU 占用低但业务完全停止。

**排查步骤**：

**用 jstack 检测死锁**：
```bash
jstack <pid> | grep -A 30 "deadlock"
```
jstack 输出里如果有死锁，会在最底部有 "Found one Java-level deadlock" 的明显标识，并列出涉及的线程和锁。

**Arthas（推荐，更方便）**：
```bash
thread -b   # 直接输出死锁线程信息
```

**分析死锁原因**：

从 jstack 输出看线程 A 持有 Lock1 等待 Lock2，线程 B 持有 Lock2 等待 Lock1，形成环路。找到对应代码逻辑。

**解决方案**：

**短期（恢复服务）**：重启服务，临时解除死锁。

**根本解决**：
1. **固定加锁顺序**：所有需要同时持有多把锁的代码，按相同顺序获取锁
2. **tryLock 超时**：用 `ReentrantLock.tryLock(timeout)` 替代 `synchronized`，超时后放弃并释放已持有的锁
3. **减少锁粒度**：拆分粗粒度锁，减少同时持有多把锁的代码路径
4. **Code Review**：在 Review 阶段识别嵌套加锁，预防胜于治疗

###### 6. 接口响应慢如何排查？

**答：**

接口响应慢排查要从全链路入手，逐层定位瓶颈所在。

**第一步：确定是哪个环节慢**

用 SkyWalking 看该接口的 Trace 详情：
- 网关到服务入口耗时（网络 + 网关处理）
- Service 内部耗时（哪个方法调用）
- DB 查询耗时（哪条 SQL）
- Redis 调用耗时
- 外部 HTTP 调用耗时

通常 Trace 会直接显示瀑布图，一眼看出最长的 Span 在哪里。

**第二步：按定位结果深入排查**

- **慢 SQL**：`EXPLAIN` 看执行计划，检查是否走了全表扫描、大量回表。看 Slow Query Log（`long_query_time=1`）。
- **GC 导致 STW**：看 GC 日志，Full GC 发生时所有线程暂停，接口超时。用 G1 或 ZGC 降低 STW 时间。
- **线程池满**：连接池等待队列堆积（`HikariCP` 的 `pending` 队列长），说明 DB 连接数不够或慢 SQL 占住连接。
- **外部依赖慢**：第三方接口超时，用 Resilience4j 加超时+降级，不让它拖垮主接口。
- **锁竞争**：高并发写操作锁等待时间长，用 Arthas `trace` 查具体加锁位置。
- **序列化/反序列化**：大对象 JSON 序列化开销，改用分页或简化响应字段。

**第三步：验证优化效果**

用 wrk 或 JMeter 压测同场景，对比优化前后 TP99。

###### 7. 如何进行线上问题回滚？

**答：**

回滚是应对上线故障的最快手段，要求：快（分钟内完成）、安全（回滚不引入新问题）、可恢复（回滚后可重新部署修复版本）。

**代码回滚**：

Jenkins + GitLab：直接在 Jenkins 上选择上一个稳定版本的 Tag 重新构建部署，整个回滚流程 5~10 分钟。
```
GitLab 上 checkout 上一版本 Tag → Jenkins Pipeline 构建镜像 → K8s 滚动更新
```
Kubernetes 也可以直接回滚：
```bash
kubectl rollout undo deployment/order-service
# 或者指定版本
kubectl rollout undo deployment/order-service --to-revision=2
```

**数据库变更回滚**：

数据库 DDL 回滚风险较高，要提前准备：
- 每次 DDL 变更前写好"回滚 SQL"（如加列的回滚就是删列，加索引的回滚是删索引）
- 用 Liquibase 或 Flyway 管理数据库变更版本，支持 `rollback` 命令
- 大表 DDL 用 `pt-online-schema-change` 在线操作，可以随时中断不影响原表

**配置回滚**：

Nacos 控制台支持一键回滚到历史版本，5 秒内生效，不需要重启服务。

**回滚决策**：

- 故障影响核心功能、短期无法定位根因 → 立即回滚
- 故障是已知 Bug、有临时降级方案 → 可以先降级，热修复后重新上线
- 数据已经被脏写 → 代码回滚后还需要数据修复，不能只回滚代码
