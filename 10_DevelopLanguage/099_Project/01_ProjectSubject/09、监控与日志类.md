# 监控与日志类

---

###### 1. 项目中如何实现日志收集和分析？

**答：**

我们用 ELK Stack（Elasticsearch + Logstash + Kibana）做日志收集和分析，整个流程如下：

**日志采集**：

各服务用 Logback 输出结构化 JSON 日志（包含 traceId、userId、接口名、耗时、错误信息等字段），输出到文件。每台机器部署 Filebeat，实时采集日志文件并发到 Kafka（加一层 Kafka 做缓冲，防止日志洪峰打垮 ES）。

**日志处理**：

Logstash 从 Kafka 消费日志，做字段解析、过滤、转换（比如提取 HTTP 状态码、解析异常栈）后写入 ES。

**存储与查询**：

ES 按天创建索引（`app-logs-2026.03.15`），配合 ILM（Index Lifecycle Management）策略：热数据保留 7 天，温数据保留 30 天，冷数据归档后删除。

Kibana 做日志检索（KQL 语法）和可视化仪表盘，出问题时用 traceId 直接过滤出完整请求链路的所有日志。

**日志规范**：
```json
{
  "timestamp": "2026-03-15T21:00:00.000+08:00",
  "level": "ERROR",
  "traceId": "a1b2c3d4e5f6",
  "service": "order-service",
  "method": "POST /orders",
  "userId": "12345",
  "duration": 256,
  "message": "下单失败：库存不足",
  "exception": "..."
}
```

###### 2. 如何实现链路追踪？

**答：**

我们用 SkyWalking 做链路追踪，Agent 方式无侵入接入，不需要改业务代码。

**接入方式**：

JVM 启动参数加：`-javaagent:/opt/skywalking-agent.jar -Dskywalking.agent.service_name=order-service`，Agent 自动拦截 Spring MVC、MySQL、Redis、HTTP、MQ 等组件，把调用链数据上报到 SkyWalking OAP Server。

**TraceId 透传**：

SkyWalking 自动在 HTTP 请求头中传递 `sw8` 头（包含 TraceId），Feign 调用时自动透传，跨服务调用形成完整链路。

为了在业务日志里也能看到 TraceId，我们在 Logback 配置里用 MDC：
```java
// 或者通过 SkyWalking 提供的 API
TraceContext.traceId(); // 获取当前 TraceId，可加到 MDC
```
Logback pattern 里加 `%X{traceId}`，这样每条日志都带 TraceId，ELK 和 SkyWalking 两边都能关联。

**核心功能**：

- 拓扑图：自动绘制服务依赖关系图
- 链路详情：查看完整调用链，每个 Span 的耗时、SQL 语句、异常信息
- 性能分析：接口 TP99、慢服务、慢 SQL 自动告警

###### 3. 项目中有哪些监控指标？

**答：**

监控体系分四个层次：

**系统层（基础设施）**：
- CPU 使用率、Load Average（超过 CPU 核数要告警）
- 内存使用率（超过 80% 告警）
- 磁盘使用率 + IO 读写速率
- 网络带宽、TCP 连接数

**JVM 层**：
- 堆内存使用率（Old Gen 持续增长 → 内存泄漏）
- GC 频率和耗时（Full GC 超过 1 次/分钟告警）
- 线程数（线程池活跃线程、等待队列长度）
- Metaspace 使用率

**应用层**：
- 接口 QPS、TPS
- 接口响应时间 TP50/TP90/TP99（TP99 > 1s 告警）
- 接口错误率（5XX 比例 > 0.1% 告警）
- 连接池使用率（HikariCP 等待队列长度）
- 缓存命中率

**业务层**：
- 下单成功率、支付成功率
- 消息积压量（MQ 消费 lag）
- 定时任务成功率
- 核心 KPI（日活、GMV、注册量）

**工具**：Prometheus 采集指标，Grafana 做可视化仪表盘，AlertManager 发告警（钉钉/飞书/邮件）。

###### 4. 如何实现告警机制？

**答：**

我们的告警体系：Prometheus 负责采集和存储指标，AlertManager 负责路由和发送告警，Grafana 用于可视化和告警规则管理。

**告警规则配置（Prometheus Rules）**：
```yaml
groups:
  - name: api_alerts
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.01
        for: 2m        # 持续 2 分钟才触发，避免抖动告警
        labels:
          severity: critical
        annotations:
          summary: "接口错误率超过 1%"
          description: "服务 {{ $labels.service }} 错误率 {{ $value | humanizePercentage }}"
```

**告警分级**：
- P0（Critical）：立即电话+钉钉，影响核心业务（支付失败、服务不可用）
- P1（Warning）：钉钉群通知，关注但可延后处理（接口慢、内存增长趋势）
- P2（Info）：只记录不告警，用于趋势分析

**防告警风暴**：
- `for: 2m`：指标异常持续超过 2 分钟才触发，过滤瞬时抖动
- AlertManager 告警聚合（同一服务的多个告警合并为一条）
- 告警抑制（已有 Critical 时，同服务的 Warning 告警静默）
- 告警静默（维护窗口期间临时关闭特定告警）

**Runbook**：每个告警关联一个处理文档（Runbook），告警消息里带链接，值班工程师收到告警后直接查处理流程，减少 MTTD（故障检测时间）。

###### 5. 线上问题如何快速定位？

**答：**

线上告警来了，定位速度直接影响故障持续时长。我们的排查流程（SOP）：

**第一步：判断影响范围（2分钟内）**

看 Grafana 监控：哪些服务、哪些接口受影响？用户量级多大？核心链路还是边缘功能？先判断是否需要立即降级或回滚。

**第二步：找到出错位置（5分钟内）**

用 SkyWalking 链路追踪：
1. 找一个有问题的用户请求，从告警 traceId 或用户反馈入手
2. 在 SkyWalking 查 traceId 的完整链路，定位是哪个服务、哪个方法耗时异常或报错

**第三步：深入排查（10分钟内）**

- **看日志**：Kibana 里用 traceId 过滤，看完整错误日志和异常堆栈
- **看指标**：GC 是否频繁？DB 连接池满了？Redis 超时？CPU/内存异常？
- **看代码**：根据日志定位到具体代码行

**第四步：处置**

- 临时处置（先止血）：降级非核心功能、扩容、回滚
- 根本处置：修 Bug、优化 SQL、扩容资源
- 事后：写 Post-mortem 故障复盘，分析根因，落地改进措施

**工具链**：SkyWalking（链路）+ ELK（日志）+ Grafana（指标）+ Arthas（线上诊断）四件套。

###### 6. 如何进行日志脱敏？

**答：**

日志里不能有敏感信息（手机号、身份证、银行卡、密码、密钥），否则违反数据安全合规要求。

**方案一：Jackson 全局序列化脱敏（推荐）**

自定义 Jackson 注解，在实体类字段上标注，序列化时自动脱敏：
```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface Sensitive {
    SensitiveType type();
}

// 实体字段
@Sensitive(type = SensitiveType.PHONE)
private String phone;  // 序列化时自动输出 "138****8888"
```
在 Jackson 序列化器里实现脱敏逻辑（手机号中间4位替换为 `****`，身份证显示前3后4，密码全 `***`）。

**方案二：Logback Converter 过滤**

在 Logback 配置里加正则过滤：
```xml
<conversionRule conversionWord="mask" converterClass="MaskConverter"/>
```
MaskConverter 对日志字符串做正则替换，匹配手机号、身份证、信用卡等 Pattern，自动脱敏。

**方案三：AOP 拦截日志入参**

Controller 层 AOP 打印入参日志前，先对敏感字段脱敏：
```java
@Around("@annotation(ApiLog)")
public Object log(ProceedingJoinPoint pjp) {
    // 入参脱敏处理
    String params = desensitize(JSON.toJSONString(pjp.getArgs()));
    log.info("请求参数：{}", params);
    return pjp.proceed();
}
```

**最佳实践**：入口统一处理（VO/DTO 层加注解），不依赖开发手动脱敏，减少遗漏风险。
