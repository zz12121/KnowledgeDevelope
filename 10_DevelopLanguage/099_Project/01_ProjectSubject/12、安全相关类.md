# 安全相关类

---

###### 1. 项目中如何防止 SQL 注入？

**答：**

SQL 注入的根本原因是把用户输入直接拼接进 SQL 字符串。防御的核心原则是：**绝不拼接，全用预编译**。

**① MyBatis 预编译（`#{}` vs `${}`）**

MyBatis 的 `#{}` 是预编译参数，底层生成 `PreparedStatement`，参数值不参与 SQL 解析，完全安全：
```xml
<select id="findUser" resultType="User">
    SELECT * FROM users WHERE name = #{name}   <!-- 安全：预编译 -->
</select>
```
`${}` 是字符串替换，直接拼接进 SQL，有注入风险。只有在 ORDER BY 列名、表名这类不能用预编译的场景才用 `${}`，且必须做白名单校验：
```java
// 白名单校验
if (!ALLOWED_COLUMNS.contains(sortColumn)) {
    throw new IllegalArgumentException("非法排序字段");
}
```

**② 参数校验**

接口入参用 Bean Validation 校验格式，非法格式请求直接拦截，连 DB 都不到：
```java
@Pattern(regexp = "^[a-zA-Z0-9_]{1,50}$", message = "用户名格式不正确")
private String username;
```

**③ 最小权限原则**

DB 账号按最小权限配置，业务账号只有 SELECT/INSERT/UPDATE/DELETE 权限，没有 DROP TABLE 等 DDL 权限，即使被注入也无法删表。

**④ WAF（Web Application Firewall）**

网关层或 CDN 层部署 WAF，检测并拦截常见 SQL 注入 Pattern（`' OR 1=1`、`UNION SELECT` 等）。

###### 2. 如何防止 XSS 攻击和 CSRF 攻击？

**答：**

**XSS（跨站脚本攻击）防御**：

XSS 是攻击者注入恶意脚本到页面，在其他用户浏览器上执行。

- **输出编码**：所有用户输入内容在输出到 HTML/JS 时进行 HTML 实体编码（`<` 转 `&lt;`），防止脚本执行。前端 Vue/React 的模板语法默认转义，只有 `v-html` / `dangerouslySetInnerHTML` 才需要额外注意。
- **内容安全策略（CSP）**：`Content-Security-Policy` 响应头限制页面只能加载可信来源的脚本，即使注入了脚本也无法执行
- **HttpOnly Cookie**：Session Cookie 设置 `HttpOnly`，JavaScript 无法读取 Cookie，窃取会话失败
- **输入过滤**：富文本内容用 OWASP Java HTML Sanitizer 过滤，白名单允许的 HTML 标签

**CSRF（跨站请求伪造）防御**：

CSRF 是攻击者诱导用户访问恶意页面，借用用户的 Cookie 向正常网站发送请求。

- **CSRF Token**：服务端生成随机 Token，放到表单或请求头，攻击者的页面无法获取这个 Token，请求会被拒绝。Spring Security 默认开启 CSRF 保护。
- **SameSite Cookie**：Cookie 设置 `SameSite=Strict` 或 `SameSite=Lax`，跨站请求不携带 Cookie，CSRF 自然失效
- **验证 Referer**：检查请求的 `Referer` 头是否来自本域（作为辅助手段，不能单独依赖）
- **前后端分离（JWT 认证）**：JWT 放在 Authorization 请求头，而不是 Cookie，浏览器不自动携带，CSRF 无法利用

###### 3. 用户密码如何存储？

**答：**

密码绝对不能明文存储，也不能用 MD5/SHA1 这类可逆或可彩虹表破解的方式。

**正确做法：BCrypt 哈希**

BCrypt 是专门为密码设计的哈希算法，有以下特点：
- **加盐（Salt）**：每次哈希结果不同，即使两个用户密码相同，哈希值也不同，防止彩虹表
- **慢哈希（Cost Factor）**：可以调节计算成本，成本因子越高，每次哈希越慢，暴力破解越难
- **无需单独存 salt**：BCrypt 哈希结果本身包含 salt

```java
// Spring Security 内置支持
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12); // cost factor=12，推荐 10~14
}

// 注册时加密
String encodedPassword = passwordEncoder.encode(rawPassword);

// 登录时验证
boolean matches = passwordEncoder.matches(rawPassword, encodedPassword);
```

**其他注意事项**：
- 传输层：密码通过 HTTPS 加密传输，不明文走网络
- 不存储原密码：即使是找回密码，也是重置密码而不是"发送原密码"
- 密码策略：最小长度 8 位，包含大小写+数字，防止弱密码

###### 4. 敏感数据如何加密？

**答：**

敏感数据（手机号、身份证、银行卡号、地址）需要在数据库层面加密存储，防止 DB 泄露导致数据暴露。

**字段级加密（AES-128/256）**：

```java
@Component
public class AesEncryptor {
    @Value("${encrypt.aes-key}")
    private String aesKey; // 密钥从配置中心/KMS获取，不硬编码

    public String encrypt(String plaintext) {
        // AES-GCM 模式，有认证，比 ECB/CBC 更安全
        byte[] iv = generateRandomIV();
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.ENCRYPT_MODE, new SecretKeySpec(aesKey.getBytes(), "AES"), new GCMParameterSpec(128, iv));
        byte[] encrypted = cipher.doFinal(plaintext.getBytes());
        return Base64.encode(concat(iv, encrypted));
    }
}
```

配合 MyBatis TypeHandler，在 SQL 读写时自动加解密，业务代码无感知。

**密钥管理（KMS）**：

加密密钥是核心资产，不能硬编码在代码里，要接入 KMS（Key Management Service）：
- 密钥存在 KMS（AWS KMS、阿里云 KMS），应用有权限的情况下运行时获取
- 轮换密钥：定期更换加密密钥（老数据用老密钥解密，写入用新密钥），降低密钥泄露影响

**传输加密**：HTTPS + TLS 1.3，敏感数据不走明文 HTTP。

**日志脱敏**：加密存储的同时，日志输出时也要脱敏（手机号显示 `138****8888`）。

###### 5. 如何实现接口鉴权？

**答：**

接口鉴权分两步：认证（你是谁）+ 授权（你能做什么）。

**认证：JWT**

用户登录后颁发 JWT（JSON Web Token），后续请求在 Authorization 请求头携带：
```
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...
```
网关层统一验签（用 RSA 非对称密钥，公钥验签，私钥在认证服务），验证通过后把用户 ID、角色、权限列表解析出来，放入请求头透传给下游服务。

```java
// 网关 GlobalFilter
@Override
public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    String token = exchange.getRequest().getHeaders().getFirst("Authorization");
    Claims claims = jwtUtil.parseToken(token);
    // 把用户信息加入请求头
    ServerHttpRequest mutatedRequest = exchange.getRequest().mutate()
        .header("X-User-Id", claims.getSubject())
        .header("X-User-Role", claims.get("role").toString())
        .build();
    return chain.filter(exchange.mutate().request(mutatedRequest).build());
}
```

**授权：RBAC + Spring Security**

后端服务用 Spring Security，在 Service 层关键方法加 `@PreAuthorize`：
```java
@PreAuthorize("hasRole('ADMIN') or hasPermission(#orderId, 'ORDER_VIEW')")
public OrderVO getOrder(Long orderId) { ... }
```

**接口白名单**：登录、注册、健康检查等接口不需要鉴权，在网关配置白名单路由跳过鉴权过滤器。

###### 6. Token 如何设计和管理？

**答：**

Token 设计要兼顾安全性和用户体验（别动不动就让用户重新登录）。

**双 Token 方案（Access Token + Refresh Token）**：

- **Access Token**：短有效期（15分钟~2小时），每次 API 请求携带，无状态验证，存在内存/localStorage
- **Refresh Token**：长有效期（7~30天），仅用于刷新 Access Token，存在 HttpOnly Cookie（安全存储），每次使用后轮换（Refresh Token Rotation），旧 Refresh Token 立即失效，防止重放攻击

**Access Token 无状态验证**（推荐）：
- JWT 签名验证，无需查 DB，性能高
- 短有效期内即使 Token 泄露，影响范围有限

**Token 失效（踢下线）**：
- JWT 无状态的问题是无法主动让 Token 失效
- 解法：维护 Token 黑名单（Redis，TTL = Token 剩余有效期）；或 Token 版本号（DB/Redis 存最新版本，JWT 携带版本，不匹配则拒绝）
- 修改密码、账号被封禁时，递增用户的 Token 版本号，所有旧 Token 立即失效

**安全注意事项**：
- Access Token 不放 Cookie（CSRF 风险），放 Authorization 请求头
- Refresh Token 放 HttpOnly Cookie（XSS 无法读取），只在刷新接口使用
- HTTPS 全程加密，防止 Token 被中间人截获

###### 7. 如何防止接口被恶意调用？

**答：**

接口被恶意调用（爬虫、刷接口、DDoS）的防御是多层的：

**① 接口签名验证（防篡改 + 防重放）**

对外 API 要求客户端做签名：请求头携带 `timestamp`、`nonce`（随机串）、`sign`（HMAC-SHA256 签名）。
- 时间戳在 10 分钟内有效（防重放）
- nonce 在 Redis 中标记已用过（防重放）
- 签名覆盖所有请求参数（防篡改）

**② 频率限制（Rate Limiting）**

网关层对单 IP + 用户 ID 做限流：
- 同一 IP，1 分钟内同接口不超过 20 次
- 同一用户，1 秒内不超过 5 次
- 用 Redis 滑动窗口实现，Lua 脚本原子操作

**③ 验证码（人机验证）**

登录、注册、发短信等敏感接口，频率触发阈值后要求滑动验证码或图形验证码（行为验证码，不可绕过）。

**④ 黑名单**

- IP 黑名单：访问频率异常的 IP 加入黑名单，网关直接 403
- 账号黑名单：检测到异常行为的账号临时封禁
- 黑名单存 Redis，动态生效

**⑤ WAF + DDoS 防护**

云厂商的 WAF（Web Application Firewall）和 DDoS 防护（流量清洗），在流量进入服务前拦截大规模攻击。
