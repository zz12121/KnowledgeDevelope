# 第三方集成类

---

###### 1. 项目中集成了哪些第三方服务？

**答：**

我们项目中集成了以下几类第三方服务，说一下每类的接入思路：

**① 支付类**：微信支付、支付宝。最核心的集成，需要特别关注安全（签名验签、HTTPS 双向认证）和幂等性（支付单不重复创建、回调幂等处理）。

**② 短信/消息类**：阿里云短信、腾讯云短信（双供应商冗余）。接入时抽象统一的 `SmsService` 接口，底层可以根据可用性自动切换供应商，防止单供应商宕机影响短信发送。

**③ 物流类**：快递鸟（聚合了多家快递）、顺丰直接对接。获取物流轨迹定时轮询 + Webhook 回调结合，轨迹数据缓存到 Redis 减少频繁调用。

**④ 文件存储类**：阿里云 OSS / 腾讯云 COS。所有文件上传走 OSS，后端不做中转，前端直传（STS 临时凭证），减少服务器带宽消耗。

**⑤ 身份认证类**：微信/QQ 第三方登录（OAuth 2.0 授权码模式）、实名认证（人脸识别 API）。

**⑥ 地图/地理服务**：高德地图 API（地址解析、附近搜索、配送距离计算）。

**接入原则**：
- 统一封装：每个第三方服务封装为独立 `Client` 类，不直接调 SDK
- 超时配置：所有外部调用设置超时（通常 3~5 秒），用 Resilience4j 加熔断降级
- 重试与幂等：非幂等操作不自动重试（如支付），幂等操作可以配置指数退避重试

###### 2. 如何对接支付平台？

**答：**

对接支付平台（以微信支付为例），核心环节：

**① 环境准备**

注册微信商户账号，获取 `appid`、`mch_id`（商户号）、API 密钥，下载并配置证书（`.p12` 文件，用于退款等高权限接口）。

**② 支付流程**

以小程序支付为例（JSAPI）：
```
小程序前端 → 我方后端 → 微信支付统一下单接口
      ↓ 返回 prepay_id
微信支付 SDK 唤起支付弹窗
      ↓ 用户支付
微信 → 我方 notify_url 回调
      ↓ 验签 + 幂等处理 + 更新订单
返回 SUCCESS 给微信（必须，否则微信会重复回调）
```

**③ 回调处理（关键）**

```java
@PostMapping("/wechat/notify")
public String notify(@RequestBody String notifyBody) {
    // 1. 验签（使用 APIV3 密钥）
    if (!wechatPayValidator.validate(notifyBody)) return "FAIL";
    
    // 2. 解析支付结果
    PayNotifyDTO notify = parseNotify(notifyBody);
    
    // 3. 幂等：检查支付单状态
    PayOrder payOrder = payOrderMapper.selectByOutTradeNo(notify.getOutTradeNo());
    if (payOrder.getStatus() == PAID) return "SUCCESS"; // 已处理，直接返回成功
    
    // 4. 更新支付单和订单状态（数据库事务）
    updatePayAndOrderStatus(payOrder, notify);
    
    // 5. 发 MQ 触发后续业务（积分、库存确认等）
    eventPublisher.publish(new OrderPaidEvent(payOrder.getOrderId()));
    
    return "SUCCESS";
}
```

**④ 安全注意事项**

- 回调接口不要做鉴权（微信不会带我们的 token），但要验签（微信的数字签名）
- 金额要以分为单位（整数），不用浮点数
- notify_url 必须是 HTTPS，且能被微信公网访问

###### 3. 如何对接物流系统？

**答：**

物流对接有两种模式：直接对接各快递公司 API，或者接入聚合平台（快递鸟、快递100）。

**推荐：聚合平台（快递鸟）**

一次接入，支持所有主流快递公司，API 统一，维护成本低。

**核心功能**：

**① 运单创建（电子面单）**

下单时调用快递鸟接口，发运单数据（收寄件地址、商品重量），获取快递单号和面单打印内容，面单直接打印贴包裹。

**② 物流轨迹查询**

- **定时轮询**：每隔 30 分钟对"已发货"状态的订单查询最新轨迹，更新到 DB，缓存到 Redis（TTL 30 分钟）
- **Webhook 推送**（推荐）：注册回调地址，快递状态变更时快递鸟主动推送，减少无效轮询：

```java
@PostMapping("/logistics/callback")
public String callback(@RequestBody LogisticsCallbackDTO dto) {
    // 验签
    // 更新轨迹，发 MQ 通知用户
    return "OK";
}
```

**③ 缓存策略**

物流轨迹变更频率低（一天几次），查询频率高（用户反复刷新），适合缓存：
- 正在运输中：TTL 30 分钟（可能有新轨迹）
- 已签收：TTL 24 小时（不会再变）
- 异常状态（快件丢失）：不缓存，实时查

###### 4. 短信和邮件服务如何实现？

**答：**

短信和邮件服务作为通知渠道，要考虑：解耦（异步发送，不影响主业务）、可靠（不丢消息）、统一管理（模板管理、发送记录、失败重试）。

**整体架构**：

业务服务发消息事件（如 `SmsEvent`）到 MQ，消息通知服务消费 MQ，调用具体短信/邮件 SDK 发送。

**短信发送**：

```java
@Service
public class SmsServiceImpl implements SmsService {
    
    // 双供应商，主备切换
    @Autowired
    private AliSmsClient aliSmsClient;
    
    @Autowired
    private TencentSmsClient tencentSmsClient;

    @Override
    public void sendSms(SmsDTO dto) {
        try {
            aliSmsClient.send(dto.getPhone(), dto.getTemplateCode(), dto.getParams());
        } catch (SmsException e) {
            log.warn("阿里云短信失败，切换腾讯云", e);
            tencentSmsClient.send(dto.getPhone(), dto.getTemplateCode(), dto.getParams());
        }
        // 记录发送流水（手机号脱敏存储）
        smsLogMapper.insert(buildSmsLog(dto));
    }
}
```

**邮件发送**：

Spring Boot `JavaMailSender`，HTML 邮件模板用 Thymeleaf 渲染：
```java
MimeMessage message = mailSender.createMimeMessage();
MimeMessageHelper helper = new MimeMessageHelper(message, true, "UTF-8");
helper.setTo(to);
helper.setSubject(subject);
helper.setText(thymeleafEngine.render("email/order-confirm", model), true); // true=HTML
mailSender.send(message);
```

**防刷**：短信发送加频率限制（同一手机号 1 分钟最多 1 条，1 天最多 5 条），Redis 计数。

###### 5. 第三方接口调用失败如何处理？

**答：**

第三方接口（支付、短信、物流）调用失败是常态，要有完善的降级和补偿机制。

**分类处理**：

**① 临时性失败（超时、服务抖动）**

配置超时 + 自动重试（带指数退避）：
```java
@Retryable(
    value = {SocketTimeoutException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 1000, multiplier = 2) // 1s, 2s, 4s
)
public void sendSms(SmsDTO dto) { ... }
```

**② 熔断降级（第三方服务故障）**

用 Resilience4j 熔断器：连续失败超过阈值，熔断打开，后续请求直接走降级逻辑，不继续打第三方：
```java
@CircuitBreaker(name = "smsService", fallbackMethod = "smsFallback")
public void sendSms(SmsDTO dto) { ... }

public void smsFallback(SmsDTO dto, Exception ex) {
    // 降级：记录到待发送队列，等第三方恢复后重发
    pendingSmsQueue.add(dto);
    log.warn("短信服务熔断，已加入待发送队列", ex);
}
```

**③ 非幂等操作（如支付）**

不自动重试！失败后记录到本地消息表，由人工确认或对账任务处理，避免重复支付。

**④ 监控告警**

第三方接口调用成功率接入 Prometheus，低于阈值（如 95%）立即告警，快速发现第三方故障。

###### 6. 如何实现第三方接口的重试机制？

**答：**

重试机制设计的核心原则：**幂等操作可以重试，非幂等操作要谨慎**。

**① Spring Retry（注解式，简单场景）**

```java
@Retryable(
    value = {IOException.class, HttpClientErrorException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 500, multiplier = 2, maxDelay = 5000)  // 500ms → 1s → 2s（上限5s）
)
public TrackingInfo queryLogistics(String trackNo) {
    return logisticsClient.query(trackNo);
}

@Recover  // 重试全部失败后的兜底
public TrackingInfo recoverQueryLogistics(Exception e, String trackNo) {
    log.error("物流查询失败，trackNo: {}", trackNo, e);
    return TrackingInfo.unknown();
}
```

**② 本地消息表 + 定时重试（可靠场景）**

对于必须成功的操作（发短信通知、同步数据），发送前先入库（status=PENDING），成功后更新 DONE，失败后保持 PENDING。定时任务扫描 PENDING 状态且过了重试间隔的消息，重新发送：

```sql
SELECT * FROM message_outbox 
WHERE status = 'PENDING' AND next_retry_time <= NOW() AND retry_count < 5
```

每次重试后更新 `retry_count` 和 `next_retry_time`（指数退避：下次重试时间 = now + 2^retryCount 分钟）。超过最大重试次数标记 FAILED，告警人工处理。

**③ 指数退避 + Jitter（避免雪崩重试）**

多个客户端同时重试会在同一时间点同时打第三方，用随机抖动（Jitter）让重试时间分散：
```java
long delay = (long)(Math.pow(2, retryCount) * 1000 + Math.random() * 1000);
```
