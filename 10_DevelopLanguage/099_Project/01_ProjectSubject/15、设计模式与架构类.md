# 设计模式与架构类

---

###### 1. 项目中用到了哪些设计模式？

**答：**

项目里用到了不少设计模式，分类说几个最典型的：

**创建型**：
- **工厂模式**：根据订单类型（普通订单/团购订单/预售订单）创建不同的 OrderHandler 实现，用 Map<OrderType, OrderHandler> 替代 if-else 枚举
- **建造者模式**：构建复杂的查询条件对象（OrderQuery.builder().userId().status().dateRange().build()），避免多参数构造器

**结构型**：
- **装饰器模式**：日志、监控、限流逻辑用装饰器包装 Service，不侵入业务代码
- **代理模式**：Spring AOP 事务管理、日志记录、权限校验，本质都是代理
- **门面模式**：PaymentFacade 聚合支付宝、微信支付多个 SDK，对上层提供统一接口

**行为型**：
- **策略模式**：优惠券计算（满减/折扣/立减），每种策略实现 `DiscountStrategy` 接口，通过 Spring 容器注入 Map，按类型路由
- **模板方法模式**：各业务的消息发送（短信/邮件/Push），抽象父类定义发送流程骨架，子类实现具体渠道
- **观察者模式**：下单成功事件（`OrderCreatedEvent`），多个监听者（库存服务、积分服务、营销服务）独立监听，解耦
- **状态机模式**：订单状态流转（待支付 → 已支付 → 已发货 → 已完成 → 已取消），状态迁移逻辑封装在状态类里

**面试加分点**：设计模式不是"用越多越好"，关键是"用对了才好"，要说清楚在什么场景下用、解决了什么问题。

###### 2. 如何进行系统架构设计？

**答：**

架构设计没有放之四海而皆准的模板，但有一套思考框架：

**第一步：理解业务需求**

- 功能需求：系统要做什么
- 非功能需求（更关键）：QPS 多少、可用性要求（99.9%？99.99%？）、数据量、延迟目标、安全级别

**第二步：选择架构模式**

- **单体应用**：团队小、业务简单、快速迭代时的首选；不要过度设计
- **微服务**：业务复杂、团队规模大、需要独立部署和扩缩容时引入；参考 DDD 划分服务边界
- **事件驱动**：有大量异步解耦需求时引入 MQ

**第三步：设计各层架构**

- **接入层**：API Gateway（鉴权、限流、路由）、CDN（静态资源）
- **服务层**：各微服务，Spring Boot + Spring Cloud，Feign 调用 + MQ 通信
- **数据层**：MySQL 主从（写/读分离）+ Redis 缓存 + ES 搜索 + OSS 文件存储
- **基础设施**：K8s 容器编排、Nacos 注册中心/配置中心、SkyWalking 链路追踪、ELK 日志

**第四步：考虑非功能性**

高可用（无单点）、可扩展（水平扩容）、可观测（日志/指标/链路）、安全（认证/鉴权/加密）。

**第五步：演化式架构**

架构不是一步到位的，从 MVP 开始，随业务增长演进。过度设计是最大的浪费。

###### 3. 如何保证系统的可扩展性？

**答：**

可扩展性是指系统能在变更需求或增长流量时，以较小代价平滑应对。有两个维度：

**水平扩缩容（Scale Out）**：

应用层设计成无状态（Session 用 Redis 共享，本地缓存接受短暂不一致），任意时候加减实例不影响功能。K8s HPA 根据 CPU/内存自动扩缩容。

数据层：MySQL 读写分离（读压力分散到从库），Redis Cluster 按需加分片，ES 分片数提前规划（分片创建后无法修改）。

**功能可扩展（开闭原则）**：

- **策略模式**：新增优惠类型，只需新写一个 `DiscountStrategy` 实现类，通过 Spring 自动注入到路由 Map，不需要改原有代码
- **插件化 / SPI**：核心框架定义接口，功能用插件扩展，如 Dubbo 的 SPI 机制
- **事件驱动**：下单成功发事件，新功能只需新增监听者，不修改下单代码（开闭原则）
- **抽象隔离**：数据库操作抽象为 Repository 接口，上层不关心底层是 MySQL 还是其他存储

**避免反可扩展性模式**：
- 大量 if-else 按类型分支：改成策略模式
- 强耦合直接调用：改成事件或 MQ
- 全局单点（单数据库、单消息队列实例）：加高可用方案

###### 4. 如何进行领域驱动设计？

**答：**

DDD（Domain-Driven Design）是一种把业务模型直接反映到软件设计中的方法。我们项目里做了部分 DDD 实践，主要体现在：

**核心概念的落地**：

- **限界上下文（Bounded Context）**：按业务领域划分微服务边界，每个服务有自己独立的数据模型和职责。比如订单服务关心"订单"，商品服务关心"商品"，两者的"价格"含义可能不同（商品服务关心展示价，订单服务关心下单时的快照价），不混用。

- **聚合根（Aggregate Root）**：保证聚合内数据一致性的入口。订单聚合中，Order 是聚合根，OrderItem 只能通过 Order 来操作（不能直接 CRUD OrderItem），聚合内的变更通过聚合根来保证业务规则。

- **领域事件（Domain Event）**：领域内发生重要事情时发出事件，比如 `OrderPaidEvent`，监听者（积分服务、发货服务）独立响应。

- **仓储（Repository）**：领域对象的持久化抽象，OrderRepository 接口定义业务查询（`findByUserId`），具体实现可以是 MyBatis 或 JPA，上层不关心。

**说明**：DDD 概念不要照本宣科背书，要结合项目说你是如何区分"领域层"和"基础设施层"的，如何用聚合根保证数据一致性的，这比背概念更有说服力。

###### 5. 微服务拆分的原则是什么？

**答：**

微服务拆分是个"分"的艺术，拆得太细维护成本爆炸，拆得太粗失去微服务的意义。

**拆分原则**：

**① 单一职责原则**：一个服务只做一件事，对应一个清晰的业务能力（用户服务管用户，订单服务管订单）。

**② DDD 限界上下文**：以业务领域为边界，每个限界上下文是一个服务候选。不同上下文里相同词语的含义可能不同（商品目录里的"商品" vs 仓储里的"货品"）。

**③ 数据自治**：每个服务拥有自己的数据库，服务间不直接 JOIN，通过接口或 MQ 交换数据。

**④ 独立部署原则**：一个服务的修改和部署不需要其他服务同时变更。如果两个服务总是要一起发布，说明边界划分不合理，应该合并。

**⑤ 团队自治（Conway's Law）**：服务边界应该和团队边界对齐，一个团队负责一个或几个服务，减少跨团队协调。

**常见反模式**：
- **过度拆分**：把一个简单的 CRUD 服务拆成 3 个，每次改需求要改 3 个服务 + 接口协调，纯粹增加复杂度
- **共享数据库**：多个服务操作同一个库，没有实现数据自治，本质还是单体
- **服务链太长**：A 调 B 调 C 调 D，链路过长，延迟叠加，任何一个超时都影响全链路

###### 6. 如何处理服务之间的依赖关系？

**答：**

微服务间的依赖关系处理不好，会形成"分布式单体"（microservices but still monolith）：

**① 明确依赖方向，避免循环依赖**

建立服务依赖层次（如：网关 → BFF → 业务服务 → 基础服务），上层依赖下层，下层不依赖上层。如果发现 A 和 B 互相调用，通常说明边界划分有问题，考虑提取公共部分到一个新服务 C，或者用事件解耦。

**② 接口版本管理**

下游服务修改接口时，必须向下兼容（新增字段而不是修改/删除），用 URL 版本号（`/v2/orders`）发布新版本，旧版本维护一段时间后下线。消费方可以按自己节奏迁移。

**③ 服务契约测试**

用 Pact（消费者驱动契约测试）：消费方定义它期望的接口格式，提供方 CI 跑契约测试验证接口兼容性，防止提供方改接口破坏消费方。

**④ 异步解耦减少直接依赖**

能用 MQ 解耦的就用 MQ，下单服务不直接调库存服务，而是发 `OrderCreatedEvent`，库存服务订阅并消费。这样库存服务暂时不可用，下单主流程不受影响。

###### 7. 如果让你重新设计这个项目，你会怎么做？

**答：**

这是一道开放性反思题，面试官想看你是否有批判性思维，能不能说出项目的不足并提出改进思路。不要说"什么都好，没有问题"，这不真实。

**参考答法（结合项目实际）**：

如果重新设计，我会在以下几个方面做不同的选择：

**① 更早引入 DDD**

项目初期用了贫血模型（Entity 只有 getter/setter，业务逻辑全堆在 Service），随着业务增长，Service 越来越臃肿，难以维护。如果重来，会用 DDD 的聚合根 + 领域服务，把业务规则内聚到领域对象里，Service 只做协调。

**② 缓存设计更系统**

现在的缓存是"哪里慢加哪里"，比较零散，没有统一的缓存管理。重新设计会在框架层统一处理缓存失效（Canal + MQ 异步），而不是每个 Service 自己在写 DB 后手动删缓存。

**③ 分库分表提前规划**

订单表上线后才做分库分表，迁移成本很高（双写、数据验证、切流）。如果重来，在数据模型设计阶段就考虑分片键（user_id），即使一开始不分，代码层面也要准备好。

**④ 更完善的可观测性**

早期日志格式不统一、缺乏 traceId，出问题很难排查。重新设计会从第一天就统一日志格式（结构化 JSON + traceId），接入 SkyWalking，让可观测性是内建的，而不是事后补救的。
