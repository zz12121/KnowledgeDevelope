# 测试相关类

---

###### 1. 项目中如何进行单元测试？

**答：**

我们用 JUnit 5 + Mockito 做单元测试，遵循 AAA（Arrange-Act-Assert）结构：

**基本示例**：
```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @InjectMocks
    private OrderService orderService;

    @Mock
    private OrderMapper orderMapper;

    @Mock
    private StockService stockService;

    @Test
    @DisplayName("下单成功：库存充足时创建订单")
    void createOrder_success_when_stock_sufficient() {
        // Arrange（准备数据和 Mock）
        CreateOrderRequest req = new CreateOrderRequest(1L, 2);
        when(stockService.checkAndDeduct(1L, 2)).thenReturn(true);
        when(orderMapper.insert(any())).thenReturn(1);

        // Act（执行）
        OrderVO result = orderService.createOrder(req);

        // Assert（断言）
        assertNotNull(result);
        assertEquals(OrderStatus.PENDING, result.getStatus());
        verify(stockService).checkAndDeduct(1L, 2);  // 验证方法被调用
    }

    @Test
    @DisplayName("下单失败：库存不足时抛出异常")
    void createOrder_fail_when_stock_insufficient() {
        when(stockService.checkAndDeduct(anyLong(), anyInt())).thenReturn(false);

        assertThrows(BusinessException.class, () -> orderService.createOrder(new CreateOrderRequest(1L, 2)));
    }
}
```

**原则（FIRST）**：
- **Fast**：单元测试要快，不连 DB，不调外部服务，全 Mock
- **Independent**：测试间互不依赖，可以单独运行
- **Repeatable**：任何环境运行结果一致
- **Self-validating**：有明确的 pass/fail 断言
- **Timely**：写代码的同时或之前写测试（TDD）

**测试覆盖率**：Service 层、工具类目标 > 80%，用 JaCoCo 生成覆盖率报告，CI 强制门禁。

###### 2. 如何进行集成测试？

**答：**

集成测试验证多个组件协同工作，包括真实的数据库、Redis 等中间件，比单元测试更接近真实环境。

**工具**：Spring Boot Test + Testcontainers

**Testcontainers 动态启动容器**：
```java
@SpringBootTest
@Testcontainers
class OrderServiceIntegrationTest {

    @Container
    static MySQLContainer<?> mysql = new MySQLContainer<>("mysql:8.0")
        .withDatabaseName("test")
        .withUsername("root")
        .withPassword("test");

    @Container
    static GenericContainer<?> redis = new GenericContainer<>("redis:7")
        .withExposedPorts(6379);

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", mysql::getJdbcUrl);
        registry.add("spring.redis.host", redis::getHost);
        registry.add("spring.redis.port", () -> redis.getMappedPort(6379));
    }

    @Autowired
    private OrderService orderService;

    @Test
    void createOrder_and_query_back() {
        OrderVO order = orderService.createOrder(new CreateOrderRequest(1L, 2));
        assertNotNull(orderService.getById(order.getId()));
    }
}
```

Testcontainers 每次测试启动真实 Docker 容器，测试完自动销毁，隔离性好，不污染环境。

**集成测试 vs 单元测试**：集成测试更全面但更慢（启动容器需要时间），一般跑在 CI 的特定阶段，不是每次 commit 都跑。

###### 3. 如何保证代码质量？

**答：**

代码质量要靠体系来保证，不能靠人工自律：

**① 静态代码分析（SonarQube）**

CI Pipeline 集成 SonarQube，每次 Push 自动扫描：
- Bug（空指针、资源未关闭等潜在 Bug）
- 安全漏洞（SQL 注入风险、硬编码密码）
- 代码异味（代码复杂度过高、重复代码）
- 覆盖率

设置质量门禁：新增 Bug 为 0、安全漏洞为 0、覆盖率不低于 70%，不达标则 Pipeline 失败，阻止合并。

**② Code Review**

所有代码通过 GitLab Merge Request 合并，强制要求 2 人 Review 通过。Review 重点：
- 逻辑正确性（有没有边界处理、并发安全）
- 性能（有没有 N+1、大对象）
- 安全（有没有注入风险、敏感信息泄露）
- 可维护性（命名、注释、过度复杂）

**③ 规范统一（Checkstyle / SpotBugs）**

代码风格用 Checkstyle 强制统一（Google Java Style），SpotBugs 静态分析常见 Bug 模式。

**④ 测试覆盖率门禁**

JaCoCo 集成 CI，核心 Service 层覆盖率低于阈值则构建失败。

**⑤ 童子军法则**

离开时，让代码比你来时更干净——每次修改顺手重构一点点，防止技术债务积累。

###### 4. 测试覆盖率如何？

**答：**

这道题要结合项目实际情况回答，不要说"我们没统计"，要给出数字和背后的考量。

**参考口径**：

我们用 JaCoCo 统计代码覆盖率，整体覆盖率约 75%。分层来看：

- **Service 层（核心业务逻辑）**：覆盖率 > 85%，这是最重要的一层，单元测试重点覆盖
- **Controller 层**：约 60%，通过集成测试覆盖主流程
- **Mapper 层（MyBatis）**：通过集成测试（Testcontainers + 真实 MySQL）覆盖，约 70%
- **工具类/公共组件**：> 90%，逻辑简单但使用频率高，要覆盖全面
- **DTO/VO/Entity**：一般不计入覆盖率统计（只有 getter/setter）

**覆盖率的正确认知**：100% 覆盖率不代表没有 Bug，覆盖率只是度量测试有没有执行到那行代码，而不是测试有没有验证正确性。重点是写有价值的测试（覆盖边界条件、异常场景），而不是为了追求数字写空断言的测试。

CI 强制门禁：核心模块覆盖率下降 > 5% 则构建失败，防止覆盖率持续下降。

###### 5. 如何进行接口测试？

**答：**

接口测试验证 HTTP 接口的正确性，覆盖正常流程和异常边界。

**工具选择**：

- **开发阶段**：Postman / ApiFox（手动调试接口，方便快速验证）
- **自动化测试**：Spring Boot Test + MockMvc（代码级集成测试）
- **接口文档联动**：Springdoc + OpenAPI，接口文档自动生成，可在 Swagger UI 直接测试

**MockMvc 接口测试示例**：
```java
@WebMvcTest(OrderController.class)
class OrderControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private OrderService orderService;

    @Test
    void createOrder_returns_201() throws Exception {
        OrderVO mockOrder = new OrderVO(1L, OrderStatus.PENDING);
        when(orderService.createOrder(any())).thenReturn(mockOrder);

        mockMvc.perform(post("/orders")
                .contentType(MediaType.APPLICATION_JSON)
                .content("{\"productId\":1,\"quantity\":2}")
                .header("X-User-Id", "12345"))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.data.status").value("PENDING"));
    }

    @Test
    void createOrder_returns_400_when_quantity_invalid() throws Exception {
        mockMvc.perform(post("/orders")
                .contentType(MediaType.APPLICATION_JSON)
                .content("{\"productId\":1,\"quantity\":-1}"))
            .andExpect(status().isBadRequest());
    }
}
```

**测试用例覆盖**：
- 正常路径：参数合法，验证返回数据结构和状态码
- 参数校验：必填字段缺失、格式不合法
- 鉴权：未登录、权限不足
- 业务异常：库存不足、订单不存在
