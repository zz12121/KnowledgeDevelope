###### 1. 什么是创建型设计模式，有哪些？

创建型设计模式关注的是**对象的创建方式**，目的是把对象的创建过程封装起来，让系统不依赖具体类的实例化过程，从而降低耦合度。

GoF 定义的五种创建型模式是：
- **单例模式（Singleton）**：确保全局只有一个实例
- **工厂方法模式（Factory Method）**：定义创建对象的接口，让子类决定实例化哪个类
- **抽象工厂模式（Abstract Factory）**：创建一系列相关联的对象，确保产品兼容性
- **建造者模式（Builder）**：分步骤构建复杂对象
- **原型模式（Prototype）**：通过克隆已有对象来创建新对象

---

###### 2. 什么是单例模式，核心要点是什么？

单例模式确保**一个类只有一个实例，并提供一个全局访问点**。

核心三要素：私有构造方法（禁止外部 new）、私有静态实例变量（保存唯一实例）、公共静态访问方法（提供全局入口）。

使用场景：数据库连接池、线程池、配置管理类、日志对象、ID 生成器等需要全局唯一的场景。

---

###### 3. 单例模式有哪些实现方式？各有什么特点？

**1. 饿汉式（推荐，简单可靠）**

```java
public class Singleton {
    // 类加载时立即创建，天生线程安全
    private static final Singleton INSTANCE = new Singleton();

    private Singleton() {}

    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```

优点：实现简单，线程安全，无同步开销。缺点：如果实例一直没被用到，会白白占内存。

**2. 懒汉式（线程不安全，不推荐用）**

```java
public class Singleton {
    private static Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton(); // 多线程下可能创建多个实例
        }
        return instance;
    }
}
```

**3. 双重检查锁 DCL（延迟加载 + 线程安全，推荐）**

```java
public class Singleton {
    // volatile 必须加，防止指令重排序导致返回未初始化完的对象
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                    // 第一次检查，避免不必要的同步
            synchronized (Singleton.class) {
                if (instance == null) {            // 第二次检查，防止多线程重复创建
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

`volatile` 关键字的必要性：`instance = new Singleton()` 这行代码编译后实际是三步（分配内存、初始化对象、赋值引用），JVM 可能指令重排序成"分配内存→赋值引用→初始化对象"，如果不加 volatile，另一个线程可能拿到一个地址非空但对象未初始化完的引用。

**4. 静态内部类（延迟加载 + 线程安全，优雅）**

```java
public class Singleton {
    private Singleton() {}

    // 静态内部类，只有在 getInstance() 第一次被调用时才会加载
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

利用 JVM 类加载机制保证线程安全，延迟加载，没有 synchronized 开销。是实际工程中最优雅的写法。

**5. 枚举（最强防护）**

```java
public enum Singleton {
    INSTANCE;

    public void doSomething() { /* 业务方法 */ }
}

// 使用
Singleton.INSTANCE.doSomething();
```

JVM 天然保证枚举值的唯一性，而且能防序列化破坏和反射破坏，是 Effective Java 推荐的写法。唯一缺点是不支持延迟加载，且代码风格对部分团队来说不够直观。

---

###### 4. 懒汉式和饿汉式单例模式的区别？

饿汉式在类加载时就创建实例，不管你用不用都提前准备好了，天生线程安全，但如果实例很大且系统运行时根本没用到，就是浪费内存。

懒汉式延迟到第一次调用时才创建，节省内存，但基础版本线程不安全，需要用 DCL 或静态内部类的方式来保证线程安全。

选择建议：实例小且一定会用，用饿汉式；实例大或创建耗时，用静态内部类方式的懒汉式。

###### 5. 如何防止单例模式被反射和序列化破坏？

**反射破坏的防御**

反射可以调用 `setAccessible(true)` 来访问私有构造方法，绕过私有化限制：

```java
// 饿汉式的防御方案——在构造方法里检查
public class Singleton {
    private static final Singleton INSTANCE = new Singleton();

    private Singleton() {
        if (INSTANCE != null) {
            throw new RuntimeException("禁止通过反射创建单例！");
        }
    }

    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```

**序列化破坏的防御**

如果单例实现了 `Serializable`，反序列化时会创建新对象，可以加 `readResolve()` 方法解决：

```java
public class Singleton implements Serializable {
    private static final Singleton INSTANCE = new Singleton();

    private Singleton() {}

    public static Singleton getInstance() { return INSTANCE; }

    // 反序列化时，JVM 会调用这个方法，返回现有实例而不是新建
    protected Object readResolve() {
        return INSTANCE;
    }
}
```

**最强方案：枚举**

枚举类型由 JVM 保证每个值唯一：反射机制遇到枚举构造方法时会直接抛出 `IllegalArgumentException`，序列化机制对枚举也有特殊处理（通过枚举名查找已有实例），两种攻击都能天然防御。

###### 6. Spring 中的单例 Bean 和设计模式中的单例有什么区别？

设计模式中的单例是**进程级别的唯一性**，同一个 JVM 进程里这个类只有一个对象，通过静态变量保证。

Spring 中的单例 Bean 是**容器级别的唯一性**，同一个 ApplicationContext 中每个 Bean 定义只有一个实例，通过 `DefaultSingletonBeanRegistry` 内部的 `ConcurrentHashMap` 来维护。如果你创建了多个 ApplicationContext，同一个 Bean 会有多个实例。

另外，Spring 单例 Bean 是可以通过反射创建的，设计模式中的单例一般防止了这一点。

###### 7. 说说工厂模式的三种形式

工厂模式有三个层次，按复杂度递增：

**简单工厂（不在 GoF 23 种里，但最常用）**

一个工厂类根据参数决定创建哪种产品：

```java
public class LoggerFactory {
    public static Logger createLogger(String type) {
        switch (type) {
            case "file":    return new FileLogger();
            case "console": return new ConsoleLogger();
            case "db":      return new DatabaseLogger();
            default: throw new IllegalArgumentException("Unknown logger type: " + type);
        }
    }
}
```

简单，但违反开闭原则——新增产品类型必须修改工厂类。

**工厂方法模式（GoF 正式成员）**

把工厂本身抽象成接口，每个产品对应一个工厂实现，新增产品只需新增工厂类：

```java
public interface LoggerFactory {
    Logger createLogger();
}

public class FileLoggerFactory implements LoggerFactory {
    public Logger createLogger() { return new FileLogger(); }
}

public class ConsoleLoggerFactory implements LoggerFactory {
    public Logger createLogger() { return new ConsoleLogger(); }
}
```

**抽象工厂模式**

工厂方法只创建一种产品，抽象工厂创建一族相关产品，下面单独讲。

---

###### 8. 什么是工厂方法模式，适用什么场景？

工厂方法模式定义一个创建对象的接口，但让子类决定实例化哪个类，把实例化推迟到子类。

核心结构：Product 产品接口 → ConcreteProduct 具体产品；Creator 抽象工厂（声明工厂方法）→ ConcreteCreator 具体工厂（实现工厂方法，返回具体产品）。

适用场景：不知道要创建的具体类；希望扩展时只加代码不改代码；需要让子类决定创建什么对象。

```java
// 产品接口
public interface Notification {
    void send(String message, String recipient);
}

// 具体产品
public class EmailNotification implements Notification {
    public void send(String message, String recipient) {
        System.out.println("发送邮件到 " + recipient + ": " + message);
    }
}

public class SMSNotification implements Notification {
    public void send(String message, String recipient) {
        System.out.println("发送短信到 " + recipient + ": " + message);
    }
}

public class WeChatNotification implements Notification {
    public void send(String message, String recipient) {
        System.out.println("发送微信消息到 " + recipient + ": " + message);
    }
}

// 抽象工厂
public abstract class NotificationFactory {
    public abstract Notification createNotification();

    // 模板方法：工厂方法创建对象后，做统一处理
    public void notifyUser(String message, String recipient) {
        Notification notification = createNotification();
        notification.send(message, recipient);
    }
}

// 具体工厂
public class EmailNotificationFactory extends NotificationFactory {
    public Notification createNotification() { return new EmailNotification(); }
}

public class SMSNotificationFactory extends NotificationFactory {
    public Notification createNotification() { return new SMSNotification(); }
}

// 使用
public class OrderService {
    private NotificationFactory notificationFactory;

    public OrderService(NotificationFactory factory) {
        this.notificationFactory = factory;
    }

    public void createOrder(Order order) {
        // 处理订单...
        notificationFactory.notifyUser("订单创建成功", order.getUserPhone());
    }
}
```

###### 9. 什么是抽象工厂模式，和工厂方法有什么区别？

抽象工厂模式提供一个创建**一系列相关或相互依赖对象**的接口，而无需指定它们的具体类。它是工厂方法的升级版，专门处理"产品家族"的创建问题。

工厂方法模式针对一种产品，抽象工厂模式针对一族产品。举个例子：工厂方法是"生产一种按钮"，抽象工厂是"生产一套 UI 主题的所有控件（按钮+输入框+下拉框），并确保这套控件风格一致"。

实际场景——不同风格 UI 组件的创建：

```java
// 抽象产品
public interface Button { void render(); void onClick(); }
public interface TextField { void render(); String getValue(); }
public interface Checkbox { void render(); boolean isChecked(); }

// macOS 风格产品族
public class MacButton implements Button {
    public void render() { System.out.println("渲染 macOS 风格按钮"); }
    public void onClick() { System.out.println("macOS 按钮点击效果"); }
}
public class MacTextField implements TextField {
    public void render() { System.out.println("渲染 macOS 风格文本框"); }
    public String getValue() { return "Mac风格输入"; }
}

// Windows 风格产品族
public class WindowsButton implements Button {
    public void render() { System.out.println("渲染 Windows 风格按钮"); }
    public void onClick() { System.out.println("Windows 按钮点击效果"); }
}
public class WindowsTextField implements TextField {
    public void render() { System.out.println("渲染 Windows 风格文本框"); }
    public String getValue() { return "Windows风格输入"; }
}

// 抽象工厂接口：创建整套 UI 产品族
public interface UIFactory {
    Button createButton();
    TextField createTextField();
    Checkbox createCheckbox();
}

// 具体工厂：macOS 工厂，创建的都是 macOS 风格，保证兼容性
public class MacUIFactory implements UIFactory {
    public Button createButton()     { return new MacButton(); }
    public TextField createTextField() { return new MacTextField(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}

// 具体工厂：Windows 工厂
public class WindowsUIFactory implements UIFactory {
    public Button createButton()     { return new WindowsButton(); }
    public TextField createTextField() { return new WindowsTextField(); }
    public Checkbox createCheckbox() { return new WindowsCheckbox(); }
}

// 应用层——依赖抽象工厂，不关心是哪个平台
public class Application {
    private final UIFactory factory;
    private Button button;
    private TextField textField;

    public Application(UIFactory factory) {
        this.factory = factory;
    }

    public void buildUI() {
        this.button = factory.createButton();
        this.textField = factory.createTextField();
        button.render();
        textField.render();
    }
}

// 根据平台选择对应工厂
UIFactory factory = System.getProperty("os.name").contains("Mac")
        ? new MacUIFactory() : new WindowsUIFactory();
Application app = new Application(factory);
app.buildUI();
```

工厂方法的扩展方向是"垂直"的（增加新产品种类），抽象工厂的扩展方向是"水平"的（增加新产品家族）。抽象工厂新增产品族容易（加一个工厂实现类），但新增产品种类很难（需要改所有工厂接口和实现）。

###### 10. 什么是建造者模式，适用什么场景？

建造者模式将**复杂对象的构建过程和它的表示分离**，使得同样的构建过程可以创建出不同的表示。

最直观的场景是创建一个有很多可选参数的复杂对象，用构造方法的话要写一堆重载，用 Setter 的话又无法保证对象一次构造完整。建造者模式的链式调用方式解决了这个问题：

```java
// 复杂的 HTTP 请求对象
public class HttpRequest {
    private final String url;
    private final String method;
    private final Map<String, String> headers;
    private final String body;
    private final int timeout;
    private final int retryCount;
    private final boolean followRedirects;

    private HttpRequest(Builder builder) {
        this.url = builder.url;
        this.method = builder.method;
        this.headers = builder.headers;
        this.body = builder.body;
        this.timeout = builder.timeout;
        this.retryCount = builder.retryCount;
        this.followRedirects = builder.followRedirects;
    }

    public static class Builder {
        private final String url;            // 必填
        private String method = "GET";       // 选填，有默认值
        private Map<String, String> headers = new HashMap<>();
        private String body;
        private int timeout = 3000;
        private int retryCount = 0;
        private boolean followRedirects = true;

        public Builder(String url) {
            if (url == null || url.isEmpty()) {
                throw new IllegalArgumentException("URL不能为空");
            }
            this.url = url;
        }

        public Builder method(String method) {
            this.method = method;
            return this;
        }

        public Builder header(String key, String value) {
            this.headers.put(key, value);
            return this;
        }

        public Builder body(String body) {
            this.body = body;
            return this;
        }

        public Builder timeout(int timeout) {
            this.timeout = timeout;
            return this;
        }

        public Builder retryCount(int retryCount) {
            this.retryCount = retryCount;
            return this;
        }

        public Builder followRedirects(boolean followRedirects) {
            this.followRedirects = followRedirects;
            return this;
        }

        public HttpRequest build() {
            // 可以在这里做整体校验
            if ("POST".equals(method) && (body == null || body.isEmpty())) {
                throw new IllegalStateException("POST 请求必须有请求体");
            }
            return new HttpRequest(this);
        }
    }
}

// 使用起来非常清晰
HttpRequest request = new HttpRequest.Builder("https://api.example.com/users")
        .method("POST")
        .header("Content-Type", "application/json")
        .header("Authorization", "Bearer token123")
        .body("{\"name\":\"张三\"}")
        .timeout(5000)
        .retryCount(3)
        .build();
```

建造者模式适合：创建对象需要很多参数（特别是有大量可选参数时）、需要对构建过程进行校验、对象需要不可变（所有属性在 `build()` 后不可改变）。Lombok 的 `@Builder` 注解本质上就是自动生成这套代码。

###### 11. 建造者模式和工厂模式的区别？

工厂模式关注的是**创建哪种对象**（What）。你告诉工厂一个类型，工厂返回对应的产品，创建过程是一步完成的。

建造者模式关注的是**如何构建这个对象**（How）。它通过多步骤分步骤构建，你可以控制每一步的细节，最终 `build()` 得到产品。

简单比喻：工厂模式是去餐厅点一份"宫保鸡丁套餐"，厨房直接端上来；建造者模式是去自助餐，自己选主食、配菜、汤、甜点，最后端走一盘你自己配的餐。

两者经常配合使用：用抽象工厂决定创建哪个系列的产品，再用建造者控制每个产品的构建细节。

###### 12. 什么是原型模式，Java 如何实现？

原型模式通过**克隆已有对象**来创建新对象，而不是通过 new 关键字重新初始化。

使用场景：对象创建成本很高（比如需要从数据库或远程服务加载数据初始化），但新对象和已有对象差别不大，复制一份再修改比重新创建更高效。

Java 实现通过 `Cloneable` 接口和 `clone()` 方法：

```java
public class UserProfile implements Cloneable {
    private String name;
    private String email;
    private List<String> roles;
    private Address address;

    // 浅拷贝：基本类型会独立，但 roles 和 address 仍然是同一个引用
    @Override
    public UserProfile clone() {
        try {
            return (UserProfile) super.clone();
        } catch (CloneNotSupportedException e) {
            throw new AssertionError("不可能发生", e);
        }
    }

    // 深拷贝：所有引用类型都创建新对象
    public UserProfile deepClone() {
        try {
            UserProfile copy = (UserProfile) super.clone();
            copy.roles = new ArrayList<>(this.roles);     // 独立的 List
            copy.address = this.address.clone();          // 独立的 Address
            return copy;
        } catch (CloneNotSupportedException e) {
            throw new AssertionError("不可能发生", e);
        }
    }
}

// 实际使用场景：游戏中创建大量相似的 NPC
public class NPCFactory {
    private static final Map<String, NPC> prototypes = new HashMap<>();

    static {
        // 预加载原型（可能涉及复杂初始化）
        prototypes.put("guard", new GuardNPC("守卫", 100, "patrol"));
        prototypes.put("merchant", new MerchantNPC("商人", 50, "sell"));
    }

    public static NPC createNPC(String type) {
        NPC prototype = prototypes.get(type);
        if (prototype == null) throw new IllegalArgumentException("未知NPC类型: " + type);
        return prototype.clone(); // 克隆原型，而不是重新创建
    }
}
```

###### 13. 深拷贝和浅拷贝的区别？

浅拷贝只复制对象本身，引用类型字段复制的是引用地址，原对象和副本共享同一个引用对象。改了副本里的引用对象，原对象也会受影响。

深拷贝不仅复制对象本身，还递归地复制所有引用类型，原对象和副本完全独立，互不影响。

```java
// 验证浅拷贝的问题
UserProfile original = new UserProfile("张三", List.of("admin"));
UserProfile shallow = original.clone(); // 浅拷贝

shallow.getRoles().add("superuser"); // 这一行同时修改了 original 的 roles！
System.out.println(original.getRoles()); // 输出 [admin, superuser] ← 原对象被污染

// 深拷贝不会有这个问题
UserProfile deep = original.deepClone();
deep.getRoles().add("superuser");
System.out.println(original.getRoles()); // 仍然输出 [admin] ← 原对象安全
```

实现深拷贝除了手动 `clone()` 递归以外，还可以用序列化/反序列化的方式（适合对象图很深的场景）：

```java
public static <T extends Serializable> T deepCopyBySerialization(T obj) throws Exception {
    ByteArrayOutputStream bos = new ByteArrayOutputStream();
    new ObjectOutputStream(bos).writeObject(obj);
    ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
    return (T) new ObjectInputStream(bis).readObject();
}
```

选择建议：对象只有基本类型字段，或引用的是不可变对象（如 String），用浅拷贝就够；有可变引用且需要完全独立时，用深拷贝。
