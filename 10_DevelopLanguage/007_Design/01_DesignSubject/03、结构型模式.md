###### 1. 什么是结构型设计模式，有哪些？

结构型设计模式关注的是**如何将类和对象组合成更大的结构**，同时保持这些结构的灵活性和高效性。

GoF 定义的七种结构型模式是：
- **代理模式（Proxy）**：为其他对象提供代理以控制访问
- **装饰器模式（Decorator）**：动态地为对象添加额外职责
- **适配器模式（Adapter）**：将不兼容的接口转换成客户端期望的接口
- **门面模式（Facade）**：为子系统提供统一的简化接口
- **桥接模式（Bridge）**：将抽象部分和实现部分分离，使它们独立变化
- **组合模式（Composite）**：将对象组合成树形结构，统一处理单个和组合对象
- **享元模式（Flyweight）**：通过共享技术减少大量细粒度对象的内存消耗

---

###### 2. 说说你对代理模式的理解

代理模式是为其他对象提供一种代理来**控制对这个对象的访问**，在不改变目标对象的前提下，在访问前后加入自定义逻辑。

代理对象和目标对象实现同一个接口，对客户端来说代理和真实对象是透明的，客户端感知不到背后换了个代理。

常见使用场景：AOP 切面（事务、日志、权限）、RPC 远程调用的本地代理、延迟加载（虚拟代理）、缓存代理、访问控制代理。

从 JVM 层面来说，静态代理在编译期就生成了代理类，动态代理在运行时动态生成字节码。

---

###### 3. 静态代理和动态代理的区别？

静态代理是手动编写代理类，每个目标类需要一个对应的代理类，接口方法一多，代理类里就一堆样板代码。接口一旦变更，所有代理类都要改。

动态代理是运行时动态生成代理类，一个 `InvocationHandler` 可以代理任意多个类，代码复用率高。Spring AOP 就是基于动态代理实现的。

本质区别：静态代理是编译期的硬绑定关系，动态代理是运行时的灵活生成。

---

###### 4. JDK 动态代理和 CGLIB 代理的区别？

JDK 动态代理基于接口实现，通过 `Proxy.newProxyInstance()` 生成代理类，底层使用反射。目标类必须实现至少一个接口，代理类继承了 `Proxy` 并实现了目标接口。

CGLIB 代理基于继承实现，通过 ASM 字节码框架在运行时生成目标类的子类，重写非 final 方法来插入增强逻辑。不需要接口，但无法代理 final 类和 final 方法。

```java
// JDK 动态代理示例
public interface UserService {
    User findById(Long id);
    void save(User user);
}

public class UserServiceImpl implements UserService {
    public User findById(Long id) { /* 查数据库 */ return new User(); }
    public void save(User user) { /* 写数据库 */ }
}

// InvocationHandler 实现（可以代理任意实现了接口的类）
public class LoggingInvocationHandler implements InvocationHandler {
    private final Object target;

    public LoggingInvocationHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        long start = System.currentTimeMillis();
        System.out.println("开始执行: " + method.getName());
        try {
            Object result = method.invoke(target, args);
            System.out.println("执行成功，耗时: " + (System.currentTimeMillis() - start) + "ms");
            return result;
        } catch (Exception e) {
            System.out.println("执行异常: " + e.getMessage());
            throw e;
        }
    }
}

// 创建代理
UserService proxy = (UserService) Proxy.newProxyInstance(
    UserServiceImpl.class.getClassLoader(),
    new Class[]{UserService.class},
    new LoggingInvocationHandler(new UserServiceImpl())
);
proxy.findById(1L);

// CGLIB 代理示例（需要引入 cglib 或 spring-core 依赖）
public class CglibProxy implements MethodInterceptor {
    @Override
    public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
        System.out.println("CGLIB 前置处理: " + method.getName());
        Object result = proxy.invokeSuper(obj, args); // 调用父类（目标类）方法
        System.out.println("CGLIB 后置处理: " + method.getName());
        return result;
    }
}

// 没有接口的类也能代理
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(UserServiceImpl.class);
enhancer.setCallback(new CglibProxy());
UserServiceImpl proxied = (UserServiceImpl) enhancer.create();
```

Spring 的选择策略：目标类实现了接口，默认用 JDK 动态代理；目标类没有实现接口，自动切换到 CGLIB。从 Spring Boot 2.0 开始，默认全部使用 CGLIB（通过 `proxyTargetClass = true` 配置），避免 JDK 代理的一些限制。

###### 5. 代理模式和装饰器模式有什么区别？

两者结构上确实很像，都持有目标对象的引用，都实现了相同的接口。区别在于**设计意图**。

代理模式的重点是**访问控制**，代理本身不增强目标对象的核心能力，而是在访问目标对象的过程中附加一些与目标功能无关的逻辑，比如权限校验、日志记录、事务管理。代理通常对客户端透明，客户端不知道自己在和代理交互。

装饰器模式的重点是**功能增强**，装饰器让对象具备了更强的能力，而且装饰是由客户端主动组装的，客户端清楚地知道自己在给对象加功能。

一句话记住：代理是"让人帮你干你不关心的杂事"，装饰器是"给自己装备升级武器"。

---

###### 6. 装饰器模式是什么，如何实现？

装饰器模式动态地给一个对象添加额外职责，比继承更灵活，因为可以运行时动态组合多个装饰器。

核心结构：Component（抽象组件，定义接口）→ ConcreteComponent（具体组件，基础实现）→ Decorator（装饰器抽象类，持有 Component 引用）→ ConcreteDecorator（具体装饰器，添加具体功能）。

实际场景——给咖啡添加配料计算价格：

```java
// 抽象组件
public interface Coffee {
    String getDescription();
    double getCost();
}

// 具体组件：黑咖啡
public class BlackCoffee implements Coffee {
    public String getDescription() { return "黑咖啡"; }
    public double getCost() { return 10.0; }
}

// 装饰器抽象类
public abstract class CoffeeDecorator implements Coffee {
    protected Coffee coffee; // 持有被装饰的 Coffee 引用

    public CoffeeDecorator(Coffee coffee) {
        this.coffee = coffee;
    }

    public String getDescription() { return coffee.getDescription(); }
    public double getCost() { return coffee.getCost(); }
}

// 具体装饰器：加牛奶
public class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee coffee) { super(coffee); }

    @Override
    public String getDescription() { return coffee.getDescription() + ", 牛奶"; }

    @Override
    public double getCost() { return coffee.getCost() + 3.0; }
}

// 具体装饰器：加糖
public class SugarDecorator extends CoffeeDecorator {
    public SugarDecorator(Coffee coffee) { super(coffee); }

    @Override
    public String getDescription() { return coffee.getDescription() + ", 糖"; }

    @Override
    public double getCost() { return coffee.getCost() + 1.0; }
}

// 具体装饰器：加奶油
public class CreamDecorator extends CoffeeDecorator {
    public CreamDecorator(Coffee coffee) { super(coffee); }

    @Override
    public String getDescription() { return coffee.getDescription() + ", 奶油"; }

    @Override
    public double getCost() { return coffee.getCost() + 5.0; }
}

// 客户端：自由组合，运行时决定装饰哪些
Coffee myCoffee = new BlackCoffee();               // 黑咖啡 10.0
myCoffee = new MilkDecorator(myCoffee);            // 加牛奶 +3.0
myCoffee = new SugarDecorator(myCoffee);           // 加糖 +1.0
myCoffee = new CreamDecorator(myCoffee);           // 加奶油 +5.0
System.out.println(myCoffee.getDescription());     // 黑咖啡, 牛奶, 糖, 奶油
System.out.println(myCoffee.getCost());            // 19.0
```

Java IO 里的 `BufferedInputStream(new FileInputStream(...))` 就是这个模式。

###### 7. 了解适配器模式吗？有哪些实现方式？

适配器模式将一个类的接口转换成客户端期望的另一个接口，解决两个不兼容接口之间的协作问题。就像电源适配器，让不同标准的插头能用同一个插座。

有两种实现方式：

**类适配器（继承方式）**

```java
// 现有的旧日志系统（Adaptee）——接口和你想要的不一样
public class OldLogger {
    public void writeLog(String level, String message) {
        System.out.println("[" + level + "] " + message);
    }
}

// 你的新系统希望用的接口（Target）
public interface Logger {
    void info(String message);
    void error(String message);
    void warn(String message);
}

// 类适配器：继承旧日志系统 + 实现新接口
public class LoggerClassAdapter extends OldLogger implements Logger {
    public void info(String message)  { writeLog("INFO", message); }
    public void error(String message) { writeLog("ERROR", message); }
    public void warn(String message)  { writeLog("WARN", message); }
}
```

缺点：Java 单继承限制，如果 Adaptee 已经有父类，这种方式就不行了。

**对象适配器（组合方式，推荐）**

```java
// 对象适配器：组合旧日志系统，更灵活
public class LoggerObjectAdapter implements Logger {
    private final OldLogger oldLogger; // 持有 Adaptee 引用

    public LoggerObjectAdapter(OldLogger oldLogger) {
        this.oldLogger = oldLogger;
    }

    public void info(String message)  { oldLogger.writeLog("INFO", message); }
    public void error(String message) { oldLogger.writeLog("ERROR", message); }
    public void warn(String message)  { oldLogger.writeLog("WARN", message); }
}

// 实际场景：整合第三方支付接口
public class AlipayAdapter implements PaymentGateway {
    private final AlipayClient alipayClient; // 支付宝 SDK（Adaptee）

    public AlipayAdapter(AlipayClient alipayClient) {
        this.alipayClient = alipayClient;
    }

    @Override
    public PaymentResult pay(PaymentRequest request) {
        // 把统一的 PaymentRequest 转换成支付宝 SDK 的参数格式
        AlipayTradePayRequest alipayRequest = new AlipayTradePayRequest();
        alipayRequest.setOutTradeNo(request.getOrderId());
        alipayRequest.setTotalAmount(request.getAmount().toString());
        AlipayTradePayResponse response = alipayClient.execute(alipayRequest);
        // 把支付宝的返回结果转换成统一的 PaymentResult
        return new PaymentResult(response.isSuccess(), response.getTradeNo());
    }
}
```

Spring MVC 中的 `HandlerAdapter` 就是适配器模式的经典应用——把不同形式的 Handler（`@Controller` 方法、`HttpRequestHandler` 实现类等）适配成统一的处理接口。

###### 8. 适配器模式和装饰器模式的区别？

适配器主要是解决**接口不兼容**的问题，把一个现有的接口转换成另一种形式，被适配的类通常和目标接口没什么关系。

装饰器主要是**增强功能**，装饰器和被装饰对象实现的是同一个接口，目的是在保持接口不变的情况下动态扩展功能。

适配器是"翻译官"（不同语言的接口之间互译），装饰器是"增强剂"（同一语言但功能更强了）。

###### 9. 什么是门面模式（外观模式）？

门面模式为子系统中一组复杂的接口提供一个简单统一的入口，隐藏子系统的复杂性，让客户端更容易使用。

实际场景——订单下单流程（需要协调多个子系统）：

```java
// 子系统1：库存服务
public class InventoryService {
    public boolean checkStock(Long productId, int quantity) { /* 检查库存 */ return true; }
    public void reduceStock(Long productId, int quantity) { /* 扣减库存 */ }
}

// 子系统2：支付服务
public class PaymentService {
    public PaymentResult processPayment(Long orderId, BigDecimal amount) { /* 处理支付 */ return new PaymentResult(true); }
}

// 子系统3：物流服务
public class ShippingService {
    public String createShipment(Long orderId) { /* 创建运单 */ return "SF-20240101"; }
}

// 子系统4：通知服务
public class NotificationService {
    public void sendOrderConfirmation(Long userId, Long orderId) { /* 发送通知 */ }
}

// 门面类：把这一套复杂的子系统调用封装成一个简单的接口
@Service
public class OrderFacade {
    private final InventoryService inventoryService;
    private final PaymentService paymentService;
    private final ShippingService shippingService;
    private final NotificationService notificationService;

    public OrderFacade(InventoryService inventoryService,
                       PaymentService paymentService,
                       ShippingService shippingService,
                       NotificationService notificationService) {
        this.inventoryService = inventoryService;
        this.paymentService = paymentService;
        this.shippingService = shippingService;
        this.notificationService = notificationService;
    }

    // 客户端只调用这一个方法，门面内部协调所有子系统
    public OrderResult placeOrder(Long userId, Long productId, int quantity, BigDecimal amount) {
        // 1. 检查库存
        if (!inventoryService.checkStock(productId, quantity)) {
            return OrderResult.fail("库存不足");
        }
        // 2. 扣减库存
        inventoryService.reduceStock(productId, quantity);
        // 3. 创建订单（省略）
        Long orderId = 1001L;
        // 4. 处理支付
        PaymentResult payResult = paymentService.processPayment(orderId, amount);
        if (!payResult.isSuccess()) {
            inventoryService.reduceStock(productId, -quantity); // 回滚库存
            return OrderResult.fail("支付失败");
        }
        // 5. 创建运单
        String trackingNo = shippingService.createShipment(orderId);
        // 6. 发送通知
        notificationService.sendOrderConfirmation(userId, orderId);

        return OrderResult.success(orderId, trackingNo);
    }
}

// 客户端极其简单，不需要知道背后涉及多少个服务
OrderResult result = orderFacade.placeOrder(userId, productId, 1, price);
```

###### 10. 门面模式和代理模式的区别？

代理模式针对的是**单个对象**，代理对象和目标对象实现同一接口，目的是控制对目标对象的访问。

门面模式针对的是**一组对象（子系统）**，它不是代替某个单一对象，而是把多个子系统的调用封装成一个简单的接口，目的是降低系统复杂度。

另外，代理通常对客户端透明（客户端以为自己在和真实对象交互），门面是显式存在的（客户端知道在调门面）。

###### 11. 什么是桥接模式？

桥接模式将**抽象部分**和**实现部分**分离，使它们可以独立地变化。核心是用组合来替代多层继承，解决"抽象和实现都有变化时，继承会导致类爆炸"的问题。

场景：消息发送系统，消息类型（普通消息/紧急消息/通知消息）和发送方式（邮件/短信/微信）都在变化，如果用继承：`EmailNormalMessage`、`EmailUrgentMessage`、`SMSNormalMessage`、`SMSUrgentMessage`……类爆炸了。桥接模式把两个维度分开：

```java
// 实现部分的接口（发送渠道）
public interface MessageSender {
    void send(String recipient, String content);
}

// 具体实现：邮件发送
public class EmailSender implements MessageSender {
    public void send(String recipient, String content) {
        System.out.println("邮件发送给 " + recipient + ": " + content);
    }
}

// 具体实现：短信发送
public class SMSSender implements MessageSender {
    public void send(String recipient, String content) {
        System.out.println("短信发送给 " + recipient + ": " + content);
    }
}

// 具体实现：微信发送
public class WeChatSender implements MessageSender {
    public void send(String recipient, String content) {
        System.out.println("微信发送给 " + recipient + ": " + content);
    }
}

// 抽象部分（消息类型）—— 持有实现部分的引用（桥）
public abstract class Message {
    protected MessageSender sender; // 这就是"桥"

    public Message(MessageSender sender) {
        this.sender = sender;
    }

    public abstract void send(String recipient, String content);
}

// 扩展抽象：普通消息
public class NormalMessage extends Message {
    public NormalMessage(MessageSender sender) { super(sender); }

    @Override
    public void send(String recipient, String content) {
        sender.send(recipient, "[普通] " + content);
    }
}

// 扩展抽象：紧急消息
public class UrgentMessage extends Message {
    public UrgentMessage(MessageSender sender) { super(sender); }

    @Override
    public void send(String recipient, String content) {
        sender.send(recipient, "【紧急】" + content + " 请立即处理！");
    }
}

// 使用：抽象和实现自由组合，互不影响
Message emailUrgent = new UrgentMessage(new EmailSender());
emailUrgent.send("boss@company.com", "服务器宕机了");

Message wechatNormal = new NormalMessage(new WeChatSender());
wechatNormal.send("team", "下午3点开会");

// 新增一种发送方式（钉钉），只加一个实现类，不改任何抽象部分
// 新增一种消息类型（提醒消息），只加一个抽象子类，不改任何实现部分
```

JDBC 是桥接模式的经典实现：`DriverManager`（抽象）通过 `Driver` 接口（桥）连接各数据库厂商的具体实现。

###### 12. 什么是组合模式？

组合模式将对象组合成**树形结构**来表示"部分-整体"的层次关系，让客户端对单个对象和组合对象的使用方式保持一致。

实际场景——文件系统：

```java
// 抽象组件：文件系统节点
public abstract class FileSystemNode {
    protected String name;

    public FileSystemNode(String name) { this.name = name; }

    public abstract long getSize();
    public abstract void display(String indent);

    // 默认不支持子节点操作（叶子节点）
    public void add(FileSystemNode node) {
        throw new UnsupportedOperationException("叶子节点不支持add");
    }

    public void remove(FileSystemNode node) {
        throw new UnsupportedOperationException("叶子节点不支持remove");
    }
}

// 叶子节点：文件
public class File extends FileSystemNode {
    private long size;

    public File(String name, long size) {
        super(name);
        this.size = size;
    }

    @Override
    public long getSize() { return size; }

    @Override
    public void display(String indent) {
        System.out.println(indent + "📄 " + name + " (" + size + " bytes)");
    }
}

// 容器节点：文件夹
public class Directory extends FileSystemNode {
    private List<FileSystemNode> children = new ArrayList<>();

    public Directory(String name) { super(name); }

    @Override
    public void add(FileSystemNode node) { children.add(node); }

    @Override
    public void remove(FileSystemNode node) { children.remove(node); }

    @Override
    public long getSize() {
        return children.stream().mapToLong(FileSystemNode::getSize).sum(); // 递归计算
    }

    @Override
    public void display(String indent) {
        System.out.println(indent + "📁 " + name + " (" + getSize() + " bytes)");
        children.forEach(child -> child.display(indent + "  ")); // 递归显示
    }
}

// 使用：客户端对文件和目录的操作方式完全一致
Directory root = new Directory("root");
Directory src = new Directory("src");
src.add(new File("Main.java", 2048));
src.add(new File("App.java", 1024));
root.add(src);
root.add(new File("README.md", 512));
root.display("");
System.out.println("总大小: " + root.getSize() + " bytes");
```

###### 13. 什么是享元模式？

享元模式通过**共享技术**减少大量细粒度对象的内存消耗。把对象的状态分成"内在状态"（不变的、可共享的部分）和"外在状态"（变化的、不共享的部分），只共享内在状态。

实际场景——游戏中渲染大量同类型树木：

```java
// 享元对象：存储内在状态（不变的树的类型数据）
public class TreeType {
    private final String name;      // 树种
    private final String texture;   // 纹理（占大内存）
    private final String color;     // 颜色

    public TreeType(String name, String texture, String color) {
        this.name = name;
        this.texture = texture;
        this.color = color;
    }

    // 渲染时，外在状态（位置）从外部传入
    public void draw(int x, int y) {
        System.out.printf("在(%d,%d)绘制 %s 树，纹理=%s, 颜色=%s%n", x, y, name, texture, color);
    }
}

// 享元工厂：管理享元对象池，确保相同内在状态的对象只创建一份
public class TreeTypeFactory {
    private static final Map<String, TreeType> treeTypes = new HashMap<>();

    public static TreeType getTreeType(String name, String texture, String color) {
        String key = name + ":" + texture + ":" + color;
        return treeTypes.computeIfAbsent(key, k -> {
            System.out.println("创建新的树类型: " + name); // 只创建一次
            return new TreeType(name, texture, color);
        });
    }

    public static int getTypesCount() { return treeTypes.size(); }
}

// 树对象（只存储外在状态：位置）
public class Tree {
    private int x, y;                   // 外在状态：每棵树的位置不同
    private TreeType type;              // 内在状态：共享的类型信息

    public Tree(int x, int y, TreeType type) {
        this.x = x; this.y = y; this.type = type;
    }

    public void draw() { type.draw(x, y); }
}

// 使用：创建 1000 棵树，但 TreeType 只有 3 个对象
public class Forest {
    private List<Tree> trees = new ArrayList<>();

    public void plantTree(int x, int y, String name, String texture, String color) {
        TreeType type = TreeTypeFactory.getTreeType(name, texture, color); // 复用
        trees.add(new Tree(x, y, type));
    }

    public void draw() { trees.forEach(Tree::draw); }
}

Forest forest = new Forest();
Random random = new Random();
for (int i = 0; i < 1000; i++) {
    // 只有3种树类型，但位置各不相同
    String[] types = {"橡树", "松树", "枫树"};
    String type = types[i % 3];
    forest.plantTree(random.nextInt(1000), random.nextInt(1000), type, type + "纹理", "绿色");
}
System.out.println("Tree对象数: 1000，TreeType对象数: " + TreeTypeFactory.getTypesCount()); // 只有3个
```

JDK 中 `Integer.valueOf(-128 到 127)`、`String` 常量池、`Boolean.TRUE`/`Boolean.FALSE`、数据库连接池都是享元思想的应用。

###### 14. 享元模式和单例模式的区别？

单例模式目的是控制实例数量，保证全局只有一个实例，一个类对应一个对象。

享元模式目的是减少内存消耗，是多个共享对象，按需创建，一个类可以对应多个共享对象（但同一种内在状态只有一个）。

单例是"这个类只能有一个对象"，享元是"相同状态的对象只创建一次，然后复用"。
