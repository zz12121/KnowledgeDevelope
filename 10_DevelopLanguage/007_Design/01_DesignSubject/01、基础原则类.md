###### 1. 为什么要学习设计模式？

学设计模式的核心价值，我觉得可以从四个维度来说。

第一，设计模式是"经过时间验证的答案"。很多软件设计中反复出现的问题，前人都总结成了模式。学了之后，遇到对应场景就知道怎么做，不用自己从零摸索。比如需要对象创建和使用解耦，工厂模式就是答案；需要对一批订阅者广播消息，观察者模式就是答案。

第二，设计模式天然促进高内聚低耦合的设计。比如策略模式把变化的算法封装成独立类，工厂模式把对象创建过程收到一处，装饰器模式用组合替代继承扩展功能——这些都是在把"容易变化的部分"和"相对稳定的部分"隔开，让代码改起来不容易连锁崩。

第三，设计模式是团队沟通的共同语言。说"这里用责任链"，所有人立刻理解了整体架构，比画几页时序图还高效。

第四，读框架源码必须会。Spring 里的工厂、模板方法、代理、观察者，MyBatis 里的建造者、代理、装饰器，Netty 里的责任链，不懂模式根本读不懂这些框架是怎么设计的。

###### 2. 什么是 SOLID 原则，能整体说说吗？

SOLID 是面向对象设计最核心的五条原则，每个字母对应一条：

- **S**（Single Responsibility）— 单一职责：一个类只做一件事，只有一个理由被修改
- **O**（Open/Closed）— 开闭：对扩展开放，对修改关闭
- **L**（Liskov Substitution）— 里氏替换：子类必须能替换父类，不破坏原有行为
- **I**（Interface Segregation）— 接口隔离：接口要小，不要让客户端依赖用不到的方法
- **D**（Dependency Inversion）— 依赖倒转：依赖抽象，不依赖具体实现

这五条原则相互协作：开闭原则是目标，依赖倒转是手段，里氏替换是约束继承的前提，接口隔离是细化抽象的方法，单一职责贯穿始终。加上后面的迪米特法则和合成复用原则，一共七条，构成设计模式的理论基础。

###### 3. 什么是开闭原则？

开闭原则说的是：软件实体应该**对扩展开放，对修改关闭**。

核心思想就是，当需求变化时，应该通过新增代码来适应变化，而不是去修改已经存在的代码。被修改的代码越少，引入 bug 的风险就越低。

实现开闭原则的关键是"依赖抽象"。把系统里容易变化的部分抽象成接口或抽象类，固定下来之后不再修改。当需求变化时，只需要新增一个实现类就行了。

举个典型的例子，面积计算器：

```java
// 违反开闭原则 ——— 每增加一种形状都要改这个方法
public class AreaCalculator {
    public double calculate(Object shape) {
        if (shape instanceof Circle) {
            Circle c = (Circle) shape;
            return Math.PI * c.getRadius() * c.getRadius();
        } else if (shape instanceof Rectangle) {
            Rectangle r = (Rectangle) shape;
            return r.getWidth() * r.getHeight();
        }
        // 新增三角形？必须改这里
        throw new IllegalArgumentException("Unknown shape");
    }
}

// 遵循开闭原则 ——— 抽象稳定，扩展用新类
public interface Shape {
    double area();
}

public class Circle implements Shape {
    private double radius;
    public Circle(double radius) { this.radius = radius; }
    @Override
    public double area() { return Math.PI * radius * radius; }
}

public class Rectangle implements Shape {
    private double width, height;
    public Rectangle(double width, double height) {
        this.width = width; this.height = height;
    }
    @Override
    public double area() { return width * height; }
}

// 新增三角形：只需加一个类，不改任何已有代码
public class Triangle implements Shape {
    private double base, height;
    public Triangle(double base, double height) {
        this.base = base; this.height = height;
    }
    @Override
    public double area() { return 0.5 * base * height; }
}

// 计算器依赖抽象，永远不需要改
public class AreaCalculator {
    public double totalArea(List<Shape> shapes) {
        return shapes.stream().mapToDouble(Shape::area).sum();
    }
}
```

Spring 的 `BeanPostProcessor` 机制就是开闭原则的典型实现。Spring 容器核心逻辑不动，只需注册新的 `BeanPostProcessor` 实现类就能扩展 Bean 的处理流程。

###### 4. 什么是依赖倒转原则？

依赖倒转原则有两条规定：高层模块不应该依赖低层模块，两者都应该依赖抽象；抽象不应该依赖细节，细节应该依赖抽象。

说白了就是：**面向接口编程，不要面向实现编程**。

"倒转"这个词的意思是，传统设计里高层直接调用低层，依赖方向是自上而下的。DIP 要求把这个关系倒过来，让高层和低层都指向一个抽象接口，接口通常由高层定义，低层的具体实现反过来依赖高层定义的抽象，所以叫"倒转"。

实现方式主要是依赖注入，通过构造器注入或 Setter 注入，由外部（Spring 容器）来决定注入哪个具体实现：

```java
// 违反依赖倒转 ——— 高层直接 new 了低层的具体类
public class OrderService {
    private MySQLOrderRepository repo = new MySQLOrderRepository(); // 耦合死了
}

// 遵循依赖倒转 ——— 高层依赖抽象接口
public interface OrderRepository {
    void save(Order order);
    Order findById(Long id);
}

// 低层实现依赖于高层定义的接口
@Repository
public class MySQLOrderRepository implements OrderRepository {
    @Override
    public void save(Order order) { /* MySQL 持久化 */ }
    @Override
    public Order findById(Long id) { /* MySQL 查询 */ }
}

// 高层模块只依赖接口，不关心用 MySQL 还是 MongoDB
@Service
public class OrderService {
    private final OrderRepository orderRepository;

    @Autowired  // 由 Spring 注入具体实现
    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    public void createOrder(Order order) {
        orderRepository.save(order);
    }
}
```

这样一来，切换数据库实现只需换一个 `@Repository` 实现类，`OrderService` 完全不用动。

###### 5. 什么是单一职责原则？

单一职责原则说的是：**一个类应该只有一个引起它变化的原因**，即一个类只负责一项职责。

判断一个类是否违反了 SRP 的方法很直接：问自己"有多少个不同的业务变化会导致这个类被修改"，如果超过一个，职责就可能太多了。

职责耦合的最大问题是：一个职责的变化会影响其他职责，哪怕那部分代码完全没动，也可能出 bug。

```java
// 违反单一职责 ——— 持久化、验证、报告生成全混在一起
public class UserManager {
    public void saveUser(User user) { /* 操作数据库 */ }      // 职责1
    public boolean validateUser(User user) { /* 验证格式 */ } // 职责2
    public void sendWelcomeEmail(User user) { /* 发邮件 */ }  // 职责3
    public String generateReport(User user) { /* 生成报告 */ } // 职责4
}
// 改数据库结构，或改邮件格式，或改报告模板，都要动这一个类

// 遵循单一职责 ——— 每个类职责清晰
public class UserRepository {      // 只负责持久化
    public void save(User user) { /* ... */ }
}
public class UserValidator {       // 只负责验证
    public boolean isValid(User user) { /* ... */ }
}
public class UserEmailService {    // 只负责邮件
    public void sendWelcomeEmail(User user) { /* ... */ }
}
public class UserReportService {   // 只负责报告
    public String generateReport(User user) { /* ... */ }
}
```

Spring 中 `@Service`、`@Repository`、`@Controller` 分层本身就是单一职责原则的体现，每层只干自己那份事。

###### 6. 什么是里氏替换原则？

里氏替换原则的核心思想是：**所有引用父类的地方，都必须能透明地换成子类，程序行为不变**。

里氏替换是对继承的约束。它要求子类在扩展父类时，不能改变父类已有的行为契约，只能在父类的基础上增加新的行为。

具体来说有几条规则：子类方法的前置条件（对输入的要求）不能比父类更严格；子类方法的后置条件（对输出的承诺）不能比父类更宽松；子类不能重写父类的已实现方法抛出父类没有的异常。

最经典的违反案例是让 `Square` 继承 `Rectangle`：

```java
// 违反里氏替换 ——— 正方形继承矩形会破坏行为契约
class Rectangle {
    protected int width, height;
    public void setWidth(int w)  { this.width = w; }
    public void setHeight(int h) { this.height = h; }
    public int getArea() { return width * height; }
}

class Square extends Rectangle {
    @Override
    public void setWidth(int w) {
        super.setWidth(w);
        super.setHeight(w); // 问题：设置宽会同时改高
    }
    @Override
    public void setHeight(int h) {
        super.setHeight(h);
        super.setWidth(h);  // 问题：设置高会同时改宽
    }
}

// 这段测试代码：传 Rectangle 能通过，传 Square 就失败
void testLSP(Rectangle r) {
    r.setWidth(5);
    r.setHeight(4);
    assert r.getArea() == 20; // 传 Square 时 area = 25，断言失败！
}
```

正确的做法是让 `Square` 和 `Rectangle` 都实现一个公共的 `Shape` 接口，而不是让 `Square` 继承 `Rectangle`。继承要表达的是真实的 "is-a" 关系，不是为了代码复用就随便继承。

###### 7. 什么是接口隔离原则？

接口隔离原则要求：**客户端不应该被迫依赖它不使用的接口**，应该把大接口拆成多个专职接口。

接口隔离和单一职责的关注点不同：单一职责关注的是类，接口隔离关注的是接口。违反 SRP 的类往往也会导致它实现的接口变得臃肿，从而违反 ISP。

```java
// 违反接口隔离 ——— 一个接口塞了所有角色的方法
public interface Worker {
    void code();   // 程序员干的
    void test();   // 测试员干的
    void design(); // 架构师干的
    void deploy(); // 运维干的
}

// 后端开发被迫实现根本不干的活
public class BackendDeveloper implements Worker {
    public void code() { /* 认真写 */ }
    public void test() { throw new UnsupportedOperationException(); } // 空实现
    public void design() { throw new UnsupportedOperationException(); }
    public void deploy() { throw new UnsupportedOperationException(); }
}

// 遵循接口隔离 ——— 按角色拆分接口
public interface Coder   { void code(); }
public interface Tester  { void test(); }
public interface Designer { void design(); }
public interface Deployer { void deploy(); }

// 后端开发只实现自己需要的
public class BackendDeveloper implements Coder, Tester {
    public void code() { /* 写代码 */ }
    public void test() { /* 自测 */ }
}
```

Spring 中大量接口都很小且职责单一，比如 `BeanNameAware`、`BeanFactoryAware`、`InitializingBean`，都是非常典型的接口隔离设计——需要哪个回调，实现哪个接口，而不是实现一个巨型接口。

###### 8. 什么是迪米特法则（最少知识原则）？

迪米特法则说的是：**一个对象应该对其他对象保持最少的了解，只和直接朋友通信，不和陌生人说话**。

"朋友"指的是：当前对象本身、方法参数传入的对象、当前对象的成员变量、当前方法内部创建的对象。凡是需要通过其他对象"中转"才能访问的，就是"陌生人"，不该直接调用。

违反迪米特法则最典型的表现是"火车式调用"：`a.getB().getC().doSomething()`。

```java
// 违反迪米特法则
public class CustomerService {
    public String getCustomerCity(Long customerId) {
        Customer customer = customerRepo.findById(customerId);
        // 深入到 customer 的 address 的 city，company 是陌生人
        return customer.getAddress().getCity().getName();
    }
}

// 遵循迪米特法则 ——— 让中间对象封装细节
public class Customer {
    private Address address;

    // Customer 自己提供获取城市名的方法，外部不用深入
    public String getCityName() {
        return address.getCityName();
    }
}

public class Address {
    private City city;

    public String getCityName() {
        return city.getName(); // Address 内部处理
    }
}

public class CustomerService {
    public String getCustomerCity(Long customerId) {
        Customer customer = customerRepo.findById(customerId);
        return customer.getCityName(); // 只和直接朋友 Customer 交互
    }
}
```

迪米特法则的目的是降低耦合度。它的代价是可能引入大量的包装方法，所以要结合实际情况，不要为了遵守原则反而让代码变得更难理解。

###### 9. 什么是合成复用原则？

合成复用原则说的是：**优先使用组合或聚合，而不是继承来实现代码复用**。

继承是"白箱复用"——子类能看见父类所有的内部实现细节，耦合极深。继承关系在编译期就确定了，运行时无法改变。而组合是"黑箱复用"——只通过接口交互，被组合的对象内部对外不可见，耦合度低，且可以在运行时动态替换组合的对象。

继承的问题：父类实现一改，子类可能全崩；类层次一深，类爆炸不可避免。

Java IO 里的 `FilterInputStream` 是合成复用原则的经典案例：

```java
// 使用继承的坏处：想给 FileInputStream 加缓冲，只能这样
class BufferedFileInputStream extends FileInputStream { ... }
// 想给 ByteArrayInputStream 也加缓冲？只能再来一个
class BufferedByteArrayInputStream extends ByteArrayInputStream { ... }
// 类爆炸了...

// Java IO 实际的做法：组合（FilterInputStream 持有 InputStream 引用）
public class FilterInputStream extends InputStream {
    protected volatile InputStream in; // 组合，不是继承

    protected FilterInputStream(InputStream in) {
        this.in = in;
    }

    public int read() throws IOException {
        return in.read(); // 委托给被组合的对象
    }
}

// 于是你可以这样：给任意 InputStream 动态加缓冲
InputStream input = new BufferedInputStream(new FileInputStream("file.txt"));
// 还可以再叠一层解密
InputStream input2 = new CipherInputStream(new BufferedInputStream(new FileInputStream("file.txt")), cipher);
```

合成复用原则不是说"永远不用继承"，而是说"只有真的是 is-a 关系且符合里氏替换原则时，才考虑继承"。大多数时候，组合更灵活。

###### 10. 七大设计原则之间有什么关系？

七大原则不是孤立的，它们相互支撑，共同指向同一个目标：让系统易于扩展、不易腐化。

开闭原则是总目标，说系统应该能扩展不能被修改。实现它的手段是依赖倒转原则，通过抽象来隔离变化。里氏替换原则是继承的约束，保证用抽象的地方可以安全地替换具体实现。接口隔离原则是细化抽象的方法，让接口保持精准而不臃肿。单一职责原则确保每个模块只有一个变化原因，职责不混淆。迪米特法则降低模块间的耦合，控制知识的传播范围。合成复用原则在具体实现时优先用组合，保持灵活性。

可以这么记：**开闭是目标，倒转是手段，里氏是约束，隔离是拆分，职责是边界，迪米特是封装，合成是实现**。
