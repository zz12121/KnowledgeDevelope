###### 1. 在实际项目中如何选择合适的设计模式？

选设计模式要以"问题驱动"而不是"模式驱动"。先看清问题，再找合适的模式，而不是先想用哪个模式再找场景套。

选择思路：识别系统中"容易变化的点"在哪里。如果对象的**创建方式**会变化，考虑工厂/建造者/原型；如果**行为/算法**会变化，考虑策略/模板方法/命令；如果需要**控制访问或增强功能**，考虑代理/装饰器；如果**对象关系复杂**，考虑观察者/中介者/责任链。

避坑原则：不要为了用模式而用模式。简单 if-else 能解决的就用 if-else；能直接 new 的就直接 new；遵循 YAGNI（You Aren't Gonna Need It）——不要为"可能的未来变化"过早抽象。过度设计比不设计更难维护。

###### 2. 设计模式在 Spring 框架中的应用有哪些？

Spring 是设计模式的最佳实践范本，几乎用到了所有经典模式。

**单例模式**：Spring Bean 默认是单例的，`DefaultSingletonBeanRegistry` 用 `ConcurrentHashMap` 缓存所有单例 Bean：

```java
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);
```

**工厂模式**：`BeanFactory` 是最基础的工厂接口，`ApplicationContext` 是它的高级实现。`FactoryBean` 接口允许自定义复杂对象的创建（MyBatis 的 `SqlSessionFactoryBean` 就是这个机制）。

**模板方法模式**：`AbstractApplicationContext.refresh()` 是最经典的模板方法，定义了容器启动的 13 步骤骨架，部分步骤（如 `onRefresh()`）留给子类实现。`JdbcTemplate`、`RedisTemplate` 也是这个模式，固定了操作流程，把变化的部分（SQL/命令）通过回调接口传入：

```java
// JdbcTemplate 内部
public <T> T query(String sql, RowMapper<T> rowMapper) {
    // 1. 获取连接（固定）
    // 2. 执行 SQL（固定）
    // 3. 调用 rowMapper 处理结果（可变，由调用者传入）
    // 4. 释放资源（固定）
}
```

**代理模式**：Spring AOP 的核心实现。有接口用 JDK 动态代理（`JdkDynamicAopProxy`），无接口用 CGLIB（`CglibAopProxy`）。`@Transactional`、`@Cacheable`、`@Async` 都是通过代理实现的。

**观察者模式**：`ApplicationEvent` + `ApplicationListener` + `ApplicationEventPublisher` 三件套构成了 Spring 的事件机制。容器启动/关闭时也会发出 `ContextRefreshedEvent`、`ContextClosedEvent` 等内置事件。

**责任链模式**：Spring MVC 的 `HandlerInterceptor` 链、Spring Security 的 `SecurityFilterChain`、AOP 的拦截器链（`ReflectiveMethodInvocation.proceed()`）。

**装饰器模式**：`BeanWrapper` 对 Bean 的包装、`HttpServletRequestWrapper` 对请求对象的增强。Spring 缓存模块中 `CaffeineCache`、`EhCacheCache` 等都装饰了底层缓存实现。

**适配器模式**：Spring MVC 中的 `HandlerAdapter`，把各种形式的 Handler（`@Controller`、`HttpRequestHandler`、`Servlet` 等）适配成统一的处理接口。

###### 3. MyBatis 中用到了哪些设计模式？

**建造者模式**：`SqlSessionFactoryBuilder` 通过读取配置文件分步骤构建 `Configuration` 对象，最终创建 `SqlSessionFactory`。`XMLConfigBuilder`、`XMLMapperBuilder` 也是建造者的实现。

**工厂模式**：`SqlSessionFactory` 创建 `SqlSession`，`MapperProxyFactory` 创建 Mapper 代理对象，`ObjectFactory` 负责创建查询结果对象。

**代理模式**：Mapper 接口没有实现类，但调用 `mapper.selectById(id)` 时却能执行 SQL。这是 `MapperProxy` 的魔法——它是一个 `InvocationHandler`，通过 JDK 动态代理把接口方法调用转发到 SQL 执行：

```java
public class MapperProxy<T> implements InvocationHandler {
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        if (Object.class.equals(method.getDeclaringClass())) {
            return method.invoke(this, args); // Object 的方法直接调用
        }
        MapperMethod mapperMethod = cachedMapperMethod(method);
        return mapperMethod.execute(sqlSession, args); // 转发到 SQL 执行
    }
}
```

**模板方法模式**：`BaseExecutor` 定义了 SQL 执行的骨架（查缓存→执行→更新缓存），把真正的数据库操作（`doQuery`、`doUpdate`）声明为抽象方法，由 `SimpleExecutor`、`ReuseExecutor`、`BatchExecutor` 分别实现。

**装饰器模式**：`Cache` 包下的 `CachingExecutor` 是 `Executor` 的装饰器，给执行器加上二级缓存功能。`cache/decorators/` 包下还有 `LoggingCache`（加日志）、`LruCache`（加 LRU 淘汰）、`SynchronizedCache`（加同步锁），可以按需叠加装饰。

**组合模式**：动态 SQL 处理。`SqlNode` 是抽象组件，`IfSqlNode`、`WhereSqlNode`、`TrimSqlNode`、`MixedSqlNode` 等形成组合树结构，共同描述一个完整的动态 SQL。

###### 4. JDK 源码中常见的设计模式有哪些？

**创建型**：
- `Runtime.getRuntime()` — 饿汉式单例
- `Calendar.getInstance()`、`NumberFormat.getInstance()` — 工厂方法，根据 Locale 返回不同实现
- `DocumentBuilderFactory.newInstance()` — 抽象工厂，创建完整的 XML 解析器产品族
- `Object.clone()` — 原型模式基础

**结构型**：
- Java IO 体系 — 装饰器模式经典实现（`BufferedInputStream`、`DataInputStream` 等装饰 `InputStream`）
- `Arrays.asList()` — 适配器，把数组适配成 `List`
- `Collections.unmodifiableList()` — 代理模式，拦截修改操作抛出异常
- `javax.xml.parsers.DocumentBuilder` — 门面，封装复杂的 XML 解析子系统

**行为型**：
- `java.util.Iterator` — 迭代器模式的标准化接口
- `java.util.Comparator` — 策略模式，把比较算法封装成独立对象
- `java.util.concurrent.Executor` — 命令模式，`Runnable` 是命令对象，线程池是调用者
- `javax.servlet.Filter` 和 `FilterChain` — 责任链模式
- `Integer.valueOf(-128~127)`、`String` 常量池 — 享元模式

###### 5. 电商系统中如何综合应用多个设计模式？

一个典型的电商下单场景，几乎用到了创建型、结构型、行为型的多个模式配合：

```java
// ====== 策略模式 + 工厂模式：促销价格计算 ======
public interface DiscountStrategy {
    BigDecimal calculate(Cart cart);
}

@Component("fullReduceStrategy")
public class FullReduceStrategy implements DiscountStrategy {
    public BigDecimal calculate(Cart cart) {
        return cart.getTotalPrice().compareTo(new BigDecimal("200")) >= 0
                ? cart.getTotalPrice().subtract(new BigDecimal("30"))
                : cart.getTotalPrice();
    }
}

@Component("memberDiscountStrategy")
public class MemberDiscountStrategy implements DiscountStrategy {
    public BigDecimal calculate(Cart cart) {
        return cart.getTotalPrice().multiply(new BigDecimal("0.9"));
    }
}

// ====== 模板方法模式：统一的支付流程骨架 ======
public abstract class AbstractPaymentProcessor {
    public final PaymentResult process(PaymentRequest request) {
        validate(request);                  // 1. 校验（固定）
        BigDecimal finalAmount = applyDiscount(request); // 2. 优惠（可变）
        PaymentResult result = doPay(request, finalAmount); // 3. 支付（可变）
        postProcess(result);                // 4. 后处理（固定）
        return result;
    }

    protected void validate(PaymentRequest request) { /* 通用校验 */ }
    protected BigDecimal applyDiscount(PaymentRequest request) { return request.getAmount(); }
    protected abstract PaymentResult doPay(PaymentRequest request, BigDecimal amount);
    protected void postProcess(PaymentResult result) { /* 通用后处理 */ }
}

@Component
public class AlipayProcessor extends AbstractPaymentProcessor {
    @Override
    protected PaymentResult doPay(PaymentRequest request, BigDecimal amount) {
        System.out.println("支付宝支付: " + amount);
        return new PaymentResult(true, "alipay-trade-" + System.currentTimeMillis());
    }
}

// ====== 观察者模式：订单创建后通知多个下游 ======
public class OrderCreatedEvent extends ApplicationEvent {
    private final Order order;
    public OrderCreatedEvent(Object source, Order order) {
        super(source); this.order = order;
    }
    public Order getOrder() { return order; }
}

@Component
public class InventoryDeductListener {
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        System.out.println("扣减库存: " + event.getOrder().getId());
    }
}

@Component
public class WeChatNotifyListener {
    @EventListener
    @Async
    public void onOrderCreated(OrderCreatedEvent event) {
        System.out.println("异步推送微信消息: " + event.getOrder().getId());
    }
}

// ====== 责任链模式：下单前的多项校验 ======
public interface OrderCheckHandler {
    boolean check(Order order, OrderCheckHandler next);
}

@Component
@Order(1)
public class StockCheckHandler implements OrderCheckHandler {
    public boolean check(Order order, OrderCheckHandler next) {
        if (!hasStock(order)) {
            throw new BusinessException("库存不足");
        }
        return next != null && next.check(order, null);
    }
    private boolean hasStock(Order order) { return true; }
}

@Component
@Order(2)
public class CreditCheckHandler implements OrderCheckHandler {
    public boolean check(Order order, OrderCheckHandler next) {
        if (!creditOk(order)) {
            throw new BusinessException("信用不足，禁止下单");
        }
        return next != null && next.check(order, null);
    }
    private boolean creditOk(Order order) { return true; }
}

// ====== 建造者模式：构建复杂的 Order 对象 ======
Order order = Order.builder()
        .userId(userId)
        .productId(productId)
        .quantity(1)
        .totalAmount(price)
        .paymentType("alipay")
        .shippingAddress(address)
        .build();
```

这几个模式的分工是：建造者负责构建复杂订单对象；责任链负责下单前的校验流程；模板方法固定支付流程骨架；策略模式处理不同的促销规则；观察者解耦下单后的各种通知逻辑。

###### 6. 过度使用设计模式会带来什么问题？

过度设计是真实存在的陷阱，主要体现在几个方面：

代码复杂度不必要地升高：一个简单的 if-else 被拆成策略接口 + N 个策略类 + 工厂类，新人维护时要追好几个文件才能看清一段逻辑。

维护成本反而上升：模式引入的抽象层越多，理解和定位问题的路径就越长。调用链深了，出 bug 还不好排查。

违反 YAGNI 原则：为"可能的未来变化"提前抽象，结果这个变化永远没来，多出来的代码反而成了累赘。

平衡原则：**从简单实现开始，等真的出现重复和痛点时再重构**。简单说就是"三次法则"——第一次直接实现，第二次遇到类似需求复制修改，第三次才提取共同模式。

###### 7. 如何判断是否需要引入某个设计模式？

可以问自己这几个问题：

这个问题是否真的存在？如果当前代码改起来不痛苦，就没有必要引入模式。

引入模式后，代码是变简单还是变复杂了？如果加了一个模式，文件数量大幅增加但逻辑并没有变清晰，说明过度设计了。

这个"变化点"真的会变化吗？策略模式适合算法经常切换的场景；如果算法基本固定，策略模式就是多此一举。

团队是否都理解这个模式？引入只有自己懂的设计模式，会给团队协作带来麻烦。

###### 8. 设计模式和架构模式的区别？

设计模式是**微观代码层面**的可复用方案，解决的是类/对象级别的问题，比如如何创建对象、如何组织类的结构、如何封装行为。

架构模式是**宏观系统层面**的结构决策，解决的是子系统/组件之间的关系，比如如何划分层次、如何组织服务、如何管理数据流。

常见架构模式：MVC/MVP/MVVM（UI 架构）、微服务（服务拆分）、分层架构（Controller/Service/Repository）、Event-Driven（事件驱动）、CQRS（命令查询职责分离）。

两者不是对立的，而是不同粒度的设计工具。架构模式给系统划定了宏观结构，设计模式在微观层面填充具体实现。比如 MVC 架构中的 View 可以用观察者模式来监听 Model 变化。

###### 9. 微服务架构中常用的设计模式有哪些？

微服务有它自己的"专属"模式体系：

**服务发现与注册**：服务启动时向注册中心（Nacos/Eureka）注册，调用方从注册中心获取服务地址。解耦了服务地址的硬编码。

**API 网关**：所有外部请求先经过网关，网关负责认证、鉴权、限流、路由、协议转换。这其实是门面模式 + 责任链模式在微服务层面的体现。

**断路器模式**：类似电路断路器，当下游服务连续失败达到阈值，自动"断开"，不再发请求，防止雪崩。Hystrix/Resilience4j 实现。

**Saga 模式**：解决微服务间的分布式事务问题。把长事务拆成一系列本地事务，每步有对应的补偿操作，失败时逐步回滚。有编排式（Orchestration）和协同式（Choreography）两种实现。

**CQRS（命令查询职责分离）**：写操作和读操作走不同的模型和数据源，写走主库命令模型，读走从库/缓存查询模型，解决读写性能差异大的问题。

**事件溯源（Event Sourcing）**：不直接存储当前状态，而是存储一系列导致状态变化的事件，当前状态由事件回放得出。天然支持审计日志和时间旅行（回到过去某个时间点的状态）。

###### 10. 如何重构代码以应用设计模式？

重构引入模式要有清晰的方法，不能一上来就大改。

**识别重构时机**：代码有很多 if-else 分支（考虑策略模式）；多个类有重复的逻辑流程（考虑模板方法）；一处改动频繁波及其他地方（考虑观察者或事件解耦）；对象创建代码散落各处（考虑工厂）；类的职责太多（考虑拆分并用门面统一入口）。

**重构流程**：先建立测试覆盖，再开始重构；小步提交，每次只做一个明确的改变；改完立刻运行测试验证行为没有改变。

```java
// 重构示例：把一段充血的 if-else 重构成策略模式

// 重构前 ——— 新增导出格式就得改这个方法
public class ReportExporter {
    public byte[] export(Report report, String format) {
        if ("excel".equals(format)) {
            return exportToExcel(report);
        } else if ("pdf".equals(format)) {
            return exportToPdf(report);
        } else if ("csv".equals(format)) {
            return exportToCsv(report);
        }
        throw new UnsupportedOperationException("不支持的格式: " + format);
    }
}

// 重构后 ——— 新增格式只加一个 Strategy 实现类
public interface ExportStrategy {
    byte[] export(Report report);
    String getFormat();
}

@Component
public class ExcelExportStrategy implements ExportStrategy {
    public byte[] export(Report report) { return exportToExcel(report); }
    public String getFormat() { return "excel"; }
}

@Component
public class ReportExporter {
    private final Map<String, ExportStrategy> strategies = new HashMap<>();

    @Autowired
    public ReportExporter(List<ExportStrategy> list) {
        list.forEach(s -> strategies.put(s.getFormat(), s));
    }

    public byte[] export(Report report, String format) {
        ExportStrategy strategy = strategies.get(format);
        if (strategy == null) throw new UnsupportedOperationException("不支持的格式: " + format);
        return strategy.export(report);
    }
}
```

重构的目标不是"用上模式"，而是让代码更容易理解、扩展和测试。如果重构后代码更难懂了，说明这个模式不适合这里。
