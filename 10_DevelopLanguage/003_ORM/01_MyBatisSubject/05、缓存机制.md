###### 1. MyBatis 的缓存机制是怎样的？
[[22_MybatisKnowledge/04_缓存机制/01、一级缓存与二级缓存#1. 缓存分层设计|📖]] MyBatis 的缓存分两层：**一级缓存（SqlSession 级别）**和**二级缓存（Mapper 命名空间级别）**，查询时先查二级缓存，再查一级缓存，最后才查数据库。

**一级缓存**默认开启且无法关闭，作用范围是当前 SqlSession。底层是 `BaseExecutor` 中的 `PerpetualCache`，本质是一个 HashMap，以 `CacheKey`（SQL + 参数 + 分页等信息的组合）为 key 缓存查询结果。同一个 SqlSession 内，相同条件的查询第二次直接从 HashMap 取，不访问数据库。

**二级缓存**需要手动配置，作用范围是 Mapper 命名空间，跨 SqlSession 共享。底层采用**装饰器模式**，`CachingExecutor` 包装基础 `Executor`，在执行查询时先检查二级缓存。有一个重要的"事务性"设计：查询结果不会立即放入二级缓存，而是暂存在 `TransactionalCache` 中，等到当前 SqlSession **commit** 之后才真正写入缓存，确保只有已提交的数据才对其他 SqlSession 可见。

###### 2. MyBatis 一级缓存和二级缓存的区别是什么？
[[22_MybatisKnowledge/04_缓存机制/01、一级缓存与二级缓存|📖]] 两者的核心区别体现在作用范围和数据共享能力上。

**一级缓存**是 SqlSession 私有的，线程隔离，不会产生跨 Session 的数据一致性问题。它始终开启，作用是减少同一个请求处理过程中的重复查询（比如同一个 HTTP 请求里多次查询同一个用户）。不需要实体类实现 `Serializable`，因为对象直接放在 JVM 内存里，不涉及序列化。

**二级缓存**是 Mapper 级别的，多个 SqlSession 共享，可以跨请求共享热点数据。需要在 Mapper XML 中显式配置 `<cache/>`，并且实体类必须实现 `Serializable`（因为缓存对象可能需要序列化到磁盘或传输）。生命周期随应用存在，除非手动清除或到期淘汰。

两者的另一个重要区别是**数据一致性**：一级缓存在 SqlSession 内隔离，安全性高；二级缓存跨 Session 共享，有潜在的脏读风险（多表关联时尤其需要注意）。

###### 3. MyBatis 一级缓存的生命周期是什么？
[[22_MybatisKnowledge/04_缓存机制/01、一级缓存与二级缓存#2. 一级缓存|📖]] 一级缓存随 SqlSession 的创建而创建，随 SqlSession 的关闭而销毁。

在此期间，有几种情况会让一级缓存提前失效：

- **执行 DML 操作**（INSERT/UPDATE/DELETE）：`BaseExecutor.update()` 源码中，每次执行写操作前都会调用 `clearLocalCache()` 清空缓存，避免缓存了旧数据而数据库已经被修改的情况
- **手动清空**：调用 `sqlSession.clearCache()`
- **查询参数变化**：CacheKey 由 SQL + 参数 + 分页偏移等信息组成，任何一项变化都导致缓存未命中（相当于新的 key，之前的缓存对这次查询无效）

###### 4. MyBatis 二级缓存如何配置？
[[22_MybatisKnowledge/04_缓存机制/01、一级缓存与二级缓存#3. 二级缓存|📖]] 开启二级缓存需要三步：全局开关（默认 true）、Mapper 级别配置、实体类序列化。

```xml
<!-- mybatis-config.xml（默认就是 true，通常可以省略） -->
<settings>
    <setting name="cacheEnabled" value="true"/>
</settings>
```

```xml
<!-- UserMapper.xml：在 <mapper> 标签内添加 <cache> 标签 -->
<cache
    eviction="LRU"           <!-- 淘汰策略：LRU(最近最少使用)、FIFO、SOFT、WEAK -->
    flushInterval="60000"    <!-- 自动刷新间隔（毫秒），60 秒清一次 -->
    size="1024"              <!-- 最多缓存 1024 个对象 -->
    readOnly="true"/>        <!-- true=所有操作返回同一个缓存对象（性能好，线程不安全）; false=返回副本（安全但有序列化开销）-->
```

```java
// 实体类必须实现 Serializable
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    // ...
}
```

**注意**：二级缓存在事务 commit 后才生效，不要在写多读少的场景滥用，多表关联场景下要特别小心缓存污染问题。

###### 5. MyBatis 缓存的失效场景有哪些？
[[22_MybatisKnowledge/04_缓存机制/01、一级缓存与二级缓存#4. 缓存失效|📖]] **一级缓存失效**：执行 DML 操作（INSERT/UPDATE/DELETE）会自动清空当前 SqlSession 的缓存；调用 `clearCache()`；SqlSession 关闭；查询参数改变导致 CacheKey 不同（缓存未命中，不是失效）。

**二级缓存失效**：当前 Mapper 命名空间下执行 DML 操作，会清空该 Mapper 的整个二级缓存；达到 `flushInterval` 配置的刷新时间；缓存条数超过 `size` 限制，按 `eviction` 策略淘汰最旧的条目。

**多表关联的缓存陷阱**：这是二级缓存最容易踩的坑。比如 OrderMapper 里有一个查询关联了 User 表数据，当 UserMapper 执行了更新操作后，OrderMapper 的二级缓存并不会自动失效——因为它们是不同的命名空间。导致 Order 缓存里的用户信息是旧的，产生脏读。解决方案是：关联多表的查询场景，禁用二级缓存（在该 `<select>` 标签加 `useCache="false"`），或者用 `<cache-ref>` 让多个 Mapper 共享同一个缓存命名空间。

###### 6. MyBatis 如何集成第三方缓存（如 Redis、Ehcache）？
[[22_MybatisKnowledge/04_缓存机制/01、一级缓存与二级缓存#5. 第三方缓存集成|📖]] 集成第三方缓存需要实现 MyBatis 的 `Cache` 接口，或使用现成的集成包。官方提供了 `mybatis-redis` 和 `mybatis-ehcache` 依赖。

**Redis 集成**：

```xml
<dependency>
    <groupId>org.mybatis.caches</groupId>
    <artifactId>mybatis-redis</artifactId>
    <version>1.0.0-beta2</version>
</dependency>
```

然后在 Mapper XML 中配置：

```xml
<cache type="org.mybatis.caches.redis.RedisCache"/>
```

**使用第三方缓存的主要动机**是支持**分布式缓存**——多实例部署时，MyBatis 默认的内存缓存是 JVM 本地的，每个实例独立，相互之间没有同步。引入 Redis 后，所有实例共享同一个缓存，避免数据不一致，也便于集中管理缓存的过期和清除。

###### 7. MyBatis 缓存的命中率如何优化？
[[22_MybatisKnowledge/04_缓存机制/01、一级缓存与二级缓存|📖]] 缓存命中率的优化主要从两个维度入手：合理选择缓存层级和优化缓存配置。

**缓存层级选择**：高频的会话内重复查询依靠一级缓存就够了；需要跨会话共享的静态/低频变化数据，才有必要开启二级缓存。不要为了缓存而缓存，频繁更新的数据开二级缓存反而会因为频繁清空而没有收益。

**配置优化**：`eviction` 推荐 LRU（最近最少使用），比 FIFO 更符合实际访问规律；`flushInterval` 根据数据更新频率合理设置，不要太短（缓存还没暖就被清掉）；`blocking="true"` 防止缓存穿透——当缓存 key 不存在时，只允许一个线程去查数据库，其他线程等待，避免大量并发请求同时打到数据库。

**分布式环境**：多实例部署时，必须用 Redis 等分布式缓存替代默认实现，否则各实例的缓存相互独立，形不成效果还可能引起数据不一致。
