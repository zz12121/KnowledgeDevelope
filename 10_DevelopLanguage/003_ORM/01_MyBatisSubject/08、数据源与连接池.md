###### 1. MyBatis 自带的连接池有什么特点？
[[22_MybatisKnowledge/06_事务与数据源/02、数据源与连接池#1. MyBatis 内置数据源|📖]] MyBatis 自带的数据源实现主要有两种：**UNPOOLED** 和 **POOLED**。

**UNPOOLED** 每次请求连接都直接通过 `DriverManager.getConnection()` 新建，用完立即关闭，没有任何连接复用。适合测试环境或对性能要求极低的场景。

**POOLED** 是 MyBatis 内置的轻量级连接池，核心实现类是 `PooledDataSource`，维护两个列表：`idleConnections`（空闲连接）和 `activeConnections`（活跃连接）。

每个连接被 `PooledConnection` 包装，这个包装类通过动态代理重写了 `close()` 方法——调用 `close()` 时实际上是把连接归还到空闲池，而不是真正关闭 JDBC 连接。这就是连接池的核心原理：**包装器模式**。

**连接获取的核心逻辑**（`popConnection()` 方法）：先看空闲池有没有，有就直接取；空闲池为空且活跃连接数未到上限就新建；活跃连接数已满就看最老的连接是否超时（`poolMaximumCheckoutTime`），超时则"征用"（回滚其事务，重新包装返回）；否则等待（`state.wait()`）。整个过程在 `synchronized` 块内，保证线程安全。

**适用建议**：POOLED 适合开发测试，生产环境强烈建议使用 HikariCP 或 Druid，功能更完善、性能更好。

###### 2. MyBatis 如何配置第三方连接池（如 HikariCP、Druid）？
[[22_MybatisKnowledge/06_事务与数据源/02、数据源与连接池#2. HikariCP 配置|📖]] 切换第三方连接池非常简单，因为 MyBatis 的 DataSource 接口是开放的，任何实现了 `javax.sql.DataSource` 接口的连接池都可以接入。

**HikariCP（Spring Boot 默认，追求极致性能）**：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/my_db
    username: root
    password: yourpassword
    driver-class-name: com.mysql.cj.jdbc.Driver
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
```

**Druid（功能全面，内置监控和防 SQL 注入）**：

```yaml
spring:
  datasource:
    type: com.alibaba.druid.pool.DruidDataSource
    druid:
      initial-size: 5
      min-idle: 5
      max-active: 20
      filters: stat,wall,log4j2  # stat=SQL监控统计, wall=防SQL注入
      stat-view-servlet:
        enabled: true             # 开启 /druid 监控页面
```

**选择建议**：追求极致性能选 HikariCP；需要 SQL 统计分析、慢 SQL 告警、防 SQL 注入、运维监控的企业系统选 Druid。

###### 3. MyBatis 如何实现多数据源配置？
[[22_MybatisKnowledge/06_事务与数据源/02、数据源与连接池#5. 多数据源静态配置|📖]] 多数据源配置的核心是：为每个数据源创建独立的 `SqlSessionFactory` 和 `TransactionManager`，通过 `@MapperScan` 的 `sqlSessionFactoryRef` 属性将不同包路径的 Mapper 绑定到对应的工厂。

```java
@Configuration
@MapperScan(basePackages = "com.example.mapper.primary",
            sqlSessionFactoryRef = "primarySqlSessionFactory")
public class PrimaryDataSourceConfig {

    @Bean @Primary
    @ConfigurationProperties(prefix = "primary.datasource")
    public DataSource primaryDataSource() {
        return DataSourceBuilder.create().type(HikariDataSource.class).build();
    }

    @Bean @Primary
    public SqlSessionFactory primarySqlSessionFactory(
            @Qualifier("primaryDataSource") DataSource dataSource) throws Exception {
        SqlSessionFactoryBean bean = new SqlSessionFactoryBean();
        bean.setDataSource(dataSource);
        bean.setMapperLocations(new PathMatchingResourcePatternResolver()
                .getResources("classpath:mapper/primary/*.xml"));
        return bean.getObject();
    }

    @Bean @Primary
    public DataSourceTransactionManager primaryTransactionManager(
            @Qualifier("primaryDataSource") DataSource dataSource) {
        return new DataSourceTransactionManager(dataSource);
    }
}
```

第二个数据源配置类似，去掉 `@Primary` 注解，换成 secondary 的包路径和前缀即可。

**关键点**：`@Primary` 标记默认 Bean（多个同类型 Bean 时的首选）；`@Qualifier` 在注入时明确指定 Bean 名称避免歧义。

###### 4. MyBatis 多数据源下如何切换数据源？
[[22_MybatisKnowledge/06_事务与数据源/03、多数据源与读写分离#2. AbstractRoutingDataSource 动态切换|📖]] 静态多数据源是启动时绑定，运行时无法切换。要实现**运行时动态切换**，核心技术是 Spring 的 `AbstractRoutingDataSource`。

它维护一个 `targetDataSources` Map（key→DataSource），每次获取连接时调用 `determineCurrentLookupKey()` 返回当前要用的 key，从 Map 里找对应的 DataSource。

结合 **ThreadLocal** 存储当前线程的数据源标识，再用 **AOP 切面 + 自定义注解**实现优雅的切换：

```java
// 1. ThreadLocal 持有数据源标识
public class DataSourceContextHolder {
    private static final ThreadLocal<String> HOLDER = new ThreadLocal<>();
    public static void set(String type) { HOLDER.set(type); }
    public static String get() { return HOLDER.get(); }
    public static void clear() { HOLDER.remove(); } // 防内存泄漏，必须清理
}

// 2. 路由数据源
public class DynamicDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return DataSourceContextHolder.get();
    }
}

// 3. AOP 切面
@Before("@annotation(ds)")
public void before(DataSource ds) { DataSourceContextHolder.set(ds.value()); }

@After("@annotation(ds)")
public void after(DataSource ds) { DataSourceContextHolder.clear(); }

// 4. 使用
@DataSource("secondary")
public User getFromSecondary(Long id) { return userMapper.selectById(id); }
```

**重要注意事项**：动态切换必须在 **Spring 事务开启之前**完成。因为事务开始时，`DataSourceTransactionManager` 就从 DataSource 获取连接并绑定到线程，此后连接已经固定，再修改 ThreadLocal 的 key 也不起作用了。通常的做法是：AOP 切换数据源的 `@Before` 优先级要高于事务 AOP 的优先级。

###### 5. MyBatis 如何实现读写分离？
[[22_MybatisKnowledge/06_事务与数据源/03、多数据源与读写分离#3. 读写分离|📖]] 读写分离是多数据源动态切换的特例：写操作路由主库，读操作路由从库。实现方案与动态数据源相同，路由逻辑更简单——利用 Spring 事务的 `readOnly` 属性自动判断：

```java
public class ReadWriteRouter extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        boolean isReadOnly = TransactionSynchronizationManager.isCurrentTransactionReadOnly();
        return isReadOnly ? "slave" : "master";
    }
}
```

在 Service 层通过 `@Transactional(readOnly = true)` 标记读操作，路由数据源会自动将这些查询路由到从库：

```java
@Transactional           // → master（写操作）
public void createUser(User user) { ... }

@Transactional(readOnly = true)  // → slave（读操作）
public User getUserById(Long id) { ... }
```

多个从库时，可以在 `determineCurrentLookupKey()` 里加轮询或随机逻辑实现负载均衡。

**注意主从延迟**：从库有复制延迟（通常毫秒级），对一致性要求高的读操作（如支付后立刻查余额）应该强制走主库，不能依赖从库的数据。
