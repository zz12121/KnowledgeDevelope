###### 1. MyBatis 的插件机制是什么？如何自定义插件？
[[22_MybatisKnowledge/05_插件与扩展/01、插件机制与拦截器#1. 插件机制原理|📖]] MyBatis 插件机制是基于**拦截器（Interceptor）**的框架扩展点，本质是 **JDK 动态代理 + 责任链模式**。它允许开发者在不修改 MyBatis 源码的情况下，拦截并增强四大核心组件的行为：Executor、StatementHandler、ParameterHandler、ResultSetHandler。

自定义插件分三步：

**第一步：实现 `Interceptor` 接口**

```java
@Intercepts({
    @Signature(type = StatementHandler.class, method = "prepare",
               args = {Connection.class, Integer.class})
})
public class MyPlugin implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 在目标方法前后加入自定义逻辑
        System.out.println("Before: " + invocation.getMethod().getName());
        Object result = invocation.proceed();  // 继续执行责任链
        System.out.println("After: " + invocation.getMethod().getName());
        return result;
    }

    @Override
    public Object plugin(Object target) {
        return Plugin.wrap(target, this);  // 固定写法，生成代理对象
    }

    @Override
    public void setProperties(Properties properties) {
        // 接收配置文件中为插件设置的属性
    }
}
```

**第二步：`@Intercepts` + `@Signature` 注解声明拦截点**，精确指定要拦截的类、方法和参数类型。

**第三步：在 `mybatis-config.xml` 中注册**：

```xml
<plugins>
    <plugin interceptor="com.example.MyPlugin">
        <property name="threshold" value="1000"/>
    </plugin>
</plugins>
```

**源码生效机制**：MyBatis 启动时，`InterceptorChain` 收集所有插件。创建 Executor/StatementHandler 等核心组件时，调用 `interceptorChain.pluginAll(target)` 对目标对象层层包装，返回代理对象。调用代理方法时，`Plugin.invoke()` 检查是否命中 `@Signature` 声明的方法，命中则调用 `intercept()`，否则直接放行。

###### 2. MyBatis 插件的拦截点有哪些？
[[22_MybatisKnowledge/05_插件与扩展/01、插件机制与拦截器#2. 四大拦截点|📖]] MyBatis 只允许拦截四大核心组件的特定方法，覆盖了 SQL 执行的完整生命周期。

**Executor** 是最上层的调度核心，可拦截 `update`、`query`、`commit`、`rollback` 等方法。适合需要全局介入的功能：分页（修改 SQL 并执行 COUNT）、数据权限过滤（在 WHERE 里追加条件）、读写分离（路由查询到从库）。

**StatementHandler** 负责 SQL 语句构建，可拦截 `prepare`（SQL 准备阶段）、`parameterize`（参数绑定）等。这里能拿到 `BoundSql` 对象修改 SQL 文本，是**SQL 重写**的最佳拦截点，PageHelper 就是在这里加 LIMIT 语句的。

**ParameterHandler** 负责参数类型转换，可拦截 `setParameters`。适合参数加密、统一参数校验等场景。

**ResultSetHandler** 负责结果集映射，可拦截 `handleResultSets`。适合结果集解密、数据脱敏、自定义类型转换等场景。

**选择拦截点的原则**：修改 SQL 语句选 StatementHandler.prepare；对所有 SQL 做全局统计选 Executor.query/update；修改参数选 ParameterHandler.setParameters；处理结果选 ResultSetHandler.handleResultSets。

###### 3. MyBatis 的分页插件（PageHelper）是如何工作的？
[[22_MybatisKnowledge/05_插件与扩展/01、插件机制与拦截器#4. PageHelper 工作流程|📖]] PageHelper 是基于 MyBatis 插件机制实现的，核心思路是拦截查询方法、重写 SQL、自动执行 COUNT 查询。

**工作流程**：

1. **设置分页参数**：调用 `PageHelper.startPage(2, 10)` 将分页信息（页码=2、每页=10）存入 **ThreadLocal**，保证线程安全
2. **拦截查询**：执行 Mapper 方法时被 PageHelper 拦截，从 ThreadLocal 取出分页参数
3. **执行 COUNT 查询**：自动生成 `SELECT COUNT(*) FROM (原始SQL)` 并执行，得到总记录数
4. **SQL 重写**：根据数据库方言（MySQL 用 LIMIT，Oracle 用 ROWNUM），在原始 SQL 后追加分页语句，改写为 `SELECT ... LIMIT 10 OFFSET 10`
5. **执行分页查询**：只从数据库返回当前页的数据
6. **封装结果**：将查询结果和总数等信息封装到 `Page` 对象，清理 ThreadLocal

从源码角度看，PageHelper 通过 `MetaObject` 反射工具获取并修改 `MappedStatement` 中 `BoundSql` 的 SQL 字符串来实现重写。

###### 4. MyBatis 如何实现物理分页和逻辑分页？
[[22_MybatisKnowledge/05_插件与扩展/01、插件机制与拦截器#5. 物理分页 vs 逻辑分页|📖]] 两者的本质区别是**分页在数据库层面还是应用层面执行**。

**物理分页**是在 SQL 里加 `LIMIT`/`ROWNUM` 等数据库分页语句，数据库只返回当前页的数据，网络传输和内存占用都很小。这是**生产环境唯一正确的选择**，尤其是数据量大的情况下。实现方式：在 XML 里手写带 LIMIT 的 SQL，或者使用 PageHelper 自动重写。

**逻辑分页**是 MyBatis 的 `RowBounds` 功能：查询时传入偏移量和条数，但 MyBatis 实际上先把所有数据都查到内存，再通过 `DefaultResultSetHandler` 用游标跳过前 N 条、只取指定条数。数据库压力没有减少，只是减少了返回给调用方的数据量。当数据量大时极容易 OOM，只适合数据量极小（几百条以内）的测试或内部场景。

###### 5. MyBatis 的拦截器（Interceptor）有什么作用？
[[22_MybatisKnowledge/05_插件与扩展/01、插件机制与拦截器#1. 插件机制原理|📖]] `Interceptor` 是 MyBatis 插件机制的核心接口，它的本质作用是提供**对 MyBatis 核心流程的无侵入式扩展能力**。

通过 `intercept()` 方法，可以在目标方法执行前后插入任意逻辑（AOP 的核心思想）；通过 `plugin()` 方法生成代理对象，将增强逻辑织入目标组件；通过 `setProperties()` 让拦截器行为可配置，适应不同场景。

典型应用场景：分页（PageHelper）、SQL 性能监控（慢 SQL 告警）、数据脱敏（查询结果的手机号打码）、数据加密（存储前加密、查询后解密）、动态数据源路由（读写分离）、操作审计日志。

###### 6. MyBatis 如何实现 SQL 性能监控？
[[22_MybatisKnowledge/05_插件与扩展/01、插件机制与拦截器#3. 自定义插件实现|📖]] SQL 性能监控的实现思路是：自定义拦截器，拦截 Executor 的 `query` 和 `update` 方法，在执行前后记录时间戳，计算耗时，超过阈值则输出慢 SQL 警告：

```java
@Intercepts({
    @Signature(type = Executor.class, method = "query",
               args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class}),
    @Signature(type = Executor.class, method = "update",
               args = {MappedStatement.class, Object.class})
})
public class SqlPerformanceInterceptor implements Interceptor {
    private static final long SLOW_SQL_THRESHOLD = 1000; // 慢 SQL 阈值 1秒

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long startTime = System.currentTimeMillis();
        try {
            return invocation.proceed();
        } finally {
            long costTime = System.currentTimeMillis() - startTime;
            MappedStatement ms = (MappedStatement) invocation.getArgs()[0];
            if (costTime > SLOW_SQL_THRESHOLD) {
                log.warn("慢SQL警告 => ID: {}, 耗时: {}ms", ms.getId(), costTime);
                // 还可以发送到 SkyWalking、Prometheus 等监控系统
            }
        }
    }
}
```

进阶方案可以维护一个 Map 统计每个 SQL 的执行频率，发现热点查询；或者拦截 ResultSetHandler 检查返回结果集大小，预防大量数据返回导致的内存问题。
