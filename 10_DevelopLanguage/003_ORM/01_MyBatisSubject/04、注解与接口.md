###### 1. MyBatis 支持哪些常用注解？
[[22_MybatisKnowledge/03_注解与接口/01、注解体系与Mapper代理#1. 注解体系|📖]] MyBatis 注解体系按功能可以分为六大类。

**CRUD 注解**：`@Select`、`@Insert`、`@Update`、`@Delete`，直接在注解值里写 SQL 语句，适合简单固定的 SQL，不需要额外的 XML 文件。

**结果映射注解**：`@Results`（等效 `<resultMap>`）、`@Result`（等效 `<result>`）、`@ResultMap`（引用已定义的 resultMap）。

**关联映射注解**：`@One`（一对一，等效 `<association>`）、`@Many`（一对多，等效 `<collection>`），配合 `@Result` 使用。

**参数处理注解**：`@Param` 为方法参数命名，`@Options` 配置 useGeneratedKeys 等选项，`@SelectKey` 处理非自增主键。

**动态 SQL 注解**：`@SelectProvider`、`@InsertProvider`、`@UpdateProvider`、`@DeleteProvider`，指向 Java 类中的方法动态构建 SQL，适合复杂条件的注解方式。

**缓存注解**：`@CacheNamespace`（等效 `<cache>`）、`@CacheNamespaceRef`（等效 `<cache-ref>`）。

**Mapper 扫描**：`@Mapper` 标记单个接口，`@MapperScan`（Spring 注解）批量扫描包路径。

###### 2. MyBatis 中 @Param 注解的作用是什么？
[[22_MybatisKnowledge/03_注解与接口/01、注解体系与Mapper代理#2. @Param 源码机制|📖]] `@Param` 的核心作用是**为方法参数命名**，让 MyBatis 能在 XML 或注解 SQL 中通过名称引用参数。

**为什么需要它**：Java 编译后默认不保留方法参数名（需要加 `-parameters` 编译参数），所以 MyBatis 默认会用 `param1`、`param2` 这样的名称代替参数名。多参数方法如果不加 `@Param`，XML 里就只能写 `#{param1}`，既不直观也容易出错。

加了 `@Param` 之后，MyBatis 内部的 `ParamNameResolver` 会用注解指定的名称覆盖默认命名，XML 里就可以用有意义的名称了：

```java
// 正确：用 @Param 明确参数名
User findUser(@Param("name") String name, @Param("age") Integer age);
```

XML 中对应：`WHERE name = #{name} AND age = #{age}`

另外，MyBatis 会同时保留 `param1`、`param2` 这样的默认名称作为兼容方案，所以两种写法都能用。

**单参数的特殊情况**：如果方法只有一个参数且是 POJO 或 Map，不需要 `@Param`，直接用属性名引用即可。

###### 3. MyBatis 的 Mapper 接口是如何工作的？
[[22_MybatisKnowledge/03_注解与接口/01、注解体系与Mapper代理#3. Mapper 动态代理原理|📖]] Mapper 接口不需要写实现类，运行时 MyBatis 会通过 **JDK 动态代理**自动生成实现。

具体来说，`MapperProxyFactory` 利用 `Proxy.newProxyInstance()` 创建了一个实现了 Mapper 接口的代理对象 `MapperProxy`。当调用 `userMapper.selectById(1L)` 时，实际上是调用了代理对象的 `invoke()` 方法。

`MapperProxy.invoke()` 内部会用 `MapperMethod` 对象处理这次调用：它根据方法名在 `Configuration` 中找到对应的 `MappedStatement`（就是 XML 中那条 SQL 的描述对象），然后转换为 `SqlSession` 的具体操作（如 `sqlSession.selectOne("com.example.UserMapper.selectById", 1L)`），最终执行 SQL 并返回结果。

整个过程开发者无感知，用接口方法的方式操作数据库，代码既简洁又类型安全。

###### 4. MyBatis 如何实现 Mapper 接口与 XML 的绑定？
[[22_MybatisKnowledge/03_注解与接口/01、注解体系与Mapper代理#3. Mapper 动态代理原理|📖]] 绑定机制通过**命名空间 + 方法名**精确匹配。XML 中的 `namespace` 属性必须等于 Mapper 接口的全限定名，XML 中 SQL 语句的 `id` 必须等于接口的方法名：

```xml
<!-- UserMapper.xml -->
<mapper namespace="com.example.mapper.UserMapper">
    <select id="selectById" resultType="User">
        SELECT * FROM users WHERE id = #{id}
    </select>
</mapper>
```

```java
package com.example.mapper;
public interface UserMapper {
    User selectById(@Param("id") Long id);
}
```

启动时，`XMLMapperBuilder` 解析 XML 文件，将每条 SQL 封装为 `MappedStatement` 注册到 `Configuration` 中，key 为 `"com.example.mapper.UserMapper.selectById"`。调用 Mapper 方法时，代理对象用同样的 key 找到对应的 `MappedStatement` 执行。

如果出现 `Invalid bound statement (not found)` 异常，99% 是 namespace 或 id 拼写错误，或者 XML 文件没被正确打包到 classpath。

###### 5. MyBatis 中使用注解和 XML 配置的优缺点是什么？
[[22_MybatisKnowledge/03_注解与接口/01、注解体系与Mapper代理#4. 注解 vs XML|📖]] 两种方式各有适用场景，核心差异在于**动态 SQL 的支持程度和可读性**。

**注解方式**的优势是简洁——代码即配置，不用在 Java 文件和 XML 文件之间来回切换。对于简单的 CRUD 操作非常方便。劣势是写复杂 SQL 很痛苦：需要借助 `<script>` 标签或 Provider 类，代码可读性很差，而且修改 SQL 需要重新编译部署。

**XML 方式**的优势是 SQL 与 Java 代码完全分离，复杂 SQL 一目了然，动态 SQL 标签支持完整，DBA 可以直接查看和优化 SQL 文件而不需要懂 Java。劣势是需要维护额外的 XML 文件，文件数量多了之后管理成本增加。

**实践建议**：简单的单表 CRUD 可以用注解；涉及多表关联、复杂动态条件、存储过程的场景，坚决用 XML。两种方式可以混用，同一个项目里既有注解 Mapper 也有 XML Mapper 是完全允许的。

###### 6. MyBatis 的 @SelectProvider、@InsertProvider 等注解的作用是什么？
[[22_MybatisKnowledge/03_注解与接口/01、注解体系与Mapper代理#5. @SelectProvider 使用|📖]] `@*Provider` 注解是注解方式下处理动态 SQL 的方案，将 SQL 构建逻辑提取到一个独立的 Java 类的方法中。

```java
// 1. 定义 SQL 构建类
public class UserSqlProvider {
    public String selectByConditions(final Map<String, Object> params) {
        return new SQL() {{
            SELECT("*");
            FROM("users");
            if (params.get("name") != null) WHERE("name = #{name}");
            if (params.get("minAge") != null) WHERE("age >= #{minAge}");
            ORDER_BY("create_time DESC");
        }}.toString();
    }
}

// 2. Mapper 接口引用
public interface UserMapper {
    @SelectProvider(type = UserSqlProvider.class, method = "selectByConditions")
    List<User> selectByConditions(@Param("name") String name, @Param("minAge") Integer minAge);
}
```

MyBatis 提供了 `SQL` 工具类（方法链风格），用于安全地构建 SQL，比字符串拼接更不容易出错。

**适用场景**：需要在注解方式下实现动态 SQL，或者希望把 SQL 构建逻辑做单元测试（Provider 类是普通 Java 类，比 XML 更容易测试）。
