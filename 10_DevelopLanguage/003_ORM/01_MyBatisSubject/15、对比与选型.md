###### 1. MyBatis 与 JPA 的区别是什么？如何选择？
[[22_MybatisKnowledge/10_最佳实践与选型/02、框架对比与选型#1. MyBatis vs JPA|📖]] 这两个框架的核心区别在于**设计哲学**。

**MyBatis** 走的是**半自动化 ORM** 路线，框架不试图屏蔽 SQL，开发者直接写 SQL，框架负责把结果集映射到 Java 对象。你对每条 SQL 有完全的控制权，想怎么优化就怎么优化。

**JPA** 走的是**全自动化 ORM** 路线，你操作对象，框架自动生成 SQL。简单 CRUD 几乎零代码，但复杂查询要靠 JPQL，SQL 生成过程不透明，出了性能问题不好排查。

具体差异体现在这几个维度：**SQL 控制力**上 MyBatis 完全掌控、JPA 有限控制；**开发效率**上 JPA 胜出（简单 CRUD 零代码）、MyBatis 需要手写 SQL；**性能调优**上 MyBatis 直接精准、JPA 需要理解其生成 SQL 和缓存机制才能调；**学习曲线**上 MyBatis 平缓（会 SQL 就能上手）、JPA 陡峭（实体状态、延迟加载等概念较多）；**数据库兼容性**上 JPA 的 Hibernate 方言自动适配各数据库，MyBatis 需要手动处理方言差异。

**选择建议**：
- **选 MyBatis**：复杂查询系统（报表、数据分析）、遗留系统改造、对 SQL 性能有极致要求的场景。
- **选 JPA**：快速原型开发、DDD 项目、以简单 CRUD 为主的系统。
- **混合使用**：核心复杂业务用 MyBatis，简单模块用 JPA，但注意跨框架的事务一致性。

**源码角度**：MyBatis 通过 `SqlSessionTemplate` → `MappedStatement` → `Executor` 执行 SQL，路径清晰；JPA 的 `EntityManager` 底层 `SessionImpl` 要管理实体状态，JPQL 解析为 AST 再转数据库 SQL，链路复杂得多。

###### 2. MyBatis 与 Spring Data JDBC 的区别是什么？
[[22_MybatisKnowledge/10_最佳实践与选型/02、框架对比与选型#2. MyBatis vs Spring Data JDBC|📖]] 这两者的定位有本质区别。

**MyBatis** 是一个完整的 SQL 映射框架，提供强大的结果集映射、缓存机制和动态 SQL 支持，专注于让你把 SQL 写好、映射做对。

**Spring Data JDBC** 是 Spring Data 家族成员，本质是 **JdbcTemplate 的封装**，提供更简洁的 Repository 抽象，故意不做 ORM 那套复杂机制，回归 SQL 本质。

几个关键特性对比：**映射复杂度**上 MyBatis 支持复杂嵌套结果映射，Spring Data JDBC 只做简单一对一映射，不支持懒加载；**SQL 支持**上 MyBatis 有完整动态 SQL 和存储过程支持，Spring Data JDBC 复杂查询得靠 `@Query` 注解手写；**缓存**上 MyBatis 有一级、二级缓存，Spring Data JDBC 没有内置缓存；**事务**两者都走 Spring 事务体系，这块没区别。

**适用场景**：MyBatis 适合需要复杂映射、动态 SQL、性能调优的企业级应用；Spring Data JDBC 适合微服务里的简单数据访问、对启动速度有要求（不想引完整 ORM 框架）的场景，以及追求"代码即文档"简洁风格的项目。

###### 3. 什么场景下适合使用 MyBatis？
[[22_MybatisKnowledge/10_最佳实践与选型/02、框架对比与选型#3. MyBatis适用场景|📖]] MyBatis 的优势场景很集中，记住这几类就够了。

**复杂 SQL 查询**：金融报表、电商商品筛选这类涉及多表关联、窗口函数、自定义函数的业务，MyBatis 直接控制 SQL 的优势无可替代，全自动 ORM 在这里生成的 SQL 往往很难看。

**性能敏感型应用**：高并发交易系统对响应时间有严格要求，MyBatis 允许写经过精心调优的 SQL，避免 ORM 自动生成 SQL 可能带来的额外开销。

**遗留系统集成**：需要调用存储过程、使用 Oracle CONNECT BY 等数据库特有功能的场景，MyBatis 的灵活性让它成为改造遗留系统的理想选择。

**大数据量处理**：需要流式查询、游标分页、批量操作的场景，MyBatis 提供更好的控制能力，比如导出百万行数据不撑爆内存。

**团队技术栈**：团队 SQL 功底强，希望直接控制数据库交互，而不是靠框架自动生成 SQL——MyBatis 更符合这类团队的工作方式。

###### 4. MyBatis 的优缺点是什么？
[[22_MybatisKnowledge/10_最佳实践与选型/02、框架对比与选型#4. MyBatis优缺点|📖]] 客观评价一下。

**核心优势**：

首先是 **SQL 灵活性**，这是 MyBatis 存在的根本价值。直接写 SQL、直接优化，动态 SQL（`<if>`、`<choose>`、`<foreach>`）能应对各种复杂业务需求。源码层面，`DynamicSqlSource` 解析动态标签，`BoundSql` 生成最终 SQL，整个链路非常清晰。

其次是**性能优化空间大**，接近 JDBC 的原始性能，没有复杂 ORM 开销。一级缓存（SqlSession 级别）和二级缓存（Mapper 级别）可按需配置。`ExecutorType.BATCH` 批处理能大幅提升写入性能。

还有**易于调试和监控**，SQL 日志清晰可见，方便性能分析和问题定位，与 Druid 等连接池集成后能拿到很详细的监控指标。

最后是**与 Spring 生态无缝集成**，`SqlSessionFactoryBean` 简化配置，与 Spring 事务管理紧密结合，几乎感觉不到整合的摩擦。

**主要局限性**：

**开发效率相对低**是最大的槽点——简单 CRUD 也要写 SQL 和映射配置，代码量和 JPA 比起来差距明显。**数据库移植性挑战**：一旦用了某个数据库特有语法，换库就要改 SQL。**维护成本**：大项目的 XML 文件会越来越长，动态 SQL 嵌套深了真的很难读。**对象关联处理弱**：相比 JPA 的自动关联管理，MyBatis 需要手动处理复杂对象关系。

MyBatis-Plus 等增强工具通过代码生成、Lambda 表达式查询等功能，正在弥补开发效率上的不足，这也是为什么目前很多项目选择 MyBatis-Plus 而不是纯 MyBatis。
