###### 1. MyBatis 如何开启 SQL 日志？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#1. 开启SQL日志|📖]] MyBatis 开启 SQL 日志主要有三种方式，根据使用场景选合适的就好。

**Spring Boot 项目（推荐）**，直接在 `application.yml` 里配置：
```yaml
# 方法一：设置Mapper包的日志级别（生产可用）
logging:
  level:
    com.example.mapper: debug  # 替换为你的Mapper包路径

# 方法二：MyBatis内置STDOUT日志（开发环境快速调试）
mybatis:
  configuration:
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl
```

**传统 MyBatis 项目**，在 `mybatis-config.xml` 中指定：
```xml
<configuration>
  <settings>
    <setting name="logImpl" value="SLF4J"/>  <!-- 可选：SLF4J、LOG4J、LOG4J2等 -->
  </settings>
</configuration>
```

**Log4j2 精细配置**，建 `log4j2.xml`：
```xml
<Configuration>
  <Appenders>
    <Console name="Console" target="SYSTEM_OUT">
      <PatternLayout pattern="%d{HH:mm:ss} [%t] %-5level %logger{36} - %msg%n"/>
    </Console>
  </Appenders>
  <Loggers>
    <Logger name="com.example.mapper" level="DEBUG" additivity="false">
      <AppenderRef ref="Console"/>
    </Logger>
    <Logger name="org.apache.ibatis" level="DEBUG" additivity="false">
      <AppenderRef ref="Console"/>
    </Logger>
    <Root level="INFO">
      <AppenderRef ref="Console"/>
    </Root>
  </Loggers>
</Configuration>
```

**源码机制**：MyBatis 的日志系统通过 `LogFactory` 在初始化时检测当前项目的日志框架，采用适配器模式统一接口。配置 `log-impl` 实际上是指定具体的日志适配器实现。

###### 2. MyBatis 如何查看执行的 SQL 语句？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#2. 查看完整SQL|📖]] 日志配置成功后，控制台会输出这样的信息：
```
==>  Preparing: SELECT * FROM users WHERE id = ? AND name = ?
==> Parameters: 1(Integer), test(String)
<==    Total: 1
```

不过默认日志里参数和 SQL 是分开的，如果想拿到**带实际参数的完整 SQL**，得用拦截器：
```java
@Intercepts({
    @Signature(type = Executor.class, method = "update", args = {MappedStatement.class, Object.class}),
    @Signature(type = Executor.class, method = "query", args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})
})
public class SqlInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        MappedStatement mappedStatement = (MappedStatement) invocation.getArgs()[0];
        Object parameter = invocation.getArgs()[1];
        BoundSql boundSql = mappedStatement.getBoundSql(parameter);
        System.out.println("SQL: " + boundSql.getSql());
        System.out.println("Parameters: " + Arrays.toString(invocation.getArgs()));
        return invocation.proceed();
    }
}
```

**IDEA 断点调试**：打开 `SimpleExecutor`，在 `doQuery` 方法里打断点，观察 `BoundSql` 对象——`sql` 字段是完整 SQL，`parameterMappings` 是参数映射列表。

###### 3. MyBatis 常见的异常有哪些？如何解决？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#3. 常见异常排查|📖]] 把几类高频异常整理一下，遇到基本都能快速定位。

**Invalid bound statement (not found)**——最常见，Mapper 接口方法和 XML 无法绑定：
- 检查 XML 的 `namespace` 是否与 Mapper 接口全限定名完全一致（大小写都得匹配）。
- 确认方法名与 XML 中 `id` 属性一致。
- 验证 Mapper XML 是否被正确打包进 classpath。
- Spring Boot 项目里检查是否有 `@Mapper` 或 `@MapperScan`。

**参数绑定异常**——`TypeException: Could not set parameter`，多半是多参数没加 `@Param`：
```java
// 错误写法：多参数未注解
User findUser(String name, Integer age);

// 正确写法：用@Param明确参数名
User findUser(@Param("userName") String name, @Param("userAge") Integer age);
```
XML 里用 `#{userName}` 和 `#{userAge}` 对应。

**结果映射异常**——`ResultMap contains duplicates` 或属性填不上：
- 检查 `resultMap` 里是否有重复的 id 配置。
- 确认数据库字段名与实体属性名的映射关系，必要时显式用 `<result>` 标签指定。

**事务相关异常**——数据改了没生效，往往是没加 `@Transactional`：
```java
@Transactional  // 别忘了这个
public void updateUser(User user) {
    userMapper.update(user);
}
```

###### 4. MyBatis 的 Mapper 找不到的问题如何排查？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#4. Mapper找不到排查|📖]] 系统化排查，按顺序走一遍基本能定位：

1. **基础配置检查**：Mapper 接口有没有 `@Mapper` 注解，或者在 `@MapperScan` 指定的包路径内？MyBatis 配置文件的 `<mappers>` 配置对不对？

2. **XML 映射文件检查**：`namespace` 必须与 Mapper 接口全限定名完全一致（包括大小写），方法名与 `id` 属性一致，XML 文件位置和加载路径要对上。

3. **Maven 构建配置**：最容易被遗忘的一步，确保 `pom.xml` 配置了资源包含：
```xml
<build>
  <resources>
    <resource>
      <directory>src/main/java</directory>
      <includes><include>**/*.xml</include></includes>
    </resource>
    <resource>
      <directory>src/main/resources</directory>
    </resource>
  </resources>
</build>
```

4. **运行时验证**：写个测试确认 Bean 是否加载成功：
```java
@SpringBootTest
class MapperLoadTest {
    @Autowired
    private ApplicationContext context;

    @Test
    void testMapperLoaded() {
        String[] beanNames = context.getBeanNamesForType(UserMapper.class);
        assertTrue(beanNames.length > 0);
    }
}
```

5. **特殊场景**：多模块项目要确保主模块能扫到子模块的 Mapper；热部署环境注意 IDE 自动编译设置；版本不兼容时也会出现此类问题。

###### 5. MyBatis 如何调试 SQL 参数绑定？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#5. 调试参数绑定|📖]] 参数绑定问题排查有几个层次，从轻到重依次用。

**日志级别调到 TRACE**——最简单，直接看详细绑定信息：
```yaml
logging:
  level:
    com.example.mapper: trace   # TRACE级别显示最详细的参数信息
    org.apache.ibatis: debug
    org.springframework.jdbc: debug
```

**自定义 TypeHandler 埋点**——在类型转换过程中加日志：
```java
public class DebugTypeHandler extends BaseTypeHandler<String> {
    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, String parameter, JdbcType jdbcType) throws SQLException {
        System.out.println("参数位置：" + i + ", 值：" + parameter);
        ps.setString(i, parameter);
    }
    // 其他方法...
}
```

**源码断点调试**：在 `DefaultParameterHandler.setParameters` 打断点，观察 `parameterMappings` 和 `parameterObject` 的实际值，跟踪 TypeHandler 的处理流程。

**注意边界情况**——null 值和复杂对象嵌套是高频坑：
```java
// XML里处理null值（别直接用，加null判断）
<if test="name != null and name != ''">AND name = #{name}</if>
<if test="age != null">AND age = #{age}</if>

// 复杂对象嵌套属性访问，用属性路径
// #{user.name}, #{createTime}
```

动态 SQL 调试时，可在 `<if>`/`<choose>` 节点旁加注释标记，配合日志输出验证分支是否走对。
