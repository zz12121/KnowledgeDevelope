###### 1. MyBatis 的最佳实践有哪些？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#6. 最佳实践总结|📖]] 从三个维度来说。

**架构设计层面**：SQL 与 Java 代码要分离，Mapper 接口定义方法，XML 写 SQL，提高可读性和可维护性。简单场景用注解减少配置，复杂 SQL 用 XML 更直观。分层架构（Entity → Mapper → Service）各层职责清晰，不要把业务逻辑塞进 Mapper 层。

**性能优化层面**：复杂条件查询优先用 `<if>`、`<choose>`、`<foreach>` 等动态 SQL，绝对不要用字符串拼接——那是 SQL 注入的温床。批量操作要用 `<foreach>` 或 `BatchExecutor`，一条一条插的写法在大数据量下会拖垮性能。二级缓存慎用，读多写少的场景可以考虑，但频繁更新的数据一定要配好缓存刷新策略，否则会读到脏数据。

**可维护性层面**：统一异常处理，Mapper 方法出的错别裸抛，该包装包装。Mapper 方法要有单元测试覆盖，别全靠集成测试才发现问题。定期代码审查，MyBatis 的 XML 容易膨胀，及时整理避免成为维护噩梦。

###### 2. MyBatis 项目中如何组织 Mapper 文件？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#7. 项目组织|📖]] 标准目录结构是这样的：
```
src/
├── main/
│   ├── java/
│   │   └── com/example/
│   │       ├── entity/          # 实体类
│   │       ├── mapper/          # Mapper接口
│   │       │   ├── UserMapper.java
│   │       │   └── OrderMapper.java
│   │       └── service/         # 业务逻辑层
│   └── resources/
│       ├── mapper/              # XML映射文件
│       │   ├── UserMapper.xml
│       │   └── OrderMapper.xml
│       └── mybatis-config.xml   # MyBatis配置
```

Maven 项目记得在 `pom.xml` 里配好资源包含，否则 XML 不会被打包进去：
```xml
<build>
    <resources>
        <resource>
            <directory>src/main/resources</directory>
            <includes>
                <include>**/*.xml</include>
            </includes>
        </resource>
    </resources>
</build>
```

**组织策略**：按业务模块划分，UserMapper 对应用户模块，OrderMapper 对应订单模块。接口和 XML 同名（`UserMapper.java` ↔ `UserMapper.xml`）是铁律，别自己发明命名规则。Spring Boot 项目用 `@MapperScan` 自动扫描，不用每个都手动注册。

###### 3. MyBatis 的命名规范有哪些建议？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#8. 命名规范|📖]] 几条核心规范：

**类与接口**：类名用帕斯卡命名法（`UserMapper`），Mapper 接口统一以 `Mapper` 结尾，清晰表明职责。

**方法命名**：用驼峰式动词短语，语义要清晰——`saveUser`、`updateUser`、`deleteById`、`findUserById`、`selectAll`。查询用 `find`/`select`，写入用 `save`/`insert`，更新用 `update`，删除用 `delete`，保持一致。

**数据库字段映射**：强烈建议开启驼峰命名自动映射，省去大量手写 `<result>` 配置：
```xml
<settings>
    <setting name="mapUnderscoreToCamelCase" value="true"/>
</settings>
```
这样 `user_name` 自动映射到 `userName`，`create_time` 到 `createTime`，非常省事。

**变量与常量**：变量驼峰式（`userName`、`orderList`），常量全大写下划线分隔（`MAX_BATCH_SIZE`）。

###### 4. MyBatis 在微服务架构中如何使用？
[[22_MybatisKnowledge/07_Spring集成/01、MyBatis-Spring集成原理#6. 微服务使用|📖]] 在微服务里用 MyBatis 和单体应用基本一样，核心是每个服务独立数据库，各自管理自己的 Mapper。

**依赖配置**：
```xml
<dependency>
    <groupId>org.mybatis.spring.boot</groupId>
    <artifactId>mybatis-spring-boot-starter</artifactId>
    <version>2.1.4</version>
</dependency>
```

**数据源配置**，连接池参数要根据服务流量合理调整：
```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/your_database
    username: your_username
    password: your_password
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
```

**Java 配置类**替代 XML，更灵活：
```java
@Configuration
@MapperScan("com.example.demo.mapper")
public class MyBatisConfig {
    @Bean
    public SqlSessionFactory sqlSessionFactory(DataSource dataSource) throws Exception {
        SqlSessionFactoryBean sessionFactory = new SqlSessionFactoryBean();
        sessionFactory.setDataSource(dataSource);
        sessionFactory.setMapperLocations(new PathMatchingResourcePatternResolver()
                .getResources("classpath*:mapper/**/*.xml"));
        return sessionFactory.getObject();
    }
}
```

**微服务数据隔离原则**：每个服务独立数据库，通过 API 而不是直接共享数据库来通信。结合 Spring Cloud 组件实现负载均衡和熔断，MyBatis 本身不需要特殊处理。

###### 5. MyBatis 如何与分库分表中间件（如 Sharding-JDBC）集成？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#9. 分库分表集成|📖]] ShardingSphere（Sharding-JDBC）和 MyBatis 集成非常顺滑，核心原理是 ShardingSphere 实现了 `DataSource` 接口，对 MyBatis 完全透明。

**依赖引入**：
```xml
<dependency>
    <groupId>org.apache.shardingsphere</groupId>
    <artifactId>sharding-jdbc-spring-boot-starter</artifactId>
    <version>${sharding-sphere.version}</version>
</dependency>
```

**分片规则配置示例**：
```yaml
spring:
  shardingsphere:
    datasource:
      names: ds0,ds1
      ds0: # 数据源0配置
      ds1: # 数据源1配置
    sharding:
      tables:
        user:
          actualDataNodes: ds${0..1}.user_${0..15}
          tableStrategy:
            inline:
              shardingColumn: user_id
              algorithmExpression: user_${user_id % 16}
```

**集成要点**：SQL 尽量避免数据库特异性语法，确保分片后能正确执行；用 ShardingSphere 的分布式主键生成器替代数据库自增 ID；分布式场景下配好柔性事务或 XA 事务保证数据一致性。

###### 6. MyBatis 的安全性注意事项有哪些？
[[22_MybatisKnowledge/10_最佳实践与选型/01、调试排查与最佳实践#10. 安全注意事项|📖]] 几个重要的安全点，一定要注意。

**SQL 注入防护**——这是重中之重：
```xml
<!-- 安全：#{}预编译，参数作为纯数据处理 -->
<select id="findUser" resultType="User">
    SELECT * FROM users WHERE name = #{name}
</select>

<!-- 危险：${}字符串拼接，直接插入SQL文本 -->
<select id="findUser" resultType="User">
    SELECT * FROM users WHERE name = '${name}'
</select>
```
原则很简单：**能用 `#{}` 就不用 `${}`**，动态表名、排序字段等不得不用 `${}` 的地方，必须加白名单校验。

**敏感数据保护**：通过自定义 TypeHandler 对敏感字段（手机号、身份证、密码）在存库前加密，读出后解密，业务层完全透明。SQL 日志里也要过滤敏感参数，别让密码明文出现在日志里。

**权限控制**：数据库用户只授最小必要权限，不要给 SELECT 权限的用户 DELETE 权限。通过 MyBatis 插件记录重要操作审计日志，实现行级数据权限控制。

**事务安全**：涉及多步数据操作的方法要加 `@Transactional(rollbackFor = Exception.class)`，确保原子性。

**配置安全**：数据库连接信息用配置中心管理（Spring Cloud Config 或 Nacos），定期轮换密码，生产环境开启 SSL 加密连接。
