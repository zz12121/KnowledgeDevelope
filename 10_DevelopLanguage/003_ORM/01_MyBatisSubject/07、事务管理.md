###### 1. MyBatis 如何管理事务？
[[22_MybatisKnowledge/06_事务与数据源/01、事务管理与传播行为#1. MyBatis 内置事务管理|📖]] MyBatis 的事务管理基于 `Transaction` 接口，有两种实现：**JdbcTransaction** 和 **ManagedTransaction**。

**JdbcTransaction** 用于 MyBatis 独立使用的场景，直接封装 JDBC 连接的事务操作。它的 `commit()` 调用 `connection.commit()`，`rollback()` 调用 `connection.rollback()`，完全由代码控制。

**ManagedTransaction** 用于容器托管（如 Spring）的场景，它的 `commit()` 和 `rollback()` 是**空实现**——事务生命周期由容器（Spring）负责，MyBatis 自己不介入。

**独立使用时的事务流程**：

```java
SqlSession sqlSession = sqlSessionFactory.openSession();  // 默认 autoCommit=false，开启事务
try {
    userMapper.insertUser(new User("张三"));
    userMapper.updateUserStatus(1L, "ACTIVE");
    sqlSession.commit();   // 手动提交
} catch (Exception e) {
    sqlSession.rollback(); // 异常时回滚
} finally {
    sqlSession.close();
}
```

`TransactionFactory`（通过 `<transactionManager type="JDBC"/>` 配置）负责创建对应的 Transaction 实例。

###### 2. MyBatis 中事务的隔离级别如何设置？
[[22_MybatisKnowledge/06_事务与数据源/01、事务管理与传播行为#1. MyBatis 内置事务管理|📖]] MyBatis 本身不实现隔离级别，而是依赖底层数据库或 Spring 框架。

**独立使用时**，通过 `openSession()` 传入 `TransactionIsolationLevel` 枚举：

```java
SqlSession session = sqlSessionFactory.openSession(TransactionIsolationLevel.READ_COMMITTED);
```

**与 Spring 集成时**（更常见），通过 `@Transactional` 的 `isolation` 属性设置：

```java
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void criticalOperation() { ... }
```

Spring 会通过 `DataSourceTransactionManager` 将隔离级别设置到当前连接上。常用的隔离级别是 `READ_COMMITTED`（避免脏读，大多数场景的默认选择）。

###### 3. MyBatis 与 Spring 集成后，事务是如何管理的？
[[22_MybatisKnowledge/06_事务与数据源/01、事务管理与传播行为#3. Spring 事务整合机制|📖]] 与 Spring 集成后，事务的主导权完全交给 Spring，MyBatis 的 `ManagedTransaction` 只是"傀儡"。

**整合的核心机制**分四层：

首先，**`DataSourceTransactionManager`** 是实际的事务管理器，负责开启/提交/回滚事务。

其次，**Spring AOP** 在方法调用时织入 `TransactionInterceptor`，当检测到 `@Transactional` 注解，事务拦截器在目标方法执行前开启事务，执行完毕后根据是否有异常决定提交还是回滚。

再次，**`TransactionSynchronizationManager`** 通过 ThreadLocal 将数据库连接（`ConnectionHolder`）绑定到当前线程，保证同一个事务中所有操作使用同一个 Connection。

最后，**`SqlSessionTemplate`** 是 MyBatis-Spring 的关键桥梁，它在每次 SQL 操作时，先从 `TransactionSynchronizationManager` 获取当前事务的 Connection，再包装成 SqlSession 使用，确保所有操作都在同一个事务内。

###### 4. MyBatis 的事务传播行为有哪些？
[[22_MybatisKnowledge/06_事务与数据源/01、事务管理与传播行为#5. Spring 七种传播行为|📖]] 事务传播行为定义了**方法间互相调用时，事务如何传播**，是 Spring 的概念，通过 `@Transactional(propagation = Propagation.XXX)` 配置。

最常用的三种：

**REQUIRED（默认）**：有事务就加入，没有就新建。绝大多数情况用这个就对了。调用方有事务，被调用方直接用，一荣俱荣一损俱损；调用方没有事务，被调用方自己开一个。

**REQUIRES_NEW**：不管当前有没有事务，都开一个**独立的新事务**，并挂起当前事务。新事务失败只回滚自己，不影响调用方的事务。典型场景：操作日志记录——主业务失败不应该影响日志写入，日志写入失败也不应该影响主业务。

**NESTED**：如果有外层事务，在外层事务内开一个**嵌套事务**（通过 Savepoint 实现）。嵌套事务可以独立回滚到 Savepoint，但最终提交依赖外层事务。外层回滚则嵌套事务也回滚。适合"尝试性操作"——嵌套事务失败可以只撤销该步骤，外层继续。

其他传播行为：**SUPPORTS**（有事务加入，没有就非事务执行）、**MANDATORY**（必须有事务，否则抛异常）、**NOT_SUPPORTED**（挂起当前事务，以非事务执行）、**NEVER**（有事务则抛异常）。

###### 5. MyBatis 如何处理事务回滚？
[[22_MybatisKnowledge/06_事务与数据源/01、事务管理与传播行为#6. 回滚规则|📖]] **独立使用 MyBatis** 时，回滚是手动编程式的，在 catch 块里调用 `sqlSession.rollback()`，如上第1题代码示例所示。

**与 Spring 集成时**，回滚由 `TransactionInterceptor` 自动处理。但有一个重要的默认行为需要记住：**Spring 默认只在抛出 `RuntimeException` 或 `Error` 时才自动回滚**，受检异常（Checked Exception）不触发回滚。

这个默认行为在业务开发中经常踩坑——自定义的业务异常（extends Exception）抛出时，事务不会回滚，数据悄悄提交了。

**最佳实践**：明确指定回滚规则：

```java
@Transactional(rollbackFor = Exception.class)  // 所有 Exception 都回滚，包括受检异常
public void updateUser(User user) throws BusinessException {
    userMapper.update(user);
    // 即使这里抛出 BusinessException（受检异常），也会触发回滚
}
```

另外，如果不想抛出异常但又需要回滚，可以用编程式标记：

```java
TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
```
