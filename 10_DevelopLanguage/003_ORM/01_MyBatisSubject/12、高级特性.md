###### 1. MyBatis 如何处理枚举类型？
[[22_MybatisKnowledge/05_插件与扩展/02、TypeHandler与高级特性#1. 枚举处理|📖]] MyBatis 提供了两种内置的枚举处理器，并支持自定义处理逻辑。

**内置枚举处理器**，选择哪个取决于你想往数据库里存什么值：
- **EnumTypeHandler**：默认处理器，将枚举的名称（`name()`）作为字符串存储，比如 `ComputerState.OPEN` 存的是 `"OPEN"`。
- **EnumOrdinalTypeHandler**：将枚举的序号（`ordinal()`）作为整数存储，比如 `ComputerState.OPEN` 存的是 `0`（第一个枚举值）。

**自定义枚举处理器**——当内置的不够用时（比如希望存枚举的自定义 code 值），可以这样做：

1. 定义标识接口，约定枚举行为：
```java
public interface BaseCodeEnum {
    int getCode(); // 约定返回枚举的自定义编码
}
```
2. 让枚举实现该接口：
```java
public enum ComputerState implements BaseCodeEnum {
    OPEN(10), CLOSE(11), OFF_LINE(12), FAULT(200), UNKNOWN(255);
    private int code;
    ComputerState(int code) { this.code = code; }
    @Override public int getCode() { return this.code; }
}
```
3. 继承 `BaseTypeHandler` 编写处理器——`setNonNullParameter` 负责写入（调 `getCode()`），`getNullableResult` 负责读出（根据 code 值还原枚举实例）。
4. 注册处理器：XML 配置 `<typeHandlers>` 标签，或 Spring Boot 中通过 `@Configuration` 动态扫描所有实现 `BaseCodeEnum` 的枚举自动注册。

**源码角度**：枚举处理的核心是 `TypeHandler`。设置参数或获取结果时，`DefaultParameterHandler` 和 `DefaultResultSetHandler` 都会通过 `TypeHandlerRegistry` 查找对应的 `TypeHandler` 做转换。

###### 2. MyBatis 如何自定义 TypeHandler？
[[22_MybatisKnowledge/05_插件与扩展/02、TypeHandler与高级特性#2. 自定义TypeHandler|📖]] 自定义 TypeHandler 是扩展 MyBatis 类型转换能力的关键方式，核心是继承 `BaseTypeHandler` 抽象类。

**实现步骤**

1. **创建处理器类**：继承 `org.apache.ibatis.type.BaseTypeHandler<T>`，重点实现四个方法：
   - `setNonNullParameter`：将 Java 类型参数转为 JDBC 类型，设置到 `PreparedStatement`。
   - 三个 `getNullableResult` 重载：分别从 `ResultSet`（按列名、按列索引）和 `CallableStatement` 中读取值并转为 Java 类型。

   下面是一个将 Java 对象与 JSON 字符串互转的典型示例：
    ```java
    @MappedTypes(MyComplexType.class)   // 指定处理的Java类型
    @MappedJdbcTypes(JdbcType.VARCHAR)  // 指定对应的JDBC类型
    public class JsonTypeHandler extends BaseTypeHandler<MyComplexType> {
        private static final ObjectMapper mapper = new ObjectMapper();
        @Override
        public void setNonNullParameter(PreparedStatement ps, int i, MyComplexType parameter, JdbcType jdbcType) throws SQLException {
            ps.setString(i, mapper.writeValueAsString(parameter)); // 对象 → JSON字符串
        }
        @Override
        public MyComplexType getNullableResult(ResultSet rs, String columnName) throws SQLException {
            String json = rs.getString(columnName);
            return json == null ? null : mapper.readValue(json, MyComplexType.class); // JSON字符串 → 对象
        }
        // ... 另外两个getNullableResult方法
    }
    ```

2. **注册处理器**，三种方式择一：
   - XML 配置 `<typeHandlers>` 标签显式注册。
   - Spring Boot 配置 `mybatis.type-handlers-package`，自动包扫描注册。
   - 配合 `@MappedTypes` + `@MappedJdbcTypes` 注解，扫描时自动识别。

**源码机制**：MyBatis 启动时 `XMLConfigBuilder` 解析配置，初始化 `TypeHandlerRegistry`，将 Java 类型、JDBC 类型与 `TypeHandler` 实例建立三元映射关系。执行 SQL 时，`StatementHandler` 通过注册表获取对应处理器完成转换。

###### 3. MyBatis 的 TypeHandler 有什么作用？
[[22_MybatisKnowledge/05_插件与扩展/02、TypeHandler与高级特性#3. TypeHandler的作用|📖]] TypeHandler 是 MyBatis 的核心组件之一，本质就是 **Java 类型与 JDBC 类型之间的双向转换桥梁**，主要体现在两个环节：

**写入时（参数设置）**：执行 `PreparedStatement` 时，TypeHandler 把 Java 属性值转为 JDBC 类型，调用 `ps.setXXX()` 设置到 SQL 占位符上。

**读取时（结果获取）**：从 `ResultSet` 拿数据时，TypeHandler 把 JDBC 类型的结果值转为 Java 对象，填充到结果对象的对应属性中。

**具体应用场景**：
- **基本类型和常见对象**：String、Integer、Date 等，MyBatis 内置了大量 TypeHandler，自动搞定。
- **枚举类型处理**：如上题所述。
- **复杂对象与特殊格式**：把对象序列化为 JSON 字符串存库，或反过来把 JSON 还原成对象；处理 BLOB/CLOB 大对象。
- **数据加密脱敏**：写入前加密，读出后解密，业务层完全透明。

###### 4. MyBatis 如何实现存储过程调用？
[[22_MybatisKnowledge/05_插件与扩展/02、TypeHandler与高级特性#4. 存储过程|📖]] MyBatis 通过 `CallableStatement` 支持存储过程调用，关键是声明 `statementType="CALLABLE"`。

**Mapper 接口**：
```java
public interface UserMapper {
    // 方式一：用Map接收OUT参数
    void callProcedureWithMap(Map<String, Object> params);
    // 方式二：注解方式
    @Select("{ CALL get_user_by_id(#{id, mode=IN}, #{userName, mode=OUT, jdbcType=VARCHAR}) }")
    @Options(statementType = StatementType.CALLABLE)
    User callProcedureWithAnnotation(@Param("id") Long id, @Param("userName") String userName);
}
```

**XML 配置**，必须加上 `statementType="CALLABLE"`：
```xml
<select id="callUserProcedure" statementType="CALLABLE" parameterType="map">
    {call getUserDetail(
        #{userId, mode=IN, jdbcType=INTEGER},
        #{userName, mode=OUT, jdbcType=VARCHAR},
        #{userAge, mode=OUT, jdbcType=INTEGER}
    )}
</select>
```

**OUT 参数处理**：MyBatis 通过 `CallableStatementHandler` 调用存储过程，`DefaultResultSetHandler` 处理输出参数，OUT 参数值会被回写到你传入的 Map 对象或注解指定的参数中。

###### 5. MyBatis 如何处理 BLOB 和 CLOB 类型？
[[22_MybatisKnowledge/05_插件与扩展/02、TypeHandler与高级特性#5. BLOB与CLOB处理|📖]] MyBatis 内置了 `BlobTypeHandler` 和 `ClobTypeHandler` 来处理大对象类型。

**基本映射方式**：
- `BLOB`（二进制大对象）→ Java 的 `byte[]` 或 `java.sql.Blob`。
- `CLOB`（字符大对象）→ Java 的 `String` 或 `java.sql.Clob`。

**流式处理（防内存溢出）**：对于超大 BLOB/CLOB，不能一次性加载，要用 `ResultHandler` 做流式处理：
```java
public interface BlobMapper {
    @Select("SELECT file_content FROM large_files WHERE id = #{id}")
    void getBlobContent(Long id, ResultHandler<byte[]> handler); // 流式处理，逐条消费
}
```

**源码角度**：`BlobTypeHandler` 的 `getResult` 方法会调用 `ResultSet.getBlob()` 拿到 `Blob` 对象，再通过 `Blob.getBinaryStream()` 流式读取。`DefaultResultSetHandler` 负责管理整个过程并确保资源正确关闭。

###### 6. MyBatis 如何实现动态表名？
[[22_MybatisKnowledge/05_插件与扩展/02、TypeHandler与高级特性#6. 动态表名|📖]] MyBatis 不直接支持 XML 中动态表名，但有三种方案，安全性依次递增。

**方案一：`${}` 字符串替换（不推荐）**——有 SQL 注入风险，只在表名来源可信时才考虑：
```xml
<select id="selectByTableName" resultType="map">
    SELECT * FROM ${tableName} WHERE id = #{id}
</select>
```

**方案二：Provider 类（MyBatis 3.4.2+）**——在 Java 代码中动态拼接 SQL：
```java
public class DynamicTableProvider {
    public String selectByTableName(Map<String, Object> params) {
        String tableName = (String) params.get("tableName");
        return "SELECT * FROM " + tableName + " WHERE id = #{id}";
    }
}
@SelectProvider(type = DynamicTableProvider.class, method = "selectByTableName")
List<Map<String, Object>> selectByTableName(@Param("tableName") String tableName, @Param("id") Long id);
```

**方案三：拦截器（最安全灵活）**——实现 `Interceptor` 接口，拦截 Executor 的查询方法，在运行时修改 `BoundSql` 中的 SQL 文本，将预定义占位符替换为实际表名，配合白名单校验杜绝注入风险。

###### 7. MyBatis 的鉴别器（discriminator）有什么作用？
[[22_MybatisKnowledge/02_映射与SQL/03、结果映射与关联查询#4. 鉴别器|📖]] `<discriminator>` 是 `<resultMap>` 中的一个强大元素，作用类似于 Java 的 switch-case——根据某列的值决定用哪套结果映射。

**典型应用场景**：单表继承，一张表存多种类型的数据，每种类型有不同的字段子集。比如 vehicle 表同时存车辆和卡车的记录，通过 `vehicle_type` 字段区分。

**配置示例**：
```xml
<resultMap id="vehicleResultMap" type="Vehicle">
    <id property="id" column="id"/>
    <result property="brand" column="brand"/>
    <!-- 根据vehicle_type列的值选择不同映射 -->
    <discriminator javaType="String" column="vehicle_type">
        <case value="CAR" resultMap="carResultMap"/>
        <case value="TRUCK" resultMap="truckResultMap"/>
    </discriminator>
</resultMap>

<resultMap id="carResultMap" type="Car" extends="vehicleResultMap">
    <result property="doorCount" column="door_count"/>
</resultMap>

<resultMap id="truckResultMap" type="Truck" extends="vehicleResultMap">
    <result property="payload" column="payload"/>
</resultMap>
```

查询时，MyBatis 根据 `vehicle_type` 字段的值，将这行数据映射为 `Car` 对象还是 `Truck` 对象，并分别填充各自特有的字段。

**源码角度**：`DefaultResultSetHandler` 处理结果集时会检查 `ResultMap` 是否包含 `Discriminator`。如果有，先拿到鉴别器列的值，匹配对应 Case，再用 Case 中指定的 `ResultMap` 创建和填充对象。

###### 8. MyBatis 如何实现结果集的嵌套映射？
[[22_MybatisKnowledge/02_映射与SQL/03、结果映射与关联查询#2. 嵌套映射|📖]] MyBatis 通过 `<association>`（一对一）和 `<collection>`（一对多）实现复杂的嵌套对象映射。

**一对一关联（`<association>`）**——比如订单关联一个用户：
```xml
<resultMap id="orderDetailMap" type="Order">
    <id property="id" column="order_id"/>
    <result property="orderNo" column="order_no"/>
    <!-- 连接查询方式：直接映射JOIN结果 -->
    <association property="user" javaType="User">
        <id property="id" column="user_id"/>
        <result property="username" column="username"/>
    </association>
    <!-- 嵌套查询方式：触发额外的SELECT（注意N+1） -->
    <association property="user" column="user_id" javaType="User" select="selectUserById"/>
</resultMap>
```

**一对多关联（`<collection>`）**——比如博客关联多篇文章：
```xml
<resultMap id="blogDetailMap" type="Blog">
    <id property="id" column="blog_id"/>
    <result property="title" column="title"/>
    <!-- 连接查询方式 -->
    <collection property="posts" ofType="Post">
        <id property="id" column="post_id"/>
        <result property="content" column="content"/>
    </collection>
    <!-- 嵌套查询方式 -->
    <collection property="posts" column="blog_id" ofType="Post" select="selectPostsByBlogId"/>
</resultMap>
```

**两种方式怎么选**：
- **连接查询（JOIN）**：单条 SQL 一次性拿所有数据，性能好，适合关联数据量不大的情况。
- **嵌套查询（select）**：先查主表再查关联表，会有 N+1 问题，但配合 `lazyLoadingEnabled=true` 可延迟加载，按需查询。

**源码角度**：`DefaultResultSetHandler` 的 `handleResultSets` → `getRowValue` 是嵌套映射的核心链路。普通属性走 `applyPropertyMappings`，嵌套映射走 `applyNestedResultMappings`。嵌套查询则通过 `Executor` 再次发起查询。
