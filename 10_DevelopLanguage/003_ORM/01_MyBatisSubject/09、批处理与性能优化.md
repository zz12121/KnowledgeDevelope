###### 1. MyBatis 如何实现批量插入？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#1. 批量插入的三种方式|📖]] MyBatis 实现批量插入主要有三种方式，性能差异显著。

**方式一：`<foreach>` 单条 SQL（最推荐）**

将多条数据合并成一条 INSERT 语句，显著减少网络往返次数：

```xml
<insert id="batchInsert" parameterType="java.util.List">
    INSERT INTO user(name, age, email) VALUES
    <foreach collection="list" item="item" separator=",">
        (#{item.name}, #{item.age}, #{item.email})
    </foreach>
</insert>
```

每批建议 500~1000 条，超过就分批调用，避免单条 SQL 太长给数据库解析带来压力。

**方式二：`ExecutorType.BATCH`（JDBC 级别批处理）**

基于 JDBC 的 `addBatch()` 机制，`BatchExecutor` 将多条 SQL 缓存后一次性提交：

```java
SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH);
UserMapper mapper = sqlSession.getMapper(UserMapper.class);
try {
    for (int i = 0; i < userList.size(); i++) {
        mapper.insert(userList.get(i));      // 加入批处理队列，不立即执行
        if ((i + 1) % 1000 == 0) {
            sqlSession.flushStatements();    // 每 1000 条提交一次
        }
    }
    sqlSession.commit();
} catch (Exception e) {
    sqlSession.rollback();
} finally {
    sqlSession.close();
}
```

MySQL 连接 URL 加 `rewriteBatchedStatements=true` 后，驱动会自动把 `executeBatch()` 重写为单条多值 INSERT，性能提升几倍到十几倍。

**方式三：循环单条插入**（不推荐）—— N 次 INSERT，N 次网络往返，数据量一大就是性能灾难。

###### 2. MyBatis 的批处理（Batch）模式是什么？如何使用？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#2. 三种执行器对比|📖]] `ExecutorType.BATCH` 批处理模式基于 JDBC 的 `addBatch()` 机制：调用 mapper 方法时不立即执行 SQL，而是加入批处理队列，调用 `flushStatements()` 时才真正执行 `executeBatch()`，一次性提交所有缓存的 SQL。

MyBatis 提供三种执行器：**SIMPLE**（默认）每次执行为 SQL 创建新的 `PreparedStatement`，通用场景；**REUSE** 复用相同 SQL 的 `PreparedStatement`，减少预编译开销，适合相同 SQL 高频执行；**BATCH** 缓存多个操作延迟执行，专门用于大数据量批量操作。

Spring Boot 中如果某个场景需要批处理，推荐局部使用（在 Service 方法内手动创建 BATCH SqlSession），而不是全局修改默认执行器类型，避免影响其他操作。

**性能优化要点**：每 500~1000 条调用一次 `flushStatements()` 避免内存溢出；MySQL URL 加 `rewriteBatchedStatements=true`；批处理必须在事务中执行确保失败时回滚。

###### 3. MyBatis 如何优化大数据量查询？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#3. 大数据量查询优化|📖]] 大数据量查询的核心问题是内存压力。如果一次性把几十万条数据加载到 JVM，很容易 OOM。

**流式查询（ResultHandler）**：`DefaultResultSetHandler` 通过 `ResultSet.TYPE_FORWARD_ONLY` 和 `fetchSize` 设置，让 JDBC 驱动按批从数据库拉取数据而不是全量加载，逐条处理后立即可以被 GC：

```java
sqlSession.select("com.example.UserMapper.streamingQuery", param,
    context -> {
        User user = (User) context.getResultObject();
        processUser(user);  // 处理后即可释放
    });
```

**游标查询（Cursor，MyBatis 3.4.6+）**：更优雅的方式，使用 try-with-resources：

```java
try (Cursor<User> cursor = userMapper.scanAll()) {
    for (User user : cursor) {
        processUser(user);
    }
}
```

**深分页优化**：传统 `LIMIT offset, size` 在 offset 很大时性能极差（MySQL 需要扫描并丢弃前 offset 行）。改用基于索引的游标分页：`WHERE id > #{lastId} ORDER BY id LIMIT #{pageSize}`，每次查询时间稳定，与总数据量无关。

###### 4. MyBatis 的 N+1 查询问题是什么？如何解决？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#4. N+1 查询问题|📖]] N+1 是一个经典的性能陷阱：查询主表 N 条记录，再对每条记录单独发起一次关联查询，共执行 1+N 次 SQL。

典型场景：用嵌套查询方式加载订单列表的用户信息，查 100 个订单 → 触发 100 次用户查询 = 101 次 SQL。随着数据量增大，问题越来越严重。

**解决方案一（推荐）：JOIN 查询一次搞定**

```xml
<select id="selectUserWithOrders" resultMap="userWithOrdersMap">
    SELECT u.id as user_id, u.name,
           o.id as order_id, o.amount as order_amount
    FROM user u LEFT JOIN orders o ON u.id = o.user_id
    WHERE u.id = #{id}
</select>
```

**解决方案二：先查主表，再批量 IN 查关联表**

```sql
-- 第一步：查所有用户
SELECT * FROM user WHERE ...
-- 第二步：一次 IN 查询搞定所有订单
SELECT * FROM orders WHERE user_id IN (1, 2, 3, ..., 100)
```

**解决方案三：延迟加载**（不能根本解决，但延迟了查询时机）—— 对列表场景意义不大，只适合少量数据且关联对象不常被访问的场景。

###### 5. MyBatis 如何避免全表扫描？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#5. SQL 调优要点|📖]] 避免全表扫描的关键是 **WHERE 条件要能命中索引**，以及**限制结果集大小**。

**MyBatis 动态 SQL 防全表扫描**：用 `<where>` + `<if>` 确保条件字段不为空时才加入 SQL，避免空条件导致没有 WHERE 子句：

```xml
<select id="searchUsers" resultType="User">
    SELECT * FROM user
    <where>
        <if test="name != null and name != ''">AND name = #{name}</if>
        <if test="age != null">AND age = #{age}</if>
    </where>
    LIMIT 1000  <!-- 大数据量查询必须限制结果集 -->
</select>
```

**SQL 层面**：避免在索引列上使用函数（`WHERE YEAR(create_time) = 2024` 会导致索引失效，应该改成范围查询）；LIKE 查询用前缀匹配（`LIKE 'value%'` 可走索引，`LIKE '%value%'` 不行）；定期用 EXPLAIN 分析执行计划。

###### 6. MyBatis 的执行器（Executor）有哪几种类型？各有什么特点？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#2. 三种执行器对比|📖]] Executor 是 MyBatis 的调度核心，负责 SQL 执行和缓存处理，有三种实现，前面已详细介绍，这里补充源码层面。

所有 Executor 都继承 `BaseExecutor`，`BaseExecutor.query()` 是查询的模板方法：先创建 CacheKey，查一级缓存（`localCache`），命中直接返回，未命中则调用子类的 `doQuery()` 方法查数据库，结果写入 `localCache`。

`SimpleExecutor.doQuery()` 每次都调用 `configuration.newStatementHandler()` 创建新的 `PreparedStatement`；`ReuseExecutor` 内部维护一个 `statementMap` 缓存，对相同 SQL 复用 `Statement`；`BatchExecutor` 的 `doUpdate()` 调用 `statement.addBatch()`，`flushStatements()` 调用 `statement.executeBatch()`。

###### 7. MyBatis 如何进行 SQL 调优？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#5. SQL 调优要点|📖]] SQL 调优从三个层次入手：SQL 本身、MyBatis 配置、连接池。

**SQL 层面**：EXPLAIN 分析执行计划，确保 WHERE 条件列上有索引；避免 `SELECT *`，只查需要的字段；深分页改游标分页；大批量操作用 `<foreach>` 或 BATCH 模式。

**MyBatis 配置**：开启 `mapUnderscoreToCamelCase`（驼峰自动映射，减少手写 resultMap）；合理配置二级缓存（读多写少的场景）；设置 `defaultFetchSize` 为合适的值（流式查询时控制每次从数据库拉取的行数）。

**监控与诊断**：开发环境用 `StdOutImpl` 打印完整 SQL；生产环境用自定义拦截器记录慢 SQL（阈值一般设 500ms~1s）；结合 Druid 监控页面查看 SQL 执行频率和平均耗时。定期 review 执行次数高但执行时间长的 SQL，这些是优化的优先目标。

###### 8. BaseExecutor 的一级缓存是如何实现的？CacheKey 由哪些部分组成？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#6.2 BaseExecutor：模板方法 + 一级缓存|📖]] BaseExecutor 使用 `PerpetualCache`（底层就是一个 `HashMap`）作为一级缓存，存储在 Executor 对象内部，与 SqlSession 同生命周期。

**query() 模板方法的缓存流程**：
1. 调用 `createCacheKey()` 创建 CacheKey 对象
2. 到 `localCache` 中查找（`localCache.getObject(key)`）
3. 命中则直接返回缓存结果；未命中则调用子类 `doQuery()` 查数据库，结果写入 `localCache`

**CacheKey 的 6 个组成维度**：
1. `MappedStatement.id`（即 Mapper 接口的全限定方法名，如 `com.example.UserMapper.findById`）
2. `RowBounds.offset`（分页偏移量）
3. `RowBounds.limit`（分页限制数）
4. `BoundSql.sql`（包含 `?` 占位符的 SQL 文本）
5. 所有实参值（按 ParameterMapping 顺序依次 update 到 CacheKey）
6. `Environment.id`（多数据源场景下区分不同数据库环境）

CacheKey 内部通过 `hashcode + checksum + count + updateList.equals()` 四维判等，即使哈希碰撞也能通过 `updateList` 精确比较。任意一个维度不同，都不会命中缓存。

**一级缓存失效的三种情况**：
- 调用 `update()` / `insert()` / `delete()` → 自动调用 `clearLocalCache()`
- `<select>` 配置了 `flushCache="true"` → 查询前清空
- `mybatis.configuration.localCacheScope=STATEMENT` → 每次查询后清空（相当于禁用一级缓存）

###### 9. ReuseExecutor 和 SimpleExecutor 的区别是什么？Statement 复用的原理？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#6.4 ReuseExecutor：Statement 复用|📖]] 两者在继承 BaseExecutor 后，核心区别在 `prepareStatement()` 方法的实现：

**SimpleExecutor**：每次调用 `doQuery()` / `doUpdate()` 都通过 JDBC Connection 创建新的 `PreparedStatement`，执行完成后立即 `close()`。优点是实现简单；缺点是相同 SQL 反复编译，有额外开销。

**ReuseExecutor**：内部维护 `Map<String, Statement> statementMap`（key 是 SQL 字符串）：
- 先检查 `statementMap` 是否已有该 SQL 对应的 Statement
- 有则复用（跳过 `connection.prepareStatement()` 的编译步骤），只重新绑定参数
- 没有则新建并放入 `statementMap`
- 事务提交/回滚时，`doFlushStatements()` 关闭所有缓存 Statement 并清空 Map

**适用场景**：在同一个事务/SqlSession 中，如果同一条 SQL 会被反复执行（如 for 循环中逐条 insert 但每次参数不同），ReuseExecutor 能减少 PreparedStatement 的编译次数。如果 SQL 种类很多（每条都不同），反而 HashMap 查找的开销得不偿失，意义不大。

###### 10. BatchExecutor 的批处理机制源码是怎样的？为何连续相同 SQL 才能合并？
[[22_MybatisKnowledge/09_性能与调优/01、批处理与性能优化#6.5 BatchExecutor：JDBC 批处理|📖]] BatchExecutor 实现了 JDBC 批处理，核心依赖两个列表：
- `List<Statement> statementList`：缓存所有 Statement 对象
- `List<BatchResult> batchResultList`：对应每个 Statement 的参数列表和执行结果

**doUpdate() 的合并判断逻辑**：

```java
// 当前 SQL + MappedStatement 与上一条完全相同 → 追加到同一 Statement
if (sql.equals(currentSql) && ms.equals(currentStatement)) {
    stmt = statementList.get(last);      // 复用最后一个 Statement
    handler.parameterize(stmt);          // 绑定新参数
    batchResultList.get(last).addParameterObject(parameterObject); // 记录参数
} else {
    // 不同 SQL → 创建新 Statement，加入 statementList
    stmt = handler.prepare(connection, transaction.getTimeout());
    statementList.add(stmt);
    batchResultList.add(new BatchResult(ms, sql, parameterObject));
    currentSql = sql;
    currentStatement = ms;
}
handler.batch(stmt);  // 调用 PreparedStatement.addBatch()
```

**为何连续相同才能合并**：判断条件是 `sql.equals(currentSql)`，即只与**上一条**比较。这是为了确保 SQL 执行的顺序语义（如先 insert 再 update 不能乱序）。若 update 和 insert 交替执行，每次都会新建 Statement，批处理效果大打折扣。

**最佳实践**：
```
✅ 推荐：1000 条 update → flush → 1000 条 insert → flush（同类型 SQL 集中执行）
❌ 避免：update → insert → update → insert 交替（每次都会新建 Statement）
```

**doFlushStatements()** 遍历 `statementList`，对每个 Statement 调用 `executeBatch()`，获得 `int[]` 更新行数数组，存入对应的 `BatchResult.updateCounts`。全部执行完后清空三个状态字段（`currentSql`、`statementList`、`batchResultList`）。
