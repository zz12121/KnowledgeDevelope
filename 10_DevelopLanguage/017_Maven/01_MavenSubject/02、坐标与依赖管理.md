# 02、坐标与依赖管理

## Q1. Maven 坐标的含义是什么？

Maven 坐标是唯一标识一个构件（jar/war/pom）的"地址"，由三个必填字段构成：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>   <!-- 组织/公司域名反写 -->
    <artifactId>spring-boot-starter-web</artifactId> <!-- 项目/模块名 -->
    <version>3.2.0</version>                        <!-- 版本号 -->
    <scope>compile</scope>                           <!-- 依赖范围（可选，默认compile）-->
    <type>jar</type>                                 <!-- 包类型（可选，默认jar）-->
    <classifier>sources</classifier>                 <!-- 区分同一构件不同形式（可选）-->
</dependency>
```

坐标与本地仓库路径的映射关系：
```
groupId/artifactId/version/artifactId-version.jar

org.springframework.boot/spring-boot-starter-web/3.2.0/
  → ~/.m2/repository/org/springframework/boot/spring-boot-starter-web/3.2.0/
    spring-boot-starter-web-3.2.0.jar
```

## Q2. Maven 的依赖 scope 有哪些？

scope 决定依赖在哪些阶段有效、是否打包进 jar：

| scope | 编译 | 测试 | 运行时 | 是否打包 | 典型示例 |
|-------|------|------|--------|---------|---------|
| **compile**（默认） | ✅ | ✅ | ✅ | ✅ | spring-boot-starter-web |
| **test** | ❌ | ✅ | ❌ | ❌ | junit, mockito |
| **provided** | ✅ | ✅ | ❌ | ❌ | servlet-api（容器提供）|
| **runtime** | ❌ | ✅ | ✅ | ✅ | JDBC 驱动 |
| **system** | ✅ | ✅ | ❌ | ❌ | 本地 jar，需配合 systemPath |
| **import** | - | - | - | - | 仅用于 BOM 导入 |

**重点记忆**：
- `test`：只在 `src/test/` 下编译测试时有效，最终 war/jar 里不包含
- `provided`：编译时要，运行时容器（Tomcat/K8s）提供，不打包
- `runtime`：代码里不直接引用类（如 JDBC 驱动通过 `Class.forName()` 加载），运行时才需要

## Q3. 依赖传递是什么？如何排除传递性依赖？

**依赖传递**：A 依赖 B，B 依赖 C → A 也自动得到 C（传递性依赖）。

优点是不用手动声明所有间接依赖，缺点是可能引入不需要的 jar，甚至产生版本冲突。

```xml
<!-- 排除特定的传递性依赖 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <!-- 排除内置Tomcat，改用Undertow -->
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-tomcat</artifactId>
        </exclusion>
    </exclusions>
</dependency>
<!-- 再加入 Undertow -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-undertow</artifactId>
</dependency>
```

## Q4. Maven 如何解决依赖冲突？两个原则是什么？

Maven 的依赖仲裁遵循两个原则：

**原则一：最短路径优先**
```
A → B → C → X(1.0)
A → D → X(2.0)       # D 到 X 路径更短（2跳 vs 3跳）
                      # 最终使用 X(2.0)
```

**原则二：第一声明优先（路径相同时）**
```
A → B → X(1.0)
A → C → X(2.0)       # 路径一样长（都是2跳）
                      # B 在 pom.xml 中先声明 → 使用 X(1.0)
```

**手动解决冲突（推荐做法）**：
```bash
# 1. 查看依赖树，找冲突
mvn dependency:tree | grep -i "xxx"
mvn dependency:tree -Dincludes=org.slf4j:slf4j-api  # 过滤特定包

# 2. 在 dependencyManagement 中锁定版本
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.slf4j</groupId>
            <artifactId>slf4j-api</artifactId>
            <version>2.0.9</version>  <!-- 强制指定版本 -->
        </dependency>
    </dependencies>
</dependencyManagement>
```

> **追问：如果多模块项目中子模块之间版本冲突怎么处理？**
> 在父 POM 的 `dependencyManagement` 中统一管理所有版本，子模块声明依赖时不写 version，由父 POM 统一锁定。这样版本升级只改父 POM 一处，所有子模块同步生效。

## Q5. 可选依赖（optional）是什么？

```xml
<dependency>
    <groupId>redis.clients</groupId>
    <artifactId>jedis</artifactId>
    <version>5.0.0</version>
    <optional>true</optional>    <!-- 可选依赖 -->
</dependency>
```

`optional=true` 的含义：这个依赖**不会传递**给依赖了我的其他项目。

**适用场景**：框架类库设计时，某些功能是可插拔的。比如 Spring Boot 的 `spring-boot-autoconfigure` 对 Redis/MongoDB 等的依赖都是 optional 的，只有用户自己在 pom.xml 里加了相关依赖才会触发自动配置。

## Q6. classifier 是什么？

classifier 用于区分**同一个坐标但内容不同**的构件：

```xml
<!-- 普通 jar -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>mylib</artifactId>
    <version>1.0.0</version>
</dependency>

<!-- 包含源码的 jar -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>mylib</artifactId>
    <version>1.0.0</version>
    <classifier>sources</classifier>
</dependency>

<!-- 包含 Javadoc 的 jar -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>mylib</artifactId>
    <version>1.0.0</version>
    <classifier>javadoc</classifier>
</dependency>

<!-- 针对特定 JDK 版本编译的 jar -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>mylib</artifactId>
    <version>1.0.0</version>
    <classifier>jdk11</classifier>
</dependency>
```

实际生成的文件：
- `mylib-1.0.0.jar`
- `mylib-1.0.0-sources.jar`
- `mylib-1.0.0-javadoc.jar`
- `mylib-1.0.0-jdk11.jar`

## Q7. dependencyManagement 和 dependencies 的区别？

这是很多人容易混淆的点：

```xml
<!-- dependencyManagement：只是声明版本，不实际引入依赖 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.mysql</groupId>
            <artifactId>mysql-connector-j</artifactId>
            <version>8.3.0</version>  <!-- 锁定版本 -->
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- dependencies：真正引入依赖，不写version会继承上面声明的版本 -->
<dependencies>
    <dependency>
        <groupId>com.mysql</groupId>
        <artifactId>mysql-connector-j</artifactId>
        <!-- 不写version，自动用 8.3.0 -->
    </dependency>
</dependencies>
```

**最佳实践**：在父 POM 的 `dependencyManagement` 中声明所有依赖版本（统一管控），子模块的 `dependencies` 中只声明 groupId+artifactId，版本由父 POM 管理。这样整个多模块项目的依赖版本一目了然。

## Q8. 如何引用本地不在仓库中的 jar？

有几种方案，推荐程度从高到低：

```xml
<!-- 方案1（不推荐）：system scope（只适合临时，路径硬编码）-->
<dependency>
    <groupId>com.oracle</groupId>
    <artifactId>ojdbc8</artifactId>
    <version>19.3</version>
    <scope>system</scope>
    <systemPath>${project.basedir}/libs/ojdbc8-19.3.jar</systemPath>
</dependency>

<!-- 方案2（推荐）：手动安装到本地仓库 -->
<!-- mvn install:install-file 
    -Dfile=ojdbc8-19.3.jar 
    -DgroupId=com.oracle 
    -DartifactId=ojdbc8 
    -Dversion=19.3 
    -Dpackaging=jar -->

<!-- 方案3（团队推荐）：上传到私服，然后正常引用 -->
<!-- mvn deploy:deploy-file 
    -Dfile=ojdbc8-19.3.jar 
    -DgroupId=com.oracle 
    -DartifactId=ojdbc8 
    -Dversion=19.3 
    -DrepositoryId=nexus 
    -Durl=http://nexus.company.com/repository/thirdparty/ -->
```
