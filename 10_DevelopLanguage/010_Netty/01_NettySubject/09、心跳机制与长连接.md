# 09、心跳机制与长连接

---

## 1. Netty 支持哪些心跳类型设置？

Netty 的心跳机制核心依赖 `IdleStateHandler`，从事件类型上支持三种基于空闲检测的心跳：

1. **读空闲（READER_IDLE）**：指定时间内没有从通道读到数据，通常意味着对端没发数据
2. **写空闲（WRITER_IDLE）**：指定时间内没有向通道写数据，通常意味着本端没发数据
3. **全部空闲（ALL_IDLE）**：指定时间内既没有读也没有写

这三种类型通过 `IdleStateEvent` 的 `state()` 方法来区分。用户在自定义 Handler 的 `userEventTriggered()` 方法里捕获这些事件，执行发心跳包、关闭连接等逻辑。

除此之外，Netty 也支持**协议级心跳**，比如 WebSocket 的 Ping/Pong 帧，或者自定义二进制协议中的心跳消息类型——这些需要用户在 ChannelHandler 里自行处理。

## 2. Netty 如何实现心跳机制？

核心是 `IdleStateHandler` + 用户自定义事件处理器，分三步搭建：

**第一步**：把 `IdleStateHandler` 加入 Pipeline：
```java
// readerIdleTime：读空闲超时；writerIdleTime：写空闲超时；allIdleTime：全部空闲超时
pipeline.addLast(new IdleStateHandler(60, 0, 0, TimeUnit.SECONDS));
```

**第二步**：在自定义 Handler 里处理空闲事件：
```java
public class HeartbeatHandler extends ChannelInboundHandlerAdapter {
    private int missedPongs = 0;

    @Override
    public void userEventTriggered(ChannelHandlerContext ctx, Object evt) throws Exception {
        if (evt instanceof IdleStateEvent) {
            IdleStateEvent e = (IdleStateEvent) evt;
            if (e.state() == IdleState.READER_IDLE) {
                missedPongs++;
                if (missedPongs >= 3) {
                    // 连续 3 次未收到对端数据，判定连接失效，主动关闭
                    ctx.close();
                } else {
                    // 发送心跳探测包
                    ctx.writeAndFlush(new HeartbeatMessage());
                }
            }
        } else {
            super.userEventTriggered(ctx, evt);
        }
    }

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) throws Exception {
        // 收到任何消息，重置心跳计数
        missedPongs = 0;
        super.channelRead(ctx, msg);
    }
}
```

**第三步**：服务端收到心跳包后回复响应，客户端在收到响应后重置计时。

**源码视角**：`IdleStateHandler` 继承自 `ChannelDuplexHandler`，在 `channelActive` 时注册定时任务（通过 `HashedWheelTimer` 调度），在 `channelRead` 和 `write` 时更新最后读写时间戳，定时任务执行时比较差值，超阈值则触发 `IdleStateEvent`。

## 3. IdleStateHandler 的作用是什么？

`IdleStateHandler` 是 Netty 提供的**连接空闲状态检测处理器**。它的作用是：在指定时间内没有读/写/读写事件发生时，向 Pipeline 触发一个 `IdleStateEvent`，供后续 Handler 处理。

**核心工作原理**：
1. `Channel` 被激活时（`channelActive`），调用 `initialize()` 为读、写、全部三种空闲分别调度定时任务
2. 每次有读操作完成（`channelReadComplete`），更新"最后读时间戳"，重新调度读空闲检测任务
3. 每次有写操作（拦截 `write` 方法），更新"最后写时间戳"，重新调度写空闲检测任务
4. 定时任务执行时比较当前时间与最后读写时间戳的差值，超阈值则调 `fireUserEventTriggered()` 触发事件

**它只负责检测和触发，不包含任何处理逻辑**——发心跳包还是关闭连接，是业务方自己在 Handler 里决定的。

由于为每个 Channel 创建了额外调度任务，连接数极大时要注意性能影响。Netty 使用高效的 `HashedWheelTimer` 管理这些任务，影响可控。

## 4. Netty 如何实现长连接？

长连接是指 TCP 连接建立后持续保持，供多次数据交换使用。Netty 天然适合长连接，实现要点如下：

**保持连接活性**：
- **应用层心跳（推荐）**：用 `IdleStateHandler` 检测读空闲，超时发 Ping，对端回 Pong，多次未收到响应则主动关闭
- **TCP Keepalive（保底）**：`ChannelOption.SO_KEEPALIVE` 启用 TCP 层保活，但默认时间 2 小时，不够灵活，仅作备用

**服务端关键设计**：
- 用 `ChannelGroup` 统一管理已认证的客户端 Channel，方便广播或分组推送：
```java
public static final ChannelGroup channelGroup = new DefaultChannelGroup(GlobalEventExecutor.INSTANCE);
// 认证成功后加入
channelGroup.add(ctx.channel());
// 广播
channelGroup.writeAndFlush(message);
```
- 用 `AttributeKey` 在 Channel 上挂业务状态（如用户 ID）：
```java
public static final AttributeKey<String> USER_ID = AttributeKey.valueOf("userId");
channel.attr(USER_ID).set("user_123");
```

**客户端关键设计**：
- 保持 Channel 引用，定时发心跳
- 实现断线重连机制（见问题6）

**资源管理**：
- 用 Netty 默认内存池减少 GC 压力
- 合理设置 `ChannelOption.SO_RCVBUF` 和 `SO_SNDBUF`
- 严格管理 ByteBuf 释放，防止内存泄漏

## 5. Netty 的连接管理是如何实现的？

Netty 本身不提供开箱即用的连接池，但提供了底层组件供用户实现连接管理。

**核心管理组件**：

**`ChannelGroup`**：线程安全的 Channel 集合，支持广播、批量关闭：
```java
ChannelGroup allChannels = new DefaultChannelGroup(GlobalEventExecutor.INSTANCE);
allChannels.add(channel);
allChannels.writeAndFlush(message);      // 广播
allChannels.close().awaitUninterruptibly(); // 关闭所有
```

**`AttributeKey`**：在 Channel 上安全存取用户自定义属性：
```java
public static final AttributeKey<Session> SESSION_KEY = AttributeKey.newInstance("session");
channel.attr(SESSION_KEY).set(new Session(userId));
Session s = channel.attr(SESSION_KEY).get();
```

**自定义 ConnectionManager**：实际业务中通常实现一个管理类，内部用 `ConcurrentHashMap` 维护"标识 → Channel"的映射，提供查找、添加、移除和广播方法，并在 `channelInactive` 时自动从映射中清除失效 Channel。

Netty 的 `DefaultChannelGroup` 内部就是 `ConcurrentMap<ChannelId, Channel>`，是个很好的参考实现。

## 6. 如何处理 Netty 中的连接断线重连？

断线重连的核心是：**在 `channelInactive` 中感知断线，用指数退避策略异步重连**。

```java
public class ClientReconnectHandler extends ChannelInboundHandlerAdapter {
    private final Bootstrap bootstrap;
    private int retries = 0;
    private static final int MAX_RETRIES = 10;
    private static final long BASE_DELAY_SECONDS = 1L;

    public ClientReconnectHandler(Bootstrap bootstrap) {
        this.bootstrap = bootstrap;
    }

    @Override
    public void channelInactive(ChannelHandlerContext ctx) throws Exception {
        if (retries < MAX_RETRIES) {
            retries++;
            long delay = Math.min(BASE_DELAY_SECONDS << retries, 60L); // 指数退避，最大 60 秒
            System.out.println("连接断开，" + delay + "秒后尝试第" + retries + "次重连...");
            // 在 EventLoop 中调度，不阻塞
            ctx.channel().eventLoop().schedule(this::reconnect, delay, TimeUnit.SECONDS);
        } else {
            System.err.println("重连次数超过上限，放弃重连");
        }
        super.channelInactive(ctx);
    }

    private void reconnect() {
        bootstrap.connect().addListener((ChannelFutureListener) future -> {
            if (future.isSuccess()) {
                System.out.println("重连成功");
                retries = 0;
                // 重连成功后通常需要在 channelActive 里做初始化（如登录、恢复会话）
            } else {
                System.err.println("重连失败: " + future.cause().getMessage());
                // 再次触发 channelInactive 逻辑，或直接再次调度
            }
        });
    }
}
```

**优化建议**：
- 将重连逻辑封装到独立的 `Reconnector` 类，与业务 Handler 解耦
- 如果服务端有多个实例，重连时可以尝试不同地址，实现简单的负载均衡
- 重连成功后在 `channelActive` 里做鉴权和会话恢复

## 7. Netty 如何实现连接池？

Netty 本身不提供连接池，但基于其异步特性可以方便地构建。连接池的核心是：**维护一组到同一远端的 Channel，按需借出、归还或新建**。

```java
public class SimpleNettyPool {
    private final Bootstrap bootstrap;
    private final BlockingQueue<Channel> idleChannels = new LinkedBlockingQueue<>();
    private final Set<Channel> allChannels = ConcurrentHashMap.newKeySet();
    private final int maxConnections;

    public Channel borrowChannel() throws InterruptedException {
        // 1. 先从空闲队列取
        Channel channel = idleChannels.poll();
        if (channel != null && channel.isActive()) {
            return channel;
        } else if (channel != null) {
            allChannels.remove(channel); // 已失效，丢弃
        }

        // 2. 未达上限则新建连接
        if (allChannels.size() < maxConnections) {
            ChannelFuture future = bootstrap.connect().sync();
            channel = future.channel();
            allChannels.add(channel);
            return channel;
        }

        // 3. 等待其他连接释放，最多等 5 秒
        channel = idleChannels.poll(5, TimeUnit.SECONDS);
        if (channel != null && channel.isActive()) return channel;
        throw new RuntimeException("获取连接超时");
    }

    public void returnChannel(Channel channel) {
        if (channel.isActive()) {
            // 清理 channel 状态（如移除本次请求特有的 Handler）
            idleChannels.offer(channel);
        } else {
            allChannels.remove(channel);
            channel.close();
        }
    }
}
```

**生产级考量**：
- 用 Apache Commons Pool 的 `GenericObjectPool` 管理 Channel，提供丰富配置（最大最小空闲、连接检测）
- 连接池应支持异步获取，返回 `Future<Channel>`
- 设置最大空闲时间，定期检查并关闭长期空闲的连接
- 在微服务架构中，通常直接用 gRPC、Dubbo 或 HTTP 客户端内置的连接池，无需自建

## 高频追问：IdleStateHandler 和 HashedWheelTimer 是怎么配合工作的？

`IdleStateHandler` 内部为每个 Channel 注册**最多三个定时任务**（读空闲、写空闲、全部空闲），这些任务通过 `HashedWheelTimer` 调度。

**HashedWheelTimer 的原理**：时间轮是一个环形数组，数组的每一格代表一个时间刻度（tick）。定时任务按"任务超时时间 / tick 间隔"取模，放到对应的格子里。时间轮的指针每个 tick 前进一格，执行当前格内的所有到期任务。

- **时间复杂度**：添加/取消任务 O(1)，执行检查 O(1)（只处理当前格）
- **适合场景**：大量定时任务但精度要求不高的场景，比如网络连接的超时检测——Netty 中可能有几万个连接，每个连接都有心跳定时器，用 HashedWheelTimer 比 Java 的 `ScheduledExecutorService`（基于堆，O(logN)）高效得多

**配合工作流程**：
1. Channel active 时，IdleStateHandler 调用 `EventLoop.schedule()` 注册读空闲检测任务
2. 这个任务最终由 EventLoop 内的 HashedWheelTimer 管理，到期后放入 EventLoop 的任务队列
3. EventLoop 执行任务时检查最后读时间戳，若空闲超时则调 `fireUserEventTriggered(idleStateEvent)`
4. 有新读操作时，IdleStateHandler 取消旧任务、注册新任务，重置计时
