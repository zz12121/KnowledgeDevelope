# 十五、协议支持

###### 1. 选 HTTP 作为数据传输有什么好处？

HTTP 最大的优势是**生态成熟、开箱即用**。

**主要好处：**

**协议标准化**：RFC 规范清晰，有完整的请求/响应格式、方法、状态码体系，浏览器、Nginx、各种语言的 HTTP 客户端库天然支持，调试工具链完备（curl、Postman、浏览器 DevTools）。

**防火墙友好**：HTTP 跑 80 端口，HTTPS 跑 443，这两个端口几乎在所有网络环境中都默认开放，无需额外配置防火墙策略，穿透能力极强。

**可读性好，易调试**：HTTP 头部是文本格式，人眼可读，用 Wireshark 抓包直接看得懂。

**扩展性强**：通过 Header 机制可以灵活附加认证（Authorization）、内容类型、缓存控制等元数据；通过 Content-Type 支持 JSON、XML、Protobuf 等多种数据格式协商。

**天然支持水平扩展**：HTTP 无状态，任意请求可路由到集群中任意节点，反向代理（Nginx）轻松做负载均衡、限流、SSL 终止。

**安全性**：HTTPS = HTTP + TLS，加密、完整性、身份认证，已是互联网默认标准。

**RESTful 天然匹配**：GET/POST/PUT/DELETE 对应 CRUD，语义清晰。

**当然 HTTP 也有局限**：

- 头部开销较大（HTTP/2 的 HPACK 头部压缩改善了这一点）
- HTTP/1.1 有队头阻塞问题
- 实时推送不是 HTTP 的强项（WebSocket/SSE 是补充方案）
- 相比定制二进制协议，序列化效率略低

**结论**：面向公网的 API、与浏览器交互的场景，HTTP(S) 是无可争议的首选。微服务内部如果对性能要求不极端苛刻，用 HTTP 也完全够用，而且可以复用所有 HTTP 基础设施（网关、链路追踪、日志）。性能敏感的内部通信，才需要考虑 gRPC 或自定义二进制协议。

###### 2. HTTP 协议和 RPC 协议的区别是什么？

这两个常被放一起比，但其实不是同一层次的东西。

**本质区别：**

HTTP 是一个**应用层网络协议**，定义了客户端和服务器之间如何交换数据，核心是"资源的请求与响应"。

RPC（Remote Procedure Call）是一种**通信范式/编程模型**，目标是让调用远程服务像调用本地方法一样透明，核心是"方法的远程调用"。

**几个关键维度对比：**

**设计目标**：HTTP 聚焦资源的表述性状态转移（REST），RPC 聚焦分布式服务间的接口调用。

**通信模型**：HTTP 是请求-响应模式；RPC 框架通常通过客户端 Stub/代理实现，代理负责序列化参数、网络传输、反序列化结果，对调用方屏蔽网络细节。

**协议栈**：HTTP 是完整的应用层协议（报文格式、方法、状态码都是 HTTP 规定的）；而 RPC 不是一个具体协议，它需要选择底层传输协议（TCP、HTTP/1.1、HTTP/2）和消息编码协议（JSON、Protobuf、Thrift）。

**消息格式**：HTTP 有固定的报文结构（起始行、头部、正文），格式固定；RPC 的消息格式由框架决定，高性能框架通常用紧凑的二进制编码，包含服务名、方法名、序列化参数。

**性能**：HTTP/1.1 有队头阻塞、头部冗余问题；专为高性能设计的 RPC 框架（Dubbo、Thrift）用二进制编码 + 长连接，延迟和吞吐量更优。

**关键联系：两者并不对立**

很多 RPC 框架直接用 HTTP 作为传输层：
- gRPC = HTTP/2 + Protobuf，是典型的 "RPC over HTTP"
- Spring Cloud 的 Feign = RESTful 风格的 RPC 实现

而 Dubbo 默认用自定义二进制协议直接跑在 TCP 上，绕开 HTTP 协议栈，追求极致性能。

**选型原则**：对外 API、浏览器/移动端交互 → HTTP(S) + REST；微服务内部高性能通信 → 二进制 RPC（gRPC、Dubbo）。

###### 3. Netty 如何支持 HTTP 协议？

Netty 通过一组内置 Handler 提供完整的 HTTP/1.x 支持，Pipeline 里几个组件配合就能搞定。

**核心组件：**

**`HttpServerCodec`**：HTTP 服务器编解码器，是 `HttpRequestDecoder` + `HttpResponseEncoder` 的组合体。
- 解码方向：把字节流解析成 `HttpRequest`、`HttpContent`、`LastHttpContent` 对象
- 编码方向：把 `HttpResponse` 对象序列化成字节流发出去

**`HttpObjectAggregator`**：HTTP 消息聚合器。HTTP 报文体可能分多个 `HttpContent` 到达，聚合器把它们合并成一个完整的 `FullHttpRequest`（构造参数是最大聚合字节数，防止 OOM）。有了它，业务 Handler 只需处理一个完整对象，大大简化逻辑。

**`HttpContentCompressor`**：支持 gzip 压缩响应，与客户端 `Accept-Encoding` 协商。

**Pipeline 配置示例：**

```java
public class HttpServerInitializer extends ChannelInitializer<SocketChannel> {
    @Override
    protected void initChannel(SocketChannel ch) {
        ChannelPipeline p = ch.pipeline();
        p.addLast(new HttpServerCodec());
        p.addLast(new HttpObjectAggregator(65536)); // 最大聚合 64KB
        p.addLast(new HttpContentCompressor());     // 可选，自动 gzip
        p.addLast(new MyHttpRequestHandler());
    }
}
```

**业务 Handler 示例：**

```java
public class MyHttpRequestHandler extends SimpleChannelInboundHandler<FullHttpRequest> {
    @Override
    protected void channelRead0(ChannelHandlerContext ctx, FullHttpRequest request) {
        String uri = request.uri();
        HttpMethod method = request.method();
        
        FullHttpResponse response = new DefaultFullHttpResponse(
            HttpVersion.HTTP_1_1,
            HttpResponseStatus.OK,
            Unpooled.copiedBuffer("Hello, Netty!", CharsetUtil.UTF_8));
        response.headers().set(HttpHeaderNames.CONTENT_TYPE, "text/plain; charset=UTF-8");
        response.headers().set(HttpHeaderNames.CONTENT_LENGTH, response.content().readableBytes());
        
        ctx.writeAndFlush(response);
    }
}
```

**HTTP/2 支持**：Netty 也提供完整的 HTTP/2 支持（`Http2FrameCodec`、`Http2MultiplexHandler`），API 更面向"帧"（Frame）设计，适合需要多路复用的场景。

###### 4. Netty 如何支持 WebSocket 协议？

WebSocket 是全双工通信协议，建立在 HTTP 之上，通过一次 HTTP Upgrade 握手完成协议升级。Netty 用 `WebSocketServerProtocolHandler` 把这一切自动搞定。

**升级流程：**

1. 客户端发送 HTTP GET 请求，带上 `Upgrade: websocket`、`Connection: Upgrade`、`Sec-WebSocket-Key` 等头部
2. 服务端验证后返回 `101 Switching Protocols`，附上 `Sec-WebSocket-Accept`
3. 之后双方直接用 WebSocket 帧通信

**关键 Handler：**

**`WebSocketServerProtocolHandler`**：核心，负责：
- 识别 WebSocket 握手请求，自动完成握手响应（计算 `Sec-WebSocket-Accept`）
- 握手成功后自动将 Pipeline 中的 HTTP 解码器替换为 WebSocket 帧解码器（`WebSocketFrameDecoder`/`WebSocketFrameEncoder`）
- 自动处理 Ping/Pong 心跳帧

**WebSocket 帧类型**：
- `TextWebSocketFrame`：文本帧
- `BinaryWebSocketFrame`：二进制帧
- `PingWebSocketFrame` / `PongWebSocketFrame`：心跳帧
- `CloseWebSocketFrame`：关闭帧

**Pipeline 配置示例：**

```java
public class WebSocketServerInitializer extends ChannelInitializer<SocketChannel> {
    @Override
    protected void initChannel(SocketChannel ch) {
        ChannelPipeline pipeline = ch.pipeline();
        pipeline.addLast(new HttpServerCodec());
        pipeline.addLast(new HttpObjectAggregator(65536));
        pipeline.addLast(new WebSocketServerProtocolHandler("/ws")); // 指定 WS 路径
        pipeline.addLast(new MyWebSocketFrameHandler());
    }
}
```

**业务 Handler 示例：**

```java
public class MyWebSocketFrameHandler extends SimpleChannelInboundHandler<WebSocketFrame> {
    @Override
    protected void channelRead0(ChannelHandlerContext ctx, WebSocketFrame frame) {
        if (frame instanceof TextWebSocketFrame) {
            String text = ((TextWebSocketFrame) frame).text();
            ctx.channel().writeAndFlush(new TextWebSocketFrame("Echo: " + text));
        } else if (frame instanceof CloseWebSocketFrame) {
            ctx.channel().close();
        }
    }
    
    @Override
    public void userEventTriggered(ChannelHandlerContext ctx, Object evt) throws Exception {
        if (evt == WebSocketServerProtocolHandler.ServerHandshakeStateEvent.HANDSHAKE_COMPLETE) {
            System.out.println("WebSocket 握手成功！");
        } else {
            super.userEventTriggered(ctx, evt);
        }
    }
}
```

握手成功事件通过 `userEventTriggered` 通知，你可以在这里做连接注册、鉴权等操作。

###### 5. Netty 如何实现自定义协议？

Netty 自定义协议的核心是两件事：**定义协议格式**（解决粘包/拆包）+ **实现编解码器**（ByteBuf ↔ 业务对象）。

**第一步：定义协议格式**

强烈推荐用**长度字段**界定消息边界，例如：

```
[魔数(4B)][版本(1B)][消息类型(1B)][序列号(4B)][数据长度(4B)][数据体]
```

**第二步：实现解码器**（继承 `ByteToMessageDecoder`）

```java
public class CustomDecoder extends ByteToMessageDecoder {
    private static final int BASE_LENGTH = 4 + 1 + 1 + 4; // 固定头部长度

    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        if (in.readableBytes() < BASE_LENGTH) return;
        
        in.markReaderIndex(); // 标记，数据不完整时可以重置
        
        int magic = in.readInt();
        byte version = in.readByte();
        byte type = in.readByte();
        int dataLength = in.readInt();

        if (in.readableBytes() < dataLength) {
            in.resetReaderIndex(); // 数据不够，等下次
            return;
        }

        ByteBuf data = ctx.alloc().buffer(dataLength);
        in.readBytes(data, dataLength);

        CustomMessage msg = new CustomMessage();
        msg.setVersion(version);
        msg.setType(type);
        msg.setData(data);
        out.add(msg);
    }
}
```

**第三步：实现编码器**（继承 `MessageToByteEncoder`）

```java
public class CustomEncoder extends MessageToByteEncoder<CustomMessage> {
    @Override
    protected void encode(ChannelHandlerContext ctx, CustomMessage msg, ByteBuf out) {
        out.writeInt(0xCAFEBABE);  // 魔数
        out.writeByte(msg.getVersion());
        out.writeByte(msg.getType());
        ByteBuf data = msg.getData();
        out.writeInt(data.readableBytes());
        out.writeBytes(data);
    }
}
```

**第四步：Pipeline 组合**

```java
pipeline.addLast(new LengthFieldBasedFrameDecoder(1024 * 1024, 8, 4, 0, 0));
pipeline.addLast(new CustomDecoder());
pipeline.addLast(new CustomEncoder());
pipeline.addLast(new CustomBusinessHandler());
```

**几个关键点：**

- **优先用 `LengthFieldBasedFrameDecoder` 做预处理**：它负责按长度字段切帧，你的 Decoder 收到的就是一个完整包，不用再处理粘包逻辑
- **魔数作用**：快速识别非法数据，直接关闭连接
- **版本号作用**：协议升级时保持向后兼容
- **内存管理**：Decoder 中手动创建的 ByteBuf 加入 `out` 列表后由 Netty 负责释放

###### 6. Netty 如何支持 SSL/TLS？

Netty 通过 `SslHandler` 实现 SSL/TLS，它必须放在 Pipeline **最前面**（addFirst）——先解密数据，后续 Handler 才能看到明文。

**使用步骤：**

**1. 构建 SslContext**

```java
// 服务端
SslContext sslCtx = SslContextBuilder
    .forServer(certChainFile, keyFile)
    .sslProvider(SslProvider.OPENSSL)  // 推荐用 OpenSSL，性能更好
    .build();

// 客户端
SslContext sslCtx = SslContextBuilder.forClient()
    .trustManager(InsecureTrustManagerFactory.INSTANCE) // 仅测试！生产用真实 CA
    .build();
```

**2. 添加 SslHandler 到 Pipeline**

```java
@Override
protected void initChannel(SocketChannel ch) {
    ChannelPipeline pipeline = ch.pipeline();
    SSLEngine engine = sslCtx.newEngine(ch.alloc());
    engine.setUseClientMode(false);
    // engine.setNeedClientAuth(true); // 双向认证
    SslHandler sslHandler = new SslHandler(engine);
    sslHandler.setHandshakeTimeout(30, TimeUnit.SECONDS);
    pipeline.addFirst("ssl", sslHandler); // 必须放第一个！
    
    pipeline.addLast(new HttpServerCodec());
    pipeline.addLast(new MyHandler());
}
```

**3. 监听握手完成**

```java
sslHandler.handshakeFuture().addListener(future -> {
    if (future.isSuccess()) {
        System.out.println("SSL 握手成功");
    } else {
        System.err.println("握手失败：" + future.cause());
        future.channel().close();
    }
});
```

**SslHandler 内部工作原理：**

- **入站方向**：调用 `SSLEngine.unwrap()` 解密 → 把明文传给下一个 Handler
- **出站方向**：调用 `SSLEngine.wrap()` 加密 → 把密文发到网络
- **握手**：在 `channelActive` 或第一次 `decode` 时自动触发，处理 NEED_TASK、FINISHED 等状态机

**注意事项：**

- 加解密是 CPU 密集型操作，高并发场景建议用 OpenSSL 引擎（`SslProvider.OPENSSL`）
- 关闭 Channel 时应调用 `ctx.close()`，让 SslHandler 发送 `close_notify` 警报，避免截断攻击
- 可通过 `SslHandler.setWrapDataSize()` 调整每次加密的数据块大小，平衡延迟和吞吐

###### 7. 高频追问：WebSocket 握手升级后，Pipeline 发生了什么变化？

这是 `WebSocketServerProtocolHandler` 最巧妙的地方：**动态替换 Handler**。

握手完成后，WebSocket 的通信单位不再是 HTTP 报文，而是 WebSocket 帧。原来用于解析 HTTP 的 `HttpRequestDecoder` 就没用了，需要换成 `WebSocket13FrameDecoder`。

整个过程：

1. 握手前，Pipeline 是 `[HttpServerCodec] → [HttpObjectAggregator] → [WebSocketServerProtocolHandler] → [业务Handler]`
2. `WebSocketServerProtocolHandler` 检测到 `Upgrade: websocket` 请求
3. 自动发送 `101 Switching Protocols` 响应
4. **动态替换**：移除 `HttpObjectAggregator`，将 `HttpRequestDecoder` 替换为 `WebSocket13FrameDecoder`，将 `HttpResponseEncoder` 替换为 `WebSocket13FrameEncoder`
5. 触发 `HANDSHAKE_COMPLETE` 用户事件通知业务 Handler

握手后 Pipeline：`[HttpServerCodec(WS帧编解码)] → [WebSocketServerProtocolHandler] → [业务Handler]`

这个设计的精妙之处在于：Pipeline 是**可动态修改**的，Handler 可以在运行时添加、移除、替换，非常灵活。
