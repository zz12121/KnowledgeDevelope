### 六、序列化与编解码

> 关联知识库：[[34_DubboKnowledge/03_通信与序列化]] — Dubbo 序列化方案对比（Hessian2/Kryo/Protobuf/FST）及安全配置

###### 1. 说说序列化

序列化是把**对象状态转换为可存储或传输格式（通常是字节流）**的过程；反序列化是逆过程，把字节流重新构造为对象。

**核心目的**：
1. **网络传输**：分布式系统中跨进程、跨网络传递对象
2. **持久化存储**：将对象状态保存到文件或数据库
3. **跨语言数据交换**：不同语言编写的系统间交换数据

**评估一个序列化方案的维度**：
- **空间效率**：编码后的字节数，直接影响网络带宽和存储成本
- **时间效率**：序列化/反序列化速度
- **兼容性**：协议版本升级后新旧数据能否互认
- **安全性**：反序列化过程是否存在 RCE 风险
- **跨语言支持**：能否被多种编程语言使用
- **易用性**：是否需要预编译或代码生成

---

###### 2. 说说影响序列化性能的关键因素

1. **数据大小（空间效率）**：编码后字节数直接影响网络带宽。二进制协议比文本协议（JSON/XML）更紧凑，因为省略了字段名等元数据，使用更高效的数值编码（如 Protobuf 的 Varint）。

2. **序列化/反序列化速度（时间效率）**：
   - **反射开销**：Java 原生序列化、Jackson 等大量使用反射，性能较差
   - **内存分配与拷贝**：序列化中创建临时对象、字节数组拷贝的次数
   - **计算复杂度**：字符串转义、Base64、压缩算法会增加 CPU 负担

3. **序列化方式**：
   - 运行时反射：灵活但慢（Java 原生序列化）
   - 预编译/代码生成：牺牲灵活性换性能（Protobuf、Thrift 编译时生成序列化代码，直接操作字节数组）
   - 字节码生成：如 Kryo 某些模式使用 ASM 生成字节码，首次慢，后续快

4. **数据模型复杂度**：循环引用、深层嵌套、多态等复杂特性需要额外处理逻辑，影响性能。

**高级优化手段**：零拷贝序列化（如 Cap'n Proto，直接在字节缓冲区上读字段）、增量编码（只序列化变化部分）、共享引用（重复对象只序列化一次）。

---

###### 3. 说说 Java 序列化的缺点是什么？

Java 原生序列化（实现 Serializable 接口）在生产级分布式系统中存在明显缺陷：

1. **性能低下**：基于反射，序列化/反序列化速度慢；生成的二进制流体积大，包含大量类描述、字段签名等元数据

2. **安全性问题（最严重）**：反序列化会调用对象的 readObject() 方法，攻击者构造恶意字节流可触发任意代码执行（RCE）。Apache Commons Collections 反序列化漏洞就是典型案例。可用 ObjectInputFilter 设置白名单缓解，但默认未启用

3. **跨语言能力差**：严重依赖 Java 特有的类信息和 JVM 环境，无法与 C++、Python、Go 等其他语言交互

4. **版本兼容性管理困难**：依赖 serialVersionUID。类结构变化（增删字段、修改类型）会导致反序列化失败（InvalidClassException）；即使 UID 一致，字段增删改也可能造成数据错乱

5. **内存消耗与 GC 压力**：反序列化时创建大量临时对象，增加 GC 压力

6. **协议不透明**：Java 专有二进制格式，难以人工阅读和调试

**结论**：高性能场景（RPC、MQ、缓存）中，Java 原生序列化基本被 Protobuf、Kryo 等取代。

---

###### 4. 除了 Java 序列化，还有哪些序列化？

**文本格式**：
- **JSON**：最流行，可读性好，跨语言支持极佳；体积较大，无 schema，数字和日期有歧义。代表库：Jackson、Gson
- **XML**：标签繁重，解析性能差，主要在遗留系统和配置场景
- **YAML/TOML**：主要用于配置文件

**二进制 Schema-based（需预定义结构）**：
- **Protobuf**：Google 出品，性能极高，二进制流极小，跨语言，强类型，需预定义 .proto 文件。**高性能 RPC 的事实标准**，gRPC 默认采用
- **Apache Thrift**：Facebook 开源，类似 Protobuf，支持更丰富的传输方式，需预定义 .thrift 文件
- **Apache Avro**：Hadoop 生态，Schema 以 JSON 格式存储，支持动态 Schema，适合大数据场景

**二进制无 Schema（运行时反射/无预定义）**：
- **MessagePack**：类 JSON 的二进制格式，跨语言，比 JSON 更紧凑
- **BSON**：MongoDB 使用，JSON 的二进制扩展
- **CBOR**：IETF 标准，类 JSON 的二进制格式

**针对 Java 优化的二进制序列化**：
- **Kryo**：速度非常快，序列化体积小；跨语言支持弱，版本间格式可能不兼容，使用前需要**预注册类**（或关闭注册，但会影响性能和体积）
- **FST**：高性能，兼容 Serializable 接口，主要面向 Java
- **Hessian**：跨语言二进制协议，Dubbo 早期默认序列化方式

**选型建议**：
- 高性能 RPC：优先 **Protobuf** 或 **Thrift**
- 大数据/动态 Schema：**Avro**
- Java 内部进程间（非跨语言）：**Kryo** 或 **FST**，注意版本兼容性
- 通用 API/对外接口：**JSON**

###### 5. Netty 支持哪些编解码器？

**内置通用编解码器**：
- `LengthFieldBasedFrameDecoder` + `LengthFieldPrepender`：处理粘包/拆包的黄金组合
- `StringEncoder` / `StringDecoder`：字符串与字节的转换
- `ObjectEncoder` / `ObjectDecoder`：基于 Java 原生序列化（**不推荐生产使用**）
- `LineEncoder` / `LineBasedFrameDecoder`：基于行分隔符

**第三方序列化框架集成（netty-codec 子模块）**：
- **Protobuf**：`ProtobufEncoder`、`ProtobufDecoder`、`ProtobufVarint32FrameDecoder`（处理粘包）、`ProtobufVarint32LengthFieldPrepender`（添加长度头）
- **JBoss Marshalling**：`CompatibleMarshallingEncoder/Decoder`，可替代 Java 原生序列化

**自定义集成**：Jackson、Kryo 等通过继承 `MessageToMessageEncoder/Decoder` 轻松集成。

所有编解码器都实现了 ChannelHandler 接口，可直接添加到 Pipeline 中。

###### 6. 如何自定义 Netty 的编解码器？

**自定义解码器（继承 ByteToMessageDecoder）**：
```java
// 协议格式：[消息类型(1B)][数据长度(4B)][数据]
public class CustomDecoder extends ByteToMessageDecoder {
    private static final int HEADER_SIZE = 5; // 1 + 4
    
    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        if (in.readableBytes() < HEADER_SIZE) return;
        
        in.markReaderIndex();                    // 标记，方便回滚
        byte type = in.readByte();
        int length = in.readInt();
        
        if (in.readableBytes() < length) {
            in.resetReaderIndex();               // 数据不够，回滚等待
            return;
        }
        
        ByteBuf data = ctx.alloc().buffer(length);
        in.readBytes(data, length);
        out.add(new MyMessage(type, data));      // 构造消息对象传给下一个 Handler
    }
}
```

**自定义编码器（继承 MessageToByteEncoder）**：
```java
public class CustomEncoder extends MessageToByteEncoder<MyMessage> {
    @Override
    protected void encode(ChannelHandlerContext ctx, MyMessage msg, ByteBuf out) {
        out.writeByte(msg.getType());
        ByteBuf data = msg.getData();
        out.writeInt(data.readableBytes());      // 写长度
        out.writeBytes(data);                    // 写数据
    }
}
```

**注意事项**：
- **粘包处理**：解码器必须用 markReaderIndex/resetReaderIndex 处理数据不完整的情况
- **资源管理**：decode 中 alloc().buffer() 创建的 ByteBuf 加入 out 后，由后续 Handler 负责释放；encode 方法传入的 out 由 Netty 管理，不需要手动释放
- **最佳实践**：通常将 LengthFieldBasedFrameDecoder 放在自定义解码器前面先处理粘包，这样自定义解码器每次收到的 ByteBuf 就是完整的应用层数据包，代码更简洁

###### 7. Protobuf 在 Netty 中如何使用？

Protobuf 是 Netty 中最常用的高效序列化方案，Netty 提供了专门的编解码器支持。

**第一步：定义 proto 文件**
```protobuf
syntax = "proto3";
option java_outer_classname = "UserProto";
message User {
  int32 id = 1;
  string name = 2;
  string email = 3;
}
```

**第二步：Pipeline 中添加编解码器**
```java
public class ServerInitializer extends ChannelInitializer<SocketChannel> {
    @Override
    protected void initChannel(SocketChannel ch) {
        ChannelPipeline pipeline = ch.pipeline();
        
        // 处理粘包：在 Protobuf 字节流前加/解 Varint32 长度头
        pipeline.addLast(new ProtobufVarint32FrameDecoder());        // 解码：读长度头，拆帧
        pipeline.addLast(new ProtobufVarint32LengthFieldPrepender()); // 编码：加长度头
        
        // Protobuf 编解码
        pipeline.addLast(new ProtobufDecoder(UserProto.User.getDefaultInstance())); // 字节→对象
        pipeline.addLast(new ProtobufEncoder());                     // 对象→字节
        
        pipeline.addLast(new ServerHandler());
    }
}
```

**第三步：业务 Handler 直接处理 Protobuf 对象**
```java
public class ServerHandler extends SimpleChannelInboundHandler<UserProto.User> {
    @Override
    protected void channelRead0(ChannelHandlerContext ctx, UserProto.User user) {
        System.out.println("Received: " + user.getId() + ", " + user.getName());
        
        UserProto.User response = UserProto.User.newBuilder()
            .setId(200).setName("Response").build();
        ctx.writeAndFlush(response);
    }
}
```

**关键说明**：
- ProtobufVarint32FrameDecoder/Prepender 专门解决 Protobuf 消息的粘包，原理是用 Varint32 编码的长度头（比固定 4 字节更紧凑）
- ProtobufDecoder 需要传入默认实例（用于确定解析的目标类型）
- Protobuf 编码后的二进制流非常紧凑，序列化速度极快（代码生成，无反射），是 gRPC 的默认选择

###### 8. 【高频追问】Kryo 在 Netty 中使用时有哪些注意事项？

Kryo 比 Java 原生序列化快 10 倍以上，体积更小，但在 Netty 中使用有几个坑：

**1. Kryo 实例不是线程安全的**

Kryo 对象不能跨线程共享。在 Netty 中，多个 Channel 的 Handler 可能在不同 EventLoop 线程执行（如果 Handler 是 @Sharable 的），必须用 ThreadLocal 或对象池来隔离 Kryo 实例：
```java
private static final ThreadLocal<Kryo> kryoLocal = ThreadLocal.withInitial(() -> {
    Kryo kryo = new Kryo();
    kryo.register(MyMessage.class);
    return kryo;
});
```

**2. 需要预注册类**（推荐）

不注册直接使用虽然可以，但序列化时会把类名写入字节流（体积大），且有安全风险。预注册后 Kryo 使用数字 ID 代替类名：
```java
kryo.register(MyMessage.class, 100);   // 注册时指定固定 ID
```
**重要**：所有通信节点的注册 ID 必须完全一致，新增类只能追加不能插入。

**3. 版本兼容性**

Kryo 不同版本序列化格式可能不兼容，不适合跨版本持久化存储，主要用于进程间通信。

**4. 循环引用**

Kryo 默认启用循环引用检测（会有少量开销），可以通过 `kryo.setReferences(false)` 关闭（确保对象图无循环才能关闭）。
