> 负载均衡算法的通用原理（轮询/最少连接/一致性哈希等）详见 [[33_DistributedKnowledge/04_分布式服务治理]]，本文聚焦 Dubbo 的具体实现。

###### 1. Dubbo 的负载均衡策略有哪些？
Dubbo 所有负载均衡策略通过 `LoadBalance` SPI 接口实现，内置四种：

**1. Random LoadBalance（随机，默认）**：
按权重设置随机概率，权重越大被选中概率越高。算法：计算总权重，产生 `[0, totalWeight)` 随机数，遍历 Provider 列表依次减去权重，随机数 < 0 时选中当前 Provider。调用量大时分布很均匀。源码：`RandomLoadBalance.doSelect()`。

**2. RoundRobin LoadBalance（轮询）**：
按权重设置轮询比率，加权轮询实现。内部维护 `ConcurrentMap<String, AtomicPositiveInteger>` 记录每个方法的调用序列。缺点：慢 Provider 容易积压请求，因为请求会被固定分配到它，直到权重轮到为止。源码：`RoundRobinLoadBalance`。

**3. LeastActive LoadBalance（最少活跃调用数）**：
优先调用当前正在处理请求数（活跃数）最少的 Provider，让慢 Provider 收到更少请求。活跃数 = 调用中的请求数（调用前 +1，完成后 -1）。活跃数相同时按权重随机。能很好反映 Provider 的实时处理能力。源码：`LeastActiveLoadBalance.doSelect()`。

**4. ConsistentHash LoadBalance（一致性哈希）**：
相同参数的请求总发到同一个 Provider，默认对第一个参数哈希。Provider 挂时，通过虚拟节点（默认 160 个）将其流量平摊到其他 Provider，变动影响最小。适用于有状态场景（如会话、缓存亲和）。源码：`ConsistentHashLoadBalance` 内维护 `ConsistentHashSelector` 哈希环。

**5. ShortestResponse LoadBalance（最短响应时间，Dubbo 2.7+）**：
优先选择最近一段时间内**平均响应时间最短**的 Provider，比 LeastActive 更能反映 Provider 的实时性能（LeastActive 只看请求数，不看耗时）。相同响应时间时按权重随机。源码：`ShortestResponseLoadBalance`。

###### 2. Dubbo 默认的负载均衡策略是什么？
**默认是 Random LoadBalance（随机）**。

选择随机作为默认的原因：
1. **简单高效**：时间复杂度 O(n)，无状态，无需维护历史信息，性能开销最小。
2. **调用量大时分布均匀**：大量请求下随机算法收敛于按权重比例分配，效果接近轮询。
3. **不容易产生热点**：相比一致性哈希（特定参数固定打某节点），随机不会因请求模式固定而产生单节点压力。
4. **权重支持良好**：权重高的节点被选中概率成比例增加，流量分配可精细控制。

SPI 扩展文件中默认扩展名 `random` 指向 `RandomLoadBalance`，未显式配置 `loadbalance` 时 `ExtensionLoader` 加载默认实现。

###### 3. 如何自定义负载均衡策略？
完全基于 SPI 机制，三步搞定：

**步骤 1：实现 LoadBalance 接口**（也可以继承 `AbstractLoadBalance` 只覆写 `doSelect`）：
```java
public class IPHashLoadBalance extends AbstractLoadBalance {
    @Override
    protected <T> Invoker<T> doSelect(List<Invoker<T>> invokers, URL url, Invocation invocation) {
        // 获取客户端 IP，Hash 后取模选 Invoker
        String clientIp = RpcContext.getContext().getRemoteAddressString();
        int index = Math.abs(clientIp.hashCode()) % invokers.size();
        return invokers.get(index);
    }
}
```

**步骤 2：注册 SPI 扩展**：
在 `resources/META-INF/dubbo/org.apache.dubbo.rpc.cluster.LoadBalance` 文件中添加：
```
iphash=com.example.IPHashLoadBalance
```

**步骤 3：启用自定义策略**：
```java
// 注解方式
@DubboReference(loadbalance = "iphash")
private DemoService demoService;
```
```yaml
# 配置文件全局生效
dubbo:
  consumer:
    loadbalance: iphash
```

源码角度：调用时 `AbstractClusterInvoker.invoke()` → `doSelect()` → `ExtensionLoader.getExtension("iphash")` 加载自定义实现，无缝融入调用链。

###### 4. 集群容错方式有哪些？
集群容错在消费者端生效，当调用某 Provider 失败时的应对策略，通过 `Cluster` SPI 实现：

| 策略 | 配置值 | 特点 | 适用场景 |
|---|---|---|---|
| **Failover（失败自动切换，默认）** | `failover` | 失败后自动切换其他节点重试，可配置重试次数 `retries` | 读操作、幂等操作 |
| **Failfast（快速失败）** | `failfast` | 失败后立即报错，不重试 | 非幂等写操作（下单、扣款） |
| **Failsafe（失败安全）** | `failsafe` | 失败后直接忽略，仅记录日志 | 非核心辅助操作（日志记录、统计） |
| **Failback（失败自动恢复）** | `failback` | 失败后后台记录，定时重发 | 消息通知、最终一致性场景 |
| **Forking（并行调用）** | `forking` | 同时调用多个 Provider，一个成功即返回，浪费资源 | 实时性要求极高的读操作 |
| **Broadcast（广播）** | `broadcast` | 逐个调用所有 Provider，任意一个报错则报错 | 通知所有 Provider 更新缓存/配置 |

容错入口：`Cluster.join(Directory)` 接收 Invoker 列表，返回包装了容错逻辑的 `ClusterInvoker`。

###### 5. 如何实现 Dubbo 的集群部署？
核心是**启动多个 Provider 实例向同一注册中心注册**，Consumer 通过负载均衡分配调用：

1. **服务无状态化**：会话、临时状态外置到 Redis/数据库，任何实例都能处理任意请求，这是集群化前提。

2. **多实例启动**：多台机器（或 K8s 多 Pod）部署相同服务，指向同一注册中心：
   ```yaml
   dubbo:
     application:
       name: demo-provider
     registry:
       address: nacos://127.0.0.1:8848
     protocol:
       name: dubbo
       port: 20880
   ```

3. **Consumer 侧**：只需配置服务接口和注册中心地址，启动后自动拉取所有 Provider 地址列表：
   ```yaml
   dubbo:
     application:
       name: demo-consumer
     registry:
       address: nacos://127.0.0.1:8848
   ```
   ```java
   @DubboReference(loadbalance = "random", cluster = "failover", retries = 2)
   private DemoService demoService;
   ```

4. **读写容错差异化**：读操作用 `failover` + `retries=2`，写操作用 `failfast` + `retries=0`，方法级别精细控制：
   ```java
   @DubboReference(methods = {
       @Method(name = "query", retries = 2),
       @Method(name = "submit", retries = 0, cluster = "failfast")
   })
   private OrderService orderService;
   ```

###### 6. Dubbo 的集群容错默认是哪种策略？
**默认是 Failover Cluster（失败自动切换）**。

为什么选 Failover 作为默认：
1. **符合普遍需求**：读操作（大多数场景）是幂等的，偶发调用失败（网络抖动/节点瞬时 GC）通过重试其他节点往往能成功，提升成功率。
2. **提高可用性**：自动屏蔽故障节点，从消费者视角服务整体可用。
3. **配置可控**：`retries` 参数（默认2次）限制重试次数，避免无限重试引发雪崩。
4. **平衡性最好**：相比 Failfast（太脆弱）、Forking（太耗资源）、Failback（逻辑复杂），Failover 是最通用的方案。

源码：`RegistryProtocol.doRefer()` 创建 Invoker 时调用 `Cluster$Adaptive.join()`，未指定 `cluster` 参数时使用默认 `failover`，生成 `FailoverClusterInvoker`。

###### 7. 服务读写推荐的容错策略是怎样的？
核心原则：**保证数据一致性，避免非幂等操作重复执行**。

**读操作（幂等）**：
- 推荐 **Failover**，`retries` 设1-2次，搭配较短 `timeout` 快速失败并重试。
- 负载均衡可选 `LeastActive` 或 `ShortestResponse`，实时反映 Provider 负载。

**写操作（非幂等）**：
- 核心写（下单/扣款/修改）：推荐 **Failfast** + `retries=0`，失败立即报错，由上层业务决策是否重试，**严防框架层面重试导致重复执行**。
- 非核心写（记录日志/更新统计）：推荐 **Failsafe**，失败忽略，不影响主流程。

**配置示例**（方法级精细控制）：
```java
@DubboReference(methods = {
    @Method(name = "queryOrders", retries = 2),           // 读操作重试
    @Method(name = "createOrder", retries = 0, cluster = "failfast"),  // 写操作快速失败
    @Method(name = "recordLog", cluster = "failsafe")     // 日志失败忽略
})
private OrderService orderService;
```

记忆口诀：**读重试，写失败快报，日志静默失败**。

###### 8. Forking 集群容错与幂等性有什么关系？
**Forking 策略本身要求被调用操作必须是幂等的**，这是一个容易踩坑的地方。

**问题根源**：Forking 同时调用多个 Provider，只取第一个返回的结果，其余调用仍会在 Provider 端执行完成（只是结果被丢弃）。也就是说，**一次 Forking 调用可能导致操作在多个 Provider 上都执行了**。

**典型踩坑场景**：
- 用 Forking 调用"扣减库存"：3个 Provider 同时执行，库存可能被扣3次。
- 用 Forking 调用"插入订单"：可能产生3条重复订单记录。

**正确使用姿势**：
- Forking **只适合读操作或幂等操作**，比如查询数据、发送通知（有去重机制）。
- 需要的是"快"而不是"对"的场景：比如查询多个缓存节点，哪个先返回用哪个。

**与 Failover 的区别**：
- Failover 是串行重试（失败了再试下一个），同一时刻只有一个 Provider 在处理请求。
- Forking 是并行发出（同时打多个），适合实时性极高但不担心重复执行的场景。

口诀：**Forking = 并行查询，不能用于写**。
