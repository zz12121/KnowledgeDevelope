###### 1. Dubbo 常见的问题及解决方案有哪些？

Dubbo 生产环境常见问题主要围绕**服务暴露、服务发现、服务调用、性能瓶颈、资源泄漏**五大方面。

**1. 服务提供者无法暴露或注册失败**
- **现象**：Provider 启动无报错，但注册中心不可见，Consumer 报 `No provider available`
- **排查**：查日志关键字 `Export dubbo service` 和 `Register`。3 条日志都有说明正常；有前两条没第三条说明服务已本地暴露但注册失败，问题在注册中心客户端；3 条都没有说明暴露流程未启动，检查 Spring 容器
- **断点**：`ServiceConfig.export()` → `RegistryProtocol.doRegister()`

**2. 服务调用超时**
- **现象**：频繁出现 `RpcException: Timeout exception`
- **排查**：先看消费者超时配置是否合理；然后 `jstack` 看提供者业务线程池（`DubboServerHandler-` 前缀），定位是数据库慢查询、外部 HTTP 调用、锁竞争还是 Full GC

**3. 线程池耗尽**
- **现象**：提供者端 `RejectedExecutionException`，消费者调用失败
- **排查**：`jstack` 分析线程阻塞点；短期可调大 `threads` 或换 `eager` 线程池；根本是优化慢业务逻辑 + 异步化 IO 操作

**4. 序列化/反序列化异常**
- **现象**：`SerializationException`、`ClassNotFoundException`
- **排查**：确认 Consumer 和 Provider 使用的序列化方式一致；DTO 类全限定名、`serialVersionUID` 完全一致；避免使用 Java 原生序列化

**5. 内存泄漏**
- **现象**：堆内存持续增长，Full GC 后不回收到基线
- **排查**：`jmap -dump:live,format=b,file=heap.hprof <pid>` 后用 MAT 分析，重点查 `Invoker`、`Channel`、`ExchangeClient` 实例数量和 `DubboProtocol` 中的静态 Map 缓存

**6. 注册中心抖动**
- **现象**：服务实例列表频繁刷新，调用时有时无
- **排查**：检查 ZK/Nacos 集群健康状态；适当调大 `dubbo.registry.timeout`，避免 GC 停顿导致会话超时被误剔除

**通用诊断工具**：
- `telnet localhost 22222`，使用 `ls`、`ps`、`count`、`trace` 命令实时诊断
- Dubbo 日志设置 `DEBUG` 级别（包路径 `org.apache.dubbo`）
- Prometheus + Grafana 监控 QPS、RT、错误率、线程池活跃度

###### 2. Dubbo 服务调用超时怎么排查？

超时排查遵循 **"先消费者，后提供者；先外部，后内部"** 的原则。

**步骤一：消费者端初步分析**

1. 确认实际生效的超时配置（方法级 > 接口级 > 全局，消费者 > 提供者）
2. 用 QoS 命令实时追踪：`trace com.xxx.Service method`
3. 排除配置太短的误报：如果超时 100ms 但网络 RTT 就 50ms，先调大 `timeout` 验证

**步骤二：提供者端深度排查（主要方向）**

绝大多数超时源于提供者处理慢：

```bash
jstack -l <provider_pid> > thread_dump.txt
```

搜索 `DubboServerHandler-` 前缀的线程，分析状态：
- `BLOCKED`/`WAITING` → 阻塞在数据库（慢 SQL/连接池满）、外部 HTTP 调用、锁竞争
- `RUNNABLE` 但 CPU 不高 → 可能 IO 等待

**步骤三：网络层分析**

- 双向 `ping` 检查网络延迟
- Dubbo 协议默认单连接，大请求/慢响应会导致队头阻塞；可谨慎增加 `connections`（如 2-3）

**步骤四：源码切入点**

消费者端：超时控制在 `HeaderExchangeChannel.request()` 中，创建 `DefaultFuture` 并提交 `TimeoutTask` 到 `HashedWheelTimer`，触发时调用 `DefaultFuture.timeoutCheck()`

提供者端：`DubboProtocol.requestHandler` 处理请求，若业务线程池已满，请求排队或被拒绝，同样导致消费者超时

**排查金三角**：业务线程栈 + 监控调用链（SkyWalking）+ 网络基础环境

###### 3. Dubbo 服务无法注册到注册中心怎么办？

按顺序排查：**配置 → 日志 → 源码 → 网络 → 容器环境**。

**Step 1：基础配置检查**
- `dubbo.registry.address` 是否正确，网络是否可达（`telnet ZK_HOST 2181`）
- `dubbo.application.name` 是否配置且符合规范
- `@EnableDubbo` 或 XML 配置是否被正确加载

**Step 2：分析启动日志**

Dubbo 在关键节点输出 INFO 级别日志：
```
[DUBBO] Export dubbo service com.xxx.XxxService to local registry...
[DUBBO] Export dubbo service com.xxx.XxxService to url dubbo://...
[DUBBO] Register dubbo service com.xxx.XxxService url dubbo://... to registry...
```
- 有前两条没第三条 → 注册中心客户端问题
- 三条都没有 → Spring 容器/Bean 初始化问题

**Step 3：注册中心客户端问题**
- Curator 5.x 与 ZK Server 3.4.x 可能存在兼容性问题
- ZK 开启了 ACL 需要配置 `dubbo.registry.username`/`password`
- 虚拟机/容器中 GC 停顿过长导致 ZK 会话超时，适当调大 `session.timeout`

**Step 4：容器环境特殊配置（最常见）**

```yaml
dubbo:
  protocol:
    host: ${POD_IP}      # 必须注册真实的 Pod IP
  registry:
    address: zookeeper://${ZOOKEEPER_HOST}:2181
```

**验证命令**：
```bash
# 连接 ZK 确认节点是否存在
ls /dubbo/com.xxx.XxxService/providers
```

###### 4. Dubbo 服务调用失败怎么排查？

核心思路：**错误码定大类 → 隔离法定小类**。

**第一步：解读错误码**

捕获 `RpcException`，打印 `code` 和 `message`：
- `403 (FORBIDDEN)`：Token 校验失败或黑白名单限制
- `502 (TIMEOUT)`：超时
- `501 (NETWORK)`：网络连接错误
- `502 (NO_PROVIDER)`：无可用提供者

**第二步：排查"无可用提供者"**

最常见错误：
1. 确认 Provider 成功注册，Consumer 成功订阅（看日志 `[DUBBO] Subscribe:`）
2. 检查 `version` 和 `group` 是否完全一致（一个字符差异也会失败）
3. 检查是否有路由规则（标签路由/条件路由）把当前消费者过滤掉了
4. QoS 命令 `ls -l` 查看 Consumer 当前缓存了哪些提供者

**第三步：直连法隔离问题**

```java
@DubboReference(url = "dubbo://192.168.1.100:20880")
private DemoService demoService;
```

直连成功说明问题在注册中心/路由；直连也失败说明问题在两点间的网络或服务本身。

**第四步：Mock 法隔离**

配置 `mock="force:return null"` 并验证 Mock 能生效，则证明是远程调用问题，非本地代码问题。

**第五步：调用链追踪**

通过 SkyWalking 查看完整调用链，确认失败发生在哪一跳，精确定位到具体的服务实例和时间点。

**源码辅助**：

- `AbstractClusterInvoker.invoke()`：负载均衡和容错策略执行过程
- `DubboInvoker.doInvoke()`：请求发送和响应接收细节

###### 5. Dubbo 如何排查线程池满的问题？

线程池满是提供者端的典型问题，根本原因是**业务线程无法及时释放**。

**Step 1：确认现象**
- Provider 日志搜索 `RejectedExecutionException`
- 监控显示线程池活跃线程数达到 `threads` 上限，队列任务积压

**Step 2：jstack 分析（最关键）**

```bash
jstack -l <provider_pid> > thread_dump.txt
```

搜索 `DubboServerHandler-{ip}:{port}-` 前缀的线程，分析状态：

| 线程状态 | 常见阻塞点 | 排查方向 |
|---|---|---|
| `WAITING (on object monitor)` | `Object.wait()`、`Future.get()` | 等待锁或异步结果 |
| `BLOCKED (on monitor entry)` | `synchronized` 或 `Lock` 竞争 | 锁持有者的栈帧 |
| `WAITING (parking)` | `LockSupport.park()` | 条件队列或 FutureTask |
| `RUNNABLE` (IO 等待) | `SocketInputStream.socketRead()` | 数据库/外部 HTTP 慢响应 |

**Step 3：对症下药**

- 发现慢 SQL → 优化查询、加索引、检查连接池
- 发现外部 HTTP 调用 → 设置合理超时 + 异步化
- 发现锁竞争 → 减少锁粒度或使用并发数据结构

**Step 4：Dubbo 配置调优（治标）**

```yaml
dubbo:
  protocol:
    threads: 500         # 增大线程数（治标）
    threadpool: eager    # 优先创建线程而非排队（适合突发流量）
```

**Step 5：根本解决（治本）**

- 对 DB/Redis/HTTP 等 IO 操作使用 `CompletableFuture` 异步化，释放 Dubbo 业务线程
- 使用 Sentinel 配置熔断，避免下游故障导致线程被长期占用
- 对特定慢方法配置 `executes` 参数，防止拖垮整个线程池

**源码关联**：线程池在 `DubboProtocol.openServer()` 中创建，通过 `ExecutorRepository` 统一管理；拒绝策略默认 `AbortPolicy`，可自定义。

###### 6. Dubbo 如何排查内存泄漏问题？

Dubbo 内存泄漏通常源于**不当的使用方式或自定义扩展**，排查有标准流程。

**Step 1：确认泄漏模式**

通过 JMX/Prometheus 观察老年代内存：Full GC 后仍不回收到基线水平，呈阶梯状持续上涨。

**Step 2：获取 Heap Dump**

```bash
jmap -dump:live,format=b,file=dubbo_heap.hprof <pid>
# 或配置 JVM 参数自动生成：-XX:+HeapDumpOnOutOfMemoryError
```

用 MAT 或 JProfiler 分析 `.hprof` 文件。

**Step 3：定位 Dubbo 可疑对象**

重点查：
- `Invoker`、`Channel`、`ExchangeClient`、`ServiceBean`、`ReferenceBean` 实例数量是否异常增长
- `DubboProtocol.referenceClientMap`（`ConcurrentHashMap`）的 size 是否异常大

**Step 4：常见泄漏场景**

**场景一：未关闭的 ReferenceConfig**
```java
// 如果通过 API 方式创建，必须手动销毁
ReferenceConfig<DemoService> reference = new ReferenceConfig<>();
// 使用完成后
reference.destroy();  // 否则 ExchangeClient 连接不会释放
```

**场景二：自定义 Filter 中持有全局引用**
```java
// 危险！静态 Map 引用 Invocation 导致泄漏
public class LeakyFilter implements Filter {
    private static Map<String, Invocation> CACHE = new HashMap<>();
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) {
        CACHE.put(invocation.getMethodName(), invocation); // 泄漏！
        return invoker.invoke(invocation);
    }
}
```

**场景三：ThreadLocal 未清理**

在 Filter 中使用 ThreadLocal 存放调用上下文，如果没有在 `finally` 块中 `remove()`，会造成内存泄漏。

**场景四：Netty ByteBuf 未释放**

自定义编解码器中 `ByteBuf` 未按规定 `release()`，导致堆外内存泄漏（进程内存持续增长但堆正常）。
开启检测：`-Dio.netty.leakDetectionLevel=PARANOID`（仅测试环境）

###### 7. Dubbo 连接数过多怎么处理？

**Step 1：诊断现状**

```bash
# 查看连接到某端口的 TCP 连接数
netstat -anp | grep :20880 | wc -l

# Dubbo QoS 查看连接信息
telnet localhost 22222
```

**Step 2：分析根因**

**Dubbo 协议（默认）**：设计上是**单连接复用**，一个 Consumer 进程对一个 Provider IP:Port 默认只建立 1 个连接（由 `connections` 参数控制）。

理论连接数 = Consumer 进程数 × `connections`

如果实际连接数远超这个值，常见原因：
- `connections` 被调得过大（不必要）
- Consumer 频繁重启，旧连接延迟关闭造成短时膨胀
- `ReferenceConfig` 未销毁导致连接未关闭（资源泄漏）

**HTTP/REST 协议**：连接数 = Consumer 实例数 × 连接池大小，连接池配置过大会导致连接数飙升。

**Step 3：解决方案**

| 原因 | 解决方式 |
|---|---|
| `connections` 设置过大 | 恢复默认值 1，只在跨机房高延迟场景下谨慎增加（2-5）|
| Consumer 频繁重启 | 优化应用生命周期，保证长运行 |
| 资源泄漏 | 确保 `ReferenceConfig.destroy()` 被调用，Spring 环境一般没问题 |
| HTTP 连接池过大 | 调优 HttpClient 连接池参数（maxTotal/maxPerRoute）+ 空闲超时 |
| Provider 保护 | 配置 `accepts` 限制最大连接数，超出时直接关闭新连接 |

###### 8. 生产环境快速诊断 Checklist

遇到 Dubbo 生产问题时，按以下 Checklist 快速定位：

**一、服务不可用（No Provider）**
- [ ] ZK/Nacos 集群是否正常？（CPU、内存、磁盘 IO）
- [ ] Provider 日志中是否有 `Register` 成功的关键日志？
- [ ] Consumer 日志中是否有 `Subscribe` 成功的关键日志？
- [ ] version 和 group 是否完全匹配？
- [ ] 是否有路由规则把 Consumer 过滤掉了？
- [ ] 容器环境：Provider 注册的是 Pod IP 还是错误的 IP？

**二、调用超时**
- [ ] `jstack` 提供者端业务线程是否大量阻塞？阻塞在哪里？
- [ ] 数据库：是否有慢 SQL？连接池是否满了？
- [ ] 网络：两端 `ping`/`traceroute` 是否正常？
- [ ] Full GC：是否有频繁/长时间 Full GC 导致的停顿？
- [ ] 消费者超时是否比提供者超时更短？（不合理配置）

**三、线程池满**
- [ ] `jstack` 查业务线程池（`DubboServerHandler-`）是否全部阻塞？
- [ ] 哪个服务方法最多调用？是否有慢方法未设置 `executes` 保护？
- [ ] 是否有下游服务响应变慢导致连锁反应？
- [ ] Sentinel 是否配置了熔断规则？

**四、序列化异常**
- [ ] Consumer 和 Provider 的序列化方式是否一致？
- [ ] DTO 类的全限定名、`serialVersionUID` 是否完全一致？
- [ ] 是否有枚举新增值或方法签名变更未同步？
