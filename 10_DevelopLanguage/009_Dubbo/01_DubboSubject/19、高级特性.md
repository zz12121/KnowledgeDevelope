###### 1. Dubbo 为什么默认用 Javassist？

Dubbo 默认使用 Javassist 而非 JDK 动态代理，基于**性能、灵活性、兼容性**的综合考量。

**性能差异（核心原因）**

- **JDK 动态代理**：基于 `java.lang.reflect.Proxy`，每次方法调用都经过 `InvocationHandler.invoke()` 反射调用，有额外开销
- **Javassist**：在**类加载阶段**直接生成代理类字节码，生成的类中每个方法体都是直接的方法调用逻辑，**运行时零反射**，性能接近原生代码

在微服务高频 RPC 场景下，这种差异会被放大：牺牲一点启动时生成字节码的时间，换取每次调用的性能提升。

**源码证明**：

```java
@SPI("javassist")   // 默认扩展名为 javassist
public interface ProxyFactory {
    @Adaptive({Constants.PROXY_KEY})
    <T> T getProxy(Invoker<T> invoker) throws RpcException;

    @Adaptive({Constants.PROXY_KEY})
    <T> Invoker<T> getInvoker(T proxy, Class<T> type, URL url) throws RpcException;
}
```

`META-INF/dubbo/internal/org.apache.dubbo.rpc.ProxyFactory` 中：
```
javassist=org.apache.dubbo.rpc.proxy.javassist.JavassistProxyFactory
jdk=org.apache.dubbo.rpc.proxy.jdk.JdkProxyFactory
```

**灵活性优势**

Javassist 提供高级 API 操作字节码，允许在生成代理类时插入自定义逻辑（如异步调用、泛化调用的特殊处理），JDK 动态代理只能通过 `InvocationHandler` 这一个入口拦截，灵活性有限。

**生成代理类核心逻辑**

`JavassistProxyFactory.getProxy()` → `Proxy.getProxy(ClassLoader cl, Class<?>... ics)`：

1. 用 Javassist 的 `ClassPool`/`CtClass`/`CtMethod` 动态构建代理类
2. 该类实现用户定义的服务接口
3. 每个方法体生成为：封装 `RpcInvocation` → 调用 `Invoker.invoke(invocation).recreate()`
4. 生成类被加载后实例化，通过 `setInvoker()` 注入 ClusterInvoker

###### 2. Dubbo 的动态代理是如何实现的？

动态代理是 Dubbo 透明 RPC 调用的基石，结合了 **Java 动态代理技术** 和 **Dubbo Invoker 调用模型**，通过 SPI 支持多种实现。

**消费者端代理创建流程**

当 `@DubboReference` 触发 `ReferenceBean.getObject()` 时：

1. **创建 ClusterInvoker**：`ReferenceConfig.init()` 调用 `Protocol.refer()`，生成包含负载均衡/容错逻辑的 `ClusterInvoker`
2. **字节码生成**：`ProxyFactory.getProxy(invoker)` 通过 Javassist 动态生成代理类

生成的代理类方法体（伪代码）：
```java
public Object methodName(Object[] args) {
    // 封装调用信息
    RpcInvocation invocation = new RpcInvocation(method, args);
    invocation.setAttachment("path", interfaceName);
    // 调用 Invoker 链，结果通过 recreate() 还原类型
    return invoker.invoke(invocation).recreate();
}
```

3. **返回代理对象**：生成的类通过 `setInvoker()` 注入 ClusterInvoker 后实例化，注入到 `@DubboReference` 字段

**调用流程**

```
proxy.method(args)
  → 动态生成类中的 methodName() 方法
    → 创建 RpcInvocation
      → ClusterInvoker.invoke()（负载均衡 + 容错）
        → 选择目标 DubboInvoker
          → Netty Channel 发送请求
            → 等待响应，recreate() 还原结果
```

**提供者端 Invoker 创建**

`ServiceConfig.export()` 调用 `ProxyFactory.getInvoker(proxy, type, url)`，生成 `AbstractProxyInvoker` 子类，其 `doInvoke()` 方法通过反射调用本地实现对象的对应方法。

**源码定位**：
- 代理生成入口：`JavassistProxyFactory.getProxy()` → `Proxy.getProxy(...)`
- 调用链：`MockClusterInvoker` → `AbstractClusterInvoker` → `DubboInvoker`

###### 3. Dubbo 的本地存根是什么？

本地存根（Local Stub）是在消费者端对远程服务代理的一层**增强包装**，允许在发起远程调用前后执行本地逻辑，常用于参数验证、客户端缓存、日志记录等非核心切面操作。

**定义 Stub**（必须有接收远程代理的构造函数）：

```java
public class UserServiceStub implements UserService {
    private final UserService userService;  // 真正的远程代理

    public UserServiceStub(UserService userService) {
        this.userService = userService;
    }

    @Override
    public User getUser(Long id) {
        // before：参数校验
        if (id == null || id <= 0) {
            throw new IllegalArgumentException("Invalid user id");
        }
        // 发起远程调用
        User user = userService.getUser(id);
        // after：本地缓存
        Cache.put("user_" + id, user);
        return user;
    }
}
```

配置：
```xml
<dubbo:reference id="userService" interface="com.example.UserService" 
                 stub="com.example.UserServiceStub"/>
```

**源码实现**：`ReferenceConfig.createProxy()` 检查 `stub` 参数后，通过 `StubProxyFactoryWrapper.getProxy()` 先创建原始远程代理，再用反射实例化 Stub 类并注入远程代理，最终返回 Stub 实例。

**与 Mock 的区别**：

| 对比 | 本地存根（Stub） | 本地伪装（Mock）|
|---|---|---|
| 目的 | 增强调用过程（Before/After 逻辑）| 服务降级（失败备用方案）|
| 触发时机 | **每次**调用 | 仅调用**失败**时 |
| 是否发起远程调用 | 总是发起 | `force:` 不发起；`fail:` 仅失败后执行 |

###### 4. Dubbo 的本地伪装是什么？

本地伪装（Local Mock）是 Dubbo 的**服务降级机制**，当远程调用失败（超时/网络异常/无提供者）时，自动返回预定义的本地模拟结果，保障系统韧性。

**触发条件**：仅在抛出 `RpcException`（框架异常）时触发，业务异常不触发 Mock。

**配置方式**：

```java
@DubboReference(mock = "fail:return null")     // 失败时返回 null
@DubboReference(mock = "force:return empty")   // 强制不发起远程调用
@DubboReference(mock = "com.example.UserServiceMock")  // 指定 Mock 类
private UserService userService;
```

Mock 类示例：
```java
public class UserServiceMock implements UserService {
    @Override
    public User getUser(Long id) {
        return new User(-1L, "降级用户");  // 降级数据
    }
}
```

**实现原理**：`MockClusterWrapper` 包装了真正的 `Cluster`（如 `FailoverCluster`）。`MockClusterInvoker.invoke()` 执行流程：

1. 检查是否 `force:` 开头 → 直接执行 Mock，不发起远程调用
2. 否则尝试远程调用
3. 抛出 `RpcException` → 执行 Mock 逻辑（`fail:` 模式）

**动态配置 Mock**：在 Dubbo Admin 控制台可以**在线动态修改** Mock 规则，实现生产环境快速降级，无需重启服务。

###### 5. Dubbo 的延迟暴露是什么？

延迟暴露（Delay Export）是指 Provider 不在 Spring 容器初始化时立即注册服务，而是等待一段时间后再暴露。

**配置方式**：

```yaml
dubbo.provider.delay: -1   # 推荐：等待 Spring 容器初始化完成后暴露
# delay = 0：立即暴露（默认）
# delay = 5000：延迟 5 秒后暴露
```

**为什么需要延迟暴露？**

1. **避免依赖未就绪**：服务 A 依赖本地 Bean B，如果 B 还没初始化完就收到远程调用，会出错。`delay=-1` 确保所有 Bean 初始化完成后才暴露
2. **避免注册风暴**：大规模集群同时重启时，设置不同的随机延迟，错峰注册，减轻注册中心压力
3. **等待预热完成**：JIT 编译、缓存预热等操作完成后再接收流量

**源码实现**：

```java
// ServiceConfig.export()
public synchronized void export() {
    if (shouldDelay()) {
        delayExportExecutor.schedule(this::doExport, getDelay(), TimeUnit.MILLISECONDS);
    } else {
        doExport();
    }
}

// ServiceBean 监听 Spring ContextRefreshedEvent（delay=-1 时）
public void onApplicationEvent(ContextRefreshedEvent event) {
    if (!isExported() && !isUnexported()) {
        export();  // Spring 容器刷新完成后触发暴露
    }
}
```

**推荐实践**：生产环境统一使用 `delay=-1`，保证所有本地依赖 Bean 就绪后再接收流量。

###### 6. Dubbo 的并发控制是如何实现的？

Dubbo 并发控制通过 `executes` 参数限制服务端某方法的**最大并行执行线程数**，防止单个慢方法压垮整个服务。

**配置**：

```xml
<dubbo:service interface="com.example.UserService" executes="100">
    <dubbo:method name="heavyQuery" executes="20"/>  <!-- 方法级优先级更高 -->
</dubbo:service>
```

**实现原理：ExecuteLimitFilter + Semaphore**

`ExecuteLimitFilter` 是服务提供者端的 Filter，使用信号量实现并发计数：

```java
public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
    int max = url.getMethodParameter(methodName, EXECUTES_KEY, 0);
    if (max > 0) {
        String key = invoker.getInterface().getName() + "." + methodName;
        Semaphore semaphore = EXECUTE_LIMIT_MAP.computeIfAbsent(key, k -> new Semaphore(max));
        
        if (!semaphore.tryAcquire()) {  // 非阻塞，立即失败
            throw new RpcException("并发数超过限制: " + max);
        }
        try {
            return invoker.invoke(invocation);
        } finally {
            semaphore.release();  // 确保释放
        }
    }
    return invoker.invoke(invocation);
}
```

**关键特性**：

- **立即失败**：超限请求直接拒绝（不排队），保护提供者
- **动态生效**：通过配置中心可以实时修改 `executes` 值，无需重启
- **与 `threads` 的区别**：`executes` 控制业务逻辑的并发执行数；`threads` 控制 Dubbo 协议层处理网络请求的 IO 线程数，两者是不同层面

###### 7. Dubbo 的连接控制是如何实现的？

连接控制通过 `accepts` 参数限制提供者接受的最大客户端连接数，防止文件描述符耗尽。

**配置**：

```xml
<dubbo:protocol name="dubbo" port="20880" accepts="1000"/>
```

**实现原理**：发生在 **TCP 连接建立阶段**（比请求处理更早）

```java
// DubboProtocol 简化逻辑
public void addServerChannel(Channel channel) {
    AtomicInteger count = connections.computeIfAbsent(key, k -> new AtomicInteger(0));
    int max = getMaxAccepts(channel);
    if (max > 0 && count.incrementAndGet() > max) {
        count.decrementAndGet();
        channel.close();  // 立即关闭新连接
        throw new RpcException("连接数超过限制: " + max);
    }
}
```

**`accepts` vs `connections` 区别**：

| 参数 | 配置端 | 作用 |
|---|---|---|
| `accepts` | 服务端（Provider）| 限制 Provider 能接受的总连接数 |
| `connections` | 客户端（Consumer）| 限制 Consumer 对每个 Provider 建立的连接数（默认 1）|

###### 8. Dubbo 的粘滞连接是什么？

粘滞连接（Sticky Connection）让来自同一消费者的请求尽可能路由到同一个提供者实例，用于有状态或会话保持的场景。

**配置**：

```java
@DubboReference(loadbalance = "sticky")
private UserService userService;
```

**工作原理**：

1. 首次调用：负载均衡正常选择一个提供者，缓存结果到 `RpcContext`（含过期时间）
2. 后续调用：检查缓存是否未过期且目标提供者仍可用 → 直接复用
3. 缓存失效（超时/提供者下线）：重新负载均衡选择，更新缓存

**源码**（`StickyLoadBalance` 装饰器模式）：
```java
protected <T> Invoker<T> doSelect(List<Invoker<T>> invokers, URL url, Invocation invocation) {
    StickyHolder holder = (StickyHolder) RpcContext.getClientAttachment()
                              .getObjectAttachment("sticky." + methodName);
    if (holder != null && holder.time > System.currentTimeMillis()
            && invokers.contains(holder.invoker)) {
        return holder.invoker;  // 返回缓存的 Invoker
    }
    // 缓存失效，重新选择
    Invoker<T> selected = innerLoadBalance.select(invokers, url, invocation);
    RpcContext.getClientAttachment().setObjectAttachment("sticky." + methodName,
            new StickyHolder(selected, System.currentTimeMillis() + stickyTimeout));
    return selected;
}
```

**注意事项**：破坏了负载均衡的均匀性，可能导致流量倾斜。强会话绑定场景应使用 `consistenthash`（一致性哈希）而不是粘滞连接。

###### 9. Dubbo 如何实现多协议支持？

多协议支持是 Dubbo **SPI 微内核架构**的典型体现，核心是 `Protocol` 接口扩展点：

```java
@SPI("dubbo")   // 默认协议
public interface Protocol {
    @Adaptive  // 根据 URL 中的 protocol 参数动态选择实现
    <T> Exporter<T> export(Invoker<T> invoker) throws RpcException;

    @Adaptive
    <T> Invoker<T> refer(Class<T> type, URL url) throws RpcException;
}
```

**配置多协议**：

```xml
<!-- 同一服务暴露在多个协议上 -->
<dubbo:service interface="com.example.AdminService" protocol="dubbo,tri"/>
```

**工作原理**：配置 `protocol="dubbo,tri"` 时，`ServiceConfig` 遍历协议列表，为每个协议调用一次 `Protocol.export()`，同一服务在两个端口（如 20880 和 50051）上同时监听。

**内置协议一览**：

| 协议名 | 特点 | 适用场景 |
|---|---|---|
| `dubbo` | TCP 二进制，单长连接多路复用，高性能 | 内部服务高频调用 |
| `tri`（Triple）| HTTP/2 + Protobuf，兼容 gRPC，支持流式 | 云原生、跨语言互通、网关场景 |
| `rest` | HTTP/JSON/XML，RESTful 风格 | 对外暴露 API |
| `hessian` | 基于 HTTP 的 Hessian 序列化 | 与老系统兼容 |

**典型应用**：
- 内部高性能调用用 `dubbo` 协议
- 对外暴露或与前端通信用 `rest` 或 `tri`
- 协议升级时双协议暴露，逐步迁移消费者

###### 10. Dubbo 如何实现多注册中心？

多注册中心允许服务**同时向多个注册中心注册**，实现多机房容灾和混合云架构。

**配置**：

```xml
<dubbo:registry id="zk" address="zookeeper://127.0.0.1:2181"/>
<dubbo:registry id="nacos" address="nacos://127.0.0.1:8848"/>

<dubbo:service interface="com.example.DemoService" registry="zk,nacos"/>
```

**实现原理**：

提供者端：`ServiceConfig.doExportUrlsFor1Protocol()` 遍历注册中心列表，对每个 `RegistryConfig` 调用 `RegistryProtocol.export()`，分别向 ZK 和 Nacos 创建注册节点。

消费者端：`ReferenceConfig.createProxy()` 为每个注册中心调用 `RegistryProtocol.refer()` 获取 Invoker 列表，再通过 `MergeableCluster` 合并成统一的集群 Invoker。

**注意事项**：
- 同一服务在不同注册中心都有提供者，消费者端需要合并去重
- 跨注册中心的路由规则和权重需要统一考虑
- 监控每个注册中心的健康状态，及时发现单注册中心故障

###### 11. Dubbo 的服务分组是如何实现的？

服务分组（Group）通过 **接口 + 分组 + 版本号**的三元组唯一标识一个服务，实现同一接口的多种实现逻辑隔离。

**配置**：

```java
@DubboService(group = "payment")        // 支付分组
public class PaymentDemoServiceImpl implements DemoService {}

@DubboService(group = "logistics")      // 物流分组
public class LogisticsDemoServiceImpl implements DemoService {}

// 消费者引用指定分组
@DubboReference(group = "payment")
private DemoService paymentService;
```

**实现原理**：`group` 信息编码到注册的 URL 中（如 `?group=payment`），消费者订阅时 `RegistryDirectory.toInvokers()` 对比消费者 URL 和提供者 URL 中的 group 参数，过滤不匹配的提供者。

**分组聚合（Group Merger）**：

```java
// 同时消费多个分组，合并结果
@DubboReference(group = "*", merger = "list")   // 合并所有分组的 List 结果
private DemoService demoService;
```

源码中 `MergeableClusterInvoker` 并发调用各分组，通过 `ListMerger`/`MapMerger` 等合并结果。

**应用场景**：
- **环境隔离**：test/staging/prod 环境分组
- **业务线隔离**：不同业务线使用同一接口的不同实现
- **灰度发布**：结合 `version` 实现多版本并行

###### 12. Mock 降级的动态配置实战

这道题考察的是 Dubbo Mock 在生产环境的实际操控能力，而不只是配置 `mock="true"` 那么简单。

**12.1 动态 Mock 的必要性**

硬编码 `mock="fail:return null"` 的问题：Mock 规则固化，无法在不重启的情况下调整降级策略。生产环境需要能**随时开启/关闭降级，甚至修改降级数据**。

**12.2 通过配置中心动态控制**

Dubbo 支持通过配置中心（Nacos/ZK）动态下发 Override 规则：

```yaml
# 在 Nacos 的 dubbo-override.yaml 中配置
configVersion: v2.7
scope: service
key: org.apache.dubbo.demo.DemoService
enabled: true
configs:
- addresses:
  - 0.0.0.0   # 对所有消费者生效
  side: consumer
  parameters:
    mock: "force:return {\"code\":200,\"msg\":\"降级数据\"}"
```

下发后，`ConsumerConfig` 实时感知变更，`MockClusterInvoker` 在下次调用时生效，**无需重启任何服务**。

**12.3 Dubbo Admin 可视化操作**

在 Dubbo Admin 控制台：
1. 找到目标服务
2. 点击「服务降级」
3. 选择降级策略（`force:` 或 `fail:`）
4. 填写降级返回值（JSON 格式）
5. 点击「执行」

Admin 底层就是通过配置中心下发 Override 规则，与上面 YAML 方式等价。

**12.4 Mock 返回值格式**

| 表达式 | 含义 |
|---|---|
| `return null` | 返回 null |
| `return empty` | 返回空对象（集合返回空列表，对象返回 new 出的默认实例）|
| `return {"id":1,"name":"mock"}` | 返回 JSON 反序列化的对象 |
| `throw` | 抛出 RpcException |
| `throw com.xxx.BizException` | 抛出指定类型的异常 |

**12.5 搭配 Sentinel 使用**

更推荐的方案是 **Sentinel 熔断 + Dubbo Mock 降级**的组合：
- Sentinel 负责判断是否触发降级（基于错误率/慢调用比例）
- Dubbo Mock 负责定义降级时返回什么数据
- 两者解耦，分别由不同团队维护
