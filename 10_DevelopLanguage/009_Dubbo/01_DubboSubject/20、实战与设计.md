###### 1. 如果让你实现一个 RPC 框架，你会怎么设计？

核心思路：**分层架构 + 微内核插件化**，参考 Dubbo 的设计理念。

**分层架构设计**：

| 层次 | 职责 | 关键抽象 |
|---|---|---|
| 服务接口层 | 定义业务接口，供开发者使用 | Java Interface |
| 代理层 | 透明化远程调用，把本地接口调用转为远程调用 | `ProxyFactory` |
| 集群容错层 | 服务目录、路由、负载均衡、集群容错策略 | `Invoker`、`Cluster`、`LoadBalance` |
| 远程调用层 | 协议编解码，统一请求/响应模型 | `Protocol`、`RpcRequest`/`RpcResult` |
| 网络传输层 | 基于 Netty 的高性能异步 NIO，连接池/心跳/重连 | `Transport`、`Channel` |
| 序列化层 | 支持多种序列化协议，SPI 可扩展 | `Serialization` |

**核心设计点**：

1. **Invoker 抽象**：所有调用统一为 `Invoker.invoke(Invocation)`，无论本地还是远程。这使得 Cluster、LoadBalance、Filter 等可以在不感知具体协议的情况下工作

2. **URL 统一配置模型**：一个 URL 携带服务地址和所有治理参数（超时、权重、版本、序列化方式等），穿透整个调用链，是动态配置的核心载体

3. **SPI 插件机制**：协议、序列化、注册中心、负载均衡、集群策略全部抽象为 SPI 扩展点，按需加载，框架核心与实现完全解耦

4. **主从 Reactor 线程模型**：Boss 线程组处理连接，Worker 线程组处理 I/O 读写，业务逻辑交给独立的业务线程池，避免阻塞 I/O 线程

5. **优雅上下线**：Provider 先注册后暴露服务；关闭时先注销，等待处理中请求完成，再释放资源。Consumer 利用本地缓存应对注册中心抖动

6. **Filter 链**：通过责任链模式串联监控、日志、权限、限流、熔断等治理逻辑，与业务代码完全解耦

**如果要超越 Dubbo，我会在以下方面创新**：

- 默认采用应用级服务发现（类似 Dubbo 3.x），天然适配 Kubernetes
- 协议层内置 TraceID 传递，暴露标准 Metrics 接口（Micrometer），与 OpenTelemetry 集成
- 治理规则基于声明式 YAML，全部通过配置中心动态下发，零重启生效

###### 2. 如何基于 Dubbo 设计一个微服务架构？

基于 Dubbo 设计微服务架构，远不止引入 RPC，而是一套涵盖**服务治理、监控、部署、运维**的完整体系。

**组件选型**：

| 职责 | 选型 | 理由 |
|---|---|---|
| 注册中心 + 配置中心 | **Nacos** | AP 高可用，服务发现 + 配置二合一，与 Dubbo/Spring Cloud Alibaba 集成最好 |
| 流量治理 + 熔断限流 | **Sentinel** | 通过 Dubbo Filter 深度集成，支持方法级规则，控制台可视化 |
| 监控指标 | **Prometheus + Grafana** | 收集 JVM 和 Dubbo Metrics（QPS/RT/错误率），制作实时大盘 |
| 链路追踪 | **SkyWalking** | 无侵入 Agent，自动注入 TraceID，支持 Dubbo 协议解析 |
| API 网关 | **Spring Cloud Gateway** 或 **Apache ShenYu** | 南北向流量，认证、限流、HTTP → Dubbo 协议转换 |

**Dubbo 关键配置策略**：

```yaml
dubbo:
  application:
    name: user-service
    register-mode: instance   # 应用级服务发现（3.x 推荐）
  provider:
    delay: -1                 # 等 Spring 初始化完成后暴露
    timeout: 3000
    retries: 2
  consumer:
    timeout: 3000
    check: false              # 启动时不检查依赖服务，避免启动顺序问题
```

**集群容错策略**：读服务用 `Failover`（失败自动重试），核心写服务用 `Failfast`（快速失败，防止重复写入）

**架构全景**：
```
外部请求 → API 网关（认证/限流）
          → Dubbo 服务集群（用户服务/订单服务/支付服务...）
          ↕ Nacos（服务发现 + 配置）
          ↕ Prometheus（Metrics）
          ↕ SkyWalking（链路追踪）
          ↕ Sentinel 控制台（流量规则）
          → Seata（分布式事务，按需）
```

成功的关键：**清晰边界 + 完善治理 + 自动化运维 + 全链路可观测性**

###### 3. Dubbo 在实际项目中如何进行性能调优？

性能调优是系统工程，从多个层面逐步推进，**监控先行，数据驱动**。

**1. 协议与序列化优化（收益最大）**

```java
// 使用 Kryo 高性能序列化，需预注册所有传输类
public class MySerializationOptimizer implements SerializationOptimizer {
    @Override
    public Collection<Class> getSerializableClasses() {
        return Arrays.asList(UserDTO.class, OrderDTO.class, ...);
    }
}
```

```yaml
dubbo.protocol.serialization: kryo
dubbo.protocol.optimizer: com.example.MySerializationOptimizer
```

性能排序：Kryo ≈ FST > Hessian2 >> JSON >> Java 原生序列化

**2. 线程模型调优**

```yaml
dubbo.protocol.threads: 400     # 根据压测结果调整，IO密集型可适当调大
dubbo.protocol.threadpool: eager # 突发流量时优先创建线程，而非排队
```

线程数参考公式：`线程数 = (等待时间 + CPU时间) / CPU时间 × CPU核数`

IO 密集型（含 DB/Redis 调用）可取 3-5 倍 CPU 核数。

**3. 超时与重试策略**

```yaml
# 分级超时：核心服务短，非核心长
dubbo.provider.timeout: 1000
# 消费者超时必须 > 提供者超时，避免提供者还在处理消费者已超时重试
dubbo.consumer.timeout: 2000  

# 写操作严禁重试！
@DubboReference(retries = 0)
private OrderService orderService;
```

**4. 避免大包传输**

Dubbo 协议不适合传输超大包（> 1MB）。大文件传输走专用通道（OSS/SFTP），只传文件标识符。

**5. JVM 调优**

```
-XX:+UseG1GC -Xms4g -Xmx4g -XX:MaxGCPauseMillis=200
```

G1 GC 目标是控制 GC 停顿时间，避免因 Full GC 导致 Dubbo 线程暂停引发超时。

**6. 调优流程**

建立基线 → 压测找瓶颈 → 针对性优化（一次只改一个变量）→ 验证效果 → 下一轮循环

**调优心法**：监控先行，数据驱动；由外而内，逐层深入；一次只改一个变量，对比验证。

###### 4. 如何设计 Dubbo 的监控告警系统？

监控三大支柱：**Metrics（指标）+ Tracing（链路）+ Logging（日志）**，缺一不可。

**1. 指标采集**

```yaml
# Dubbo 3.x 内置 Metrics 支持
dubbo:
  metrics:
    protocol: prometheus  # 暴露 /metrics 端点
    aggregation:
      enabled: true
```

核心指标：
- **QPS/吞吐量**：每个服务/方法的调用次数/秒
- **响应时间**：P50/P90/P99/P999 分位值（P99 是核心 SLA 指标）
- **错误率**：调用失败比例
- **线程池**：活跃线程数、队列积压、拒绝数
- **连接数**：活跃 TCP 连接数

**2. 链路追踪**

SkyWalking Agent 方式（推荐，零侵入）：

```yaml
# JVM 启动参数
-javaagent:/path/to/skywalking-agent.jar
-Dskywalking.agent.service_name=user-service
-Dskywalking.collector.backend_service=oap-server:11800
```

TraceID 自动通过 Dubbo Attachment 在服务间传递，无需修改业务代码。

**3. 日志与 TraceID 关联**

```xml
<!-- logback.xml：MDC 中注入 TraceID，所有日志自动包含 -->
<pattern>%d{yyyy-MM-dd HH:mm:ss} [%X{SW_CTX}] [%thread] %-5level %logger - %msg%n</pattern>
```

结合 ELK，通过 TraceID 可以串联一次请求跨服务的全链路日志。

**4. 告警规则设计**

```yaml
# Prometheus AlertManager 规则示例
groups:
- name: dubbo-alerts
  rules:
  - alert: DubboHighErrorRate
    expr: rate(dubbo_provider_requests_total{result="fail"}[5m]) / rate(dubbo_provider_requests_total[5m]) > 0.01
    for: 2m
    labels:
      severity: critical
    annotations:
      summary: "服务 {{ $labels.service }} 错误率超过 1%"

  - alert: DubboHighP99Latency
    expr: histogram_quantile(0.99, rate(dubbo_provider_request_duration_seconds_bucket[5m])) > 1
    for: 3m
    annotations:
      summary: "服务 P99 响应时间超过 1 秒"
      
  - alert: DubboThreadPoolSaturation
    expr: dubbo_thread_pool_active_count / dubbo_thread_pool_max_size > 0.9
    for: 1m
    annotations:
      summary: "线程池使用率超过 90%，即将满载"
```

**5. 整体架构**

```
Dubbo 应用
  ├── (Metrics) → Prometheus → Grafana（大盘 + 告警）
  ├── (Traces) → SkyWalking Agent → OAP Server → SW UI
  └── (Logs) → Filebeat → Elasticsearch → Kibana

告警通知：AlertManager → 企业微信/钉钉/短信/电话
```

###### 5. Dubbo 的最佳实践有哪些？

**接口设计**：
- API 模块分离：接口/DTO/异常独立成 `xxx-api` Maven 模块，版本严格锁定
- 使用专门的 DTO 传输对象，不要把 JPA 实体直接丢给 RPC（延迟加载代理无法序列化）
- 返回值包装成 `Result<T>` 或使用受检异常，清晰表达错误
- 每次不兼容变更必须升 `version`，保留旧版本供过渡期使用

**配置管理**：
- 注册中心地址、超时等配置外部化到 Nacos，实现动态刷新
- 读操作 `retries=2`（默认）；**写操作严格设置 `retries=0`**，防止重复提交
- 关闭启动检查 `check=false`，避免因依赖服务未就绪导致自身启动失败

**服务治理**：
- 读服务用 `Failover`；核心写服务用 `Failfast` 或 `Failsafe`
- 新机器上线设置低权重（如 10），配置 `warmup` 时间缓慢增加流量，避免冷启动压垮
- 务必监控业务线程池活跃度，提前发现线程池饱和预警

**代码规范**：
- 避免 RPC 接口重载方法（某些序列化框架无法区分）
- 谨慎使用 `RpcContext` 隐式传参，保持接口语义清晰
- 所有 RPC 接口用 JavaDoc 说明超时预期、可重试性、幂等性

###### 6. 在大流量场景下如何使用 Dubbo？

大流量场景的核心挑战：**高并发低延迟 + 系统不崩溃**。

**架构层面**：
- 无状态服务设计（会话状态外置 Redis），支持水平扩展
- 多级缓存：本地 Guava Cache（极少变化数据）+ Redis（热点数据）
- 数据库层提前完成读写分离/分库分表

**流量控制**：
- Provider 端：`executes` 限制方法并发 + Sentinel QPS 限流
- Consumer 端：`actives` 限制对单 Provider 的并发请求，防止压垮
- 非核心服务预先准备 Mock 降级策略，Sentinel 熔断后自动触发

**性能极致优化**：
- 使用 Kryo/FST 序列化，传输 DTO 精简字段
- `LeastActive` 负载均衡策略，将请求路由到当前最空闲的实例
- 核心服务超时设置短（如 200ms），快速失败而不长时间占用线程

**弹性高可用**：
- 提供者至少 3 实例，跨可用区部署
- 新版本发布通过 `warmup` + 标签路由灰度验证，确认稳定后全量
- K8s HPA 基于 Dubbo QPS 指标自动扩容

**大促预案**：
- 提前全链路压测，确认单机容量和系统瓶颈
- 准备好降级预案（可以关闭哪些非核心功能）
- 大促期间 7×24 小时监控大屏，核心服务设置秒级告警

###### 7. Dubbo 服务如何进行压力测试？

专业压测 = 真实场景模拟 + 全链路监控 + 明确性能基线 + 精准瓶颈定位。

**环境准备**：
- 压测环境硬件/中间件配置与生产一致
- 使用 Dubbo 的 `group` 机制隔离压测流量，`group="pressure-test"` 避免影响线上

**工具选型**：
- **JMeter + ApacheJMeter_dubbo 插件**：最常用，支持注册中心发现或直连
- **自研（泛化调用）**：大型企业常见，基于 `GenericService` 灵活模拟复杂场景
- **dubbo-benchmark**：官方基准测试，适合框架层性能对比

**脚本设计要点**：
- 模拟完整链路（登录 → 查询 → 下单），而不是单接口
- CSV 数据驱动，每次请求用不同的用户/商品 ID，避免缓存失真
- 阶梯式加压（50 → 100 → 200 → 400 并发），找到性能拐点

**瓶颈分析**：

| 现象 | 可能瓶颈 | 定位手段 |
|---|---|---|
| QPS 上不去，CPU 低 | 外部依赖（DB/Redis）| 慢 SQL 日志、连接池监控 |
| QPS 上不去，CPU 高 | 计算密集或序列化瓶颈 | Async-Profiler 火焰图 |
| 响应时间随压力线性增长 | 锁竞争或资源排队 | jstack 线程分析 |
| 线程池满 | 业务慢、线程数配置小 | jstack + 监控大盘 |

**容量规划**：压测得到单机最大安全 QPS（错误率 < 0.1% 的最高值），根据预估流量计算所需机器数，再保留 30-50% 余量。

###### 8. 如何设计 Dubbo 的服务拆分策略？

服务拆分的好坏决定了系统的**可维护性、可扩展性和团队协作效率**。

**指导原则：DDD（领域驱动设计）**

不要按技术层（Controller/Service/DAO）拆，要按**业务领域**拆：
- 识别**限界上下文**（Bounded Context）：订单域、支付域、用户域、库存域、商品域
- 每个上下文对应一个服务，内部高内聚，接口低耦合

**拆分粒度原则**：

- **初期宜粗不宜细**：3-5 个核心服务起步，避免过度的分布式复杂性
- **随业务成长细化**：当某服务代码库过大、发布冲突频繁时，再二次拆分
- **单服务规模**：一个团队（5-9 人）能完整负责的规模，符合康威定律

**Dubbo 视角的服务设计规范**：

```java
// 1. API 模块独立，接口设计稳定
// user-api 模块
public interface UserService {
    Result<UserDTO> getUser(Long userId);  // 返回包装类型，清晰错误处理
    Result<UserDTO> createUser(CreateUserRequest request);  // 使用 Request 对象，便于扩展
}

// 2. 服务自治：独立数据库，禁止跨服务直接访问他人 DB
@DubboService(version = "1.0.0", group = "user-core")
public class UserServiceImpl implements UserService {}
```

**依赖与通信模式选择**：

| 场景 | 通信方式 |
|---|---|
| 需要立即得到结果 | Dubbo RPC 同步调用，设置合理超时+熔断 |
| 非关键路径，提升吞吐 | Dubbo 异步调用（`CompletableFuture`）|
| 数据最终一致性、解耦 | 消息队列（RocketMQ/Kafka），事件驱动 |

**数据一致性**：
- **强一致**（付款、库存扣减）：Seata AT/TCC 模式，但性能有损耗，谨慎使用
- **最终一致**（推荐）：基于消息队列的可靠事件模式（发件箱模式）或 SAGA

**拆分陷阱**：
- 过度拆分 → 运维复杂度剧增，网络延迟成为瓶颈
- 分布式事务滥用 → 性能差且复杂，多用最终一致性
- 写操作未设置 `retries=0` → RPC 重试导致重复数据
- 缺乏全局 TraceID → 跨服务问题难以定位

**实施步骤**（以电商单体拆分为例）：

1. 识别核心域：商品、订单、用户、支付
2. 优先拆边界清晰、变化少的服务（如商品服务）
3. 数据迁移用双写方案保障同步
4. 调用方改为 Dubbo RPC，灰度验证
5. 重复直到完成

###### 9. Dubbo 与 Seata 分布式事务集成

这是一道高频深挖题，考察对分布式事务架构的理解。

**9.1 为什么 Dubbo + Seata 是常见组合？**

Dubbo 解决了服务间**高效通信**的问题，但当一个业务操作需要跨多个服务原子执行时（如：订单服务扣库存 + 支付服务扣余额 + 积分服务加积分），任何一步失败需要全部回滚，这就需要分布式事务框架 Seata。

**9.2 集成方式**

添加依赖：
```xml
<dependency>
    <groupId>io.seata</groupId>
    <artifactId>seata-spring-boot-starter</artifactId>
    <version>${seata.version}</version>
</dependency>
```

配置：
```yaml
seata:
  application-id: order-service
  tx-service-group: order-tx-group
  service:
    vgroup-mapping:
      order-tx-group: default   # 指向 Seata Server 集群
  registry:
    type: nacos                 # 通过 Nacos 发现 Seata Server
```

业务代码：
```java
@Service
public class OrderServiceImpl implements OrderService {
    
    @DubboReference
    private StockService stockService;
    
    @DubboReference  
    private AccountService accountService;
    
    @GlobalTransactional(timeoutMills = 5000, name = "create-order-tx")
    public void createOrder(OrderRequest request) {
        // 1. 创建订单（本地 DB）
        orderRepository.save(new Order(request));
        
        // 2. 扣减库存（跨服务 Dubbo 调用，自动被 Seata 纳入全局事务）
        stockService.deduct(request.getProductId(), request.getQuantity());
        
        // 3. 扣减账户余额（另一个跨服务调用）
        accountService.deduct(request.getUserId(), request.getAmount());
        
        // 任何一步失败，@GlobalTransactional 自动回滚所有操作
    }
}
```

**9.3 工作原理**

Seata AT 模式工作流程：

1. `@GlobalTransactional` 开启全局事务，Seata TC（事务协调器）分配全局事务 ID（XID）
2. XID 通过 Dubbo `RpcContext` 的 Attachment 自动在服务间传递
3. 每个服务的本地操作被 Seata DataSource 代理：执行前记录 before image，执行后记录 after image，生成 undo log
4. 全部成功 → TC 通知各服务提交
5. 任意失败 → TC 通知各服务根据 undo log 执行反向 SQL 回滚

**9.4 XID 传递机制（源码层面）**

Seata 提供了专门的 Dubbo Filter：
```java
// seata-dubbo 模块中的 ApacheDubboTransactionPropagationFilter
@Activate(group = {CONSUMER, PROVIDER}, order = 100)
public class ApacheDubboTransactionPropagationFilter implements Filter {
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) {
        String xid = RootContext.getXID();
        String rpcXid = RpcContext.getContext().getAttachment(RootContext.KEY_XID);
        
        if (StringUtils.isNotEmpty(xid) && StringUtils.isEmpty(rpcXid)) {
            // Consumer 端：将本地 XID 放入 Attachment，传递给 Provider
            RpcContext.getContext().setAttachment(RootContext.KEY_XID, xid);
        } else if (StringUtils.isEmpty(xid) && StringUtils.isNotEmpty(rpcXid)) {
            // Provider 端：从 Attachment 中取出 XID，绑定到当前线程
            RootContext.bind(rpcXid);
        }
        // ...
    }
}
```

**9.5 使用建议**

- Seata AT 模式适合对**强一致性要求高**但能容忍一定性能损耗的场景（如支付、库存）
- 对于**高并发写入**场景，AT 模式的行锁开销较大，考虑 TCC 模式（需要业务实现 try/confirm/cancel）
- **最终一致**的场景（如发送通知、更新统计）优先使用消息队列，不要滥用 Seata
- Seata Server 本身需要高可用部署，避免成为单点
