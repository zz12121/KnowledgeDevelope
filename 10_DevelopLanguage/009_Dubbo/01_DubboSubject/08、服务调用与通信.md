###### 1. Dubbo 服务之间的调用是阻塞的吗？
取决于调用方式，**默认是同步阻塞调用**，但 Dubbo 完整支持**异步非阻塞调用**。

**同步阻塞（默认）**：Consumer 线程发起调用后阻塞等待，直到 Provider 返回结果或超时抛出异常。底层在 `DubboInvoker.doInvoke()` 中调用 `DefaultFuture.get()` 进行阻塞——`DefaultFuture` 内部用 `Lock + Condition`，I/O 线程收到响应后唤醒等待线程。

**异步非阻塞**：需显式配置或使用 `CompletableFuture` 接口。`DubboInvoker.doInvoke()` 立即返回 `AsyncRpcResult`（内持 `CompletableFuture`），I/O 线程收到响应后调用 `DefaultFuture.received()` 完成 Future，触发回调。

选择原则：逻辑简单、响应快、需顺序执行 → 同步；I/O 密集、响应慢、需并行调用多服务 → 异步。

###### 2. Dubbo 服务调用默认几次？
需区分**首次调用**和**失败重试**：

- 首次调用：总发生一次。
- 默认 **`retries=2`**，即首次失败后最多重试 2 次，**最多调用 3 次**。

注意：只有可重试的异常（网络超时、连接异常）会触发重试，业务异常（`RpcException` 但非网络超时）不重试。**写操作必须设 `retries=0`！**

源码：`FailoverClusterInvoker.doInvoke()` 根据 `retries` 循环，捕获可重试异常时选下一个 Invoker 继续尝试。`retries=0` 表示不重试，共调用 1 次。

###### 3. 如何解决 Dubbo 的服务调用超时问题？
系统性排查五步走：

**1. 定位超时环节**：接入 SkyWalking/Zipkin 链路追踪，确认超时在网络传输、服务端处理还是数据库等下游依赖；用 QoS 命令 `telnet localhost 22222 → count [service]` 查看实时耗时分布。

**2. 合理配置超时**：遵循**消费者超时 > 提供者超时**原则，方法级别精细设置：
```java
@DubboReference(methods = {
    @Method(name = "heavyQuery", timeout = 10000),
    @Method(name = "simpleQuery", timeout = 1000)
})
private DemoService demoService;
```

**3. 分析慢服务**：`jstack` 查看线程堆栈，排查锁竞争、慢 SQL、Full GC；调整线程池 `threads`/`dispatcher` 参数，避免任务队列积压。

**4. 网络与序列化优化**：检查网络延迟丢包率；大数据量换高效序列化（Kryo/FST），避免传大对象。

**5. 容错配置兜底**：读操作 `retries=1-2`，写操作 `retries=0` + `cluster=failfast`；配置 Mock 降级，超时返回兜底数据。

源码：超时控制在 `HeaderExchangeChannel.request()` 创建请求时提交 `HashedWheelTimer` 延迟任务，超时后主动触发 `DefaultFuture` 标记超时并抛 `TimeoutException`，是**消费者端主动判断**，不依赖 Provider。

###### 4. Dubbo 的服务暴露和引用过程是怎样的？
**服务暴露（Provider 端）**，核心调用链：`ServiceConfig.export() → RegistryProtocol.export() → DubboProtocol.export()`

1. Spring 解析 `@DubboService`，生成 `ServiceBean`（继承 `ServiceConfig`）。
2. `ServiceBean.afterPropertiesSet()` 调用 `export()`。
3. `ProxyFactory`（默认 `JavassistProxyFactory`）将业务实现包装成 `AbstractProxyInvoker`。
4. `ProtocolFilterWrapper.export()` 将 Invoker 包装进 Filter 链。
5. `DubboProtocol.export()` 为协议+端口创建（或复用）`ExchangeServer`，启动 Netty 监听端口。
6. `RegistryProtocol.export()` 将服务 URL（含 IP/Port/接口名）注册到注册中心。

**服务引用（Consumer 端）**，核心调用链：`ReferenceConfig.get() → RegistryProtocol.refer() → Cluster.join() → ProxyFactory.getProxy()`

1. Spring 解析 `@DubboReference`，生成 `ReferenceBean`（实现 `FactoryBean`）。
2. `ReferenceBean.getObject()` → `ReferenceConfig.get()` → `init()`。
3. `RegistryProtocol.refer()` 订阅注册中心，将每个 Provider 地址转成 `Invoker`。
4. `FailoverCluster.join()` 将多个 Invoker 合并成一个集群 Invoker（内含负载均衡和容错逻辑）。
5. `JavassistProxyFactory.getProxy()` 生成目标接口的代理对象注入到字段。

###### 5. 什么是 Dubbo 的泛化调用？
泛化调用允许**消费者不依赖服务接口 JAR 包**，通过方法名、参数类型和参数值发起远程调用。常用于**服务测试、网关代理、动态路由**等无法预先引入接口包的场景。

**使用示例**：
```java
ReferenceConfig<GenericService> reference = new ReferenceConfig<>();
reference.setInterface("com.example.DemoService");
reference.setGeneric(true);  // 声明泛化调用
GenericService genericService = reference.get();

// 发起泛化调用
Object result = genericService.$invoke(
    "sayHello",
    new String[]{"java.lang.String"},
    new Object[]{"world"}
);
```

**实现原理**：
- **Consumer 端**：`GenericFilter` 将 POJO 参数标准化为 `Map`，封装成 `RpcInvocation`。
- **Provider 端**：收到请求后同样经过 `GenericFilter`，通过反射调用真实业务实现，结果泛化转换后返回。
- **序列化要求**：参数和返回值必须是基本类型、Map、List 或 `Serializable` 对象。

核心类：`GenericFilter`、`GenericService`（接口）、`GenericServiceImpl`（Provider 端泛化处理实现）。

###### 6. Dubbo 如何实现异步调用？
**推荐方式：接口定义返回 CompletableFuture（Dubbo 2.7+）**

```java
// 接口定义
public interface DemoService {
    CompletableFuture<String> sayHelloAsync(String name);
}

// Provider 实现
@DubboService
public class DemoServiceImpl implements DemoService {
    @Override
    public CompletableFuture<String> sayHelloAsync(String name) {
        return CompletableFuture.supplyAsync(() -> {
            // 异步执行耗时逻辑
            return "Hello " + name;
        });
    }
}

// Consumer 调用
@DubboReference
private DemoService demoService;

public void doAsync() {
    CompletableFuture<String> future = demoService.sayHelloAsync("world");
    future.whenComplete((result, ex) -> {
        if (ex != null) ex.printStackTrace();
        else System.out.println("结果：" + result);
    });
    // 主线程继续执行其他任务，不阻塞
}
```

**传统方式（RpcContext 异步，不推荐）**：
```java
@DubboReference(async = true)
private DemoService demoService;

demoService.sayHello("world");  // 立即返回 null
Future<String> future = RpcContext.getContext().getFuture();
String result = future.get();  // 按需阻塞获取
```

**底层实现**：异步调用时 `DubboInvoker.doInvoke()` 返回 `AsyncRpcResult`（内持 `CompletableFuture`），I/O 线程收到响应后 `DefaultFuture.received()` 完成 Future，触发注册的回调。

###### 7. Dubbo 的同步调用和异步调用有什么区别？

| 特性 | 同步调用 | 异步调用 |
|---|---|---|
| **调用行为** | 调用线程**阻塞**等待结果 | 调用线程**立即返回**，通过 Future 获取结果 |
| **线程利用率** | 等待期间线程被占用，高并发易耗尽线程池 | 不阻塞线程，可承载更高并发 |
| **编程复杂度** | 简单直观，顺序执行 | 需处理 CompletableFuture 回调，略复杂 |
| **适用场景** | 逻辑简单、响应快、需顺序执行 | I/O 密集、响应慢、需并行调用多服务 |
| **实现方式** | 默认，`DefaultFuture.get()` 阻塞 | 接口返回 CompletableFuture 或 `async=true` |
| **性能** | 延迟 = 网络 + 服务处理 | 提升吞吐量，降低感知延迟 |

**适合异步的典型场景**：同时调用多个服务聚合结果（如查询多个子系统，最后汇总）、发起调用后不需要立即用结果（如发送通知异步等待确认）。

###### 8. Dubbo 如何实现服务回调？
服务回调允许**Provider 调用 Consumer 端的方法**，实现反向调用（类似订阅通知模式）。

**实现步骤**：
```java
// 1. 定义回调接口（放在 API 模块共享）
public interface DataChangeListener {
    void onChanged(String data);
}

// 2. 服务接口中携带回调参数
public interface DataService {
    void subscribe(String topic, DataChangeListener listener);
}

// 3. Consumer 实现回调并注册
@DubboReference
private DataService dataService;

dataService.subscribe("order-topic", event -> {
    System.out.println("收到变更: " + event);
});

// 4. Provider 保存回调并在数据变化时触发
@DubboService
public class DataServiceImpl implements DataService {
    private final ConcurrentMap<String, List<DataChangeListener>> listeners = new ConcurrentHashMap<>();

    @Override
    public void subscribe(String topic, DataChangeListener listener) {
        listeners.computeIfAbsent(topic, k -> new CopyOnWriteArrayList<>()).add(listener);
    }

    public void publishChange(String topic, String data) {
        listeners.getOrDefault(topic, Collections.emptyList())
                 .forEach(l -> l.onChanged(data));  // 反向 RPC 调用
    }
}
```

**底层原理**：Consumer 传递回调对象时，Dubbo 在 Consumer 端暴露一个临时服务监听端口，将回调对象包装成 `Invoker`传给 Provider。Provider 保存的是指向 Consumer 的 `Invoker`引用，调用 `listener.onChanged()` 时实际发起了一次 Provider → Consumer 的 RPC。

**注意事项**：依赖双向长连接；需管理回调列表生命周期（提供取消订阅方法），防止内存泄漏。

###### 9. Dubbo 的参数回调是什么？
参数回调是服务回调的具体形式，特指**将回调对象作为方法参数传递，由 Provider 在未来某时间点调用**。上面第8题就是典型的参数回调示例。

关键区分：
- **参数回调**：回调接口作为方法参数传入，Provider 保存引用，后续主动调用（推 push 模式）。
- **事件通知（oninvoke/onreturn/onthrow）**：Consumer 本地的钩子，不涉及 Provider 回调，只在本地执行增强逻辑。

参数回调的实现依赖 `ProxyFactory` 在创建代理时检测参数中的回调接口，为其生成指向 Consumer 的代理 `Invoker` 并传递给 Provider。

###### 10. Dubbo 如何实现事件通知？
事件通知在消费者**本地**执行，在调用前（`oninvoke`）、成功后（`onreturn`）、异常时（`onthrow`）注入自定义逻辑，类似 AOP 切面。

**配置方式**：
```java
// 通知类
@Component("notify")
public class DemoServiceNotify {
    // 调用前：参数与原方法一致
    public void onInvoke(String name) {
        log.info("准备调用 sayHello，参数: {}", name);
    }
    // 成功返回：第1个参数是返回值，后续是原方法参数
    public void onReturn(String result, String name) {
        log.info("sayHello 成功，结果: {}", result);
    }
    // 异常时：第1个参数是异常，后续是原方法参数
    public void onThrow(Throwable ex, String name) {
        log.error("sayHello 异常: {}", ex.getMessage());
    }
}
```
```xml
<dubbo:reference id="demoService" interface="com.example.DemoService">
    <dubbo:method name="sayHello"
        oninvoke="notify.onInvoke"
        onreturn="notify.onReturn"
        onthrow="notify.onThrow" />
</dubbo:reference>
```

**实现原理**：`FutureFilter`（消费者端内置 Filter）拦截调用，在前置阶段通过反射调用 `oninvoke` 方法；根据调用结果（成功/异常）分别调用 `onreturn`/`onthrow` 方法。

**与回调的本质区别**：事件通知是 Consumer 本地执行的钩子；回调是 Provider 发起的反向 RPC 调用。

###### 11. Dubbo 单连接多路复用原理是什么？
Dubbo 协议默认对同一个 Provider 地址只维护**一个 TCP 长连接**，所有并发请求都复用这个连接。

**实现核心**：
1. **请求 ID 唯一标识**：每个 RPC 请求在发送时分配一个唯一的 `requestId`（64位长整型），写入协议头。Provider 返回响应时携带相同的 `requestId`。

2. **DefaultFuture 映射表**：Consumer 端的 `DefaultFuture` 类维护一个 `ConcurrentHashMap<Long, DefaultFuture>`，以 `requestId` 为 Key。发送请求时注册，收到响应时从 Map 中取出对应的 Future 并完成它。

3. **I/O 线程分发**：I/O 线程（Netty EventLoop）读取完整响应后，根据 `requestId` 找到等待的 `DefaultFuture`，唤醒业务线程（同步调用）或触发回调（异步调用）。

```
Consumer 线程1 →─┐
Consumer 线程2 →─┤─→ 同一个 TCP 连接 →─→ Provider
Consumer 线程3 →─┘
              requestId区分不同请求
```

**何时需要多连接**：请求量极大导致单个连接成为 I/O 瓶颈时，可配置 `connections=2-5`（轻量连接池）。绝大多数场景默认单连接完全够用。

**源码**：`DubboProtocol.getSharedClient()` 从 `referenceClientMap` 缓存中获取或创建 `ReferenceCountExchangeClient`（引用计数包装的连接），同一地址的所有 Invoker 共享。
