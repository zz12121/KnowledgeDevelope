###### 1. Dubbo 如何实现服务鉴权？

Dubbo 2.7.5+ 引入了基于 **AK/SK 签名认证**的鉴权机制，为敏感业务提供身份验证和防篡改保障。

**核心流程**：

**① 消费者端（签名生成）**
在调用需要鉴权的服务前，消费者用 SK（Secret Key）对请求元数据（接口名、方法名、时间戳，可选加入请求参数）做 HMAC-SHA256 哈希，生成签名（Signature）。时间戳用于防重放攻击（通常允许±5分钟的时间偏差）。

**② 传递**
将 AK（Access Key）、签名、时间戳通过 Dubbo Attachment 机制（`RpcContext`/`Invocation` 的附件）随请求发送。

**③ 提供者端（验签）**
`AuthFilter`（Provider 端 Filter）从附件中取出 AK，查询对应的 SK（可从鉴权中心或本地缓存获取），用相同算法重新计算签名，与传来的签名比对。一致且时间戳有效则通过，否则抛 `RpcException`。

**配置**：
```java
@DubboService(parameters = {"service.auth", "true", "param.sign", "true"})
public class PaymentServiceImpl implements PaymentService { ... }
```

**高级功能**：AK/SK 可由鉴权服务中心动态下发，无需重启服务刷新密钥，避免硬编码。

###### 2. Dubbo 如何实现 Token 验证？

Token 验证是 Dubbo 最简单的访问控制机制，**核心目的是防止消费者绕过注册中心直连提供者**。

**工作原理**：

1. **Provider 配置 Token**：Token 作为服务元数据发布到注册中心。
   ```java
   @DubboService(token = "true") // true表示自动生成UUID作为Token
   // 或者
   @DubboService(token = "my-fixed-token")
   public class UserServiceImpl implements UserService { ... }
   ```

2. **Consumer 自动获取 Token**：Consumer 订阅服务时从注册中心拿到 Token，调用时自动附在 Attachment 里。

3. **Provider 端 TokenFilter 验证**：
```java
@Activate(group = PROVIDER, value = TOKEN_KEY) // 有token参数才激活
public class TokenFilter implements Filter {
    @Override
    public Result invoke(Invoker<?> invoker, Invocation inv) throws RpcException {
        String token = invoker.getUrl().getParameter(TOKEN_KEY);
        if (ConfigUtils.isNotEmpty(token)) {
            String remoteToken = (String) inv.getObjectAttachmentWithoutConvert(TOKEN_KEY);
            if (!token.equals(remoteToken)) {
                throw new RpcException("Invalid token! Forbid invoke...");
            }
        }
        return invoker.invoke(inv);
    }
}
```

**Token 的局限性**：只能防止直连绕过，不能防止合法 Consumer 伪造调用。对安全要求高的场景应使用 AK/SK 签名认证。

###### 3. Dubbo 如何防止服务被非法调用？

安全是纵深防御体系，Dubbo 提供四道防线：

**第一道：认证与鉴权**
- **Token 验证**：防止绕过注册中心直连，简单易用
- **AK/SK 签名认证**：身份验证 + 防篡改 + 防重放，适合对外暴露的核心接口

**第二道：访问控制（ACL）**
- **IP 白名单**（自定义 Filter）：
```java
@Activate(group = PROVIDER)
public class IpWhiteListFilter implements Filter {
    private Set<String> whiteList = Set.of("10.0.0.1", "10.0.0.2");
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) {
        String clientIp = RpcContext.getContext().getRemoteHost();
        if (!whiteList.contains(clientIp)) {
            throw new RpcException(RpcException.FORBIDDEN_EXCEPTION, "IP not allowed: " + clientIp);
        }
        return invoker.invoke(invocation);
    }
}
```
- **条件路由黑白名单**：在 Dubbo Admin 中配置"禁止应用 B 调用服务 X"的路由规则，应用级别的访问控制

**第三道：传输安全（SSL/TLS）**
Dubbo 2.7.5+ 支持为 Netty 通道加 TLS：
```yaml
dubbo:
  protocol:
    ssl-enabled: true
    ssl-server-key-cert-chain-path: classpath:server-cert.pem
    ssl-server-private-key-path: classpath:server-key.pem
```
防止通信内容被窃听或中间人攻击。

**第四道：流量防护**
- `executes` + `actives` 限制并发，防止恶意流量打垮服务
- 集成 Sentinel 做更精细的限流熔断
- 时间戳防重放（AK/SK 鉴权中内置）

**最佳实践**：内网可信环境用 IP 白名单 + Token；对外接口必须 SSL/TLS + AK/SK 签名。

###### 4. Dubbo 支持 SSL/TLS 加密吗？

**支持，Dubbo 2.7.5+ 为 Dubbo 协议（Netty）和 Triple 协议提供了完整的 TLS 支持。**

**配置方式**（Spring Boot）：
```yaml
dubbo:
  protocol:
    name: dubbo
    port: 20880
    ssl-enabled: true
  ssl:
    # Provider端证书
    server-key-cert-chain-path: classpath:certs/server.crt
    server-private-key-path: classpath:certs/server.key
    # 双向TLS（mTLS）：还需要配置信任的CA
    server-trust-cert-collection-path: classpath:certs/ca.crt
    # Consumer端
    client-trust-cert-collection-path: classpath:certs/ca.crt
    client-key-cert-chain-path: classpath:certs/client.crt  # mTLS才需要
    client-private-key-path: classpath:certs/client.key
```

**源码实现**：`NettyServer`/`NettyClient` 初始化时检查 `SslConfig`，通过 `SslContextBuilder` 创建 `SslContext`，添加 `SslHandler` 到 Netty ChannelPipeline 的首位，在握手阶段完成 TLS 协商。

**性能影响**：TLS 握手（首次）有开销，但 Dubbo 使用长连接，后续请求复用已建立的 TLS 会话，额外延迟约 1~2ms，对高频调用影响可接受。

###### 5. Dubbo 如何实现接口级别的访问控制？

接口级访问控制要求精细到**特定接口甚至特定方法**的调用鉴权，最灵活的方式是**自定义 Filter**：

```java
@Activate(group = PROVIDER, order = -10000) // order很小，放在链头，最先鉴权
public class InterfaceAuthFilter implements Filter {
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        String serviceName = invoker.getInterface().getName();
        String methodName = invocation.getMethodName();
        // 消费者应用名通过ContextFilter设置到RpcContext
        String consumerApp = invocation.getAttachment("application");
        
        // 从配置中心或本地规则查询权限
        if (!permissionService.hasPermission(consumerApp, serviceName, methodName)) {
            throw new RpcException(RpcException.FORBIDDEN_EXCEPTION,
                String.format("[%s] is not allowed to call [%s.%s]",
                    consumerApp, serviceName, methodName));
        }
        return invoker.invoke(invocation);
    }
    
    private PermissionService permissionService; // 注入Spring Bean
    public void setPermissionService(PermissionService ps) { this.permissionService = ps; }
}
```

**优势**：
- **动态生效**：权限规则存 Nacos，修改无需重启
- **粒度极细**：可到方法级，甚至基于参数值判断
- **集成灵活**：可对接公司统一权限管理系统（RBAC、ABAC）

**基于注解的声明式控制（进阶）**：
```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequiresPermission {
    String value(); // 如 "order:create"
}
```
在 Filter 中用反射扫描接口方法上的 `@RequiresPermission`，结合 Spring Security 校验。这种方式更声明式，但需要接口方法上加注解，对 API 模块有侵入。

###### 6. Token 验证和 AK/SK 鉴权如何选择？（新增）

这两种机制经常一起问，面试时要能清晰对比：

| 对比维度 | Token 验证 | AK/SK 签名认证 |
|---|---|---|
| **防护目标** | 防止**绕过注册中心直连** | 防止**未授权调用** + **请求篡改** + **重放攻击** |
| **安全强度** | 低（Token 是静态字符串，长期有效） | 高（签名动态生成，时间戳防重放） |
| **实现复杂度** | 简单，Dubbo 内置支持 | 较复杂，需要鉴权服务中心管理 AK/SK |
| **适用场景** | 内网服务间隔离，防止误调用 | 对外暴露的敏感接口，公有云环境 |
| **动态管理** | 需要重新发布服务或配置中心更新 | AK/SK 可由鉴权中心动态下发，不用重启 |
| **性能开销** | 极小（字符串比对） | 稍大（HMAC 计算 + 可能的远程查询 SK） |

**选择建议**：
- **内网微服务之间**：Token 验证够用，主要防止直连绕过注册中心的误操作
- **跨公司/对外 API**：必须 AK/SK + TLS，Token 太弱
- **高安全场景（金融、支付）**：AK/SK + TLS + IP 白名单三重保障

**实际项目常见做法**：大多数内部微服务只开 Token（甚至不开），核心对外接口才上 AK/SK。安全措施的粒度要和接口的敏感度匹配，不要所有接口一刀切上最强安全，会带来不必要的性能和维护成本。
