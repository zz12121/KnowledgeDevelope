###### 1. Dubbo 的 Filter 是什么？有什么作用？

Dubbo Filter 是基于**责任链 + 装饰器模式**实现的调用拦截器，允许在服务调用的关键节点插入自定义逻辑。它是 Dubbo SPI 最典型的应用。

**核心接口**：
```java
@SPI
public interface Filter {
    Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException;
}
```
`invoker` 是链中的下一个节点（框架注入），`invocation` 封装了本次调用的方法名、参数和附件。

**五大核心用途**：
1. **服务治理**：限流（`ExecuteLimitFilter`）、熔断降级（Sentinel集成）、鉴权（`TokenFilter`）
2. **监控统计**：收集 QPS/耗时（`MonitorFilter`）
3. **链路追踪**：TraceID 注入和传递
4. **日志记录**：访问日志（`AccessLogFilter`）
5. **参数校验**：JSR-303 参数校验（`ValidationFilter`）

**架构定位**：`ProtocolFilterWrapper` 是构建 Filter 链的入口，它作为 `Protocol` 的 Wrapper（SPI 装饰器），在 `export()` 和 `refer()` 时调用 `buildInvokerChain()` 将 Filter 组装成链，包装在原始 `Invoker` 外面。

###### 2. Dubbo 有哪些内置的 Filter？

Dubbo 内置 Filter 通过 `@Activate` 注解在特定条件下自动激活：

**消费者端**：
- `ConsumerContextFilter`：设置/清理调用前后的 `RpcContext` 上下文
- `FutureFilter`：支持事件通知（`oninvoke`/`onreturn`/`onthrow`）
- `ActiveLimitFilter`：`actives` 参数控制消费者端并发数

**提供者端**：
- `ContextFilter`：接收消费者传来的附件，设置到本线程 `RpcContext`
- `AccessLogFilter`：记录调用日志到文件（`accesslog` URL 参数触发）
- `TimeoutFilter`：监控超时调用，打印告警日志
- `ExceptionFilter`：**关键 Filter**——将不在接口中定义的受检异常包装为 `RuntimeException`，防止消费者端因找不到异常类而反序列化失败
- `ExecuteLimitFilter`：`executes` 参数控制提供者端方法并发数（`Semaphore` 实现）
- `TpsLimitFilter`：限制每秒事务数
- `TokenFilter`：Token 验证（仅当 URL 有 `token` 参数时激活）

**双向（Consumer + Provider）**：
- `MonitorFilter`：收集调用统计并上报监控中心
- `GenericFilter`：处理泛化调用的参数和返回值转换
- `EchoFilter`：处理 `$echo` 心跳/回声测试请求
- `ValidationFilter`：JSR-303 参数校验（引入 `validation-api` 依赖后激活）

###### 3. 如何自定义 Dubbo 的 Filter？

三步走，参考内置 Filter 的写法：

**步骤一：实现 Filter 接口**
```java
package com.example.filter;

import org.apache.dubbo.common.extension.Activate;
import org.apache.dubbo.rpc.*;
import static org.apache.dubbo.common.constants.CommonConstants.PROVIDER;

@Activate(group = PROVIDER, order = 100) // Provider端自动激活，order越小越靠外
public class MetricsFilter implements Filter {

    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        long start = System.currentTimeMillis();
        String interfaceName = invoker.getInterface().getSimpleName();
        String methodName = invocation.getMethodName();
        try {
            Result result = invoker.invoke(invocation);
            // 调用成功：记录耗时
            long elapsed = System.currentTimeMillis() - start;
            metricsService.record(interfaceName, methodName, elapsed, true);
            return result;
        } catch (RpcException e) {
            // 调用失败：记录错误
            metricsService.record(interfaceName, methodName, System.currentTimeMillis() - start, false);
            throw e;
        }
    }
    
    private MetricsService metricsService; // 通过Setter注入Spring Bean
    public void setMetricsService(MetricsService metricsService) {
        this.metricsService = metricsService;
    }
}
```

**步骤二：创建 SPI 配置文件**
`resources/META-INF/dubbo/org.apache.dubbo.rpc.Filter`：
```
metricsFilter=com.example.filter.MetricsFilter
```

**步骤三：启用（有了 @Activate 会自动激活，也可显式指定）**
```yaml
dubbo:
  provider:
    filter: metricsFilter,-accesslog  # 启用metricsFilter，同时关闭accesslog
```

**关键点**：Filter 的 `finally` 块是后置逻辑执行的地方；如果想中断调用链（如权限校验失败），直接抛 `RpcException` 即可，不调用 `invoker.invoke(invocation)`。

###### 4. Dubbo Filter 的执行顺序是怎样的？

**决定因素**：`@Activate` 的 `order` 属性（越小越靠外）+ 手动配置的 Filter 顺序。

**构建链的逻辑（`ProtocolFilterWrapper.buildInvokerChain()`）**：

```java
// 简化的构建逻辑
Invoker<T> last = invoker; // 原始Invoker（链的末端）
// 逆序遍历Filter列表
for (int i = filters.size() - 1; i >= 0; i--) {
    final Filter filter = filters.get(i);
    final Invoker<T> next = last;
    // 用匿名Invoker包装：调用filter.invoke(next, invocation)
    last = new Invoker<T>() {
        public Result invoke(Invocation invocation) {
            return filter.invoke(next, invocation);
        }
    };
}
return last; // 返回链头
```

**结论**：`order` 越小 → 在 Filter 列表中越靠前 → 构建时越晚被包装 → 成为链头 → **前置逻辑最先执行，后置逻辑最后执行**（像洋葱圈）。

**执行示例**（FilterA order=1, FilterB order=2, FilterC 手动配置）：
```
FilterA.before → FilterB.before → FilterC.before → 业务方法
                                               ↓
FilterA.after  ← FilterB.after  ← FilterC.after  ← 返回结果
```

###### 5. Dubbo 的拦截器链是如何工作的？

Filter 链的工作本质是**动态组装 + 洋葱模型的递归调用**。

**链的执行流程**：
1. 消费者调用动态代理，代理构造 `RpcInvocation` 并调用链头 `Invoker.invoke()`
2. 链头是第一个 Filter 的匿名 `Invoker`，执行 `filter.invoke(next, invocation)`
3. Filter 执行前置逻辑，然后调用 `next.invoke(invocation)` 传递给下一个节点
4. 递归调用直到链末端的原始 `Invoker`（执行实际业务或发送网络请求）
5. 结果逐层返回，每个 Filter 执行后置逻辑

**关键设计**：
- **装饰器模式**：每个 Filter 持有 `next` 的引用（框架构建链时注入），可以在不修改原有代码的情况下增强功能
- **递归调用栈**：利用方法调用栈实现顺序执行（前置）和反向返回（后置）
- **`Invoker` 统一抽象**：无论是原始业务、远程代理还是 Filter 包装，都是 `Invoker`，链式调用成为可能
- **任意 Filter 可以中断链**：不调用 `next.invoke()` 直接返回，即可阻止后续调用（如鉴权失败）

###### 6. Filter 与 AOP 拦截器有什么区别？（新增）

这是面试中容易被追问的比较题。

| 维度 | Dubbo Filter | Spring AOP |
|---|---|---|
| **作用范围** | RPC 调用层面（Consumer/Provider 端） | Spring Bean 方法调用层面（本地） |
| **触发时机** | 发生远程调用时 | 调用任意 Spring Bean 方法时 |
| **上下文** | 可访问 `RpcContext`（远端 IP、附件等 RPC 上下文） | 只有本地 JVM 上下文 |
| **组装方式** | `@Activate` 条件激活 + `buildInvokerChain()` 逆序构建 | `@Aspect` + Pointcut 表达式，Spring 代理动态织入 |
| **适用场景** | 分布式治理：鉴权、限流、链路追踪、监控 | 本地切面：事务、缓存、日志、权限 |

**两者互补**，不是替代关系：
- **Dubbo Filter** 处理 RPC 调用的横切关注点，在服务边界上生效，能感知 RPC 上下文（超时、重试、附件）。
- **Spring AOP** 处理本地方法的横切关注点，更细粒度，能基于方法签名、注解进行切面。

**常见组合**：在 Dubbo Filter 中做 TraceID 的跨服务传递（RPC 层），在 Spring AOP `@Around` 中做本地方法的耗时日志和缓存（业务层）。两层拦截，各司其职。
