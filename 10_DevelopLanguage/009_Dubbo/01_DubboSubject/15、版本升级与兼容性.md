###### 1. Dubbo 3.0 相比之前版本有哪些重要更新？

Dubbo 3.0 是面向云原生时代的重大架构升级，核心变化集中在**服务发现模型、通信协议、云原生集成、架构重构**四个维度。

**1.1 全新的应用级服务发现（最革命性的变化）**

Dubbo 2.x 是**接口级**服务发现，每个接口独立注册。一个应用有 50 个接口、10 个实例，就会在注册中心写入 500 条记录，重复存放大量相同的应用级配置，极度冗余。

Dubbo 3.0 改为**应用级**服务发现：注册中心只存应用实例信息（IP:Port），接口元数据通过独立的 **MetadataService** 同步。这样上面的例子只需 10 条记录，压力降低 90% 以上。

源码实现：`ServiceDiscoveryRegistry.doRegister()` 只注册应用实例 URL；接口元数据由 `MetadataServiceExporter` 发布到元数据中心（Remote 模式）或通过内置 RPC 接口暴露（Local 模式，默认）。

**1.2 Triple 协议（下一代 RPC 协议）**

Dubbo 2.x 的私有协议在网关和 Mesh 环境中穿透性差。3.0 推出 **Triple 协议**，基于 HTTP/2 + Protobuf，完全兼容 gRPC：

- **网关友好**：任何支持 HTTP/2 的网关（Envoy、Nginx）都能直接解析
- **流式支持**：原生支持单向/双向流，适合大文件传输、实时推送等场景
- **多语言互通**：基于 Protobuf IDL，可直接与 Go/Python 的 gRPC 服务互通

核心类：`TripleProtocol`，底层用 Netty 处理 HTTP/2 帧，`TripleCodec` 负责 Attachment 与 HTTP Header 的转换。

**1.3 深度云原生集成**

- 支持 Kubernetes Service 作为注册中心（`dubbo.registry.address=kubernetes://`），无需额外部署 ZK/Nacos
- 提供 **双模 Service Mesh**：Sidecar 模式（Envoy 代理）+ Proxyless 模式（SDK 内置 xDS 客户端）
- QoS 端点（`/live`、`/ready`）与 K8s 健康探针对齐

**1.4 模块化重构**

核心包拆分，`org.apache.dubbo:dubbo` 不再包含所有可选组件，Redis 注册中心、Hessian 协议等需单独引入，架构更轻量、更清晰。

**1.5 统一流量治理规则**

废弃 2.x 中分散的路由和配置规则，引入统一的声明式治理规则（兼容 VirtualService 格式），可同时覆盖 SDK 和 Mesh 部署形态。

###### 2. Dubbo 2.x 和 Dubbo 3.x 有什么区别？

一句话：从**面向接口的 RPC 框架**进化为**面向应用的云原生微服务框架**。

| 对比维度 | Dubbo 2.x | Dubbo 3.x |
|---|---|---|
| **服务发现模型** | 接口级：每个接口独立注册，路径 `/dubbo/com.xxx.UserService/providers/{url}` | 应用级：以应用为粒度注册，路径 `/services/user-app/providers/{instance-info}`，接口信息通过元数据服务同步 |
| **默认 RPC 协议** | Dubbo 私有协议（TCP 二进制），穿透性差 | Triple 协议（HTTP/2），兼容 gRPC，支持流式，网关/Mesh 友好 |
| **注册中心数据量** | N个接口 × M个实例 = N*M 条冗余数据 | 仅 M 条实例记录，压力降低 90% 以上 |
| **云原生支持** | 有限，需外部适配 | 原生深度集成 K8s，提供双模 Mesh 方案 |
| **消费者内存** | 每个接口维护独立地址列表，内存与接口数线性相关 | 只维护应用实例列表，内存降低约 50% |
| **兼容性** | 2.x 终点版本 | 完全兼容 2.x，提供双注册/双订阅迁移机制 |

**源码层关键差异**：

- 2.x 的 `ZookeeperRegistry.doRegister()` 为每个接口创建路径节点
- 3.x 的 `ServiceDiscoveryRegistry.doRegister()` 只注册应用实例；接口元数据走另一条链路（`MetadataService`）
- 2.x 的 `RegistryDirectory` 监听接口路径；3.x 的 `ServiceDiscoveryDirectory` 监听应用路径，再通过服务自省获取接口详情

###### 3. Dubbo 如何实现版本兼容？

Dubbo 通过**多版本共存 + 服务唯一标识三元组 + 灰度迁移规则**来保障版本兼容。

**3.1 核心机制：服务唯一标识 = 接口 + 分组 + 版本号**

不同版本的服务在注册中心被视为完全不同的服务，消费者通过指定 `version` 参数路由到对应版本：

```java
// 提供者：同时暴露 v1 和 v2
@DubboService(version = "1.0.0")
public class UserServiceV1Impl implements UserService {}

@DubboService(version = "2.0.0")
public class UserServiceV2Impl implements UserService {}

// 消费者：按需引用
@DubboReference(version = "1.0.0")
private UserService userServiceV1;

@DubboReference(version = "2.0.0")
private UserService userServiceV2;
```

**3.2 注册中心隔离**

- v1.0.0 注册路径：`/dubbo/com.xxx.UserService/providers/{url}?version=1.0.0`
- v2.0.0 注册路径：`/dubbo/com.xxx.UserService/providers/{url}?version=2.0.0`

消费者订阅时，`RegistryDirectory` 根据配置的 version 和 group 过滤匹配的提供者 URL。

**3.3 灰度发布与迁移规则（3.x 新增）**

Dubbo 3.x 为应用级服务发现升级专门设计了**三态迁移状态机**：

- `FORCE_INTERFACE`：强制消费接口级地址（兼容 2.x 消费者）
- `APPLICATION_FIRST`：优先尝试应用级地址，失败则降级接口级（双订阅，默认过渡态）
- `FORCE_APPLICATION`：强制消费应用级地址（迁移完成）

通过配置中心动态下发 `MigrationRule`，可以实现全集群流量的平滑切换，不需要重启任何服务。

**3.4 接口设计规范（减少版本升级频率）**

- **可以不升版本**：新增方法、新增 POJO 字段（需注意序列化兼容）
- **必须升版本**：删除方法、修改方法签名、修改枚举值（可能引发反序列化异常）

建议参数和返回值使用 POJO 而非接口或枚举的裸类型，提升序列化兼容性。

###### 4. Dubbo 升级需要注意哪些问题？

从 2.x 升级到 3.x 是系统性工程，需要从**依赖管理、配置兼容、发布策略、监控验证**多角度周密规划。

**4.1 依赖管理**

```xml
<!-- 用 BOM 统一管理版本，避免依赖冲突 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.apache.dubbo</groupId>
            <artifactId>dubbo-dependencies-bom</artifactId>
            <version>3.3.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

- **注册中心客户端**：Nacos 必须先将 Server 升级到 2.x 再升 Client；ZK 无需升级 Server
- **模块化注意**：Redis 注册中心、Hessian 协议等需单独引入对应子模块

**4.2 服务发现迁移开关**

升级后默认进入双注册/双订阅的兼容模式，需要通过以下配置明确迁移意图：

```yaml
# 提供者：决定注册行为
dubbo.application.register-mode: all  # all=双注册，instance=仅应用级，interface=仅接口级

# 消费者：决定订阅和选址行为
dubbo.application.service-discovery.migration: APPLICATION_FIRST  # 推荐过渡值
```

**4.3 灰度发布策略（关键！）**

绝对不要一次性全量升级，按照**先提供者，后消费者**的顺序分批进行：

1. 升级 30% 提供者实例，开启双注册
2. 升级对应消费者实例，配置 `APPLICATION_FIRST`
3. 观察关键指标（成功率、RT）稳定后，逐步扩大范围
4. 全量完成后，将提供者改为 `instance`，消费者改为 `FORCE_APPLICATION`，清理旧的接口级数据

**4.4 自定义扩展兼容性检查**

- 自定义 Filter、Router、LoadBalance 需检查与 3.x API 的兼容性（部分扩展点签名有变化）
- `EventDispatcher` 在 3.x 已被移除，相关代码需要重写

**4.5 回滚预案**

发布前必须准备回滚方案：一旦核心指标（成功率下降 > 1%）异常，立即回滚旧版本。双注册/双订阅模式本身就是最好的回滚保障。

###### 5. Dubbo 的应用级服务发现是什么？

应用级服务发现是 Dubbo 3.0 引入的全新模型，核心思想是**将实例寻址与接口元数据解耦**，以应用为粒度进行服务注册与发现。

**5.1 两级发现机制**

**第一级：地址发现**
提供者启动时，只将应用实例信息（应用名、IP、端口）注册到注册中心。消费者拿到的是目标应用的实例列表，而不是接口列表。

**第二级：服务自省**
消费者拿到实例列表后，通过以下方式获取接口详情：
- **Remote 模式**：从独立元数据中心（ZK/Nacos）查询接口元数据
- **Local 模式（默认）**：直接通过 RPC 调用目标实例内置的 `MetadataService` 获取接口信息（点对点，更高效）

**5.2 接口级 vs 应用级数据量对比**

假设一个应用 10 个实例，50 个接口：

| 方面 | 接口级（2.x） | 应用级（3.x） |
|---|---|---|
| 注册数据条数 | 50 × 10 = 500 条（每条重复应用配置） | 10 条实例记录 |
| 消费者 Watch 数 | 需要订阅 50 个接口路径 | 只订阅 1 个应用路径 |
| 实例上下线推送 | 1 个实例下线 → 推送 50 次（推送风暴） | 1 个实例下线 → 推送 1 次 |

**5.3 源码流程**

提供者注册：
1. `ServiceConfig.export()` 触发 `MetadataServiceExporter` 收集接口元数据，生成 `revision`
2. `ServiceDiscoveryRegistry.doRegister()` 向注册中心写入简化的实例 URL

消费者自省：
1. `ServiceDiscoveryRegistryDirectory.subscribe()` 订阅应用路径，获取实例列表
2. 触发 `refreshInvoker()`，调用 `MetadataUtils.getMetadataServiceProxy()` 获取 MetadataService 代理
3. 通过代理调用 `getServiceDefinition()` 获取接口元数据，生成可调用的 `Invoker`

**5.4 优势总结**

- 注册中心存储和推送压力降低 90% 以上，支持百万实例集群
- 与 Kubernetes Service、Service Mesh 天然契合
- 容量评估更直观：只与实例数相关，不受业务接口数量影响

###### 6. 从 FORCE_INTERFACE 到 FORCE_APPLICATION：迁移状态机的实际操作指南

这是一道高频追问题，考察你对 Dubbo 2.x → 3.x 迁移的实战理解。

**6.1 三态状态机**

`MigrationRule` 定义了三种迁移状态，控制消费者如何选择地址：

```
FORCE_INTERFACE (兼容态)
    ↓ 逐步切换
APPLICATION_FIRST (过渡态，默认)
    ↓ 验证稳定
FORCE_APPLICATION (完成态)
```

**6.2 各状态详解**

| 状态 | 行为 | 适用阶段 |
|---|---|---|
| `FORCE_INTERFACE` | 强制使用接口级地址（2.x 模式），忽略应用级地址 | 完全兼容 2.x 消费者，过渡初期 |
| `APPLICATION_FIRST` | 优先使用应用级地址，无可用时降级到接口级（双订阅） | 主迁移阶段，提供者已双注册 |
| `FORCE_APPLICATION` | 强制使用应用级地址，不再感知接口级地址 | 迁移完成，全面 3.x 后 |

**6.3 实际操作步骤**

**Step 1：升级提供者，开启双注册**
```yaml
dubbo.application.register-mode: all  # 同时注册接口级和应用级
```

**Step 2：逐步升级消费者，使用过渡态**
```yaml
dubbo.application.service-discovery.migration: APPLICATION_FIRST
```

消费者会同时订阅两套地址，优先走应用级，回退接口级，形成平滑过渡。

**Step 3：通过配置中心动态下发 MigrationRule（无需重启）**
```yaml
# 可在 Nacos/ZK 上动态下发，实时生效
key: migration-rule
step: APPLICATION_FIRST
threshold: 1.0  # 应用级地址覆盖率达到 100% 才切换
```

**Step 4：全量完成后，收尾**
```yaml
# 提供者：只写应用级
dubbo.application.register-mode: instance

# 消费者：只读应用级  
dubbo.application.service-discovery.migration: FORCE_APPLICATION
```

最后手动清理注册中心上遗留的 2.x 接口级路径数据。

**6.4 关键注意事项**

- `APPLICATION_FIRST` 模式下，如果应用级地址数为 0（提供者未升级），会自动降级到接口级，**不会导致服务中断**
- 迁移过程中要密切监控 `dubbo.application.service-discovery.migration` 指标，确认地址来源正常
- 建议生产环境先在非核心服务上完成迁移演练，积累经验后再推广到核心链路
