###### 1. 说说你对 Dubbo 序列化的了解
序列化是将对象转换为字节流进行网络传输的过程，反序列化则是逆过程。在 Dubbo 中，序列化处于架构最底层的 Serialize 层，直接关系到 RPC 调用的吞吐量、延迟和跨语言能力。

几个关键点：

**定位**：Consumer 端将 `RpcInvocation` 对象序列化成字节发给 Provider，Provider 将 `RpcResult` 序列化后返回。序列化的速度和产出的字节大小直接影响网络带宽消耗和 CPU 开销。

**抽象与扩展**：Dubbo 将序列化抽象为 `org.apache.dubbo.common.serialize.Serialization` SPI 接口，通过 SPI 机制可随意切换。编解码器（`DubboCodec`）根据协议头中的 `serializationId` 查找对应实现，协议层与序列化层解耦。

**四大考量因素**：
- **性能**：序列化速度（CPU）和产出字节大小（网络 IO）。
- **跨语言性**：支持多语言互通（如 Protobuf/JSON/Hessian），适合异构系统。
- **兼容性**：服务接口 DTO 字段增减时，新老版本能否平滑通信。
- **易用性**：是否需要预编译或预定义 Schema（Protobuf 需要 `.proto`，Kryo 需要注册类）。

**源码切入**：`DubboCodec.encodeRequestData()` 获取 `Serialization` 实例，通过 `ObjectOutput` 将数据写入网络缓冲区，这就是序列化在调用链中的位置。

###### 2. Dubbo 支持哪些序列化方式？
通过 `<dubbo:protocol serialization="..." />` 或 `<dubbo:provider serialization="..." />` 配置，主要支持：

| 序列化方式 | 配置值 | 编号 | 特点 |
|---|---|---|---|
| **Hessian2（默认）** | `hessian2` | 2 | 跨语言二进制，兼容性好，性能均衡，无需 Schema |
| **Java 原生** | `java` | 3 | JDK 自带，**不推荐**，性能差体积大，无跨语言能力 |
| **Fastjson** | `fastjson` | 6 | JSON 格式，人类可读，跨语言，性能不及二进制 |
| **Kryo** | `kryo` | 8 | 高性能Java二进制，速度快体积小，需显式注册类，对字段增减不友好 |
| **FST** | `fst` | - | 性能与 Kryo 相当，无需预注册类，需引入依赖 |
| **Protobuf** | `protobuf` | - | Google 出品，跨语言高性能，需定义 `.proto` 文件，兼容性最好，与 Triple 协议完美搭配 |

SPI 扩展文件位于 `META-INF/dubbo/internal/org.apache.dubbo.common.serialize.Serialization`，每种实现按编号注册。

###### 3. 如何优化 Dubbo 的序列化性能？
从四个层面入手：

**1. 选对序列化协议**：
- 纯 Java 环境追求极致性能：选 **Kryo** 或 **FST**，序列化速度和字节大小都显著优于 Hessian2。
- 跨语言场景：**Protobuf + Triple 协议**，兼顾性能和兼容性。
- 默认 **Hessian2** 是稳妥折中，不折腾就用它。

**2. Kryo 类注册优化**：预注册序列化类可大幅提升性能并减少字节体积：
```java
// 实现 SerializationOptimizer 接口
public class MyOptimizer implements SerializationOptimizer {
    @Override
    public Collection<Class> getSerializableClasses() {
        return Arrays.asList(UserDTO.class, OrderDTO.class, ProductDTO.class);
    }
}
```
配置：`<dubbo:protocol serialization="kryo" optimizer="com.example.MyOptimizer" />`

**3. 精简 DTO 对象**：
- 只传 RPC 必需字段，避免传完整的 Entity（含数据库字段、关联对象）。
- 避免嵌套过深和复杂集合结构。
- 优先使用基本类型（`int/long`）而非包装类（`Integer/Long`），减少装箱开销。
- 用 `transient` 关键字排除不需要传输的字段。

**4. 架构层面**：
- 大数据不走 RPC，走 OSS/文件服务；分页或分片替代大对象传输。
- 考虑升级 Triple 协议（HTTP/2 + Protobuf），整体传输效率更高。
- 用 Dubbo Monitor 或 SkyWalking 监控实际序列化耗时，定位瓶颈后针对性优化。

###### 4. Hessian 序列化的优缺点是什么？
**优点**：
1. 二进制协议，性能和体积远优于 Java 原生序列化。
2. 跨语言支持（Java/Python/C++ 等）。
3. 兼容性较好：反序列化时多出的字段被忽略，缺少的字段设为默认值，对字段增减有容忍度，方便服务平滑升级。
4. 无需预定义 Schema，直接序列化 POJO，开箱即用。
5. 经历多年生产验证，稳定可靠。

**缺点**：
1. 性能不是顶尖：相比 Kryo/FST/Protobuf，速度和压缩率已经落后。
2. 对复杂 Java 类型支持有坑：
   - `LinkedHashMap`、`LinkedHashSet` 等特殊集合支持不好。
   - 某些枚举类型序列化存在兼容性问题。
   - `BigDecimal` 序列化可能有精度问题。
3. 版本兼容性：Hessian 与 Hessian2 不兼容，需要客户端和服务端版本匹配。

实际使用中，遇到序列化异常，第一步要排查是否与 Hessian 的上述已知缺陷有关。

###### 5. 为什么 Dubbo 默认使用 Hessian 序列化？
这是**历史选择 + 技术权衡**的结果：

1. **历史背景**：Dubbo 诞生于 2011 年，当时成熟的高性能跨语言序列化选择不多，Hessian 在那个年代已经久经考验，是相对优秀的选择。

2. **跨语言平衡**：阿里早期系统有 Java/C++ 等多语言，Hessian 跨语言特性（相比纯 Java 的 Kryo）更适合异构系统。

3. **开箱即用**：无需 `.proto` 文件编译（像 Protobuf），无需手动注册类（早期 Kryo 需要），直接序列化 POJO，上手成本低。

4. **稳定性**：大量生产案例验证，选它风险低。

**现状与建议**：保留 Hessian2 为默认更多是**向后兼容**考量，避免给海量现存系统带来升级风险。新项目完全可以直接配置 Kryo（纯 Java 场景）或 Protobuf + Triple（跨语言场景），通过一个配置项就能切换，这正是 Dubbo SPI 扩展性的价值所在。

###### 6. 序列化安全漏洞如何防范？
反序列化安全是 Java 生态的老大难问题，Dubbo 也不例外：

**常见攻击方式**：
- **Java 原生反序列化漏洞**（CVE 系列）：攻击者构造恶意序列化数据，在反序列化时触发 RCE（远程代码执行）。典型如 Apache Commons Collections 利用链。
- **Hessian 反序列化漏洞**：攻击者通过精心构造的 Hessian 数据包触发 gadget chain。

**Dubbo 防护措施**：

1. **禁用 Java 原生序列化**：强烈建议不使用 `serialization="java"`，这是最危险的方式。
   ```yaml
   dubbo:
     protocol:
       serialization: hessian2  # 不要用 java
   ```

2. **开启 Token 认证**：在生产环境启用 Token，防止非 Dubbo 客户端直接构造恶意请求：
   ```xml
   <dubbo:service token="true" />
   ```

3. **类白名单（Dubbo 2.7.13+ / 3.x）**：Dubbo 引入反序列化安全机制，只允许白名单类进行反序列化：
   ```yaml
   dubbo:
     application:
       serialize-check-status: STRICT  # STRICT 严格模式，只允许白名单
   ```
   或添加自定义允许类：
   ```
   # META-INF/dubbo/security/serialize.allowlist
   com.example.dto.UserDTO
   com.example.dto.OrderDTO
   ```

4. **升级序列化框架到安全版本**：及时跟进 Hessian/Fastjson 等框架的 CVE 修复版本。

5. **网络层隔离**：Dubbo 服务端口不对公网开放，只在内部网络可访问，从网络层降低攻击面。

###### 7. 主流序列化框架性能对比
在常见业务 DTO 场景下（中等复杂度对象，包含字符串、数字、嵌套对象）：

| 序列化框架 | 序列化速度 | 反序列化速度 | 字节大小 | 跨语言 | 需 Schema | 兼容性 |
|---|---|---|---|---|---|---|
| **Protobuf** | 极快 | 极快 | 最小 | ✅ | 需要 `.proto` | 最好 |
| **Kryo** | 很快 | 很快 | 小 | ❌ | 推荐注册类 | 一般 |
| **FST** | 很快 | 很快 | 较小 | ❌ | 不需要 | 一般 |
| **Hessian2** | 较快 | 较快 | 中等 | ✅ | 不需要 | 好 |
| **Fastjson** | 中等 | 中等 | 较大 | ✅ | 不需要 | 好 |
| **Java 原生** | 慢 | 慢 | 最大 | ❌ | 不需要 | 一般 |

**选型建议**：
- Java 内部服务 → **Kryo** 或 **FST**
- Dubbo 3 + 跨语言 → **Protobuf + Triple**
- 存量系统不想折腾 → 保持 **Hessian2**
- **坚决不用** Java 原生序列化
