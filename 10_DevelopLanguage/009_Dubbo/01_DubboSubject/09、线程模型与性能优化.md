###### 1. Dubbo 的线程模型是怎样的？

Dubbo 的线程模型核心思路是**把网络 I/O 处理和业务逻辑执行分开**，防止慢业务阻塞网络通道，保证高并发吞吐。整体分为三层：

**① Netty I/O 线程（NIO EventLoop）**

由 Netty 的 `EventLoopGroup` 管理，负责 TCP 连接建立、数据收发和协议编解码（`DubboCodec`）。这些线程是非阻塞的，默认数量是 CPU 核数 × 2。它们只做轻量的网络工作，不碰业务逻辑。

**② 线程派发（Dispatcher）**

I/O 线程解码出完整的 `Request`/`Response` 后，由 `Dispatcher` 决定接下来由谁执行：

| 策略 | 说明 |
|---|---|
| `message`（**默认**）| 请求和响应消息派发到线程池，连接/断连事件在 I/O 线程处理 |
| `all` | 所有消息全部派发到线程池 |
| `direct` | 所有消息直接在 I/O 线程执行（谨慎使用，业务慢会拖死网络） |
| `execution` | 仅请求消息派发到线程池，响应和事件在 I/O 线程处理 |
| `connection` | 已废弃 |

**③ 业务线程池**

被派发过来的消息在此执行。对于 Provider，线程池跑服务实现方法；对于 Consumer，线程池处理异步回调。默认是 `FixedThreadPool`，200 个线程。

**总结**：`Netty I/O 线程（收发+编解码）→ Dispatcher（按策略派发）→ 业务线程池（跑业务）`，这三层清晰分离，I/O 高效，业务隔离。

###### 2. Dubbo 的线程池策略有哪些？

Dubbo 通过 `org.apache.dubbo.common.threadpool.ThreadPool` SPI 提供四种业务线程池：

1. **FixedThreadPool（默认）**：核心线程数 = 最大线程数，固定 200 个线程，搭配无界队列。优点是稳定、无频繁创建销毁；缺点是无界队列在突发流量下容易 OOM。

2. **CachedThreadPool**：线程数弹性伸缩，空闲 60 秒自动回收，用 `SynchronousQueue`。适合短生命周期任务，但线程数无上限，峰值可能耗尽资源。

3. **LimitedThreadPool**：核心 0、最大 200，空闲 60 秒回收，用 `SynchronousQueue`。结合了 Fixed 的有界性和 Cached 的弹性，是更安全的"弹性线程池"。

4. **EagerThreadPool**：当核心线程满时**优先创建新线程**而非入队，直到达到最大线程数后才入队。通过自定义 `TaskQueue.offer()` 返回 `false` 来欺骗线程池创建线程，适合不能让任务排队的低延迟场景。

配置方式：`dubbo.protocol.threadpool=eager`、`dubbo.protocol.threads=300`。

**记忆口诀**：Fixed 稳、Cached 弹、Limited 限弹、Eager 急。

###### 3. Dubbo 如何配置线程池大小？

线程池配置主要针对 Provider 端的业务线程池：

**协议级全局配置（最常用）**：
```yaml
dubbo:
  protocol:
    threads: 500
    threadpool: fixed
```

**服务级覆盖**：
```xml
<dubbo:service interface="com.example.SlowService" threads="100"/>
```

**配置建议**：
- I/O 密集型（大量 DB/Redis 调用）：可以设得大些，200~800。
- CPU 密集型（计算）：`CPU 核数 + 1` 左右。
- 没有公式，**必须通过压测**找最优值——监控线程池活跃度（`active`）和队列积压情况，活跃度持续 > 80% 就该调大或优化业务。

**源码路径**：`DubboProtocol.openServer()` → `Exchangers.bind()` → `NettyServer` 初始化时，通过 `ExtensionLoader.getExtension(threadpoolName)` 获取 `ThreadPool` 实现，调用 `getExecutor(url)` 创建线程池。

###### 4. Dubbo 的 IO 模型是什么？

Dubbo 默认使用 **Netty 4 NIO 模型**，底层是主从 Reactor 多线程模式：

- **bossGroup**（主 Reactor）：通常 1 个线程，专门接受客户端 TCP 连接。
- **workerGroup**（从 Reactor）：默认 CPU × 2 个线程，处理已建立连接的读写事件。

所有网络操作（connect、read、write）都是非阻塞的，通过 `ChannelFuture` 提供异步通知，事件驱动。

**Dubbo 里的体现**：
- 客户端：`NettyClient` 用 `NioSocketChannel` + `NioEventLoopGroup`
- 服务端：`NettyServer` 用 `NioServerSocketChannel` + 两个 `NioEventLoopGroup`
- 编解码：`NettyCodecAdapter` 的 `InternalEncoder`/`InternalDecoder` 挂在 ChannelPipeline 上

**IO 模型 vs 线程模型的关系**：IO 模型解决"数据如何高效传输"，线程模型解决"消息解码后由谁执行业务"，两者通过 `Dispatcher` 衔接。

###### 5. 如何优化 Dubbo 的性能？

性能优化是系统工程，分五个层面：

**① 协议与序列化（收益最大）**
- 将默认 `hessian2` 换成 `kryo` 或 `fst`，性能提升明显；用 `SerializationOptimizer` 预注册自定义类。
- 避免传大对象（Dubbo 协议不适合大包，单包超 1MB 建议走专用通道）。

**② 线程模型**
- 根据压测结果调整 `threads`；考虑用 `eager` 线程池减少排队延迟。
- Dispatcher 保持默认 `message` 即可。

**③ 连接管理**
- Dubbo 协议单连接多路复用，**绝大多数场景保持默认 `connections=1`** 即可，盲目加连接只会增加 Provider 压力。
- 心跳间隔（`heartbeat`）可适当调大到 120~180 秒，减少无效心跳包。

**④ 超时与重试**
- 按服务 P99 耗时合理设置 `timeout`，确保**消费者超时 > 提供者超时**，避免 Provider 还在跑、Consumer 已重试。
- 非幂等写操作必须 `retries=0`。

**⑤ JVM 与操作系统**
- 用 G1 GC，合理设置堆大小，减少 Full GC 停顿（GC 停顿会直接导致 Dubbo 线程暂停、请求超时）。
- 调整 Linux 文件描述符上限、`net.ipv4.tcp_tw_reuse` 等 TCP 参数。

**调优闭环**：监控先行（QoS + Prometheus），数据驱动，一次只改一个变量，压测验证。

###### 6. Dubbo 的连接池是如何管理的？

Dubbo 连接管理主要在消费者端，目标是复用 TCP 连接：

1. **单一长连接（默认）**：Dubbo 协议下，一个消费者进程对一个 `host:port` **默认只建 1 个 TCP 连接**，所有并发调用共享这条连接（多路复用）。

2. **懒加载**：首次调用某个 Provider 时才触发连接建立。`DubboProtocol.getSharedClient(url)` 查缓存，不存在则调 `initClient()` 创建 `NettyClient`。

3. **连接缓存**：`DubboProtocol` 维护静态 Map `referenceClientMap: ConcurrentHashMap<String, List<ReferenceCountExchangeClient>>`，key 是服务器地址。配置 `connections=n`（n>1）时，同一地址会有 n 个连接。

4. **心跳保活**：`HeaderExchangeClient` 内置 `HeartbeatTimerTask`，默认 60 秒发送一次心跳，防止连接被防火墙或 NAT 断开。

5. **优雅关闭**：消费者销毁时，`DubboProtocol.destroy()` 逐步关闭所有缓存的连接，`ReferenceCountExchangeClient` 通过引用计数确保所有引用都释放后才真正关闭。

**关键参数**：`connections`（连接数，默认 1）、`heartbeat`（心跳间隔，默认 60000ms）。

###### 7. Dubbo 的长连接和短连接有什么区别？

Dubbo **强制使用长连接**，没有短连接选项，这是其高性能的基础。

| 特性 | 长连接（Dubbo 实际采用） | 短连接（Dubbo 不用） |
|---|---|---|
| **连接建立** | 一次建立，多次复用 | 每次调用新建，用完即关 |
| **资源消耗** | 少，避免反复三次握手/四次挥手 | 多，建连开销大 |
| **性能** | 高，延迟低 | 低，每次调用有建连开销 |
| **服务器压力** | 稳定，连接数 = 消费者进程数 × connections | 瞬间峰值高 |

**Dubbo 能用长连接的三个原因**：
1. **多路复用**：每个请求有唯一 `requestId`，在同一连接上区分不同请求的响应（`DefaultFuture` 维护 requestId → Future 的映射）。
2. **NIO 异步**：非阻塞 I/O 让单线程管理多个并发请求成为可能。
3. **心跳保活**：定期心跳防止连接被中间设备断开。

**结论**：在 Dubbo 体系里，根本不需要考虑"用长连接还是短连接"——就是长连接，没得选，也不需要选。

###### 8. 如何设计业务线程池隔离？（新增）

**问题背景**：Dubbo 默认所有服务共享同一个业务线程池（`FixedThreadPool`，200 线程）。如果某个慢服务（如大量 DB 查询）把线程池塞满，其他快速服务也会被拖累，出现"线程池污染"问题。

**隔离方案**：为不同服务或方法分配独立的线程池，隔离慢服务对快服务的影响。

**实现方式**：

**方式一：多协议端口（彻底隔离）**
为核心服务单独开一个协议端口，配置独立的线程池：
```yaml
dubbo:
  protocols:
    dubbo-core:
      name: dubbo
      port: 20880
      threads: 200
    dubbo-slow:
      name: dubbo
      port: 20881
      threads: 50  # 慢服务用小线程池，防止无限扩张
```
```java
@DubboService(protocol = "dubbo-slow")
public class ReportServiceImpl implements ReportService { ... }

@DubboService(protocol = "dubbo-core")
public class OrderServiceImpl implements OrderService { ... }
```

**方式二：自定义 Filter + Executor（细粒度）**
```java
@Activate(group = PROVIDER, order = -10000)
public class ThreadIsolationFilter implements Filter {
    private static final Map<String, ExecutorService> POOL_MAP = new ConcurrentHashMap<>();

    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        String serviceKey = invoker.getInterface().getSimpleName();
        // 为慢服务单独分配线程池
        if ("ReportService".equals(serviceKey)) {
            ExecutorService pool = POOL_MAP.computeIfAbsent(serviceKey,
                k -> Executors.newFixedThreadPool(20));
            Future<Result> future = pool.submit(() -> invoker.invoke(invocation));
            try {
                return future.get(30, TimeUnit.SECONDS);
            } catch (Exception e) {
                throw new RpcException("Isolated pool execution failed", e);
            }
        }
        return invoker.invoke(invocation);
    }
}
```

**方式三：Dubbo 3.x 服务级线程池配置**
Dubbo 3.x 支持在 `@DubboService` 上直接指定 executor bean：
```java
@DubboService(executor = "slowServiceExecutor")
public class ReportServiceImpl implements ReportService { ... }

@Bean("slowServiceExecutor")
public Executor slowServiceExecutor() {
    return new ThreadPoolExecutor(10, 20, 60L, TimeUnit.SECONDS, new SynchronousQueue<>());
}
```

**隔离策略建议**：
- 核心交易链路（下单/支付）：200 线程，`eager` 线程池
- 报表/导出等慢任务：30~50 线程，限制并发防止 DB 压力过大
- 通知/消息推送：异步化，不占用 Dubbo 业务线程
