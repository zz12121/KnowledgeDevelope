###### 1. 如何监控 Dubbo 服务的运行状态？

监控 Dubbo 是多层次的工作，生产上一般组合使用：

**① Dubbo 内置 Monitor 模块**
`MonitorFilter` 拦截所有调用，收集调用次数、成功/失败数、耗时，定时聚合上报到监控中心。配置：`dubbo.monitor.protocol=registry`（从注册中心获取监控地址）。轻量但功能有限，现代项目主要靠 Metrics API 替代。

**② QoS 在线运维命令**
服务启动后，默认在 22222 端口开启 QoS（HTTP + Telnet）：
```bash
# Telnet方式
telnet localhost 22222
ls            # 列出所有服务
ps            # 查看服务端口
count com.xxx.UserService getUserById  # 实时调用次数和平均耗时
trace com.xxx.UserService getUserById  # 实时追踪某方法调用
```
这是线上应急排查的利器，无需重启服务。

**③ 第三方 APM（生产推荐）**
通过 Filter SPI 机制或 Java Agent 接入 SkyWalking、Zipkin。Agent 方式（如 SkyWalking）无需修改代码，字节码增强自动注入拦截逻辑。

**④ Spring Boot Actuator 健康检查**
自定义 `HealthIndicator`，检查注册中心连接、核心依赖服务是否可用，通过 `/actuator/health` 对外暴露。K8s 的 `readinessProbe` 可直接对接。

**⑤ Prometheus + Grafana（云原生标准）**
Dubbo 3.x 的 `dubbo-metrics-api` 基于 Micrometer，直接将 `rpc_requests_total`、`rpc_request_duration_seconds` 等指标暴露给 Prometheus，配合 Grafana 面板展示实时 QPS、P99 响应时间、错误率。

**关键源码**：`MonitorFilter.invoke()` 记录开始时间，调用结束后通过 `DubboMonitor.collect()` 累加统计；`DubboMonitor` 内置定时任务每分钟 `send()` 一次聚合数据。

###### 2. Dubbo 如何实现服务链路追踪？

链路追踪的核心是在分布式调用链中**传递唯一的 TraceID 和 SpanID**，Dubbo 通过两个机制实现：

**① RpcContext Attachment 传递**
`RpcContext` 是 `ThreadLocal` 的调用上下文，存储 `attachments: Map<String, Object>`。消费者在发起调用前将 TraceID 放入 Attachment，Dubbo 协议编码时（`DubboCodec.encodeRequestData()`）会把 Attachment 序列化进请求体；Provider 解码后从 `Invocation.getAttachments()` 取出，再放回本线程的 `RpcContext`。

**② Filter 机制注入/提取上下文**
```java
// 消费者端Filter：注入TraceID
if (RpcContext.getContext().isConsumerSide()) {
    String traceId = MDC.get("traceId"); // 从当前线程MDC取
    if (traceId != null) {
        invocation.setAttachment("traceId", traceId);
    }
}
// 提供者端Filter：提取TraceID并绑定到MDC
if (RpcContext.getContext().isProviderSide()) {
    String traceId = invocation.getAttachment("traceId");
    if (traceId != null) {
        MDC.put("traceId", traceId);
    }
}
```

**注意事项**：
- **异步线程丢失 TraceID 问题**：MDC 基于 `ThreadLocal`，提交异步任务时 TraceID 会丢失。需要包装 `Runnable`，在任务提交时拷贝 MDC，执行前恢复。
- **Filter 的 finally 块必须清理 MDC**，防止线程池复用时日志污染。

###### 3. Dubbo 与 Zipkin、SkyWalking 如何集成？

**与 Zipkin 集成（显式配置，有侵入性）**：

```xml
<dependency>
    <groupId>io.zipkin.brave</groupId>
    <artifactId>brave-instrumentation-dubbo</artifactId>
</dependency>
```
然后在 Dubbo 消费者和提供者配置中添加 Filter：
```yaml
dubbo:
  consumer:
    filter: tracing
  provider:
    filter: tracing
```
还需配置 `Tracing` Bean 指定 Zipkin 服务地址和采样率。Brave 的 `TracingFilter` 会在调用前后创建/结束 Span，并异步上报。

**与 SkyWalking 集成（无侵入，推荐）**：

```bash
java -javaagent:/path/to/skywalking-agent/skywalking-agent.jar \
     -Dskywalking.agent.service_name=your-service \
     -Dskywalking.collector.backend_service=localhost:11800 \
     -jar your-app.jar
```

SkyWalking Agent 通过字节码增强（Byte Buddy）自动拦截 Dubbo 关键类，无需修改任何配置或代码，且功能更全（指标监控、服务拓扑图、性能分析）。

**对比**：
- **Zipkin + Brave**：轻量，适合只需要链路追踪的场景；需要手动配置 Filter。
- **SkyWalking Agent**：无代码侵入，功能更强大，生产首选；需要部署 OAP 后端。

###### 4. Dubbo 的 Monitor 模块是做什么的？

Monitor 模块的职责是**度量**——回答"这个服务跑得怎么样"。

**核心架构**：
- `MonitorFilter`（数据入口）：拦截所有调用，记录耗时、成功/失败，调用 `Monitor.collect()` 上报单次数据。
- `DubboMonitor`（聚合器）：内部维护 `ConcurrentMap<Statistics, AtomicReference<long[]>>` 作为滑动时间窗口。每隔 1 分钟（`interval` 可配）执行 `send()` 方法，将过去一分钟的累计数据发送到监控服务器，然后重置计数器。
- 监控服务器（`dubbo-monitor-simple`或现代监控系统）：接收数据，计算 QPS = `count / interval`，提供可视化。

**现代演进**：Dubbo 3.x 的 Metrics API（基于 Micrometer）逐渐替代了 Monitor 模块，直接暴露标准 Counter/Gauge/Histogram 指标，更容易接入 Prometheus 生态，PromQL `rate(rpc_requests_total[1m])` 比 Monitor 的定时聚合更灵活。

###### 5. Dubbo 如何统计服务调用的 QPS？

**Monitor 模块统计**：

`MonitorFilter` 每次调用时原子地累加计数器（用 `AtomicReference<long[]>` + CAS 保证线程安全）：
```java
// long[0] = 调用次数，long[1] = 总耗时，long[2] = 失败次数
update[0] = current[0] + 1;  // 调用次数+1
update[1] = current[1] + duration;  // 累加耗时（用于计算平均值）
```

`DubboMonitor` 每分钟将 `count / 60`（秒）得出该窗口的平均 QPS，上报给监控服务。

**Prometheus 方式（推荐）**：
```yaml
dubbo:
  metrics:
    enabled: true
    protocol: prometheus
    port: 20888
```
Dubbo 3.x 会将 `rpc_requests_total`（Counter）暴露给 Prometheus，通过 PromQL 计算任意时间窗口的 QPS：
```promql
rate(dubbo_provider_requests_total[1m])   # 过去1分钟的QPS
```
这比 Monitor 的固定1分钟窗口更精确、更灵活。

###### 6. Dubbo 如何实现日志追踪？

**TraceID 贯穿全链路的标准做法**：

1. **在请求入口生成 TraceID**：在 Web 层 Filter 中生成 UUID 作为 TraceID，存入 `MDC.put("traceId", traceId)`，并设置到 `RpcContext.getContext().setAttachment("traceId", traceId)`。

2. **自定义 TraceFilter 跨服务传递**：
```java
@Activate(group = {PROVIDER, CONSUMER})
public class TraceFilter implements Filter {
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        if (RpcContext.getContext().isConsumerSide()) {
            // 消费者：将MDC中的traceId放入Attachment
            String traceId = MDC.get("traceId");
            if (traceId != null) invocation.setAttachment("traceId", traceId);
        } else {
            // 提供者：从Attachment取出traceId绑定到MDC
            String traceId = invocation.getAttachment("traceId");
            if (traceId != null) MDC.put("traceId", traceId);
        }
        try {
            return invoker.invoke(invocation);
        } finally {
            if (RpcContext.getContext().isProviderSide()) {
                MDC.remove("traceId");  // 必须清理，防止线程复用时污染
            }
        }
    }
}
```

3. **Logback 日志模板加入 TraceID**：
```xml
<pattern>%d{HH:mm:ss.SSS} [%thread] [%X{traceId}] %-5level %logger - %msg%n</pattern>
```

4. **SPI 注册**：`META-INF/dubbo/org.apache.dubbo.rpc.Filter` 文件中加入 `traceFilter=com.example.TraceFilter`，配置 `@Activate` 自动激活。

**使用 APM 更省力**：SkyWalking Agent 会自动完成 TraceID 生成、传递和 MDC 设置，日志模板里用 `%X{tid}` 即可，完全不需要手写 TraceFilter。

###### 7. TraceID 在异步线程中如何正确传递？（新增）

**问题**：MDC 和 `RpcContext` 都是基于 `ThreadLocal` 的。当业务代码中用 `ExecutorService` 提交异步任务时，子线程拿不到父线程的 TraceID，导致日志断链。

**解法一：包装 Runnable/Callable（手动）**
```java
public class TraceRunnable implements Runnable {
    private final Runnable delegate;
    private final Map<String, String> mdcContext;
    
    public TraceRunnable(Runnable delegate) {
        this.delegate = delegate;
        this.mdcContext = MDC.getCopyOfContextMap(); // 提交时拷贝父线程MDC
    }
    
    @Override
    public void run() {
        MDC.setContextMap(mdcContext != null ? mdcContext : Collections.emptyMap());
        try {
            delegate.run();
        } finally {
            MDC.clear();
        }
    }
}

// 使用
executor.submit(new TraceRunnable(() -> {
    // 子线程里能正常打出traceId
    log.info("async task running");
}));
```

**解法二：TransmittableThreadLocal（TTL，推荐）**
引入 Alibaba 的 `transmittable-thread-local` 库，替换 `ThreadLocal`，可以自动在线程切换时传递上下文，无需手动包装：
```xml
<dependency>
    <groupId>com.alibaba</groupId>
    <artifactId>transmittable-thread-local</artifactId>
</dependency>
```
搭配 `TtlExecutors.getTtlExecutorService(executor)` 包装线程池，所有提交到该线程池的任务都会自动继承父线程的 `TransmittableThreadLocal` 值。

**解法三：使用 SkyWalking Agent**
SkyWalking 内置了对异步线程的处理，通过字节码增强自动透传 TraceContext，基本上是最省心的方案。
