###### 1. Dubbo 的服务治理功能包括哪些？
Dubbo 服务治理覆盖分布式服务的完整生命周期：

1. **服务注册与发现**：Provider 自动注册，Consumer 动态发现，支持多种注册中心（Nacos/ZK/Consul）。
2. **负载均衡**：内置随机/轮询/最少活跃/一致性哈希/最短响应五种策略，支持自定义扩展。
3. **集群容错**：Failover/Failfast/Failsafe/Failback/Forking/Broadcast 六种容错策略。
4. **服务路由**：条件路由、标签路由，实现流量调度、灰度发布、蓝绿部署。
5. **动态配置**：运行时动态调整超时、权重、负载均衡策略等，无需重启。
6. **服务降级与 Mock**：非核心服务故障时返回兜底数据或本地 Mock 结果。
7. **服务限流与熔断**：通过集成 Sentinel 实现 QPS 控制、熔断降级。
8. **优雅上下线**：Provider 预热（warmup 参数）、延迟暴露（delay 参数）、优雅停机（DubboShutdownHook）。
9. **访问控制**：基于 IP 的黑白名单机制。
10. **服务监控**：调用次数、耗时、成功率等统计信息。
11. **服务分组与版本**：通过 group/version 实现环境隔离和多版本共存。

这些功能大部分通过 SPI 机制实现：`Router`、`ConfigCenter`、`MetadataReport` 等都是扩展点。

###### 2. Dubbo 如何实现服务降级和容错？
**服务降级**通过在消费者端配置 `mock` 实现：

```java
// 方式一：失败后执行 Mock 类（接口名 + Mock 后缀）
@DubboReference(mock = "true")
private DemoService demoService;
// 则需要有 DemoServiceMock implements DemoService

// 方式二：强制返回，不发起远程调用
@DubboReference(mock = "force:return null")
private DemoService demoService;

// 方式三：失败后返回固定值（执行后降级）
@DubboReference(mock = "fail:return null")
private DemoService demoService;
```

源码：`MockClusterWrapper` 包装原始 Invoker，`invoke()` 方法根据配置和调用结果决定是否执行 Mock 逻辑。

**集群容错**通过 `Cluster` SPI 实现。以 `FailoverClusterInvoker.doInvoke()` 为例：捕获 `RpcException` 后，根据 `retries` 参数选择其他 Provider 重试。`Failfast` 直接抛异常，`Failsafe` 捕获后记录日志并返回空结果。

###### 3. Dubbo 的服务分组和版本号有什么作用？
**分组（group）**：区分同一接口的不同实现。常用于：
- 环境隔离（test/staging/prod 环境不同实现）。
- 同一接口多个业务实现（如支付服务：微信支付组 vs 支付宝支付组）。
- Consumer 必须指定分组才能调用对应服务，不同分组完全隔离。

**版本号（version）**：用于接口不兼容升级时的多版本共存和灰度过渡。例如同时暴露 v1.0.0（老版本）和 v2.0.0（新版本），Consumer 按需引用指定版本，逐步迁移。

**源码角度**：版本和分组都会体现在服务 URL 中，`RegistryDirectory.notify()` 在更新 Invoker 列表时，根据 Consumer URL 中的 group 和 version 参数过滤 Provider 列表，保证 Consumer 只拿到匹配的服务实例。

**配置示例**：
```java
// Provider 暴露 v2.0.0
@DubboService(version = "2.0.0", group = "payService")
public class OrderServiceV2Impl implements OrderService {}

// Consumer 引用 v2.0.0
@DubboReference(version = "2.0.0", group = "payService")
private OrderService orderService;
```

###### 4. 服务上线怎么兼容旧版本？
采用**多版本并行 + 渐进式迁移**策略：

1. **接口设计兼容**：新接口只添加方法，不修改原有方法签名；DTO 用 Hessian/Protobuf 等兼容性好的序列化，添加字段时设默认值。

2. **同时部署新老版本**：新版本以新版本号（如 v2.0.0）部署，与老版本（v1.0.0）并存：
   ```java
   // 新版本 Provider
   @DubboService(version = "2.0.0")
   public class DemoServiceV2Impl implements DemoService {}
   ```

3. **消费者逐步迁移**：
   - 内部服务先升级 v2.0.0，验证稳定性。
   - 通过 Dubbo 标签路由，将特定流量（灰度用户/测试流量）路由到 v2.0.0。
   - 全部验证无误后，所有 Consumer 切换至 v2.0.0，下线 v1.0.0。

4. **版本回滚预案**：发现问题快速将 Consumer 版本号配置回 v1.0.0，老版本 Provider 仍在线，秒级回滚。

###### 5. Dubbo 的服务上下线如何优雅处理？
**优雅上线**：
- **延迟暴露**：`delay` 参数（毫秒）让服务在 Spring 容器初始化完成后延迟暴露，避免 Consumer 在 Provider 未完全就绪时调用。`ServiceConfig.export()` 中 `delay > 0` 时用定时器延迟执行。
- **服务预热**：`warmup` 参数设置预热时间（如 10 分钟），此期间 Provider 权重从1线性增长至设定值，流量缓慢增加，避免新节点瞬间被打满。

**优雅停机**（`DubboShutdownHook` 触发）：
1. **拒绝新请求**：关闭所有 `Protocol` 的 `Server`，不再接受新调用。
2. **等待进行中的请求完成**：通过线程池计数等待，超时时间默认 10 秒（`dubbo.service.shutdown.wait` 配置）。
3. **注销服务**：向注册中心注销，让 Consumer 感知并更新本地列表。
4. **释放资源**：关闭连接池、线程池等。

`DubboProtocol.destroy()` 遍历 `serverMap`，`ExchangeServer` 关闭时先停止新请求，再等待 `executor` 任务完成。

###### 6. 怎么实现动态感知服务下线的？
通过**注册中心的 Watcher/Listener 机制**实现，以 ZooKeeper 为例完整流程：

1. Provider 启动时在 `/dubbo/{service}/providers` 下创建**临时节点**。
2. Consumer 订阅该路径，注册 `ChildListener`。
3. Provider 正常下线或进程崩溃（会话断开），ZooKeeper 自动删除其临时节点。
4. ZooKeeper 触发 `ChildListener.childChanged()` 回调，`ZookeeperRegistry` 收到变更事件。
5. 调用 `RegistryDirectory.notify(List<URL> urls)` 更新本地 `methodInvokerMap`，移除已下线的 `Invoker`。
6. 后续调用路由时，该节点已从可用列表中消失，不会再被选中。

Nacos 的机制类似：Provider 注册实例时 Nacos Server 主动心跳检测，心跳超时后摘除实例，通过长轮询推送变更通知给 Consumer。

源码：`ZookeeperRegistry.doSubscribe()` 注册 `ChildListener`，`childChanged()` 最终调用 `notify()`，`RegistryDirectory.notify()` 重建 `Invoker` 列表并刷新 `RouterChain`。

###### 7. Dubbo 如何优雅停机？
优雅停机核心是**先停流量，再等完成，最后释放资源**，由 `DubboShutdownHook` 触发：

```
收到 SIGTERM/SIGINT
    ↓
关闭 Server 拒绝新请求
    ↓
从注册中心注销（Consumer 感知，停止路由到本节点）
    ↓
等待进行中的请求完成（最长 dubbo.service.shutdown.wait 时间）
    ↓
关闭线程池、连接池
    ↓
JVM 退出
```

**关键配置**：
```yaml
dubbo:
  service:
    shutdown:
      wait: 10000   # 等待进行中请求完成的最大时间，默认 10秒
```

**踩坑点**：K8s 环境下，需配合 `preStop` 钩子，先等待 readinessProbe 失败（让 Service 不再路由新流量），再等一段时间（如5秒）后才真正发 SIGTERM，给 Consumer 端感知下线的时间窗口：
```yaml
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 5"]
```

###### 8. Dubbo 的管理控制台（Admin）能做什么？
Dubbo Admin 提供可视化服务治理：

1. **服务查询**：查看所有 Provider/Consumer 列表，含 IP、端口、版本、分组等详情。
2. **服务测试**：手动发起调用，支持参数输入和结果查看（非常便于联调排查）。
3. **动态配置下发**：
   - 调整 Provider 权重，实时影响负载均衡。
   - 动态修改超时、重试次数等参数。
   - 配置条件路由和标签路由规则。
   - 配置动态 Mock 规则。
4. **服务治理操作**：强制下线 Provider，启用/禁用服务，配置 IP 黑白名单。
5. **监控统计**：查看服务 QPS、平均响应时间、成功率等。
6. **依赖拓扑**：可视化展示服务间调用依赖图，帮助分析影响范围。

控制台通过读取注册中心数据获取服务信息，通过操作配置中心下发动态配置。

###### 9. Dubbo 如何实现服务限流？
→ 限流算法通用原理详见 [[33_DistributedKnowledge/04_分布式服务治理]]

Dubbo 本身未内置完整限流，主要有三种方式：

**1. 集成 Sentinel（推荐）**：
```xml
<dependency>
    <groupId>org.apache.dubbo</groupId>
    <artifactId>dubbo-sentinel-support</artifactId>
</dependency>
```
引入依赖后，Sentinel 的 `DubboFilter` 通过 SPI 自动注入调用链。在 Sentinel 控制台配置 QPS/线程数规则，`before()` 方法中进行流控检查，超限时抛出 `BlockException`。

**2. 内置 TpsLimitFilter**：Dubbo 自带简单 TPS 限流，通过 `tps` 参数配置：
```xml
<dubbo:service tps="100" tpsInterval="1000" />
```
功能简单，只支持固定窗口计数，生产环境建议用 Sentinel。

**3. 自定义 Filter**：实现 `Filter` 接口，在 `invoke()` 中加令牌桶/滑动窗口逻辑，通过 SPI 注册后在服务上启用：
```java
@Activate(group = Constants.PROVIDER)
public class CustomRateLimitFilter implements Filter {
    private final RateLimiter limiter = RateLimiter.create(100); // 100 QPS
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        if (!limiter.tryAcquire()) {
            throw new RpcException("Rate limit exceeded");
        }
        return invoker.invoke(invocation);
    }
}
```

###### 10. Dubbo 如何实现服务熔断？
→ 熔断器三态（CLOSED/OPEN/HALF-OPEN）原理详见 [[33_DistributedKnowledge/04_分布式服务治理]]

Dubbo 原生无熔断器，主要通过集成 Sentinel 实现：

**集成 Sentinel 熔断**：
在 Sentinel 控制台针对 Dubbo 服务配置熔断降级规则（慢调用比例/异常比例/异常数），当触发时 `SentinelDubboConsumerFilter` 抛出 `BlockException`，配合 Dubbo Mock 返回降级数据：

```java
@DubboReference(mock = "fail:return null")
private DemoService demoService;
```

**熔断规则配置（代码方式）**：
```java
DegradeRule rule = new DegradeRule("com.example.DemoService");
rule.setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType());  // 按异常比例
rule.setCount(0.5);   // 50% 异常率触发熔断
rule.setTimeWindow(10); // 熔断持续10秒
DegradeRuleManager.loadRules(Collections.singletonList(rule));
```

Sentinel 熔断器实现了标准的三态状态机（关闭→打开→半开），比手动在 Filter 中统计失败率可靠和完善得多，生产环境直接用 Sentinel。

###### 11. Dubbo 的服务路由规则有哪些？
Dubbo 支持两种主要路由规则：

**1. 条件路由**：通过条件表达式匹配消费者和提供者，格式 `[consumer条件] => [provider条件]`：
```
# 消费者IP 10.20.153.10 的请求只调用 IP 10.20.153.11 的 Provider
host = 10.20.153.10 => host = 10.20.153.11

# find* 开头的方法只调用指定两个 Provider
method = find* => host = 10.20.153.10,10.20.153.11

# 排除指定 Provider（容量小的机器）
=> host != 10.20.153.11
```

**2. 标签路由（推荐，用于灰度发布）**：为 Provider 打标签，Consumer 请求携带对应标签路由到匹配节点：
```java
// 在请求入口（网关 Filter）携带灰度标签
RpcContext.getContext().setAttachment("dubbo.tag", "gray");
```
`TagRouter` 根据标签选择匹配的 Provider；Consumer 无标签则选无标签 Provider，实现流量隔离。

路由规则存储在注册中心（`/dubbo/{service}/routers`）或配置中心，在 `RegistryDirectory.notify()` 中加载，构建 `RouterChain`。调用时 `AbstractClusterInvoker` 调用 `RouterChain.route()` 过滤 Invoker 列表。

###### 12. Dubbo 如何实现灰度发布？
利用**标签路由**，完整流程：

1. **部署灰度版本**：新版本 Provider 启动时打灰度标签（`tag=gray`），旧版本无标签或 `tag=stable`。
   ```yaml
   # dubbo.properties 或启动参数
   dubbo.provider.tag=gray
   ```

2. **配置路由规则**（通过 Admin 或配置中心下发）：指定哪些消费者请求携带灰度标签（可按 IP/用户 ID/比例）。

3. **网关/入口携带标签**：灰度流量进入时在 Filter 中设置：
   ```java
   // 灰度用户携带 gray 标签
   RpcContext.getContext().setAttachment("dubbo.tag", "gray");
   ```

4. **路由匹配**：`TagRouter` 将带 `gray` 标签的请求路由到打了 `gray` 标签的 Provider，无标签请求路由到 `stable` Provider，两条流量完全隔离。

5. **逐步全量**：灰度验证通过后，扩大灰度范围，最终全量升级，移除灰度标签，下线旧版本。

**多级调用链灰度**：灰度标签会通过 Attachment 在调用链上传递，下游服务也能感知并路由到对应版本，整条调用链都灰度化。

###### 13. Dubbo 服务自省是什么？
服务自省（Service Introspection）是 Dubbo 2.7 引入、Dubbo 3 全面升级的**新服务发现模型**，解决传统接口级服务发现的注册中心压力问题。

**传统接口级服务发现的问题**：Provider 将所有接口的完整 URL 元数据发布到注册中心，一个应用有100个接口就存100条记录，数据量大，注册中心压力大，在 K8s 等大规模场景下不可持续。

**应用级服务发现（服务自省）**：
- 注册中心只存**应用实例**基本信息（IP+端口+应用名），轻量。
- 详细的接口元数据推送到独立的**元数据中心**（如 Nacos/ZK 元数据存储）。
- Consumer 先从注册中心获取应用实例，再从元数据中心获取接口信息。

```properties
dubbo.metadata-report.address=nacos://127.0.0.1:8848
```

**优势**：
- 注册中心数据量减少 ~100 倍，可承载更大规模服务。
- 与 K8s Service 原生对齐（K8s Service 也是应用粒度），云原生友好。
- 元数据中心独立部署，扩展性更好。

这也是 Dubbo 3 应用级服务发现的基础，兼容接口级模式（可通过 `dubbo.registry.simplified` 配置切换）。

###### 14. 如何通过标签路由实现流量治理实战？
以大促流量保障场景为例，完整的标签路由实战：

**场景**：大促期间把 VIP 用户流量路由到高配机器，普通用户路由到普通机器。

**步骤1：Provider 集群打标签**：
```yaml
# 高配机器 dubbo.properties
dubbo.provider.tag=vip

# 普通机器  
dubbo.provider.tag=normal
```

**步骤2：网关入口 Filter 注入标签**：
```java
@Component
public class TagInjectFilter implements Filter {
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        // 从请求上下文获取用户等级
        String userLevel = UserContext.getUserLevel();
        if ("VIP".equals(userLevel)) {
            RpcContext.getContext().setAttachment("dubbo.tag", "vip");
        } else {
            RpcContext.getContext().setAttachment("dubbo.tag", "normal");
        }
        return invoker.invoke(invocation);
    }
}
```

**步骤3：配置 TagRouter 路由规则**（通过 Admin 下发）：
```yaml
configVersion: v3.0
force: false  # 无匹配时是否降级到无标签节点
enabled: true
key: demo-provider
tags:
  - name: vip
    addresses: ["192.168.1.10:20880", "192.168.1.11:20880"]
  - name: normal
    addresses: ["192.168.1.20:20880", "192.168.1.21:20880"]
```

**步骤4：`force=false` 的降级保障**：如果 VIP 标签节点全部不可用，`TagRouter` 会降级到无标签或其他标签的节点，保证可用性（`force=true` 则严格匹配，无则报错）。

这种方案还可以扩展到**流量隔离**（测试流量 → 测试环境 Provider）、**灰度发布**（新版本 → gray 标签 Provider）等场景，是 Dubbo 流量治理的核心手段。
