###### 1. 说几个 Dubbo 核心的配置
Dubbo 核心配置围绕五个维度展开，对应五大配置类：

**应用级别**：
- `ApplicationConfig`：应用名（`name`，用于监控和链路追踪）、`qosEnable`/`qosPort`（22222，运维命令端口）。

**注册中心**：
- `RegistryConfig`：`address`（如 `nacos://127.0.0.1:8848`）、`timeout`（默认5000ms）、`check`（启动时检查注册中心可用性）。

**协议**：
- `ProtocolConfig`：`name`（默认 `dubbo`）、`port`（默认 20880）、`threads`（业务线程池大小，默认200）、`serialization`（序列化方式，默认 `hessian2`）。

**Provider 侧**：
- `ProviderConfig`：`timeout`（默认1000ms）、`retries`（默认2，不含第一次）、`loadbalance`（默认 `random`）、`cluster`（默认 `failover`）。
- `ServiceConfig`：`interface`、`ref`（实现 Bean）、`version`、`group`、`delay`（延迟暴露，-1表示Spring初始化完成后）。

**Consumer 侧**：
- `ConsumerConfig`：`timeout`、`retries`、`loadbalance`、`check`（默认 `true`，启动时检查依赖服务）。
- `ReferenceConfig`：`interface`、`version`、`group`、`url`（直连地址）、`check`、`timeout`、`retries`。

**方法级别**（最细粒度）：
- `MethodConfig`：`name`（方法名）、`timeout`、`retries`、`loadbalance`、`actives`（Consumer 最大并发调用数）。

###### 2. Dubbo 有哪几种配置方式？
**1. XML 配置（传统 Spring 方式）**：
```xml
<dubbo:application name="demo-provider"/>
<dubbo:registry address="nacos://127.0.0.1:8848"/>
<dubbo:protocol name="dubbo" port="20880"/>
<dubbo:service interface="com.example.DemoService" ref="demoService"/>
```

**2. 注解配置（Spring Boot 主流）**：
```java
@DubboService(version = "1.0.0", timeout = 3000)
public class DemoServiceImpl implements DemoService {}

@DubboReference(version = "1.0.0", retries = 2)
private DemoService demoService;
```

**3. 属性文件配置（Spring Boot application.yml）**：
```yaml
dubbo:
  application:
    name: demo-provider
  registry:
    address: nacos://127.0.0.1:8848
  protocol:
    name: dubbo
    port: 20880
  scan:
    base-packages: com.example
```
最简洁，与 Spring Boot 自动装配配合最佳，是新项目首选。

**4. API 编程配置**（无 Spring 容器或动态生成服务场景）：
```java
ServiceConfig<DemoService> service = new ServiceConfig<>();
service.setApplication(new ApplicationConfig("demo-provider"));
service.setRegistry(new RegistryConfig("nacos://127.0.0.1:8848"));
service.setInterface(DemoService.class);
service.setRef(new DemoServiceImpl());
service.export();
```

**5. 外部化配置（配置中心）**：从 Nacos/Apollo/ZK 拉取配置，支持动态推送刷新，优先级高于本地配置文件。

实际项目通常混用：属性文件定义全局参数，注解定义服务暴露和引用，配置中心管理动态参数。

###### 3. 说说 Dubbo 的优先级配置
Dubbo 配置优先级从高到低：

```
JVM -D 参数
    ↓
配置中心（Nacos/Apollo/ZK）
    ↓
本地代码/配置文件（API/YAML/XML/注解）
    ↓
Dubbo SPI 内置默认值
```

**同源内部的细粒度优先级**（细粒度覆盖粗粒度）：
```
方法级（MethodConfig）
    > 接口级（ServiceConfig / ReferenceConfig）
    > 全局级（ProviderConfig / ConsumerConfig）
```

**示例**：如果全局设 `timeout=1000ms`，某服务设 `timeout=3000ms`，某方法设 `timeout=5000ms`，最终该方法调用超时时间是 5000ms。

**源码实现**：配置合并在 `AbstractConfig.refresh()` 方法中实现，从各配置源收集参数并按优先级合并，最终汇聚到 `URL` 对象中（`ServiceConfig.export()` 或 `ReferenceConfig.get()` 触发）。URL 中的参数就是最终生效的配置。

###### 4. 如何禁用某个服务的启动检查？
禁用启动检查主要是消费者端，防止依赖服务未启动导致自身启动失败：

**全局禁用**：
```yaml
dubbo:
  consumer:
    check: false
```

**针对特定服务引用**：
```java
@DubboReference(check = false)
private DemoService demoService;
```

**禁用注册中心连通性检查**：
```yaml
dubbo:
  registry:
    check: false
```

**原理**：`ReferenceConfig.init()` → `createProxy()` → `RegistryProtocol.refer()` 会检查注册中心可用性和 Provider 列表是否为空。`check=false` 时跳过这些检查，Consumer 正常启动（但调用时可能失败）。适用于服务循环依赖、开发调试等场景。

**注意**：生产环境不建议全局禁用，可能掩盖真实的服务依赖问题。建议针对具体非关键服务禁用，关键依赖保持 `check=true`。

###### 5. Dubbo 中有哪些调优参数？
调优参数按层面分类：

**线程池**：
```yaml
dubbo:
  protocol:
    threads: 200        # 业务线程池大小，I/O密集可适当调大（如500）
    threadpool: fixed   # fixed（默认）/cached/limited
    accepts: 1000       # Provider 最大接受连接数，0不限制
```

**超时与重试**：
```yaml
dubbo:
  provider:
    timeout: 3000      # 超时时间（ms），按服务实际耗时P99设定
    retries: 2         # 写操作必须设为0！读操作1-2次
```

**连接**：
```yaml
dubbo:
  protocol:
    connections: 1     # 每服务对每 Provider 的长连接数，默认1；高吞吐可调为2-5
    heartbeat: 60000   # 心跳间隔（ms），空闲场景可调大减少心跳包
```

**并发控制**：
```yaml
# Provider 端服务限流（服务器保护自身）
dubbo:
  provider:
    executes: 200      # 每服务最大并行执行数

# Consumer 端限流（防止压垮服务端）
dubbo:
  consumer:
    actives: 50        # 每服务消费者最大并发调用数
```

**序列化**：
```yaml
dubbo:
  protocol:
    serialization: kryo  # 切换高性能序列化，需添加依赖
```

**JVM 层面**：堆内存、GC 算法（G1/ZGC）、GC 暂停时间调优，减少 Full GC 对 Dubbo 线程的影响，避免超时。

调优原则：先监控定位瓶颈（QoS 端口、Dubbo Admin、SkyWalking），再针对性调优，没有万能参数。

###### 6. Dubbo 配置的加载顺序是怎样的？
Dubbo 配置完整加载顺序（从最先到最后）：

```
1. JVM 系统参数（-Ddubbo.xxx）           最高优先级
2. 配置中心（Nacos/Apollo）推送的配置     支持动态刷新
3. Spring 环境变量（环境变量/命令行参数）
4. application.properties/yml 文件
5. Dubbo XML/注解/API 配置
6. Dubbo SPI 内置默认值                  最低优先级
```

**配置合并过程**：在 `ServiceConfig.export()` 或 `ReferenceConfig.get()` 时，调用 `checkAndUpdateSubConfigs()` 方法，从各配置源收集参数，按优先级合并，最终生成一个完整的 `URL` 对象。URL 中的参数就是最终生效的配置。

**重要提示（Dubbo 2.7+）**：推荐将**动态配置**（超时、权重、路由规则）放在配置中心，将**服务发现相关**（注册中心地址）放在注册中心，各司其职，层次清晰。

###### 7. Dubbo 如何实现配置中心？
→ Nacos 配置中心深入原理详见 [[27_SpringCloudKnowledge/05_配置中心与消息总线/01、配置中心与消息总线]]

Dubbo 配置中心通过 `ConfigCenter` SPI 接口实现，支持：

**支持的实现**：Nacos（主流）、ZooKeeper（ZNode 存配置+Watcher监听）、Apollo（携程开源）、本地文件（测试）。

**核心功能**：
- 应用级配置（`dubbo.properties` 全局参数）。
- 服务级配置（针对特定接口的超时、重试规则）。
- 动态推送：配置变更实时推送到客户端，下次调用立即生效，无需重启。

**配置方式**：
```yaml
dubbo:
  config-center:
    address: nacos://127.0.0.1:8848
    namespace: dubbo-prod
    group: DEFAULT_GROUP
```

**工作原理**：
1. 应用启动时连接配置中心，订阅预定义配置节点。
2. 配置变更时配置中心推送变更，Dubbo 客户端调用 `ConfigurationListener.process()` 更新内部配置（如更新 `timeout` 值）。
3. 配置更新实时生效，下一次调用使用新配置。

**配置中心 vs 注册中心的区别**：
- **注册中心**：存储易失的、动态的服务实例地址（Provider IP/Port）。
- **配置中心**：存储持久化的、稳定的配置信息（超时/权重/路由规则）。

###### 8. 配置优先级完整链路解析
一个 Dubbo 服务调用的最终配置，经历如下合并路径（以 `timeout` 为例）：

```
方法级 timeout（@DubboReference methods={@Method(timeout=5000)}）
    ↓ 若未设置
接口级 timeout（@DubboReference(timeout=3000)）
    ↓ 若未设置
Consumer 全局 timeout（dubbo.consumer.timeout=2000）
    ↓ 若未设置
Provider 方法级 timeout（@DubboService methods={@Method(timeout=4000)}）
    ↓ 若未设置
Provider 接口级 timeout（@DubboService(timeout=3000)）
    ↓ 若未设置
Provider 全局 timeout（dubbo.provider.timeout=2000）
    ↓ 若未设置
Dubbo 默认值（1000ms）
```

**Consumer 侧配置优先于 Provider 侧**：Consumer 离调用更近，它的超时设置直接决定本次调用会等多久。

**动态配置覆盖**：配置中心下发的动态配置可以在运行时覆盖上述所有本地配置（优先级在 JVM -D 参数之下，代码/文件配置之上），这正是动态配置的价值——不重启就能调整超时和权重。

###### 9. 多环境配置隔离如何实现？
→ Apollo/Nacos 多环境隔离详细方案见 [[27_SpringCloudKnowledge/05_配置中心与消息总线/01、配置中心与消息总线]]

Dubbo 视角下的多环境隔离方案：

**方案一：注册中心 Namespace 隔离（Nacos）**：
```yaml
# dev 环境
dubbo:
  registry:
    address: nacos://127.0.0.1:8848?namespace=dev

# prod 环境
dubbo:
  registry:
    address: nacos://127.0.0.1:8848?namespace=prod
```
不同 Namespace 完全隔离，dev 的 Provider 不会被 prod 的 Consumer 发现。

**方案二：服务 group 隔离**：
```java
// dev 环境 Provider
@DubboService(group = "dev")
public class DemoServiceImpl {}

// dev 环境 Consumer
@DubboReference(group = "dev")
private DemoService demoService;
```
group 方式粒度到接口级别，灵活但维护成本略高。

**方案三：注册中心地址隔离**：dev/test/prod 各自独立的注册中心集群（成本高但最彻底）。

生产推荐：**Nacos Namespace 隔离**，一个 Nacos 集群服务多套环境，通过 Namespace 完全隔离，既节省基础设施成本，又保证环境安全。
