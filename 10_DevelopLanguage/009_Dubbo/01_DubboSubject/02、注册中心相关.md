###### 1. Dubbo 的注册中心有哪些实现方式？各有什么优缺点？
Dubbo 通过 `RegistryFactory` SPI 接口支持多种注册中心，常见实现：

**ZooKeeper（早期默认推荐）**
- **优点**：CP 模型，强一致性（ZAB 协议），临时节点天然实现健康检测——Provider 会话断开节点自动删除，Consumer 立即感知；目录结构清晰便于监控。
- **缺点**：写操作需集群过半节点确认，性能低于 AP 型；部署运维相对复杂；会话超时设置不当可能导致服务被误剔除。
- **源码**：`ZookeeperRegistry.doSubscribe()` 为 `/dubbo/{service}/providers` 路径注册 `ChildListener`，子节点变化时触发 `notify()` 更新本地 `Invoker` 列表。

**Nacos（当前主流推荐）**
- **优点**：默认 AP（Distro 协议），可切 CP（Raft），高可用性能好；内置 TCP/HTTP 健康检查；同时支持配置中心，一套基础设施解决两类问题；云原生友好，UI 完善。
- **缺点**：CP 模式下性能会下降；客户端 SDK 相对较重。
- **源码**：`NacosRegistry` 调用 `NamingService.registerInstance()` 注册，`subscribe()` 订阅，通过 `EventListener` 接收变更通知。

**Redis**
- **优点**：部署简单，利用 Key 过期特性实现 TTL 心跳。
- **缺点**：主从复制异步，可能有短暂数据不一致窗口；Pub/Sub 连接断开有消息丢失风险；可靠性不如专用组件。
- **源码**：`RedisRegistry` 后台有守护线程（`expireExecutor`）定期 `EXPIRE` 续期，用 `PSUBSCRIBE` 监听键空间事件。

**Consul/etcd**：CP 型，基于 Raft 协议，强一致性。etcd 在 K8s 生态地位重要；Consul 额外提供多数据中心和 HTTP 健康检查特性。

###### 2. 注册中心宕机了，有什么影响？
分**两个角色**、**两个阶段**分析：

**对 Provider**：
- 启动时宕机 → 无法注册 → Consumer 发现不了该实例，此 Provider 不可用。
- 运行期宕机 → 已运行的 Provider 不受影响，继续提供服务，但新 Provider 无法注册。

**对 Consumer**：
- 启动时宕机 → 无法拉取服务列表，默认会抛异常（可配置 `check=false` 跳过）。
- 运行期宕机 → **影响有限**。因为 Consumer 已将服务列表缓存在本地 `RegistryDirectory` 的 `methodInvokerMap` 中，已建立连接的调用**继续正常进行**。但无法感知新 Provider 上线或已有 Provider 下线（宕机）。

源码关键点：`RegistryDirectory.notify()` 负责更新本地缓存。注册中心断开时，`ZookeeperRegistry` 不会主动清空缓存，调用链 `ClusterInvoker → Directory → 本地缓存 Invoker` 仍能正常工作。

###### 3. 注册中心挂了，Consumer 还能调用 Provider 吗？挂 30 秒呢？挂一天呢？
**挂 30 秒**：完全可以正常调用。Consumer 使用本地缓存的服务列表，Provider 列表无变化，完全透明。注册中心恢复后会自动重连并接收通知。

**挂一天**：已建立连接的调用通常能继续，但整体健壮性下降，取决于 Provider 集群变化情况：
1. **所有 Provider 一直健康**：调用持续正常，就像没有注册中心一样，能撑很久。
2. **有新 Provider 上线**：Consumer 无法感知，新节点无流量，负载不均。
3. **有 Provider 宕机**（最严重）：Consumer 无法及时剔除故障节点，请求打到宕机节点会失败。此时依赖 Failover 集群容错重试其他节点——只要集群还有存活节点，最终能成功，但会有失败和延迟。若全部 Provider 宕机，服务完全不可用。

本质上：注册中心挂机时间越长，Consumer 本地缓存与真实集群状态**偏差越大**，但并非立即完全不可用。

###### 4. 如何设计一个注册中心？
设计一个适用于 Dubbo 的注册中心，需考虑以下核心要素：

**功能模型**：
1. **服务注册表**：高可用分布式数据库，存储 `<服务名 → 实例地址列表>` 映射，实例包含协议、版本、权重等元数据。
2. **服务注册**：Provider 启动时登记自身信息，支持**心跳续约**，会话断开或超时自动删除（临时性注册）。
3. **服务发现**：Consumer 通过拉取（Pull）或订阅推送（Push）获取服务实例列表。
4. **健康检查**：主动探活或被动心跳判断 Provider 健康状态，及时摘除不健康实例。
5. **变更通知**：实例列表变化时，实时可靠地通知所有订阅的 Consumer 更新本地缓存。

**架构设计**：
- **高可用集群**：多副本部署，避免单点故障，采用主从或 Raft 多副本架构。
- **一致性模型选择**：CP（ZooKeeper/etcd）保证强一致性，AP（Nacos/Eureka）保证高可用；Dubbo 的 Registry 抽象能适配两者。
- **缓存持久化**：Client 将最后一次成功的服务列表持久化到本地文件，注册中心不可用时降级使用。

**与 Dubbo 集成**：
- 实现 `Registry` 和 `RegistryFactory` SPI 接口，核心是实现 `register/unregister/subscribe/unsubscribe` 方法。
- 在 `subscribe` 中将原生变更事件转换为 `NotifyListener.notify()` 回调，以更新 `RegistryDirectory`。
- 继承 `FailbackRegistry` 获得开箱即用的失败重试模板。

###### 5. Dubbo 支持哪些注册中心？推荐使用哪个？
支持：**ZooKeeper、Nacos、Redis、Multicast（测试用）、Consul、etcd、SOFARegistry** 等。

**当前最推荐 Nacos**，理由：
1. **功能融合**：服务注册发现 + 动态配置中心，一套基础设施搞定两件事。
2. **模型灵活**：默认 AP 高可用，需要时可切 CP，适合大多数场景。
3. **云原生**：与 Spring Cloud Alibaba、K8s、Dubbo 深度集成，维护活跃。
4. **UI 完善**：管理控制台功能全，从 ZooKeeper 迁移成本低。

ZooKeeper 在需要强一致性的特定场景仍可考虑，但运维复杂性和性能瓶颈使其在新项目中不再是首选。

###### 6. ZooKeeper 在 Dubbo 中扮演什么角色？
ZooKeeper 在 Dubbo 中扮演**服务注册与发现协调者**和**元数据存储中心**的角色：

1. **服务地址注册**：Provider 启动时在 `/dubbo/{serviceName}/providers` 下创建**临时子节点**，节点值为 Provider URL（如 `dubbo://192.168.1.1:20880/...`）。
2. **服务发现与订阅**：Consumer 订阅 `/dubbo/{serviceName}/providers` 父节点，通过 Watcher 监听子节点变化。
3. **动态感知**：Provider 下线（正常退出或异常宕机会话断开），ZooKeeper 自动删除临时节点，Watcher 触发通知 Consumer 更新本地 Invoker 列表。
4. **配置存储（可选）**：早期用于存储路由规则、权重覆盖等动态配置，现多由独立配置中心承担。

源码角度：`ZookeeperRegistry.doSubscribe()` 添加 `ChildListener`，`childChanged()` 回调最终调用 `RegistryDirectory.notify()`，完成整个服务列表更新。**临时节点机制**是实现服务自动上下线的核心。

###### 7. Dubbo 可以不使用注册中心吗？
可以，这种方式叫**点对点直连**。

**配置方式**：在 `ReferenceConfig` 中直接指定 Provider 地址，绕过注册中心：
```xml
<dubbo:reference id="demoService"
    interface="com.example.DemoService"
    url="dubbo://192.168.1.1:20880" />
```
或多地址：`url="dubbo://192.168.1.1:20880,dubbo://192.168.1.2:20880"`

**原理**：Dubbo 直接解析 URL 创建连接，`RegistryDirectory` 中持有这些固定地址的 Invoker，没有注册订阅过程。

**适用场景**：本地开发调试、POC 验证、服务地址极少变动的点对点场景。

**局限性**：完全丧失动态服务治理能力。Provider 地址变更（扩容/缩容/迁移/故障）时必须手动修改所有 Consumer 配置并重启，无法自动发现和容错。生产分布式系统**强烈不推荐**直连，注册中心是必要组件。

###### 8. Nacos 和 ZooKeeper 作为注册中心有什么核心区别？
→ 深入对比详见 [[27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现原理]]

这里给出 Dubbo 视角下的核心对比：

| 维度 | ZooKeeper | Nacos |
|---|---|---|
| **一致性模型** | CP（ZAB 协议） | 默认 AP（Distro），可切 CP（Raft） |
| **健康检测** | 临时节点自动删除（被动） | 主动探活 + 心跳（TCP/HTTP/MySQL） |
| **性能** | 写操作需半数确认，偏慢 | AP 模式性能更高 |
| **附加功能** | 仅协调服务，无配置中心 | 兼具配置中心、元数据管理 |
| **云原生** | 一般 | 优秀，K8s 集成好 |
| **推荐场景** | 强一致性要求（如选举、分布式锁） | 绝大多数微服务注册发现场景 |

Nacos 的 AP 模式意味着在网络分区时优先保证可用性（Consumer 还能调到服务），ZooKeeper 的 CP 模式则优先保证一致性（宁可不可用，也不给出不一致的地址），在微服务注册发现场景中，AP 更合适。
