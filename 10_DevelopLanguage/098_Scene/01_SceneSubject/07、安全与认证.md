# 七、安全与认证

###### 1. 你项目用的 JWT，那如何保证安全性？别人拿到你这个 Token 不就可以直接登录你的账号了吗？怎么办？

这道题问的是 JWT 的**安全加固**方案。

**JWT 的先天问题：** JWT 是无状态的，只要 Token 有效，持有者就能访问。Token 被窃取确实是风险，但有多种防御手段：

**① 短有效期 + Refresh Token（核心方案）**

```
Access Token：有效期 15 分钟（短），存于内存
Refresh Token：有效期 7 天（长），存于 HttpOnly Cookie（防 XSS 读取）

客户端 → API：携带 Access Token
Access Token 过期 → 客户端自动用 Refresh Token 换新的 Access Token
Refresh Token 过期 → 用户重新登录
```

即使 Access Token 被窃取，攻击者最多有 15 分钟窗口，危害大幅降低。

**② HTTPS（传输层防窃取）**

所有接口强制走 HTTPS，防止中间人攻击在传输途中窃取 Token。不用 HTTP 的项目，Token 安全性根本无从谈起。

**③ Token 黑名单（实现主动吊销）**

JWT 无状态的缺点是服务端无法主动让 Token 失效（如用户退出登录、改密码后旧 Token 依然有效）。解决方案：

```java
// 退出登录时，将该 Token 的 jti（JWT ID）加入 Redis 黑名单
public void logout(String token) {
    Claims claims = parseToken(token);
    String jti = claims.getId();
    long remainingTtl = claims.getExpiration().getTime() - System.currentTimeMillis();
    // 黑名单只需存到 Token 自然过期为止
    redisTemplate.opsForValue().set("jwt:blacklist:" + jti, "1", 
                                    remainingTtl, TimeUnit.MILLISECONDS);
}

// 验证 Token 时检查黑名单
public boolean isTokenValid(String token) {
    String jti = parseToken(token).getId();
    return !Boolean.TRUE.equals(redisTemplate.hasKey("jwt:blacklist:" + jti));
}
```

**④ 绑定请求上下文（防 Token 被其他设备使用）**

在 Token 中加入发行时的 IP、User-Agent 指纹，每次请求校验是否匹配。但 IP 可能变化（移动网络），可以适当放宽，只做异常检测（而非严格校验）。

**⑤ Refresh Token 轮换（Rotation）**

每次使用 Refresh Token 时，颁发新的 Refresh Token，旧的立即作废。如果攻击者使用旧的 Refresh Token，服务端检测到令牌已轮换，立即吊销所有相关 Token，并告警。

###### 2. 把单体项目进行了多机部署，多台服务器是如何共享用户登录信息的？

单体项目多实例部署后，Session 共享是必须解决的问题。

**方案一：Session 集中存储（Redis）——推荐**

```java
// Spring Session + Redis，一行配置搞定
@EnableRedisHttpSession
public class SessionConfig {}
```

```yaml
spring:
  session:
    store-type: redis
    timeout: 30m
  redis:
    host: redis-server
    port: 6379
```

用户登录后，Session 不存在本地内存里，而是存在 Redis 中。所有实例连接同一个 Redis，任意节点都能读到 Session。

**方案二：JWT（无状态，不需要 Session 共享）**

彻底放弃服务端 Session，改用 JWT Token。登录成功后颁发 JWT，之后每次请求携带 JWT，服务端从 Token 中解析用户信息，不需要 Session 存储。

适合前后端分离架构，但需要处理 Token 吊销问题（见上题）。

**方案三：Session 粘性（Sticky Session）**

```nginx
upstream backend {
    ip_hash;  # 同一 IP 的请求始终路由到同一台服务器
    server app1:8080;
    server app2:8080;
}
```

Nginx 用 `ip_hash` 将同一用户的请求路由到同一台服务器，不需要共享 Session。缺点：某台服务器宕机时，该服务器上的 Session 丢失，用户被踢；负载不均衡（同 IP 的请求集中在一台）。不推荐，仅作为临时过渡方案。

**方案四：Session 广播复制（Tomcat Cluster）**

Tomcat 集群模式下，Session 数据在所有节点间复制同步。适合小规模集群，节点多时同步开销大，不适合大规模。

**工程实践：** 现代项目基本都是 Spring Session + Redis 或 JWT，前者对传统 Session 代码改动最小，后者彻底解耦状态。

###### 3. 现在需要你设计一个将已登录的用户踢下线的功能，如何实现？

"踢下线"本质是**主动让用户的登录状态失效**，有两种主要技术栈的做法：

**方案一：Session 模式下的踢下线**

```java
// 方式1：直接删除 Redis 中的 Session
@Service
public class KickOutService {
    @Autowired
    private FindByIndexNameSessionRepository sessionRepository;
    
    public void kickUser(Long userId) {
        // 找到该用户的所有 Session
        Map<String, Session> sessions = sessionRepository
            .findByPrincipalName(userId.toString());
        // 逐一失效
        sessions.forEach((id, session) -> sessionRepository.deleteById(id));
    }
}
```

Spring Session 提供了按用户名查找所有 Session 的接口，删除即踢下线。

**方案二：JWT 模式下的踢下线**

JWT 无状态，无法直接失效，通过以下方式实现：

```java
// 方式1：黑名单（精确踢特定 Token）
public void kickByToken(String token) {
    String jti = parseJwt(token).getId();
    long remainTtl = calculateRemainingTtl(token);
    redisTemplate.opsForValue().set("jwt:blacklist:" + jti, "1", 
                                    remainTtl, TimeUnit.MILLISECONDS);
}

// 方式2：版本号机制（踢该用户所有 Token）
public void kickUser(Long userId) {
    // 在 Redis 中递增该用户的 tokenVersion
    redisTemplate.opsForValue().increment("user:token:version:" + userId);
    // 登录时在 JWT claims 中写入当前版本号
    // 验证时对比 JWT 中的版本号与 Redis 中的版本号，不匹配则拒绝
}
```

版本号方案可以踢该用户的所有 Token（比如改密码后踢所有已登录设备），且 Redis 只存一个整数，非常轻量。

**方案三：强制重新认证标志**

```java
// Redis 中存储用户的"强制登出"标志
public void forceLogout(Long userId) {
    redisTemplate.opsForValue().set("force:logout:" + userId, 
                                    System.currentTimeMillis(), 1, TimeUnit.HOURS);
}

// Token 验证时检查
public void validateToken(Long userId, long tokenIssuedAt) {
    String forceLogoutTime = redisTemplate.opsForValue().get("force:logout:" + userId);
    if (forceLogoutTime != null && Long.parseLong(forceLogoutTime) > tokenIssuedAt) {
        throw new TokenInvalidException("您已被踢下线，请重新登录");
    }
}
```

**功能扩展：** 系统可以提供管理员踢人、用户自助下线所有设备、异地登录踢下旧设备等功能，底层都是上述机制。

###### 4. 如何设计一个 OAuth2.0 授权服务？Token 如何管理？

**OAuth2.0 四种授权模式选择：**

- **授权码模式（Authorization Code）**：最安全，适合有服务端的第三方登录（微信登录、GitHub 登录）
- **客户端凭证模式（Client Credentials）**：服务间调用（M2M），不涉及用户
- **密码模式（Resource Owner Password）**：用户直接把密码给客户端，安全性差，仅在完全可信的一方应用中使用（如自家 APP）
- **PKCE 扩展授权码模式**：移动端/单页应用无法安全存储 client_secret 时使用

**授权码流程（最重要）：**

```
用户 → 点击"微信登录"
  → 重定向到授权服务器 /authorize?response_type=code&client_id=xxx&redirect_uri=xxx
用户 → 在授权服务器登录并同意授权
  → 授权服务器重定向回 redirect_uri，携带 code
业务服务器 → 用 code + client_secret 换 access_token（后端对后端，不经过浏览器）
  → 获得 access_token + refresh_token
业务服务器 → 用 access_token 访问资源服务器获取用户信息
```

**Spring Authorization Server（推荐实现）：**

```java
@Configuration
public class AuthorizationServerConfig {
    
    @Bean
    public RegisteredClientRepository registeredClientRepository() {
        RegisteredClient client = RegisteredClient.withId(UUID.randomUUID().toString())
            .clientId("web-app")
            .clientSecret(passwordEncoder.encode("secret"))
            .authorizationGrantType(AuthorizationGrantType.AUTHORIZATION_CODE)
            .authorizationGrantType(AuthorizationGrantType.REFRESH_TOKEN)
            .redirectUri("https://app.example.com/callback")
            .scope("openid").scope("profile").scope("email")
            .tokenSettings(TokenSettings.builder()
                .accessTokenTimeToLive(Duration.ofMinutes(15))
                .refreshTokenTimeToLive(Duration.ofDays(30))
                .build())
            .build();
        return new InMemoryRegisteredClientRepository(client);
    }
}
```

**Token 管理策略：**

- **Access Token**：JWT 格式（无状态，资源服务器本地验证）或 Opaque Token（透明令牌，需要 introspection 端点验证）
- **Refresh Token**：不透明字符串，存 DB 或 Redis，有效期长，用一次就轮换
- **PKCE（公共客户端必用）**：生成 code_verifier + code_challenge，防止授权码被截获

###### 5. 有人恶意攻击我们的注册功能，伪造手机号大量下发验证码，除了限流之外，如何解决这个问题？

**多层防御体系：**

**① 图形验证码 / 行为验证码（前置拦截）**

发验证码前，先完成行为验证。腾讯云验证码（CAPTCHA）、网易易盾支持滑块验证、无感验证，正常用户几乎无感知，机器人通过率极低。

**② 手机号真实性校验**

```java
// 手机号格式校验（正则）
if (!phone.matches("^1[3-9]\\d{9}$")) {
    throw new InvalidPhoneException("手机号格式错误");
}

// 空号检测（调用运营商 API 或第三方服务，如阿里云号码认证）
// 空号/注销号 → 直接拒绝，不发短信
boolean isRealPhone = phoneCheckService.isValid(phone);
```

**③ 发送频率控制（比限流更细粒度）**

```java
// 同一手机号：1分钟内只能发1条，24小时内最多5条
String key1min = "sms:1min:" + phone;
String key24h = "sms:24h:" + phone;

Long count1min = redisTemplate.opsForValue().increment(key1min);
if (count1min == 1) redisTemplate.expire(key1min, 1, TimeUnit.MINUTES);
if (count1min > 1) throw new TooManyRequestsException("发送太频繁，请1分钟后重试");

Long count24h = redisTemplate.opsForValue().increment(key24h);
if (count24h == 1) redisTemplate.expire(key24h, 24, TimeUnit.HOURS);
if (count24h > 5) throw new TooManyRequestsException("今日发送次数已达上限");
```

**④ 设备指纹 + IP 限制**

```java
// 同一 IP 每天最多给不同手机号发N条（防止用代理批量轮换手机号）
String ipKey = "sms:ip:24h:" + clientIp;
// 同一设备指纹（User-Agent + 浏览器特征）限制
String deviceKey = "sms:device:24h:" + deviceFingerprint;
```

**⑤ 短信费用预警**

配置短信通道的每日预算上限告警，超过阈值立即暂停发送并人工介入。

**⑥ 验证码有效性**

```java
// 验证码5分钟内有效，且只能用一次（验证成功后立即删除）
redisTemplate.delete("sms:code:" + phone); // 验证成功后删除，防止重放
```

**⑦ 黑名单机制**

检测到异常行为（短时间大量请求）的手机号/IP/设备，加入临时黑名单，直接拒绝后续请求。

###### 6. 如何防止 SQL 注入？

**SQL 注入原理：** 攻击者将恶意 SQL 语句拼接到查询中，改变 SQL 的语义：

```java
// ❌ 危险：拼接字符串
String sql = "SELECT * FROM users WHERE username='" + username + "'";
// 输入 username = "' OR '1'='1"，变成：
// SELECT * FROM users WHERE username='' OR '1'='1'  → 返回所有用户
```

**防止方法：**

**① 预编译语句（Prepared Statement）——根本解法**

```java
// ✅ JDBC 预编译
String sql = "SELECT * FROM users WHERE username = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, username); // 参数化绑定，不会被解释为 SQL
```

MyBatis 默认的 `#{}` 也是预编译，会自动转义：

```xml
<!-- ✅ 使用 #{} 参数占位符，预编译处理 -->
<select id="findUser" resultType="User">
    SELECT * FROM users WHERE username = #{username}
</select>

<!-- ❌ 使用 ${} 字符串替换，有注入风险（只用于动态表名/列名场景） -->
<select id="findUser">
    SELECT * FROM users WHERE username = '${username}'
</select>
```

**② 输入验证和白名单**

```java
// 对不能参数化的场景（如动态排序列名），使用白名单校验
List<String> allowedColumns = Arrays.asList("name", "age", "create_time");
if (!allowedColumns.contains(orderByColumn)) {
    throw new IllegalArgumentException("非法的排序字段");
}
```

**③ 最小权限原则**

数据库账号只授予必要的权限（只查询的账号不给 DELETE/DROP 权限），即使注入成功，危害也有限。

**④ ORM 框架天然防护**

JPA、MyBatis 正确使用时，参数化查询是默认行为，只要不用字符串拼接就是安全的。

**⑤ WAF（Web 应用防火墙）**

在网络层识别并拦截 SQL 注入攻击特征（`OR 1=1`、`UNION SELECT` 等），作为兜底防护层。

###### 7. 如何防止 XSS 和 CSRF 攻击？

**XSS（跨站脚本攻击）**

攻击者将恶意脚本注入网页，用户浏览时脚本执行，窃取 Cookie、劫持操作。

防御方案：

**① 输出转义（最根本）**

所有用户输入的内容，在输出到 HTML 时进行 HTML 实体转义：
- `<` → `&lt;`
- `>` → `&gt;`
- `"` → `&quot;`
- `'` → `&#x27;`

前端框架（React/Vue）默认转义，危险的是 `v-html` / `dangerouslySetInnerHTML`，要谨慎使用。

**② CSP（Content Security Policy）**

```nginx
# Nginx 添加安全响应头
add_header Content-Security-Policy "default-src 'self'; script-src 'self'; object-src 'none'";
```

限制页面只能加载同源的脚本，即使注入了 `<script>` 标签，浏览器也拒绝执行。

**③ HttpOnly Cookie**

```java
response.setHeader("Set-Cookie", "sessionId=xxx; HttpOnly; Secure; SameSite=Strict");
```

`HttpOnly` 让 JavaScript 无法读取 Cookie，即使 XSS 攻击成功，也无法窃取 Session。

**CSRF（跨站请求伪造）**

攻击者诱导用户访问恶意页面，该页面自动发送请求到目标网站，利用用户已登录的 Cookie 执行恶意操作。

防御方案：

**① CSRF Token（经典方案）**

```java
// Spring Security 默认开启 CSRF 保护
@Configuration
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.csrf().csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse());
        // 前端从 Cookie 读取 CSRF Token，每次请求放在 Header 中
        return http.build();
    }
}
```

**② SameSite Cookie**

```
Set-Cookie: sessionId=xxx; SameSite=Strict
```

`SameSite=Strict`：Cookie 只在同站请求时发送，跨站请求不带 Cookie，从根本上阻断 CSRF。
`SameSite=Lax`：更宽松，允许导航类跨站请求携带 Cookie（如点链接），但阻止 POST 跨站请求。

**③ 检查 Referer / Origin Header**

服务端校验请求头中的 `Referer` 和 `Origin`，只接受来自合法源的请求（但 Referer 可能被屏蔽）。

###### 8. 如何实现接口签名验证？

接口签名验证常用于**开放平台 API**（如微信支付、阿里云 API），防止请求被篡改、防重放攻击。

**签名流程（以阿里云风格为例）：**

```
请求参数 = {businessParam1=v1, businessParam2=v2, appId=xxx, timestamp=1710000000, nonce=uuid}

Step1: 将参数按字典序排列并拼接
      appId=xxx&businessParam1=v1&businessParam2=v2&nonce=uuid&timestamp=1710000000

Step2: 加入 SecretKey 计算 HMAC-SHA256
      sign = HMAC-SHA256(sortedParams + "&secretKey=xxx")

Step3: 将 sign 加入请求头
      X-Signature: sign值
```

**服务端验证：**

```java
@Component
public class SignatureFilter implements Filter {
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        HttpServletRequest req = (HttpServletRequest) request;
        
        String appId = req.getHeader("X-App-Id");
        String timestamp = req.getHeader("X-Timestamp");
        String nonce = req.getHeader("X-Nonce");
        String signature = req.getHeader("X-Signature");
        
        // 1. 时间戳校验（防重放：请求必须在5分钟内）
        long reqTime = Long.parseLong(timestamp);
        if (Math.abs(System.currentTimeMillis() / 1000 - reqTime) > 300) {
            sendError(response, "请求已过期");
            return;
        }
        
        // 2. Nonce 防重放（同一 nonce 只能用一次）
        if (Boolean.FALSE.equals(redisTemplate.opsForValue()
                .setIfAbsent("nonce:" + nonce, "1", 10, TimeUnit.MINUTES))) {
            sendError(response, "重放请求");
            return;
        }
        
        // 3. 查询 AppId 对应的 SecretKey
        String secretKey = appKeyService.getSecretKey(appId);
        
        // 4. 重新计算签名，对比
        String expectedSign = calculateSign(req, secretKey, timestamp, nonce);
        if (!MessageDigest.isEqual(signature.getBytes(), expectedSign.getBytes())) {
            sendError(response, "签名验证失败");
            return;
        }
        
        chain.doFilter(request, response);
    }
}
```

**安全要点：**
- 签名比较用 `MessageDigest.isEqual`（时间恒定比较），防止时序攻击
- SecretKey 不能放在请求中传递，只参与计算，不外泄
- nonce + timestamp 双重防重放
- HTTPS 传输，防止签名本身被窃取

###### 9. 如何设计一个权限系统（RBAC）？

**RBAC（Role-Based Access Control）——基于角色的访问控制**

```
用户（User）→ 角色（Role）→ 权限（Permission）→ 资源（Resource）

用户属于多个角色
角色拥有多个权限
权限对应某个资源的某个操作（CREATE/READ/UPDATE/DELETE）
```

**核心数据模型：**

```sql
-- 用户表
CREATE TABLE user (id BIGINT PRIMARY KEY, username VARCHAR(50));

-- 角色表
CREATE TABLE role (id BIGINT PRIMARY KEY, role_code VARCHAR(50), role_name VARCHAR(100));

-- 权限表（资源 + 操作）
CREATE TABLE permission (
    id BIGINT PRIMARY KEY,
    resource_code VARCHAR(100) COMMENT '资源标识，如 order:manage',
    action VARCHAR(20) COMMENT '操作，如 READ/WRITE/DELETE'
);

-- 用户-角色关联（多对多）
CREATE TABLE user_role (user_id BIGINT, role_id BIGINT);

-- 角色-权限关联（多对多）
CREATE TABLE role_permission (role_id BIGINT, permission_id BIGINT);
```

**Spring Security + RBAC 实现：**

```java
// 方法级权限控制
@PreAuthorize("hasPermission('order:manage', 'READ')")
public List<Order> queryOrders() { ... }

@PreAuthorize("hasRole('ADMIN') or hasPermission('order:manage', 'WRITE')")
public void createOrder(Order order) { ... }

// 自定义权限判断
@Component
public class CustomPermissionEvaluator implements PermissionEvaluator {
    @Override
    public boolean hasPermission(Authentication auth, Object targetDomainObject, 
                                  Object permission) {
        String username = auth.getName();
        String resource = (String) targetDomainObject;
        String action = (String) permission;
        // 查询用户的权限列表，判断是否包含此权限
        return permissionService.hasPermission(username, resource, action);
    }
}
```

**菜单权限（前端控制）：**

登录成功后，后端返回当前用户拥有的菜单列表和按钮权限列表，前端动态渲染菜单和控制按钮显隐。

**数据权限（行级控制）：**

```java
// 普通用户只能看自己部门的数据，管理员能看所有
// 通过 MyBatis 拦截器动态加 WHERE 条件
@DataScope(deptAlias = "d", userAlias = "u")
public List<User> selectUserList(User user) { ... }
// → WHERE d.dept_id IN (当前用户可访问的部门ID列表)
```

**三权分立建议（生产环境）：** 管理员角色只管用户/角色配置，不接触业务数据；审计员角色只能查看操作日志；普通角色按最小权限原则分配。

###### 10. 敏感数据如何加密存储和传输？

**传输层：HTTPS + TLS 1.3**

所有 API 强制 HTTPS，配置现代 TLS 1.3，禁用 TLS 1.0/1.1 和弱加密套件。

**存储加密：**

**① 密码字段——单向哈希（不可逆）**

```java
// 正确做法：BCrypt（自动加盐，抗彩虹表）
String hashedPassword = BCrypt.hashpw(rawPassword, BCrypt.gensalt(12));
// 验证
boolean valid = BCrypt.checkpw(inputPassword, hashedPassword);

// ❌ 错误做法：MD5、SHA1（可被彩虹表破解）
// ❌ 更错误：明文存储密码
```

**② 手机号/身份证/银行卡——对称加密 + 部分展示脱敏**

```java
// AES-256-GCM 加密（认证加密，防篡改）
@Service
public class EncryptService {
    
    public String encrypt(String plaintext) {
        SecretKeySpec key = new SecretKeySpec(masterKey, "AES");
        GCMParameterSpec params = new GCMParameterSpec(128, generateIV());
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.ENCRYPT_MODE, key, params);
        byte[] ciphertext = cipher.doFinal(plaintext.getBytes(StandardCharsets.UTF_8));
        return Base64.encode(IV + ciphertext); // IV 与密文一起存储
    }
}

// 展示时脱敏
String masked = phone.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2");
// 138****8888
```

**③ 加密密钥管理**

密钥不能硬编码在代码中，应该：
- 从环境变量或 K8s Secret 注入
- 使用 KMS（Key Management Service，如 AWS KMS、阿里云 KMS）管理密钥生命周期
- 定期轮换密钥（密钥轮换时需要重加密历史数据）

**④ 数据库字段加密（透明加密）**

```java
// MyBatis TypeHandler 透明加密/解密
@MappedTypes(EncryptedString.class)
public class EncryptTypeHandler extends BaseTypeHandler<String> {
    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, String param, JdbcType jdbcType) {
        ps.setString(i, encryptService.encrypt(param)); // 写入时加密
    }
    
    @Override
    public String getNullableResult(ResultSet rs, String columnName) {
        return encryptService.decrypt(rs.getString(columnName)); // 读取时解密
    }
}
```

**⑤ 审计与监控**

对敏感数据的访问做审计日志（谁、什么时候、查看了哪些敏感数据），异常访问触发告警。
