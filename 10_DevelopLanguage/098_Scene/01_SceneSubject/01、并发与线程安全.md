# 一、并发与线程安全

###### 1. 在 Java 中，不使用锁如何实现一个线程安全的单例？

最经典的答案是用**静态内部类**或**枚举**。

**静态内部类（推荐）：**

```java
public class Singleton {
    private Singleton() {}

    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

原理：JVM 在加载 `Singleton` 类时，不会立即加载 `Holder`，只有第一次调用 `getInstance()` 时才触发 `Holder` 的类加载。类加载是线程安全的（由 JVM 类加载锁保证），所以 `INSTANCE` 只会被初始化一次，且没有显式的锁开销。这同时满足了**懒加载**和**线程安全**。

**枚举（防反射/反序列化攻击）：**

```java
public enum Singleton {
    INSTANCE;

    public void doSomething() { ... }
}
```

枚举类在 JVM 层面天然是单例，序列化由 JVM 保证，反射也无法破坏（枚举不允许反射创建实例）。是 Effective Java 中推荐的方式。

**双重检查锁（DCL）——需要 volatile：**

```java
public class Singleton {
    // volatile 禁止指令重排，防止拿到半初始化对象
    private static volatile Singleton INSTANCE;

    private Singleton() {}

    public static Singleton getInstance() {
        if (INSTANCE == null) {
            synchronized (Singleton.class) {
                if (INSTANCE == null) {
                    INSTANCE = new Singleton();
                }
            }
        }
        return INSTANCE;
    }
}
```

注意：**volatile 不可省略**。`new Singleton()` 在字节码层面分三步：①分配内存 ②初始化对象 ③将引用指向内存地址，JIT 可能重排成 ①③②，导致另一个线程看到非 null 但未初始化的对象。

**面试结论：** 不需要显式锁的推荐方案是静态内部类；需要防止序列化/反射破坏的用枚举；DCL 也是无锁路径下的常见方案但必须带 volatile。

###### 2. 主线程如何得知线程池中多个并行任务的完成并收集它们的结果？

**方案一：CompletableFuture.allOf + thenApply（最推荐）**

```java
ExecutorService executor = Executors.newFixedThreadPool(4);

CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> queryUserInfo(), executor);
CompletableFuture<Integer> f2 = CompletableFuture.supplyAsync(() -> queryOrderCount(), executor);
CompletableFuture<BigDecimal> f3 = CompletableFuture.supplyAsync(() -> queryBalance(), executor);

// allOf 等待全部完成
CompletableFuture.allOf(f1, f2, f3).join();

// 分别获取结果（join() 不抛受检异常，比 get() 更方便）
String userInfo = f1.join();
int orderCount = f2.join();
BigDecimal balance = f3.join();
```

`allOf` 等待所有任务完成，`join()` 在 allOf 之后调用不会阻塞。如果某个任务抛异常，会在 `join()` 时以 `CompletionException` 包装抛出。

**方案二：Future + List（传统方式）**

```java
List<Future<String>> futures = new ArrayList<>();
futures.add(executor.submit(() -> task1()));
futures.add(executor.submit(() -> task2()));

List<String> results = new ArrayList<>();
for (Future<String> f : futures) {
    results.add(f.get()); // 依次阻塞等待
}
```

缺点是 `get()` 串行等待，如果前面的任务慢，后面已完成的任务结果也要等着。

**方案三：CompletionService（先完成先处理）**

```java
CompletionService<String> cs = new ExecutorCompletionService<>(executor);
for (int i = 0; i < 10; i++) {
    cs.submit(() -> doTask());
}
for (int i = 0; i < 10; i++) {
    String result = cs.take().get(); // 哪个先完成就先拿哪个
    process(result);
}
```

**方案四：CountDownLatch（只需等待完成，不需要返回值）**

```java
CountDownLatch latch = new CountDownLatch(3);
executor.submit(() -> { try { task1(); } finally { latch.countDown(); } });
executor.submit(() -> { try { task2(); } finally { latch.countDown(); } });
executor.submit(() -> { try { task3(); } finally { latch.countDown(); } });
latch.await(10, TimeUnit.SECONDS); // 带超时等待
```

**实际项目选择：** 现代 Java 项目中首选 CompletableFuture，链式编程 + 异常处理 + 超时控制（`orTimeout`）一套组合下来非常优雅。

###### 3. 如何设计一个线程安全的计数器，支持高并发场景？

**层层递进的方案：**

**方案一：synchronized（最简单，高竞争下性能差）**

```java
public class SyncCounter {
    private long count = 0;
    public synchronized void increment() { count++; }
    public synchronized long get() { return count; }
}
```

**方案二：AtomicLong（CAS 无锁，中低并发首选）**

```java
private final AtomicLong count = new AtomicLong(0);
public void increment() { count.incrementAndGet(); }
public long get() { return count.get(); }
```

底层是 CAS（Compare-And-Swap）操作，在竞争不激烈时性能极好。高竞争时 CAS 失败重试多，性能下降。

**方案三：LongAdder（高并发计数首选）**

```java
private final LongAdder count = new LongAdder();
public void increment() { count.increment(); }
public long get() { return count.sum(); }
```

JDK 8 引入，核心思路是**分段竞争**：每个线程尽量操作自己的 Cell（类似 ConcurrentHashMap 分段锁思想），读取时 `sum()` 把所有 Cell 累加。高并发写多读少的场景性能碾压 AtomicLong。

**方案四：分布式场景——Redis INCR**

```java
// Redis 原子操作，单线程模型天然保证原子性
redisTemplate.opsForValue().increment("counter:orderCount", 1);
```

Redis 的 INCR/INCRBY 命令是原子操作，适合需要跨 JVM 实例共享的计数器（比如限流计数、访问量统计）。

**选型建议：**
- 单 JVM 低竞争 → AtomicLong
- 单 JVM 高并发写 → LongAdder
- 需要分布式共享 → Redis INCR
- 需要持久化 → 数据库 + 定期批量刷新（本地 LongAdder 累积，定时写 DB）

###### 4. 多线程环境下如何保证订单号的唯一性？

**方案一：雪花算法（Snowflake）——分布式场景主流**

```
64位 = 1位符号 + 41位时间戳(ms) + 10位机器ID + 12位序列号
```

- 41位时间戳支持约69年
- 10位机器ID支持1024个节点（可拆分成5位数据中心+5位机器）
- 12位序列号每毫秒每节点可生成4096个ID

特点：趋势递增（对索引友好）、不依赖数据库、高性能（每秒数百万）。

注意时钟回拨问题：检测到时钟回拨时，可以等待时钟追上、或拒绝生成并报警。Leaf（美团）、Uid-Generator（百度）都是雪花算法的生产级实现。

**方案二：数据库自增 ID**

主键 AUTO_INCREMENT，简单可靠，但依赖数据库成为瓶颈。分库分表时需要用"号段模式"：从 DB 批量申请一段 ID（如每次取1000个），内存中消费完再申请。

**方案三：UUID**

```java
String orderId = UUID.randomUUID().toString().replace("-", "");
```

优点是本地生成，不依赖任何外部服务。缺点是36位字符串无序，对 InnoDB 聚簇索引不友好（频繁页分裂），且不含业务语义。

**方案四：业务前缀 + 时间戳 + 随机数**

```
ORD-20260315-1450-8392745
```

可读性好，但并发极高时随机数可能重复，需要加数据库唯一索引兜底。

**单 JVM 多线程场景：**

直接用 `AtomicLong` 递增序号，配合时间戳前缀拼装。

**实际项目推荐：** 分布式环境用雪花算法（或 Leaf/Uid-Generator）；单机简单场景用 `AtomicLong` + 时间戳；有现成的分布式 ID 服务直接接入，不要重复造轮子。

###### 5. 分布式环境下如何实现分布式锁？有哪些实现方案？

**方案一：Redis 分布式锁（最常用）**

**基础版 SET NX EX：**

```java
// 原子命令：SET key value NX PX milliseconds
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent("lock:order:123", requestId, 30, TimeUnit.SECONDS);
if (Boolean.TRUE.equals(success)) {
    try {
        // 执行业务逻辑
    } finally {
        // Lua 脚本原子删除（验证是自己的锁才删）
        String luaScript = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                          "return redis.call('del', KEYS[1]) else return 0 end";
        redisTemplate.execute(new DefaultRedisScript<>(luaScript, Long.class),
                             List.of("lock:order:123"), requestId);
    }
}
```

关键点：
1. 用 UUID 作为 value，删锁时验证是否是自己持有的，防误删
2. 删锁用 Lua 脚本保证原子性（check-and-delete 两步要原子）
3. 设置合理的过期时间，避免锁永远不释放

**生产级：Redisson（强烈推荐）**

```java
RLock lock = redissonClient.getLock("lock:order:123");
try {
    // 尝试加锁，最多等待3秒，锁自动释放时间30秒
    boolean locked = lock.tryLock(3, 30, TimeUnit.SECONDS);
    if (locked) {
        // 业务逻辑
    }
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

Redisson 内置看门狗机制（Watchdog）：业务未完成时自动续期，解决锁超时释放的问题。还支持 RedLock（多节点）、可重入锁、读写锁等。

**方案二：ZooKeeper 分布式锁**

利用 ZK 的临时有序节点：所有客户端在同一路径下创建 `lock-0000000001` 这样的临时有序节点，序号最小的获得锁，其他节点 Watch 前一个节点，前一个节点删除时依次唤醒。

优点：ZK 会话断开时临时节点自动删除，不会出现死锁；有 Watch 机制，不需要轮询。
缺点：ZK 性能不如 Redis，且需要维护额外的 ZK 集群。

**方案三：数据库乐观锁**

```sql
UPDATE inventory SET stock = stock - 1, version = version + 1
WHERE product_id = 1 AND version = #{version} AND stock > 0
```

适合并发量不高的场景，实现简单，但高并发时大量 UPDATE 失败导致重试，性能差。

**方案四：数据库唯一索引**

```sql
INSERT INTO distributed_lock(lock_key, owner, expire_time) 
VALUES ('order:123', 'node1', NOW() + INTERVAL 30 SECOND)
-- 唯一索引在 lock_key 上
```

靠唯一索引冲突来实现互斥，简单但有性能瓶颈和过期释放复杂性。

**选型总结：**
- 高并发、对性能要求高 → Redis（Redisson）
- 对一致性要求极高、能接受性能损耗 → ZooKeeper（Curator）
- 并发量低、不想引入中间件 → 数据库乐观锁或唯一索引
- 生产项目 99% 的场景用 Redisson 就够了
