###### 1. JDK 序列化问题排查

JDK 序列化（ObjectOutputStream / ObjectInputStream）在分布式系统中引发问题的场景，以及排查思路。

**常见问题一：serialVersionUID 不一致**

```
java.io.InvalidClassException: com.example.User; local class incompatible: 
stream classdesc serialVersionUID = 12345, local class serialVersionUID = 67890
```

原因：类修改后（加字段、改方法）没有显式声明 `serialVersionUID`，JVM 自动重新计算，与之前序列化的版本不同。

解决：始终显式声明 `serialVersionUID`：

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L; // 明确声明，不依赖自动生成
    private String name;
    // 新增字段不影响反序列化（会用默认值）
    private String email; // 新加的字段，旧版序列化数据反序列化时 email=null
}
```

**常见问题二：序列化对象过大（网络传输 / Redis 存储）**

排查：计算序列化后字节大小：

```java
ByteArrayOutputStream baos = new ByteArrayOutputStream();
ObjectOutputStream oos = new ObjectOutputStream(baos);
oos.writeObject(obj);
System.out.println("序列化大小: " + baos.size() + " bytes");
```

优化：改用 JSON（Jackson）、Protobuf 或 Kryo 序列化，体积更小，速度更快。

**常见问题三：Redis 中反序列化失败（ClassNotFoundException）**

Redis 中存了旧版本类的 JDK 序列化字节，新版本代码反序列化时找不到类或字段不匹配。

解决：Redis 缓存的值改用 JSON 序列化（Jackson2JsonRedisSerializer），不依赖类结构：

```java
@Bean
public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
    RedisTemplate<String, Object> template = new RedisTemplate<>();
    template.setConnectionFactory(factory);
    Jackson2JsonRedisSerializer<Object> serializer = 
        new Jackson2JsonRedisSerializer<>(Object.class);
    template.setValueSerializer(serializer);
    return template;
}
```

**常见问题四：Transient 字段丢失**

```java
private transient String sensitiveField; // transient 字段不会被序列化
```

反序列化后 `sensitiveField` 是 null，如果业务逻辑依赖这个字段就会出错。排查时注意检查 transient 修饰的字段。

###### 2. 如何设计一个短链系统？

短链系统的核心：**将长 URL 映射为短 URL，访问短 URL 时重定向到长 URL**。

**核心数据结构：**

```sql
CREATE TABLE short_url (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    short_code VARCHAR(10) UNIQUE NOT NULL COMMENT '短码，如 abc123',
    long_url TEXT NOT NULL COMMENT '原始长 URL',
    create_time DATETIME,
    expire_time DATETIME COMMENT '过期时间，null表示永不过期',
    access_count BIGINT DEFAULT 0 COMMENT '访问次数'
);
-- short_code 有唯一索引
```

**短码生成方案：**

**方案一：自增 ID 转 Base62（推荐）**

```java
private static final String CHARS = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";

public String encode(long id) {
    StringBuilder sb = new StringBuilder();
    while (id > 0) {
        sb.append(CHARS.charAt((int)(id % 62)));
        id /= 62;
    }
    return sb.reverse().toString(); // 如：id=10000 → "2Bi"
}
```

6位 Base62 可以表示 62^6 ≈ 568亿个短链，足够用很长时间。

**方案二：MD5/SHA1 取前几位**

对长 URL 做 MD5，取前6个字符。可能有哈希碰撞（两个不同 URL 映射到同一短码），需要碰撞时追加内容重试。

**重定向处理：**

```java
@GetMapping("/{shortCode}")
public ResponseEntity<Void> redirect(@PathVariable String shortCode,
                                      HttpServletRequest request) {
    // 1. 先查 Redis 缓存
    String longUrl = redisTemplate.opsForValue().get("url:" + shortCode);
    
    if (longUrl == null) {
        // 2. 缓存未命中，查 DB
        ShortUrl entity = shortUrlMapper.findByCode(shortCode);
        if (entity == null || isExpired(entity)) {
            return ResponseEntity.notFound().build();
        }
        longUrl = entity.getLongUrl();
        redisTemplate.opsForValue().set("url:" + shortCode, longUrl, 1, TimeUnit.DAYS);
    }
    
    // 3. 异步统计访问次数（不阻塞重定向）
    asyncService.incrementAccessCount(shortCode);
    
    // 301 永久重定向（浏览器缓存，减少重复请求）
    // 302 临时重定向（需要统计点击量时用302，浏览器不会缓存）
    return ResponseEntity.status(HttpStatus.MOVED_PERMANENTLY)
                         .header("Location", longUrl)
                         .build();
}
```

**关键细节：**

- 301 vs 302：301 浏览器会缓存，之后直接访问长链，节省服务器压力但无法统计点击；302 每次都打服务器，可以统计。通常短链用302。
- 缓存：热门短链放 Redis，避免每次都查 DB（访问量高的短链延迟从几十ms降到1ms以内）
- 自定义短码：允许用户指定短码（如品牌活动），需要检查是否冲突
- 安全检测：长 URL 应通过安全API检测是否是恶意链接

###### 3. 如何设计一个延迟任务调度系统？

延迟任务 = 在某个时间点执行某个任务，比如订单30分钟未支付自动取消、7天后自动确认收货。

**方案一：Redis ZSet（中小量级，最常用）**

```java
// 添加延迟任务
public void addTask(String taskId, String taskData, long executeAtMs) {
    redisTemplate.opsForZSet().add("delay:tasks", 
                                    taskId + ":" + taskData, 
                                    executeAtMs); // score = 执行时间戳
}

// 调度器：每秒扫描到期任务
@Scheduled(fixedDelay = 1000)
public void processTasks() {
    long now = System.currentTimeMillis();
    // 查询 [0, now] 范围内的任务
    Set<ZSetOperations.TypedTuple<String>> tasks = 
        redisTemplate.opsForZSet().rangeByScoreWithScores("delay:tasks", 0, now, 0, 100);
    
    tasks.forEach(tuple -> {
        String value = tuple.getValue();
        // 原子删除，防止多节点重复执行
        Long removed = redisTemplate.opsForZSet().remove("delay:tasks", value);
        if (removed == 1) {
            executeTask(value);
        }
    });
}
```

**方案二：时间轮（高性能，低延迟要求场景）**

Netty 的 `HashedWheelTimer` 是时间轮的经典实现，在内存中高效管理大量定时任务：

```java
HashedWheelTimer timer = new HashedWheelTimer(100, TimeUnit.MILLISECONDS, 512);

timer.newTimeout(timeout -> {
    // 30分钟后执行
    cancelUnpaidOrder(orderId);
}, 30, TimeUnit.MINUTES);
```

缺点：纯内存，不持久化，JVM 重启任务丢失。适合不需要持久化的短期任务（如连接超时、心跳检测）。

**方案三：MQ 延迟消息（可持久化，可靠）**

- RocketMQ 原生支持延迟级别
- RabbitMQ 插件支持任意延迟

适合需要可靠持久化的延迟任务。

**方案四：XXL-Job / Quartz（企业级任务调度）**

```java
@XxlJob("cancelUnpaidOrderHandler")
public ReturnT<String> cancelUnpaidOrders() {
    // 扫描超时未支付订单
    List<Order> orders = orderMapper.findUnpaidExpired();
    orders.forEach(order -> orderService.cancelOrder(order.getId()));
    return ReturnT.SUCCESS;
}
```

XXL-Job 支持分布式任务分片、失败重试、可视化控制台，适合有复杂调度需求的场景。

**方案选择：**

| 场景 | 推荐方案 |
|------|----------|
| 量少，简单，不需要持久化 | 时间轮（Netty）|
| 量中等，需要持久化 | Redis ZSet |
| 量大，需要高可靠 | MQ 延迟消息 |
| 复杂定时任务（周期性）| XXL-Job / Quartz |

###### 4. 如何设计一个评论系统？

评论系统的特点：**层级关系（楼层评论+楼中楼）、读多写少、某些内容评论量巨大（热门文章/视频）**。

**数据模型：**

```sql
CREATE TABLE comment (
    id BIGINT PRIMARY KEY,
    content_id BIGINT NOT NULL COMMENT '被评论的内容ID（文章/视频）',
    content_type TINYINT NOT NULL COMMENT '内容类型',
    user_id BIGINT NOT NULL COMMENT '评论者',
    parent_id BIGINT DEFAULT 0 COMMENT '父评论ID，0表示根评论（楼层）',
    root_id BIGINT DEFAULT 0 COMMENT '根评论ID，用于快速查楼中楼',
    reply_user_id BIGINT COMMENT '回复的目标用户ID',
    content TEXT NOT NULL,
    like_count INT DEFAULT 0,
    status TINYINT DEFAULT 1 COMMENT '1正常，0已删除',
    create_time DATETIME,
    INDEX idx_content(content_id, content_type, status, create_time),
    INDEX idx_root(root_id, create_time)
);
```

**接口设计：**

- 获取评论列表：先查根评论（parent_id=0），分页返回
- 获取楼中楼：根据 root_id 查子评论，展示前几条，"查看更多"再分页

```java
// 查某文章的评论（两级结构）
public CommentPageResult getComments(Long contentId, int type, int page) {
    // 1. 查根评论，分页
    List<Comment> roots = commentMapper.findRootComments(contentId, type, page, PAGE_SIZE);
    
    // 2. 批量查每个根评论的前N条楼中楼（减少 N+1 查询）
    List<Long> rootIds = roots.stream().map(Comment::getId).collect(toList());
    Map<Long, List<Comment>> subCommentsMap = commentMapper.findSubComments(rootIds, 3);
    
    // 3. 组装
    return assembleResult(roots, subCommentsMap);
}
```

**热门评论处理：**

- 热门文章下可能有几十万条评论，深翻页性能差
- 前N页（前20页）走 Redis 缓存（按时间或按热度的评论 ID 列表），后面的页走 DB 游标分页
- 评论的点赞数用 Redis 计数，定时同步回 DB

**内容审核：**

评论发布时，调用内容安全 API（阿里云绿网、腾讯云）进行违禁词/违规内容检测，通过后才展示。对高风险用户可以先展示给自己，异步审核完再公开（微博的做法）。

###### 5. 如何设计一个点赞系统？

点赞系统的挑战：**高频写入（每次点赞都是写操作）、读多（展示点赞数）、同一用户不能重复点赞**。

**设计方案：**

```java
// 点赞/取消点赞（Redis BitMap 记录用户是否点赞）
public boolean toggleLike(Long contentId, String contentType, Long userId) {
    String likedKey = "liked:" + contentType + ":" + contentId;
    String countKey = "like:count:" + contentType + ":" + contentId;
    
    // BitMap offset = userId（注意：用户ID范围大时改为 Set）
    Boolean isLiked = redisTemplate.opsForValue().getBit(likedKey, userId);
    
    if (Boolean.TRUE.equals(isLiked)) {
        // 取消点赞
        redisTemplate.opsForValue().setBit(likedKey, userId, false);
        redisTemplate.opsForValue().decrement(countKey);
        return false;
    } else {
        // 点赞
        redisTemplate.opsForValue().setBit(likedKey, userId, true);
        redisTemplate.opsForValue().increment(countKey);
        return true;
    }
}

// 查询点赞数（直接读 Redis）
public long getLikeCount(Long contentId, String contentType) {
    String countKey = "like:count:" + contentType + ":" + contentId;
    Object count = redisTemplate.opsForValue().get(countKey);
    return count == null ? 0L : Long.parseLong(count.toString());
}
```

**大 UserId 问题：** BitMap 按 userId 做偏移量，如果 userId 是雪花算法的 long（很大），直接用会撑爆内存。改用 Redis Set 存储点赞用户 ID（`SADD`/`SISMEMBER`），或者先将 userId 映射为连续序号。

**延迟写入 DB：**

点赞数不需要实时入 DB，可以定时（每分钟/每5分钟）将 Redis 中的点赞数同步到 DB。用户实时看到的是 Redis 中的数据。

**防刷赞：** 同一用户在短时间内连续点赞/取消点赞，可能是机器人。添加频率限制：同一用户同一内容，10秒内最多操作1次。

###### 6. 如何设计一个消息推送系统？

消息推送场景：APP 推送通知（离线推送）、IM 即时消息（在线推送）、系统广播（业务通知）。

**架构分层：**

```
业务服务 → 消息路由服务 → 推送网关
                              ├── APNs（iOS 推送）
                              ├── FCM（Android 推送）
                              ├── WebSocket（在线用户实时推送）
                              └── SMS（短信，重要通知）
```

**在线用户实时推送（WebSocket / SSE）：**

```java
// 用户连接 WebSocket，注册到推送服务
@ServerEndpoint("/ws/notify/{userId}")
public class NotifyEndpoint {
    
    @OnOpen
    public void onOpen(Session session, @PathParam("userId") Long userId) {
        // 维护 userId → Session 的映射
        SessionManager.register(userId, session);
    }
    
    @OnClose
    public void onClose(@PathParam("userId") Long userId) {
        SessionManager.unregister(userId);
    }
}

// 推送消息给指定用户
public void pushToUser(Long userId, Message message) {
    Session session = SessionManager.getSession(userId);
    if (session != null && session.isOpen()) {
        session.getAsyncRemote().sendText(JSON.toJSON(message));
    } else {
        // 用户不在线，走离线推送（APNs/FCM）或落库待用户上线拉取
        offlinePushService.push(userId, message);
    }
}
```

**多节点 WebSocket（Session 共享问题）：**

多台服务器，用户 A 连接在服务器1，但消息从服务器2发出——服务器2没有用户 A 的 Session。

解决：所有服务器订阅 Redis Pub/Sub，推送时广播消息，拥有该用户 Session 的服务器负责推送：

```java
// 发消息时，通过 Redis Pub/Sub 广播
redisTemplate.convertAndSend("push:channel:" + userId, JSON.toJSON(message));
// 每台服务器订阅所有用户频道，有对应 Session 的就推，没有的忽略
```

**离线推送（APNs / FCM）：**

用户不在线时，通过设备厂商的推送通道发送通知，用户设备接收到后展示在通知栏。需要在用户登录时将 deviceToken 存入 DB，推送时查出目标用户的 deviceToken 调用推送 API。

**消息可靠性：**

重要消息（如支付成功通知）要保证至少一次送达：
1. 推送后记录"已推送"状态
2. 用户端收到后回 ACK
3. 未收到 ACK 的消息，定时重推（上限N次）

###### 7. 如何设计一个搜索系统？

**全文搜索架构：**

```
用户输入 → 查询分析（分词/纠错/同义词扩展）→ ES 查询 → 结果排序 → 返回
                                                    ↑
数据变更 → Canal 监听 binlog → MQ → 索引更新服务 → ES 索引同步
```

**ElasticSearch 核心设计：**

```json
PUT /products
{
  "mappings": {
    "properties": {
      "name": {"type": "text", "analyzer": "ik_max_word", "search_analyzer": "ik_smart"},
      "description": {"type": "text", "analyzer": "ik_max_word"},
      "category": {"type": "keyword"},
      "price": {"type": "scaled_float", "scaling_factor": 100},
      "sales": {"type": "long"},
      "create_time": {"type": "date"}
    }
  }
}
```

**查询设计：**

```json
GET /products/_search
{
  "query": {
    "bool": {
      "must": [
        {"multi_match": {"query": "手机 华为", "fields": ["name^3", "description"]}}
      ],
      "filter": [
        {"term": {"category": "手机"}},
        {"range": {"price": {"gte": 1000, "lte": 5000}}}
      ]
    }
  },
  "sort": [{"_score": "desc"}, {"sales": "desc"}],
  "from": 0, "size": 20,
  "highlight": {"fields": {"name": {}}}
}
```

**数据同步（DB → ES）：**

- 全量同步：初始化或 ES 索引重建时，从 DB 分批导入
- 增量同步：Canal 监听 binlog，实时同步数据变更到 ES（延迟通常在秒级以内）

**搜索优化技巧：**

- **搜索建议（Suggest）**：`search_as_you_type` 字段类型，实时补全输入
- **词典扩展**：IK 分词器支持自定义词典（品牌名、专业术语）
- **同义词**：配置同义词词典（手机=手持电话=移动电话）
- **拼写纠错**：`fuzzy` 查询处理拼写错误（iphon → iphone）

###### 8. 如何设计一个日志收集系统？

**ELK / EFK 是标准方案：**

```
应用日志 → Filebeat/Logback → Kafka → Logstash/Flink → Elasticsearch → Kibana
```

**各组件职责：**

- **Filebeat**：轻量日志采集器，监控日志文件变化，实时推送
- **Logback → Kafka Appender**：Java 应用直接将日志发到 Kafka，不落本地文件（适合容器化）
- **Kafka**：缓冲高峰期日志洪峰，解耦生产和消费
- **Logstash / Flink**：日志清洗、结构化解析、字段提取、过滤
- **Elasticsearch**：日志存储和全文检索（倒排索引）
- **Kibana**：可视化查询、仪表盘、告警

**日志规范（Java 应用）：**

```java
// 结构化日志（JSON 格式），便于解析
log.info("order created, orderId={}, userId={}, amount={}", orderId, userId, amount);

// 错误日志必须带异常堆栈
log.error("order create failed, orderId={}", orderId, e);

// 包含 TraceId（链路追踪）
// 通过 MDC 注入 TraceId，日志自动携带
MDC.put("traceId", TraceContext.getTraceId());
```

**日志索引策略：**

```
按天创建索引：app-logs-2026.03.15
保留30天，超过30天的自动删除（ILM 策略）
热索引：最近7天，高性能 SSD
温索引：7-30天，HDD
冷索引/删除：30天后
```

**告警集成：**

Kibana 或 ElastAlert 配置规则：ERROR 日志每分钟超过N条，触发钉钉/邮件告警。

###### 9. 如何设计一个配置中心？

配置中心解决的问题：**配置与代码分离、配置变更不需要重启应用、多环境配置管理**。

**主流选择：Nacos Config（推荐）**

```yaml
# Spring Boot 接入 Nacos 配置中心
spring:
  cloud:
    nacos:
      config:
        server-addr: nacos:8848
        namespace: ${NACOS_NAMESPACE:dev}
        group: DEFAULT_GROUP
        file-extension: yaml
        refresh-enabled: true  # 启用动态刷新
```

```java
// @RefreshScope：配置变更时自动刷新 Bean
@RefreshScope
@RestController
public class OrderController {
    
    @Value("${order.timeout:1800}")
    private int orderTimeout; // Nacos 中修改后，下次请求自动用新值
}
```

**配置中心核心功能设计：**

**① 配置版本管理**

每次配置变更保存历史版本，支持回滚。数据结构：

```sql
CREATE TABLE config_history (
    id BIGINT PRIMARY KEY,
    config_id BIGINT,
    content TEXT,
    operator VARCHAR(50),
    change_reason VARCHAR(200),
    create_time DATETIME
);
```

**② 环境隔离**

通过 Namespace（dev/test/staging/prod）隔离不同环境的配置，防止开发配置影响生产。

**③ 权限控制**

生产环境配置需要审批流程，防止误操作。

**④ 配置推送**

Nacos 客户端通过长轮询监听配置变更，服务端配置变化后，主动通知客户端拉取新配置（延迟约1秒内）。

**⑤ 加密配置**

密码、密钥等敏感配置加密存储（可用 Jasypt 透明解密），或直接通过 K8s Secret 注入，不存配置中心。

###### 10. 如何设计一个灰度发布系统？

灰度发布（Canary Release / Feature Flag）：**新版本先对小部分用户开放，验证没问题后再全量发布**，降低发布风险。

**方案一：基于网关的流量切分（最常用）**

```yaml
# Spring Cloud Gateway 流量染色
spring:
  cloud:
    gateway:
      routes:
        - id: order-service-canary
          uri: lb://order-service
          predicates:
            - Path=/api/order/**
          filters:
            # 将5%的流量路由到 v2 版本（带 version=v2 的 Header）
            - name: RequestRateLimiter
            - name: AddRequestHeader
              args:
                name: X-Version
                value: "#{T(java.util.concurrent.ThreadLocalRandom).current().nextInt(100) < 5 ? 'v2' : 'v1'}"
```

或者用 Nginx + Lua 脚本按比例切流。

**方案二：基于用户属性的灰度（精准灰度）**

```java
// 灰度规则配置（存配置中心）
// {"type":"userId", "value":"[1000,2000]"}  // userId 在1000-2000之间的用户
// {"type":"city", "value":"北京,上海"}        // 北京和上海的用户

@Component
public class GrayFilter implements GlobalFilter {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        Long userId = extractUserId(exchange.getRequest());
        boolean isGray = grayRuleEngine.isGrayUser(userId);
        
        String targetVersion = isGray ? "v2" : "v1";
        ServerHttpRequest request = exchange.getRequest().mutate()
            .header("X-Version", targetVersion)
            .build();
        return chain.filter(exchange.mutate().request(request).build());
    }
}
```

**方案三：Feature Flag（代码级灰度）**

```java
// 用 Feature Toggle 控制功能开关，不需要重启
if (featureToggleService.isEnabled("new-checkout-flow", userId)) {
    return newCheckoutService.checkout(order);
} else {
    return oldCheckoutService.checkout(order);
}
```

FeatureFlag 的状态存 Redis 或配置中心，实时开关功能，适合精细化的功能灰度。

**灰度发布流程：**

1. 发布 v2 实例（不接流量）
2. 按比例放流：1% → 10% → 50% → 100%
3. 每个阶段观察错误率、响应时间、业务指标（下单成功率）
4. 异常则立即回滚（切回 v1）
5. 指标正常则继续扩大比例
