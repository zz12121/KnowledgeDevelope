# 四、分布式系统

###### 1. 现在我让你把单体服务拆成多个微服务，这些服务之间如何自动发现彼此？具体原理是什么？

**服务自动发现的核心组件：服务注册中心**

微服务的自动发现依赖注册中心，有 Nacos、Eureka、Consul、ZooKeeper 几种选择，目前国内主流是 **Nacos**。

**工作原理：**

```
                        ① 注册（实例启动时）
服务提供者 ──────────────────────────────→ 注册中心（Nacos）
服务提供者 ←────────────────────────────── ② 心跳续约（每5秒）
服务消费者 ──────────────────────────────→ 注册中心（Nacos）③ 拉取/订阅服务列表
服务消费者 ←────────────────────────────── ④ 推送变更（服务上下线时）
服务消费者 ──────────────────────────────→ 服务提供者 ⑤ 直接调用
```

**① 服务注册：** 服务实例启动时，将自己的 IP、端口、服务名等信息注册到 Nacos
**② 心跳续约：** 每隔 5 秒向 Nacos 发送心跳，证明自己还活着；Nacos 超过 15 秒未收到心跳，标记为不健康；超过 30 秒将实例从注册表中删除
**③ 服务发现：** 消费者从 Nacos 获取服务实例列表，缓存到本地
**④ 实时推送：** Nacos 基于长轮询（HTTP Long Polling）或 gRPC 推送变更，服务实例上下线时消费者及时感知
**⑤ 本地调用：** 消费者本地缓存实例列表，结合负载均衡（Round Robin / 权重）选一个实例直接发请求，不经过注册中心

**Nacos vs Eureka 核心差异：**

Nacos 同时支持 AP 模式（临时实例，服务发现用）和 CP 模式（持久实例，配置中心用），Eureka 只有 AP 模式。Nacos 还内置了配置管理，一套中间件两用。Eureka 已停止更新，新项目选 Nacos。

**配置示例（Spring Cloud Alibaba）：**

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-server:8848
        namespace: production
      config:
        server-addr: nacos-server:8848
        file-extension: yaml
```

###### 2. 当前有个本地操作 A，远程操作 B，我需要保证 A 和 B 事务的一致性，你会如何实现？

这道题是分布式事务的经典场景，描述的是本地 DB 操作和远程 RPC/HTTP 调用需要保持一致。

**方案一：本地消息表（最推荐，可靠性高）**

核心思路：A 和"发消息"这两步放在同一个本地事务里，通过异步消息驱动 B 操作。

```java
@Transactional
public void doLocalAndSendMessage(Order order) {
    // 1. 执行本地操作 A
    orderMapper.insert(order);
    
    // 2. 在同一事务内写入本地消息表（而不是直接发 MQ）
    MessageRecord msg = new MessageRecord("DEDUCT_STOCK", order.getProductId(), 
                                          order.getQuantity(), "PENDING");
    messageRecordMapper.insert(msg);
    // 事务提交：A 操作 + 消息记录 原子成功或失败
}

// 定时任务扫描消息表，发送未处理的消息
@Scheduled(fixedDelay = 1000)
public void sendPendingMessages() {
    List<MessageRecord> pending = messageRecordMapper.findPending();
    pending.forEach(msg -> {
        try {
            mqTemplate.send(msg); // 发送到 MQ
            messageRecordMapper.markSuccess(msg.getId());
        } catch (Exception e) {
            // 失败重试，下次定时任务继续处理
        }
    });
}
```

操作 B（库存扣减）的消费者做幂等处理，消费成功后删除/标记消息，实现最终一致性。

**方案二：RocketMQ 事务消息（更优雅）**

```java
// 发送半消息（Half Message）
TransactionSendResult result = rocketMQTemplate.sendMessageInTransaction(
    "deduct-stock-topic",
    MessageBuilder.withPayload(stockEvent).build(),
    null
);

// 本地事务执行器
@RocketMQTransactionListener
public class OrderTransactionListener implements RocketMQLocalTransactionListener {
    
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            orderMapper.insert(order); // 执行本地操作 A
            return RocketMQLocalTransactionState.COMMIT; // 通知 RocketMQ 提交半消息
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK; // 回滚，B 不执行
        }
    }
    
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        // RocketMQ 回查：检查本地操作 A 是否成功
        return orderMapper.existsById(orderId) ? COMMIT : ROLLBACK;
    }
}
```

**方案三：Seata AT 模式（通用分布式事务框架）**

```java
@GlobalTransactional
public void businessMethod() {
    localService.operationA(); // 本地 DB 操作
    remoteService.operationB(); // 远程服务调用
    // Seata 保证两者的原子性
}
```

Seata AT 通过 undo_log 实现回滚，对业务代码改动极小，一个注解搞定。但需要部署 Seata Server，且 undo_log 会有额外存储开销。

**选型建议：**
- 简单场景（两个操作，可以用 MQ 解耦）→ 本地消息表或 RocketMQ 事务消息
- 复杂多步分布式操作 → Seata AT 模式
- 高性能要求（如秒杀扣库存）→ TCC 模式（手动实现 Try-Confirm-Cancel）

###### 3. 横跨十几个分布式服务的慢请求要如何排查？

**① 全链路追踪系统（必备基础设施）**

首先要有完善的链路追踪：SkyWalking、Jaeger、Zipkin，国内常用 **SkyWalking**。

每个请求从入口（网关）开始生成一个全局 `TraceId`，通过 Header 传递，所有服务记录 `TraceId`、`SpanId`、服务名、开始时间、耗时、状态码。

```
Trace: abc123
├── API Gateway (0ms ~ 800ms): 总耗时800ms
│   ├── order-service (5ms ~ 300ms): 耗时295ms
│   │   ├── MySQL query (5ms ~ 250ms): 耗时245ms ← 慢SQL！
│   │   └── redis-get (251ms ~ 252ms): 耗时1ms
│   ├── stock-service (300ms ~ 400ms): 耗时100ms
│   └── notification-service (400ms ~ 800ms): 耗时400ms ← 慢！
```

通过 TraceId 在 SkyWalking 控制台直接可视化看到每个服务、每次 DB 调用的耗时，一目了然定位瓶颈。

**② 没有链路追踪时的手工排查步骤**

1. **确定入口耗时**：看 Nginx/网关的 access log，找到这个请求的总响应时间和 TraceId（如果有的话）
2. **逐服务排查**：根据 TraceId 或请求时间戳，在各服务日志中检索，看哪个服务耗时异常
3. **分析慢服务**：对慢服务做进一步分析：
   - 慢 SQL？查 MySQL `slow_query_log`
   - 线程阻塞？`jstack` 看线程 dump
   - 依赖下游慢？递归排查下游服务
   - GC 停顿？看 GC 日志
4. **网络问题**：`ping`/`traceroute` 检查网络延迟，`tcpdump` 抓包分析

**③ 诊断工具组合**

```bash
# 查看 SkyWalking 慢链路（P99 > 1s 的 Trace）
# 在 SkyWalking UI: Trace → 按响应时间排序

# 定位 JVM 问题（Arthas）
arthas> trace com.example.OrderService createOrder  # 方法级耗时分析
arthas> watch com.example.InventoryService deductStock returnObj  # 查看入参出参

# 数据库慢查询
SHOW VARIABLES LIKE 'slow_query_log%';
tail -f /var/log/mysql/slow.log | grep -A5 "Query_time"
```

**④ 建立 SLA 基线**

正常情况下记录各接口的 P50/P95/P99 响应时间基线，出现慢请求时有参照对比。Prometheus + Grafana 监控 HTTP 请求耗时分布。

###### 4. 分布式事务有哪些实现方案？各有什么优缺点？

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **2PC（两阶段提交）** | 协调者 Prepare + Commit 两阶段 | 强一致性 | 阻塞、单点故障、性能差 | 跨库操作，XA 标准 |
| **TCC** | Try-Confirm-Cancel 三阶段，业务侵入 | 高性能，灵活 | 代码改动大，需实现3个方法 | 高性能支付、库存 |
| **Seata AT** | 自动生成 undo_log，框架层实现回滚 | 业务无侵入（加注解即可） | 需要 Seata Server，有额外开销 | 通用业务场景 |
| **本地消息表** | 消息与本地事务共同提交，MQ 异步驱动 | 简单可靠，不依赖中间件 | 有延迟，最终一致性 | 非实时要求的业务 |
| **事务消息（RocketMQ）** | 半消息 + 本地事务 + 回查 | 可靠，自动回查 | 依赖 RocketMQ | 异步跨服务事务 |
| **Saga 模式** | 一系列本地事务 + 补偿事务（正向/反向） | 长事务，无长时间锁定 | 补偿逻辑复杂，难以实现 | 长业务流程（如退款） |

**重点掌握：**

**2PC（了解即可，实际少用）：**
- 第一阶段：协调者问所有参与者"能提交吗？"，参与者执行操作但不提交，返回 Yes/No
- 第二阶段：全部 Yes 则 Commit，有任何 No 则 Rollback
- 缺点：协调者单点故障导致参与者永久阻塞；所有参与者在第二阶段完成前都持有锁

**TCC（高性能场景用）：**
- Try：预留资源（冻结库存）
- Confirm：真正扣减（执行业务）
- Cancel：释放预留资源（解冻库存）
- 需要业务代码实现三个接口，开发成本高，但性能最好（资源锁定时间短）

**Seata AT（推荐日常使用）：**
- 框架自动解析 SQL，生成前后镜像存入 undo_log
- 提交时删除 undo_log；回滚时用 undo_log 反向操作还原数据
- 一个 `@GlobalTransactional` 注解搞定，业务代码无需修改

###### 5. 如何实现分布式 ID 生成？有哪些方案？

**方案一：雪花算法（Snowflake）——主流方案**

```
64位 = 1位符号位 + 41位毫秒时间戳 + 10位机器ID + 12位序列号
```

特点：
- 趋势递增，对 B+ 树索引友好（减少页分裂）
- 纯内存计算，每秒生成约400万个 ID
- 不依赖任何外部服务

关键问题——时钟回拨：

```java
// 检测到时钟回拨时的处理策略
if (currentTimestamp < lastTimestamp) {
    long offset = lastTimestamp - currentTimestamp;
    if (offset <= 5) {
        // 等待时钟追上（≤5ms 的小回拨）
        Thread.sleep(offset * 2);
    } else {
        // 大回拨直接报异常，触发告警
        throw new RuntimeException("时钟回拨超过5ms，拒绝生成ID");
    }
}
```

**生产级实现：** 美团 Leaf（双模式：雪花算法 + 号段模式）、百度 UidGenerator。

**方案二：号段模式（Segment）**

```sql
-- DB 中维护一个号段分配表
CREATE TABLE id_alloc (
    biz_type VARCHAR(64) PRIMARY KEY COMMENT '业务类型',
    max_id BIGINT NOT NULL COMMENT '当前最大ID',
    step INT NOT NULL COMMENT '每次步长',
    version INT NOT NULL COMMENT '乐观锁版本'
);
```

服务启动时从 DB 批量申请一段 ID（如每次取步长1000），内存中消费，消费完再申请下一段。
- 优点：简单可靠，ID 是纯递增数字
- 缺点：依赖 DB（虽然访问频率低），重启时有 ID 跳号（浪费一段未用的号段）

**方案三：Redis INCR**

```java
long id = redisTemplate.opsForValue().increment("global:order:id");
```

利用 Redis 原子自增。优点是简单；缺点是 Redis 持久化问题（RDB 可能丢失部分 INCR），Redis 重启后可能 ID 重复，需要配合持久化策略。

**方案四：UUID**

纯本地生成，无序，36个字符，对 InnoDB 聚簇索引性能差（频繁页分裂）。不推荐作为主键，但可以作为业务流水号使用。

**选型建议：**

| 需求 | 推荐方案 |
|------|----------|
| 高性能、分布式、趋势递增 | 雪花算法（Leaf/UidGenerator）|
| 纯递增、简单可靠 | 号段模式（Leaf Segment）|
| 有意义的业务单号 | 时间前缀 + 雪花ID + 校验码 |
| 只要唯一，不关心规律 | UUID（但不推荐做 DB 主键）|

###### 6. CAP 理论在实际项目中如何应用？

**CAP 理论**：分布式系统中，一致性（C，Consistency）、可用性（A，Availability）、分区容忍性（P，Partition Tolerance）三者不能同时满足，最多只能满足两个。

**现实情况是：P 无法舍弃**

网络分区（节点间通信故障）在分布式系统中是必然会发生的，所以 P 是前提，实际选择的是**CP 还是 AP**。

**CP 系统（一致性优先）**

```
节点 A ──×── 节点 B（网络分区）

CP 行为：节点 A 拒绝响应（牺牲可用性），直到分区恢复，确保读取一致
```

典型：ZooKeeper、Consul（Raft 算法，写操作需要多数派确认）

适用场景：分布式锁（绝对不能产生两个锁持有者）、金融交易（宁可不服务，不能数据不一致）

**AP 系统（可用性优先）**

```
节点 A ──×── 节点 B（网络分区）

AP 行为：A 和 B 都继续响应（牺牲一致性），分区期间数据可能不同步
```

典型：Eureka、Redis Sentinel（默认配置）、Cassandra、DynamoDB

适用场景：服务注册中心（实例多/少一个，只是负载均衡受影响，服务不可用代价更大）、商品库存展示（允许短暂不一致，最终一致即可）

**BASE 理论（CAP 的工程实践延伸）**

互联网系统大多选 AP + BASE（基本可用、软状态、最终一致性）：
- 允许系统在某段时间内数据不一致（软状态）
- 通过异步对账、MQ 消息最终达到一致（最终一致性）
- 在一致性和可用性之间取了个工程上的平衡点

**实际项目举例：**

- Nacos 服务发现模块：AP 模式（临时实例，哨兵心跳，可用性优先）
- Nacos 配置中心：CP 模式（持久化配置，一致性优先，Raft 协议）
- Redis 主从：AP（主节点故障时从节点可以读，但可能读到旧数据）
- MySQL 同步复制：CP（从库同步确认后才提交，牺牲性能换一致性）

###### 7. 如何保证分布式系统的幂等性？

**幂等性定义：** 同一个操作执行一次和多次，产生的效果相同。

重要性：网络重试、MQ 消费、用户重复点击，都可能导致同一操作被执行多次。

**实现方案：**

**① 唯一 ID + 数据库唯一索引（最通用）**

```java
// 请求中携带唯一请求ID（由客户端或网关生成）
public Result createOrder(CreateOrderRequest req) {
    // 利用唯一索引防重（idempotency_key 有唯一索引）
    try {
        orderMapper.insertWithIdempotencyKey(req.getIdempotencyKey(), req);
        return Result.success();
    } catch (DuplicateKeyException e) {
        // 重复请求，返回第一次的处理结果
        return orderMapper.findByIdempotencyKey(req.getIdempotencyKey()).getResult();
    }
}
```

**② Redis SetNX（高性能防重）**

```java
// 在 Redis 中设置幂等键，成功才执行
Boolean isNew = redisTemplate.opsForValue()
    .setIfAbsent("idempotent:" + requestId, "1", 5, TimeUnit.MINUTES);
if (!Boolean.TRUE.equals(isNew)) {
    return Result.duplicate("重复请求，已忽略");
}
// 执行业务逻辑
```

**③ 数据库乐观锁（防并发更新）**

```sql
-- 只有 version 匹配时才更新，同一 version 只能更新一次
UPDATE orders SET status = 'PAID', version = version + 1 
WHERE id = ? AND version = ? AND status = 'PENDING'
```

**④ 状态机（业务逻辑层幂等）**

```java
// 通过状态流转约束，拒绝非法的重复操作
if (order.getStatus() != OrderStatus.PENDING) {
    return; // 已经支付过，直接返回，不重复处理
}
order.setStatus(OrderStatus.PAID);
orderMapper.updateById(order);
```

**⑤ MQ 消费幂等**

```java
@RocketMQMessageListener(topic = "order-topic")
public class OrderConsumer implements RocketMQListener<OrderMessage> {
    
    @Override
    public void onMessage(OrderMessage msg) {
        // 用消息 ID 做幂等检查
        if (redisTemplate.opsForValue()
                .setIfAbsent("consumed:" + msg.getMsgId(), "1", 1, TimeUnit.DAYS)) {
            processOrder(msg); // 首次消费
        }
        // 重复消费：直接忽略
    }
}
```

**幂等性设计原则：**
- GET 请求天然幂等（只查不改）
- DELETE 操作可以设计成幂等（删除不存在的资源返回成功）
- POST（创建）和 PUT（更新）需要主动实现幂等
- 幂等 key 的存储时间要超过最大重试窗口期

###### 8. 分布式系统中如何实现限流？

**为什么需要限流：** 保护系统不被超过承载能力的流量压垮，平滑流量洪峰。

**四种限流算法：**

**① 固定窗口（计数器）**

每个时间窗口内计数，超过阈值拒绝请求。简单但有临界问题（两个窗口交界处可能放过双倍流量）。

**② 滑动窗口**

更精确的计数，记录每个请求的时间戳，窗口随时间滑动。Redis ZSet 实现：

```java
public boolean tryAcquire(String key, int limit, int windowSeconds) {
    long now = System.currentTimeMillis();
    long windowStart = now - windowSeconds * 1000;
    
    // 清理过期数据
    redisTemplate.opsForZSet().removeRangeByScore(key, 0, windowStart);
    // 计数
    Long count = redisTemplate.opsForZSet().count(key, windowStart, now);
    if (count < limit) {
        redisTemplate.opsForZSet().add(key, UUID.randomUUID().toString(), now);
        redisTemplate.expire(key, windowSeconds, TimeUnit.SECONDS);
        return true;
    }
    return false;
}
```

**③ 令牌桶（Token Bucket）——推荐**

以固定速率向桶中放入令牌，请求来时取令牌，有令牌就通过，没有就拒绝。允许一定程度的突发流量（桶中有积累的令牌）。Google Guava `RateLimiter` 就是令牌桶实现。

```java
// Guava RateLimiter：每秒100个请求
RateLimiter rateLimiter = RateLimiter.create(100.0);
if (rateLimiter.tryAcquire()) {
    // 处理请求
} else {
    // 限流拒绝
}
```

**④ 漏桶（Leaky Bucket）**

请求进入队列，以固定速率流出处理。平滑流量，不允许突发。适合需要严格控制处理速率的场景（如调用第三方 API 有 QPS 限制）。

**分布式限流（跨节点限流）：**

**Redis + Lua 脚本（原子操作）：**

```lua
-- 滑动窗口限流 Lua 脚本
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', key, 0, now - window * 1000)
local count = redis.call('ZCARD', key)
if count < limit then
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, window)
    return 1
end
return 0
```

**Sentinel（阿里巴巴，推荐生产使用）：**

```java
@SentinelResource(value = "orderCreate", 
                  blockHandler = "orderCreateBlocked",
                  fallback = "orderCreateFallback")
public Order createOrder(CreateOrderRequest req) { ... }

// 限流降级处理
public Order orderCreateBlocked(CreateOrderRequest req, BlockException ex) {
    throw new ServiceBusyException("系统繁忙，请稍后重试");
}
```

Sentinel 支持 QPS 限流、线程数限流、熔断降级、热点参数限流，控制台可视化配置，Spring Cloud 集成非常方便。
