# 十三、AI 与新技术

###### 1. 如果一个 GPU 集群的 LLM 处理能力为 1000 tokens/s，那 1000 个用户同时并发访问，响应给每个用户的性能只有 1 token/s 吗？怎么分析性能瓶颈？

**直觉分析：不是简单的平均分配**

"1000 tokens/s / 1000 用户 = 1 token/s"的分析是对的，但这只是最悲观情况。实际上：

1. **1000 个请求不会同时到达**（泊松分布，存在时间错开）
2. **每个请求的处理是批处理（Batching）**，GPU 有并行度，可以同时处理多个 token 生成
3. **流式输出（Streaming）**：用户看到的是逐 token 流式回答，首 token 延迟（Time to First Token, TTFT）比总吞吐量更重要

**LLM 推理的两个阶段和性能指标：**

- **Prefill 阶段（处理 Prompt）**：计算 KV Cache，高度并行，GPU 利用率高
- **Decode 阶段（逐 token 生成）**：自回归生成，每次只生成一个 token，串行性强，是吞吐瓶颈所在

**三个关键指标：**
- **TTFT（Time to First Token）**：用户发请求到看到第一个字的时间，影响体验
- **TBT（Time Between Tokens）**：相邻两个 token 之间的间隔，影响生成流畅感
- **吞吐量（Throughput）**：系统每秒处理的 token 总数（全局）

**如何分析 1000 QPS 的性能瓶颈：**

**第一步：区分是 Prefill 瓶颈还是 Decode 瓶颈**

- 长 prompt 短回答 → Prefill 是瓶颈（GPU 算力不足）
- 短 prompt 长回答 → Decode 是瓶颈（KV Cache 显存不足、自回归串行）

**第二步：分析显存（KV Cache）**

```
每个请求的 KV Cache 大小 ≈ 2 × num_layers × num_heads × head_dim × seq_len × dtype_size
```

A100 80GB 显存，模型占用 40GB，剩余 40GB 用于 KV Cache。如果每个请求需要 40MB KV Cache，最多同时处理 1000 个请求。

**第三步：优化手段**

- **Continuous Batching（连续批处理）**：不等一批全完成再开始下一批，动态插入新请求，提升 GPU 利用率（vLLM 核心优化）
- **PagedAttention**（vLLM）：KV Cache 用分页管理，减少内存碎片，同时处理的请求数提升数倍
- **量化（Quantization）**：int8/int4 量化，显存占用减半，可以放更多并发
- **多卡并行**：张量并行（TP）/ 流水线并行（PP）

**对用户的实际影响：** 1000 QPS 时每个用户感受不是 1 token/s，而是系统先把已有请求服务完，新请求在队列等待（排队延迟）。vLLM 的最大理论并发比朴素实现高 24x。

###### 2. 如何设计一个支持大模型推理的服务架构？

**挑战：** LLM 推理是 GPU 密集型且有长时流式输出，与传统 HTTP 请求-响应模式差异很大。

**整体架构：**

```
客户端 → API 网关（鉴权/限流/路由）
         → 推理调度层（队列/负载均衡/优先级）
         → 推理节点（vLLM/TGI，GPU 服务器）
         → 流式输出（SSE / WebSocket）

辅助服务：
  Prompt 管理/模板渲染
  RAG 检索服务（向量数据库）
  结果缓存（相同 Prompt 的结果）
  审计日志
```

**推理框架选型：**

- **vLLM**：高吞吐，PagedAttention，生产首选
- **TGI（Text Generation Inference）**：Hugging Face 出品，生态好
- **Ollama**：本地开发和轻量部署

**关键设计点：**

**① 流式输出（SSE）**

```java
@GetMapping(value = "/chat", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> chat(@RequestBody ChatRequest request) {
    return chatService.streamChat(request)
        .map(token -> ServerSentEvent.<String>builder()
            .data(token)
            .build());
}
```

**② 请求队列（控制并发）**

GPU 资源有限，多于并发上限的请求需要排队：

```java
// 限制推理并发数（如单卡最多处理50个并发请求）
private final Semaphore concurrencySemaphore = new Semaphore(50);

public void inference(InferenceRequest req) {
    if (!concurrencySemaphore.tryAcquire(5, TimeUnit.SECONDS)) {
        throw new ServiceBusyException("推理服务繁忙，请稍后重试");
    }
    try {
        vllmClient.generate(req);
    } finally {
        concurrencySemaphore.release();
    }
}
```

**③ RAG（检索增强生成）**

```java
// 用户问题 → 向量化 → 检索相关文档 → 拼入 Prompt → LLM 推理
String userQuestion = "我们公司的请假流程是什么？";
List<Document> relevant = vectorStore.similaritySearch(userQuestion, 5);
String context = buildContext(relevant);
String prompt = promptTemplate.format(context, userQuestion);
return llmClient.chat(prompt);
```

**④ 多模型路由**

```java
// 简单问题走小模型（快、便宜），复杂问题走大模型（准、慢）
ModelRouter router = question -> {
    if (question.length() < 50 && !question.contains("分析")) {
        return "qwen-7b"; // 7B 模型
    }
    return "qwen-72b"; // 72B 模型
};
```

###### 3. 如何优化大模型的推理性能？

**从多个维度优化：**

**① 模型量化（减少显存，提高吞吐）**

```bash
# GPTQ 量化（4bit），显存减少约75%，精度损失约1-2%
python -m auto_gptq --model-name qwen-72b --bits 4 --group-size 128

# AWQ（Activation-aware Weight Quantization），量化质量更好
```

**② 硬件加速**

- **Flash Attention 2**：重写 Attention 计算，减少 HBM 读写，提速 2-4x
- **CUDA Graphs**：减少 CPU-GPU 调度开销（对 Decode 阶段效果显著）
- **Speculative Decoding**：用小草稿模型快速生成多个候选 token，大模型一次验证多个，提升 decode 速度 2-3x

**③ 推理框架优化**

- **vLLM Continuous Batching**：动态批处理，GPU 利用率从 30% 提升到 80%+
- **PagedAttention**：KV Cache 分页管理，减少显存碎片，并发能力提升数倍
- **Tensor Parallelism**：多卡切分模型，每张卡处理部分参数

**④ 缓存优化**

- **KV Cache 缓存（Prefix Caching）**：相同前缀（如 System Prompt）的 KV Cache 复用，减少重复计算
- **语义缓存（Semantic Cache）**：将历史问答对存入向量库，相似问题直接返回缓存结果（GPTCache）

**⑤ 模型蒸馏/小模型**

用大模型的输出结果训练小模型（Student Model），特定任务上小模型性能接近大模型，但推理成本低 10x。

**性能目标参考（A100 80GB）：**

| 配置 | 吞吐量 |
|------|--------|
| 原始 Llama-70B FP16 | ~200 tokens/s |
| Llama-70B INT4 量化 | ~600 tokens/s |
| vLLM + Continuous Batching | ~1500 tokens/s（高并发场景）|

###### 4. 如何实现向量数据库的相似度搜索？

向量数据库用于存储和检索高维向量（Embedding），核心操作是**最近邻搜索（ANN，Approximate Nearest Neighbor）**。

**工作原理：**

```
文本/图片 → Embedding 模型（如 text-embedding-ada-002）→ 高维向量（如 1536维 float32）
→ 存入向量数据库（附带元数据）

查询时：
查询文本 → 向量化 → 在数据库中找 Top K 最相似向量（余弦相似度/欧式距离）
```

**主流向量数据库：**

| 数据库 | 特点 |
|--------|------|
| **Milvus** | 开源，高性能，生产首选 |
| **Qdrant** | Rust 实现，性能好，API 友好 |
| **Pinecone** | 全托管，开箱即用 |
| **Weaviate** | 内置向量化，多模态支持 |
| **pgvector** | PostgreSQL 插件，已有 PG 的场景 |
| **Redis** | 简单场景，支持向量搜索（RedisVL）|

**Spring AI + Qdrant 示例：**

```java
@Configuration
public class VectorStoreConfig {
    @Bean
    public VectorStore vectorStore(EmbeddingClient embeddingClient) {
        // Qdrant 向量数据库
        return new QdrantVectorStore(qdrantClient, embeddingClient);
    }
}

// 存储文档
vectorStore.add(List.of(
    new Document("这是第一篇文章的内容...", Map.of("source", "doc1")),
    new Document("另一篇关于Java的文章...", Map.of("source", "doc2"))
));

// 相似度搜索（返回最相关的5篇文档）
List<Document> results = vectorStore.similaritySearch(
    SearchRequest.query("如何学习Java").withTopK(5)
);
```

**ANN 算法（快速相似度搜索）：**

暴力搜索（计算所有向量的相似度）在百万级数据时太慢，ANN 算法用近似方法大幅提速：

- **HNSW（Hierarchical Navigable Small World）**：图结构索引，搜索速度极快（毫秒级），是最常用的 ANN 算法
- **IVF（Inverted File Index）**：向量聚类后建倒排索引，适合大规模数据
- **FAISS（Facebook AI）**：高性能 ANN 库，是很多向量数据库的底层

**向量搜索 + 关键词搜索（混合搜索）：**

```java
// 混合搜索：向量相似度 + BM25 关键词匹配，综合打分更准确
SearchResult hybridSearch(String query) {
    List<Document> vectorResults = vectorStore.similaritySearch(query, 20);
    List<Document> keywordResults = esClient.keywordSearch(query, 20);
    return mergeAndRerank(vectorResults, keywordResults); // RRF 重排
}
```
