# 二、缓存设计与优化

###### 1. MySQL 里有 2000w 数据，Redis 中只存 20w 的数据，如何保证 Redis 中的数据都是热点数据？

核心思路：**让 Redis 自动淘汰冷数据，保留热数据**。

**① 配置 Redis 内存淘汰策略**

在 `redis.conf` 中设置：

```conf
maxmemory 4gb                    # 限制 Redis 最大内存
maxmemory-policy allkeys-lru     # 淘汰策略
```

常用淘汰策略说明：
- `allkeys-lru`：对所有 key 使用 LRU 算法淘汰（**推荐用于缓存场景**）
- `allkeys-lfu`：对所有 key 使用 LFU（最近最少使用频率）淘汰，比 LRU 更精准
- `volatile-lru`：只对设置了 TTL 的 key 进行 LRU 淘汰
- `allkeys-random`：随机淘汰（几乎不用）

**② 给所有缓存 key 设置合理的 TTL**

```java
// 基础 TTL + 随机抖动，防止同一时刻大量 key 过期
int ttl = 3600 + ThreadLocalRandom.current().nextInt(600);
redisTemplate.opsForValue().set(key, value, ttl, TimeUnit.SECONDS);
```

不设 TTL 的 key 一旦写入就不会自然过期，容易堆积冷数据。

**③ 业务层主动识别热点（可选增强）**

统计访问频次（Redis INCR 记录 key 访问次数），定期分析报告，对高频访问的 key 主动延长 TTL，对低频 key 缩短或立即删除。

**④ 多级缓存降低 Redis 压力**

L1 本地缓存（Caffeine）缓存最热 TOP 1000 的数据，Redis 作为 L2 存热点数据，超出内存的长尾数据走 DB。

**面试重点：** 不是手动维护20w条——配置 `maxmemory` + `allkeys-lfu/lru`，让 Redis 自动保留访问最频繁的 key，这就是热点数据自动筛选的机制。

###### 2. 项目用了 Redis 缓存来提升并发度，假设 Redis 挂了怎么办？

这道题考的是**高可用设计**和**降级策略**，要从预防、发现、兜底三个层面回答。

**① 预防——Redis 高可用架构**

- **主从复制 + Sentinel（哨兵）**：主节点故障时哨兵自动选主，适合中等规模
- **Redis Cluster**：数据分片 + 自动故障转移，适合大规模场景
- **客户端多读节点**：读请求分散到多个从节点，单点故障影响面小

**② 发现——熔断与监控**

接入 Resilience4j 或 Sentinel 对 Redis 操作进行熔断，连续失败超过阈值后触发熔断，避免大量请求堆积等待超时把线程池打满。

```java
// Resilience4j CircuitBreaker 示例
CircuitBreaker circuitBreaker = CircuitBreaker.ofDefaults("redis");
Try.of(CircuitBreaker.decorateCheckedSupplier(circuitBreaker, 
    () -> redisTemplate.opsForValue().get(key)))
   .recover(e -> null) // 降级：返回 null，走 DB
   .get();
```

**③ 兜底——降级策略**

Redis 不可用时的降级方案（优先级从高到低）：

1. **本地缓存兜底**：L1 Caffeine 缓存可以撑一段时间，Redis 故障不会立刻击穿到 DB
2. **直接查 DB**：对核心业务允许降级查库，控制并发（加数据库连接池限流）
3. **返回默认值/兜底数据**：非核心业务（商品推荐、广告等）直接返回预设兜底数据
4. **服务降级**：对非核心功能直接关闭，返回"功能暂不可用"

**④ 恢复——快速回切**

Redis 恢复后，避免流量洪峰直接打到 DB，需要做缓存预热（见下题）。同时监控告警及时通知，控制恢复速度（逐步放流）。

**面试答法框架：** 预防（高可用架构）→ 感知（熔断/告警）→ 降级（本地缓存/直查DB/兜底数据）→ 恢复（预热+逐步放流）。

###### 3. 现在让你负责一个核心系统，需要实现核心数据的缓存预热，如何做？

缓存预热是指**系统启动或 Redis 恢复后，提前将热点数据加载到缓存**，避免冷启动时的缓存雪崩。

**方案一：启动时预热（适合数据量较小的核心数据）**

```java
@Component
public class CacheWarmupRunner implements ApplicationRunner {
    
    @Override
    public void run(ApplicationArguments args) throws Exception {
        log.info("开始缓存预热...");
        // 加载热门商品（TOP 1000）
        List<Product> hotProducts = productService.queryTop1000();
        hotProducts.forEach(p -> {
            int ttl = 7200 + ThreadLocalRandom.current().nextInt(600); // 随机抖动防雪崩
            redisTemplate.opsForValue().set("product:" + p.getId(), p, ttl, TimeUnit.SECONDS);
        });
        log.info("缓存预热完成，共加载 {} 条", hotProducts.size());
    }
}
```

用 `ApplicationRunner` 或 `CommandLineRunner`，在 Spring Boot 启动完成后、接受流量前执行预热。

**方案二：异步预热 + 分批加载（数据量大时）**

```java
@Async
public void warmupAsync() {
    // 分批加载，每批500条，避免一次性压垮 DB 和 Redis
    int page = 0, pageSize = 500;
    while (true) {
        List<Product> batch = productService.queryHotProducts(page++, pageSize);
        if (batch.isEmpty()) break;
        batch.forEach(p -> cacheProduct(p));
        Thread.sleep(100); // 控制预热速度
    }
}
```

**方案三：定时预热（热点数据定期刷新）**

```java
@Scheduled(cron = "0 0 3 * * ?") // 每天凌晨3点预热次日热点
public void scheduledWarmup() {
    // 分析昨天访问日志，提取 TOP N 热点 key
    List<String> hotKeys = analyzeHotKeys();
    hotKeys.forEach(key -> preloadToCache(key));
}
```

**方案四：懒加载 + 互斥锁防击穿（数据量太大，不适合全量预热）**

对于无法全量预热的场景，第一个缓存未命中时加分布式锁，只让一个请求穿透到 DB，其他请求等待或返回空，加载完成后所有请求从缓存取。

**预热顺序建议：** 核心配置数据（字典表、规则配置）→ 热门商品/活动 → 用户会话 → 次要数据。

**关键细节：** 预热时给不同 key 设置随机过期时间（TTL ± 随机抖动），防止预热的 key 同时到期引发雪崩。

###### 4. Redis 缓存穿透、缓存击穿、缓存雪崩分别是什么？如何解决？

**缓存穿透——查不存在的 key**

问题：请求的 key 在缓存和数据库中都不存在，每次都穿过缓存直打 DB，被恶意利用可以打垮 DB。

解决：
1. **空值缓存**：DB 查询为空时，也向 Redis 写入 `null`，TTL 设短一些（如60秒），下次同样的 key 直接从缓存返回 null，不打 DB
2. **布隆过滤器**：在缓存前加一层 Bloom Filter，对不存在的 ID 直接拦截，不请求 Redis 和 DB

```java
// 布隆过滤器（Redisson 实现）
RBloomFilter<Long> bloomFilter = redissonClient.getBloomFilter("product:ids");
bloomFilter.tryInit(10_000_000L, 0.01); // 1000w个元素，误差率1%

if (!bloomFilter.contains(productId)) {
    return null; // 直接拦截
}
// 再走缓存逻辑
```

**缓存击穿——热点 key 过期瞬间**

问题：某个热点 key 刚好过期，大量并发请求同时穿透到 DB，DB 瞬间压力飙升。

解决：
1. **互斥锁（分布式锁）**：缓存未命中时，只允许一个线程去 DB 加载，其他线程等待或重试
2. **逻辑过期（不设物理 TTL）**：在 value 中存储逻辑过期时间，过期后异步更新缓存，旧数据继续提供服务（牺牲一致性换取可用性）

```java
public Product getProduct(Long id) {
    String json = redis.get("product:" + id);
    if (json == null) {
        // 缓存不存在，加锁重建
        String lockKey = "lock:product:" + id;
        if (tryLock(lockKey)) {
            try {
                // 双重检查
                json = redis.get("product:" + id);
                if (json == null) {
                    Product p = db.findById(id);
                    redis.setex("product:" + id, 3600, JSON.toJSON(p));
                    return p;
                }
            } finally {
                unlock(lockKey);
            }
        } else {
            Thread.sleep(50);
            return getProduct(id); // 重试
        }
    }
    return JSON.parse(json, Product.class);
}
```

**缓存雪崩——大量 key 同时过期**

问题：大量缓存 key 在同一时刻集中过期，所有请求同时穿透到 DB，DB 压力骤增甚至崩溃。

解决：
1. **TTL 随机抖动**：设置过期时间时加随机量，`TTL = 基础时间 + random(0, 600)`，错开过期时间
2. **多级缓存**：L1 本地缓存（Caffeine）+ L2 Redis，Redis 雪崩时本地缓存可以撑一段时间
3. **Redis 高可用**：Sentinel/Cluster 防止 Redis 宕机引发雪崩
4. **限流降级**：在缓存层之前加限流，DB 压力超阈值时快速失败，保护 DB

**三者对比速记：**

| 问题 | 原因 | 核心解法 |
|------|------|----------|
| 穿透 | key 根本不存在 | 空值缓存 + 布隆过滤器 |
| 击穿 | 热点 key 过期 | 互斥锁 / 逻辑过期 |
| 雪崩 | 大量 key 同时过期 | TTL 随机抖动 + 多级缓存 |

###### 5. 如何设计一个多级缓存架构？

多级缓存的核心思想：**离用户越近，速度越快，容量越小；越远速度越慢，容量越大**。

**典型三级架构：**

```
请求 → L1 本地缓存（JVM 内存，Caffeine）
      ↓ 未命中
      → L2 分布式缓存（Redis）
      ↓ 未命中
      → L3 数据库 / 搜索引擎
```

**L1 本地缓存（Caffeine）**

```java
@Bean
public Cache<Long, Product> localCache() {
    return Caffeine.newBuilder()
        .maximumSize(10_000)          // 最多缓存1万条
        .expireAfterWrite(5, TimeUnit.MINUTES)
        .recordStats()               // 开启统计，监控命中率
        .build();
}
```

特点：纳秒级访问速度，零网络开销，但容量受限于 JVM 堆内存，多节点之间数据不一致。

**L2 Redis**

毫秒级访问，容量大，多节点共享，支持丰富的数据结构。适合热点数据的分布式共享缓存。

**Cache Aside 策略（读写逻辑）：**

```java
public Product getProduct(Long id) {
    // 1. 查本地缓存
    Product product = localCache.getIfPresent(id);
    if (product != null) return product;
    
    // 2. 查 Redis
    product = redisTemplate.opsForValue().get("product:" + id);
    if (product != null) {
        localCache.put(id, product); // 回填本地缓存
        return product;
    }
    
    // 3. 查 DB
    product = productMapper.selectById(id);
    if (product != null) {
        redisTemplate.opsForValue().set("product:" + id, product, 1, TimeUnit.HOURS);
        localCache.put(id, product);
    }
    return product;
}
```

**缓存一致性难题：**

多级缓存最大的挑战是一致性。常见方案：
- **更新时删除所有缓存**（懒加载，下次查询重建）
- **MQ 广播失效**：数据更新时发 MQ 消息，所有实例订阅并清除本地缓存

```java
// 数据变更时
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onProductUpdate(ProductUpdatedEvent event) {
    redisTemplate.delete("product:" + event.getProductId());
    mqTemplate.send("cache.invalidate", event.getProductId()); // 广播通知其他节点
}
```

**进阶：CDN 作为第零层缓存**

对于静态资源或不常变化的数据（商品图片、详情页），在 CDN 层加缓存，把请求拦截在边缘节点，减少回源请求。

###### 6. 缓存与数据库数据一致性如何保证？

这是缓存系统中最难的问题，核心矛盾：**如何让缓存和数据库的数据保持同步**。

**Cache Aside Pattern（旁路缓存）——最常用**

读：先读缓存 → 命中返回 → 未命中读 DB → 写入缓存
写：先写 DB → 再删除缓存（不是更新缓存）

为什么是**删除**而不是**更新**缓存？
- 更新缓存有竞态条件：A 写完 DB 还没更新缓存，B 也写完 DB 并更新了缓存，然后 A 用旧值覆盖了 B 的缓存值
- 删除是幂等的，下次查询自然重建，逻辑简单

**先写 DB 还是先删缓存？**

推荐：**先写 DB，再删缓存**（Cache Aside 标准做法）。

先删缓存的问题：删缓存后、写 DB 前，另一个线程读到 DB 旧值并写入缓存，之后 DB 才写入新值——缓存里是旧数据。

先写 DB 的问题：写 DB 成功，删缓存失败（网络抖动等）——缓存里是旧数据。解决方案：**延迟双删** 或 **异步删缓存**。

**延迟双删：**

```java
// 写 DB 前删一次
redisTemplate.delete(cacheKey);
// 写数据库
productMapper.updateById(product);
// 延迟一段时间再删一次（兜底，清除并发读写入的旧值）
scheduler.schedule(() -> redisTemplate.delete(cacheKey), 500, TimeUnit.MILLISECONDS);
```

**异步删除（Canal + MQ，强一致场景）：**

```
DB 变更 → MySQL Binlog → Canal 监听 → 发 MQ 消息 → 消费者删除/更新缓存
```

Canal 监听 MySQL binlog，感知数据变更，异步删除对应缓存。优点是不侵入业务代码；缺点是有一定延迟，在延迟窗口内缓存和 DB 不一致。

**实际项目的权衡：**

大多数业务场景接受**最终一致性**（Cache Aside + 合理 TTL），极少数场景（如账户余额）需要强一致性（读写串行化 + 分布式锁）。

###### 7. 用 Bitmap 来存储 100 个用户 ID，但是这个 ID 的范围很大，此时会有什么问题？如何解决？

**问题分析：**

Redis Bitmap 本质是字节数组，偏移量（offset）就是用户 ID。如果用户 ID 是雪花算法生成的（如 `1649000000000000001`），那么 `SETBIT user:login 1649000000000000001 1` 会分配一个偏移量约为 `1.6 × 10^18` 位的 Bitmap，换算成字节大约是 **200 PB**——Redis 直接 OOM。

**根本原因：** Bitmap 的内存消耗 = `最大偏移量 / 8` 字节，与存储的数据量无关，与 ID 范围有关。100 个用户 ID，但 ID 最大值是 10^18，内存就是 10^18/8 字节。

**解决方案：**

**方案一：ID 映射压缩（最常用）**

不直接用原始 ID 作为 offset，而是先将 ID 映射到一个连续的小序号：

```java
// 用 Redis Hash 存 ID → 序号 的映射
long seq = redisTemplate.opsForValue().increment("user:seq"); // 自增序号
redisTemplate.opsForHash().put("user:id2seq", String.valueOf(userId), seq);
// Bitmap 用序号作为 offset
redisTemplate.opsForValue().setBit("user:login", seq, true);
```

100个用户就只用 100 个 bit，极其紧凑。

**方案二：分桶 Bitmap**

将 ID 按范围分桶，每个桶是一个小 Bitmap：

```java
long bucketId = userId / 1_000_000; // 每桶100万
long offset = userId % 1_000_000;
String key = "user:login:bucket:" + bucketId;
redisTemplate.opsForValue().setBit(key, offset, true);
```

每个桶最多 1MB（100万/8），总内存取决于实际分布，避免超大稀疏 Bitmap。

**方案三：用 Set 或 HyperLogLog 替代**

如果只是 100 个用户，直接用 Redis Set 存储用户 ID 就行，比 Bitmap 简单直接。如果只需要估算去重数量（不需要精确），HyperLogLog 固定 12KB 解决亿级 UV 统计。

**方案四：RoaringBitmap（大数据量 + 大 ID 范围）**

RoaringBitmap 是一种压缩位图，对稀疏数据非常高效，Java 有现成的库（`org.roaringbitmap`）。适合在应用层处理，不适合直接放 Redis。

**面试总结：** 关键是意识到 Bitmap 的内存取决于最大 offset，而非元素数量。稀疏 Bitmap 内存浪费极其严重，解法是映射压缩（ID→小序号）或分桶。
