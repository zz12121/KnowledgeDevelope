# 八、高并发场景

###### 1. 如何在附近 100w 的商户中，快速找到离你最近的 5 家商户？

这道题考的是**地理位置检索**，核心数据结构是 **GeoHash** 或 **Redis GEO**。

**方案：Redis GEO（最简单实用）**

Redis GEO 底层基于 ZSet + GeoHash 实现，支持：
- `GEOADD`：添加地理坐标
- `GEORADIUS` / `GEODIST`：范围查询和距离计算
- `GEOPOS`：获取某个 key 的坐标

```java
// 初始化：批量导入商户坐标（一般在后台管理时维护）
List<RedisGeoCommands.GeoLocation<String>> locations = merchants.stream()
    .map(m -> new RedisGeoCommands.GeoLocation<>(
        String.valueOf(m.getId()), 
        new Point(m.getLongitude(), m.getLatitude())))
    .collect(toList());
redisTemplate.opsForGeo().add("geo:merchant", locations);

// 查询：以用户当前坐标为圆心，5km 以内最近的5家商户
Circle circle = new Circle(new Point(userLng, userLat), 
                            new Distance(5, Metrics.KILOMETERS));
GeoResults<RedisGeoCommands.GeoLocation<String>> results = redisTemplate.opsForGeo()
    .radius("geo:merchant", circle, 
             GeoRadiusCommandArgs.newGeoRadiusArgs()
                 .includeDistance()  // 返回距离
                 .sortAscending()    // 按距离从近到远
                 .limit(5));         // 取前5个
```

**GeoHash 原理：**

将经纬度编码成一个字符串（GeoHash），相邻区域的 GeoHash 前缀相同。查询"附近"本质是查 GeoHash 前缀匹配，走 ZSet 有序索引，时间复杂度 O(log N)。

**百万级数据的分区优化：**

单个 Key 存 100 万条数据性能没问题（ZSet 支持），但如果全国商户都在一个 Key 里，数据量很大。可以按城市/区域分拆：

```java
// 按城市分片，key = "geo:merchant:{cityCode}"
String key = "geo:merchant:" + cityCode;
```

用户查询时先确定所在城市，再查该城市的商户。

**补充：ElasticSearch Geo Point（更复杂场景）**

如果需要附加筛选条件（附近5km内评分>4.5星的商户，且支持外卖），可以用 ES 的 Geo Query：

```json
GET /merchants/_search
{
  "query": {
    "bool": {
      "must": { "term": { "supports_delivery": true }},
      "filter": {
        "geo_distance": {
          "distance": "5km",
          "location": {"lat": 39.9, "lon": 116.4}
        }
      }
    }
  },
  "sort": [{"_geo_distance": {"location": {"lat": 39.9, "lon": 116.4}, "order": "asc"}}],
  "size": 5
}
```

###### 2. 会员表中有 500 万条会员数据，有什么方式可以对即将过期的会员提前 7 天进行提醒？

**方案一：定时任务扫表（简单可靠）**

```java
@Scheduled(cron = "0 0 9 * * ?") // 每天上午9点执行
@Async
public void remindExpiringMembers() {
    LocalDate remindDate = LocalDate.now().plusDays(7);
    // 分批查询即将过期的会员（7天后到期）
    int page = 0, pageSize = 1000;
    while (true) {
        List<Member> batch = memberMapper.findExpiringMembers(remindDate, page++, pageSize);
        if (batch.isEmpty()) break;
        batch.forEach(m -> {
            notificationService.sendExpireReminder(m);
        });
        if (batch.size() < pageSize) break;
    }
}
```

```sql
-- 索引：expire_date 要有索引
SELECT * FROM member 
WHERE expire_date = #{remindDate} AND status = 'ACTIVE'
LIMIT #{offset}, #{pageSize}
```

优点：简单，数据量小时完全够用。缺点：500万数据每天全扫一次，I/O 压力大；凌晨任务时间长。

**优化：只查当天需要提醒的数据（精确到天），加 expire_date 索引，不是全表扫，而是范围查。**

**方案二：Redis ZSet 延迟队列（高效）**

在会员创建/续费时，将"到期日 - 7天"的时间戳作为 score，加入 ZSet：

```java
// 会员创建/续费时
public void onMemberCreated(Member member) {
    long remindTimestamp = member.getExpireDate().minusDays(7)
                                 .atStartOfDay()
                                 .toEpochSecond(ZoneOffset.UTC);
    redisTemplate.opsForZSet().add("member:remind", 
                                    String.valueOf(member.getId()), 
                                    remindTimestamp);
}

// 定时任务每小时扫描到期的提醒
@Scheduled(fixedDelay = 3600_000)
public void processReminders() {
    long now = Instant.now().getEpochSecond();
    // 查询 score <= now 的元素（已经到提醒时间）
    Set<ZSetOperations.TypedTuple<String>> members = 
        redisTemplate.opsForZSet().rangeByScoreWithScores("member:remind", 0, now, 0, 1000);
    members.forEach(tuple -> {
        Long memberId = Long.valueOf(tuple.getValue());
        notificationService.sendReminder(memberId);
        redisTemplate.opsForZSet().remove("member:remind", tuple.getValue());
    });
}
```

优点：只处理当天到期的记录，效率高，无全表扫描。

**方案三：RocketMQ 延迟消息**

会员创建时发送延迟消息（到期前7天投递），消费者收到后发提醒。

**方案四：MQ + Canal（数据变更驱动）**

Canal 监听 member 表的 `expire_date` 变更，计算提醒时间，向 MQ 发送延迟消息。

**实际项目建议：** 500万且每天只需跑一次：方案一（定时任务 + 合理索引）就够了，简单维护成本低。需要高精度提醒（精确到小时）或频繁续费场景，用方案二（Redis ZSet）。

###### 3. 商家想要知道自己店铺卖的最好的 Top 50 商品，如何实现这个功能？

**方案：Redis ZSet 实时排行榜**

```java
// 每次下单时，更新对应商品的销量 ZSet
public void onOrderCreated(Order order) {
    order.getItems().forEach(item -> {
        // 以店铺ID为命名空间，商品ID为成员，销量为分数
        String key = "rank:shop:" + order.getShopId();
        redisTemplate.opsForZSet().incrementScore(key, 
                                                   String.valueOf(item.getProductId()), 
                                                   item.getQuantity());
    });
}

// 查询 Top 50（分数从高到低）
public List<ProductSales> getTop50(Long shopId) {
    String key = "rank:shop:" + shopId;
    Set<ZSetOperations.TypedTuple<String>> top50 = 
        redisTemplate.opsForZSet().reverseRangeWithScores(key, 0, 49);
    
    return top50.stream().map(tuple -> {
        Long productId = Long.valueOf(tuple.getValue());
        long sales = tuple.getScore().longValue();
        Product product = productService.getById(productId);
        return new ProductSales(product, sales);
    }).collect(toList());
}
```

ZSet 的 `ZREVRANGEBYSCORE` 操作时间复杂度 O(log N + M)，百万级商品仍然很快。

**实时性与精确性的权衡：**

- **完全实时**（每单更新 ZSet）：销量实时精准，但写操作频繁（高并发下压力大）
- **定时更新**（每分钟/每小时统计一次，从 DB 查询写入 ZSet）：延迟几分钟，但 DB 和 Redis 数据可以对账，更可靠

**生产建议：**

1. 下单时异步更新 ZSet（通过 MQ 解耦，不影响下单主流程）
2. 定时任务每天凌晨从 DB 统计全天数据，校正 ZSet（防止因 Redis 故障丢失数据）
3. 多个时间维度：今日销量榜、本周销量榜、历史总销量榜，分别用不同 Key 维护

**分时段统计（今日榜、本周榜）：**

```java
// 今日榜 key 带日期
String key = "rank:shop:" + shopId + ":" + LocalDate.now().toString();
redisTemplate.expire(key, 8, TimeUnit.DAYS); // 保留8天数据
```

###### 4. 每秒 200 笔订单请求到支付服务，支付服务需调用第三方支付，限流 100 笔每秒。请你设计支付服务，要求分布式的，并且是 FIFO，不超过第三方限流量，但要用满第三方 100 笔每秒的量，请问如何设计？

**设计思路：消息队列 + 令牌桶限流 + 有序消费**

```
支付请求 (200 QPS)
    ↓
消息队列（单分区，保证 FIFO 顺序）
    ↓
消费者（令牌桶，100 QPS）→ 第三方支付 API (100 QPS)
    ↓
支付结果回调（异步通知订单服务）
```

**① 接收层：支付请求入队（异步解耦）**

```java
@PostMapping("/pay")
public ResponseEntity<PayResponse> pay(@RequestBody PayRequest req) {
    // 生成支付流水号，幂等 key
    String payNo = idGenerator.generate();
    
    // 将请求写入 MQ，立即返回"处理中"
    mqTemplate.send("payment-queue", new PayMessage(payNo, req));
    
    // 轮询或 WebSocket 通知支付结果
    return ResponseEntity.ok(PayResponse.processing(payNo));
}
```

**② MQ 单分区（保证 FIFO）**

使用 Kafka 或 RocketMQ，将支付请求发到**同一个 Partition**（或 RocketMQ 同一个 MessageQueue），保证 FIFO 顺序。

```java
// Kafka：指定 partition 0，全局有序
producer.send(new ProducerRecord<>("payment-queue", 0, key, value));
```

**③ 消费层：令牌桶控制速率**

```java
// 消费者以令牌桶限速，精确控制每秒消费100个
RateLimiter rateLimiter = RateLimiter.create(100.0); // 100 tokens/second

@KafkaListener(topicPartitions = @TopicPartition(topic="payment-queue", partitions="0"),
               concurrency = "1") // 单消费者，保证顺序
public void consumePayment(PayMessage msg) {
    rateLimiter.acquire(); // 阻塞直到拿到令牌（平滑限速到100 QPS）
    
    try {
        ThirdPartyPayResult result = thirdPartyPayClient.pay(msg);
        payResultService.handleSuccess(msg.getPayNo(), result);
    } catch (RateLimitException e) {
        // 第三方限流异常：延迟重试
        mqTemplate.sendDelay("payment-queue", msg, 1000);
    }
}
```

**④ 分布式场景（多消费者节点）**

如果单消费者 100 QPS 不够（第三方限流是全局 100 QPS），多节点消费时需要**分布式令牌桶**：

```java
// Redis 滑动窗口限流，所有消费者节点共享
public boolean tryAcquire(String key, int limit, int window) {
    String luaScript = "..."; // 滑动窗口 Lua 脚本
    return redisTemplate.execute(luaScript, List.of(key), limit, window, 
                                 System.currentTimeMillis()) == 1L;
}
```

**关键注意点：**

- 幂等处理：支付请求重试时不能重复扣款（以 payNo 做幂等 key）
- 超时处理：第三方支付超时（如30s），需要查询支付结果接口确认状态
- 消息保障：支付结果通过 MQ 异步通知订单服务，不能丢失
- 监控：实时监控消费队列积压量（Kafka Consumer Lag），积压过多告警

###### 5. 如何设计一个秒杀系统？

秒杀系统的核心挑战：**极短时间内超高并发，库存有限，不能超卖**。

**架构设计三原则：请求漏斗（层层过滤）、快速失败、异步化**

```
CDN（静态页面缓存）
    ↓
网关（IP限流/用户限流）
    ↓
秒杀服务（库存预扣/资格校验）
    ↓
MQ（削峰，异步创建订单）
    ↓
订单服务（真正创建订单/扣库存）
```

**① 流量层过滤**

- 静态页面提前推到 CDN，秒杀开始前不打后端
- 前端倒计时结束后才允许点击，减少提前请求
- 网关层对同一用户限流（1秒只允许1次请求）

**② 库存预扣（Redis）**

```java
// 初始化：将库存预加载到 Redis
redisTemplate.opsForValue().set("seckill:stock:" + productId, stockCount);

// 秒杀下单时，原子减库存
public boolean seckill(Long userId, Long productId) {
    // Lua 脚本原子操作：减库存 + 标记用户已购买（防止同一用户重复购买）
    String lua = 
        "local stock = redis.call('get', KEYS[1]) " +
        "if tonumber(stock) <= 0 then return -1 end " +
        "if redis.call('sismember', KEYS[2], ARGV[2]) == 1 then return -2 end " +
        "redis.call('decrby', KEYS[1], 1) " +
        "redis.call('sadd', KEYS[2], ARGV[2]) " +
        "return 1";
    
    Long result = redisTemplate.execute(new DefaultRedisScript<>(lua, Long.class),
        List.of("seckill:stock:" + productId, "seckill:buyers:" + productId),
        productId.toString(), userId.toString());
    
    return result != null && result == 1L;
}
```

**③ 异步创建订单（MQ）**

Redis 减库存成功后，向 MQ 发送"创建订单"消息，立即返回"排队中"给用户。订单服务消费 MQ 消息，真正写 DB 创建订单并扣减真实库存。

**④ DB 层兜底防超卖**

```sql
-- 数据库乐观锁保底，即使 Redis 出问题，DB 层也不会超卖
UPDATE seckill_stock SET stock = stock - 1 
WHERE product_id = ? AND stock > 0
```

**⑤ 防刷措施**

- 用户需要登录才能参与
- 同一用户限购1件（Redis Set 记录已购用户）
- 限流：网关 + Sentinel，单用户 1 QPS，单接口 N QPS
- 前端防刷：随机的请求参数 token（服务端颁发，一次有效）

###### 6. 如何设计一个抢红包系统？

抢红包的核心挑战：**并发分配随机金额、不超总额、高并发场景下的性能**。

**预拆包方案（最优）**

```java
// 发红包时：预先把金额拆好，存入 Redis List
public void createRedPacket(CreateRedPacketRequest req) {
    List<Integer> amounts = splitRedPacket(req.getTotalAmount(), req.getCount());
    
    String key = "redpacket:" + redPacketId;
    // amounts 是已拆好的随机金额列表（分，整数）
    redisTemplate.opsForList().leftPushAll(key, 
        amounts.stream().map(String::valueOf).toArray(String[]::new));
    redisTemplate.expire(key, 24, TimeUnit.HOURS); // 24小时过期
    
    // 同时存红包基本信息到 DB
    redPacketMapper.insert(redPacket);
}

// 抢红包时：原子 POP（先到先得，无并发问题）
public Integer grabRedPacket(Long redPacketId, Long userId) {
    String key = "redpacket:" + redPacketId;
    
    // 幂等检查：同一用户只能抢一次
    if (Boolean.FALSE.equals(redisTemplate.opsForSet()
            .add("redpacket:grabbed:" + redPacketId, String.valueOf(userId)))) {
        throw new AlreadyGrabbedException("您已经抢过了");
    }
    
    // LPOP 原子操作，线程安全
    String amountStr = redisTemplate.opsForList().leftPop(key);
    if (amountStr == null) {
        // 红包已被抢完，回滚刚加入的 userId
        redisTemplate.opsForSet().remove("redpacket:grabbed:" + redPacketId, userId);
        throw new RedPacketEmptyException("红包已抢完");
    }
    
    int amount = Integer.parseInt(amountStr);
    // 异步写入数据库记录
    mqTemplate.send("redpacket-record", new GrabRecord(redPacketId, userId, amount));
    return amount;
}
```

**随机拆包算法（二倍均值法，保证公平性）**

```java
public List<Integer> splitRedPacket(int totalAmount, int count) {
    List<Integer> amounts = new ArrayList<>();
    int remaining = totalAmount;
    
    for (int i = 0; i < count - 1; i++) {
        // 随机金额在 [1, 剩余金额/(剩余人数) * 2] 之间
        int max = remaining / (count - i) * 2;
        int amount = Math.max(1, ThreadLocalRandom.current().nextInt(max) + 1);
        amounts.add(amount);
        remaining -= amount;
    }
    amounts.add(remaining); // 最后一个拿剩余金额
    
    // 打乱顺序，避免最后一个人必然拿到特定金额
    Collections.shuffle(amounts);
    return amounts;
}
```

**异步入账（DB 操作异步化）**

抢红包成功后，通过 MQ 异步将记录写入 DB，不阻塞抢红包流程。

**过期处理：** 24小时未被抢完的红包，退款给发红包的用户。

###### 7. 如何防止超卖问题？

超卖是指商品库存为0后还继续出售，核心是**并发场景下的库存扣减原子性**。

**方案一：数据库乐观锁（低并发场景）**

```sql
-- 乐观锁：只有库存 > 0 时才更新（CAS 操作）
UPDATE inventory SET stock = stock - 1 
WHERE product_id = ? AND stock > 0

-- 检查 affected rows
if (affectedRows == 0) {
    throw new StockInsufficientException("库存不足");
}
```

并发情况下，只有一个请求能成功扣减，其他请求因 `stock > 0` 条件失败而得到正确的"库存不足"提示。简单可靠，适合并发量不高的场景。

**方案二：数据库悲观锁（SELECT FOR UPDATE）**

```java
@Transactional
public void deductStock(Long productId, int quantity) {
    // 加行锁，防止并发
    Inventory inventory = inventoryMapper.selectForUpdate(productId);
    if (inventory.getStock() < quantity) {
        throw new StockInsufficientException("库存不足");
    }
    inventory.setStock(inventory.getStock() - quantity);
    inventoryMapper.update(inventory);
}
```

适合对一致性要求极高的场景，但行锁会影响并发性能。

**方案三：Redis 原子操作（高并发场景首选）**

```java
// Lua 脚本原子扣减
String lua = 
    "local stock = tonumber(redis.call('get', KEYS[1])) " +
    "if stock == nil or stock < tonumber(ARGV[1]) then return 0 end " +
    "redis.call('decrby', KEYS[1], ARGV[1]) " +
    "return 1";

Long result = redisTemplate.execute(
    new DefaultRedisScript<>(lua, Long.class),
    List.of("stock:" + productId),
    String.valueOf(quantity));

if (result == 0) throw new StockInsufficientException("库存不足");
```

Redis 单线程模型保证 Lua 脚本原子执行，高并发下不会超卖。DB 层再加一层乐观锁兜底。

**方案四：分布式锁（保守方案）**

```java
RLock lock = redissonClient.getLock("lock:stock:" + productId);
if (lock.tryLock(1, 5, TimeUnit.SECONDS)) {
    try {
        checkAndDeductStock(productId, quantity);
    } finally {
        lock.unlock();
    }
}
```

串行化处理，彻底避免并发，但吞吐量低，适合极低并发场景。

**秒杀场景的最佳实践：**

Redis 预扣（乐观）→ MQ 异步下单 → DB 最终扣减（乐观锁兜底），三层保障。

###### 10. 你项目中做过高并发优化吗？是怎么做的？——高频面试引导问题

面试官问这个问题是想了解你是否有过**真实的线上高并发场景经验**，以及你是否能描述清楚问题、方案和效果。

**回答框架**：

1. **描述业务场景** - 什么业务？并发量多大？
2. **遇到了什么问题** - QPS多少？瓶颈在哪里？
3. **做了哪些优化** - 分层解决方案
4. **最终效果** - QPS提升多少？响应时间降低多少？

**常见优化案例**：

| 场景 | 问题 | 优化方案 | 效果 |
|------|------|---------|------|
| 热点商品查询 | DB压力过大 | Redis缓存 + 本地缓存二级缓存 | DB QPS从5000降到100 |
| 秒杀下单 | 库存超卖 | Redis Lua原子预扣 + MQ异步下单 | 0超卖，TPS提升10倍 |
| 首页接口 | 响应慢 | 接口拆分 + 异步加载 + CDN | 接口RT从2s降到200ms |
| 大量用户同时登录 | Session瓶颈 | Redis共享Session + 分布式锁 | 支持水平扩展 |
| 瞬时流量洪峰 | 服务被打挂 | 限流 + 熔断 + 降级 | 系统稳定性提升 |

**回答示例**：

> 我们之前做过秒杀系统的高并发优化。最初的问题是：秒杀活动开始时，10000 QPS直接打到数据库，数据库连接被打满，接口超时。
>
> 我们的优化方案是**分层拦截**：
> 1. **CDN层**：秒杀页面静态化，用户就近访问，不打后端
> 2. **Redis层**：活动开始前把库存加载到Redis，下单时Redis原子扣库存，成功后才写MQ
> 3. **MQ层**：削峰填谷，订单服务异步消费，数据库压力降低80%
> 4. **数据库层**：乐观锁兜底，防止Redis失效导致超卖
>

###### 8. 如何实现限流？有哪些限流算法？

见第四章第8题（分布式系统中的限流），此处补充**本地限流（单机）**的场景。

**Guava RateLimiter（单机 QPS 限流）**

```java
// 全局限流器（单例）
private final RateLimiter rateLimiter = RateLimiter.create(1000.0); // 每秒1000个

public Response handleRequest(Request req) {
    if (!rateLimiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
        return Response.tooManyRequests();
    }
    return processRequest(req);
}
```

**Resilience4j RateLimiter（Spring Boot 集成）**

```java
@RateLimiter(name = "orderApi", fallbackMethod = "orderApiFallback")
public Order createOrder(CreateOrderRequest req) { ... }

public Order orderApiFallback(CreateOrderRequest req, RequestNotPermitted ex) {
    throw new TooManyRequestsException("系统繁忙，请稍后重试");
}
```

```yaml
resilience4j:
  ratelimiter:
    instances:
      orderApi:
        limit-for-period: 100          # 每个周期最多100个请求
        limit-refresh-period: 1s        # 周期长度：1秒
        timeout-duration: 0s            # 等待超时：0秒（不等待，直接拒绝）
```

**四种算法速查：**

| 算法 | 特点 | 适用 |
|------|------|------|
| 固定窗口 | 简单，有临界突发问题 | 简单场景 |
| 滑动窗口 | 精确，无临界问题 | 精确限流 |
| 令牌桶 | 允许突发，平均限速 | API 限流（推荐）|
| 漏桶 | 严格匀速，不允许突发 | 控制处理速率 |

###### 9. 如何设计一个排行榜系统？

**Redis ZSet 是排行榜的标准解法**

```java
// 更新用户积分（加减分）
public void updateScore(String leaderboardKey, Long userId, double delta) {
    redisTemplate.opsForZSet().incrementScore(leaderboardKey, 
                                               String.valueOf(userId), delta);
}

// 查询 Top N
public List<RankItem> getTopN(String key, int n) {
    Set<ZSetOperations.TypedTuple<String>> top = 
        redisTemplate.opsForZSet().reverseRangeWithScores(key, 0, n - 1);
    
    return top.stream()
        .map(tuple -> new RankItem(Long.valueOf(tuple.getValue()), 
                                   tuple.getScore().longValue()))
        .collect(toList());
}

// 查询某用户的排名（从1开始）
public Long getUserRank(String key, Long userId) {
    Long rank = redisTemplate.opsForZSet().reverseRank(key, String.valueOf(userId));
    return rank == null ? null : rank + 1; // ZSet 排名从0开始，+1转为1开始
}
```

**多维度排行榜（如游戏中按战力+等级排序）**

ZSet score 是 double，精度限制。可以通过编码将多字段合并为一个 long：

```java
// 高32位存主排序字段（战力），低32位存次排序字段（等级）
// 同战力时，等级高的排在前面
long score = (long) power << 32 | level;
```

**分时段排行榜（日榜/周榜/月榜）**

```java
// Key 带时间维度
String dayKey = "rank:day:" + LocalDate.now().toString();
String weekKey = "rank:week:" + LocalDate.now().format(DateTimeFormatter.ofPattern("YYYY-ww"));
redisTemplate.expire(dayKey, 8, TimeUnit.DAYS);   // 保留8天
redisTemplate.expire(weekKey, 60, TimeUnit.DAYS);  // 保留60天

// ZUNIONSTORE 聚合周榜 = 7天日榜求并集+分数相加
```

**大规模排行榜（千万级用户）**

ZSet 支持百万级数据无压力。如果全球排行榜有数千万用户，可以分片：只把活跃用户（近7天有积分变动）放 ZSet，长期不活跃用户走 DB 查询。
