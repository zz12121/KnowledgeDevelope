# 五、消息队列

###### 1. 第三方上游接口接入做异步处理，是否可以使用 MQ？主要考虑什么？如果不用 MQ，还有什么方式？

**是否适合用 MQ？**

先问几个问题：
1. 这个操作是否必须同步完成？（用户下单后必须马上知道结果 → 不适合 MQ 异步）
2. 第三方接口是否有延迟/不稳定？（有延迟 → MQ 非常合适）
3. 失败后是否需要重试？（需要重试 → MQ 的重试机制很有价值）
4. 是否需要削峰？（流量突发 → MQ 做缓冲层）

**用 MQ 时主要考虑：**

**① 消息可靠性（不丢消息）**
- 生产者：开启 Producer Confirm/ACK，确认消息成功写入 Broker
- Broker：开启持久化（RabbitMQ 设置 durable=true；Kafka replication-factor≥2）
- 消费者：业务处理完成后再手动 ACK，避免消费失败但消息被标记成功

**② 幂等性（消息重复消费）**
MQ 的 at-least-once 语义决定了消息可能重复，消费者必须做幂等处理（上面第四章已详述）。

**③ 顺序性**
第三方接口调用是否有顺序要求？如果有，需要保证同一业务的消息路由到同一个 partition/queue（Kafka 同一 partitionKey，RocketMQ 同一 MessageQueue）。

**④ 死信队列**
消费失败超过重试次数后，消息进入死信队列（DLQ），便于后续人工处理或告警，不要让消息静默丢失。

**⑤ 消费能力匹配**
估算第三方接口的 QPS 上限，设置消费者并发数，避免超过第三方限流。可以结合漏桶算法控制消费速率。

**不用 MQ 的替代方案：**

**① 本地消息表（轻量级可靠异步）**
本地 DB 存消息，定时任务轮询发送，适合不想引入 MQ 中间件的场景。

**② 异步线程池（ThreadPoolExecutor）**
```java
@Async
public void asyncCall(Request req) {
    thirdPartyService.call(req);
}
```
简单但不可靠——JVM 宕机时内存中的任务丢失，无法重试。只适合无关紧要的操作（如发通知）。

**③ Spring Event（进程内异步）**
```java
applicationEventPublisher.publishEvent(new ThirdPartyCallEvent(data));
```
同进程内的异步解耦，本质还是线程池，和 `@Async` 有同样的可靠性问题。

**④ 定时任务扫表**
将待处理任务写入 DB，定时任务定期扫表调用第三方，失败可以重试，适合实时性要求不高的场景。

**面试结论：** 如果对可靠性有要求、需要重试、有流量波动，用 MQ 是合适的。MQ 的主要成本是引入了额外的中间件和复杂性（幂等、顺序、监控）。对简单低频场景，本地消息表可能更合适。

###### 2. 如何保证消息队列的消息不丢失？

消息从生产到消费，有三个环节可能丢失：**生产者 → Broker → 消费者**，每个环节都要处理。

**生产者端：确认机制**

**RabbitMQ — Publisher Confirm：**
```java
channel.confirmSelect();
channel.basicPublish(exchange, routingKey, props, body.getBytes());
if (!channel.waitForConfirms(5000)) {
    // 未确认，重发或报警
    log.error("消息发送失败，需要重试");
}
```

**Kafka — acks 参数：**
```java
props.put("acks", "all"); // 所有 ISR 副本确认才算成功（最强可靠性）
props.put("retries", 3);  // 发送失败自动重试
props.put("enable.idempotence", "true"); // 幂等发送，防止重试导致重复
```

**RocketMQ — 同步发送：**
```java
SendResult result = producer.send(message); // 同步发送，等待 Broker 响应
if (result.getSendStatus() != SendStatus.SEND_OK) {
    // 处理发送失败
}
```

**Broker 端：持久化**

- RabbitMQ：Exchange 和 Queue 设置 `durable=true`，消息 `deliveryMode=2`（持久化到磁盘）
- Kafka：`replication.factor=3`（3副本），`min.insync.replicas=2`（至少2个副本同步）
- RocketMQ：同步刷盘（`flushDiskType=SYNC_FLUSH`）确保消息落盘再响应

**消费者端：手动 ACK**

```java
// RabbitMQ 手动 ACK
@RabbitListener(queues = "order-queue", ackMode = "MANUAL")
public void consume(OrderMessage msg, Channel channel, 
                    @Header(AmqpHeaders.DELIVERY_TAG) long tag) {
    try {
        processOrder(msg);
        channel.basicAck(tag, false); // 确认消费成功
    } catch (Exception e) {
        // 处理失败，消息重入队列（requeue=true），或发送到死信队列（requeue=false）
        channel.basicNack(tag, false, true);
    }
}
```

**兜底：死信队列 + 告警**

消息多次重试仍失败后进入死信队列，触发告警通知人工介入，确保消息最终不会静默丢失。

**总结三段保证：**
```
生产者: 同步发送 + Confirm/ACK + 本地消息表兜底
Broker: 持久化 + 多副本 + 选主策略
消费者: 手动 ACK + 幂等处理 + 死信队列
```

###### 3. 如何保证消息的顺序性？

**为什么消息会乱序？**

多分区/多队列并行消费时，同一业务的消息可能被不同消费者处理，顺序无法保证。

**Kafka 顺序保证：**

Kafka 只保证同一个 Partition 内消息有序。

```java
// 发送时指定相同的 key（如 orderId），同一 key 路由到同一 Partition
producer.send(new ProducerRecord<>("order-topic", orderId.toString(), orderEvent));
```

消费时，每个 Partition 只分配给一个消费者（Consumer Group 内），单消费者串行消费保证顺序。

注意：Rebalance 发生时会短暂乱序，解决方案是 Partition Sticky Assignment（粘性分配）减少 Rebalance 频率。

**RocketMQ 顺序消费：**

```java
// 发送：同一订单的消息放入同一个 MessageQueue
SendResult result = producer.send(message, (mqs, msg, arg) -> {
    Long orderId = (Long) arg;
    int index = (int)(orderId % mqs.size());
    return mqs.get(index); // 同一 orderId 路由到同一 Queue
}, orderId);

// 消费：使用顺序消费监听器
@RocketMQMessageListener(
    topic = "order-topic",
    consumerGroup = "order-group",
    consumeMode = ConsumeMode.ORDERLY // 顺序消费
)
public class OrderConsumer implements RocketMQListener<OrderEvent> { ... }
```

**全局有序 vs 分区有序：**

- **全局有序**：只用一个 Partition/Queue，单消费者串行处理。吞吐量极低，基本不用
- **分区有序（实际采用）**：按业务 key（如 orderId）分区，同一业务的消息在同一分区内有序

**业务层面的顺序保证（防止网络乱序）：**

即使 MQ 保证了顺序投递，如果消费者处理失败并重试，重试的消息可能跑到后续消息前面。解决方案：

```java
// 在消息中加入序列号，消费时校验顺序
if (msg.getSeq() != expectedSeq) {
    // 消息乱序，延迟重试或放入等待队列
    throw new MessageOutOfOrderException();
}
```

###### 4. 如何处理消息积压问题？

**消息积压的成因：**
- 消费者处理速度跟不上生产速度
- 消费者宕机/重启，积压大量待消费消息
- 下游依赖（DB、第三方接口）变慢，消费者处理耗时增加

**紧急处理（快速消峰）：**

**① 扩充消费者（最直接）**

```bash
# Kafka：增加消费者实例（不超过 Partition 数量）
kubectl scale deployment order-consumer --replicas=20

# RabbitMQ：临时增加消费者线程数
# spring.rabbitmq.listener.simple.concurrency=20
# spring.rabbitmq.listener.simple.max-concurrency=50
```

注意：Kafka 消费者数量不能超过 Partition 数，超出的消费者会空转。如果 Partition 数不够，需要提前做好容量规划。

**② 临时扩容分区（Kafka）**

```bash
kafka-topics.sh --alter --topic order-topic --partitions 50 --bootstrap-server broker:9092
```

增加 Partition 数，让更多消费者并行消费。注意：已存在的消息按原有分区分布，扩容后新消息才会分布到新分区。

**③ 消费者快速转储**

如果消费者处理逻辑很重（如写 DB），可以先将消息快速转存到别处（另一个 Topic 或临时 DB 表），解除积压，再慢慢从临时存储消费处理。

**长期优化：**

**① 提升单消费者处理效率**
- 批量处理代替逐条处理（`spring.kafka.listener.type=batch`）
- 异步写 DB（批量 INSERT 代替单条）
- 减少不必要的业务逻辑

**② 增加 Partition/Queue 数量**
在架构设计阶段就要预留足够的分区数，Kafka 推荐每个 Broker 的 Partition 数在几百到一千以内。

**③ 消费者降级**
积压量超过阈值时，非核心消费逻辑（如积分累积、发通知）可以暂时跳过或降级，保证核心业务（下单、支付）优先消费。

**④ 监控告警**
提前配置消费延迟告警（Kafka Consumer Lag > N），提前介入，不要等到积压严重才发现：

```yaml
# Prometheus Alert Rule
- alert: KafkaConsumerLag
  expr: kafka_consumer_group_lag > 10000
  for: 5m
  annotations:
    summary: "Kafka消费积压超过10000条，需要处理"
```

###### 5. 如何保证消息的幂等性消费？

本质和前面"分布式幂等性"一样，但 MQ 场景有其特点：at-least-once 投递语义下，消息**必然会重复**，消费者必须设计幂等。

**常见重复消费场景：**
- 消费者处理完但 ACK 之前崩溃 → Broker 重投
- 消费者处理超时 → Broker 认为失败，重投
- 消费者网络超时 → 两条消息都被处理

**方案一：数据库唯一索引（最可靠）**

```java
public void consumeOrder(OrderCreatedEvent event) {
    try {
        // 以 msgId 或业务唯一键为幂等键，唯一索引防重复
        orderMapper.insertIgnoreDuplicate(event.getOrderId(), event.getMsgId(), event);
    } catch (DuplicateKeyException e) {
        log.info("重复消费，忽略。msgId: {}", event.getMsgId());
    }
}
```

**方案二：Redis 记录已消费的消息 ID**

```java
public void onMessage(Message message) {
    String msgId = message.getMsgId();
    // SET NX 原子操作
    Boolean isFirst = redisTemplate.opsForValue()
        .setIfAbsent("consumed:" + msgId, "1", 24, TimeUnit.HOURS);
    if (Boolean.FALSE.equals(isFirst)) {
        log.info("重复消息，跳过: {}", msgId);
        return;
    }
    processMessage(message);
}
```

**方案三：状态机检查（业务逻辑层）**

```java
public void consumeOrderPaid(OrderPaidEvent event) {
    Order order = orderMapper.selectById(event.getOrderId());
    if (order.getStatus() != OrderStatus.PENDING_PAYMENT) {
        // 已经处理过（状态已经不是"待支付"），直接返回
        return;
    }
    // 幂等更新：只有 PENDING_PAYMENT 状态才能更新
    int rows = orderMapper.updateStatus(event.getOrderId(), 
        OrderStatus.PENDING_PAYMENT, OrderStatus.PAID);
    if (rows == 0) {
        // 并发导致的重复，已被其他线程处理
        return;
    }
}
```

**方案四：版本号乐观锁**

```sql
UPDATE orders SET status = 'PAID', version = version + 1
WHERE order_id = ? AND version = ? AND status = 'PENDING_PAYMENT'
```

只有 version 匹配时才更新，并发或重复消费时只有一次能成功。

**幂等设计要点：**
- 消息 ID（`msgId`）是天然的幂等键，各 MQ 框架都有
- 结合业务语义设计幂等（状态机是最自然的方式）
- 幂等检查要和业务操作保证原子性（用 DB 事务或 Redis Lua 脚本）

###### 6. RabbitMQ、Kafka、RocketMQ 如何选型？

**三者核心对比：**

| 维度 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| 定位 | 消息中间件（AMQP） | 分布式日志/流处理 | 金融级消息队列 |
| 吞吐量 | 万级/s | 百万级/s | 十万级/s |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 消息顺序 | 单队列有序 | 分区内有序 | 分区内有序 |
| 延迟消息 | 需要插件 | 不支持 | 原生支持（18个精度级别）|
| 事务消息 | 不支持 | 支持 | 原生支持 |
| 死信队列 | 支持 | 不支持（需自己实现）| 支持 |
| 运维复杂度 | 中（有管理控制台）| 高（依赖 ZooKeeper）| 中（NameServer）|
| 社区生态 | 成熟（Spring 支持好）| 最活跃（大数据生态）| 阿里巴巴（国内活跃）|

**选型建议：**

**选 RabbitMQ，如果：**
- 需要复杂的消息路由（Exchange + Binding，如 Topic/Fanout/Direct 路由）
- 团队熟悉 AMQP 协议
- 需要低延迟消息传递（微秒级）
- 消息量不大（日均百亿以内）

**选 Kafka，如果：**
- 需要极高吞吐（日志收集、用户行为数据、流处理）
- 需要消息持久化很长时间（日志审计、数据管道）
- 对接大数据生态（Flink、Spark、Hadoop）
- 需要精确的消息重放（消费者可以重置 offset）

**选 RocketMQ，如果：**
- 在 Java/Spring 生态，对国内支持要求高
- 需要事务消息（分布式事务）
- 需要精确延迟消息（定时任务场景）
- 金融/电商场景，对消息可靠性要求极高

**实际项目经验：** 传统 Java 项目用 RabbitMQ；电商/金融核心业务用 RocketMQ；大数据管道、日志收集用 Kafka。不要只看性能数字，也要看团队熟悉度和运维成本。

###### 7. 消息队列如何实现延迟消息？

延迟消息是指消息发送后，延迟一段时间才被消费者接收到。常见场景：订单超时取消（30分钟未支付自动关闭）、定时发送通知。

**RocketMQ（原生支持延迟消息）**

```java
Message msg = new Message("order-delay-topic", orderEvent.getBytes());
// 延迟级别：1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h
msg.setDelayTimeLevel(16); // 第16级 = 30分钟
producer.send(msg);
```

RocketMQ 5.x 支持任意精度延迟（不再限于18个级别）：

```java
msg.setDelayTimeMills(30 * 60 * 1000L); // 30分钟后投递
```

**RabbitMQ（插件：rabbitmq-delayed-message-exchange）**

```java
// 安装插件后，声明一个 x-delayed-message 类型的 Exchange
Map<String, Object> args = new HashMap<>();
args.put("x-delayed-type", "direct");
channel.exchangeDeclare("delayed-exchange", "x-delayed-message", true, false, args);

// 发送时在 Header 中指定延迟时间（毫秒）
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .headers(Collections.singletonMap("x-delay", 30 * 60 * 1000))
    .build();
channel.basicPublish("delayed-exchange", "order-timeout", props, body);
```

**Kafka（不原生支持，需要应用层实现）**

常见方案：
1. **多 Topic 分级延迟**：创建 `delay-1min`、`delay-5min`、`delay-30min` 等 Topic，消息发到对应 Topic，消费者轮询检查是否到期
2. **时间轮 + 外部存储**：消息存入 Redis（ZSet，score=到期时间戳），时间轮扫描到期消息，再推到真正的消费 Topic

**基于 Redis ZSet 实现通用延迟队列：**

```java
// 生产者：消息写入 ZSet，score=执行时间戳
public void addDelayedJob(String jobData, long delayMs) {
    double score = System.currentTimeMillis() + delayMs;
    redisTemplate.opsForZSet().add("delay-queue", jobData, score);
}

// 消费者：定时扫描到期消息
@Scheduled(fixedDelay = 1000)
public void processDelayedJobs() {
    long now = System.currentTimeMillis();
    Set<String> jobs = redisTemplate.opsForZSet()
        .rangeByScore("delay-queue", 0, now, 0, 100);
    jobs.forEach(job -> {
        // 原子删除，防止并发消费
        Long removed = redisTemplate.opsForZSet().remove("delay-queue", job);
        if (removed == 1) {
            processJob(job);
        }
    });
}
```

**选型建议：** 有 RocketMQ 就用原生延迟消息（最简单）；只有 Kafka 或需要精确任意延迟的场景，用 Redis ZSet + 定时扫描；数据量小可以用 DB + 定时任务。
