# 十、数据处理

###### 1. ArrayList 里有 1 亿条数据，要怎么去重？

面试场景题，考的是**大数据量去重的思维**，不同场景有不同答案。

**内存充足时：HashSet 去重**

```java
List<String> list = ...; // 1亿条数据
Set<String> set = new HashSet<>(list.size());
set.addAll(list);
List<String> deduplicated = new ArrayList<>(set);
```

直接将数据放入 HashSet，利用哈希表天然去重。
内存占用估算：1亿条字符串（假设每条10字节），HashSet 约需 3-5 GB 内存（哈希表有额外空间开销）。如果 JVM 堆足够，这是最简单的方案。

**内存不足时：分块 + 哈希分桶**

```java
// 1. 按 hashCode 分成 N 份（假设分100桶）
List<List<String>> buckets = new ArrayList<>();
for (int i = 0; i < 100; i++) buckets.add(new ArrayList<>());

for (String item : list) {
    int bucketIdx = Math.abs(item.hashCode() % 100);
    buckets.get(bucketIdx).add(item);
}

// 2. 对每个桶单独去重（每桶约100万条，内存可接受）
List<String> result = new ArrayList<>();
for (List<String> bucket : buckets) {
    Set<String> bucketSet = new HashSet<>(bucket);
    result.addAll(bucketSet);
}
```

相同的元素一定落在同一个桶里，每个桶内去重，最后合并。

**极限内存场景：外排序 + 流式处理**

数据写到磁盘，按哈希值分片，每次只加载一个分片到内存去重，写回磁盘。类似 MapReduce 的思想。

**并行去重（多核加速）：**

```java
// Java 8 parallel stream + ConcurrentHashMap 去重
Map<String, Boolean> seen = new ConcurrentHashMap<>();
List<String> deduplicated = list.parallelStream()
    .filter(item -> seen.putIfAbsent(item, Boolean.TRUE) == null)
    .collect(Collectors.toList());
```

**布隆过滤器（允许一定误判率）：**

如果可以接受极小概率的误判（把不重复的判为重复），布隆过滤器内存消耗极小（1亿数据约 240MB）。

###### 2. 如何快速定位到五分钟内重复登录了两次的 QQ 号？

**数据结构分析：**

- 需要记录每个 QQ 号的登录时间序列
- 需要查找：在任意5分钟窗口内有≥2次登录的 QQ 号
- 数据量大（每秒可能百万级登录事件）

**方案一：Redis Hash + 滑动时间窗口（实时检测）**

```java
// 每次登录事件处理
public void onLogin(String qqId, long loginTimestamp) {
    String key = "login:times:" + qqId;
    long windowStart = loginTimestamp - 5 * 60 * 1000; // 5分钟前
    
    // 用 ZSet：member=timestamp，score=timestamp
    redisTemplate.opsForZSet().add(key, String.valueOf(loginTimestamp), loginTimestamp);
    // 清理5分钟前的旧记录
    redisTemplate.opsForZSet().removeRangeByScore(key, 0, windowStart);
    // 检查窗口内登录次数
    Long count = redisTemplate.opsForZSet().count(key, windowStart, loginTimestamp);
    
    if (count >= 2) {
        // 触发告警或风控处理
        alertService.sendAlert(qqId, "5分钟内多次登录");
    }
    
    redisTemplate.expire(key, 10, TimeUnit.MINUTES); // 10分钟后自动清理
}
```

**方案二：离线批处理（Flink / Spark）**

适合日志文件分析场景（事后分析，不是实时检测）：

```java
// Flink：5分钟滑动窗口，统计每个QQ号登录次数
DataStream<LoginEvent> loginStream = ...;

loginStream
    .keyBy(LoginEvent::getQqId)
    .window(SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1))) // 5分钟窗口，1分钟滑动
    .process(new ProcessWindowFunction<LoginEvent, Alert, String, TimeWindow>() {
        @Override
        public void process(String qqId, Context ctx, 
                           Iterable<LoginEvent> events, Collector<Alert> out) {
            long count = StreamSupport.stream(events.spliterator(), false).count();
            if (count >= 2) {
                out.collect(new Alert(qqId, "多次登录", ctx.window().getEnd()));
            }
        }
    });
```

**方案三：SQL 方式（数据量较小时）**

```sql
-- 找出5分钟内登录≥2次的QQ号
SELECT qq_id, COUNT(*) as login_count
FROM login_log
WHERE login_time >= DATE_SUB(NOW(), INTERVAL 5 MINUTE)
GROUP BY qq_id
HAVING COUNT(*) >= 2
```

###### 3. 现在有一天内的大量日志，每条日志记录了 {userid, login_time, logout_time}，时间单位是秒。怎么求出一天内的最大在线人数，以及维持最大在线人数的最长持续时间？

**算法思路：事件扫描法（差分思想）**

将每个用户的在线时间段拆分成两个事件：
- login_time：+1（在线人数加1）
- logout_time：-1（在线人数减1）

将所有事件按时间排序后扫描，维护当前在线人数，找最大值。

```java
public class MaxOnlineAnalyzer {
    
    public Result analyze(List<Log> logs) {
        // 1. 构造时间点事件
        List<int[]> events = new ArrayList<>(); // [时间戳, 类型(1=登录/-1=登出)]
        for (Log log : logs) {
            events.add(new int[]{log.loginTime, 1});   // 登录：+1
            events.add(new int[]{log.logoutTime, -1}); // 登出：-1
        }
        
        // 2. 按时间排序（同时刻：登出事件先处理，避免瞬间人数虚高）
        events.sort((a, b) -> a[0] != b[0] ? Integer.compare(a[0], b[0]) 
                                             : Integer.compare(a[1], b[1])); // -1 < 1，登出先
        
        // 3. 扫描求最大值和最长持续时间
        int maxOnline = 0;
        int current = 0;
        int maxStart = -1, maxEnd = -1;
        int maxDuration = 0;
        int currentMaxStart = -1;
        
        for (int[] event : events) {
            current += event[1];
            
            if (current > maxOnline) {
                maxOnline = current;
                currentMaxStart = event[0]; // 新的最大值开始时间
                maxDuration = 0;
            }
            
            if (current == maxOnline) {
                if (event[1] == -1) { // 下降，这段最大区间结束
                    int duration = event[0] - currentMaxStart;
                    if (duration > maxDuration) {
                        maxDuration = duration;
                        maxStart = currentMaxStart;
                        maxEnd = event[0];
                    }
                }
            }
        }
        
        return new Result(maxOnline, maxDuration, maxStart, maxEnd);
    }
}
```

**另一种更简洁的思路：时间轴计数**

如果时间范围不大（一天 = 86400秒），可以用数组直接计数：

```java
int[] timeline = new int[86400]; // 每秒的在线人数变化量

for (Log log : logs) {
    timeline[log.loginTime]++;
    if (log.logoutTime < 86400) timeline[log.logoutTime]--;
}

// 前缀和求每秒在线人数
int maxOnline = 0, currentOnline = 0;
for (int sec = 0; sec < 86400; sec++) {
    currentOnline += timeline[sec];
    maxOnline = Math.max(maxOnline, currentOnline);
}
```

时间复杂度 O(N + T)，N 是日志条数，T 是时间范围（86400）。

###### 4. 如何实现海量数据的 Top K 问题？

**经典解法：小顶堆（Min-Heap）**

维护一个大小为 K 的小顶堆，遍历所有数据，堆顶是当前前 K 个数中最小的：

```java
public List<Integer> topK(int[] data, int k) {
    // 小顶堆（默认）
    PriorityQueue<Integer> minHeap = new PriorityQueue<>(k);
    
    for (int num : data) {
        if (minHeap.size() < k) {
            minHeap.offer(num);
        } else if (num > minHeap.peek()) {
            // 当前数比堆中最小的大，替换
            minHeap.poll();
            minHeap.offer(num);
        }
    }
    
    return new ArrayList<>(minHeap); // 堆中 K 个数就是 Top K
}
```

时间复杂度：O(N log K)，空间复杂度：O(K)。适合单机内存可以容纳所有数据的场景。

**分布式场景（数据分布在多台机器）：**

1. 每台机器独立计算本地 Top K
2. 将各机器的 Top K 汇总（总共 M×K 个数据）
3. 再对汇总数据跑一次 Top K

如 10 台机器，每台取 Top 100，汇总 1000 个数据后取全局 Top 100。

**Redis ZSet（实时 Top K）：**

```java
// 每次操作后更新分数
redisTemplate.opsForZSet().incrementScore("rank:product:sales", productId, quantity);
// 取 Top K
Set<ZSetOperations.TypedTuple<String>> topK = 
    redisTemplate.opsForZSet().reverseRangeWithScores("rank:product:sales", 0, k - 1);
```

###### 5. 如何实现海量数据的去重？

**场景：数据量超过内存限制，无法全部加载**

**方案一：布隆过滤器（允许极小误判率）**

```java
// Google Guava BloomFilter
BloomFilter<String> bf = BloomFilter.create(
    Funnels.stringFunnel(StandardCharsets.UTF_8),
    100_000_000, // 期望元素数量：1亿
    0.001        // 允许误判率：0.1%（约1.2GB内存）
);

for (String item : dataStream) {
    if (bf.mightContain(item)) {
        // 可能重复（有0.1%误判率）
        continue;
    }
    bf.put(item);
    process(item); // 只处理新数据
}
```

内存占用：1亿元素，1%误判率约需 120MB；0.1%误判率约需 175MB。远小于存储1亿个字符串的内存。

**方案二：哈希分桶 + 逐桶去重（精确去重）**

```java
// 按哈希值将数据分到 N 个文件（桶）
int N = 100;
List<BufferedWriter> writers = new ArrayList<>();
for (int i = 0; i < N; i++) {
    writers.add(new BufferedWriter(new FileWriter("bucket_" + i + ".txt")));
}

for (String item : bigDataSource) {
    int bucket = Math.abs(item.hashCode() % N);
    writers.get(bucket).write(item + "\n");
}
writers.forEach(w -> w.close());

// 对每个桶（约总数据量/N）加载到内存，用 HashSet 去重
Set<String> result = new HashSet<>();
for (int i = 0; i < N; i++) {
    Set<String> bucketSet = new HashSet<>(Files.readAllLines(Path.of("bucket_" + i + ".txt")));
    result.addAll(bucketSet);
}
```

相同元素一定分到同一个桶，每桶内存去重，最后合并。

**方案三：排序去重（适合可排序数据）**

外排序（将数据分块排序，多路归并），排序后相同元素相邻，线性扫描去重：

```
大文件 → 分块排序（内存可承受的块）→ 多路归并 → 有序大文件 → 扫描去重
```

###### 6. 如何判断一个元素是否在海量数据中存在？

**布隆过滤器（BloomFilter）——最优解**

在第5题中已介绍，这里补充原理：

布隆过滤器是一个 m 位的比特数组 + k 个哈希函数：

```
插入元素 x：
  对 x 用 k 个不同哈希函数计算 k 个位置，将这些位置的 bit 置 1

查询元素 y：
  对 y 计算 k 个位置，如果所有位置都是 1 → 可能存在（有误判）
                        如果任意一个位置是 0 → 一定不存在
```

**特性：**
- 不存在 → 必定返回"不存在"（零误判）
- 存在 → 可能返回"不存在"（有误判率，但可以控制）
- **只能添加元素，不能删除**（删除会影响其他元素的查询）

**需要删除的场景：Counting Bloom Filter**

每个位置不是 bit 而是计数器，删除时计数器减1。空间翻 4 倍，但支持删除。

**Redis BitMap（小ID范围场景）**

当元素是整数且 ID 范围不大时，Redis BitMap 是极简方案：

```java
// 设置：ID 1000 的用户已处理
redisTemplate.opsForValue().setBit("processed:users", 1000L, true);

// 查询：ID 1000 是否已处理
boolean processed = redisTemplate.opsForValue().getBit("processed:users", 1000L);
```

1亿个 bit = 12.5MB，如果 ID 范围在1亿以内，极其节省空间。

###### 7. 如何实现大文件的上传和下载？

**大文件上传：分片上传**

```java
// 前端将文件切片（每片 5MB），分批上传
// 后端接口：接收分片

@PostMapping("/upload/chunk")
public ResponseEntity<ChunkResult> uploadChunk(
        @RequestParam("file") MultipartFile chunkFile,
        @RequestParam String fileHash,    // 文件整体的 MD5（用于合并和秒传）
        @RequestParam int chunkIndex,     // 当前分片序号
        @RequestParam int totalChunks     // 总分片数
) {
    String chunkPath = tempDir + "/" + fileHash + "_" + chunkIndex;
    chunkFile.transferTo(new File(chunkPath));
    
    // 记录已上传的分片（Redis Set）
    redisTemplate.opsForSet().add("upload:chunks:" + fileHash, String.valueOf(chunkIndex));
    
    return ResponseEntity.ok(new ChunkResult(chunkIndex, "uploaded"));
}

// 合并分片
@PostMapping("/upload/merge")
public ResponseEntity<String> mergeChunks(
        @RequestParam String fileHash,
        @RequestParam int totalChunks,
        @RequestParam String fileName) {
    
    // 检查所有分片是否都上传了
    Long uploadedCount = redisTemplate.opsForSet().size("upload:chunks:" + fileHash);
    if (!uploadedCount.equals((long) totalChunks)) {
        return ResponseEntity.badRequest().body("分片未上传完整");
    }
    
    // 合并文件
    mergeChunks(fileHash, totalChunks, fileName);
    redisTemplate.delete("upload:chunks:" + fileHash);
    return ResponseEntity.ok("上传成功");
}
```

**秒传（MD5 查重）：**

上传前先将文件 MD5 发给服务端，服务端检查是否已存在，存在则直接返回成功，不需要重新上传。

**断点续传：** 上传前查询已上传的分片列表，只上传缺失的分片。

**大文件下载：HTTP Range 分段下载**

```java
@GetMapping("/download/{fileId}")
public ResponseEntity<Resource> download(
        @RequestHeader(value = "Range", required = false) String range,
        @PathVariable Long fileId,
        HttpServletResponse response) {
    
    File file = fileService.getFile(fileId);
    long fileLength = file.length();
    
    if (range != null) {
        // 解析 Range: bytes=0-1023
        long[] rangeParts = parseRange(range, fileLength);
        long start = rangeParts[0], end = rangeParts[1];
        long contentLength = end - start + 1;
        
        response.setStatus(HttpServletResponse.SC_PARTIAL_CONTENT); // 206
        response.setHeader("Content-Range", "bytes " + start + "-" + end + "/" + fileLength);
        response.setContentLengthLong(contentLength);
        
        // 只发送请求的范围
        try (InputStream is = new FileInputStream(file)) {
            is.skip(start);
            StreamUtils.copyRange(is, response.getOutputStream(), start, end);
        }
    } else {
        // 全量下载
        response.setContentLengthLong(fileLength);
        Files.copy(file.toPath(), response.getOutputStream());
    }
    return null;
}
```

支持 Range 请求后，下载客户端可以多线程并行下载不同分段，速度大幅提升。

**生产建议：** 大文件上传和下载不要经过应用服务器，直接走对象存储（OSS/S3）：

```java
// 前端直传 OSS：后端只生成预签名 URL
String presignedUrl = ossClient.generatePresignedUrl(bucketName, objectKey, expiryDate);
return presignedUrl; // 前端拿到 URL 直接向 OSS 上传，不经过应用服务器
```

###### 8. 如何实现数据的实时统计？

实时统计的典型场景：实时 UV/PV、实时销售额、实时在线人数、流量监控看板。

**方案一：Redis 原子操作（简单实时统计）**

```java
// 实时 PV（页面访问次数）
redisTemplate.opsForValue().increment("stat:pv:" + pageId + ":" + todayDate);

// 实时 UV（去重访客数），HyperLogLog（误差约0.81%，1000万用户只需12KB）
redisTemplate.opsForHyperLogLog().add("stat:uv:" + pageId + ":" + todayDate, userId);
long uv = redisTemplate.opsForHyperLogLog().size("stat:uv:" + pageId + ":" + todayDate);

// 实时在线人数，ZSet + 时间戳维护（5分钟内活跃的用户）
String key = "online:users";
long now = System.currentTimeMillis();
redisTemplate.opsForZSet().add(key, String.valueOf(userId), now);
redisTemplate.opsForZSet().removeRangeByScore(key, 0, now - 5 * 60 * 1000);
long onlineCount = redisTemplate.opsForZSet().zCard(key);
```

**方案二：Flink 实时流处理（复杂实时统计）**

```java
// Flink：实时统计每分钟的订单金额
DataStream<OrderEvent> orders = env.addSource(new KafkaOrderSource());

orders
    .assignTimestampsAndWatermarks(WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(5)))
    .keyBy(order -> order.getShopId())
    .window(TumblingEventTimeWindows.of(Time.minutes(1))) // 每分钟滚动窗口
    .aggregate(new SumAggregator())
    .addSink(new RedisSink()); // 写入 Redis 供前端查询
```

**方案三：ClickHouse（海量实时写入 + 快速聚合查询）**

```sql
-- ClickHouse 的 MergeTree 引擎支持每秒百万级写入
CREATE TABLE order_stats (
    order_id UInt64,
    shop_id UInt64,
    amount Decimal(18, 2),
    create_time DateTime
) ENGINE = MergeTree()
ORDER BY (shop_id, create_time);

-- 实时聚合查询极快（列存储，向量化执行）
SELECT shop_id, sum(amount) 
FROM order_stats 
WHERE create_time >= now() - INTERVAL 1 HOUR
GROUP BY shop_id;
```

**前端实时展示：** WebSocket 推送 + 服务端定期（每秒/每5秒）从 Redis 或 ClickHouse 读取最新数据并推送给前端。
