# 三、数据库设计与优化

###### 1. 暂时先不考虑分库分表，如果单表的数据量很大，如何优化？

单表数据量大时，按优先级从低成本到高成本逐步优化：

**① 索引优化（首先排查）**

- 确保查询条件字段有合适索引，避免全表扫描
- 使用联合索引，遵循最左前缀原则
- 用 `EXPLAIN` 分析执行计划，关注 `type`（至少达到 `ref`）、`key`、`rows`
- 删除无用索引，减少写入开销

**② SQL 优化**

- 避免 `SELECT *`，只查需要的列
- 避免在 WHERE 子句中对索引列做函数运算（`WHERE YEAR(create_time) = 2025` → 改为范围查询）
- 大分页改用游标分页（`WHERE id > last_id LIMIT 20` 而非 `LIMIT 1000000, 20`）
- 复杂聚合查询考虑预计算/汇总表

**③ 冷热数据分离（归档历史数据）**

```sql
-- 将3年前的历史订单归档到 order_archive 表
INSERT INTO order_archive SELECT * FROM orders WHERE create_time < '2023-01-01';
DELETE FROM orders WHERE create_time < '2023-01-01';
```

业务上区分"活跃数据"和"历史数据"，历史数据移到归档表（或压缩存储），主表始终保持在可控规模。

**④ 分区表（Partition）**

```sql
CREATE TABLE orders (
    id BIGINT,
    create_time DATETIME,
    ...
) PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026)
);
```

MySQL 分区表将数据按规则分散到不同物理文件，查询时可以**分区裁剪**（Partition Pruning），只扫描相关分区。对按时间范围查询的表效果显著。但注意：分区键必须包含在唯一索引中，且跨分区查询不一定有优化效果。

**⑤ 读写分离**

写操作走主库，读操作走从库，分散主库压力。适合读多写少的场景。

**⑥ 增加缓存层**

热点查询结果缓存到 Redis，减少数据库压力。

**面试关键：** 先问慢在哪——全表扫描（加索引）、深分页（游标分页）、历史数据堆积（归档）、读压力大（读写分离+缓存）。分库分表是最后手段，优先上面这些低成本方案。

###### 2. 设计数据库表时，关联表和在一个表中加冗余字段关联各有什么优势？

**关联表（规范化设计）**

```sql
-- 用户表 + 订单表分离
CREATE TABLE user (id BIGINT, name VARCHAR(50), ...);
CREATE TABLE order (id BIGINT, user_id BIGINT, ...); -- 通过 user_id 关联
```

优点：
- **数据不重复**：用户信息只存一份，修改用户名不需要同步多处
- **维护方便**：数据一致性由数据库 FK（外键）或业务逻辑保证
- **存储空间小**：没有冗余数据

缺点：
- **查询复杂**：需要 JOIN，多表关联性能差，尤其是大表 JOIN
- **高并发瓶颈**：JOIN 操作在 DB 层执行，难以拆分到多个节点（分库分表后跨库 JOIN 基本不可行）

**冗余字段（反规范化设计）**

```sql
-- 订单表直接冗余用户名
CREATE TABLE order (
    id BIGINT,
    user_id BIGINT,
    user_name VARCHAR(50),  -- 冗余字段，避免 JOIN
    ...
);
```

优点：
- **查询简单快速**：单表查询，无需 JOIN
- **适合读多写少**：查询性能好，特别适合高并发读场景
- **分布式友好**：分库分表后跨库查询无需 JOIN

缺点：
- **数据冗余**：同一信息存多处，浪费存储空间
- **一致性问题**：源数据变更时，需要同步更新冗余字段（容易漏更新导致数据不一致）
- **维护成本高**：业务逻辑更复杂

**选择原则：**

| 场景 | 推荐方案 |
|------|----------|
| 数据经常变更、一致性要求高 | 关联表 |
| 读多写少、高并发查询 | 冗余字段 |
| 会分库分表的系统 | 冗余字段（跨库 JOIN 代价太大） |
| 数据量小、OLTP 场景 | 关联表 |
| 报表/分析场景 | 可以用宽表（大量冗余字段）|

**实际项目：** 往往是混用——核心主表规范化设计，为了查询性能冗余少量关键字段（如用户名、商品名），并在业务层保证冗余字段的同步更新（通过事件/MQ）。

###### 3. 项目中同样的功能，可能是环境或其他要求原因，需要适配多种数据库，如何实现？

**方案一：JPA/Hibernate（ORM 层屏蔽差异）**

JPA 会将 JPQL 转换为各数据库的方言（Dialect），配置不同的 `spring.jpa.database-platform` 即可切换：

```yaml
# MySQL
spring.jpa.database-platform: org.hibernate.dialect.MySQL8Dialect
# PostgreSQL
spring.jpa.database-platform: org.hibernate.dialect.PostgreSQLDialect
# Oracle
spring.jpa.database-platform: org.hibernate.dialect.Oracle12cDialect
```

大部分简单 CRUD 操作无需修改代码，复杂 SQL 需要用 `@Query` 注解并注意方言兼容。

**方案二：MyBatis + 条件 SQL（灵活可控）**

利用 MyBatis 的动态 SQL + 数据库类型枚举：

```xml
<!-- 使用数据库类型判断，生成不同分页 SQL -->
<select id="queryPage">
    SELECT * FROM orders
    <if test="dbType == 'mysql'">
        LIMIT #{offset}, #{pageSize}
    </if>
    <if test="dbType == 'oracle'">
        WHERE ROWNUM BETWEEN #{offset} AND #{offset + pageSize}
    </if>
    <if test="dbType == 'postgresql'">
        LIMIT #{pageSize} OFFSET #{offset}
    </if>
</select>
```

或者更优雅的方式：用 `@MappedStatement` 拦截器在运行时动态修改 SQL（PageHelper 分页插件就是这样实现的）。

**方案三：ShardingSphere 或 Druid SQL Parser**

ShardingSphere 有 SQL 方言转换能力，可以统一写标准 SQL，由框架负责转换成目标数据库的方言。

**方案四：抽象 DAO 层 + 多实现**

```java
public interface UserRepository {
    List<User> findByCondition(UserQuery query);
}

@Repository
@ConditionalOnProperty(name = "db.type", havingValue = "mysql")
public class MySqlUserRepository implements UserRepository { ... }

@Repository
@ConditionalOnProperty(name = "db.type", havingValue = "postgresql")
public class PostgreSqlUserRepository implements UserRepository { ... }
```

通过 Spring `@Conditional` 注解，根据配置项激活不同实现。代码量多但最灵活可控。

**实际建议：** 尽量避免使用数据库私有特性（如 MySQL 的 `GROUP_CONCAT`、Oracle 的 `ROWNUM`）。优先用标准 SQL + ORM，必须用特性时用上面的方案封装。

###### 4. 数据库表上新增一个字段，如果这个表正在进行读写操作，如何处理才能不影响现有读写操作？

这道题的本质是 **Online DDL（在线 DDL）**，核心问题是普通 `ALTER TABLE` 会锁表，导致业务中断。

**MySQL 5.6+ 原生 Online DDL**

```sql
-- 指定 ALGORITHM=INPLACE, LOCK=NONE，尽量不锁表
ALTER TABLE orders 
ADD COLUMN remark VARCHAR(200) DEFAULT NULL COMMENT '备注',
ALGORITHM=INPLACE, LOCK=NONE;
```

MySQL 8.0 对大多数 DDL 操作（加列、改默认值、加索引）都支持 `ALGORITHM=INPLACE, LOCK=NONE`，不阻塞读写。但某些操作（如改列类型、加全文索引）仍需要锁表，需要提前评估。

**gh-ost（GitHub 开源，生产推荐）**

```bash
gh-ost \
  --host="db-host" --database="shop" --table="orders" \
  --alter="ADD COLUMN remark VARCHAR(200)" \
  --execute
```

原理：
1. 在原表旁边创建一张 `_orders_gho` 影子表（加上新列）
2. 通过 binlog 实时同步原表的增量数据到影子表
3. 全量拷贝原表存量数据到影子表（低速，不锁表）
4. 确认同步追上后，原子性地重命名（影子表 → 原表）

全程不锁表，对业务无感知，还支持限速控制（避免拖慢主库）。是 GitHub、各大互联网公司的标准 Online Schema Change 工具。

**pt-online-schema-change（Percona，传统方案）**

原理类似：创建新表 → 触发器同步增量 → 全量拷贝 → rename。不如 gh-ost，但比直接 ALTER TABLE 安全。

**蓝绿发布 / 双写过渡（应用层方案）**

如果 DDL 风险太高，可以：
1. 新建一张含新字段的新表
2. 代码双写：写入时同时写旧表和新表（新字段写空/默认值）
3. 存量数据迁移到新表
4. 切换读取来源到新表
5. 确认无误后切换写入，下线旧表

这种方案不依赖 DDL 工具，但代码改动大，适合高风险大表。

**面试重点：** 直接 `ALTER TABLE` 加列在小表（几十万行以内）可能几秒内完成没问题，但千万级大表必须用 gh-ost 或 pt-osc。说出工具名称和原理，加分很多。

###### 5. MySQL 数据库一亿条数据，怎么快速加索引？

一亿条数据直接 `CREATE INDEX` 会有以下问题：
- 执行时间可能几小时，期间锁表（5.5以前）或影响 I/O 性能
- 占用大量临时表空间
- 影响在线业务

**方案一：Online DDL（MySQL 5.6+ 内置）**

```sql
-- 创建索引，不锁写（ALGORITHM=INPLACE，lock=NONE）
ALTER TABLE big_table 
ADD INDEX idx_user_create(user_id, create_time),
ALGORITHM=INPLACE, LOCK=NONE;
```

MySQL 8.0 的加索引操作基本都是 INPLACE + NONE，不阻塞 DML，但仍会有一定 I/O 压力。

**方案二：gh-ost 在线 DDL（生产首选）**

如上题所述，gh-ost 通过影子表 + binlog 同步，全程无锁。对一亿级大表加索引是最安全的方案，还支持限速：

```bash
gh-ost --max-load="Threads_running=50" --critical-load="Threads_running=100" ...
```

**方案三：主从切换法（停机窗口受限时）**

1. 在从库上执行 `CREATE INDEX`（不影响主库业务）
2. 从库加完索引后，提升为新主库
3. 老主库作为新从库同步数据并加索引
4. 全部完成后恢复

**方案四：分批构建（配合分区表）**

如果表已经分区，可以按分区逐个加索引，每次只影响一个分区，降低风险：

```sql
ALTER TABLE orders PARTITION (p2023) ADD INDEX idx_user(user_id);
```

**控制加索引影响 I/O 的参数：**

```sql
-- 降低 innodb_sort_buffer_size 减少内存占用
-- 控制 innodb_online_alter_log_max_size 允许的 DML 变更大小
SET GLOBAL innodb_sort_buffer_size = 67108864; -- 64MB
```

**面试总结：** 一亿数据加索引，直接 ALTER TABLE 风险高，生产上用 gh-ost；说出 Online DDL 的 ALGORITHM=INPLACE + LOCK=NONE；提到可以在从库先加，再做主从切换。

###### 6. 如何设计一个支持千万级数据的分页查询方案？

**问题根源：深度分页性能极差**

```sql
-- 翻到第100万页：先扫描1000020行，再丢弃前1000000行
SELECT * FROM orders LIMIT 1000000, 20; -- 极慢！
```

`LIMIT offset, size` 的代价随着 offset 增大而线性增长，offset=1000000 时要扫描并丢弃前一百万行。

**方案一：游标分页（基于 ID 的范围查询）——最推荐**

```sql
-- 第一页
SELECT * FROM orders WHERE id > 0 ORDER BY id ASC LIMIT 20;

-- 下一页（记录上次最后一条的 id）
SELECT * FROM orders WHERE id > #{lastId} ORDER BY id ASC LIMIT 20;
```

优点：无论翻到第几页，始终只扫描 20 行（走主键索引范围查询）。
缺点：只能顺序翻页（上一页/下一页），不支持跳页（直接跳到第1000页）。适合移动端无限滚动、APP 下拉加载。

**方案二：子查询 + 覆盖索引优化**

```sql
-- 先用覆盖索引快速定位主键，再回表取数据
SELECT * FROM orders 
WHERE id >= (
    SELECT id FROM orders ORDER BY create_time DESC LIMIT 1000000, 1
)
ORDER BY create_time DESC
LIMIT 20;
```

内层子查询走覆盖索引（只读索引列，不回表），定位到起始 id 很快；外层再用主键 id 查完整数据。比直接 LIMIT 1000000, 20 快很多，但深翻页仍有性能损耗。

**方案三：限制最大页数**

产品层面限制最多翻到 500 页（每页20条，即最多查10000条），超出范围提示"请使用搜索缩小范围"。这是很多大型平台（如淘宝）的实际做法。

**方案四：ElasticSearch 搜索翻页**

对于需要全文搜索 + 大量翻页的场景，将数据同步到 ES，使用 ES 的 `search_after` 游标分页：

```json
GET /orders/_search
{
  "query": { "match_all": {} },
  "sort": [{"create_time": "desc"}, {"id": "desc"}],
  "search_after": [1709000000000, 12345678],
  "size": 20
}
```

`search_after` 基于排序值定位，翻多深都是固定性能。

**方案五：Redis 缓存分页结果**

热门列表（如商品排行榜、帖子列表）用 Redis ZSet 存储，按分数排序，`ZREVRANGEBYSCORE` 取指定范围：

```java
// 商品按销量排行，获取第2页（每页20条）
Set<ZSetOperations.TypedTuple<Long>> result = 
    redisTemplate.opsForZSet().reverseRangeWithScores("product:rank", 20, 39);
```

**选型建议：**

| 场景 | 方案 |
|------|------|
| 移动端下拉加载 | 游标分页（id > lastId）|
| 后台管理系统翻页 | 子查询优化 + 限制最大页 |
| 商品搜索/内容搜索 | ElasticSearch search_after |
| 排行榜 | Redis ZSet |

###### 7. 数据库连接池如何设置合理的参数？

连接池参数设置不当是很多性能问题的根源，核心参数有三个：最小连接数、最大连接数、超时时间。

**HikariCP（Spring Boot 默认，推荐）**

```yaml
spring:
  datasource:
    hikari:
      minimum-idle: 10           # 最小空闲连接数
      maximum-pool-size: 50      # 最大连接数（核心参数）
      connection-timeout: 30000  # 获取连接的最大等待时间(ms)，超时抛异常
      idle-timeout: 600000       # 空闲连接超过此时间被回收(ms)
      max-lifetime: 1800000      # 连接最大生命周期，防止 MySQL wait_timeout 断开
      connection-test-query: SELECT 1  # 连接有效性检测 SQL（若驱动支持 isValid 可不配）
```

**最大连接数怎么算？**

经验公式（HikariCP 官方建议）：

```
connections = (core_count * 2) + effective_spindle_count
```

对于纯 I/O 密集型（如简单 CRUD，线程大部分时间在等 DB 响应）：

```
max-pool-size = 并发线程数 × (DB等待时间 / 总请求时间)
```

实践中的建议值：
- 8核 CPU 的应用服务器：最大连接数设 50~100
- 不是越大越好：连接数过多，DB 端线程切换开销反而增大
- 用压测工具找到最优值（在最大吞吐点附近）

**wait_timeout 陷阱（最常见的坑）**

MySQL 默认 `wait_timeout=28800`（8小时），空闲超过8小时的连接会被 MySQL 断开，但连接池不知道，下次使用时会报 `Communications link failure`。

解决：
1. 设置 `max-lifetime < wait_timeout`（如 max-lifetime=1800000 即30分钟）
2. 配置 `keepalive-time` 定期发心跳（HikariCP 4.0+）：`keepalive-time: 60000`

**Druid 额外配置：**

```yaml
spring:
  datasource:
    druid:
      initial-size: 10
      min-idle: 10
      max-active: 50
      test-while-idle: true
      validation-query: SELECT 1
      time-between-eviction-runs-millis: 60000
      min-evictable-idle-time-millis: 300000
```

###### 8. 如何进行数据库的读写分离？主从同步延迟如何处理？

**读写分离实现方式：**

**方案一：ShardingSphere-JDBC（推荐，无代理层）**

```yaml
spring:
  shardingsphere:
    datasource:
      names: master,slave1,slave2
      master:
        url: jdbc:mysql://master:3306/shop
      slave1:
        url: jdbc:mysql://slave1:3306/shop
      slave2:
        url: jdbc:mysql://slave2:3306/shop
    rules:
      readwrite-splitting:
        data-sources:
          shop-ds:
            type: STATIC
            props:
              write-data-source-name: master
              read-data-source-names: slave1,slave2
            load-balancer-name: round-robin
```

**方案二：MyBatis + 多数据源 + AOP**

通过 AOP 拦截 Service 方法，读操作切到从库，写操作切到主库：

```java
@Around("@annotation(ReadOnly)")
public Object switchToSlave(ProceedingJoinPoint pjp) {
    DataSourceContextHolder.setSlave(); // 切换到从库
    try {
        return pjp.proceed();
    } finally {
        DataSourceContextHolder.clear();
    }
}
```

**方案三：MySQL Router / ProxySQL（代理层，对应用透明）**

在应用和 DB 之间加一层代理，代理负责路由读写请求。应用只连代理，不感知主从拓扑。

**主从同步延迟处理：**

主从延迟是读写分离最大的挑战——刚写入主库的数据，立刻从从库读可能读到旧值。

**① 写后读强制走主库（最简单）**

```java
// 刚写完数据后，强制这次读走主库
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);
    // 立刻读取自己刚写入的数据，强制走主库
    DataSourceContextHolder.setMaster();
    Order created = orderMapper.selectById(order.getId());
}
```

**② 半同步复制（MySQL 5.5+）**

主库提交事务时，等待至少一个从库确认收到 binlog 后才返回成功。延迟从毫秒级降到微秒级，但会增加写入延迟。

**③ 业务层容忍延迟**

大多数业务场景可以接受几十毫秒的延迟（用户下单后立刻查列表，商品列表里没有当前订单用户也不一定察觉）。对用户体验影响不大的查询，直接走从库。

**④ 监控延迟**

```sql
-- 在从库执行，查看主从延迟秒数
SHOW SLAVE STATUS\G
-- 关注 Seconds_Behind_Master 字段
```

延迟超过阈值（如5秒）时告警，必要时临时将流量全切回主库。

###### 9. 分库分表后如何进行跨库查询和跨库事务？

分库分表后，很多在单库时轻松实现的操作变得复杂，这是分库分表最大的代价。

**跨库查询的处理方案：**

**① 字段冗余（反规范化）**

将跨库需要 JOIN 的字段冗余在同一个表里，避免跨库关联：

```sql
-- 订单表冗余用户名、商品名，不需要跨库 JOIN 用户表、商品表
CREATE TABLE order (
    id BIGINT,
    user_id BIGINT,
    user_name VARCHAR(50),    -- 冗余字段
    product_id BIGINT,
    product_name VARCHAR(200), -- 冗余字段
    ...
)
```

**② 全局表（字典表广播）**

数据量小且不常变化的表（如地区表、字典表），在每个分库都保存一份完整副本：

```yaml
# ShardingSphere 广播表配置
rules:
  sharding:
    broadcast-tables:
      - area_dict
      - system_config
```

查询时直接 JOIN 本地的广播表，无需跨库。

**③ 应用层聚合（最通用）**

```java
// 不在 DB 层做 JOIN，在应用层聚合
List<Order> orders = orderService.queryByUserId(userId);
List<Long> productIds = orders.stream().map(Order::getProductId).collect(toList());
Map<Long, Product> productMap = productService.queryByIds(productIds); // 并行查询
orders.forEach(o -> o.setProduct(productMap.get(o.getProductId())));
```

缺点是需要多次网络请求，且聚合逻辑在应用层实现，代码复杂。

**④ 引入搜索引擎（复杂查询场景）**

将需要多维度查询的数据同步到 ElasticSearch，复杂搜索走 ES，简单 CRUD 走 DB。ES 天然支持全文检索、多字段聚合。

**跨库事务处理：**

跨库事务是分布式事务，不能直接使用 DB 的本地 `@Transactional`。

**① Seata AT 模式（业务无侵入，推荐）**

```java
@GlobalTransactional // 替换 @Transactional
public void createOrderAndDeductStock(Order order) {
    orderService.createOrder(order);         // 写订单库
    inventoryService.deductStock(order);      // 写库存库（可能在不同库）
}
```

Seata AT 模式自动生成 undo_log，分布式提交/回滚，对业务代码基本无侵入。

**② TCC 模式（高性能场景）**

手动实现 Try-Confirm-Cancel 三个阶段，性能比 AT 模式高，但业务代码改动大。

**③ 最终一致性（业务允许时的最优解）**

通过 MQ 异步消息保证最终一致性。比如下单时先写订单库，发一条扣库存的 MQ 消息，库存服务消费消息扣减。如果失败就重试（幂等处理），最终保持一致。

**实际建议：** 能不跨库就不跨库。设计分片键时，尽量把同一业务的相关数据分到同一分片（如按 user_id 分片，用户的订单、地址、积分都在同一个数据库节点）。

###### 10. 如何设计一个数据归档方案？

数据归档是解决主表数据量膨胀的关键手段，目标是将历史/冷数据迁移出主表，同时保证业务连续性。

**归档策略设计：**

**① 确定归档规则**

- 时间维度：N年前的历史订单（如3年前）
- 状态维度：已关闭/已完成/已退款的订单
- 业务维度：访问频率低的用户行为日志

**② 归档目标存储**

| 归档目标 | 适用场景 |
|----------|----------|
| 同库归档表（`order_archive`）| 偶尔还需查询历史数据 |
| 独立归档库 | 大量历史数据，单独维护 |
| 对象存储（OSS/S3）| 日志类数据，只需下载查看 |
| 列存储数据库（ClickHouse）| 需要做历史数据分析报表 |
| 冷存储（Glacier）| 合规要求保存但几乎不查询 |

**③ 归档流程（以迁移到归档表为例）**

```java
@Scheduled(cron = "0 2 * * * ?")  // 每天凌晨2点执行
public void archiveOldOrders() {
    LocalDate cutoffDate = LocalDate.now().minusYears(3);
    
    int batchSize = 1000;
    long lastId = 0;
    
    while (true) {
        // 分批查询（避免一次性锁定大量行）
        List<Order> batch = orderMapper.findOldOrders(cutoffDate, lastId, batchSize);
        if (batch.isEmpty()) break;
        
        // 写入归档表
        orderArchiveMapper.batchInsert(batch);
        
        // 从主表删除（待验证归档成功后）
        List<Long> ids = batch.stream().map(Order::getId).collect(toList());
        orderMapper.batchDelete(ids);
        
        lastId = batch.get(batch.size() - 1).getId();
        Thread.sleep(100); // 限速，避免影响业务
    }
}
```

**④ 归档的关键注意事项**

- **先写归档再删主表**：保证数据不丢失，即使删除失败数据也存在归档表中
- **分批小批量迁移**：每批1000条，配合限速（sleep/令牌桶），避免影响在线业务
- **验证归档完整性**：迁移后 COUNT 对比，确认数据无遗漏再删除
- **保留查询入口**：业务系统提供"查看历史订单"功能，路由到归档表查询
- **定期清理**：根据监管要求确定保留年限，超期数据可彻底清除

**⑤ 使用 pt-archiver（Percona 工具）**

```bash
pt-archiver \
  --source h=master,D=shop,t=orders \
  --dest h=archive-db,D=shop_archive,t=orders \
  --where "create_time < '2023-01-01'" \
  --limit 1000 --sleep 0.1 \
  --no-check-charset
```

pt-archiver 是专业的归档工具，支持限速、断点续传、在线归档，生产推荐使用。
