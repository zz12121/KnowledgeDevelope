# 十二、监控与运维

###### 1. 新项目要上线了，你会关注哪些指标，为什么？

上线后第一时间需要建立的监控体系，按层次来看：

**① 系统层指标（基础设施）**

- **CPU 使用率**：超过 70% 持续时间长 → 需要扩容或排查代码 CPU 热点
- **内存使用率**：持续增长不回落 → 内存泄漏
- **磁盘 I/O / 磁盘使用率**：磁盘满会导致应用崩溃（日志写不进去）
- **网络流量/带宽**：出口带宽打满 → CDN 或多节点分担

**② JVM 层指标（Java 应用必看）**

- **GC 频率和停顿时间**：Full GC 频率过高 → 堆太小或内存泄漏；停顿时间长（>500ms）→ 用户体验受损
- **堆内存使用率**：持续接近 `-Xmx` → 快 OOM 了
- **线程数**：线程池满 → 新请求入队或被拒绝
- **类加载数量**：持续增长 → 动态类加载泄漏（Metaspace 泄漏前兆）

**③ 应用层指标（最重要）**

- **QPS / TPS**：请求量，判断流量是否符合预期
- **响应时间（P99/P95/P50）**：P99 > 1s 说明有问题；监控百分位而非平均值（平均值掩盖长尾）
- **错误率（5xx / 4xx）**：上线后 5xx 率突增 → 有 Bug；4xx 突增 → API 文档有误或客户端问题
- **接口成功率**：业务成功率（下单成功/支付成功）比 HTTP 状态码更能反映业务健康

**④ 业务层指标（体现业务价值）**

- **核心流程转化率**：注册率、下单成功率、支付成功率
- **核心业务量**：订单量/支付金额，对比上线前是否有异常波动（上线后某指标突然下跌 → 可能引入了 Bug）

**⑤ 中间件指标**

- DB：慢查询数量、连接池使用率、主从延迟
- Redis：内存使用率、命中率、连接数
- MQ：消费积压量（Consumer Lag）

**面试答法建议：** 从底层（服务器）到上层（业务）分层说，重点强调业务指标（纯技术指标好但业务指标才是真正的目的）。

###### 2. 如何搭建一套完整的监控告警系统？

**完整监控架构：**

```
数据采集          数据存储          可视化          告警
Prometheus ──→ Prometheus TSDB ──→ Grafana ──→ AlertManager
ELK Stack  ──→ Elasticsearch  ──→ Kibana  ──→ 规则告警
SkyWalking ──→ ES / OAP       ──→ UI      ──→ 内置告警
```

**Prometheus + Grafana（指标监控标配）：**

```yaml
# Docker Compose 快速部署
services:
  prometheus:
    image: prom/prometheus
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
  
  grafana:
    image: grafana/grafana
    ports:
      - "3000:3000"
  
  alertmanager:
    image: prom/alertmanager
    volumes:
      - ./alertmanager.yml:/etc/alertmanager/alertmanager.yml
```

```yaml
# prometheus.yml 抓取配置
scrape_configs:
  - job_name: 'spring-boot'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['app:8080']
    
  - job_name: 'jvm'
    static_configs:
      - targets: ['app:8080']
```

```java
// Spring Boot 应用接入 Prometheus
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

**Grafana Dashboard：**

导入社区已有的 Dashboard（JVM 监控 Dashboard ID：4701；Spring Boot Dashboard ID：12685），开箱即用。

**AlertManager 告警规则：**

```yaml
# prometheus/rules/app.yml
groups:
  - name: app-alerts
    rules:
      - alert: HighErrorRate
        expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m]) > 0.1
        for: 2m
        annotations:
          summary: "接口错误率超过10%"
          
      - alert: HighResponseTime
        expr: histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m])) > 1
        for: 3m
        annotations:
          summary: "P99 响应时间超过1秒"
```

**告警通知（AlertManager）：**

```yaml
# alertmanager.yml
receivers:
  - name: 'dingtalk'
    webhook_configs:
      - url: 'http://dingtalk-webhook/dingtalk/robot1/send'
  - name: 'email'
    email_configs:
      - to: 'ops@company.com'
```

###### 3. 如何进行日志分析和问题排查？

**ELK/EFK 标准日志分析流程：**

```
1. 日志产生：应用 → Logback/Log4j2
2. 日志采集：Filebeat 采集文件 / Logback Kafka Appender 直接投 Kafka
3. 日志处理：Logstash 解析/过滤/转换（Grok 正则提取字段）
4. 日志存储：Elasticsearch（全文索引）
5. 日志查询：Kibana 界面搜索/分析
```

**快速排查问题（Kibana）：**

```
# Kibana KQL 查询语法
# 查某个接口的所有 ERROR 日志
level: ERROR AND path: "/api/order/create"

# 查某个 TraceId 的完整链路日志
traceId: "a1b2c3d4e5f6"

# 查某段时间内的异常
@timestamp: [2026-03-15T10:00:00 TO 2026-03-15T11:00:00] AND level: ERROR
```

**常用命令行工具（无 ELK 时）：**

```bash
# 查找今天的 ERROR 日志
grep "ERROR" /logs/app.log | tail -100

# 统计各接口错误次数（按 URL 分组）
grep "ERROR" /logs/access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -20

# 查找 GC 停顿时间超过1s的记录
grep "GC pause" /logs/gc.log | awk '$NF+0 > 1000 {print}' | head -20

# 实时监控 ERROR 日志
tail -f /logs/app.log | grep --color "ERROR\|Exception"
```

**日志规范（减少排查成本）：**

- 关键操作记录入参、出参、耗时
- 异常必须打完整堆栈（`log.error("msg", e)`，不要只 `log.error(e.getMessage())`）
- 包含 TraceId（MDC 注入）
- 业务关键节点（下单、支付）打业务日志，方便对账

###### 4. 如何实现全链路监控？

全链路监控 = 链路追踪（Tracing）+ 指标监控（Metrics）+ 日志（Logging）三者打通（Observability 三大支柱）。

**三者打通的关键：TraceId**

```
请求进入 → 生成 TraceId → 注入 HTTP Header（X-TraceId）
         → 注入 MDC（日志自动携带 TraceId）
         → 注入 Span（追踪系统记录调用链）
         → 注入 Metrics Label（指标可按 TraceId 查询）
```

**技术选型（推荐组合）：**

| 职责 | 工具 |
|------|------|
| 链路追踪 | SkyWalking（Java 无侵入）/ Zipkin / Jaeger |
| 指标监控 | Prometheus + Grafana |
| 日志 | ELK（Elasticsearch + Logstash + Kibana）|
| 统一平台 | 阿里云 ARMS / 腾讯云 APM（托管方案）|

**从 Grafana 告警跳到 SkyWalking 链路（打通关键）：**

```yaml
# Grafana 告警里带上 traceId URL
annotations:
  description: "响应时间超标，查链路: http://skywalking:8080/trace?traceId={{$labels.traceId}}"
```

**SkyWalking 关键配置：**

```bash
# 采样率（高并发时不需要100%采样）
# skywalking-agent.config
agent.sample_n_per_3_secs=30  # 每3秒最多采样30个 Trace（按比例）
```

**OpenTelemetry（未来趋势）：**

OpenTelemetry 是 CNCF 项目，统一了 Tracing/Metrics/Logs 的数据模型和 SDK，与后端（SkyWalking/Jaeger/Prometheus）解耦，是未来可观测性的标准。

###### 5. 线上故障如何快速定位和恢复？

**黄金原则：先恢复，后分析（MTTR 比 MTBF 更重要）**

用户在受损，恢复服务是第一优先级，根因分析等服务稳定后再进行。

**故障定位 SOP（标准操作程序）：**

**第一步（0-2分钟）：确认故障范围**

```bash
# 监控看板：确认哪些服务/接口受影响
# 错误率、响应时间、QPS 图表
# 是否全量用户还是部分用户？某个接口还是全部接口？
```

**第二步（2-5分钟）：快速恢复**

- 最近是否有发布？**立即回滚**（最常见的原因，回滚最快）
- 数据库连接池满？**重启应用服务**（快速释放连接）
- Redis 内存满？**临时清理或扩容**
- 流量突增打垮？**临时限流降级**

**第三步（5-15分钟）：日志/链路分析**

```bash
# 查 ERROR 日志（SkyWalking 或 ELK）
# 找到第一条异常发生的时间（根因往往在最早的那条异常）
# 找高频出现的异常类型

# 关注：异常发生时间是否与发布时间吻合？
```

**第四步（恢复后）：根因分析**

保留现场（dump 文件、日志快照），写故障报告：

```
故障报告模板：
- 故障时间：开始/结束
- 故障影响：受影响用户数/接口/业务指标
- 根本原因：具体原因（如：新版本引入的 N+1 查询导致 DB 连接池耗尽）
- 处理过程：时间线记录
- 改进措施：技术改进 + 流程改进（监控/告警/测试覆盖）
```

**常见故障和快速恢复：**

| 故障 | 快速恢复 |
|------|----------|
| 代码 Bug | 回滚 |
| DB 慢查询 / 死锁 | kill 慢查询进程；加索引（emergency）|
| 内存泄漏 OOM | 重启应用；增加堆内存（临时）|
| 第三方服务超时 | 熔断降级；超时缩短；临时关闭依赖 |
| 流量洪峰 | 限流；扩容（K8s 快速扩容）|

###### 6. 如何设计告警规则避免告警风暴？

**告警风暴的成因：** 一个根因问题触发了大量关联告警（如 DB 挂掉 → 100个服务的 DB 连接告警同时触发），运维被淹没在告警噪音中，找不到真正的问题。

**解决方案：**

**① 告警分级（区分紧急程度）**

- **P0（立即响应）**：核心功能完全不可用、大量 5xx 错误、关键数据丢失风险
- **P1（1小时内）**：性能明显下降、部分功能异常、预警性告警
- **P2（工作时间处理）**：优化建议、非核心功能降级
- P0/P1 电话/钉钉紧急通知，P2 邮件聚合发送

**② 告警去重（AlertManager Inhibit）**

```yaml
# alertmanager.yml：如果父告警触发，抑制子告警
inhibit_rules:
  - source_match:
      alertname: 'DatabaseDown'
    target_match:
      severity: 'warning'
    equal: ['service']  # 同一服务的 warning 告警被 DatabaseDown 抑制
```

**③ 告警聚合（同类告警合并通知）**

```yaml
# AlertManager 路由配置：同 service 的告警聚合发送
route:
  group_by: ['service', 'alertname']    # 按服务和告警名分组
  group_wait: 30s                        # 等待30秒聚合同组告警
  group_interval: 5m                     # 同组告警最多5分钟发一次
  repeat_interval: 4h                    # 未恢复的告警每4小时重复一次
```

**④ 设置合理阈值（避免误报）**

```yaml
# 不是瞬时触发，而是持续一段时间才告警
- alert: HighCpuUsage
  expr: avg(cpu_usage) > 0.85
  for: 5m  # 持续5分钟CPU > 85%才告警（避免瞬时峰值误报）
```

**⑤ 告警后的降噪**

- 已知问题（维护窗口）临时屏蔽告警（AlertManager silence）
- 告警后 N 分钟内同类告警不重复通知（repeat_interval）
- 定期复查告警规则，删除长期没有触发价值的告警

**⑥ 运营手册（Runbook）**

每条告警关联一个 Runbook（处理手册），告警消息中包含直接跳转链接，减少排查时间：

```yaml
annotations:
  runbook: "https://wiki.company.com/runbook/high-error-rate"
  summary: "接口错误率超过阈值，参考 Runbook 处理"
```
