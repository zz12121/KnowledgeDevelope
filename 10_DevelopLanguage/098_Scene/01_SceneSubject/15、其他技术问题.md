# 十五、其他技术问题

###### 1. 如何实现接口版本管理？

接口版本管理用于在迭代 API 时保持向后兼容，让新旧客户端可以同时工作。

**方案一：URL 路径版本号（最直观）**

```java
@RestController
@RequestMapping("/api/v1/user")
public class UserControllerV1 {
    @GetMapping("/{id}")
    public UserV1DTO getUser(@PathVariable Long id) { ... }
}

@RestController
@RequestMapping("/api/v2/user")
public class UserControllerV2 {
    @GetMapping("/{id}")
    public UserV2DTO getUser(@PathVariable Long id) { ... } // 返回字段有变化
}
```

优点：简单直观，Swagger 文档清晰。缺点：URL 里有版本号，不够"RESTful"。

**方案二：请求头版本号（推荐）**

```java
@GetMapping(value = "/user/{id}", headers = "X-API-Version=1")
public UserV1DTO getUserV1(@PathVariable Long id) { ... }

@GetMapping(value = "/user/{id}", headers = "X-API-Version=2")
public UserV2DTO getUserV2(@PathVariable Long id) { ... }
```

客户端在 Header 中指定版本：`X-API-Version: 2`，URL 不变，更优雅。

**方案三：Accept Header 内容协商**

```java
@GetMapping(value = "/user/{id}", produces = "application/vnd.company.v1+json")
public UserV1DTO getUserV1(@PathVariable Long id) { ... }
```

请求头：`Accept: application/vnd.company.v2+json`，符合 HTTP 标准但客户端使用稍复杂。

**版本管理最佳实践：**

- 明确版本废弃策略（如旧版本维护N个月后下线）
- 做好版本变更文档（什么版本增加/删除了什么字段）
- 新版本只做加法（新增字段），不改已有字段的类型和语义
- 必须有 breaking change 时，版本号升级（v1 → v2）

###### 2. 如何实现多租户系统？

多租户（Multi-Tenancy）是 SaaS 系统的核心架构，多个客户（租户）共享同一套系统但数据相互隔离。

**数据隔离三种方案：**

**方案一：独立数据库（Strong Isolation）**

每个租户一个独立数据库，数据完全隔离，故障互不影响。
- 优点：安全性高，可以为大客户定制
- 缺点：租户多时数据库实例数爆炸，运维成本高

**方案二：共享数据库，独立 Schema**

多个租户共享 MySQL 实例，每个租户一个 Schema（database）：
```sql
USE tenant_001; -- 租户1
SELECT * FROM orders;

USE tenant_002; -- 租户2
SELECT * FROM orders;
```

**方案三：共享数据库和 Schema，行级隔离（最常用）**

所有租户的数据在同一张表，通过 `tenant_id` 字段区分：

```sql
-- 所有业务表加 tenant_id 字段
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    tenant_id BIGINT NOT NULL COMMENT '租户ID',
    ...
    INDEX idx_tenant(tenant_id, ...)
);
```

**MyBatis 插件自动注入 tenant_id（防止代码遗漏）：**

```java
@Component
public class TenantInterceptor implements Interceptor {
    
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 自动在 SQL 的 WHERE 条件中注入当前租户的 tenant_id
        // 避免每个 SQL 都手动加 WHERE tenant_id = ?
        MappedStatement ms = (MappedStatement) invocation.getArgs()[0];
        String sql = ms.getBoundSql(invocation.getArgs()[1]).getSql();
        // 解析 SQL，注入 tenant_id 条件
        return invocation.proceed();
    }
}
```

**TenantContext（线程本地存储）：**

```java
// 请求拦截器：从 JWT 或 Header 中提取 tenant_id 放入 ThreadLocal
public class TenantInterceptorFilter implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest req, ...) {
        Long tenantId = extractTenantId(req);
        TenantContext.setTenantId(tenantId); // ThreadLocal 存储
        return true;
    }
    
    @Override
    public void afterCompletion(HttpServletRequest req, ...) {
        TenantContext.clear(); // 请求结束后清除，防止线程池复用时污染
    }
}
```

###### 3. 如何实现数据脱敏？

数据脱敏用于保护用户隐私，展示数据时隐藏部分敏感信息。

**脱敏类型：**

```java
public class MaskUtils {
    // 手机号：138****8888
    public static String maskPhone(String phone) {
        if (phone == null || phone.length() != 11) return phone;
        return phone.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2");
    }
    
    // 身份证：110101********0019
    public static String maskIdCard(String idCard) {
        if (idCard == null || idCard.length() < 6) return idCard;
        return idCard.replaceAll("(\\w{6})\\w*(\\w{4})", "$1********$2");
    }
    
    // 银行卡：6222 **** **** 1234
    public static String maskBankCard(String cardNo) {
        return cardNo.replaceAll("(\\d{4})\\d+(\\d{4})", "$1 **** **** $2");
    }
    
    // 姓名：张*
    public static String maskName(String name) {
        if (name == null || name.length() == 0) return name;
        return name.charAt(0) + "*".repeat(name.length() - 1);
    }
    
    // 邮箱：zh***@example.com
    public static String maskEmail(String email) {
        int atIdx = email.indexOf('@');
        if (atIdx <= 2) return "***" + email.substring(atIdx);
        return email.substring(0, 2) + "***" + email.substring(atIdx);
    }
}
```

**Spring Boot 自动脱敏（Jackson 注解）：**

```java
// 自定义脱敏注解
@JacksonAnnotationsInside
@JsonSerialize(using = SensitiveSerializer.class)
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface Sensitive {
    SensitiveType type() default SensitiveType.PHONE;
}

// 自定义序列化器
public class SensitiveSerializer extends JsonSerializer<String> {
    @Override
    public void serialize(String value, JsonGenerator gen, 
                          SerializerProvider provider) throws IOException {
        Sensitive annotation = ...; // 获取注解
        gen.writeString(MaskUtils.mask(value, annotation.type()));
    }
}

// 在 DTO 字段上使用
public class UserDTO {
    private String name;
    
    @Sensitive(type = SensitiveType.PHONE)
    private String phone;  // 序列化时自动脱敏
    
    @Sensitive(type = SensitiveType.ID_CARD)
    private String idCard;
}
```

###### 4. 如何实现幂等性？

见第四章第7题（分布式系统幂等性），此处补充 **HTTP 接口层面的幂等实现**：

**Token 机制（前端防重复提交）：**

```java
// 1. 前端发起操作前，先获取一个 idempotency token
@GetMapping("/api/token")
public String getToken() {
    String token = UUID.randomUUID().toString();
    redisTemplate.opsForValue().set("idempotency:" + token, "unused", 5, TimeUnit.MINUTES);
    return token;
}

// 2. 前端提交请求时携带 token
@PostMapping("/api/order/create")
public Order createOrder(@RequestHeader("X-Idempotency-Token") String token,
                          @RequestBody CreateOrderRequest req) {
    // 3. 原子消费 token（确保同一 token 只处理一次）
    Boolean consumed = redisTemplate.opsForValue()
        .setIfAbsent("idempotency:consumed:" + token, "1", 5, TimeUnit.MINUTES);
    if (!Boolean.TRUE.equals(consumed)) {
        // 重复请求：返回第一次的结果（从缓存取）
        return redisTemplate.opsForValue().get("idempotency:result:" + token);
    }
    
    // 4. 执行业务逻辑
    Order order = orderService.create(req);
    
    // 5. 缓存结果（重复请求时返回）
    redisTemplate.opsForValue().set("idempotency:result:" + token, order, 5, TimeUnit.MINUTES);
    
    return order;
}
```

###### 5. 如何处理时区问题？

时区问题是跨国系统的常见坑，核心原则：**存储用 UTC，展示时转换为用户本地时区**。

**数据库层：**

```sql
-- 存储时间戳（与时区无关）而非 DATETIME（依赖服务器时区）
create_time BIGINT COMMENT '毫秒时间戳（UTC）'
-- 或
create_time DATETIME(3) COMMENT '始终存储 UTC 时间'
```

```yaml
# MySQL 连接时区配置
spring.datasource.url=jdbc:mysql://host/db?serverTimezone=UTC
```

**Java 层：**

```java
// 应用服务器统一设置 UTC 时区
@SpringBootApplication
public class App {
    @PostConstruct
    public void init() {
        TimeZone.setDefault(TimeZone.getTimeZone("UTC"));
    }
}

// 或在启动参数中设置
java -Duser.timezone=UTC -jar app.jar

// 存储：统一转 UTC 再存
public Instant toUtc(LocalDateTime localTime, String userTimezone) {
    return localTime.atZone(ZoneId.of(userTimezone)).toInstant();
}

// 展示：从 UTC 转为用户本地时区
public LocalDateTime toUserTime(Instant utcTime, String userTimezone) {
    return utcTime.atZone(ZoneId.of(userTimezone)).toLocalDateTime();
}
```

**接口层：**

- 接口返回时间用 ISO 8601 格式并带时区：`"2026-03-15T10:00:00Z"`（Z 表示 UTC）
- 或返回毫秒时间戳，由前端根据浏览器时区转换展示

**常见坑：**

```java
// ❌ new Date() 依赖 JVM 时区
Date date = new Date();

// ✅ Instant 是 UTC 时间戳，与 JVM 时区无关
Instant now = Instant.now();

// ❌ LocalDateTime 不含时区信息，不要直接存数据库
LocalDateTime ldt = LocalDateTime.now(); // 这是本地时间，没有时区

// ✅ ZonedDateTime / OffsetDateTime 含时区
ZonedDateTime zdt = ZonedDateTime.now(ZoneOffset.UTC);
```

###### 7. 如何进行代码重构？

代码重构是在不改变外部行为的前提下，改善代码内部结构。

**重构的前提：有测试覆盖**

没有测试保障的重构是危险的，先补充单元测试/集成测试，再动代码。

**常见重构技术（《重构》书中经典手法）：**

```java
// ❌ 提炼函数前：一个方法做了太多事
public void processOrder(Order order) {
    // 验证
    if (order.getAmount().compareTo(BigDecimal.ZERO) <= 0) throw new InvalidOrderException();
    if (order.getItems().isEmpty()) throw new EmptyOrderException();
    
    // 计算
    BigDecimal total = order.getItems().stream()
        .map(item -> item.getPrice().multiply(new BigDecimal(item.getQuantity())))
        .reduce(BigDecimal.ZERO, BigDecimal::add);
    order.setTotalAmount(total);
    
    // 持久化
    orderMapper.insert(order);
    sendOrderCreatedEvent(order);
}

// ✅ 提炼函数后：每个方法职责单一
public void processOrder(Order order) {
    validateOrder(order);
    calculateTotal(order);
    saveOrder(order);
}
```

**代码坏味道和对应手法：**

| 坏味道 | 重构手法 |
|--------|----------|
| 方法太长（>20行）| 提炼函数（Extract Method）|
| 类太大（God Class）| 提炼类（Extract Class）|
| 重复代码 | 提炼方法/引入模板方法 |
| 过长参数列表 | 引入参数对象（Parameter Object）|
| 条件表达式复杂 | 以多态替换条件（Replace Conditional with Polymorphism）|
| 魔法数字 | 以常量替换魔法数（Replace Magic Number with Constant）|

**重构节奏：** 小步重构，每次只做一件事，随时可以提交。不要一次性大规模重构，风险太高。

###### 8. 如何进行技术选型？

技术选型的本质是**在约束条件下，选择最适合团队和业务的技术方案**。

**选型维度：**

**① 业务需求匹配度（首要）**

技术是手段，业务是目的。候选方案能不能解决当前的业务问题？能解决多少？

**② 团队熟悉度**

团队没人熟悉的技术，引入后学习曲线高，踩坑多，风险大。优先选团队已有一定了解的技术。

**③ 社区生态**

- Issue 是否活跃（GitHub star 数、近期 commit）
- 文档是否完整
- 有没有商业公司长期维护（Apache/CNCF/大公司背书）
- 国内有没有用户群、技术文章（出问题能找到资料）

**④ 性能和规模**

当前业务规模用哪个都够，考虑未来1-2年的增长，不要过度设计也不要选会成为瓶颈的方案。

**⑤ 运维成本**

引入一个新中间件，需要有人会运维（部署、监控、故障处理）。无运维经验的团队，托管云服务（RDS、Redis 云服务）优于自建。

**⑥ 实际测试（选型前的 POC）**

对于关键技术，做 Proof of Concept（概念验证），用接近生产的数据量测试性能，不要只看官方 benchmark。

**常见选型对话思路：**

> 面试官："为什么选 Kafka 而不是 RabbitMQ？"
> 
> 好的回答：说出场景（高吞吐日志收集/大数据管道），说出对比两者的差异，说出团队背景（熟悉度），说出是否有 POC 验证，最后给出结论。

###### 9. 如何设计接口文档？

好的接口文档能大幅降低前后端联调成本，减少沟通摩擦。

**OpenAPI / Swagger（推荐自动生成）：**

```java
// Spring Boot 3.x + springdoc-openapi
@Operation(summary = "创建订单", description = "根据购物车创建新订单")
@ApiResponse(responseCode = "200", description = "订单创建成功")
@ApiResponse(responseCode = "400", description = "参数错误")
@PostMapping("/order")
public ResponseEntity<OrderDTO> createOrder(
    @Parameter(description = "创建订单请求体", required = true)
    @Valid @RequestBody CreateOrderRequest req) { ... }
```

**接口文档规范内容：**

- **接口名称和说明**：这个接口做什么
- **请求方式和 URL**：GET/POST/PUT/DELETE + 路径
- **请求参数**：Header/Path/Query/Body 各类型参数，说明是否必填、数据类型、示例值
- **响应格式**：成功响应和错误响应的字段说明、示例
- **错误码**：业务错误码列表及含义
- **变更记录**：版本号、日期、变更内容

###### 10. 如何进行单元测试？

**单元测试三原则（AAA 模式）：**

```java
@Test
void testCalculateDiscount_FullReductionCoupon() {
    // Arrange（准备）：创建测试数据
    Order order = Order.builder().totalAmount(new BigDecimal("200")).build();
    CouponTemplate coupon = CouponTemplate.builder()
        .type(CouponType.FULL_REDUCTION)
        .discountAmount(new BigDecimal("30"))
        .threshold(new BigDecimal("100"))
        .build();
    
    // Act（执行）：调用被测方法
    BigDecimal discount = discountService.calculate(order, coupon);
    
    // Assert（断言）：验证结果
    assertThat(discount).isEqualByComparingTo(new BigDecimal("30"));
}
```

**Mock 外部依赖（Mockito）：**

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {
    
    @Mock
    private OrderMapper orderMapper;  // Mock 数据库操作
    
    @Mock
    private InventoryService inventoryService; // Mock 外部服务
    
    @InjectMocks
    private OrderService orderService; // 注入 Mock 的被测类
    
    @Test
    void testCreateOrder_Success() {
        // 设置 Mock 行为
        when(inventoryService.checkStock(anyLong(), anyInt())).thenReturn(true);
        doNothing().when(orderMapper).insert(any());
        
        // 执行
        Order result = orderService.createOrder(buildRequest());
        
        // 验证
        assertThat(result).isNotNull();
        verify(inventoryService).checkStock(anyLong(), anyInt()); // 验证调用了库存服务
        verify(orderMapper).insert(any(Order.class)); // 验证写了数据库
    }
}
```

**测试覆盖率：**

- 核心业务逻辑（计算折扣、状态变更）：90%+
- 工具类：100%
- Controller 层（薄层逻辑）：集成测试覆盖即可

**好的单元测试特征（FIRST 原则）：**
- **Fast**：运行快（毫秒级，不连 DB/Redis）
- **Isolated**：独立，不依赖外部环境，不依赖执行顺序
- **Repeatable**：可重复，任何环境跑结果相同
- **Self-validating**：有明确的 pass/fail 结论（不是打印日志手动看）
- **Timely**：及时，与代码同步编写（TDD）

###### 11. 如何实现自动化部署（CI/CD）？

**CI/CD 流程：**

```
代码提交 → Git Push → CI（自动构建+测试）→ CD（自动部署）
```

**Jenkins Declarative Pipeline（标准方案）：**

```groovy
pipeline {
    agent any
    
    environment {
        DOCKER_REGISTRY = 'registry.example.com'
        IMAGE_NAME = 'order-service'
    }
    
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        
        stage('Build & Test') {
            steps {
                sh 'mvn clean test'
                junit 'target/surefire-reports/**/*.xml'
                jacoco execPattern: 'target/jacoco.exec'
            }
        }
        
        stage('Build Docker Image') {
            steps {
                sh "docker build -t ${DOCKER_REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER} ."
                sh "docker push ${DOCKER_REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER}"
            }
        }
        
        stage('Deploy to Test') {
            steps {
                sh "kubectl set image deployment/order-service " +
                   "order-service=${DOCKER_REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER} " +
                   "-n test"
                sh "kubectl rollout status deployment/order-service -n test"
            }
        }
        
        stage('Deploy to Production') {
            when {
                branch 'main'
            }
            steps {
                input message: '确认部署到生产环境?', ok: '确认'
                sh "kubectl set image deployment/order-service ..."
            }
        }
    }
    
    post {
        failure {
            dingtalkSend message: "构建失败：${BUILD_URL}"
        }
    }
}
```

**GitHub Actions（代码托管在 GitHub 时更方便）：**

###### 12. 如何进行代码 Review？

代码 Review 的目的：**发现潜在问题、传播最佳实践、知识共享、保持代码库质量**。

**Review 关注点（优先级从高到低）：**

**① 正确性（最重要）**

- 代码逻辑是否正确？边界条件处理了吗？
- 并发安全性？（多线程场景下是否有竞态条件）
- 异常处理是否完整？资源是否释放？
- SQL 是否正确？有没有 N+1 问题？

**② 安全性**

- SQL 注入？XSS？
- 权限校验有没有漏洞？
- 敏感信息有没有打到日志？

**③ 可维护性**

- 代码可读性（命名是否清晰、注释是否必要）
- 有没有重复代码（DRY 原则）
- 方法是否太长、太复杂（圈复杂度）
- 测试覆盖是否充分

**④ 性能**

- 是否有明显的性能问题（全表扫描、N+1、重复计算）
- 是否有不必要的资源消耗（过多对象创建、大集合操作）

**Review 礼仪（影响协作效率）：**

- 用"建议"而非"命令"：`"考虑使用 Optional"` 而非 `"你应该用 Optional"`
- 具体说明原因：不只说"这里不好"，说明为什么不好、怎么改更好
- 区分必须修改（阻塞合并）和建议优化（可选）
- 肯定好的代码（不只找问题）
- 及时 Review（不要让 PR 等超过1天）

###### 13. 如何管理技术债务？

**技术债务的形成原因：** 历史遗留代码、临时方案变长期方案、业务压力下的快速交付、技术选型不当。

**识别和量化技术债务：**

- **代码质量分析工具**：SonarQube 扫描（bug、code smell、重复代码、覆盖率）
- **人工识别**：在 Code Review 中标记 `// TODO: 技术债 - 原因 - 计划时间`
- **问题追踪**：将技术债整理成 Issue/Ticket，进行定期评估

**技术债务管理策略：**

**① 童子军法则（随手清理）**

每次修改代码时，顺手改进周围的代码（重命名、提炼方法、补测试），让代码库逐步变好，而不是等到专项重构。

**② 专项重构 Sprint**

```
每个季度安排1个 Sprint 专门处理技术债
债务类型：优先解决影响开发效率的债（慢速构建、CI不稳定、关键路径代码难以理解）
```

**③ 不允许债务滚雪球**

定期（每月/每季度）评审技术债列表，对每项债务进行：
- 偿还（安排开发时间）
- 接受（低优先级，暂时搁置）
- 删除（已不相关的技术债）

**④ 技术债防止积累（代码门禁）**

在 CI 中加入 SonarQube 质量门禁：

```yaml
# sonar-project.properties
sonar.qualitygate.wait=true
# 质量门禁规则：
# - 新增代码覆盖率 > 70%
# - 新增代码没有阻断级别的 bug
# - 代码重复率 < 3%
```

未通过质量门禁的代码不能合并到主干，防止债务继续增加。

**沟通技术债的价值（面向管理层）：**

将技术债转化为业务语言：`"这部分代码每次需求都需要额外1天时间才能理解上下文，每季度在这里损失约5个人天。专项重构投入5天，可以在1个季度内回收成本。"` 用 ROI 说服管理层批准技术债偿还。
