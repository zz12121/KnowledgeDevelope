# 十一、网络与运维

###### 1. 在 Linux 命令行敲下一行命令，会进行哪些事情？

这道题问的是 Shell 命令的**完整执行流程**，考察操作系统基础知识。

以 `ls -la /home` 为例：

**① Shell 解析命令行**

1. Shell（bash/zsh）接收键盘输入
2. **词法分析**：将命令行按空白分割成 token：`["ls", "-la", "/home"]`
3. **语法分析**：识别命令名、选项、参数
4. **环境变量/别名展开**：如果 `ls` 有 alias（`alias ls='ls --color=auto'`），替换为实际命令
5. **通配符展开**：`*.java` 展开为匹配的文件列表

**② 查找可执行文件**

Shell 在 `$PATH` 中按顺序查找 `ls` 命令：
```
/usr/local/bin/ls → 不存在
/usr/bin/ls → 存在！→ 确定可执行文件路径
```

**③ 创建子进程（fork + exec）**

```c
pid_t pid = fork(); // 复制当前 Shell 进程，创建子进程
if (pid == 0) {
    // 子进程：用 ls 替换子进程的程序映像
    execve("/usr/bin/ls", ["ls", "-la", "/home"], envp);
}
```

- `fork`：复制父进程（Shell）的地址空间，写时复制（COW）
- `execve`：加载 `/usr/bin/ls` 的代码和数据，替换子进程的内存空间（地址空间完全替换）

**④ 程序执行**

1. 内核加载 ELF 可执行文件，设置程序计数器
2. 动态链接器（`ld.so`）加载依赖的共享库（如 `libc.so`）
3. `ls` 开始执行：调用 `opendir`/`readdir` 系统调用读取目录内容
4. 系统调用：用户态（Ring3）→ 内核态（Ring0），内核执行文件系统操作，返回结果
5. `ls` 格式化输出，调用 `write` 系统调用写到标准输出（stdout）

**⑤ 输出显示**

终端接收 stdout 的内容，渲染到屏幕。

**⑥ 进程退出**

`ls` 执行完毕，`exit(0)` 退出，内核回收进程资源。Shell（父进程）调用 `wait` 收取子进程退出状态，打印下一个命令提示符。

###### 2. 当客户端有重连机制，当服务意外宕机重新部署启动时，怎么避免流量洪峰？

服务重启后，所有等待中的客户端同时重连，形成"惊群效应"（Thundering Herd），可能再次压垮刚起来的服务。

**方案一：随机退避（Exponential Backoff with Jitter）**

客户端不是立即重连，而是等待一段随机时间：

```java
// 指数退避 + 随机抖动
public long getRetryDelay(int retryCount) {
    long baseDelay = 1000L; // 基础等待1秒
    long maxDelay = 60_000L; // 最大等待60秒
    // 指数增长：1s, 2s, 4s, 8s, ...
    long exponentialDelay = baseDelay * (1L << Math.min(retryCount, 6));
    // 加入随机抖动（0 ~ exponentialDelay/2），打散客户端重连时刻
    long jitter = ThreadLocalRandom.current().nextLong(exponentialDelay / 2);
    return Math.min(exponentialDelay + jitter, maxDelay);
}
```

大量客户端因为随机抖动分散在不同时间重连，流量洪峰被平摊。

**方案二：服务端限流（服务启动时限制接入速率）**

```java
// 服务启动时，设置较低的连接限制，随时间线性提升（流量预热）
@PostConstruct
public void startupWarmup() {
    int initialLimit = 100; // 初始只接受100 QPS
    int targetLimit = 5000; // 目标 5000 QPS
    
    // 30秒内从初始限制线性增长到目标
    scheduler.scheduleWithFixedDelay(() -> {
        currentLimit.incrementAndGet();
    }, 1, 1, TimeUnit.SECONDS);
}
```

Sentinel 支持"冷启动"（WarmUp）模式，服务刚启动时限流严格，慢慢放开：

```java
FlowRule rule = new FlowRule("myService");
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_WARM_UP);
rule.setWarmUpPeriodSec(30); // 30秒预热
rule.setCount(5000); // 目标 QPS
```

**方案三：负载均衡层控制（网关预热）**

新实例刚启动时，Nginx/网关给它分配较少的权重，随时间逐步增加：

```nginx
upstream backend {
    server new-instance:8080 weight=1;   # 新实例低权重
    server stable-instance:8080 weight=9; # 稳定实例高权重
}
```

**方案四：健康检查 + 就绪探针（K8s 场景）**

```yaml
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 30  # 等待30秒再开始检查
  periodSeconds: 5
```

服务启动完成（Spring Boot 应用完全初始化、缓存预热完成）后才通过就绪检查，K8s 才向该实例转发流量。

###### 3. 我们线上软件安装包地址，这几天访问量暴增，现在 CDN 流量一直异常，这是什么原因？如何解决？

**异常流量分析：**

CDN 流量异常暴增，常见原因：

**① 盗链（Hotlinking）**

其他网站/APP 直接引用你的下载 URL，他们的用户实际在消耗你的 CDN 流量。

排查：查 CDN 访问日志，分析 Referer 来源，找到异常 Referer 来源。

**② 爬虫/恶意下载**

自动化脚本大量重复下载同一文件。排查：查 UA（User-Agent）是否是脚本特征，IP 分布是否集中。

**③ DDoS 攻击**

攻击者大量请求下载地址，耗尽带宽。

**④ 正常业务增长**

产品真的火了，用户暴增。

**解决方案：**

**① 防盗链（最直接）**

```nginx
# Nginx 配置 Referer 白名单
location /download/ {
    valid_referers none blocked *.example.com;
    if ($invalid_referer) {
        return 403;
    }
}
```

CDN 厂商也提供 Referer 白名单配置，在 CDN 边缘节点拦截。

**② URL 鉴权（带时效签名的下载链接）**

```java
// 生成有时效性的下载 URL（有效期1小时）
public String generateSignedUrl(String objectKey, long expireMs) {
    long expire = System.currentTimeMillis() / 1000 + expireMs / 1000;
    String sign = md5(secretKey + objectKey + expire); // 签名
    return "https://cdn.example.com/" + objectKey + "?sign=" + sign + "&expire=" + expire;
}
```

CDN 或应用层验证签名和时间，过期或签名错误的请求直接拒绝。

**③ IP 限速和封禁**

CDN 控制台配置单 IP 下载速率限制和访问频率限制。对已识别的恶意 IP，加入黑名单。

**④ 登录鉴权**

软件下载需要先登录/注册，登录状态才能获取有效的下载链接，大幅提高爬虫成本。

**⑤ 成本控制（降级方案）**

流量异常时，临时切换到对象存储（OSS/S3）直传（成本更低），或暂停下载功能，等分析清楚原因后恢复。

###### 4. 公司目前有一些海外业务，从国内将数据发送到海外延迟比较大，如何处理这个问题？

**国际网络延迟的根本原因：** 物理距离（中国到欧美约10000km）、海底光缆带宽限制、路由节点多跳、GFW 影响跨境流量。

**解决方案：**

**① 就近部署（最根本的解法）**

在目标地区（如北美、欧洲、东南亚）的云厂商（AWS、Azure、GCP、阿里云）建立 Region，业务数据就近处理，不经过跨境网络：

```
用户请求 → 就近 CDN 节点 → 当地 Region 服务 → 当地 DB
                                    ↑
                              数据同步（异步，可接受延迟）
                              ← 国内 Region
```

**② CDN 加速（静态/准静态资源）**

将 API 网关、静态资源推到 CDN 边缘节点，用户请求就近响应，减少回源次数。

**③ 专线/加速网络（企业网络层方案）**

- **阿里云全球加速（GA）**：在云厂商骨干网内传输，绕过公网拥塞
- **SD-WAN**：企业自建软件定义广域网，优化跨境路由
- 相比公网，专线延迟可降低 40-60%

**④ 协议优化**

- HTTP/2 多路复用，减少连接建立次数
- QUIC/HTTP/3，TCP 改 UDP，减少握手轮次，对高延迟网络更友好
- 消息压缩（Gzip/Brotli），减少传输数据量

**⑤ 数据同步优化（异步化 + 最终一致性）**

如果业务允许，核心操作在当地 Region 完成，跨境数据同步异步进行：

```
国内写操作 → 国内DB → Binlog → Kafka（国内）→ 国际专线 → Kafka（海外）→ 海外DB
```

延迟可能几秒到几分钟，但用户操作响应是毫秒级的（不需要等跨境同步）。

###### 5. 如何实现服务的优雅上下线？

**优雅上线（预热后接流量）**

1. 新实例启动，完成初始化（连接池、缓存预热、配置加载）
2. 就绪探针通过后，注册到注册中心，通知 LB 加入集群
3. 可配合流量预热（低权重逐步增加），避免冷启动压力

**优雅下线（不中断进行中的请求）**

```java
// Spring Boot 2.3+ 内置优雅停机
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 等待进行中请求完成的超时时间
server:
  shutdown: graceful  # 开启优雅停机
```

优雅停机流程：
1. 收到 SIGTERM 信号
2. **从注册中心注销**（Nacos/Eureka），停止接受新请求
3. **等待进行中的请求完成**（最多等待 `timeout-per-shutdown-phase` 时间）
4. 关闭连接池、MQ 消费者、定时任务
5. 进程退出

**注意服务注册的传播延迟：** 服务注销后，消费者的本地缓存不会立即更新（Nacos 推送有延迟），可能还会有1-2秒的请求路由到下线服务。解决：下线前先从注册中心注销，等待几秒，再停止接受请求。

```bash
# K8s preStop hook：给一段时间让流量自然停止
lifecycle:
  preStop:
    exec:
      command: ["sh", "-c", "curl -X POST http://localhost:8080/actuator/shutdown; sleep 15"]
```

###### 6. 如何进行容量规划？

容量规划是估算系统在目标流量下所需的服务器资源，核心是**找到系统的性能瓶颈，然后按比例扩容**。

**步骤：**

**① 明确指标**

- 峰值 QPS：活动大促期间最高 QPS 是多少？
- 响应时间目标：P99 < 200ms
- 可用性目标：99.9%（一年不超过8.76小时不可用）

**② 基准压测（找单实例性能上限）**

```bash
# wrk 压测，观察吞吐量和响应时间
wrk -t8 -c200 -d60s http://service:8080/api/order/create

# 关键指标
# - 最大 QPS（吞吐量开始下降的点）
# - P99 响应时间
# - CPU 利用率、内存使用率
```

假设单实例在 CPU ~70% 时能稳定 500 QPS，P99 < 100ms。

**③ 计算所需实例数**

```
需要实例数 = 峰值 QPS / 单实例 QPS × 安全系数（一般1.5~2）
           = 5000 QPS / 500 QPS × 2
           = 20 个实例
```

**④ 预留弹性容量**

- 正常情况运行在 60-70% 负载（留 30-40% 余量应对突发）
- 配置 HPA（K8s Horizontal Pod Autoscaler）自动扩缩容
- 大促前预先扩容（提前评估预期流量倍数）

**⑤ 依赖资源容量**

- 数据库：根据 TPS 和并发连接数评估
- Redis：根据命令/s 和内存使用评估
- MQ：根据消息/s 和存储需求评估

**实际建议：** 不要只算 CPU，还要评估 DB 的 TPS、Redis 的 QPS、网络带宽。瓶颈往往不在应用服务器，而在数据库。

###### 7. 如何设计一个高可用的系统架构？

高可用（HA）的目标：**消除单点故障，保证系统在部分组件故障时仍能服务**。

**核心原则：**

**① 冗余（无单点）**

每个组件至少部署2个实例：
- 应用服务：多实例 + 负载均衡（Nginx、K8s Service）
- 数据库：主从 + 自动故障切换（MGR/MHA/PXC）
- Redis：Sentinel（3节点）或 Cluster（6节点）
- Nginx：主备（keepalived + VIP 漂移）

**② 熔断降级（故障隔离）**

下游服务超时/故障时，不让故障传播到上游，保护核心功能：

```java
@HystrixCommand(fallbackMethod = "getRecommendFallback",
                commandProperties = {
                    @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", value = "20"),
                    @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", value = "50")
                })
public List<Product> getRecommend(Long userId) { ... }
```

**③ 限流（防止过载）**

流量超出系统承载时，优先保证核心功能可用，非核心功能降级。

**④ 超时控制（快速失败）**

所有远程调用设置合理超时（如 HTTP 调用超时3秒），不要无限等待。超时后快速失败，释放线程资源。

**⑤ 健康检查 + 自动恢复**

```yaml
livenessProbe:   # 存活探针：失败则重启容器
  httpGet:
    path: /actuator/health/liveness
readinessProbe:  # 就绪探针：失败则从 LB 摘除
  httpGet:
    path: /actuator/health/readiness
```

**⑥ 异地多活（极高可用要求）**

两个及以上地域部署完整服务，流量就近接入，任一地域故障时，流量切换到其他地域（30秒内完成）。是大型互联网公司的标配，但成本高，数据一致性复杂。

###### 8. 如何实现服务的降级和熔断？

**降级（Degradation）：** 非核心功能在压力大或依赖不可用时，返回降级结果（默认值、缓存数据、静态页面），保证核心功能可用。

**熔断（Circuit Breaker）：** 监控调用失败率，超过阈值时"断开电路"，快速返回失败（不再实际调用），防止故障蔓延。

**三种状态：**

```
关闭（CLOSED）→ 正常调用，统计失败率
     ↓ 失败率超阈值
打开（OPEN）→ 直接快速失败（不实际调用）
     ↓ 等待恢复时间（如30秒）
半开（HALF_OPEN）→ 放少量请求尝试调用
     ↓ 成功率高，回到 CLOSED
     ↓ 失败，回到 OPEN
```

**Resilience4j 实现（Spring Boot）：**

```java
@CircuitBreaker(name = "inventoryService", fallbackMethod = "fallbackDeductStock")
@TimeLimiter(name = "inventoryService")
public CompletableFuture<Boolean> deductStock(Long productId, int quantity) {
    return CompletableFuture.supplyAsync(() -> 
        inventoryClient.deduct(productId, quantity));
}

// 降级方法：熔断/超时时执行
public CompletableFuture<Boolean> fallbackDeductStock(Long productId, int quantity, 
                                                       Throwable ex) {
    log.error("库存服务不可用，进入降级: {}", ex.getMessage());
    // 降级策略：走本地缓存估算、返回乐观结果（配合补偿）
    return CompletableFuture.completedFuture(true);
}
```

```yaml
resilience4j:
  circuitbreaker:
    instances:
      inventoryService:
        sliding-window-size: 10
        failure-rate-threshold: 50        # 失败率超50%触发熔断
        wait-duration-in-open-state: 30s   # OPEN 状态持续30秒
        permitted-number-of-calls-in-half-open-state: 3
```

**Sentinel（阿里巴巴，国内更流行）：**

```java
@SentinelResource(value = "queryProductDetail",
                  blockHandler = "blockHandler",     // 限流/熔断处理
                  fallback = "fallback")             // 降级处理
public Product queryProductDetail(Long productId) { ... }
```

Sentinel 提供实时控制台，可以动态调整规则（QPS阈值、熔断策略），生产运维很方便。

###### 9. 如何进行链路追踪？

链路追踪（Distributed Tracing）用于在微服务架构中跟踪一次请求经过的所有服务和调用链路。

**核心概念：**

- **Trace**：一次完整请求的全链路，由唯一 TraceId 标识
- **Span**：链路中的一个操作单元（一次 RPC 调用、一次 DB 查询），有起止时间
- **SpanContext**：通过 HTTP Header 在服务间传递 TraceId 和 SpanId

**Spring Boot 集成 SkyWalking（无侵入）：**

```bash
# JVM 启动参数添加 SkyWalking Agent（无需修改代码）
java -javaagent:/opt/skywalking/agent/skywalking-agent.jar \
     -Dskywalking.agent.service_name=order-service \
     -Dskywalking.collector.backend_service=skywalking-oap:11800 \
     -jar app.jar
```

SkyWalking Agent 通过字节码增强（ByteBuddy）自动拦截 HTTP 请求、数据库调用、RPC 调用等，无需业务代码改动。

**手动埋点（需要自定义 Span）：**

```java
// 使用 OpenTelemetry API
Tracer tracer = OpenTelemetry.getTracer("order-service");

Span span = tracer.spanBuilder("queryUserPoints").startSpan();
try (Scope scope = span.makeCurrent()) {
    span.setAttribute("userId", userId);
    long points = pointsService.queryPoints(userId);
    span.setAttribute("points", points);
    return points;
} catch (Exception e) {
    span.recordException(e);
    span.setStatus(StatusCode.ERROR);
    throw e;
} finally {
    span.end();
}
```

**日志与链路关联：**

通过 MDC 将 TraceId 注入到日志中，日志和链路追踪关联起来：

```java
// 在请求拦截器中注入 TraceId 到 MDC
MDC.put("traceId", span.getSpanContext().getTraceId());
// logback pattern: [%X{traceId}] %msg%n
// 日志：[a1b2c3d4e5f6] 订单创建成功
```

在 Kibana 搜索日志时可以按 TraceId 过滤，找到完整的请求日志。
