# 六、性能优化

###### 1. 一条 1s 请求的响应，怎么优化成 1ms？

这道题考的是**全链路性能优化思维**，从几个维度系统性地回答：

**先拆解 1s 花在哪里（分析是前提）**

用链路追踪（SkyWalking/Jaeger）拆解每段耗时，常见分布：
- 网络 IO（跨机房、TLS 握手）
- 数据库查询（未走索引/全表扫描）
- 外部调用（RPC、第三方 HTTP）
- 业务计算（CPU 密集/序列化）
- 锁等待/线程阻塞

**① 缓存（最有效的1000x提速手段）**

把慢查询（DB/ES/外部接口）的结果缓存到 Redis 或本地 Caffeine。DB 查询 100ms → Redis 命中 0.1ms。

```java
// 本地缓存：纳秒级
@Cacheable(cacheNames = "product", key = "#id", cacheManager = "caffeineCacheManager")
public Product getProduct(Long id) { ... }
```

**② 索引优化（DB 1000ms → 1ms）**

为慢 SQL 加合适的索引，`EXPLAIN` 检查执行计划。一条全表扫描 1s 的 SQL，加索引后可能 1ms 以内。

**③ 异步化（串行 → 并行）**

```java
// 原来串行：A(200ms) + B(300ms) + C(400ms) = 900ms
// 并行化后：max(A, B, C) = 400ms
CompletableFuture<A> fa = CompletableFuture.supplyAsync(() -> queryA());
CompletableFuture<B> fb = CompletableFuture.supplyAsync(() -> queryB());
CompletableFuture<C> fc = CompletableFuture.supplyAsync(() -> queryC());
CompletableFuture.allOf(fa, fb, fc).join();
```

**④ 减少 IO 次数**

- N+1 问题：一次查询拿所有 id，再批量 `IN` 查询，代替循环单条查询
- HTTP 合并：多次 API 调用合并成一次批量接口
- 数据库批量操作：`INSERT ... VALUES (...),(...),...` 代替循环 INSERT

**⑤ 预计算（把运行时计算变成提前算）**

排行榜不要每次实时聚合，用 Redis ZSet 维护，写入时更新，查询时 `O(log N)` 直接取。

**⑥ CDN + 静态化**

接口返回的数据如果变化不频繁，做 HTTP 缓存（`Cache-Control: max-age=60`）或 CDN 缓存，请求不到后端。

**⑦ 减少序列化开销**

JSON 序列化/反序列化有一定 CPU 开销，高频接口可以考虑 Protobuf（比 JSON 快 5-10x，但可读性差）。

**面试结语：** "1ms"是目标，方向是缓存+索引+并行化+减 IO。具体优化多少取决于瓶颈在哪，先用工具找瓶颈，再针对性优化，不要盲目优化。

###### 2. 除了代码，还有哪些地方可以优化性能？

这道题考的是**系统视角的性能优化**，不只是代码层面。

**① 基础设施层**

- **硬件升级**：SSD 替换 HDD（I/O 性能 10-100x 提升）；增加内存（减少 GC 频率、增大 Buffer Pool）；网卡升级（10Gb 以太网代替 1Gb）
- **CPU 绑定**：对延迟敏感应用，CPU 亲和性设置，减少跨 NUMA 节点的内存访问

**② 操作系统层**

```bash
# 调整 TCP 连接参数，提升网络性能
sysctl -w net.core.somaxconn=65535
sysctl -w net.ipv4.tcp_tw_reuse=1  # TIME_WAIT 连接复用

# 调整文件描述符上限
ulimit -n 1048576

# 大内存页（HugePage）减少 TLB miss，对内存密集型应用有帮助
```

**③ 网络层**

- **CDN**：将静态资源下沉到边缘节点，减少跨地域延迟
- **HTTP/2 or gRPC**：多路复用，减少连接建立开销
- **连接复用（Keep-Alive）**：HTTP 长连接、数据库连接池，避免频繁 TCP 握手

**④ 数据库层**

- **主从架构**：读写分离，扩展读能力
- **分库分表**：水平扩展，突破单库限制
- **数据库参数调优**：`innodb_buffer_pool_size` 设为物理内存的 60-80%；`max_connections` 合理配置

**⑤ 中间件层**

- **Redis 集群**：横向扩展缓存容量和读写能力
- **MQ 消峰**：用 MQ 削平流量峰值，保护下游
- **Nginx 负载均衡**：upstream 多节点，合理配置 `keepalive`、`worker_processes`

**⑥ 架构层**

- **微服务拆分**：细粒度扩缩容，热点服务独立扩容
- **异步解耦**：同步调用改成 MQ 异步，降低响应时间
- **读写分离**：查询和写入流量走不同路径
- **多级缓存**：CDN + 本地缓存 + Redis 三级架构

**⑦ 部署层**

- **就近部署**：将服务部署在靠近用户的区域（多 Region）
- **K8s HPA**：根据 CPU/QPS 自动水平扩缩容，弹性应对流量峰值
- **资源隔离**：CPU Request/Limit、内存限制，防止 noisy neighbor 问题

###### 3. 4核8G 和 8核16G，这两个配置的服务器如何选择？

这道题考的是**服务器配置与业务场景的匹配**，没有通用答案，要根据服务性质判断。

**分析维度：**

**① 服务类型**

- **CPU 密集型**（图像处理、加密解密、AI 推理、数据压缩）→ 更多核更重要，选 8核16G
- **I/O 密集型**（CRUD、查 DB、调外部接口）→ 线程大多在等待，CPU 利用率不高，4核8G 够用，更多靠增加内存和网络

**② 内存使用**

- Java 应用堆内存至少 4G 起步（GC 停顿频率）
- 如果有 Redis、Elasticsearch 等内存密集型服务，内存是瓶颈
- JVM Metaspace、Off-heap Buffer 也占内存，实际可用 Heap 远小于物理内存

**③ 并发量**

- 低并发（< 100 QPS）→ 4核8G 绰绰有余
- 中等并发（100-1000 QPS）→ 4核8G 可以，看响应时间
- 高并发（> 1000 QPS）→ 8核16G 或多实例水平扩展

**④ 成本**

8核16G 的价格通常是 4核8G 的 2-3 倍。两台 4核8G（总成本相当）配合负载均衡，不仅性能不亚于一台 8核16G，还有高可用保障。

**工程师答法：**

单台服务器升配（Scale Up）和多台低配服务器（Scale Out）的取舍：
- 无状态服务（API 服务）→ 优先水平扩展（多台 4核8G + LB），可用性更好
- 有状态服务（DB 主库）→ 垂直升配（8核16G 更稳定，减少节点数复杂性）

**实际建议：** 先用 4核8G 跑压测，看 CPU 和内存的实际利用率，CPU 跑满就加核，内存跑满就加内存。不要提前过度配置。

###### 4. 如何进行 JVM 性能调优？

JVM 调优的目标：**降低 GC 停顿时间、减少 Full GC 频率、避免 OOM**。

**① 堆内存大小设置**

```bash
# 堆内存初始值和最大值设为相同，避免动态扩容的 STW
-Xms4g -Xmx4g

# 年轻代占总堆的 1/3 ~ 1/2（应用以短生命周期对象为主）
-Xmn2g

# 或者使用比例控制（G1 不需要手动设置年轻代大小）
-XX:NewRatio=2
```

**② 选择合适的 GC 算法**

```bash
# G1（JDK 9+ 默认，推荐大堆）
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200  # 目标最大停顿时间200ms
-XX:G1HeapRegionSize=16m  # 适合 8G 以上的堆
-XX:InitiatingHeapOccupancyPercent=45  # 堆占用45%时触发并发标记

# ZGC（JDK 15+ 稳定，超低延迟，适合大内存）
-XX:+UseZGC
-XX:ZCollectionInterval=120  # 定期GC间隔（秒）

# CMS（已废弃，JDK 14 删除，不要再用）
```

**③ 监控与分析**

```bash
# 查看 GC 实时日志（JDK 9+）
-Xlog:gc*:file=/logs/gc.log:time,uptime:filecount=10,filesize=100m

# jstat 实时监控
jstat -gcutil <pid> 1000  # 每秒打印GC统计

# 分析 GC 日志工具：GCEasy（在线）、GCViewer（客户端）
```

**④ 常见问题排查**

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| Full GC 频繁 | 老年代太小/内存泄漏 | 增大堆/查内存泄漏 |
| Young GC 频繁但停顿短 | 对象创建速率高 | 增大年轻代/减少对象创建 |
| GC 停顿过长 | 堆太大/GC 算法不合适 | 换 G1/ZGC |
| OOM: Java heap space | 对象太多/堆太小 | 增大堆/排查内存泄漏 |
| OOM: Metaspace | 类加载过多（动态生成类）| 增大 Metaspace/-XX:MaxMetaspaceSize |

**⑤ JVM 参数模板（生产 Spring Boot 应用）**

```bash
JAVA_OPTS="-server
  -Xms4g -Xmx4g
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=200
  -XX:+HeapDumpOnOutOfMemoryError
  -XX:HeapDumpPath=/logs/heap-dump.hprof
  -Xlog:gc*:file=/logs/gc.log:time,uptime:filecount=10,filesize=50m
  -Djava.security.egd=file:/dev/./urandom"
```

###### 5. 如何定位和解决 CPU 100% 的问题？

**三步定位法（Linux + Java）：**

**第一步：找到 CPU 高的 Java 进程**

```bash
top  # 找到 CPU 占用最高的 Java 进程 PID，假设是 12345
```

**第二步：找到 CPU 高的线程**

```bash
top -H -p 12345  # 显示进程内所有线程的 CPU 占用
# 找到 CPU 最高的线程 TID，假设是 12389

# 将线程 TID 转为16进制（jstack 里用16进制）
printf "%x\n" 12389   # 结果：3065
```

**第三步：jstack 分析线程在做什么**

```bash
jstack 12345 > /tmp/stack.txt
grep -A30 "nid=0x3065" /tmp/stack.txt
```

看到线程栈，分析 CPU 高的原因：
- 死循环（while/for 没有退出条件）→ 找到对应代码修复
- 频繁 GC（大量线程在 GC）→ 堆内存不足，排查内存泄漏
- 正常业务计算（加密、序列化）→ 优化算法

**Arthas 一键分析（更方便）：**

```bash
# 启动 Arthas
curl -O https://arthas.aliyun.com/arthas-boot.jar && java -jar arthas-boot.jar

# 查看 CPU 最高的线程
thread -n 5  # 打印CPU占用最高的5个线程及其调用栈，无需手动转十六进制
```

**常见原因和解法：**

| 原因 | 排查方式 | 解决方案 |
|------|----------|----------|
| 死循环 | thread -n 看调用栈 | 修复循环逻辑 |
| 频繁 Full GC | jstat -gcutil 看 FGC | 增大堆/排查内存泄漏 |
| 正则表达式回溯 | thread 看方法 | 优化正则/增加长度限制 |
| 大量序列化 | trace 看耗时 | 减少序列化/换更快的库 |
| 线程创建过多 | thread 看线程数 | 限制线程池大小 |

###### 6. 如何定位和解决内存泄漏问题？

**内存泄漏**：对象不再被业务使用，但 GC Roots 仍有引用，无法被 GC 回收，导致堆内存持续增长，最终 OOM。

**定位步骤：**

**① 确认是否内存泄漏（还是内存不足）**

```bash
# 观察堆内存趋势（如果持续增长，Full GC 后也不回落）
jstat -gcutil <pid> 5000  # 每5秒打印一次
# Old 区持续增长 + Full GC 后也无法降低 → 内存泄漏
```

**② 获取堆 Dump**

```bash
# 方式1：手动触发（不停服）
jmap -dump:format=b,file=/tmp/heap.hprof <pid>

# 方式2：OOM 时自动 Dump（JVM 参数）
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/

# 方式3：Arthas
heapdump /tmp/heap.hprof
```

**③ 分析 Dump（MAT/VisualVM）**

用 Eclipse MAT（Memory Analyzer Tool）分析 heap.hprof：
- 查看 "Leak Suspects" 报告，自动推断疑似泄漏
- 查看 "Dominator Tree"，找到占用内存最大的对象
- 用 "OQL" 查询特定类的实例数量

```
# MAT 找到内存最大的对象链
SELECT * FROM java.util.HashMap WHERE size > 100000
```

**常见内存泄漏场景：**

**① ThreadLocal 未 remove**

```java
// 错误示例：线程池中 ThreadLocal 不会随请求结束而清理
private static final ThreadLocal<UserContext> context = new ThreadLocal<>();
// 使用完必须 remove，否则线程复用时旧值残留，且对象无法回收
context.remove(); // 务必在 finally 中调用
```

**② 静态集合无限增长**

```java
// 静态 Map 生命周期和 JVM 一样长，如果一直 put 但不 remove，就是泄漏
private static final Map<String, Object> CACHE = new HashMap<>();
// 应该用 Caffeine/Guava Cache（有淘汰策略）代替
```

**③ 未关闭的资源**

数据库连接、InputStream、Socket 未 close，资源对象无法被 GC，且操作系统的文件描述符耗尽。

**④ 内部类持有外部类引用**

非静态内部类/匿名类持有外部类的引用，外部类无法被回收（尤其是 Activity/Fragment 中的匿名 Handler——Android 常见问题）。

⑤ **WebappClassLoader 泄漏（Tomcat 重部署）**

热部署时 WebappClassLoader 没有正确被 GC，导致 Metaspace 增长。原因通常是第三方库的线程/定时器持有类加载器引用（如 JDBC Driver、log4j 定时任务）。

###### 7. Java 应用的内存持续性增长，但是监控显示堆内存没有什么变化，可能的原因有哪些？

堆内存正常，但进程总内存持续增长，说明增长的是**堆外内存（Off-Heap Memory）**。

**可能原因：**

**① Metaspace 持续增长**

动态生成类（如反射、CGLIB、Groovy/Groovy 脚本、JSP）导致 Metaspace 持续增长。

```bash
jstat -gcmetacapacity <pid>  # 查看 Metaspace 使用情况
```

解决：设置 `-XX:MaxMetaspaceSize=256m` 限制上限（超出会 OOM，至少能快速发现问题）；检查是否在循环中不断动态生成代理类。

**② 直接内存（DirectByteBuffer）**

NIO、Netty、MapDB 等大量使用堆外直接内存（Direct Memory），不受 `-Xmx` 限制。

```bash
# 设置直接内存上限
-XX:MaxDirectMemorySize=1g
```

```java
// 监控直接内存使用
MBeanServer mbeanServer = ManagementFactory.getPlatformMBeanServer();
ObjectName objectName = new ObjectName("java.nio:type=BufferPool,name=direct");
long directMemoryUsed = (long) mbeanServer.getAttribute(objectName, "MemoryUsed");
```

**③ 线程栈内存**

每个线程默认占 512KB-1MB 栈空间（`-Xss` 参数控制）。如果线程数持续增加（线程泄漏），总内存会增长。

```bash
jstack <pid> | grep "java.lang.Thread.State" | wc -l  # 统计线程数
```

**④ JNI/Native 内存**

应用使用了 JNI 调用本地库（如 libssl、图像处理库），本地内存分配不经过 JVM，无法用 JVM 工具监控。

**⑤ 内存映射文件（MappedByteBuffer）**

操作大文件时使用 `MappedByteBuffer`，文件内容映射到进程地址空间，不在堆内，但占用进程 RSS。

**诊断工具：**

```bash
# NMT（Native Memory Tracking）：追踪 JVM 内部各区域内存使用
-XX:NativeMemoryTracking=detail
jcmd <pid> VM.native_memory detail

# pmap：查看进程内存映射
pmap -x <pid> | sort -n -k 3 | tail -20

# 用 jemalloc/gperftools 分析 malloc 调用
```

###### 8. 接口响应时间慢如何排查和优化？

**排查步骤（从外到内）：**

**① 确定是偶发慢还是持续慢**

用监控（Prometheus + Grafana）看 P95/P99 响应时间趋势。偶发慢可能是 GC 停顿、网络抖动；持续慢是代码/DB 问题。

**② 链路追踪定位慢点**

SkyWalking 直接看该接口的 Trace，哪个 Span 耗时最多就优先优化那里。

**③ 常见慢点和解法**

| 慢点 | 排查 | 优化 |
|------|------|------|
| SQL 慢 | MySQL slow log + EXPLAIN | 加索引、优化 SQL、读写分离 |
| 缓存未命中 | 看缓存命中率 | 扩大缓存、预热、增大 TTL |
| 外部 HTTP 调用慢 | trace 看耗时 | 设置合理超时、异步化、熔断降级 |
| 串行改并行 | 看调用顺序 | CompletableFuture 并行调用 |
| 线程池满 | 看 waiting 任务数 | 扩大线程池/优化耗时操作 |
| GC 停顿 | GC 日志 | JVM 调优 |
| 大对象序列化 | arthas trace 看序列化耗时 | 减少数据量/换序列化框架 |

**④ Arthas trace（精确方法级分析）**

```bash
# 追踪 OrderService.createOrder 方法的每个子调用耗时
trace com.example.OrderService createOrder -n 5
```

**优化优先级（ROI 从高到低）：**

1. **加缓存**：DB 查询改为 Redis，效果立竿见影
2. **加索引**：慢 SQL 加索引，一行命令 1000x 提速
3. **并行化**：串行改并行，响应时间取决于最长的那个调用
4. **减少调用**：N+1 查询改批量，减少 DB/Redis 请求次数
5. **代码层优化**：算法优化、减少内存分配（效果最小，放最后）

###### 9. 如何优化 SQL 查询性能？

**① 走索引（最重要）**

```sql
-- 检查执行计划
EXPLAIN SELECT * FROM orders WHERE user_id = 123 AND status = 'PAID';
-- 关注：type（至少 ref/range，避免 ALL）、key（实际使用的索引）、rows（预估扫描行数）

-- 创建联合索引（区分度高的列放前面，常一起查询的列组合）
CREATE INDEX idx_user_status ON orders(user_id, status);
```

**② 避免索引失效**

```sql
-- ❌ 函数/运算导致索引失效
WHERE DATE(create_time) = '2026-03-15'   -- 对索引列用了函数
WHERE age + 1 = 18                        -- 对索引列做运算

-- ✅ 改写为范围查询
WHERE create_time >= '2026-03-15' AND create_time < '2026-03-16'
WHERE age = 17

-- ❌ 前导通配符 LIKE 索引失效
WHERE name LIKE '%张三%'   -- 前缀通配符，走不了索引

-- ✅ 改为后缀匹配（能走索引）或走全文索引/ES
WHERE name LIKE '张三%'
```

**③ 优化深分页**

```sql
-- ❌ 深度分页：LIMIT 1000000, 20 要扫描并丢弃100w行
SELECT * FROM orders ORDER BY id DESC LIMIT 1000000, 20;

-- ✅ 游标分页：WHERE id < 上次最小ID LIMIT 20
SELECT * FROM orders WHERE id < #{lastId} ORDER BY id DESC LIMIT 20;

-- ✅ 子查询优化（需要跳页时）
SELECT * FROM orders 
WHERE id >= (SELECT id FROM orders ORDER BY id LIMIT 1000000, 1)
LIMIT 20;
```

**④ 避免 SELECT ***

```sql
-- ❌ SELECT * 可能导致回表（覆盖索引失效）且传输不必要的数据
SELECT * FROM orders WHERE user_id = 123;

-- ✅ 只查需要的列，如果列都在索引里，可以走覆盖索引（不需要回表）
SELECT id, order_no, status, amount FROM orders WHERE user_id = 123;
```

**⑤ JOIN 优化**

```sql
-- 小表驱动大表（MySQL Nested Loop Join 外层行数越少越快）
-- 确保被 JOIN 的字段有索引
SELECT o.*, u.name FROM orders o 
INNER JOIN users u ON o.user_id = u.id  -- user.id 有主键索引，OK
WHERE o.status = 'PAID';

-- 避免笛卡尔积（忘了写 ON 条件）
```

**⑥ 批量操作**

```sql
-- ❌ 循环单条 INSERT
INSERT INTO orders VALUES (1, ...);
INSERT INTO orders VALUES (2, ...);

-- ✅ 批量 INSERT（减少网络往返和事务开销）
INSERT INTO orders VALUES (1, ...), (2, ...), (3, ...);

-- ❌ IN 子句元素过多（>1000个）会很慢
SELECT * FROM orders WHERE id IN (1, 2, ..., 10000);

-- ✅ 分批查询（每批500个）
```
