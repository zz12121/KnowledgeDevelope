###### 1. 什么是主从复制？

[[../../../21_MySqlKnowledge/04_日志与高可用/02、主从复制原理|📖]]
主从复制是把一个数据库服务器（主库）的数据实时同步到一个或多个其他服务器（从库）的技术。它解决了三个核心问题：

**高可用**：主库挂了可以快速切换到从库继续服务，减少停机时间。

**读写分离**：写操作走主库，大量读操作分摊到多个从库，显著提升读并发能力。电商、内容网站这类读多写少的业务非常依赖这个架构。

**数据安全与备份**：在从库上做备份，不影响主库的线上服务；从库本身也是一份实时热备数据。

###### 2. MySQL 主从复制的原理是什么？

[[../../../21_MySqlKnowledge/04_日志与高可用/02、主从复制原理|📖]]
主从复制的本质是：主库把所有数据变更记录到 Binlog，从库把 Binlog 同步过来并重放，两边数据就保持一致了。这个过程依赖三个线程：

**主库的 Binlog Dump 线程**：从库连上来之后，主库为每个从库创建一个 Dump 线程，负责读取 Binlog 事件并发送给从库的 I/O 线程。

**从库的 I/O 线程**：接收主库发来的 Binlog 事件，写入本地的 Relay Log（中继日志）。Relay Log 格式和 Binlog 完全一样，是主库 Binlog 在从库的本地副本。

**从库的 SQL 线程**：读取 Relay Log 里的事件，在本地执行，让从库的数据追上主库。

I/O 线程和 SQL 线程分工的好处是解耦——网络传输和执行可以独立进行，互不等待。

###### 3. 主从复制有哪些模式？

[[../../../21_MySqlKnowledge/04_日志与高可用/02、主从复制原理|📖]]
主从复制可以从两个维度分类：

**按同步强度**分三种：异步复制（主库不等从库，性能最好但可能丢数据）、半同步复制（主库等至少一个从库确认收到，平衡性能和安全）、全同步复制（等所有从库都执行完才返回，最安全但最慢）。

**按 Binlog 格式**分三种：Statement（记原始 SQL，日志小但可能主从不一致）、Row（记行数据变化，精准但日志量大）、Mixed（自动选，推荐的默认方式）。

**按架构**分：一主多从（最常见）、级联复制（从库再接从库，减轻主库 Dump 线程压力）、双主（互为主从，用于多活架构，需要处理冲突）。

###### 4. 什么是异步复制、半同步复制、全同步复制？

[[../../../21_MySqlKnowledge/04_日志与高可用/02、主从复制原理|📖]]
区别核心在于：主库提交事务后，要等到什么程度才向客户端返回成功。

**异步复制**：主库提交事务，立刻返回客户端成功，不管从库有没有收到 Binlog。性能最好，但如果主库在 Binlog 发出去之前崩溃，已提交的数据就可能丢失，切换到从库时数据会缺失。

**半同步复制**：主库提交事务后，要等至少一个从库确认"已收到并写入 Relay Log"了，才返回客户端成功。至少保证了数据有一个副本，主库崩溃切换时不会丢数据。性能有一定损失（多了一个网络往返），生产上推荐使用。MySQL 5.7 引入了**无损半同步**，把等待时机提前到存储引擎提交之前，安全性进一步提升。

**全同步复制**：等所有从库都执行完事务才返回。数据安全性最高，但性能极差，从库越多越慢，几乎不在生产中用。

###### 5. 主从复制的延迟问题如何解决？

[[../../../21_MySqlKnowledge/04_日志与高可用/02、主从复制原理|📖]]
主从延迟是生产中最常见的痛点，`SHOW SLAVE STATUS` 里的 `Seconds_Behind_Master` 反映延迟时间。

延迟的主要原因：从库硬件（尤其是磁盘 I/O）不如主库；主库有大事务，从库 SQL 线程要等一样长的时间重放；传统模式下从库 SQL 线程是单线程，跟不上主库并发写入速度。

解决手段：

**开启多线程复制（MTS）** 是最有效的。MySQL 5.7 的 `LOGICAL_CLOCK` 并行模式让同一时间窗口内没有冲突的事务并行重放，显著缩短延迟。配置：`slave_parallel_workers = 8`，`slave_parallel_type = LOGICAL_CLOCK`。

**拆分大事务**：批量 DML 分批执行，每批不超过 1000~5000 行，避免单个事务太大拖慢从库。

**确保从库硬件不弱于主库**：特别是 SSD 磁盘，I/O 性能差的从库永远追不上。

**主从同机房低延迟网络**：网络延迟直接叠加到复制延迟上。

**监控告警**：用 `pt-heartbeat` 工具精确测量实际延迟（比 `Seconds_Behind_Master` 更可靠），延迟超阈值立即告警。

###### 6. 如何保证主从数据的一致性？

[[../../../21_MySqlKnowledge/04_日志与高可用/02、主从复制原理|📖]]
除了减少延迟，还要主动做一致性校验：

**使用 GTID 复制**：GTID（全局事务标识符）给每个事务分配全局唯一 ID，主从切换时从库可以自动定位同步点，不用人工找 Binlog 文件名和 Position，避免了人为操作出错。这是现代 MySQL 主从配置的标准做法。

**定期校验**：用 Percona Toolkit 的 `pt-table-checksum` 工具定期校验主从数据是否一致，发现不一致后用 `pt-table-sync` 修复从库数据。

**从库设为只读**：设置 `read_only = ON`，防止误操作在从库写入数据导致不一致。

**读写分离用中间件管控**：应用层通过中间件路由，写操作只能走主库，从根源上防止误写从库。

###### 7. 什么是读写分离？如何实现？

[[../../../21_MySqlKnowledge/04_日志与高可用/02、主从复制原理|📖]]
读写分离建立在主从复制基础上，把写操作路由到主库，读操作路由到从库，分摊主库压力、提升读并发能力。

实现方式主要三种：

**应用层封装**：在 Java 代码里用 Spring 的 `AbstractRoutingDataSource` 做多数据源路由，写操作走主库数据源，读操作走从库数据源。结合 AOP + `@Transactional(readOnly = true)` 注解自动切换。好处是灵活可控，缺点是有一定代码入侵。

**数据库中间件代理**：在应用和数据库之间加一层代理（ShardingSphere-Proxy、MyCat、MySQL Router），代理自动解析 SQL 并路由，对应用完全透明。适合需要同时做分库分表的场景，但增加了运维复杂度。

**驱动层路由**：MySQL 官方 Connector/J 支持在连接字符串里配置主从地址，驱动根据 `readOnly` 属性自动路由。简单但功能有限，不够灵活。

###### 8. 读写分离可能遇到的问题有哪些？

[[../../../21_MySqlKnowledge/04_日志与高可用/02、主从复制原理|📖]]
读写分离最常见的坑是**主从延迟导致读到旧数据**：用户刚写入一条数据，马上查询，请求落到还没同步好的从库，就看不到刚写的数据了。

应对策略：对"刚写完立刻要读"的核心操作（比如下单后跳转订单详情页），强制走主库读；允许短暂延迟的非核心读请求走从库；用半同步复制缩小延迟窗口。

其他问题：**事务内必须走主库**，事务里的读操作如果路由到从库，读到的可能是旧数据；**多个从库负载均衡**，用加权轮询或随机策略分配；**从库宕机**，中间件要有健康检查和自动摘除能力，保证服务不中断。

###### 9. 什么是 MySQL 的高可用方案（如 MHA、MGR）？

[[../../../21_MySqlKnowledge/04_日志与高可用/03、高可用方案（MHA_MGR）|📖]]
主从复制是基础，但主库挂了需要自动切换才叫高可用，这就需要高可用方案。

**MHA（Master High Availability）**：由 Node 节点（部署在每台 MySQL 服务器上）和 Manager 节点（监控和调度）组成。Manager 定期探测主库状态，发现主库故障后自动选出数据最新的从库晋升为新主库，其他从库指向新主。优点是成熟稳定，切换快（一般 30 秒内），对现有架构改动小。缺点是需要配合 VIP 或 DNS 做服务发现，运维配置有一定复杂度，Manager 本身也是单点（需要额外保障）。

**MGR（MySQL Group Replication）**：基于 Paxos 协议的分布式状态机复制，多个节点组成一个复制组，事务提交必须经过多数节点（N/2+1）同意才能提交，数据强一致性由协议保证。任意节点都可以读写，故障时自动选举新主，对应用透明。优点是原生高可用，数据强一致，不依赖外部组件；缺点是对网络质量要求高，写操作有多节点协商的延迟，并发写入多时可能有冲突回滚。

选型建议：对稳定性和可控性要求高、有完善 DBA 团队的场景用 MHA；新建系统追求原生高可用和运维自动化，推荐用 MGR 或 InnoDB Cluster（MGR 的官方封装版）。
