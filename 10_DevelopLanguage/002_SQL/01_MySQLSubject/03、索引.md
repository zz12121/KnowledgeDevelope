###### 1. 什么是索引？索引的作用是什么？

[[../../../21_MySqlKnowledge/02_索引与查询优化/01、索引原理与数据结构|📖]]
索引是一种帮助 MySQL 高效获取数据的数据结构，类比书籍的目录——你不会为了找一篇文章从第一页翻到最后，而是先看目录，直接定位到对应页码。索引的本质就是以空间换时间，把查询复杂度从 O(n) 降到 O(log n)。

索引的好处显而易见：**提升查询速度**（避免全表扫描）、**保证唯一性**（唯一索引）、**加速 JOIN**（关联字段有索引时效率提升明显）、**优化排序和分组**（避免额外的 filesort 操作）。

但代价也要记住：每个索引都占额外磁盘空间，而且每次 INSERT、UPDATE、DELETE 都要同步维护索引结构，写操作会变慢。所以索引不是越多越好，要根据查询模式有的放矢地建。

###### 2. MySQL 有哪些索引类型？

[[../../../21_MySqlKnowledge/02_索引与查询优化/02、索引类型与使用策略|📖]]
按功能和约束来划分，MySQL 主要有这几种索引：

**主键索引（PRIMARY KEY）**：唯一且不允许 NULL，每张表只能有一个。在 InnoDB 里，主键索引就是聚簇索引，数据行直接存在叶节点里。

**唯一索引（UNIQUE）**：保证字段值唯一，允许有一个 NULL。适合邮箱、身份证号、手机号这类有唯一性要求的字段。

**普通索引（INDEX）**：最基本的索引，没有任何约束，就是为了加速查询用的。

**联合索引（复合索引）**：在多列上联合建索引，遵循最左前缀原则，能同时优化多条件查询，合理设计还能实现覆盖索引。

**全文索引（FULLTEXT）**：用于全文搜索，支持自然语言分析，适合文章、评论这类长文本内容的模糊搜索。注意 MySQL 的全文索引对中文支持有限，生产中大规模全文检索一般用 Elasticsearch。

**空间索引（SPATIAL）**：用于地理空间数据类型（GEOMETRY、POINT 等），GIS 系统会用到。

###### 3. B+树索引的原理是什么？

[[../../../21_MySqlKnowledge/02_索引与查询优化/01、索引原理与数据结构|📖]]
B+树是 InnoDB 默认的索引数据结构，是一种为磁盘 I/O 优化设计的多路平衡查找树。它有几个关键特点：

**非叶节点只存 Key 和指针，不存数据**。这样每个节点能装更多 Key，树的高度更矮，意味着查询时磁盘 I/O 更少。一个 16KB 的数据页能存几百个 Key，三层 B+树大约能索引 2000 万行数据，查找只需最多 3 次 I/O。

**所有数据都在叶节点**。叶节点存储完整数据行（聚簇索引）或主键值（二级索引），非叶节点只起导航作用。

**叶节点之间用双向链表连接**。这是 B+树支持高效范围查询的关键——找到范围起点后，沿着链表顺序扫描就行，不需要回到根节点重新遍历。

查找过程：从根节点出发，每个节点内做二分查找定位区间，顺着指针往下走，直到叶节点，找到数据或主键值。高度固定，查询性能稳定。

###### 4. 为什么 InnoDB 选择 B+树而不是 B树或哈希表？

[[../../../21_MySqlKnowledge/02_索引与查询优化/01、索引原理与数据结构|📖]]
这是一个数据结构选型的经典问题，核心考量是磁盘数据库的访问特性。

**B树 vs B+树**：B 树的每个节点（包括非叶节点）都存数据，这导致每个节点能存的 Key 数量减少，树高度增加，磁盘 I/O 次数变多。而且 B 树不支持叶节点链表，范围查询要来回遍历树，效率很低。B+树非叶节点只存 Key，扇出更大，树更矮，范围查询也因为链表而效率很高，综合来看 B+树完胜。

**哈希表 vs B+树**：哈希表的等值查询确实是 O(1)，极快，InnoDB 也有自适应哈希索引作为补充。但哈希表有致命缺点：不支持范围查询和排序、不支持最左前缀匹配，而这些是业务中非常普遍的需求。所以哈希索引无法作为主要的索引结构，只能做缓存层的补充加速。

结论：B+树在范围查询、排序、磁盘 I/O 优化这几个维度的综合表现最佳，是数据库索引的最优解。

###### 5. 什么是聚簇索引和非聚簇索引？

[[../../../21_MySqlKnowledge/02_索引与查询优化/01、索引原理与数据结构|📖]]
这是根据数据存储方式做的划分，理解这个是理解 InnoDB 索引的基础。

**聚簇索引**：索引结构和数据行存在一起，叶节点直接存完整数据行。InnoDB 里主键索引就是聚簇索引，数据按主键顺序物理排列，一张表只能有一个聚簇索引。通过主键查询非常快，只需一次 B+树查找。

**非聚簇索引（二级索引）**：索引和数据分开存，叶节点只存"索引列的值 + 对应行的主键值"，不存完整数据。查询时先在二级索引树里找到主键，再拿主键去聚簇索引里找完整数据，这个过程叫**回表**，需要两次 B+树查找。

所以二级索引查询有个优化方向：如果查询需要的所有列都在索引里（称为覆盖索引），就不需要回表，性能大幅提升。

###### 6. 主键索引和普通索引的区别？

[[../../../21_MySqlKnowledge/02_索引与查询优化/01、索引原理与数据结构|📖]]
区别主要体现在三个维度：

**约束性**：主键索引唯一且不允许 NULL；普通索引允许重复值，也允许 NULL。

**数量**：每张表只能有一个主键索引；普通索引可以建多个。

**存储结构**（InnoDB 特有）：主键索引就是聚簇索引，数据行直接存在叶节点里；普通索引是二级索引，叶节点存的是主键值，查完整数据需要回表。

###### 7. 什么是覆盖索引？

[[../../../21_MySqlKnowledge/02_索引与查询优化/02、索引类型与使用策略|📖]]
覆盖索引是一种优化技术，指查询所需要的全部列都包含在索引里，MySQL 可以直接从索引返回结果，不需要回表读完整行。

举个例子，如果在 `users` 表的 `(name, age)` 上建了联合索引，那么 `SELECT name, age FROM users WHERE name = 'Alice'` 就是覆盖索引查询——要的 `name` 和 `age` 都在索引的叶节点里，不需要拿主键回表。用 `EXPLAIN` 看的话，`Extra` 列会显示 `Using index`。

覆盖索引能显著减少 I/O，对于大表查询效果尤其明显。设计联合索引时，把查询频率高的列组合进去，尽量实现覆盖索引，是一种常见的优化套路。

###### 8. 什么是联合索引？最左前缀原则是什么？

[[../../../21_MySqlKnowledge/02_索引与查询优化/02、索引类型与使用策略|📖]]
**联合索引**就是在多列上联合建的一个索引。比如 `INDEX idx_a_b_c (a, b, c)`，这三列按顺序组合成一个索引树，先按 a 排序，a 相同再按 b，b 相同再按 c。

**最左前缀原则**：使用联合索引必须从最左边的列开始匹配，不能跳过中间的列。举几个具体例子：

`WHERE a = 1` 能用索引（用到 a 列）；`WHERE a = 1 AND b = 2` 能用（用到 a、b 列）；`WHERE a = 1 AND b = 2 AND c = 3` 能用（三列全用到）。

`WHERE b = 2` 不能用索引（没有从 a 开始）；`WHERE a = 1 AND c = 3` 只能用到 a 列，c 用不到（跳过了 b）。

`WHERE a = 1 AND b > 2 AND c = 3` 用到 a 和 b，c 用不到（b 是范围查询，后面的列无法继续利用索引排序）。

这个原则的本质是 B+树的排序特性——索引先按 a 排，a 相同才看 b，所以跳过 a 直接用 b 查，根本没有有效的排序规律可以利用。

###### 9. 索引失效的常见场景有哪些？

[[../../../21_MySqlKnowledge/02_索引与查询优化/02、索引类型与使用策略|📖]]
即使建了索引，写法不对也会让优化器放弃使用索引，退化成全表扫描。常见的几种情况：

**对索引列使用函数或表达式**：`WHERE YEAR(create_time) = 2023` 不走索引，改成 `WHERE create_time >= '2023-01-01' AND create_time < '2024-01-01'` 才行。

**隐式类型转换**：`code` 是 `varchar` 类型，却写 `WHERE code = 100`（没加引号），MySQL 会把 code 列的值都转成数字再比较，索引失效。

**LIKE 以 `%` 开头**：`WHERE name LIKE '%张'` 不走索引，`WHERE name LIKE '张%'` 才走。前缀模糊查询破坏了 B+树的有序性。

**OR 一侧无索引**：`WHERE a = 1 OR b = 2`，如果 b 没有索引，整个查询可能退化为全表扫描（MySQL 会根据代价决定）。

**违反最左前缀原则**：联合索引没从最左列开始用，前面已详细说过。

**索引列参与运算**：`WHERE id + 1 = 5` 不走索引，改成 `WHERE id = 4`。

**NOT IN 和 NOT EXISTS**：在某些情况下会导致索引失效，尽量用等值条件替代。

###### 10. 如何创建和删除索引？

```sql
-- 创建普通索引
CREATE INDEX idx_email ON users(email);

-- 创建唯一索引
CREATE UNIQUE INDEX uni_mobile ON users(mobile);

-- 创建联合索引
CREATE INDEX idx_name_age ON users(name, age);

-- 删除索引
DROP INDEX idx_email ON users;

-- 查看表索引
SHOW INDEX FROM users;
```

###### 11. 什么情况下不适合建立索引？

[[../../../21_MySqlKnowledge/02_索引与查询优化/02、索引类型与使用策略|📖]]
索引不是免费的，以下几种情况建了反而是负担：

**数据量很小的表**：几百行的表全表扫描比走索引还快，因为走索引额外有一次回表开销。

**写多读少的表**：每次写操作都要维护索引，如果写操作占大头，大量索引反而拖慢整体性能。

**区分度极低（选择性差）的列**：比如"性别"列只有两个值，即使走了索引，每次还是要扫描一半的数据，优化器一般会直接选择全表扫描。

**很少出现在 WHERE 条件里的列**：建了也没人用，白占空间。

###### 12. 索引下推（Index Condition Pushdown）是什么？

[[../../../21_MySqlKnowledge/02_索引与查询优化/03、查询优化器与执行计划|📖]]
索引下推（ICP）是 MySQL 5.6 引入的优化，核心思想是把 WHERE 里的过滤条件尽量"下推"到存储引擎层处理，减少回表次数。

以联合索引 `(name, age)` 为例，查询 `WHERE name LIKE '张%' AND age > 20`：

**没有 ICP**：存储引擎根据 `name` 条件定位到所有姓张的记录，不管 age 是多少，每条都先回表读完整行，再由 Server 层用 `age > 20` 过滤。如果姓张的人很多但年龄不符合条件的多，就做了大量无效回表。

**有 ICP**：存储引擎在索引内部就用 `age > 20` 做过滤，只有同时满足 name 和 age 两个条件的记录才回表查完整行。回表次数大幅减少，I/O 节省明显。

用 `EXPLAIN` 看，Extra 列显示 `Using index condition` 就说明 ICP 生效了。
