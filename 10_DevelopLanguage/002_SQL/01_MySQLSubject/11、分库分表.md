###### 1. 什么是分库分表？为什么需要分库分表？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
分库分表是把一个大数据库按业务或数据量拆分成多个小数据库/小表，分布到不同的服务器上，解决单机数据库的性能和存储瓶颈。

**为什么需要**：单表数据量超千万行后，B+树高度增加，查询慢；高并发写入导致大量锁冲突；单机存储容量跟不上数据增长；大数据库备份恢复耗时太长。

但这是最后的手段，代价是系统复杂度急剧上升。上分库分表之前，先把索引优化、读写分离、缓存、冷热数据归档、升级硬件这些方案都试一遍。只有当这些都无法解决问题了，才考虑分库分表。

###### 2. 垂直分库和水平分库的区别？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
**垂直分库**：按业务模块拆，不同业务的表分到不同的数据库。比如电商系统拆成用户库、商品库、订单库、支付库，各库之间库表结构不同，各自独立演进。好处是业务清晰、单库压力降低；坏处是跨库 JOIN 做不了，跨库事务变成分布式事务，复杂度大增。

**水平分库**：把同一张表的数据按规则（按 ID 哈希、按时间范围等）拆分到多个库，每个库表结构完全相同，只是存的数据行不同。比如订单表按 `order_id % 4` 分到 4 个库里，每个库只存四分之一的订单。解决单表数据量过大问题，但跨分片的查询、排序、分页都很麻烦。

两者经常结合使用：先垂直分库按业务解耦，再对单业务内的大表做水平分表。

###### 3. 垂直分表和水平分表的区别？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
**垂直分表**：把一张宽表按列拆分成多张表，通常按字段访问频率把常用字段放一张表（主表），把大字段、不常用字段放另一张表（扩展表）。两表通过相同的主键关联。

比如 `user` 表拆成 `user_basic`（id、username、phone，高频查询）和 `user_profile`（user_id、bio、avatar，低频访问）。好处是主表变小，单数据页存更多行，读写性能提升，也方便冷热数据分别缓存。

**水平分表**：把一张大表的数据行拆到多张结构相同的小表里，比如 `orders` 按时间拆成 `orders_2024`、`orders_2025`。解决单表行数过多的问题。查同一年的订单只扫一张表，效率高；但跨年查询就要查多张表再合并。

###### 4. 分库分表的策略有哪些？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
水平拆分的核心是选择分片策略，主要有三种：

**范围分片**：按分片键的连续区间划分，比如 id 在 0~100万 的数据放分片1，100万~200万 放分片2；或者按时间分月分年。优点是扩容方便，只需加新分片；缺点是容易出热点，最新数据都集中在最后一个分片，写压力不均衡。

**哈希取模分片**：对分片键的值做哈希再取模，比如 `order_id % 4` 决定放哪个分片。优点是数据分布均匀，不容易出热点；缺点是扩容很痛苦——分片数量一变，大量数据要重新迁移。

**一致性哈希分片**：把节点和数据都映射到一个哈希环上，数据顺时针找到第一个节点存储。增减节点时，只影响相邻节点的一小部分数据，迁移量大幅减少。适合对扩展性要求高的场景，但需要引入虚拟节点来保证数据分布均匀，实现相对复杂。

###### 5. 什么是分片键？如何选择分片键？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
分片键是决定一行数据分配到哪个分片的字段，是分库分表最关键的设计决策。

选择原则：**数据分布要均匀**，分片键的值域要足够分散，避免数据倾斜导致某个分片成为热点；**要覆盖高频查询条件**，这样查询可以直接路由到单个分片，不用扫全部分片；**尽量减少跨分片操作**，跨分片的 JOIN、排序、分页都非常复杂。

一个常见的陷阱：选了时间作为分片键，所有最新数据都往最后一个分片写，热点问题严重。更好的做法是选业务 ID（如 user_id、order_id），取模分片，分布均匀。

对于既要按买家查又要按卖家查的场景（比如"我的订单"和"我的卖出"），可以对订单数据按 buyer_id 和 seller_id 各做一份冗余存储，以空间换查询便利。

###### 6. 分库分表后如何进行分页查询？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
分页查询是分库分表最棘手的问题，根据场景有不同方案：

**全局汇总**：从所有分片各取数据，在内存里合并排序再分页。结果准确但性能极差，offset 越大、分片越多就越慢，基本不可用于生产高并发场景。

**游标分页（推荐常规场景）**：不用 LIMIT offset，改为记录上一页最后一条记录的分片键和排序键，下次从这个位置往后取。只支持"上一页/下一页"，不能跳页，但性能非常好。

**业务折衷**：允许分页结果有轻微误差，或限制只能查最近 N 页，超出范围的数据导出下载。

**异构索引（大厂方案）**：把数据同步到 Elasticsearch 或 TiDB 里，用这些专门的检索系统来做分页查询，MySQL 负责事务写，ES 负责复杂查询，是处理分库分表查询问题的最优解。

###### 7. 分库分表后如何处理跨库 JOIN？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
跨库 JOIN 要尽量避免，几个常用策略：

**全局广播表**：对数据量小、变化少、各分片都要用到的字典表（省市区、商品类目等），每个分片都冗余一份，关联时直接在本分片内 JOIN。

**字段冗余**：把常用的关联字段冗余到主表。比如订单表里冗余商品名称，查订单时不用再关联商品表（商品名称基本不变，冗余成本低）。

**应用层 JOIN**：先查主表数据，再根据关联 ID 批量查关联表，最后在应用内存里拼接。会多几次查询，但好过跨库 JOIN。

**表分组（co-location）**：关联紧密的表（订单表和订单明细表）使用相同的分片键，保证同一个订单的数据总落在同一个分片里，可以直接在分片内 JOIN。

###### 8. 分库分表后如何生成全局唯一 ID？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
分库后数据库自增 ID 会重复，必须用分布式 ID 方案。

**UUID**：全局唯一，但完全随机，插入 B+树导致大量页分裂，性能差；字符串类型，存储空间大，索引效率低。不推荐作为主键。

**数据库号段表**：一张独立的 ID 生成表，应用批量取号段（比如一次取 1000 个 ID），自己递增使用。ID 有序，实现简单；缺点是这张表是单点，虽然可以用双主来规避，但仍有复杂度。

**雪花算法（推荐）**：64 位长整型，高位是时间戳（毫秒），中间是机器 ID，低位是序列号。趋势递增（对 B+树友好）、全局唯一、本地生成（无网络依赖）、每毫秒可生成 4096 个 ID，性能极高。唯一的问题是时钟回拨会导致重复，需要在实现时处理（回拨时等待或抛异常）。这是生产中最常用的方案。

###### 9. 常见的分库分表中间件有哪些？

[[../../../21_MySqlKnowledge/05_架构与运维/01、分库分表|📖]]
主流分两类：

**Client 模式（Jar 包嵌入应用）**：代表是 **ShardingSphere-JDBC**，在应用层拦截 JDBC 调用，自动路由分片，性能好（没有额外网络层），与应用强耦合。适合 Java 项目。

**Proxy 模式（独立代理服务）**：代表是 **ShardingSphere-Proxy** 和 **MyCat**，应用连接代理像连接普通 MySQL 一样，代理解析 SQL 路由。对应用完全透明，语言无关；缺点是多了一层网络跳转，需要单独运维代理服务。

选型建议：Java 技术栈首选 ShardingSphere-JDBC，轻量无侵入，社区活跃，文档完善；需要对应用完全透明或多语言接入的场景用 ShardingSphere-Proxy；MyCat 功能成熟但近年更新放缓，新项目不太推荐。
