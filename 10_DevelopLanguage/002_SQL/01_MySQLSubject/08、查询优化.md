###### 1. 如何分析 SQL 查询性能？

[[../../../21_MySqlKnowledge/02_索引与查询优化/03、查询优化器与执行计划|📖]]
分析 SQL 性能是一个系统性过程，分几个层次来做：

第一步，**发现慢 SQL**。开启慢查询日志（`slow_query_log=ON`，`long_query_time` 设为 1~2 秒），MySQL 会自动记录执行超时的 SQL。用 `pt-query-digest` 分析日志，按总时间排序，最耗时的 SQL 优先处理。

第二步，**分析执行计划**。用 `EXPLAIN` 查看 MySQL 打算怎么执行这条 SQL——走哪个索引、扫描多少行、有没有全表扫描、有没有 filesort 或 temporary table，这些都是性能瓶颈的信号。

第三步，**验证优化效果**。MySQL 8.0 有 `EXPLAIN ANALYZE`，会实际执行 SQL 并输出预估行数和实际行数的对比，直接看到优化前后的差距。

第四步，**监控系统资源**。一条 SQL 如果导致 CPU 或 I/O 飙升，结合 `SHOW ENGINE INNODB STATUS` 和 Performance Schema 查看具体等待事件，找到真正的瓶颈所在。

###### 2. EXPLAIN 命令的作用是什么？如何使用？

[[../../../21_MySqlKnowledge/02_索引与查询优化/03、查询优化器与执行计划|📖]]
`EXPLAIN` 是性能分析最重要的工具，它展示 MySQL 优化器为某条 SQL 生成的执行计划，不真正执行 SQL，告诉你 MySQL 打算怎么跑。

```sql
EXPLAIN SELECT * FROM users WHERE age > 25 AND name = '张三';
```

高级用法：`EXPLAIN FORMAT=JSON` 输出更详细的 JSON 格式，包含成本估算数据；`EXPLAIN ANALYZE`（MySQL 8.0.18+）真正执行 SQL 并对比预估和实际执行情况，排查计划偏差很有用。

###### 3. EXPLAIN 输出结果中各字段的含义？

[[../../../21_MySqlKnowledge/02_索引与查询优化/03、查询优化器与执行计划|📖]]
EXPLAIN 输出的关键字段，从最重要的开始说：

**type（访问类型）**：判断查询效率的核心指标，从好到坏依次是：`system`（只有一行）> `const`（主键/唯一索引等值查找，极快）> `eq_ref`（关联查询中唯一索引匹配）> `ref`（非唯一索引匹配）> `range`（索引范围扫描）> `index`（全索引树扫描）> `ALL`（全表扫描，需要优化）。出现 `ALL` 或 `index` 就要警惕了。

**key**：实际使用的索引，`NULL` 表示没走索引。

**rows**：MySQL 预估要扫描的行数，越小越好。如果 rows 很大但最终返回行数很少，说明有大量过滤，要考虑优化索引。

**Extra（额外信息）**：`Using index`（覆盖索引，不用回表，很好）；`Using where`（在 Server 层还有过滤，正常）；`Using filesort`（需要额外排序操作，尽量消除）；`Using temporary`（用了临时表，GROUP BY 或 ORDER BY 时常见，需要优化）；`Using join buffer`（JOIN 时没走索引，用了 join buffer，性能差）。

**key_len**：使用的索引长度，可以判断联合索引用了几列。

**id 和 select_type**：id 相同的按顺序执行；id 不同则 id 越大越先执行。select_type 是 SIMPLE 表示简单查询，SUBQUERY 表示子查询，DERIVED 表示派生表（FROM 里的子查询）。

###### 4. 什么是查询优化器？

[[../../../21_MySqlKnowledge/02_索引与查询优化/03、查询优化器与执行计划|📖]]
查询优化器是 MySQL 的核心组件，负责把一条 SQL 翻译成它认为代价最低的执行计划。MySQL 用的是基于成本的优化器（CBO）。

工作流程大致是：解析 SQL 之后，优化器先做逻辑优化（条件化简、常量替换、子查询转换等），再做物理优化——枚举可能的执行路径（走哪个索引、表连接顺序怎么安排），向存储引擎询问每种方案的 I/O 和 CPU 代价，选代价最低的方案执行。

优化器的决策依赖统计信息，如果统计信息过期，可能选出次优计划。用 `ANALYZE TABLE` 可以刷新统计信息。有时候你会发现优化器没有选择你认为最优的索引，这时可以用 `FORCE INDEX` 或 `USE INDEX` 提示优化器，但一般先用 EXPLAIN 确认是不是统计信息的问题。

###### 5. 常见的 SQL 优化技巧有哪些？

[[../../../21_MySqlKnowledge/02_索引与查询优化/04、SQL优化实战|📖]]
几个最实用的方向：

**索引相关**：为 WHERE、JOIN ON、ORDER BY、GROUP BY 的列建索引；用覆盖索引减少回表；避免索引失效（函数操作、类型不匹配、前缀模糊）。

**SQL 写法**：避免 `SELECT *`，只取需要的列，减少网络传输和内存消耗；`UNION ALL` 比 `UNION` 快（少一次去重）；`EXISTS` 通常比 `IN` 更高效，尤其是子查询结果集很大时；避免在 WHERE 里对字段做函数或运算。

**大数据量分页**：`LIMIT 100000, 20` 这种大偏移量会扫描大量数据，改成延迟关联：先用覆盖索引取出主键，再关联拿完整数据。

```sql
-- 原始写法，慢
SELECT * FROM orders ORDER BY id LIMIT 100000, 20;
-- 优化后
SELECT o.* FROM orders o 
JOIN (SELECT id FROM orders ORDER BY id LIMIT 100000, 20) t ON o.id = t.id;
```

**COUNT 优化**：`COUNT(*)` 和 `COUNT(1)` 性能基本一样；`COUNT(字段)` 会忽略 NULL，语义不同，而且如果字段没有索引会更慢。

###### 6. 什么是慢查询？如何定位和优化慢查询？

[[../../../21_MySqlKnowledge/02_索引与查询优化/04、SQL优化实战|📖]]
执行时间超过 `long_query_time` 阈值的 SQL 就是慢查询。

定位步骤：开启慢查询日志 → 用 `pt-query-digest` 分析（按 Query_time 倒序排列） → 拿到最慢的 SQL，用 `EXPLAIN` 分析执行计划 → 找到问题点（全表扫描、索引失效、filesort 等）。

优化思路：大多数情况是索引问题，按照 EXPLAIN 的结果补索引或调整 SQL 写法；排除索引问题后，考虑业务逻辑优化（比如能否缓存、能否拆分查询）；最后才考虑架构层面（读写分离、分库分表）。

###### 7. 如何避免全表扫描？

[[../../../21_MySqlKnowledge/02_索引与查询优化/04、SQL优化实战|📖]]
EXPLAIN 里 type 是 ALL 就是全表扫描，影响性能的第一大敌。

对策：WHERE 和 JOIN ON 的列确保有索引，且写法不要导致索引失效；如果优化器选错了索引，用 `FORCE INDEX` 强制指定；跑 `ANALYZE TABLE` 更新统计信息，帮助优化器做出正确判断。

不过也不用过于紧张：很小的表（几百行）全表扫描比走索引还快，这是正常的，不需要强行建索引。

###### 8. 什么是子查询？如何优化子查询？

[[../../../21_MySqlKnowledge/02_索引与查询优化/04、SQL优化实战|📖]]
子查询就是嵌套在外层查询里的查询，便于表达复杂逻辑，但性能有时候不如 JOIN。

优化策略：最常见的是**把子查询改写成 JOIN**，大多数情况下更快，因为优化器对 JOIN 的优化更成熟；用 `EXISTS` 替代 `IN`，尤其是当子查询结果集很大时；避免在 FROM 子句里写复杂子查询（派生表），尽量提前物化或用临时表。

MySQL 5.6 之后优化器会尝试自动把某些子查询转化为半连接（semi-join）来优化，但不是所有情况都能转，复杂子查询还是要手动优化。

###### 9. JOIN 的类型有哪些？如何优化 JOIN 查询？

[[../../../21_MySqlKnowledge/02_索引与查询优化/04、SQL优化实战|📖]]
JOIN 类型：`INNER JOIN`（只返回两表都有的匹配行）、`LEFT/RIGHT OUTER JOIN`（保留一侧全部行）、`CROSS JOIN`（笛卡尔积，谨慎使用）。

优化策略：**JOIN ON 条件的列必须有索引**，这是最关键的；**小表驱动大表**，优化器一般会自动选，但可以用 `STRAIGHT_JOIN` 强制顺序；减少 JOIN 的表数量，超过 5 个表的 JOIN 要考虑拆分逻辑。

MySQL 的 JOIN 算法：最常用的是 NLJ（Nested-Loop Join，驱动表的每一行去匹配被驱动表，被驱动表有索引时效率很好）；如果被驱动表没索引会退化为 BNL（Block NLJ，把驱动表数据放到 join buffer 里批量匹配，EXPLAIN 里显示 Using join buffer）；MySQL 8.0.18+ 支持 Hash Join，对大表等值连接更友好。

###### 10. 什么是索引合并（Index Merge）？

[[../../../21_MySqlKnowledge/02_索引与查询优化/02、索引类型与使用策略|📖]]
索引合并是 MySQL 优化器的一种策略：同一张表的一个查询，同时使用多个索引，把各自的结果集取交集（AND）或并集（OR），再返回结果。比如 `WHERE a = 1 OR b = 2`，如果 a 和 b 分别有索引，优化器可能各走一次索引再合并结果。

EXPLAIN 里会显示 `Using sort_union(...)` 或 `Using union(...)` 或 `Using intersect(...)`。

不过索引合并实际上并不总是性能更好，有时候合并的开销反而比全表扫描还大。如果常见查询需要用到索引合并，更好的方式是建一个覆盖这些列的联合索引。

###### 11. LIMIT 分页如何优化？

[[../../../21_MySqlKnowledge/02_索引与查询优化/04、SQL优化实战|📖]]
`LIMIT offset, size` 在 offset 很大时非常慢，因为 MySQL 要先扫描 offset 行再丢掉，只返回后面的 size 行。

**延迟关联**是最常用的优化：先用覆盖索引拿到主键列表，再关联查完整数据：

```sql
SELECT * FROM orders o
JOIN (SELECT id FROM orders WHERE status = 1 ORDER BY id LIMIT 100000, 20) t 
ON o.id = t.id;
```

**游标分页**（更推荐）：记录上一页最后一条记录的 ID，下次查询用 `WHERE id > last_id LIMIT 20`，这样直接定位，不需要扫描前面的数据。适合按 ID 顺序翻页的场景。

###### 12. COUNT(*) 和 COUNT(1) 和 COUNT(字段) 的区别？

[[../../../21_MySqlKnowledge/02_索引与查询优化/04、SQL优化实战|📖]]
**COUNT(*) 和 COUNT(1)** 性能基本一样，都是统计总行数（包含 NULL），现代 MySQL 对两者的处理方式相同，不用纠结哪个更快。

**COUNT(字段)** 只统计这个字段不为 NULL 的行数，语义不同。如果字段上没有索引，还会导致查询更慢。

如果需要快速统计总行数且不带 WHERE 条件，MyISAM 存储的是行数可以直接返回，InnoDB 则需要扫描。对于 InnoDB 的大表总行数统计，可以考虑用单独的计数表维护，或者用近似值 `SELECT TABLE_ROWS FROM information_schema.TABLES WHERE TABLE_NAME = 'your_table'`（不精确但很快）。

###### 13. 如何优化大数据量的查询？

[[../../../21_MySqlKnowledge/02_索引与查询优化/04、SQL优化实战|📖]]
针对大数据量场景，从多个层次来考虑：

**索引优化**：建好合适的索引是基础，大表没走索引就是灾难。

**分批处理**：避免一次性处理大量数据，拆分成小批次。比如批量更新/删除按主键范围分批：

```sql
-- 每次处理 1000 行，循环直到处理完
DELETE FROM logs WHERE create_time < '2023-01-01' LIMIT 1000;
```

**分区表**：对按时间或范围查询的超大表，用 RANGE PARTITION 按年月分区，查询时可以裁剪掉大量不相关的分区。

**读写分离**：把大查询、报表查询路由到从库，不影响主库的写性能。

**冷热分离**：历史冷数据归档到冷存储或独立表，主表只保留近期热数据，大幅缩减扫描范围。

**缓存**：对读多写少且允许有一定延迟的数据，用 Redis 缓存查询结果，减少数据库压力。
