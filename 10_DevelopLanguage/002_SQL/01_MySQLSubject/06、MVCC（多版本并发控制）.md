###### 1. 什么是 MVCC？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
MVCC（Multi-Version Concurrency Control，多版本并发控制）是 InnoDB 实现高并发的核心机制。它的思路是：通过维护数据行的多个历史版本，让读操作和写操作可以并发执行，互不阻塞——写的时候不影响读，读的时候也不挡住写。

从 Java 开发者的视角，你可以把 MVCC 理解为一种乐观锁思想。普通 `SELECT`（快照读）无需加锁，读到的是某个历史时刻的数据快照，不受其他事务正在进行的写操作影响。这解决了读写冲突，写写冲突则依然要靠锁来解决。

MVCC 只在 RC 和 RR 两个隔离级别下工作。RC 下每次查询都生成新快照，可以读到最新提交的数据；RR 下整个事务用同一个快照，保证可重复读。

###### 2. MVCC 的实现原理是什么？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
MVCC 依赖三个核心组件协同工作：**隐藏字段**、**Undo Log 版本链**和 **ReadView**。

**隐藏字段**：InnoDB 在每行数据末尾自动添加两个关键隐藏字段——`DB_TRX_ID`（记录最后修改这行数据的事务ID）和 `DB_ROLL_PTR`（指向 Undo Log 中该行上一个版本的指针）。如果表没有显式主键，还会有一个 `DB_ROW_ID`。

**Undo Log 版本链**：每次对一行数据做修改，InnoDB 不会直接覆盖旧版本，而是把旧版本存到 Undo Log 里，通过 `DB_ROLL_PTR` 把所有历史版本串成一个链表，最新版本在链头，最早版本在链尾。

**ReadView**：事务做快照读时生成一个 ReadView，里面记录了当前系统中所有活跃事务的 ID 列表。读数据时，从版本链头开始遍历，用可见性算法判断哪个版本对当前事务可见，找到第一个可见版本就返回。

###### 3. 什么是隐藏字段？包括哪些？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
InnoDB 在每行数据末尾自动添加三个隐藏字段，对用户不可见，但是 MVCC 的基础：

**DB_TRX_ID**（6字节）：最近修改事务ID，记录最后一次插入或更新这行数据的事务ID。DELETE 在 InnoDB 内部也是特殊的 UPDATE（打删除标记），所以也会更新这个字段。

**DB_ROLL_PTR**（7字节）：回滚指针，指向这行数据上一个历史版本在 Undo Log 中的位置，是构成版本链的关键。

**DB_ROW_ID**（6字节）：隐含自增行ID，仅当表没有显式主键时，InnoDB 自动生成这个字段作为聚簇索引的键。有主键的表一般不存在这个字段。

###### 4. 什么是 Undo Log？它的作用是什么？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
Undo Log 是逻辑日志，记录数据修改之前的状态，有两个核心用途：

**事务回滚（原子性）**：ROLLBACK 时根据 Undo Log 记录的反向操作还原数据——INSERT 对应记录主键（回滚时删它），DELETE 对应保存完整行（回滚时重新插），UPDATE 对应保存修改前的字段值（回滚时恢复旧值）。

**实现 MVCC**：Undo Log 里存着数据行的历史版本，当其他事务做快照读时，沿着 `DB_ROLL_PTR` 指针链找到对自己可见的历史版本，不需要加锁，读写就不会互相阻塞了。

Undo Log 分两类：Insert Undo Log 事务提交后可以立即清理（回滚期过了就没用了）；Update Undo Log 不能立即清理，因为还有其他事务的快照读可能要用它，得等到所有活跃事务都不再需要它时，Purge 线程才会清理。这也是为什么长事务会导致 Undo Log 堆积、表空间膨胀的原因。

###### 5. 什么是 ReadView（读视图）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
ReadView 是事务做快照读时生成的一个数据结构，用于判断版本链上每个版本对当前事务是否可见，四个关键字段：

**m_ids**：生成 ReadView 时所有活跃（已开始但未提交）事务的 ID 集合。

**min_trx_id**：m_ids 里最小的事务ID，比这个小的事务都已提交了。

**max_trx_id**：生成 ReadView 时系统即将分配的下一个事务 ID（当前最大值+1），大于等于这个值的事务都是在 ReadView 生成之后才开启的。

**creator_trx_id**：创建这个 ReadView 的事务自身的 ID。

###### 6. ReadView 在不同隔离级别下的生成时机？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
RC 和 RR 的核心区别就在这里：

**RC（读已提交）**：每次执行快照读都生成一个全新的 ReadView。所以它每次都能看到最新已提交的数据，但同一个事务里两次相同的查询可能结果不同——这就是不可重复读。

**RR（可重复读）**：只在事务里第一次做快照读时生成 ReadView，之后所有快照读都复用这同一个。整个事务里看到的都是同一个历史快照，保证了可重复读，也在一定程度上避免了幻读。

READ UNCOMMITTED 不用 ReadView（直接读最新版本，脏读由此而来），SERIALIZABLE 直接加锁串行执行，也不走 MVCC。

###### 7. MVCC 如何判断数据的可见性？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
从版本链头开始，对每个版本的 `DB_TRX_ID`（记为 T）用 ReadView 做如下判断：

**1. T == creator_trx_id**：这个版本是当前事务自己改的，**可见**。

**2. T < min_trx_id**：生成这个版本的事务在 ReadView 创建之前就已经提交了，**可见**。

**3. T >= max_trx_id**：生成这个版本的事务在 ReadView 创建之后才开启的，**不可见**。

**4. min_trx_id <= T < max_trx_id**：需要进一步看 T 是不是在 m_ids（活跃事务列表）里。在里面说明这个事务创建 ReadView 时还没提交，**不可见**；不在里面说明已经提交了，**可见**。

如果当前版本不可见，就顺着 `DB_ROLL_PTR` 去看上一个版本，重复判断，直到找到可见版本或版本链遍历完。

###### 8. 版本链是如何形成的？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
版本链是通过 `DB_ROLL_PTR` 把这行数据的所有历史版本串起来的单向链表，链头是最新版本，链尾是最早版本。

形成过程：事务 A 插入一行，生成版本 V1，`DB_ROLL_PTR` 为空。事务 B 来更新这行时，先把当前版本 V1 完整拷贝到 Undo Log 里作为旧版本，然后在原数据页上修改，生成新版本 V2，`DB_ROLL_PTR` 指向 Undo Log 里的 V1。事务 C 再更新，同理生成 V3 指向 V2。最终链条是 V3 → V2 → V1。

###### 9. MVCC 解决了哪些并发问题？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
MVCC 主要解决的是读写冲突，让读写不互相阻塞，写写冲突仍然需要锁。

**脏读**：MVCC 快照读只能看到 ReadView 创建前已提交的事务，看不到未提交的数据，完全解决脏读。

**不可重复读**：RR 级别下复用同一个 ReadView，每次读到的都是同一个快照，解决了不可重复读；RC 级别下每次重新生成 ReadView，不可重复读仍然存在。

**幻读**：这个要分情况讨论。对于**快照读**，RR 级别下复用 ReadView，看不到新插入的行，从快照读角度没有幻读；对于**当前读**（SELECT FOR UPDATE、UPDATE 等），MVCC 不管用，要靠 Next-Key Lock 封住间隙防止插入，才能防止幻读。所以 MVCC 解决了快照读的幻读，但不能单独解决当前读的幻读。

###### 10. MVCC 的优缺点是什么？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
**优点**：读写不互相阻塞，高并发性能好；减少了锁的使用，死锁概率降低；为快照读提供一致的数据视图，方便做报表和分析。

**缺点**：维护版本链需要额外的存储空间（Undo Log）和 CPU 开销；需要 Purge 线程定期清理旧版本，如果有长事务一直活着，旧版本无法清理，Undo 表空间会不断膨胀，这是生产中要特别注意的问题；当前读的幻读问题 MVCC 自己解决不了，需要锁的配合。
