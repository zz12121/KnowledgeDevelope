###### 1. MySQL 有哪些备份方式？

[[../../../21_MySqlKnowledge/05_架构与运维/04、备份恢复与安全|📖]]
MySQL 备份从三个维度来理解：

**按备份内容**分逻辑备份和物理备份。逻辑备份输出 SQL 文件（`mysqldump`），可读性好，跨平台；物理备份直接拷贝数据文件（`XtraBackup`），速度快，适合大数据量。

**按服务状态**分热备份（不停服备份，业务无感知）、温备份（短暂锁表只读）和冷备份（停服后拷文件，最简单但停机时间长）。生产上绝大多数用热备份。

**按备份量**分全量备份（完整数据快照，定期做）、增量备份（只备份上次以来的变化，频繁做，恢复时要全量 + 所有增量逐个应用）和差异备份（自上次全量以来的所有变化，恢复时只需全量 + 最新差异，平衡恢复复杂度和备份大小）。

###### 2. 逻辑备份和物理备份的区别？

[[../../../21_MySqlKnowledge/05_架构与运维/04、备份恢复与安全|📖]]
**逻辑备份（mysqldump 等）**：输出 SQL 语句（CREATE TABLE、INSERT 等）。灵活，支持单表、单库、带条件的部分数据备份；可移植性高，跨版本、跨平台都能用；但备份和恢复都慢（要逐条执行 SQL 重建索引），大数据量时很慢；输出是文本文件，体积比物理文件大。

**物理备份（XtraBackup 等）**：直接拷贝 `.ibd` 等数据文件。速度快，对服务影响小（XtraBackup 可以热备，锁定时间极短）；恢复也快，类似文件复制；但可移植性差，依赖 MySQL 版本和文件格式，不能跨大版本恢复；粒度粗，通常是实例级别。

选择建议：百 GB 以下、需要灵活性的场景，逻辑备份够用；TB 级别、对备份窗口和恢复时间有严格要求的核心库，必须用物理备份。

###### 3. mysqldump 的作用和使用方法？

[[../../../21_MySqlKnowledge/05_架构与运维/04、备份恢复与安全|📖]]
mysqldump 是 MySQL 官方自带的逻辑备份工具，把数据库的结构和数据导出成 SQL 文本文件。

```bash
# 备份单个数据库
mysqldump -u root -p mydb > mydb_backup.sql

# 备份所有数据库
mysqldump -u root -p --all-databases > full_backup.sql

# 只备份表结构不备份数据
mysqldump -u root -p mydb --no-data > schema.sql

# 恢复
mysql -u root -p mydb < mydb_backup.sql
```

几个重要参数：

`--single-transaction`：对 InnoDB 表必加，开启事务后做一致性快照备份，备份过程中不锁表，实现热备。

`--master-data=2`：在备份文件里以注释形式记录当前 Binlog 位置，做 PITR（时间点恢复）时知道从哪里开始应用 Binlog。

`--routines`、`--triggers`、`--events`：分别备份存储过程/函数、触发器、事件，默认不备份，生产上要加上。

`--quick`：大表逐行读取，避免把整张表读到内存里撑爆内存。

###### 4. 如何实现热备份？

[[../../../21_MySqlKnowledge/05_架构与运维/04、备份恢复与安全|📖]]
热备份有两条路：

**Percona XtraBackup（物理热备，推荐）**：边拷贝数据文件边持续收集 Redo Log，备份完成后通过 prepare 阶段把备份期间的 Redo Log 应用到数据文件上，使备份达到一致状态。全程不锁表（极短时间的共享锁用于获取 Binlog 位置），对业务几乎无影响。

```bash
# 备份
xtrabackup --backup --user=root --password=xxx --target-dir=/backup/2024/

# 准备（恢复前必须执行，应用 redo log 使备份一致）
xtrabackup --prepare --target-dir=/backup/2024/

# 恢复（先停 MySQL，清空数据目录）
xtrabackup --copy-back --target-dir=/backup/2024/
chown -R mysql:mysql /var/lib/mysql
```

**mysqldump + --single-transaction（逻辑热备）**：对 InnoDB 表在事务中做快照读，备份期间其他写操作不受影响，适合数据量不太大的场景。

###### 5. 如何进行数据恢复？

[[../../../21_MySqlKnowledge/05_架构与运维/04、备份恢复与安全|📖]]
根据备份类型不同，恢复方式也不同。

**从 mysqldump 逻辑备份恢复**：直接把 SQL 文件导入，速度慢（要重建索引），但操作简单，跨版本也没问题。

**从 XtraBackup 物理备份恢复**：停 MySQL → 清空数据目录 → `--copy-back` 复制文件 → 修正文件权限 → 启动 MySQL。速度快，几百 GB 通常几分钟内恢复。

**时间点恢复（PITR）**：这是最精细的恢复，适合"误删了某张表/某些数据，想恢复到误操作之前的状态"场景。步骤：

1. 先恢复最近的全量备份
2. 用 `mysqlbinlog` 重放全备之后到误操作发生前的 Binlog

```bash
# 按时间范围重放 Binlog
mysqlbinlog --start-datetime="2024-01-01 10:00:00" \
            --stop-datetime="2024-01-01 10:30:00" \
            mysql-bin.000001 | mysql -u root -p

# 按位置点精确恢复
mysqlbinlog --start-position=154 --stop-position=5000 \
            mysql-bin.000001 | mysql -u root -p
```

前提是必须开启了 Binlog，并且日志文件完整没有被清理。

###### 6. 什么是增量备份和全量备份？

[[../../../21_MySqlKnowledge/05_架构与运维/04、备份恢复与安全|📖]]
**全量备份**：某个时间点数据库全部数据的完整快照。是恢复的基础锚点，备份量大，时间长，通常按周或按天做一次。

**增量备份**：只备份上次备份（全量或增量）之后变化的数据。备份量小，速度快，但恢复时要把全量和所有中间的增量备份按顺序逐个应用，链条越长恢复越复杂，任何一个环节损坏都会导致后续无法恢复。

**差异备份**：备份自上次**全量**备份以来所有变化的数据（不管中间做了多少次增量）。恢复时只需全量 + 最新差异，比增量简单；但随着时间推移，差异越来越大，备份时间也越来越长，到了一定程度要重新做一次全量。

实际生产中常见的策略：每周全量备份一次 + 每天增量备份 + Binlog 实时归档，PITR 可以恢复到任意分钟级时间点。

###### 7. 如何使用 Binlog 进行数据恢复？

[[../../../21_MySqlKnowledge/05_架构与运维/04、备份恢复与安全|📖]]
Binlog 是 PITR 的基础，它记录了所有数据变更操作，配合全量备份可以恢复到任意时间点。

首先确保 Binlog 已开启（`log_bin` 参数），并且历史 Binlog 文件完整保留。

恢复步骤：先恢复全量备份 → 找到全备结束时对应的 Binlog 位置（用 XtraBackup 备份会自动记录在 `xtrabackup_binlog_info` 里，用 mysqldump 时加 `--master-data=2` 也会记录）→ 用 `mysqlbinlog` 从那个位置开始，重放到故障前一刻：

```bash
# 先确认要恢复到哪个时间点（比如误删操作前 10:30）
# 然后从全备的 Binlog 位置重放到目标时间
mysqlbinlog --start-position=4  \
            --stop-datetime="2024-01-01 10:30:00" \
            mysql-bin.000002 mysql-bin.000003 | mysql -u root -p
```

**最佳实践**：把 Binlog 实时同步到独立的存储（对象存储或另一台服务器），防止因磁盘故障导致日志和数据一起丢失。同时设置合理的 `expire_logs_days`（MySQL 8.0 用 `binlog_expire_logs_seconds`），保留足够长的历史日志。
