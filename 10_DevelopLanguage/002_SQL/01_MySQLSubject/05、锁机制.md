###### 1. MySQL 有哪些锁类型？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
MySQL 的锁体系比较复杂，可以从三个维度来理解：

**按粒度**分为全局锁、表级锁和行级锁。全局锁让整个数据库只读，一般用于做全库逻辑备份；表级锁粒度大，并发性差；行级锁粒度最细，并发性最好，是 InnoDB 的主战场。

**按属性**分为共享锁（S锁/读锁）和排他锁（X锁/写锁）。

**按思想**分为悲观锁和乐观锁，这两个更多是编程模式层面的概念。

InnoDB 的行级锁还进一步细分为：记录锁（Record Lock）、间隙锁（Gap Lock）、临键锁（Next-Key Lock）和插入意向锁（Insert Intention Lock），各有各的使用场景。

###### 2. 什么是共享锁（S锁）和排他锁（X锁）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
**共享锁（S锁/读锁）**：允许多个事务同时持有同一资源的共享锁，读读共享，互不影响。但持有 S 锁的事务不能修改数据，也会阻止其他事务加 X 锁。可以通过 `SELECT ... LOCK IN SHARE MODE` 或 MySQL 8.0 的 `FOR SHARE` 显式加共享锁。

**排他锁（X锁/写锁）**：写独占，同一时刻只能有一个事务持有某行的 X 锁，持有期间其他事务既不能加 S 锁也不能加 X 锁，必须等待。`INSERT`、`UPDATE`、`DELETE` 会自动加 X 锁，也可以通过 `SELECT ... FOR UPDATE` 显式加。

###### 3. 什么是表锁、行锁、页锁？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
**表级锁**：锁定整张表，加锁快，开销小，不会发生死锁，但并发度最低——一个写操作把整张表锁住，其他所有读写都得等。MyISAM 用表级锁，这也是它并发性差的根本原因。

**行级锁**：锁定单行记录，粒度最细，并发度最高，其他事务可以同时操作不同的行。代价是加锁开销大，管理复杂，会发生死锁。InnoDB 支持行级锁，这是它并发性能好的核心原因。

**页级锁**：介于两者之间，锁定相邻的一组记录（一页），BDB 存储引擎使用，现在很少见了。

###### 4. 什么是意向锁（Intention Lock）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
意向锁是 InnoDB 自动维护的表级锁，是为了让表级锁和行级锁共存而设计的协调机制。

它解决了一个问题：当有事务想锁整张表时，如何快速判断表里是否有行已经被锁了？如果没有意向锁，就要逐行检查，效率极低。有了意向锁，事务在加行锁之前，必须先在表上加对应的意向锁，这样检查表锁是否冲突时，只需要看表上有没有意向锁就够了。

分两种：**意向共享锁（IS）**，表示该事务准备在某些行上加 S 锁；**意向排他锁（IX）**，表示该事务准备在某些行上加 X 锁。意向锁之间互相兼容，但 IX 和表级 S 锁不兼容，IS/IX 和表级 X 锁不兼容。整个过程对用户透明，由 InnoDB 自动管理。

###### 5. 什么是记录锁（Record Lock）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
记录锁是最基本的行级锁，锁定的是**单个索引记录**。比如 `SELECT * FROM users WHERE id = 1 FOR UPDATE`，就在 id=1 这条索引记录上加了记录锁，其他事务不能修改或删除这条记录，直到当前事务结束。

注意：行锁锁的是索引记录，不是物理行。如果查询条件没有用到索引，InnoDB 可能升级为锁整张表。

###### 6. 什么是间隙锁（Gap Lock）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
间隙锁锁定的是**索引记录之间的区间**，但不包括区间端点的记录本身。它的目的是防止其他事务在这个区间内插入新记录，从而解决幻读问题。

举个例子：如果表里有 id 为 1、5、10 的记录，`SELECT * FROM users WHERE id BETWEEN 5 AND 10 FOR UPDATE` 可能锁住 (5, 10) 这个开区间，阻止其他事务插入 id 为 6、7、8、9 的记录。

间隙锁在**读已提交（RC）隔离级别下会自动失效**，只在 RR 级别下生效，这也是为什么 RC 下死锁概率更低——锁的范围更小。

###### 7. 什么是临键锁（Next-Key Lock）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
临键锁 = 记录锁 + 间隙锁，锁的是一个**左开右闭区间**。这是 InnoDB 在**可重复读（RR）级别下的默认锁算法**，既锁住了已有的记录，又封堵了间隙，从而防止幻读。

举个例子：表里有 id 为 1、4、7、10 的记录，临键锁的区间就是 (-∞, 1]、(1, 4]、(4, 7]、(7, 10]、(10, +∞)。如果某个查询命中了 id=7，临键锁就会锁住 (4, 7] 这个区间，既锁住了 id=7 这条记录，也锁住了 id 在 4~7 之间的间隙。

###### 8. 什么是插入意向锁（Insert Intention Lock）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
插入意向锁是一种特殊的间隙锁，专门在 INSERT 操作时使用。当多个事务要在同一个间隙里插入数据时，如果它们插入的位置不冲突（比如一个插 id=3，另一个插 id=6，都在同一个间隙内但位置不同），这些事务可以同时持有插入意向锁，互不等待，从而提高并发插入性能。

但如果间隙上有间隙锁（Gap Lock），插入意向锁会被阻塞，这就是为什么并发插入会导致锁等待的原因。

###### 9. 什么是自增锁（AUTO-INC Lock）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
自增锁是针对 AUTO_INCREMENT 字段的特殊表级锁，保证自增值的生成是连续的。当一个事务插入数据时，需要先拿到自增锁获取下一个 ID，其他事务的插入操作必须等待。

MySQL 5.1.22 之后引入了更轻量的轻量自增锁模式（`innodb_autoinc_lock_mode`），批量插入时只锁自增值的分配过程，不锁整个插入操作，提升了并发性。不过这可能导致主从复制时，binlog 用 statement 模式会有主键不一致的问题，生产上 binlog 一般用 row 模式来规避。

###### 10. 行锁是如何实现的？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
InnoDB 的行锁是通过锁定索引记录来实现的，而不是直接锁定物理数据行。

这意味着：如果 SQL 没有走索引（全表扫描），InnoDB 无法精确定位到具体的索引记录，就会退化为锁定所有扫描到的记录，极端情况下相当于锁了整张表。这就是为什么一定要确保 WHERE 条件用到了索引，否则本以为是行锁，实际上变成了表锁，并发性大打折扣。

###### 11. 什么情况下行锁会升级为表锁？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
主要有以下几种情况：

**查询没用到索引**：InnoDB 扫描全表，会对所有扫描到的记录加锁，等效于表锁。

**查询条件区分度太低**：优化器判断全表扫描比走索引更快，放弃了索引，同样导致锁范围扩大。

**显式的表级锁操作**：比如 `LOCK TABLES users WRITE` 直接加表锁。

**DDL 操作**：ALTER TABLE 这类操作会持有 MDL（元数据锁）的写锁，阻塞所有对该表的读写。

###### 12. 乐观锁和悲观锁的区别？MySQL 如何实现？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
这是两种并发控制思想，不是 MySQL 特有的概念，但在业务开发里都会用到。

**悲观锁**：悲观地认为并发冲突会经常发生，所以在操作数据之前就先加锁。在 MySQL 里用 `SELECT ... FOR UPDATE` 实现，拿到排他锁后其他事务就得等。适合写操作多、冲突频繁的场景，但并发性差，容易死锁，事务要保持尽量短。

**乐观锁**：乐观地认为冲突很少发生，所以操作时不加锁，提交时再检查数据有没有被改过。通常在表里加一个 `version` 字段，更新时带上旧版本号作为条件：

```sql
UPDATE orders 
SET status = 2, version = version + 1 
WHERE id = 1 AND version = 5;
```

如果这条 UPDATE 影响了 0 行，说明 version 已经被别人改了，需要重试。适合读多写少、冲突概率低的场景，并发性好，但需要业务层处理重试逻辑。

###### 13. 什么是死锁？如何避免死锁？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
死锁是指两个或多个事务互相持有对方需要的锁，形成循环等待，谁都无法继续推进。

经典场景：事务 A 锁住了 id=1 的行，想要再锁 id=2；同时事务 B 锁住了 id=2 的行，想要再锁 id=1。两个事务互相等待，永远无法结束，这就是死锁。

InnoDB 会自动检测死锁，选择代价最小的事务（通常是撤销操作少的那个）回滚，让另一个事务继续执行。可以用 `SHOW ENGINE INNODB STATUS` 查看死锁信息。

避免死锁的几个实用方法：**保持事务短小精悍**，尽快提交，减少锁持有时间；**多个程序访问多张表时，约定以相同的顺序加锁**，避免环形等待；**为 WHERE 条件列建合适的索引**，避免全表扫描扩大锁范围；**考虑使用乐观锁**替代悲观锁；**适当降低隔离级别**（从 RR 降到 RC 可以消除间隙锁，大幅减少死锁概率）。

###### 14. MySQL 如何检测和处理死锁？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
InnoDB 内部维护一个等待图（Wait-for Graph），每当一个事务尝试获取某个锁时，如果被阻塞了，就在等待图里记录"我在等谁"。InnoDB 会定期检测等待图里是否有环，如果有环就说明发生了死锁。

检测到死锁后，InnoDB 选择回滚"代价最小"的事务（一般是 Undo Log 最少的那个），并返回错误 `ERROR 1213: Deadlock found when trying to get lock`，被选中回滚的事务需要应用层捕获这个错误并进行重试。

`innodb_deadlock_detect`（默认开启）控制是否启用死锁检测，高并发下死锁检测本身也有性能开销，某些极端场景下会考虑关掉，改用锁等待超时（`innodb_lock_wait_timeout`）来处理。

###### 15. SELECT ... FOR UPDATE 的作用是什么？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
`SELECT ... FOR UPDATE` 对查询结果集加排他锁（X锁），相当于一次当前读。被锁住的行（在 RR 级别下还包括间隙）在当前事务结束前，其他事务既不能对这些行加 S 锁，也不能加 X 锁，不能修改也不能删除。

典型使用场景是"先读后写，要保证读到的数据在写之前不被改掉"，比如扣库存：先查库存是否充足，再扣减，这两步必须作为一个原子操作，中间不能让别人把库存改了。

```sql
START TRANSACTION;
SELECT stock FROM products WHERE id = 1 FOR UPDATE;
-- 检查库存是否充足
UPDATE products SET stock = stock - 1 WHERE id = 1;
COMMIT;
```

MySQL 8.0 还支持 `SKIP LOCKED`（跳过被锁行，适合抢单场景）和 `NOWAIT`（锁不到立即报错，不等待）。

###### 16. SELECT ... LOCK IN SHARE MODE 的作用是什么？

[[../../../21_MySqlKnowledge/03_事务与并发控制/02、锁机制深度解析|📖]]
`SELECT ... LOCK IN SHARE MODE`（MySQL 8.0 也写作 `FOR SHARE`）对查询结果加共享锁（S锁）。允许其他事务同时加 S 锁读取同样的行，但阻止其他事务加 X 锁修改这些行，直到当前事务结束。

适合需要"读取数据并确保读取期间数据不被修改，但允许其他事务同时读"的场景。比如统计某个范围内的数据时，需要保证统计期间不会有数据变化。不过在实际业务里，这个语句用得相对少，更多情况会用 `FOR UPDATE`。
