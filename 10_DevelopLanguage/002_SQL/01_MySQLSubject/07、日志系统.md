###### 1. MySQL 有哪些日志？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
MySQL 的日志体系分两个层面：

**存储引擎层（InnoDB 特有）**：Redo Log（重做日志，保证持久性，崩溃恢复）和 Undo Log（回滚日志，保证原子性，实现 MVCC）。这两个是 InnoDB 事务机制的核心。

**Server 层（所有存储引擎共用）**：Binlog（二进制日志，主从复制和数据恢复）、Error Log（错误日志，故障排查首要入口）、Slow Query Log（慢查询日志，SQL 调优利器）、General Query Log（通用查询日志，记录所有 SQL，生产慎用）。

此外还有主从复制架构里从库用的 Relay Log（中继日志）。

对于 Java 开发者来说，日常最常打交道的是 Slow Query Log（排查慢 SQL）、Binlog（数据变更订阅，Canal 就是基于它）和对 Redo/Undo 的概念理解（理解事务可靠性的基础）。

###### 2. 什么是 Redo Log（重做日志）？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
Redo Log 是 InnoDB 的物理日志，保证事务持久性的关键，核心设计是 **WAL（Write-Ahead Logging，预写式日志）**——先写日志，再写数据页。

工作流程：执行 DML 操作时，先把数据页加载到内存 Buffer Pool 里修改（产生脏页），同时把"在某个页的某个偏移量做了什么修改"记录到 Redo Log Buffer。事务提交时，把 Redo Log Buffer 强制刷盘（fsync）。后台线程再把脏页异步刷到磁盘数据文件。

如果刷脏页之前宕机了，重启时 InnoDB 扫描 Redo Log，把已提交但未落盘的事务重新应用一遍（redo），数据就不会丢了。这就是"Crash-Safe"的来源。

Redo Log 是固定大小循环写的（比如两个各 512MB 的文件），有 write pos（当前写入位置）和 checkpoint（已落盘的位置）。当 write pos 追上 checkpoint 时，必须先推进 checkpoint（即触发脏页刷盘）才能继续写，这也是为什么 Redo Log 配太小会影响性能。

###### 3. 什么是 Undo Log（回滚日志）？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
Undo Log 是逻辑日志，记录数据修改前的状态，有两个职责：

**保证原子性**：事务回滚时，按照 Undo Log 里记录的反向操作还原数据。INSERT 对应删除，DELETE 对应重新插入，UPDATE 对应恢复旧值。

**实现 MVCC**：Undo Log 存着数据行的历史版本，通过 `DB_ROLL_PTR` 串成版本链，供快照读使用，让读操作不需要加锁就能读到一致的历史数据。

Undo Log 分 Insert Undo Log 和 Update Undo Log。前者事务提交后可以立即清理；后者因为还可能被其他事务的快照读用到，必须等到所有活跃事务都不需要它了，Purge 后台线程才会清理。所以长事务会导致 Undo Log 堆积，这是生产中要极力避免长事务的原因之一。

###### 4. 什么是 Binlog（二进制日志）？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
Binlog 是 MySQL Server 层的逻辑日志，记录所有引起或可能引起数据变更的操作（INSERT/UPDATE/DELETE/DDL），不记录 SELECT 和 SHOW。

最核心的两个用途：**主从复制**——主库把 Binlog 发给从库，从库重放这些事件保持数据同步；**数据恢复**——配合全量备份，可以把数据库恢复到任意时间点（PITR）。

Binlog 是追加写的（和 Redo Log 循环写不同），文件达到大小限制会切换新文件，旧文件保留，这让历史数据恢复成为可能。Binlog 也是 Canal、Maxwell 等 CDC 工具订阅数据变更的数据源。

###### 5. Redo Log 和 Binlog 的区别？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
这是面试高频题，有几个维度的区别要说清楚：

**所属层级**：Redo Log 是 InnoDB 存储引擎特有的，Binlog 是 MySQL Server 层的，所有存储引擎都可以使用。

**日志性质**：Redo Log 是物理日志，记录"在哪个数据页的哪个偏移量做了什么修改"；Binlog 是逻辑日志，记录 SQL 语句的原始逻辑或行数据的变化。

**写入方式**：Redo Log 循环写，固定大小，会覆盖旧日志；Binlog 追加写，旧文件永久保留，适合数据恢复。

**功能**：Redo Log 负责崩溃恢复（Crash-Safe），确保事务持久性；Binlog 负责主从复制和数据恢复。

**事务绑定**：Redo Log 和 InnoDB 事务紧密绑定，每个事务的修改都对应 Redo Log；Binlog 在事务提交时整体写入，记录的是最终结果。

###### 6. 什么是两阶段提交（Two-Phase Commit）？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
两阶段提交保证 Redo Log 和 Binlog 的内容严格一致，分 Prepare 和 Commit 两阶段：

**Prepare 阶段**：InnoDB 把事务的 Redo Log 写入磁盘，状态标记为 PREPARE，此时修改已持久化但未最终提交。

**Commit 阶段**：Server 层把 Binlog 写入磁盘并 sync。Binlog 写成功后，InnoDB 把 Redo Log 状态改为 COMMIT，事务才算真正完成。

###### 7. 为什么需要两阶段提交？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
不用两阶段提交，Redo Log 和 Binlog 就可能出现不一致，进而导致主从不一致：

**先写 Redo Log 再写 Binlog，中间崩溃**：主库重启后靠 Redo Log 恢复了数据，但 Binlog 里没有这条记录，从库不会同步，主从分叉。

**先写 Binlog 再写 Redo Log，中间崩溃**：Binlog 里有，从库会执行，但主库因为没有 Redo Log 认为事务没提交，数据回滚了，主从同样分叉。

两阶段提交的解决思路：以 Binlog 是否完整作为事务是否提交的最终判断依据。崩溃恢复时，如果 Redo Log 是 Prepare 状态，就去看 Binlog 里有没有对应的 XID 事件——有就提交（因为 Binlog 代表主库已决定提交），没有就回滚。这样主从永远一致。

###### 8. 什么是 Error Log（错误日志）？

错误日志记录 MySQL 服务器从启动、运行到关闭过程中遇到的错误、警告和关键提示信息，是故障排查的第一入口。服务起不来、表无法加载、发生崩溃——第一时间看这个文件。默认在数据目录下，文件名是 `host_name.err`，可以通过 `log_error` 参数配置路径。

###### 9. 什么是 Slow Query Log（慢查询日志）？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
慢查询日志记录执行时间超过 `long_query_time`（默认 10 秒，生产上通常调到 1~2 秒）的 SQL，是 SQL 性能调优最直接的切入点。

几个重要配置：`slow_query_log = ON` 开启；`long_query_time` 设置阈值；`log_queries_not_using_indexes = ON` 同时记录没有用到索引的查询（哪怕执行很快，没走索引本身就是个隐患）。

分析工具推荐 `pt-query-digest`（Percona 出品），比自带的 `mysqldumpslow` 更强大，能按慢查询次数、总时间等维度排序汇总，快速定位最需要优化的 SQL。

###### 10. 什么是 General Query Log（通用查询日志）？

通用查询日志记录所有客户端连接和执行的 SQL，包括 SELECT、SHOW 这类不修改数据的操作。因为记录量极大，I/O 压力很高，**生产环境不建议长期开启**，只在需要审计某个特定问题时临时打开。和慢查询日志的区别是：慢查询日志只记录"慢"的，通用查询日志记录"所有"的。

###### 11. Binlog 的三种格式是什么？

[[../../../21_MySqlKnowledge/04_日志与高可用/01、日志系统（Redo_Undo_Binlog）|📖]]
Binlog 有三种格式，各有利弊：

**Statement（SBR）**：直接记录原始 SQL 语句，日志文件小，I/O 小。但如果 SQL 里用了 `NOW()`、`RAND()`、`CURRENT_USER()` 这类不确定性函数，在从库重放时结果可能和主库不一样，导致主从不一致。

**Row（RBR）**：记录每一行数据修改前后的完整镜像，精准无歧义，是**MySQL 5.7.7 之后的默认格式**。缺点是批量操作时日志量极大（UPDATE 1000 行就有 1000 条记录），I/O 压力大，日志文件增长快。

**Mixed（MBR）**：混合模式，MySQL 根据 SQL 类型自动判断：对确定性语句用 Statement，对涉及不确定性函数的语句用 Row。兼顾了性能和安全，但仍有少量边角情况存在不确定性风险。

生产上用 Row 格式最安全，Canal 等数据订阅工具也只支持 Row 格式。
