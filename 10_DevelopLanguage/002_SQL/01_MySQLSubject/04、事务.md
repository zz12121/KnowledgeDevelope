###### 1. 什么是事务？

[[../../../21_MySqlKnowledge/03_事务与并发控制/01、事务与ACID|📖]]
事务是数据库中一组操作的逻辑工作单元，这些操作要么全部成功，要么全部失败回滚，不存在"做了一半"的中间状态。

从业务角度理解，事务就是一个不可分割的操作边界。经典例子是银行转账：扣款和加款是两条 SQL，但它们必须作为一个整体来执行——扣款成功了加款失败是不可接受的。事务就是保证这种原子性的机制。

###### 2. 事务的 ACID 特性是什么？

[[../../../21_MySqlKnowledge/03_事务与并发控制/01、事务与ACID|📖]]
ACID 是事务四个核心特性的首字母缩写，是衡量事务可靠性的标准。

**原子性（Atomicity）**：事务里的所有操作是一个整体，要么全部成功提交，要么全部失败回滚，不允许部分成功。银行转账里，扣款和加款必须同时成功或同时回滚，不能出现只扣不加的情况。

**一致性（Consistency）**：事务执行前后，数据库必须从一个合法状态变到另一个合法状态，不能破坏数据的完整性约束。转账前后两个账户的总金额必须不变，这就是一致性。

**隔离性（Isolation）**：并发执行的事务之间互不干扰，每个事务看起来像是在独占数据库执行一样。你的转账操作不应该受到其他人同时转账的影响。

**持久性（Durability）**：事务一旦提交，对数据的修改就是永久性的，哪怕系统立刻崩溃，重启后数据也不会丢失。

一致性是目标，原子性、隔离性、持久性是手段，三者共同保证一致性。

###### 3. MySQL 如何实现原子性（Atomicity）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/01、事务与ACID|📖]]
MySQL 通过 **Undo Log** 实现原子性。

Undo Log 是逻辑日志，记录的是每个操作的"逆操作"：INSERT 对应记录主键（回滚时执行 DELETE）；DELETE 对应记录完整的行数据（回滚时执行 INSERT）；UPDATE 对应记录修改前的字段值（回滚时执行反向 UPDATE）。

当事务执行失败或调用 ROLLBACK 时，InnoDB 按照 Undo Log 的记录反向执行，把数据恢复到修改前的状态。

在 Java 里，你调用 `connection.rollback()`，JDBC 驱动通知 MySQL 执行回滚，MySQL 就根据 Undo Log 把数据复原。

###### 4. MySQL 如何实现一致性（Consistency）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/01、事务与ACID|📖]]
一致性是 ACID 的最终目标，它不是由某个单一机制保证的，而是原子性、隔离性、持久性协同作用的结果，同时也需要应用层配合。

数据库层面，主键约束、唯一约束、外键约束、NOT NULL 约束、数据类型检查等都是一致性的防线——违反了这些约束的操作会直接报错拒绝执行。

业务层面，你需要在代码里保证业务规则正确。比如转账前要检查余额是否充足，不足要抛出异常触发回滚，而不是让扣出负数的账户存在数据库里。

###### 5. MySQL 如何实现隔离性（Isolation）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/01、事务与ACID|📖]]
MySQL 通过**锁机制**和 **MVCC（多版本并发控制）** 两套机制共同实现隔离性。

锁机制是悲观思路：读时加共享锁（S锁，读读共享，读写互斥），写时加排他锁（X锁，完全独占）。但纯加锁的方式并发性较差，读写会互相阻塞。

MVCC 是 InnoDB 的核心并发机制，乐观思路：给每行数据维护一条版本链，每次修改生成新版本，老版本留在 Undo Log 里。读操作（快照读）根据 ReadView 找到对自己可见的版本，不需要加锁，读写互不阻塞。每行有两个隐藏字段：`DB_TRX_ID`（最后修改的事务ID）和 `DB_ROLL_PTR`（指向 Undo Log 旧版本的指针）。

两套机制分工：快照读（普通 SELECT）用 MVCC，当前读（SELECT FOR UPDATE、UPDATE、DELETE）用锁。

###### 6. MySQL 如何实现持久性（Durability）？

[[../../../21_MySqlKnowledge/03_事务与并发控制/01、事务与ACID|📖]]
MySQL 通过 **Redo Log + WAL 策略**实现持久性。

背景问题：InnoDB 为了性能，数据修改先在内存 Buffer Pool 里做，再异步刷到磁盘。如果刷盘前宕机，内存里的修改就丢了。

解决方案是 WAL（Write-Ahead Logging，预写式日志）：**先写日志，再写数据页**。具体流程：

1. 修改 Buffer Pool 中的数据页
2. 把本次修改记到 Redo Log Buffer
3. 事务提交时，将 Redo Log Buffer 强制刷盘（`fsync`）
4. 后台线程再把脏页异步刷到数据文件

如果步骤 4 之前宕机，重启时 MySQL 会扫描 Redo Log，把已提交但未落盘的事务重新应用，保证数据不丢失。

Redo Log 是顺序写，数据页是随机写，顺序写的 I/O 性能远高于随机写，所以 WAL 不仅保证了持久性，还提升了写入性能。

###### 7. 什么是脏读、不可重复读、幻读？

[[../../../21_MySqlKnowledge/03_事务与并发控制/04、隔离级别与并发问题|📖]]
这是并发事务的三种经典问题，理解它们是理解隔离级别的基础。

**脏读**：事务 A 读到了事务 B 还未提交的数据。如果 B 最后回滚了，A 读到的就是一条根本不存在的"幻象"数据。这是最严重的问题，会导致业务逻辑基于错误数据做决策。

**不可重复读**：事务 A 在同一个事务里两次读同一行数据，结果不一样——因为中间事务 B 提交了对这行数据的修改。A 第一次查余额是 100，B 转账后变成了 80，A 再查就是 80 了。重点是同一行被**修改**了。

**幻读**：事务 A 两次查询同一个范围，结果集的行数不一样——因为中间事务 B 提交了**新增或删除**。A 第一次统计年龄在 20~30 的用户有 10 个，B 新增了一个这个年龄段的用户，A 再统计变成 11 个了。重点是结果集里多了或少了行，而不是某行数据变了。

###### 8. MySQL 的事务隔离级别有哪些？

[[../../../21_MySqlKnowledge/03_事务与并发控制/04、隔离级别与并发问题|📖]]
SQL 标准定义了四种隔离级别，从低到高，对并发问题的防护越来越全面，但并发性能也越来越低。

**读未提交（Read Uncommitted）**：最低级别，允许读到其他事务未提交的数据，脏读、不可重复读、幻读都可能发生。几乎没有业务会用这个级别。

**读已提交（Read Committed）**：只能读到其他事务已提交的数据，解决了脏读，但不可重复读和幻读仍然存在。这是 Oracle 和 PostgreSQL 的默认级别，在互联网业务里用得也很多（很多公司把 MySQL 的隔离级别从 RR 改成了 RC，因为 RC 下间隙锁少，死锁概率更低）。

**可重复读（Repeatable Read）**：同一个事务内多次读同一行，结果一致，解决了不可重复读。这是 **MySQL 的默认隔离级别**。InnoDB 在 RR 级别下通过 MVCC + Next-Key Lock，实际上也解决了大部分幻读场景。

**串行化（Serializable）**：最高级别，所有事务串行执行，完全避免了三种并发问题，但并发性能极差，只在对数据一致性要求极高、并发量极低的场景才会用。

###### 9. 如何查看和设置事务隔离级别？

```sql
-- 查看当前会话隔离级别
SELECT @@transaction_isolation;

-- 查看全局隔离级别
SELECT @@global.transaction_isolation;

-- 设置当前会话隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置全局隔离级别
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

在 Java 应用里，一般通过 Spring 的 `@Transactional` 注解来设置：

```java
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void transferMoney(String from, String to, BigDecimal amount) {
    // 转账业务逻辑
}
```

###### 10. 可重复读（RR）如何解决幻读问题？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
InnoDB 在 RR 级别下通过两套机制来防幻读，要根据读的类型分开讨论：

**快照读（普通 SELECT）用 MVCC**：事务第一次查询时创建一个 ReadView 快照，之后整个事务里的查询都读这个快照，看不到其他事务新增或删除的行，自然不会幻读。

**当前读（SELECT FOR UPDATE、UPDATE、DELETE）用 Next-Key Lock**：Next-Key Lock = 记录锁 + 间隙锁，不仅锁住当前存在的行，还锁住行之间的间隙，防止其他事务在这个范围内插入新行。

举个例子：

```sql
-- 事务A执行范围查询并加锁
SELECT * FROM users WHERE age BETWEEN 20 AND 30 FOR UPDATE;
```

InnoDB 会锁住 age 在 20~30 范围内的现有记录，同时锁住这个区间里的间隙，让其他事务无法在这个范围内插入新用户，从而防止幻读。

###### 11. 什么是当前读和快照读？

[[../../../21_MySqlKnowledge/03_事务与并发控制/03、MVCC多版本并发控制|📖]]
InnoDB 里有两种读取数据的方式：

**快照读**：读的是数据的历史版本（通过 ReadView + Undo Log 构建），不加锁，读写不互相阻塞。普通的 `SELECT * FROM table WHERE ...` 就是快照读。RR 级别下，快照在事务第一次查询时创建，之后固定不变；RC 级别下，每条语句都创建新快照。

**当前读**：读的是数据的最新提交版本，并且加锁防止其他事务并发修改。`SELECT ... FOR UPDATE`（加 X 锁）、`SELECT ... LOCK IN SHARE MODE`（加 S 锁）、`UPDATE`、`DELETE`、`INSERT` 都是当前读。

两者的区别：快照读更快、并发性更好，但读到的可能不是最新数据；当前读保证读到最新数据，但需要等锁。业务里需要先读后写保证一致性的场景（比如扣库存）要用当前读，不能用快照读。
