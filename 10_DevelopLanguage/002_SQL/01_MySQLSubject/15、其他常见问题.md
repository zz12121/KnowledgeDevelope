###### 1. VARCHAR 和 CHAR 的区别？
[[../../../21_MySqlKnowledge/01_MySQL基础/02、数据类型与字符集|📖]]

CHAR 和 VARCHAR 的核心区别在于**存储方式**。CHAR 是**固定长度**类型，定义为 CHAR(10) 就一定占 10 个字符的空间，不足部分用空格填充，检索时自动去掉尾部空格。VARCHAR 是**可变长度**类型，实际存储的是真实字符数据加上 1~2 字节的长度前缀（≤255 用 1 字节，>255 用 2 字节），更加节省空间。

从性能角度看，CHAR 由于长度固定，InnoDB 可以直接通过偏移量定位数据，读写速度略快；VARCHAR 需要计算偏移量，稍慢一些。从适用场景看，**CHAR 适合长度固定的字段**，比如 MD5 哈希值、身份证号、邮政编码；**VARCHAR 适合长度变化较大的字段**，比如姓名、地址、描述信息，用它来存储可以有效节省磁盘空间。

**存储机制深度解析**：
- **CHAR**：在`storage/innobase/include/dict0mem.h`中，InnoDB会将CHAR字段填充到定义长度，便于直接通过偏移量定位数据
- **VARCHAR**：需要额外的1-2字节存储实际长度（≤255用1字节，>255用2字节），数据存储更加紧凑
###### 2. DATETIME 和 TIMESTAMP 的区别？
[[../../../21_MySqlKnowledge/01_MySQL基础/02、数据类型与字符集|📖]]

DATETIME 和 TIMESTAMP 最本质的区别是**时区处理方式**和**存储范围**。

DATETIME 存储的是"字面时间"，你存什么取出来就是什么，不做任何时区转换，支持从 `1000-01-01` 到 `9999-12-31` 的宽广范围，占用 8 字节（MySQL 5.6.4 以后是 5 字节加小数秒）。TIMESTAMP 则不同，它在存储时会调用 `my_timestamp_to_utc()` 将本地时间转换为 UTC，读取时再通过 `my_timestamp_from_utc()` 转回当前会话的时区，因此**同一条数据在不同时区的客户端显示的时间是不同的**。TIMESTAMP 只占 4 字节，但范围只到 2038 年（即著名的 2038 年问题）。

另一个常用特性是**自动更新**：TIMESTAMP 支持 `DEFAULT CURRENT_TIMESTAMP` 和 `ON UPDATE CURRENT_TIMESTAMP`，插入或更新时自动记录当前时间，非常适合 `created_at`、`updated_at` 这类字段；DATETIME 默认不支持自动更新。还有一点值得注意，往 TIMESTAMP 字段插入 NULL 时，会自动存储当前时间，而不是真正的 NULL。

**使用建议**：需要跨时区的国际化应用用 TIMESTAMP，本地化业务或需要存储历史日期用 DATETIME。

**时区处理源码角度**：TIMESTAMP在存储时调用`my_timestamp_to_utc()`转换为UTC时间，读取时通过`my_timestamp_from_utc()`转换回当前时区。
###### 3. TEXT 和 BLOB 的区别？
[[../../../21_MySqlKnowledge/01_MySQL基础/02、数据类型与字符集|📖]]

TEXT 和 BLOB 都是用来存储大数据的类型，核心区别是**存储的是字符数据还是二进制数据**。TEXT 有字符集概念，存储的是文本，支持字符集转换，排序按字符集规则不区分大小写；BLOB 存储的是原始字节，没有字符集，排序按字节数值二进制比较，大小写敏感。

场景上很好区分：**TEXT 适合文章正文、用户评论、配置文件**这类长文本；**BLOB 适合图片、音视频文件、加密数据、序列化对象**等二进制内容。

从存储引擎实现看，InnoDB 处理 TEXT/BLOB 时，当值超过一定大小，前 768 字节存储在主记录中，剩余部分存放在溢出页（off-page）里，这意味着大 TEXT/BLOB 字段会触发额外的 IO 操作，对性能有影响。**实际开发中建议避免在数据库中存储大文件**，应改用对象存储（如 OSS、S3），数据库只存 URL。
###### 4. NULL 和空字符串的区别？
[[../../../21_MySqlKnowledge/01_MySQL基础/02、数据类型与字符集|📖]]

NULL 和空字符串 `''` 在语义和行为上有本质区别，搞混会踩很多坑。

**语义**上，NULL 表示"未知/缺失/不适用"，是一种特殊状态；空字符串 `''` 表示"已知的空值"，是一个确定的零长度字符串。**存储**上，NULL 用 NULL 位图标记，几乎不占额外空间；空字符串需要存储长度信息，占用少量空间。

**查询**方式不同：NULL 必须用 `IS NULL` 或 `IS NOT NULL` 判断，用 `= NULL` 永远返回 FALSE；空字符串用 `= ''` 或 `!= ''` 判断。

**聚合函数**的处理也不同：`COUNT(col)` 会忽略 NULL 值，但会计入空字符串；`COUNT(*)` 则统计所有行包括 NULL。这一点在统计时经常造成结果不符合预期。

**索引行为**方面，唯一索引允许存在多个 NULL（NULL 被认为互不相等），但只允许一个空字符串 `''`。

**设计建议**：如果字段可能为空，优先用 `NOT NULL DEFAULT ''` 而非允许 NULL，这样查询条件更简单，也能避免 NULL 参与运算时整个表达式返回 NULL 的陷阱（比如 `NULL + 1 = NULL`）。
###### 5. DELETE、TRUNCATE、DROP 的区别？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]

这三个操作经常被混用，但在语句类型、可回滚性、性能上差别很大。

**DELETE** 是 DML 语句，逐行删除数据，每删一行都会记录 Undo Log，因此**可以回滚**，也会触发触发器，但不重置自增计数器。由于是逐行处理，大量数据删除时性能较慢。

**TRUNCATE** 是 DDL 语句，直接释放数据页，相当于把表快速清空——在 InnoDB 内部实际是先执行 DROP TABLE 再重新 CREATE TABLE 的操作，所以速度极快。但也因此**不可回滚**，不触发触发器，同时会重置自增计数器。

**DROP** 也是 DDL，直接删除整张表（包括表结构、数据、索引、触发器），速度最快，**完全不可回滚**。

**选择原则**：只删部分数据用 DELETE；清空整张表同时保留表结构用 TRUNCATE；连表都不要了用 DROP。生产环境对 TRUNCATE 和 DROP 要格外慎重，执行前务必确认有备份。

**TRUNCATE实现原理**：在InnoDB中，TRUNCATE实际上是通过先执行DROP TABLE再CREATE TABLE来实现的快速清空，这也是为什么它不能回滚的原因。
###### 6. UNION 和 UNION ALL 的区别？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]
**核心区别**：UNION会去除重复行并进行排序，而UNION ALL直接合并所有结果。
**性能影响**：
- **UNION**：需要创建临时表进行去重排序，性能开销大
- **UNION ALL**：直接返回合并结果，性能更高
**使用建议**：除非确实需要去重，否则优先使用UNION ALL，特别是在大数据量时性能差异显著。
###### 7. HAVING 和 WHERE 的区别？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]

WHERE 和 HAVING 的核心区别是**执行时机不同**：WHERE 在 GROUP BY 之前过滤行，HAVING 在 GROUP BY 之后过滤分组结果。

正因为执行时机不同，它们能用的字段也不同。WHERE 只能引用表中的原始字段，**不能使用聚合函数**（因为聚合还没发生）；HAVING 可以引用分组后的字段以及聚合函数的结果，比如 `HAVING COUNT(*) > 5` 或 `HAVING AVG(salary) > 8000`。

**性能建议**：尽量把过滤条件写在 WHERE 里，因为 WHERE 先过滤能减少进入 GROUP BY 的数据量，而 HAVING 是在分组后才过滤，等于白做了一些计算。只有必须基于聚合结果做过滤时才用 HAVING。
###### 8. EXISTS 和 IN 的区别？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]

EXISTS 和 IN 都能实现"子查询过滤"，但执行逻辑不同，适用场景也有差异。

**IN** 的执行方式是先将子查询的结果集物化（即完整执行子查询，把结果放到临时集合中），再逐行与外表比较。它**适合子查询结果集较小**的场景，结果集越大，内存和比较开销越高。另外要注意，如果子查询结果包含 NULL，`col IN (子查询)` 可能返回 UNKNOWN（即既不是 TRUE 也不是 FALSE），导致过滤行为不符合预期。

**EXISTS** 是半连接语义，外表驱动，对外表每一行都去执行子查询，**只要找到一条匹配就立即停止**（short-circuit），不需要完整遍历子查询。EXISTS 不受子查询中 NULL 的影响，逻辑更加清晰。它**适合外表小、子查询大**的场景。

不过现代 MySQL 优化器（5.6+）会对 IN 进行半连接转换优化，很多时候两者生成的执行计划已经相差不大。但从**业务语义**角度，"判断是否存在"这类逻辑用 EXISTS 表达更自然，可读性更好。

**优化器策略**：现代MySQL优化器会对IN进行半连接转换优化，但EXISTS的逻辑更符合"存在性检查"的业务语义。
###### 9. 如何查看表的结构和索引信息？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]
**常用命令**：
```sql
-- 查看表结构
DESC table_name;
DESCRIBE table_name;
SHOW COLUMNS FROM table_name;

-- 查看建表语句（包含索引）
SHOW CREATE TABLE table_name;

-- 查看索引信息
SHOW INDEX FROM table_name;
ANALYZE TABLE table_name;  -- 更新索引统计信息

-- 查看表状态信息
SHOW TABLE STATUS LIKE 'table_name';
```
**Information Schema查询**：

```sql
-- 查看列信息
SELECT * FROM information_schema.COLUMNS 
WHERE TABLE_SCHEMA = 'your_db' AND TABLE_NAME = 'your_table';

-- 查看索引统计
SELECT * FROM information_schema.STATISTICS 
WHERE TABLE_SCHEMA = 'your_db' AND TABLE_NAME = 'your_table';
```

###### 11. 什么是存储过程和触发器？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]

**存储过程**是预编译并存储在数据库中的 SQL 语句集合，通过名称显式调用（`CALL proc_name()`）。它支持输入输出参数、循环判断等编程结构，可以返回结果集，并且支持完整的事务控制，适合封装复杂的业务逻辑或批量处理任务。

**触发器**是一种特殊的存储过程，在特定数据库事件（INSERT / UPDATE / DELETE）发生时**自动执行**，无法直接调用。它通过 NEW 和 OLD 关键字访问被操作的行数据（NEW 是新值，OLD 是旧值），与触发语句处于同一事务，不能直接返回结果集。触发器常用于**数据审计**（记录变更日志）和**完整性约束**（如阻止薪资降低）。

两者对比来看：存储过程是主动调用的，灵活性强；触发器是被动触发的，对业务代码透明但调试较困难，滥用会带来隐蔽的副作用，需谨慎使用。

**触发器示例与源码机制：**
```sql
DELIMITER //
CREATE TRIGGER before_salary_update 
BEFORE UPDATE ON employees FOR EACH ROW
BEGIN
    IF NEW.salary < OLD.salary THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Salary cannot decrease';
    END IF;
END//
```
**源码角度**：触发器定义存储在`mysql.triggers`系统表中，当DML操作执行时，查询优化器会检查是否存在相关触发器并将其逻辑注入执行计划。
###### 12. 什么是视图？视图的优缺点？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]
**视图**是虚拟表，其内容由查询定义，不存储实际数据，在查询时动态生成。
**优点：**
- **安全性**：隐藏敏感数据，限制用户只能访问特定列
- **简化复杂性**：封装复杂查询逻辑，简化应用代码
- **数据抽象**：底层表结构变化时，视图接口可保持不变
**缺点：**
- **性能开销**：复杂视图查询可能转换为复杂连接，影响性能
- **更新限制**：不是所有视图都支持更新操作
**可更新视图条件：**
- 必须包含基表的所有非空列
- 不能使用聚合函数、DISTINCT、GROUP BY等
- 来自单一基表或使用可更新的连接
###### 13. MySQL 8.0 有哪些新特性？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]
**1. 窗口函数**
支持OVER()子句，实现复杂分析查询，如排名、累计汇总等。
**2. 通用表表达式**
```sql
WITH department_stats AS (
    SELECT department_id, AVG(salary) avg_sal
    FROM employees GROUP BY department_id
)
SELECT * FROM department_stats WHERE avg_sal > 5000;
```
**3. 原子DDL操作**
数据定义语句完全原子化，避免部分成功导致的元数据不一致。
**4. 增强的JSON支持**
```sql
SELECT json_extract(data, '$.user.name') FROM documents;
```
**5. 不可见索引**
`ALTER TABLE t ADD INDEX idx_name (name) INVISIBLE;`可临时禁用索引测试性能影响。
**源码角度**：窗口函数在`sql/window_functions.cc`中实现，使用基于排序的流式处理，避免中间结果物化，提升内存使用效率。
###### 14. 什么是 CTE（公用表表达式）？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]
CTE是临时命名结果集，在单个语句执行范围内有效，支持递归查询。
**非递归CTE：**
```sql
WITH regional_sales AS (
    SELECT region, SUM(amount) total_sales
    FROM orders GROUP BY region
)
SELECT region, total_sales FROM regional_sales 
WHERE total_sales > 100000;
```
**递归CTE（组织结构查询）：**
```sql
WITH RECURSIVE employee_tree AS (
    -- 锚点查询
    SELECT employee_id, name, manager_id, 1 level
    FROM employees WHERE manager_id IS NULL
    UNION ALL
    -- 递归成员
    SELECT e.employee_id, e.name, e.manager_id, et.level + 1
    FROM employees e JOIN employee_tree et 
    ON e.manager_id = et.employee_id
)
SELECT * FROM employee_tree;
```
**源码角度**：CTE在`sql/sql_cte.cc`中实现，递归CTE使用临时表存储中间结果，通过迭代计算直到满足终止条件。
###### 15. 什么是窗口函数？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]
窗口函数对一组相关行（窗口）进行计算，同时保留原始行细节。
**核心语法：**
```sql
SELECT 
    employee_name,
    department,
    salary,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) as dept_rank,
    AVG(salary) OVER (PARTITION BY department) as dept_avg_salary
FROM employees;
```
**常用窗口函数：**
- **排名函数**：ROW_NUMBER(), RANK(), DENSE_RANK()
- **分布函数**：PERCENT_RANK(), CUME_DIST()
- **偏移函数**：LAG(), LEAD(), FIRST_VALUE(), LAST_VALUE()
- **聚合函数**：SUM(), AVG() 配合OVER()
**性能优化**：窗口函数通过`filesort`操作实现分区排序，合理使用索引可显著提升性能。
###### 16. 如何进行全文检索？
[[../../../21_MySqlKnowledge/01_MySQL基础/03、SQL语言与执行流程|📖]]
MySQL支持基于倒排索引的全文检索功能。
**创建全文索引：**
```sql
ALTER TABLE articles ADD FULLTEXT(title, content);
```
**全文检索查询：**
```sql
-- 自然语言模式
SELECT * FROM articles 
WHERE MATCH(title, content) AGAINST('database optimization');

-- 布尔模式（高级查询）
SELECT * FROM articles 
WHERE MATCH(title, content) 
AGAINST('+MySQL -Oracle' IN BOOLEAN MODE);
```
**InnoDB全文索引源码机制**：在`storage/innobase/fts`目录下，FTS索引维护倒排列表，将文档分解为词元并建立映射关系。查询时通过倒排索引快速定位包含关键词的文档。
**优化技巧**：
- 设置合适的`ft_min_word_len`参数控制最小词长
- 使用查询扩展提高召回率
- 对于大规模应用，考虑专用搜索引擎（如Elasticsearch）
###### 17. MySQL 如何处理表情符号（emoji）？
[[../../../21_MySqlKnowledge/01_MySQL基础/02、数据类型与字符集|📖]]
表情符号处理核心在于字符集和排序规则的支持。
**UTF8与UTF8MB4：**
- **utf8**：最多3字节，支持基本多语言平面（BMP）字符
- **utf8mb4**：4字节编码，完全支持Unicode，包括表情符号
**配置步骤：**
1. **修改字符集**：
```sql
ALTER DATABASE database_name CHARACTER SET = utf8mb4 COLLATE = utf8mb4_unicode_ci;
ALTER TABLE table_name CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```
2. **连接配置**：确保应用程序连接字符串指定UTF-8编码
3. **验证支持**：
```sql
SELECT * FROM test_emoji WHERE content LIKE '%😊%';
```
**Java应用配置：**
```java
// JDBC连接字符串
String url = "jdbc:mysql://localhost:3306/db?useUnicode=true&characterEncoding=UTF-8";
```
**源码角度**：在`strings/ctype-utf8mb4.c`中实现UTF8MB4编码处理，表情符号被识别为4字节序列，存储引擎在索引和比较操作中正确处理这些字符。
