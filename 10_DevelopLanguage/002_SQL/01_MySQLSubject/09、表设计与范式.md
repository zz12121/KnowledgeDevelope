###### 1. 什么是数据库范式？

[[../../../21_MySqlKnowledge/05_架构与运维/03、表设计与范式|📖]]
数据库范式是关系型数据库设计的规范体系，核心目标是通过合理的表结构设计来消除数据冗余、减少数据异常（插入异常、更新异常、删除异常），保证数据的一致性和完整性。

范式的核心思想是"一事一地"——一张表只描述一个实体或一种关系。范式从 1NF 到 5NF 递进，高级包含低级的要求。实际业务中，满足到**第三范式（3NF）通常就够了**，更高范式理论上完美但实践中可能过度设计，反而影响查询性能。

###### 2. 第一范式、第二范式、第三范式是什么？

[[../../../21_MySqlKnowledge/05_架构与运维/03、表设计与范式|📖]]
**第一范式（1NF）**：字段必须是原子的，不可再分。就是说不能在一个字段里存多个值，比如把电话和地址都塞进"联系信息"一个字段里，这就违反了 1NF。这是关系数据库的最基本要求，现代数据库通常都满足。

**第二范式（2NF）**：在满足 1NF 的基础上，所有非主键列必须完全依赖于整个主键，不能只依赖主键的一部分。这个问题主要出现在联合主键的情况下。比如选课表用（学号, 课程号）作为联合主键，但"课程学分"只依赖"课程号"，和"学号"无关——这就是部分依赖，违反了 2NF。把课程学分单独拆到课程表里就解决了。

**第三范式（3NF）**：在满足 2NF 的基础上，任何非主键列不能传递依赖于主键（不能存在 A → B → C 的依赖关系）。比如员工表里有"部门ID"和"部门名称"，部门名称依赖部门ID，部门ID才依赖员工ID，这就是传递依赖。解决方法是把部门信息拆到单独的部门表里。

###### 3. 什么是反范式化？为什么需要反范式化？

[[../../../21_MySqlKnowledge/05_架构与运维/03、表设计与范式|📖]]
反范式化是故意在表中引入冗余数据，用空间换查询时间。

为什么需要？过度范式化导致表数量多，一个复杂查询可能要 JOIN 七八张表，JOIN 操作昂贵，而且优化空间有限。有些高频查询如果可以通过冗余一个字段来避免 JOIN，性能提升非常明显。

典型的反范式化场景：在订单明细表里冗余商品名称，避免每次查询都要关联商品表（商品名称基本不变，冗余成本极低）；为复杂报表创建预汇总表，定期计算好结果存起来，查询时直接读；把访问频率极高的 1:1 关系合并到一张表，减少关联。

反范式化是双刃剑：读性能提升了，但写操作需要同时维护冗余数据，一致性保障变复杂了。所以不要为了性能随意冗余，要有明确的查询场景驱动，而且要在应用层或触发器里保证冗余字段的同步更新。

###### 4. 如何设计一个好的数据库表结构？

[[../../../21_MySqlKnowledge/05_架构与运维/03、表设计与范式|📖]]
好的表结构设计需要在规范性、性能和可扩展性之间平衡，几个核心原则：

**范式为基础，反范式为调优**：先按 3NF 设计，保证数据一致性；再针对高频查询场景按需冗余。

**字段类型选小不选大**：能用 `SMALLINT` 不用 `INT`，能用 `VARCHAR(20)` 不用 `VARCHAR(255)`，越小的类型意味着更少的磁盘 I/O 和内存占用。整型运算比字符串快，能用整型的不用字符串。

**尽量 NOT NULL + 默认值**：NULL 值让索引统计和值比较都变复杂，尽量给字段加 NOT NULL 约束并设默认值。

**命名清晰规范**：表名和字段名要有自解释性，统一小写下划线风格（如 `user_profile`），不用数据库关键字。

**提前考虑扩展性**：如果将来可能分库分表，主键要用分布式 ID（雪花算法），不要用单机自增 ID，不然分库后 ID 冲突会非常麻烦。

###### 5. 主键如何选择？自增 ID 和 UUID 的优缺点？

[[../../../21_MySqlKnowledge/05_架构与运维/03、表设计与范式|📖]]
这是一个经典的权衡问题，没有绝对的答案，要看业务场景。

**自增 ID（AUTO_INCREMENT）**：数值类型，索引效率高；INSERT 时顺序写入 B+树叶节点，页分裂少，写入性能优秀；易读易排序；但只在单库单表内唯一，分库分表时会有 ID 冲突问题；连续 ID 容易被爬虫推测数据量。

**UUID**：128 位全球唯一，完全没有冲突问题，安全性高，天然适合分布式；但字符串类型，索引体积大；最关键的是完全随机，每次 INSERT 都要在 B+树中间随机插入，导致大量页分裂和索引碎片，写入性能差，表越大越明显。

**推荐方案**：单体应用无分库分表规划，用 `BIGINT` 自增 ID。有分布式需求，用**雪花算法（Snowflake）**——生成的是 64 位 BIGINT，趋势递增（时间戳高位），全局唯一，兼具自增 ID 的性能和 UUID 的全局唯一性，是工程实践中的最优解。

###### 6. 什么是外键？使用外键的优缺点？

[[../../../21_MySqlKnowledge/05_架构与运维/03、表设计与范式|📖]]
外键是数据库层面的约束，保证子表的外键字段值必须在主表的主键中存在（参照完整性）。比如订单表的 `user_id` 是外键，指向用户表的 `id`，这样就不可能有"对应用户不存在"的订单。

**使用外键的好处**：数据库层面强制数据一致性，不依赖应用层逻辑，写代码时不用担心误删了关联数据；数据文档化，看表结构就知道表之间的关系。

**外键的代价**：每次 INSERT/UPDATE/DELETE 都要检查关联约束，额外的 I/O 开销；维护外键关系容易导致锁范围扩大，高并发下是性能瓶颈；对分库分表、微服务拆分的支持很差——表都不在同一个库里了，外键根本没法用。

**互联网行业的主流实践**：**不在数据库层面使用外键**，而是在应用层通过代码逻辑保证数据一致性。这样牺牲了一定的数据库层面安全感，但换来了灵活性和性能，更适合高并发和分布式架构。
