###### 1. MySQL 有哪些常见的存储引擎？

[[../../../21_MySqlKnowledge/01_MySQL基础/01、MySQL架构与存储引擎|📖]]
MySQL 支持多种存储引擎，最常用的几个：

**InnoDB** 是 MySQL 5.5 起的默认引擎，也是生产中用得最多的。它支持事务（ACID）、行级锁、外键约束和 MVCC，崩溃恢复能力强，适合一切需要数据可靠性的 OLTP 业务，比如电商、金融系统。

**MyISAM** 不支持事务和外键，用的是表级锁，写操作会锁整张表，并发性差。但它的 `COUNT(*)` 极快（保存了行数），支持全文索引，读多写少、对一致性要求不高的场景可以考虑，比如内容管理系统。不过现在 InnoDB 的全文索引已经很完善了，MyISAM 的适用场景越来越窄。

**Memory** 把数据全存在内存里，速度极快，但重启后数据就没了。适合存会话、临时数据、缓存等非持久化内容。

**Archive** 有极高的压缩比，只支持插入和查询，不支持索引和事务，适合存大量只写不改的历史归档数据，比如操作日志、审计记录。

结论是：新项目除非有非常特殊的需求，一律用 InnoDB。

###### 2. InnoDB 和 MyISAM 的区别？

[[../../../21_MySqlKnowledge/01_MySQL基础/01、MySQL架构与存储引擎|📖]]
这是面试最高频的问题之一，核心区别有以下几点：

**事务支持**：InnoDB 完整支持 ACID 事务，意外宕机后数据一致性有保障；MyISAM 完全不支持事务，宕机后表容易损坏。

**锁粒度**：InnoDB 支持行级锁，多个事务可以并发操作不同的行，并发性能高；MyISAM 只有表级锁，一个写操作会阻塞整张表的所有读写，并发性极差。

**外键**：InnoDB 支持外键约束，保证数据参照完整性；MyISAM 不支持，需要应用层自己维护关联逻辑。

**存储结构**：InnoDB 采用聚簇索引，数据行就存在主键索引的叶节点里，按主键顺序物理存放；MyISAM 的数据文件（.MYD）和索引文件（.MYI）是分开的，索引叶节点存的是数据行的物理地址。

**COUNT(\*)**：MyISAM 保存了总行数，`COUNT(*)` 直接返回，非常快；InnoDB 没有保存行数，每次都要扫描统计。

**崩溃恢复**：InnoDB 有 Redo Log 和 Double Write Buffer，崩溃后能自动恢复；MyISAM 崩溃后表容易损坏，需要用 `CHECK TABLE` 和 `REPAIR TABLE` 手动修复。

从 Java 开发者角度，InnoDB 的 MVCC 机制非常重要：写操作不会阻塞读，读操作也不会阻塞写，这靠的是 Undo Log 版本链——修改数据时不直接覆盖旧版本，而是创建新版本，老版本留在 Undo Log 里，其他事务仍然可以读到它们。

###### 3. 如何选择合适的存储引擎？

[[../../../21_MySqlKnowledge/01_MySQL基础/01、MySQL架构与存储引擎|📖]]
选择逻辑其实很简单：

需要事务、需要高并发读写、需要外键、需要崩溃恢复——选 **InnoDB**，这覆盖了 99% 的业务场景。

纯读多写少、对一致性要求很低、需要 `COUNT(*)` 极快——可以考虑 **MyISAM**，但这种场景越来越少。

需要存临时数据、访问频率极高但可以接受重启丢失——用 **Memory**。

需要存大量归档日志、几乎不查询——用 **Archive**。

给 Java 开发者的建议：用 Spring Boot 做业务开发，默认选 InnoDB，不要为了一个 `COUNT(*)` 快就换成 MyISAM，因为你失去的远比得到的多。

###### 4. InnoDB 的架构是怎样的？

[[../../../21_MySqlKnowledge/01_MySQL基础/01、MySQL架构与存储引擎|📖]]
InnoDB 架构分内存和磁盘两大部分。

**内存部分**：核心是 **Buffer Pool**，这是 InnoDB 的心脏。读数据先把数据页从磁盘加载到 Buffer Pool，改数据先改内存里的页（称为脏页），再由后台线程异步刷到磁盘。Buffer Pool 大小对性能影响极大，生产上通常设为系统内存的 50%~80%（`innodb_buffer_pool_size`）。此外还有 **Log Buffer**，用于暂存 Redo Log，减少频繁刷盘；以及 **Change Buffer**，缓存对非唯一二级索引的写操作，等到数据页被读进内存时再合并，减少随机 I/O。

**磁盘部分**：数据存在**表空间**里，MySQL 5.6.6 之后默认每张表有自己独立的 `.ibd` 文件（独立表空间），方便单独备份和空间回收。**Redo Log** 是崩溃恢复的关键，采用 WAL 策略（先写日志再写数据），循环写两个文件，事务提交时 Redo Log 必须先落盘。**Undo Log** 存在撤销表空间里，负责事务回滚和 MVCC 版本链。

###### 5. 什么是表空间（Tablespace）？

[[../../../21_MySqlKnowledge/01_MySQL基础/01、MySQL架构与存储引擎|📖]]
表空间是 InnoDB 存储逻辑结构的最顶层，可以理解为 InnoDB 自己管理的"虚拟文件系统"，所有数据和索引都放在表空间里。

常见的几种类型：

**系统表空间**（ibdata1）：存放 InnoDB 数据字典、双写缓冲等元数据。MySQL 5.6 之前所有表的数据默认都在这里，会越来越大难以收缩，现在不推荐把数据表放这里。

**独立表空间**（.ibd 文件）：MySQL 5.6.6 之后的默认配置，每张表有独立的 `.ibd` 文件。好处是可以单独备份、传输、回收空间，强烈推荐保持这个默认配置。

**通用表空间**：允许把多张表放到一个共享文件里，适合需要统一管理多张表的场景。

**临时表空间**：存用户创建的临时表和内部临时表，会话结束自动清理。
