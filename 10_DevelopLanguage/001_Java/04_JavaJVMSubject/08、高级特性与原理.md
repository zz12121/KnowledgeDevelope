###### 1. 什么是 JMM（Java 内存模型）？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
JMM（Java Memory Model，Java 内存模型）是 Java 定义的一套**多线程间内存访问规范**，规定了一个线程对变量的写操作何时、以何种方式对其他线程可见。

JMM 的核心抽象是：每个线程有自己的**工作内存**（类似 CPU 缓存），对变量的操作先在工作内存里进行，再同步回**主内存**。线程之间不能直接访问对方的工作内存。

JMM 通过三个关键特性来保证多线程的正确性：
- **原子性**：一个操作要么全部完成，要么完全没有发生，不会被中断
- **可见性**：一个线程对共享变量的修改能被其他线程及时看到
- **有序性**：程序的执行顺序按照代码的书写顺序（可被编译器和处理器的指令重排序打乱）

###### 2. happens-before 原则是什么？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#五、执行引擎|📖]]
happens-before 是 JMM 的核心规则，定义了操作间的**可见性和顺序保证**。如果操作 A happens-before 操作 B，那么 A 的结果对 B 一定可见，且 A 在 B 之前发生。

**主要的 happens-before 规则**：

1. **程序顺序规则**：在同一个线程内，代码书写顺序靠前的操作 happens-before 靠后的操作
2. **监视器锁规则**：解锁（unlock）happens-before 后续的加锁（lock）操作
3. **volatile 变量规则**：对 volatile 变量的写 happens-before 后续对该变量的读
4. **线程启动规则**：`Thread.start()` happens-before 新线程里的任意操作
5. **线程终止规则**：线程中所有操作 happens-before 其他线程检测到该线程结束（join/isAlive）
6. **中断规则**：对线程 interrupt() 的调用 happens-before 被中断线程检测到中断
7. **对象 finalizer 规则**：对象构造函数完成 happens-before finalize() 方法的开始
8. **传递性**：如果 A happens-before B，B happens-before C，那么 A happens-before C

###### 3. volatile 关键字的底层实现原理？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#五、执行引擎|📖]]
volatile 通过两个机制保证可见性和有序性：

**可见性**：JVM 会在 volatile 变量的写操作后插入一条**写屏障（Store Barrier）**，强制把工作内存中的值刷新回主内存；在 volatile 变量的读操作前插入**读屏障（Load Barrier）**，强制从主内存重新加载最新值，而不是读工作内存中的缓存。这样每次读 volatile 变量都是最新值。

**有序性（禁止指令重排）**：volatile 写操作前后会分别插入 StoreStore 屏障和 StoreLoad 屏障，volatile 读操作后会插入 LoadLoad 和 LoadStore 屏障，防止编译器和 CPU 把 volatile 操作和它前后的其他操作重排序，保证了 volatile 写操作对其后的 volatile 读操作的可见性（happens-before 保证）。

**volatile 不保证原子性**：`i++` 这种操作（read-modify-write）不是原子的，即使 i 是 volatile 也不行，多线程下会有问题。需要原子性要用 `AtomicInteger` 或 `synchronized`。

###### 4. synchronized 的底层实现原理？

[[../../../20_JavaKnowledge/04_JVM/04、对象的创建与内存布局#四、逃逸分析（Escape Analysis）|📖]]
synchronized 在字节码层面对应 **monitorenter** 和 **monitorexit** 指令（用于同步块），或者方法的 **ACC_SYNCHRONIZED** 标志（用于同步方法）。

底层依赖**对象监视器（Monitor）**，每个对象都有一个 Monitor，Monitor 里有计数器和持有线程引用：
- 线程执行 monitorenter 时尝试获取 Monitor 的持有权，如果计数器为0则获取成功，计数器+1；同一线程可重入，计数器继续+1
- 执行 monitorexit 时计数器-1，减到0时释放 Monitor
- 其他线程尝试获取已被持有的 Monitor 时会阻塞，进入该对象的等待队列（EntryList）

JDK 6 以后对 synchronized 做了大量优化，引入了锁升级机制（见下题）。

###### 5. 什么是偏向锁、轻量级锁、重量级锁？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#三、字节码指令分类|📖]]
这是 synchronized 为了减少锁开销而引入的**三级锁升级机制**，根据竞争激烈程度动态升级：

**偏向锁**：适合只有一个线程访问同步代码的场景。第一次加锁时，把线程 ID 记录在对象头的 Mark Word 里，下次这个线程再来，发现 Mark Word 里的线程 ID 就是自己，直接进入，完全无锁操作，**连 CAS 都不用**。多个线程竞争时才撤销偏向锁。

**轻量级锁**：有多个线程交替执行同步块（没有真正的并发冲突）。用 **CAS（Compare-And-Swap）** 操作把 Mark Word 替换成指向线程栈帧里锁记录的指针来加锁，加锁失败时自旋等待（不挂起线程）。自旋超过一定次数或自旋时又有第三个线程来，升级为重量级锁。

**重量级锁**：多线程真正并发竞争。依赖操作系统的互斥量（Mutex），竞争失败的线程被挂起进入内核态等待。**上下文切换开销大**，但不占用 CPU，适合锁竞争激烈的场景。

###### 6. 锁升级的过程是怎样的？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

**升级过程**：
1. 对象刚创建，是**无锁**状态
2. 第一个线程加锁，Mark Word 记录线程 ID，进入**偏向锁**状态
3. 第二个线程竞争，偏向锁被撤销，原线程和新线程都尝试用 CAS 替换 Mark Word，进入**轻量级锁**状态，失败的线程自旋等待
4. 自旋超过阈值（JDK 6+ 是自适应自旋，由 JVM 根据历史情况决定），或者有第三个线程加入竞争，升级为**重量级锁**，失败的线程被挂起

**锁只能升级不能降级**（但是偏向锁可以被重置）。

###### 7. 什么是锁消除和锁粗化？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
**锁消除（Lock Elimination）**：JIT 编译器在运行时分析，如果发现一段加了锁的代码里，被锁住的对象根本不会被多线程共享（逃逸分析表明不存在竞争），就自动把这个锁去掉，减少不必要的同步开销。

例如：`StringBuffer.append()` 里有 synchronized，但如果 StringBuffer 对象是方法内的局部变量、没有逃逸，JIT 就会把这些锁消除。

**锁粗化（Lock Coarsening）**：如果一段代码里反复对同一个对象加锁解锁（比如循环内的 synchronized），JIT 会把这些细粒度的锁合并成一个更大范围的锁，减少加锁解锁的次数。

例如：循环里的 `synchronized(obj) { append(); }` 会被优化成锁整个循环。

###### 8. 什么是内存屏障？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
内存屏障（Memory Barrier）是 CPU 指令，用于**控制特定操作的执行顺序**，防止 CPU 和编译器对内存读写操作进行重排序，保证内存操作的可见性和顺序性。

主要分四种：
- **LoadLoad 屏障**：屏障前的读操作先于屏障后的读操作完成
- **StoreStore 屏障**：屏障前的写操作先于屏障后的写操作完成（并刷新到主存）
- **LoadStore 屏障**：屏障前的读操作先于屏障后的写操作完成
- **StoreLoad 屏障**：屏障前的写操作先于屏障后的读操作完成，且保证写操作对所有 CPU 可见。开销最大，也最安全。

Java 的 `volatile` 底层就是通过插入内存屏障来实现有序性和可见性的。

###### 9. 什么是伪共享？如何避免？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
**伪共享（False Sharing）**：CPU 缓存以**缓存行（Cache Line，通常64字节）**为单位加载数据。如果两个无关的变量恰好在同一个缓存行里，一个线程修改其中一个变量，会导致整个缓存行失效，另一个线程的缓存也失效了，即使它访问的是另一个变量。两个线程互相使对方的缓存失效，性能急剧下降，这就是伪共享。

**如何避免**：
- **手动填充**：在变量前后加 padding 字段，把两个热点变量分离到不同缓存行
- **@Contended 注解**（JDK 8+）：JVM 会自动在 `@Contended` 注解的字段或类的前后插入128字节的 padding，把它和其他变量隔离到不同缓存行。需要开启 `-XX:-RestrictContended`。`LongAdder`、`ConcurrentHashMap` 内部的 Cell 数组就用了 @Contended。

###### 10. 什么是对象的内存布局（对象头、实例数据、对齐填充）？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
HotSpot 里每个对象内存布局分三部分：

**对象头（12字节，开启指针压缩）**：
- **Mark Word（8字节）**：存对象的运行时状态，根据锁状态存不同内容：无锁时存哈希码和 GC 分代年龄；偏向锁时存线程 ID；轻量级锁时存栈帧里锁记录的指针；重量级锁时存 Monitor 的指针。
- **类型指针（4字节）**：指向方法区里该类的元数据。开启指针压缩（-XX:+UseCompressedClassPointers）是4字节，否则8字节。
- **数组长度（4字节，仅数组对象有）**

**实例数据**：对象真正存储的字段，按 JVM 规定的顺序排列（long/double → int/float → short/char → byte/boolean → 引用，父类字段在前）。

**对齐填充**：HotSpot 要求对象大小是8字节的整数倍，不足的用填充字节补齐，没有实际含义，纯占位。

###### 11. 什么是指针压缩？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
**指针压缩（Compressed Oops）**：在64位 JVM 上，正常情况对象引用是8字节，但大多数应用的堆不超过32GB，完全不需要64位的完整寻址空间。JVM 利用对象8字节对齐的特性（末3位始终为0），把32位压缩指针左移3位就能覆盖 32GB 的寻址空间（2^32 × 8 = 32GB）。

开启 `-XX:+UseCompressedOops`（JDK 6u23 后默认开启，堆 <= 32GB 时有效）后，对象引用从8字节压缩到4字节，堆内存占用通常减少 20~30%。

**注意**：堆超过32GB时，压缩指针无法覆盖所有地址，JVM 会自动关闭压缩，引用变回8字节。所以有时候堆设31GB比设33GB反而更省内存（因为33GB会关闭指针压缩）。

###### 12. 什么是即时编译（JIT）？C1 和 C2 编译器的区别？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
**JIT（Just-In-Time Compilation，即时编译）**：Java 程序启动时先用解释器执行字节码，JVM 监控代码执行情况，发现某些方法或循环被频繁执行（热点代码），就用 JIT 编译器把它们编译成本地机器码缓存起来，后续直接运行机器码，速度大幅提升。

**C1 编译器（Client Compiler）**：编译速度快，优化程度较低，适合桌面应用这种启动速度重要的场景。生成的机器码质量一般。

**C2 编译器（Server Compiler）**：编译速度慢，优化非常激进（方法内联、逃逸分析、循环展开等），生成的机器码质量高，执行效率高。适合长期运行的服务端应用。

JDK 7 引入了分层编译（Tiered Compilation），同时用两个编译器，兼顾了启动速度和长期运行效率。

###### 13. 什么是分层编译？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
分层编译（Tiered Compilation）把代码的执行分为5个层次，从快到高质量：

- **Level 0**：解释执行，同时收集 profiling 数据（方法调用次数、分支判断统计等）
- **Level 1**：C1 编译，不收集 profiling，执行简单代码
- **Level 2**：C1 编译 + 少量 profiling
- **Level 3**：C1 编译 + 完整 profiling（最常见的 C1 层次）
- **Level 4**：C2 编译，基于 profiling 数据做激进优化，生成最高质量机器码

代码从 Level 0 开始，随着执行次数增加逐步晋升到更高层次。C1 编译让代码快速启动，C2 编译让代码长期运行性能最优，两者各司其职。JDK 8 开始默认开启分层编译。

###### 14. 什么是内联优化？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
内联优化（Method Inlining）是 JIT 最重要的优化手段之一：把**被调用方法的代码直接嵌入到调用处**，省掉方法调用的开销（栈帧创建、参数复制、返回值处理等），同时让编译器能跨方法边界做更多优化。

例如：
```java
// 原始代码
int result = Math.abs(-5);

// 内联后（编译器视角）
int result = (-5 < 0) ? 5 : -5;  // Math.abs 的代码被内嵌进来
```

内联后，`-5 < 0` 是编译期可以确定的条件（常量折叠），整个表达式可以直接优化为 `result = 5`。

###### 15. 什么是方法内联的条件？

[[../../../20_JavaKnowledge/04_JVM/10、字节码与执行引擎#一、Java 字节码概述|📖]]
JIT 并不是所有方法都会内联，需要满足几个条件：

**可以内联的方法**：
- **热点方法**：调用次数超过阈值（默认10000次，通过 `-XX:CompileThreshold` 控制）
- **方法体足够小**：字节码长度不超过 `-XX:MaxInlineSize`（默认35字节）或 `-XX:InlineSmallCode`（默认1000字节的已编译代码）
- **非虚方法**：static 方法、private 方法、final 方法可以直接内联；虚方法需要通过类型分析（CHA，类层次分析）确认只有一个实现才能内联

**阻止内联的情况**：
- 方法体太大（循环、复杂逻辑）
- 虚方法有多个实现类（多态调用，无法确定要调用哪个）
- `synchronized` 方法（同步方法内联代价高）
- 方法调用频率不够（没有到达热点阈值）

可以用 `-XX:+PrintInlining` 参数让 JIT 打印内联日志，看哪些方法被内联了，哪些没有及原因。
