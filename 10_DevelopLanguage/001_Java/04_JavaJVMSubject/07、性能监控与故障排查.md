###### 1. 什么是 jps 命令？如何使用？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
`jps`（JVM Process Status Tool）用于**列出当前机器上所有 Java 进程的 PID 和主类名**，是排查问题的第一步。

常用选项：
- `jps`：列出 PID 和主类名
- `jps -v`：显示启动时的 JVM 参数（看 -Xms、-Xmx 等配置）
- `jps -l`：显示完整的主类全限定名（方便区分相同进程名的不同应用）
- `jps -m`：显示传给 main 方法的参数

使用示例：
```bash
$ jps -l
12345 org.springframework.boot.loader.JarLauncher
23456 sun.tools.jps.Jps
```

###### 2. 什么是 jstat 命令？常用参数有哪些？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#二、JFR（Java Flight Recorder）|📖]]
`jstat`（JVM Statistics Monitoring Tool）用于**实时监控 JVM 的内存和 GC 统计信息**，线上排查 GC 问题的利器。

最常用的用法：
```bash
jstat -gcutil <pid> <interval_ms> <count>
```
- `-gcutil <pid> 1000`：每1秒输出一次各内存区使用率（%）和 GC 次数/累计时间

输出字段含义：
- **S0/S1**：两个 Survivor 区的使用率
- **E**：Eden 区使用率
- **O**：Old（老年代）使用率，持续增长不降说明有内存泄漏
- **M**：Metaspace（元空间）使用率
- **YGC/YGCT**：Young GC 次数/累计时间
- **FGC/FGCT**：Full GC 次数/累计时间，FGC 频繁是大警报

其他常用选项：
- `-gc <pid>`：输出各区的实际大小（字节）而不是百分比
- `-gccapacity <pid>`：各区的容量上下限

###### 3. 什么是 jmap 命令？如何生成堆转储文件？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#三、Arthas 深度使用|📖]]
`jmap`（JVM Memory Map）用于**查看堆内存使用情况或生成 Heap Dump 文件**。

**查看堆信息**：
```bash
jmap -heap <pid>    # 打印堆的配置、使用情况、GC 算法
```

**生成 Heap Dump 文件**（OOM 排查必用）：
```bash
jmap -dump:format=b,file=/tmp/heap.hprof <pid>
# live 参数只 dump 存活的对象，文件更小
jmap -dump:live,format=b,file=/tmp/heap-live.hprof <pid>
```

**注意**：`jmap -dump` 在执行时会短暂暂停 JVM（STW），生产环境要在低峰期执行或用其他方式（提前配 -XX:+HeapDumpOnOutOfMemoryError）。

dump 文件用 **MAT（Eclipse Memory Analyzer）** 或 **VisualVM** 打开分析。

###### 4. 什么是 jstack 命令？如何分析线程堆栈？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#五、故障排查标准流程|📖]]
`jstack`（JVM Stack Trace）用于**打印 JVM 进程的所有线程快照**，排查 CPU 100% 和死锁最常用。

```bash
jstack <pid>        # 打印所有线程栈
jstack -l <pid>     # 包含锁信息
```

**分析 CPU 100% 问题**：
1. `top` 找到 CPU 最高的 Java 进程 PID
2. `top -H -p <pid>` 找到该进程内 CPU 最高的线程 TID
3. 把 TID 转成16进制：`printf "%x" <tid>` → 得到 `0x1234`
4. `jstack <pid> | grep -A 50 "0x1234"` 找到对应线程的栈帧，定位到热点代码

**分析死锁**：`jstack -l <pid>` 输出末尾会有 "Found 1 deadlock" 字样，下面直接列出死锁的两个线程和它们持有/等待的锁。

###### 5. 什么是 jinfo 命令？

[[../../../20_JavaKnowledge/04_JVM/13、内存泄漏排查实战#二、常见内存泄漏场景|📖]]
`jinfo`（JVM Configuration Info）用于**查看和动态修改 JVM 的配置参数**。

```bash
jinfo <pid>                    # 打印所有 JVM 参数和系统属性
jinfo -flag MaxHeapSize <pid>  # 查看某个参数的值
jinfo -flag +PrintGCDetails <pid>  # 动态开启某个参数（部分参数支持热更新）
```

常用场景：不重启应用动态开启 GC 日志（仅支持 Manageable 类型的参数）。

###### 6. 如何使用 VisualVM 进行性能分析？

[[../../../20_JavaKnowledge/04_JVM/13、内存泄漏排查实战#五、内存泄漏排查完整流程|📖]]
VisualVM 是 JDK 自带的图形化监控工具（JDK 9 后独立发布），功能相当全面：

**连接方式**：
- 本地：直接在 Applications 面板选择进程
- 远程：需要 JVM 开启 JMX 或 jstatd

**主要功能**：
- **Overview**：查看 JVM 参数、系统属性
- **Monitor 标签**：实时监控 CPU 使用率、堆内存使用曲线、GC 活动、线程数量
- **Threads 标签**：查看所有线程状态，检测死锁（"Detect Deadlock" 按钮）
- **Profiler 标签**：方法级别 CPU 和内存 Profiling，找出热点方法和内存分配大户
- **Heap Dump**：生成并分析 Heap Dump 文件，找内存泄漏

**使用步骤（内存泄漏排查）**：
1. 在 Monitor 标签观察堆内存，如果 Full GC 后堆仍然居高不下说明有泄漏
2. 点击 "Heap Dump" 生成快照
3. 在 Heap Dump 里看 Classes 按大小排序，找到占内存最多的类
4. 查看实例列表，找具体的对象和引用链

###### 7. 如何使用 JConsole 监控 JVM？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#五、故障排查标准流程|📖]]
JConsole 是 JDK 自带的更轻量的 JMX 监控工具（`jconsole` 命令启动）：

- **连接**：选择本地进程或输入远程 JMX 地址
- **Memory 标签**：查看各内存区（堆、元空间等）的使用曲线和 GC 情况，可以手动触发 GC
- **Threads 标签**：查看线程数量曲线，检测死锁（"Detect Deadlock" 按钮）
- **Classes 标签**：监控已加载类的数量，持续增长可能是类加载泄漏
- **MBeans 标签**：访问所有 JMX MBean，可以查看和修改各种配置

VisualVM 功能更强，JConsole 更轻量，生产环境远程监控时 JConsole 够用。

###### 8. 什么是 MAT（Memory Analyzer Tool）？如何使用？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
MAT（Eclipse Memory Analyzer Tool）是分析 Java Heap Dump 文件的**专业工具**，排查内存泄漏神器。

**主要功能**：
- **Leak Suspects（泄漏嫌疑）报告**：自动分析 Heap Dump，标记出最可能的内存泄漏点，一键生成报告
- **Dominator Tree（支配树）**：按对象占用内存从大到小排列，直接找到占内存最多的"大头"
- **Histogram（直方图）**：按类名统计对象数量和占用内存，快速定位"哪个类的实例最多"
- **OQL（Object Query Language）**：SQL 语法查询 Heap 里的对象，非常灵活

**使用流程（排查内存泄漏）**：
1. 用 `jmap` 或 OOM 自动 dump 获得 `.hprof` 文件
2. MAT 打开文件，先看 "Leak Suspects" 报告
3. 查 Dominator Tree，找到内存占用前几名的对象
4. 选中可疑对象，右键 → "Path to GC Roots" → 找到引用链，看谁持有了这个对象导致无法回收

###### 9. 什么是 GCViewer？如何分析 GC 日志？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
GCViewer 是一个开源的 GC 日志分析工具（或直接用在线版 **gceasy.io**），把 GC 日志文件可视化，让你更容易看出问题。

**使用方式**：把 `-Xloggc` 指定的 GC 日志文件上传到 gceasy.io，或者用 GCViewer.jar 打开。

**关键指标看点**：
- **总 GC 时间 / 吞吐量**：吞吐量 = 用户线程运行时间 / 总时间，低于95%就要关注
- **停顿时间分布**：P99/P999 停顿时间，是否满足业务 SLA
- **GC 原因**：Allocation Failure（Eden 满了）是正常；Metadata GC Threshold（元空间不足）、System.gc() 是问题信号
- **堆使用趋势**：如果 Full GC 后老年代使用率仍然不断增长，基本就是内存泄漏了

###### 10. 如何诊断 CPU 100% 的问题？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
CPU 100% 是最常见的线上问题之一，排查步骤：

1. **定位进程**：`top` 命令找到 CPU 占用最高的 PID
2. **定位线程**：`top -H -p <pid>` 找到该进程里 CPU 最高的线程 TID
3. **转换线程 ID**：`printf "%x\n" <tid>` 把十进制 TID 转成十六进制（jstack 里是十六进制）
4. **抓线程栈**：`jstack <pid> > /tmp/stack.txt`
5. **搜索线程**：在 stack.txt 里搜索转换后的十六进制 TID，找到对应线程的栈
6. **分析代码**：看栈顶几帧，定位到具体的方法和代码行

**常见 CPU 100% 原因**：
- **死循环**：while(true) 或者循环条件写错
- **HashMap 死循环（JDK 1.7）**：多线程 put 触发扩容形成链表环，现在用 ConcurrentHashMap 解决
- **频繁 GC**：GC 线程本身也消耗 CPU，GC 后堆内存回收效果差说明有泄漏或内存设置太小
- **大量正则匹配**：复杂正则表达式的回溯可能导致 CPU 飙升
- **序列化/反序列化密集**：高频 JSON/XML 处理

###### 11. 如何诊断内存溢出问题？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
OOM 分几类，原因和解决方法不同：

**堆内存溢出（java.lang.OutOfMemoryError: Java heap space）**：
- 原因：内存泄漏（对象无法回收）、或者堆设置太小、或者一次性加载了过多数据
- 排查：获取 Heap Dump，用 MAT 分析，找 GC Roots 引用链

**元空间溢出（java.lang.OutOfMemoryError: Metaspace）**：
- 原因：加载了过多的类（动态代理、反射、Groovy 等），或 MaxMetaspaceSize 设置太小
- 解决：增大 `-XX:MaxMetaspaceSize`，或检查类加载是否失控

**栈溢出（java.lang.StackOverflowError）**：
- 原因：递归太深，或者方法调用链太长
- 解决：检查递归是否有正确的终止条件，适当增大 `-Xss`

**直接内存溢出（OutOfMemoryError: Direct buffer memory）**：
- 原因：NIO 的 DirectByteBuffer 用太多
- 解决：增大 `-XX:MaxDirectMemorySize`，或检查 DirectBuffer 是否及时释放

###### 12. 如何诊断死锁问题？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
**方法一：jstack 检测**
```bash
jstack -l <pid>
```
输出末尾会有 "Found X deadlock(s)" 明确提示，并列出死锁的线程及各自持有和等待的锁。

**方法二：JConsole / VisualVM 的 Detect Deadlock 功能**，点一下按钮就能自动检测。

**死锁发生的条件（必须同时满足）**：
1. 互斥（锁只能被一个线程持有）
2. 请求与保持（已持有锁的线程再请求其他锁）
3. 不可抢占（锁不能被强制释放）
4. 循环等待（线程A等B的锁，线程B等A的锁）

**预防死锁的实践**：
- 固定加锁顺序（所有线程按相同顺序获取锁）
- 使用 `tryLock(timeout)` 代替无限等待
- 缩小锁的粒度和持有时间

###### 13. 什么是 Arthas？常用命令有哪些？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
Arthas 是阿里开源的 Java **在线诊断工具**，无需重启 JVM，可以动态 trace 方法、查看类信息、监控性能指标，是生产环境排查问题的神器。

启动：`java -jar arthas-boot.jar`，选择要诊断的 Java 进程。

**最常用命令**：
```bash
dashboard    # 实时显示 JVM 状态（CPU、内存、线程、GC 概览）
thread       # 查看所有线程，thread -n 3 看 CPU 占用最高的3个线程
thread <id>  # 查看指定线程的栈
trace com.example.Service methodName  # trace 某个方法的调用链和耗时
watch com.example.Service methodName "{params, returnObj}" -x 2  # 观察方法入参和返回值
jad com.example.Service  # 反编译某个类（看实际运行的是哪个版本的代码）
classloader  # 查看类加载器层次结构
redefine /tmp/NewService.class  # 热替换某个类的字节码（谨慎使用）
heapdump /tmp/heap.hprof  # 生成 Heap Dump
```

###### 14. 如何使用 JFR（Java Flight Recorder）？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
JFR（Java Flight Recorder）是 JDK 11+ 免费提供（JDK 8 需要商业授权）的**低开销性能分析工具**，可以记录 JVM 运行期间的详细事件（GC、线程、IO、方法调用等），开销极低（< 1%）。

**开启方式**：
```bash
# 启动时开启
java -XX:StartFlightRecording=duration=120s,filename=/tmp/recording.jfr MyApp

# 运行中开启（用 jcmd）
jcmd <pid> JFR.start duration=60s name=profile filename=/tmp/recording.jfr
jcmd <pid> JFR.dump filename=/tmp/snapshot.jfr
jcmd <pid> JFR.stop
```

**分析工具**：用 **JDK Mission Control（JMC）** 打开 `.jfr` 文件，可以看到：
- 方法热点（Hotspots），哪些方法最耗 CPU
- 内存分配热点，哪些代码分配了最多对象
- GC 事件详情
- 线程状态变化、锁竞争情况
- I/O 操作延迟

###### 15. 如何进行 JVM 性能基准测试？

[[../../../20_JavaKnowledge/04_JVM/11、JVM监控与故障排查#一、JDK 内置命令行工具|📖]]
JVM 性能基准测试推荐用 **JMH（Java Microbenchmark Harness）**，这是 OpenJDK 官方的微基准测试框架，会正确处理 JVM 预热、JIT 优化等影响因素。

```java
@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.MICROSECONDS)
@State(Scope.Thread)
public class MyBenchmark {

    @Benchmark
    public void testMethod() {
        // 被测试的代码
    }
    
    // 基准方法，用来对比
    @Benchmark
    public void baseline() {
        // 什么都不做
    }
}
```

**注意事项**：
- 必须充分预热（`@Warmup`）让 JIT 编译稳定后再测量
- 使用足够多的迭代（`@Measurement`）减少随机误差
- 防止 JIT 把你的测试代码优化掉（用 `Blackhole` 消费结果）
- 在目标运行环境（同等 CPU、JVM 版本、参数）上测
