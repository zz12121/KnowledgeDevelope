###### 1. 什么是垃圾回收？

[[../../../20_JavaKnowledge/04_JVM/05、垃圾回收算法#一、对象的生死判断|📖]]
垃圾回收（GC，Garbage Collection）是 JVM 的**自动内存管理机制**，负责识别并回收程序中不再使用的对象所占用的堆内存，防止内存泄漏和 OOM，让开发者不需要像 C/C++ 那样手动管理内存。

###### 2. 如何判断对象是否可以被回收？

[[../../../20_JavaKnowledge/04_JVM/05、垃圾回收算法#一、对象的生死判断|📖]]
**引用计数法**：给每个对象加一个计数器，有引用就加1，引用失效就减1，计数为0就可以回收。简单高效，但有致命缺陷：**无法解决循环引用问题**（A 引用 B，B 引用 A，但两者都不被其他对象引用，计数永远不为0，永远回收不了）。Python 用了这个方法但同时有额外处理，Java 没有用。

**可达性分析算法（Java 采用的方案）**：以一组称为 **GC Roots** 的根对象为起点，沿着引用关系往下搜索，能搜索到的对象就是存活的，搜索不到的就是"垃圾"（不可达对象），可以被回收。这个方法能正确处理循环引用。

###### 5. 什么是 GC Roots？哪些对象可以作为 GC Roots？

[[../../../20_JavaKnowledge/04_JVM/06、垃圾收集器全览#一、收集器全览|📖]]
**GC Roots** 是可达性分析的起点，是"肯定活着的对象"，从它们开始遍历引用链：

- **虚拟机栈（栈帧的本地变量表）中引用的对象**：方法里正在用的局部变量指向的对象
- **方法区中类静态属性引用的对象**：类的 static 字段指向的对象
- **方法区中常量引用的对象**：常量池里的字符串常量等
- **本地方法栈中 JNI 引用的对象**：Native 方法里引用的 Java 对象
- **被 synchronized 持有的对象**：正在被同步锁持有的对象
- **JVM 内部引用**：Class 对象、异常对象、类加载器等系统级引用

###### 7. 什么是强引用、软引用、弱引用、虚引用？

[[../../../20_JavaKnowledge/04_JVM/06、垃圾收集器全览#三、ParNew / Parallel|📖]]
Java 有四种引用强度，GC 对它们的态度各不相同：

**强引用（Strong Reference）**：最普通的引用，`Object obj = new Object()` 这就是强引用。只要强引用存在，GC **永远不会回收**这个对象，宁愿 OOM 也不回收。

**软引用（SoftReference）**：用 `SoftReference<T>` 包装，告诉 JVM 这个对象"可以回收但不着急"。只有**内存不足即将 OOM 时**，GC 才会回收软引用对象。适合做内存敏感的缓存，内存充足时一直能用，内存紧张时自动清理。

**弱引用（WeakReference）**：用 `WeakReference<T>` 包装，比软引用更弱。**下一次 GC 运行时，不管内存是否充足，都会回收弱引用对象**。`WeakHashMap` 的 Key 用的就是弱引用，ThreadLocal 的 Entry 也是。适合生命周期较短的辅助对象。

**虚引用（PhantomReference）**：最弱，**不能通过虚引用获取对象本身**（get() 永远返回 null）。它的唯一意义是：当对象被 GC 回收时，虚引用会被放入 `ReferenceQueue`，程序通过监听这个队列得到通知，做一些清理工作。适合管理堆外内存（直接内存）的回收时机。

###### 8. finalize() 方法的作用是什么？

[[../../../20_JavaKnowledge/04_JVM/06、垃圾收集器全览#四、CMS 收集器|📖]]
`finalize()` 是对象被 GC 回收前的**最后一次自救机会**。

当一个对象不可达时，GC 先判断它是否需要执行 `finalize()`（没覆盖过该方法，或已经调用过了，就不需要）。需要执行的话，对象会被放进一个低优先级队列，由 JVM 的 Finalizer 线程去执行 `finalize()`。在这个方法里，对象可以通过把自己赋给某个强引用来"自救"，成功的话第二次标记时会把它从"即将回收"的集合里移除。

但是 `finalize()` **不推荐使用**：
- 执行时机不确定，不知道什么时候 Finalizer 线程才会跑到
- 只能执行一次，对象自救成功后再次不可达就无法再救了
- 运行代价高昂，拖慢 GC 性能
- 替代方案：使用 `try-finally` 或 `try-with-resources` 进行资源清理，JDK 9 后推荐用 `Cleaner` 替代

###### 9. 什么是垃圾回收算法？

[[../../../20_JavaKnowledge/04_JVM/07、G1收集器深度解析#一、G1 概述|📖]]
**标记-清除（Mark-Sweep）**：分两步，先标记所有存活对象，再清除未被标记的对象。简单，但会产生大量**内存碎片**，大对象找不到连续空间时麻烦。适合老年代（CMS 用了这个）。

**标记-复制（Copying）**：把内存分两半，只用一半，存活对象复制到另一半后清理原来那半。**无碎片，效率高**，但浪费一半内存。对象存活率低的新生代非常适合用这个（Eden + Survivor 就是这个思路）。

**标记-整理（Mark-Compact）**：先标记，然后让所有存活对象往一端移动，清理边界以外的内存。**无碎片**，但移动对象要更新所有引用地址，**效率比标记-清除低**。适合对象存活率高的老年代。

**分代收集（Generational）**：基于"绝大多数对象朝生夕死"的规律，把堆分成新生代（用复制算法）和老年代（用标记-整理或标记-清除），根据各自特点用不同算法。这是**现代商用 JVM 的主流策略**。

###### 13. 分代收集理论是什么？

[[../../../20_JavaKnowledge/04_JVM/05、垃圾回收算法#一、对象的生死判断|📖]]
分代收集建立在两个假说上：
1. **弱分代假说**：绝大多数对象都是朝生夕死的（生命周期极短）
2. **强分代假说**：熬过越多次 GC 的对象，就越难以消亡

基于此，把堆分成新生代和老年代，新生代用复制算法（对象大多短命，存活少，复制代价低），老年代用标记-整理（对象存活率高，复制代价大）。

**Minor GC**：只回收新生代，非常频繁，速度快（100ms 级别）。

**Major GC / Full GC**：Full GC 清理整个堆（包括新生代、老年代、元空间），速度慢（秒级甚至分钟级），应尽量避免频繁发生。

**STW（Stop-The-World）**：GC 时为了保证对象引用关系不变化，必须暂停所有用户线程。所有收集器都无法完全消除 STW，优化目标是把 STW 时间缩到最短。

###### 16. 有哪些垃圾收集器？

[[../../../20_JavaKnowledge/04_JVM/05、垃圾回收算法#一、对象的生死判断|📖]]
**Serial GC**：单线程，GC 时完全 STW，JVM 版的"一个人扫地"。适合客户端模式或单核 CPU 的小内存应用。

**ParNew GC**：Serial 的多线程版，多个线程同时 GC，但仍然 STW。常和 CMS 搭配处理新生代。

**Parallel Scavenge GC**：多线程，吞吐量优先（高吞吐 = 用户线程运行时间/总时间）。JDK 8 默认收集器，适合批处理、后台计算这类注重吞吐的场景，对响应时间不敏感。

**CMS（Concurrent Mark Sweep）**：老年代收集器，目标是**低停顿**，大部分工作和用户线程并发进行，只有初始标记和重新标记需要短暂 STW。适合重视响应速度的 B/S 系统。缺点是产生内存碎片、对 CPU 敏感、有浮动垃圾。

**G1（Garbage First）**：JDK 9 默认收集器，把堆划分成等大的 Region，不再固定新老代的物理位置，每次 GC 优先回收"垃圾最多、收益最大"的 Region。**可预测停顿时间**，在吞吐量和低延迟之间取得好的平衡。适合大内存服务端应用。

垃圾收集流程：

- **新生代（复制算法）**：STW → 标记 Eden+Survivor From 的存活对象 → 复制到 Survivor To → 清理 Eden 和 Survivor From → 恢复线程
- **老年代 CMS（标记-清除）**：初始标记（STW）→ 并发标记 → 重新标记（STW）→ 并发清除
- **G1**：初始标记（STW）→ 并发标记 → 最终标记（STW）→ 筛选回收（STW，把选中 Region 的存活对象复制到空 Region）

###### 25. G1 的工作原理是什么？

[[../../../20_JavaKnowledge/04_JVM/05、垃圾回收算法#一、对象的生死判断|📖]]
G1 的核心设计是把堆划分成大量（默认2048个）等大的 **Region**（1~32MB），每个 Region 根据需要扮演 Eden、Survivor、Old 或 Humongous（大对象）的角色，不固定。

工作流程：
1. **初始标记（STW，很短）**：标记 GC Roots 直接关联的对象
2. **并发标记**：从 GC Roots 出发做可达性分析，与用户线程并发，耗时最长但不停顿
3. **最终标记（STW，很短）**：处理并发标记期间因用户线程运行导致的引用变化（用 SATB 记录）
4. **筛选回收（STW）**：计算每个 Region 的回收价值（垃圾量/回收耗时），按价值从高到低排序，在满足目标停顿时间（`-XX:MaxGCPauseMillis`）的前提下选择尽可能多的 Region 回收，把存活对象复制到空 Region，然后清理旧 Region

G1 的关键特点：**可预测停顿时间**（通过控制每次回收的 Region 数量来控制停顿时间）、**整体无内存碎片**（Region 之间复制后是整块连续内存）。
