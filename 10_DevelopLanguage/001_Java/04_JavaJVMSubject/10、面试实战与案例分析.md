###### 1. 线上应用频繁 Full GC，如何排查？

[[../../../20_JavaKnowledge/04_JVM/13、内存泄漏排查实战#一、Java 内存泄漏的本质|📖]]
频繁 Full GC 是高频生产事故，排查思路：

**第一步：确认问题**
```bash
jstat -gcutil <pid> 1000  # 每秒看一次 GC 情况
```
观察 FGC 次数和 FGCT 时间，判断频繁程度（每分钟超过1次就要排查）。

**第二步：看 GC 日志分析原因**，Full GC 有几种触发原因：
- **Allocation Failure（分配失败）**：老年代满了，通常是内存泄漏或堆太小
- **System.gc()调用**：代码里或第三方库里显式调用了 `System.gc()`，用 `-XX:+DisableExplicitGC` 禁掉
- **Metadata GC Threshold**：元空间不足，调大 `-XX:MaxMetaspaceSize`
- **Concurrent Mode Failure（CMS 特有）**：CMS 并发清理时老年代满了，需要提前触发 CMS

**第三步：分析老年代增长趋势**，如果老年代使用率在每次 Full GC 后仍然不断增长，基本是内存泄漏。用 `jmap` dump heap，MAT 分析，找 GC Roots 引用链。

**第四步：调整 GC 参数**（在排除泄漏后）：增大堆大小、调整新生代大小、换 G1 收集器。

###### 2. 应用启动后立即 OOM，可能的原因？

[[../../../20_JavaKnowledge/04_JVM/13、内存泄漏排查实战#四、OOM 类型与排查对策|📖]]
启动即 OOM，说明在初始化阶段就耗尽了内存，常见原因：

- **堆设置太小**：`-Xmx` 太小，Spring Boot 应用加载大量 Bean、初始化缓存就能撑满
- **元空间不足**：加载了大量框架类（Spring、Hibernate 等）耗尽元空间，报 `OutOfMemoryError: Metaspace`
- **直接内存不足**：报 `OutOfMemoryError: Direct buffer memory`，NIO 框架初始化时申请了大量直接内存
- **栈溢出**：启动线程太多，线程栈耗尽了内存（`-Xss` 太大且线程数多）
- **一次性加载大数据**：初始化时加载了大量数据到内存（比如预热缓存把数据库全表数据读到 Map 里）

**快速确认**：看 OOM 的详细信息（`java.lang.OutOfMemoryError: xxx`），不同的后缀对应不同的内存区域。

###### 3. 如何优化应用的启动时间？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#四、GC 调优|📖]]
启动慢的几个维度：

**减少 Bean 初始化**：Spring Boot 开启懒加载 `spring.main.lazy-initialization=true`，非关键 Bean 延迟到第一次使用时才初始化。

**使用 CDS / AppCDS**：把 JDK 核心类和应用类打进共享归档，跳过重复的类加载步骤，通常减少 20~40% 启动时间。

**减少 classpath 扫描**：避免全包扫描，明确指定需要扫描的包，减少不必要的类加载。

**JIT 预热设置**：`-XX:TieredStopAtLevel=1` 只用 C1 编译，牺牲运行时峰值性能换取更快的启动速度（适合一次性任务）。

**容器场景**：考虑 GraalVM Native Image，毫秒级启动，但需要处理动态特性的配置。

###### 4. 如何减少 GC 停顿时间？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
**换低延迟 GC**：G1（目标停顿 100ms 级别）→ ZGC/Shenandoah（目标停顿 < 1ms）。收集器选择是对停顿时间影响最大的单一因素。

**合理设置停顿目标**：G1 的 `-XX:MaxGCPauseMillis=200`，不要设太小（JVM 会频繁 GC 导致吞吐量下降），根据业务 SLA 设置合理值。

**减少需要 GC 的对象**：复用对象（对象池）、避免大量短生命周期对象、减少大对象分配（大数组、大 Map）。

**增大堆大小**：堆更大 → GC 触发频率更低 → 总停顿时间更少，但单次 Full GC 的停顿可能更长。

**减少对象晋升**：适当增大新生代，让更多短生命周期对象在新生代消亡，减少晋升到老年代的量，降低 Full GC 频率。

###### 5. 大对象如何优化？

[[../../../20_JavaKnowledge/04_JVM/07、G1收集器深度解析#三、G1 回收流程|📖]]
**什么是大对象**：需要大量连续内存的对象，比如大数组、存储大量数据的 Map/List。

**问题**：大对象直接进老年代（或在 Eden 无法找到连续空间时触发 GC），频繁创建大对象会导致频繁 Full GC。

**优化方案**：
- **拆分大对象**：把一个大 Map 拆成多个小 Map（分片），或分批处理大数据而不是一次性加载
- **对象复用**：大对象放到对象池里复用，避免频繁创建销毁（比如大的 byte[]）
- **流式处理**：用 Stream/Iterator 逐条处理数据，而不是一次性加载到内存
- **调整阈值**：`-XX:PretenureSizeThreshold` 设合理的大对象阈值（只对 Serial/ParNew 有效）

###### 6. 如何处理元空间溢出问题？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
**症状**：`java.lang.OutOfMemoryError: Metaspace`

**原因**：
- 加载了过多的类（框架反射、动态代理 CGLIB/JDK 代理、Groovy/Kotlin 动态脚本）
- 应用频繁重部署但类加载器没有被回收（Web 应用在 Tomcat 里更新但老的 ClassLoader 未 GC）
- `MaxMetaspaceSize` 设置太小

**排查步骤**：
```bash
jstat -gcmetacapacity <pid>  # 查看元空间容量和使用情况
# 或
jcmd <pid> VM.native_memory  # 查看本地内存使用（需要开启 NativeMemoryTracking）
```

**解决**：
- 调大 `-XX:MaxMetaspaceSize`（比如从默认的无限制改为明确的 512m）
- 检查是否有类加载泄漏（jmap + MAT 查看 ClassLoader 数量）
- 减少不必要的动态代理和类生成

###### 7. 如何处理直接内存溢出问题？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
**症状**：`java.lang.OutOfMemoryError: Direct buffer memory`

**原因**：`ByteBuffer.allocateDirect()` 申请的堆外内存超过了 `-XX:MaxDirectMemorySize` 设置的上限（默认和 -Xmx 相同）。常见于 NIO 框架（Netty）、文件操作、gRPC 等。

**排查**：
```bash
# 查看直接内存使用量
java.lang.management.BufferPoolMXBean
# 或者 Arthas
memory | grep direct
```

**解决**：
- 调大 `-XX:MaxDirectMemorySize`
- 确保 DirectBuffer 及时释放（用 `((DirectBuffer)buffer).cleaner().clean()`，或等待 GC 触发 Phantom Reference 回调）
- Netty 应用注意 ByteBuf 的引用计数，避免 ByteBuf 泄漏

###### 8. 线程池参数与 JVM 堆大小的关系？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
每创建一个线程，JVM 需要分配一块独立的**线程栈空间**（大小由 `-Xss` 控制，默认512KB~1MB，Linux 64位 HotSpot 默认1MB）。

关系计算：**最大线程数 ≈ (OS 总内存 - JVM 堆 - 元空间 - 其他内存) / 线程栈大小**

例如：机器8GB内存，堆设6GB，元空间512MB，其他1GB，每个线程栈1MB → 最多约512个线程。

**实践建议**：
- 不要盲目增大线程池，线程本身就是内存消耗
- CPU 密集型任务：线程数 = CPU 核数 + 1
- IO 密集型任务：线程数 = CPU 核数 × (1 + 等待时间/计算时间)，通常是 CPU 核数的 2~10 倍
- 堆不够用时，可以适当减小 `-Xss`（但不要低于256KB，容易 StackOverflow）

###### 9. 如何进行 JVM 调优的一般步骤？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
JVM 调优不是乱调参数，要有系统性方法：

1. **明确目标**：是降低延迟（P99停顿时间）还是提升吞吐量？目标不同，优化方向不同。
2. **建立基线**：在调优前记录当前指标（GC 停顿时间、吞吐量、内存使用等），有对比才知道有没有改善。
3. **分析 GC 日志**：收集并分析 GC 日志，用 GCViewer/gceasy.io 可视化，找出瓶颈（Full GC 频率、停顿时间、内存增长趋势）。
4. **一次只改一个参数**：每次调整一个参数，对比前后效果，才能知道是哪个参数起了作用。
5. **压测验证**：在模拟生产环境的压力测试下验证效果，而不是仅凭理论推断。
6. **监控上线**：调优后部署到生产环境，持续监控确保稳定，没有出现新问题。

###### 10. 什么是 GC 调优的黄金法则？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
几条实践中验证有效的原则：

**1. 先排除内存泄漏**：调参之前先确认没有内存泄漏。有泄漏的话无论怎么调参都是治标不治本，只是拖延崩溃时间。

**2. 减少对象分配比优化 GC 更有效**：最好的 GC 是不发生 GC。优化代码减少不必要的对象创建，比调 GC 参数收益更大。

**3. 堆大小不是越大越好**：堆越大 → Full GC 时间越长 → STW 停顿更久。要根据应用的实际对象存活量合理设置，通常设为存活对象占用内存的 3~4 倍比较合适。

**4. 不要过度调优**：系统满足业务需求就够了，过度追求极致参数反而可能引入不稳定性。

###### 11. 如何评估 GC 的健康状态？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
GC 健康的参考指标：

- **吞吐量 > 95%**：用户线程运行时间占总时间的比例，低于95%说明 GC 太频繁
- **Minor GC 频率**：对于中等流量的 Web 服务，每隔几秒到几十秒一次是正常的
- **Full GC 频率**：生产环境里 Full GC 应该极少发生（每天几次以内），频繁 Full GC 一定是有问题
- **GC 停顿时间**：看 P99/P999，是否满足业务的延迟 SLA
- **老年代使用率**：Full GC 后老年代应该降到一个稳定水位，如果持续增长说明有泄漏
- **GC 后老年代恢复率**：老年代能回收多少，回收率低说明对象生命周期设计不合理

###### 12. 生产环境中推荐的 JVM 参数配置？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
一个中等规模 Java 服务的参考配置（4核8GB容器）：

```bash
# 堆内存
-Xms2g -Xmx4g
# 或容器感知方式
-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0

# 元空间
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=512m

# 垃圾收集器（G1，JDK 9+ 默认）
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:G1HeapRegionSize=16m

# GC 日志（JDK 11+）
-Xlog:gc*:file=/logs/gc.log:time,uptime:filecount=10,filesize=50m

# 故障时堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/logs/heap.hprof

# 性能优化
-XX:+UseCompressedOops
-XX:+UseCompressedClassPointers
```

###### 13. 如何处理内存泄漏与内存溢出的区别？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
**内存泄漏（Memory Leak）**：对象不再被程序使用，但 GC 无法回收（因为仍然有引用指向它）。表现为内存使用量随时间持续增长，最终导致 OOM。根本原因是代码问题。

**内存溢出（Memory Overflow / OOM）**：需要分配的内存超过了 JVM 或操作系统能提供的内存上限。可能是泄漏的最终结果，也可能单纯是堆设置太小。

**区别要点**：
- 内存泄漏一定会最终导致 OOM，但 OOM 不一定是内存泄漏（可能就是数据量太大）
- 内存泄漏需要修代码，OOM 可能只需要调大堆

**如何区分**：观察老年代使用率趋势，Full GC 后如果老年代占用持续增长不降就是泄漏；如果 Full GC 后能恢复到稳定水位，只是偶发尖峰撑爆，可能只是需要加内存或减少峰值内存消耗。

###### 14. JVM 调优的常见误区有哪些？

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
**误区一：堆越大越好** — 堆大了 Full GC 停顿更长，也不是越大越好。要根据实际存活对象量合理设置。

**误区二：一遇到 GC 问题就调参数** — 应该先排查代码层面的问题（内存泄漏、大对象、不必要的对象创建），调参只是锦上添花，治不了根本。

**误区三：调优在开发环境验证就够了** — 开发环境流量、数据量、并发度都和生产不同，结果未必能在生产上复现，一定要在模拟生产的压测环境验证。

**误区四：一次调多个参数** — 看不出哪个参数有效，要一次只改一个，对比前后效果。

**误区五：追求极致的停顿时间而忽视吞吐量** — 停顿时间越短往往吞吐量越低（GC 更频繁），要根据业务优先级选择合适的平衡点。

**误区六：线上直接调参不测试** — JVM 参数调整必须在非生产环境充分验证，错误的参数可能导致 OOM 或服务不稳定。

###### 15. 描述一次你实际进行 JVM 调优的经历

[[../../../20_JavaKnowledge/04_JVM/09、JVM调优实战#五、实战案例|📖]]
（面试中的开放题，按以下结构回答，结合自己的实际经历）

**标准回答框架**：

**背景**：某 XX 服务每天业务高峰期（上午10点）会出现接口响应时间突然飙升，P99 从正常的20ms上升到2000ms，持续约30秒后恢复，日志里能看到有 Full GC 发生。

**排查过程**：
1. 收集 GC 日志，发现高峰期 Full GC 频率约每10分钟一次，每次停顿 800ms~1.5s
2. 用 `jstat -gcutil` 观察，发现 Full GC 前老年代使用率已经到92%以上
3. Full GC 后老年代使用率降到65%，说明没有明显内存泄漏，但降下去的幅度不够，下次 GC 很快再满
4. 分析对象晋升速率，发现有大量生命周期在5~15秒的"中龄"对象在新生代存活太久晋升到老年代（是 HTTP 连接池 + 缓存导致的）

**解决方案**：
1. 适当增大新生代（从原来堆的1/4改到1/3），让中龄对象有更多机会在新生代消亡
2. 把收集器从 CMS 换成 G1，设置 `-XX:MaxGCPauseMillis=200`
3. 把 `MaxTenuringThreshold` 从默认15调低到10，让确实长期存活的对象更快进老年代，减少 Survivor 区的压力

**效果**：Full GC 频率从10分钟一次降到约1小时一次，P99 停顿从2000ms降到200ms以内，业务高峰期接口响应时间稳定在正常水平。
