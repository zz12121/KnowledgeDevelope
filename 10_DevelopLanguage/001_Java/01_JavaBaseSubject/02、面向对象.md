###### 1. 面向对象和面向过程的区别？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#一、封装（Encapsulation）|📖]]
这两个是两种完全不同的编程思想。

**面向过程（POP）** 以步骤为中心，把问题拆成一系列步骤，用函数来实现，按顺序调用。数据和功能是分开的，C语言就是典型代表。它的优势是性能通常更好，流程直接，适合嵌入式这种性能敏感的场景。

**面向对象（OOP）** 以对象为中心，分析问题时先想"有哪些实体参与进来、这些实体有什么属性和行为"，然后把数据和行为封装在对象里。Java、C++是典型代表。它的优势是代码易维护、易复用、易扩展，更适合复杂的大型系统。

一句话区别：面向过程是"第一步做什么，第二步做什么"；面向对象是"谁来做，怎么配合"。

性能上，面向过程通常快一些，因为没有对象创建销毁的开销。但在现代硬件条件下，这个差距在大多数业务场景里可以忽略不计。

###### 2. 说说面向对象编程三大特性？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#一、封装（Encapsulation）|📖]]
**封装**：把数据和操作数据的方法绑定在一起，对外隐藏实现细节，只暴露有限的接口。通俗说就是"把内部的东西藏起来，只开一个口子给外面用"。好处是提高了安全性，内部改了不影响外部调用。

**继承**：子类可以复用父类的属性和方法，在此基础上扩展新功能。表达的是"is-a"的关系，比如狗是动物。好处是代码复用，减少重复。Java是单继承，一个类只能有一个直接父类。

**多态**：同一个方法调用，根据实际对象类型的不同，表现出不同的行为。比如父类引用指向子类对象，调用方法时执行的是子类重写的方法。这种机制叫做**动态绑定**或者**运行时多态**。好处是提高了代码的扩展性和灵活性。

###### 3. 什么是多态机制？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#三、多态（Polymorphism）|📖]]
多态的核心是：**用父类（或接口）引用指向子类对象，调用方法时执行子类的具体实现**。

这种行为叫**动态绑定**，JVM在运行时根据对象的实际类型决定调用哪个方法，而不是编译期就确定了。

多态要成立需要三个条件：继承（或实现接口）、方法重写、父类引用指向子类对象。

多态的价值在于：写代码时面向父类或接口，不用关心具体是哪个子类；运行时自动调到正确的实现。这让代码扩展起来特别方便，加新子类不需要改调用方代码。

###### 4. 说说重载和重写的区别？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#四、重载（Overload）vs 重写（Override）|📖]]
这两个概念都跟"多态"有关，但是完全不同的东西：

**重载（Overload）** 发生在**同一个类里**，方法名相同但参数列表不同（类型、个数、顺序不同都算），返回值和访问权限可以随意改。本质是**编译时多态**，编译期就根据参数类型确定调哪个方法了。

**重写（Override）** 发生在**父子类之间**，子类对父类方法的重新实现。方法名和参数列表必须完全相同，返回类型要相同或是父类方法返回类型的子类，访问权限只能比父类更宽松（不能更严格），抛出的异常只能比父类更少或更具体。本质是**运行时多态**，运行时才决定调哪个版本。

记忆口诀：重载是"同名不同参，一个类里干"；重写是"继承后推翻，运行时才判"。

###### 5. 重载的方法能否根据返回类型进行区分？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#四、重载（Overload）vs 重写（Override）|📖]]
**不能**。原因很直接：编译器在判断调用哪个重载方法时，依据的是你传进来的**参数列表**，跟返回类型没有关系。

假如允许只靠返回类型来区分，那写`int result = calculate(1, 2);`这行代码时，编译器无法判断你要调哪个方法，因为看不出来你想要`int`还是`double`返回值。

所以同一个类里出现方法名相同、参数列表也相同、只有返回类型不同的两个方法，直接编译报错。

###### 6. 简单说说 Java 中接口和抽象类的区别？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#五、抽象类 vs 接口|📖]]
接口和抽象类都不能直接实例化，都可以包含抽象方法，但设计目的和特性差别挺大。

**语法层面**：接口用`interface`，抽象类用`abstract class`。接口里的变量默认是`public static final`常量，方法默认是`public abstract`（Java 8后可以有`default`和`static`方法实现）；抽象类可以有普通成员变量、有构造方法，可以有任意访问权限的方法。

**继承上**：一个类可以**实现多个接口**，但只能**继承一个抽象类**。这是最核心的区别，Java单继承的限制可以靠接口来突破。

**设计目的不同**：接口强调的是"能做什么"，描述的是一种能力或行为规范，是"has-a"关系，比如`Flyable`、`Serializable`；抽象类强调的是"是什么"，描述的是类之间的层次关系，是"is-a"关系，比如`Animal`是动物类的抽象。

实际选择原则：**优先用接口**定义行为规范，获得最大灵活性；用抽象类来**复用代码**、提供模板方法。Spring框架就是大量接口定义契约、抽象类提供默认实现的典范。

###### 7. 抽象类能使用 final 修饰吗？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#五、抽象类 vs 接口|📖]]
**不能**，这两个关键字根本是矛盾的。

`abstract`要求类**必须被继承**才能被使用（因为它有未实现的方法）；`final`要求类**不允许被继承**。两者同时出现就是说"你必须继承我，但我不让你继承我"，逻辑上自相矛盾，Java编译器直接报错。

###### 8. 抽象的（abstract）方法是否可同时是静态的？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#五、抽象类 vs 接口|📖]]
**不能**，原因也是逻辑矛盾。

`abstract`方法是要求子类来重写实现的，调用时通过对象的多态机制动态绑定；`static`方法属于类本身，可以直接通过类名调用，不依赖对象，而且静态方法不能被重写（只能被隐藏）。

一个方法不可能既需要子类去实现（abstract的要求），又不依赖对象可以直接类名调用（static的特性），这两个要求放一起语义冲突，Java不允许。

###### 9. 什么是封装？如何实现封装？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#一、封装（Encapsulation）|📖]]
封装的核心思想是两点：**把数据和操作捆绑在一起**，**隐藏内部实现细节**。

实现方式很标准：把成员变量声明为`private`，然后提供`public`的getter/setter方法。在setter里可以加数据校验逻辑，控制数据的有效性。

```java
public class BankAccount {
    private double balance; // 私有化数据
    
    public void deposit(double amount) { // 控制修改的接口
        if (amount > 0) balance += amount;
    }
    
    public double getBalance() { return balance; } // 只读接口
    // 没有setBalance，防止余额被随意修改
}
```

封装的好处：提高安全性（外部不能随意修改内部状态）、提高可维护性（内部实现改了，只要接口不变，外部不受影响）。

###### 10. 接口和抽象类各自的使用场景是什么？

[[../../../20_JavaKnowledge/01_Java基础/02、面向对象三大特性#五、抽象类 vs 接口|📖]]
**优先用接口的场景**：
- 定义一组不相关类都要具备的某种能力，比如`Flyable`、`Comparable`、`Serializable`
- 一个类需要具备多种不同的能力（因为可以实现多接口）
- 框架设计中定义模块间的通信契约

**优先用抽象类的场景**：
- 多个有继承关系的类需要共享代码，把公共实现放抽象类里，子类继承复用
- 需要提供模板方法，规定算法骨架，让子类填充具体步骤
- 描述严格的"is-a"层次关系

现代设计有一个趋势：**面向接口编程**，尽量用接口来定义类型，获得最大灵活性，然后再用抽象类或具体类去实现。
