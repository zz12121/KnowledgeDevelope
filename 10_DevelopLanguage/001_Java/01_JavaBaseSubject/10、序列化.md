###### 1. 什么是序列化和反序列化？Serializable 接口的作用是什么？

[[../../../20_JavaKnowledge/01_Java基础/11、序列化与反序列化#一、序列化概述|📖]]
**序列化**：把内存中的Java对象转成字节序列的过程。字节序列可以持久化到磁盘（数据持久化），也可以通过网络传到另一台机器（远程传输）。

**反序列化**：序列化的逆过程，把字节序列还原成Java对象，恢复原始对象的状态。

`Serializable`接口是个**标记接口**，里面什么方法都没有，只是作为一个标记告诉JVM：这个类的对象可以被序列化。序列化机制（`ObjectOutputStream`）在运行时检查对象是否实现了这个接口，没实现就抛`NotSerializableException`。

###### 2. Java 序列化中如果有些字段不想进行序列化，怎么办？

[[../../../20_JavaKnowledge/01_Java基础/11、序列化与反序列化#二、Java 原生序列化（Serializable）|📖]]
主要三种方式：

**`transient`关键字（最常用）**：加在字段声明前，序列化时跳过这个字段，反序列化后该字段恢复默认值（null或0等）。适合密码、敏感信息等不想持久化的字段：
```java
public class User implements Serializable {
    private String name;
    private transient String password; // 不序列化
}
```

**`static`修饰**：静态变量属于类不属于对象实例，序列化针对的是实例状态，所以静态变量天然不被序列化。

**自定义序列化**：实现`Externalizable`接口，重写`writeExternal`和`readExternal`方法，完全自己控制序列化哪些字段、怎么序列化。也可以在实现`Serializable`的类中定义`writeObject`/`readObject`方法来自定义逻辑。

###### 3. serialVersionUID 的作用是什么？

[[../../../20_JavaKnowledge/01_Java基础/11、序列化与反序列化#二、Java 原生序列化（Serializable）|📖]]
`serialVersionUID`是序列化版本号，用来确保序列化和反序列化时的**类版本兼容性**。

工作机制：序列化时，JVM把当前类的`serialVersionUID`写入字节流。反序列化时，JVM比较字节流里的版本号和当前类的版本号，一致才能成功还原，不一致就抛`InvalidClassException`。

**为什么要显式声明**：如果你不声明，JVM会根据类的结构（类名、方法、字段等）自动生成。问题是，一旦类的结构有任何改动（哪怕加个方法），自动生成的ID就变了，之前序列化的对象就无法反序列化了。

**最佳实践**：在可序列化的类中显式声明一个固定值的`serialVersionUID`，这样类改了（比如加了新字段）也能向后兼容地反序列化旧版本的对象（新字段用默认值）：
```java
private static final long serialVersionUID = 1L;
```

###### 4. 序列化的底层实现原理是什么？

[[../../../20_JavaKnowledge/01_Java基础/11、序列化与反序列化#二、Java 原生序列化（Serializable）|📖]]
序列化底层依赖`ObjectOutputStream`和`ObjectInputStream`，遵循特定的二进制协议：

**序列化流程**：调用`writeObject(obj)`时，序列化器会递归地遍历对象图，处理所有关联对象。写入的内容包括：魔数和版本号（标识这是Java序列化流）、类的描述信息（类名、serialVersionUID、字段信息）、对象的实际字段值。如果同一个对象被多次引用，用句柄机制处理，只写一份数据，避免循环引用。

**反序列化流程**：根据类名找到并加载对应的Class对象，然后**不调用构造方法**，直接在JVM层面分配内存创建空对象，再通过反射把字节流中的字段值逐一填充进去。

"不走构造方法"这点很关键，意味着反序列化可以绕过构造方法里的逻辑，创建出"构造不允许创建"的对象，这也是某些安全漏洞的来源。

如果类里定义了`writeObject`/`readObject`方法，序列化引擎会通过反射调用这些方法实现自定义序列化逻辑。
