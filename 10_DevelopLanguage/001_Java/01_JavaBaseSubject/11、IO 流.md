###### 1. 说说 Java 中 IO 流？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#三、传统 IO 体系（java.io）|📖]]
IO流是程序和外界（文件、网络、控制台等）交换数据的通道。

按数据单位分两大类：

**字节流**：以字节（8位）为单位操作数据。根类是`InputStream`/`OutputStream`。处理所有二进制数据，如图片、视频、音频文件。代表类：`FileInputStream`/`FileOutputStream`、`BufferedInputStream`/`BufferedOutputStream`。

**字符流**：以字符（16位Unicode）为单位操作数据。根类是`Reader`/`Writer`。处理文本数据，自动处理字符编码转换，避免乱码。代表类：`FileReader`/`FileWriter`、`BufferedReader`/`BufferedWriter`、`InputStreamReader`（转换流）。

字节流是基础，字符流是上层建筑，底层还是字节流，只是封装了编码/解码逻辑。

###### 2. 既然有了字节流，为什么还要有字符流？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#三、传统 IO 体系（java.io）|📖]]
字符流是为了解决**文本处理的痛点**。

想象用字节流读中文文本：UTF-8下一个汉字是3字节，你得自己拼字节、处理编码、处理字符跨字节边界的情况，非常繁琐且容易出错。

字符流把这些复杂性封装起来，读的时候自动把字节解码成字符，写的时候自动把字符编码成字节，你只需要操作字符就行，完全不用操心编码细节。

结论：**字节流能处理一切，字符流让文本处理更方便**。实际开发中：非文本文件用字节流，文本文件用字符流（或转换流指定编码）。

###### 3. 字节流如何转为字符流？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#三、传统 IO 体系（java.io）|📖]]
通过**转换流**来桥接：`InputStreamReader`（字节输入流 → 字符输入流）和`OutputStreamWriter`（字符输出流 → 字节输出流）。

关键是可以**指定字符编码**，这是解决跨系统乱码问题的关键：

```java
// 用UTF-8编码读文本文件
try (BufferedReader reader = new BufferedReader(
        new InputStreamReader(new FileInputStream("file.txt"), StandardCharsets.UTF_8))) {
    String line;
    while ((line = reader.readLine()) != null) {
        // 处理每行
    }
}
```

`FileReader`是转换流的快捷版，但使用系统默认编码，跨环境可能乱码。**需要明确控制编码时用转换流，不要用FileReader**。

###### 4. Files 的常用方法都有哪些？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#三、传统 IO 体系（java.io）|📖]]
Java 7引入的`java.nio.file.Files`工具类，是现代Java文件操作的首选：

**读写文件**：`readAllBytes(Path)`一次读所有字节（小文件），`readAllLines(Path)`读所有行到List，`write(Path, byte[])`写字节，`newBufferedReader(Path)`建字符读取器。

**文件操作**：`exists(Path)`检查是否存在，`createFile(Path)`创建文件，`createDirectories(Path)`创建目录（含父目录），`copy(Path, Path)`复制，`move(Path, Path)`移动/重命名，`delete(Path)`删除（不存在抛异常），`deleteIfExists(Path)`安全删除。

**获取信息**：`size(Path)`获取文件大小，`isDirectory(Path)`判断是否目录，`getLastModifiedTime(Path)`获取最后修改时间。

###### 5. FileInputStream 和 FileReader 的区别是什么？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#三、传统 IO 体系（java.io）|📖]]
**FileInputStream** 继承自InputStream（字节流），读的是字节（byte），不处理编码，得到的是文件原始字节。适合复制图片、视频等二进制文件，或者你需要精确控制每个字节的场景。

**FileReader** 继承自Reader（字符流），读的是字符（char），自动用系统默认编码把字节转成字符。适合读文本文件，但因为使用默认编码，跨系统可能乱码，**建议用InputStreamReader替代并指定编码**。

核心结论：**处理二进制文件用FileInputStream，处理文本文件用InputStreamReader（指定编码）**。

###### 6. 简单说说 Java IO 与 NIO 的区别？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#一、IO 模型全景|📖]]
**IO（BIO，Blocking IO）**：面向流，每次读一点数据；阻塞模式，线程等数据时被挂起什么都干不了；一个连接需要一个线程，高并发时线程数量爆炸，资源耗尽。

**NIO（New IO）**：面向缓冲区，数据先读到Buffer里再处理；支持非阻塞模式，线程发起IO请求后可以继续干其他事；Selector选择器可以让一个线程监控多个Channel，一个线程处理多个连接，适合高并发。核心三件套：`Channel`（双向数据通道）、`Buffer`（数据缓冲区）、`Selector`（多路复用选择器）。

比喻：IO是服务员一直站在顾客旁边等点餐（阻塞）；NIO是一个服务员同时服务多桌顾客，谁准备好了再去处理（多路复用）。

###### 7. BIO、NIO、AIO 有什么区别？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#一、IO 模型全景|📖]]
这三种代表了Java处理IO的三种模型：

**BIO（同步阻塞）**：发起IO请求后，线程阻塞等待，直到操作完成。简单直观，适合连接数少的场景。

**NIO（同步非阻塞）**：发起IO请求后，线程立即返回，可以去做其他事；定期轮询或通过Selector事件通知来检查是否就绪。适合高并发、连接数多的场景。

**AIO（异步非阻塞）**：发起IO请求后立即返回；操作系统完成IO后，**主动回调**通知应用程序。不需要应用层轮询，真正的异步。适合大规模文件IO或需要高性能异步处理的场景。

关键区分：BIO/NIO都是**同步**的（调用方要主动等待或轮询结果）；AIO是**异步**的（操作系统主动通知）。

###### 8. Java IO 中用到了哪些设计模式？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#三、传统 IO 体系（java.io）|📖]]
**装饰器模式（最核心）**：IO流的精髓所在。通过组合而不是继承来动态添加功能。比如：
- `new BufferedInputStream(new FileInputStream("file"))`，给基础流套上缓冲层
- `new ObjectInputStream(new BufferedInputStream(new FileInputStream("file")))`，再套反序列化层

各种流可以像搭积木一样自由组合，灵活性极高，比继承灵活多了。

**适配器模式**：转换流`InputStreamReader`/`OutputStreamWriter`就是适配器，把字节流和字符流这两个不兼容的接口适配到一起，让它们可以协作。

###### 9. 如何正确关闭 IO 流？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#三、传统 IO 体系（java.io）|📖]]
**强烈推荐Try-with-Resources（Java 7+）**：实现`AutoCloseable`接口的资源在try括号里声明，JVM自动在try块结束后关闭，即使出现异常也会关闭，代码简洁安全：

```java
try (FileInputStream fis = new FileInputStream("source.jpg");
     FileOutputStream fos = new FileOutputStream("target.jpg")) {
    // 操作流
    byte[] buf = new byte[1024];
    int len;
    while ((len = fis.read(buf)) != -1) {
        fos.write(buf, 0, len);
    }
} // 自动关闭，无需手写finally
```

传统的try-catch-finally方式太繁琐，而且嵌套close也可能抛异常，容易写漏，现代Java代码都应该用try-with-resources。

###### 10. Java 有几种文件拷贝方式，哪一种效率最高？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#一、IO 模型全景|📖]]
主要四种方式，效率从低到高：

**基础字节流**：手动读写，最基础，性能一般。

**缓冲字节流**：`BufferedInputStream/BufferedOutputStream`，减少系统调用次数，比基础流明显快，大多数场景的推荐选择。

**`Files.copy()`**：Java 7+的高级API，代码最简洁，底层有优化。

**NIO的`FileChannel.transferTo()`**：效率最高，特别是大文件。底层利用操作系统的**零拷贝技术**，数据不经过用户态缓冲区，直接在内核态完成传输，CPU开销和上下文切换大幅减少。

**结论**：大文件拷贝用`FileChannel.transferTo()`效率最高；日常开发用`Files.copy()`代码最简洁。

###### 11. 什么是零拷贝技术？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#一、IO 模型全景|📖]]
零拷贝是减少数据拷贝次数的IO优化技术。

**传统文件传输的问题**：数据从磁盘到网卡需要经历4次拷贝（磁盘→内核缓冲区→用户空间→Socket缓冲区→网卡）和4次上下文切换，CPU开销很大。

**零拷贝的优化**：通过`sendfile`系统调用（Linux），数据可以直接从内核的文件缓冲区传到网卡，不需要经过用户空间，通常只需要2次DMA拷贝，大幅减少CPU负担和上下文切换。

Java的`FileChannel.transferTo()`在底层支持零拷贝，Kafka、Nginx等高性能中间件都大量使用了这个技术。

###### 12. RandomAccessFile 的使用场景是什么？

[[../../../20_JavaKnowledge/01_Java基础/10、IO与NIO深度解析#三、传统 IO 体系（java.io）|📖]]
`RandomAccessFile`支持**随机访问**文件——可以通过移动文件指针（seek）到任意位置进行读写，不像普通流只能顺序处理。

典型使用场景：
- **断点续传**：记录已下载的字节位置，下次续传时直接从该位置继续写
- **多线程下载**：把文件分成多块，每个线程负责一块，通过指针定位并行写入同一个文件
- **日志追加**：移动指针到文件末尾追加内容
- **定长记录数据库**：存储固定长度的记录，通过计算偏移量直接定位到某条记录
