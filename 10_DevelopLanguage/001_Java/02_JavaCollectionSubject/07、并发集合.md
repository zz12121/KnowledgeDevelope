###### 1. 说说 Java 中的并发集合有哪些？

[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#三、CopyOnWriteArrayList|📖]]
Java 并发集合主要在 `java.util.concurrent` 包里，设计目标是在保证线程安全的同时最大化并发性能，核心思路有三种：**锁分离**（不同区域用不同锁）、**无锁算法**（CAS 操作）、**写时复制**（COW）。

主要的并发集合：

**并发 Map**：`ConcurrentHashMap` — 高并发的键值对存储，JDK 1.8 之后采用 synchronized + CAS + 红黑树，性能很强，是 HashMap 在多线程场景下的首选替代品。`ConcurrentSkipListMap` — 基于跳表实现，有序且线程安全。

**并发 Queue**：`ArrayBlockingQueue`（有界阻塞队列）、`LinkedBlockingQueue`（可选有界的阻塞队列，两把锁）、`ConcurrentLinkedQueue`（无界非阻塞，CAS 实现）、`SynchronousQueue`（不存元素，直接传递）、`DelayQueue`（延迟队列）。

**并发 List**：`CopyOnWriteArrayList` — 写时复制，读操作完全无锁，适合读多写极少的场景。

**并发 Set**：`CopyOnWriteArraySet`（内部基于 CopyOnWriteArrayList）、`ConcurrentSkipListSet`（有序并发 Set，基于跳表）。

###### 2. CopyOnWriteArrayList 的原理和应用场景？

[[../../../20_JavaKnowledge/02_集合框架/05、ConcurrentHashMap深度解析#一、整体架构|📖]]
CopyOnWriteArrayList 的核心是**写时复制**策略：

**读操作**完全无锁，直接读当前数组引用，速度极快，可以多线程并发读。

**写操作**（add、set、remove 等）先加锁，然后把底层数组完整复制一份，在新数组上做修改，改完再把 volatile 修饰的数组引用切换到新数组，最后释放锁。由于数组引用是 volatile 的，新数组对所有读线程立即可见。

**适合的场景**：读多写极少，比如事件监听器列表、系统配置项、权限白名单。每次改配置才写一次，日常都是读。

**不适合的场景**：
- 写操作频繁 — 每次写都要复制整个数组，内存和性能开销很大
- 需要强一致性读 — 读到的是写操作前的快照，写完之前的读看不到新数据
- 数据量非常大 — 复制代价太高

###### 3. ConcurrentHashMap 的原理？

[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#四、阻塞队列（BlockingQueue）|📖]]
**JDK 1.7 — 分段锁时代**：把整个 Map 分成若干个 Segment，每个 Segment 相当于一个小 HashMap，有自己的 ReentrantLock。不同 Segment 的操作可以并发进行，默认并发度是16。

**JDK 1.8 — synchronized + CAS + 红黑树**：彻底废弃了 Segment，改为直接锁住单个桶的头节点（synchronized），锁粒度从"段"缩小到"单个桶"。同时大量使用 CAS 操作做无锁化尝试，减少线程阻塞。结构上也引入了红黑树（和 HashMap 1.8 一样），链表过长时自动树化，防止极端哈希冲突下性能退化。

###### 4. 如何选择合适的并发集合？

[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#五、ConcurrentLinkedQueue|📖]]
- **高频键值存取** → `ConcurrentHashMap`，全能选手，并发高性能好
- **有序的键值存取** → `ConcurrentSkipListMap`，线程安全且有序
- **生产者-消费者模式** → `ArrayBlockingQueue` 或 `LinkedBlockingQueue`，天然阻塞协调
- **高吞吐的直接任务传递** → `SynchronousQueue`，不存元素，直接手递手
- **读极多写极少** → `CopyOnWriteArrayList` / `CopyOnWriteArraySet`，读完全无锁
- **高性能非阻塞队列** → `ConcurrentLinkedQueue`，CAS 实现，无锁高吞吐

**重要原则**：多线程场景下优先用 JUC 的并发容器，不要用 `Collections.synchronizedXXX()` 粗暴包装，后者性能差很多。

###### 5. 为什么不推荐使用 Vector 和 Hashtable？

[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#六、ConcurrentSkipListMap / Set|📖]]
Vector 和 Hashtable 是 JDK 1.0 时代的老古董，它们实现线程安全的方式是在**几乎所有公共方法上都加 synchronized**。这是典型的粗粒度锁，同一时刻只有一个线程能访问，性能极差。

在高并发场景下，这种方式根本扛不住，远不如 JUC 里针对并发专门设计的集合。`ConcurrentHashMap` 锁的粒度细到单个桶，`CopyOnWriteArrayList` 读完全无锁，两者在并发性能上甩 Hashtable 和 Vector 几条街。

现在基本上没有理由再用 Vector 和 Hashtable 了，知道它们存在就好，代码里别写。

###### 6. Collections.synchronizedXXX 方法的原理是什么？

[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#八、如何选择并发集合|📖]]
`Collections.synchronizedList()` 等方法是**装饰器模式**的典型应用。它返回一个包装类，内部持有原集合对象，把所有方法都用 synchronized 包一层。

比如 synchronizedList 的包装类内部长这样：
```java
public boolean add(E e) {
    synchronized (mutex) { return c.add(e); }
}
public E get(int index) {
    synchronized (mutex) { return c.get(index); }
}
```

本质上是方法级别的粗粒度锁，能用，但性能不如 JUC 并发容器。

**有一个坑要注意**：即使用了 synchronizedList 包装，**迭代时也需要手动加锁**，否则还是线程不安全：
```java
List<String> syncList = Collections.synchronizedList(new ArrayList<>());
synchronized (syncList) {  // 迭代时必须手动同步
    for (String s : syncList) { ... }
}
```
