###### 1. 如何提高集合的性能？

[[../../../20_JavaKnowledge/02_集合框架/04、HashMap深度解析#三、扩容机制|📖]]
集合性能优化核心就四个维度：

**选对集合**：根据你的访问模式选合适的类型。随机访问多就用 ArrayList，键值查找用 HashMap，需要排序用 TreeMap/TreeSet，多线程用 ConcurrentHashMap。选错了底层数据结构，再怎么优化都是事倍功半。

**设好初始容量**：ArrayList、HashMap 这类集合都会自动扩容，但扩容要复制数据，很耗性能。如果能预估数据量，提前指定初始容量可以显著减少甚至避免扩容。这是成本最低、收益最明显的优化。

**用高效 API**：比如 Java 8 的 `removeIf()`、Stream API，内部做了优化，比自己写循环更高效。遍历 HashMap 用 `entrySet()` 而不是 `keySet()` + `get(key)`。

**并发控制选对**：多线程场景别用 `Collections.synchronizedXXX()` 粗暴包装，用 JUC 的并发集合，根据读写比例选 ConcurrentHashMap 还是 CopyOnWriteArrayList。

###### 2. 集合的初始容量应该如何设置？

[[../../../20_JavaKnowledge/02_集合框架/02、ArrayList深度解析#三、扩容机制|📖]]
**为什么要设置**：ArrayList 扩容是1.5倍，HashMap 扩容是2倍，每次扩容都要新建数组、复制旧数据，O(n) 的操作，数据量大的时候很明显。

**ArrayList 的初始容量**：直接按预估元素数量设置就好，比如预计存1000个就 `new ArrayList<>(1000)`。

**HashMap 的初始容量**：因为有负载因子（默认0.75），实际触发扩容的阈值是 `容量 × 0.75`，所以要把预估数量除以负载因子再取2的幂：

预计存1000个元素 → `1000 / 0.75 ≈ 1334` → 取最近的2的幂次方 → **设为2048**

也可以用 Google Guava 的 `Maps.newHashMapWithExpectedSize(n)` 帮你自动计算，不用自己手算。

**一个实用建议**：初始容量宁可略大也不要略小，多用点内存换来的是减少扩容和更少的哈希冲突，总体性价比很高。

###### 3. ArrayList 和 LinkedList 在不同场景下的性能对比？

[[../../../20_JavaKnowledge/02_集合框架/01、集合框架总览#七、集合选型指南|📖]]
核心结论：**绝大多数情况下 ArrayList 更快**，不是因为理论时间复杂度，而是因为**内存连续、CPU 缓存友好**。

- **随机访问（按索引）**：ArrayList O(1) 直接定位，LinkedList O(n) 要从头遍历。ArrayList 完胜。

- **尾部追加**：两者都是 O(1)（ArrayList 不扩容时），性能接近，ArrayList 稍好（缓存友好）。

- **中间插入删除**：理论上 LinkedList 只需改指针 O(1)，ArrayList 要移动元素 O(n)。**但实际情况是**，LinkedList 要先遍历找到那个位置（O(n)），加上每个节点对象的内存分配开销，和 ArrayList 的内存复制相比没有明显优势，有时甚至更慢。

- **内存占用**：ArrayList 内存连续、紧凑。LinkedList 每个节点要多存前后指针，内存占用更大且离散，GC 压力也更大。

**实用结论**：用 ArrayList 作为默认选择。只有需要频繁操作**头部**（addFirst/removeFirst）或把 List 当 Deque 用，才考虑 LinkedList 或直接用 `ArrayDeque`。

###### 4. HashMap 的性能优化技巧有哪些？

[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#八、如何选择并发集合|📖]]
**1. 合理设置初始容量**：如前所述，根据预估元素数量计算，避免扩容。这是最有效的优化手段。

**2. 保证 Key 的 hashCode 质量**：HashMap 性能高度依赖哈希分布均匀性。Key 的 hashCode 如果大量返回相同值，会导致大量哈希冲突，链表变长（甚至树化），性能从 O(1) 退化到 O(log n) 甚至 O(n)。写自定义类作 Key 时，hashCode 方法一定要写好，让不同对象的哈希值尽量分散。

**3. 遍历用 entrySet()**：遍历键值对时，用 `entrySet()` 一次拿到 key 和 value，不要先 `keySet()` 再 `get(key)` — 后者相当于查了两次哈希表：

```java
// 低效：查了两次哈希
for (String key : map.keySet()) {
    Object value = map.get(key);
}

// 高效：一次拿到 Entry
for (Map.Entry<String, Object> entry : map.entrySet()) {
    String key = entry.getKey();
    Object value = entry.getValue();
}
```

**4. 多线程换 ConcurrentHashMap**：单线程用 HashMap，多线程必须换 ConcurrentHashMap，不要用 `Collections.synchronizedMap()` 包装，那是粗暴整体锁，并发性能差。

**5. 了解树化机制做排查**：链表长度超8且数组容量 >= 64 才会树化。如果频繁触发树化，说明哈希冲突很严重，需要排查 Key 的 hashCode 方法或考虑扩大初始容量。
