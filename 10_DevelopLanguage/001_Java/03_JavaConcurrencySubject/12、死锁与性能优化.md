###### 1. 什么是死锁？死锁产生的条件是什么？
[[../../../20_JavaKnowledge/03_并发编程/12、死锁分析与性能优化#一、死锁（Deadlock）|📖]]
死锁是两个或多个线程互相持有对方需要的资源，谁都不肯先放，导致永远卡住的状态。

死锁必须同时满足四个条件（**破坏任意一个都能预防死锁**）：

1. **互斥条件**：资源同一时刻只能被一个线程使用（锁本身的特性，通常没法消除）
2. **请求与保持**：线程在持有资源的同时，去请求其他线程占用的资源，请求不到就等着，但不放已有的
3. **不可剥夺**：线程已经拿到的资源，没用完之前别人抢不走
4. **循环等待**：线程之间形成环形的等待链：A 等 B 的锁，B 等 C 的锁，C 等 A 的锁

###### 2. 如何避免死锁？
[[../../../20_JavaKnowledge/03_并发编程/12、死锁分析与性能优化#二、死锁检测与诊断|📖]]
破坏四个条件中的一个即可：

**固定加锁顺序**（破坏循环等待）：所有线程按同样的顺序获取多把锁，比如始终先锁 A 再锁 B，不会形成环形等待。这是最常用的方法。

**使用 tryLock 超时获取**（破坏请求与保持）：用 `ReentrantLock.tryLock(timeout)` 代替 `lock()`，拿不到锁就主动释放已有的锁，退出重试，而不是一直等。

**一次性申请所有资源**（破坏请求与保持）：把需要的所有锁一次性全部申请，要么全拿到要么全不拿，不会出现"拿着一个等另一个"的情况。代价是并发度低。

**避免嵌套锁**：能不在持锁状态下再去获取其他锁就不要。

###### 3. 如何检测和定位死锁？
[[../../../20_JavaKnowledge/03_并发编程/12、死锁分析与性能优化#三、活锁（Livelock）与饥饿（Starvation）|📖]]
**jstack**（最常用）：`jstack <pid>` 生成线程转储，JVM 会在输出中**明确标出"Found one Java-level deadlock"**，并列出相关线程和各自持有/等待的锁，定位非常直接。

**JConsole / JVisualVM**：图形化工具，"线程"页面有"检测死锁"按钮，点一下就能看到死锁链。线上不方便用，本地排查很方便。

**Arthas**（生产环境推荐）：`thread -b` 命令直接定位阻塞其他线程最多的那个线程，排查死锁非常高效。

**代码埋点**：用 `ThreadMXBean.findDeadlockedThreads()` 定期扫描，检测到死锁后告警，适合要求高可用的系统。

###### 4. 死锁、活锁、饥饿有什么区别？
[[../../../20_JavaKnowledge/03_并发编程/12、死锁分析与性能优化#五、并发性能优化策略|📖]]
三个都是线程"卡住"的问题，但性质不同：

**死锁**：线程全部**阻塞**，静止不动，互相等待，什么都不做。

**活锁**：线程**没有阻塞**，都在运行，但在互相"谦让"或不断重试，谁都没有实质性推进。比如两个人在走廊里都向同一侧让路，结果谁也过不去，但两个人都还在动。解决方法：引入随机退避时间，打破同步。

**饥饿**：某个线程**长期得不到资源**，其他线程都在跑，就它一直在等。通常是因为线程优先级太低或者某个线程长期霸占锁。解决方法：用公平锁或合理设置优先级。

###### 5. 如何提高并发程序的性能？
[[../../../20_JavaKnowledge/03_并发编程/12、死锁分析与性能优化#六、常见性能陷阱|📖]]
几个核心方向：

**减少锁的范围**：临界区只包含真正需要同步的代码，不要把不需要加锁的逻辑也放进去，让锁持有的时间尽量短。

**降低锁的粒度**：用更细粒度的锁，比如 `ConcurrentHashMap` 锁单个桶而不是整个 map，增加并发度。

**用无锁方案**：能用 `AtomicXxx` 或 `ConcurrentHashMap` 就别用 `synchronized`，无锁性能更好。

**减少共享状态**：能用局部变量、不可变对象或 `ThreadLocal` 的地方就不要共享，从根本上消除同步需求。

**合理使用线程池**：避免频繁创建销毁线程，控制并发数量，防止过多线程争抢 CPU。

**读多写少的场景**：用 `ReadWriteLock` 或 `StampedLock` 代替独占锁，让读操作并发执行。

###### 6. 什么是无锁编程？
[[../../../20_JavaKnowledge/03_并发编程/12、死锁分析与性能优化#四、锁竞争分析|📖]]
无锁编程用**CAS 原子操作**代替传统互斥锁实现线程安全，线程不阻塞，操作冲突时重试。

**优点**：没有线程上下文切换开销，在低竞争时吞吐量很高，也不会有死锁。

**缺点**：高竞争时 CAS 频繁失败，线程一直自旋空转，反而比加锁更耗 CPU；存在 ABA 问题；代码逻辑比加锁复杂。

**适用场景**：竞争不激烈的计数器（`AtomicLong`）、无锁队列（`ConcurrentLinkedQueue`）、状态标志（`AtomicBoolean`）。

###### 7. 线程数应该设多少？线程过多或过少会怎样？
[[../../../20_JavaKnowledge/03_并发编程/12、死锁分析与性能优化#一、死锁（Deadlock）|📖]]
**线程太少**：CPU 有空闲，特别是 I/O 密集型任务，大部分时间在等待，浪费了 CPU 资源，吞吐量低。

**线程太多**：线程间上下文切换开销大；内存占用高（每个线程有自己的栈，默认 512KB~1MB）；GC 压力增大；线程调度本身也消耗 CPU。

**通用参考值**：CPU 密集型设 CPU 核心数 + 1；I/O 密集型设 CPU 核心数 × 2（或根据等待/计算比例计算）。无论理论算法，最终都要在生产负载下压测验证。
