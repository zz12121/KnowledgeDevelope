###### 1. 什么是线程池？为什么要使用线程池？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#一、线程池概述|📖]]
线程池是一种**线程复用机制**，预先创建一批线程放在"池子"里，有任务来了直接分配给空闲线程执行，而不是每来一个任务就 new 一个线程、完了就销毁。

为什么要用？主要三个原因：

第一，**减少开销**：创建和销毁线程涉及系统调用，是有成本的。复用线程可以省掉这些开销。

第二，**提高响应速度**：任务来了直接有线程接，不用等创建。

第三，**控制系统资源**：如果来多少任务就创建多少线程，线程数量失控会把内存吃光（OOM），还有大量上下文切换。线程池能把并发数控制在合理范围内。

###### 2. 线程池的核心参数有哪些？各有什么作用？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#二、ThreadPoolExecutor 核心参数|📖]]
`ThreadPoolExecutor` 有七个核心参数，都很重要：

**corePoolSize（核心线程数）**：线程池长期保留的线程数，即使空闲也不销毁（除非开启了 `allowCoreThreadTimeout`）。

**maximumPoolSize（最大线程数）**：当队列满了之后，允许新建临时线程，但总数不能超过这个值。

**keepAliveTime + unit（空闲线程存活时间）**：超过 corePoolSize 的那些临时线程，空闲时间超过这个值就会被销毁。

**workQueue（任务队列）**：放不下的任务先进队列等着。常用的有 `ArrayBlockingQueue`（有界）、`LinkedBlockingQueue`（可以无界）、`SynchronousQueue`（不存储，直接传递）。

**threadFactory（线程工厂）**：用于创建新线程，可以自定义线程名、优先级等，线上排查问题时非常有用。

**handler（拒绝策略）**：当线程数达到最大值且队列也满了，无法接收新任务时的处理方式。

###### 3. 线程池的工作流程是怎样的？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#三、任务提交流程|📖]]
任务提交进来，线程池按以下顺序处理：

1. 当前工作线程数 < `corePoolSize`：**直接新建核心线程**执行任务，哪怕有空闲线程
2. 线程数 ≥ `corePoolSize`，队列没满：**任务进队列**等待
3. 队列满了，线程数 < `maximumPoolSize`：**新建临时线程**执行任务
4. 队列满了，线程数也到上限了：**执行拒绝策略**

记住关键点：**新建线程 → 入队 → 扩展到最大线程 → 拒绝**，这个顺序不能搞反。

###### 4. 线程池有哪些拒绝策略？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#四、拒绝策略|📖]]
JDK 内置了四种：

**AbortPolicy（默认）**：直接抛 `RejectedExecutionException`，让调用方感知到任务被拒绝了。

**CallerRunsPolicy**：让提交任务的那个线程（比如主线程）自己来执行这个任务。这相当于一种"反压"机制，提交方会变慢，自然就降低了提交速度。

**DiscardPolicy**：悄悄丢掉任务，什么也不做，也不通知。小心数据丢失。

**DiscardOldestPolicy**：丢掉队列里最老的那个任务（队头），然后再试着提交当前任务。

生产环境一般自定义拒绝策略，比如记录日志、写入消息队列做异步补偿。

###### 5. 线程池有哪些常见类型？为什么不建议用 Executors 创建线程池？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#五、线程池的关闭|📖]]
Executors 提供了几个快捷方法：`FixedThreadPool`（固定线程数）、`CachedThreadPool`（动态线程数）、`SingleThreadExecutor`（单线程顺序执行）、`ScheduledThreadPool`（定时任务）。

**为什么不推荐用？** 主要是两个 OOM 风险：

`FixedThreadPool` 和 `SingleThreadExecutor` 用的是**无界 `LinkedBlockingQueue`**，队列可以无限增长，任务堆积会把内存撑爆。

`CachedThreadPool` 和 `ScheduledThreadPool` 的最大线程数是 `Integer.MAX_VALUE`，极端情况下会创建大量线程，同样可能 OOM。

**推荐做法**：直接用 `ThreadPoolExecutor` 构造函数，手动指定有界队列和合理的最大线程数，一切都显式可控。

###### 6. 如何合理配置线程池的大小？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#六、Executors 工厂方法|📖]]
没有通用公式，要根据任务类型来：

**CPU 密集型任务**（数学计算、加解密等）：设 CPU 核心数 + 1。多一个是为了应对偶发的 Page Fault 等暂停，保持 CPU 不空转。

**I/O 密集型任务**（数据库、网络请求等）：线程大部分时间在等 I/O，CPU 很闲，可以多开线程。粗略公式是 CPU 核心数 × 2，精确一点是 CPU 核心数 × (1 + 等待时间/计算时间)。

无论理论上算出什么，最后都要**压测验证**，观察实际的 CPU 利用率、吞吐量和响应时间。

###### 7. execute() 和 submit() 有什么区别？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#七、线程池监控与调优|📖]]
`execute()` 只能提交 `Runnable`，没有返回值，任务里抛的异常会直接传播到线程的未捕获异常处理器。

`submit()` 可以提交 `Runnable` 或 `Callable`，返回一个 `Future` 对象。可以通过 `future.get()` 获取执行结果，如果任务抛了异常，异常会被包在 `ExecutionException` 里，调用 `get()` 时才会抛出。

如果你不关心结果也不想处理异常，用 `execute()`；需要获取结果或者想统一处理异常，用 `submit()`。

###### 8. shutdown() 和 shutdownNow() 的区别？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#七、线程池监控与调优|📖]]
两者都会拒绝接收新任务，区别在于怎么处理现有的任务：

**shutdown()**：温柔关闭。等所有**已提交的任务（包括队列里排队的）都执行完**，线程池才真正停止。适合正常关闭场景。

**shutdownNow()**：强制关闭。立刻**中断所有正在执行的线程**（发送 interrupt 信号），并**返回队列中还没执行的任务列表**。注意：中断只是发信号，任务内部要响应 interrupt 才能真正停下来。

推荐搭配 `awaitTermination()` 使用，等线程池真正关闭后再继续后续逻辑：
```java
executor.shutdown();
if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
    executor.shutdownNow(); // 超时还没停就强制关
}
```

###### 9. 线程池的工作线程抛出异常会怎样？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#二、ThreadPoolExecutor 核心参数|📖]]
要区分 `execute()` 和 `submit()` 两种情况：

用 `execute()` 提交的任务，如果运行时抛出未捕获异常，**这个线程会终止**，线程池会新建一个线程补上。异常会传递给线程的 `UncaughtExceptionHandler`，如果没设置就会打印到控制台，**不影响其他线程**。

用 `submit()` 提交的任务，异常会被**静默吞掉**，包装在 `Future` 里。只有调用 `future.get()` 时才会以 `ExecutionException` 的形式抛出。如果你用 `submit()` 提交任务但从不调用 `get()`，异常就永远不会被发现，这是一个常见的坑。

###### 10. 什么是 ForkJoinPool 和工作窃取算法？
[[../../../20_JavaKnowledge/03_并发编程/08、线程池深度解析#六、Executors 工厂方法|📖]]
`ForkJoinPool` 是专门为"分治型"并行计算设计的线程池，适合把大任务递归拆分成小任务并行执行。

**工作窃取算法**：每个工作线程都有自己的双端队列。自己生成的子任务放在自己队列的头部，自己也从头部取。当自己的队列空了，就去**其他线程队列的尾部**偷任务来执行。从尾部偷而不是头部，是为了减少竞争，提高吞吐量。

Java 8 的 `parallelStream()` 底层默认就用的 `ForkJoinPool.commonPool()`。
