###### 1. 什么是原子类？Java 中有哪些原子类？
[[../../../20_JavaKnowledge/03_并发编程/05、CAS与原子类#一、CAS 原理|📖]]
原子类是 Java 并发包（`java.util.concurrent.atomic`）提供的一组**无锁线程安全类**，底层依赖 CPU 的 **CAS 指令**和 `volatile` 语义，对单个共享变量的读-改-写操作做到了原子性，性能比加锁好很多。

主要分五大类：
- **基本类型**：`AtomicInteger`、`AtomicLong`、`AtomicBoolean`
- **数组类型**：`AtomicIntegerArray`、`AtomicLongArray`、`AtomicReferenceArray`（原子更新数组中的某个元素）
- **引用类型**：`AtomicReference`、`AtomicStampedReference`（带版本号，解决 ABA）、`AtomicMarkableReference`（带标记位）
- **字段更新器**：`AtomicIntegerFieldUpdater` 等，对现有类的 `volatile` 字段做原子操作，不需要改源码
- **累加器**：`LongAdder`、`DoubleAdder`、`LongAccumulator`，高并发求和专用，性能远超 `AtomicLong`

###### 2. AtomicInteger 的实现原理是什么？
[[../../../20_JavaKnowledge/03_并发编程/05、CAS与原子类#二、原子类全览|📖]]
核心就三个东西：

一个用 `volatile` 修饰的 `int value` 字段，保证可见性；一个 `valueOffset` 常量，记录 `value` 字段在内存中的偏移量（类加载时通过 `Unsafe` 获取）；以及 `Unsafe` 类，提供直接操作内存的 CAS 指令。

以 `incrementAndGet()` 为例，它的逻辑是：不断读取当前值，计算新值（加 1），然后 CAS 尝试把旧值换成新值。如果 CAS 失败（说明有其他线程先改了），就重新读取再试，直到成功为止。这就是**自旋 CAS**。

###### 3. CAS 算法是什么？它的优缺点是什么？
[[../../../20_JavaKnowledge/03_并发编程/05、CAS与原子类#二、原子类全览|📖]]
CAS（Compare-And-Swap）是一种**乐观锁**思想：操作前先读取旧值，修改时再检查内存里的值有没有被改过，如果没变就更新，如果变了就重试。整个操作由 CPU 保证原子性（CMPXCHG 指令）。

**优点**：不阻塞线程，竞争不激烈时性能远好于加锁；也不存在死锁问题。

**缺点**有三个：
一是 **ABA 问题**：值从 A 变成 B 再变回 A，CAS 以为没变，其实已经被动过了。
二是**自旋开销**：竞争激烈时 CAS 频繁失败，线程一直在空转（自旋），白白消耗 CPU。
三是**只保证单变量原子性**：如果需要同时原子性地更新多个变量，CAS 做不到，得用锁或者 `AtomicReference` 把多个变量包一起。

###### 4. CAS 的 ABA 问题是什么？怎么解决？
[[../../../20_JavaKnowledge/03_并发编程/05、CAS与原子类#三、LongAdder vs AtomicLong|📖]]
ABA 问题是指：值最初是 A，线程 1 准备 CAS 改成 C，但还没执行。这期间线程 2 把它改成了 B，然后又改回了 A。线程 1 再执行 CAS，发现值是 A 就以为没人动过，操作成功了，但其实中间已经被改过两次。

**解决方案是加版本号**，每次修改值时版本号也递增。即使值从 A 变 B 变 A，版本号从 1 变 2 变 3，CAS 检查时版本号对不上就知道被改过了。

Java 里有两个类解决这个问题：

**`AtomicStampedReference`**：用一个 `int` 版本号（stamp）跟踪每次变更，能知道改了多少次。

**`AtomicMarkableReference`**：只用一个 `boolean` 标记位，不关心改了几次，只关心"有没有被改过"。比如标记一个节点是否已被逻辑删除。

###### 5. AtomicStampedReference 和 AtomicMarkableReference 怎么选？
[[../../../20_JavaKnowledge/03_并发编程/05、CAS与原子类#一、CAS 原理|📖]]
看需求：

如果你需要知道值被修改了几次，或者需要防止反复修改的 ABA 问题，用 **`AtomicStampedReference`**。

如果你只需要知道"这个引用有没有被动过"（比如节点是否已处理），用 **`AtomicMarkableReference`** 更轻量。

###### 6. LongAdder 和 AtomicLong 的区别是什么？
[[../../../20_JavaKnowledge/03_并发编程/05、CAS与原子类#四、原子引用类型|📖]]
在低并发下两者差不多，高并发写多的场景下 `LongAdder` 性能**远好于** `AtomicLong`。

**原因**：`AtomicLong` 让所有线程竞争同一个 `value` 变量，CAS 失败率高，自旋严重。`LongAdder` 用了**分散竞争**的思路：内部有一个 `base` 和一个 `Cell[]` 数组，线程来了先尝试更新 `base`，CAS 失败了就根据线程哈希找到对应的 `Cell`，在那个 `Cell` 上累加，不同线程打到不同的 `Cell` 上，竞争分散了。求最终值时把 `base + 所有 Cell` 相加即可。

**权衡**：`LongAdder` 的 `sum()` 是**近似值**，计算期间可能还有其他线程在更新。`AtomicLong` 的读取是精确实时的，还支持 `compareAndSet` 等复杂操作。

**选择原则**：高并发计数器（统计 QPS、点击量）用 `LongAdder`；需要精确值或做 CAS 比较的场景用 `AtomicLong`。

###### 7. 什么是字段更新器？使用场景是什么？
[[../../../20_JavaKnowledge/03_并发编程/05、CAS与原子类#一、CAS 原理|📖]]
字段更新器（`AtomicIntegerFieldUpdater`、`AtomicLongFieldUpdater`、`AtomicReferenceFieldUpdater`）允许你以原子方式更新**已有类中的 `volatile` 字段**，不需要把字段类型改成原子类。

典型场景：你在用第三方类或者不想修改已有实体类的情况下，需要对某个字段做原子操作。

要求：目标字段必须是 `volatile` 修饰的，且对更新器所在类可见。

###### 8. 原子类的典型应用场景有哪些？
[[../../../20_JavaKnowledge/03_并发编程/05、CAS与原子类#一、CAS 原理|📖]]
- **计数器**：网站访问量、接口调用次数，`LongAdder` 是首选
- **状态标志**：用 `AtomicBoolean` 做"只执行一次"的开关，避免 `synchronized`
- **序列号生成**：用 `AtomicLong.getAndIncrement()` 生成递增唯一 ID
- **无锁数据结构**：无锁队列、无锁栈等高级并发数据结构的基础，依赖 `AtomicReference` + CAS
