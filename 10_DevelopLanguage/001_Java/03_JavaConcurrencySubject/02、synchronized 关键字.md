###### 1. 说说 synchronized 关键字的理解？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#一、synchronized 的三种用法|📖]]
`synchronized` 是 Java 内置的同步机制，核心就是**同一时刻只允许一个线程执行被保护的代码**，解决多线程访问共享资源时的安全问题。

它能给我们提供三大保证：**原子性**（锁定的操作不可分割）、**可见性**（线程释放锁前会强制把修改刷回主内存，下一个拿锁的线程能看到最新值）、**有序性**（防止指令重排序干扰并发逻辑）。

###### 2. synchronized 的底层原理是什么？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#二、底层原理：对象头（Object Header）|📖]]
底层依赖 **Java 对象头** 和 **Monitor（监视器锁）** 机制。

每个 Java 对象的对象头里有一个 **Mark Word**，里面存着锁状态、持有锁的线程 ID 等信息，对象还跟一个 Monitor 关联。

当 `synchronized` 修饰代码块时，编译后字节码会插入 `monitorenter`（进入同步块时尝试获取锁）和 `monitorexit`（退出时释放锁）两条指令。修饰方法时，JVM 则通过方法上的 `ACC_SYNCHRONIZED` 访问标志来识别，原理类似——实例方法锁的是 `this`，静态方法锁的是当前类的 `Class` 对象。

###### 3. synchronized 可以修饰哪些地方？有什么区别？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#一、synchronized 的三种用法|📖]]
有三种用法，锁的目标不同：

**修饰实例方法**：锁是当前对象实例（`this`），不同线程访问同一个对象的该方法会互斥，但不同对象之间不影响。

**修饰静态方法**：锁是当前类的 `Class` 对象，因为静态方法属于类而不属于某个实例，所以不同对象调用也会互斥。

**修饰代码块**：锁是你指定的那个对象，比如 `synchronized(this)` 或 `synchronized(MyClass.class)`，粒度最细，灵活性最高。

有个容易踩坑的地方：一个线程访问实例同步方法，另一个线程访问静态同步方法，它们拿的是**不同的锁**，不会互斥。

###### 4. synchronized 的锁升级过程是怎样的？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#三、锁升级过程（偏向→轻量→重量）|📖]]
JVM 为了减少加锁开销，锁会随竞争激烈程度逐步升级，且**不可逆**。

整个过程是：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**。

一开始对象没有锁，当第一个线程来访问时，锁就偏向它；如果偶尔有第二个线程来竞争，升级为轻量级锁，通过 CAS 自旋尝试；如果竞争很激烈，自旋失败，就膨胀为重量级锁，没拿到锁的线程直接被挂起进入阻塞队列，等待被唤醒。

###### 5. 什么是偏向锁、轻量级锁、重量级锁？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#三、锁升级过程（偏向→轻量→重量）|📖]]
**偏向锁**：专为"只有一个线程反复使用同一把锁"的场景设计。第一次获取锁时把线程 ID 写入 Mark Word，之后同一线程再来，直接拿，连 CAS 都不用做。开销极小，但一旦出现竞争就需要撤销偏向锁，有一定代价。

**轻量级锁**：多个线程交替使用、竞争不激烈时适用。线程在自己的栈帧里创建一个 Lock Record，然后用 CAS 把对象的 Mark Word 替换为指向该 Lock Record 的指针。CAS 失败则自旋等一等，自旋到一定次数还没拿到才升级。

**重量级锁**：竞争激烈、锁持有时间较长时适用。底层依赖操作系统的互斥量（Mutex），没拿到锁的线程会被挂起，等锁释放时再被唤醒。上下文切换开销大，但不会浪费 CPU 空转。

###### 6. synchronized 和 Lock 的区别是什么？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#八、synchronized vs ReentrantLock|📖]]
两者都能实现同步，但差异挺明显的。

首先从**实现层次**看，`synchronized` 是 JVM 层面的内置关键字，而 `Lock`（比如 `ReentrantLock`）是 JDK 层面的 API 接口。

其次从**使用方式**看，`synchronized` 会自动加锁和释放锁，代码块结束或抛异常时自动解锁；`ReentrantLock` 需要手动调用 `lock()` 和 `unlock()`，通常要放在 `finally` 块里确保能解锁，稍微麻烦一点但更灵活——它支持可中断获取锁、超时获取锁、尝试非阻塞获取锁，还支持公平锁。

最后从**等待机制**看，`synchronized` 只能用 `Object.wait/notify`，所有等待线程共用一个等待队列；`ReentrantLock` 可以通过 `Condition` 绑定多个等待队列，能做精确的线程通知，比如生产者和消费者分开等待，各自唤醒。

简单说：**能用 `synchronized` 就用它，代码更简洁；需要高级控制才换 `ReentrantLock`。**

###### 7. synchronized 和 volatile 的区别是什么？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#八、synchronized vs ReentrantLock|📖]]
两者都是解决并发问题的工具，但侧重点完全不同。

`volatile` 是**轻量级**的，它只保证可见性和有序性，**不保证原子性**。比如 `i++` 这种复合操作加了 `volatile` 也是线程不安全的。它的优势是不会引起线程阻塞，性能好。

`synchronized` 是**重量级**的，可见性、有序性、原子性**三样都能保证**。但代价是有可能引起线程阻塞和上下文切换。

一句话总结：**状态标志这种简单场景用 `volatile`，涉及复合操作或多个变量联动就必须用 `synchronized`。**

###### 8. synchronized 的可见性和有序性如何保证？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#二、底层原理：对象头（Object Header）|📖]]
**可见性**来自 happens-before 原则：对一把锁的解锁操作 happens-before 后续对这把锁的加锁操作。也就是说，前一个线程解锁前对共享变量做的所有修改，下一个拿到锁的线程一定能看到。

**有序性**的保证逻辑是：同一时刻只允许一个线程进入同步块，所以在单个线程的视角里执行是有序的，不会因指令重排序导致数据不一致。

###### 9. synchronized 的锁优化有哪些？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#五、JDK 6+ 的四项关键优化|📖]]
JVM 和 JIT 编译器对 `synchronized` 做了多项优化，让它没有想象中那么"重"：

**锁消除**：JIT 通过逃逸分析发现锁对象不会逃逸出当前线程（比如方法内的局部 `StringBuffer`），会直接消除这个没有竞争的锁。

**锁粗化**：如果对同一个对象连续地加锁解锁，JIT 会把这些操作合并为一次，避免频繁开销。

**自适应自旋**：JDK 6 以后，自旋次数不是固定的，JVM 会根据历史经验动态调整——上次自旋成功就多转几圈，上次失败就少转甚至直接挂起。

###### 10. synchronized 能防止指令重排序吗？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#五、JDK 6+ 的四项关键优化|📖]]
**可以**。`synchronized` 通过 happens-before 规则（监视器解锁 happens-before 加锁）确保了同步代码块内部的代码不会被重排序到块外。对外部线程来说，同步块内的操作像是一个原子整体，顺序是确定的。

###### 11. synchronized 的 wait/notify 使用有什么注意事项？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#七、synchronized 与 wait/notify 协作模式|📖]]
三个必须牢记的要点：
1. `wait()`、`notify()`、`notifyAll()` **必须在 synchronized 块内调用**，否则会抛 `IllegalMonitorStateException`。
2. 判断等待条件必须用 **while 循环，不能用 if**。原因是存在"虚假唤醒"（操作系统可能无故唤醒线程），以及多消费者场景下被唤醒后条件可能已经被别人消耗掉了。
3. **尽量用 notifyAll() 而不是 notify()**，因为 notify() 只随机唤醒一个线程，可能唤醒了错误的线程导致死锁。
