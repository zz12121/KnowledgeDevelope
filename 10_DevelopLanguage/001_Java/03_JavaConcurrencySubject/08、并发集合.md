###### 1. ConcurrentHashMap 的实现原理是什么？JDK 1.7 和 1.8 有什么区别？
[[../../../20_JavaKnowledge/02_集合框架/05、ConcurrentHashMap深度解析#一、整体架构|📖]]
`ConcurrentHashMap` 是线程安全的高性能哈希表，两个版本的实现思路差别很大：

**JDK 1.7（分段锁）**：把整个 Map 分成多个 `Segment`（默认 16 个），每个 `Segment` 是一个独立的哈希表，继承自 `ReentrantLock`。写操作只锁住对应的 `Segment`，不同 `Segment` 之间可以并发，最多支持 16 个线程同时写。结构是 `Segment 数组 + HashEntry 链表`。

**JDK 1.8（CAS + synchronized）**：彻底抛弃了 `Segment`，结构和 `HashMap` 一样（数组 + 链表 + 红黑树），但写操作改用 **CAS 先无锁尝试，失败再用 synchronized 锁单个桶的头节点**。锁粒度从 Segment 级别（每个 Segment 管多个桶）降到单个桶，并发度大幅提升，理论上可以有和数组大小一样多的线程同时写。

1.8 还引入了红黑树优化长链表查询，以及**多线程协同扩容**（线程在 put 时发现正在扩容，会主动帮忙迁移数据）。

###### 2. ConcurrentHashMap 的 put 操作流程是怎样的？（JDK 1.8）
[[../../../20_JavaKnowledge/02_集合框架/05、ConcurrentHashMap深度解析#二、JDK 1.7 实现|📖]]
1. 计算 key 的 hash 值
2. 如果 table 还没初始化，先初始化（CAS 操作）
3. 找到目标桶，如果**桶为空**，用 **CAS 直接插入**新节点，不需要加锁
4. 如果桶的头节点是 `ForwardingNode`（扩容标志），说明正在扩容，**当前线程协助迁移**
5. 如果桶有数据，用 **synchronized 锁住头节点**，再遍历链表/红黑树执行插入或更新
6. 插入后检查是否需要树化（链表长度 ≥ 8 且数组长度 ≥ 64）
7. 检查是否需要扩容

###### 3. ConcurrentHashMap 如何保证线程安全？
[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#三、CopyOnWriteArrayList|📖]]
三层保障：

**CAS**：桶为空时的首次插入用 CAS，无锁化，性能最好。

**synchronized**：桶有数据时锁头节点（也可能是树的根节点），粒度精确到单个桶。JDK 6+ 的 synchronized 经过大量优化，在低竞争时性能接近 CAS。

**volatile**：Node 节点的 `value` 和 `next` 用 volatile 修饰，保证节点值的修改对其他线程立即可见，读操作不需要加锁。

###### 4. ConcurrentHashMap 能保证复合操作的原子性吗？
[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#四、阻塞队列（BlockingQueue）|📖]]
**单个方法调用（put/get/remove）是线程安全的，但多个方法组合起来的复合操作不是原子的。**

比如这个经典错误写法：
```java
if (!map.containsKey(key)) {
    map.put(key, value); // 两步之间可能有其他线程也 put 了
}
```

应该用 `ConcurrentHashMap` 提供的原子复合操作：
- `putIfAbsent(key, value)`：key 不存在才放入
- `computeIfAbsent(key, fn)`：key 不存在则计算值再放入（懒加载缓存常用）
- `replace(key, oldValue, newValue)`：只有当前值等于 oldValue 才替换

###### 5. CopyOnWriteArrayList 的实现原理是什么？
[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#六、ConcurrentSkipListMap / Set|📖]]
核心思想是**写时复制（Copy-On-Write）**：

**读操作**：直接在当前数组上读，完全无锁，不需要任何同步，速度极快。

**写操作**：先加锁，然后**复制一份底层数组**，在新数组上修改，改完把数组引用指向新数组。整个过程读操作看到的都是旧数组，不受影响，读写完全不冲突。

因此：**读操作零开销，写操作有一次数组复制的开销。**

###### 6. CopyOnWriteArrayList 的优缺点和适用场景是什么？
[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#一、为什么需要并发集合|📖]]
**优点**：读性能极高（无锁）；迭代时不会抛 `ConcurrentModificationException`（迭代的是创建时刻的快照）；读写互不阻塞。

**缺点**：写操作每次都要复制整个数组，数组大时内存开销和 GC 压力很大；读到的数据可能不是最新的（弱一致性，看到的是写之前的快照）。

**适用场景**：**读多写极少**的场景，且对数据实时性要求不高，比如事件监听器列表、黑白名单、系统配置信息等。如果写操作频繁，用 `ConcurrentHashMap` 或加锁更合适。

###### 7. BlockingQueue 有哪些常用实现？怎么选择？
[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#一、为什么需要并发集合|📖]]
**`ArrayBlockingQueue`**：有界队列，数组实现，一把锁（生产者消费者共用），内存连续，适合需要明确边界控制的场景。

**`LinkedBlockingQueue`**：默认无界（可指定容量），链表实现，**两把锁**（`putLock` 和 `takeLock`），生产者和消费者互不干扰，高并发下吞吐量更高。`FixedThreadPool` 和 `SingleThreadExecutor` 默认用的就是它（要注意默认无界会有 OOM 风险）。

**`SynchronousQueue`**：不存储元素，put 必须有线程在 take，直接传递任务，吞吐量极高。`CachedThreadPool` 用的就是它。

**`PriorityBlockingQueue`**：无界，按优先级出队，适合需要任务优先级的场景。

**`DelayQueue`**：元素必须到期才能取出，适合定时任务、缓存过期清理。

**选择原则**：需要限流/防止 OOM 用有界队列（ArrayBlockingQueue）；追求高吞吐用 LinkedBlockingQueue；任务直接传递不存储用 SynchronousQueue。

###### 8. ArrayBlockingQueue 和 LinkedBlockingQueue 有什么区别？
[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#一、为什么需要并发集合|📖]]
**底层结构**：前者是定长数组，初始化就分配好内存，连续；后者是链表，动态创建节点，每次操作有额外的节点对象开销。

**锁机制**：前者生产者和消费者共用一把锁，高并发时可能争抢；后者分两把锁（`putLock`/`takeLock`），生产和消费可以并行，高并发吞吐量更高。

**容量**：前者必须指定容量（有界）；后者默认无界，可选指定容量。

**选择**：需要严格边界控制或要求公平性时选 `ArrayBlockingQueue`；追求更高吞吐量选 `LinkedBlockingQueue`（记得指定容量！）。

###### 9. ConcurrentLinkedQueue 的特点是什么？
[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#一、为什么需要并发集合|📖]]
基于 **CAS 无锁算法** 实现的非阻塞线程安全队列（FIFO）。

特点：不会阻塞线程，操作失败立刻重试；非常适合高并发且生产消费速度大致匹配的场景；无界。

注意：**`size()` 方法需要遍历整个链表，开销大，结果也可能不准确（弱一致性）**。判断是否为空用 `isEmpty()` 而不是 `size() == 0`。

###### 10. ConcurrentHashMap在JDK 1.7和1.8有什么区别？——高频面试引导问题

**版本对比**：

| 特性 | JDK 1.7 | JDK 1.8 |
|------|---------|---------|
| 数据结构 | Segment + HashEntry | Node + 红黑树 |
| 并发原理 | 分段锁 | CAS + synchronized |
| 锁粒度 | 整个Segment | 每个桶 |
| 查询复杂度 | O(n) | O(log n) |

**回答示例**：

> 1.7用的是Segment分段锁，16个Segment，每个Segment相当于一个小的HashMap，锁住一段。1.8改成CAS + synchronized，锁的粒度更细，直接锁桶里的头结点。
>
> 为什么1.8要加红黑树？因为链表太长查询会退化到O(n)，加红黑树保证O(log n)。
>
> 追问：synchronized用在什么时候？
> CAS用于保证头结点不变，synchronized用于真正发生hash冲突需要写链表/红黑树时。

###### 11. ConcurrentHashMap的put流程是怎样的？——高频面试引导问题

**put流程（jdk 1.8）**：

```
1. 计算hash
2. if 当前位置空？→ CAS尝试插入
   ↓ 否
3. if 头结点正在扩容？→ 帮助扩容
   ↓ 否
4. synchronized锁住头结点
5. 插入链表 or 树化
6. CAS更新计数
```

**回答示例**：

> 核心流程：
> 1. 先用CAS尝试插入空位置
> 2. 如果CAS失败（说明有冲突），用synchronized锁住头结点
> 3. 插入后检查链表长度，超过8且数组>=64则转红黑树
> 4. 最后用addCount更新元素计数（LongAdder实现）

###### 12. HashMap是线程安全的吗？有什么问题？——高频面试引导问题

**HashMap并发问题**：

| 问题 | 原因 | 后果 |
|------|------|------|
| 数据丢失 | 多线程同时put，resize时数据覆盖 | 丢数据 |
| 死循环 | 1.7头插法 + 并发resize | CPU 100% |
| 可见性 | 无volatile | 其他线程看不到最新值 |

**回答示例**：

> HashMap不是线程安全的，并发场景下：
> 1. 数据覆盖：多个线程同时往同一个桶放数据，后写的覆盖前面的
> 2. 死循环：JDK 1.7的头插法+并发resize会导致环形链表，get时死循环
> 3. 扩容丢数据：resize时数据迁移可能丢失
>
> 追问：并发场景用什么Map？
> - 高并发读：ConcurrentHashMap
> - 强一致性：Hashtable / Collections.synchronizedMap
> - 计数统计：ConcurrentHashMap + LongAdder

###### 13. CopyOnWriteArrayList适合什么场景？为什么？——高频面试引导问题

**特点**：

| 特性 | 说明 |
|------|------|
| 读无锁 | 读操作完全无锁，性能高 |
| 写时复制 | 每次写都复制新数组，内存开销大 |
| 弱一致性 | 迭代器遍历的是快照，不保证最新 |

**回答示例**：

> 适合场景：
> - **读多写少**：如配置中心、黑名单、白名单
> - **迭代频繁**：迭代时不会抛ConcurrentModificationException
> - **不关心实时性**：能接受短暂的数据不一致
>
> 不适合：
> - 写多读少：每次写都复制数组，内存压力大
> - 数据量大：复制成本太高
> - 实时性要求高：读到的可能是旧数据
