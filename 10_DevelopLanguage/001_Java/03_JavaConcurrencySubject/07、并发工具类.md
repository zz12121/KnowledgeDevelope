###### 1. CountDownLatch 的作用和使用场景是什么？
[[../../../20_JavaKnowledge/03_并发编程/09、并发工具类（CountDownLatch-CyclicBarrier-Semaphore）#一、CountDownLatch|📖]]
`CountDownLatch` 就是一个**倒计时门闩**。构造时给一个初始计数 N，每个子线程完成任务后调用 `countDown()` 让计数减 1，等待方调用 `await()` 阻塞直到计数归零才继续。

典型场景：**主线程等待所有子任务完成**。比如启动时需要等 N 个服务都初始化完成，或者并发压测时让所有线程都准备好再同时开始。

有个重要限制：**一次性的，用完不能重置**。计数到 0 就永远是 0 了，不能复用。

###### 2. CyclicBarrier 的作用和使用场景是什么？
[[../../../20_JavaKnowledge/03_并发编程/09、并发工具类（CountDownLatch-CyclicBarrier-Semaphore）#二、CyclicBarrier|📖]]
`CyclicBarrier` 是**循环屏障**，让一批线程彼此等待，等所有人都到达"集合点"（屏障）后，大家再一起继续下一步。

构造时可以指定一个 `barrierAction`（Runnable），当所有线程都到达屏障后，由最后一个到达的线程来执行这个汇总操作。

典型场景：**多阶段并行计算**。比如把一个大矩阵分给多个线程分块计算，第一阶段各自算完后在屏障汇合做归并，再开始第二阶段。

**可以反复使用（Cyclic）**，一轮结束后自动重置，可以开始下一轮。

###### 3. CountDownLatch 和 CyclicBarrier 有什么区别？
[[../../../20_JavaKnowledge/03_并发编程/09、并发工具类（CountDownLatch-CyclicBarrier-Semaphore）#三、Semaphore|📖]]
两者都用于线程间同步等待，但方向正好相反：

**CountDownLatch** 是"等别人完成"——通常是一个或少数几个线程等待一批工作线程干完活。计数只能减不能加，一次性。

**CyclicBarrier** 是"大家互相等"——一批线程彼此等待，都到了再一起出发。可以循环复用，每轮结束自动重置。还支持指定一个屏障动作。

记忆口诀：**CountDownLatch 是"一等多"，CyclicBarrier 是"多等多"。**

###### 4. Semaphore 的作用和使用场景是什么？
[[../../../20_JavaKnowledge/03_并发编程/09、并发工具类（CountDownLatch-CyclicBarrier-Semaphore）#四、三者对比|📖]]
`Semaphore` 是**信号量**，通过控制"许可证数量"来限制同时访问某个资源的线程数，本质是**限流工具**。

用法：`acquire()` 获取一个许可（没有就阻塞等待），用完资源后 `release()` 归还许可。

典型场景：**数据库连接池限制并发数**（比如连接池只有 10 个连接，用 `new Semaphore(10)` 控制最多 10 个线程同时获取连接）；**接口限流**（控制同时处理的请求数）。

###### 5. Exchanger 的作用是什么？
[[../../../20_JavaKnowledge/03_并发编程/10、Fork-Join框架#一、Fork-Join 概述|📖]]
`Exchanger` 专门用于**两个线程之间交换数据**。当两个线程都调用 `exchange()` 方法时，它们会互相把各自的数据传给对方，然后都继续执行。如果只有一个线程调用，它会一直阻塞等待另一个。

场景：遗传算法中两个种群交换染色体、双人数据校对等。

###### 6. Phaser 和 CyclicBarrier 有什么区别？
[[../../../20_JavaKnowledge/03_并发编程/11、CompletableFuture异步编程#一、Future 的缺陷 → CompletableFuture 的诞生|📖]]
`Phaser` 是 JDK 7 引入的，可以看作是功能更强的 CyclicBarrier：

**动态参与者**：`CyclicBarrier` 的参与线程数在构造时固定；`Phaser` 允许运行时**动态注册和注销**参与者（`register()`/`arriveAndDeregister()`），适合参与方数量不确定的场景。

**多阶段支持**：`Phaser` 有明确的"相位"（phase）概念，每完成一轮同步进入下一个相位，可以精确控制每个阶段的行为，支持树形结构降低大规模并发的同步开销。

`Phaser` 更灵活强大，但用法也更复杂。一般场景用 `CyclicBarrier` 就够了。

###### 7. CompletableFuture 的作用和核心优势是什么？
[[../../../20_JavaKnowledge/03_并发编程/13、ThreadLocal深度解析#一、ThreadLocal 是什么|📖]]
`CompletableFuture` 是 JDK 8 的**异步编程框架**，解决了 `Future` 接口只能阻塞等待结果、无法组合多个异步任务的缺陷。

核心优势：
- **链式调用**：`thenApply`/`thenAccept`/`thenRun` 把多个异步操作串成流水线，不需要嵌套回调
- **组合操作**：`thenCombine` 等待两个任务都完成，`allOf` 等待所有任务，`anyOf` 等任意一个完成
- **异常处理**：`exceptionally` 捕获链路中的异常，`handle` 无论成功失败都能处理
- **非阻塞**：整个链路都是异步的，不需要用 `get()` 阻塞线程

```java
CompletableFuture.supplyAsync(() -> fetchUserInfo(userId))
    .thenApplyAsync(user -> enrichWithOrders(user))
    .thenAcceptAsync(result -> saveToCache(result))
    .exceptionally(e -> { log.error("Failed", e); return null; });
```

###### 8. Future 和 CompletableFuture 的区别是什么？
[[../../../20_JavaKnowledge/03_并发编程/13、ThreadLocal深度解析#四、内存泄漏问题（高频面试题！）|📖]]
**`Future`（JDK 5）** 的局限性很明显：`get()` 会阻塞当前线程，必须同步等待结果；没有回调机制，无法在任务完成时自动触发后续逻辑；多个 `Future` 之间无法方便地组合（等所有完成、等任意一个完成都需要手写逻辑）；异常处理不便。

**`CompletableFuture`（JDK 8）** 全部解决了这些问题：支持回调（不需要阻塞等结果）；支持链式调用和组合；内置丰富的异常处理；可以手动 `complete()` 设置结果。

一句话：**`Future` 是被动等待，`CompletableFuture` 是主动编排。**

###### 9. CompletableFuture 的常用 API 怎么区分？
[[../../../20_JavaKnowledge/03_并发编程/09、并发工具类（CountDownLatch-CyclicBarrier-Semaphore）#一、CountDownLatch|📖]]
记住几个核心方法的规律：

- **`thenApply`**：有入参有返回值（转换结果），像 `map`
- **`thenAccept`**：有入参无返回值（消费结果）
- **`thenRun`**：无入参无返回值（执行后续动作，不关心前一步结果）
- 带 **`Async`** 后缀的（如 `thenApplyAsync`）：在新线程里执行，不阻塞当前线程
- **`thenCompose`**：把两个有依赖关系的 `CompletableFuture` 串联（前一个的结果作为后一个的输入），避免嵌套
- **`thenCombine`**：合并两个**独立**的 `CompletableFuture` 的结果
- **`allOf`**：等所有完成；**`anyOf`**：等任意一个完成

###### 10. CompletableFuture 如何处理异常？
[[../../../20_JavaKnowledge/03_并发编程/09、并发工具类（CountDownLatch-CyclicBarrier-Semaphore）#一、CountDownLatch|📖]]
有三种异常处理方式：

**`exceptionally(fn)`**：只在发生异常时触发，可以返回一个默认值，类似 try-catch。

**`handle(fn)`**：无论成功还是失败都会触发，fn 接收结果和异常两个参数（正常时异常为 null，异常时结果为 null）。

**`whenComplete(fn)`**：也是无论成功失败都触发，但不能改变结果或异常（只能消费，不能转换）。

注意：异常如果没有被处理，会在最终 `get()` 时以 `ExecutionException` 的形式抛出。
