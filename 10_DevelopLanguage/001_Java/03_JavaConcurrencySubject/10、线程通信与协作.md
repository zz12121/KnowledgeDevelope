###### 1. 线程间通信的方式有哪些？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#七、synchronized 与 wait/notify 协作模式|📖]]
线程间通信主要解决两类问题：协调执行顺序，以及安全共享数据。常用的方式有以下几种：

**等待/通知（wait/notify）**：最基础的机制，线程在条件不满足时主动 `wait()` 释放锁进入等待，条件就绪后由其他线程 `notify()`/`notifyAll()` 唤醒。经典的生产者-消费者模型就是这么做的。

**Lock + Condition**：`ReentrantLock` 配合 `Condition` 实现，比 `wait/notify` 更灵活——一把锁可以绑定多个 `Condition`，能精确唤醒特定条件的等待线程，不需要 `notifyAll` 唤醒所有人。

**并发工具类**：`CountDownLatch`（倒计时等待）、`CyclicBarrier`（循环屏障）、`Semaphore`（信号量限流）、`Exchanger`（两线程交换数据）等，把常见的协作模式直接封装好了。

**线程安全容器**：`BlockingQueue` 把生产者-消费者的同步细节全藏进去了，直接用 `put/take` 就行，最简洁实用。

**ThreadLocal**：严格来说不是"通信"，而是数据隔离——每个线程有自己的变量副本，避免共享冲突，用于传递线程上下文（如用户 ID、数据库连接）。

###### 2. wait()、notify() 和 notifyAll() 怎么用？有什么注意事项？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#七、synchronized 与 wait/notify 协作模式|📖]]
三个方法的作用：
- `wait()`：当前线程释放锁，进入等待，直到被唤醒
- `notify()`：随机唤醒一个在该对象上等待的线程
- `notifyAll()`：唤醒所有在该对象上等待的线程

**标准使用范式**（等待方必须用 while）：
```java
// 等待方
synchronized (lock) {
    while (!condition) {  // 必须是 while，不能是 if！
        lock.wait();
    }
    doWork();
}

// 通知方
synchronized (lock) {
    changeCondition();
    lock.notifyAll();
}
```

**关键注意点**：
1. 这三个方法**必须在 synchronized 块内调用**，否则抛 `IllegalMonitorStateException`
2. 等待条件必须用 **while 循环**，不能用 if——因为存在"虚假唤醒"（OS 可能无故唤醒线程），也因为多消费者时被唤醒后条件可能已被别人消耗
3. 一般用 **notifyAll() 而不是 notify()**，除非所有等待线程的条件完全相同（否则 notify 可能唤醒了错误的线程）

###### 3. 为什么 wait() 和 notify() 必须在 synchronized 块内调用？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#七、synchronized 与 wait/notify 协作模式|📖]]
两个原因：

**防止"信号丢失"（Lost Wakeup）**：线程 A 检查条件不满足，准备调用 `wait()` 的瞬间，线程 B 修改了条件并调用了 `notify()`，B 的通知在 A `wait()` 之前就发出了，A 随后进入等待，永远等不到通知了。同步块保证了"检查条件"和"进入等待"是原子的，不会被打断。

**逻辑要求**：`wait()` 需要先释放锁（不持有锁怎么释放），`notify()` 需要持有锁才能通知。JVM 强制要求调用这些方法前必须持有该对象的监视器锁，否则抛异常。

###### 4. notify() 和 notifyAll() 的区别是什么？什么时候用哪个？
[[../../../20_JavaKnowledge/03_并发编程/03、synchronized深度解析#七、synchronized 与 wait/notify 协作模式|📖]]
`notify()` 只随机唤醒**一个**等待线程，开销小，但有风险——如果唤醒的那个线程条件不满足（比如多个消费者等待队列非空，但生产的数据类型不匹配），它会重新等待，而真正需要唤醒的线程一直等着，可能造成"信号丢失"甚至死锁。

`notifyAll()` 唤醒**所有**等待线程，都去竞争锁，只有条件满足的那个能继续，其他的重新等待。虽然多了些竞争开销，但更安全。

**选择原则**：除非你能确定所有等待线程的条件完全相同（任意唤醒哪个都可以），否则**默认用 notifyAll()**。

###### 5. 如何实现生产者-消费者模式？
[[../../../20_JavaKnowledge/02_集合框架/10、并发集合全览（CopyOnWriteArrayList等）#四、阻塞队列（BlockingQueue）|📖]]
有两种主流实现：

**方式一：BlockingQueue（推荐）**
直接用现成的，生产者 `put()`（满了阻塞），消费者 `take()`（空了阻塞），同步都在内部处理好了：
```java
BlockingQueue<String> queue = new LinkedBlockingQueue<>(10);
// 生产者线程
queue.put("item");  // 阻塞直到有位置
// 消费者线程  
String item = queue.take();  // 阻塞直到有数据
```

**方式二：wait/notifyAll（理解原理用）**
手写控制逻辑，需要注意 while 循环和 notifyAll()，适合理解底层原理，实际开发优先用 BlockingQueue。

###### 6. ThreadLocal 的作用和原理是什么？
[[../../../20_JavaKnowledge/03_并发编程/13、ThreadLocal深度解析#一、ThreadLocal 是什么|📖]]
`ThreadLocal` 提供**线程本地变量**，让每个线程都有自己独立的变量副本，互不干扰，彻底避免了多线程访问共享变量的同步问题。

**底层结构**：每个 `Thread` 对象内部都有一个 `ThreadLocalMap`（key 是 `ThreadLocal` 实例本身，value 是该线程存的值）。调用 `threadLocal.set(value)` 时，实际上是把 value 存到当前线程的 `ThreadLocalMap` 里；调用 `get()` 时从当前线程自己的 map 里取。

**内存泄漏风险**：`ThreadLocalMap` 的 key 是 **WeakReference（弱引用）**。如果 `ThreadLocal` 对象没有外部强引用了，GC 时会把 key 回收掉，变成 null key，但 value 是强引用，不会被回收，留在线程的 map 里无法访问也无法清除——**如果这个线程是线程池里的长期存活线程，这些 value 就会一直占着内存，形成内存泄漏**。

**防范方法**：用完 `ThreadLocal` 必须在 `finally` 块里调用 `remove()` 清除，尤其是在线程池环境下。

###### 7. ThreadLocal 的典型使用场景有哪些？
[[../../../20_JavaKnowledge/03_并发编程/13、ThreadLocal深度解析#七、ThreadLocal vs synchronized|📖]]
**数据库连接/Session 管理**：在同一个请求的整个处理链路里，把 `Connection` 或 `Session` 存在 `ThreadLocal` 中，Service、DAO 层都能取到，而不用层层传参，也保证了同一个线程用的是同一个连接（事务保证）。

**用户身份信息传递**：拦截器解析 JWT 后把用户 ID 存入 `ThreadLocal`，后续的业务逻辑直接从 `ThreadLocal` 取，不用在方法参数里层层传递。

**SimpleDateFormat 线程安全问题**：`SimpleDateFormat` 非线程安全，通过 `ThreadLocal<SimpleDateFormat>` 每个线程维护自己的实例，既安全又避免了每次 new 对象的开销。
