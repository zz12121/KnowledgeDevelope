###### 1. 什么是 Java 内存模型（JMM）？
[[../../../20_JavaKnowledge/03_并发编程/02、Java内存模型（JMM）#一、为什么需要 JMM？|📖]]
JMM（Java Memory Model）是一套**规范**，不是真实存在的内存结构。它的目的是屏蔽不同 CPU 和操作系统的内存访问差异，告诉我们：**在多线程环境下，什么时候一个线程的写操作对另一个线程是可见的，以及代码的执行顺序是否能被保证。**

JMM 规定每个线程都有自己的**工作内存**（类比 CPU 缓存），共享变量存在**主内存**里。线程只能操作工作内存里的副本，操作完成后才同步回主内存。这种设计提升了性能，但也带来了**可见性**和**有序性**问题。

###### 2. JMM 中的主内存和工作内存是什么？
[[../../../20_JavaKnowledge/03_并发编程/02、Java内存模型（JMM）#二、主内存与工作内存|📖]]
**主内存**：所有线程共享，存放共享变量（实例字段、静态字段、数组元素）。

**工作内存**：每个线程私有，存放该线程用到的共享变量的**副本**。线程的所有操作都在工作内存里进行，不能直接读写主内存的变量。

注意这和 JVM 内存区域（堆、栈）是不同层次的概念：JMM 是抽象的并发内存模型，从物理角度看主内存对应硬件内存，工作内存对应 CPU 缓存和寄存器。

###### 3. 什么是内存可见性问题？怎么解决？
[[../../../20_JavaKnowledge/03_并发编程/02、Java内存模型（JMM）#三、三大特性|📖]]
可见性问题的根源是：线程 A 改了共享变量写回了自己的工作内存，但还没同步到主内存，或者线程 B 读的是工作内存里的旧副本，没有从主内存刷新。结果就是 B 看不到 A 的修改。

解决方案：
- **`volatile`**：写操作立刻刷回主内存，读操作强制从主内存取，保证可见性
- **`synchronized`**：解锁前强制将修改刷回主内存，加锁后强制从主内存加载，happens-before 保证可见性
- **`final`**：对象构造完成后，final 字段对其他线程可见，无需同步
- **显式内存屏障**：框架层面直接控制，一般业务代码用不到

###### 4. 什么是指令重排序？为什么会有指令重排序？
[[../../../20_JavaKnowledge/03_并发编程/02、Java内存模型（JMM）#四、happens-before 规则|📖]]
为了提高 CPU 执行效率，编译器和处理器可以对没有数据依赖关系的指令**重新排序执行**。单线程下结果不变（as-if-serial 语义），但多线程下可能破坏预期的执行顺序。

三个层面都可能发生重排序：编译器优化重排、指令级并行重排（CPU 流水线）、内存系统重排（Store Buffer 导致写操作延迟可见）。

典型案例：单例的 DCL 模式里，`new Singleton()` 的三步（分配内存→初始化→赋引用）可能被重排成（分配内存→赋引用→初始化），导致其他线程拿到未完全初始化的对象。

###### 5. 什么是 happens-before 原则？
[[../../../20_JavaKnowledge/03_并发编程/02、Java内存模型（JMM）#五、内存屏障（Memory Barrier）|📖]]
happens-before 是 JMM 的**核心可见性规则**：如果 A happens-before B，那么 A 的操作结果对 B 一定可见，而且 A 的执行顺序在 B 之前。

有八大规则，最常考的六个：

**程序顺序规则**：单线程内，前面的操作 happens-before 后面的操作（但注意这只是可见性保证，不是说不会重排序，只是重排后结果要和顺序执行一样）。

**volatile 规则**：volatile 写 happens-before 后续的 volatile 读。

**监视器规则（synchronized）**：unlock happens-before 后续的 lock。

**线程启动规则**：`Thread.start()` happens-before 该线程里的每一个操作。

**线程终止规则**：线程的所有操作 happens-before `Thread.join()` 返回。

**传递性**：A hb B，B hb C，则 A hb C。这个传递性非常强大，可以通过 volatile 变量"传递"普通变量的可见性。

###### 6. final 关键字的内存语义是什么？
[[../../../20_JavaKnowledge/03_并发编程/04、volatile深度解析#三、禁止指令重排序|📖]]
正确构造对象（构造函数里没有 this 引用逸出）的情况下，JMM 保证：

**写保证**：构造函数里对 final 字段的写操作，不会被重排序到构造函数之外。

**读保证**：其他线程通过正常引用访问这个对象时，一定能看到 final 字段的初始化值，不需要加任何同步。

这就是为什么不可变对象（所有字段都是 final）天然线程安全——只要对象引用被安全发布，字段值就一定是正确的。

###### 7. 什么是伪共享？如何避免？
[[../../../20_JavaKnowledge/03_并发编程/02、Java内存模型（JMM）#三、三大特性|📖]]
现代 CPU 以**缓存行**（64 字节）为单位加载数据。如果两个线程各自频繁修改的两个不相关变量恰好落在同一个缓存行里，即使它们逻辑上完全无关，一个线程修改变量时也会让另一个线程的整个缓存行失效，被迫重新从内存加载。两个无关变量互相拖累对方，这就是**伪共享（False Sharing）**，是多线程程序的性能杀手。

解决方法：
- **JDK 8+ 用 `@sun.misc.Contended` 注解**：JVM 自动在变量两侧填充 128 字节，使其独占缓存行，这是推荐做法
- **手动填充**：在变量旁边声明几个 padding 字段（不够优雅，但能起作用）

`LongAdder` 的 `Cell` 数组就用了 `@Contended`，这是它高并发性能好的原因之一。

###### 8. JMM 的 MESI 协议是什么？
[[../../../20_JavaKnowledge/03_并发编程/02、Java内存模型（JMM）#一、为什么需要 JMM？|📖]]
MESI 是 Intel 等 CPU 的**缓存一致性协议**，保证多核 CPU 缓存里的数据一致。

四种状态：
- **M（Modified）**：此缓存行数据被修改了，只在本核心缓存里，还没写回主内存
- **E（Exclusive）**：此缓存行只在本核心，数据和主内存一致（干净）
- **S（Shared）**：此缓存行在多个核心的缓存里都有，数据一致
- **I（Invalid）**：此缓存行无效，下次访问需要从主内存重新加载

当一个核心写入变量时，MESI 通过总线嗅探把其他核心里的同一缓存行标记为 Invalid，这就是 `volatile` 写操作强制可见的硬件基础。
