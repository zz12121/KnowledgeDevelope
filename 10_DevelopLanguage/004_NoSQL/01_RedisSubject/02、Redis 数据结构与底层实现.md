###### 1. Redis 的 String 类型底层是如何实现的?
Redis 的 String 类型通过 **redisObject 对象系统**进行封装，根据存储内容动态选择最优编码方式，包含三种底层实现：

**int 编码**：当字符串值可以表示为 64 位有符号整数时，Redis 直接将整数值存储在 redisObject 的 ptr 指针位置（刚好 8 字节），不再需要额外的 SDS 结构，完全避免了额外内存开销。

**embstr 编码**：针对短字符串（≤44 字节）的优化编码，将 redisObject 和 SDS 结构在内存中连续分配，只需一次内存分配操作，内存局部性更好。但因为是只读的，任何修改操作都会导致其转换为 raw 编码。

**raw 编码**：长字符串（>44 字节）的标准编码方式，redisObject 和 SDS 结构分开存储，支持动态修改。

**44 字节阈值的由来**：jemalloc 的最小内存分配单元是 64 字节，64 - 16（redisObject 大小）- 3（sdshdr8 结构）- 1（`\0`）= 44 字节，刚好能装进一个 64 字节的内存块里，不浪费。

###### 2. Redis 的 List 类型底层实现原理?
Redis 的 List 类型经历了显著的结构演进，当前核心实现是 **quicklist**，它平衡了内存效率和操作性能。

**quicklist 架构**：是一个双向链表，每个链表节点都是一个 ziplist（Redis 7.0+ 换成了 listpack）。这种设计把大列表分割成多个小块，有效控制了连锁更新的影响范围。

**历史演进**：Redis 3.2 之前分别用 linkedlist（内存开销大、碎片多）和 ziplist（内存紧凑但连锁更新有风险）两种结构，3.2 版本引入 quicklist 把两者的优点结合起来。Redis 7.0 把内部的 ziplist 替换为 listpack，彻底解决了连锁更新问题。

**listpack 的改进**：每个 entry 只记录自身的长度，不再记录前一个 entry 的长度，这样修改某个 entry 时就不会触发后续节点的连锁更新。

###### 3. Redis 的 Hash 类型底层实现原理?
Redis Hash 类型根据数据特征在 **listpack（ziplist）和 hashtable** 两种编码间动态切换。

**listpack 编码**（小数据时用）：所有键值对连续存储在 listpack 里，键和值相邻排列，内存非常紧凑，但查询效率是 O(N)，适合元素少且键值短的场景。

**hashtable 编码**（数据量大时用）：使用字典结构，查询 O(1)，字典内部有两张哈希表用于渐进式 rehash，通过链地址法解决哈希冲突。

**切换条件**（以下任一不满足则转 hashtable）：所有键值对的键和值字符串长度都小于 64 字节，且键值对数量小于 512 个（这两个阈值可通过 `hash-max-listpack-entries` 和 `hash-max-listpack-value` 配置）。注意编码升级是单向的，不会降级回去。

###### 4. Redis 的 Set 类型底层实现原理?
Set 类型根据元素内容和数量在 **intset 和 hashtable** 两种编码间切换。

**intset 编码**（纯整数时用）：结构包含编码类型、长度和整数数组，元素按升序排列，支持二分查找。内存密度极高，适合全是整数的小集合。

**hashtable 编码**（通用）：字典的 key 存储 Set 元素，value 设为 NULL。支持 O(1) 的增删查，但内存开销相对大。

**切换时机**：添加了非整数元素，或者元素数量超过 `set-max-intset-entries`（默认 512），就会从 intset 升级为 hashtable，同样不可逆。

###### 5. Redis 的 Sorted Set (ZSet) 底层实现原理?
ZSet 通过 **listpack（小数据）和 skiplist+dict 组合（大数据）** 两种结构实现。

**listpack 编码**（小数据时用）：元素和分值相邻存储，按分值排序，内存紧凑，但随数据量增长性能下降。

**skiplist + dict 组合**是 ZSet 的核心实现，也是最有意思的设计：同时维护一个跳跃表和一个字典，两个结构**共享同一份数据**（通过指针引用，不复制），不会浪费额外内存。

为什么要两个结构？跳跃表负责按分值排序，支持 O(logN) 的范围查询（`ZRANGE`、`ZRANK` 等）；字典负责 O(1) 的成员查找（`ZSCORE`、`ZRANK` 的点查询）。单独用跳跃表，点查询是 O(logN)；单独用字典，范围查询和排序要 O(N logN)。组合起来，两种操作都是最优复杂度。

**切换条件**：元素数量超过 `zset-max-listpack-entries`（默认 128）或任意成员长度超过 `zset-max-listpack-value`（默认 64 字节）时，转为 skiplist+dict 编码。

###### 6. 什么是 SDS (Simple Dynamic String)?
SDS 是 Redis 设计的**二进制安全动态字符串**，解决了 C 字符串的多个固有缺陷。

**核心数据结构**：有 sdshdr8/16/32/64 四种 header，根据字符串长度自动选择最小的 header 类型，加上一个柔性字节数组存实际内容。

**相比 C 字符串的优势**：
- **O(1) 获取长度**：header 里直接存了 len，`strlen` 是 O(N)，SDS 是 O(1)
- **杜绝缓冲区溢出**：修改前自动检查空间并扩容
- **二进制安全**：不用 `\0` 标识结尾，可以存储含 `\0` 的二进制数据
- **空间预分配**：长度 <1MB 时预分配双倍空间，>=1MB 时每次多分配 1MB，减少频繁扩容的内存重分配次数
- **惰性空间释放**：字符串缩短时不立即归还多余空间，留着备用

###### 7. 什么是跳跃表 (Skip List)?
跳跃表是**有序数据结构**，通过维护多级索引实现高效查询，Redis 将其作为 Sorted Set 的底层实现之一。

**结构原理**：底层是一个有序链表，在它之上建立多级"快速通道"，高层节点稀疏、跨度大，低层节点密集、跨度小。查找时从高层开始，快速跳过大段不匹配的区间，最终在底层精确定位，平均时间复杂度 O(logN)。

**层高是随机的**：每个节点的层数通过概率算法生成，期望层数是 O(logN)，最大层数是 32。正是这种随机性，使跳跃表无需维护严格的平衡条件，实现比红黑树简单得多。

**Redis 跳跃表节点的特殊设计**：除了正向的多级前进指针外，底层还有一个**后退指针**，方便逆序遍历（实现 `ZREVRANGE` 等命令）。每个节点还记录了到下一个节点的**跨度（span）**，方便 O(logN) 时间内计算排名。

**为什么不用红黑树**？跳跃表在范围查询时比红黑树更友好——找到起始节点后直接顺序遍历即可，不需要中序遍历；实现简单，代码量少；另外 Redis 作者本人表示他觉得跳跃表更容易理解和调试。

###### 8. Redis 的压缩列表 (ziplist) 是什么?
ziplist 是**内存紧凑型顺序结构**，为小规模数据提供高内存密度存储。

**整体结构**：依次包含 zlbytes（总字节数）、zltail（尾节点偏移量）、zllen（节点数量）、entries（节点列表）、zlend（结束标志 `0xFF`）。不使用指针，只靠偏移量定位，内存连续。

**entry 结构**：每个 entry 包含 prevlen（前驱节点长度）、encoding（编码类型）和 entry-data（实际数据）。prevlen 变长编码：前驱节点 <254 字节时用 1 字节存，>=254 字节用 5 字节存。

**连锁更新问题（cascade update）**：正是因为 prevlen 的变长编码，当插入或删除某个节点导致其长度变化时，后续节点的 prevlen 字段可能都需要扩展，引发连锁的内存重分配。最坏情况 O(N²)，这就是为什么 ziplist 只用于小数据量场景，以及 Redis 7.0 为什么要用 listpack 替代它。

###### 9. Redis 的快速列表 (quicklist) 是什么?
quicklist 是 **ziplist 的双向链表**，平衡了内存效率和操作性能，是 List 类型的底层实现。

**结构设计**：每个链表节点（quicklistNode）包含前驱指针、后继指针、指向 ziplist（或 listpack）的指针，以及节点的大小、元素个数等元数据。

**关键控制参数**：
- `list-max-listpack-size`（原 `list-max-ziplist-size`）：控制每个节点的最大大小，可以是字节数（负值）或元素个数（正值），默认 -2 表示每个节点不超过 8KB
- `list-compress-depth`：两端不压缩的节点数，中间节点用 LZF 压缩，默认 0（不压缩）

quicklist 限制了每个 ziplist 节点的大小，即使发生连锁更新也只影响一个小节点内部，不会蔓延到整个链表。

###### 10. Redis 的字典 (dict) 是如何实现的?
Redis 字典采用**哈希表实现**，通过链地址法解决冲突，支持渐进式 rehash 确保高性能。

**核心结构**：字典包含两个哈希表 ht[0] 和 ht[1]，平时只用 ht[0]，rehash 时才用 ht[1]。每个哈希表有一个桶数组，每个桶存储 dictEntry 链表。

**渐进式 rehash**：当负载因子（used/size）超过阈值（默认 1，BGSAVE 期间是 5）时触发 rehash。关键点在于不是一次性把 ht[0] 所有数据搬到 ht[1]，而是**每次命令操作时顺带迁移一个桶**，分摊开销，避免单次大批量迁移导致的性能抖动。

rehash 期间：查找同时查 ht[0] 和 ht[1]；新插入的 key 直接写入 ht[1]，保证 ht[0] 只减不增。全部迁移完成后，ht[1] 变成新的 ht[0]，旧的 ht[0] 释放。

###### 11. Redis 的整数集合 (intset) 是什么?
intset 是**纯整数集合的高效存储结构**，当 Set 类型的元素均为整数且数量不多时使用。

**结构**：包含编码类型（encoding）、元素数量（length）和整数数组（contents[]）。元素在数组中**升序排列**，查找用二分查找，O(logN)。

**编码升级机制**：intset 支持 int16、int32、int64 三种编码。当添加一个超出当前范围的整数时（比如 int16 范围是 -32768~32767，加了一个 100000），整个 intset 升级为更宽的编码，所有元素重新转换格式。升级是**不可逆的**，不会因为删除了那个大整数而降级。

这种结构最大的优点是内存极其紧凑，没有指针开销，比 hashtable 节省几倍内存。

###### 12. Redis 对象的类型和编码是什么关系?
Redis 的**类型（type）和编码（encoding）是解耦的**，这种设计提供了极大的灵活性和优化空间。

**类型（type）定义对外接口**：string、list、hash、set、zset 五种，决定了用户可以对这个 key 执行哪些命令（比如 String 类型就不能执行 LPUSH）。

**编码（encoding）决定内部实现**：同一种类型可以有多种编码。比如 String 有 int/embstr/raw 三种；ZSet 有 listpack 和 skiplist+dict 两种。Redis 会根据数据特征自动选择并在必要时升级编码，对用户完全透明。

这种解耦让 Redis 可以针对小数据用内存紧凑的编码，数据量大了再升级到高性能编码，兼顾了内存效率和操作性能。

```bash
# 查看某个 key 的当前编码
OBJECT ENCODING mykey
```
