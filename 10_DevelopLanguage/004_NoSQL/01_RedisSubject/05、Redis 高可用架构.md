###### 1. Redis 的主从复制原理是什么?
Redis 主从复制是一种**异步的数据同步机制**，核心目标是实现数据冗余、故障恢复和读写分离。整个过程分三个阶段：连接建立、数据同步和命令传播。

**第一阶段：连接建立**

从节点通过 `replicaof <masterip> <masterport>` 配置主节点地址，内部定时任务检测到后向主节点发起 Socket 连接。连接建立后发送 PING 命令检测连通性，如果主节点设置了密码，从节点还需要用 `masterauth` 配置的密码发送 AUTH 命令验证身份。

**第二阶段：数据同步（PSYNC 命令）**

Redis 2.8+ 使用 `PSYNC` 命令，支持**全量复制**和**部分复制**两种模式：

**全量复制**（首次连接或偏移量不在缓冲区内时触发）：从节点发送 `PSYNC ? -1`，主节点 BGSAVE 生成 RDB 文件，传给从节点；从节点清空自身数据，加载 RDB；期间主节点新写命令缓存在复制客户端缓冲区，RDB 加载完后补传给从节点。

**部分复制**（断线重连时优先尝试）：依赖三个核心要素：
1. **复制偏移量**：主从各自维护，偏移量一致表示数据同步
2. **复制积压缓冲区**：主节点维护的固定大小环形队列（默认 1MB），存储最近传播的写命令；重连时，如果从节点的偏移量还在这个缓冲区里，就可以只补发缺失部分，不需要全量复制
3. **Run ID**：Redis 实例唯一标识，重连时如果 Run ID 不匹配（主节点重启了），必须全量复制

**第三阶段：命令传播**

同步完成后，主节点每执行一个写命令，就**异步**广播给所有从节点。注意是异步的，主节点不等从节点确认就返回给客户端，这意味着从节点数据可能短暂滞后。

###### 2. 说一说 Redis 分布式场景下数据同步?
Redis 分布式场景下，数据同步的核心机制是**基于主从复制的异步同步**，属于**最终一致性**模型。主节点处理写命令后立即返回客户端，异步地将命令发给从节点，因此从节点数据可能短暂落后。

在网络延迟、从节点高负载或网络分区等情况下，从节点可能读到旧数据。

如果业务对一致性有要求，Redis 提供了 `WAIT` 命令：`WAIT <numreplicas> <timeout>`，阻塞当前客户端直到至少 `numreplicas` 个从节点确认复制完成，或超时返回。这是一种**弱一致性**保证，不等同于强一致性（WAIT 超时后不保证数据已复制，且 WAIT 期间可以正常服务其他请求）。

###### 3. Redis 哨兵模式的工作原理?
哨兵（Sentinel）是 Redis 的高可用方案，负责监控主从节点并在主节点故障时**自动完成故障转移**。

**监控**：每个 Sentinel 节点每秒向所有主节点、从节点和其他 Sentinel 发送 PING 命令，检测存活状态。

**主观下线（Subjectively Down）**：如果某实例在 `down-after-milliseconds` 配置的时间内没有有效响应 PING，发起检测的这个 Sentinel 就会将该实例标记为"主观下线"。这只是单个 Sentinel 的判断，可能是网络抖动造成的误判。

**客观下线（Objectively Down）**：当某 Sentinel 将主节点标记为主观下线后，会向其他 Sentinel 询问意见。如果达到法定数量（Quorum）的 Sentinel 都认为主节点下线，就标记为"客观下线"，触发故障转移流程。

**Sentinel 领导者选举（Raft 算法）**：客观下线后，多个 Sentinel 通过 Raft 算法选出一个 Leader 负责执行故障转移，避免多个 Sentinel 同时操作。

**故障转移**：Leader Sentinel 根据规则（复制偏移量最大的优先、其次 run id 字典序等）从从节点中选出最优的晋升为新主节点，发送 `REPLICAOF NO ONE` 命令，然后让其他从节点复制新主节点，最后通知客户端新的主节点地址。

###### 4. Redis 集群模式的工作原理?
Redis Cluster 是**去中心化的分片集群**方案，解决了单机内存不够和读写吞吐量不足的问题，同时内置了高可用机制。

**数据分片**：整个数据集划分为 16384 个哈希槽，每个 key 通过 `CRC16(key) % 16384` 计算所属槽，集群中每个主节点负责一部分槽。

**节点互联（Gossip 协议）**：所有节点通过 Gossip 协议互相传播集群状态（谁负责哪些槽、谁下线了等），每个节点都有完整的集群元数据，客户端连接任意节点都能访问数据。

**请求路由**：客户端发送命令后，节点计算 key 的哈希槽；如果是自己负责的槽就直接执行；如果不是，返回 `MOVED` 重定向错误，告知正确节点地址。智能客户端（如 JedisCluster）会缓存槽位映射，下次直接连正确节点。

**高可用**：每个主节点可以有一个或多个从节点，主节点故障后从节点自动选举接管，集群能自动完成故障转移，无需外部哨兵。

> 📖 [[../../../23_RedisKnowledge/05_高可用架构/01、主从复制与高可用#5-集群模式|知识库：集群模式原理]]

###### 5. Redis 集群的数据分片方式?
Redis Cluster 采用**虚拟哈希槽分区**，而非一致性哈希。

**固定 16384 个槽**，key 的分片算法是 `HASH_SLOT = CRC16(key) % 16384`。集群管理员手动或自动将这 16384 个槽分配给不同的主节点，比如三主集群通常是 0-5460、5461-10922、10923-16383 各一份。

**选择 16384 而不是更大的数字**：CRC16 产生 65536 种可能值，16384 个槽心跳包占用空间是 `16384/8 = 2KB`，如果用 65536 个槽则需要 8KB，心跳包太大；另外 Redis 官方认为集群节点数不会超过 1000 个，16384 个槽已经足够。

**哈希标签**：如果 key 里有 `{}` 括起来的内容，CRC16 只计算 `{}` 内的部分，比如 `user:{1000}:name` 和 `user:{1000}:email` 都会分到同一个槽，方便同一用户的数据落在同一节点，支持跨 key 的事务操作。

**扩缩容的优势**：数据与节点解耦，扩容时只需迁移部分槽到新节点，不需要重新计算所有 key 的哈希，迁移开销小，且对客户端影响很小。

###### 6. 什么是哈希槽 (Hash Slot)?
哈希槽是 Redis Cluster 进行数据分片和路由的基本单位，可以把 16384 个槽的集合理解为一个逻辑完整的数据库，物理上由多个 Redis 节点共同分担。

**关键特性**：
- 每个 key 唯一属于一个槽，一个槽可以有多个 key
- 节点的增删只会引起槽在节点间的重分布，不影响 key 到槽的计算规则
- 槽的迁移是渐进式的，迁移期间集群正常服务，迁移中的 key 会收到 ASK 重定向

**MOVED vs ASK 重定向**：`MOVED` 表示 key 永久属于那个节点（槽分配固定了），客户端应更新缓存；`ASK` 表示 key 正在迁移到那个节点，只对这次请求有效，客户端不更新缓存。

###### 7. Redis 集群如何进行故障转移?
集群故障转移是**全自动的**，内置于集群机制中，无需外部哨兵。

1. **故障检测**：节点间定期互发 PING，如果某主节点超时未响应，其他主节点将其标记为"疑似下线"，通过 Gossip 协议传播这一状态
2. **故障确认**：当大多数主节点都确认该主节点下线，集群将其标记为"已下线"，触发故障转移
3. **从节点选举**：该主节点的从节点们会竞选，请求其他主节点投票，第一个获得超过半数主节点投票的从节点当选新主节点
4. **接管**：新主节点接管原主节点的所有哈希槽，更新集群元数据，通过 Gossip 传播变更

从故障被确认到完成故障转移，通常需要几秒到十几秒。这段时间访问故障节点的槽会报错，可以通过增加从节点数量来缩短选举时间。

###### 8. Redis 主从复制的延迟问题如何解决?
主从复制延迟是异步复制的固有代价，不能完全消除，只能降低和适配。

**降低延迟**：确保主从节点网络低延迟高带宽；调大复制积压缓冲区（`repl-backlog-size`，建议设为 512MB 以上），减少断线重连时全量复制的概率；`repl-disable-tcp-nodelay` 设为 `no`（默认），让主节点尽快发送小数据包。

**监控延迟**：`INFO replication` 里对比 `master_repl_offset` 和 `slave_repl_offset`，差值就是复制延迟的字节数；也可以看 `slave_lag` 字段，单位是秒。

**业务层面适配**：对于一致性要求高的读操作，直接读主节点；对于允许短暂延迟的业务，接受从节点读；有条件的话，用 `WAIT` 命令做弱一致性保证。

###### 9. Redis 哨兵模式的脑裂问题是什么?
脑裂是网络分区时出现**多个主节点**的现象，会导致数据丢失。

**场景还原**：一主一从 + 三个 Sentinel，网络分区把主节点和 1 个 Sentinel 隔在 A 区，从节点和 2 个 Sentinel 在 B 区。B 区的 2 个 Sentinel 达到 Quorum，判定主节点客观下线，把从节点提升为新主节点。此时 A 区的旧主节点还在正常工作，客户端继续往 A 区写数据，B 区新主节点也在接收写操作。网络恢复后，旧主节点降为从节点，全量同步新主节点数据，**A 区在分区期间写入的数据全部丢失**。

**解决方案**：配置 `min-replicas-to-write` 和 `min-replicas-max-lag`：

```bash
min-replicas-to-write 1      # 必须至少有 1 个从节点在线
min-replicas-max-lag 10      # 从节点延迟不超过 10 秒
```

这样，当主节点与所有从节点失联（即 A 区被隔离的旧主节点失去从节点），它会**主动拒绝写操作**，减少脑裂期间的数据丢失。代价是可用性略有下降，但保护了数据一致性。

###### 10. Redis 集群的扩容和缩容如何操作?
Redis Cluster 的扩缩容核心是**哈希槽的重新分配**，用 `redis-cli --cluster` 工具完成。

**扩容流程**：
```bash
# 1. 启动新节点
redis-server /etc/redis/redis-new.conf

# 2. 加入集群（新节点暂时没有槽）
redis-cli --cluster add-node 新节点IP:端口 已有节点IP:端口

# 3. 槽重分配（迁移部分槽到新节点）
redis-cli --cluster reshard 已有节点IP:端口
# 按提示输入要迁移的槽数量和目标节点 ID

# 4. 为新主节点添加从节点（高可用）
redis-cli --cluster add-node 从节点IP:端口 已有节点IP:端口 --cluster-slave --cluster-master-id 新主节点ID
```

**缩容流程**：
```bash
# 1. 先把要删除节点的槽迁移到其他节点
redis-cli --cluster reshard 已有节点IP:端口

# 2. 槽迁移完成后移除节点
redis-cli --cluster del-node 已有节点IP:端口 要删除的节点ID
```

迁移过程中客户端会收到 ASK 重定向，智能客户端会自动处理，对业务几乎无感知。

###### 11. Redis 集群支持事务吗?
Redis Cluster **支持事务，但有严格限制**：事务块（`MULTI/EXEC`）内所有命令涉及的 key 必须落在同一个节点的同一个哈希槽内，因为事务要求原子性，而跨节点操作无法保证原子性。

**实现方式：哈希标签**。`{}`括起来的部分才参与哈希计算，比如：

```redis
user:{1000}:name    → 哈希 "1000" → 槽 X
user:{1000}:age     → 哈希 "1000" → 槽 X（同一槽）
user:{2000}:name    → 哈希 "2000" → 槽 Y（不同槽）
```

这样 `user:1000` 的所有字段都会落在同一个节点，可以在一个事务里操作。设计 key 的时候提前规划好哈希标签，是集群模式下保证事务可用的关键。

###### 12. Redis 的读写分离如何实现?
读写分离的核心是**写操作发往主节点，读操作分发到从节点**，提升读吞吐量。

**方式一：客户端直连**（最简单）：代码里维护两个连接池，写连接池指向主节点，读连接池指向从节点。简单直接，但需要应用层感知节点状态。

**方式二：代理中间件**（对应用透明）：引入 Twemproxy、Codis 或 ProxySQL 等中间件，应用只连接代理，代理按规则路由（写 → 主，读 → 从）。引入了额外网络跳转，有一定延迟。

**方式三：框架集成**（Java 推荐）：Spring Data Redis + Lettuce 支持读写分离配置，只需声明主从节点地址，框架自动路由，配合 Lettuce 的响应式 API 效果最好。

```java
@Bean
public RedisClient redisClient() {
    // Lettuce 配置读写分离
    ClientOptions options = ClientOptions.builder()
        .readFrom(ReadFrom.REPLICA_PREFERRED)  // 优先读从节点，不可用时读主节点
        .build();
    // ...
}
```

**注意**：读写分离有数据一致性风险，从节点数据可能滞后几毫秒到几秒，业务层必须能接受读到旧数据。
