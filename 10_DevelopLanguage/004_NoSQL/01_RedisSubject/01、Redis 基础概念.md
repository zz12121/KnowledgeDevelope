###### 1. 你觉得 Redis 是什么?
Redis 是一个**开源的内存数据结构存储系统**，核心价值在于提供高性能的键值存储与多种数据结构操作能力。从架构视角看，它不只是个缓存，可以同时扮演数据库、缓存、消息中间件三个角色。

**核心定位与设计哲学**：
- **内存优先架构**：数据主要存储在内存中，读写操作直接操作内存数据结构，这是高性能的根本。
- **数据持久化支持**：通过 RDB 快照和 AOF 日志两种方式将内存数据持久化到磁盘，保证数据安全。
- **键值存储模型**：键是字符串，值支持多种数据结构（字符串、哈希、列表等），操作语义由值的数据类型决定。

**源码视角**：在 Redis 源码中，所有数据存储在全局的 `redisDb` 结构中，其核心是哈希表，用于存储键到值的映射。值的底层通过 `redisObject` 结构体表示，其 `type` 字段标识具体数据类型（如 `REDIS_STRING`、`REDIS_LIST`），`ptr` 指针指向存储该数据类型的实际数据结构（如 SDS、字典、跳跃表等）。

###### 2. Redis 有哪些主要特点?
Redis 最核心的特点是**高性能**，读写速度可以达到 10 万次/秒以上，这得益于纯内存操作。数据结构方面，Redis 不只是简单的 key-value，它支持字符串、哈希、列表、集合、有序集合等多种数据结构，可以直接映射到业务模型，大大减少了应用层的复杂度。

持久化方面，Redis 提供 RDB 快照和 AOF 日志两种方式，前者适合快速恢复，后者保证更高的数据安全性。高可用方面，支持主从复制、哨兵模式、集群模式，能满足各种规模的生产需求。

原子性是 Redis 很重要的一个特性，所有单个命令都是原子的，再加上 `MULTI/EXEC` 事务和 Lua 脚本，并发编程场景下非常实用。此外还有发布订阅、键过期、Lua 脚本等扩展功能，适用场景非常广泛。

**源码角度（持久化举例）**：RDB 持久化通过 `rdbSave` 函数实现，它遍历数据库，将每个键值对按特定格式序列化到磁盘。AOF 持久化则在 `propagate` 函数中，将写命令追加到 AOF 缓冲区，根据 `appendfsync` 配置（always/everysec/no）决定刷盘策略。

###### 3. Redis 和 Memcached 有什么区别?
这两个是最常被拿来对比的内存缓存系统，核心区别在**数据结构和功能丰富度**上。

Redis 支持字符串、哈希、列表、集合、有序集合等多种数据类型，而 Memcached 只支持简单的字符串键值对，这意味着很多业务逻辑在 Redis 里可以直接用数据结构命令实现，Memcached 则需要把逻辑放到应用层处理。

持久化方面，Redis 提供 RDB 和 AOF 两种机制，Memcached 完全不支持持久化，数据纯内存存储，重启即丢失。

线程模型方面，Redis 6.0 之前命令处理是单线程的（通过 I/O 多路复用处理并发），避免了锁竞争，在高并发下反而更稳定；Memcached 是多线程模型，理论上能更好地利用多核 CPU，但锁竞争是个问题。

功能方面，Redis 支持发布订阅、事务、Lua 脚本等高级特性，Memcached 功能聚焦于缓存，相对简单。单个值的大小限制上，Redis 支持到 512MB，Memcached 通常限制在 1MB 以内。

**选择建议**：需要复杂数据结构、持久化或高级功能（如发布订阅）就选 Redis；纯缓存场景、值较小且无需持久化可以考虑 Memcached，但目前大多数场景 Redis 都是首选。

###### 4. Redis 支持哪些数据类型?
Redis 有五大基本数据类型，加上三种特殊类型，总共八种。

**五大基本类型**：**String** 是最基础的，底层是 SDS，支持 `SET/GET/INCR` 等，常用于缓存、计数器、分布式锁。**Hash** 底层是 listpack 或哈希表，支持 `HSET/HGET/HGETALL`，适合存储对象（如用户信息）。**List** 底层是 quicklist，支持 `LPUSH/RPOP/LRANGE`，常用于消息队列和最新动态列表。**Set** 底层是 intset 或哈希表，支持 `SADD/SMEMBERS/SINTER`，适合标签、好友关系和去重场景。**ZSet（有序集合）** 底层是跳跃表加字典的双重结构，支持 `ZADD/ZRANGE/ZREVRANK`，最典型的应用是排行榜和延迟队列。

**三种特殊类型**：**Bitmap** 本质是 String 的位操作，`SETBIT/GETBIT/BITCOUNT`，用于用户签到和活跃统计。**HyperLogLog** 基于稀疏矩阵编码，`PFADD/PFCOUNT`，用于大数据量的基数（不重复元素数）估算，约 12KB 固定内存。**GEO** 底层是 ZSet + Geohash，用于地理位置存储和范围查询。

**源码角度（ZSet 实现）**：有序集合同时使用跳跃表和字典。跳跃表支持范围查询（`ZRANGE`），字典支持按成员快速查找分数（`ZSCORE`）。这种双重结构以空间换时间，确保各操作都高效。

###### 5. Redis 为什么这么快?
Redis 快的核心原因有五个，每一个都是真实有效的加速因素，不是噱头。

**第一，纯内存操作**。数据存在内存里，读写直接操作内存数据结构，内存访问速度是磁盘的几个数量级，这是最根本的原因。

**第二，单线程避免锁竞争**。命令处理是单线程的，不需要加锁，没有上下文切换开销，代码执行路径简单直接。很多人以为单线程是缺点，其实恰恰是 Redis 高性能的关键设计。

**第三，I/O 多路复用**。虽然是单线程处理命令，但用 epoll（Linux）/kqueue（macOS）实现了 I/O 多路复用，单个线程可以同时监听成千上万个客户端连接，不会因为等待网络 I/O 而阻塞。

**第四，高效的数据结构**。SDS 获取长度是 O(1)，跳跃表范围查询是 O(logN)，字典查找是 O(1)，这些都是专门为 Redis 场景优化过的实现。

**第五，RESP 协议简单高效**。客户端和服务端之间用的是 Redis 自定义的 RESP 协议，格式简单，解析快，序列化/反序列化开销小。

**源码角度（事件循环）**：Redis 启动后进入 `aeMain` 函数的事件循环（`ae.c`）。它通过 `aeApiPoll`（封装 epoll 等）监听所有套接字的事件。当客户端命令到达，文件事件处理器将命令读入缓冲区，解析后调用对应命令函数（如 `setCommand`）执行，最后将结果写回客户端。整个过程在单线程内顺序执行，高效且无锁。

###### 6. Redis 的应用场景有哪些?
Redis 的典型场景其实和它的数据结构一一对应，理解了数据结构就理解了应用场景。

**缓存**是最常见的场景，把数据库查询结果、热点数据存入 Redis，设置合理的过期时间，能极大降低数据库压力，注意要处理好缓存穿透、击穿、雪崩三个问题。

**分布式 Session** 用 Hash 结构存储用户 Session，所有服务实例共享同一个 Redis，解决了水平扩展时 Session 不同步的问题。

**消息队列**可以用 List 的 `LPUSH/BRPOP` 实现简单的生产消费模型，或者用更完善的 Redis Stream 实现带消费者组的消息队列。

**排行榜**天生是 ZSet 的场景，`ZADD` 更新分数，`ZREVRANGE` 获取 TopN，`ZREVRANK` 查询个人排名，一套命令全搞定。

**分布式锁**用 `SET key value NX PX timeout` 原子命令加锁，Lua 脚本原子解锁，是目前最常用的分布式锁方案。

**实时统计**方面，HyperLogLog 适合 UV 统计（大量用户去重），Bitmap 适合签到统计（按位存储每天的签到状态），GEO 适合附近的人功能。

###### 7. Redis 是单线程还是多线程?
这道题要分清楚**单线程/多线程指的是哪部分**，不能一刀切地说单线程或多线程。

**Redis 6.0 之前**：网络 I/O 和命令处理这一整个核心流程是单线程的。这里的"单线程"特指命令的读取、解析、执行、返回结果，这四步都在主线程里顺序完成。

**但 Redis 从来不是纯单线程的**：持久化（RDB 快照生成、AOF 重写）、异步删除（`UNLINK` 命令）、集群同步等操作一直都是由**后台线程或子进程**执行，以避免阻塞主线程。

**Redis 6.0 之后**：引入了多线程 I/O，主要用于**网络数据的读取和发送**，命令的实际执行逻辑依然是单线程。简单说就是 I/O 线程负责搬数据，主线程负责处理命令，这样在大响应包场景下能大幅提升吞吐量。

**源码体现**：在 `src/networking.c` 中，`readQueryFromClient`（读命令）和 `addReply`（写回复）等函数都在主线程上下文中被调用（6.0 前）。而像 `rdbSaveBackground`（后台 RDB 保存）则会 `fork` 子进程来执行。

###### 8. Redis 6.0 为什么引入多线程?
背景是随着网络硬件性能的提升（万兆网卡普及），在高吞吐量场景下，**网络 I/O 的处理开销本身成了瓶颈**。主线程要独自完成"读请求 → 解析请求 → 执行命令 → 发送响应"整套流程，当返回数据量很大时（比如 `HGETALL` 一个几百个字段的 Hash），发送响应这一步就会占用大量主线程时间，导致后续请求排队等待。

**Redis 6.0 的多线程模型是这样工作的**：多线程主要负责**网络 I/O**，即并行读取多个客户端的请求数据和并行发送响应；**命令的执行本身仍然是单线程**的，这保证了命令操作的原子性，无需引入复杂锁机制。主线程接受连接后，把 I/O 任务分发给 I/O 线程处理，等数据读完再串行执行命令，执行完后再由 I/O 线程并行发送结果。

**默认是关闭的**，需要手动开启：

```bash
io-threads-do-reads yes
io-threads 4   # 建议设置为 CPU 核心数的 3/4
```

对于延迟敏感的简单 GET/SET 操作，多线程提升不明显；对于需要返回大量数据的命令（`LRANGE`、`HGETALL`、`ZRANGE` 等），效果显著。
