###### 1. 什么是缓存穿透?如何解决?
**缓存穿透**是指查询一个缓存和数据库中**都不存在**的数据，每次请求都直接打到数据库，完全绕过了缓存。这通常是恶意攻击者构造大量无效 key 造成的，危害极大。

**解决方案 1：布隆过滤器**（推荐）

在缓存前加一层布隆过滤器，启动时把所有合法的 key 加进去，每次请求先过布隆过滤器，若确定不存在直接拒绝，根本不让请求到达缓存和数据库。

```java
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    1000000,  // 预期元素数量
    0.01      // 误判率（1% 的概率把不存在的 key 判断为存在）
);
// 启动时预热所有合法 key
bloomFilter.put("user:1000");
// 请求时先过滤
if (!bloomFilter.mightContain(key)) {
    return null;  // 直接拦截，不查缓存也不查 DB
}
```

**解决方案 2：缓存空对象**

数据库查不到时，把空值（null 或 ""）也缓存起来，设置较短的过期时间（比如 5 分钟），避免相同请求反复打数据库。

```java
Object data = db.query(key);
if (data == null) {
    redis.set(key, "", Duration.ofMinutes(5));  // 缓存空值
    return null;
}
```

空对象缓存的缺点是会占用一些内存，而且如果 key 种类无限多，依然防不住。

**解决方案 3：接口层参数校验**

对请求参数做合法性检查，比如 ID 为负数、格式不对的直接在接口层拒绝，最简单但覆盖范围有限。

**实际项目通常是布隆过滤器 + 空值缓存组合使用**，前者拦截随机无效 key，后者应对少量有规律的无效 key。

###### 2. 什么是缓存击穿?如何解决?
**缓存击穿**是指一个**热点 key 突然失效**时，大量并发请求同时访问这个 key，请求全部穿透到数据库，数据库瞬间压力激增。注意和缓存穿透的区别：穿透是 key 根本不存在，击穿是 key 过期失效。

**解决方案 1：互斥锁**（一致性优先）

缓存失效时，只允许一个线程重建缓存，其他线程等待重试。用 Redis 的 `SETNX` 实现分布式锁：

```java
if (cacheValue == null) {
    // 尝试获取分布式锁
    boolean locked = redis.setIfAbsent(lockKey, "1", Duration.ofSeconds(10));
    if (locked) {
        try {
            // 双重检查，可能已有其他线程重建完了
            cacheValue = redis.get(key);
            if (cacheValue == null) {
                cacheValue = db.query(key);
                redis.set(key, cacheValue, Duration.ofMinutes(30));
            }
        } finally {
            redis.del(lockKey);  // 释放锁
        }
    } else {
        Thread.sleep(50);
        return queryById(id);  // 未抢到锁，稍后重试
    }
}
```

**解决方案 2：逻辑过期**（可用性优先）

key 不设置物理过期时间，在 value 里存一个逻辑过期时间。查询时发现逻辑上过期了，**异步启动一个线程**去重建缓存，当前请求直接返回旧数据。这样不会有大量线程等待，可用性高，但会短暂返回旧数据。

**解决方案 3：永不过期 + 异步更新**

对极热点数据（如首页轮播、秒杀商品详情）直接设置永不过期，通过定时任务或 MQ 在后台更新缓存内容，彻底避免失效瞬间的压力。

###### 3. 什么是缓存雪崩?如何解决?
**缓存雪崩**是指**大量缓存数据在同一时间失效**，或 Redis 服务直接宕机，导致所有请求同时打向数据库，可能引发数据库雪崩式宕机。

缓存击穿是单个热点 key 失效，雪崩是大批 key 同时失效，范围更大，危害更严重。

**预防措施**：

**随机 TTL**：给不同 key 的过期时间加上随机偏移量，避免同批次导入的数据同时过期。

```java
int baseExpire = 30 * 60;  // 基础 30 分钟
int randomOffset = new Random().nextInt(10 * 60);  // 0-10 分钟随机
redis.set(key, value, Duration.ofSeconds(baseExpire + randomOffset));
```

**多级缓存**：本地缓存（JVM 内的 Caffeine/Guava Cache）+ Redis 分布式缓存，即使 Redis 整体挂了，本地缓存还能撑一段时间。

**Redis 高可用**：主从 + 哨兵或集群模式，避免 Redis 单点故障。

**熔断降级**：引入 Sentinel 或 Resilience4j 等熔断框架，当数据库压力超过阈值时快速返回兜底数据（如默认值、降级提示），保护数据库不被打死。

###### 4. 什么是热点数据?如何处理?
热点数据是指被**极高频率访问**的数据，比如双十一秒杀商品、首页轮播图、热搜榜单等。热点数据的核心问题是单个 key 的并发访问量太高，可能把 Redis 单个节点打满，甚至引发击穿。

**处理策略**：
1. **缓存预热**：系统启动或大促活动前，提前把热点数据加载到缓存，避免冷启动时的数据库压力
2. **多级缓存**：本地 JVM 缓存 + Redis，热点数据优先从本地缓存取，完全不走网络，响应最快
3. **热点 Key 探测**：通过 `redis-cli --hotkeys` 或自定义中间件监控 key 的访问频率，发现热点后特殊处理（比如在应用层本地缓存该 key，或把该 key 分散到多个 Redis 节点）
4. **逻辑分片**：将同一个热点 key 复制成 N 份（如 `hotkey_0` 到 `hotkey_9`），访问时随机选一份，把压力分摊到多个 slot 甚至多个节点
5. **永不过期 + 异步更新**：对极核心的热点数据不设过期时间，后台任务负责定期更新

###### 5. 布隆过滤器是什么?如何使用?
布隆过滤器是一种**空间极度高效的概率型数据结构**，用于快速判断一个元素是否在集合中。

**核心特性**：
- **可能误判（假阳性）**：说元素存在，但实际不存在（误判率可配置）
- **不会漏判（假阴性）**：说元素不存在，那它一定不存在

**原理**：一个很大的 bit 数组 + 多个哈希函数。添加元素时，通过 k 个哈希函数计算 k 个位置，全部置 1；查询时，如果这 k 个位置全是 1，认为元素可能存在（因为这些位置可能是其他元素设置的，所以有误判）；只要有一个是 0，元素一定不存在。

**Java 使用（Guava）**：

```java
BloomFilter<String> filter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    1000000,  // 预期插入的元素数量
    0.01      // 期望误判率 1%
);
filter.put("key1");
filter.mightContain("key1")  // true（可能存在）
filter.mightContain("key999")  // false（一定不存在）
```

**Redis 集成**：用 RedisBloom 模块，适合分布式场景：

```redis
BF.ADD myfilter "key1"
BF.EXISTS myfilter "key1"   # 1 或 0
```

**局限性**：布隆过滤器**不支持删除**（删除一个元素会影响其他元素的判断），需要支持删除场景的话用 Counting Bloom Filter 或 Cuckoo Filter。

###### 6. 缓存预热是什么?如何实现?
缓存预热是指**系统启动或低峰期**提前把热点数据加载到缓存，避免冷启动时大量请求直接打数据库。

**实现方式一：应用启动时加载**（Spring Boot 示例）：

```java
@Component
public class CacheWarmUp implements CommandLineRunner {
    @Override
    public void run(String... args) {
        List<HotItem> hotItems = itemService.getHotItems();
        hotItems.forEach(item ->
            redis.opsForValue().set(
                "item:" + item.getId(),
                item,
                Duration.ofHours(2)
            )
        );
        log.info("缓存预热完成，共预热 {} 条热点数据", hotItems.size());
    }
}
```

**实现方式二：定时任务刷新**：在业务低峰期（凌晨 2-4 点）定时执行预热任务，保证缓存中始终有热点数据。

**实现方式三：手动触发**：提供运维接口，大促活动开始前人工触发预热，保证活动时缓存已就绪。

**注意**：预热的数据量不能太大，否则启动时间会很长；也要考虑数据的时效性，预热进去的数据要设置合理的 TTL。

###### 7. 缓存更新策略有哪些?
三种主流的缓存更新模式：

**Cache Aside Pattern（旁路缓存，最常用）**：读操作先查缓存，未命中再查 DB 并写入缓存；写操作先更新 DB，再删除缓存。注意是**删除缓存**而不是更新缓存，因为更新缓存在并发场景下更容易出现数据不一致。

**Read/Write Through（读写穿透）**：应用只和缓存组件交互，缓存组件负责与数据库同步。读时缓存未命中由缓存组件从 DB 加载；写时应用写缓存，缓存组件同步写 DB。代码简洁，但需要缓存中间件支持，目前 Redis 原生不支持，需要自己封装。

**Write Behind（异步写回）**：写操作只更新缓存，缓存异步批量写入 DB。写性能极高，适合写多读少场景（如日志聚合），但缓存宕机时可能丢失未写入 DB 的数据，一致性最弱。

> 📖 [[../../../23_RedisKnowledge/06_缓存应用/01、缓存模式与问题解决#5-缓存更新模式|知识库：三种缓存更新模式]]

###### 8. Cache Aside Pattern 是什么?
Cache Aside 是**最常用的缓存模式**，应用层自己负责缓存和数据库之间的同步。

**读操作**：先查 Redis，命中直接返回；未命中则查数据库，查到后写入 Redis 再返回。

**写操作**：先更新数据库，再删除（不是更新）Redis 缓存。

**为什么是删除而不是更新？** 在并发写场景下，如果两个线程同时更新数据库，但更新缓存的顺序可能和数据库不一致，导致缓存里存了旧值。删除缓存就不存在这个问题，下次读时自然会从数据库重新加载最新值。

**潜在问题**：先写 DB 后删缓存，在极端情况下（删缓存失败、或者恰好有另一个线程在写 DB 之前读了缓存并在写 DB 之后写入了旧值）仍然可能出现短暂不一致，这时候就需要延迟双删或 Binlog 方案来兜底。

> 📖 [[../../../23_RedisKnowledge/06_缓存应用/01、缓存模式与问题解决#5-缓存更新模式|知识库：Cache Aside 模式]]

###### 9. Read/Write Through Pattern 是什么?
应用层把**缓存当作主要数据源**，所有读写操作都发给缓存，由缓存组件负责与数据库同步，对应用透明。

**读穿透（Read Through）**：读缓存命中直接返回；未命中时，由缓存组件自己去数据库查并加载，应用感知不到这个过程。

**写穿透（Write Through）**：应用写缓存，缓存组件同步写数据库，两步都完成才返回成功，保证缓存和数据库的强一致。

这种模式的优点是代码简洁，缓存组件封装了复杂的同步逻辑；缺点是需要有成熟的缓存中间件支持（比如 Tair、NearCache），Redis 原生不提供这层封装，需要自己实现。

> 📖 [[../../../23_RedisKnowledge/06_缓存应用/01、缓存模式与问题解决#5-缓存更新模式|知识库：Read/Write Through 模式]]

###### 10. Write Behind Pattern 是什么?
**写操作只更新缓存，由缓存组件异步批量写入数据库**，写操作无需等待数据库完成，性能极高。

**优点**：写吞吐量极大，数据库压力小，适合写多读少且允许短暂不一致的场景（如访问计数、消息已读状态）。

**缺点**：缓存宕机时，还未写入数据库的数据会丢失；数据一致性最弱，中间状态不可见；实现复杂，需要处理写失败重试和顺序性问题。

实际应用中，Write Behind 通常用于对数据丢失不敏感的计数类数据，不适合订单、资金等关键业务数据。

###### 11. 如何保证缓存和数据库的数据一致性?
这是业界公认的难题，没有完美方案，只有适合业务的权衡方案。

**方案一：延迟双删**。先删缓存，再更新数据库，延迟几百毫秒（等主从同步完成）再删一次缓存。解决了"更新数据库期间有读请求把旧值写入缓存"的问题，但依赖时间估算，不够精确。

```java
redis.del(key);
db.update(data);
Thread.sleep(500);  // 等主从同步
redis.del(key);     // 二次删除
```

**方案二：Binlog 异步同步**（最终一致，推荐）。应用只更新数据库，通过 Canal 等工具订阅 MySQL Binlog，捕获数据变更后异步更新 Redis。优点是与业务代码解耦，数据库事务提交后再更新缓存，不存在"数据库成功但删缓存失败"的问题。

**方案三：版本号控制**。在缓存值里存版本号，更新时只允许新版本覆盖旧版本，避免并发写导致的数据倒退。

**实践建议**：允许短暂不一致时，Cache Aside + 合理 TTL 就够了；要求高一致性时，用 Binlog 异步同步；绝对不能接受不一致时，考虑加分布式锁串行化写操作。

###### 12. 双写一致性问题如何解决?
双写一致性是指同时写缓存和数据库时，如何保证两者最终一致。

**最佳实践：先写数据库，后删缓存（Cache Aside 模式）**，这是目前公认的最合适做法。不要"先删缓存再写数据库"，因为删缓存和写数据库之间有时间窗口，期间有读请求会把旧数据写回缓存，造成缓存污染。

**兜底方案：消息队列保证删缓存可靠性**。如果删缓存失败，把删除操作发送到 MQ，通过重试机制保证最终一定删除成功。

**进阶方案：Binlog + Canal**。数据库事务提交是可靠的，通过订阅 Binlog 来异步更新缓存，天然不存在双写不一致问题，因为缓存更新是数据库变更的下游。

**版本号/时间戳**：在缓存值里加版本号，写入时做 CAS（Compare-And-Swap），只有版本更新的数据才能覆盖，防止并发写时旧数据覆盖新数据。
