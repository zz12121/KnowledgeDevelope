###### 1. Redis 有哪些持久化方式?
Redis 提供三种持久化策略，各有侧重。

**RDB（快照持久化）** 是定时生成内存数据的时间点快照，以二进制文件形式存储，数据安全等级中等（可能丢失最后一次快照后的数据），但恢复速度快，性能影响小，所有版本都支持。

**AOF（追加日志）** 是记录每个写操作命令的日志文件，数据安全等级高（everysec 策略最多丢失 1 秒数据），但文件比 RDB 大、恢复慢，Redis 2.0+ 支持。

**混合持久化** 是 Redis 4.0 引入的创新，结合了 RDB 快速恢复和 AOF 数据安全的优势，文件结构是 RDB 格式的全量数据 + AOF 格式的增量命令，是目前推荐的生产配置。

**设计哲学**：RDB 优先考虑性能，AOF 优先考虑数据安全，混合模式找平衡点。

###### 2. RDB 持久化的原理是什么?
RDB 持久化的核心是**内存快照**，通过 fork 子进程实现非阻塞的后台持久化。

**触发方式**：
- **手动触发**：`SAVE`（阻塞）或 `BGSAVE`（非阻塞后台执行）
- **自动触发**：根据 `save` 配置（如 `save 900 1` 表示 900 秒内至少 1 个 key 变化就触发）
- **其他场景**：执行 `SHUTDOWN`、`FLUSHALL` 时也会触发

**BGSAVE 的核心机制（源码）**：

```c
int rdbSaveBackground(char *filename) {
    pid_t childpid;
    if ((childpid = fork()) == 0) {
        // 子进程：执行实际 RDB 文件生成
        int retval = rdbSave(filename);
        exit((retval == C_OK) ? 0 : 1);
    } else {
        // 父进程：记录信息，继续处理客户端请求
        server.stat_fork_time = ustime() - start;
        return C_OK;
    }
}
```

**写时复制（Copy-on-Write）**：fork 之后父子进程共享同一份物理内存页，只有父进程修改某页时，操作系统才为子进程复制该页。这样子进程看到的始终是 fork 那一刻的内存快照，同时父进程可以正常处理新的写请求，内存开销也大幅降低。

**RDB 文件结构**：以 `REDIS` 魔数开头，包含版本信息、数据库数据（各键值对及其过期时间）、结束标记和 CRC64 校验和。

###### 3. AOF 持久化的原理是什么?
AOF 持久化通过**记录写操作命令**来保证数据持久性，重放这些命令即可恢复数据。

**工作流程**：
1. 客户端写命令执行后，以 RESP 协议格式追加到 `aof_buf` 缓冲区
2. 根据 `appendfsync` 配置，将缓冲区内容 fsync 到磁盘
3. 定期或手动触发 AOF 重写，消除冗余命令

**AOF 文件格式（RESP 协议）**：

```
*3        ← 3 个参数
$3        ← 第 1 个参数长度 3 字节
SET
$5        ← 第 2 个参数长度 5 字节
mykey
$7        ← 第 3 个参数长度 7 字节
myvalue
```

**AOF 重写机制**：重写不是扫描旧 AOF 文件，而是**遍历当前数据库状态**，为每个键生成最小命令集。比如对一个键先 `SET count 1`、再 `INCR count` 十次，重写后只保留 `SET count 11`，大幅压缩文件大小。

重写期间的新写命令会**同时写入 AOF 缓冲区和 AOF 重写缓冲区**，子进程完成重写后，主进程把重写缓冲区的内容追加到新 AOF 文件，再原子性地替换旧文件，确保不丢数据。

###### 4. RDB 和 AOF 的优缺点对比?
**数据安全性**方面，RDB 可能丢失最后一次快照后的所有数据（取决于 save 间隔），AOF 使用 everysec 策略最多丢 1 秒数据，安全性明显更高。

**恢复速度**方面，RDB 直接加载二进制数据，速度极快；AOF 需要逐条重放写命令，数据量大时恢复可能要几分钟甚至更长。

**磁盘占用**方面，RDB 是二进制压缩格式，体积小；AOF 是文本格式，文件大，需要靠重写机制来控制大小。

**性能影响**方面，RDB 写入时需要 fork 子进程，会有短暂阻塞，但后续持久化不影响主线程；AOF 需要持续写入，fsync 策略会影响写性能，always 策略影响最大。

**可读性**方面，AOF 文本格式可以直接打开查看和手动修改（比如误删了一个 key，直接从 AOF 里把那条命令删掉再恢复），RDB 是二进制格式，不可读。

**运维**方面，RDB 文件易迁移和备份；AOF 文件可能损坏，需要用 `redis-check-aof` 工具修复。

###### 5. 如何选择 RDB 和 AOF?
选择策略应该基于业务对数据安全的要求：

**纯缓存场景**：数据可以从数据库重建，丢了也没关系，就用 RDB，追求最高性能。配置示例：`save 900 1`、`save 300 10`。

**需要数据安全**：大多数业务场景，推荐开启混合持久化（`appendonly yes` + `aof-use-rdb-preamble yes`），兼顾恢复速度和数据安全。

**金融/交易系统**：数据绝对不能丢，用 AOF 的 `appendfsync always` 策略，每个写命令都立即 fsync，但性能是最低的，要做好性能规划。

**数据量很大**：如果关注启动速度，优先 RDB 或混合持久化，纯 AOF 恢复太慢。

**生产最佳实践**：混合持久化 + 定期把 RDB 文件备份到异地，同时用 `INFO persistence` 监控持久化状态。

###### 6. 什么是 AOF 重写?
AOF 重写是解决 AOF 文件膨胀问题的核心机制，本质是**基于当前数据状态生成最小命令集**，而不是压缩旧 AOF 文件。

**触发条件**：
- **自动触发**：`auto-aof-rewrite-percentage 100`（文件增长了 100%）且 `auto-aof-rewrite-min-size 64mb`（文件至少 64MB）同时满足
- **手动触发**：`BGREWRITEAOF` 命令

**重写过程**：
1. 主进程 fork 子进程，子进程拿到当前数据库快照
2. 子进程遍历所有数据库，为每个键生成对应的写命令（String 生成 SET，List 生成 RPUSH，等等）
3. **重点**：子进程工作期间，主进程继续处理客户端请求，新写命令**同时**写入旧 AOF 缓冲区（保证旧文件正常工作）和 AOF 重写缓冲区
4. 子进程写完新 AOF 文件后通知主进程
5. 主进程把重写缓冲区里的增量命令追加到新 AOF 文件
6. 原子性地用新文件替换旧文件（`rename` 系统调用）

**优化效果举例**：对同一个 key 执行了 100 次 `INCR count`，重写后只保留 `SET count 100`，一行命令替代了 100 行。

###### 7. Redis 的混合持久化是什么?
混合持久化是 Redis 4.0 引入的方案，结合了 RDB 快速恢复和 AOF 数据安全的优势。

**实现原理**：AOF 文件的结构变了，变成 `[RDB 格式的全量数据][AOF 格式的增量命令]`。重写时先把当前数据库快照以 RDB 格式写入文件头，然后把重写期间的增量操作以 AOF 格式追加在后面。

**启用配置**：

```bash
appendonly yes
aof-use-rdb-preamble yes   # Redis 4.0+ 默认开启
```

**优势**：恢复时先加载 RDB 部分（速度极快），再重放 AOF 部分（数据量少）。相比纯 AOF，启动速度大幅提升；相比纯 RDB，数据安全性更高（最多丢失 AOF 部分，通常只有几秒）。

**注意**：混合持久化生成的 AOF 文件不能被 Redis 4.0 以下版本加载，升级 Redis 版本前注意兼容性。

###### 8. RDB 持久化会阻塞主进程吗?
要分两种情况说：

**`SAVE` 命令**：**完全阻塞**，执行期间 Redis 不响应任何其他请求，直到持久化完成。只适合停机维护或手动数据备份，生产环境基本不用。

**`BGSAVE` 命令（以及自动触发）**：**fork 阶段有短暂阻塞**，fork 本身是一个系统调用，需要复制父进程的内存页表，内存越大 fork 越慢。fork 完成后子进程做实际的持久化工作，主进程恢复正常服务。

**fork 阶段阻塞时间估算**：经验值大约是每 GB 内存 10-20ms，10GB 内存可能需要 100-200ms，期间主进程完全阻塞，所有请求排队。

**写时复制的内存开销**：fork 期间如果主进程有大量写请求，会产生内存页复制，最坏情况下内存使用量可能翻倍，需要预留足够的内存空间。

**优化建议**：
- 单实例内存不要超过 20GB，控制 fork 时间
- 监控 fork 耗时：`INFO stats` 里的 `latest_fork_usec` 字段
- 用 SSD 硬盘减少持久化时间
- 合理配置 `save` 参数，避免在业务高峰期触发

###### 9. AOF 的三种同步策略是什么?
AOF 的同步策略通过 `appendfsync` 配置控制，直接影响数据安全性和写性能的权衡。

**`always`**：每个写命令执行完立即调用 `fsync` 同步到磁盘。数据安全性最高，最多丢失一个命令；但每次写都要等 fsync 完成，性能最低，磁盘 I/O 是瓶颈。适合金融、交易等对数据一致性要求极高的场景。

**`everysec`**（默认推荐）：后台线程每秒执行一次 fsync。最多丢失 1 秒数据，性能和安全性的最佳平衡点，是大多数生产环境的选择。

**`no`**：Redis 不主动 fsync，完全依赖操作系统决定刷盘时机。性能最高，但操作系统崩溃时可能丢失较多数据（通常是几十秒内的数据）。适合对数据安全性要求不高的缓存场景。

**源码实现**：

```c
void flushAppendOnlyFile(int force) {
    if (server.aof_fsync == AOF_FSYNC_ALWAYS) {
        fsync(server.aof_fd);  // 每次写入后同步
    } else if (server.aof_fsync == AOF_FSYNC_EVERYSEC) {
        // 后台线程每秒执行一次 fsync
    }
    // no 策略：不执行主动 fsync
}
```

###### 10. Redis 持久化数据丢失的场景有哪些?
**RDB 的数据丢失场景**：两次快照之间 Redis 宕机，则这段时间的所有写操作都丢失。`save` 间隔配置越长，丢失的数据窗口越大。如果 `SHUTDOWN` 命令没能正常执行（比如 kill -9），也不会触发最后一次 RDB。

**AOF 的数据丢失场景**：everysec 策略下，宕机可能丢失最近 1 秒的写操作。如果使用 no 策略，可能丢失操作系统缓存里几十秒的数据。AOF 文件在写入过程中宕机还可能产生不完整的文件（`redis-check-aof` 可以修复）。

**混合持久化的数据丢失场景**：重写完成的那一刻是安全的，但 AOF 增量部分也会因为 fsync 策略有丢失风险，和纯 AOF 一样最多丢 1 秒（everysec 策略）。

**数据安全最佳实践**：
- 混合持久化 + everysec（兼顾速度和安全）
- 定期将 RDB/AOF 文件备份到异地（防止服务器物理故障）
- `INFO persistence` 监控持久化健康状态
- 主从复制作为额外的数据冗余层
- 定期演练数据恢复流程，确认备份有效性
