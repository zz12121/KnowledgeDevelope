###### 1. Zookeeper 的典型应用场景

ZooKeeper 的所有应用场景，本质上都是利用这两个核心特性：**临时节点（故障自动感知）** + **Watcher（变化主动推送）**。

**七大典型场景：**

| 场景 | 使用的特性 | 简述 |
|------|---------|------|
| 服务注册与发现 | 临时节点 + Watcher | 服务上线创建临时节点，下线自动删除 |
| 配置中心 | 持久节点 + Watcher | 配置变更后推送给所有监听者 |
| 分布式锁 | 临时顺序节点 | 序号最小者持锁，宕机自动释放 |
| Leader 选举 | 临时节点抢创建 | 第一个创建成功的节点当选 Leader |
| 集群管理 | 临时节点 + Watcher | 感知集群成员上下线 |
| 命名服务 | 持久顺序节点 | 全局唯一 ID 生成 |
| 分布式队列 | 持久顺序节点 | 按序号消费，实现 FIFO |

###### 2. Zookeeper 和 Dubbo 的关系?

ZooKeeper 是 Dubbo 默认的**服务注册中心**，两者是配合关系，不是竞争关系。

**ZooKeeper 在 Dubbo 中的角色：**

```
服务提供者（Provider）
    ↓ 启动时注册
ZooKeeper /dubbo/{服务名}/providers/
    ↑ 拉取可用地址列表 + 监听变化
服务消费者（Consumer）
```

**具体机制：**
1. **服务注册**：Provider 启动时在 `/dubbo/{interface}/providers/` 下创建**临时节点**，内容是 URL（包含 IP/端口/参数）
2. **服务订阅**：Consumer 订阅 `/dubbo/{interface}/providers/`，获取所有 Provider 地址列表，并注册 Watcher
3. **动态感知**：Provider 宕机后 Session 超时，临时节点消失，Consumer 通过 Watcher 感知到，更新本地路由表
4. **负载均衡**：Consumer 持有 Provider 列表，按配置的负载均衡策略（Random/RoundRobin/ConsistentHash）选择目标发起调用

**Dubbo 的 ZK 路径结构：**
```
/dubbo
  /{interface}
    /providers      # 服务提供者注册
    /consumers      # 服务消费者注册（可选）
    /routers        # 路由规则
    /configurators  # 配置覆盖规则
```

**补充：** Dubbo 3.x 应用级服务发现模式下，ZK 路径改为 `/services/{applicationName}/`，不再以接口为维度注册，减少 ZNode 数量。

###### 3. 如何使用 Zookeeper 实现服务注册与发现?

**注册中心设计思路：**

```java
// 服务注册（Provider 启动时）
public void register(String serviceName, String address) throws Exception {
    // 路径：/services/{serviceName}/{address}
    String path = "/services/" + serviceName + "/" + address;
    // 创建临时节点，服务宕机后自动删除
    client.create()
        .creatingParentsIfNeeded()
        .withMode(CreateMode.EPHEMERAL)
        .forPath(path, address.getBytes());
}

// 服务发现（Consumer 启动时）
public List<String> discover(String serviceName) throws Exception {
    String path = "/services/" + serviceName;
    return client.getChildren().forPath(path);
}

// 监听服务变化（Consumer 持续订阅）
public void subscribe(String serviceName, ServiceChangeListener listener) {
    String path = "/services/" + serviceName;
    PathChildrenCache cache = new PathChildrenCache(client, path, true);
    cache.getListenable().addListener((c, event) -> {
        List<String> addresses = client.getChildren().forPath(path);
        listener.onServiceChange(addresses);
    });
    cache.start();
}
```

**完整流程：**
1. Provider 启动 → 创建临时节点 `/services/{name}/{ip:port}`
2. Consumer 启动 → `getChildren("/services/{name}")` 获取地址列表
3. Consumer 用 `PathChildrenCache` 监听子节点变化
4. Provider 宕机 → 临时节点消失 → Consumer 收到 `CHILD_REMOVED` 事件 → 更新本地缓存
5. 新 Provider 上线 → 临时节点创建 → Consumer 收到 `CHILD_ADDED` 事件 → 更新本地缓存

###### 4. 如何使用 Zookeeper 实现配置中心?

配置中心的核心是：**配置存在 ZNode，变更后推送给所有监听者**。

```java
// 发布配置
public void publishConfig(String key, String value) throws Exception {
    String path = "/config/" + key;
    if (client.checkExists().forPath(path) == null) {
        client.create().creatingParentsIfNeeded()
            .withMode(CreateMode.PERSISTENT)
            .forPath(path, value.getBytes());
    } else {
        client.setData().forPath(path, value.getBytes());
    }
}

// 监听配置（NodeCache，持续监听，无间隙）
public void watchConfig(String key, ConfigChangeCallback callback) throws Exception {
    String path = "/config/" + key;
    NodeCache cache = new NodeCache(client, path);
    cache.getListenable().addListener(() -> {
        String newValue = new String(cache.getCurrentData().getData());
        callback.onConfigChanged(key, newValue);
    });
    cache.start(true);  // true 表示立即加载当前值
}
```

**配置中心实现的要点：**
- 使用**持久节点**（配置不依赖客户端会话存活）
- 使用 Curator 的 **NodeCache**（持续监听，解决 Watcher 间隙问题）
- 可以使用**版本号**做乐观锁，防止并发写入冲突
- 生产中更推荐用 Nacos / Apollo，它们在 ZooKeeper 的基础上加了 UI、权限、历史版本等运维能力

###### 5. 如何使用 Zookeeper 实现命名服务?

命名服务就是生成**全局唯一、有序的 ID**，利用顺序节点的自增特性实现：

```java
// 生成全局唯一 ID
public long generateId(String namespace) throws Exception {
    String path = "/naming/" + namespace + "/id-";
    // 创建持久顺序节点
    String createdPath = client.create()
        .creatingParentsIfNeeded()
        .withMode(CreateMode.PERSISTENT_SEQUENTIAL)
        .forPath(path, new byte[0]);
    // 提取序号（路径末尾10位数字）
    String sequenceStr = createdPath.replace(path, "");
    long id = Long.parseLong(sequenceStr);
    // 可以删除节点节省空间（只需要序号）
    client.delete().forPath(createdPath);
    return id;
}
```

**优缺点：**
- **优点**：简单可靠，全局唯一，有序
- **缺点**：每次生成 ID 都要创建/删除 ZNode，性能不高（每秒几百次）
- **适用场景**：低频 ID 生成，如分布式 Job ID、批次号
- **高频场景**推荐：雪花算法（Snowflake）或 Redis INCR

###### 6. 如何使用 Zookeeper 实现集群管理?

集群管理的核心是感知**集群成员的加入和退出**，利用临时节点 + Watcher 实现：

```java
// 节点加入集群
public void joinCluster(String nodeId, String nodeInfo) throws Exception {
    String path = "/cluster/nodes/" + nodeId;
    client.create()
        .creatingParentsIfNeeded()
        .withMode(CreateMode.EPHEMERAL)
        .forPath(path, nodeInfo.getBytes());
}

// 监控集群成员变化
public void monitorCluster(ClusterChangeListener listener) throws Exception {
    PathChildrenCache cache = new PathChildrenCache(client, "/cluster/nodes", true);
    cache.getListenable().addListener((c, event) -> {
        List<ChildData> nodes = cache.getCurrentData();
        switch (event.getType()) {
            case CHILD_ADDED:
                listener.onNodeJoin(new String(event.getData().getData()));
                break;
            case CHILD_REMOVED:
                listener.onNodeLeave(event.getData().getPath());
                break;
        }
    });
    cache.start();
}
```

实际应用：Kafka 用 ZooKeeper 管理 Broker 注册，HBase 用 ZooKeeper 管理 RegionServer 注册，都是这套模式。

###### 7. 如何使用 Zookeeper 实现 Master 选举?

**方式一：抢创建临时节点（简单场景）**

```java
public boolean tryBecomeMaster() {
    try {
        client.create()
            .withMode(CreateMode.EPHEMERAL)
            .forPath("/election/master", nodeId.getBytes());
        return true;  // 创建成功，当选 Master
    } catch (NodeExistsException e) {
        return false;  // 节点已存在，有其他人是 Master
    }
}

// 监听 Master 节点，Master 宕机后重新竞选
public void watchMaster() {
    NodeCache cache = new NodeCache(client, "/election/master");
    cache.getListenable().addListener(() -> {
        if (cache.getCurrentData() == null) {
            // Master 节点消失，重新竞选
            tryBecomeMaster();
        }
    });
}
```

**方式二：Curator LeaderSelector（推荐，Curator 封装）**

```java
LeaderSelector selector = new LeaderSelector(client, "/election/leader",
    new LeaderSelectorListenerAdapter() {
        @Override
        public void takeLeadership(CuratorFramework client) throws Exception {
            System.out.println(nodeId + " 成为 Leader");
            // 在这里执行 Leader 独有的工作
            // 方法返回时，自动放弃 Leader 角色
            Thread.sleep(Long.MAX_VALUE);  // 一直保持直到异常/shutdown
        }
    });
selector.autoRequeue();  // Leader 放弃后自动重新加入队列
selector.start();
```

###### 8. Zookeeper 在 Kafka 中的应用是什么?

> 双链：[[31_KafkaKnowledge]]

**Kafka（3.0 之前）大量依赖 ZooKeeper 做集群协调：**

**① Broker 注册**
- 每个 Broker 启动时在 `/brokers/ids/{brokerId}` 创建临时节点
- Broker 宕机 → 临时节点消失 → Controller 感知 → 触发 Leader 重新选举

**② Topic 和分区元数据**
- `/brokers/topics/{topicName}` 存储 Topic 的分区分配信息
- `/brokers/topics/{topicName}/partitions/{pid}/state` 存储分区 Leader 和 ISR 信息

**③ Controller 选举**
- Kafka 的 Controller 是一个特殊 Broker，负责分区 Leader 选举
- 通过抢创建 `/controller` 临时节点来竞选 Controller，ZooKeeper 保证只有一个 Controller

**④ 消费者组管理（老版本）**
- `/consumers/{groupId}/offsets/{topic}/{partition}` 存储消费位移
- 新版本改为存在 Kafka 内部 Topic `__consumer_offsets`

**Kafka 3.0+ 去 ZooKeeper：**
- Kafka 引入了 **KRaft 模式**（基于 Raft 的内置元数据管理）
- 不再依赖外部 ZooKeeper，自己管理集群元数据
- ZooKeeper 模式在 Kafka 4.0 中完全移除

###### 9. Zookeeper 在 HBase 中的应用是什么?

HBase 是强依赖 ZooKeeper 的系统，ZooKeeper 承担了 HBase 的**核心协调角色**：

**① Master 选举和 HA**
- HBase 的 HMaster 通过 ZooKeeper 进行主备选举
- Active Master 在 ZK 上创建临时节点，Standby Master 监听该节点
- Active Master 宕机 → 节点消失 → Standby Master 感知并接管

**② RegionServer 注册与健康监测**
- RegionServer 启动时创建临时节点 `/hbase/rs/{serverName}`
- RegionServer 宕机 → 临时节点消失 → HMaster 触发 Region 迁移

**③ ROOT/META 表地址**
- `hbase:meta` 表的地址存储在 ZooKeeper 中
- 客户端访问 HBase 的第一步：从 ZooKeeper 获取 `hbase:meta` 的位置，再查找目标 Region 的地址

**④ 集群状态管理**
- 集群启动/关闭状态、Recovery 状态等都通过 ZooKeeper 协调

---

> **追问：既然 Kafka 3.0 去掉了 ZooKeeper，Dubbo 3.x 也支持了 Nacos，ZooKeeper 的未来怎么样？**
>
> ZooKeeper 确实在新生态中的占比在下降，但不会消亡：
>
> 1. **老系统的既有投资**：大量生产中的 Kafka 2.x、Dubbo 2.x、HBase 系统仍在用，迁移成本高
> 2. **新场景被替代**：服务注册发现（Nacos更好用）、配置中心（Apollo/Nacos）、分布式锁（Redis更快）都有更好的替代方案
> 3. **核心优势依然存在**：ZooKeeper 的 CP 模型、强一致性、ZAB 协议在**需要强一致协调**的场景（如 Hadoop YARN 的 HA、HBase）仍然是最可靠的选择
>
> 结论：ZooKeeper 在大数据生态（Hadoop/HBase/早期Kafka）中地位依然稳固，在微服务领域逐渐被 Nacos 取代，学好原理更重要。
