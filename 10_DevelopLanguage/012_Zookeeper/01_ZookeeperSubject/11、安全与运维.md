###### 1. Zookeeper 如何保证数据安全?

ZooKeeper 的数据安全从三个层面保障：

**① 访问控制（ACL）**
- 每个 ZNode 独立配置 ACL（schema:id:permissions）
- 支持 `world`/`auth`/`digest`/`ip` 四种认证方案
- 通过 ACL 限制哪些客户端可以读/写/创建/删除节点

**② 传输安全（SSL/TLS）**
- ZooKeeper 3.5.5+ 支持客户端-服务端、服务端-服务端的 SSL/TLS 加密传输
- 配置 `ssl.keyStore.location`、`ssl.trustStore.location` 等参数
- 客户端连接时使用 `zookeeper.ssl.trustStore.location` 等 JVM 参数

**③ 认证机制（SASL）**
- 支持 SASL（Simple Authentication and Security Layer）
- 可集成 Kerberos 实现强身份认证（适合企业级安全要求）
- 配置 `authProvider.1=org.apache.zookeeper.server.auth.SASLAuthenticationProvider`

**④ 数据持久化保障**
- 事务日志 + 快照双重持久化，防止数据丢失
- ZAB 协议保证已提交数据不会因为 Leader 切换而丢失

###### 2. Zookeeper 的身份认证机制有哪些?

**① Digest 认证（最常用）**

用户名+密码认证，密码 SHA1 + Base64 编码存储：

```java
// 客户端添加认证信息
zk.addAuthInfo("digest", "admin:password123".getBytes());

// 创建节点时设置 ACL
List<ACL> acls = new ArrayList<>();
acls.add(new ACL(ZooDefs.Perms.ALL,
    new Id("digest", DigestAuthenticationProvider.generateDigest("admin:password123"))));
zk.create("/protected", data, acls, CreateMode.PERSISTENT);
```

**② IP 认证**

根据客户端 IP 地址限制访问：
```java
// 只允许 192.168.1.0/24 网段的客户端访问
List<ACL> acls = Arrays.asList(
    new ACL(ZooDefs.Perms.READ | ZooDefs.Perms.WRITE,
            new Id("ip", "192.168.1.0/24"))
);
```

**③ SASL/Kerberos 认证**

企业级强认证，适合 Hadoop 生态：
```properties
# zoo.cfg 配置
authProvider.1=org.apache.zookeeper.server.auth.SASLAuthenticationProvider
requireClientAuthScheme=sasl
jaasLoginRenewWindow=0
```

**④ X.509 证书认证（mTLS）**
- 双向 SSL/TLS，客户端和服务端都需要证书
- 适合高安全要求的生产环境

**注意：** `world:anyone:rwcda`（全开放）是默认 ACL，生产环境**必须**修改！

###### 3. 如何监控 Zookeeper 集群的健康状态?

**① 四字命令（快速检查）**

```bash
# 查看服务器状态（最常用）
echo ruok | nc zoo1 2181  # 返回 imok 表示存活

# 查看统计信息（连接数、延迟、节点数等）
echo stat | nc zoo1 2181

# 查看当前连接详情
echo dump | nc zoo1 2181

# 查看环境变量（Java版本、ZK版本等）
echo envi | nc zoo1 2181

# 查看内存使用和性能指标
echo mntr | nc zoo1 2181
```

**② mntr 指标监控（接入 Prometheus）**

`mntr` 命令返回丰富的 Key-Value 指标，可接入监控系统：

| 关键指标 | 含义 | 阈值参考 |
|---------|------|---------|
| `zk_avg_latency` | 平均请求延迟(ms) | < 10ms |
| `zk_max_latency` | 最大请求延迟(ms) | < 100ms |
| `zk_outstanding_requests` | 等待处理的请求数 | < 100 |
| `zk_open_file_descriptor_count` | 打开的文件描述符数 | < 系统限制的 80% |
| `zk_server_state` | 节点角色（leader/follower/observer） | 集群只有一个 leader |
| `zk_znode_count` | ZNode 总数 | < 100w |
| `zk_watch_count` | Watcher 总数 | < 100w |
| `zk_followers` | Follower 数量 | 等于预期值 |

**③ 接入 Prometheus + Grafana**

ZooKeeper 3.6+ 内置 Prometheus 格式 metrics 端点：
```properties
# zoo.cfg
metricsProvider.className=org.apache.zookeeper.metrics.prometheus.PrometheusMetricsProvider
metricsProvider.httpPort=7000
```

访问 `http://zoo1:7000/metrics` 获取 Prometheus 格式的监控数据。

###### 4. Zookeeper 的四字命令有哪些?

ZooKeeper 支持通过 `echo cmd | nc host port` 方式发送四字命令（Four Letter Words）快速获取集群信息：

| 命令 | 作用 |
|------|------|
| `ruok` | 检查服务存活，返回 `imok`（Run OK → I'm OK，ZK 工程师的幽默） |
| `stat` | 集群状态（角色/版本/连接数/延迟/请求统计） |
| `mntr` | 监控指标（比 stat 更详细，返回 KV 格式，适合接入监控系统） |
| `conf` | 查看当前配置（tickTime/dataDir/clientPort 等） |
| `envi` | JVM 环境信息 |
| `dump` | 当前 Session 列表和临时节点信息 |
| `cons` | 当前活跃连接详情（IP/延迟/接收发送字节数） |
| `srvr` | 服务器信息（`stat` 的精简版，不包含连接信息） |
| `wchs` | Watcher 汇总（总 Watcher 数、监听路径数） |
| `wchc` | 按连接列出 Watcher（Session → 监听的路径列表） |
| `wchp` | 按路径列出 Watcher（路径 → 监听该路径的 Session） |
| `dirs` | 日志目录和快照目录大小 |
| `crst` | 重置连接统计 |
| `isro` | 当前是否只读模式（少数派节点在网络分区时进入只读） |

**3.5.3+ 需要在 zoo.cfg 中白名单开启：**
```properties
4lw.commands.whitelist=stat,ruok,conf,isro,mntr
# 或全部开启
4lw.commands.whitelist=*
```

###### 5. 如何进行 Zookeeper 的数据备份与恢复?

**数据备份：**

ZooKeeper 的数据持久化文件包含：
- **快照文件**：`dataDir/version-2/snapshot.*`
- **事务日志**：`dataLogDir/version-2/log.*`

备份策略：
```bash
# 方式1：直接复制快照和日志文件（需要停止 ZK 或使用文件系统快照）
rsync -av /data/zookeeper/data/version-2/ /backup/zookeeper/data/
rsync -av /data/zookeeper/log/version-2/ /backup/zookeeper/log/

# 方式2：使用 ZooKeeper 内置备份工具（导出 ZNode 数据）
# 无官方工具，第三方工具 zkdump：
java -jar zkdump.jar -zookeeper zoo1:2181 -path / > zk_backup.json
```

**数据恢复：**

```bash
# 停止 ZooKeeper 服务
bin/zkServer.sh stop

# 清理当前数据目录
rm -rf /data/zookeeper/data/version-2/*
rm -rf /data/zookeeper/log/version-2/*

# 恢复快照和日志
cp /backup/zookeeper/data/snapshot.* /data/zookeeper/data/version-2/
cp /backup/zookeeper/log/log.* /data/zookeeper/log/version-2/

# 重启服务，ZK 会自动加载快照+重放日志
bin/zkServer.sh start
```

**注意：** 集群恢复时建议所有节点使用同一个备份时间点的数据，避免数据不一致。

###### 6. Zookeeper 的日志如何管理?

ZooKeeper 有两类日志：

**① 应用日志（zookeeper.out / zookeeper-xx.log）**

运行日志，记录 ZooKeeper 服务的运行状态、警告、错误：
```properties
# log4j.properties 配置
log4j.rootLogger=INFO, ROLLINGFILE
log4j.appender.ROLLINGFILE=org.apache.log4j.RollingFileAppender
log4j.appender.ROLLINGFILE.File=/data/zookeeper/logs/zookeeper.log
log4j.appender.ROLLINGFILE.MaxFileSize=10MB
log4j.appender.ROLLINGFILE.MaxBackupIndex=10
```

**② 事务日志（dataLogDir/version-2/log.*）**

数据持久化日志，不是给人看的，是 ZAB 协议的 WAL：
- 定期自动清理：`autopurge.purgeInterval` + `autopurge.snapRetainCount`
- 手动清理：`zkCleanup.sh` 或 `PurgeTxnLog` 工具

**日志级别调整（临时调试）：**
```bash
# 通过 JMX 动态调整日志级别
java -jar jmxterm.jar
open zoo1:9999
bean org.apache.ZooKeeperService:name0=StandaloneServer_port-1
run setLogLevel DEBUG
```

###### 7. 如何排查 Zookeeper 的常见问题?

**排查思路：四字命令 → 应用日志 → 业务影响**

**① 集群是否正常**
```bash
# 检查各节点是否存活
for host in zoo1 zoo2 zoo3; do
    echo -n "$host: "; echo ruok | nc $host 2181
done

# 检查集群角色（确认只有一个 leader）
for host in zoo1 zoo2 zoo3; do
    echo -n "$host: "; echo stat | nc $host 2181 | grep Mode
done
```

**② 查看关键指标**
```bash
# 连接数、延迟、Outstanding 请求数
echo mntr | nc zoo1 2181 | grep -E "zk_(avg_latency|outstanding|znode_count|watch_count)"
```

**③ 常见问题排查**

| 现象 | 可能原因 | 排查方法 |
|------|---------|---------|
| `ruok` 不返回 | 进程崩溃 / 端口占用 | `ps aux | grep zookeeper`，检查 `zookeeper.out` |
| 选举不结束（一直 LOOKING） | 网络不通 / 端口3888阻塞 | `telnet zoo1 3888`，检查防火墙 |
| 写操作超时 | 磁盘 I/O 慢 / Follower 无法同步 | `iostat -x 1`，检查磁盘性能 |
| 大量 Session Expired | SessionTimeout 太短 / 网络抖动 | 增大 sessionTimeout，检查网络质量 |
| 内存告警 | ZNode 过多 / 数据过大 | `echo mntr | nc` 查看 `zk_znode_count` |

###### 8. Zookeeper 集群升级需要注意什么?

**升级原则：滚动升级，逐台进行，先 Observer → 再 Follower → 最后 Leader**

**升级步骤：**

1. **备份数据**：升级前完整备份快照和事务日志
2. **查看版本兼容性**：确认新旧版本兼容（ZooKeeper 一般向下兼容）
3. **Observer 先升级**（如有）：风险最低，不影响投票
4. **逐台升级 Follower**：
   - 停止一台 Follower
   - 替换 ZooKeeper 包
   - 启动，观察日志是否正常，`echo stat` 确认角色为 FOLLOWING
   - 等待数据同步完成后再处理下一台
5. **最后升级 Leader**：
   - 停止 Leader → 触发重新选举 → 新 Leader 产生
   - 升级原 Leader → 以 Follower 身份重新加入
6. **验证集群**：`echo mntr`，检查 leader/follower 角色、zxid 一致性

**注意事项：**
- 每次只停一台，保证集群始终有多数派存活
- 升级 Leader 前确认多数派 Follower 已完成升级（防止旧版本当选新 Leader）
- 跨大版本升级（如 3.4 → 3.6）前先查阅 Release Notes，可能需要配置文件调整
- 升级后观察 1~2 个业务高峰期，确认无异常后视为升级完成

> **追问：ZooKeeper 运维中最容易忽略的配置是什么？**
>
> **最容易被忽略的三个配置：**
>
> 1. **`dataLogDir` 与 `dataDir` 分开**：很多人图简单用同一个目录，结果事务日志和快照互相抢 I/O，性能大幅下降
>
> 2. **`autopurge.purgeInterval` 为 0（默认关闭）**：忘记开启自动清理，日志积累几个月把磁盘撑满，线上报警才发现
>
> 3. **ACL 全是 `world:anyone:rwcda`**：默认没有任何访问限制，所有人都能读写删节点。生产环境必须配置 Digest ACL 或 IP 白名单，否则任何人都能清空 ZooKeeper 数据
