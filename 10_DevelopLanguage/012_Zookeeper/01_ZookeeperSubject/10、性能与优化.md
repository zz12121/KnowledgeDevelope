# 十、性能与优化

> 双链：[[37_ZookeeperKnowledge/06_性能优化与运维]]

---

###### 1. Zookeeper 的性能瓶颈在哪里?

ZooKeeper 的性能瓶颈集中在**写操作**，读性能相对较好。

**写性能瓶颈：**
1. **单 Leader 写**：所有写请求必须经过 Leader，Leader 是写的单点，无法水平扩展
2. **ZAB 两阶段广播**：每次写需要 Leader 把 Proposal 广播给所有 Follower，等超过半数 ACK，再广播 COMMIT，至少两个 RTT
3. **全同步事务日志（fsync）**：每次写事务都必须 fsync 到磁盘（WAL），强制持久化，磁盘 I/O 是瓶颈
4. **Follower 数量影响写延迟**：Follower 越多，Leader 需要等待的 ACK 越多，延迟线性上升

**读性能相对较好的原因：**
- 读请求可以被任意 Follower/Observer 处理，水平扩展
- 全部数据在内存中，读操作无磁盘 I/O
- 通过增加 Observer 可以线性扩展读吞吐

**实际压测指标（参考值）：**
- 写 QPS：~5000~15000（单集群，依赖磁盘速度）
- 读 QPS：~10w+（内存读，Observer 扩展后更高）

###### 2. 如何优化 Zookeeper 的性能?

**① 分离事务日志磁盘**
```properties
# 快照和事务日志分开存放，避免 I/O 竞争
dataDir=/ssd/zookeeper/data     # 快照（读多）
dataLogDir=/ssd/zookeeper/log   # 事务日志（写多，最好用 SSD）
```
事务日志的 fsync 是写性能的核心瓶颈，单独挂载高速 SSD 磁盘效果显著。

**② 使用 Observer 扩展读能力**
- 增加 Observer 节点处理读请求，不影响写吞吐
- 适合读多写少场景（如配置中心、服务注册发现）

**③ 合理设置 tickTime**
```properties
# 根据集群间网络延迟调整，不要设置过小（频繁 PING）或过大（故障检测慢）
tickTime=2000  # 局域网集群；跨机房可适当增大到 5000
```

**④ 控制 ZNode 数量和数据大小**
- 单集群 ZNode 数量控制在 100w 以内
- 每个 ZNode 数据不超过 1MB（越小越好，理想情况下几百字节）
- 避免在 ZK 上存业务数据（只存元数据和协调数据）

**⑤ 启用自动快照清理**
```properties
autopurge.snapRetainCount=3
autopurge.purgeInterval=1
```

**⑥ 调整 JVM 堆内存**
```bash
# ZooKeeper 全量数据在内存，根据数据量设置合适的堆大小
export JVMFLAGS="-Xmx4g -Xms4g"
# 使用 G1GC 减少 GC 停顿（ZK 对延迟敏感）
export JVMFLAGS="$JVMFLAGS -XX:+UseG1GC -XX:MaxGCPauseMillis=200"
```

###### 3. Zookeeper 的写性能为什么不高?

ZooKeeper 的写性能天花板大约在 **5000~15000 TPS**，根本原因是设计目标不同——ZooKeeper 不是高吞吐 KV 存储，而是**强一致分布式协调服务**，一致性和高写吞吐是矛盾的。

**具体原因：**

1. **同步写日志（fsync）**：写操作必须先 fsync 到磁盘才能响应，一次 fsync 在 HDD 上约 5~10ms，在 SSD 上约 0.1~1ms，磁盘 I/O 直接限制写 QPS

2. **ZAB 两阶段提交**：Propose → 等 ACK → Commit，至少两次网络 RTT，局域网 RTT 约 1ms，每次写至少 2ms

3. **单点 Leader**：所有写串行经过 Leader，无法并行，Leader 的单线程处理成为瓶颈

4. **广播开销**：Follower 越多，Leader 的广播开销越大，写延迟随 Follower 数量增加而上升

**类比：** 这就好比一家公司的每个决策都要开全体会议（ZAB 广播），并且所有决议都要存档盖章（fsync），效率自然不高，但确保每个决策合法有记录。

###### 4. Zookeeper 的事务日志和快照机制是什么?

ZooKeeper 的持久化依赖两种文件：**事务日志（Transaction Log）** 和 **快照（Snapshot）**，两者共同保证崩溃后数据可恢复。

**事务日志（WAL）：**
- 存储路径：`dataLogDir`
- 每次写操作先写事务日志（WAL），然后再应用到内存
- 采用**预分配**策略：每次新建日志文件时预先分配 64MB 空间，减少磁盘扩容操作
- 文件名格式：`log.zxid`（以该文件第一条事务的 ZXID 命名）

**快照（Snapshot）：**
- 存储路径：`dataDir`
- 定期将内存中的**完整数据树**序列化到磁盘
- 触发条件：写事务数量超过阈值（`snapCount`，默认 100000 次）
- 文件名格式：`snapshot.zxid`
- 快照是内存的全量镜像，但不保证精确对应某个 ZXID（因为快照过程中可能还有写操作进来）

**崩溃恢复时的数据加载：**
```
1. 加载最新的快照文件（snapshot.zxid）→ 恢复大部分数据
2. 找到 zxid 大于 snapshot.zxid 的事务日志文件
3. 重放这些日志中的事务 → 恢复快照之后的增量数据
4. 完成数据恢复，进入选举或同步阶段
```

###### 5. 如何进行 Zookeeper 的数据清理?

ZooKeeper 的事务日志和快照持续增长，不清理会把磁盘撑满。

**方式一：配置自动清理（推荐）**
```properties
# zoo.cfg
autopurge.snapRetainCount=3     # 保留最新 3 个快照（至少要保留3个以备回滚）
autopurge.purgeInterval=1       # 每1小时自动清理一次（0表示关闭自动清理）
```

**方式二：手动执行清理脚本**
```bash
# ZooKeeper 内置清理工具
java -cp /opt/zookeeper/lib/*:/opt/zookeeper/zookeeper-*.jar \
     org.apache.zookeeper.server.PurgeTxnLog \
     /data/zookeeper /data/zookeeper/log -n 3
```

**方式三：cron 定期清理（老版本无自动清理时）**
```bash
# 每天凌晨2点清理，保留最新5个快照
0 2 * * * /opt/zookeeper/bin/zkCleanup.sh /data/zookeeper 5
```

**注意：**
- 不要把所有快照都删了，至少保留 **3个快照**，以便故障时数据回滚
- 清理操作要在业务低峰期，避免清理期间 I/O 压力影响服务
- 监控磁盘使用率，在快满（>80%）时提前告警，不要等到撑满

###### 6. Zookeeper 的内存管理机制是怎样的?

**ZooKeeper 全量数据存在内存（DataTree），这是其读性能高的根本原因。**

**内存数据结构：**
- `DataTree`：整棵 ZNode 树，用 `ConcurrentHashMap<String, DataNode>` 存储（路径→节点）
- `DataNode`：包含 data（byte[]）、stat（StatPersisted）、children（HashSet<String>）
- `WatchManager`：Watcher 注册表（path → Set<Watcher>）
- `SessionTracker`：会话表（sessionId → SessionImpl）

**内存估算：**
- 每个 ZNode 基础开销约 200 字节（不含数据）
- 1万个 ZNode 约 2MB，100万个 ZNode 约 200MB
- 生产中 ZK 通常配置 4~8GB 堆内存

**JVM GC 对 ZooKeeper 的影响：**
- Full GC 期间 ZooKeeper 停止响应，可能导致 Session 超时告警
- 推荐使用 **G1GC**（-XX:+UseG1GC），控制停顿时间 < 200ms
- 避免使用 CMS（已废弃）和 ZGC（生产环境 ZK 版本可能不够新）

```bash
# 推荐 JVM 参数
ZK_SERVER_HEAP="-Xmx4g -Xms4g -XX:+UseG1GC -XX:MaxGCPauseMillis=200 \
  -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/data/zookeeper/logs/"
```

###### 7. 什么情况下会导致 Zookeeper 性能下降?

**常见的性能下降场景及原因：**

**① 大量小写请求涌入**
- 写请求集中在 Leader，ZAB 广播开销叠加，TPS 下降
- 解决：业务层批量写（Multi 操作），或评估是否真的需要 ZK

**② Watcher 过多/频繁触发**
- 服务注册发现场景下，大量客户端监听同一个父路径
- 该路径子节点频繁变化（服务频繁上下线），触发大量 Watcher 通知
- 导致 Leader 的推送压力大，网络带宽打满
- 解决：减少不必要的 Watcher，使用推拉结合（订阅+定时拉取）

**③ 数据节点过大**
- 某个 ZNode 存了大量数据（接近 1MB）
- 每次读写都需要传输大块数据，影响吞吐
- 解决：ZK 只存元数据，大数据存 DB/OSS，ZK 存引用路径

**④ 磁盘 I/O 成为瓶颈**
- 事务日志和快照在同一磁盘，互相抢 I/O
- 解决：`dataLogDir` 单独挂 SSD

**⑤ GC 停顿**
- ZooKeeper 进程 Full GC 时完全停止，所有客户端的心跳超时，大量 Session 超时
- 解决：合理配置 JVM 堆内存 + G1GC

**⑥ 网络抖动（跨机房部署）**
- Leader 需要等待 Follower ACK，跨机房高延迟直接影响写延迟
- 解决：同机房部署 Leader 和 Follower，跨机房的节点配置为 Observer

> **追问：ZooKeeper 的 snapCount 参数是什么？应该怎么调？**
>
> `snapCount` 控制"每多少次写事务触发一次快照"，默认值是 **100000**。
>
> **触发机制：**
> 实际上不是精确在第 snapCount 次触发，而是在 `snapCount/2 ~ snapCount` 之间随机触发（加入随机性防止集群中所有节点同时生成快照，引发 I/O 峰值）。
>
> **调整建议：**
> - 写频繁的场景：适当**增大** snapCount（如 500000），减少快照频率，降低 I/O 压力
> - 磁盘空间紧张：**减小** snapCount，配合 `autopurge` 快速清理旧日志
> - 崩溃恢复慢（日志太多）：**减小** snapCount，加快恢复速度（快照越新，需要重放的日志越少）
>
> 一般不需要调，默认 100000 在大多数场景下够用。
