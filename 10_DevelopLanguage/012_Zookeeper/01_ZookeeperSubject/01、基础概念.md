# 一、基础概念

> 双链：[[37_ZookeeperKnowledge/01_核心概念与数据模型]]

---

###### 1. ZooKeeper 是什么?

ZooKeeper 是 Apache 的一个**分布式协调服务**，本质上是一个**高可用的分布式数据存储系统**，专门用来解决分布式系统中的协调问题。

你可以把它理解成一个"分布式的配置中心 + 通知系统 + 锁服务"的合体。它提供一个类似文件系统的树形数据模型，客户端可以读写节点、监听节点变化，以此来实现各种分布式协调场景。

核心特点：
- **树形数据模型**：数据组织成类似文件系统的路径结构，如 `/app/config/db`
- **强一致性**：基于 ZAB 协议，保证顺序一致性（CP 模型）
- **Watcher 机制**：客户端可以订阅节点变化，服务端主动推送通知
- **高可用集群**：主从架构（Leader/Follower/Observer），半数以上节点存活即可工作
- **轻量级**：不适合存储大量业务数据，每个节点（ZNode）最大只能存 **1MB**

类比：ZooKeeper 就像一个"分布式公告栏"——谁想干什么先在上面写一笔，其他人订阅这个公告，有变动立刻通知。

---

###### 2. ZooKeeper 能做什么?

ZooKeeper 的核心价值在于提供**分布式协调能力**，具体能做：

| 能力 | 典型应用 |
|------|---------|
| 服务注册与发现 | Dubbo 服务注册、Spring Cloud ZK |
| 配置中心 | 动态配置下发、热更新 |
| 分布式锁 | 基于临时顺序节点实现互斥锁 |
| Leader 选举 | Kafka、HBase 的主节点选举 |
| 集群管理 | 监控节点存活、感知节点变化 |
| 命名服务 | 全局唯一 ID 生成 |
| 分布式队列 | 基于顺序节点实现 FIFO 队列 |

记忆口诀：**配注锁选管命队**（配置/注册/锁/选举/集群管理/命名/队列）

###### 3. 你说说 Zookeeper 文件系统

ZooKeeper 使用**树形层次化的命名空间**（Hierarchical Name Space），和 Unix 文件系统非常相似：

```
/
├── app
│   ├── config
│   │   ├── db-url  (存储数据库地址)
│   │   └── redis-url
│   └── services
│       ├── order-service-192.168.1.1  (临时节点，服务注册)
│       └── order-service-192.168.1.2
└── locks
    └── order-lock  (分布式锁节点)
```

核心特点：
- 路径必须是**绝对路径**，从 `/` 开始
- 每个路径就是一个 **ZNode**（数据节点）
- ZNode 既可以存数据（最大1MB），也可以有子节点
- 路径唯一标识一个 ZNode，类似文件的绝对路径

和普通文件系统不同的是：ZooKeeper 的每个"文件夹"本身也可以存数据，不像 Linux 中目录和文件是分开的。

---

###### 4. Zookeeper 的系统架构又是怎么样的?

ZooKeeper 集群是**主从架构**，包含三种角色：

```
客户端 → Leader ← → Follower × N
                 ↘  Observer × N (可选)
```

**Leader（领导者）**：
- 集群唯一，处理所有**写请求**
- 负责发起事务提议（Proposal）并广播给 Follower
- 负责协调 Follower 进行数据同步

**Follower（跟随者）**：
- 处理**读请求**（直接响应客户端）
- 写请求转发给 Leader
- 参与 **Leader 选举**和**写事务投票**（半数以上同意才提交）

**Observer（观察者）**：
- 只处理读请求，不参与选举和投票
- 作用是**扩展读性能**，不影响写性能（因为不参与投票，增加 Observer 不影响写吞吐）

客户端连接任意节点，读请求就地处理，写请求转发给 Leader，Leader 走 ZAB 协议完成两阶段提交。

###### 5. 说说 zookeeper 有哪些数据节点

ZooKeeper 的 ZNode 分 **4 种类型**，按"是否持久" × "是否有序"组合：

| 类型 | 命令标志 | 特点 |
|------|---------|------|
| **持久节点**（PERSISTENT） | 默认 | 客户端断开后依然存在，需手动删除 |
| **持久顺序节点**（PERSISTENT_SEQUENTIAL） | `-s` | 持久 + 路径末尾自动追加递增序号，如 `/lock-0000000001` |
| **临时节点**（EPHEMERAL） | `-e` | 客户端会话结束自动删除，常用于服务注册 |
| **临时顺序节点**（EPHEMERAL_SEQUENTIAL） | `-es` | 临时 + 自动递增序号，**分布式锁的核心利器** |

> 从 3.6.0 版本起新增了 **Container 节点**（容器节点，子节点全部删除后自己也会被删除）和 **TTL 节点**（超时自动删除），但用得少。

临时节点是 ZooKeeper 最精妙的设计：服务宕机 → 会话超时 → 临时节点自动消失 → Watcher 触发 → 其他节点感知。整个过程无需任何主动心跳判断。

###### 6. Znode 里面都存储了什么?

每个 ZNode 包含两部分：**数据内容（data）** 和 **元数据（stat 结构）**。

stat 元数据字段：

| 字段 | 含义 |
|------|------|
| `czxid` | 节点被创建时的事务 ID |
| `mzxid` | 节点最后一次被修改时的事务 ID |
| `ctime` | 创建时间戳（ms） |
| `mtime` | 最后修改时间戳（ms） |
| `version` | 数据版本号（每次修改 +1） |
| `cversion` | 子节点版本号 |
| `aversion` | ACL 版本号 |
| `ephemeralOwner` | 临时节点的 Session ID；持久节点为 0 |
| `dataLength` | 数据长度（字节） |
| `numChildren` | 子节点数量 |
| `pzxid` | 子节点最后一次变更的事务 ID |

version 字段是实现**乐观锁**的基础：修改时指定期望的 version，ZooKeeper 内部做 CAS 校验，version 不匹配则拒绝修改。

---

###### 7. Zookeeper 都有哪些功能?

ZooKeeper 提供的核心功能：

1. **统一命名服务**：树形路径作为全局唯一名称，用于服务寻址
2. **配置管理**：动态配置下发，客户端通过 Watcher 感知配置变化
3. **集群管理**：通过临时节点感知集群成员的加入/离开
4. **分布式锁**：基于临时顺序节点实现公平互斥锁，天然防死锁
5. **Master 选举**：通过抢创建临时节点来决定谁是 Master
6. **发布/订阅**：数据发布到 ZNode，订阅者通过 Watcher 感知变化
7. **分布式队列**：顺序节点天然保证入队顺序

###### 8. Zookeeper 的 ACL 权限控制机制是什么?

ACL（Access Control List）是 ZooKeeper 的**节点级别权限控制机制**，每个 ZNode 都有自己的 ACL 列表。

ACL 由三部分组成：`schema:id:permissions`

**schema（认证方案）**：
- `world`：任何人，`world:anyone` 代表所有客户端
- `auth`：已认证的用户（当前会话认证过的用户）
- `digest`：用户名+密码认证（密码 SHA1 + Base64 加密）
- `ip`：基于 IP 地址认证，如 `ip:192.168.1.0/24`

**permissions（权限）**：
| 权限 | 缩写 | 含义 |
|------|------|------|
| CREATE | c | 创建子节点 |
| READ | r | 读取节点数据和子节点列表 |
| WRITE | w | 修改节点数据 |
| DELETE | d | 删除子节点 |
| ADMIN | a | 设置 ACL |

注意：ZooKeeper 的 ACL 只对单个节点有效，**不递归**——父节点有权限不代表子节点也有权限，每个节点的 ACL 独立设置。

###### 9. 什么是 ZNode 的版本号(version)?

ZooKeeper 的版本号是**乐观并发控制（OCC）**的核心机制，每个 ZNode 有三个版本号：

- **version**：节点数据的版本，每次调用 `setData` 成功后 +1
- **cversion**：子节点列表的版本，每次子节点增删后 +1
- **aversion**：ACL 版本，每次修改 ACL 后 +1

**版本号的作用——乐观锁：**

```java
// 读取当前 version
Stat stat = new Stat();
byte[] data = zk.getData("/config/db", false, stat);
int currentVersion = stat.getVersion();  // 假设为 3

// 修改时传入期望 version，若 ZK 上 version 不等于 3 则抛异常
zk.setData("/config/db", newData, currentVersion);
```

类比数据库的 `WHERE version = 3` 条件更新，CAS 思想，避免丢失更新。

version 从 0 开始，-1 表示"不关心版本，强制覆盖"。

###### 10. Zookeeper 的数据模型是什么样的?

ZooKeeper 的数据模型是一棵**有根节点（/）的树**，官方叫做 **ZNode Tree**（ZNode 树）。

核心特征：
1. **路径唯一性**：每个 ZNode 用绝对路径唯一标识，路径不允许重复
2. **数据与结构共存**：每个 ZNode 既可以存数据（最大 1MB），也可以有子节点
3. **类型多样**：持久/临时/顺序/TTL 四大类型，组合出不同的生命周期和行为
4. **版本控制**：data/children/acl 三个维度的版本号，支持乐观锁
5. **原子操作**：ZooKeeper 3.4+ 支持 Multi 批量原子操作，要么全成功要么全失败
6. **不适合大数据**：节点数据上限 1MB，设计目标是存少量协调数据，不是 KV 存储

整棵树存在内存中（同时 WAL 持久化），所以读取极快，但内存限制了存储容量——生产中 ZNode 数量通常控制在几十万级别。

---

> **追问：ZooKeeper 的 ZNode 为什么限制最大 1MB？不能存更多吗？**
>
> 这是个设计权衡。ZooKeeper 把所有数据存在**内存**中以保证读性能，同时通过 ZAB 协议把写操作广播给所有 Follower。
>
> 如果允许大数据：
> - 内存占用爆炸（几万个节点 × 几MB = GB 级内存）
> - 每次写广播需要把整个数据包发给所有 Follower，带宽压力和延迟都会变高
> - 崩溃恢复时重放日志也会更慢
>
> 所以 ZooKeeper 的定位很明确：**存协调元数据，不存业务数据**。需要大文件？存 HDFS/OSS，把路径存 ZooKeeper 就够了。
