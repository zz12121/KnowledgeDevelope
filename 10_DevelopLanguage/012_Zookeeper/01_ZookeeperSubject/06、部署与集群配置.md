# 六、部署与集群配置

> 双链：[[37_ZookeeperKnowledge/04_会话管理与集群配置]]

---

###### 1. Zookeeper 有哪几种几种部署模式?

ZooKeeper 有 **3 种部署模式**：

**① 单机模式（Standalone）**
- 只有一台服务器，适合开发调试
- 无高可用，宕机即全挂
- 配置简单，`zoo.cfg` 里不需要 `server.x` 配置项

```bash
# 启动单机模式
bin/zkServer.sh start
```

**② 伪集群模式（Pseudo-Cluster）**
- 一台物理机上运行多个 ZooKeeper 实例
- 每个实例用不同的端口（clientPort、数据端口、选举端口）
- 适合本地开发测试，不适合生产

```
# 3个实例，端口分别：
# instance1: clientPort=2181, 2881, 3881
# instance2: clientPort=2182, 2882, 3882
# instance3: clientPort=2183, 2883, 3883
```

**③ 集群模式（Cluster）**
- 多台物理机各运行一个 ZooKeeper 实例，组成集群
- 生产环境标准部署，建议**奇数台**（3/5/7台）
- 半数以上节点存活即可正常工作

---

###### 2. 集群最少要几台机器,集群规则是怎样的?集群中有 3 台服务器,其中一个节点宕机,这个时候 Zookeeper 还可以使用吗?

**集群最少需要 3 台**（2 台没意义，1 台宕机后剩 1 台，1≤2/2，无法构成多数派）。

**集群规则（半数以上原则）：**

存活节点数 > 总节点数 / 2 时，集群正常工作

| 总节点数 | 需要存活 | 允许宕机 |
|---------|---------|---------|
| 3 | ≥ 2 | 1 |
| 5 | ≥ 3 | 2 |
| 7 | ≥ 4 | 3 |

**3 台服务器，1 台宕机：还可以使用。**

- 存活：2台，总数：3台
- 2 > 3/2 = 1.5 → **满足多数派**，集群正常工作
- 写请求：Leader 获得 2 个 ACK（包括自己），超过半数，可以提交
- 读请求：正常服务

如果再宕机 1 台（只剩 1 台）：1 ≤ 1.5，不满足多数派，集群进入不可用状态。

###### 3. 集群支持动态添加机器吗?

**支持，但不同版本有差异。**

**ZooKeeper 3.5 之前：不支持动态扩容**
- 必须停掉整个集群，修改所有节点的 `zoo.cfg`，重启
- 维护成本高，有短暂停机时间

**ZooKeeper 3.5+ 支持动态重配置（Dynamic Reconfiguration）**
- 通过 `reconfig` 命令在线添加/删除节点，无需停机
- 需要开启 `reconfigEnabled=true`

```bash
# 动态添加节点（使用 zkCli.sh）
reconfig -add server.4=hostname4:2888:3888:participant

# 动态删除节点
reconfig -remove 4

# 查看当前配置
config
```

**注意事项：**
- 动态变更会触发选举（因为集群构成改变）
- 建议每次只增减一台，一次性变更过多节点可能导致不稳定
- Curator 3.x 以上对动态重配置有完整支持

###### 4. Zookeeper 负载均衡和 nginx 负载均衡区别

这是一道"比较不同层次负载均衡"的题，两者根本不是同一种东西。

**ZooKeeper 的"负载均衡"：**
- ZooKeeper 本身不做负载均衡，它提供的是**服务注册与发现**的基础设施
- 基于 ZooKeeper 的负载均衡流程：
  1. 服务提供方注册到 ZooKeeper（临时节点）
  2. 服务消费方从 ZooKeeper 获取可用节点列表
  3. 消费方在**客户端**按某种策略（轮询/随机/权重）选择目标节点
- 这是**客户端负载均衡**，均衡逻辑在调用方，中间无代理
- 代表：Dubbo + ZooKeeper，Ribbon

**Nginx 负载均衡：**
- Nginx 是**反向代理**，请求先到 Nginx，Nginx 再转发给后端
- 这是**服务端负载均衡**，调用方不感知后端节点列表
- 均衡策略：轮询/权重/ip_hash/least_conn 等
- 代表：Web 集群、API Gateway

| 对比维度 | ZooKeeper 负载均衡 | Nginx 负载均衡 |
|---------|------------------|--------------|
| 均衡位置 | 客户端 | 服务端（代理层） |
| 节点发现 | 动态（ZK 实时更新） | 静态（配置文件，需手动更新或配合服务发现） |
| 网络跳数 | 1跳（直连目标） | 2跳（经过Nginx） |
| 适用场景 | RPC/微服务内部调用 | HTTP 请求/南北向流量 |

###### 5. Zookeeper 的配置文件(zoo.cfg)中重要参数有哪些?

`zoo.cfg` 是 ZooKeeper 的核心配置文件，关键参数：

**基础参数：**
```properties
# 基本时间单位（毫秒），其他时间参数的倍数基准
tickTime=2000

# 快照文件和 myid 文件存储目录（必填）
dataDir=/var/lib/zookeeper/data

# 事务日志存储目录（建议和 dataDir 分开，避免 I/O 竞争）
dataLogDir=/var/lib/zookeeper/log

# 客户端连接端口
clientPort=2181

# Follower 初始连接 Leader 的超时：initLimit × tickTime
initLimit=10

# Follower 与 Leader 保持同步的超时：syncLimit × tickTime
syncLimit=5
```

**集群配置：**
```properties
# 集群成员配置：server.myid=host:数据端口:选举端口[:observer]
server.1=zoo1:2888:3888
server.2=zoo2:2888:3888
server.3=zoo3:2888:3888
server.4=zoo4:2888:3888:observer  # Observer 节点
```

**性能调优参数：**
```properties
# 是否允许 Leader 接受客户端连接（关闭让 Leader 专注协调）
leaderServes=yes

# 快照保留数量（防止磁盘积累太多快照）
autopurge.snapRetainCount=3

# 自动清理触发间隔（小时），0表示不自动清理
autopurge.purgeInterval=1

# 每个客户端最大连接数（防止单个客户端占满连接）
maxClientCnxns=60

# 最大数据包大小（字节），默认 1MB
jute.maxbuffer=1048576
```

###### 6. Zookeeper 如何进行扩容和缩容?

**扩容（添加节点）：**

ZooKeeper 3.5 以前（停机扩容）：
1. 停止所有 ZooKeeper 服务
2. 在新节点上安装 ZooKeeper，创建 `myid` 文件
3. 修改所有节点的 `zoo.cfg`，加入新节点的 `server.x` 配置
4. 重启所有节点

ZooKeeper 3.5+ （动态扩容）：
```bash
# 新节点先以 Observer 模式加入（更安全，不影响投票）
# 新节点 zoo.cfg 临时配置：
peerType=observer

# 等数据同步完成后，使用 reconfig 提升为 Participant
zkCli.sh -server zoo1:2181
reconfig -add server.4=zoo4:2888:3888:participant
```

**缩容（删除节点）：**

ZooKeeper 3.5+ ：
```bash
# 优先将目标节点降为 Observer 再移除（减少影响）
reconfig -add server.3=zoo3:2888:3888:observer
# 等待数据稳定后
reconfig -remove 3
```

**注意事项：**
- 扩容前确认新节点的数据目录是空的（全新加入），或者与 Leader 兼容
- 缩容时注意不要让存活节点数降到半数以下
- 生产环境变更建议在业务低峰期操作

###### 7. 什么是 Zookeeper 的 Quorum(法定人数)?

**Quorum（法定人数）** 是 ZooKeeper 集群做出决策所需的**最小同意节点数**，公式：

```
Quorum = N/2 + 1（向下取整的 N/2 再加1）
```

即：超过半数节点。

**Quorum 的应用场景：**
1. **Leader 选举**：候选节点获得 ≥ Quorum 票才能成为 Leader
2. **写事务提交**：Leader 收到 ≥ Quorum 个 ACK 才提交事务
3. **集群可用性判断**：存活节点数 ≥ Quorum 时集群才对外服务

**为什么是半数以上？**

多数派原则保证：任意两个满足 Quorum 的集合，必定有交集（共同节点）。

这个共同节点保证了"旧 Leader 提交的最后一个事务"至少被一台参与新 Leader 选举的节点看到，从而不会丢失。

举例（5台）：
- 写操作要3台确认 → 至少有3台有最新数据
- 选举要3票 → 新 Leader 必须从3台中选出
- 3台和3台必有交集（至少1台重叠）
- 这台重叠的节点有最新数据，确保新 Leader 不丢数据

###### 8. 为什么 Zookeeper 集群建议使用奇数台服务器?

核心原因：**奇数台和偶数台（少1台）的容错能力相同，但成本更低**。

ZooKeeper 集群正常工作需要：**存活节点数 > 总节点数 / 2**

| 节点数 | 允许宕机数 | 容错率 |
|-------|---------|------|
| 3 | 1 | 33% |
| **4** | **1** | **25%** |
| 5 | 2 | 40% |
| **6** | **2** | **33%** |
| 7 | 3 | 43% |

- **4 台 vs 3 台**：容错数都是 1，4 台多花一台机器却没有提升容错能力
- **6 台 vs 5 台**：容错数都是 2，同样浪费一台机器

所以**偶数台是浪费**——花了钱，没买到额外的安全。奇数台是性价比最优解。

另外，奇数台在网络分区时也更安全：
- 5台 发生 3+2 分区：3台那侧可以选出 Leader，2台那侧不行
- 4台 发生 2+2 分区：两侧都无法选出 Leader，整个集群不可用

所以奇数台还能更好地处理对称分区。

> **追问：ZooKeeper 集群中某个节点磁盘满了怎么处理？**
>
> ZooKeeper 的事务日志和快照会持续增长，磁盘满会导致节点无法写入日志，进而退出集群。
>
> **预防措施：**
> 1. 配置 `autopurge.snapRetainCount=3` 和 `autopurge.purgeInterval=1`，让 ZooKeeper 自动清理旧快照
> 2. `dataDir`（快照）和 `dataLogDir`（事务日志）分开到不同磁盘，避免日志和快照互相抢空间
> 3. 配置磁盘使用率监控告警（如 Prometheus + Node Exporter，磁盘使用率 > 80% 告警）
>
> **应急处理：**
> 1. 临时清理：手动删除旧快照，保留最新 3 个；`zkCleanup.sh` 或手动执行
>    ```bash
>    # 清理快照和日志（保留最新3个）
>    java -cp zookeeper.jar:lib/* org.apache.zookeeper.server.PurgeTxnLog dataDir dataLogDir -n 3
>    ```
> 2. 扩展磁盘：挂载新磁盘，迁移 dataLogDir
> 3. 先将该节点从集群摘除（reconfig -remove），处理完后重新加入
