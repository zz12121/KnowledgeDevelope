# 五、会话管理

> 双链：[[37_ZookeeperKnowledge/04_会话管理与集群配置]]

###### 1. 熟悉会话管理吗

ZooKeeper 客户端与服务端之间通过**会话（Session）** 建立关联，会话是 ZooKeeper 运作的基础。

**会话的作用：**
- 维持客户端与服务端的连接状态
- 临时节点（EPHEMERAL）的生命周期绑定到 Session——Session 失效，临时节点自动删除
- 保证请求的顺序性（同一 Session 内的请求按顺序处理）
- 存储 Watcher 注册信息（Session 断开时，服务端自动清理该 Session 的 Watcher）

**会话的关键属性：**
- **Session ID**：全局唯一的会话标识符（64位Long），由 Leader 分配
- **Session Timeout**：客户端设置的超时时间（建议 30s，最小 2×tickTime）
- **isClosing**：会话是否正在关闭（用于服务端不再接受该会话的新请求）
- **Tick**：会话在时间桶（ExpiryQueue）中的过期时间槽

**会话创建流程：**
1. 客户端发起 TCP 连接到任意 ZooKeeper 服务器
2. 发送 `ConnectRequest`，携带上次的 Session ID（0表示新建）和 Session Timeout
3. 服务端创建 Session，分配全局唯一 Session ID，响应 `ConnectResponse`
4. 连接建立，客户端进入 `CONNECTED` 状态

###### 2. 了解 Chroot 特性吗

Chroot（Change Root，改变根目录）是 ZooKeeper 客户端的一个**命名空间隔离**特性。

**作用：**
限定客户端只能操作 ZooKeeper 数据树的某个子树，实现多租户隔离，防止误操作影响其他应用的数据。

**使用方式：**
在连接字符串中指定路径作为根：
```java
// 不使用 Chroot：可以访问整个 ZK 数据树
ZooKeeper zk = new ZooKeeper("host1:2181,host2:2181", 30000, watcher);

// 使用 Chroot：所有操作都限定在 /myapp 下
ZooKeeper zk = new ZooKeeper("host1:2181,host2:2181/myapp", 30000, watcher);

// 此时 getData("/config") 实际访问的是 /myapp/config
zk.getData("/config", false, null);  // 等价于不带Chroot时的 /myapp/config
```

**效果：**
- 客户端路径 `/config` → 实际路径 `/myapp/config`
- 客户端看不到 `/myapp` 以外的路径
- 实现应用间的命名空间隔离，互不干扰

**Dubbo 的实践：** Dubbo 默认使用 `/dubbo` 作为 Chroot 根节点，所有服务注册信息都在 `/dubbo` 下，不影响其他应用。

###### 3. Zookeeper 的会话超时机制是什么?

会话超时是 ZooKeeper 保证**临时节点自动清理**和**故障检测**的核心机制。

**超时判断原理：**
- 客户端每隔 `tickTime/3` 发送一次心跳（PING）到服务端
- 服务端使用**分桶策略**（ExpiryQueue）管理会话超时：
  - 按 `expirationInterval`（默认等于 tickTime=2000ms）将时间划分为多个桶
  - 每个 Session 放入其对应的过期时间桶中
  - 服务端定期检查当前时间对应的桶，该桶内的 Session 全部视为超时

**心跳与超时时序：**
```
客户端：─PING─────────PING─────────PING── ...
服务端：       刷新Session过期时间
超时：  未在 SessionTimeout 内收到心跳 → 关闭Session → 删除临时节点 → 触发Watcher
```

**超时时间协商：**
- 客户端创建连接时设置期望的超时时间
- 服务端有最小超时（2×tickTime）和最大超时（20×tickTime）限制
- 服务端返回实际生效的超时时间（在限制范围内协商）

**客户端断开重连时：**
- 断开 ≠ 超时，只要在超时时间内重连，Session 继续有效，临时节点保留
- 超时后才视为 Session 失效，临时节点被清除

###### 4. 什么是 Session ID 和 Session Password?

**Session ID：**
- 64位全局唯一标识符，由 Leader 生成
- 生成规则：`高位是 myid（防止不同机器生成相同ID），低位是递增计数器`
- 格式：`0x myid(1字节) + 时间戳 + 自增序号`
- 客户端用 Session ID 在重连时恢复会话（避免创建新 Session，保留临时节点）

**Session Password：**
- 16字节的随机密码，创建 Session 时服务端生成并返回给客户端
- 客户端需要妥善保存（ZooKeeper 原生客户端自动保存）
- 重连时客户端携带 Session ID + Session Password，服务端验证后恢复 Session
- 防止其他客户端伪造 Session ID 来"冒充"并恢复他人的会话

**实际使用场景（Curator 封装了这些细节）：**
```java
// 原生 API 获取 Session ID 和 Password
long sessionId = zk.getSessionId();
byte[] sessionPassword = zk.getSessionPasswd();

// 使用已有 Session 重建连接（适用于会话迁移）
ZooKeeper zk2 = new ZooKeeper(connectString, sessionTimeout, watcher, sessionId, sessionPassword);
```

###### 5. Zookeeper 客户端与服务端的心跳机制是怎样的?

**心跳机制的目的：**
- 告诉服务端"我还活着"，刷新 Session 过期时间
- 检测网络故障，触发重连

**心跳实现（原生客户端 SendThread）：**

```
readTimeout  = sessionTimeout × 2/3  （读超时，超过此时间没收到任何数据则重连）
pingInterval = sessionTimeout / 3     （主动发 PING 的间隔，默认 sessionTimeout=30s 则每10s PING一次）
```

**核心逻辑：**
1. `SendThread` 维护一个定时器，计算"最近一次收到服务端消息的时间"
2. 如果当前时间 - 最近接收时间 > `pingInterval`，发送 PING 请求
3. 如果当前时间 - 最近接收时间 > `readTimeout`，认为连接异常，触发重连

**PING/PONG 流程：**
```
客户端 SendThread:
  每隔 pingInterval 发送空的 RequestHeader（xid=-2，type=PING）
  
服务端 NIOServerCnxn:
  收到 PING，直接回复空响应
  
服务端 SessionTracker:
  每次收到任何请求（包括 PING），都刷新该 Session 的过期时间
```

**重连策略：**
- 断开后随机延迟重连（避免惊群）
- 重连到集群中随机一台服务器（不一定是原来的那台）
- 重连成功后，服务端验证 Session ID + Password，若 Session 未超时则恢复

###### 6. 会话的生命周期包含哪些状态?

ZooKeeper 客户端 Session 的生命周期状态（`KeeperState`）：

```
NOT_CONNECTED（初始）
      ↓  发起连接
CONNECTING
      ↓  连接建立、Session 创建成功
CONNECTED  ←─────────────── 正常工作
      ↓  网络断开               ↑
CONNECTING（尝试重连）──────────┘  重连成功
      ↓  超时时间内未重连成功
CLOSED / SESSION_EXPIRED
```

**各状态说明：**

| 状态 | 含义 |
|------|------|
| `NOT_CONNECTED` | 初始状态，尚未建立连接 |
| `CONNECTING` | 正在尝试连接/重连服务端 |
| `CONNECTED` | 已连接，Session 有效，正常工作 |
| `CONNECTEDREADONLY` | 连接到只读节点（少数派节点，只允许读） |
| `CLOSED` | 客户端主动关闭 Session（调用 `zk.close()`） |
| `AUTH_FAILED` | 认证失败 |
| `NOT_CONNECTED`（Session Expired后） | Session 超时，需要重新创建 Session |

**Session Expired 的影响：**
- 所有该 Session 创建的临时节点被删除
- 监听这些节点的 Watcher 收到 `NodeDeleted` 通知
- 客户端收到 `SessionExpiredException`，必须创建新的 ZooKeeper 实例重新连接

###### 7. Zookeeper 如何处理会话迁移?

**会话迁移**是指客户端从一台 ZooKeeper 服务器断开，然后重连到另一台服务器，原有 Session 继续有效的过程。

**会话迁移的前提：**
ZooKeeper 是集群，所有服务器共享相同的数据和 Session 信息（通过 ZAB 协议同步），所以换一台服务器连接，Session 依然有效。

**迁移流程：**
1. 客户端与服务器 A 断开连接（网络故障/主动断开/A 宕机）
2. 客户端进入 `CONNECTING` 状态，尝试连接集群中其他服务器
3. 客户端连接到服务器 B，发送 `ConnectRequest`，携带原 Session ID + Session Password
4. 服务器 B 查询本地 Session 表，验证 Session ID 和 Password
   - 如果 Session 仍有效（未超时）：恢复 Session，继续有效，临时节点保留
   - 如果 Session 已超时：返回 `SESSION_EXPIRED`，客户端需重新创建
5. 恢复成功：客户端回到 `CONNECTED` 状态，Watcher 自动重新注册（Curator 会处理）

**特殊情况——Owner 节点不匹配：**
- 如果某个临时节点是由特定 Session 创建的，Session 迁移后归属关系不变
- 新的服务器会在与 Leader 的数据同步后，正确维护该临时节点

**Curator 对会话迁移的封装：**
Curator 提供 `SessionFailRetryPolicy`，会话迁移和重建对业务层完全透明，业务只需处理 `SUSPENDED`（暂时断开）和 `LOST`（Session 超时）两个事件。

> **追问：会话超时时间设置多少合适？太长太短有什么问题？**
>
> **太短的问题：**
> - 网络抖动可能导致 Session 频繁超时，临时节点被误删，触发不必要的服务上下线事件
> - 分布式锁可能因为网络波动丢失，导致锁失效
>
> **太长的问题：**
> - 节点真正宕机后，临时节点长时间不消失，其他客户端感知延迟大
> - 分布式锁被持有的时间更长（死锁风险）
>
> **生产建议：**
> - 常规场景：`30秒`（tickTime=2000ms，sessionTimeout=30000ms）
> - 对故障感知要求高的场景（如服务发现）：`10~15秒`
> - 网络质量差的跨机房场景：`60~90秒`
> - 确保 `sessionTimeout >= 2 × tickTime`（ZooKeeper 的最小限制）
> - Curator 建议：`sessionTimeout` 至少是 `connectionTimeout` 的3倍
