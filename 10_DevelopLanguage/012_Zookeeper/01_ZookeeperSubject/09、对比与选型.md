###### 1. Zookeeper 和 Eureka、Consul、Nacos 有什么区别?

这是服务注册中心的经典对比题，核心是 **CAP 模型的选择差异**：

| 对比维度 | ZooKeeper | Eureka | Consul | Nacos |
|---------|-----------|--------|--------|-------|
| CAP 模型 | CP | AP | CP（可配置为AP） | CP + AP 可切换 |
| 一致性协议 | ZAB | 无（最终一致） | Raft | Raft |
| 健康检查 | Session 超时（临时节点） | 心跳 | 多种（HTTP/TCP/TTL/Script） | 心跳 + 主动检测 |
| 多数据中心 | 不支持 | 不支持 | 支持 | 支持 |
| 配置中心 | 基础支持 | 不支持 | 支持（KV） | 支持（完善UI） |
| 控制台 | 无（需第三方） | 有（基础） | 有 | 有（功能完善） |
| 生态 | 大数据/老一代微服务 | Spring Cloud Netflix | Hashicorp 生态 | Spring Cloud Alibaba |
| 推荐场景 | 强一致协调/大数据 | 微服务注册（容忍不一致） | 混合云/多DC | 国内微服务（推荐） |

**关键区别解读：**
- **ZooKeeper vs Eureka**：ZK 优先保证一致性（CP），Leader 选举期间服务发现不可用；Eureka 优先可用（AP），节点故障时旧数据依然可读
- **ZooKeeper vs Nacos**：Nacos 支持 CP/AP 切换，内置配置中心，UI 更好用，在新项目中 Nacos 是更优选择
- **ZooKeeper vs Consul**：Consul 同是 CP，但多数据中心支持更好，且有 Service Mesh（Consul Connect）能力

###### 2. chubby 是什么,和 zookeeper 比你怎么看?

**Chubby 是什么：**
Chubby 是 Google 内部的**分布式锁服务**，2006 年 Mike Burrows 在论文 "The Chubby Lock Service for Loosely-Coupled Distributed Systems" 中描述。
是 GFS、BigTable、MapReduce 等 Google 核心系统的协调基础设施，ZooKeeper 的设计灵感就来自 Chubby。

**Chubby vs ZooKeeper：**

| 维度 | Chubby | ZooKeeper |
|------|--------|-----------|
| 来源 | Google 内部闭源 | Apache 开源（Chubby 论文后开发） |
| 核心定位 | 分布式锁服务 | 分布式协调服务（更通用） |
| 数据模型 | 类似文件系统（File + Lock） | 树形 ZNode（含锁语义） |
| 一致性协议 | Paxos | ZAB（ZooKeeper Atomic Broadcast） |
| 客户端缓存 | 支持客户端缓存（需一致性协议同步） | 不支持客户端缓存 |
| 事件通知 | 事件通知机制（类似 Watcher） | Watcher 机制 |
| 锁的语义 | 一等公民（显式的锁 API） | 通过临时顺序节点模拟 |

**总结：** ZooKeeper 是 Chubby 的开源"精神继承者"，API 和功能比 Chubby 更通用灵活，但由于 Chubby 是 Google 内部系统，外部无法直接使用，工程上 ZooKeeper 才是事实标准。

###### 3. Zookeeper 和 etcd 的区别是什么?

**etcd** 是 CoreOS（现为 CNCF 项目）开发的分布式 KV 存储，是 Kubernetes 的核心元数据存储。

| 对比维度 | ZooKeeper | etcd |
|---------|-----------|------|
| 一致性协议 | ZAB（ZooKeeper 专用） | Raft（更通用、易理解） |
| 数据模型 | 树形 ZNode（有层级结构） | 扁平 KV（可用前缀模拟层级） |
| Watch 机制 | 一次性 Watcher（3.6+ 支持持久） | 持久的 Watch，支持前缀监听 |
| Lease（租约） | Session 超时机制（临时节点） | Lease 机制（更灵活，可绑定多个 key） |
| 性能 | 写性能一般（ZAB 两阶段） | 写性能更好（Raft 优化，顺序写） |
| 生态 | Hadoop/大数据/旧一代微服务 | Kubernetes/云原生（CNCF） |
| 语言 | Java | Go |
| 部署复杂度 | 较复杂（Java 环境/Quorum 配置） | 相对简单（单二进制文件） |
| 适用场景 | 大数据协调/分布式锁 | K8s 元数据存储/云原生服务发现 |

**选择建议：**
- 云原生/K8s 场景：etcd（K8s 内置，生态天然集成）
- 大数据/Hadoop/HBase：ZooKeeper（天然支持，深度集成）
- 新的微服务注册中心：Nacos（功能更完善，运维更友好）

###### 4. 在什么场景下应该选择 Zookeeper?

选 ZooKeeper 的几个明确信号：

**① 已有大数据生态**
- 使用 HBase、Hadoop YARN、旧版 Kafka 的项目，ZooKeeper 是必选

**② 强一致性的分布式协调**
- 分布式锁（对锁的可靠性要求高，不能因为 Redis 主从切换导致锁丢失）
- Master 选举（K8s 环境用 etcd，非 K8s 用 ZooKeeper）
- 跨服务的配置协调（需要保证所有节点一致后才能继续）

**③ 依赖 Dubbo 的老系统**
- Dubbo 默认注册中心是 ZooKeeper，老系统迁移成本高

**不适合选 ZooKeeper 的场景：**
- 高吞吐写场景（ZK 写性能瓶颈在 Leader 单点 + ZAB 广播）
- 大量数据存储（节点上限 1MB，不是 KV 数据库）
- 需要多数据中心（etcd / Consul 更合适）
- 新的微服务注册中心（Nacos 更推荐）

---

###### 5. Zookeeper 的 CP 特性适合什么场景?

ZooKeeper 是 CP 模型（牺牲可用性换一致性），适合以下场景：

**① 分布式锁**
- 对锁的正确性要求高：不能出现两个客户端同时持锁
- ZooKeeper 的 CP 保证：即使在网络分区时，少数派节点也不提供服务，不会出现双 Leader

**② 服务协调和 Master 选举**
- 要求全局只有一个 Master：ZooKeeper 的强一致性保证选举结果全局可见

**③ 配置中心（强一致要求）**
- 数据库连接串、加密密钥等关键配置，需要保证所有节点同时看到最新值
- ZooKeeper 写操作强一致，适合这类场景

**④ 顺序保证场景**
- 需要全局事件顺序的协调逻辑，ZooKeeper 的 ZXID 全局有序天然满足

**不适合的场景：**
- 服务注册发现（注册中心对可用性要求更高，宁可读到稍旧的列表也不能拒绝服务）
- 超高频读写（CP 的代价是写吞吐低）

###### 6. Zookeeper 与 Redis 分布式锁的区别?

> 双链：[[33_DistributedKnowledge/04_分布式服务治理]]

| 对比维度 | ZooKeeper 分布式锁 | Redis 分布式锁（Redisson） |
|---------|------------------|--------------------------|
| 实现原理 | 临时顺序节点 + Watcher | SET key ... NX EX / Lua 脚本 |
| 锁自动释放 | Session 超时后临时节点自动删除（无需设置超期时间，天然防死锁） | 必须设置 TTL，时间难以评估（太长死锁，太短误释放） |
| 可靠性（主从切换） | 高（CP 模型，多数派写入后才成功） | 中（主从切换时可能锁丢失，RedLock 可缓解但有争议） |
| 性能 | 低（ZAB 两阶段提交，每次加锁至少需要一次 Leader 写） | 高（内存操作，单机 10w+ QPS） |
| 公平性 | 天然公平（顺序节点，FIFO 排队） | 默认不公平，Redisson 提供公平锁（开销更大） |
| 可重入性 | Curator InterProcessMutex 支持可重入 | Redisson RLock 支持可重入 |
| Watch 支持 | 天然支持（Watcher 通知前一节点释放） | 需要轮询或 Pub/Sub |
| 部署复杂度 | 需要 ZooKeeper 集群 | 需要 Redis 集群 |
| 适用场景 | 对锁可靠性要求极高、可接受性能损失 | 高并发、对性能敏感、可接受极低概率锁丢失 |

**选型口诀：**
- 钱（金融/订单）→ ZooKeeper（宁可慢，不能错）
- 量（秒杀/限流）→ Redis（宁可快，不过精确）

---

> **追问：为什么说 Redis 主从切换时分布式锁可能丢失？**
>
> Redis 主从同步是**异步的**：客户端在 Master 写入锁成功后，Master 立刻响应客户端"加锁成功"，此时 Slave 可能还未收到这条写入。
>
> 如果 Master 在同步完成前宕机，Slave 被提升为新 Master，新 Master 上没有这把锁的记录。此时另一个客户端可以成功加锁，出现两个客户端同时持锁的情况。
>
> **Martin Kleppmann vs Redis 作者（Antirez）的红锁之争：**
> - Antirez 提出 RedLock：在多个独立 Redis 主节点上加锁，超过半数成功才算加锁成功，降低单节点故障的影响
> - Martin 指出：即使 RedLock 也无法在进程暂停（GC Stop-The-World）+ 时钟漂移的情况下保证锁的正确性，建议用基于 token 的 fencing 方案
>
> 结论：Redis 分布式锁适合"工程上够用"的场景，不适合需要绝对严格正确性的场景（那就用 ZooKeeper）。
