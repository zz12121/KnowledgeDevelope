###### 1. Zookeeper 连接超时如何处理?

**连接超时的两个层次：**

**① ConnectionTimeout（连接建立超时）**
- 客户端发起 TCP 连接，在 `connectionTimeout` 内未能建立连接
- 原因：服务端不可达、端口被防火墙拦截、服务端负载过高
- 表现：`java.net.SocketTimeoutException` 或 `ConnectionLoss`

**② SessionTimeout（会话超时）**
- 连接成功后，在 `sessionTimeout` 内没有收到服务端心跳响应
- 原因：网络抖动、服务端 GC 停顿、服务端过载

**排查步骤：**
```bash
# 1. 检查服务端是否存活
echo ruok | nc zoo1 2181

# 2. 检查端口是否通
telnet zoo1 2181

# 3. 检查服务端负载
echo mntr | nc zoo1 2181 | grep outstanding

# 4. 检查网络延迟
ping zoo1
```

**解决方案：**
- 适当增大 `connectionTimeout`（客户端配置）：`CuratorFrameworkFactory.builder().connectionTimeoutMs(10000)`
- 增大 `sessionTimeout`：如果是网络抖动导致的，30s → 60s
- 检查服务端 JVM GC 停顿：GC 超过 sessionTimeout 就会触发大量 Session 过期
- 客户端配置重试策略：`ExponentialBackoffRetry` 自动重连

###### 2. 如何处理 Zookeeper 的 Session Expired?

**Session Expired 是比连接断开更严重的状态**——连接断开是临时的，Session 过期意味着服务端已经清理了该 Session 的所有临时节点和 Watcher。

**Session Expired 的影响：**
- 该 Session 创建的**所有临时节点被删除**
- 分布式锁丢失（临时节点被删除意味着锁被释放）
- 服务注册节点消失（其他节点认为该服务宕机）

**处理方式：**

**① 使用 Curator，监听 ConnectionState**
```java
client.getConnectionStateListenable().addListener((c, state) -> {
    switch (state) {
        case SUSPENDED:
            // 连接暂时断开，先暂停业务（锁可能还没过期）
            log.warn("ZK 连接暂时断开，暂停业务操作");
            pauseBusiness();
            break;
        case RECONNECTED:
            // 重连成功，Session 仍有效，恢复业务
            log.info("ZK 重连成功");
            resumeBusiness();
            break;
        case LOST:
            // Session 过期！必须重新初始化所有 ZK 相关资源
            log.error("ZK Session 过期，重新初始化");
            reinitializeZkResources();
            break;
    }
});
```

**② Session 过期后的重建流程**
```java
private void reinitializeZkResources() {
    // 1. 关闭旧的 Curator 客户端
    // 2. 重新创建 CuratorFramework
    // 3. 重新注册服务（临时节点）
    // 4. 重新抢锁（如果持有分布式锁）
    // 5. 重新注册 Watcher
}
```

**预防措施：**
- 合理设置 `sessionTimeout`（建议 30s~60s）
- 优化 JVM GC，避免 Stop-The-World 超过 sessionTimeout
- 使用 Curator 封装，避免手动处理复杂的重连逻辑

###### 3. Zookeeper 数据不一致如何排查?

ZooKeeper 保证的是**顺序一致性**（非强一致性），特定场景下确实可能读到旧数据。

**常见的"数据不一致"场景：**

**① 读 Follower 读到旧数据**
- 原因：写操作成功后，COMMIT 到 Follower 的传播有延迟
- 排查：比较 Leader 和 Follower 的同一路径数据
  ```bash
  # 直接连接 Leader 查询
  zkCli.sh -server leader_host:2181 getData /path
  # 连接 Follower 查询
  zkCli.sh -server follower_host:2181 getData /path
  ```
- 解决：读取强一致数据时，先调用 `sync()` 再读

**② 网络分区后少数派节点读到旧数据**
- 原因：少数派节点可能还在提供只读服务（`zookeeper.standalone.enabled=false` 且 `readonlymode.enabled=true`）
- 解决：确认 `isro` 四字命令返回，`rw` 表示可读写，`ro` 表示只读模式，只读模式下数据可能是旧的

**③ 客户端 Watcher 丢失，本地缓存和实际不一致**
- 原因：Session 重连后，Watcher 未重新注册
- 解决：使用 Curator NodeCache/PathChildrenCache，内部自动重新注册

**排查工具：**
```bash
# 检查各节点的 zxid 是否一致（zxid 越大越新）
echo stat | nc zoo1 2181 | grep Zxid
echo stat | nc zoo2 2181 | grep Zxid
echo stat | nc zoo3 2181 | grep Zxid
```

###### 4. 如何解决 Zookeeper 的羊群效应(Herd Effect)?

**羊群效应（Herd Effect）定义：**
当某个资源释放时，大量等待的客户端同时被唤醒，蜂拥而至竞争同一个资源，产生大量无效竞争，浪费系统资源。

**在 ZooKeeper 中的表现：**
如果用最朴素的方式实现分布式锁：所有等待锁的客户端都监听同一个锁节点，锁释放时所有等待者同时被唤醒，但只有一个能抢到锁，其他的重新等待。10个等待者就有10次无效唤醒。

**解决方案：临时顺序节点 + 只监听前一个节点**

```java
// 每个客户端创建临时顺序节点
String myNode = client.create().withMode(CreateMode.EPHEMERAL_SEQUENTIAL)
    .forPath("/locks/lock-");
// myNode 例如 /locks/lock-0000000003

// 获取所有节点排序
List<String> nodes = client.getChildren().forPath("/locks");
Collections.sort(nodes);
int myIndex = nodes.indexOf(myNode.replace("/locks/", ""));

if (myIndex == 0) {
    // 我是最小节点，获取锁
    acquireLock();
} else {
    // 监听前一个节点（不是最小节点）
    String prevNode = "/locks/" + nodes.get(myIndex - 1);
    // 等前一个节点消失，再重新判断
    watchAndWait(prevNode);
}
```

**效果：**
- 锁释放时，只有"前一个"等待者被唤醒
- 每次只唤醒一个客户端，形成有序排队，无羊群效应
- Curator `InterProcessMutex` 内部就是这样实现的

###### 5. 什么是 Zookeeper 的惊群问题?

**惊群（Thundering Herd）问题**与羊群效应类似，但更泛化：指一个事件触发导致大量进程/线程同时被唤醒，但只有少数能成功处理，其余白白消耗 CPU 和资源。

**在 ZooKeeper 中的具体表现：**

**场景一：大量客户端监听同一个节点**
- 如服务注册中心：100 个消费者都在 `getChildren` 监听 `/services/order/providers`
- 一个 Provider 上线，100 个 Watcher 同时触发，100 个客户端同时去 `getChildren` 拉新列表
- ZooKeeper 服务端同时处理 100 个读请求，瞬间压力倍增

**缓解方案：**
1. **随机延迟**：收到 Watcher 通知后，随机延迟 100~500ms 再去拉数据，将并发分散
2. **推拉结合**：不是每次 Watcher 触发都立刻拉数据，而是标记"有变化"，由定时任务定期批量拉取
3. **客户端本地缓存**：使用 Curator 的 PathChildrenCache，内部自动合并多个通知

**场景二：Leader 宕机后大量客户端同时重试**
- Leader 宕机 → 选举期间写失败 → 大量客户端收到 `ConnectionLoss`，立刻重试
- 新 Leader 上任后瞬间收到大量重试请求，可能过载

**缓解：**
- 客户端配置指数退避重试：`ExponentialBackoffRetry(1000, 5)` — 每次重试间隔指数增长

###### 6. Zookeeper 网络分区如何处理?

**网络分区（Network Partition）的影响：**

假设 5 台节点发生 3+2 分区：

**多数派（3台）侧：**
- 有 Leader 或重新选举出 Leader
- 正常提供读写服务
- 写操作只需 3 台 ACK，满足多数派要求

**少数派（2台）侧：**
- 默认情况下，无法达到多数派，拒绝所有写请求
- 如果开启了只读模式（`readonlymode.enabled=true`），允许读请求（但数据可能是旧的）
- 客户端会收到 `CONNECTING` 或只读模式通知

**恢复后的处理：**
1. 分区恢复，少数派节点重新连接到多数派的 Leader
2. Leader 根据少数派节点的 ZXID，决定 DIFF/TRUNC/SNAP 同步方式
3. 少数派节点同步完成后，重新加入集群提供服务

**脑裂防护（回顾）：**
- 只有多数派才能选出 Leader（半数以上原则），不可能出现两个 Leader
- epoch 机制保证旧 Leader 在分区恢复后主动降级

**运维建议：**
- 关闭只读模式（默认已关闭）：避免客户端读到分区期间的旧数据
- 跨机房部署：同机房 3 台，另一机房 2 台（Observer），保证主机房分区时另一机房不影响整体可用性

###### 7. Zookeeper 磁盘空间不足如何处理?

磁盘满是 ZooKeeper 最常见的运维故障之一，处理不当会导致节点崩溃。

**应急处理步骤：**

**① 立刻清理旧快照和日志（不要停服）**
```bash
# 确认当前磁盘使用情况
df -h /data/zookeeper
du -sh /data/zookeeper/log/version-2/*

# 保留最新3个快照，清理其余（使用官方工具，不要手动 rm）
java -cp /opt/zookeeper/lib/*:/opt/zookeeper/zookeeper-*.jar \
     org.apache.zookeeper.server.PurgeTxnLog \
     /data/zookeeper/data /data/zookeeper/log -n 3

# 或者使用 zkCleanup.sh
/opt/zookeeper/bin/zkCleanup.sh -n 5
```

**② 检查是否有异常大的日志文件**
```bash
# 查找大文件
find /data/zookeeper -name "*.log" -o -name "log.*" | xargs du -sh | sort -rh | head -20
```

**③ 临时挂载额外存储**
```bash
# 将 dataLogDir 迁移到更大的磁盘
# 1. 停止 ZK（如果来得及）
# 2. 拷贝现有日志到新磁盘
cp -r /data/zookeeper/log /new_disk/zookeeper/log
# 3. 修改 zoo.cfg: dataLogDir=/new_disk/zookeeper/log
# 4. 重启 ZK
```

**④ 永久解决：开启自动清理**
```properties
# zoo.cfg
autopurge.snapRetainCount=3
autopurge.purgeInterval=1
```

**预防措施：**
- `dataDir`（快照）和 `dataLogDir`（事务日志）分开到不同磁盘
- 配置磁盘使用率告警（建议 > 70% 告警，> 85% 触发自动清理）
- 生产环境每台 ZK 节点至少预留 **50GB** 专用磁盘空间（根据写入量调整）

> **追问：ZooKeeper 节点 CPU 突然飙高怎么排查？**
>
> **排查步骤：**
>
> 1. **确认是什么类型的高负载**
>    ```bash
>    top -H -p $(pgrep -f zookeeper)  # 查看 ZK 进程的线程 CPU 占用
>    ```
>
> 2. **检查请求是否异常增多**
>    ```bash
>    echo mntr | nc zoo1 2181 | grep outstanding  # 等待队列是否堆积
>    echo cons | nc zoo1 2181 | wc -l  # 当前连接数是否异常多
>    ```
>
> 3. **常见原因**
>    - **大量 Watcher 同时触发**：某个高频变化的节点有大量 Watcher（如服务注册路径），触发大量通知和重查询
>    - **大量写请求涌入**：业务写入突增，ZAB 广播消耗 CPU
>    - **GC 频繁**：堆内存不足触发频繁 GC，表现为 `java` 进程 CPU 高
>    - **Leader 选举风暴**：网络抖动导致频繁触发选举，CPU 被选举投票消耗
>
> 4. **GC 问题排查**
>    ```bash
>    jstat -gcutil $(pgrep -f zookeeper) 1000  # 每秒打印 GC 情况
>    ```
>    如果 FGC 频繁，需要增大 JVM 堆内存或优化业务减少 ZNode 数量。
