# 07、性能监控与优化

## Q1. 如何快速查看系统整体性能状况？

三个入门命令：`top`、`vmstat`、`iostat`

**`top` — 实时进程视图**：
```bash
top
# 常用交互键
# P  按CPU排序  M  按内存排序
# k  杀进程      r  调整nice值
# 1  显示每核CPU  H  显示线程

# 输出头部解读
top - 20:00:00 up 10 days, load average: 0.52, 0.48, 0.45
#                             ↑1分钟    ↑5分钟  ↑15分钟
# load average > CPU核数 → 系统过载

Tasks: 200 total, 1 running, 199 sleeping
%Cpu(s): 15.2 us, 3.1 sy, 0.0 ni, 79.5 id, 1.8 wa, 0.2 hi, 0.2 si
#         ↑用户  ↑系统          ↑空闲  ↑IO等待  ↑硬中断 ↑软中断
# wa 持续高 → I/O瓶颈  sy 高 → 内核态操作多（系统调用/中断）

KiB Mem: 16GB total, 2GB free, 8GB used, 6GB buff/cache
```

**`vmstat 1` — 系统综合快照**：
```bash
vmstat 1 5   # 每秒输出，共5次
# procs: r=运行队列长度（>CPU核数说明过载），b=等待IO的进程数
# memory: swpd=已用swap（非零说明内存不足），free/buff/cache
# io: bi=块设备读，bo=块设备写（KB/s）
# system: in=中断/s，cs=上下文切换/s
# cpu: us/sy/id/wa/st
```

## Q2. CPU 性能分析有哪些工具？

```bash
# 查看 CPU 核数和型号
nproc                          # 核数
lscpu                          # 详细CPU信息
cat /proc/cpuinfo | grep "model name" | head -1

# 查看 CPU 使用率（每个核）
top → 按1
mpstat -P ALL 1               # 每核详细统计（需安装 sysstat）

# 找到 CPU 消耗最高的线程
top → 按H（显示线程）
ps -eLf | sort -k3 -rn | head  # 按CPU排序的线程

# Java 应用 CPU 问题排查
# 1. top 找到高CPU的Java进程（PID=1234）
# 2. top -H -p 1234 找到高CPU的线程（TID=5678）
# 3. TID转十六进制：printf "%x\n" 5678 → 0x162a
# 4. jstack 1234 | grep -A20 "0x162a"  → 找到具体代码位置

# 性能热点分析（perf，内核级）
perf top -p 1234              # 实时热点函数
perf record -g -p 1234 sleep 10  # 采样10秒
perf report                   # 分析报告

# 系统调用分析
strace -cp 1234               # 统计各系统调用耗时
strace -p 1234                # 实时系统调用跟踪（慎用，有性能损耗）
```

## Q3. 内存性能如何分析和优化？

```bash
# 查看内存使用概览
free -h                        # 简洁
cat /proc/meminfo             # 详细

# 关键字段解读（free -h）
#              total  used  free  shared  buff/cache  available
# Mem:          16G   8G    2G    500M      6G           7.5G
# ↑ available 才是实际可用（free + 可回收的cache）

# 找内存消耗大的进程
ps aux --sort=-%mem | head
top → 按M

# 内存详细分析
smem -r -p -s rss | head      # 按RSS排序（更精准）
pmap -x 1234                  # 进程内存映射详情

# 查看内存碎片（如果碎片严重可能影响大内存分配）
cat /proc/buddyinfo

# 查看 Swap 使用情况
swapon -s
vmstat 1 | grep -E "si|so"   # si=swap in，so=swap out（持续非零说明内存紧张）

# 查看 OOM 历史
dmesg | grep -i "out of memory"

# 临时清理页缓存（不影响正在使用的）
sync && echo 3 > /proc/sys/vm/drop_caches  # 1=页缓存 2=slab 3=全部
# 注意：正常情况不需要手动清理，内核会自动管理
```

**Linux 内存分配策略**：
```bash
# overcommit 策略（/proc/sys/vm/overcommit_memory）
# 0 = 允许合理超额分配（默认）
# 1 = 总是允许超额分配（危险，可能OOM）
# 2 = 禁止超额分配（分配量不超过 RAM + swap）
cat /proc/sys/vm/overcommit_memory
```

## Q4. 磁盘 I/O 性能如何分析？

```bash
# 磁盘整体状态
df -h                          # 空间使用
df -i                          # inode 使用
lsblk                          # 块设备树状视图

# I/O 实时监控
iostat -x 1                    # 详细I/O统计（sysstat包）
iotop                          # 按进程的I/O排行（类似top）
iotop -o                       # 只显示正在I/O的进程

# iostat 关键字段
# tps        每秒I/O操作次数
# kB_read/s  每秒读取KB数
# kB_wrtn/s  每秒写入KB数
# %util      设备利用率（接近100%说明I/O瓶颈）
# await      平均I/O等待时间（ms，机械盘<20ms算正常）
# svctm      实际服务时间（ms）
# r_await/w_await 读/写平均等待时间

# 找I/O消耗大的进程
iotop -bo -d 2    # -b 批处理，-o 只显示活跃进程
# 或
pidstat -d 1      # 每秒输出进程的磁盘I/O

# 具体文件的I/O
lsof +D /tmp      # 查看哪些进程在访问目录

# 磁盘性能测试（生产慎用）
dd if=/dev/zero of=/tmp/test bs=1G count=1 oflag=direct   # 顺序写
hdparm -Tt /dev/sda                                         # 读速度
fio --name=test --rw=randread --bs=4k --size=1G --direct=1 # 随机读（推荐）
```

## Q5. 如何使用 `sar` 进行系统历史性能分析？

`sar`（System Activity Reporter）是 sysstat 套件中最强大的历史分析工具：

```bash
# 安装 sysstat（会自动每10分钟采集一次数据）
yum install sysstat
systemctl enable --now sysstat

# 查看 CPU 历史（今天）
sar -u
# 查看特定时间段
sar -u -s 08:00:00 -e 10:00:00

# 查看内存历史
sar -r                 # 内存使用
sar -B                 # 页面活动（缺页中断/页面换出）

# 查看磁盘I/O历史
sar -b                 # 块设备I/O统计
sar -d -p             # 各磁盘详细I/O

# 查看网络历史
sar -n DEV             # 网络接口流量
sar -n SOCK            # socket统计

# 查看历史某一天的数据
sar -u -f /var/log/sa/sa14    # 查看14号的CPU历史

# 负载历史
sar -q                 # 运行队列和负载平均值历史
```

## Q6. 网络性能如何监控？

```bash
# 实时网络流量（按接口）
iftop -i eth0                  # 实时带宽（类似top）
nethogs eth0                   # 按进程的实时带宽
watch -n1 cat /proc/net/dev   # 原始数据

# 网络流量统计
sar -n DEV 1 5                # 1秒采样5次，看每个接口的收发速率

# 连接数统计
ss -s                          # 各状态的连接数汇总
ss -tn | awk '{print $1}' | sort | uniq -c   # 按状态统计

# TIME_WAIT 连接数监控（大量时需要调优）
ss -tn state time-wait | wc -l

# 查看网络错误
ip -s link show eth0           # 看 errors/dropped/overrun
netstat -s | grep -E "error|fail|drop"
```

## Q7. 系统负载（Load Average）该如何解读？

```bash
# 查看负载
uptime
# 20:00:00 up 10 days, load average: 2.50, 1.80, 1.20
#                                    1分钟  5分钟  15分钟

# 负载 = 正在运行的进程数 + 等待运行（可运行态）的进程数 + 等待I/O的进程数
# 关键：和 CPU 核数对比，不是和1对比！
nproc  # 查看核数
```

**负载解读**：
- 负载 < CPU核数：系统空闲，没压力
- 负载 ≈ CPU核数：满载，充分利用
- 负载 > CPU核数：过载，有进程在等待 CPU
- 负载 >> CPU核数 × 2：严重过载，需要扩容或排查

**负载高但CPU使用率不高** → 通常是 I/O wait，大量进程在等磁盘/网络：
```bash
# 确认是否I/O问题
iostat -x 1          # 看 %util 和 await
vmstat 1 | grep -v procs  # 看 b 列（等待I/O的进程数）
```

## Q8. 生产环境 Java 应用常见性能问题如何排查？

**CPU 100% 排查**：
```bash
# 1. 找到高CPU的Java进程
top -c   # 找到PID，如 1234

# 2. 找到高CPU的线程
top -H -p 1234
# 或
ps -mp 1234 -o THREAD,tid,time | sort -rn | head

# 3. 线程ID转16进制（TID 5678 → 0x162e）
printf "%x\n" 5678

# 4. jstack分析
jstack 1234 | grep -A 30 "nid=0x162e"

# 常见原因：死循环/频繁GC（GC线程CPU高）/正则回溯/JSON序列化大对象
```

**内存泄漏排查**：
```bash
# 1. 确认JVM堆内存增长
jstat -gcutil 1234 1000   # 每秒输出GC统计，观察老年代(O列)是否持续增长

# 2. 导出堆快照
jmap -dump:format=b,file=/tmp/heap.hprof 1234
# 或启动时加参数：-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/

# 3. MAT/VisualVM/JProfiler分析 heap.hprof

# 4. 查看对象统计（快速定位大对象类）
jmap -histo 1234 | head -20
```

**频繁 Full GC 排查**：
```bash
# GC日志分析（启动时加）
-Xlog:gc*:file=/app/logs/gc.log:time,uptime:filecount=5,filesize=20m

# 常见原因
# - 大对象直接进老年代（-XX:PretenureSizeThreshold）
# - 年轻代太小，GC频繁晋升
# - 内存泄漏导致老年代满
# - 大量反射/动态代理导致元空间(MetaSpace)满
```

> **追问：如何在不影响性能的情况下持续监控生产 Java 应用？**
> 推荐 **Arthas**（阿里开源）：`java -jar arthas-boot.jar`，支持 `dashboard`（实时资源视图）、`thread -n 3`（CPU占用前3线程）、`trace`（方法调用链耗时）、`watch`（方法入参/返回值观测）、`ognl`（执行表达式）等功能，不需要重启应用，对生产影响极小。
