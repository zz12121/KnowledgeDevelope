# 05、系统架构与原理

## Q1. Linux 内核的主要组成部分有哪些？

Linux 内核是操作系统的核心，负责管理硬件资源并提供系统调用接口：

```
┌─────────────────────────────────────────┐
│              用户空间 (User Space)        │
│   应用程序 / Shell / 系统工具 / 库文件     │
├─────────────────────────────────────────┤
│           系统调用接口 (Syscall)          │
├─────────────────────────────────────────┤
│              内核空间 (Kernel Space)      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │ 进程调度  │ │ 内存管理  │ │ 文件系统  │ │
│  │(Scheduler)│ │  (MM)   │ │  (VFS)  │ │
│  └──────────┘ └──────────┘ └──────────┘ │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │ 网络子系统│ │ 设备驱动  │ │ IPC机制   │ │
│  │(TCP/IP)  │ │(Drivers) │ │(管道/信号)│ │
│  └──────────┘ └──────────┘ └──────────┘ │
├─────────────────────────────────────────┤
│              硬件抽象层 (HAL)             │
├─────────────────────────────────────────┤
│                 硬件                     │
└─────────────────────────────────────────┘
```

**五大核心子系统**：
- **进程调度（Scheduler）**：CFS（完全公平调度）决定 CPU 时间分配
- **内存管理（MM）**：虚拟内存、分页、缺页中断、OOM Killer
- **虚拟文件系统（VFS）**：统一文件系统接口（ext4/xfs/tmpfs/procfs 都通过 VFS）
- **网络子系统**：TCP/IP 协议栈、Socket、网络设备驱动
- **IPC（进程间通信）**：管道、信号、共享内存、消息队列、信号量

## Q2. 用户空间和内核空间的区别是什么？

**核心区别**：访问权限和内存隔离。

| 维度 | 用户空间 | 内核空间 |
|------|---------|---------|
| CPU 特权级 | Ring 3（受限） | Ring 0（最高权限） |
| 内存访问 | 只能访问自己的虚拟地址空间 | 可访问所有物理内存 |
| 直接访问硬件 | ❌ 不允许 | ✅ 允许 |
| 崩溃影响 | 只影响该进程 | 整个系统 Kernel Panic |
| 典型代表 | Java JVM、Nginx、bash | 内核代码、驱动程序 |

**切换过程（系统调用）**：
```
应用程序调用 read()
    ↓
触发软中断/syscall 指令（陷入内核）
    ↓
CPU 特权级 Ring3 → Ring0，切换页表
    ↓
内核执行具体操作（I/O 读取）
    ↓
结果复制到用户空间缓冲区
    ↓
返回用户空间，特权级恢复 Ring0 → Ring3
```

**性能影响**：频繁的用户/内核切换（上下文切换）开销大，这也是 Java NIO、零拷贝（sendfile）、mmap 等技术的优化出发点。

## Q3. Linux 系统启动流程是什么？

```
BIOS/UEFI 上电自检 (POST)
    ↓
读取 MBR/GPT，找到引导程序
    ↓
GRUB2 引导加载器
    ↓
加载 Linux 内核（vmlinuz）到内存
加载初始内存文件系统（initramfs）
    ↓
内核初始化（内存/CPU/设备驱动）
    ↓
挂载 initramfs，运行 /init（找到真正的根文件系统）
    ↓
挂载真正的根文件系统（/）
    ↓
启动 PID=1 的进程
 ├─ 老系统：SysV init → 执行 /etc/inittab → 运行 rc 脚本
 └─ 新系统：systemd → 按依赖关系并行启动服务
    ↓
启动各类服务（网络/SSH/应用等）
    ↓
显示登录提示（getty/GDM）
```

> **追问：Linux 系统卡在启动时如何排查？**
> ①GRUB 界面按 `e` 进入编辑模式，在内核行末加 `systemd.unit=rescue.target` 进入单用户模式；②用 `journalctl -b` 查看本次启动日志；③用 `systemctl list-units --failed` 查看失败的服务；④常见原因：磁盘满/fstab 配置错误/关键服务崩溃。

## Q4. Linux 的运行级别（runlevel）和 systemd target 是什么？

传统 SysV init 的运行级别（CentOS 6 及之前）：

| 级别 | 含义 |
|------|------|
| 0 | 关机 |
| 1 | 单用户模式（维护/恢复） |
| 2 | 多用户无网络 |
| 3 | 多用户命令行（服务器常用） |
| 5 | 多用户图形界面 |
| 6 | 重启 |

**systemd targets（CentOS 7/Ubuntu 现代系统）**：
```bash
# 查看当前目标
systemctl get-default
# 切换目标
systemctl isolate multi-user.target    # 相当于 runlevel 3
systemctl isolate graphical.target     # 相当于 runlevel 5
# 设置默认目标
systemctl set-default multi-user.target

# 常用 target 对应关系
# poweroff.target    = 0
# rescue.target      = 1
# multi-user.target  = 3
# graphical.target   = 5
# reboot.target      = 6
```

## Q5. systemd 的核心概念是什么？

systemd 是现代 Linux 的 PID 1 进程，负责系统初始化和服务管理。

**核心概念 Unit（单元）**：
```bash
# 常见 unit 类型
service   # 服务（最常用）
target    # 目标/里程碑（相当于runlevel）
socket    # socket 激活
timer     # 定时任务（cron 替代品）
mount     # 挂载点
path      # 路径监视

# 基本命令
systemctl start/stop/restart/reload nginx
systemctl enable/disable nginx    # 开机自启
systemctl status nginx            # 查看状态（含最近日志）
systemctl is-active nginx
systemctl list-units --type=service --state=running

# 查看服务日志
journalctl -u nginx -f            # 实时查看
journalctl -u nginx --since "1 hour ago"
```

**编写自定义 service 文件**：
```ini
# /etc/systemd/system/myapp.service
[Unit]
Description=My Java Application
After=network.target

[Service]
Type=simple
User=appuser
WorkingDirectory=/app
ExecStart=/usr/bin/java -jar /app/myapp.jar
ExecStop=/bin/kill -TERM $MAINPID
Restart=on-failure
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```
```bash
systemctl daemon-reload
systemctl enable --now myapp
```

## Q6. 文件描述符（File Descriptor）是什么？有什么限制？

文件描述符（fd）是进程访问文件/socket/管道等 I/O 资源的整数句柄。

**预留的 fd**：
- 0 → 标准输入（stdin）
- 1 → 标准输出（stdout）
- 2 → 标准错误（stderr）

**fd 限制**：
```bash
# 查看当前进程的fd限制
ulimit -n                     # 软限制（当前用户）
ulimit -Hn                    # 硬限制

# 查看系统级fd限制
cat /proc/sys/fs/file-max

# 修改单个进程的限制（重启前有效）
ulimit -n 65535

# 永久修改（/etc/security/limits.conf）
# appuser soft nofile 65535
# appuser hard nofile 65535

# systemd 服务需要在 [Service] 中额外设置
# LimitNOFILE=65535

# 查看某个进程当前打开的fd
ls -la /proc/1234/fd | wc -l
lsof -p 1234 | wc -l
```

**Java 应用常见问题**：
- 大量 TCP 连接 + 文件操作 → 容易触发 `Too many open files`
- Netty/Tomcat 高并发时每个连接占一个 fd，必须调大 `nofile` 限制

## Q7. Linux 文件系统的 inode 是什么？

**inode（索引节点）** 是文件系统中存储文件元数据的数据结构：

```
inode 存储的信息：
  - 文件类型（普通/目录/符号链接...）
  - 权限（rwxrwxrwx）
  - 所有者（uid/gid）
  - 文件大小
  - 时间戳（atime/mtime/ctime）
  - 硬链接数
  - 数据块指针（指向实际数据）

inode 不存储：
  - 文件名（文件名存在目录项里，指向inode编号）
```

```bash
# 查看inode信息
stat /etc/hosts
ls -i /etc/hosts         # 显示inode编号

# 查看inode使用情况（有时磁盘空间够但inode满了）
df -i

# inode满的症状：
# "No space left on device" 但 df -h 显示空间充足
# 解决：清理小文件（如 /tmp 下的大量临时文件）
```

**硬链接 vs 软链接 vs inode 的关系**：
```
硬链接 → 创建一个新的目录项，指向同一个 inode
        （inode 硬链接计数 +1，删除原文件不影响数据）

软链接（符号链接）→ 创建新的 inode，内容是目标路径字符串
        （原文件删除后，软链接失效）

ln /etc/hosts /tmp/hosts_hard    # 硬链接
ln -s /etc/hosts /tmp/hosts_soft # 软链接
```

## Q8. `/proc` 和 `/sys` 目录的作用是什么？

这两个是虚拟文件系统，内容在内存中生成，不占磁盘空间。

**`/proc`** — 进程和内核信息：
```bash
/proc/cpuinfo           # CPU信息
/proc/meminfo           # 内存使用详情
/proc/loadavg           # 负载平均值（1/5/15分钟）
/proc/net/tcp           # TCP连接表（ss/netstat底层数据源）
/proc/sys/              # 内核参数（可读写）

/proc/[PID]/            # 每个进程一个目录
/proc/1234/cmdline      # 进程启动命令
/proc/1234/fd/          # 已打开的文件描述符
/proc/1234/maps         # 内存映射
/proc/1234/status       # 进程状态信息
/proc/1234/net/tcp      # 该进程的网络连接
```

**`/sys`**（sysfs）— 设备和内核对象：
```bash
/sys/class/net/eth0/    # 网卡属性
/sys/block/sda/         # 磁盘属性
/sys/kernel/mm/         # 内存管理参数（如transparent_hugepage）
```

**通过 `/proc/sys` 动态调整内核参数**：
```bash
# 直接写入（临时，重启失效）
echo 65535 > /proc/sys/fs/file-max
echo 1 > /proc/sys/net/ipv4/ip_forward    # 开启IP转发

# 用 sysctl 命令（等价）
sysctl -w net.ipv4.ip_forward=1

# 永久修改（写入 /etc/sysctl.conf）
echo "net.ipv4.ip_forward = 1" >> /etc/sysctl.conf
sysctl -p   # 立即生效
```

## Q9. Linux 中的信号（Signal）有哪些常用信号？

信号是进程间异步通知的机制：

```bash
# 常用信号
SIGTERM (15)  # 优雅终止，允许进程清理后退出（kill pid 默认发送）
SIGKILL (9)   # 强制终止，内核直接杀死，进程无法捕获/忽略
SIGHUP  (1)   # 挂起，常用于让进程重新加载配置（nginx -s reload）
SIGINT  (2)   # 键盘中断（Ctrl+C）
SIGQUIT (3)   # 键盘退出（Ctrl+\），会产生 core dump
SIGUSR1 (10)  # 用户自定义信号1（如 nginx 日志切割用 USR1）
SIGUSR2 (12)  # 用户自定义信号2（如 nginx 热升级用 USR2）
SIGCHLD (17)  # 子进程状态变化通知父进程

# 发送信号
kill -15 1234      # 发送 SIGTERM
kill -9 1234       # 强制杀死
kill -HUP 1234     # 发送 SIGHUP
kill -USR1 $(cat /var/run/nginx.pid)   # nginx 日志切割

# 杀死同名所有进程
killall nginx
pkill -f "java.*myapp"   # 按进程名模式匹配
```

> **追问：Java 应用收到 SIGTERM 后如何优雅退出？**
> 注册 JVM Shutdown Hook：`Runtime.getRuntime().addShutdownHook(new Thread(() -> { /* 关闭线程池、等待请求处理完、释放连接 */ }))；`  
> Spring Boot 应用在 JDK 9+ 中可以直接配置 `server.shutdown=graceful`，配合 `spring.lifecycle.timeout-per-shutdown-phase=30s` 设置超时时间，收到 SIGTERM 后不再接收新请求，等待当前请求处理完再退出。

## Q10. Linux 中 OOM Killer 是什么机制？

当系统内存耗尽时，内核的 OOM Killer（Out Of Memory Killer）会选择一个进程强制杀死以释放内存：

**OOM Score（oom_score）**：
```bash
# 查看进程的 OOM 分数（越高越容易被杀）
cat /proc/1234/oom_score
# 查看 oom_score_adj（可调整，-1000=禁止被杀, 1000=最容易被杀）
cat /proc/1234/oom_score_adj

# 保护关键进程不被OOM杀
echo -1000 > /proc/1234/oom_score_adj
# 或用 systemd service
# OOMScoreAdjust=-1000
```

**查看 OOM 事件**：
```bash
dmesg | grep -i "oom\|kill"
journalctl -k | grep -i oom
```

**Java 应用 OOM 场景**：
- JVM 堆内存 `-Xmx` 设置太大，超过物理内存 → 触发 OOM Killer
- 最佳实践：`-Xmx` 设置为容器内存的 75%，留余量给 OS 和 JVM off-heap
- 容器环境中要设置正确的内存限制（Docker `-m`/K8s `resources.limits.memory`）

## Q11. 什么是 Linux 的上下文切换？

上下文切换是 CPU 从一个进程/线程切换到另一个的过程，需要保存和恢复寄存器、程序计数器、内存映射等状态。

**分类**：
- **进程上下文切换**：开销大，需要切换虚拟内存空间（TLB 刷新）
- **线程上下文切换**：同进程内线程切换，共享虚拟内存，开销小
- **中断上下文切换**：硬件中断触发，优先级最高

```bash
# 查看系统上下文切换率
vmstat 1
# cs列：每秒上下文切换次数
# in列：每秒中断次数

# 查看单个进程的上下文切换
pidstat -w -p 1234 1
# cswch/s：自愿上下文切换（进程主动等待I/O等资源）
# nvcswch/s：非自愿切换（时间片用完被强制切换）
```

**Java 线程过多的影响**：
- 每个线程 512KB~1MB 栈内存
- 大量线程竞争 CPU 时非自愿切换增多，性能下降
- 用 `pidstat -w` 发现非自愿切换高 → 线程数过多或锁竞争严重 → 考虑异步/响应式架构
