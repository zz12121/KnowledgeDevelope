###### 1. 什么是服务降级？
服务降级是**主动的、有计划的**系统容错策略：当系统整体资源紧张或某个非核心服务故障时，为保证核心服务可用，有策略地暂时简化或关闭非核心功能，返回预设的兜底数据。

**关键要点**：
- 降级是**主动的**，可以预先制定预案（如大促前提前关闭评论功能）。
- 降级是**有选择的**，只降非核心功能，保障核心业务。
- 实现方式：返回默认值、调用本地备用逻辑、直接屏蔽功能。

```java
// Sentinel 降级示例
@SentinelResource(value = "getRecommend", fallback = "defaultRecommend")
List<Item> getRecommend(Long userId) { ... }

List<Item> defaultRecommend(Long userId) {
    return Collections.emptyList(); // 兜底：返回空列表
}
```

###### 2. 什么是服务熔断？
服务熔断是**被动的、自动的**故障隔离机制，灵感来源于电路熔断器：当对某个目标服务的调用失败率超过阈值时，熔断器"跳闸"进入**打开状态**，后续请求不再真实调用，直接快速失败，防止故障扩散、资源耗尽（雪崩）。

**熔断器三态状态机**：
1. **CLOSED（关闭）**：正常状态，请求正常通过，统计失败率。
2. **OPEN（打开）**：失败率超阈值，所有请求快速失败，执行降级逻辑。等待冷却时间（`sleepWindowInMilliseconds`）后进入半开。
3. **HALF-OPEN（半开）**：放行少量试探请求，成功则关闭熔断器，失败则再次打开。

###### 3. 熔断和降级的区别是什么？
两者经常配合使用，但关注点不同：

| 维度 | 熔断 | 降级 |
|------|------|------|
| 目标 | 故障隔离，防止级联故障 | 弃车保帅，保障核心功能 |
| 触发条件 | 下游服务**调用失败率**自动触发 | **系统负载**或人工配置触发 |
| 作用层次 | 针对**单个服务依赖** | 可针对单个服务也可针对整个功能模块 |
| 主被动 | 被动自动（监测到故障才触发） | 主动有计划（可提前设置） |

**关系**：熔断通常是触发降级的原因之一，熔断器打开后执行预定的降级逻辑。

###### 4. Hystrix 的工作原理是什么？
Hystrix 用**命令模式**将外部调用封装在 `HystrixCommand` 中，执行流程：

1. **检查缓存**：若启用请求缓存，相同 key 直接返回缓存。
2. **检查熔断器**：若 OPEN 状态，直接执行 `getFallback()` 降级。
3. **检查线程池/信号量**：舱壁隔离策略，资源耗尽则拒绝执行并降级。
4. **执行业务**：在独立线程池（线程隔离）或调用线程（信号量隔离）执行实际请求。
5. **统计健康度**：无论成功失败，上报给熔断器，统计时间窗口内错误率。
6. **触发熔断**：错误率 > 阈值（默认 50%）且请求数 > 最低阈值（默认 20），CLOSED → OPEN。
7. **Fallback**：任何失败（异常/超时/拒绝）都执行 `getFallback()`。

> 注意：Hystrix 已进入维护模式（不再主动开发），Spring Cloud 2020 默认集成 Sentinel 替代。

###### 5. Sentinel 和 Hystrix 的区别是什么？
| 维度 | Hystrix | Sentinel |
|------|---------|---------|
| 设计核心 | 以**隔离**为中心，防止故障传播 | 以**流量**为切入点，多维度保护 |
| 隔离机制 | 线程池隔离（隔离性好但开销大） | 信号量隔离（轻量），支持慢调用熔断 |
| 流量控制 | 只有线程数/信号量总量限制 | QPS/并发线程数/冷启动/匀速排队 |
| 实时监控 | 需结合 Dashboard，规则静态配置 | 开箱即用控制台，**动态规则推送**（Nacos 数据源） |
| 维护状态 | 进入维护，不推荐新项目使用 | 阿里云维护，活跃开发 |
| 扩展性 | 扩展复杂 | 开放 SPI 接口，数据源/规则/适配层均可扩展 |

###### 6. 常见限流算法有哪些？
**四种主流限流算法**：

1. **固定窗口计数器**：将时间划分为固定窗口（如 1s），窗口内请求超过阈值则拒绝。简单但存在"边界突刺"：窗口切换瞬间可能放过 2 倍流量。
2. **滑动窗口计数器**：维护一个滑动的时间窗口（通常分成多个小格），统计窗口内总请求数，比固定窗口平滑，解决了边界突刺问题。Sentinel 使用此方式。
3. **令牌桶（Token Bucket）**：以固定速率向桶中添加令牌，每个请求消耗一个令牌，桶空则限流。**允许一定程度的突发流量**（桶中积累的令牌可以短时消耗完）。Guava `RateLimiter` 是经典实现。
4. **漏桶（Leaky Bucket）**：请求进入漏桶队列，以**固定速率**流出处理，超出容量则丢弃。严格限制输出速率，**不允许突发**。适合保护后端的稳定输出场景。

**选型口诀**：令牌桶允许突发（适合 API 限流），漏桶严格匀速（适合保护下游），滑动窗口是大多数场景的推荐选择。

###### 7. 分布式限流如何实现？
分布式限流需要跨实例协调计数器，核心挑战是**原子性和一致性**。

**常用方案**：

**Redis + Lua 脚本（最常用）**：利用 Redis 单线程执行 Lua 脚本的原子性，实现滑动窗口限流：
```lua
-- 滑动窗口限流（ZSET 实现）
local key = KEYS[1]
local now = tonumber(ARGV[1])      -- 当前时间戳(ms)
local window = tonumber(ARGV[2])   -- 窗口大小(ms)
local limit = tonumber(ARGV[3])    -- 限流阈值
local reqId = ARGV[4]              -- 请求唯一ID

redis.call('ZREMRANGEBYSCORE', key, 0, now - window)  -- 移除窗口外旧记录
local count = redis.call('ZCARD', key)                -- 统计窗口内请求数
if count < limit then
    redis.call('ZADD', key, now, reqId)               -- 添加当前请求
    redis.call('EXPIRE', key, math.ceil(window/1000))  -- 设置过期
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```

**网关层统一限流**：在 API 网关（Spring Cloud Gateway / Nginx+Lua）做集群级别统一限流，配置简单，对业务代码无侵入。

**Sentinel 集群限流**：Sentinel 提供 Token Server 模式，各实例向 Token Server 申请令牌，实现精确的集群级别限流。

###### 8. 单机限流和分布式限流的区别是什么？
| 维度 | 单机限流 | 分布式限流 |
|------|---------|---------|
| 控制范围 | 单个 JVM 进程 | 整个集群的全局总流量 |
| 实现复杂度 | 简单（AtomicLong / Guava RateLimiter） | 复杂（需 Redis 等外部存储 + 网络协调） |
| 精确性 | 精确控制本机流量 | 网络延迟导致可能有瞬时毛刺 |
| 性能开销 | 几乎零开销 | 每次请求额外一次 Redis 网络 RPC |
| 适用场景 | 单机保护阈值、服务实例级别 | API 总量控制、防恶意攻击 |

**混合策略（推荐）**：粗粒度限流放网关（集群级），细粒度业务限流（如用户级）放客户端，各层协同。

###### 9. 如何设计一个限流系统？
一个生产级限流系统的设计要点：

1. **限流对象**：按 API 路径、用户 ID、IP、租户维度分别设置不同阈值。
2. **算法选择**：API 限流用令牌桶（允许突发）；保护下游服务用漏桶（匀速）；QPS 统计用滑动窗口。
3. **原子操作**：分布式环境用 **Redis + Lua 脚本**，单机用 `AtomicLong` 或 `Semaphore`。
4. **动态配置**：阈值存配置中心（Nacos），支持实时调整无需重启。
5. **限流容错**：Redis 不可用时，**降级为本地限流**，而非让所有请求都通过（防止雪崩）。
6. **监控告警**：实时展示各资源的限流次数、QPS，触发限流时及时告警。
7. **分层防护**：接入层（Nginx）→ 网关层（Gateway）→ 服务层（Sentinel），层层过滤，形成纵深防御。

###### 10. 什么是服务雪崩？如何预防？
服务雪崩是指分布式系统中，**一个服务的故障导致依赖它的服务相继超时/失败，故障沿调用链扩散**，最终整个系统不可用的现象。

**根本原因**：同步调用 + 资源无限等待（没有超时控制）+ 没有隔离机制。

**预防策略（防雪崩三件套）**：
1. **超时控制**：所有远程调用设置合理超时时间（如 HTTP 请求 2s 超时），防止线程无限阻塞。
2. **熔断器**：失败率超阈值后自动熔断，快速失败，不让故障累积。
3. **舱壁隔离（Bulkhead）**：为每个依赖服务分配独立的线程池或信号量，一个服务满载不影响其他服务（Hystrix 线程池隔离的核心价值）。
4. **限流**：防止流量洪峰压垮下游，是防雪崩的第一道防线。
5. **降级**：提前准备兜底逻辑，故障时立即降级，保证系统整体可用。
