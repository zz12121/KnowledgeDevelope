###### 1. 分布式系统有哪些核心特征？
分布式系统的核心特征源于"将多台独立计算机组合成一个协同工作的整体"这个本质。

- **分布性与对等性**：系统组件物理上分散在不同网络节点，节点之间具有自治性，能够独立运作。比如 Eureka/Nacos 集群，每个节点就是平等自治的。
- **并发性与透明性**：多个节点可以同时执行任务，对外暴露统一视图（位置透明、访问透明）。比如通过 RMI 调用远程方法，代码层面跟调用本地方法没有区别。
- **可扩展性与高可靠性**：通过加节点水平扩展处理能力；通过冗余部署使单点故障不影响整体，比如 ZooKeeper 只要超过半数节点存活集群就能继续服务。

###### 2. 什么是 CAP 理论？在实际中如何选择？
CAP 理论是分布式系统设计的基础约束，指出以下三者**不可同时满足**：
- **C（一致性）**：任何时刻所有节点看到的数据完全相同（强一致性）。
- **A（可用性）**：每个请求都能收到响应（不保证是最新数据）。
- **P（分区容错性）**：网络分区发生时系统仍能继续工作。

> 在真实的分布式系统中，网络分区是客观存在的，**P 是必须保留的**，因此实际只在 C 和 A 之间做取舍：

| 类型 | 代表产品 | 场景 |
|------|---------|------|
| **CP**（舍 A 保 C） | ZooKeeper、HBase、etcd | 强一致性要求，如分布式锁、元数据管理 |
| **AP**（舍 C 保 A） | Eureka、Cassandra、CouchDB | 高可用要求，如服务注册发现、购物车 |

**注意**：CAP 中的一致性是"强一致性"，BASE 理论为 AP 系统提供了"最终一致性"的工程实践路径。

###### 3. 什么是 BASE 理论？和 ACID 有什么区别？
BASE 是 eBay 架构师 Dan Pritchett 提出的分布式系统设计原则，是 CAP 中 AP 方案的工程延伸：
- **Basically Available（基本可用）**：出现故障时，允许损失部分可用性（如响应时间变长、降级页面），但核心流程必须保障。
- **Soft State（软状态）**：允许数据在同步过程中存在中间状态，比如 MySQL 主从异步复制时从库短暂落后。
- **Eventually Consistent（最终一致性）**：经过一段时间后，所有副本最终会达成一致。

**ACID vs BASE**：
| | ACID | BASE |
|--|------|------|
| 目标 | 强一致性 | 最终一致性 |
| 适用场景 | 银行转账等强一致场景 | 电商、社交等高可用场景 |
| 性能 | 低吞吐、高延迟 | 高吞吐、低延迟 |

###### 4. 分布式系统和集中式系统的核心区别是什么？
两者的根本区别在于架构哲学——集中式是"一个大脑"，分布式是"多个大脑协同"。

| 特征 | 集中式系统 | 分布式系统 |
|------|-----------|-----------|
| **架构本质** | 所有计算、存储由单一中心节点完成 | 多节点通过网络协作 |
| **数据一致性** | 天然强一致性 | 需引入复杂协议（Raft/Paxos）保证 |
| **可靠性** | 存在单点故障风险 | 冗余部署，高可靠性 |
| **扩展性** | 垂直扩展，成本高且有上限 | 水平扩展，灵活经济 |
| **复杂度** | 架构简单，易于管理 | 需要处理网络延迟、分区、数据一致性等问题 |

###### 5. 什么是分布式 ID？常见的生成方案有哪些？
分布式 ID 是在分布式系统中**全局唯一、趋势递增、高性能**的标识符，用于替代单库自增主键。

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **UUID** | 本地生成128位随机数 | 无依赖，简单 | 无序，索引性能差，长度大 |
| **数据库号段** | 批量申请号段到内存，用完再申请 | 有序，简单可靠 | 依赖 DB，号段用完有短暂延迟 |
| **Redis INCR** | 利用 Redis 原子递增 | 有序，性能高 | 依赖 Redis，需处理持久化 |
| **雪花算法（Snowflake）** | 64位 = 时间戳(41) + 机器ID(10) + 序列号(12) | 高性能，趋势递增，含时间信息 | 依赖时钟，时钟回拨会产生重复ID |
| **美团 Leaf** | 号段 + 雪花双模式 | 生产级，高可用 | 引入额外依赖 |

**雪花算法结构**（以 Snowflake 为例）：
```
符号位(1) | 时间戳ms(41) | 数据中心ID(5) | 机器ID(5) | 序列号(12)
```
理论上每毫秒可生成 4096 个 ID，单机 TPS 约 400 万/秒。

**时钟回拨问题解决方案**：
- 发现时钟回拨时，等待时间补偿（回拨时间短的情况）
- 使用 `NTP` 保持时钟同步，设置最大容忍回拨阈值
- 美团 Leaf 方案：通过 Zookeeper 记录最后使用的时间戳，启动时检查

###### 6. 分布式架构的演进路径是什么？
分布式架构随着业务规模扩大而逐步演进：

1. **单体架构 → 垂直分层**：按功能层（Web/Service/DB）分别部署，物理分离但逻辑仍集中。
2. **引入分布式组件 → SOA**：读写分离、独立缓存服务、服务总线（ESB），Java 生态常用 Dubbo RPC 框架。
3. **微服务架构**：以业务能力为边界，拆分为小而自治的独立服务，每服务独立部署扩展。Spring Cloud + Kubernetes 是主流技术栈。
4. **云原生 + 服务网格**：将流量控制、安全、遥测等通用能力从业务代码中剥离，下沉到基础设施层（如 Istio），开发者专注业务逻辑。

###### 7. 分布式系统中什么是节点？节点间如何通信？
**节点**是分布式系统中最基本的操作实体，通常是一台通过网络互联的独立服务器，或运行在 JVM 中的一个独立进程（如一个微服务实例）。

**节点间通信方式**：
- **HTTP/REST**：简单通用，适合跨语言，延迟稍高。
- **gRPC**：基于 HTTP/2 + Protobuf，性能高，适合内部服务调用。
- **RPC 框架**：如 Dubbo，封装了序列化、路由、负载均衡等能力。
- **消息队列**：Kafka/RocketMQ，异步解耦，适合事件驱动架构。

###### 8. 什么是分布式计算？MapReduce 是怎么工作的？
分布式计算的核心思想是**"分而治之"**——把大任务分解成多个子任务并行处理，再汇总结果。

**MapReduce 工作流程**（以 Hadoop 为例）：
1. **Map 阶段**：将输入数据拆分，分配到各节点并行执行 Map 函数，产出 `<Key, Value>` 的中间结果。
2. **Shuffle 阶段**：将相同 Key 的结果汇聚到同一个 Reducer 节点。
3. **Reduce 阶段**：对每个 Key 的所有 Value 进行聚合计算，输出最终结果。

**词频统计示意**：
```java
// Mapper：对每个单词输出 (word, 1)
mapper: "Hello World" → [("Hello", 1), ("World", 1)]

// Shuffle：按 Key 聚合
shuffle: ("Hello", [1, 1, 1]), ("World", [1, 1])

// Reducer：汇总计数
reducer: ("Hello", 3), ("World", 2)
```
框架处理了分布式执行、容错、任务调度等复杂细节，开发者只需实现 `Mapper` 和 `Reducer` 两个类。
