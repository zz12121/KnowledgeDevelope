###### 1. 常见的链路追踪系统有哪些？

主流的开源链路追踪系统都源于 Google Dapper 论文的设计理念，选型时主要关注**侵入性、功能深度和生态成熟度**三个维度。

- **Zipkin**：Twitter 开源，基于 Dapper 的经典实现。设计简洁，通过 `Brave` 客户端库埋点，支持 Elasticsearch/Cassandra 多种存储后端。适合作为理解链路追踪原理的入门，或轻量级集成的起点。

- **SkyWalking**：国人开源，Apache 顶级项目。核心亮点是**基于字节码增强的无侵入探针**，对业务代码零修改即可实现链路、指标、拓扑图的全量采集。UI 强大，支持告警和性能剖析，是目前国内微服务监控的主流选择。

- **Jaeger**：Uber 开源，CNCF 项目。兼容 OpenTracing API，强调云原生和可扩展性，原生支持 gRPC 通信。如果团队已深度使用 OpenTelemetry 标准，或技术栈以 Go 为主，Jaeger 是自然之选。

- **Pinpoint**：Naver 开源，与 SkyWalking 同为无侵入方案，但将调用链与代码级 Profiling 深度结合，提供方法栈粒度的性能洞察，代价是架构较重、资源消耗相对较高。

- **Cat**：美团开源，更偏向实时应用监控平台，不仅包含链路追踪，还集成了丰富的指标统计和告警功能，但客户端需要代码埋点。

**选型参考**：追求无侵入、功能全面 → **SkyWalking**；已用 OpenTelemetry/OpenTracing → **Jaeger**；轻量快速集成或学习原理 → **Zipkin**。

###### 2. Zipkin 的工作原理是什么？

Zipkin 遵循经典的**收集 → 传输 → 存储 → 查询**模型，核心是 **Span** 的生成、传播和上报。

**1. 数据生成与传播（客户端 Brave 库）**

- 请求进入服务时，`TracingFilter` 检查请求头（如 `X-B3-TraceId`）：不存在则生成全局唯一的 **Trace ID**，已存在则沿用（说明是某个追踪链路的一环）。
- 为本次服务调用创建一个 **Span**，并通过 `Inject` 操作将 TraceId、SpanId、ParentSpanId 等注入到 HTTP Header 中，随请求传递至下游。
- 请求处理完成后，Span 被标记结束时间，并**异步**上报给 Zipkin Collector，不阻塞主流程。

**2. 服务端组件**

| 组件 | 职责 |
|---|---|
| Collector | 接收 HTTP/Kafka 上报的 Span，验证并存储 |
| Storage | 可插拔后端：内存（测试）、Elasticsearch（生产）、Cassandra（高写入） |
| Query Service | 提供 API 供 UI 查询 Trace 数据 |
| UI | 可视化调用链和时序图 |

**源码角度（Brave 核心）**：
- `Tracer.newTrace()` / `nextSpan()` 创建或继承追踪上下文；
- `B3Propagation` 定义了如何将上下文编解码到 HTTP 头；
- `AsyncReporter` 使用后台线程批量将 Span 发送给 Zipkin，降低对主流程的性能影响。

###### 3. SkyWalking 和 Zipkin 的区别是什么？

区别主要体现在设计理念、埋点方式和功能生态上。

| 维度 | Zipkin | SkyWalking |
|---|---|---|
| **核心定位** | 专注追踪，轻量组件 | APM 平台，链路+指标+拓扑+告警一体化 |
| **埋点方式** | **库模式/手动埋点**，引入 Brave SDK，对代码有侵入 | **探针模式/无侵入**，Java Agent 字节码增强，业务代码零修改 |
| **数据模型** | 核心是 Span，一个 Trace 由多个 Span 组成 | 引入 **Segment** 概念：一个 Segment 是单服务内一次请求的完整轨迹，包含多个代表方法调用的 Span，更适合做代码级剖析 |
| **上下文传播** | B3 格式 HTTP Header（`X-B3-TraceId` 等） | 自研 Context Carrier，支持 HTTP/gRPC/MQ 多种载体 |
| **性能开销** | 较低，埋点轻量 | 略高，字节码增强有额外开销，但经过优化生产环境通常 <3% |
| **UI 能力** | 基础，专注链路展示 | 强大：拓扑图、实例/端点指标仪表盘、性能剖析、日志关联、告警 |

**结论**：Zipkin 是轻量的追踪组件，适合嵌入现有体系。SkyWalking 是一体化 APM 平台，无侵入特性大幅降低接入和维护成本，国内微服务场景中占主导地位。

###### 4. 什么是 TraceId 和 SpanId？

这是链路追踪数据模型的两个核心标识符，用于将散落在各个服务的日志和调用记录串联起来。

- **TraceId**：**全局链路 ID**。在一次完整的分布式业务请求中（如用户下单，涉及网关→订单服务→库存服务→支付服务），所有相关调用共享**同一个 TraceId**。它像一根绳子，把各个服务的数据串成一条完整的调用链。TraceId 在请求入口处（通常是网关）生成。

- **SpanId**：**工作单元 ID**。代表一次具体的操作（服务内某个方法调用，或调用下游服务的过程）。每个 Span 有唯一的 SpanId 和 **ParentSpanId**（指向上游调用者），通过这个父子关系可以还原出**调用树**。

**示例**：`用户 → 网关(A) → 订单服务(B) → 库存服务(C)`

```
TraceId: T1 贯穿全程
网关A:    SpanId=S1,  Parent=null  (RootSpan)
订单B:    SpanId=S2,  Parent=S1
库存C:    SpanId=S3,  Parent=S2
```

通过 `T1` 可以查到整条链路 `[S1 → S2 → S3]`，并通过 Parent 关系还原调用树和耗时分布。SpanId 通常用雪花算法或 UUID 生成，保证全局唯一。

###### 5. 如何在微服务中集成链路追踪？

集成链路追踪是系统性工程，分为**探针接入**、**上下文传播**和**后端部署**三部分。

**以 SkyWalking 为例（无侵入推荐方案）**

**1. 部署后端服务**
- 部署 OAP Server（负责接收、聚合、分析数据）和 UI；
- 配置存储，生产环境推荐 Elasticsearch。

**2. 应用接入（零代码修改）**

```bash
# 在应用启动脚本 JAVA_OPTS 中加入：
-javaagent:/path/to/skywalking-agent.jar
-DSW_AGENT_NAME=order-service
-DSW_AGENT_COLLECTOR_BACKEND_SERVICES=oap-host:11800
```

Agent 会自动增强 Spring MVC、Feign、JDBC、Redis 等主流组件，无需改动业务代码。K8s 环境可通过 Init Container 挂载 Agent JAR。

**3. 上下文跨边界传播**
- HTTP 调用：Agent 自动拦截并注入/提取 `sw8` 头；
- 消息队列：SkyWalking Agent 已支持 Kafka/RocketMQ 客户端的自动增强；
- **跨线程传播**：使用 `@TraceCrossThread` 注解或 `RunnableWrapper`/`CallableWrapper` 包装异步任务，确保 TraceId 不丢失。

**以 Zipkin + Spring Cloud Sleuth（库模式）为例**

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-sleuth-zipkin</artifactId>
</dependency>
```

```yaml
# application.yml
spring:
  zipkin:
    base-url: http://zipkin-server:9411
  sleuth:
    sampler:
      probability: 0.1   # 采样率 10%，生产环境不要全量采集
```

Sleuth 会自动为 RestTemplate、Feign、Spring MVC 等注入追踪逻辑，并将 TraceId 注入 MDC，让日志自动携带 TraceId。

> Spring Boot 3.x 后 Sleuth 已迁移为 **Micrometer Tracing**，API 类似但包名变化，详见 [[27_SpringCloudKnowledge]]。

**核心要点**：无论哪种方案，关键在于保证 **TraceId 和 Span 上下文能够跨服务、跨线程、跨异步边界正确传递**。

###### 6. 如何设计链路追踪的采样率？

**为什么需要采样？**

全量采集意味着每次请求都会生成、上报、存储 Span 数据，在高 QPS 系统中（如日均亿级请求），存储和带宽成本极高，而且大多数正常请求的数据价值有限。采样的目标是**以最小的开销，覆盖最有价值的请求**。

**常见采样策略**

| 策略 | 原理 | 适用场景 |
|---|---|---|
| **固定比例（头部采样）** | 随机抽取 N% 的请求，在入口处决定采样与否，并传递给下游 | 低成本，但可能漏掉偶发故障 |
| **自适应采样** | 根据系统负载、QPS 动态调整采样率，高峰期降低，低谷期升高 | 兼顾全面性和开销 |
| **尾部采样（Tail Sampling）** | 先收集所有请求的 Span，完成后根据结果（如错误、高延迟）决定是否保留 | 最准确，能覆盖所有异常，但内存消耗高 |
| **强制采样** | 特定请求（如错误请求、超时请求）强制采样，无视采样率 | 与固定比例结合使用 |

**最佳实践**
- **生产环境推荐固定比例 1%~10%**，错误请求强制采样（100%）；
- 重要业务接口（支付、下单）可单独设置更高采样率；
- Skywalking 支持在 OAP 端配置动态采样策略，无需重启应用；
- **头部采样必须在入口传递采样决策**（通过 Header 的 Sampled 字段），确保一条链路要么全部采集，要么全部不采集，否则链路数据会不完整。

###### 7. TraceId 如何跨线程和跨进程传播？

**跨进程传播（服务间调用）**

这是链路追踪的基础场景，通过在请求中注入特定 Header 实现：

- HTTP 调用：Zipkin 使用 B3 格式（`X-B3-TraceId`、`X-B3-SpanId`、`X-B3-Sampled`），SkyWalking 使用 `sw8` 头；
- gRPC 调用：通过 Metadata 传递；
- 消息队列：通过消息 Header 传递（MQ 的 Property），消费者提取后创建子 Span。

**跨线程传播（同进程内异步场景）**

这是更容易被忽视的问题。Java 默认的 `ThreadLocal` 在切换线程时会丢失上下文。

```java
// 问题场景：异步线程中 TraceId 丢失
@Async
public void asyncMethod() {
    // 这里 MDC 中的 traceId 是 null！
    log.info("处理异步任务，traceId={}", MDC.get("traceId"));
}
```

**解决方案**

```java
// 方案1：SkyWalking 提供 RunnableWrapper，自动传递 Span 上下文
ExecutorService executor = Executors.newFixedThreadPool(4);
executor.submit(RunnableWrapper.of(() -> {
    // 此处 Span 上下文已正确传递
    doSomething();
}));

// 方案2：使用 Transmittable ThreadLocal（TTL）库，支持 Executor 线程池场景
// 用 TtlExecutors 包装线程池，TraceId 自动随任务传递
ExecutorService ttlExecutor = TtlExecutors.getTtlExecutorService(executor);

// 方案3：手动在提交任务前捕获上下文，在任务内恢复
String traceId = MDC.get("traceId");
executor.submit(() -> {
    MDC.put("traceId", traceId);
    try {
        doSomething();
    } finally {
        MDC.remove("traceId");
    }
});
```

**最佳实践**：优先使用框架或中间件的标准包装方案（SkyWalking RunnableWrapper / TTL），避免手动 MDC 传递导致泄漏。

###### 8. 什么是 OpenTelemetry？它与 Zipkin/SkyWalking 的关系？

**背景：为什么需要 OpenTelemetry？**

在 OpenTelemetry 出现之前，链路追踪领域存在两套竞争标准：**OpenTracing**（CNCF，专注追踪）和 **OpenCensus**（Google，覆盖追踪+指标）。两套 API 不兼容，厂商各行其是，用户切换工具成本极高。**OpenTelemetry（OTel）** 是两者合并的产物，是 CNCF 支持的统一可观测性标准。

**OpenTelemetry 是什么**

OTel 提供**厂商中立的标准 API、SDK 和数据协议（OTLP）**，覆盖 Traces（链路）、Metrics（指标）、Logs（日志）三大支柱（即所谓的可观测性三剑客）。它只定义标准，不提供存储和 UI。

```
应用 ─→ OTel SDK（采集、处理）─→ OTel Collector（接收、转换、导出）─→ 后端（Jaeger/Zipkin/SkyWalking/Prometheus...）
```

**与 Zipkin/SkyWalking 的关系**

| | OpenTelemetry | Zipkin | SkyWalking |
|---|---|---|---|
| 定位 | 数据采集标准 + SDK | 追踪后端（存储+UI）| APM 平台（后端+UI） |
| 是否提供存储/UI | 否 | 是 | 是 |
| 关系 | OTel 可以将数据导出到 Zipkin/Jaeger/SkyWalking | 可作为 OTel 的存储后端 | 支持接收 OTel 协议数据 |

**实际应用建议**：新项目推荐使用 OTel SDK 做埋点，通过 OTel Collector 将数据路由到 Jaeger 或 SkyWalking，实现"采集与存储解耦"。这样未来更换后端存储时，业务代码无需改动。

```yaml
# Spring Boot 3.x + Micrometer Tracing（基于 OTel）
management:
  tracing:
    sampling:
      probability: 0.1
  otlp:
    tracing:
      endpoint: http://otel-collector:4318/v1/traces
```
