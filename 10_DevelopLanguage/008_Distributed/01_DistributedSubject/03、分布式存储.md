###### 1. 什么是分布式文件系统？
分布式文件系统是指文件系统管理的物理存储资源不直接连接在本地节点，而是通过网络与节点相连，对外提供**统一逻辑视图**，让用户能像访问本地文件一样访问网络上分散的文件。

**三大核心特性**：
- **透明性**：访问透明（远程文件与本地操作方式一致）、位置透明（物理位置对用户隐藏）。
- **并发控制**：多客户端并发访问同一文件时，通过锁机制（如 HDFS 的租约机制 Lease）保证一致性。
- **容错与冗余**：通过多副本（如 HDFS 默认 3 副本）或纠删码确保节点故障时数据不丢失。

**典型系统**：HDFS（大数据场景）、Ceph（通用分布式存储）、GlusterFS（扩展性强）。

###### 2. HDFS 的架构和工作原理是什么？
HDFS 采用**主从架构**，核心三个角色：

- **NameNode（主节点）**：管理文件系统命名空间，存储元数据（文件名、权限、块映射），常驻内存。通过 FSImage（快照）+ EditLog（操作日志）持久化，Secondary NameNode 定期合并两者。
- **DataNode（从节点）**：存储实际数据块（默认 128MB/块），定期向 NameNode 发送心跳和块报告。
- **Secondary NameNode**：定期合并 FSImage 和 EditLog，减轻 NameNode 负担，**不是 NameNode 的热备**。

**写入流程（Pipeline 机制）**：
1. 客户端向 NameNode 申请写入，获得一组 DataNode 列表（管道）。
2. 客户端将数据分割成 packet，流式写入管道第一个 DataNode。
3. 第一个 DataNode 接收并存储，同时转发给下一个，依次传递形成流水线。
4. 管道中最后一个 DataNode 完成后，反向发送确认信号。

**读取流程**：客户端从 NameNode 获取块位置，直连最近的 DataNode 并行读取，最后在本地合并。

###### 3. 什么是对象存储？
对象存储是以**对象**为基本存储单元的分布式存储架构。每个对象包含三部分：数据本身 + 元数据（描述数据的属性）+ 全局唯一标识符（Key）。通过 **RESTful API（HTTP PUT/GET/DELETE）** 访问。

**核心特征**：
- **扁平命名空间**：使用 Bucket（桶）组织对象，无复杂目录层级，易于水平扩展。
- **无限扩展**：单个对象存储集群可扩展至 EB 级别。
- **适合冷数据**：对象存储访问延迟比块存储高，但成本极低，适合图片、视频、日志、备份等非结构化数据。

**典型产品**：AWS S3、阿里云 OSS、腾讯云 COS、MinIO（开源私有化部署）。

###### 4. 什么是块存储和文件存储的区别？
| 特征 | 块存储 | 文件存储 | 对象存储 |
|------|--------|----------|---------|
| 访问接口 | iSCSI/FC 底层块协议 | NFS/SMB 文件级协议 | HTTP RESTful API |
| 数据组织 | 原始磁盘块，OS 格式化 | 文件/目录树结构 | 扁平 Bucket/Key |
| 适用场景 | 数据库、高 IOPS 应用 | 文件共享、企业文档 | 图片、视频、备份 |
| 性能 | 高（低延迟） | 中等 | 较低（访问延迟高） |
| 扩展性 | 有限 | 有限 | 近乎无限 |

简单记：**块存储像硬盘**（格式化后用），**文件存储像 NAS**（挂载后按目录访问），**对象存储像网盘**（通过 URL 访问）。

###### 5. 分布式存储系统如何保证数据的可靠性？
主要通过两种技术：

**1. 多副本机制（Replication）**：
- 同一份数据复制成多个副本（通常 3 个），存储在不同节点、不同机架甚至不同数据中心。
- HDFS 管道写入：先写第一个 DataNode，再由它转发，保证写入原子性。
- 当副本数低于阈值，NameNode 自动触发重复制，从健康副本复制到新节点。
- **代价**：3 副本意味着 3 倍存储成本。

**2. 纠删码（Erasure Coding）**：
- 将原始数据分成 k 个数据块，通过编码计算生成 m 个校验块，共 n=k+m 个块。
- 任意 k 个块即可恢复完整数据，最多可容忍 m 个块失效。
- 例如 RS-6-3 配置：6 个数据块 + 3 个校验块，存储开销约 1.5 倍（vs 3 副本的 3 倍），容忍任意 3 块失效。
- **代价**：编码/解码需要计算资源，读取时延比多副本高，适合**访问频率低的冷数据**。

###### 6. 什么是副本机制？
副本机制是通过**主动复制数据**来提高可用性和可靠性的技术，是分布式存储最基础的手段。

**三个目标**：高可用（某副本不可用时其他副本继续服务）、负载均衡（读请求分发到不同副本）、容错（硬件故障不丢数据）。

**一致性挑战**：多副本的核心难题是保证写一致性。常见策略：
- **WARO（Write All Read One）**：写时必须所有副本成功才返回，一致性最强但可用性差。
- **Quorum 机制**：写 W 个、读 R 个，W+R>N 时保证强一致。可以灵活调整 W、R 平衡性能和一致性。

###### 7. 什么是纠删码（Erasure Code）？
纠删码是基于编码理论的数据保护技术，以**更低存储开销实现更高可靠性**。核心是里德-所罗门码（Reed-Solomon）：

**工作原理**：
1. **编码**：将 k 个数据块通过范德蒙矩阵运算生成 m 个校验块，共 n=k+m 个块。
2. **存储**：n 个块分散存储在不同节点上。
3. **恢复**：任意 m 个块丢失时，只需从剩余块中任取 k 个，通过解码算法完全恢复原始数据。

**HDFS 中的应用**：HDFS 3.x 支持纠删码策略，冷数据从 3 副本（300% 开销）迁移到 RS-6-3 策略（150% 开销），大幅降低存储成本，且可容忍任意 3 个块失效。

**适用场景**：对象存储（AWS S3 大量使用）、HDFS 冷数据、磁盘 RAID 阵列（本质上也是纠删码的应用）。

###### 8. 分布式存储中的数据分片策略有哪些？
数据分片是将大型数据集分散到多个存储节点的技术，是水平扩展和并行处理的基础。

**三种主要策略**：

**1. 固定大小分块（Range Partitioning）**：
- 按固定大小切割（如 HDFS 的 128MB/块）或按 key 范围分片（如 HBase 的 Region）。
- 优点：实现简单，顺序扫描效率高；缺点：热点数据可能集中在某个分片（如单调递增 ID 导致所有写入到最后一个 Region）。

**2. 哈希分片（Hash Partitioning）**：
- 对 key 取哈希值，路由到对应节点（如 Redis Cluster 的哈希槽：CRC16(key) % 16384）。
- 优点：数据分布均匀；缺点：范围查询需要扫描所有节点。
- 一致性哈希：增减节点时只影响相邻区间，大幅减少数据迁移量。

**3. 内容定义分块（Content-Defined Chunking）**：
- 用滑动窗口 + Rabin 指纹动态确定分块边界，相同内容产生相同边界。
- 优点：局部修改不影响其他块，重删效果极好；缺点：计算开销大，块大小不固定。

**9. 什么是一致性哈希？和普通哈希分片有什么区别？**
详见 [[02、分布式一致性与共识#13. 什么是一致性哈希算法？]] — 一致性哈希通过虚拟哈希环，将节点增减时的数据迁移量从 O(N) 降低到 O(1/N)，是分布式缓存和存储的标配。

**与普通哈希（key % N）的对比**：普通哈希节点变化时需迁移几乎所有 key；一致性哈希节点变化时只影响相邻区间的 key，代价极低。Redis Cluster 使用的哈希槽（16384 个槽固定分配）是一致性哈希的简化变体，牺牲了动态扩展的灵活性，换来了实现的简单性。
