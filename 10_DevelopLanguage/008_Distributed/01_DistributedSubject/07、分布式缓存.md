###### 1. 什么是分布式缓存？
分布式缓存是将缓存数据**分散存储在多台独立服务器**上，构成逻辑统一的缓存池，解决本地缓存的数据孤岛、容量受限、一致性和扩展性问题。核心技术挑战：数据分片（一致性哈希/哈希槽）、高可用（主从复制 + 故障转移）、数据一致性（最终一致性为主）。

代表产品：Redis Cluster（去中心化分片，Gossip 协议 + CRC16 哈希槽路由）、Memcached 集群（客户端一致性哈希分片）。

详见 → [[../../../20_RedisKnowledge/23_RedisKnowledge/01、Redis基础概念|Redis 集群架构与分片机制]]

###### 2. Redis 和 Memcached 的区别是什么？
- **数据结构**：Redis 支持 String/Hash/List/Set/ZSet/Bitmap/HyperLogLog 等丰富结构；Memcached 仅支持 String 键值。
- **持久化**：Redis 提供 RDB（快照）+ AOF（操作日志）两种持久化；Memcached 无持久化。
- **集群**：Redis 原生支持哨兵（高可用）和 Cluster（分片扩展）；Memcached 需客户端实现一致性哈希。
- **线程模型**：Redis 6.0 前单线程，6.0 后多线程处理网络 IO，核心命令执行仍单线程；Memcached 多线程。
- **选型**：需要复杂数据结构/持久化/高级特性 → Redis；纯简单 KV 缓存且需多线程极致性能 → Memcached（现在基本选 Redis）。

###### 3. 什么是缓存穿透？如何解决？
**缓存穿透**：查询**根本不存在**的数据，Cache Miss 后直接打到数据库，可能被恶意利用发起大量无效查询。

**解决方案**：
1. **布隆过滤器**：请求到来前先查布隆过滤器，判断 Key 是否存在。不存在直接拒绝，不穿透到 Cache 和 DB。Redisson 提供生产级 `RBloomFilter`。
2. **缓存空值**：查 DB 结果为空时，向 Cache 写入空值（如 `"NULL"`），设置短 TTL（30s），下次同 Key 请求直接命中 Cache。
3. **接口鉴权 + 限流**：拦截非法请求，防止恶意参数洗刷。

###### 4. 什么是缓存击穿？如何解决？
**缓存击穿**：某个**热点 Key** 过期瞬间，大量并发请求同时 Cache Miss，直接涌向数据库。与雪崩的区别：击穿针对**单个热点 Key**，雪崩是**大批 Key** 同时失效。

**解决方案**：
1. **互斥锁**：Cache Miss 时用 `SETNX` 加锁，只有一个线程回源 DB，其他线程等待重试。适合对一致性要求高的场景。
2. **逻辑过期**：Value 中封装过期时间，业务线程发现逻辑过期时异步触发重建，期间继续返回旧数据（最终一致，高可用）。
3. **永不过期 + 定时更新**：对核心热点 Key 不设 TTL，后台任务定期刷新（适合不频繁变化的配置类数据）。

详见 → [[../../../20_RedisKnowledge/23_RedisKnowledge/06、缓存应用|缓存击穿与热点 Key 解决方案]]

###### 5. 什么是缓存雪崩？如何解决？
**缓存雪崩**：大量 Key 在**同一时间大面积失效**，或 Redis 集群宕机，导致所有请求涌向数据库。

**解决方案**：
1. **差异化 TTL**：过期时间加随机值（`TTL = baseTTL + random(60s)`），避免集中失效。
2. **集群高可用**：Redis Cluster 或哨兵模式，节点故障自动转移。本地缓存（Caffeine）兜底，多级缓存架构。
3. **熔断降级**：Sentinel/Hystrix 对数据库访问熔断，失败时返回兜底数据。
4. **持久化快速恢复**：AOF/RDB 持久化，Redis 重启后快速加载历史数据。

###### 6. 缓存更新策略有哪些？
- **Cache Aside（旁路缓存）**：读时 Cache Miss → 查 DB → 写 Cache；写时先更新 DB → 再删 Cache（不是更新 Cache）。最常用。并发问题用延迟双删兜底：`DB.update → Cache.del → sleep → Cache.del`。
- **Read/Write Through**：缓存作为代理层，读写均经缓存，由缓存同步维护 DB。侵入性强，实现复杂。
- **Write Behind（异步写回）**：写 Cache 后异步批量写 DB，吞吐量高，但有数据丢失风险。Redis AOF 是类似机制。
- **定时刷新**：监听 DB Binlog（Canal 组件），异步刷新 Cache，适合读多写少的配置类数据。

**最佳实践**：大多数业务用 Cache Aside；写多读少 + 允许短暂不一致 → Write Behind；强一致性 → Read/Write Through（性能损耗大）。

###### 7. 什么是缓存预热？
缓存预热指系统上线前或 Redis 重启后，**主动将热点数据加载到缓存**，避免冷启动瞬间大量 Cache Miss 打穿数据库。

实现方式：
- **静态预热**：启动时通过 `ApplicationRunner` 预加载已知热点 Key。
- **动态预热**：分析历史访问日志，识别 Top N 热点数据，自动预热。

```java
@Component
public class CacheWarmUp implements ApplicationRunner {
    @Override
    public void run(ApplicationArguments args) {
        List<HotData> hotList = db.queryTopNHotData(1000);
        hotList.forEach(data -> redis.set(data.key, data.value, 3600));
    }
}
```

###### 8. 什么是缓存降级？
缓存降级指在 Redis 高负载或故障时，暂时关闭部分非核心缓存或直接返回降级数据，保证核心服务可用：

- **开关降级**：通过配置中心动态开关，直接返回默认值或走兜底接口（不查 Redis）。
- **容错降级**：Redis 访问超时时自动降级到数据库，记录告警日志。
- **数据降级**：返回简化版数据（如只返回基础字段，不返回详情）。

###### 9. Redis 集群模式有哪些？
- **主从复制**：一主多从，读写分离，主节点异步复制给从节点。
- **哨兵模式（Sentinel）**：在主从基础上增加哨兵节点监控，实现主节点故障自动切换。
- **Cluster 模式**：去中心化分片集群，16384 个哈希槽分布到各节点，Gossip 协议维护集群状态。

详见 → [[../../../20_RedisKnowledge/23_RedisKnowledge/05、高可用架构|Redis 三种集群模式对比]]

###### 10. Redis 哨兵模式的原理是什么？
哨兵通过 PING 心跳检测主从节点，发现主节点故障后：主观下线（单 Sentinel 判断）→ 客观下线（多数 Sentinel 确认）→ Raft 选举 Leader Sentinel → 从节点中选最优者提升为主 → 通知其他从节点和客户端。

详见 → [[../../../20_RedisKnowledge/23_RedisKnowledge/05、高可用架构|Redis 哨兵模式原理与故障转移]]

###### 11. Redis Cluster 的数据分片机制是什么？
Redis Cluster 将全部数据划分为 16384 个哈希槽，每个节点负责一段槽位。路由规则：`slot = CRC16(key) & 16383`，客户端直连对应节点。节点不匹配时返回 MOVED（永久重定向）或 ASK（迁移中临时重定向）。

详见 → [[../../../20_RedisKnowledge/23_RedisKnowledge/05、高可用架构|Redis Cluster 分片与路由机制]]

###### 12. 如何保证缓存和数据库的一致性？
分布式环境下强一致性成本极高，通常追求**最终一致性**：

1. **延迟双删**：`DB.update → Cache.del → sleep(100ms) → Cache.del`，应对并发读导致旧值回写。
2. **异步监听 Binlog**：Canal 监听 MySQL Binlog，解析变更后异步删除/更新 Redis，解耦且延迟低。
3. **版本号控制**：缓存数据携带版本号，更新时比对版本，防止旧数据覆盖新数据。

**选择依据**：读多写少、容忍短暂不一致 → Cache Aside + 延迟双删；写多、一致性要求较高 → Binlog 异步监听。
