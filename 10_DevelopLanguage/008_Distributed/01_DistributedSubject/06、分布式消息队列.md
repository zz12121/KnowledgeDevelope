###### 1. 什么是消息队列？有什么作用？
消息队列是**分布式系统中实现应用间异步通信的中间件**，采用生产者-消费者模型，核心是在消息传输过程中提供临时存储容器。

**四大核心价值**：
- **异步处理**：生产者无需等待消费者处理完成，提升系统响应速度（如下单后异步发短信）。
- **系统解耦**：服务间通过消息通信，消除接口级强耦合（如订单服务和库存服务解耦）。
- **流量削峰**：高并发请求先进队列，消费者按处理能力消化（如秒杀缓冲）。
- **消息通讯**：实现点对点或发布订阅模式的数据传递。

详见 → [[../../../20_MessageQueueKnowledge/28_MessageQueueKnowledge/01、消息队列核心概念|消息队列三大价值与核心概念]]

###### 2. 消息队列的应用场景有哪些？
- **异步处理**：用户注册后，发送邮件和短信通知异步执行，缩短主流程响应时间。
- **应用解耦**：订单服务完成下单后发消息，库存服务、物流服务、积分服务各自异步消费。
- **流量削峰**：秒杀高并发请求先进 MQ，后端服务按能力消费，避免系统崩溃。
- **日志处理**：Kafka 接收分布式系统日志，供 ELK、流处理平台消费。
- **事件驱动**：领域事件驱动微服务间的状态同步（如订单创建 → 通知用户 → 更新统计）。

###### 3. 如何保证消息不丢失？
三个层面协同保障，详见 → [[../../../20_MessageQueueKnowledge/28_MessageQueueKnowledge/02、消息可靠性|消息可靠性与不丢失保证]]:

- **生产者层**：使用 Publisher Confirm（RabbitMQ）或 `acks=all`（Kafka）确保消息成功写入 Broker；重要消息先落本地 DB 再发送，定时重试。
- **Broker 层**：开启持久化（队列 + 消息都持久化）；集群副本（RabbitMQ 镜像队列 / Kafka 多副本）防单点丢失。
- **消费者层**：关闭自动提交，手动 ACK，处理完成后再确认；消费者幂等设计防重复处理。

###### 4. 如何保证消息不重复消费？
重复消费的根源是网络重传、消费者重启或 Rebalance 导致的重复投递。解决核心是**幂等性设计**：

- **唯一消息 ID**：每条消息携带全局唯一 ID，消费前查 Redis 或 DB 判重，已处理则跳过。
- **数据库唯一约束**：消费结果写库时利用唯一索引防重（INSERT IGNORE 或唯一键冲突直接忽略）。
- **乐观锁**：通过版本号控制并发更新，确保只处理一次。

详见 → [[../../../20_MessageQueueKnowledge/28_MessageQueueKnowledge/02、消息可靠性|幂等性与不重复消费]]

###### 5. 如何保证消息的顺序性？
顺序性场景：订单创建 → 支付 → 发货，必须严格按顺序处理。

- **RabbitMQ**：单队列单消费者（吞吐低）；或按业务字段（如订单 ID）哈希路由到同一队列。
- **Kafka**：同一分区内消息按偏移量有序，通过指定相同 Key 让顺序消息落入同一分区；一个分区只分配一个消费者处理。
- **RocketMQ**：`MessageQueueSelector` 将顺序消息投递到同一队列。

详见 → [[../../../20_MessageQueueKnowledge/28_MessageQueueKnowledge/03、消息顺序性|消息顺序性实现方案]]

###### 6. 消息积压怎么处理？
消息积压 = 消费速度 < 生产速度，分应急处理和根因治理：

**应急**：扩容消费者实例（Kafka 增加分区数同步扩容）；批量消费（加大 `max.poll.records`）；暂时降级非核心消息处理逻辑。

**根因治理**：优化消费逻辑（耗时操作异步化）；设置积压阈值监控告警；合理设置消费者并发线程数。

详见 → [[../../../20_MessageQueueKnowledge/28_MessageQueueKnowledge/04、消息积压与监控|消息积压处理]]

###### 7. 什么是消息的持久化？
消息持久化是将消息保存到磁盘，防止 Broker 重启时数据丢失：

- **RabbitMQ**：队列声明 `durable=true`，消息 `deliveryMode=2`，消息和元数据都持久化到磁盘。
- **Kafka**：所有消息持久化到磁盘日志文件（分段存储 + 稀疏索引），通过 `sendfile` 零拷贝提升读写性能。

详见 → [[../../../20_MessageQueueKnowledge/29_RabbitMQKnowledge/03、消息可靠性|RabbitMQ 消息持久化]] | [[../../../20_MessageQueueKnowledge/31_KafkaKnowledge/01、核心架构|Kafka 存储机制]]

###### 8. RabbitMQ 的工作模式有哪些？
RabbitMQ 六种工作模式（按 Exchange 类型区分）：简单模式、工作队列、发布订阅（Fanout）、路由模式（Direct）、主题模式（Topic）、RPC 模式。

详见 → [[../../../20_MessageQueueKnowledge/29_RabbitMQKnowledge/02、Exchange与路由|RabbitMQ Exchange 与路由模式详解]]

###### 9. Kafka 的架构是怎样的？
Kafka 采用**分布式提交日志架构**：Producer → Broker（Topic/Partition）← Consumer Group，ZooKeeper 管理元数据（新版本已迁移到 KRaft 模式，不再依赖 ZooKeeper）。

详见 → [[../../../20_MessageQueueKnowledge/31_KafkaKnowledge/01、核心架构|Kafka 核心架构详解]]

###### 10. Kafka 如何保证高吞吐量？
六大技术手段：**顺序读写**（磁盘顺序 IO）、**页缓存**（OS page cache）、**批量处理**（生产者批量发 + 消费者批量拉）、**零拷贝**（sendfile 系统调用）、**分区并行**（多分区并行生产消费）、**压缩传输**（Snappy/LZ4/Zstd）。

详见 → [[../../../20_MessageQueueKnowledge/31_KafkaKnowledge/04、生态与实战|Kafka 高性能原理]]

###### 11. 什么是 Kafka 的 ISR 机制？
ISR（In-Sync Replicas）是**与 Leader 副本保持同步的副本集合**。Follower 与 Leader 差距在 `replica.lag.time.max.ms` 内时留在 ISR；滞后太多则移出 ISR。消息提交需 ISR 中所有副本确认（`acks=all`）。

详见 → [[../../../20_MessageQueueKnowledge/31_KafkaKnowledge/03、高可用与可靠性|Kafka ISR 与高可用机制]]

###### 12. RocketMQ 的事务消息如何实现？
RocketMQ 通过**半消息（Half Message）+ 二阶段提交**实现分布式事务最终一致性：发半消息（消费者不可见）→ 执行本地事务 → 提交（消息可见）或回滚（消息丢弃）→ 服务端定时回查未确认的半消息。

详见 → [[../../../20_MessageQueueKnowledge/30_RocketMQKnowledge/03、消息类型|RocketMQ 事务消息原理]]

###### 13. 什么是死信队列？
死信队列（DLQ）是**存储处理失败或过期消息的特殊队列**，触发条件：消息被拒绝且不重新入队（NACK）、消息 TTL 过期、队列达到最大长度。

核心价值：故障隔离（异常消息不阻塞正常队列）、重试机制（结合延迟队列）、问题排查（失败消息可回溯）。

详见 → [[../../../20_MessageQueueKnowledge/29_RabbitMQKnowledge/05、高级特性|RabbitMQ 死信队列与延迟队列]]

###### 14. 消息队列在分布式系统中如何实现最终一致性？
消息队列是分布式事务最终一致性的核心工具，典型模式是**本地消息表**：

1. 业务操作和"待发送消息"写同一个本地事务（原子性）。
2. 定时任务扫描待发送消息，投递到 MQ。
3. 消费方幂等处理，成功后回调确认。
4. 发送方更新消息状态为"已完成"。

这比 2PC/TCC 侵入性小，比纯 MQ 事务消息更通用，是最常用的最终一致性方案。

###### 15. RabbitMQ、RocketMQ、Kafka 如何选型？
- **RabbitMQ**：消息路由复杂（需要 Topic/Direct/Fanout 灵活路由）、延迟低（单机数万 QPS）、团队熟悉 AMQP 协议。适合企业内部系统、业务消息。
- **RocketMQ**：需要事务消息、顺序消息、延迟消息，Java 技术栈，阿里云生态，高可靠业务场景。
- **Kafka**：超高吞吐（百万 QPS）、日志收集、流式计算（Kafka Streams）、大数据管道。不适合消息量少但可靠性要求极高的场景。
