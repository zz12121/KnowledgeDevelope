###### 1. 什么是分布式事务？为什么单机事务不够用？
分布式事务是指**跨多个独立数据节点（数据库、服务、消息队列）的事务操作**，需要保证这些操作要么全部成功，要么全部回滚。

**单机事务不够用的原因**：
- 微服务架构下，不同服务有独立的数据库，无法用单个数据库事务跨库协调。
- 分库分表后，同一业务的数据分散在多个数据库实例。
- 涉及 MQ、外部 API 等非数据库资源时，传统 ACID 事务无法覆盖。

**分布式事务的挑战**：网络延迟和故障不可避免，节点可能部分失败，强一致性成本极高，必须在一致性、可用性、性能之间做权衡。

###### 2. 什么是 2PC（两阶段提交）？
2PC 是最基础的分布式事务协议，由**事务协调者（TM）**控制多个**资源管理者（RM）** 完成两阶段操作：

**阶段一（Prepare）**：协调者向所有参与者发送 Prepare 请求，各参与者执行事务但不提交，将结果（Vote Yes/No）返回给协调者。

**阶段二（Commit/Rollback）**：若所有参与者都投 Yes，协调者发送 Commit；否则发送 Rollback。参与者执行对应操作并返回 ACK。

**核心问题**：
- **同步阻塞**：Prepare 阶段参与者持有锁等待协调者指令，高并发下容易成为瓶颈。
- **单点故障**：协调者宕机后参与者无限等待（虽有超时机制，但状态不确定）。
- **数据不一致**：Phase 2 网络异常可能导致部分提交部分回滚（脑裂）。

###### 3. 什么是 TCC 事务模型？
TCC（Try-Confirm-Cancel）是应用层设计的柔性分布式事务，通过**业务逻辑分解**实现最终一致性，对 DB 锁依赖极少。

**三阶段**：
- **Try（资源预留）**：检查并预留业务资源。例如下单：冻结库存（可用库存-N，冻结库存+N）、预扣余额（余额-N，冻结金额+N）。**必须幂等**。
- **Confirm（确认执行）**：Try 全部成功后执行，清除冻结，提交业务。**必须幂等且不失败**。
- **Cancel（取消撤销）**：Try 任一失败后执行，释放预留资源。**必须幂等**。

```java
// Seata TCC 示例（注解方式）
@TwoPhaseBusinessAction(name = "inventoryTCC", commitMethod = "confirm", rollbackMethod = "cancel")
boolean try(BusinessActionContext context, @BusinessActionContextParameter(paramName = "productId") Long productId, int count);
boolean confirm(BusinessActionContext context);
boolean cancel(BusinessActionContext context);
```

**TCC 特有问题**：
- **空回滚**：Try 未执行（网络超时）但 Cancel 被调用，Cancel 要能识别并直接返回成功。
- **幂等**：Confirm/Cancel 可能被重试调用多次，必须保证幂等。
- **悬挂**：Cancel 先于 Try 执行（网络超时后 Cancel 先到达），Try 要能识别已取消并直接返回失败。

###### 4. TCC 和 2PC 的区别是什么？
| 维度 | 2PC（XA） | TCC |
|------|----------|-----|
| 协议层次 | 数据库/资源管理器层 | 应用业务层 |
| 锁定粒度 | 数据库行锁（Prepare → Commit 持续占用） | 业务层软锁（状态字段，非 DB 锁） |
| 一致性 | 强一致性（ACID） | 最终一致性（允许中间状态） |
| 业务侵入 | 几乎无侵入 | 需显式实现 Try/Confirm/Cancel 三接口 |
| 性能 | 低（同步阻塞，锁竞争） | 高（预留阶段快速，二阶段可异步） |
| 适用 | 短事务、低并发、强一致（传统系统） | 高并发、长事务、互联网场景 |

###### 5. 什么是 Seata？它支持哪些事务模式？
Seata 是阿里开源的分布式事务框架，四种模式：

1. **AT 模式**（默认推荐）：基于数据库本地 ACID 事务，自动解析 SQL 生成前后镜像（undo_log），实现自动补偿。业务**零代码侵入**，只需添加 `undo_log` 表并配置数据源代理。一阶段提交本地事务，二阶段异步删除镜像（成功）或根据镜像回滚（失败）。
2. **TCC 模式**：如上所述，需要业务实现三接口，Seata 负责协调全局事务。
3. **Saga 模式**：长事务解决方案，每步提交本地事务，失败时逆序调用补偿。通过状态机引擎（StateMachineEngine）编排，支持可视化流程定义。
4. **XA 模式**：基于数据库原生 XA 协议，Seata 作为事务管理器（TM）协调，强一致性但性能较低。

###### 6. AT 模式和 TCC 模式的区别是什么？
| 维度 | AT 模式 | TCC 模式 |
|------|---------|---------|
| 实现原理 | SQL 解析 + undo_log 自动补偿 | 业务接口显式定义 Try/Confirm/Cancel |
| 业务侵入 | 零侵入（加 undo_log 表即可） | 强侵入（需实现三个方法） |
| 适用资源 | 仅支持关系型数据库（ACID） | 任何可定义预留/确认/取消语义的资源（Redis、MQ、外部 API） |
| 锁机制 | 一阶段提交后释放 DB 锁，全局锁（global_lock 表）防并发冲突 | 业务状态字段软锁，无 DB 锁竞争 |
| 性能 | SQL 解析和镜像生成有开销 | 无 SQL 解析，性能更优 |
| 开发成本 | 低，配置即用 | 高，需仔细设计空回滚/幂等/悬挂处理 |

**选型建议**：系统以 MySQL 为主、希望低成本接入 → AT；需要高性能或操作非数据库资源 → TCC；超长业务流程 → Saga。

###### 7. 本地消息表方案的实现原理是什么？
本地消息表是利用**数据库事务 + 消息队列**实现最终一致性的经典方案，核心是"将分布式事务拆为本地事务 + 异步消息"：

**实现步骤**：
1. **业务操作和写消息在同一本地事务**：业务 SQL + `INSERT INTO local_message(biz_data, status='PENDING')` 同一事务提交。
2. **定时任务投递**：后台定时扫描 `status='PENDING'` 的消息，发送到 MQ，发送成功则更新 `status='SENT'`；失败则重试（限 retry_count）。
3. **消费方幂等处理**：消费者处理完成后发送确认，生产方更新 `status='DONE'`。

**关键设计**：生产方写消息与业务操作原子性（同一 DB 事务）；消费方必须幂等（消息 ID 去重）；消息 ID 应包含业务语义，便于对账。

###### 8. Saga 模式的工作原理是什么？
Saga 模式是解决**长业务流程**分布式事务的方案，核心思想：将大事务拆分为一系列**本地事务序列**，每个本地事务提交后立即可见，失败时**逆序执行补偿操作**。

**两种实现方式**：
- **编排（Orchestration）**：一个中心协调器（Seata 的状态机引擎）按顺序调用各服务，失败时调用补偿接口。适合复杂流程，可视化清晰。
- **编舞（Choreography）**：服务间通过事件/消息链式触发，每个服务自己判断下一步和补偿。去中心化，但流程难以追踪。

**与 TCC 的区别**：TCC 的 Try 阶段会预留资源（有隔离性），Saga 的每步直接提交（中间状态可见），适合允许中间状态的长事务场景（如出行预订：订机票 → 订酒店 → 租车，任一失败逆序取消）。

###### 9. 分布式事务的性能开销如何优化？
1. **选合适模式**：非强一致场景优先选本地消息表（最终一致）或 Saga；AT 模式次之；TCC 高性能；XA/2PC 最后考虑。
2. **减少事务粒度**：将大事务拆小，减少锁占用时间；TCC 的 Try 阶段尽量轻量。
3. **异步化非核心**：日志记录、通知等非核心操作异步处理，不纳入分布式事务范围。
4. **Seata AT 优化**：定期清理 undo_log 历史数据；优化 undo_log 表索引；使用 Seata Server 集群；减少对热点数据的并发更新。
5. **避免全局锁热点**：AT 模式默认对更新操作加全局锁，通过业务设计减少对同一行数据的并发写。

###### 10. 分布式事务的常见陷阱有哪些？
1. **忘记幂等**：Confirm/Cancel 在网络超时后可能被重试，不幂等会导致数据错乱。
2. **忘记空回滚**：Cancel 先于 Try 到达（网络分区后 Cancel 先执行），Cancel 要能检测到未执行的 Try 并直接成功。
3. **悬挂问题**：Try 在 Cancel 之后才到达（已超时回滚了），Try 要能检测到已取消状态并直接失败，否则资源永久预留。
4. **事务范围过大**：将非必要的操作纳入分布式事务，导致锁持有时间过长，拖垮系统性能。
5. **忽略补偿的失败**：Saga/TCC 的 Confirm/Cancel 也可能失败，需要重试机制兜底，最终仍可能需要人工处理（引入告警 + 对账）。
