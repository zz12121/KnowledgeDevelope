###### 1. 什么是配置中心？

配置中心是微服务架构中的关键基础设施，用于**集中、统一、动态地管理所有微服务应用的配置信息**。

传统配置方式（本地配置文件、环境变量）的痛点：配置分散难维护、修改后需重启应用、缺乏版本历史和权限管控、生产配置易误操作。配置中心正是解决这些痛点而生。

**核心价值**

| 能力 | 说明 |
|---|---|
| **集中管理** | 所有环境（开发/测试/生产）在一个平台统一管理，按应用、环境、集群隔离 |
| **动态热更新** | 配置修改后实时推送到应用实例，无需重启（热更新） |
| **版本与审计** | 记录所有修改历史，支持一键回滚，满足合规审计要求 |
| **灰度发布** | 新配置仅推送给部分实例，验证通过后再全量推送 |
| **高可用** | 作为关键基础设施，必须保证自身高可用 |

**常见配置类型**：数据库连接串、线程池大小、功能开关（Feature Flag）、业务规则（活动规则、费率）、环境差异化配置等。

###### 2. Apollo 的架构是怎样的？

Apollo（携程开源）采用经典分层架构，核心目标是实现**配置发布 + 实时推送 + 高可用**。

**核心组件**

```
Portal（管理 UI）
    ↓ 增删改查、发布
Admin Service（配置管理 API）
    ↓ 写入数据库、发布变更消息
    ↑ 服务注册
Config Service（配置查询 API）   ← Eureka（服务发现）
    ↑ 长轮询监听
Apollo 客户端（集成在应用中）
    ↓ 本地文件缓存（容灾）
```

**配置动态推送流程（核心）**

1. 用户在 Portal 修改配置并发布，Admin Service 将新配置写入 MySQL，并向 `ReleaseMessage` 表插入变更消息；
2. Config Service 内有 `ReleaseMessageScanner` 线程，周期性扫描该表，发现新消息后通知所有监听此 Namespace 的客户端；
3. 客户端启动时通过 `RemoteConfigLongPollService` 向 Config Service 发起**长轮询**请求（Hold 住连接）；
4. Config Service 收到新消息后，**中断长轮询**连接返回变更的 Namespace 信息；
5. 客户端收到通知后，主动拉取最新配置，更新内存中的 `Config` 对象，并触发 Spring 的 `RefreshEvent`，刷新 `@RefreshScope` 注解的 Bean。

**高可用设计**：Config Service 无状态，可水平扩展。客户端有**本地文件缓存**（`apollo-config` 目录），即使所有 Config Service 宕机，应用也能用缓存正常启动。

###### 3. Nacos 和 Apollo 的区别是什么？

| 维度 | Apollo | Nacos |
|---|---|---|
| **核心定位** | **专注配置管理**，功能深度突出 | **一站式**服务发现 + 配置管理双合一 |
| **配置模型** | Namespace → AppId → Cluster，模型严谨 | Namespace → Group → DataId，更灵活但需自行规划 |
| **推送机制** | **HTTP 长轮询**，稳定成熟 | 支持 **UDP 推送**（延迟极低，可能丢包）和 **HTTP 长轮询** |
| **格式支持** | properties、yaml | properties、yaml、xml、json、html、text 多格式原生支持 |
| **部署依赖** | 依赖 Eureka（服务发现）+ MySQL，组件较多 | **无外部依赖**，单机用内嵌 Derby，集群用 MySQL，部署更简单 |
| **生态集成** | 与 Spring Cloud 集成良好 | Spring Cloud Alibaba 核心组件，与 Sentinel/Seata 生态更紧密 |
| **权限管控** | 完善的用户/角色/权限体系，多团队协作更成熟 | 早期较弱，后续版本增强，但整体上 Apollo 更成熟 |

**选型建议**：
- 只需强大配置管理，已有服务发现方案 → **Apollo**（更专业、稳健）
- 追求架构简化，统一服务发现和配置管理，或在 Spring Cloud Alibaba 生态中 → **Nacos**（开箱即用）

###### 4. 配置中心如何实现配置的动态更新？

动态更新的核心是让客户端**感知**到服务端配置变化，并**安全地**将新配置应用到运行中的应用中。

**服务端：变更感知与通知**

```
用户发布新配置
    ↓
持久化到数据库 + 发布变更事件
（DB轮询 / MQ广播 / UDP推送）
    ↓
通知订阅客户端
```

**客户端：监听与热更新**

1. **长连接/长轮询**：客户端启动后向配置中心发起长轮询，服务端将请求挂起（Hold），有变更时主动返回响应；
2. **主动拉取**：客户端收到"有变更"通知后，主动调用接口拉取最新配置（避免推送丢包导致数据不一致）；
3. **线程安全更新内存**：SDK 将新配置更新到内存配置对象，必须保证线程安全，避免读取时脏读；
4. **触发框架刷新**：发布 `RefreshEvent` / `EnvironmentChangeEvent`，被 `@RefreshScope` 注解的 Bean 会被销毁并重建，注入新配置值。

```java
// Spring Cloud 中，@RefreshScope 让 Bean 在配置变更时自动重建
@RestController
@RefreshScope   // 关键注解
public class FeatureController {
    @Value("${feature.new-ui.enabled:false}")
    private boolean newUiEnabled;

    @GetMapping("/ui-mode")
    public String getUiMode() {
        return newUiEnabled ? "new" : "legacy";
    }
}
```

> **注意**：不是所有 Bean 都需要 `@RefreshScope`。只对那些需要热更新的配置项的 Bean 加此注解，否则会增加不必要的 Bean 重建开销。

###### 5. 配置中心如何保证高可用？

**服务端高可用**

- **集群部署**：核心组件（Apollo 的 Config/Admin Service，Nacos Server）都以多实例集群方式部署，通过负载均衡或服务发现消除单点；
- **多活数据中心**：跨机房部署，数据通过 MySQL 主从复制或 Raft（Nacos 基于 Raft）同步；
- **存储高可用**：MySQL 主从复制 + 读写分离，Nacos 内置 Raft 保证分布式一致性；
- **无状态化**：服务端组件设计为无状态，方便水平扩展和快速故障转移。

**客户端高可用（最关键）**

配置中心高可用设计的精髓是：**"服务端可降级，客户端须容灾"**。

- **本地文件缓存**：客户端首次成功获取配置后，将配置**持久化到本地磁盘**。服务端完全不可用时，客户端自动降级使用本地缓存启动，确保应用不受影响；
- **内存快照**：运行期配置保存在内存中，与服务器断连时，应用仍基于内存中最新配置运行；
- **重试与保护阈值**：服务端返回错误时，不轻易清空本地缓存，防止网络抖动导致配置丢失；
- **容灾演练**：定期模拟配置中心宕机，验证客户端降级和恢复机制是否有效。

###### 6. 配置中心如何实现配置加密？

生产环境中，数据库密码、第三方 API Key 等敏感配置**绝对不能明文存储**在配置中心。

**常见加密方案**

**方案1：客户端透明解密（推荐）**

```yaml
# 配置中心存储密文（约定前缀 ENC(...)）
datasource:
  password: ENC(xTvb4eWJ/qeQJPMkH0N8SQ==)
```

```xml
<!-- 引入 Jasypt 自动解密依赖 -->
<dependency>
    <groupId>com.github.ulisesbocchio</groupId>
    <artifactId>jasypt-spring-boot-starter</artifactId>
</dependency>
```

```yaml
# 解密密钥通过环境变量注入（不放在配置文件中！）
JASYPT_ENCRYPTOR_PASSWORD: your-secret-key
```

Jasypt 会在 Spring 环境初始化时自动解密 `ENC(...)` 格式的值，对业务代码完全透明。

**方案2：KMS（Key Management Service）**

使用云厂商的 KMS（如腾讯云 KMS、AWS KMS）或 HashiCorp Vault 管理密钥：
- 配置中心只存储密文和 KMS Key ID；
- 应用启动时向 KMS 申请解密，KMS 校验 IAM 权限后返回明文；
- 优点：密钥的生命周期（生成、轮换、吊销）由专业系统管理，安全级别更高。

**方案3：Nacos 内置加密插件（Nacos 2.x）**

Nacos 2.x 支持通过 SPI 扩展接口自定义加密插件，实现配置的加密存储和透明解密。

**最佳实践**：
- 数据库密码、私钥等高敏感配置 → KMS；
- 普通 API Key → Jasypt 客户端加密；
- **密钥本身绝不放配置中心**，通过环境变量或 K8s Secret 注入。

###### 7. 如何设计多环境（dev/test/prod）的配置隔离？

多环境隔离是配置中心的核心应用场景之一，目标是让不同环境的配置互不干扰，同时减少重复配置。

**Apollo 的多环境方案**

Apollo 以 **Environment（环境）+ Cluster（集群）+ Namespace（命名空间）** 三级隔离：
- 不同 Environment（DEV/FAT/UAT/PRO）部署独立的 Config Service，彻底物理隔离；
- 公共配置放在 `application` Namespace，各服务私有配置放在各自的 Namespace；
- 环境间配置不会互相影响，生产环境变更需独立审批。

**Nacos 的多环境方案**

利用 **Namespace** 实现环境隔离（每个环境一个 Namespace），利用 **Group** 进一步按业务线或集群分组：

```yaml
spring:
  cloud:
    nacos:
      config:
        namespace: prod-namespace-id   # 生产环境 Namespace ID
        group: ORDER-SERVICE           # 服务分组
        file-extension: yaml
```

**通用最佳实践**

1. **配置继承（公共 + 私有）**：将通用配置（如日志格式、公共超时）放在 `common` Namespace/文件，各服务私有配置继承后覆盖差异项，减少重复；
2. **生产环境保护**：生产配置只读给开发人员，只有运维或上线负责人有修改权限；
3. **配置版本管理**：生产环境每次变更必须有关联的变更单号，并记录变更原因，方便回溯；
4. **不同环境变量化**：数据库地址、域名等环境差异配置通过变量占位符管理，环境间切换只需修改对应 Namespace 下的变量值。
