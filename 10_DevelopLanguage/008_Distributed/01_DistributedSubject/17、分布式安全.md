###### 1. 分布式系统中常见的安全威胁有哪些？

分布式系统因组件众多、网络通信复杂、边界模糊，面临比单体系统更严峻的安全挑战。

**1. 身份认证与授权漏洞**
- **弱认证**：简单密码、默认凭证、认证逻辑缺陷
- **越权访问**：水平越权（访问他人数据）和垂直越权（获取管理员功能）
- **凭证泄露**：API Key、JWT 令牌被窃取或不当存储（如写入代码库）

**2. 网络与通信攻击**
- **中间人攻击（MITM）**：拦截、窃听或篡改未加密的明文通信
- **重放攻击**：截获有效请求包，之后重复发送冒充合法操作
- **DDoS 攻击**：海量恶意流量耗尽 API 网关、认证服务等关键组件资源

**3. 数据安全威胁**
- **敏感数据泄露**：数据库拖库、日志记录敏感信息、配置文件硬编码密码
- **注入攻击**：SQL 注入、NoSQL 注入、命令注入等

**4. API 与接口滥用**
- **接口未保护**：内部或管理接口暴露公网且无认证
- **无速率限制**：API 被爬虫或恶意程序刷取

**5. 基础设施与配置安全**
- **不安全配置**：容器使用 root 用户、开放不必要端口、使用有漏洞的依赖组件
- **供应链攻击**：第三方库（如 Log4j）存在严重安全漏洞

**6. 分布式特有威胁**
- **服务间信任问题**：缺乏服务间认证（mTLS），内部网络被横向移动攻击
- **配置中心/注册中心攻击**：这些核心组件一旦被攻击，可导致服务发现混乱或配置被恶意修改

防御需建立**纵深安全体系**：网络隔离 + 传输加密 + 强身份认证 + 最小权限授权 + 输入校验 + 安全依赖管理 + 持续漏洞扫描。

###### 2. 什么是 OAuth 2.0？

OAuth 2.0 是一个**授权框架**，而非认证协议。它解决的核心问题是：**如何安全地让第三方应用在用户授权后，代表用户访问其资源，而无需向第三方暴露用户密码**。

**四大角色**
- **资源所有者（用户）**：拥有被访问资源的人
- **客户端（第三方应用）**：试图访问用户资源的应用
- **授权服务器**：验证用户身份并颁发访问令牌
- **资源服务器**：存放受保护资源的 API 服务器

**授权码模式流程（最安全，推荐）**

```
客户端 → 授权服务器（用户登录并授权）
授权服务器 → 客户端重定向回调 URL + 授权码
客户端后端 → 授权服务器（用授权码 + ClientID + ClientSecret 换 AccessToken）
授权服务器 → 颁发 AccessToken（+ 可选 RefreshToken）
客户端 → 资源服务器（携带 AccessToken 请求资源）
```

**其他授权模式**
- **隐式模式**：直接返回 AccessToken（安全性差，已不推荐，被 PKCE 替代）
- **密码模式**：用户直接提供用户名密码给客户端（仅适用于高度信任的自家客户端）
- **客户端凭证模式**：服务间调用，客户端代表自己而非用户访问资源

**OAuth 2.0 vs OIDC**：OAuth 2.0 只处理授权，OpenID Connect（OIDC）是基于 OAuth 2.0 的认证层扩展，在 AccessToken 之外额外颁发 **ID Token**（含用户身份信息），解决了"我知道你可以访问，但我不知道你是谁"的问题。

###### 3. JWT 的工作原理是什么？

JWT 是一种**紧凑、自包含**的令牌格式，用于在各方之间安全传输 JSON 格式的信息。可被签名（HMAC / RSA），确保信息可信。

**JWT 结构**：`Header.Payload.Signature`（三段 Base64URL 编码，用 `.` 分隔）

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9  ← Header（算法+类型）
.eyJzdWIiOiJ1c2VyMTIzIiwiZXhwIjoxNzAwMDAwMDAwfQ  ← Payload（声明）
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c    ← Signature（签名）
```

**Payload 常用声明**：`sub`（主体/用户ID）、`exp`（过期时间）、`iss`（签发者）、`iat`（签发时间）、自定义业务声明（如 `roles`）。

**工作流程**

```
用户登录 → 认证服务颁发 JWT（含用户信息+过期时间）→ 客户端存储
后续请求：Authorization: Bearer <token>
资源服务器验证：重算签名比对 + 检查 exp/iss 声明
```

**优点**：无状态，服务端不需要存储 Session，天然适合分布式横向扩展。

**关键注意事项**
- **Payload 是明文**（只是 Base64URL 编码，不是加密），**绝不能存放密码、手机号**等敏感信息
- JWT 一旦签发**无法主动吊销**（除非配合黑名单机制），因此 `exp` 不宜设置太长（一般 AccessToken 15 分钟，RefreshToken 7天）
- 密钥（Secret）的安全管理至关重要，建议使用非对称签名（RS256）：认证服务持有私钥签发，各资源服务仅持有公钥验证

```java
// 使用 jjwt 创建和验证
String token = Jwts.builder()
    .setSubject("user123")
    .claim("roles", List.of("ADMIN"))
    .setExpiration(new Date(System.currentTimeMillis() + 15 * 60 * 1000))
    .signWith(privateKey, SignatureAlgorithm.RS256)
    .compact();

// 验证（只需公钥）
Claims claims = Jwts.parserBuilder()
    .setSigningKey(publicKey)
    .build()
    .parseClaimsJws(token)
    .getBody();
```

###### 4. 如何防止分布式系统中的重放攻击？

重放攻击防御核心是确保每个请求的**唯一性（Nonce）+ 时效性（Timestamp）**。

**组合策略（业界最佳实践）**

每个 API 请求必须携带：
- `X-Timestamp`：请求发起的 UTC 时间戳
- `X-Nonce`：一次性随机字符串（UUID）
- `X-Signature`：对 `方法 + URI + 时间戳 + Nonce + 请求体` 用客户端密钥计算的 HMAC 签名

**服务端校验逻辑**：

```java
public void validateRequest(HttpServletRequest request) {
    String timestamp = request.getHeader("X-Timestamp");
    String nonce = request.getHeader("X-Nonce");
    String signature = request.getHeader("X-Signature");

    // 1. 校验时间戳窗口（±5分钟）
    long requestTime = Long.parseLong(timestamp);
    if (Math.abs(System.currentTimeMillis() / 1000 - requestTime) > 300) {
        throw new SecurityException("请求已过期");
    }

    // 2. 校验 Nonce 唯一性（用 Redis，过期时间略大于时间窗口）
    String nonceKey = "nonce:" + clientId + ":" + nonce;
    if (!redis.setIfAbsent(nonceKey, "1", Duration.ofMinutes(6))) {
        throw new SecurityException("重放请求被拒绝");
    }

    // 3. 验证签名完整性
    String expectedSig = hmacSHA256(method + uri + timestamp + nonce + body, clientSecret);
    if (!MessageDigest.isEqual(expectedSig.getBytes(), signature.getBytes())) {
        throw new SecurityException("签名验证失败");
    }
}
```

**四种防御手段对比**

| 手段 | 原理 | 局限 |
|---|---|---|
| **Nonce + Redis** | 一次性随机数，用后标记已使用 | Redis 需要高可用，存在短暂窗口 |
| **时间戳窗口** | 拒绝过期请求，简单有效 | 需要时钟同步（NTP），窗口内仍可重放 |
| **序列号** | 递增序列，只接受更大序号 | 分布式下序号维护复杂 |
| **HTTPS** | 防止传输中截获 | 不能防止合法客户端恶意重放 |

**生产推荐**：时间戳（5分钟窗口）+ Nonce（Redis 去重）+ HMAC 签名，三者结合，相互补充。

###### 5. 什么是 API 网关的安全策略？

API 网关是所有流量的入口，是实施统一安全策略的最佳位置，构成保护后端微服务的第一道防线。

**核心安全能力**

**1. 统一认证**：网关集中验证令牌（JWT / OAuth AccessToken / API Key），通过后透传用户身份给下游，避免每个微服务重复实现认证逻辑。

**2. 粗粒度授权**：基于 JWT 中的 `roles` / `scope` 声明，做路径级别的权限校验（精细的资源级授权仍由各业务服务负责）。

**3. 传输安全**：强制 HTTPS（TLS 终止），内部服务间可采用 HTTP 或 mTLS（Service Mesh 场景）。

**4. 限流防刷**：基于 IP / 用户 ID / API Key 实施速率限制，防 DDoS 和资源滥用。

**5. 防重放**：集成时间戳 + Nonce 校验（见上题）。

**6. 请求脱敏**：响应返回给客户端前，对敏感字段（手机号、身份证号）进行掩码处理。

**Spring Cloud Gateway 实现示例**

```java
@Component
public class JwtAuthFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = extractToken(exchange.getRequest());
        if (token == null) {
            return unauthorized(exchange, "缺少认证令牌");
        }
        return jwtDecoder.decode(token)
            .flatMap(jwt -> {
                // 权限校验
                if (!hasRequiredRole(jwt, exchange.getRequest().getPath().value())) {
                    return forbidden(exchange, "权限不足");
                }
                // 将用户信息透传给下游
                ServerHttpRequest mutatedRequest = exchange.getRequest().mutate()
                    .header("X-User-Id", jwt.getSubject())
                    .header("X-User-Roles", String.join(",", getRoles(jwt)))
                    .build();
                return chain.filter(exchange.mutate().request(mutatedRequest).build());
            })
            .onErrorResume(e -> unauthorized(exchange, "令牌无效或已过期"));
    }
}
```

###### 6. 什么是零信任安全架构？在分布式系统中如何落地？

**零信任的核心理念**

传统安全模型假设"内网是可信的"，只在边界做防御。零信任彻底颠覆这一假设：**"永不信任，持续验证"（Never Trust, Always Verify）**——无论请求来自内网还是外网，每次访问都必须经过身份验证和授权。

**为什么微服务场景特别需要零信任**

微服务之间频繁互相调用，传统方案假设"内网调用是安全的"，但一旦某个服务被攻击，攻击者可以在内网横向移动，访问所有其他服务。

**零信任在分布式系统中的落地方案**

**1. 服务间 mTLS（双向 TLS）**

每个服务都有独立的身份证书，服务间调用时双方互相验证对方证书，确保"我知道我在和谁说话"。在 Kubernetes + Istio 场景下，mTLS 由 Service Mesh 自动处理，业务代码无感知。

```yaml
# Istio PeerAuthentication：强制命名空间内所有服务间使用 mTLS
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT
```

**2. 最小权限原则（Least Privilege）**

每个服务只授予完成其职责所需的最小权限。在 K8s 中通过 RBAC 和 ServiceAccount 实现服务身份隔离：

```yaml
# 每个服务绑定独立 ServiceAccount，只授予需要的权限
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
rules:
- apiGroups: [""]
  resources: ["configmaps", "secrets"]
  verbs: ["get", "list"]   # 只读，不允许修改
```

**3. 动态访问控制**

基于请求上下文（用户身份 + 设备状态 + 地理位置 + 时间）动态决定是否允许访问，而不是静态的"白名单"规则。

**4. 持续监控与审计**

所有服务间调用都记录审计日志，异常行为（如深夜访问、高频调用）触发告警，做到"事后可追溯"。

###### 7. 如何实现分布式系统的审计日志？

审计日志记录**"谁在什么时候对什么资源做了什么操作，结果如何"**，用于合规审计和安全取证。

**设计要点**

1. **审计事件模型**：每条审计记录至少包含：
   - **主体**：用户ID、用户名、客户端IP、来源服务
   - **时间**：精确时间戳（毫秒级）
   - **操作**：类型（增/删/改/查/登录）+ 对象（资源类型 + 资源ID）
   - **变更内容**：更新操作的前后值（diff）
   - **结果**：成功/失败 + 失败原因
   - **追踪标识**：**TraceId**（关联请求链）

2. **采集方式（AOP + 异步）**

```java
@Aspect
@Component
public class AuditLogAspect {
    @Around("@annotation(auditLog)")
    public Object around(ProceedingJoinPoint pjp, AuditLog auditLog) throws Throwable {
        AuditEvent event = AuditEvent.builder()
            .operator(SecurityContext.getCurrentUserId())
            .operation(auditLog.value())
            .resourceId(parseResourceId(pjp.getArgs()))
            .traceId(MDC.get("traceId"))   // 从 MDC 获取链路 ID
            .startTime(Instant.now())
            .build();
        try {
            Object result = pjp.proceed();
            event.setSuccess(true);
            return result;
        } catch (Exception e) {
            event.setSuccess(false);
            event.setFailReason(e.getMessage());
            throw e;
        } finally {
            event.setEndTime(Instant.now());
            eventPublisher.publishEvent(event);  // 异步发布，不阻塞主流程
        }
    }
}

@Async
@EventListener
public void handleAuditEvent(AuditEvent event) {
    // 通过 Kafka 发送到审计服务，再写入 Elasticsearch
    kafkaTemplate.send("audit-log", event);
}
```

3. **存储选型**：审计日志是写多读少、按时间范围查询的场景，推荐 **Elasticsearch**（支持全文搜索、按操作类型聚合分析）；按月分 Index，控制单个 Index 大小。

4. **可靠性保障**：通过 **Kafka** 作为中间缓冲，避免 ES 抖动时丢失审计记录；审计日志不允许删除，需有防篡改机制（只读存储或区块链存证）。
