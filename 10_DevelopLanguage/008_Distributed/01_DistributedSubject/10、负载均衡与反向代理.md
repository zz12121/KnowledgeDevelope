###### 1. 什么是负载均衡？
负载均衡是将网络流量或计算任务**智能分发到多个后端服务器**的技术，目标是优化资源使用、最大化吞吐量、最小化响应时间、避免单点过载。

核心是一个调度器，根据预设算法和实时健康状态，在多个后端资源之间分配请求。按位置分：可以在网络层（L4，如 LVS）、应用层（L7，如 Nginx）或客户端（如 Ribbon）实现。

###### 2. 负载均衡的分类有哪些？
**按实现位置**：
- **硬件负载均衡**：F5 BIG-IP、A10，性能极高，成本昂贵，通常用于金融核心系统。
- **软件负载均衡**：Nginx、LVS、HAProxy，灵活成本低，是互联网公司主流选择。

**按网络层次（OSI 层）**：
- **四层（L4）**：基于 IP + TCP/UDP 端口转发，对应用透明，效率极高。代表：LVS、F5。
- **七层（L7）**：基于应用层协议（HTTP/HTTPS/gRPC）内容转发，可根据 URL、Header、Cookie 精细路由。代表：Nginx、HAProxy。

**按部署位置**：
- **服务端负载均衡**：独立部署在客户端与服务端之间（如 Nginx 前置），所有流量都经过它。
- **客户端负载均衡**：逻辑集成在消费者内（如 Spring Cloud LoadBalancer/Ribbon），客户端自行选择目标实例。

###### 3. Nginx 的负载均衡策略有哪些？
Nginx 在 `ngx_http_upstream_module` 模块实现多种策略：

1. **轮询（默认）**：按时间顺序逐一分配，实现简单，适合后端服务能力相同的场景。
2. **加权轮询**：通过 `weight` 参数设置权重，内部用平滑加权算法（Smooth Weighted Round Robin）避免短时集中，适合后端性能不同的场景。
3. **IP 哈希（ip_hash）**：根据客户端 IP 哈希值固定路由到同一后端，用于**会话保持**（但更推荐会话外置到 Redis）。
4. **最少连接（least_conn）**：优先分配给当前活跃连接数最少的后端，适合后端处理时间差异大的场景。
5. **响应时间（least_time，Nginx Plus）**：商业版，优先分配给响应时间最短的后端。

**源码角度**：加权轮询核心在 `ngx_http_upstream_get_peer` 函数，动态计算 `effective_weight` 和 `current_weight` 实现平滑分发。

###### 4. LVS 的工作模式有哪些？
LVS（Linux Virtual Server）是内核级四层负载均衡，三种工作模式：

**NAT 模式**：
- 原理：LVS 修改数据包目标 IP → 真实服务器 IP，响应时再改回源地址。双向流量经过 LVS。
- 优点：真实服务器配置简单，只需将网关指向 LVS。
- 缺点：LVS 是双向流量瓶颈，大流量下容易成为性能短板。

**TUN 模式（IP 隧道）**：
- 原理：LVS 将请求封装在新 IP 包中（IP-in-IP 隧道），真实服务器解封后**直接响应客户端**（不经过 LVS）。
- 优点：响应流量不经过 LVS，性能更高；支持跨网段部署。
- 缺点：真实服务器需支持 IP 隧道协议，配置复杂。

**DR 模式（直接路由，最常用）**：
- 原理：LVS 只修改数据帧 MAC 地址（不修改 IP），转发给真实服务器。服务器配置与 LVS 相同的 VIP（但通过 ARP 抑制不响应 ARP 请求），处理后**直接响应客户端**。
- 优点：性能最高（只有入站流量经过 LVS）；真实服务器不需修改路由。
- 缺点：LVS 与真实服务器必须在同一二层网络（不能跨路由）。

###### 5. 什么是反向代理？
反向代理代表**后端服务器**接收客户端请求，转发给内部服务器并将响应返回客户端。对客户端而言，它就是原始服务器。

核心功能：负载均衡（分发到多后端）、安全隔离（隐藏真实 IP）、缓存加速（静态资源缓存）、SSL 终结（HTTPS 在代理处解密，后端走 HTTP）、压缩优化。

代表：Nginx、Apache HTTP Server。

###### 6. 正向代理和反向代理的区别是什么？
**本质区别**：代理的对象不同。

- **正向代理**：代理**客户端**，代表客户端向外部服务器发请求。客户端需要显式配置代理地址。典型：科学上网工具、企业出口代理。
- **反向代理**：代理**服务器**，代表服务器接收客户端请求。客户端无感知，认为直接访问目标服务器。典型：Nginx 作为 Web 服务器入口、CDN 边缘节点。

**记忆口诀**：正向代理"隐藏客户端"，反向代理"隐藏服务器"。

###### 7. 如何实现动态负载均衡？
动态负载均衡能根据后端服务器实时状态自动调整流量分配：

1. **健康检查**：定期主动（HTTP 探测/TCP SYN）或被动（观察错误）检查后端健康状态，自动剔除故障节点。
2. **服务发现集成**：客户端负载均衡器（Ribbon/Spring Cloud LoadBalancer）从服务注册中心（Eureka/Nacos）动态获取实例列表，感知实例上下线。
3. **权重动态调整**：集成 Prometheus 收集后端性能指标，通过 API 动态调整 Nginx 权重（响应慢的降低权重）。
4. **自适应算法**：Nginx Plus 的 `least_time` 策略根据实时响应时间自动调整。

###### 8. 什么是会话保持（Session Sticky）？
会话保持将**同一客户端的连续请求路由到同一后端服务器**，用于有状态服务（如 Session 存在服务器内存）的负载均衡场景。

**三种实现方式**：
1. **IP 哈希**：Nginx `ip_hash`，简单但同一 NAT 网络下多用户会集中到同一后端。
2. **Cookie 注入**：负载均衡器在第一次响应中植入含后端 ID 的 Cookie，后续请求根据 Cookie 路由（如 `JSESSIONID=node01`）。
3. **URL 参数**：在 URL 中附加服务器标识参数进行路由。

**缺点**：破坏负载均衡纯粹性，目标服务器宕机时该服务器的 Session 丢失。

**最佳实践**：现代架构推荐**无状态服务 + Session 外置（Redis）**，彻底不需要会话保持。

###### 9. 四层负载均衡和七层负载均衡的区别是什么？
| 维度 | 四层（L4） | 七层（L7） |
|------|----------|----------|
| 工作层次 | 网络层/传输层（IP + TCP/UDP 端口） | 应用层（HTTP/HTTPS/gRPC 内容） |
| 路由依据 | 源/目标 IP + 端口 | URL 路径、Header、Cookie、Body |
| 处理开销 | 极低（内核直接处理，如 LVS） | 较高（需要解析应用层协议） |
| 功能 | 简单转发，对应用透明 | 精细路由、SSL 卸载、内容改写、限流 |
| 代表 | LVS、F5 硬件 | Nginx、HAProxy、Spring Cloud Gateway |
| 适用 | 高流量、低延迟、纯 TCP 转发 | HTTP API、微服务网关、精细流量控制 |

**实际架构**：通常 L4（LVS）→ L7（Nginx）→ 后端服务，形成两层负载均衡。LVS 承担大流量入口，Nginx 做精细路由和 SSL 卸载。

###### 10. Ribbon 客户端负载均衡的原理是什么？
Ribbon 是 Spring Cloud 中的客户端负载均衡组件，集成在服务消费者内部：

**工作原理**：
1. `ServerList` 接口从服务注册中心（Eureka/Nacos）拉取并缓存可用实例列表（定期刷新）。
2. `IPing` 接口定期心跳检测实例健康状态，剔除不健康实例。
3. `IRule` 接口根据策略选择目标实例（轮询/随机/加权响应时间/区域感知等）。
4. 发起真实 HTTP 请求到选中的实例。

**内置策略**：
- `RoundRobinRule`：轮询（默认）。
- `RandomRule`：随机。
- `WeightedResponseTimeRule`：根据平均响应时间计算权重，响应快的权重大。
- `ZoneAwareLoadBalancer`：区域感知，优先同区域实例，避免跨区域延迟。

> 注意：Spring Cloud 2020 以后 Ribbon 已被弃用，推荐使用 Spring Cloud LoadBalancer，与 Ribbon 逻辑类似但更轻量。
