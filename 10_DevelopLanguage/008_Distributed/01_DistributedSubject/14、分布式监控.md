###### 1. 分布式系统监控的指标有哪些？

一个完善的分布式监控体系需要覆盖从基础设施到业务逻辑的多个层面，核心遵循**可观测性三剑客：Metrics（指标）+ Logs（日志）+ Traces（链路追踪）**。

**1. 系统资源指标**
- **CPU**：使用率、负载（Load Average）、上下文切换次数
- **内存**：使用量、剩余量、Swap 使用情况、GC 相关（JVM 场景）
- **磁盘**：I/O 吞吐量、读写延迟、使用率、inode 使用情况
- **网络**：带宽使用率、TCP 连接数（ESTABLISHED/TIME_WAIT）、丢包率、重传率

**2. 应用运行时指标（以 JVM 为例）**
- **堆内存**：Eden、Survivor、Old Gen 的使用情况
- **垃圾回收**：Young GC 次数/耗时、Full GC 次数/耗时（Full GC 频繁是性能瓶颈信号）
- **线程**：活跃线程数、阻塞线程数、死锁检测
- **类加载**：已加载类数量、卸载类数量

**3. 微服务与中间件指标（黄金四指标：延迟/流量/错误/饱和度）**
- **流量**：QPS（每秒请求数）、TPS（每秒事务数）
- **延迟**：平均响应时间、P95/P99 分位值、最大响应时间（平均值会掩盖毛刺，**P99 更能反映真实体感**）
- **错误率**：HTTP 5xx 比例、异常抛出次数、超时次数
- **饱和度**：数据库连接池使用率、线程池队列深度、Redis 内存使用率、消息队列堆积数

**4. 业务指标（最贴近用户价值）**
- 订单创建成功率、支付成功率、登录成功率
- 每秒订单数、新增用户数、DAU/MAU

**5. 分布式系统特有指标**
- 副本同步延迟（MySQL 主从延迟、Redis 主从偏移量）
- ZooKeeper/etcd 请求延迟、Watch 数量
- Elasticsearch 分片状态、Kafka 分区 Leader 切换次数、消费者 Lag

> **面试口诀**：SRE 黄金四指标 = **延迟（Latency）+ 流量（Traffic）+ 错误（Errors）+ 饱和度（Saturation）**，简称 L.T.E.S.

###### 2. Prometheus 的工作原理是什么？

Prometheus 是基于**拉取模型（Pull-based）**的开源监控系统，与大多数推送模型的监控系统不同。

**数据模型**

所有数据存储为**时间序列**（带时间戳的数值流），每个时间序列由**指标名称 + 标签（key=value）**唯一标识：

```
http_requests_total{method="POST", handler="/api/orders", status="200"}
```

标签是 Prometheus 强大聚合能力的基础，可以随意组合做 `sum()/rate()/avg()` 等聚合计算。

**核心工作流程**

```
Target（暴露 /metrics）← Scrape（拉取） ← Prometheus Server
                                          ↓ 存储 TSDB
告警规则 → Alertmanager → 多渠道通知
Grafana ← PromQL 查询 ← Prometheus Server
```

1. **目标发现与抓取**：Prometheus Server 按 `scrape_interval`（默认 15s）周期性向各个 Target 的 `/metrics` 端点发 HTTP 请求，拉取指标数据。Target 列表通过静态配置或动态服务发现（Kubernetes、Consul）获取。

2. **指标暴露**：被监控的应用需运行 HTTP 服务暴露指标。通常通过客户端库（`prometheus-client-java`）集成，或通过 Exporter（如 `node_exporter` 采集系统指标）实现。

3. **存储（TSDB）**：样本先写入内存（headBlock），定期以块的形式持久化磁盘。采用自定义压缩格式，针对时间范围查询进行了大量优化。

4. **查询（PromQL）**：强大的查询语言，支持实时聚合和多维度分析。

5. **告警（Alertmanager）**：根据 `alerting rules` 触发告警，Alertmanager 负责**去重**（防重复通知）、**分组**（相关告警合并）、**抑制**（高级别触发时压制低级别）、**静默**，并通过邮件、钉钉、Webhook 等路由发送。

**推模型 vs 拉模型**

| | 拉模型（Prometheus） | 推模型（InfluxDB / StatsD） |
|---|---|---|
| 控制权 | 监控中心控制 | 被监控方控制 |
| 发现 | 需要服务发现 | 只要知道推送地址即可 |
| 短周期任务 | 不适合（任务结束时来不及抓取） | 适合 |
| 防火墙穿透 | 拉取需要网络可达 | 推送更容易穿越防火墙 |

> 短期批处理任务可通过 **Pushgateway** 将指标推送到网关，再由 Prometheus 从网关拉取。

###### 3. Grafana 是如何与 Prometheus 配合使用的？

经典组合：**Prometheus 负责数据的采集与存储，Grafana 负责可视化展示与告警**，两者解耦、各司其职。

**配合流程**

1. **数据源配置**：在 Grafana UI 中将 Prometheus Server 地址（如 `http://prometheus:9090`）添加为数据源。
2. **创建仪表盘**：仪表盘由多个**面板**组成，每个面板对应一个 PromQL 查询。
3. **面板查询示例**：

```promql
# 所有实例的 CPU 使用率（5分钟均值）
100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# 接口 P99 响应时间
histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service))
```

4. **可视化渲染**：Grafana 将时序数据渲染为图表、仪表、热图等，支持时间范围选择、自动刷新和模板变量（变量可以动态切换查看不同实例或服务）。
5. **告警集成**：推荐在 Prometheus 端定义告警规则（`alerting_rules.yml`），由 Alertmanager 发送通知。Grafana 可将 Alertmanager 配置为数据源，集中展示所有告警状态。

**实际落地经验**：用好 Grafana 的关键是建立分层仪表盘——从大盘（集群概览）→ 服务大盘 → 实例大盘 → 接口详情，层层下钻定位问题。

###### 4. 什么是 APM（应用性能监控）？

APM 是对应用程序性能、可用性、用户体验进行监控、管理和诊断的实践与技术体系。它超越了基础资源监控（CPU、内存），深入到**应用代码内部**，回答的是"为什么应用慢了"或"错误发生在哪里"这类更深层的问题。

**APM 三大支柱**

| 支柱 | 说明 |
|---|---|
| **指标（Metrics）** | 量化性能的数值：TPS、P99 响应时间、GC 次数、连接池使用率等，是判断系统健康的"体温计" |
| **链路追踪（Traces）** | 记录端到端请求在分布式系统中的完整路径，通过 TraceId/SpanId 还原调用拓扑，精确定位慢调用或故障点 |
| **日志（Logs）** | 运行时产生的文本事件记录。现代 APM 强调**日志与链路的关联**：将 TraceId 注入日志（MDC），可以从错误日志一键跳转到对应请求链路 |

**APM 与基础监控的区别**

- **基础监控**：关注"机器是否活着"、"资源是否够用"（CPU 高了）
- **APM**：关注"业务交易是否成功"、"代码哪里慢了"（因为订单服务某个 DAO 方法有慢 SQL，导致 CPU 高）

**主流工具**：SkyWalking、Pinpoint（无侵入式），以及商业版的 New Relic、AppDynamics、Dynatrace。

###### 5. 如何设计一个分布式监控系统？

生产级分布式监控系统需要平衡功能性、性能、可靠性和可维护性，核心架构分五层：

**1. 数据采集层**
- **代理模式**：在主机或容器内部署轻量级 Agent（`Prometheus Node Exporter`、`Telegraf`），采集系统和自定义指标；
- **无侵入探针**：Java Agent（如 SkyWalking）字节码增强，自动采集 JVM 指标和调用链；
- **日志收集器**：Filebeat / Fluentd 从日志文件采集、解析、结构化日志；
- **关键设计**：采集端尽量轻量，支持批量压缩上报，具备本地缓存和断点续传能力。

**2. 数据传输与聚合层**
- 指标常用**拉模型**（Prometheus），超大规模时可通过消息队列（Kafka）做削峰缓冲；
- 高基数数据（如全量访问日志）需在传输前**预聚合或采样**，降低后端存储压力。

**3. 数据存储层**

| 数据类型 | 推荐存储 | 特点 |
|---|---|---|
| 时序指标 | Prometheus TSDB / VictoriaMetrics | 时间范围查询、压缩优化 |
| 链路追踪 | Elasticsearch / Jaeger 后端 | 支持全文搜索和复杂过滤 |
| 日志 | Elasticsearch（ELK）/ Loki | ES 功能强大，Loki 索引少成本低 |

- 根据数据热度设计分层存储：热数据在本地 SSD，冷数据归档到对象存储（S3/COS）。

**4. 计算分析与查询层**
- **实时计算**：对流入的指标流做实时聚合、异常检测；
- **统一查询**：提供跨数据源关联查询（通过 TraceId 同时查询链路和日志），Grafana 可关联多数据源实现联动。

**5. 可视化与告警层**
- **告警规则**：在 Prometheus 定义基于指标的规则；
- **告警处理**：Alertmanager 做**去重、分组、抑制、静默**；
- **多渠道通知**：邮件、钉钉、企业微信、Webhook；
- **自愈联动**：高级场景中，告警可自动触发修复动作（重启 Pod、扩容节点）。

###### 6. 什么是 ELK Stack？与 Loki 有什么区别？

**ELK Stack**

ELK 是 Elasticsearch + Logstash + Kibana 的组合，是最成熟的日志管理方案：

```
日志文件 → Filebeat（采集）→ Logstash（解析/过滤/转换）→ Elasticsearch（存储/索引）→ Kibana（可视化）
```

- **Elasticsearch**：全文索引每条日志的所有字段，支持复杂的关键词搜索和聚合分析；
- **Logstash**：支持丰富的过滤器（grok 解析非结构化日志），但资源消耗较重，轻量场景可用 Filebeat 直接写 ES；
- **Kibana**：日志搜索、仪表盘、告警一体化 UI。

**Loki（Grafana Labs 出品）**

Loki 的设计哲学是"**像 Prometheus 一样处理日志**"：

- **只对日志的元数据（标签）建索引**，不对日志内容建全文索引，存储成本比 ELK 低 5~10 倍；
- 日志内容以压缩块形式存储（类似 Prometheus 时序块）；
- 通过 **LogQL** 查询，语法与 PromQL 相似，学习成本低；
- 与 Grafana 原生集成，指标+日志+链路可在同一界面联动查询。

**两者对比**

| | ELK | Loki |
|---|---|---|
| **索引方式** | 全文索引，查询强大 | 仅索引标签，查询受限（无法全文搜索） |
| **存储成本** | 高（全量倒排索引） | 低（标签索引 + 压缩存储） |
| **查询性能** | 复杂查询快 | 日志量大时扫描慢 |
| **运维复杂度** | 较高（ES 集群调优难） | 较低（可用对象存储） |
| **适用场景** | 需要复杂日志分析、全文搜索 | 云原生场景、成本敏感、配合 Grafana |

**选型建议**：需要强大日志分析和搜索能力 → **ELK**；追求低成本、与 Prometheus+Grafana 统一观测 → **Loki**。

###### 7. 如何设计告警系统避免告警风暴？

**告警风暴的成因**

当某个核心依赖（如数据库、网络）出现故障时，大量上层服务同时触发告警，运维人员被淹没在告警洪流中，反而难以快速定位根因。

**防治策略**

**1. 告警分组（Grouping）**：将相同来源或相同类型的告警合并为一条通知。例如，同一集群的 20 个节点同时 CPU 高，合并为一条"集群 X 中 20 个节点 CPU 使用率 >90%"的告警，而不是发 20 条独立通知。Alertmanager 的 `group_by` 配置实现此功能。

**2. 告警抑制（Inhibition）**：高级别告警触发时，自动压制关联的低级别告警。例如，某机器宕机（高级别），则自动抑制该机器上所有服务的响应时间告警（低级别）。Alertmanager 的 `inhibit_rules` 配置实现此功能。

**3. 分级告警（Severity）**：告警分为 Critical / Warning / Info 三级，不同级别走不同通知渠道：
- Critical → 电话/短信，要求 5 分钟内响应；
- Warning → 钉钉群，需 30 分钟内处理；
- Info → 仅记录，次日巡检。

**4. 告警降噪（Pending → Firing）**：设置 `for` 参数，要求条件持续满足一定时间才真正触发告警，避免偶发毛刺触发大量误报。

```yaml
# Prometheus 告警规则示例
- alert: HighErrorRate
  expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05
  for: 2m         # 条件持续 2 分钟才触发，过滤偶发毛刺
  labels:
    severity: critical
  annotations:
    summary: "服务 {{ $labels.service }} 错误率超过 5%"
    description: "当前错误率: {{ $value | humanizePercentage }}"
```

**5. 告警收敛（静默+值班）**：维护窗口期间设置告警静默（Silence）；建立值班制度，确保每条 Critical 告警都有人负责跟进，避免告警成为"狼来了"。
