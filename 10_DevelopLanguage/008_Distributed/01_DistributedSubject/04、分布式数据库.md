###### 1. 什么是分库分表？
分库分表是解决单数据库性能瓶颈的分布式数据存储方案。**分库**将数据分散到多个数据库，**分表**将单个表拆分为多个结构相同的子表。核心目标是通过水平或垂直拆分，解决数据量过大和高并发导致的性能瓶颈。

**实践策略**：通常先垂直拆分（按业务模块解耦），再水平拆分（按数据量扩展）。垂直拆分是架构设计阶段做的事，水平拆分是数据量到极限后的扩展手段。

###### 2. 垂直分库和水平分库的区别是什么？
- **垂直分库**：按业务模块划分，将不同业务的表放到不同数据库（如用户库、订单库、商品库）。不同库的表结构不同，核心目标是**业务解耦**。挑战：跨库 JOIN 复杂，需通过接口替代 SQL 关联。
- **水平分库**：按数据行分布规则划分，同一业务的数据按规则（如用户 ID 取模）拆分到多个数据库。不同库的表结构完全一致，核心目标是**扩展容量、提升并发**。挑战：跨库事务、全局排序分页困难，数据迁移扩容成本高。

两者常组合使用：先垂直分库按业务隔离，再对热点库进行水平分库。

###### 3. 垂直分表和水平分表的区别是什么？
- **垂直分表**：按字段（列）拆分，将宽表拆为核心字段表 + 扩展字段表（如商品基础信息 + 商品详情）。适合表字段多、含大字段（TEXT/BLOB）或冷热字段明显的场景。好处是减少单行数据宽度，提升单页数据行数，减少 IO。
- **水平分表**：按数据行拆分，将大表数据分散到多个结构相同的子表（如 order_0、order_1...order_N）。适合单表数据量超过千万的场景。挑战：跨表查询需合并结果，分页与排序复杂。

**口诀**：垂直是"瘦身"（减少列），水平是"分身"（减少行）。

###### 4. 分库分表后如何进行分页查询？
分页查询在分片环境下是典型难题，核心问题是数据分散在多个分片，无法直接 `LIMIT offset, count`。

**三种方案**：
- **全局视野法**：每个分片返回前 `(page * size)` 条数据，在内存中汇总排序后截取当前页。准确但资源消耗随页码线性增长，大页码时性能很差。
- **二次查询法（优化）**：第一次查各分片当前页数据边界值，第二次精确查询修正偏移。减少数据传输，但需两次查询。
- **游标/书签法**：记录上一页最后一条记录的排序字段值（如 id > 最后一条 id），下次查询用 WHERE 条件替代 OFFSET，性能最优，但只支持顺序翻页，不支持跳页。

**ShardingSphere 处理方式**：将 `LIMIT 10000, 20` 改写为各分片 `LIMIT 0, 10020`，内存归并后截取结果。

###### 5. 分库分表后如何进行排序？
排序的核心挑战是**全局有序**——数据分散在多个分片，每个分片只能保证分片内有序。

- **排序字段为分片键**：若排序与分片规则一致（如按时间分片并按时间排序），可直接路由到特定分片，效率最高。
- **排序字段非分片键**：需从各分片拉取数据，在内存中归并排序。ShardingSphere 的 `OrderByStreamMergedResult` 用最小堆多路归并，流式返回，内存占用可控。
- **性能优化**：避免 `SELECT *`；若排序需求频繁，考虑将排序字段设为分片键；或将搜索/排序需求迁移到 Elasticsearch。

###### 6. 什么是数据库中间件？常见的有哪些？
数据库中间件是应用与数据库之间的抽象层，让开发者像操作单库单表一样操作分布式数据库集群。两大类型：

- **JDBC 驱动层（客户端）**：以 JAR 包嵌入应用，对 JDBC 操作透明拦截。代表：ShardingSphere-JDBC、TDDL。优点：性能损耗小（<7%）；缺点：仅支持 Java。
- **Proxy 代理层**：独立进程部署，应用连接代理，代理转发请求。代表：MyCat、ShardingSphere-Proxy。优点：语言无关；缺点：增加网络跳转，有性能损耗。

**选型原则**：Java 技术栈且追求性能选 JDBC 驱动层；多语言异构环境选 Proxy 层。

###### 7. ShardingSphere 的核心功能是什么？
ShardingSphere 最核心的是**数据分片**，工作流程经典的五步走：

1. **SQL 解析**：ANTLR 解析 SQL 生成抽象语法树（AST），提取表名、分片键条件等。
2. **SQL 路由**：根据分片策略和分片键值，确定目标数据节点（支持精确路由、范围路由、广播路由）。
3. **SQL 改写**：将逻辑表名改写为物理表名（如 `order` → `order_0`），补充排序字段等。
4. **SQL 执行**：通过 ExecutorEngine 并行执行，支持连接限制模式（OLTP）和内存限制模式（OLAP）。
5. **结果归并**：流式归并（OrderByStreamMergedResult）或内存归并（MemoryMergedResult），对多分片结果做聚合、排序、分页。

此外还支持：读写分离（自动路由 DML→主库 / DQL→从库）、分布式事务（XA/Seata AT）、分布式主键（内置雪花算法）。

###### 8. 分库分表后如何解决跨库事务问题？
跨库事务本质是分布式事务问题，详见 [[08、分布式事务]]。两大方向：

- **强一致性（刚性事务，XA）**：基于两阶段提交，事务管理器协调多个数据库。一致性强但同步阻塞，性能低，不适合高并发。ShardingSphere 集成 Atomikos 实现。
- **最终一致性（柔性事务）**：
  - **TCC**：业务层实现 Try/Confirm/Cancel 三接口，侵入性强但性能好。
  - **Saga**：长事务拆解为多个本地事务，失败时逆序补偿。适合业务流程长的场景。
  - **本地消息表**：业务 DB 和消息表同一事务写入，定时任务投递消息，消费方幂等处理。

**最佳实践**：强资金相关用 TCC；大多数业务接受最终一致性，本地消息表 + 消息队列是最通用的方案。

###### 9. 如何平滑地进行数据库迁移？
平滑迁移核心原则是**避免停机**和**保证数据一致性**，标准方案是**双写迁移**：

1. **全量迁移**：将旧库历史数据同步到新库（DataX、Canal 等工具），记录同步截止点（Binlog 位点/GTID）。
2. **增量双写**：开启双写，应用同时向旧库和新库写入，通过数据对比工具校验一致性，修复差异。
3. **灰度切读**：先将部分非核心业务读请求切到新库，验证稳定性后逐步扩大比例。
4. **停旧库写**：确认新库完全一致且稳定后，停止旧库写入，迁移完成。

**注意事项**：双写需处理写失败回滚；迁移过程需可逆，准备好一键回切方案；切流量时做好监控告警。

###### 10. 什么是读写分离？如何实现？
读写分离将写操作（INSERT/UPDATE/DELETE）指向主库，读操作（SELECT）指向从库，通过主从复制同步数据，核心价值是**分摊主库压力，提升读吞吐量**。

**实现方式**：
- **应用层**：Spring `AbstractRoutingDataSource` 根据方法注解或 SQL 类型动态切换数据源。
- **中间件层**：ShardingSphere-JDBC 自动路由；MyCat Proxy 层转发。
- **数据库驱动层**：MySQL Connector/J 的 ReplicationDriver，根据读写操作类型路由。

###### 11. 读写分离可能带来哪些问题？
- **主从延迟**：主从异步复制导致从库数据滞后，用户可能读不到刚写的数据（如下单后立即查询订单）。解决：对一致性要求高的读（如订单详情）强制走主库；或用半同步复制牺牲部分写性能。
- **负载不均**：读远大于写时，从库可能成为瓶颈。解决：增加从库数量，读负载均衡。
- **从库故障**：影响读可用性。解决：读负载均衡需具备健康检查和自动剔除能力。

###### 12. 什么是分布式数据库？
分布式数据库是数据**物理分散**在多个节点，但对外提供**逻辑统一**访问的数据库系统。核心特性：数据分片存储、透明访问（用户无需关心物理位置）、高可用副本容错。

分类：
- **分库分表 + 中间件**：MySQL + ShardingSphere，成本低，运维复杂。
- **NewSQL**：TiDB、CockroachDB，原生分布式 + ACID 事务，见下题。
- **云原生分布式数据库**：AWS Aurora、PolarDB，计算与存储分离，共享存储池。

###### 13. NewSQL 数据库是什么？
NewSQL 是兼具 **NoSQL 扩展性**和**传统关系型数据库 ACID 特性**的现代分布式数据库，目标是"鱼与熊掌兼得"。

**核心特性**：原生水平扩展 + 强一致性（Raft/Paxos 多副本）+ 完整 SQL 兼容 + 高可用自动切换。

**TiDB 架构**（典型代表）：
- **TiDB Server**：无状态 SQL 计算层，处理连接和 SQL 解析/优化。
- **PD（Placement Driver）**：元数据管理中心 + 调度中心，负责 Region 分布和热点均衡。
- **TiKV**：分布式 KV 存储引擎，数据按 Region（默认 96MB）分片，每个 Region 通过 **Raft** 协议保证一致性。

**适用场景**：需要兼顾强一致性和水平扩展的业务，如金融核心系统从 Oracle 迁移到国产分布式数据库。

###### 14. 分库分表如何生成全局唯一 ID？
分库分表后各库各表自增 ID 会冲突，必须使用分布式 ID 生成方案。常见五种：

1. **UUID**：本地生成，无序，索引效率差，UUID v4 完全随机，生产环境不推荐作为主键。
2. **数据库号段**：批量从一张号段表申请 ID 范围（如一次取 1000 个），本地自增使用，DB 压力小但有单点风险。美团 Leaf-Segment 是成熟实现。
3. **Redis INCR**：原子递增，性能高，但 Redis 需持久化，宕机可能有 ID 空洞。
4. **雪花算法（Snowflake）**：64 位 Long = 1位符号 + 41位时间戳(ms) + 5位数据中心 + 5位机器ID + 12位序列号，每毫秒可产生 4096 个 ID，趋势递增，性能极高。主要问题是**时钟回拨**（机器时钟向前回拨时，可能产生重复 ID）。解决方案：回拨时等待（回拨 < 5ms）或抛出异常。
5. **美团 Leaf/百度 UidGenerator**：雪花算法的生产级改进，解决了机器 ID 注册和时钟回拨问题。
