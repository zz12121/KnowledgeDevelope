###### 1. 什么是容器化技术？

容器化技术是一种**操作系统级虚拟化**技术，将应用程序及其所有依赖项（库、环境变量、配置文件）打包成一个标准化、轻量级、可移植的单元（**容器**）。容器共享宿主机操作系统内核，但运行在彼此隔离的用户空间中。

**两大核心技术基石**
- **命名空间（Namespace）**：为进程提供独立的系统视图，实现进程（PID）、网络（NET）、文件系统（MNT）、IPC 等资源的隔离；
- **控制组（cgroups）**：限制进程组的 CPU、内存等物理资源使用，防止某个容器耗尽宿主机资源；
- **联合文件系统（OverlayFS）**：将多个只读层和一个可写层叠加，形成容器的根文件系统，镜像层可在多个容器间共享，极大节省存储空间。

**核心价值**
- **环境一致性**：消除"在我机器上能跑"的困境
- **轻量高效**：秒级启动，资源开销极小（相比 VM 的 GB 级 OS 开销，容器镜像通常 MB 级）
- **可移植性**：标准化镜像格式，运行在任何支持容器运行时的环境

###### 2. Docker 和虚拟机的区别是什么？

两者都提供隔离的运行环境，但实现原理和资源开销有根本差异。

| 维度 | 虚拟机 | Docker 容器 |
|---|---|---|
| **虚拟化层级** | **硬件级**：Hypervisor 模拟完整虚拟硬件 + 独立 OS | **OS 级**：共享宿主机内核，Namespace + cgroups 隔离 |
| **启动速度** | 分钟级（需启动完整 OS） | 秒级/毫秒级（直接启动进程） |
| **性能开销** | 高（每个 VM 独立 OS，大量 CPU/内存占用） | 极低（与宿主机进程共享内核） |
| **隔离性** | 强（完全 OS 隔离，安全性更高） | 较弱（共享内核，存在潜在安全风险） |
| **镜像大小** | GB 级（含完整 OS） | MB 级（仅含应用和依赖） |
| **代表技术** | VMware / KVM / VirtualBox | Docker / containerd / Podman |

**原理层面**：
- VM：Hypervisor 截获并翻译客户机特权指令，产生上下文切换开销；
- Docker：`docker run` → dockerd → containerd → runc（OCI 标准实现）→ 调用 `clone()` 系统调用创建带 Namespace 的进程 → cgroups 设置资源限制 → OverlayFS 挂载文件系统。

###### 3. 什么是 Kubernetes？

Kubernetes 是开源的**容器编排引擎**，用于自动化容器化应用的部署、扩展和管理。源于 Google 的 Borg 系统，现由 CNCF 托管，是云原生领域的基石。

**核心定位**：提供**声明式**的基础设施管理（通过 YAML 描述"期望状态"），K8s 的控制器持续工作确保实际状态与期望状态一致，自动处理节点故障、容器重启、负载均衡等复杂操作。

**核心价值**：服务发现与负载均衡 / 存储编排 / 自动部署与回滚 / 自我修复 / 密钥与配置管理 / 弹性伸缩

简单说，Kubernetes 是管理大规模容器化应用的"操作系统"。

###### 4. Kubernetes 的核心组件有哪些？

K8s 集群由**控制平面（大脑）**和**工作节点（肢体）**两部分组成。

**控制平面组件**

| 组件 | 职责 |
|---|---|
| **kube-apiserver** | 集群唯一入口，所有资源操作的 RESTful API 网关，是其他组件交互的中枢 |
| **etcd** | 高可用 KV 存储，保存整个集群的所有配置和状态数据，是集群的"唯一真相来源" |
| **kube-scheduler** | 调度器，为新 Pod 选择合适节点运行（基于资源需求、亲和性、污点等策略） |
| **kube-controller-manager** | 运行各种控制器（节点控制器、副本控制器、端点控制器），持续驱动集群向期望状态收敛 |

**工作节点组件**

| 组件 | 职责 |
|---|---|
| **kubelet** | 节点代理，与容器运行时交互，确保 Pod 中的容器健康运行，向 apiserver 汇报状态 |
| **kube-proxy** | 网络代理，通过 iptables/ipvs 规则实现 Service 虚拟 IP 到后端 Pod 的流量转发 |
| **容器运行时** | 实际运行容器的软件（containerd / CRI-O） |

**一次完整的部署流程**：
```
kubectl apply deployment.yaml
→ apiserver 存入 etcd
→ Deployment Controller 创建 ReplicaSet
→ ReplicaSet Controller 创建 Pod（未调度状态）
→ Scheduler 选择节点，更新绑定信息到 apiserver
→ 目标节点 kubelet 监听到属于自己的 Pod
→ kubelet 调用容器运行时拉镜像、启容器
→ kube-proxy 更新 iptables 转发规则
```

###### 5. K8s 的健康检查探针有哪几种？有什么区别？

K8s 提供三种探针，用于检测容器的健康状态和就绪状态：

| 探针 | 作用 | 失败时的行为 |
|---|---|---|
| **livenessProbe（存活探针）** | 检测容器是否存活（是否需要重启）| 重启容器 |
| **readinessProbe（就绪探针）** | 检测容器是否准备好接受流量 | 将 Pod 从 Service Endpoints 中摘除，不再路由流量，但不重启 |
| **startupProbe（启动探针）** | 检测容器是否已经启动完成（专为慢启动应用设计） | 未通过前，liveness/readiness 探针不会执行；超出最大失败次数则重启 |

**三者的关系**：startupProbe 先运行，通过后 liveness 和 readiness 才开始生效。

**探针检测方式**

```yaml
livenessProbe:
  httpGet:              # HTTP GET 检查
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 30   # 容器启动后等待多久开始探测
  periodSeconds: 10          # 探测间隔
  failureThreshold: 3        # 连续失败多少次才判定为失败

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
```

**实践要点**：
- Spring Boot 2.3+ 的 Actuator 已内置 `/health/liveness` 和 `/health/readiness` 端点，直接使用；
- `readinessProbe` 对于**优雅上线**至关重要：只有当应用真正准备好（如数据库连接池初始化完成、缓存预热完成）才向 readiness 端点返回 UP，避免上线初期大量报错；
- `livenessProbe` 不宜设置得太激进（`failureThreshold` 不要太小），否则应用偶发 GC 停顿就会被误杀重启。

###### 6. K8s 滚动更新、蓝绿部署、金丝雀发布有什么区别？

三种策略的核心差异在于**新旧版本共存的方式**和**流量切换的粒度**。

| 策略 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **滚动更新（Rolling Update）** | 逐步用新 Pod 替换旧 Pod，一边上线新版，一边下线旧版，新旧同时在线 | 简单，不需要额外资源，无需停机 | 新旧版本同时提供服务，可能引发兼容性问题；回滚速度相对较慢 |
| **蓝绿部署（Blue-Green）** | 同时运行两套完整的环境（蓝=旧，绿=新），验证通过后将 Service 的流量瞬间从蓝切到绿 | 切换瞬间完成，回滚只需把流量切回蓝；新旧版本完全隔离 | 需要双倍资源；切换后若发现问题，已有用户受影响 |
| **金丝雀发布（Canary）** | 将少量流量（如 5%）导入新版本，观察稳定后逐步扩大比例，直到全量替换 | 风险最小，问题只影响小部分用户；可精细控制流量百分比 | 管理复杂，需要流量分割能力（如 Istio VirtualService 或 Argo Rollouts） |

**Kubernetes 中的实现**

```yaml
# 滚动更新配置（Deployment 默认策略）
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # 允许超出期望副本数的最大数量
      maxUnavailable: 0    # 更新过程中允许不可用的最大 Pod 数（0 表示始终保持全量在线）
```

```yaml
# 蓝绿部署：通过切换 Service 的 selector 实现
# 绿环境部署完后，修改 Service selector: version: green
```

```yaml
# 金丝雀发布（Istio 方式）
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
spec:
  http:
  - route:
    - destination:
        host: my-service
        subset: stable
      weight: 90   # 90% 流量到旧版本
    - destination:
        host: my-service
        subset: canary
      weight: 10   # 10% 流量到新版本
```

**选型建议**：日常迭代用**滚动更新**；重大变更用**蓝绿**（需要快速回滚保障）；不确定稳定性的新功能用**金丝雀**。

###### 7. 什么是 Service Mesh？

Service Mesh 是**专门处理服务间通信的基础设施层**。它以轻量级网络代理（Sidecar）的形式与每个服务实例协同部署，所有服务间流量都流经这个代理，由代理实现流量控制、安全、可观测性等功能，而**无需在应用代码中实现**。

**核心思想**：将网络治理能力从业务代码中解耦，下沉到基础设施层。

| 层次 | 组件 | 职责 |
|---|---|---|
| **数据平面** | Envoy / Linkerd（Sidecar 代理） | 直接处理所有入站/出站流量，执行路由、负载均衡、熔断、mTLS 等 |
| **控制平面** | Istio Pilot / Citadel / Galley | 管理所有代理的配置，下发策略，收集遥测数据，不直接处理数据包 |

**解决的问题**：流量管理（金丝雀/故障注入）、弹性（熔断/重试/超时）、安全（mTLS/访问控制）、可观测性（指标/链路追踪/日志）。

**代表产品**：Istio（数据平面用 Envoy）、Linkerd（CNCF 毕业项目，更轻量）。

###### 8. 什么是 CI/CD？

| 概念 | 说明 |
|---|---|
| **持续集成（CI）** | 开发者频繁将代码集成到主干，每次集成都通过自动化构建和测试验证，核心目标是**快速反馈**，避免"集成地狱" |
| **持续交付（CD - Delivery）** | CI 的延伸，确保软件随时可以手动部署到生产环境，最终发布决策由人工触发 |
| **持续部署（CD - Deployment）** | 通过所有自动化测试后，代码变更**自动**部署到生产环境，无需人工干预，是最高程度的自动化 |

**CI/CD 流水线典型阶段**：

```
代码提交 → 构建（编译+单测） → 静态扫描 → 镜像构建&推送 
→ 部署到测试环境 → 集成/API测试 → 部署到预发 → 验收测试 
→ 部署到生产（CD Delivery 手动 / CD Deployment 自动）
```

**工具链**：Jenkins / GitLab CI / GitHub Actions（CI）；Argo CD / Flux（GitOps 风格 CD）；Helm / Kustomize（K8s 配置管理）。

###### 9. 如何实现应用的自动化部署？

现代云原生应用最佳实践是 **GitOps + Kubernetes**：

**核心思路：Git 仓库是唯一真相来源**

1. 开发者提交代码，触发 CI 流水线；
2. CI 完成构建和测试，生成新 Docker 镜像（如 `myapp:v1.3`）；
3. CI 自动更新 **Git 配置仓库**中对应环境的镜像标签（这是 GitOps 的关键：基础设施变更也走 Git）；
4. **Argo CD** 持续监控 Git 配置仓库，检测到配置与集群实际状态不一致后，自动（或手动确认后）同步到 K8s，触发滚动更新。

```
应用代码仓库（Dev） → CI（构建/测试/推镜像） → 配置仓库（Ops）
                                                        ↓ 监听变更
                                                   Argo CD → K8s 集群
```

**部署策略选择**（在 K8s 中）：
- 日常功能迭代 → **滚动更新**（默认）
- 重大架构变更 → **蓝绿部署**（快速回滚保障）
- 不确定稳定性的功能 → **金丝雀发布**（小流量验证）

**安全与审计**：所有变更通过 Git 提交记录审计；部署到生产前可在流水线中设置**人工审批关卡**；集成 Trivy 等工具做镜像安全扫描。
