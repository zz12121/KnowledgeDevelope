###### 1. 如何设计一个高可用的分布式系统？

高可用的核心目标是确保系统在面对各类故障时仍能持续提供服务。设计遵循**"冗余 + 解耦 + 自愈 + 容错"**四大原则。

**1. 消除单点故障**
- **无状态服务多副本**：通过负载均衡器将流量分发到多个相同实例，任一实例故障，流量自动导向健康实例；K8s 中 `Deployment` 通过 `ReplicaSet` 确保副本数
- **有状态服务集群化**：数据库（主从复制 + MHA 自动切换）、缓存（Redis Sentinel/Cluster）、消息队列（Kafka 分区多副本 + ISR 机制）

**2. 快速故障转移**
- **健康检查**：负载均衡器或注册中心（Eureka/Nacos）定期探测，及时剔除不健康节点；K8s 的 `livenessProbe` / `readinessProbe` 精确控制 Pod 生命周期
- **服务发现**：服务实例通过心跳维持注册，下线时自动注销；客户端本地缓存注册表，服务中心短暂不可用时仍能正常调用

**3. 应用层容错**
- **熔断 + 降级**：Sentinel / Hystrix，依赖服务失败率达阈值后熔断，执行降级逻辑（返回缓存数据或默认值），防止雪崩
- **重试 + 退避**：失败时自动重试其他实例，配合指数退避 + 抖动（Jitter），避免重试风暴

**4. 数据冗余与备份**
- 多副本策略（MySQL 主从、Redis AOF+RDB、HDFS 三副本）
- 定期全量 + 增量备份，并演练恢复流程

**5. 架构解耦**
- 消息队列解耦（Kafka/RocketMQ）：生产者不直接依赖消费者，消费者暂时不可用不影响主流程
- 最终一致性：跨服务数据通过异步消息同步，避免分布式事务带来的复杂性和性能瓶颈

> **面试口诀**：高可用 = 冗余（消灭单点）+ 解耦（消息队列）+ 容错（熔断降级）+ 快恢复（健康检查+故障转移）

###### 2. 如何设计一个可扩展的分布式系统？

可扩展性指系统能通过增加资源平滑提升处理能力，分布式系统的目标是**水平扩展（Scale-out）**。

**AKF 扩展立方（指导性模型）**
- **X 轴（水平复制）**：同一服务多实例，通过负载均衡分发流量
- **Y 轴（功能拆分）**：按业务领域拆分微服务，每个服务独立扩展
- **Z 轴（数据分片）**：按用户ID/地理位置等维度分片，每个分片服务部分数据

**可扩展性关键设计**

1. **无状态服务**：请求可路由到任意实例，是水平扩展的前提。Session 存 Redis，配置用配置中心，文件用对象存储

2. **数据层分片**：数据库分库分表（ShardingSphere）、Redis Cluster（16384 哈希槽）、ES 分片

3. **弹性伸缩**：K8s HPA（Horizontal Pod Autoscaler）基于 CPU/内存/自定义指标自动扩缩容：
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70   # CPU 超 70% 就扩容
```

4. **多层缓存**：本地缓存（Caffeine）→ 分布式缓存（Redis）→ 数据库，逐层拦截读请求

5. **异步化**：非核心流程异步处理（发邮件、生成报表），消息队列削峰填谷

###### 3. 什么是服务拆分的原则？

服务拆分是微服务架构设计的核心，目标是建立**高内聚、低耦合、自治**的服务单元。

**核心原则**

1. **单一职责 + 高内聚（DDD 限界上下文）**：一个服务只负责一个明确的业务领域，相关性强、变更原因相同的功能放在同一个服务。订单和支付虽然相关，但业务上下文不同，应拆分

2. **服务自治**：
   - **独立部署**：不影响其他服务的情况下独立发布
   - **独立数据库**：服务拥有私有数据库，不直接共享 DB（共享 DB 是微服务最大的反模式）
   - **独立技术栈**：可按需选择最合适的技术（在团队能力允许下）

3. **松耦合**：服务间通过 API 或事件通信，优先异步（发布/订阅事件），避免直接同步调用链过长

4. **基于业务能力，而非技术层级**：不要按"Web层、Service层、DAO层"拆分，而是按业务领域（用户、商品、风控、营销）

5. **渐进式拆分**：不要从零开始就设计出"完美"的微服务架构，从单体出发，随业务复杂度和团队规模增长，识别高频变更、高耦合的模块逐步拆分

**拆分过细的代价**：服务数量激增会带来部署、测试、监控、分布式事务、网络延迟等方面的显著复杂度。需在**自治性**和**工程复杂度**之间取得平衡。

###### 4. 如何避免分布式系统中的单点故障？

避免 SPOF（Single Point of Failure）意味着系统中不存在任何"一旦失效则整个核心功能不可用"的组件。

**分层消除单点**

| 层次 | 方案 |
|---|---|
| **接入层** | 多 Nginx 实例 + Keepalived + VIP，或直接用云厂商 ELB/SLB（天然高可用） |
| **应用层** | 至少 2 个副本，服务注册发现（Eureka/Nacos 集群），负载均衡分发流量 |
| **数据库层** | MySQL 主从复制 + MHA 自动切换，或 MySQL MGR（Group Replication） |
| **缓存层** | Redis Sentinel（主从自动切换）或 Redis Cluster（分片 + 多副本） |
| **消息队列** | Kafka 多 Broker + 分区多副本，Leader 失效后自动选举新 Leader |
| **配置/注册中心** | Nacos/Apollo 集群部署，客户端本地缓存兜底 |
| **分布式协调** | ZooKeeper/etcd 奇数节点集群（ZAB/Raft 协议保证容忍少数节点故障） |

**设计模式层面**
- **重试 + 退避**：瞬时故障自动重试，指数退避 + 随机抖动防止重试风暴
- **熔断 + 降级**：依赖链中某环节故障，快速失败 + 降级，避免故障扩散
- **跨 AZ 部署**：关键服务跨多个可用区部署，机房级故障时流量自动切至其他可用区

###### 5. 什么是异地多活架构？

异地多活是指在多个地理区域（不同城市）部署独立的数据中心，每个区域同时处理用户请求，是最高级别的容灾架构。

**与同城双活的区别**
- **同城双活**：距离 <100km，延迟 <2ms，可共享存储，接近强一致性
- **异地多活**：跨城市，延迟几十到几百毫秒，**必须接受最终一致性**

**核心挑战：数据一致性**

跨地域延迟下，强一致性（分布式事务）性能极差，通常采用**最终一致性 + 冲突解决**：

| 同步方案 | 原理 | 特点 |
|---|---|---|
| **异步复制（主流）** | 通过 MQ 或 Binlog 变更数据异步同步到其他中心 | 有短暂延迟，但性能好 |
| **同步双写** | 写入时同步写多中心 | 延迟高，任一中心故障则写入失败 |
| **最终一致（CRDT）** | 使用 CRDT 数据结构自动合并并发修改 | 适合特定场景（如购物车商品合并） |

**用户分区策略（最常用方案）**

将用户按 ID 哈希固定路由到某个主场机房，该用户的读写都发生在主场，然后数据异步复制到其他机房。这大大减少了数据冲突概率。

**流量调度**：DNS 解析（切换慢，受 TTL 限制）、HTTP 302 重定向（更灵活）、客户端 SDK 内置路由逻辑。

**全局 ID 生成**：必须使用包含数据中心信息的分布式 ID（如雪花算法中嵌入机房 ID 位），避免多中心 ID 冲突。

###### 6. 如何设计一个秒杀系统？

秒杀的核心矛盾：**瞬间超高并发 vs 有限库存**。设计目标是**系统稳定、防超卖、保证公平**。

**设计思路：分层过滤，逐层限流**

```
前端（静态化+CDN+按钮防重）
   ↓ 过滤掉 99% 流量
网关（限流+防刷+黑名单）
   ↓ 再过滤 90%
服务层（Redis原子扣库存）
   ↓ 成功扣减的请求
消息队列（异步落库）
   ↓ 慢慢处理
数据库（创建订单+扣 DB 库存）
```

**防超卖核心：Redis Lua 脚本原子操作**

```lua
-- 秒杀扣库存 Lua 脚本（原子执行，防超卖+防重复购买）
local stock_key = KEYS[1]   -- 库存 key
local user_key = KEYS[2]    -- 已购用户集合 key
local user_id = ARGV[1]

-- 检查是否已购
if redis.call('SISMEMBER', user_key, user_id) == 1 then
    return 2  -- 已购买，不允许重复
end
-- 检查库存
local stock = tonumber(redis.call('GET', stock_key))
if not stock or stock <= 0 then
    return 0  -- 库存不足
end
-- 扣库存 + 记录已购
redis.call('DECR', stock_key)
redis.call('SADD', user_key, user_id)
return 1  -- 秒杀成功
```

**完整流程**：Redis 成功扣减 → 发消息到 RocketMQ → 订单服务异步消费消息 → 创建订单 + 扣减 DB 库存 → 通知用户。

**其他关键设计**
- **缓存预热**：活动开始前将库存加载到 Redis；
- **队列泄洪**：通过内存队列（Disruptor）或 MQ 平滑流量，控制处理速度；
- **幂等性**：同一用户的秒杀请求确保幂等（Redis SISMEMBER 检查 + 数据库唯一索引兜底）；
- **独立部署**：秒杀系统独立集群，避免影响主站正常业务。

> **一句话总结**：秒杀 = 前端拦截大量请求 + Redis 内存操作核心逻辑 + 消息队列异步落库 + 分层限流保护。

###### 7. 如何保证分布式系统接口的幂等性？

**什么是幂等性**

幂等性是指对同一个请求执行一次和执行多次，产生的结果完全相同。在分布式系统中，由于网络超时、消息重投递、用户重复点击等原因，幂等性设计至关重要。

**非幂等操作的典型场景**
- 用户点击支付后网络超时，客户端重试，导致重复扣款
- 消息队列消费者消费失败重试，导致重复处理同一条消息
- 定时任务因机器切换导致任务被执行两次

**幂等性解决方案**

**1. 全局唯一 ID + 数据库唯一索引（最通用）**

```java
// 创建订单接口，客户端传入全局唯一的 clientOrderId
@PostMapping("/orders")
public Result createOrder(@RequestBody OrderRequest request) {
    // clientOrderId 在数据库设有唯一索引
    try {
        orderService.createOrder(request.getClientOrderId(), request);
        return Result.success();
    } catch (DuplicateKeyException e) {
        // 重复请求，直接返回成功（幂等）
        return Result.success("订单已存在");
    }
}
```

**2. Redis 分布式锁 + 状态机**

```java
// 防止并发重复请求
String lockKey = "order:create:" + clientOrderId;
if (!redisLock.tryLock(lockKey, 5, TimeUnit.SECONDS)) {
    throw new RuntimeException("请求处理中，请勿重复提交");
}
try {
    // 查询是否已处理
    if (orderRepository.existsByClientOrderId(clientOrderId)) {
        return existingOrder;
    }
    return createNewOrder(request);
} finally {
    redisLock.unlock(lockKey);
}
```

**3. 消息消费幂等（消息队列场景）**

```java
@KafkaListener(topics = "order-events")
public void handleOrderEvent(OrderEvent event) {
    String idempotentKey = "mq:consumed:" + event.getMessageId();
    // 利用 Redis SETNX 做消费去重
    Boolean isFirst = redisTemplate.opsForValue()
        .setIfAbsent(idempotentKey, "1", Duration.ofDays(7));
    if (Boolean.FALSE.equals(isFirst)) {
        log.info("消息 {} 已消费，跳过", event.getMessageId());
        return;
    }
    // 业务处理
    processOrderEvent(event);
}
```

**4. 乐观锁（数据库状态机）**

```sql
-- 订单状态机转换，每次只能向前推进，天然幂等
UPDATE orders 
SET status = 'PAID', version = version + 1 
WHERE order_id = ? AND status = 'PENDING' AND version = ?
-- 返回 0 行说明已处理过（状态已变更）
```

**幂等性设计三要素**：**唯一标识**（请求 ID / 消息 ID）+ **状态检查**（避免重复执行）+ **原子操作**（检查和执行不可分割）

###### 8. 如何设计一个分布式任务调度系统？

分布式任务调度需要解决**任务精准触发、负载均衡、高可用、故障转移和可视化管控**。

**两种核心架构模式**

| 模式 | 代表 | 特点 |
|---|---|---|
| **中心式调度** | XXL-Job、Quartz 集群 | 逻辑简单，状态集中；中心节点可能成为性能瓶颈 |
| **去中心化调度** | Elastic-Job、PowerJob | 扩展性好，无中心节点瓶颈；实现更复杂 |

**关键技术挑战**

**1. 任务不重复、不遗漏触发**
- **数据库行锁（Quartz）**：触发前竞争数据库行锁，抢到锁的节点获得触发权（DB 压力大）
- **分布式锁（ZooKeeper/Redis）**：通过竞争临时节点或 Redis NX 锁决定谁触发
- **分片广播（Elastic-Job）**：任务触发时，所有在线执行器同时收到通知，每个执行器根据自身分片序号处理对应的数据子集（最优雅，天然负载均衡）

**2. 故障转移**
- 执行器定期向注册中心发心跳；
- 调度器监控心跳超时，自动将失联执行器上的运行中任务重新分配给其他健康节点；
- 任务失败支持自动重试（立即/间隔/指数退避）。

**XXL-Job 工作原理（中心式典型实现）**

```
调度中心（JobScheduleHelper 线程）
    ↓ 扫描 DB 中即将触发的任务
    ↓ 生成触发日志
    ↓ 通过 RPC（HTTP）推送任务到执行器
执行器（ExecutorBizImpl）
    ↓ 在线程池中执行任务
    ↓ 回调执行结果到调度中心
调度中心
    ↓ 更新执行日志状态
```

调度与执行分离：调度中心专注触发和分发，执行器专注业务逻辑，彼此独立扩展。

**生产选型**：简单定时任务用 **XXL-Job**（成熟稳定、开箱即用）；大规模分片任务用 **PowerJob**（支持 MapReduce 模型）。
