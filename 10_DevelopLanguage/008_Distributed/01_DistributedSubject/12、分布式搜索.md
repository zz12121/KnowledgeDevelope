###### 1. 什么是 Elasticsearch？
Elasticsearch 是基于 Lucene 的**开源分布式 RESTful 搜索和分析引擎**，以 JSON 文档为基本单元。本质是一个 NoSQL 文档数据库，但其核心能力是全文搜索和聚合分析。

**核心特性**：
- **分布式高可用**：数据自动分片 + 多副本，水平扩展和故障转移开箱即用。
- **近实时（NRT）**：写入后默认 1 秒内可搜索（由 `refresh_interval` 控制），而非实时。
- **全文搜索**：基于 Lucene 倒排索引，支持复杂相关性评分。
- **ELK 生态**：与 Logstash（数据采集）+ Kibana（可视化）组成 Elastic Stack，广泛用于日志分析。

###### 2. Elasticsearch 的核心概念有哪些？
| 概念 | 说明 | 类比 |
|------|------|------|
| 索引（Index） | 一组相似文档的集合，是逻辑命名空间 | 数据库/表 |
| 文档（Document） | 基本数据单元，JSON 格式 | 行 |
| 字段（Field） | 文档中的键值对，有类型（text/keyword/long/date） | 列 |
| 映射（Mapping） | 字段类型定义，类似表结构 Schema | 表结构 |
| 分片（Shard） | 索引的水平子集，每个分片是完整的 Lucene 索引 | 分表 |
| 副本（Replica） | 分片的复制品，提高读吞吐和高可用 | 从库 |
| 节点（Node） | 运行中的 ES 实例，有主节点、数据节点、协调节点等角色 | 服务器 |

注意：`type` 概念在 ES 7.0 已废弃，每个索引只有一个隐含的 `_doc` 类型。

###### 3. Elasticsearch 的倒排索引是什么？
倒排索引是全文搜索的核心数据结构，与传统"正排索引"（文档 → 内容）相反，是**词项 → 文档列表**的映射。

**构建过程**：
1. **文本分析**：`text` 字段被分析器处理（分词 → 归一化 → 去停用词），生成词项（Token）列表。
2. **建立倒排表**：每个词项记录其出现的文档 ID 列表、词频（TF）和位置信息（用于短语查询）。

**示例**：
```
文档1: "The quick brown fox"
文档2: "The quick dog"

倒排索引：
"quick" → [doc1(pos:2), doc2(pos:2)]
"brown" → [doc1(pos:3)]
"fox"   → [doc1(pos:4)]
"dog"   → [doc2(pos:3)]
```

**Lucene 实现**：词项字典（Term Dictionary）用 FST（有限状态转换器）存储，支持高效前缀查询；倒排列表（Postings List）用跳表优化大文档集合的 AND/OR 操作。

###### 4. Elasticsearch 如何实现分布式？
ES 的分布式能力通过四个机制协同：

1. **节点发现与集群形成**：节点启动时通过 `discovery.seed_hosts` 种子节点发现集群，使用 Zen Discovery（旧版）或基于 Raft 的内置选举（7.x+）选出主节点。
2. **主节点管理**：唯一的 Master 节点负责索引创建/删除、分片分配、节点状态追踪，将 `ClusterState` 发布给所有节点。
3. **数据路由**：写入时，协调节点根据路由公式 `shard = hash(_routing) % num_primary_shards` 确定目标主分片，转发请求。
4. **Scatter/Gather 分布式搜索**：搜索请求发给任意节点（协调节点），协调节点广播给所有相关分片，各分片本地搜索后返回，协调节点合并排序后返回客户端。

###### 5. Elasticsearch 的分片和副本机制是什么？
**主分片（Primary Shard）**：
- 索引创建时指定数量，之后**不可更改**（只能 Reindex）。
- 数据写入必须先到主分片。
- 决定了数据分布粒度和最大并行度。

**副本分片（Replica Shard）**：
- 主分片的完整拷贝，可动态调整数量。
- 提高**读取并发**（搜索和 GET 可路由到任意副本）。
- 提高**高可用**（主分片故障时，副本晋升为主）。
- 副本永远不与其对应的主分片在同一节点。

**写入流程**：写入先在主分片执行 → 并行复制到所有副本 → 多数副本（`wait_for_active_shards`）确认后返回客户端。

###### 6. 如何优化 Elasticsearch 的查询性能？
**硬件与配置**：使用 SSD；JVM 堆内存不超过 31GB（利用压缩指针）；读多写少场景调大 `refresh_interval`（如 30s）降低合并压力。

**索引设计**：合理设置分片数（建议单分片 10-50GB）；精确匹配字段用 `keyword`，全文搜索用 `text`，两者都需要用 `multi-field`；禁用不需要索引的字段（`"index": false`）。

**查询优化**：
- **用 filter 替代 query**：filter 上下文结果可缓存，且不计算相关性评分，性能大幅提升。
- **避免深度分页**：`from + size` 深翻页代价很高，改用 `search_after`（推荐）或 `scroll`（批量导出）。
- **限制返回字段**：`_source` 过滤，只返回需要的字段。
- **避免前缀通配符**：`*text` 这种查询会扫描整个词项字典，效率极低。

**利用缓存**：节点查询缓存（filter 结果）、分片请求缓存（聚合结果，索引数据不变时有效）。

###### 7. Elasticsearch 如何保证数据一致性？
ES 遵循**最终一致性**模型：

**写入一致性**：通过 `wait_for_active_shards` 参数控制，默认 `1`（只需主分片确认），可设置为 `quorum`（多数分片）或 `all`（所有分片），写入可靠性逐级提升，性能逐级降低。

**近实时性**：写入内存缓冲区后，默认每 1 秒 refresh 一次，写入新 Lucene segment，此后数据才可搜索。这是"近实时"不是"实时"的原因。

**持久化保证**：事务日志（Translog）先写磁盘（默认每次写或每 5s fsync），保证宕机重启不丢已确认的写入。定期 flush 将内存 segment 持久化并清空 Translog。

**读一致性**：默认读请求可路由到任意分片（主或副本），由于复制异步，可能读到略旧数据。对一致性要求高时设 `preference=_primary` 只读主分片。

###### 8. Elasticsearch 的近实时性是怎么实现的？
ES 的近实时性核心是 **refresh 机制**：

1. 写入请求先进内存缓冲区（Index Buffer）和 Translog。
2. 每 `refresh_interval`（默认 1s），Index Buffer 的内容写入新的 Lucene **Segment**（内存 + 操作系统页缓存）。
3. Segment 建立索引后，该批数据立即可搜索（但还没 fsync 到磁盘）。
4. 后台**合并（Merge）** 线程定期合并小 Segment 为大 Segment，减少 Segment 数量，提升查询性能。
5. 定期或手动 flush 将 Segment 持久化到磁盘（fsync），并清空 Translog。

**调优技巧**：大批量写入时，可临时将 `refresh_interval=-1`（禁用自动 refresh），写完后手动 refresh，大幅提升写入吞吐量。

###### 9. Elasticsearch 集群如何扩容？
**水平扩容**（增加数据节点）：
1. 新节点加入集群，Master 节点自动触发分片重均衡（Rebalance），将部分分片迁移到新节点。
2. 分片迁移是在线的，不影响正常读写。
3. `cluster.routing.allocation` 参数控制迁移速度和并发数（避免影响性能）。

**分片数限制**：主分片数在索引创建时固定，是扩容上限。若数据量超过预期，需要：创建新索引（更多分片）→ 用 Reindex API 将数据迁移 → 更新别名指向新索引。因此建议预留足够的主分片数。

**最佳实践**：使用**索引别名（Alias）** 对外暴露逻辑索引名，后端实际索引可随时切换（如按月滚动索引），业务代码无需改动。

###### 10. 如何在分布式系统中集成 Elasticsearch 实现搜索？
**典型架构**：业务 DB（MySQL）→ 数据同步 → Elasticsearch → 搜索 API → 前端。

**数据同步方案**：
- **双写**：写 DB 时同步写 ES，简单但增加写入延迟，DB 和 ES 事务无法保证原子性。
- **异步消息**：写 DB 后发 MQ 消息，ES 消费者异步更新。解耦但有延迟。
- **Binlog 监听**（推荐）：Canal 监听 MySQL Binlog，捕获数据变更后同步到 ES。不侵入业务代码，延迟低（毫秒级）。

**索引设计原则**：
- 搜索字段用 `text` 类型并选合适分析器（中文用 IK 分析器）。
- 过滤/排序字段用 `keyword` 类型。
- 冗余常用查询字段（避免跨索引 JOIN）。
- 按时间范围分割索引（如 `order-2026-03`），便于冷数据管理和滚动删除。
