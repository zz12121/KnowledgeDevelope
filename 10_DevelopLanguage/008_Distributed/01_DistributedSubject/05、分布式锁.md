###### 1. 什么是分布式锁？
分布式锁是**在分布式环境下控制并发访问共享资源的机制**，保证同一时刻只有一个进程（一台机器的一个线程）能执行特定操作。

**五大核心特性**：
- **互斥性**：同一时刻只有一个客户端持有锁，这是最基本的要求。
- **可重入性**：同一线程可以多次获取同一把锁而不产生死锁（通过计数器实现）。
- **锁超时**：持有锁的客户端崩溃时，锁能自动释放（超时机制），防止死锁。
- **高可用**：锁服务本身需要高可用，单点故障不影响业务。
- **公平性（可选）**：按申请顺序依次获取锁，避免饥饿（ZooKeeper 临时顺序节点天然支持）。

**典型场景**：电商秒杀扣库存、分布式任务调度防重复执行、资源访问排他控制。

###### 2. 分布式锁有哪些实现方式？
三种主流实现各有优劣：

| 实现方式 | 性能 | 可靠性 | 实现复杂度 | 适用场景 |
|---------|------|--------|----------|---------|
| Redis | 高（内存操作） | 中（主从异步复制有风险） | 中 | 高并发、对一致性要求不极端 |
| ZooKeeper | 中（需持久化） | 高（ZAB 强一致） | 高 | 强一致性要求、金融场景 |
| 数据库 | 低（磁盘 IO） | 高（事务保证） | 低 | 并发不高、依赖少 |

**实际选择**：大多数互联网业务用 Redis（Redisson），金融级强一致场景用 ZooKeeper，无其他依赖的轻量场景用数据库。

###### 3. 基于 Redis 实现分布式锁需要注意什么？
Redis 分布式锁的三个关键点：

**1. 原子性加锁（SET NX PX）**：
```java
// 正确：SET key value NX PX expireMillis —— 原子操作
String result = jedis.set(lockKey, requestId, SetParams.setParams().nx().px(expireTime));
// 错误：SETNX + EXPIRE 分两步，宕机可能导致永久锁
```

**2. 防误删（Lua 脚本原子释放）**：释放锁时必须验证持有者是自己，否则可能误删他人的锁：
```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

**3. 锁续期（看门狗机制）**：业务执行时间可能超过锁超时时间，Redisson 的看门狗会每隔 1/3 过期时间检查并续期：
```java
RLock lock = redisson.getLock("myLock");
lock.lock(); // 默认 30s，看门狗每 10s 续期一次
try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```

**4. 集群场景**：主从架构下，主节点宕机前数据未同步到从节点，可能导致锁丢失。高可靠场景考虑 RedLock 或 ZooKeeper。

###### 4. 什么是 Redlock 算法？
Redlock 是 Redis 官方提出的**多实例分布式锁**算法，通过在多个独立 Redis 实例上加锁，解决单点故障问题。

**算法步骤**（N=5 个独立实例，非主从）：
1. 记录当前时间戳 t1。
2. 依次向 5 个 Redis 实例发送加锁请求（每个设置相同过期时间）。
3. 如果在 **多数（N/2+1=3）个**以上实例加锁成功，且总耗时 < 锁过期时间，则加锁成功。
4. 加锁失败（或成功后业务执行完毕），向所有 5 个实例释放锁。

**争议**：Martin Kleppmann 认为 Redlock 在时钟漂移场景下仍不安全。实际业务中，Redisson 的 Redlock 实现已够用，极致安全场景建议 ZooKeeper。

###### 5. 基于 ZooKeeper 实现分布式锁的原理是什么？
ZooKeeper 通过**临时顺序节点 + Watch 机制**实现公平分布式锁，天然支持锁释放感知：

**加锁流程**：
1. 客户端在锁路径（如 `/lock`）下创建**临时顺序节点**（如 `/lock/lock-0000000001`）。
2. 获取 `/lock` 下所有子节点并排序。
3. 若自己是序号最小的节点，获锁成功。
4. 否则，**监听（Watch）比自己序号小的前一个节点**的删除事件，等待通知。
5. 前一个节点删除（持锁客户端释放锁或宕机），Watch 触发，当前客户端重新检查。

**优点**：避免羊群效应（每个客户端只监听前一个节点）；临时节点自动清理，天然防死锁；强一致性保证。Curator 的 `InterProcessMutex` 是生产级实现。

###### 6. 基于数据库实现分布式锁的方式有哪些？
**方式一：唯一约束（INSERT 竞争）**
```sql
CREATE TABLE distributed_lock (
    lock_key VARCHAR(64) PRIMARY KEY,
    request_id VARCHAR(64),
    expire_time BIGINT
);
-- 插入成功则获锁，唯一约束保证互斥
INSERT INTO distributed_lock(lock_key, request_id, expire_time) VALUES (?, ?, ?)
```

**方式二：悲观锁（FOR UPDATE 行锁）**
```sql
-- lock_key 必须有索引，否则退化为表锁
SELECT * FROM distributed_lock WHERE lock_key = ? FOR UPDATE;
```

**方式三：乐观锁（版本号）**
```sql
-- 更新成功（影响行数=1）则获锁
UPDATE distributed_lock SET version = version + 1
WHERE lock_key = ? AND version = ?
```

数据库锁的共同缺点是性能较低，且需要处理锁超时（定时清理过期记录）和锁失效问题。

###### 7. Redis 分布式锁和 ZooKeeper 分布式锁的区别是什么？
- **性能**：Redis（内存操作）远高于 ZooKeeper（需持久化 + 网络广播）。
- **一致性**：ZooKeeper 基于 ZAB 协议保证强一致；Redis 主从异步复制，极端情况下锁可能丢失。
- **锁特性**：ZooKeeper 临时顺序节点天然支持公平锁、锁自动清理；Redis 需要自己实现可重入和公平性。
- **可靠性**：ZooKeeper 专为分布式协调而生，天然高可靠；Redis 是缓存系统，锁是附加功能。
- **适用场景**：高并发、对一致性要求不极端 → Redis（Redisson）；强一致性要求、金融场景 → ZooKeeper（Curator）。

###### 8. 分布式锁如何解决死锁问题？
死锁主要来自：持有锁的进程崩溃、网络故障导致无法释放。防御策略：

1. **锁超时（TTL）**：设置合理过期时间，持有锁的进程崩溃后自动释放。
2. **看门狗续期**：Redisson 看门狗周期性续期，保证业务运行中锁不过期但崩溃后正常超时。
3. **ZooKeeper 临时节点**：客户端会话断开时节点自动删除，锁自动释放，天然解决死锁。
4. **超时重试 + 退避**：获取锁失败后有限次重试，配合随机退避避免竞争加剧：
```java
RLock lock = redisson.getLock("myLock");
// tryLock：最多等待 3s，持有锁最多 10s
boolean acquired = lock.tryLock(3, 10, TimeUnit.SECONDS);
```

###### 9. 什么是锁的可重入性？
可重入性指**同一线程可以多次获取同一把锁而不死锁**。实现的关键是在锁中记录持有者信息和重入计数。

**Redisson 实现原理**（用 Hash 结构）：
```
HSET lockKey threadId reentrantCount
```
- 加锁时：检查 Hash 中是否已有自己的记录，有则 count+1；没有则竞争锁。
- 释放时：count-1，count 归零才真正删除 Key。

```java
RLock lock = redisson.getLock("myLock");
lock.lock();    // count = 1
lock.lock();    // count = 2（可重入，不阻塞）
lock.unlock();  // count = 1
lock.unlock();  // count = 0，释放锁
```

**ZooKeeper 实现**：在节点数据中存储线程标识和重入次数，同一线程再次获取时直接增加计数，不创建新节点。

###### 10. 如何设计一个高可用的分布式锁服务？
生产环境的分布式锁服务需考虑以下方面：

1. **基础实现选型**：优先使用成熟框架（Redisson / Curator）而非自行实现，它们已处理了大量边界情况。
2. **Redis 集群**：使用 Redisson + Redis Cluster（或哨兵模式），避免单点故障。极端场景考虑 Redlock（多独立实例）。
3. **锁超时策略**：合理设置 TTL，使用看门狗自动续期，防止业务执行时间不确定导致的锁提前释放。
4. **锁监控告警**：监控锁等待时间（过长说明竞争激烈）、锁持有时间（过长说明业务超时或死锁）。
5. **兜底降级**：锁服务（Redis/ZooKeeper）不可用时，通过熔断器快速失败并返回业务默认值，而非无限等待。
6. **锁粒度**：尽量缩小锁的粒度（如从"订单服务锁"缩小到"具体商品 ID 锁"），减少锁竞争范围，提升并发度。
