###### 1. 什么是 BASE 理论？
BASE 理论是分布式系统设计的实践原则，由 eBay 架构师 Dan Pritchett 在 2008 年提出，是 CAP 中 AP 方案的工程延伸。核心思想是**放弃强一致性，允许系统在一段时间内处于中间状态，但最终保证数据一致**，从而获得高可用性。

- **基本可用（Basically Available）**：系统发生故障时，允许损失部分非核心功能，但核心服务必须可用。例如大促期间搜索可能降级为粗糙结果，但下单流程不受影响。
- **软状态（Soft State）**：允许数据同步过程中存在中间状态，不同节点的副本可以暂时不一致。例如 MySQL 主从异步复制期间，从库可能短暂落后于主库。
- **最终一致性（Eventually Consistent）**：系统保证在经过一段时间的同步后，所有数据副本最终会达成一致状态。这个时间窗口取决于网络延迟和系统负载。

**与 ACID 的对比**：ACID 面向传统关系型数据库，强调强一致性，适合银行等强一致业务；BASE 面向高并发分布式系统，强调可用性和扩展性，是互联网业务的主流选择。

###### 2. CAP 和 BASE 的关系是什么？
CAP 定理定义了分布式系统的根本约束，BASE 理论是 CAP 中 **AP 方案的工程实践延伸**。

- **CAP 定理**指出一致性（C）、可用性（A）、分区容错性（P）三者不可兼得。分布式系统中网络分区是必然的，所以 P 必须保留，实际上只能在 C 和 A 之间取舍：
  - **CP 系统**（如 ZooKeeper）：保证一致性，节点故障时可能暂时不可用。
  - **AP 系统**（如 Eureka）：保证可用性，允许数据短暂不一致。
- **BASE 理论**为 AP 系统提供了设计范式——通过**最终一致性替代强一致性**，在高可用和一致性之间找到平衡点。

###### 3. 什么是最终一致性？
最终一致性是弱一致性的特殊形式，指系统在数据更新后不保证立即一致，但承诺**最终所有副本会达成一致**。

**客户端层面的五种变种**：
1. **因果一致性**：有因果关系的操作必须按顺序可见（如 A 回复 B 后，B 总能看到 A 的回复）。
2. **读己之所写**：用户总能读到自己刚写的数据（如发帖后立即查看）。
3. **会话一致性**：同一会话内保证"读己之所写"。
4. **单调读一致性**：每次读到的数据不会比上次更旧。
5. **单调写一致性**：同一用户的写操作按顺序执行。

**服务端 NWR 模型**：N 为副本总数，W 为写成功的最小副本数，R 为读的副本数。当 **W + R > N** 时可保证强一致性；否则为最终一致性。

###### 4. 什么是强一致性、弱一致性？
- **强一致性**：任意时刻读写都能返回最新数据，所有节点数据实时同步。实现方式有 WARO（写所有副本成功才返回）和 Quorum 机制（读写副本数超过半数即可）。适用于银行转账、库存扣减等金融业务。
- **弱一致性**：更新后不承诺立即同步，系统可能返回旧数据。最终一致性是其子集，但额外保证了"最终会收敛"的结果。适用于社交点赞、消息推送等可以容忍延迟的场景。

简单说：强一致性保证 "任何时刻读都是最新的"，弱一致性只保证 "有读到旧数据的可能性，但不会永远如此"。

###### 5. 什么是 Paxos 算法？
Paxos 是分布式共识算法的鼻祖，用于在不可靠网络中让多个节点对某个值达成一致。三个角色：

- **Proposer（提议者）**：接收客户端请求，发起提案。
- **Acceptor（接受者）**：对提案进行投票表决，是共识的核心。
- **Learner（学习者）**：同步已通过的提案值，感知最终结果。

**两阶段流程**：
1. **Prepare 阶段**：Proposer 生成全局递增提案 ID（n），向多数 Acceptor 发送 Prepare(n) 请求。Acceptor 若 n 大于已响应的最大编号，则承诺不再接受编号更小的提案，并返回已接受的最大提案值。
2. **Accept 阶段**：Proposer 收到多数承诺后，选定提案值，发送 Accept(n, value)。Acceptor 若仍未接受更大编号的 Prepare，则接受该提案。

Paxos 最大的问题是工程实现复杂，活锁难以避免（两个 Proposer 互相 Prepare）。Multi-Paxos 通过选出稳定的 Leader 解决活锁，但实现难度仍然很高，这催生了更易理解的 Raft 算法。

###### 6. 什么是 Raft 算法？
Raft 是 Paxos 的工程友好版，通过**将问题分解**（领导选举 + 日志复制 + 安全性）大幅提升可理解性。

**三种角色**：Leader（处理所有写请求）、Follower（响应 Leader）、Candidate（发起选举中的临时状态）。

**领导选举**：Follower 超过随机选举超时（150-300ms）未收到心跳，转为 Candidate 并发起投票。获得多数票者成为 Leader，任期（term）+1。投票规则：只投给日志比自己新的 Candidate，防止日志落后节点当选。

**日志复制**：Leader 收到写请求，追加日志后向所有 Follower 并行发送 AppendEntries RPC。多数节点确认后，Leader 提交并应用到状态机，再通知 Follower 提交。

**Java 实现**：Apache Ratis 是 Raft 的 Java 生产级实现，被 Apache Ozone 等项目使用。

###### 7. Paxos 和 Raft 的区别是什么？
两者都是分布式共识算法，目标相同，但设计哲学不同：

- **可理解性**：Paxos 理论抽象，工程实现需要大量优化才能落地；Raft 将问题分解为独立子问题，论文即可直接指导实现。
- **Leader 模型**：Paxos 理论上允许多个 Proposer 并发提案（Multi-Paxos 才引入 Leader 概念）；Raft 强 Leader 机制，所有写请求都必须经过 Leader，大幅简化逻辑。
- **日志顺序**：Paxos 允许日志空洞（乱序接受提案），需要额外机制填补；Raft 日志必须连续提交，顺序保证内置于协议。
- **工程选择**：etcd、CockroachDB、TiKV 等主流分布式系统都选择 Raft，而非直接实现 Paxos。

###### 8. 什么是拜占庭将军问题？
拜占庭将军问题是分布式容错的经典抽象，描述了在**节点可能恶意发送错误信息**（拜占庭错误）的情况下，如何达成共识。

- **非拜占庭容错（CFT）**：如 Paxos/Raft，只处理节点宕机或网络故障，假设所有节点行为诚实。
- **拜占庭容错（BFT）**：需要容忍节点作恶，系统需要 **3f+1** 个节点才能容忍 f 个拜占庭故障节点。PBFT（实用拜占庭容错）通过三阶段协议（Pre-prepare/Prepare/Commit）验证消息真实性。
- **应用场景**：区块链（如比特币的 PoW 共识、以太坊的 PoS）天然面对拜占庭问题；而数据中心内部的分布式系统通常只需 CFT 算法即可。

###### 9. 什么是 ZAB 协议？
ZAB（ZooKeeper Atomic Broadcast）是 ZooKeeper 专用共识协议，专为**主从架构**设计，保证写操作的全局顺序。与 Raft 类似但更早出现。

**四个阶段**：
1. **选举阶段**：通过 Fast Leader Election，优先选择 ZXID 最大的节点作为 Leader（数据最新者优先）。
2. **发现阶段**：Follower 与新 Leader 同步最新事务 ID（ZXID），确定起始状态。
3. **同步阶段**：Leader 将未提交的日志同步给所有 Follower，超过半数确认后提交。
4. **广播阶段**：正常工作模式，Leader 接收写请求，按 ZXID 全局顺序广播，类似二阶段提交。

**与 Raft 的区别**：ZAB 在选举时优先比较 ZXID（事务 ID），Raft 优先比较 term（任期）。ZAB 的 ZXID 由 epoch（高 32 位）+ 计数器（低 32 位）组成，epoch 在每次 Leader 切换时递增。

###### 10. 分布式系统中如何实现顺序一致性？
顺序一致性要求**所有节点看到相同的操作顺序**，且顺序符合程序逻辑。常用方案：

1. **全局有序广播**：由单一 Leader 序列化所有写请求（ZAB、Raft），确保全局一致顺序。
2. **逻辑时钟（Lamport Clock）**：为每个事件分配逻辑时间戳，建立 happened-before 关系，但无法区分并发事件。
3. **向量时钟（Vector Clock）**：每个节点维护一个向量，记录所有节点的逻辑时钟，可以精确识别因果关系和并发事件，适合冲突检测（如 Dynamo 的版本冲突解决）。
4. **全局时钟**：Google Spanner 使用原子钟 + GPS 提供 TrueTime API，实现接近物理时间的全局顺序，但成本极高。

**实践建议**：大多数业务场景用 ZooKeeper 临时顺序节点或数据库行锁即可保证局部有序；全局有序对性能影响很大，应按需使用。

###### 11. 什么是逻辑时钟和向量时钟？
这两者都是在没有可靠全局物理时钟的情况下，建立分布式事件因果关系的工具。

**逻辑时钟（Lamport Clock）**：
- 规则：发送消息前，发送方将本地时钟 +1 并附在消息中；接收方取 max(本地时钟, 消息时间戳) + 1。
- 保证了 happened-before 的顺序性：若 A 发生在 B 之前，则 ts(A) < ts(B)。
- 缺陷：反之不成立，ts(A) < ts(B) 不能证明 A 一定发生在 B 之前（可能是并发事件）。

**向量时钟（Vector Clock）**：
- 每个节点维护一个长度为 n（节点数）的向量。发消息时，将自己的计数 +1，接收时对每个分量取 max。
- 可以精确判断：A < B（A 因果先于 B）、A > B（B 因果先于 A）、A || B（A 和 B 是并发的，没有因果关系）。
- 应用：Amazon DynamoDB 用向量时钟检测数据副本冲突，当检测到并发写冲突时，由应用层或最后写入获胜策略解决。

###### 12. 什么是 PACELC 模型？
PACELC 是对 CAP 定理的扩展，由 Daniel Abadi 在 2012 年提出，用于更全面地描述分布式系统的权衡。

**核心思想**：
- **在发生网络分区（P）时**：必须在可用性（A）和一致性（C）之间二选一（即 CAP 的核心）。
- **在无网络分区（E，Else）时**：系统仍然需要在延迟（L，Latency）和一致性（C）之间做权衡。

**为什么 CAP 不够**：CAP 只讨论了分区时的行为，但正常运行时（大多数时间），同步复制带来的额外延迟也是真实存在的代价。PACELC 将这个权衡显式化。

**典型系统分类**：
- **PA/EL**（如 DynamoDB、Cassandra）：分区时选可用性，正常时选低延迟，典型的 AP 系统。
- **PC/EC**（如 ZooKeeper、HBase）：分区时选一致性，正常时也选强一致，典型的 CP 系统。
- **PA/EC**（如 MySQL 主从）：分区时选可用性，正常时选一致性（半同步复制）。

###### 13. 什么是一致性哈希算法？
一致性哈希（Consistent Hashing）是一种特殊的哈希算法，用于解决分布式系统中**节点增减时大规模数据迁移**的问题。

**传统哈希的问题**：对 N 个节点取模（key % N），当 N 变化时，几乎所有的 key 都需要重新映射，导致大规模缓存失效。

**一致性哈希原理**：
1. 将整个哈希值空间（0 ~ 2^32-1）构建为一个虚拟的**哈希环**。
2. 每个节点通过哈希函数映射到环上的一个位置。
3. 每个 key 也映射到环上，顺时针找到的第一个节点就是其存储节点。
4. 当节点增减时，只有相邻区间的 key 需要迁移，影响范围约为 **1/N**。

**虚拟节点解决数据倾斜**：真实节点数少时，哈希环上分布可能不均，导致某些节点负载过重。解决方案是每个真实节点对应多个虚拟节点（如 150 个），虚拟节点均匀分布在哈希环上，使数据分布更均匀。

**应用场景**：Redis Cluster（哈希槽是一致性哈希的变体）、Nginx upstream 的 hash 指令、Cassandra 的数据分区。
