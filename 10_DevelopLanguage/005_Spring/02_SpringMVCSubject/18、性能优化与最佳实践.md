###### 1. SpringMVC 的性能优化方法有哪些？

SpringMVC 的性能优化是个系统工程，需要从请求处理链路的多个环节入手。以下是几个最有价值的优化方向。

**1. 解决 HandlerMapping 匹配效率问题（最核心的性能瓶颈）**

SpringMVC 在处理 RESTful 风格的 URL（比如 `/users/{id}`）时，由于无法做直接匹配，必须遍历所有注册的 Mapping 做 Ant 路径模式匹配，时间复杂度从 O(1) 退化到 O(n)。Controller 越多，这个开销越明显。

优化方案一是减少带路径变量的接口数量，用查询参数代替路径变量；方案二是在接口较多时，升级到 Spring 5.3+ 使用 `PathPatternParser`（性能远好于 `AntPathMatcher`），或者自定义 HandlerMapping 加速匹配。

**2. 开启异步处理提升吞吐量**

对于 IO 密集型操作（调外部接口、等数据库），用 `Callable` 或 `DeferredResult` 把请求线程还给容器，避免线程空等：

```java
@GetMapping("/async-data")
public DeferredResult<ResponseEntity<Data>> getDeferredResult() {
    DeferredResult<ResponseEntity<Data>> deferredResult = new DeferredResult<>(5000L);
    asyncService.execute(data -> deferredResult.setResult(ResponseEntity.ok(data)));
    deferredResult.onTimeout(() ->
        deferredResult.setErrorResult(ResponseEntity.status(HttpStatus.REQUEST_TIMEOUT).build()));
    return deferredResult;
}
```

**3. 合理配置各组件参数**

```java
@Configuration
public class WebMvcOptimizationConfig implements WebMvcConfigurer {

    // 视图解析器开启缓存（生产环境）
    @Bean
    public ViewResolver viewResolver() {
        InternalResourceViewResolver resolver = new InternalResourceViewResolver();
        resolver.setCache(true);
        resolver.setCacheLimit(100);
        return resolver;
    }

    // 静态资源配置强缓存，减少重复请求
    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        registry.addResourceHandler("/static/**")
                .addResourceLocations("classpath:/static/")
                .setCacheControl(CacheControl.maxAge(30, TimeUnit.DAYS));
    }

    // 文件上传：超过 40KB 才写入磁盘，避免小文件频繁 IO
    @Bean
    public MultipartResolver multipartResolver() {
        CommonsMultipartResolver resolver = new CommonsMultipartResolver();
        resolver.setMaxUploadSize(10 * 1024 * 1024); // 10MB 上限
        resolver.setMaxInMemorySize(40 * 1024);       // 40KB 内存阈值
        return resolver;
    }
}
```

**4. 应用级缓存**

高频只读数据加 `@Cacheable`，避免重复查数据库：

```java
@Service
public class UserService {
    @Cacheable(value = "users", key = "#id", unless = "#result == null")
    public User getUserById(Long id) {
        return userRepository.findById(id);
    }

    @CacheEvict(value = "users", key = "#user.id")
    public void updateUser(User user) {
        userRepository.update(user);
    }
}
```

这四个方向综合施策，特别是路径匹配优化 + 异步处理这两块，在流量较大的场景下效果最明显。

###### 2. 如何避免重复提交？

防止重复提交是 Web 应用里的经典问题，最可靠的方案是 **Token 机制**，辅以业务层和前端的双重兜底。

**核心思路——一次一证：**

每次进入表单页面时，服务端生成一个唯一 Token 存到 Redis，并把 Token 返给前端。前端提交时带上这个 Token，服务端验证通过后立即删除，之后再来同一个 Token 就会失败。

**拦截器实现：**

```java
@Component
public class DuplicateSubmitInterceptor extends HandlerInterceptorAdapter {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response, Object handler) throws Exception {
        if (handler instanceof HandlerMethod) {
            HandlerMethod handlerMethod = (HandlerMethod) handler;
            PreventDuplicateSubmit annotation =
                handlerMethod.getMethod().getAnnotation(PreventDuplicateSubmit.class);

            if (annotation != null) {
                String tokenKey = "submit:token:" + request.getSession().getId()
                                   + ":" + request.getRequestURI();
                String clientToken = request.getHeader("X-Submit-Token");
                String serverToken = (String) redisTemplate.opsForValue().get(tokenKey);

                if (clientToken == null || !clientToken.equals(serverToken)) {
                    throw new DuplicateSubmitException("请勿重复提交");
                }
                redisTemplate.delete(tokenKey); // 验证通过立即删除，一次有效
            }
        }
        return true;
    }
}
```

**Token 生成接口：**

```java
@GetMapping("/token")
public ResponseEntity<Map<String, String>> generateToken(HttpServletRequest request) {
    String token = UUID.randomUUID().toString();
    String key = "submit:token:" + request.getSession().getId() + ":" + request.getRequestURI();
    redisTemplate.opsForValue().set(key, token, Duration.ofSeconds(5));
    return ResponseEntity.ok(Map.of("token", token));
}
```

**业务层兜底（幂等查重）：**

```java
@Service
public class OrderService {
    public Order createOrder(OrderDTO orderDTO) {
        // 先查，有就直接返回，没有才新建
        Order existing = orderRepository.findByUniqueId(orderDTO.getUniqueId());
        if (existing != null) return existing;
        return orderRepository.save(Order.fromDTO(orderDTO));
    }
}
```

**前端辅助（体验优化）：**

提交后立即禁用按钮，等响应回来再恢复，防止用户手速太快绕过 Token 机制。

###### 3. 如何实现请求幂等性？

幂等性是分布式系统的核心要求：**同一个操作不管执行几次，结果都和执行一次一样**。对于 HTTP 接口来说，GET/PUT/DELETE 天然是幂等的，POST 不是，需要主动保障。

**基于 Redis SETNX 的幂等控制（AOP 实现）：**

```java
@Component
public class IdempotentAspect {
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Around("@annotation(idempotent)")
    public Object checkIdempotent(ProceedingJoinPoint joinPoint, Idempotent idempotent) throws Throwable {
        String key = "idempotent:" + generateKey(joinPoint, idempotent);

        // SETNX：只有第一次能设置成功
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(key, "processing", Duration.ofSeconds(idempotent.expireTime()));

        if (Boolean.FALSE.equals(success)) {
            throw new IdempotentException("重复请求，请稍后再试");
        }

        return joinPoint.proceed();
    }
}
```

**数据库层面的幂等保障（兜底）：**

在订单表上给业务 ID 加唯一约束，数据库层面彻底拦住重复插入：

```java
@Entity
@Table(name = "orders")
public class Order {
    @Column(unique = true) // 数据库唯一约束
    private String orderNo;

    @Version // 乐观锁，防并发修改
    private Long version;
}
```

**HTTP 方法与幂等性对照：**

- `GET`/`HEAD`：查询，天然幂等
- `PUT`：全量更新，幂等（每次都是覆盖写）
- `DELETE`：删完就没了，再删还是没，幂等
- `POST`：创建资源，非幂等，需要额外控制（Token 或业务 ID 去重）
- `PATCH`：部分更新，通常非幂等（加1这类操作）

多层保障（Redis 限流 + 数据库唯一约束）是生产环境的标准做法，任何一层失效了还有另一层兜底。

###### 4. SpringMVC 的最佳实践有哪些？

结合生产环境经验，SpringMVC 的最佳实践可以从配置、控制器设计、异常处理、安全几个维度来说。

**1. 控制器设计规范**

```java
@Validated
@RestController
@RequestMapping("/api/v1")  // API 版本化，便于后续迭代
public class UserController {

    @GetMapping("/users/{id}")
    public ResponseEntity<User> getUser(@PathVariable @Min(1) Long id) {
        return ResponseEntity.ok(userService.findById(id));
    }

    @PostMapping("/users")
    public ResponseEntity<User> createUser(@Valid @RequestBody UserCreateDTO dto) {
        User user = userService.create(dto);
        URI location = ServletUriComponentsBuilder.fromCurrentRequest()
                .path("/{id}").buildAndExpand(user.getId()).toUri();
        return ResponseEntity.created(location).body(user); // 201 Created，Location 头
    }
}
```

**2. 全局异常处理统一化**

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult().getFieldErrors().stream()
                .map(e -> e.getField() + ": " + e.getDefaultMessage())
                .collect(Collectors.toList());
        return ResponseEntity.badRequest().body(new ErrorResponse("VALIDATION_ERROR", errors));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleSystem(Exception ex) {
        log.error("系统异常: ", ex);
        return ResponseEntity.status(500)
                .body(new ErrorResponse("SYSTEM_ERROR", "系统繁忙，请稍后重试"));
    }
}
```

**3. 跨域、缓存、安全统一在配置类里管理**

```java
@Configuration
@EnableWebMvc
public class WebMvcConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("https://trusted-domain.com")
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .maxAge(3600);
    }

    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        registry.addResourceHandler("/static/**")
                .addResourceLocations("classpath:/static/")
                .setCacheControl(CacheControl.maxAge(365, TimeUnit.DAYS));
    }
}
```

**4. 其他几个常踩的坑：**
- 返回值用 `ResponseEntity<T>` 而不是裸对象，方便控制状态码和响应头
- 日期统一用 ISO 格式，在 `FormatterRegistry` 里配好，不要在每个字段上单独加 `@DateTimeFormat`
- 使用 AOP 记录请求耗时，生产环境必备
- 静态资源开启 HTTP 强缓存，动态接口响应头不要带 Cache-Control: max-age
