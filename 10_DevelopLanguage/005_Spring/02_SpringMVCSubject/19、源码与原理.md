###### 1. DispatcherServlet 的源码分析

`DispatcherServlet` 是 SpringMVC 的前端控制器，整个框架的请求处理中枢。它的继承链是 `HttpServlet` → `HttpServletBean` → `FrameworkServlet` → `DispatcherServlet`，体现了经典的模板方法模式。

**初始化过程**在 `initStrategies()` 方法里，WebApplicationContext 刷新时回调，一口气初始化九大组件：

```java
protected void initStrategies(ApplicationContext context) {
    initMultipartResolver(context);           // 文件上传解析器
    initLocaleResolver(context);              // 本地化解析器
    initThemeResolver(context);               // 主题解析器
    initHandlerMappings(context);             // 处理器映射器
    initHandlerAdapters(context);             // 处理器适配器
    initHandlerExceptionResolvers(context);   // 异常解析器
    initRequestToViewNameTranslator(context); // 视图名转换器
    initViewResolvers(context);               // 视图解析器
    initFlashMapManager(context);             // FlashMap 管理器
}
```

**请求处理核心流程**在 `doDispatch()` 方法里，这是整个 MVC 框架最重要的方法：

```java
protected void doDispatch(HttpServletRequest request, HttpServletResponse response) throws Exception {
    HandlerExecutionChain mappedHandler = null;
    try {
        // 1. 检查是否是文件上传请求
        HttpServletRequest processedRequest = checkMultipart(request);

        // 2. 找到处理器（Handler + 拦截器链）
        mappedHandler = getHandler(processedRequest);
        if (mappedHandler == null) {
            noHandlerFound(processedRequest, response);
            return;
        }

        // 3. 找到对应的处理器适配器
        HandlerAdapter ha = getHandlerAdapter(mappedHandler.getHandler());

        // 4. 执行所有拦截器的 preHandle
        if (!mappedHandler.applyPreHandle(processedRequest, response)) return;

        // 5. 真正执行 Controller 方法
        ModelAndView mv = ha.handle(processedRequest, response, mappedHandler.getHandler());

        // 6. 执行所有拦截器的 postHandle
        mappedHandler.applyPostHandle(processedRequest, response, mv);

        // 7. 视图渲染（或直接写响应体）
        processDispatchResult(processedRequest, response, mappedHandler, mv, null);
    } catch (Exception ex) {
        processDispatchResult(processedRequest, response, mappedHandler, null, ex);
    }
}
```

整个流程可以概括为：找处理器 → 找适配器 → 拦截器前置 → 执行方法 → 拦截器后置 → 渲染视图。

###### 2. HandlerMapping 的实现原理是什么？

`HandlerMapping` 负责建立**请求 URL 与处理器方法的映射关系**，核心接口只有一个方法：

```java
public interface HandlerMapping {
    HandlerExecutionChain getHandler(HttpServletRequest request) throws Exception;
}
```

现代 SpringMVC 主要用 `RequestMappingHandlerMapping`，它的工作分两个阶段：

**初始化阶段（容器启动时）：**

通过 `detectHandlerMethods()` 扫描所有带 `@Controller` 注解的 Bean，把每个 `@RequestMapping` 方法封装成 `HandlerMethod` 对象并注册到 `MappingRegistry` 里。

**请求匹配阶段（每次请求时）：**

先尝试直接用 URL 查（O(1)），精确命中直接返回；找不到再遍历所有 Mapping 做 Ant 路径模式匹配（O(n)）。这就是带路径变量的接口在 Controller 很多时会有性能损耗的原因。

```java
protected HandlerMethod getHandlerInternal(HttpServletRequest request) throws Exception {
    String lookupPath = getUrlPathHelper().getLookupPathForRequest(request);
    this.mappingRegistry.acquireReadLock();
    try {
        HandlerMethod handlerMethod = lookupHandlerMethod(lookupPath, request);
        return handlerMethod != null ? handlerMethod.createWithResolvedBean() : null;
    } finally {
        this.mappingRegistry.releaseReadLock();
    }
}
```

注意 `lookupHandlerMethod` 最终返回的处理器方法还要通过 `createWithResolvedBean()` 把 Bean 引用解析为实际的 Spring Bean 实例。

###### 3. HandlerAdapter 的实现原理是什么？

`HandlerAdapter` 解决的是**不同类型处理器统一调用**的问题，是适配器模式的经典应用。SpringMVC 历史上有多种处理器类型（实现 Controller 接口、实现 HttpRequestHandler 接口、基于注解的方法），HandlerAdapter 屏蔽了这些差异。

核心接口：

```java
public interface HandlerAdapter {
    boolean supports(Object handler);   // 是否支持这种处理器
    ModelAndView handle(...) throws Exception;
    long getLastModified(...);
}
```

现代项目主要用 `RequestMappingHandlerAdapter`，它的执行流程：

1. **参数解析**：遍历方法参数，找到对应的 `HandlerMethodArgumentResolver` 解析每个参数值
2. **方法调用**：通过反射调用 Controller 方法，得到返回值
3. **返回值处理**：找到对应的 `HandlerMethodReturnValueHandler` 处理返回值（写响应体或设置视图）

整个执行链路都在 `invokeHandlerMethod()` 方法里完成，这个方法是 Spring MVC 处理请求最核心的实现。

###### 4. 参数解析器的工作原理是什么？

参数解析器通过 `HandlerMethodArgumentResolver` 接口实现，责任链模式，每个解析器负责一类参数类型：

```java
public interface HandlerMethodArgumentResolver {
    boolean supportsParameter(MethodParameter parameter); // 是否认识这个参数
    Object resolveArgument(MethodParameter parameter,
                           ModelAndViewContainer mavContainer,
                           NativeWebRequest webRequest,
                           WebDataBinderFactory binderFactory) throws Exception;
}
```

常见的解析器和它们负责的注解：

- `RequestParamMethodArgumentResolver` → `@RequestParam`
- `PathVariableMethodArgumentResolver` → `@PathVariable`
- `RequestResponseBodyMethodProcessor` → `@RequestBody`（同时也处理 `@ResponseBody`）
- `ModelAttributeMethodProcessor` → `@ModelAttribute`

执行时，`InvocableHandlerMethod.getMethodArgumentValues()` 方法依次遍历方法的每个参数，找到能 `supports` 它的解析器，调用 `resolveArgument()` 拿到参数值，最后通过反射把所有参数一起传给 Controller 方法：

```java
Object[] args = getMethodArgumentValues(request, mavContainer, providedArgs);
return doInvoke(args); // 反射调用
```

如果加了 `@Valid` 或 `@Validated`，在参数解析后还会触发 Bean Validation 校验，校验失败抛出 `MethodArgumentNotValidException`。

###### 5. 返回值处理器的工作原理是什么？

返回值处理器通过 `HandlerMethodReturnValueHandler` 接口实现，和参数解析器是对称的设计：

```java
public interface HandlerMethodReturnValueHandler {
    boolean supportsReturnType(MethodParameter returnType);
    void handleReturnValue(Object returnValue, MethodParameter returnType,
                           ModelAndViewContainer mavContainer,
                           NativeWebRequest webRequest) throws Exception;
}
```

几个关键实现：

- `ViewNameMethodReturnValueHandler`：返回值是字符串，当视图名用
- `ModelAndViewMethodReturnValueHandler`：返回 `ModelAndView`，直接渲染
- `RequestResponseBodyMethodProcessor`：处理 `@ResponseBody`，用 `HttpMessageConverter` 把对象序列化写到响应体

`@ResponseBody` 的处理最有代表性，它会把视图渲染这条路完全短路掉：

```java
public void handleReturnValue(Object returnValue, MethodParameter returnType,
                               ModelAndViewContainer mavContainer,
                               NativeWebRequest webRequest) throws Exception {
    mavContainer.setRequestHandled(true); // 标记：不需要视图渲染了
    writeWithMessageConverters(returnValue, returnType, webRequest); // 直接写响应
}
```

这也是为什么 `@RestController` 的方法不会走视图解析流程的底层原因——`@ResponseBody` 让 `mavContainer.isRequestHandled()` 返回 `true`，DispatcherServlet 检测到就直接跳过视图渲染。

###### 6. SpringMVC 的九大组件是什么？

SpringMVC 的九大组件在 `DispatcherServlet.initStrategies()` 里统一初始化，共同构成完整的 Web 请求处理体系：

1. **MultipartResolver（文件上传解析器）**：检测并解析 multipart 请求，把它包装成 `MultipartHttpServletRequest`，让 Controller 可以方便地拿到上传的文件

2. **LocaleResolver（本地化解析器）**：从请求中提取语言环境信息，支撑整个国际化体系

3. **ThemeResolver（主题解析器）**：支持 UI 主题切换，现在用得比较少，大多数项目已经前后端分离了

4. **HandlerMapping（处理器映射器）**：建立请求与处理器的映射关系，是整个请求分发的起点

5. **HandlerAdapter（处理器适配器）**：统一不同类型处理器的执行方式，屏蔽底层差异

6. **HandlerExceptionResolver（异常解析器）**：统一处理执行过程中抛出的异常，责任链模式

7. **RequestToViewNameTranslator（视图名转换器）**：当处理器没有明确返回视图名时，根据请求 URL 生成默认视图名

8. **ViewResolver（视图解析器）**：把逻辑视图名（字符串）解析为具体的 View 对象，触发模板渲染

9. **FlashMapManager（Flash 属性管理器）**：管理重定向时的临时数据，解决 redirect 后参数传递的问题（Post/Redirect/Get 模式）

这九个组件的协作链路：HandlerMapping 找处理器 → HandlerAdapter 执行处理器 → 有异常就交给 HandlerExceptionResolver → 没异常走 ViewResolver 渲染 → LocaleResolver 提供语言信息给渲染过程。
