###### 1. SpringMVC 如何支持异步请求？

SpringMVC 对异步请求的支持建立在 **Servlet 3.0 规范**的异步处理能力之上，整体分三层来实现：Servlet 容器层、Spring MVC 配置层、控制器层。

第一层是 **Servlet 容器配置**，必须在 web.xml 或 Servlet 初始化器里显式开启异步支持：

```xml
<servlet>
    <servlet-name>dispatcher</servlet-name>
    <servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
    <async-supported>true</async-supported>  <!-- 关键配置 -->
</servlet>
```

第二层是 **Spring MVC 配置**，通过 `WebMvcConfigurer` 的 `configureAsyncSupport` 方法设置超时时间和自定义线程池：

```java
@Configuration
@EnableWebMvc
public class AsyncConfig implements WebMvcConfigurer {
    @Override
    public void configureAsyncSupport(AsyncSupportConfigurer configurer) {
        configurer.setDefaultTimeout(30000L);       // 默认超时 30 秒
        configurer.setTaskExecutor(taskExecutor()); // 自定义线程池
    }

    @Bean
    public ThreadPoolTaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        return executor;
    }
}
```

第三层是**控制器层**，支持返回 `Callable`、`DeferredResult`、`WebAsyncTask` 等异步类型，框架会自动识别并走异步流程。

底层原理上，Spring MVC 通过 `WebAsyncManager` 作为异步处理的核心管理器。在 `DispatcherServlet.doDispatch()` 里，每次请求进来都会先检查是否已经启动了异步处理，如果已启动就直接返回，不再走同步处理链路：

```java
if (WebAsyncUtils.getAsyncManager(request).isConcurrentHandlingStarted()) {
    return; // 已启动异步处理，本次直接返回
}
```

当控制器方法返回异步类型时，`RequestMappingHandlerAdapter` 会找到对应的 `HandlerMethodReturnValueHandler`（比如 `CallableMethodReturnValueHandler`），负责启动整个异步处理流程。

###### 2. Callable 和 DeferredResult 的作用是什么？

`Callable` 和 `DeferredResult` 是 Spring MVC 异步处理的两种核心编程模型，目标都是实现异步响应，但在控制机制和使用场景上有明显区别。

**Callable —— 内部线程池模型**

控制器返回 `Callable`，Spring 会用它配置好的 `TaskExecutor` 在新线程里执行 `call()` 方法，执行完再把结果重新分派给 `DispatcherServlet` 走后续处理。适合的是那种"内部有个耗时任务，让 Spring 帮我异步跑一下"的场景：

```java
@GetMapping("/async/callable")
public Callable<String> handleWithCallable() {
    return () -> {
        Thread.sleep(2000); // 模拟耗时操作
        return "Callable result after 2 seconds";
    };
}
```

**DeferredResult —— 外部事件驱动模型**

`DeferredResult` 更像是一个"占位符"，控制器先把它返回出去，然后由外部线程（比如消息队列的消费者、推送服务）在任意时刻调用 `setResult()` 填入结果。适合长轮询、跨系统事件通知等场景：

```java
@GetMapping("/async/deferred")
public DeferredResult<String> handleWithDeferred() {
    DeferredResult<String> deferredResult = new DeferredResult<>(5000L, "Timeout result");

    deferredResult.onTimeout(() -> deferredResult.setErrorResult("Operation timeout"));
    resultQueue.add(deferredResult);
    return deferredResult;
}

// 外部事件来了，主动填入结果
@PostMapping("/trigger")
public String triggerResult(@RequestParam String data) {
    resultQueue.forEach(r -> r.setResult("Processed: " + data));
    return "Triggered";
}
```

两者的核心区别可以概括为：`Callable` 的执行权在 Spring 手里，`DeferredResult` 的结果设置权在应用代码手里。`DeferredResult` 实现了**请求线程与结果生产线程的完全解耦**，特别适合需要跨系统协作的异步场景。

###### 3. @Async 注解在 SpringMVC 中的应用是什么？

`@Async` 注解主要用于 **Service 层方法的异步化**，它和控制器层的 `Callable`/`DeferredResult` 是两个不同层次的概念，不要混淆。

`@Async` 的典型用法是在 Service 方法上打注解，方法在一个独立线程里运行，返回 `CompletableFuture` 供调用方组合：

```java
@Service
public class BusinessService {

    @Async
    public CompletableFuture<String> processData(String data) {
        Thread.sleep(3000); // 模拟耗时处理
        return CompletableFuture.completedFuture("Processed: " + data);
    }
}

@RestController
public class AsyncIntegrationController {
    @Autowired
    private BusinessService businessService;

    @GetMapping("/async/service")
    public CompletableFuture<String> handleAsyncService() {
        return businessService.processData("test data");
    }
}
```

使用 `@Async` 需要在配置类上加 `@EnableAsync`，并建议配置一个自定义线程池，否则会用 `SimpleAsyncTaskExecutor`（每次新建线程，生产不推荐）。

总结来说，两者的定位是：
- **控制器异步**（Callable/DeferredResult）：解决 HTTP 请求线程被阻塞的问题，提升服务器吞吐量
- **Service 异步**（@Async）：让业务方法在后台线程里跑，优化内部任务处理

两者可以组合使用，覆盖更复杂的异步场景。

###### 4. 异步请求处理的原理是什么？

Spring MVC 异步请求处理的原理基于 **Servlet 3.0 异步规范**，核心思想是**请求线程与结果处理线程分离**，通过回调机制在任务完成后重新触发请求处理。

整个流程分三个阶段：

**阶段一：异步处理启动**

控制器返回 `Callable` 或 `DeferredResult` 时，`RequestMappingHandlerAdapter` 识别到异步类型，调用 `WebAsyncManager.startCallableProcessing()`，在 Servlet 层面调用 `asyncContext.startAsync()`，把请求线程还给线程池。

**阶段二：异步任务执行**

对于 `Callable`，Spring 把它提交到配置的 `TaskExecutor`，在新线程里执行业务逻辑；对于 `DeferredResult`，框架只是等待，不占用线程，直到外部代码调用 `setResult()`。

```java
// WebAsyncManager 内部简化逻辑
taskExecutor.submit(() -> {
    Object result = callable.call(); // 执行业务
    setConcurrentResultAndDispatch(result); // 结果完成后重新派发
});
```

**阶段三：结果回调与请求恢复**

任务完成后，`setConcurrentResultAndDispatch()` 把结果保存起来，然后调用 `asyncContext.dispatch()`，将请求重新派发给 Servlet 容器，DispatcherServlet 再次接收到请求，这次直接拿到之前保存的结果，走正常的视图渲染或响应写出流程。

这套机制的性能价值在于：在 IO 密集型场景下，请求线程在等待结果期间已经被释放回线程池，可以处理其他请求，服务器吞吐量可以得到显著提升，同时线程资源消耗保持在较低水平。
