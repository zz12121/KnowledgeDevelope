###### 1. SpringMVC 支持哪些视图技术？

SpringMVC 通过可插拔的视图架构支持多种视图技术，核心设计基于 `View` 接口抽象，各种视图实现与框架解耦，遵循开闭原则。

主流视图技术一览：

- **JSP/JSTL**：传统 Java Web 应用，`InternalResourceView` / `JstlView`，模板后缀 `.jsp`
- **Thymeleaf**：现代 HTML5 应用，`ThymeleafView`，模板后缀 `.html`，支持自然模板（直接用浏览器打开不报错）
- **FreeMarker**：高性能模板引擎，`FreeMarkerView`，模板后缀 `.ftl`
- **JSON**：RESTful API 场景，`MappingJackson2JsonView`，通过 Jackson 序列化对象
- **PDF / Excel**：报表导出，`AbstractPdfView` / `AbstractExcelView`

所有视图技术都实现统一的 `View` 接口：

```java
public interface View {
    String getContentType();
    void render(Map<String, ?> model,
                HttpServletRequest request,
                HttpServletResponse response) throws Exception;
}
```

###### 2. InternalResourceViewResolver 的作用是什么？

`InternalResourceViewResolver` 是 SpringMVC 用来解析 JSP 视图的核心组件，把控制器返回的逻辑视图名（如 `"user/detail"`）拼接成完整的物理路径（如 `/WEB-INF/views/user/detail.jsp`），然后通过 `RequestDispatcher.forward()` 做服务器端转发。

解析原理很简单：**prefix（前缀） + 视图名 + suffix（后缀）= 物理路径**。

```java
// 源码中的核心逻辑
public class InternalResourceViewResolver extends UrlBasedViewResolver {
    protected AbstractUrlBasedView buildView(String viewName) throws Exception {
        String path = getPrefix() + viewName + getSuffix();
        // 如："/WEB-INF/views/" + "user/detail" + ".jsp"
        InternalResourceView view = (InternalResourceView) super.buildView(viewName);
        view.setUrl(path);
        return view;
    }
}
```

配置示例：

```java
@Bean
public InternalResourceViewResolver jspViewResolver() {
    InternalResourceViewResolver resolver = new InternalResourceViewResolver();
    resolver.setPrefix("/WEB-INF/views/");
    resolver.setSuffix(".jsp");
    resolver.setViewClass(JstlView.class);  // 支持 JSTL 标签
    resolver.setOrder(1);
    return resolver;
}
```

这种设计让控制器只需要返回逻辑名，不用关心视图文件放在哪里，实现了控制器与视图路径的解耦。

###### 3. 如何配置视图解析器？

支持 XML 和 Java 两种配置方式，现代项目推荐 Java 配置。多个视图解析器通过 `order` 属性控制优先级，值越小越先尝试。

```java
@Configuration
@EnableWebMvc
public class WebMvcConfig implements WebMvcConfigurer {

    // JSP 解析器，优先级最高
    @Bean
    public InternalResourceViewResolver jspViewResolver() {
        InternalResourceViewResolver resolver = new InternalResourceViewResolver();
        resolver.setPrefix("/WEB-INF/views/");
        resolver.setSuffix(".jsp");
        resolver.setViewClass(JstlView.class);
        resolver.setOrder(1);
        return resolver;
    }

    // Thymeleaf 解析器，优先级次之
    @Bean
    public ThymeleafViewResolver thymeleafViewResolver() {
        ThymeleafViewResolver resolver = new ThymeleafViewResolver();
        resolver.setTemplateEngine(templateEngine());
        resolver.setCharacterEncoding("UTF-8");
        resolver.setOrder(2);
        return resolver;
    }

    @Bean
    public SpringTemplateEngine templateEngine() {
        SpringTemplateEngine engine = new SpringTemplateEngine();
        engine.setTemplateResolver(templateResolver());
        return engine;
    }

    private ITemplateResolver templateResolver() {
        SpringResourceTemplateResolver resolver = new SpringResourceTemplateResolver();
        resolver.setPrefix("/WEB-INF/templates/");
        resolver.setSuffix(".html");
        resolver.setTemplateMode(TemplateMode.HTML);
        return resolver;
    }
}
```

DispatcherServlet 会按 order 从小到大依次尝试每个解析器，直到某个解析器成功返回 View 对象为止。

###### 4. 什么是视图名称解析？

视图名称解析就是 SpringMVC 把控制器返回的逻辑视图名转换成具体 View 对象的过程，由 `ViewResolver` 接口负责。

完整流程：控制器方法返回视图名（如 `"userList"`）→ DispatcherServlet 调用 `resolveViewName()` 遍历所有已注册的 `ViewResolver` → 找到第一个能处理这个名称的解析器 → 返回 View 对象 → 调用 `view.render()` 完成渲染。

```java
protected View resolveViewName(String viewName, Map<String, Object> model,
                                Locale locale, HttpServletRequest request) {
    for (ViewResolver viewResolver : this.viewResolvers) {
        View view = viewResolver.resolveViewName(viewName, locale);
        if (view != null) return view;  // 第一个成功的就用，不再继续
    }
    return null;
}
```

这种设计的价值在于：控制器只负责告诉 Spring "我要渲染 userList 这个页面"，至于用 JSP 还是 Thymeleaf，文件在哪里，控制器完全不需要知道，完全由配置决定。

###### 5. 如何返回 JSON 数据？

返回 JSON 数据主要靠 `@ResponseBody` 注解（或 `@RestController`）加上 Jackson 的 `MappingJackson2HttpMessageConverter` 配合完成。

当方法有 `@ResponseBody` 注解时，`RequestResponseBodyMethodProcessor` 会根据请求的 `Accept` 头选择合适的 `HttpMessageConverter`，然后把返回值序列化写入响应体。

```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping("/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.findById(id);  // 自动序列化为 JSON
    }

    @PostMapping
    public ResponseEntity<User> createUser(@RequestBody User user) {
        User savedUser = userService.save(user);
        return ResponseEntity.ok(savedUser);
    }

    @GetMapping
    public List<User> getUsers() {
        return userService.findAll();  // 自动序列化为 JSON 数组
    }
}
```

如果需要定制 JSON 输出（比如日期格式、字段命名策略、忽略空字段），在配置类中自定义 `ObjectMapper` 即可：

```java
@Bean
public MappingJackson2HttpMessageConverter jackson2HttpMessageConverter() {
    MappingJackson2HttpMessageConverter converter = new MappingJackson2HttpMessageConverter();
    ObjectMapper mapper = new ObjectMapper();
    mapper.setSerializationInclusion(JsonInclude.Include.NON_NULL);  // 忽略 null 字段
    mapper.configure(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS, false);
    converter.setObjectMapper(mapper);
    return converter;
}
```

###### 6. 如何返回 XML 数据？

返回 XML 数据与返回 JSON 思路一样，换成 `MarshallingHttpMessageConverter` 配合 JAXB 处理就行。实体类需要加 JAXB 注解，方法上通过 `produces = MediaType.APPLICATION_XML_VALUE` 指定响应类型。

```java
// 实体类加 JAXB 注解
@XmlRootElement
@XmlAccessorType(XmlAccessType.FIELD)
public class User {
    private Long id;
    private String name;
}

@RestController
@RequestMapping("/api")
public class XmlController {

    @GetMapping(value = "/user/{id}", produces = MediaType.APPLICATION_XML_VALUE)
    public User getUserXml(@PathVariable Long id) {
        return userService.findById(id);  // 自动序列化为 XML
    }
}
```

如果希望同一个接口既能返回 JSON 又能返回 XML（根据客户端 `Accept` 头自动切换），可以用 `ContentNegotiatingViewResolver` 做内容协商，客户端发 `Accept: application/json` 就返回 JSON，发 `Accept: application/xml` 就返回 XML。

###### 7. ModelAndView 的作用是什么？

`ModelAndView` 是同时封装模型数据和视图信息的容器，让控制器方法在一个返回值里把视图名称、模型数据、HTTP 状态码都塞进去，适合需要精细控制响应的场景。

```java
@Controller
public class UserController {

    @GetMapping("/user/{id}")
    public ModelAndView getUserDetail(@PathVariable Long id) {
        ModelAndView mav = new ModelAndView("user/detail");

        mav.addObject("user", userService.findById(id));
        mav.addObject("profile", profileService.findByUserId(id));
        mav.addObject("pageTitle", "用户详情");
        mav.setStatus(HttpStatus.OK);

        return mav;
    }

    @PostMapping("/user")
    public ModelAndView createUser(@Valid User user, BindingResult result) {
        ModelAndView mav = new ModelAndView();

        if (result.hasErrors()) {
            mav.setViewName("user/form");
            mav.addObject("errors", result.getAllErrors());
            mav.setStatus(HttpStatus.BAD_REQUEST);
        } else {
            User savedUser = userService.save(user);
            mav.setViewName("redirect:/user/" + savedUser.getId());
            mav.setStatus(HttpStatus.CREATED);
        }

        return mav;
    }
}
```

当然，现代项目更多用 `Model` 参数 + 返回视图名字符串的方式，`ModelAndView` 在传统 MVC 模板渲染场景中用得多。

###### 8. Model、ModelMap 和 Map 的区别是什么？

三者都是用来往视图传数据的，底层都是 `BindingAwareModelMap` 这同一个实现，实际效果完全等价，区别只在 API 设计层面：

- `Model` 接口：最常用，API 最清晰（`addAttribute()`），推荐使用
- `ModelMap`：继承 `LinkedHashMap`，适合需要用 Map 操作的场景
- `Map<String, Object>`：最灵活，但语义不如 `Model` 明确

```java
@Controller
public class ModelController {

    // 三种写法效果完全相同

    @GetMapping("/test1")
    public String test1(Model model) {
        model.addAttribute("message", "Hello Model");
        return "view";
    }

    @GetMapping("/test2")
    public String test2(ModelMap model) {
        model.addAttribute("message", "Hello ModelMap");
        return "view";
    }

    @GetMapping("/test3")
    public String test3(Map<String, Object> model) {
        model.put("message", "Hello Map");
        return "view";
    }
}
```

Spring 内部通过 `ModelMethodProcessor` 统一处理这三种参数类型，注入的都是同一个 `BindingAwareModelMap` 实例。日常开发直接用 `Model` 就好了。

###### 9. RedirectAttributes 的作用是什么？

`RedirectAttributes` 解决的是重定向时数据传递的问题——普通的 `Model` 属性在重定向后会丢失，因为重定向是一次新的 HTTP 请求，原来的请求属性就没了。

`RedirectAttributes` 提供两种传递方式：

- `addAttribute()`：把数据拼接到重定向 URL 的查询参数里（能在 URL 里看到）
- `addFlashAttribute()`：把数据存入 Session 的 FlashMap（重定向后自动读取并清除，不会出现在 URL 里）

Flash 属性是 Post-Redirect-Get 模式的标配，用来在重定向后展示"操作成功"或"操作失败"的提示消息：

```java
@Controller
public class RedirectController {

    @PostMapping("/user")
    public String createUser(@Valid User user, RedirectAttributes redirectAttributes) {
        try {
            User savedUser = userService.save(user);
            // Flash 属性：重定向后自动注入 Model，不显示在 URL 里
            redirectAttributes.addFlashAttribute("successMessage", "用户创建成功");
            // URL 参数：拼到 URL 里
            redirectAttributes.addAttribute("userId", savedUser.getId());
            return "redirect:/user/{userId}";
        } catch (Exception e) {
            redirectAttributes.addFlashAttribute("errorMessage", "创建失败: " + e.getMessage());
            return "redirect:/user/form";
        }
    }

    @GetMapping("/user/{userId}")
    public String getUser(@PathVariable Long userId, Model model) {
        // successMessage 已经自动从 FlashMap 中注入到 Model 里了
        model.addAttribute("user", userService.findById(userId));
        return "user/detail";
    }
}
```

###### 10. 如何实现页面重定向？

SpringMVC 里实现重定向最常见的方式是在返回值前加 `"redirect:"` 前缀：

```java
@Controller
public class RedirectController {

    // 重定向到应用内 URL
    @PostMapping("/process")
    public String processForm(Form form) {
        return "redirect:/result";
    }

    // 重定向时携带路径变量
    @PostMapping("/user")
    public String createUser(User user) {
        User savedUser = userService.save(user);
        return "redirect:/user/" + savedUser.getId();
    }
}
```

底层原理：`UrlBasedViewResolver` 检测到 `"redirect:"` 前缀后，创建 `RedirectView` 实例，`RedirectView.render()` 最终调用 `HttpServletResponse.sendRedirect()` 发送 302 响应给浏览器，浏览器再发起新的 GET 请求。

###### 11. 如何实现请求转发？

请求转发通过 `"forward:"` 前缀实现，在服务器内部把请求转发到其他资源，浏览器地址栏不变，是同一次 HTTP 请求：

```java
@Controller
public class ForwardController {

    // 转发到其他控制器
    @GetMapping("/preprocess")
    public String preprocess() {
        // 前置处理...
        return "forward:/main/handle";
    }

    // 转发到 WEB-INF 下的受保护资源（客户端直接访问不到）
    @GetMapping("/protected")
    public String protectedResource() {
        return "forward:/WEB-INF/protected/data.xml";
    }
}
```

底层通过 `RequestDispatcher.forward()` 实现，和重定向的关键区别是：转发是服务器内部行为，共享同一个请求对象，浏览器只发了一次请求，地址栏不变。

###### 12. redirect 和 forward 的区别是什么？

redirect 和 forward 是两种根本不同的跳转机制，核心区别在于**请求次数和数据共享**：

**forward（转发）**：服务器内部操作，客户端只发一次请求；URL 不变；前后共享同一个 Request 对象，Request 属性可以传递；只能访问应用内部资源；性能更好（没有网络往返）。

**redirect（重定向）**：客户端收到 302 后再发一次新请求；URL 改变；两次请求完全独立，Request 属性无法共享（可以用 Flash 属性补救）；可以跳转到任何 URL，包括外部站点；需要一次网络往返。

**选择建议**：
- 用 forward：需要保持请求上下文（比如把错误信息传给视图）、访问 WEB-INF 受保护资源
- 用 redirect：防止表单重复提交（Post-Redirect-Get 模式）、操作完成后跳转（避免刷新重复提交）、跨应用跳转
