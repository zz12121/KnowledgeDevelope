###### 1. 什么是拦截器？

拦截器是 Spring MVC 中基于 AOP 思想的横切处理机制，核心价值是把公共业务逻辑（身份验证、日志记录、性能监控等）从控制器方法里抽出来，集中统一处理，避免代码重复。

Spring MVC 通过 `HandlerExecutionChain` 管理拦截器链，它持有一个 `HandlerInterceptor` 列表，按注册顺序形成调用链。每个 `HandlerInterceptor` 提供三个生命周期钩子：

- `preHandle`：控制器方法执行前调用，返回 false 可以中断后续流程
- `postHandle`：控制器方法执行后、视图渲染前调用
- `afterCompletion`：整个请求处理完成后调用（包括视图渲染），类似 finally 块

拦截器与过滤器（Filter）的关键区别：拦截器是 Spring MVC 的概念，只作用于控制器请求，可以注入 Spring Bean；过滤器是 Servlet 规范的概念，作用于所有请求（包括静态资源），不依赖 Spring 容器。

###### 2. 拦截器和过滤器的区别是什么？

两者都是 Web 应用的横切处理组件，但所属体系不同，能力也有差异：

**作用范围**：过滤器作用于所有 Web 请求（包括静态资源），拦截器只针对 Spring MVC 控制器方法。

**规范依赖**：过滤器是 Servlet 规范标准组件，由 Servlet 容器管理；拦截器是 Spring 框架特有组件，由 Spring 容器管理，支持依赖注入。

**访问能力**：过滤器只能访问原生 Servlet API；拦截器可以访问 `HandlerMethod`、`ModelAndView` 等 Spring 特有对象，能知道"这次请求是哪个控制器的哪个方法在处理"。

**执行位置**：从 `DispatcherServlet` 视角看，过滤器在 `DispatcherServlet` 外层；拦截器在 `DispatcherServlet` 内部，介于路由匹配和控制器方法执行之间。

**选择原则**：需要操作 Spring 特有对象（HandlerMethod、ModelAndView）或需要注入 Spring Bean —— 用拦截器；需要处理所有请求（包括静态资源）或与 Spring 无关的通用处理（比如字符编码转换）—— 用过滤器。

###### 3. 如何自定义拦截器？

实现 `HandlerInterceptor` 接口，重写需要的方法，再注册到配置类就完成了。

```java
@Component
public class AuthenticationInterceptor implements HandlerInterceptor {

    @Autowired
    private UserService userService;  // 拦截器支持依赖注入

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                             Object handler) throws Exception {
        // 只拦截控制器方法，静态资源等直接放行
        if (!(handler instanceof HandlerMethod)) {
            return true;
        }

        String token = request.getHeader("Authorization");
        if (!StringUtils.hasText(token)) {
            sendErrorResponse(response, 401, "未提供认证令牌");
            return false;
        }

        User user = userService.validateToken(token);
        if (user == null) {
            sendErrorResponse(response, 401, "认证令牌无效");
            return false;
        }

        UserContext.setCurrentUser(user);
        return true;
    }

    @Override
    public void postHandle(HttpServletRequest request, HttpServletResponse response,
                          Object handler, ModelAndView modelAndView) throws Exception {
        // 向所有视图模型注入当前用户信息
        if (modelAndView != null) {
            modelAndView.addObject("currentUser", UserContext.getCurrentUser());
        }
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                               Object handler, Exception ex) throws Exception {
        // 清理 ThreadLocal，防止内存泄漏——这一步很容易忘，但很重要！
        UserContext.clear();
    }
}

// 注册拦截器
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {
    @Autowired
    private AuthenticationInterceptor authInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(authInterceptor)
                .addPathPatterns("/api/**")
                .excludePathPatterns("/api/auth/login", "/api/public/**");
    }
}
```

###### 4. HandlerInterceptor 接口有哪些方法？

`HandlerInterceptor` 接口定义了三个方法，覆盖请求处理的不同生命周期阶段：

**preHandle**：控制器方法执行之前调用。返回 `true` 继续后续流程，返回 `false` 中断整个请求（后续拦截器和控制器方法都不再执行）。典型用途：身份验证、权限检查、限流。

**postHandle**：控制器方法执行后、视图渲染前调用。可以访问和修改 `ModelAndView` 对象。注意：只有控制器方法正常返回（未抛出异常）时才会执行。典型用途：向 Model 添加公共数据、修改视图名称。

**afterCompletion**：整个请求处理完成后调用（视图渲染完毕，或控制器方法抛出异常后）。有一个 `Exception ex` 参数，可以获取处理过程中的异常信息。无论处理成功与否都会执行，类似 finally 块。典型用途：资源释放、ThreadLocal 清理、记录完整请求日志。

```java
public interface HandlerInterceptor {

    default boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                               Object handler) throws Exception {
        return true;
    }

    default void postHandle(HttpServletRequest request, HttpServletResponse response,
                            Object handler, @Nullable ModelAndView modelAndView) throws Exception {}

    default void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                 Object handler, @Nullable Exception ex) throws Exception {}
}
```

###### 5. preHandle、postHandle 和 afterCompletion 方法的执行顺序是什么？

三个方法遵循固定的生命周期顺序，并且多个拦截器时有明确的执行方向：

**正常流程**：
```
请求到达
  ↓
InterceptorA.preHandle()   → 正序（按注册顺序）
InterceptorB.preHandle()
InterceptorC.preHandle()
  ↓
控制器方法执行
  ↓
InterceptorC.postHandle()  → 逆序
InterceptorB.postHandle()
InterceptorA.postHandle()
  ↓
视图渲染
  ↓
InterceptorC.afterCompletion()  → 逆序
InterceptorB.afterCompletion()
InterceptorA.afterCompletion()
```

**异常情况**：

- 如果某个拦截器的 `preHandle` 返回 false，后续的拦截器和控制器不执行，但**已经成功执行 preHandle 的拦截器**会执行 `afterCompletion`
- 如果控制器方法抛出异常，`postHandle` 不执行（因为没有正常的 ModelAndView），但 `afterCompletion` 仍然执行
- `afterCompletion` 的可靠执行保证了资源释放不会被遗漏

这种设计让拦截器链的执行类似一个洋葱结构，外层的拦截器最先开始、最后结束，符合资源管理的最佳实践。

###### 6. 多个拦截器的执行顺序是怎样的？

多个拦截器形成拦截器链，执行顺序由注册顺序决定，遵循**先进后出**的栈式原则：

- `preHandle`：按注册顺序**正序**执行
- `postHandle`：按注册顺序**逆序**执行
- `afterCompletion`：按注册顺序**逆序**执行

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new LoggingInterceptor()).order(1);
        registry.addInterceptor(new AuthInterceptor()).order(2);
        registry.addInterceptor(new PermissionInterceptor()).order(3);
    }
}
```

执行序列：

```
请求 → LoggingInterceptor.preHandle → AuthInterceptor.preHandle → PermissionInterceptor.preHandle
     → 控制器执行
     → PermissionInterceptor.postHandle → AuthInterceptor.postHandle → LoggingInterceptor.postHandle
     → 视图渲染
     → PermissionInterceptor.afterCompletion → AuthInterceptor.afterCompletion → LoggingInterceptor.afterCompletion
```

这种洋葱结构确保了：日志拦截器最先开始记录，最后结束记录，能获取完整的请求耗时；认证拦截器在权限拦截器之前执行，保证逻辑顺序正确。

###### 7. 如何配置拦截器？

通过 `WebMvcConfigurer` 的 `addInterceptors()` 方法配置，支持细粒度的路径匹配和排除规则：

```java
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {

    @Autowired
    private LoggingInterceptor loggingInterceptor;
    @Autowired
    private AuthInterceptor authInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        // 日志拦截器：拦截所有请求
        registry.addInterceptor(loggingInterceptor)
                .addPathPatterns("/**")
                .order(1);

        // 认证拦截器：只拦截 API 路径，排除公开接口
        registry.addInterceptor(authInterceptor)
                .addPathPatterns("/api/**")
                .excludePathPatterns("/api/auth/login", "/api/auth/register")
                .excludePathPatterns("/api/public/**")
                .order(2);

        // 权限拦截器：只拦截管理端
        registry.addInterceptor(new PermissionInterceptor())
                .addPathPatterns("/admin/**")
                .order(3);
    }
}
```

路径匹配规则：`/**` 匹配所有路径（含多级），`/api/*` 只匹配 /api 下的一级路径，精确路径如 `/api/login` 只匹配该路径。

###### 8. 拦截器的应用场景有哪些？

拦截器在企业应用中的典型使用场景：

**JWT 身份认证**：在 preHandle 中验证 Token，认证失败直接返回 401，认证成功将用户信息存入 ThreadLocal，控制器可以直接取用。

**接口性能监控**：preHandle 记录开始时间，afterCompletion 计算耗时，超过阈值打印慢接口告警日志。

**链路追踪**：preHandle 生成 TraceId 写入 MDC，在响应头中返回，afterCompletion 清理 MDC。

**防重复提交**：preHandle 检查幂等 Token，首次提交成功写入 Redis 标记，相同 Token 再次提交直接拦截。

```java
// 性能监控拦截器示例
@Component
public class PerformanceInterceptor implements HandlerInterceptor {
    private final ThreadLocal<Long> startTime = new ThreadLocal<>();

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                             Object handler) throws Exception {
        startTime.set(System.currentTimeMillis());
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                               Object handler, Exception ex) throws Exception {
        long duration = System.currentTimeMillis() - startTime.get();
        startTime.remove();  // 清理 ThreadLocal

        if (duration > 1000) {
            log.warn("慢接口: {} {} 耗时: {}ms",
                request.getMethod(), request.getRequestURI(), duration);
        }
    }
}
```
