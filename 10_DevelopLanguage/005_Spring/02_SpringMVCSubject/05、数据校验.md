###### 1. SpringMVC 如何进行数据校验？

SpringMVC 的数据校验基于两条路线：**JSR-303/349 Bean Validation 标准注解校验**（主流）和**Spring Validator 接口编程式校验**（适合复杂业务逻辑）。

底层通过 `LocalValidatorFactoryBean` 集成 JSR 标准，委托给 Hibernate Validator 做实际校验；当方法参数带 `@Valid` 或 `@Validated` 时，`MethodValidationInterceptor` 会 AOP 拦截并触发校验逻辑。

```java
// 方式一：Bean Validation 注解校验（最常用）
@RestController
public class UserController {

    @PostMapping("/users")
    public ResponseEntity<?> createUser(@Valid @RequestBody User user,
                                        BindingResult result) {
        if (result.hasErrors()) {
            return ResponseEntity.badRequest().body(result.getAllErrors());
        }
        return ResponseEntity.ok(userService.save(user));
    }
}

// 方式二：实现 Validator 接口（适合复杂逻辑）
public class UserValidator implements Validator {

    @Override
    public boolean supports(Class<?> clazz) {
        return User.class.equals(clazz);
    }

    @Override
    public void validate(Object target, Errors errors) {
        User user = (User) target;
        if (user.getAge() != null && user.getAge() < 18) {
            errors.rejectValue("age", "age.min", "年龄必须大于 18 岁");
        }
        // 可以在这里写任意复杂的业务校验逻辑
    }
}
```

两种方式可以组合使用：注解处理通用格式校验，Validator 接口处理涉及业务规则的复杂校验。

###### 2. @Valid 和 @Validated 的区别是什么？

这两个注解都能触发校验，但有几个关键区别：

`@Valid` 是 JSR-303 标准注解，由 Bean Validation 规范定义，Spring 兼容支持。它可以用在参数、方法、字段、构造器上，**支持嵌套对象校验**（在嵌套字段上加 `@Valid` 即可级联校验），但**不支持分组校验**。

`@Validated` 是 Spring 自己的扩展注解，可以用在类、方法、参数上，最大的亮点是**支持分组校验**（通过 `value` 属性指定校验分组），另外加在类上还能让整个类的方法参数都被校验（Service 层常用这个特性）。

```java
// @Valid 用在控制器层参数校验
@RestController
public class UserController {

    @PostMapping("/users")
    public User createUser(@Valid @RequestBody User user) {
        return userService.save(user);
    }
}

// @Validated 用在 Service 层，或分组校验场景
@Service
@Validated  // 开启方法级别参数校验
public class UserService {

    // 分组校验：创建时只走 CreateGroup 的规则
    public User createUser(@Validated(CreateGroup.class) User user) {
        return userRepository.save(user);
    }

    // 分组校验：更新时走 UpdateGroup 的规则
    public User updateUser(@Validated(UpdateGroup.class) User user) {
        return userRepository.save(user);
    }
}
```

简单记：控制器层参数校验用 `@Valid` 就够了；需要分组校验或 Service 层方法参数校验，用 `@Validated`。

###### 3. BindingResult 的作用是什么？

`BindingResult` 是数据绑定和校验的结果容器，它把整个绑定过程中产生的所有错误都收集起来，让开发者可以精细控制错误的处理方式。

有几个重要注意事项：`BindingResult` 参数必须紧跟在被校验参数的后面，否则 SpringMVC 会直接抛出 `MethodArgumentNotValidException`，而不是把错误塞进 `BindingResult`。

```java
@Controller
public class UserController {

    @PostMapping("/register")
    public String registerUser(@Valid @ModelAttribute User user,
                               BindingResult result, Model model) {

        // 除了注解校验，还可以手动追加业务逻辑错误
        if (userService.existsByUsername(user.getUsername())) {
            result.rejectValue("username", "duplicate", "用户名已存在");
        }

        if (result.hasErrors()) {
            // 可以按字段获取错误
            List<FieldError> fieldErrors = result.getFieldErrors();
            fieldErrors.forEach(e ->
                log.error("字段: {}, 错误: {}", e.getField(), e.getDefaultMessage()));

            model.addAttribute("errors", result.getAllErrors());
            return "user/register-form";
        }

        userService.save(user);
        return "redirect:/success";
    }
}

// 如果不想每个方法都写 BindingResult，可以用全局异常处理统一拦截
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, String>> handleValidationException(
        MethodArgumentNotValidException ex) {

        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getFieldErrors().forEach(error ->
            errors.put(error.getField(), error.getDefaultMessage()));

        return ResponseEntity.badRequest().body(errors);
    }
}
```

###### 4. 如何自定义校验规则？

当内置注解（`@NotBlank`、`@Size`、`@Pattern` 等）满足不了业务需求时，可以自定义校验注解。步骤很固定：**定义注解 → 实现 ConstraintValidator → 应用注解**。

```java
// 第一步：定义校验注解
@Target({ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PhoneNumberValidator.class)
public @interface PhoneNumber {
    String message() default "手机号码格式不正确";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
    String regionCode() default "86";  // 可自定义参数
}

// 第二步：实现校验逻辑
public class PhoneNumberValidator implements ConstraintValidator<PhoneNumber, String> {
    private Pattern pattern;

    @Override
    public void initialize(PhoneNumber annotation) {
        // 根据参数初始化校验规则
        this.pattern = "86".equals(annotation.regionCode())
            ? Pattern.compile("^1[3-9]\\d{9}$")
            : Pattern.compile("^\\d{10,15}$");
    }

    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        if (value == null) return true;  // 空值交给 @NotNull 处理
        return pattern.matcher(value).matches();
    }
}

// 第三步：使用
public class User {
    @PhoneNumber(regionCode = "86", message = "请输入正确的中国大陆手机号")
    private String phone;
}
```

**进阶：跨字段校验**（比如"密码"和"确认密码"是否一致）需要把注解放在类级别：

```java
@Target({TYPE})
@Retention(RUNTIME)
@Constraint(validatedBy = PasswordMatchesValidator.class)
public @interface PasswordMatches {
    String message() default "两次密码不一致";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

public class PasswordMatchesValidator implements ConstraintValidator<PasswordMatches, User> {
    public boolean isValid(User user, ConstraintValidatorContext context) {
        if (!user.getPassword().equals(user.getConfirmPassword())) {
            // 把错误绑定到具体字段，方便前端展示
            context.disableDefaultConstraintViolation();
            context.buildConstraintViolationWithTemplate("密码确认不匹配")
                   .addPropertyNode("confirmPassword")
                   .addConstraintViolation();
            return false;
        }
        return true;
    }
}

@PasswordMatches  // 类级别注解
public class RegisterRequest {
    private String password;
    private String confirmPassword;
}
```

###### 5. JSR-303 规范是什么？

JSR-303 是 Java 规范请求（Java Specification Request），定义了 **Bean Validation 1.0 标准**，后续 JSR-349 是 1.1 版本，JSR-380 是 2.0 版本（Bean Validation 2.0）。这套规范的目标是：通过注解声明式地定义校验规则，让校验逻辑独立于业务代码和特定框架。

规范的实现体系：JSR 定义标准接口和注解 → Hibernate Validator 提供参考实现 → Spring Validation 在此基础上做框架整合。

Spring 通过 `LocalValidatorFactoryBean` 桥接 JSR 规范和 Spring 体系，委托给底层的 Hibernate Validator 实际执行校验：

```java
public class LocalValidatorFactoryBean implements Validator, ValidatorFactory {
    private ValidatorFactory validatorFactory;  // 持有 Hibernate Validator 工厂

    public void afterPropertiesSet() {
        this.validatorFactory = buildValidatorFactory();  // 初始化时构建
    }
}
```

所有自定义校验器都要实现规范定义的核心接口：

```java
public interface ConstraintValidator<A extends Annotation, T> {
    default void initialize(A constraintAnnotation) {}  // 初始化，可以读取注解参数
    boolean isValid(T value, ConstraintValidatorContext context);  // 实际校验逻辑
}
```

###### 6. 常用的校验注解有哪些？

JSR-303 标准提供了一批基础注解，Hibernate Validator 在此基础上扩展了更多实用注解：

**JSR-303 标准注解**（任何实现都支持）：

- `@NotNull` / `@Null`：非空 / 必须为空
- `@NotEmpty`：字符串、集合非空（不能是 null 或空集合/空字符串）
- `@Size(min, max)`：字符串长度或集合大小范围
- `@Min(value)` / `@Max(value)`：数值最小值/最大值
- `@DecimalMin` / `@DecimalMax`：小数的最小/最大值
- `@Pattern(regexp)`：正则表达式匹配
- `@Past` / `@Future`：必须是过去/将来时间
- `@Positive` / `@Negative`：必须是正数/负数

**Hibernate Validator 扩展注解**：

- `@NotBlank`：字符串非空且非纯空格（比 `@NotEmpty` 更严格）
- `@Email`：邮箱格式校验
- `@Length(min, max)`：字符串长度（只用于字符串，比 `@Size` 语义更明确）
- `@Range(min, max)`：数值范围（整合了 `@Min` + `@Max`）
- `@URL`：URL 格式校验

```java
public class Product {
    @NotBlank(message = "产品名称不能为空")
    @Length(min = 2, max = 50, message = "产品名称长度为 2-50 个字符")
    private String name;

    @NotNull(message = "价格不能为空")
    @DecimalMin(value = "0.0", inclusive = false, message = "价格必须大于 0")
    @Digits(integer = 10, fraction = 2, message = "价格最多保留 2 位小数")
    private BigDecimal price;

    @Email(message = "邮箱格式不正确")
    private String supplierEmail;

    @Pattern(regexp = "^(ACTIVE|INACTIVE)$", message = "状态值不合法")
    private String status;

    @Future(message = "过期时间必须是将来时间")
    private Date expiryDate;
}
```

###### 7. 如何实现分组校验？

分组校验解决的问题是：同一个实体类在不同场景下校验规则不同——比如创建用户时密码必填，更新用户时密码可以不传。

实现分为三步：**定义分组接口 → 在实体字段上指定分组 → 控制器用 `@Validated` 指定分组**。

```java
// 第一步：定义分组接口（就是普通的空接口，用作分组标识）
public interface CreateGroup {}
public interface UpdateGroup {}

// 第二步：在实体类字段上指定分组
public class User {
    // ID：更新时必须有，创建时由系统生成所以不需要
    @Null(groups = CreateGroup.class, message = "创建时不能传 ID")
    @NotNull(groups = UpdateGroup.class, message = "更新时 ID 不能为空")
    private Long id;

    @NotBlank(groups = CreateGroup.class, message = "用户名创建时必填")
    @Size(min = 3, max = 20, groups = {CreateGroup.class, UpdateGroup.class})
    private String username;

    @NotBlank(groups = CreateGroup.class, message = "密码创建时必填")
    @Size(min = 6, groups = CreateGroup.class)
    private String password;

    @Email(groups = {CreateGroup.class, UpdateGroup.class})
    private String email;
}

// 第三步：控制器指定分组
@RestController
@Validated  // 类级别加这个才能让 @Validated 在方法参数上生效
public class UserController {

    @PostMapping("/users")
    public ResponseEntity<User> createUser(
        @Validated(CreateGroup.class) @RequestBody User user) {
        return ResponseEntity.ok(userService.create(user));
    }

    @PutMapping("/users/{id}")
    public ResponseEntity<User> updateUser(
        @PathVariable Long id,
        @Validated(UpdateGroup.class) @RequestBody User user) {
        user.setId(id);
        return ResponseEntity.ok(userService.update(user));
    }
}
```

分组还支持继承——让 `DetailedGroup extends UpdateGroup`，那么 `DetailedGroup` 的校验会自动包含 `UpdateGroup` 的所有规则，方便做规则叠加。
