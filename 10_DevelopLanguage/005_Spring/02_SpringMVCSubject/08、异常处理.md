###### 1. SpringMVC 如何处理异常？

SpringMVC 的异常处理基于**责任链模式**，核心是 `HandlerExceptionResolver` 接口。当控制器方法抛出异常时，`DispatcherServlet` 捕获异常并依次遍历所有注册的 `HandlerExceptionResolver`，第一个能处理的解析器生效，返回一个 `ModelAndView` 表示异常已被处理。

```java
protected void doDispatch(HttpServletRequest request, HttpServletResponse response) {
    try {
        mv = ha.handle(processedRequest, response, mappedHandler.getHandler());
    } catch (Exception ex) {
        dispatchException = ex;
    }
    processDispatchResult(processedRequest, response, mappedHandler, mv, dispatchException);
}

protected ModelAndView processHandlerException(..., Exception ex) {
    for (HandlerExceptionResolver resolver : this.handlerExceptionResolvers) {
        ModelAndView exMv = resolver.resolveException(request, response, handler, ex);
        if (exMv != null) return exMv;  // 责任链，找到第一个能处理的就停
    }
    throw ex;  // 没有任何解析器处理，重新抛出，由 Servlet 容器兜底
}
```

默认注册的解析器执行顺序：
1. `ExceptionHandlerExceptionResolver`：处理 `@ExceptionHandler` 注解的方法
2. `ResponseStatusExceptionResolver`：处理带 `@ResponseStatus` 注解的异常
3. `DefaultHandlerExceptionResolver`：处理 Spring 内置异常（如 404、405、400 等）

这套机制实现了业务逻辑与异常处理的完全解耦，控制器方法只管正常流程，异常交给处理器统一处置。

###### 2. @ExceptionHandler 注解的作用是什么？

`@ExceptionHandler` 是方法级注解，用于在控制器内定义特定异常的处理逻辑，避免在业务方法里写大量 try-catch 块。

工作机制：当控制器抛出异常时，`ExceptionHandlerExceptionResolver` 扫描当前控制器类中所有带 `@ExceptionHandler` 的方法，选择**最匹配的那个**（异常类型最接近的优先）执行。如果当前控制器找不到，再去找 `@ControllerAdvice` 里的全局处理器。

`@ExceptionHandler` 标注的方法签名很灵活，可以接受多种参数（`HttpServletRequest`、`Model`、`WebRequest` 等），返回值也可以是 `ModelAndView`、`ResponseEntity`、视图名字符串等。

```java
@Controller
public class UserController {

    // 处理特定异常，可以一次处理多种类型
    @ExceptionHandler({UserNotFoundException.class, DataAccessException.class})
    public ResponseEntity<ErrorResponse> handleUserRelatedExceptions(
        Exception ex, WebRequest request) {

        ErrorResponse error = new ErrorResponse("USER_ERROR", ex.getMessage());
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(error);
    }
}
```

**局限**：默认只对当前控制器有效，要跨控制器共享异常处理逻辑，需要配合 `@ControllerAdvice`。

###### 3. @ControllerAdvice 注解的作用是什么？

`@ControllerAdvice` 是一个组件级注解，定义全局异常处理器，统一处理整个应用所有控制器的异常，同时还可以包含全局的 `@ModelAttribute` 和 `@InitBinder` 方法。

它支持通过属性精确控制作用范围：

```java
// 只对 com.example.api 包下的 @RestController 生效
@ControllerAdvice(annotations = RestController.class, basePackages = "com.example.api")
public class GlobalApiExceptionHandler {

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiError> handleAll(Exception ex) {
        // 全局异常处理
    }
}
```

底层原理：Spring 启动时扫描所有带 `@ControllerAdvice` 的类，由 `ExceptionHandlerExceptionResolver` 统一管理其中的 `@ExceptionHandler` 方法。当异常发生时，优先查找当前控制器的本地处理器，未找到时再遍历全局处理器。

###### 4. @RestControllerAdvice 和 @ControllerAdvice 的区别是什么？

`@RestControllerAdvice` 是 `@ControllerAdvice + @ResponseBody` 的组合注解，专为 RESTful API 场景设计。

核心区别就一点：`@RestControllerAdvice` 的方法返回值会自动序列化为 JSON/XML 写入响应体；`@ControllerAdvice` 的方法返回值默认走视图解析（可以返回视图名或 `ModelAndView`）。

```java
// 传统 MVC 应用：返回错误页面
@ControllerAdvice
public class WebExceptionHandler {

    @ExceptionHandler(Exception.class)
    public ModelAndView handleException(Exception ex) {
        ModelAndView mav = new ModelAndView("error/500");
        mav.addObject("message", ex.getMessage());
        return mav;
    }
}

// RESTful API 应用：返回 JSON 错误信息
@RestControllerAdvice
public class ApiExceptionHandler {

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiError> handleException(Exception ex) {
        ApiError error = new ApiError("INTERNAL_ERROR", "系统繁忙，请稍后重试");
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(error);
    }
}
```

前后端分离项目清一色用 `@RestControllerAdvice`，传统模板渲染项目用 `@ControllerAdvice`。

###### 5. SimpleMappingExceptionResolver 的作用是什么？

`SimpleMappingExceptionResolver` 是一个基于配置的异常解析器，通过预定义异常类到视图名的映射关系，实现不写一行代码的简单异常页面跳转。

它的核心逻辑是：根据异常类型查找映射表，找到对应视图名后，把异常对象放进 Model，渲染指定的错误页面。

```java
@Bean
public SimpleMappingExceptionResolver simpleMappingExceptionResolver() {
    SimpleMappingExceptionResolver resolver = new SimpleMappingExceptionResolver();

    Properties mappings = new Properties();
    mappings.put("com.example.NotFoundException", "error/404");   // 特定异常 → 404 页面
    mappings.put("java.lang.Exception", "error/generic");          // 兜底 → 通用错误页

    resolver.setExceptionMappings(mappings);
    resolver.setDefaultErrorView("error/default");  // 未匹配时的默认视图
    resolver.setExceptionAttribute("ex");           // 异常对象在 Model 中的属性名
    return resolver;
}
```

适用场景：传统模板渲染项目需要简单的异常 → 错误页映射。灵活性不如 `@ControllerAdvice`，现代项目用得比较少了，了解即可。

###### 6. 如何实现全局异常处理？

全局异常处理的标准方案是 `@RestControllerAdvice` + `@ExceptionHandler`，核心思路是：定义一个专门的异常处理类，覆盖应用中可能出现的各类异常，保证所有接口返回统一格式的错误响应。

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    private final Logger logger = LoggerFactory.getLogger(getClass());

    // 处理自定义业务异常
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiResponse<Void>> handleBusinessException(BusinessException ex) {
        ApiResponse<Void> response = ApiResponse.error(ex.getCode(), ex.getMessage());
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(response);
    }

    // 处理参数校验异常（@Valid 触发）
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiResponse<Map<String, String>>> handleValidationException(
        MethodArgumentNotValidException ex) {

        Map<String, String> errors = ex.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(
                FieldError::getField,
                FieldError::getDefaultMessage,
                (a, b) -> a  // 同一字段多个错误时取第一个
            ));
        return ResponseEntity.badRequest()
            .body(ApiResponse.error("VALIDATION_FAILED", "参数校验失败", errors));
    }

    // 处理资源不存在
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ApiResponse<Void>> handleNotFoundException(ResourceNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(ApiResponse.error("NOT_FOUND", ex.getMessage()));
    }

    // 兜底：处理所有未显式处理的异常
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Void>> handleUnknownException(Exception ex) {
        logger.error("未预期的系统异常: ", ex);  // 记录完整堆栈
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(ApiResponse.error("INTERNAL_ERROR", "系统繁忙，请稍后重试"));
        // 注意：生产环境绝对不能把详细错误信息返回给客户端
    }
}
```

几个关键注意事项：
- **异常匹配精确度优先**：子类异常优先于父类
- **兜底处理器不能省**：没有兜底，未知异常会直接返回 500 且没有统一格式
- **生产环境隐藏敏感信息**：通用异常处理不要把堆栈信息返回给客户端

###### 7. 如何自定义异常类？

自定义异常类通常继承 `RuntimeException`（运行时异常，不需要强制捕获），结合 `@ResponseStatus` 注解绑定 HTTP 状态码，再配合全局异常处理器使用。

```java
// 简单版：用 @ResponseStatus 直接绑定状态码
@ResponseStatus(value = HttpStatus.NOT_FOUND, reason = "用户不存在")
public class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(Long userId) {
        super("用户 ID: " + userId + " 不存在");
    }
}

// 企业级版：携带业务错误码，配合统一响应体
public class BusinessException extends RuntimeException {
    private final String errorCode;
    private final Object data;  // 可以携带额外上下文信息

    public BusinessException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
        this.data = null;
    }

    // getter...
}

// Service 层直接抛出
@Service
public class UserService {
    public User findById(Long id) {
        return userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
    }

    public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new BusinessException("INVALID_AMOUNT", "转账金额必须大于 0");
        }
        // ...
    }
}
```

自定义异常的价值在于：让代码更有表达力，同时把 HTTP 状态码与业务语义解耦——"用户不存在"就是 404，"余额不足"就是 400，而不是所有错误都返回 500。

###### 8. HandlerExceptionResolver 的作用是什么？

`HandlerExceptionResolver` 是 SpringMVC 异常处理机制的核心接口，定义了将异常转换为统一错误响应（`ModelAndView`）的标准契约，避免异常直接传播到 Servlet 容器。

接口只有一个方法，返回 `null` 表示当前解析器无法处理，异常传递给下一个：

```java
public interface HandlerExceptionResolver {
    @Nullable
    ModelAndView resolveException(HttpServletRequest request, HttpServletResponse response,
                                   @Nullable Object handler, Exception ex);
}
```

几个关键实现类：

- `ExceptionHandlerExceptionResolver`：通过反射找到并调用 `@ExceptionHandler` 方法，内部维护异常类型到方法的缓存，支持当前控制器和 `@ControllerAdvice` 两级查找
- `ResponseStatusExceptionResolver`：检查异常类的 `@ResponseStatus` 注解，调用 `response.sendError()` 设置对应的 HTTP 状态码
- `DefaultHandlerExceptionResolver`：处理 Spring 内置异常（如 `HttpRequestMethodNotSupportedException` → 405，`HttpMediaTypeNotSupportedException` → 415 等）

自定义解析器只需实现接口并注册为 Bean，通过 `Ordered` 接口控制优先级：

```java
@Component
public class CustomHandlerExceptionResolver implements HandlerExceptionResolver, Ordered {

    @Override
    public ModelAndView resolveException(HttpServletRequest request, HttpServletResponse response,
                                         Object handler, Exception ex) {
        if (ex instanceof CustomException) {
            response.setStatus(HttpStatus.BAD_REQUEST.value());
            return new ModelAndView("error/custom");
        }
        return null;  // 不能处理，传给下一个
    }

    @Override
    public int getOrder() {
        return 0;  // 最高优先级
    }
}
```
