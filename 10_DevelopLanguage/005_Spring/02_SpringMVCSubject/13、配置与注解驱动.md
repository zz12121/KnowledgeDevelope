###### 1. mvc:annotation-driven 的作用是什么？

`mvc:annotation-driven` 是 Spring MVC XML 配置中**启用注解驱动的核心标签**，加上它就等于激活了整套 Spring MVC 注解处理能力。

它背后会自动注册一系列关键 Bean：

- `RequestMappingHandlerMapping`：处理 `@RequestMapping` 注解，建立 URL 到控制器方法的映射
- `RequestMappingHandlerAdapter`：调用控制器方法，处理参数绑定、类型转换、返回值处理
- 默认的 `HandlerMethodArgumentResolver` 和 `HandlerMethodReturnValueHandler` 列表
- `ExceptionHandlerExceptionResolver` 等异常解析器

如果不加这个标签，所有的 `@Controller`、`@RequestMapping`、`@Valid` 等注解都不会生效。

```xml
<mvc:annotation-driven>
    <!-- 还可以自定义消息转换器 -->
    <mvc:message-converters>
        <bean class="org.springframework.http.converter.StringHttpMessageConverter">
            <property name="supportedMediaTypes">
                <list>
                    <value>text/plain;charset=UTF-8</value>
                    <value>application/json;charset=UTF-8</value>
                </list>
            </property>
        </bean>
    </mvc:message-converters>
</mvc:annotation-driven>
```

###### 2. @EnableWebMvc 注解的作用是什么？

`@EnableWebMvc` 是 Java Config 时代对应 `mvc:annotation-driven` 的注解版本，功能完全等价。

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Import(DelegatingWebMvcConfiguration.class)  // 关键：导入 MVC 配置支撑类
public @interface EnableWebMvc {}
```

它通过 `@Import` 引入 `DelegatingWebMvcConfiguration`，后者继承自 `WebMvcConfigurationSupport`，通过依赖注入收集所有 `WebMvcConfigurer` 实现类的配置，集中管理。

**一个重要的坑**：Spring Boot 项目**不要随便加 `@EnableWebMvc`**！Spring Boot 的自动配置有一个条件 `@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)`，加了 `@EnableWebMvc` 就会导入 `WebMvcConfigurationSupport`，触发这个条件，把 Spring Boot 的自动配置全部禁掉（包括静态资源处理、`/error` 映射等）。Spring Boot 项目里只需要实现 `WebMvcConfigurer` 接口做扩展就够了。

###### 3. WebMvcConfigurer 接口的作用是什么？

`WebMvcConfigurer` 是 Spring MVC 的核心配置扩展接口，提供一系列默认方法（Java 8 接口默认方法），开发者只需重写需要的方法来定制 MVC 行为，不需要覆盖不相关的配置。

这是典型的开闭原则——对扩展开放，对修改关闭：不改 Spring MVC 的核心代码，只通过接口扩展需要的部分。

主要方法分类：

- `addInterceptors()`：注册拦截器，配置路径匹配规则
- `addResourceHandlers()`：配置静态资源映射和缓存策略
- `addCorsMappings()`：全局 CORS 跨域配置
- `addViewControllers()`：简单页面跳转，不需要写控制器
- `configureMessageConverters()`：配置消息转换器（会替换所有默认转换器）
- `extendMessageConverters()`：在默认转换器基础上扩展（推荐）
- `addFormatters()`：注册类型转换器和格式化器
- `configureContentNegotiation()`：配置内容协商策略

```java
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {

    // 简单页面跳转：不写控制器直接映射 URL 到视图名
    @Override
    public void addViewControllers(ViewControllerRegistry registry) {
        registry.addViewController("/").setViewName("home");
        registry.addViewController("/login").setViewName("login");
        registry.addRedirectViewController("/index", "/");  // 重定向
    }
}
```

###### 4. 如何自定义 SpringMVC 配置？

自定义 Spring MVC 配置的标准方式是实现 `WebMvcConfigurer` 接口，按需重写方法：

```java
@Configuration
public class CustomWebMvcConfig implements WebMvcConfigurer {

    // 拦截器
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new LoggingInterceptor())
                .addPathPatterns("/api/**")
                .order(1);

        registry.addInterceptor(new AuthInterceptor())
                .addPathPatterns("/api/**")
                .excludePathPatterns("/api/auth/**")
                .order(2);
    }

    // 静态资源
    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        registry.addResourceHandler("/static/**")
                .addResourceLocations("classpath:/static/")
                .setCachePeriod(3600);
    }

    // 跨域
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("https://example.com")
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowCredentials(true);
    }

    // 日期格式化
    @Override
    public void addFormatters(FormatterRegistry registry) {
        registry.addFormatter(new DateTimeFormatterRegistrar()
            .setDateTimeFormatter(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss")));
    }
}
```

有多个 `WebMvcConfigurer` 实现时，Spring 会按 `@Order` 注解或 Bean 定义顺序依次调用每个实现的方法，配置是**累加的**，不会互相覆盖。

###### 5. 如何配置消息转换器？

消息转换器控制 `@RequestBody` / `@ResponseBody` 的序列化反序列化行为，有两种配置策略：

**`configureMessageConverters()`**：完全自定义，会**替换所有默认转换器**，包括 Spring Boot 自动注册的那些。

**`extendMessageConverters()`**：在默认转换器列表上**追加或修改**，保留已有配置。Spring Boot 项目推荐这种方式。

```java
@Configuration
public class MessageConverterConfig implements WebMvcConfigurer {

    // 推荐：扩展方式，保留默认配置，只替换 Jackson 转换器
    @Override
    public void extendMessageConverters(List<HttpMessageConverter<?>> converters) {
        // 找到默认的 Jackson 转换器并替换
        converters.removeIf(c -> c instanceof MappingJackson2HttpMessageConverter);

        MappingJackson2HttpMessageConverter customConverter =
            new MappingJackson2HttpMessageConverter();
        customConverter.setObjectMapper(customObjectMapper());
        converters.add(0, customConverter);  // 插到最前面，优先使用
    }

    @Bean
    public ObjectMapper customObjectMapper() {
        ObjectMapper mapper = new ObjectMapper();
        // 忽略未知字段，避免接口升级导致反序列化失败
        mapper.configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
        // 注册 Java 8 时间类型模块
        mapper.registerModule(new JavaTimeModule());
        // 日期序列化为字符串而非时间戳
        mapper.configure(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS, false);
        return mapper;
    }
}
```

###### 6. HttpMessageConverter 的作用是什么？

`HttpMessageConverter` 是 Spring MVC 处理 HTTP 请求/响应体数据转换的核心接口，负责 Java 对象和 HTTP 消息体之间的双向转换（读 = 反序列化，写 = 序列化）。

接口的四个核心方法：

```java
public interface HttpMessageConverter<T> {
    boolean canRead(Class<?> clazz, MediaType mediaType);   // 能否读这个类型
    boolean canWrite(Class<?> clazz, MediaType mediaType);  // 能否写这个类型
    T read(Class<? extends T> clazz, HttpInputMessage inputMessage);  // 反序列化
    void write(T t, MediaType contentType, HttpOutputMessage outputMessage);  // 序列化
}
```

常用的内置实现：

- `MappingJackson2HttpMessageConverter`：处理 `application/json`，最常用
- `StringHttpMessageConverter`：处理 `text/plain`、`text/html`
- `FormHttpMessageConverter`：处理 `application/x-www-form-urlencoded`（表单提交）
- `ByteArrayHttpMessageConverter`：处理字节数组（文件下载）
- `ResourceHttpMessageConverter`：处理 `Resource` 类型（文件流）

工作流程：`@ResponseBody` 标注的方法返回后，`RequestMappingHandlerAdapter` 遍历已注册的转换器列表，选择第一个 `canWrite()` 返回 true 的转换器执行序列化，然后写入响应体。选择依据是返回值类型 + 请求的 `Accept` 头。

###### 7. 如何配置内容协商？

内容协商让同一个接口根据客户端的偏好返回不同格式（JSON、XML 等），由 `ContentNegotiationManager` 管理协商策略。

```java
@Configuration
public class ContentNegotiationConfig implements WebMvcConfigurer {

    @Override
    public void configureContentNegotiation(ContentNegotiationConfigurer configurer) {
        configurer
            .favorParameter(true)          // 支持 URL 参数指定格式：?format=json
            .parameterName("format")
            .ignoreAcceptHeader(false)      // 同时也看 Accept 头
            .defaultContentType(MediaType.APPLICATION_JSON)  // 默认返回 JSON
            .mediaType("json", MediaType.APPLICATION_JSON)
            .mediaType("xml", MediaType.APPLICATION_XML);
    }

    @Override
    public void configureMessageConverters(List<HttpMessageConverter<?>> converters) {
        converters.add(new MappingJackson2HttpMessageConverter());
        converters.add(new Jaxb2RootElementHttpMessageConverter());
    }
}
```

配置后，客户端可以通过三种方式指定期望的响应格式：

- **URL 参数**：`GET /api/users?format=xml` → 返回 XML
- **Accept 头**：`Accept: application/json` → 返回 JSON
- **URL 扩展名**：`GET /api/users.xml`（不推荐，Spring 5.3 已废弃）

现代项目通常只用 `Accept` 头方式，最符合 HTTP 语义；API 客户端明确时可以只返回 JSON，不需要配置内容协商。
