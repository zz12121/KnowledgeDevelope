###### 1. 什么是微服务？

**微服务**是一种云原生架构方法论，把一个应用拆解为**一组小型、松耦合的独立服务**，每个服务单独运行在自己的进程里，围绕具体业务能力构建，通过轻量级机制（通常是 HTTP REST）相互通信。

说白了就是：不把所有东西堆一个大 jar 包里，而是按业务功能分成独立小服务，各自部署、各自扩展、互不影响。

从架构演进角度看，微服务是对**单体架构**和 **SOA** 的进一步演进——比 SOA 粒度更细，去掉了笨重的 ESB 总线，更轻量、更灵活。

###### 2. 什么是微服务架构？它有哪些优缺点？

微服务架构是把应用构建为**一组可以独立部署和扩展的小服务集合**，服务间通过轻量级通信协议协作。

**优点：**
- **技术多样性**：不同服务可以用最适合的技术栈，订单服务用 Java，报表服务用 Python
- **独立部署**：改一个服务只需重新部署它，不影响整个系统
- **精确扩展**：只对瓶颈服务扩容，"双11" 时单独扩展库存服务
- **容错隔离**：服务故障不会蔓延，单个服务挂了不会把整个系统拖垮

**缺点：**
- **分布式复杂性**：要处理网络延迟、分布式事务等问题
- **运维要求高**：管理大量服务实例，离不开容器化
- **数据一致性难**：跨服务数据同步只能靠最终一致性
- **测试复杂**：需要模拟分布式环境

###### 3. 单体应用、SOA、微服务架构的区别是什么？

三者是架构演进的三个阶段，核心差异在于拆分粒度和通信方式：

- **单体架构**：所有功能堆在一个进程里，共用同一个数据库，部署必须整体重启
- **SOA**：按业务拆分服务，但通过 ESB（企业服务总线）集中通信，总线本身容易成为瓶颈
- **微服务**：比 SOA 粒度更细，每个服务有自己的数据库，去掉了 ESB，服务间直接轻量通信

SOA 更关注企业级服务复用和集成，微服务聚焦应用解耦和快速交付。

###### 4. 微服务架构的核心特征有哪些？

- **组件化服务**：每个服务是可替换、可升级的独立单元
- **围绕业务能力**：按业务领域划分，而不是按技术层次（别再搞前端层/业务层/数据层了）
- **分散数据管理**：每个服务有自己的数据存储，不共享数据库
- **容错设计**：假设服务会故障，提前准备熔断机制
- **演进式设计**：服务可以随业务需求替换或重构

###### 5. 微服务的设计原则有哪些？

- **单一职责**：每个服务只负责一个明确的业务领域
- **服务自治**：独立开发、测试、部署和扩展
- **轻量级通信**：REST、gRPC 等跨语言协议
- **接口版本化**：稳定、向后兼容的 API 契约
- **有界上下文**：基于 DDD 划定服务边界
- **可观测性**：必须集成日志、监控和链路追踪

###### 6. 什么是服务拆分？如何进行合理的服务拆分？

服务拆分就是把单体应用按**业务边界**拆成多个微服务的过程。

合理拆分的依据：
- **领域驱动设计（DDD）**：按限界上下文划分，电商拆成订单、商品、用户等上下文
- **单一职责**：一个服务只负责一件事
- **松耦合高内聚**：服务间依赖最小，服务内功能高度相关
- **数据独立性**：每个服务有自己的数据源，绝对不能共享数据库

拆分方式用**绞杀者模式**——逐步从单体里剥离功能，不要推翻重来。

###### 7. 微服务架构中的领域驱动设计（DDD）是什么？

DDD 是一种**用领域模型指导复杂系统设计的方法论**，在微服务拆分中尤其重要。

**核心概念：**
- **限界上下文**：定义领域模型的边界，天然就是服务拆分的边界
- **聚合根**：数据修改的唯一入口，维护业务一致性。`Order` 就是聚合根
- **领域事件**：领域中发生的重要事件，服务间通信的载体。`OrderCreatedEvent` 触发库存扣减
- **值对象**：没有唯一标识的属性集合，如地址、金额

###### 8. 微服务架构中如何处理数据一致性问题？

微服务不追求强一致性，而是**最终一致性**。主要方案：

- **Saga 模式**：把长事务拆成多个本地事务，失败时执行补偿事务回滚（订单创建→库存扣减→支付，任一失败都有对应补偿）
- **TCC 模式**：Try（预留资源）→ Confirm（确认）→ Cancel（释放），业务手动控制
- **CQRS**：写操作和读操作分离，读模型可以异步同步
- **事件溯源**：存事件序列而非当前状态，通过重放事件重建状态

###### 9. 什么是服务编排和服务编制？

- **服务编制（Orchestration）**：通过中心化协调器控制流程，有一个"指挥官"统一调度各服务
- **服务编排（Choreography）**：服务间通过事件自主协作，没有中心，支付完成后发布事件，其他服务自己响应

编制适合有明确流程控制的场景，编排更符合微服务去中心化理念。实践中常用**混合模式**：核心流程用编制保证一致性，辅助流程用编排提升灵活性。

###### 10. 微服务架构中的康威定律是什么？

康威定律说的是："**组织设计的系统结构会复制该组织的沟通结构**"。

在微服务里的体现：
- 集中式大团队更容易做出单体应用
- 小团队自治更适合微服务架构
- 团队间沟通成本高的地方，往往就是服务边界所在

实践建议：按业务能力组建跨职能团队，一个团队负责"用户"相关的全部服务，从开发到运维都自己管。

###### 11. 微服务的粒度如何把握？

粒度没有标准答案，实用参考标准：
- 一个服务由 **2-5 人小团队**维护
- 代码可以在 **2 小时内**完全理解
- API 接口**不超过 10 个**

影响粒度的因素：业务复杂度、团队规模、性能要求（高频调用可以合并）、事务边界、部署频率。

###### 12. 微服务架构的演进路径是什么？

典型演进路径（别一上来就全微服务化）：

1. **单体阶段**：先快速验证业务
2. **垂直拆分**：前后端分离
3. **服务化探索**：把稳定、高复用的功能抽为服务（如用户服务）
4. **领域驱动拆分**：按 DDD 限界上下文深入拆分
5. **微服务成熟**：完善监控、治理、链路追踪等基础设施

全程用**绞杀者模式**，逐步替换，不要一口气全推倒重来。

###### 13. 微服务架构适合什么样的业务场景？

**适合：**
- 高并发，需要独立扩展核心服务（电商、直播平台）
- 快速迭代，多团队并行开发
- 复杂业务领域，适合 DDD 细分
- 技术异构需求，不同服务用不同技术栈

**不适合：**
- 业务简单稳定、团队规模小（微服务的治理成本会拖垮小团队）
- 强事务要求高的场景（分布式事务很麻烦）

📖 [[../../../../27_SpringCloudKnowledge/01_微服务架构基础/01、微服务架构基础概念#四、微服务设计原则]]

###### 14. 微服务架构的反模式有哪些？

- **分布式单体**：服务拆分了但还是高度耦合，必须一起部署，白拆了
- **过细粒度**：服务太小，网络开销比业务处理还大
- **共享数据库**：服务间通过数据库耦合，一改全改，违背自治原则
- **硬编码 IP 调用**：直接写死 IP，没用服务发现，扩缩容就麻烦了
- **薄弱监控**：没有链路追踪和指标监控，出问题根本找不到在哪

###### 15. 如何评估是否适合采用微服务架构？

评估维度：
- **业务复杂度**：业务足够复杂、需要快速迭代才值得上微服务
- **技术能力**：团队要能驾驭分布式系统、容器化、CI/CD
- **团队结构**：跨职能小团队才适合微服务自治
- **运维能力**：没有自动化运维体系，微服务会让运维崩溃
- **成本**：初期基础设施投入比单体高得多

**核心判断**：单体应用维护成本已高到难以接受、扩展性成瓶颈，且团队有能力驾驭，才是迁移的正确时机。
