###### 1. 说说微服务之间是如何独立通讯的？

微服务通信主要两种模式：

**同步调用**：
- **RESTful HTTP**：最常用，HTTP + JSON，简单直接，Spring Cloud 默认方案
- **RPC 框架**（gRPC、Dubbo）：二进制协议，性能更高，像调本地方法一样调远程

**异步消息**：
- **消息队列**（Kafka、RabbitMQ）：服务解耦，高吞吐，需处理消息可靠性
- **事件驱动**：发布/订阅，服务自主协作

选型原则：要实时响应结果 → 同步；允许延迟、追求高吞吐和解耦 → 异步。

###### 2. SpringCloud 有几种调用接口方式？

三种：

**RestTemplate + @LoadBalanced**：模板方式，需手动拼 URL，用服务名替代 IP。

```java
@Bean
@LoadBalanced
public RestTemplate restTemplate() { return new RestTemplate(); }

String result = restTemplate.getForObject("http://user-service/users/1", String.class);
```

**DiscoveryClient 手动获取实例**：自己从注册中心拿实例列表，自己实现负载均衡，灵活但繁琐，不常用。

**OpenFeign 声明式调用**（推荐）：接口 + 注解，最简洁，内置负载均衡。

###### 3. 什么是 RestTemplate？

RestTemplate 是 Spring 提供的**同步 HTTP 客户端工具类**，封装了 HTTP 连接管理，支持 GET、POST、PUT、DELETE 等方法，可配置消息转换器（JSON/XML）。

加上 `@LoadBalanced` 后，URL 中可直接用服务名，Ribbon 会拦截并做负载均衡。

###### 4. RestTemplate 的常用方法有哪些？

```java
// GET
User user = restTemplate.getForObject("http://api/users/1", User.class);
ResponseEntity<User> resp = restTemplate.getForEntity("http://api/users/1", User.class);
```

###### 5. 什么是 Ribbon？

Ribbon 是 Netflix 开源的**客户端负载均衡器**，嵌入在服务消费者中（不是独立服务），解决"从多个服务实例中选哪个"的问题。

核心特点：
- 从注册中心获取实例列表
- 基于负载均衡策略选择一个实例
- 检测实例健康状态，自动剔除故障节点

📖 [[../../../../27_SpringCloudKnowledge/03_服务调用与负载均衡/01、服务调用与负载均衡#三、Ribbon]]

---

###### 6. Ribbon 底层实现原理

Ribbon 通过**拦截器机制**实现客户端负载均衡：

```java
// LoadBalancerInterceptor 拦截 RestTemplate 请求
public ClientHttpResponse intercept(HttpRequest request, ...) {
    String serviceName = request.getURI().getHost(); // 提取服务名 "user-service"
    return loadBalancer.execute(serviceName, ...);   // 交 Ribbon 做负载均衡
}
```

**执行流程**：
1. 从 URL 提取服务名
2. 从注册中心查询健康实例列表
3. 根据负载均衡策略选一个实例
4. 将服务名替换为实际 IP:Port
5. 失败时自动重试其他实例

📖 [[../../../../27_SpringCloudKnowledge/03_服务调用与负载均衡/01、服务调用与负载均衡#三、Ribbon]]

###### 7. Ribbon 的负载均衡策略有哪些？

- **RoundRobinRule（轮询，默认）**：按顺序依次选择
- **RandomRule（随机）**：随机选择
- **RetryRule（重试）**：轮询失败后在指定时间内重试
- **BestAvailableRule（最低并发）**：选并发请求数最小的实例
- **WeightedResponseTimeRule（响应时间加权）**：响应越快被选中概率越高
- **AvailabilityFilteringRule（可用过滤）**：先过滤故障实例，再选并发较小的

```yaml
user-service:
  ribbon:
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RandomRule
```

###### 8. 如何自定义 Ribbon 的负载均衡策略？

实现 `IRule` 接口：

```java
public class MyLoadBalancingRule extends AbstractLoadBalancerRule {
    @Override
    public Server choose(Object key) {
        ILoadBalancer lb = getLoadBalancer();
        List<Server> allServers = lb.getAllServers();
        // 自定义逻辑：比如优先选配置高的服务器
        return selectByCustomLogic(allServers);
    }
}

// 注册
@Bean
public IRule myRule() { return new MyLoadBalancingRule(); }
```

###### 9. Ribbon 的饥饿加载是什么？

Ribbon 默认**懒加载**，首次请求时才初始化负载均衡客户端，可能导致第一次请求较慢。

**饥饿加载**在应用启动时就初始化，解决首次请求慢的问题：

```yaml
ribbon:
  eager-load:
    enabled: true
    clients: user-service,order-service
```

###### 10. Nginx 与 Ribbon 的区别

两者都做负载均衡，但层次不同：

**Nginx（服务端负载均衡）**：独立中间件，客户端 → Nginx → 服务实例（两跳），适合南北流量（外部入口），有单点风险。

**Ribbon（客户端负载均衡）**：嵌入在应用中，客户端 → 服务实例（直连），适合东西流量（服务间调用），去中心化无单点。

📖 [[../../../../27_SpringCloudKnowledge/03_服务调用与负载均衡/01、服务调用与负载均衡#三、Ribbon]]

###### 11. 为什么需要负载均衡？常见的负载均衡算法有哪些？

**为什么需要**：高可用（避免单点）、横向扩展（水平加实例）、平滑运维（滚动升级）。

**常见算法**：

- **轮询（Round Robin）**：请求按顺序依次分配，最简单，适合性能接近的服务器
- **加权轮询**：性能高的服务器权重更大，处理更多请求
- **随机**：随机选，实现简单
- **最少连接（Least Connections）**：选当前处理连接数最少的，适合请求耗时差异大的场景
- **哈希/源地址哈希**：根据 IP 或参数哈希，相同来源总打到同一台，实现会话保持
- **一致性哈希**：哈希算法改良版，节点变化时只重定位少量请求，避免雪崩

###### 12. 什么是客户端负载均衡和服务端负载均衡？

**服务端负载均衡**：独立设备或软件（F5、Nginx）位于客户端和服务之间，客户端请求先打到负载均衡器，再转发。优点：对客户端透明；缺点：单点风险、可能成瓶颈。

**客户端负载均衡**：逻辑嵌入在客户端应用中（Ribbon），客户端从注册中心拿实例列表，本地选一个直接连。优点：去中心化、延迟低；缺点：客户端复杂度高。

###### 13. 什么是 Netflix Feign？它的优点是什么？

Netflix Feign 是**声明式 HTTP 客户端**，核心目标是让调用远程服务像调本地方法一样简单。

主要优点：
- **声明式接口**：接口 + 注解定义调用，代码极简
- **集成 Ribbon**：天生负载均衡，无需额外编码
- **可维护性好**：契约集中在接口中

📖 [[../../../../27_SpringCloudKnowledge/03_服务调用与负载均衡/01、服务调用与负载均衡#四、OpenFeign]]

###### 14. 说说 Feign 实现原理

**动态代理 + 注解处理**，两个阶段：

**启动阶段**：`FeignClientFactoryBean` 扫描所有 `@FeignClient` 接口，通过 JDK 动态代理生成代理对象，注入 Spring 容器。

**调用阶段**：
1. `Contract` 组件解析方法注解，构建 `RequestTemplate`
2. `Encoder` 将参数填入模板，生成完整 HTTP 请求
3. `LoadBalancerFeignClient` 委托 Ribbon 做负载均衡，选目标实例
4. 真正的 HTTP 客户端发送请求
5. `Decoder` 将响应解码为返回类型

📖 [[../../../../27_SpringCloudKnowledge/03_服务调用与负载均衡/01、服务调用与负载均衡#四、OpenFeign]]

###### 15. Feign 是如何实现负载均衡的？

Feign 通过集成 **Ribbon** 实现，关键是 `LoadBalancerFeignClient`：

1. 调用时提取服务名（`"user-service"`）
2. 委托 `RibbonLoadBalancerClient.execute()` 做负载均衡
3. Ribbon 的 `ILoadBalancer` 从注册中心获取实例列表
4. 根据 `IRule` 策略选一个实例
5. `LoadBalancerFeignClient` 拿到实际 IP:Port，用底层 HTTP 客户端发请求

整个过程对 Feign 核心逻辑透明，Feign 只管契约，Ribbon 只管选实例。

###### 16. Feign 的性能优化方法有哪些？

**替换 HTTP 客户端**（默认 HttpURLConnection → OKHttp）：

```yaml
feign:
  okhttp:
    enabled: true
  httpclient:
    enabled: false
```

**配置超时时间**：

```yaml
feign:
  client:
    config:
      default:
        connectTimeout: 5000
        readTimeout: 5000
```

**开启 GZIP 压缩**：

```yaml
feign:
  compression:
    request:
      enabled: true
    response:
      enabled: true
```

**合理设置日志级别**：生产环境用 `BASIC`，不要用 `FULL`。

###### 17. Feign 与 RestTemplate 的区别是什么？

- **编程模型**：RestTemplate 模板化，手动拼 URL；Feign 声明式，接口 + 注解
- **负载均衡**：RestTemplate 需手动加 `@LoadBalanced`；Feign 内置，开箱即用
- **可读性**：RestTemplate HTTP 细节散落代码各处；Feign 契约集中在接口，更清晰

结论：现代微服务开发首选 Feign/OpenFeign，RestTemplate 更多用于简单场景或遗留代码。

###### 18. Ribbon 和 Feign 的区别？

它们不是竞争关系，是**协作关系**：

- **Ribbon**：负载均衡基础设施，解决"选哪个实例"
- **Feign**：声明式 HTTP 客户端，解决"如何优雅调用"，内部使用 Ribbon

Feign 是上层建筑，Ribbon 是底层基础。

###### 19. Ribbon 和 Feign 调用服务的区别

**Ribbon + RestTemplate**：
```java
@Bean @LoadBalanced
public RestTemplate restTemplate() { return new RestTemplate(); }

// 调用时手动拼 URL
return restTemplate.getForObject("http://user-service/users/{id}", User.class, userId);
```

**Feign**：
```java
@FeignClient(name = "user-service")
public interface UserServiceClient {
    @GetMapping("/users/{id}")
    User getUser(@PathVariable("id") Long userId);
}

// 注入后直接调，像本地方法
return userServiceClient.getUser(userId);
```

Feign 更简洁、类型安全。

###### 20. openFeign 是什么？

OpenFeign 是 **Spring Cloud 对 Netflix Feign 的增强封装**，Netflix Feign 已停更，OpenFeign 是当前推荐版本。

主要增强：
- 支持 Spring MVC 注解（`@RequestMapping`、`@PathVariable`），不用学 Feign 原生注解
- 集成 Spring Cloud LoadBalancer（替代 Ribbon）
- 与 Sentinel/Resilience4j 熔断组件更好集成
- `@EnableFeignClients` 自动扫描配置

###### 21. Feign 和 openFeign 有什么区别？

- **注解**：Netflix Feign 用原生注解（`@RequestLine`）；OpenFeign 支持 Spring MVC 注解
- **集成**：Netflix Feign 需自行集成 Spring Cloud 其他组件；OpenFeign 深度集成 Spring Cloud 生态
- **维护**：Netflix Feign 已停更；OpenFeign 持续活跃更新

在 Spring Cloud 中使用的都是 OpenFeign，不用管老版本的 Netflix Feign。

###### 22. openFeign 底层内部使用什么通信协议？

底层协议是 **HTTP/HTTPS**。

默认实现是 Java 标准库的 `HttpURLConnection`，可以替换为 OKHttp 或 Apache HttpClient 提升性能，但协议始终是 HTTP/HTTPS。

###### 23. 如何开启 openFeign 日志增强？

日志级别：`NONE`（默认）→ `BASIC`（请求/响应摘要）→ `HEADERS`（含请求头）→ `FULL`（全量）

```java
    }

```

```yaml
logging:
  level:
    com.yourcompany.client.UserServiceClient: DEBUG  # 必须设置为 DEBUG，否则 Feign 日志不输出
```

###### 24. openFeign 如何熔断降级？

以 Sentinel 为例：

```yaml
feign:
  sentinel:
    enabled: true
```

```java
// 编写降级类
@Component
public class UserServiceFallback implements UserServiceClient {
    @Override
    public User getUser(Long userId) {
        return new User(-1L, "降级用户", "服务暂时不可用");
    }
}

// Feign 客户端指定降级类
@FeignClient(name = "user-service", fallback = UserServiceFallback.class)
public interface UserServiceClient {
    @GetMapping("/users/{id}")
    User getUser(@PathVariable("id") Long userId);
}
```

调用失败达到阈值 → 熔断器打开 → 后续请求直接走 fallback，防止雪崩。

###### 25. Nacos 中的负载均衡底层是如何实现的？

Nacos 的负载均衡通过 `NacosRule`（实现 Ribbon 的 `IRule` 接口）完成：

1. **实例列表获取**：`NacosServiceDiscovery` 从 Nacos Server 订阅并缓存服务实例列表，Nacos Server 主动推送变更
2. **集群优先**：优先选择与消费者在同一集群（`cluster-name`）内的实例，降低跨机房延迟
3. **权重随机**：在同集群内，根据实例权重进行随机选择，权重越高被选中概率越大
4. **权重动态调整**：通过 Nacos 控制台可以动态修改实例权重，实现灰度发布
