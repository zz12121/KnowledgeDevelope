###### 1. 什么是服务注册与发现？

**服务注册与发现**是微服务架构中的核心机制，解决实例动态变化带来的寻址问题。

- **服务注册**：实例启动时向注册中心上报自己的元数据（IP、端口、服务名等），并通过心跳维持租约
- **服务发现**：消费者不写死目标地址，而是查注册中心获取可用实例列表，结合负载均衡选一个调

一句话：让服务找到彼此，而不是靠写死 IP。

###### 2. 服务注册与发现的实现方式有哪些？

两种模式：

**客户端发现模式（Spring Cloud 默认）**：消费者从注册中心拿到实例列表，自己在本地用 Ribbon/LoadBalancer 做负载均衡，直接连服务实例。代表：Eureka + Ribbon。优点：灵活、去中心化；缺点：客户端需要集成负载均衡逻辑。

**服务端发现模式**：消费者把请求打给负载均衡器（如 Nginx、API 网关），负载均衡器查注册中心并转发。代表：Kubernetes Service。优点：客户端完全透明；缺点：需要维护额外中间件。

📖 [[../../../../27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现#一、核心概念]]

###### 3. 什么是 Eureka？

Eureka 是 Netflix 开源的服务发现框架，遵循 **AP 模型**（高可用 + 分区容错）。

核心定位：基于 HTTP REST 协议，客户端以库的形式嵌入到服务中，通过注解驱动。包含两个组件：
- **Eureka Server**：注册中心，维护所有服务实例的注册表
- **Eureka Client**：集成在服务中，负责注册、续约和服务发现

📖 [[../../../../27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现#二、Eureka]]

---

###### 4. Eureka 的工作原理是什么？

Eureka 的机制围绕**心跳维持**和**注册表同步**展开：

1. **注册**：Client 启动时向 Server 发 POST 请求注册自身元数据
2. **续约（心跳）**：Client 默认每 **30 秒**发一次心跳，Server 收到后重置租约
3. **拉取列表**：Client 每 **30 秒**从 Server 拉取服务注册表，缓存到本地
4. **调用**：消费者用本地缓存 + Ribbon 做负载均衡选实例
5. **下线**：Client 关闭时发 DELETE 请求，Server 移除实例

**关键点**：由于缓存机制，服务状态更新最多有 90 秒延迟，这是 Eureka AP 设计的代价。

📖 [[../../../../27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现#二、Eureka]]

###### 5. Eureka 的核心概念有哪些（注册中心、服务提供者、服务消费者）？

**Eureka Server（注册中心）**：维护注册表（双层 ConcurrentHashMap），接收注册请求，管理租约，节点间异步同步数据。

```java
@SpringBootApplication
@EnableEurekaServer  // 启用注册中心
public class EurekaServerApplication { ... }
```

**服务提供者（Client）**：注册自身，维持心跳。

```yaml
eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka
  instance:
    lease-renewal-interval-in-seconds: 30    # 心跳间隔
    lease-expiration-duration-in-seconds: 90 # 超时时间
```

**服务消费者（Client）**：从注册中心拉列表，结合 Ribbon/Feign 负载均衡调用。

```java
@FeignClient(name = "user-service")
public interface UserServiceClient {
    @GetMapping("/users/{id}")
    User getUser(@PathVariable("id") Long id);
}
```

###### 6. Eureka 自我保护机制是什么？

**触发条件**：15 分钟内心跳丢失比例超过 **15%**（即不到 85% 的实例正常心跳）。

**工作机制**：进入保护模式后，不再剔除因长时间未心跳的服务实例，但继续接受新注册。网络恢复后自动退出。

**设计目的**：防止网络分区时把健康的服务实例误删。宁可暂时保留"可能已挂"的实例，也不大规模剔除导致消费者完全找不到服务。

```java
// AbstractInstanceRegistry 自我保护检查
public boolean isLeaseExpirationEnabled() {
    if (!selfPreservationModeEnabled) {
        return true; // 未开启自我保护，始终允许过期
    }
    return numberOfRenewsPerMinThreshold > 0
        && getNumOfRenewsInLastMin() > numberOfRenewsPerMinThreshold;
}
```

###### 7. Eureka 怎么实现高可用？

**对等集群**：Eureka Server 节点间互相注册，形成去中心化集群，数据双向同步。

```yaml
# 节点1 指向其他节点
eureka:
  client:
    service-url:
      defaultZone: http://eureka-node2:8762/eureka,http://eureka-node3:8763/eureka
```

**客户端容错**：
- 配置多个 Server 地址，连接失败自动切换
- 注册表本地缓存，Server 全挂时仍能用缓存数据
- 定时刷新确保数据最新

**区域感知**：支持 Region 和 Zone 概念，消费者优先调用同 Zone 的 Server，降低延迟。

###### 8. Eureka 的缓存机制是什么？

Eureka 采用**三级缓存架构**：

1. **ReadWriteCache**（读写缓存）：Guava Cache 实现，直接响应客户端查询
2. **ReadOnlyCache**（只读缓存）：定时（默认 30s）与读写缓存同步
3. **Client 本地缓存**：客户端内存缓存，减少网络调用

```java
// ResponseCacheImpl 缓存实现
private final LoadingCache<Key, Value> readWriteCacheMap;

// 注册表变更时失效缓存
public void invalidate(Key key) {
    readWriteCacheMap.invalidate(key);
}
```

三级缓存带来的数据延迟最多 90 秒，这是 Eureka AP 设计的代价——优先保证可用性，接受数据的最终一致性。

###### 9. Eureka Server 和 Eureka Client 的交互流程是什么？

**注册阶段**：Client 启动 → 初始化 EurekaClientConfigBean → 通过 DiscoveryClient 发 POST 注册请求 → Server 写入注册表并同步集群

**心跳阶段**：Client 启动心跳线程（`HeartbeatThread`），每 30s 发一次续约请求：

```java
scheduler.schedule(new TimedSupervisorTask("heartbeat", ..., new HeartbeatThread()),
    renewalIntervalInSecs, TimeUnit.SECONDS);
```

**拉取阶段**：Client 定时（30s）调 REST API 获取全量或增量注册表：

```java
Applications applications = eurekaTransport.queryClient.getDelta(); // 增量获取
```

**下线阶段**：Client 关闭时通过 shutdown hook 发 DELETE 请求：

```java
@PreDestroy
public void shutdown() { discoveryClient.shutdown(); }
```

###### 10. Eureka 的服务剔除机制是什么？

Eureka Server 通过定时任务（`EvictionTask`）检查实例心跳状态：

```java
public void run() {
    for (InstanceInfo instance : registry.getExpiredInstances()) {
        if (!isLeaseExpirationEnabled()) {
            return; // 自我保护模式下不剔除
        }
        registry.evict(instance);
    }
}
```

**判断标准**：默认 **90 秒**未收到心跳视为过期实例，触发自我保护时暂停剔除。

```yaml
eureka:
  instance:
    lease-expiration-duration-in-seconds: 90
  server:
    eviction-interval-timer-in-ms: 60000
```

###### 11. 为什么会出现 Nacos？

几个核心驱动因素：

- **Netflix 组件停更**：Eureka、Config 等进入维护模式，需要新的替代方案
- **阿里大规模实战验证**：阿里内部数万实例的微服务场景打磨出来，比 Eureka 更能抗
- **一站式需求**：Nacos = Eureka + Config + Bus，一个组件搞定注册 + 配置 + 推送，简化技术栈
- **功能更强**：AP/CP 可切换、原生配置推送、可视化控制台、持久化支持大规模集群

📖 [[../../../../27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现#三、Nacos]]

###### 12. Nacos 与 Eureka 的区别是什么？

核心差异对比：

**定位**：Eureka 专注服务发现；Nacos 是注册 + 配置一体的平台。

**CAP 模型**：Eureka 只支持 AP；Nacos 支持 AP/CP 切换。

**健康检查**：Eureka 客户端心跳（30s）；Nacos 临时实例心跳（5s）+ 非临时实例服务端主动探测（20s）。

**服务发现延迟**：Eureka 客户端定时拉取（30s 延迟）；Nacos 拉取 + 服务端主动推送（变更更及时）。

**配置管理**：Eureka 需要额外的 Spring Cloud Config；Nacos 原生支持，开箱即用。

**持久化**：Eureka 内存存储，集群规模受限；Nacos 支持 MySQL 持久化，可撑超大规模。

###### 13. Nacos 服务是如何判定服务实例的状态？

根据**实例类型**采用不同机制：

**临时实例（默认）**：
- 客户端 `BeatReactor` 线程每 **5 秒**向 Server 发心跳
- **15 秒**未收到心跳 → 标记为不健康
- **30 秒**未收到心跳 → **直接剔除**实例

**非临时实例**（`ephemeral=false`）：
- Server 端 `HealthCheckProcessor` 每 **20 秒**主动探测客户端健康端点
- 探测失败 → 标记不健康，**但不剔除**，等待恢复

一句话总结：临时实例是"临时工"，不汇报就被开除；非临时实例是"正式工"，领导主动来查，即使状态不佳也不立即开除。

###### 14. Nacos 的架构设计是什么？

Nacos 采用模块化、可扩展的分布式架构，分四层：

- **客户端层**：`NacosClient`，通过 HTTP/gRPC 与 Server 通信
- **服务端层**：Nacos Server 集群，包含 Naming Service（服务注册/发现/健康检查）和 Config Service（配置管理/推送/版本管理）
- **一致性协议层**：AP 场景用 Distro（自研最终一致性），CP 场景用 Raft（强一致性）
- **持久化层**：支持内嵌 Derby 或生产级 MySQL

`ServiceManager` 是注册中心的核心类，管理所有服务实例信息；`ConfigService` 是配置管理的核心接口。

📖 [[../../../../27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现#三、Nacos]]

###### 15. Nacos 的服务注册流程是什么？

1. **客户端发起注册**：`NacosDiscoveryAutoConfiguration` 创建 `NacosServiceRegistry`，调用 `register()` 把服务元数据封装成 `Instance` 对象，通过 `NamingProxy` 向 `/nacos/v1/ns/instance` 发 POST 请求

2. **服务端处理**：`InstanceController` 接收请求，`ServiceManager` 写入内存注册表（分层 ConcurrentHashMap）

3. **数据同步**：根据实例类型和 AP/CP 模式，选对应协议（Distro 或 Raft）同步到集群其他节点

4. **注册后行为**：
   - 临时实例 → 客户端启动 `BeatReactor` 心跳线程
   - 非临时实例 → 服务端启动主动健康检查
   - 发布注册事件通知所有订阅方（其他 Client 能及时感知变化）

📖 [[../../../../27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现#三、Nacos]]

###### 16. Nacos 的健康检查机制有哪些？

**机制一：客户端心跳（临时实例）**

客户端 `BeatReactor` 每 **5 秒**发心跳，Server 端 `HealthCheckTask` 检查超时：15s 未心跳 → 不健康，30s → 剔除。

优点：轻量，减少服务端压力；缺点：依赖客户端主动上报，客户端假死时无法及时感知。

**机制二：服务端主动探测（非临时实例）**

`HealthCheckProcessor` 每 **20 秒**主动向客户端 `/actuator/health` 等端点发请求。探测失败 → 标记不健康，但不剔除。

优点：服务端掌握主动权，更可靠；缺点：增加服务端负载。

###### 17. 为什么 Nacos 需要一致性协议？

Nacos 是分布式集群，多节点之间要保证数据一致，否则：

- 不同节点看到的服务实例列表不一样 → 消费者调到错误的实例
- 主节点宕机后数据丢失 → 服务发现失效
- 脑裂时多个节点都以为自己是主 → 数据混乱

一致性协议就是解决"多节点如何就某个值达成共识"的问题，Nacos 通过 `ConsistencyService` 接口抽象，屏蔽了具体协议的复杂性。

📖 [[../../../../27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现#三、Nacos]]

###### 18. 为什么 Nacos 选择了 Raft 以及 Distro？

Nacos 没有单一选择，根据场景同时用了两种协议：

**Raft（CP）**：用于非临时实例注册和配置管理。这类数据要求强一致性——配置必须准确，非临时实例上下线需严谨确认。Raft 通过 Leader 选举 + 日志复制，写请求必须经多数节点确认才返回成功。

**Distro（AP）**：用于临时实例注册。这类场景注重高可用，允许短暂不一致——大量无状态实例频繁增删，要求写入快、可用性高。Distro 是阿里自研最终一致性协议，每节点负责一部分数据，写任意节点后通过 Gossip 异步同步，可用性极高。

📖 [[../../../../27_SpringCloudKnowledge/02_服务注册与发现/01、服务注册与发现#三、Nacos]]

###### 19. 说说 Nacos 的 CP 模型和 AP 模型

Nacos 创新性地支持 CP/AP 切换，根据实例类型自动选择：

**AP 模型（默认，用于临时实例）**：
- 底层协议：Distro（异步复制，最终一致）
- 特点：网络分区时仍可注册新实例，保证高可用；可能读到过期数据

**CP 模型（用于非临时实例和配置管理）**：
- 底层协议：Raft（多数派同步，强一致）
- 特点：保证数据准确；网络分区时如无法与多数节点通信，则拒绝写入

**切换方式**：
```bash
curl -X PUT 'localhost:8848/nacos/v1/ns/operator/switches?entry=serverMode&value=CP'
```

注意：集群中存在非临时实例时，Nacos 会自动切到 CP 模式。

###### 20. Nacos 的命名空间、分组、集群是什么？

这是 Nacos 的**三层隔离模型**，层级关系：`命名空间 > 分组 > 服务 > 集群 > 实例`

**命名空间（Namespace）**：最顶层环境隔离，不同命名空间的服务和配置完全隔离，互相发现不到。典型用法：dev / test / prod 三个命名空间。

**分组（Group）**：同一命名空间内的逻辑分组。通常按项目或业务模块分组，如 `ORDER_GROUP`、`USER_GROUP`。

**集群（Cluster）**：同一服务的不同部署集合，通常按机房划分。消费者优先调用同集群的提供者，降低跨机房延迟。通过 `spring.cloud.nacos.discovery.cluster-name` 配置。
