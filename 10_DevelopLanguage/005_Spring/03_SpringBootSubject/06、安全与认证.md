###### 1. Spring Boot 如何集成 Spring Security?

Spring Boot 通过自动配置极大简化了 Spring Security 的集成。只需引入 `spring-boot-starter-security` 依赖，Spring Boot 就会自动开启所有 URL 的认证要求、提供默认表单登录页，并生成一个随机密码打印在启动日志里。

**核心集成步骤：**

第一步，引入依赖：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

第二步，创建安全配置类（现代推荐方式，定义 `SecurityFilterChain` Bean）：

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(authz -> authz
                .requestMatchers("/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .formLogin(withDefaults())
            .httpBasic(withDefaults());
        return http.build();
    }
}
```

**底层原理：** `@EnableWebSecurity` 导入 `WebSecurityConfiguration`，构建 `FilterChainProxy`（Spring Security 的核心过滤器链）。`SecurityFilterChain` Bean 定义 URL 模式与安全规则的映射，最终注入到 `FilterChainProxy` 中拦截所有请求。

第三步，配置用户认证源（生产环境实现 `UserDetailsService` 从数据库加载）：

```java
@Bean
public UserDetailsService userDetailsService() {
    var user = User.builder()
            .username("user")
            .password("{noop}password")  // {noop} 表示明文，生产禁止
            .roles("USER")
            .build();
    return new InMemoryUserDetailsManager(user);
}

@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();  // 生产必须用强密码编码器
}
```

###### 2. 如何实现基于角色的权限控制（RBAC）?

RBAC 的核心思想是把权限赋予角色，用户通过分配角色间接获得权限，实现权限与用户的解耦。

**数据模型设计（三张表）：**

```java
@Entity
public class User {
    @ManyToMany(fetch = FetchType.EAGER)
    private Set<Role> roles;
}

@Entity
public class Role {
    private String name;  // 如：ROLE_ADMIN, ROLE_USER
    @ManyToMany
    private Set<Permission> permissions;
}
```

**实现 `UserDetailsService`（连接用户数据和 Spring Security 的桥梁）：**

```java
@Service
public class CustomUserDetailsService implements UserDetailsService {

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        User user = userRepository.findByUsername(username)
                .orElseThrow(() -> new UsernameNotFoundException("用户不存在：" + username));
        return org.springframework.security.core.userdetails.User.builder()
                .username(user.getUsername())
                .password(user.getPassword())
                .authorities(user.getRoles().stream()
                        .map(r -> new SimpleGrantedAuthority(r.getName()))
                        .collect(Collectors.toList()))
                .build();
    }
}
```

**URL 级别权限控制：**

```java
http.authorizeHttpRequests(authz -> authz
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
    .requestMatchers("/api/**").hasAuthority("API_WRITE")
    .anyRequest().authenticated()
);
```

**方法级别权限控制（更精细）：**

```java
@Configuration
@EnableMethodSecurity(prePostEnabled = true)  // 开启方法级注解
public class MethodSecurityConfig {}

@Service
public class SomeService {
    @PreAuthorize("hasRole('ADMIN') or #userId == authentication.principal.id")
    public User getUserProfile(Long userId) { ... }  // 只有管理员或本人可访问

    @PostAuthorize("returnObject.owner == authentication.name")
    public SensitiveData getData() { ... }  // 方法执行后检查返回值权限
}
```

###### 3. 如何实现 JWT 认证?

JWT 是无状态认证机制，服务器不存储会话，所有必要信息都编码在令牌本身。与 Spring Security 集成主要通过自定义过滤器实现。

**JWT 认证完整流程：**

1. 客户端发送用户名/密码到登录接口
2. 服务端验证通过，生成 JWT 返回
3. 客户端后续请求携带 `Authorization: Bearer <token>`
4. 自定义过滤器验证 JWT，将认证信息写入 SecurityContext

**核心过滤器实现：**

```java
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {
        String authHeader = request.getHeader("Authorization");
        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            String jwt = authHeader.substring(7);
            String username = jwtUtil.extractUsername(jwt);

            if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
                UserDetails userDetails = userDetailsService.loadUserByUsername(username);
                if (jwtUtil.validateToken(jwt, userDetails)) {
                    // 构建认证对象，写入 SecurityContext（ThreadLocal 存储，本次请求有效）
                    UsernamePasswordAuthenticationToken authToken =
                        new UsernamePasswordAuthenticationToken(userDetails, null, userDetails.getAuthorities());
                    authToken.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
                    SecurityContextHolder.getContext().setAuthentication(authToken);
                }
            }
        }
        filterChain.doFilter(request, response);
    }
}
```

**Spring Security 配置关键三点：**

```java
http
    .csrf(csrf -> csrf.disable())  // JWT 不依赖 Cookie，禁用 CSRF
    .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))  // 无状态
    .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);  // 插入过滤器
```

###### 4. JWT 和 Session 的区别是什么?

JWT 和 Session 代表了两种截然不同的会话管理设计哲学，核心差异在于**状态存储位置**。

**Session 的核心逻辑（有状态）：** 服务器存储会话数据，客户端只持有 Session ID（相当于一把钥匙）。每次请求都要拿着这把钥匙到服务器查对应的会话信息。在集群环境下，需要 Redis 等集中式存储来共享会话，否则会出现同一个用户每次请求可能打到不同服务器、导致会话找不到的问题。

**JWT 的核心逻辑（无状态）：** 所有必要信息（用户 ID、角色、过期时间等）都编码在令牌本身，服务器只需验证签名。这意味着任何一台服务器实例都能独立验证 JWT，天然适合分布式架构。

**两者对比：**

- **扩展性**：JWT 天生支持水平扩展，Session 需要额外配置集中式存储
- **注销**：Session 直接删除记录立即生效；JWT 无法主动撤销，需要维护黑名单（引入了状态）或设置较短过期时间
- **安全风险**：Session 主要是 CSRF（浏览器自动携带 Cookie）；JWT 主要是 XSS（令牌存储在 LocalStorage 可能被窃取），且 JWT Payload 只是 Base64 编码，不是加密，不能放敏感信息
- **性能**：Session 每次都要查存储（可能成为瓶颈）；JWT 基于密码学计算，但令牌体积较大

**选型建议：** 传统 Web 应用、需要即时撤销权限 → Session；前后端分离 API、微服务、移动端、SSO → JWT。

###### 5. 如何实现 OAuth2.0 认证?

OAuth 2.0 是授权框架，允许第三方应用在获得用户明确授权后，有限度地访问用户在某服务上的资源，而无需暴露用户密码。

**四个核心角色：** 客户端（第三方应用）、资源所有者（用户）、授权服务器（如 GitHub、Google 登录端点）、资源服务器（存放受保护资源的 API）。

**授权码模式完整流程：** 用户访问应用 → 重定向到授权服务器 → 用户登录授权 → 携带授权码回调 → 应用用授权码换取访问令牌 → 建立本地会话。

**Spring Boot 作为 OAuth2 客户端（接入第三方登录）：**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-client</artifactId>
</dependency>
```

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          github:
            client-id: your-client-id
            client-secret: your-client-secret
            scope: user:read
```

```java
http.oauth2Login(withDefaults());  // 一行搞定 OAuth2 登录
```

**Spring Boot 作为 OAuth2 资源服务器（保护自己的 API，验证外部颁发的 JWT）：**

```java
http.oauth2ResourceServer(oauth2 -> oauth2.jwt(withDefaults()));
// 还需要配置 JWT 签发者 URI（issuer-uri）用于验证令牌签名
```

###### 6. 如何防止 CSRF 攻击?

CSRF 攻击诱骗已认证用户在不知情的情况下，向 Web 应用提交恶意请求，利用的是浏览器会自动携带目标站点 Cookie 的特性。

**Spring Security 的防护机制（同步器令牌模式）：**

1. 服务器为每个会话生成唯一、不可预测的 CSRF 令牌，存入 Session
2. 令牌嵌入到 HTML 表单隐藏域或 Meta 标签（供 Ajax 框架读取）
3. 客户端提交 POST/PUT/DELETE 请求时必须携带这个令牌
4. 服务器比较请求中的令牌和 Session 中的令牌，不匹配则拒绝

Spring Security 默认开启 CSRF 保护，Thymeleaf 等模板引擎会自动注入令牌。

**JWT 无状态 API 可以禁用：**

```java
http.csrf(csrf -> csrf.disable());
```

原因是 JWT 通常手动放在 `Authorization` 请求头，浏览器不会自动发送，CSRF 攻击的前提条件不成立。

###### 7. 如何防止 XSS 攻击?

XSS 攻击通过在网页中注入恶意脚本，窃取用户信息或执行未授权操作。

**Spring Security 通过 HTTP 安全响应头提供防御：**

```java
http.headers(headers -> headers
    .contentSecurityPolicy(csp -> csp.policyDirectives("default-src 'self'"))  // 最强防御
    .frameOptions(frame -> frame.sameOrigin())  // 防点击劫持
    .xssProtection(xss -> xss.block(true))      // 浏览器内置 XSS 过滤
);
```

**Content-Security-Policy 是核心防御：** 告诉浏览器只执行/渲染来自特定可信来源的脚本和样式，`default-src 'self'` 表示只信任同源内容，能有效阻止内联脚本和外域资源注入。

**最根本的防御是输出编码：** 将数据渲染给浏览器前，根据上下文（HTML/JS/URL）进行正确编码，把 `<` `>` 等危险字符转成 HTML 实体，使其不被解释为代码。Thymeleaf 默认会自动 HTML 转义。

Spring Security 的安全头是重要补充，但根源还是「永远不要信任用户输入，输出前必须编码」。

###### 8. 如何实现接口签名验证?

接口签名验证常用于开放 API 场景，确保请求来源合法、参数未被篡改，防止中间人攻击。

**核心机制：** 双方共享一个 secretKey，客户端用 secretKey 对请求参数计算签名，服务端用同样算法验签。

**客户端生成签名流程：**

1. 将所有请求参数（排除 sign 本身）按字典序排序
2. 拼接成 `key1=value1&key2=value2...` 格式的字符串
3. 加上时间戳（timestamp）、随机数（nonce）、secretKey 一起参与计算
4. 用 HMAC-SHA256 生成签名，放入请求头 `X-Api-Sign`

**服务端验签（拦截器实现）：**

```java
@Component
public class ApiSignInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request, ...) throws Exception {
        String clientSign = request.getHeader("X-Api-Sign");
        String timestamp  = request.getHeader("X-Api-Timestamp");
        String nonce      = request.getHeader("X-Api-Nonce");

        // 1. 检查时间戳，防重放攻击（5 分钟窗口）
        if (Math.abs(System.currentTimeMillis() - Long.parseLong(timestamp)) > 300_000) {
            throw new RuntimeException("请求已过期");
        }

        // 2. 检查 Nonce 是否已使用（Redis 短期缓存，防重复请求）
        if (nonceUsed(nonce)) throw new RuntimeException("重复请求");

        // 3. 按同样规则生成签名，比对
        String serverSign = generateSign(request, secretKey);
        if (!serverSign.equals(clientSign)) {
            throw new RuntimeException("签名无效");
        }

        return true;
    }
}
```

###### 9. 如何实现密码加密存储?

数据库里绝对不能明文存储密码，必须使用单向哈希函数处理。Spring Security 通过 `PasswordEncoder` 接口抽象密码编码过程。

**推荐使用 BCryptPasswordEncoder：**

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();  // 默认强度 10，也可用 12
    // 或者 DelegatingPasswordEncoder：支持多种算法，便于系统迁移
}
```

**存储密码时编码（用户注册/修改密码）：**

```java
String encodedPassword = passwordEncoder.encode(user.getPlainTextPassword());
user.setPassword(encodedPassword);
userRepository.save(user);
```

**验证密码时，Spring Security 自动调用 `passwordEncoder.matches(rawPassword, encodedPassword)`**，不需要手动处理。

**为什么 BCrypt 是黄金标准：**

- **自动生成随机盐**：相同密码每次哈希结果不同，彻底防彩虹表攻击
- **可调工作因子**：算法内置迭代次数（2^cost 次），成本因子每加 1 计算时间翻倍，可随硬件发展调整保持强度
- **自包含输出**：60 位字符串包含算法版本、成本因子、盐和哈希，验证时不需要额外存储盐

###### 10. BCrypt 加密的原理是什么?

BCrypt 是专为密码哈希设计的自适应哈希函数，其核心设计思路是**故意引入大量计算开销**来抵御暴力破解。

**三个关键机制：**

**自动随机盐：** 每次哈希都自动生成一个随机盐并参与计算，所以即使两个用户密码完全相同，哈希结果也完全不同。这彻底废掉了彩虹表攻击——攻击者无法预计算。

**工作因子（cost factor）：** 通常用 cost = 12 表示，意味着内部执行 2^12 = 4096 次迭代。成本因子每增加 1，计算时间约翻倍。随着 CPU 越来越快，可以通过提高 cost 来维持「破解一个密码需要花很多时间」的目标。

**自包含输出格式：** BCrypt 输出是一个 60 位字符串，格式类似 `$2a$12$R9h/cIPz0gi.URNNguA0wu.$L9YOuoSjwJkBWQ68N8bfp`，包含了算法版本（`$2a$`）、成本因子（`12`）、22 位盐和 31 位哈希。所以数据库里只需存这一个字段，验证时能自行解析出盐和成本因子重新计算比对。

**Spring Security 调用流程：**

- `encode(raw)` → 生成随机盐 → 用 Blowfish 密钥调度做 2^cost 次迭代 → 输出 60 位字符串
- `matches(raw, encoded)` → 从 encoded 解析出盐和 cost → 用相同参数重新哈希 → 比对结果

###### 11. 如何实现单点登录（SSO）?

SSO 允许用户在一个系统登录后，无需再次登录即可访问其他信任的应用。现代 SSO 基于 OAuth 2.0 / OpenID Connect 协议实现。

**核心架构：** 建立一个中央认证服务器（Keycloak、Spring Authorization Server 等），所有应用都信任它，登录请求都由它处理。

**SSO 工作流程：**

1. 用户访问应用 A，未登录，重定向到中央认证中心
2. 用户在认证中心完成登录
3. 认证中心重定向回应用 A，携带授权码
4. 应用 A 用授权码换取访问令牌 + ID 令牌，建立本地会话
5. 用户访问应用 B，未登录，重定向到认证中心
6. 认证中心检测到用户已登录（Cookie），直接重定向回应用 B，不再要求输入密码
7. 应用 B 完成令牌交换，建立本地会话

**每个应用的配置（指向同一个认证中心）：**

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          my-sso-provider:
            provider: keycloak
            client-id: app-a-client-id
            authorization-grant-type: authorization_code
            scope: openid, profile
        provider:
          keycloak:
            issuer-uri: https://central-auth-server.com/realms/my-realm
```

###### 12. 如何实现动态权限管理?

动态权限是指权限规则存储在数据库，管理员可以在不重启应用的情况下实时修改 URL-角色映射，立即生效。

**实现方案：**

第一步，数据库存储权限规则（URL pattern、HTTP method、required role）。

第二步，实现 `FilterInvocationSecurityMetadataSource`，动态从数据库加载访问某 URL 所需的角色：

```java
@Component
public class DynamicSecurityMetadataSource implements FilterInvocationSecurityMetadataSource {

    @Override
    public Collection<ConfigAttribute> getAttributes(Object object) {
        String url = ((FilterInvocation) object).getRequest().getRequestURI();
        List<String> requiredRoles = permissionService.findRequiredRolesByUrl(url);
        return SecurityConfig.createListFromCommaDelimitedString(String.join(",", requiredRoles));
    }
}
```

第三步，实现 `AccessDecisionManager`，判断当前用户是否具备所需角色：

```java
@Component
public class DynamicAccessDecisionManager implements AccessDecisionManager {

    @Override
    public void decide(Authentication auth, Object object,
                       Collection<ConfigAttribute> configAttributes) {
        for (ConfigAttribute attr : configAttributes) {
            for (GrantedAuthority authority : auth.getAuthorities()) {
                if (attr.getAttribute().equals(authority.getAuthority())) return;  // 有权限，放行
            }
        }
        throw new AccessDeniedException("权限不足");
    }
}
```

第四步，通过 `withObjectPostProcessor` 注入到 `FilterSecurityInterceptor`。

**性能优化：** 权限规则加 Redis 缓存，管理员修改权限后发布事件清缓存，使新规则生效。
