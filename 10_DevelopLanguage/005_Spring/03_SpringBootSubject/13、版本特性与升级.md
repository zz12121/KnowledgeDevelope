###### 1. Spring Boot 2.x 有什么新特性?

Spring Boot 2.x 是一次重大版本跨越，底层切换到 Spring Framework 5，最低要求 JDK 8，整体定位是现代化、响应式、云原生。主要带来了这几方面变化：

**响应式编程支持**：引入了完整的 WebFlux 响应式 Web 框架，基于 Project Reactor，底层运行在 Netty 上，支持非阻塞 I/O。控制器方法可以直接返回 `Mono`/`Flux` 类型，配合 R2DBC 等响应式数据库驱动，整个请求链路都能做到非阻塞。

**HTTP/2 支持**：内嵌 Tomcat 8.5+、Jetty、Undertow 都支持 HTTP/2，在配置里一行开关：`server.http2.enabled: true`，利用多路复用降低连接开销。

**Actuator 大改版**：端点统一收敛到 `/actuator` 路径下，不再默认全量暴露，需要明确配置；集成了 Micrometer 作为指标门面，天然支持接入 Prometheus、Grafana 等监控系统，`/actuator/prometheus` 端点直接输出标准格式。

**默认连接池换成 HikariCP**：1.x 默认用 Tomcat JDBC Pool，2.x 换成了性能更好的 HikariCP，基准测试下吞吐量提升明显。

**配置属性绑定改进**：引入了 `Binder` API，属性绑定更灵活；配置属性来源可追踪，通过 `ConfigurationPropertiesReportEndpoint` 能看到每个配置从哪来的。

**其他升级**：Thymeleaf 升到 3.x，性能提升 50%+ ；支持 Kotlin 协程；Quartz 调度器通过 `spring-boot-starter-quartz` 开箱即用。

###### 2. Spring Boot 2.x 与 1.x 有什么区别?

两个版本之间的差距比较大，从底层框架到默认行为都有变化，核心差异如下：

**JDK 要求**：1.x 最低支持 JDK 6，2.x 要求 JDK 8+，因为用到了 `java.time`、Lambda、Stream 等特性。这也意味着升到 2.x 必须先把 JDK 升上去。

**Spring Framework 版本**：1.x 基于 Spring 4.x，2.x 基于 Spring 5.x。Spring 5 带来的最大变化就是响应式编程栈（WebFlux），2.x 完整继承了这一能力。

**内嵌服务器**：1.x 用 Tomcat 7.x/8.x，2.x 升到 Tomcat 8.5+，原生支持 HTTP/2；新增了 WebFlux 下的 Netty 内嵌服务器选项。

**默认连接池**：1.x 默认用 Tomcat JDBC Pool，2.x 改为 HikariCP，性能显著更好。

**Actuator 端点**：1.x 端点直接暴露（如 `/health`、`/info`），默认大量端点开放，有安全风险；2.x 统一收到 `/actuator` 路径下，默认只暴露 `health` 和 `info`，安全性更好。

**配置属性**：1.x 属性绑定相对简单；2.x 引入了 `PropertyMapper` API，支持来源追踪，属性松散绑定（kebab-case、camelCase、underscore 都能匹配）更宽松。

**循环依赖**：2.6.0 之前默认允许循环依赖，2.6.0 起默认禁止，应用启动时若检测到循环依赖直接报错，强制开发者重构设计。

###### 3. Spring Boot 哪个版本禁止循环依赖?

从 **Spring Boot 2.6.0** 开始，默认禁止循环依赖。

**原因**：循环依赖往往意味着组件设计不合理，两个 Bean 互相依赖是违背单一职责原则的信号。强制禁止是为了推动开发者改善设计，而不是靠三级缓存把问题"绕过去"。

**表现**：启动时如果检测到循环依赖，会抛 `BeanCurrentlyInCreationException`，并给出提示：

```
The dependencies of some of the beans in the application context form a cycle:
   userService → orderService → userService
```

**底层机制**：在 `AbstractAutowireCapableBeanFactory.doCreateBean()` 里，是否将未完成初始化的 Bean 放入三级缓存，受 `allowCircularReferences` 字段控制：

```java
boolean earlySingletonExposure = (mbd.isSingleton() && this.allowCircularReferences
    && isSingletonCurrentlyInCreation(beanName));
if (earlySingletonExposure) {
    addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));
}
```

2.6.0 起 `allowCircularReferences` 默认为 `false`，三级缓存失效，循环依赖无法被自动解决。

**临时兼容**：如果项目短期内无法重构，可以在 `application.properties` 里加：

```yaml
spring:
  main:
    allow-circular-references: true
```

但这只是临时过渡方案，根本解法还是通过提取共用逻辑、引入事件驱动等手段消除循环依赖。

###### 4. Spring Boot 3.x 有什么新特性?

Spring Boot 3.x 是又一次大跨越，基于 Spring Framework 6，对运行环境和生态的要求都提高了。

**最低 Java 17**：3.x 要求 JDK 17+，可以用 Record、Sealed Class、Text Block 等现代 Java 特性，代码更简洁。

**全面迁移到 Jakarta EE**：Java EE 捐给 Eclipse 基金会后改名 Jakarta EE，包名从 `javax.*` 全部变成 `jakarta.*`。比如 `javax.servlet.HttpServlet` 变成了 `jakarta.servlet.HttpServlet`，`javax.persistence.Entity` 变成了 `jakarta.persistence.Entity`。这是升级 3.x 最大的改动量所在。

**GraalVM Native Image 原生支持**：3.x 内置了对 GraalVM 原生编译的支持，通过 AOT（提前编译）处理反射、代理等动态特性的配置，可以把 Spring Boot 应用编译成原生可执行文件，启动时间从秒级降到毫秒级，内存占用也大幅减少。

**自动配置注册方式变化**：从 `META-INF/spring.factories` 迁移到 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件，格式更简洁，每行一个类名。

**安全配置简化**：OAuth2 支持更内聚，`OAuth2AuthorizationServerConfiguration` 提供了更多默认配置，减少样板代码。

**问题诊断增强**：`/actuator/sbom`（软件物料清单）端点，可以查看应用的完整依赖图谱，辅助安全审计。

###### 5. Spring Boot 3.x 与 2.x 有什么区别?

3.x 和 2.x 之间的差异主要体现在运行环境要求和底层生态的切换上，直接影响到升级成本。

**Java 版本**：2.x 最低 JDK 8，3.x 要求 JDK 17+。不升 JDK 就没法用 3.x。

**EE 规范**：这是最大的破坏性变更。2.x 基于 Java EE（`javax.*`），3.x 基于 Jakarta EE 9+（`jakarta.*`）。所有用到 Servlet、JPA、Bean Validation、JAX-RS 等规范的地方，导入语句都要改一遍。自动化迁移工具（如 OpenRewrite）能帮你批量替换，但仍然要仔细验证。

**GraalVM 原生编译**：2.x 需要手动配置 `native-image-maven-plugin`，支持有限；3.x 内置完整的 AOT 支持，`mvn spring-boot:build-image` 直接出原生镜像，用起来流畅得多。

**Spring Framework 版本**：2.x 基于 Spring Framework 5.x，3.x 基于 6.x。一些过时的 API 被移除，某些行为也有变化。

**自动配置文件位置**：2.x 在 `META-INF/spring.factories` 里注册自动配置类，3.x 改到 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`。两者暂时共存，但 `spring.factories` 的方式在未来可能会被废弃。

###### 6. 如何从 Spring Boot 2.x 升级到 3.x?

2.x 升 3.x 的工作量不小，需要系统化地分步推进，不能一上来就改 pom 版本。

**第一步：升级 JDK**。先把 JDK 升到 17，确保编译、测试都能通过。这步完成后才有资格升 Spring Boot。

**第二步：升级 Spring Boot 版本**。把父 POM 改到 3.x：

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.0.0</version>
</parent>
```

**第三步：处理 Jakarta EE 迁移**。这是工作量最大的部分——把所有 `javax.*` 的导入改成 `jakarta.*`。OpenRewrite 工具提供了自动化迁移方案，能处理大部分情况：

```xml
<plugin>
    <groupId>org.openrewrite.maven</groupId>
    <artifactId>rewrite-maven-plugin</artifactId>
    <configuration>
        <activeRecipes>
            <recipe>org.openrewrite.java.spring.boot3.UpgradeSpringBoot_3_0</recipe>
        </activeRecipes>
    </configuration>
</plugin>
```

**第四步：更新第三方依赖**。与 Jakarta EE 相关的依赖要换成支持 Jakarta 命名空间的版本，比如 Hibernate 6.x、Flyway 9.x 等。

**第五步：检查废弃 API**。Spring Boot 3.x 移除了一些在 2.x 中标记为 `@Deprecated` 的 API，编译报错逐一解决。

**第六步：处理循环依赖**。3.x 继承了 2.6+ 的循环依赖禁止策略，升级后如果有循环依赖报错，需要重构。

**第七步：完整测试**。跑全量集成测试，用 Actuator 的 `/health` 端点验证运行状态，在测试环境充分验证后再发布生产。

###### 7. Jakarta EE 与 Java EE 的区别是什么?

两者本质上是同一套企业 Java 规范在不同历史阶段的延续，但归属方和包名发生了变化。

**历史背景**：Java EE 一直由 Oracle 管理，2017 年 Oracle 将其捐赠给 Eclipse 基金会，改名 Jakarta EE。由于 Oracle 保留了 `javax` 商标，Eclipse 基金会不能继续使用 `javax.*` 命名空间，所以 Jakarta EE 9 起把所有包名从 `javax.*` 改成了 `jakarta.*`。

**核心区别**：

最直接的影响就是包名变了：
- `javax.servlet.HttpServlet` → `jakarta.servlet.HttpServlet`
- `javax.persistence.Entity` → `jakarta.persistence.Entity`
- `javax.validation.Valid` → `jakarta.validation.Valid`

API 功能本身基本一致，但迁移时所有的 import 语句都要改。

**功能增强**：Jakarta EE 9+ 在 MicroProfile 集成、容错、配置管理方面有更好的支持；JPA 在 Jakarta EE 中增加了更灵活的缓存机制；安全配置也有简化。

**对 Spring Boot 的影响**：Spring Boot 3.x 只支持 Jakarta EE，放弃了对 Java EE（`javax.*`）的兼容，这就是为什么从 2.x 升到 3.x 时改包名的工作量那么大。

###### 8. GraalVM Native Image 的优势是什么?

GraalVM Native Image 是一种 AOT（提前编译）技术，把 Java 应用在编译期就生成机器码，不再依赖 JVM 运行时解释执行，在云原生场景下有显著优势。

**启动速度极快**：传统 Spring Boot 应用启动通常需要几秒，原生镜像可以降到 100ms 以内甚至更短。这对 Serverless 场景（FaaS）非常关键，冷启动时间直接影响用户体验和成本。

**内存占用大幅降低**：不需要 JVM 的运行时开销（JIT 编译器、解释器、类加载器等），内存占用通常能减少 50% 以上。容器化部署时镜像体积也更小，只包含必要的代码和依赖。

**镜像体积小**：AOT 编译时会通过静态分析移除不可达代码，最终生成的可执行文件比 Fat JAR + JVM 的组合小很多，拉取镜像更快。

**限制和代价**：
- **不支持动态类加载**：反射、代理等动态特性需要在编译期通过 `reflect-config.json` 显式声明，Spring Boot 3.x 的 AOT 处理帮你自动生成了大部分配置，但某些三方库可能还需要手动补充
- **编译时间长**：AOT 编译要做全程序分析，比普通 Java 编译慢很多，不适合频繁开发迭代
- **调试相对困难**：原生镜像的调试体验不如 JVM 上丰富

**Spring Boot 3.x 如何支持**：3.x 内置了 AOT 处理器，在构建阶段分析应用的 Bean 定义、反射用法、代理配置，自动生成必要的 GraalVM 配置文件，然后用 `native-maven-plugin` 编译成原生可执行文件：

```bash
mvn spring-boot:build-image -Pnative  # 构建原生镜像
```
