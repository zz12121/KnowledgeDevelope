###### 1. Spring Boot 的自动配置原理是什么？

Spring Boot 自动配置的核心思路是：**读取预定义的配置类列表 → 条件过滤 → 注册满足条件的 Bean**。

完整链路：

```
@SpringBootApplication
    ↓ 包含 @EnableAutoConfiguration
    ↓ 通过 @Import 导入 AutoConfigurationImportSelector
    ↓ 从 META-INF/spring.factories（旧版）
      或 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports（新版）
      读取 200+ 个候选自动配置类
    ↓ 经过 @Conditional 系列条件注解过滤
    ↓ 注册满足条件的 Bean
```

`AutoConfigurationImportSelector` 是核心实现类，它实现了 `DeferredImportSelector` 接口（延迟导入，确保用户定义的 Bean 优先生效），在 `selectImports()` 方法里完成加载、去重、条件过滤的全部逻辑。

自动配置类通常长这样：

```java
@Configuration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class }) // 类路径有 DataSource 才生效
@ConditionalOnMissingBean(DataSource.class)                           // 用户没自定义才生效
public class DataSourceAutoConfiguration {
    @Bean
    public DataSource dataSource(DataSourceProperties properties) {
        return properties.initializeDataSourceBuilder().build();
    }
}
```

`@ConditionalOnMissingBean` 是最常见的条件注解，作用是"如果你自己配了就用你的，你没配我帮你配"，这就是自动配置不会覆盖用户配置的原因。

###### 2. @SpringBootApplication 注解的作用是什么？

`@SpringBootApplication` 是个组合注解，展开来是三个：

```java
@SpringBootConfiguration  // 本质是 @Configuration，标记为配置类
@EnableAutoConfiguration  // 启用自动配置机制
@ComponentScan            // 扫描同级包及子包下的 @Component、@Service、@Controller 等
public @interface SpringBootApplication {}
```

三个注解各司其职：`@SpringBootConfiguration` 让主类能作为配置类使用；`@EnableAutoConfiguration` 触发整个自动配置流程；`@ComponentScan` 让 Spring 能发现你写的所有 Bean。

可以通过属性自定义行为：

```java
@SpringBootApplication(
    scanBasePackages = "com.example",  // 指定扫描范围
    exclude = {DataSourceAutoConfiguration.class}  // 排除特定自动配置
)
```

###### 3. @EnableAutoConfiguration 注解的工作原理是什么？

`@EnableAutoConfiguration` 通过 `@Import(AutoConfigurationImportSelector.class)` 把自动配置导入器引入进来，核心工作都在 `AutoConfigurationImportSelector` 里完成。

它还包含 `@AutoConfigurationPackage`，这个注解会把主类所在的包注册为"自动配置基础包"，让一些需要扫描用户包的自动配置类（比如 JPA 的 Entity 扫描）知道该扫哪里：

```java
@AutoConfigurationPackage  // 注册主类所在包
@Import(AutoConfigurationImportSelector.class)  // 导入自动配置选择器
public @interface EnableAutoConfiguration {}
```

使用 `DeferredImportSelector`（延迟导入）的原因很关键：延迟到用户定义的 Bean 都处理完之后再导入自动配置，确保 `@ConditionalOnMissingBean` 能正确判断用户有没有自定义配置，不会让自动配置覆盖用户配置。

###### 4. 如何自定义 Spring Boot 的自动配置？

自定义自动配置的标准流程分四步：

**第一步：定义配置属性类**

```java
@ConfigurationProperties(prefix = "myapp.service")
public class MyServiceProperties {
    private String name = "default";
    private int timeout = 5000;
    // getters/setters
}
```

**第二步：写业务服务类（略，正常的 Service 即可）**

**第三步：写自动配置类**

```java
@Configuration
@ConditionalOnClass(MyService.class)          // 有 MyService 类才生效
@EnableConfigurationProperties(MyServiceProperties.class)
public class MyServiceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean                 // 用户没有自己配才自动配
    public MyService myService(MyServiceProperties props) {
        return new DefaultMyService(props.getName(), props.getTimeout());
    }
}
```

**第四步：注册自动配置类**

Spring Boot 3.x 在 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 里添加一行：

```
com.example.MyServiceAutoConfiguration
```

之后其他项目引入这个 Starter 依赖，配好 `myapp.service.name=xxx`，就能直接用 `@Autowired MyService` 了。

###### 5. 如何禁用特定的自动配置类？

有三种常用方式：

**方式一：注解属性排除（代码层面）**

```java
@SpringBootApplication(exclude = {
    DataSourceAutoConfiguration.class,
    SecurityAutoConfiguration.class
})
public class Application {}
```

**方式二：配置文件排除（最灵活，不用改代码）**

```yaml
spring:
  autoconfigure:
    exclude:
      - org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
      - org.springframework.boot.autoconfigure.security.servlet.SecurityAutoConfiguration
```

**方式三：通过条件属性按需开关**

```java
@Configuration
@ConditionalOnProperty(name = "myapp.feature.enabled", havingValue = "true")
public class MyFeatureConfig {
    // 只有配置了 myapp.feature.enabled=true 才生效
}
```

实际项目里，测试环境不想启动某些组件（比如数据库、Redis）时，用配置文件排除最方便，不需要修改代码。

###### 6. @Conditional 系列注解有哪些，作用是什么？

`@Conditional` 系列注解是自动配置的决策核心，控制 Bean 什么时候创建、什么时候不创建。

常用注解速查：

- `@ConditionalOnClass`：类路径有某个类才生效 → 用来判断某个技术栈是否引入
- `@ConditionalOnMissingBean`：容器里没有某个 Bean 才创建 → 让用户配置优先，自动配置兜底
- `@ConditionalOnProperty`：某个配置属性满足条件才生效 → 通过配置文件开关功能
- `@ConditionalOnWebApplication`：是 Web 应用才生效 → Web 相关组件的自动配置
- `@ConditionalOnBean`：容器里已有某个 Bean 才生效 → 依赖其他 Bean 的配置

底层都基于 `Condition` 接口实现，`matches()` 方法返回 true 才生效。

自定义条件注解也很简单，实现 `Condition` 接口，然后用 `@Conditional(自定义条件类.class)` 标注就好：

```java
public class OnProductionCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        return Arrays.asList(context.getEnvironment().getActiveProfiles()).contains("prod");
    }
}

@ConditionalOnProduction  // 只在 prod 环境生效
@Bean
public DataSource productionDataSource() { ... }
```

###### 7. Spring Boot 启动流程是怎样的？

`SpringApplication.run()` 方法是启动入口，整个流程分几个关键阶段：

1. **创建 SpringApplication 实例**：推断应用类型（SERVLET/REACTIVE/NONE），从 `spring.factories` 加载初始化器和监听器
2. **环境准备**：加载 `application.properties`/`.yml`，处理命令行参数，发布 `EnvironmentPreparedEvent`
3. **打印 Banner**（可自定义 banner.txt）
4. **创建 ApplicationContext**：根据应用类型选择对应的上下文实现
5. **刷新上下文**（最重要的一步）：调用 `AbstractApplicationContext.refresh()`，完成 Bean 定义加载、自动配置触发、所有单例 Bean 的实例化
6. **执行 Runner**：所有 `CommandLineRunner` 和 `ApplicationRunner` 按 `@Order` 顺序执行
7. **发布 `ApplicationReadyEvent`**，应用完全就绪

其中 `refresh()` 是整个框架最核心的方法，里面依次完成了 BeanFactory 初始化、BeanPostProcessor 注册、MessageSource 初始化、事件广播器初始化、非懒加载单例 Bean 实例化等所有重要操作。

###### 8. SpringApplication.run() 方法做了什么？

`SpringApplication.run()` 是整个应用的启动入口，它负责协调所有组件完成应用的初始化：

核心步骤：
1. 创建计时器开始计时
2. 获取所有 `SpringApplicationRunListener`，发布 `starting` 事件
3. 准备 `Environment`（加载配置文件、命令行参数）
4. 打印 Banner
5. 创建 `ApplicationContext`
6. 应用所有 `ApplicationContextInitializer` 对上下文做初始化处理
7. 调用 `refreshContext()`，这是最重的步骤——完成全部 Bean 的加载与实例化
8. 发布 `started` 事件
9. 执行所有 `Runner`（`CommandLineRunner` + `ApplicationRunner`）
10. 发布 `running` 事件，返回 `ApplicationContext`

如果在任何阶段抛出异常，会调用 `handleRunFailure()` 发布失败事件并打印错误信息。

###### 9. 如何自定义 Spring Boot 的启动 Banner？

Banner 是启动时打印的那段 ASCII 艺术字，自定义方式有几种：

**方式一：在 `src/main/resources` 放 `banner.txt` 文件**，Spring Boot 启动时自动读取并打印，支持 `${spring-boot.version}` 等变量。

**方式二：配置文件指定位置**

```properties
spring.banner.location=classpath:my-banner.txt
```

**方式三：编程式自定义**

```java
SpringApplication app = new SpringApplication(Application.class);
app.setBanner((env, sourceClass, out) -> {
    out.println("╔═══════════════════════╗");
    out.println("║     MY APPLICATION    ║");
    out.println("╚═══════════════════════╝");
});
app.run(args);
```

**方式四：直接关掉**

```java
app.setBannerMode(Banner.Mode.OFF);
```

###### 10. 如何在 Spring Boot 启动时执行特定代码？

有四种常用方式，按使用频率排序：

**`@PostConstruct`**（在 Bean 初始化后立即执行）：适合 Bean 自身的初始化逻辑，执行时机早于 Runner。

**`CommandLineRunner`**（最简单）：

```java
@Component
@Order(1)
public class DataInitRunner implements CommandLineRunner {
    @Override
    public void run(String... args) throws Exception {
        // 初始化基础数据
    }
}
```

**`ApplicationRunner`**（参数处理更强大）：

```java
@Component
@Order(2)
public class CacheWarmupRunner implements ApplicationRunner {
    @Override
    public void run(ApplicationArguments args) throws Exception {
        if (args.containsOption("skip-warmup")) return; // 支持 --key=value 格式
        warmUpCache();
    }
}
```

**`@EventListener(ApplicationReadyEvent.class)`**（应用完全就绪后触发，比 Runner 稍晚）：

```java
@EventListener(ApplicationReadyEvent.class)
public void onReady() {
    log.info("应用完全就绪，可以处理请求了");
}
```

执行顺序：`@PostConstruct`（各 Bean 初始化时）→ `Runner`（按 `@Order` 顺序）→ `ApplicationReadyEvent`。

###### 11. CommandLineRunner 和 ApplicationRunner 的区别是什么？

两者都是在 Spring Boot 应用完全启动后执行代码的接口，区别只在于**参数的处理方式**：

`CommandLineRunner` 拿到的是 `String...`，就是 main 方法原始的字符串数组，需要自己解析。`ApplicationRunner` 拿到的是 `ApplicationArguments` 对象，对命令行参数做了封装，特别是支持 `--key=value` 格式的选项参数，可以用 `args.containsOption("key")` 和 `args.getOptionValues("key")` 方便地获取：

```java
// 比如启动命令是：java -jar app.jar --db.init=true someArg

@Override
public void run(ApplicationArguments args) throws Exception {
    args.getSourceArgs();                   // ["--db.init=true", "someArg"]
    args.getOptionNames();                  // ["db.init"]
    args.getOptionValues("db.init");        // ["true"]
    args.getNonOptionArgs();                // ["someArg"]
}
```

简单场景用 `CommandLineRunner` 就够了；需要解析结构化的命令行参数时用 `ApplicationRunner`。两者都支持 `@Order` 控制执行顺序。

###### 12. Spring Boot 的条件注解如何实现？

条件注解的底层基于 Spring 的 `Condition` 接口：

```java
public interface Condition {
    boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata);
}
```

`ConditionContext` 提供了判断所需的全部信息：BeanFactory、Environment（配置属性）、ClassLoader（类路径检查）、ResourceLoader（资源检查）。

Spring Boot 内置的条件注解（如 `@ConditionalOnClass`）对应各自的 `Condition` 实现类，在 `matches()` 里做具体的条件判断，比如 `OnClassCondition` 就是用 `ClassUtils.isPresent()` 检查类路径里有没有目标类。

自定义条件注解三步走：

```java
// 1. 实现 Condition 接口
public class OnProductionCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        return Arrays.asList(context.getEnvironment().getActiveProfiles()).contains("prod");
    }
}

// 2. 定义注解，关联条件实现
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Conditional(OnProductionCondition.class)
public @interface ConditionalOnProduction {}

// 3. 使用
@Bean
@ConditionalOnProduction
public DataSource productionDataSource() { ... }
```
