###### 1. Spring Boot 支持哪些打包方式?

Spring Boot 提供了灵活的打包机制，根据部署环境和需求选择合适的方式。

**四种核心打包格式：**

- **可执行 JAR（默认）**：Fat JAR，包含内嵌服务器和所有依赖，`java -jar` 直接运行，微服务首选
- **传统 WAR 包**：部署到外部 Servlet 容器（Tomcat/JBoss），传统企业环境使用
- **Docker 镜像**：容器化打包，环境一致性最好，云原生架构首选
- **分层 JAR（Spring Boot 2.3+）**：优化 Docker 镜像构建，分离依赖和应用代码，提升构建缓存利用率

**高级打包特性：**

- **瘦身 JAR（spring-boot-thin-launcher）**：依赖外部化，JAR 文件极小，启动时从 Maven 仓库下载依赖
- **GraalVM 原生镜像**：编译为原生可执行文件，启动时间从秒级降到毫秒级，内存占用大幅减少（Spring Boot 3.x 原生支持）

###### 2. JAR 和 WAR 的区别是什么?

JAR 和 WAR 是 Spring Boot 应用的两种主要打包格式，代表了不同的部署模式。

**核心架构差异：**

可执行 JAR 是「自包含」的——把 Tomcat 塞进去了，一个 JAR 就是一个完整的应用，`java -jar` 直接跑；WAR 包是「依赖外部」的——需要 Tomcat、JBoss 等 Servlet 容器来承载，多个应用共享一个容器。

**结构对比：**

```
spring-boot-app.jar（可执行 JAR）         app.war（传统 WAR）
├── META-INF/MANIFEST.MF（主类 JarLauncher） ├── META-INF/
├── BOOT-INF/                               ├── WEB-INF/
│   ├── classes/（应用代码）                 │   ├── classes/（应用代码）
│   └── lib/（所有依赖）                     │   ├── lib/（依赖）
└── org/springframework/boot/loader/        │   └── web.xml（可选）
    └── JarLauncher.class（Spring Boot 加载器） └── 静态资源/
```

**其他差异：**

- **启动速度**：JAR 更快（内嵌服务器优化）；WAR 依赖外部容器启动时间
- **资源占用**：每个 JAR 应用独立占用内存；WAR 多应用共享容器
- **依赖管理**：JAR 所有依赖打包在内，独立完整；WAR 部分依赖由容器提供（如 Servlet API）

**选择策略：**

- 微服务架构、容器化部署、云原生 → **JAR**
- 传统企业环境、已有 Servlet 容器投资、遗留系统集成 → **WAR**

###### 3. 如何将 Spring Boot 应用打包成可执行 JAR?

Spring Boot Maven 插件的 `repackage` 目标会将普通 JAR 重新打包为可执行的 Fat JAR。

**Maven 配置：**

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>2.7.0</version>
</parent>

<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <configuration>
                <executable>true</executable>   <!-- 生成可执行脚本，Linux 下可直接 ./app.jar 运行 -->
                <layers>
                    <enabled>true</enabled>     <!-- 启用分层，优化 Docker 构建 -->
                </layers>
            </configuration>
        </plugin>
    </plugins>
</build>
```

**打包和运行：**

```bash
# 打包（跳过测试）
mvn clean package -DskipTests

# 运行
java -jar target/myapp-0.0.1-SNAPSHOT.jar

# 带配置参数运行
java -jar target/myapp.jar --spring.profiles.active=prod --server.port=8081
```

**分层 JAR 原理（Spring Boot 2.3+）：**

将 JAR 分为四层，变化频率从低到高：

1. `dependencies`：稳定的第三方依赖
2. `spring-boot-loader`：Spring Boot 加载器代码
3. `snapshot-dependencies`：快照依赖
4. `application`：应用代码和资源（最频繁变化）

Docker 构建时，只有应用代码变了需要重建最后一层，前三层走缓存，大幅提升构建速度。

###### 4. 如何将 Spring Boot 应用部署到 Tomcat?

将 Spring Boot 部署到外部 Tomcat 需要将打包方式改为 WAR，并做以下调整：

**第一步，修改打包方式并排除内嵌 Tomcat：**

```xml
<packaging>war</packaging>

<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <exclusions>
            <exclusion>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-starter-tomcat</artifactId>  <!-- 排除内嵌 Tomcat -->
            </exclusion>
        </exclusions>
    </dependency>
    <dependency>
        <groupId>javax.servlet</groupId>
        <artifactId>javax.servlet-api</artifactId>
        <scope>provided</scope>  <!-- 由外部 Tomcat 提供 -->
    </dependency>
</dependencies>
```

**第二步，启动类继承 `SpringBootServletInitializer`（这是关键，替代 web.xml）：**

```java
@SpringBootApplication
public class Application extends SpringBootServletInitializer {

    @Override
    protected SpringApplicationBuilder configure(SpringApplicationBuilder builder) {
        return builder.sources(Application.class);  // 外部容器启动入口
    }

    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);  // 保留，支持 java -jar 运行
    }
}
```

**第三步，部署：**

```bash
mvn clean package -DskipTests
cp target/myapp.war $TOMCAT_HOME/webapps/
$TOMCAT_HOME/bin/startup.sh
```

访问路径：`http://localhost:8080/myapp`（由 WAR 文件名决定）。

**注意：** 应用路径由 WAR 文件名决定，可以重命名为 `ROOT.war` 让应用在根路径下访问。

###### 5. 如何使用 Docker 部署 Spring Boot 应用?

Docker 部署提供环境一致性，是现代微服务的标配部署方式。推荐使用多阶段构建减小镜像体积。

**多阶段 Dockerfile（推荐）：**

```dockerfile
# 第一阶段：构建
FROM maven:3.8.6-openjdk-17 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B   # 先下载依赖，充分利用 Docker 缓存
COPY src ./src
RUN mvn clean package -DskipTests

# 第二阶段：运行（镜像更小）
FROM openjdk:17-jre-slim
WORKDIR /app

# 创建非 root 用户，增强安全性
RUN groupadd -r spring && useradd -r -g spring spring
USER spring

COPY --from=builder --chown=spring:spring /app/target/*.jar app.jar

ENV JAVA_OPTS="-Xmx512m -Xms256m -XX:+UseG1GC"

HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:8080/actuator/health || exit 1

EXPOSE 8080
ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar /app/app.jar"]
```

**构建和运行：**

```bash
docker build -t my-spring-app:1.0.0 .

docker run -d -p 8080:8080 \
  -e "SPRING_PROFILES_ACTIVE=prod" \
  --name my-app \
  my-spring-app:1.0.0
```

**生产最佳实践：**

- 使用具体版本的基础镜像，不用 `latest`
- 非 root 用户运行（安全）
- 合理配置 JVM 内存参数（`-Xmx` 设为容器内存的 70-80%）
- 配置健康检查端点

###### 6. 如何编写 Dockerfile?

一个生产级的 Dockerfile 需要关注安全性、构建效率和运行时性能。

**几个核心技巧：**

**利用 Docker 缓存层：** 把变化频率低的步骤放前面。先复制 pom.xml 下载依赖（这一层很少变），再复制源码编译，这样依赖下载那一层会被缓存，代码改动后重新构建时只需要重新编译。

**多阶段构建减小镜像：** 构建阶段用 JDK（含编译工具），运行阶段只需 JRE，镜像体积可以减小 50% 以上。

**分层 JAR 进一步优化：**

```dockerfile
COPY --from=builder /app/target/*.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract  # 解压分层

FROM openjdk:17-jre-slim
COPY --from=builder /app/dependencies/ ./
COPY --from=builder /app/spring-boot-loader/ ./
COPY --from=builder /app/snapshot-dependencies/ ./
COPY --from=builder /app/application/ ./           # 只有这层经常变化
ENTRYPOINT ["java", "org.springframework.boot.loader.JarLauncher"]
```

**`.dockerignore` 配置（减少构建上下文）：**

```gitignore
**/target/
**/.git/
**/.idea/
**/node_modules/
Dockerfile
```

###### 7. 如何使用 Docker Compose 编排多个服务?

Docker Compose 用来定义和管理多容器的本地开发/测试环境，一个 `docker-compose.yml` 文件描述整个应用栈。

**典型微服务编排（应用 + MySQL + Redis + Nginx）：**

```yaml
version: '3.8'

services:
  app-service:
    image: my-spring-app:1.0.0
    environment:
      - SPRING_PROFILES_ACTIVE=docker
      - SPRING_DATASOURCE_URL=jdbc:mysql://mysql-db:3306/appdb
      - SPRING_REDIS_HOST=redis-cache
    ports:
      - "8080:8080"
    depends_on:
      mysql-db:
        condition: service_healthy  # 等待 MySQL 健康再启动
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
      interval: 30s
      retries: 3
    networks:
      - app-network
    volumes:
      - ./logs:/app/logs  # 日志持久化到宿主机

  mysql-db:
    image: mysql:8.0
    environment:
      MYSQL_DATABASE: appdb
      MYSQL_ROOT_PASSWORD: rootpass
    volumes:
      - mysql-data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      retries: 10
    networks:
      - app-network

  redis-cache:
    image: redis:7-alpine
    networks:
      - app-network

volumes:
  mysql-data:

networks:
  app-network:
    driver: bridge
```

**常用命令：**

```bash
docker-compose up -d          # 后台启动所有服务
docker-compose ps             # 查看服务状态
docker-compose logs -f app    # 实时查看日志
docker-compose down -v        # 停止并删除容器和数据卷
docker-compose up -d --scale app-service=3  # 水平扩展
```

**多环境管理：** `docker-compose.yml`（公共配置）+ `docker-compose.override.yml`（开发环境覆盖）+ `docker-compose.prod.yml`（生产环境）。

###### 8. 如何部署到 Kubernetes?

Kubernetes 是生产环境容器编排的标准平台，提供自动调度、扩缩容、滚动更新、服务发现等能力。

**核心资源配置（Deployment + Service + HPA）：**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: spring-boot-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: spring-boot-app
  template:
    spec:
      containers:
      - name: app
        image: my-registry.com/spring-app:1.0.0
        resources:
          requests:
            memory: "512Mi"
            cpu: "250m"
          limits:
            memory: "1Gi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 60
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 30
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: spring-app-hpa
spec:
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70  # CPU 超过 70% 开始扩容
```

**敏感配置管理：** 使用 ConfigMap 存储普通配置，Secret 存储密码/密钥（Base64 编码），通过环境变量或 Volume 挂载注入容器。

**常用运维命令：**

```bash
kubectl apply -f deployment.yaml                 # 应用配置
kubectl set image deployment/app app=image:v2    # 滚动更新
kubectl rollout undo deployment/app              # 回滚上一版本
kubectl logs -l app=spring-boot-app -f           # 查看日志
```

###### 9. 如何实现应用的灰度发布?

灰度发布让新版本先对少量用户开放，验证没问题后再全量推广，降低发布风险。

**金丝雀发布（基于流量权重）：**

通过 Istio VirtualService 配置流量比例：90% 流量走稳定版，10% 走金丝雀版。观察错误率、延迟等指标，逐步提高金丝雀流量比例，直到 100%。

**基于 Header 的路由（精准灰度）：**

```yaml
# Istio VirtualService 配置
http:
- match:
  - headers:
      canary:
        exact: "true"      # 请求头带 canary:true 的走新版
  route:
  - destination:
      host: myapp
      subset: canary
- route:                   # 其他流量走稳定版
  - destination:
      host: myapp
      subset: stable
```

这种方式可以做到精准灰度：只有特定用户（如内部测试人员、VIP 用户）的请求走新版本。

**Flagger 自动化金丝雀：** 可以根据成功率、P99 延迟等指标自动决定是否继续推进灰度或回滚，实现全自动的渐进式发布。

###### 10. 如何实现应用的蓝绿部署?

蓝绿部署通过同时维护两个版本（蓝色=当前生产，绿色=新版本），实现零停机切换和快速回滚。

**工作流程：**

1. 蓝色（Blue）版本承载所有生产流量
2. 部署绿色（Green）新版本，完成后做功能和性能验证
3. 通过 Service 的 selector 标签，将流量从蓝色切换到绿色（切换几乎即时）
4. 观察一段时间，如果有问题，把 selector 改回蓝色立即回滚
5. 确认稳定后，缩减/删除旧蓝色版本

**K8s 流量切换核心：**

```yaml
# 只需修改 Service 的 selector 标签
apiVersion: v1
kind: Service
spec:
  selector:
    app: myapp
    version: blue   # 改成 green 即切换流量
```

**自动化切换脚本关键步骤：**

```bash
# 部署新版本（Green）
kubectl apply -f deployment-green.yaml

# 等待 Green 就绪
kubectl rollout status deployment/myapp-green --timeout=300s

# 切换流量
kubectl patch service myapp-service -p '{"spec":{"selector":{"version":"green"}}}'

# 观察一段时间后，缩减旧版本
kubectl scale deployment/myapp-blue --replicas=0
```

**数据库兼容性注意事项：** 蓝绿切换期间两个版本可能同时访问数据库，Schema 变更需要向前兼容（新增字段允许 NULL，不立即删除旧字段）。
