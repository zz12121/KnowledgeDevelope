###### 1. Spring Boot 支持哪些配置文件格式？

Spring Boot 支持两种配置文件格式：**Properties** 和 **YAML**，功能完全等价，只是语法不同。

Properties 格式简单直观，每行一个 key=value；YAML 格式通过缩进表示层级关系，配置项较多时结构更清晰。

此外还有 `bootstrap.yml`（Spring Cloud 使用，在 application 配置之前加载，优先级更高）和 `application-{profile}.yml`（环境特定配置）。

实际项目建议全团队统一一种格式，不要混用。

###### 2. application.properties 和 application.yml 的区别是什么？

两者功能完全相同，区别只在语法和可读性：

```properties
# .properties 格式
server.port=8080
spring.datasource.url=jdbc:mysql://localhost:3306/test
spring.datasource.username=root
```

```yaml
# .yml 格式
server:
  port: 8080
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test
    username: root
```

配置项少的时候两者差不多；配置层级深的时候 YAML 可读性明显更好，不用反复写前缀。另外 YAML 支持在一个文件里用 `---` 分隔多个文档块，可以在单文件里定义多环境配置，这是 properties 格式做不到的。

###### 3. Spring Boot 配置文件的加载顺序是怎样的？

Spring Boot 配置采用**分层覆盖机制**，优先级由高到低：

1. 命令行参数（`--server.port=9090`）—— 最高优先级
2. Java 系统属性（`-Dserver.port=9090`）
3. 操作系统环境变量（`SERVER_PORT=9090`）
4. jar 包外部的 Profile 特定配置（`./config/application-prod.yml`）
5. jar 包外部的通用配置（`./application.yml`）
6. jar 包内部的 Profile 特定配置（`classpath:application-prod.yml`）
7. jar 包内部的通用配置（`classpath:application.yml`）
8. `@PropertySource` 指定的文件
9. 默认属性

核心原则：外部配置优先于内部配置，Profile 特定配置优先于通用配置，命令行参数优先于一切。这让运维可以在不修改 jar 包的情况下动态调整配置。

###### 4. 如何实现多环境配置？

Spring Boot 通过 **Profile 机制**实现多环境配置，核心是 `spring.profiles.active`。

多环境文件命名规范：`application-{环境名}.yml`，比如 `application-dev.yml`、`application-prod.yml`。

激活 Profile 的方式：
- 在 `application.yml` 里设置默认：`spring.profiles.active: dev`
- 启动命令行指定：`java -jar app.jar --spring.profiles.active=prod`
- 环境变量（Docker/K8s 场景）：`SPRING_PROFILES_ACTIVE=prod`

最佳实践是把公共配置放 `application.yml`，各环境差异配置放对应的 Profile 文件，避免在每个 Profile 文件里重复写相同的配置。

###### 5. @ConfigurationProperties 注解的作用是什么？

`@ConfigurationProperties` 用于**批量绑定配置属性到 Java 对象**，支持类型安全访问、松散绑定（`pool-size` 自动映射到 `poolSize`）和 JSR-303 校验。

```yaml
app:
  datasource:
    url: jdbc:mysql://localhost/test
    pool-size: 10
    timeout: 5000
```

```java
@Component
@ConfigurationProperties(prefix = "app.datasource")
@Validated  // 开启校验
public class DataSourceProperties {
    @NotBlank
    private String url;
    
    @Min(1) @Max(100)
    private int poolSize;   // pool-size 自动映射
    
    private Duration timeout; // 自动转换为 Duration 类型
}
```

这比 `@Value` 好用得多，特别是配置项多的时候，不用一个个字段加 `@Value`，统一在 Properties 类里管理，还能加校验。

###### 6. @Value 和 @ConfigurationProperties 的区别是什么？

两者都用于注入配置属性，但适用场景不同：

`@Value` 是字段级别的注入，适合只需要一两个配置值的简单场景，直接标注在字段上就行，但不支持松散绑定，不支持 JSR-303 校验，复杂类型（List/Map）支持有限。

`@ConfigurationProperties` 是类级别的批量绑定，适合有一组相关配置的场景，支持松散绑定、类型转换（Duration/DataSize）、JSR-303 校验，代码也更整洁，配置变更只需要改 Properties 类一处地方。

**经验法则**：一两个配置值用 `@Value`，一组相关配置用 `@ConfigurationProperties`。

###### 7. 如何读取自定义配置文件？

默认情况下 Spring Boot 只加载 `application.*`，读取自定义文件用 `@PropertySource`：

```java
@Configuration
@PropertySource(value = "classpath:custom.properties", encoding = "UTF-8")
public class CustomConfig {
    @Value("${custom.api.url}")
    private String apiUrl;
}
```

注意 `@PropertySource` 只支持 `.properties` 格式，不支持 YAML。如果需要加载 YAML 格式的自定义文件，需要自定义 `PropertySourceFactory` 来处理。

###### 8. 如何加密配置文件中的敏感信息？

生产环境的密码、密钥不能明文存在配置文件里，常用 **Jasypt** 库来加密：

配置文件里用 `ENC(密文)` 格式存储：
```yaml
spring:
  datasource:
    password: ENC(加密后的密文)
jasypt:
  encryptor:
    password: ${JASYPT_PASSWORD}  # 加密密钥从环境变量读取
```

运行时 Jasypt 自动识别 `ENC(...)` 并用密钥解密，应用代码感知不到加密的存在。

密钥本身通过环境变量或 K8s Secret 注入，不写死在配置文件里，这样即使配置文件泄露也不会暴露真实密码。

###### 9. Spring Boot 如何实现配置的动态刷新？

配置动态刷新需要结合 Spring Cloud Config 和 `@RefreshScope` 注解实现，无需重启应用即可更新配置。

使用了 `@RefreshScope` 的 Bean，在收到刷新信号时会被销毁并重新创建，重新读取最新的配置值：

```java
@RestController
@RefreshScope
public class ConfigController {
    @Value("${dynamic.config.value}")
    private String dynamicValue;
}
```

触发刷新：向 `/actuator/refresh` 发一个 POST 请求，Spring Cloud Bus 场景下可以广播给所有实例。

注意 `@RefreshScope` 有一定的性能开销，不要滥用，只对真正需要动态刷新的 Bean 加。

###### 10. Profile 的作用是什么，如何使用？

Profile 是 Spring Boot 的**环境隔离机制**，让不同环境（开发/测试/生产）使用不同的配置和 Bean。

三种激活方式：
```bash
# 命令行
java -jar app.jar --spring.profiles.active=prod

# 系统属性
-Dspring.profiles.active=prod

# 环境变量（Docker 最常用）
export SPRING_PROFILES_ACTIVE=prod
```

`@Profile` 注解还能用于条件化 Bean，只在特定环境创建某个 Bean，比如开发环境用 H2 内存库，生产环境用 MySQL，配置好 Profile 切换一下命令行参数就能搞定。

###### 11. 如何在代码中动态切换 Profile？

通过注入 `ConfigurableEnvironment` 可以编程式切换 Profile：

```java
@Service
public class ProfileService {
    @Autowired
    private ConfigurableEnvironment environment;

    public void switchToProfile(String profile) {
        environment.setActiveProfiles(profile);
    }

    public String getActiveProfiles() {
        return String.join(",", environment.getActiveProfiles());
    }
}
```

不过这种方式在实际生产中用得不多，运行期切换 Profile 并不会自动重新加载对应的配置文件，更常见的是在启动时通过命令行参数或环境变量指定 Profile。

###### 12. 外部化配置的优先级是怎样的？

核心原则：**命令行 > 系统属性 > 环境变量 > 外部配置文件 > 内部配置文件 > 默认值**。

对于配置文件来说，jar 包外面的配置文件优先于 jar 包内的；Profile 特定的配置文件（`application-prod.yml`）优先于通用配置文件（`application.yml`）。

这套优先级设计的目的是：让运维能在不修改代码、不重新打包的情况下，通过环境变量或命令行参数覆盖任何配置，实现"一次打包，多处运行"的目标。

###### 13. 如何使用命令行参数覆盖配置？

命令行参数优先级最高，用 `--key=value` 格式：

```bash
# 覆盖服务端口
java -jar app.jar --server.port=9090

# 覆盖数据库连接
java -jar app.jar --spring.datasource.url=jdbc:mysql://prod-db:3306/app

# 激活生产环境
java -jar app.jar --spring.profiles.active=prod

# 多个参数
java -jar app.jar --server.port=9090 --spring.profiles.active=prod
```

这是运维在不修改配置文件的情况下动态调整应用行为的最常用手段，特别适合临时覆盖某个配置值。

###### 14. Spring Cloud Config 与 Spring Boot 配置的关系是什么？

Spring Cloud Config 是 Spring Boot 配置能力的**分布式扩展**，把配置集中存储在 Git/SVN 等远程仓库，多个微服务实例从同一个 Config Server 拉取配置。

使用时在 `bootstrap.yml` 里配置 Config Server 地址（注意是 bootstrap 不是 application，因为需要在应用初始化前就加载远程配置）：

```yaml
spring:
  application:
    name: my-service
  cloud:
    config:
      uri: http://config-server:8888
      fail-fast: true  # 连接失败时快速失败，不启动
```

配置优先级：远程 Config Server 配置 > 本地 application.yml > 默认配置。配合 `@RefreshScope` 和 Spring Cloud Bus，还能实现配置动态推送，不需要重启服务。
