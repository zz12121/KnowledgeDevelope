###### 1. Spring Boot 如何集成 MyBatis?

Spring Boot 通过 `mybatis-spring-boot-starter` 提供了开箱即用的 MyBatis 集成方案，底层依赖自动配置机制。

**自动配置核心：**

`MybatisAutoConfiguration` 通过 `@ConditionalOnClass` 检测类路径下存在 `SqlSessionFactory` 和 `SqlSessionFactoryBean` 时自动生效，排在 `DataSourceAutoConfiguration` 之后，自动注册 `SqlSessionFactory` 和 `SqlSessionTemplate`。

**完整集成步骤：**

第一步，引入 starter 依赖：

```xml
<dependency>
    <groupId>org.mybatis.spring.boot</groupId>
    <artifactId>mybatis-spring-boot-starter</artifactId>
    <version>2.3.1</version>
</dependency>
```

第二步，配置数据源和 MyBatis 参数：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test
    username: root
    password: 123456
    driver-class-name: com.mysql.cj.jdbc.Driver

mybatis:
  mapper-locations: classpath:mapper/*.xml
  type-aliases-package: com.example.entity
  configuration:
    map-underscore-to-camel-case: true
    cache-enabled: true
```

第三步，配置 Mapper 扫描（二选一）：

```java
// 方式1：启动类加 @MapperScan，一次配置全局生效
@SpringBootApplication
@MapperScan("com.example.mapper")
public class Application { }

// 方式2：每个 Mapper 接口单独加 @Mapper 注解
@Mapper
public interface UserMapper {
    User selectById(Long id);
}
```

第四步，编写 XML 映射文件：

```xml
<mapper namespace="com.example.mapper.UserMapper">
    <select id="selectById" resultType="User">
        SELECT * FROM user WHERE id = #{id}
    </select>
</mapper>
```

###### 2. Spring Boot 如何集成 JPA?

Spring Data JPA 通过 Repository 抽象层极大简化了数据访问代码，底层基于 Hibernate 实现。

**自动配置核心：**

`JpaRepositoriesAutoConfiguration` 检测到 `EntityManagerFactory` 和 `PlatformTransactionManager` 时自动激活，完成实体管理器工厂和事务管理器的配置。

**集成步骤：**

第一步，引入依赖：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
```

第二步，配置 JPA 参数：

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: update  # 开发环境可用，生产建议 validate
    show-sql: true
    properties:
      hibernate:
        dialect: org.hibernate.dialect.MySQL8Dialect
        format_sql: true
```

第三步，定义实体类：

```java
@Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true)
    private String username;

    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
    private List<Order> orders = new ArrayList<>();
}
```

第四步，定义 Repository 接口：

```java
public interface UserRepository extends JpaRepository<User, Long> {

    // 方法名自动推导查询
    List<User> findByUsernameContaining(String keyword);

    // 分页查询
    Page<User> findByAgeGreaterThan(int age, Pageable pageable);

    // 自定义 JPQL
    @Query("SELECT u FROM User u WHERE u.email LIKE %:email%")
    List<User> findByEmailPattern(@Param("email") String email);
}
```

###### 3. MyBatis 和 JPA 的区别是什么?

两者代表了不同的持久层设计哲学，从架构设计到适用场景都有显著差异。

**设计理念上，** MyBatis 是 SQL 映射框架，把 SQL 可控性放在第一位，你写什么 SQL 就执行什么 SQL，非常直观；JPA 是 ORM 框架，强调面向对象，尽量让开发者不用关心底层 SQL，Repository 里写个方法名就能自动生成查询。

**开发效率上，** JPA 简单 CRUD 几乎零编码，定义个 Repository 接口就完事；MyBatis 每个操作都要手写 SQL 或注解，但在复杂报表查询、联表查询这类场景反而更高效，SQL 可以精准调优。

**性能调优上，** MyBatis 直接优化 SQL 语句即可，非常清晰；JPA 需要理解 Hibernate 的一级/二级缓存、懒加载、N+1 查询等机制，门槛更高。

**底层架构上，** MyBatis 通过 `SqlSessionTemplate` 管理会话，`MapperProxy` 实现接口动态代理；JPA 基于 `EntityManager` 持久化上下文，`SimpleJpaRepository` 提供默认实现。

**选型建议：** 业务以复杂查询为主、需要精细控制 SQL 的选 MyBatis；追求开发效率、领域模型复杂、偏向 DDD 风格的选 JPA。

###### 4. 如何配置多数据源?

多数据源配置的核心是手动定义多个 DataSource Bean，并明确指定各自的 EntityManagerFactory 和 TransactionManager，因为 Spring Boot 的自动配置在多数据源场景下会失效。

**主数据源配置：**

```java
@Configuration
@EnableTransactionManagement
@EnableJpaRepositories(
    basePackages = "com.example.primary.repository",
    entityManagerFactoryRef = "primaryEntityManagerFactory",
    transactionManagerRef = "primaryTransactionManager"
)
public class PrimaryDataSourceConfig {

    @Bean
    @Primary  // 主数据源必须加 @Primary
    @ConfigurationProperties("spring.datasource.primary")
    public DataSource primaryDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @Primary
    public LocalContainerEntityManagerFactoryBean primaryEntityManagerFactory(
            EntityManagerFactoryBuilder builder) {
        return builder
            .dataSource(primaryDataSource())
            .packages("com.example.primary.entity")
            .persistenceUnit("primary")
            .build();
    }
}
```

**次数据源配置：**

```java
@Configuration
@EnableJpaRepositories(
    basePackages = "com.example.secondary.repository",
    entityManagerFactoryRef = "secondaryEntityManagerFactory",
    transactionManagerRef = "secondaryTransactionManager"
)
public class SecondaryDataSourceConfig {

    @Bean
    @ConfigurationProperties("spring.datasource.secondary")
    public DataSource secondaryDataSource() {
        return DataSourceBuilder.create().build();
    }
}
```

**配置文件：**

```yaml
spring:
  datasource:
    primary:
      url: jdbc:mysql://localhost:3306/primary_db
      username: user1
      password: pass1
    secondary:
      url: jdbc:mysql://localhost:3306/secondary_db
      username: user2
      password: pass2
```

**注意事项：** 多数据源环境下必须手动配置，`@Primary` 标注主数据源避免注入歧义，每个数据源需要独立的 TransactionManager，不能共用。

###### 5. Spring Boot 默认使用的数据库连接池是什么?

Spring Boot 2.x 及以上版本默认使用 **HikariCP** 作为数据库连接池，之所以选它，是因为它在各项基准测试中性能遥遥领先，同时代码量极少、依赖极简。

**自动配置逻辑：**

`DataSourceConfiguration.Hikari` 通过 `@ConditionalOnClass(HikariDataSource.class)` 和 `@ConditionalOnProperty`（`matchIfMissing = true`）实现默认选择，只要不显式指定其他类型，就会自动走 HikariCP。

**HikariCP 三大性能优势：**

1. **无锁设计**：`ConcurrentBag` 结合 ThreadLocal 和 CopyOnWriteArrayList，当前线程优先复用自己之前归还的连接，几乎零竞争
2. **字节码优化**：通过 Javassist 对关键方法做字节码增强，减少动态分派开销
3. **智能清理**：自动检测并回收泄漏连接，配置 `leak-detection-threshold` 即可开启

如果需要更强的监控能力（SQL 统计、慢查询、防 SQL 注入），可以替换为阿里的 Druid 连接池。

###### 6. HikariCP 的优势是什么?

HikariCP 的高性能来自「极简设计 + 算法优化」的组合，是目前 Java 生态里最快的连接池。

**核心优势：**

**无锁连接获取：** `ConcurrentBag` 是 HikariCP 自研的数据结构，先从当前线程的 ThreadLocal 里找空闲连接，找不到再去共享队列，基本消除了多线程锁竞争。

**字节码优化：** 对 `PreparedStatement`、`Connection` 等关键类做了字节码增强，减少方法调用层级，降低 CPU 开销。

**配置极简：** 核心配置就这几项，没有一堆令人困惑的参数：

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20         # CPU 核数 × 2 + 1 是经验值
      minimum-idle: 5               # 最小空闲连接
      connection-timeout: 30000     # 获取连接最大等待时间 30s
      idle-timeout: 600000          # 空闲超时 10min
      max-lifetime: 1800000         # 连接最大存活时间 30min
      leak-detection-threshold: 60000  # 超过 60s 未归还则告警
```

**实际使用建议：** `maximum-pool-size` 不是越大越好，超过 CPU 处理能力反而因线程调度开销降低吞吐量；`leak-detection-threshold` 建议开发环境一定要配，能帮你快速发现连接泄漏问题。

###### 7. 如何实现数据库读写分离?

读写分离的核心是根据操作类型（读/写）动态路由到不同的数据库实例，Spring 提供了 `AbstractRoutingDataSource` 抽象类来实现这个能力。

**实现步骤：**

第一步，定义路由键上下文（用 ThreadLocal 存当前线程应该走哪个库）：

```java
public class DatabaseContextHolder {
    private static final ThreadLocal<DatabaseType> context = new ThreadLocal<>();

    public static void setDatabaseType(DatabaseType type) { context.set(type); }
    public static DatabaseType getDatabaseType() {
        return context.get() == null ? DatabaseType.MASTER : context.get();
    }
    public static void clear() { context.remove(); }
}
```

第二步，继承 `AbstractRoutingDataSource` 实现路由逻辑：

```java
public class ReadWriteRoutingDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return DatabaseContextHolder.getDatabaseType();
    }
}
```

第三步，AOP 切面自动切换路由：

```java
@Aspect
@Component
public class ReadWriteDataSourceAspect {

    @Around("@annotation(readOnly)")
    public Object routeDataSource(ProceedingJoinPoint point, ReadOnly readOnly) throws Throwable {
        try {
            DatabaseContextHolder.setDatabaseType(DatabaseType.SLAVE);
            return point.proceed();
        } finally {
            DatabaseContextHolder.clear();  // 必须清理，线程池复用会污染路由
        }
    }
}
```

**几个坑要注意：**

- `finally` 里必须清理 ThreadLocal，否则线程池里的线程复用会导致路由到错误的库
- 在事务内不要切换数据源，`@Transactional` 开启后已绑定了数据库连接
- 主从延迟问题：写完立刻读可能读到旧数据，这种情况需要强制走主库

###### 8. Spring Boot 如何集成 Redis?

Spring Boot 通过 `spring-boot-starter-data-redis` 提供 Redis 自动配置，默认使用 **Lettuce** 客户端（线程安全，支持响应式编程）。

**集成步骤：**

第一步，引入依赖：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
```

第二步，配置连接参数：

```yaml
spring:
  redis:
    host: localhost
    port: 6379
    password: 123456
    lettuce:
      pool:
        max-active: 8
        max-idle: 8
        min-idle: 0
        max-wait: -1ms
```

第三步，自定义 RedisTemplate 序列化（默认 JDK 序列化，存到 Redis 里是乱码，强烈建议替换）：

```java
@Configuration
public class RedisConfig {

    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);

        // Key 用 String 序列化，Value 用 Jackson JSON 序列化
        Jackson2JsonRedisSerializer<Object> serializer = new Jackson2JsonRedisSerializer<>(Object.class);
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(serializer);
        template.setHashKeySerializer(new StringRedisSerializer());
        template.setHashValueSerializer(serializer);

        return template;
    }
}
```

###### 9. 如何使用 Redis 实现缓存?

Spring Cache 提供了统一的缓存抽象层，通过注解驱动缓存操作，底层可以无缝切换 Redis、Caffeine、Ehcache 等实现。

**第一步，开启缓存并配置 RedisCacheManager：**

```java
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))  // 缓存 30 分钟
            .disableCachingNullValues();       // 不缓存 null，防止缓存穿透

        return RedisCacheManager.builder(factory)
            .cacheDefaults(config)
            .build();
    }
}
```

**第二步，在 Service 层使用缓存注解：**

```java
@Service
public class UserService {

    @Cacheable(value = "users", key = "#id")
    public User getUserById(Long id) {
        // 缓存命中时直接返回，不执行方法体
        return userRepository.findById(id);
    }

    @CachePut(value = "users", key = "#user.id")
    public User updateUser(User user) {
        // 始终执行方法，并把结果更新到缓存
        return userRepository.save(user);
    }

    @CacheEvict(value = "users", key = "#id")
    public void deleteUser(Long id) {
        // 删除数据库记录，同时清除对应缓存
        userRepository.deleteById(id);
    }
}
```

**三个核心注解的区别：** `@Cacheable` 是「先查缓存再执行」，`@CachePut` 是「先执行再更新缓存」，`@CacheEvict` 是「执行后清缓存」——记住这个顺序就不会混淆。

###### 10. @Cacheable、@CachePut、@CacheEvict 的作用是什么?

这三个注解是 Spring Cache 的核心，分别对应缓存的不同操作场景。

**底层机制：** Spring 通过 `CacheInterceptor` 拦截注解方法，在 `CacheAspectSupport.invoke()` 里根据注解类型分发到对应的执行逻辑。

**三个注解详解：**

`@Cacheable` —— 查询场景的缓存注解。方法执行前先检查缓存，命中直接返回，不命中才执行方法并把结果写入缓存。适合幂等的查询操作，是最常用的缓存注解。

`@CachePut` —— 更新场景的缓存注解。始终执行方法，执行完后把最新结果更新到缓存。适合更新操作，确保缓存和数据库保持同步。

`@CacheEvict` —— 删除场景的缓存注解。方法执行后清理指定缓存。`allEntries = true` 可以一次性清空整个缓存空间，适合批量刷新场景。

**高级用法：**

```java
// 条件缓存：只有 id > 100 时才缓存
@Cacheable(value = "users", key = "#id", condition = "#id > 100")
public User getUserConditional(Long id) { ... }

// 清空整个 users 缓存空间
@CacheEvict(value = "users", allEntries = true)
public void refreshAllUsers() { ... }

// 组合注解：同时做多个缓存操作
@Caching(
    evict = {@CacheEvict(value = "user-list", allEntries = true)},
    put   = {@CachePut(value = "users", key = "#user.id")}
)
public User updateUserWithListEvict(User user) { ... }
```

`@Caching` 是一个容器注解，可以组合多个缓存操作，在更新单个对象同时需要刷新列表缓存时特别有用。

###### 11. Spring Boot 如何实现分布式事务?

分布式事务是跨多个服务或数据库的事务一致性保证，常见方案有以下几种：

**Seata AT 模式（推荐）：** 自动补偿模式，对业务代码侵入最小，在方法上加 `@GlobalTransactional` 即可。Seata 通过分析 SQL 自动生成回滚日志（undo_log），失败时自动执行逆向补偿。适合强一致性要求且不想改业务代码的场景。

**TCC 模式（Try-Confirm-Cancel）：** 手动实现三个阶段。Try 阶段预留资源；Confirm 阶段提交；Cancel 阶段回滚。灵活性最高，但需要为每个操作编写三套逻辑，业务侵入性强。

**Saga 模式：** 将长事务拆分为多个本地事务，通过补偿事务实现回滚。适合微服务链路长、跨多个服务的场景，最终一致性。

**消息事务（最终一致性）：** 利用 RocketMQ 的事务消息机制。本地事务和消息发送捆绑在一起，通过消息的可靠投递保证最终一致性。性能最好，但只能保证最终一致，不能保证强一致。

**如何选择：** 强一致性用 Seata AT；需要灵活控制用 TCC；长链路用 Saga；性能优先且能接受短暂不一致用消息事务。

###### 12. @Transactional 注解的使用和注意事项是什么?

`@Transactional` 是 Spring 声明式事务的核心，底层通过 AOP 代理实现，在方法执行前后自动开启和提交/回滚事务。

**核心属性：**

- `propagation`：传播行为，默认 `REQUIRED`（有事务加入，没有新建）
- `isolation`：隔离级别，默认 `DEFAULT`（跟随数据库设置）
- `rollbackFor`：触发回滚的异常类型，默认只回滚 `RuntimeException` 和 `Error`
- `readOnly`：只读标记，数据库可以优化执行计划，查询操作建议加上
- `timeout`：事务超时秒数，超时自动回滚

**常见失效场景（面试高频）：**

1. **同类内部调用**：`this.methodA()` 绕过了 Spring 代理，AOP 拦截器不会触发。解决方案是注入自身 Bean 或把方法抽到另一个 Bean 里
2. **非 public 方法**：Spring AOP 只代理 public 方法，protected/private 上的 `@Transactional` 不生效
3. **异常被捕获**：方法内 catch 后没有重新抛出，Spring 感知不到异常，事务正常提交
4. **异常类型不匹配**：受检异常（如 `Exception`）默认不触发回滚，需要显式指定 `rollbackFor = Exception.class`
5. **数据库引擎不支持**：MySQL MyISAM 引擎不支持事务，InnoDB 才支持

###### 13. 事务传播行为有哪些?

Spring 定义了 7 种事务传播行为，控制嵌套调用时事务的合并与隔离方式。

**REQUIRED（默认）：** 外部有事务则加入，没有则新建。这是最常用的传播行为，大多数场景用这个就够了。

**REQUIRES_NEW：** 无论外部有没有事务，都新建一个独立事务，并把外部事务挂起。内层事务和外层事务完全隔离，互不影响。适合操作日志记录——即使主业务回滚，日志也要记下来。

**NESTED：** 嵌套事务，使用数据库 Savepoint 实现。内层可以独立回滚，但外层回滚会连带内层也回滚。和 REQUIRES_NEW 的区别是：NESTED 是同一个事务里的嵌套，REQUIRES_NEW 是两个独立事务。

**SUPPORTS：** 有事务则加入，没有则以非事务方式执行。适合读操作，有事务上下文时一起提交，没有也无所谓。

**NOT_SUPPORTED：** 以非事务方式执行，如果当前有事务则挂起。适合不需要事务保护的操作。

**MANDATORY：** 必须在事务中执行，如果外部没有事务则抛 `IllegalTransactionStateException`。适合强制要求调用方提供事务上下文的场景。

**NEVER：** 不能在事务中执行，如果外部有事务则抛异常。

**实际开发中，** 90% 的场景用 `REQUIRED`，需要独立事务（如审计日志）用 `REQUIRES_NEW`，需要嵌套回滚用 `NESTED`。

###### 14. 如何解决事务失效问题?

事务失效是 Spring 开发中的高频坑点，根本原因是 `@Transactional` 依赖 AOP 代理，绕过代理就会失效。

**同类内部调用问题（最常见）：**

```java
@Service
public class UserService {

    @Autowired
    private UserService self;  // 注入自身代理

    public void methodA() {
        self.methodB();  // 通过代理调用，事务生效
        // this.methodB();  // 直接调用，事务失效！
    }

    @Transactional
    public void methodB() { ... }
}
```

或者使用 `AopContext.currentProxy()` 获取当前代理对象（需要开启 `@EnableAspectJAutoProxy(exposeProxy = true)`）。

**异常被吞掉的问题：**

```java
@Transactional
public void method() {
    try {
        // 业务操作
    } catch (Exception e) {
        log.error("error", e);
        // 不能只 log，要重新抛出让 Spring 感知到异常
        throw new RuntimeException(e);  // 或自定义业务异常
    }
}
```

**受检异常不回滚的问题：**

```java
// 默认只回滚 RuntimeException，要回滚 Exception 需要显式声明
@Transactional(rollbackFor = Exception.class)
public void method() throws Exception { ... }
```

**几个总结原则：**

- 涉及事务的方法必须是 public 方法
- 不要在同类里直接调用有 `@Transactional` 的方法
- 异常要么不 catch，要么 catch 后重新抛出
- 受检异常要显式指定 `rollbackFor`

###### 17. 如何实现数据库迁移（Flyway/Liquibase）?

数据库迁移工具解决的是「代码版本和数据库 Schema 版本同步」的问题，避免手动执行 SQL 脚本导致环境不一致。

**Flyway（推荐，更简单）：**

```xml
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
```

SQL 脚本放在 `src/main/resources/db/migration/` 目录下，命名规范严格：

```
V1__Create_user_table.sql       # V + 版本号 + __ + 描述
V2__Add_email_column.sql
V3__Create_order_table.sql
```

Spring Boot 启动时 Flyway 自动检测 `flyway_schema_history` 表，执行所有未执行过的脚本，版本号严格递增不可回退。

**Liquibase（功能更强）：**

Liquibase 支持 XML/YAML/JSON/SQL 格式的变更集，最大优势是**支持回滚**，Flyway 不支持回滚。

```yaml
# db/changelog/001-create-user.yaml
databaseChangeLog:
  - changeSet:
      id: 1
      author: developer
      changes:
        - createTable:
            tableName: users
            columns:
              - column:
                  name: id
                  type: BIGINT
                  autoIncrement: true
```

**如何选择：** 项目简单、纯 SQL 管理选 Flyway；需要回滚能力、支持多种数据库格式选 Liquibase。

###### 18. 如何优化数据库查询性能?

数据库查询优化是个系统工程，要从多个层面入手。

**索引层面：** 为查询条件字段、排序字段、关联字段建立合适索引。避免在索引列上做函数运算（会导致索引失效），复合索引注意最左前缀原则。

**SQL 层面：** 避免 `SELECT *`，只查需要的字段；深分页（LIMIT 100000, 10）改用游标分页；避免在循环里执行 SQL，改用批量查询。

**JPA 特有问题：**

- **N+1 问题**：懒加载触发 N 次额外查询，用 `@EntityGraph` 或 `JOIN FETCH` 解决
- **分页 count 查询**：`Page` 接口会额外执行 count 查询，如果只需要滚动翻页用 `Slice` 替代
- **批量更新**：用 `@Modifying @Query` 执行 UPDATE 语句，而不是先查出来再逐个 save

**连接池层面：** HikariCP 的 `maximum-pool-size` 根据 CPU 核数调整，不是越大越好；开启 `leak-detection-threshold` 监控连接泄漏。

**架构层面：** 读写分离分担主库压力；热点数据加 Redis 缓存；非核心统计查询异步化。

**监控层面：** 开启慢查询日志，定期用 EXPLAIN 分析执行计划，Druid 连接池的监控页面能直观看到慢 SQL。

###### 19. 如何实现数据库连接池监控?

连接池监控可以帮助你及时发现连接泄漏、连接不足等问题，有两种常用方案。

**方案一：Druid 内置监控（最方便）**

换用 Druid 连接池，它自带 Web 监控页面，功能极其丰富：

```xml
<dependency>
    <groupId>com.alibaba</groupId>
    <artifactId>druid-spring-boot-starter</artifactId>
    <version>1.2.18</version>
</dependency>
```

```yaml
spring:
  datasource:
    type: com.alibaba.druid.pool.DruidDataSource
    druid:
      stat-view-servlet:
        enabled: true
        url-pattern: /druid/*
        login-username: admin
        login-password: 123456
      filter:
        stat:
          slow-sql-millis: 1000  # 超过 1s 记录为慢 SQL
          log-slow-sql: true
```

访问 `http://localhost:8080/druid` 即可看到 SQL 统计、连接池状态、慢查询列表。

**方案二：HikariCP + Actuator + Prometheus**

HikariCP 自带 Micrometer 指标，配合 Spring Boot Actuator 可以暴露给 Prometheus 采集，再通过 Grafana 展示：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: metrics,health
  metrics:
    tags:
      application: ${spring.application.name}
```

关注的核心指标：`hikaricp.connections.active`（活跃连接数）、`hikaricp.connections.pending`（等待连接数）、`hikaricp.connections.timeout.total`（超时次数）。

###### 20. Spring Data JPA 的常用注解有哪些?

JPA 注解分为实体映射注解和查询相关注解两大类。

**实体映射注解：**

- `@Entity`：标记为 JPA 实体类，对应数据库表
- `@Table(name = "t_user")`：指定表名，不加默认用类名
- `@Id`：主键字段
- `@GeneratedValue(strategy = GenerationType.IDENTITY)`：主键生成策略，IDENTITY 对应数据库自增
- `@Column(name = "user_name", nullable = false, length = 50)`：字段映射，可指定列名、非空约束、长度
- `@Transient`：标记该字段不映射到数据库

**关联关系注解：**

- `@OneToOne`、`@OneToMany`、`@ManyToOne`、`@ManyToMany`：四种关联关系
- `@JoinColumn(name = "user_id")`：指定外键列
- `@JoinTable`：多对多中间表配置
- `fetch = FetchType.LAZY/EAGER`：懒加载/立即加载，`@OneToMany` 默认 LAZY，`@ManyToOne` 默认 EAGER

**查询相关注解：**

- `@Query("SELECT u FROM User u WHERE u.name = :name")`：自定义 JPQL
- `@Query(value = "SELECT * FROM users WHERE name = ?1", nativeQuery = true)`：原生 SQL
- `@Modifying`：UPDATE/DELETE 操作必须加，配合 `@Transactional` 使用
- `@Param("name")`：命名参数绑定
- `@EntityGraph(attributePaths = {"orders"})`：指定急加载的关联属性，解决 N+1 问题

**实体生命周期回调：**

- `@PrePersist`：插入前回调
- `@PostPersist`：插入后回调
- `@PreUpdate`：更新前回调，常用于自动设置 updateTime
- `@PreRemove`：删除前回调
