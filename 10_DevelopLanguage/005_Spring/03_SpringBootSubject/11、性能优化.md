###### 1. 如何优化 Spring Boot 应用启动速度?

Spring Boot 应用启动慢，通常是扫描范围太广、Bean 初始化太重、JVM 配置没有针对性调整这几个原因造成的，可以从三个层面系统下手。

**类加载与扫描优化**方面，最直接的就是限定 `@ComponentScan` 的扫描范围，不要让 Spring 扫遍整个类路径：

```java
@ComponentScan(basePackages = {"com.example.service", "com.example.controller"})
```

另外，对于非核心的 Bean，可以用 `@Lazy` 注解延迟初始化，只有第一次用到的时候才实例化，而不是一启动就全部创建。排除用不到的自动配置也很重要，比如项目里没用数据库，就可以直接排掉：

```java
@EnableAutoConfiguration(exclude = {DataSourceAutoConfiguration.class})
```

**Bean 初始化优化**方面，要避免在 `@PostConstruct` 方法里做耗时操作，比如读远程配置、初始化连接池这些事情，应该移到 `ApplicationRunner` 或 `CommandLineRunner` 里——让应用先把框架部分跑起来，再执行业务初始化逻辑。Spring Boot 2.2+ 还支持全局懒加载：

```java
SpringApplication app = new SpringApplication(MyApp.class);
app.setLazyInitialization(true);   // 全局懒加载
app.setLogStartupInfo(false);      // 关闭启动日志，稍微快一丢丢
```

**JVM 层面**，有两个小技巧值得注意：一是用 `/dev/urandom` 替代 `/dev/random`，避免随机数生成阻塞启动：`-Djava.security.egd=file:/dev/./urandom`；二是利用 AppCDS（Application Class-Data Sharing）预先归档类数据，后续启动直接加载归档文件，类加载时间能缩短不少。

**架构层面**，大单体应用可以拆成微服务，每个服务启动负担小得多；Spring Boot 2.1+ 也支持部分 Bean 的异步初始化，利用多核并行初始化缩短整体时间。

###### 2. 如何减少内存占用?

内存优化要从对象创建、JVM 参数、缓存管理三个方向同时入手。

**对象模型优化**：在循环或频繁调用的方法里，要尽量复用对象而不是不停 new；对于大量数值计算的场景，用基本类型 `int` 而不是包装类 `Integer`，内存省得相当可观；集合初始化的时候给个合适的初始容量，避免扩容带来的内存波动，比如 `new HashMap<>(expectedSize / 0.75 + 1)`。

**JVM 内存结构调优**：

```bash
-Xms512m -Xmx512m      # 初始值和最大值设一样，避免运行时动态调整
-Xmn256m               # 新生代占堆的约 1/2
-XX:SurvivorRatio=8    # Eden:Survivor = 8:1:1
-XX:MaxMetaspaceSize=256m
-Xss256k               # 减小线程栈大小，大量线程时效果明显
```

**缓存与会话管理**：避免一口气把大量数据加载到内存，要用分页查询；HTTP Session 设置合理的超时时间，别让会话无限堆积；本地缓存配置 LRU 淘汰策略，及时清理冷数据。

**监控**：上 VisualVM 或 JProfiler 实时看堆内存；Actuator 的 `/actuator/metrics` 端点可以直接暴露内存指标，接 Prometheus 监控也很方便。

###### 3. 如何优化数据库查询性能?

数据库查询往往是性能瓶颈的重灾区，得从 SQL 本身、ORM 框架、连接池几个层面一起抓。

**SQL 优化基础**：禁止 `SELECT *`，只查需要的字段；为 `WHERE`、`JOIN`、`ORDER BY` 字段建合适的索引；深分页场景用游标分页（`WHERE id > ? LIMIT ?`）替代偏移分页（`LIMIT offset, size`），百万数据量以上的偏移分页会慢得离谱。

**JPA/Hibernate 常见坑**：N+1 查询问题是最典型的，查询一个列表后再一条条查关联对象，几十条记录瞬间变几百次 SQL。用 `@EntityGraph` 或者 JPQL 里的 `JOIN FETCH` 一次性把关联数据拉出来：

```java
@EntityGraph(attributePaths = {"orderItems"})
List<Order> findByUserId(Long userId);
```

批量操作用 Hibernate 的 `batch_size` 配置，把多条 INSERT/UPDATE 合并成一批执行：

```yaml
spring.jpa.properties.hibernate.jdbc.batch_size: 20
```

**连接池调优**：

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
```

连接池的大小不是越大越好，实际上公式是 `连接数 = CPU核心数 × 2 + 有效磁盘数`，过多的连接反而会导致数据库端竞争加剧。开慢查询日志可以帮你定位问题 SQL：

```yaml
spring.jpa.properties.hibernate.session.events.log.LOG_QUERIES_SLOWER_THAN_MS: 1000
```

###### 4. 如何实现接口响应时间优化?

接口慢通常是多个因素叠加的结果，要全链路分析，找到真正的瓶颈再下手。

**后端优化**：能异步的操作就异步处理，比如发邮件、推消息这些不影响主流程的操作；查询结果做缓存，避免重复计算；数据库查询加索引、改 SQL。

**网络传输优化**：开启服务端 GZIP 压缩，对于 JSON 响应体来说压缩率通常能达到 70% 以上：

```yaml
server:
  compression:
    enabled: true
    mime-types: text/html,text/xml,text/plain,text/css,text/javascript,application/javascript,application/json
    min-response-size: 1024
```

**并发处理**：根据任务特性调好线程池参数，避免线程不足或过多；接入限流保护，用令牌桶算法防止突发流量把系统打垮。

**监控与诊断**：接入 Sleuth + Zipkin 做链路追踪，每个请求的每一段耗时都一目了然，快速定位慢在哪一层；用 Actuator 的 `http.server.requests` 指标监控 P99 响应时间。

###### 5. 如何使用缓存提升性能?

缓存是提升系统性能最直接的手段，但选对层级和策略很关键。

**缓存层级怎么选**：本地缓存（Caffeine/Guava Cache）访问速度最快，没有网络开销，适合数据量小、读多写少、集群一致性要求不高的场景；分布式缓存（Redis）适合数据量大、需要集群间共享、容量可弹性扩展的场景；两者结合就是分层缓存——本地缓存当 L1，Redis 当 L2，能兼顾速度和一致性，代价是架构复杂度更高。

**Spring Cache 配置**：

```java
@Configuration
@EnableCaching
public class CacheConfig {
    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))
            .disableCachingNullValues();
        return RedisCacheManager.builder(factory)
            .cacheDefaults(config).build();
    }
}
```

**注解实战**：

```java
@Cacheable(value = "users", key = "#id", unless = "#result == null")
public User getUserById(Long id) {
    return userRepository.findById(id).orElse(null);
}
```

`@Cacheable` 用于查询，先查缓存再查库；`@CachePut` 用于更新，写库同时更新缓存；`@CacheEvict` 用于删除，数据删了同步清缓存。

**进阶策略**：TTL 要根据数据更新频率来定，更新频繁的数据 TTL 设短，反之设长；也可以基于事件驱动，数据变更时主动推送缓存失效；对于热点数据，可以在缓存快到期时异步刷新，避免缓存击穿。

###### 6. 如何实现异步处理?

异步处理能把耗时操作从主线程里剥离出去，让接口快速返回，同时提升系统整体吞吐量。

**Spring 异步支持**：先开启 `@EnableAsync`，然后配置专用线程池：

```java
@Configuration
@EnableAsync
public class AsyncConfig {
    @Bean("taskExecutor")
    public TaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("Async-");
        executor.initialize();
        return executor;
    }
}
```

**异步方法**：标记 `@Async` 后，方法会在配置的线程池里异步执行：

```java
@Async("taskExecutor")
public CompletableFuture<Order> processOrderAsync(Order order) {
    // 耗时处理
    return CompletableFuture.completedFuture(processedOrder);
}

@EventListener
@Async
public void handleOrderEvent(OrderEvent event) {
    // 异步处理领域事件，不阻塞事件发布方
}
```

**异步通信模式**：应用内的异步解耦用 Spring 事件机制；跨服务或需要持久化的异步任务用消息队列（RabbitMQ、Kafka）。

**注意事项**：异步方法的异常不会传播到调用方，必须在方法内部自行处理；`ThreadLocal` 里的数据在异步线程里拿不到，需要手动传递上下文；`@Async` 和 `@Transactional` 配合使用时，事务会在异步方法执行前就已提交，这一点要格外小心。

###### 7. @Async 注解的使用和注意事项是什么?

`@Async` 是 Spring 提供的异步执行注解，用对了能大幅提升吞吐量，但有几个坑必须提前知道。

**基本用法**：

```java
@Service
public class NotificationService {
    @Async
    public void sendEmail(String to, String subject, String content) {
        // 发邮件，不阻塞调用方
    }
}
```

**最关键的注意事项**：

第一，**代理机制限制**。`@Async` 基于 AOP 代理实现，同一个类内部调用异步方法不会走代理，也就不会异步执行。必须从外部通过 Bean 调用才有效。

第二，**返回值处理**。无返回值用 `void`，有返回值必须用 `Future` 或 `CompletableFuture`：

```java
@Async
public CompletableFuture<String> asyncMethod() {
    try {
        // 业务逻辑
        return CompletableFuture.completedFuture("success");
    } catch (Exception e) {
        return CompletableFuture.failedFuture(e);
    }
}
```

第三，**异常处理**。异步方法抛出的异常不会传播到调用方，得在方法内部 catch 住，或者配置 `AsyncUncaughtExceptionHandler` 统一处理未捕获的异常。

第四，**与 `@Transactional` 协作**。在事务方法里调用 `@Async` 方法时，事务会在异步方法开始执行之前就提交，所以异步方法能读到事务提交后的数据。但如果想在同一个事务里操作，这样是行不通的。

第五，**线程池配置**。最好为不同类型的异步任务配置专用线程池，避免所有任务挤在默认的 `SimpleAsyncTaskExecutor`（它每次都创建新线程，没有线程复用）。

###### 8. 如何优化线程池配置?

线程池调不好，要么线程不够用请求堆积，要么线程太多资源浪费还上下文切换频繁。关键是根据任务类型来定参数。

**核心参数调优**：

```java
@Bean("ioTaskExecutor")
public TaskExecutor ioTaskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(Runtime.getRuntime().availableProcessors() * 2);  // IO 密集型
    executor.setMaxPoolSize(50);
    executor.setQueueCapacity(1000);
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    executor.setThreadNamePrefix("IO-Thread-");
    executor.initialize();
    return executor;
}
```

**IO 密集型 vs CPU 密集型**：IO 密集型任务（数据库查询、HTTP 调用等）线程大部分时间在等待，可以设更多线程，公式是 `核心数 × (1 + IO等待时间/CPU时间)`，一般直接用 `核心数 × 2` 作为核心线程数；CPU 密集型任务线程数过多反而会导致上下文切换开销，通常设 `CPU 核心数 + 1` 就够了。

**拒绝策略选择**：
- `AbortPolicy`（默认）：直接抛异常，适合不允许丢任务的场景
- `CallerRunsPolicy`：让调用方线程去执行，能起到反压效果，适合大多数业务场景
- `DiscardPolicy`：静默丢弃，谨慎使用
- `DiscardOldestPolicy`：丢弃队列里最老的任务

**监控**：监控线程池的活跃线程数、队列积压量、拒绝任务数这几个指标，结合 Actuator 或自定义端点，甚至可以做到运行时动态调整核心线程数。

###### 9. 如何进行 JVM 调优?

JVM 调优没有万能公式，要结合应用特性、堆内存大小和 GC 目标来针对性配置。

**内存区域调优**：

```bash
java -server \
     -Xms2g -Xmx2g           # 初始值和最大值一致，避免运行中动态扩缩堆
     -Xmn1g                  # 新生代约堆的一半
     -XX:MetaspaceSize=256m  # 元空间初始大小
     -XX:MaxMetaspaceSize=256m
     -Xss256k                # 线程栈大小
     -XX:SurvivorRatio=8     # Eden:Survivor = 8:1:1
     -XX:+UseG1GC            # 使用 G1 垃圾收集器
     -jar application.jar
```

**垃圾收集器选择**：
- **G1 GC**：JDK 9+ 默认，适合大堆内存（4GB+）、追求低停顿时间的场景
- **ZGC**：JDK 11+ 可用，超大堆内存、停顿时间要求极低（< 10ms）的场景
- **CMS**：JDK 8 遗留方案，JDK 14 已移除，别用了

**G1 专项调优**：

```bash
-XX:MaxGCPauseMillis=200               # 目标最大停顿时间，G1 会自动调整 Region 大小
-XX:G1HeapRegionSize=16m               # Region 大小，建议 1M~32M 之间
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发 GC 的堆占用率阈值
```

**监控工具**：`jstat -gcutil <pid> 1000` 实时查看 GC 统计；`jmap -dump:format=b,file=heap.bin <pid>` 生成堆快照；VisualVM 或 JProfiler 做图形化分析。

###### 10. 如何定位内存泄漏问题?

内存泄漏的典型症状是老年代占用率持续上升，Full GC 后也没有明显回落，堆内存随时间呈阶梯状增长。

**诊断步骤**：

第一步，确认是不是内存泄漏——看 GC 日志，如果 Full GC 之后老年代回收效果很差，才是真正的泄漏。

第二步，生成堆转储快照：

```bash
jmap -dump:format=b,file=heap.bin <pid>
# 或者先看对象直方图
jmap -histo <pid> | head -30
```

第三步，用 Eclipse MAT 或 VisualVM 分析 heap.bin，找到 Retained Heap 最大的对象，顺着引用链往上找到 GC Root，基本就能定位到泄漏点。

**常见内存泄漏场景**：

- **静态集合**：静态 `Map`、`List` 里的对象无法被 GC 回收，是最常见的泄漏源头
- **未关闭资源**：数据库连接、IO 流没有在 finally 块或 try-with-resources 里正确关闭
- **监听器未注销**：注册了事件监听器或回调函数，但忘了在对象销毁时移除，导致对象一直被 GC Root 引用
- **ThreadLocal 滥用**：线程池里的线程生命周期很长，`ThreadLocal` 用完不 remove，对象会一直挂在线程上

**预防手段**：代码审查时重点关注资源管理；接 Actuator 的 `/actuator/metrics/jvm.memory.used` 监控堆内存趋势；做长时间压测时关注内存曲线是否收敛。

###### 11. 如何进行性能压测?

压测是验证系统能不能扛得住预期负载的必要手段，不压测就上线等于盲飞。

**压测工具选型**：JMeter 功能最全，有 GUI、支持多种协议，适合复杂场景和团队共用；Gatling 基于 Scala、DSL 脚本化，代码即测试，适合 CI/CD 集成；wrk/wrk2 轻量级命令行工具，配合 Lua 脚本，适合快速压单个 HTTP 接口。

**压测场景设计**：不同阶段测试目的不同——基准测试是单用户验证功能正确性；负载测试是模拟正常业务流量，验证达到设计指标；压力测试是超过正常负载持续打压，找出系统的性能拐点；耐力测试是低负载长时间跑（8小时+），专门用来发现内存泄漏、连接泄漏这类积累性问题。

**关键指标**：吞吐量（TPS/QPS）、响应时间分布（重点看 P95、P99，不要只看平均值）、错误率（超过 0.1% 就要警惕）、资源利用率（CPU、内存、带宽、连接池）。

**Spring Boot 接入监控**：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: metrics,httptrace
  metrics:
    export:
      prometheus:
        enabled: true
```

压测期间结合 Grafana 大盘实时观察，比事后分析报告直观得多。

###### 12. 如何实现静态资源压缩?

静态资源压缩是前端性能优化里投入产出比很高的一项，改几行配置，传输数据量能减少 60%~80%。

**服务端 GZIP 压缩**（Spring Boot 内置支持）：

```yaml
server:
  compression:
    enabled: true
    min-response-size: 1024   # 小于 1KB 的不压缩，压缩本身也有开销
    mime-types:
      - text/html
      - text/css
      - text/javascript
      - application/javascript
      - application/json
```

**前端构建优化**：用 Webpack 在构建阶段预先压缩，部署时直接上传 `.gz` 文件，服务器直接返回，省去了运行时压缩的 CPU 开销：

```javascript
const CompressionPlugin = require('compression-webpack-plugin');
module.exports = {
  plugins: [new CompressionPlugin({ algorithm: 'gzip' })]
};
```

**CDN 与缓存策略**：静态资源上 CDN，边缘节点缓存，离用户更近延迟更低；同时为静态资源配置长期强缓存（文件名带 hash，内容变了文件名就变）：

```java
@Override
public void addResourceHandlers(ResourceHandlerRegistry registry) {
    registry.addResourceHandler("/static/**")
            .addResourceLocations("classpath:/static/")
            .setCacheControl(CacheControl.maxAge(365, TimeUnit.DAYS));
}
```

**图片优化**：根据场景选格式——展示类图片用 WebP（比 JPEG 小 25%+），图标用 SVG；`srcset` 为不同分辨率设备提供不同尺寸；非首屏图片做懒加载，用 `IntersectionObserver` 实现。
