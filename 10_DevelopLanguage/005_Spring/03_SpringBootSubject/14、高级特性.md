###### 1. 如何实现自定义 Starter?

自定义 Starter 是 Spring Boot 最精髓的扩展机制，本质上就是把"自动配置 + 属性绑定 + 条件装配"打包成一个即插即用的依赖。别人引入你的 Starter，零配置就能用上功能，这就是 Spring Boot "约定优于配置"的精髓。

**项目结构**：建议分两个模块——`my-spring-boot-autoconfigure` 负责自动配置逻辑，`my-spring-boot-starter` 只做依赖聚合（引入 autoconfigure 模块和必要的三方依赖）：

```
my-starter-parent/
├── my-spring-boot-autoconfigure   # 核心：自动配置、条件判断
└── my-spring-boot-starter         # 壳：依赖聚合器
```

**配置属性类**：用 `@ConfigurationProperties` 绑定外部化配置，支持 IDE 自动补全（引入 `spring-boot-configuration-processor`）：

```java
@ConfigurationProperties(prefix = "my.service")
public class MyServiceProperties {
    private String prefix = "Hello";
    private String suffix = "!";
    private boolean enabled = true;
    // getters / setters
}
```

**自动配置类**（核心）：

```java
@Configuration
@EnableConfigurationProperties(MyServiceProperties.class)
@ConditionalOnClass(MyService.class)          // 类路径有这个类才生效
@ConditionalOnProperty(prefix = "my.service", name = "enabled", matchIfMissing = true)
public class MyServiceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean    // 容器里没有才创建，给用户覆盖的空间
    public MyService myService(MyServiceProperties props) {
        return new MyService(props.getPrefix(), props.getSuffix());
    }
}
```

**注册自动配置类**：Spring Boot 2.7+ 推荐方式：

```properties
# META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.example.autoconfigure.MyServiceAutoConfiguration
```

兼容旧版本则用 `META-INF/spring.factories`。

**关键设计原则**：
- 一定要用 `@ConditionalOnMissingBean`，让用户能覆盖你的默认 Bean
- 条件注解组合使用，避免 Starter 在不适合的环境里生效
- 属性要有合理的默认值，做到零配置可用

###### 2. 如何实现 SPI 扩展机制?

SPI（Service Provider Interface）是 Java 提供的服务发现机制，核心思想是接口与实现分离——框架定义接口，具体实现由外部提供，运行时动态发现加载。

**Java 标准 SPI**：定义好接口后，实现类在 `META-INF/services/` 目录下注册：

```java
// 接口
public interface LoggerPlugin {
    void log(String message);
    boolean supports(String type);
}

// 注册文件：META-INF/services/com.example.LoggerPlugin
com.example.FileLogger
com.example.ConsoleLogger
```

加载时用 `ServiceLoader.load(LoggerPlugin.class)` 遍历所有实现，按 `supports()` 判断用哪个。Java SPI 的缺点是每次都全量遍历所有实现，没有选择性加载。

**Spring Boot 的 SPI 增强**：`spring.factories` 机制在 Java SPI 基础上扩展了很多类型的扩展点：

```properties
# META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.MyAutoConfiguration

org.springframework.context.ApplicationListener=\
com.example.MyApplicationListener

org.springframework.boot.env.EnvironmentPostProcessor=\
com.example.MyEnvironmentPostProcessor
```

`SpringFactoriesLoader` 会在应用启动时扫描所有 JAR 包里的 `spring.factories` 文件，把所有实现聚合起来。这是 Spring Boot 自动配置的基础机制，所有的 `AutoConfiguration` 类都是通过这套 SPI 被发现和加载的。

**自定义 SPI 扩展点**：在自己的框架里也可以模仿这个模式——在 Starter 里定义扩展接口，让使用者在各自的 `spring.factories` 里注册实现，框架启动时统一加载。这样就实现了真正的插件化架构，新增实现完全不需要修改框架代码。

###### 3. Spring Boot 的事件机制是怎样的?

Spring Boot 的事件机制基于观察者模式，让应用内各模块之间通过事件解耦通信，发布方不需要知道有谁在监听，监听方也不需要主动轮询。

**事件体系**：Spring Boot 内置了一套完整的应用生命周期事件，按启动顺序依次触发：

```
ApplicationStartingEvent         → 最早，Bean 工厂还没初始化
ApplicationEnvironmentPreparedEvent → 环境配置加载完成
ApplicationPreparedEvent         → ApplicationContext 准备完成
ApplicationStartedEvent          → 应用启动完成（可接请求前）
ApplicationReadyEvent            → 应用完全就绪（推荐在这里做初始化任务）
ApplicationFailedEvent           → 启动失败时触发
ContextRefreshedEvent            → 容器刷新完成（也是常用的初始化时机）
ContextClosedEvent               → 容器关闭时
```

**事件多播器**：`SimpleApplicationEventMulticaster` 负责把事件分发给所有注册的监听器。默认是同步执行，如果给它配一个 `TaskExecutor`，就变成异步执行：

```java
@Bean
public ApplicationEventMulticaster applicationEventMulticaster() {
    SimpleApplicationEventMulticaster multicaster = new SimpleApplicationEventMulticaster();
    multicaster.setTaskExecutor(new SimpleAsyncTaskExecutor()); // 开启异步分发
    return multicaster;
}
```

**特性总结**：监听器可以通过 `@Order` 控制执行顺序；同步模式下监听器可以参与当前事务；通过 `@Async` 可以让单个监听器方法异步执行，而不影响其他监听器。

###### 4. 如何自定义事件和监听器?

自定义事件三步走：定义事件类、实现监听器、发布事件。

**定义事件**：继承 `ApplicationEvent`：

```java
public class UserRegisteredEvent extends ApplicationEvent {
    private final String username;
    private final String email;

    public UserRegisteredEvent(Object source, String username, String email) {
        super(source);
        this.username = username;
        this.email = email;
    }
    // getters
}
```

**实现监听器**（推荐用 `@EventListener` 注解方式，比实现接口更简洁）：

```java
@Component
public class EmailNotificationListener {

    @EventListener
    @Async  // 异步处理，不阻塞事件发布方
    public void handleUserRegistered(UserRegisteredEvent event) {
        // 发送欢迎邮件
        emailService.sendWelcome(event.getEmail());
    }

    @EventListener(condition = "#event.email != null")  // SpEL 条件过滤
    public void handleValidEmail(UserRegisteredEvent event) {
        // 只有 email 不为空时才处理
    }
}
```

**发布事件**：通过 `ApplicationEventPublisher` 发布，Spring 会自动注入：

```java
@Service
public class UserService {
    private final ApplicationEventPublisher publisher;

    public UserService(ApplicationEventPublisher publisher) {
        this.publisher = publisher;
    }

    public void registerUser(String username, String email) {
        // 用户注册逻辑...
        publisher.publishEvent(new UserRegisteredEvent(this, username, email));
    }
}
```

这种设计的好处是 `UserService` 完全不依赖 `EmailNotificationListener`，两者解耦。如果以后要加短信通知、积分发放，只要再加一个监听器就行，不需要改 `UserService`。

###### 5. 如何实现条件装配?

条件装配通过 `@Conditional` 系列注解控制 Bean 的创建条件，是自动配置的核心机制——根据类路径、配置属性、Bean 存在性等条件，决定某个配置是否生效。

**常用条件注解**：
- `@ConditionalOnClass`：类路径下有某个类时生效（用于判断依赖是否引入）
- `@ConditionalOnMissingBean`：容器里没有这个 Bean 时生效（给用户覆盖空间）
- `@ConditionalOnProperty`：配置属性满足条件时生效
- `@ConditionalOnWebApplication`：是 Web 应用时生效
- `@ConditionalOnExpression`：SpEL 表达式为 true 时生效

**自定义条件注解**：当内置注解不够用时，可以实现 `Condition` 接口自定义逻辑：

```java
public class DatabaseTypeCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        Map<String, Object> attrs = metadata.getAnnotationAttributes(
            ConditionalOnDatabaseType.class.getName());
        String required = (String) attrs.get("value");
        String actual = context.getEnvironment().getProperty("app.database.type");
        return required.equalsIgnoreCase(actual);
    }
}

@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Conditional(DatabaseTypeCondition.class)
public @interface ConditionalOnDatabaseType {
    String value();
}
```

然后就可以这样用：

```java
@Bean
@ConditionalOnDatabaseType("mysql")
public DataSource mysqlDataSource() { ... }

@Bean
@ConditionalOnDatabaseType("postgresql")
public DataSource postgresDataSource() { ... }
```

条件判断发生在 Bean 定义注册阶段，不满足条件的 Bean 压根不会被注册，不是创建了又销毁，性能很高效。用 `debug=true` 或者 Actuator 的 `/actuator/conditions` 端点可以查看每个条件的评估结果，排查条件不生效的问题特别方便。

###### 6. 如何使用 @Import 导入配置?

`@Import` 让你可以把配置类模块化，主配置只需要 `@Import` 进来就行，不需要把所有配置堆在一起。

**基本用法**：直接导入配置类：

```java
@Configuration
@Import({DatabaseConfig.class, CacheConfig.class})
public class AppConfig { }
```

**三种导入方式**：

第一种，**直接导入普通配置类**，就是上面那种用法。

第二种，**导入 `ImportSelector` 实现类**，可以根据条件动态决定导入哪些配置：

```java
public class FeatureImportSelector implements ImportSelector {
    @Override
    public String[] selectImports(AnnotationMetadata metadata) {
        List<String> imports = new ArrayList<>();
        if (isCacheEnabled()) imports.add(CacheConfig.class.getName());
        if (isAsyncEnabled()) imports.add(AsyncConfig.class.getName());
        return imports.toArray(new String[0]);
    }
}
```

这种方式是 `@EnableAutoConfiguration` 的底层实现——`AutoConfigurationImportSelector` 就是一个 `ImportSelector`，负责把 `spring.factories` 里所有符合条件的自动配置类动态导入进来。

第三种，**导入 `ImportBeanDefinitionRegistrar` 实现类**，可以编程式直接向容器注册 Bean 定义，粒度更细：

```java
public class CustomBeanRegistrar implements ImportBeanDefinitionRegistrar {
    @Override
    public void registerBeanDefinitions(AnnotationMetadata metadata, BeanDefinitionRegistry registry) {
        RootBeanDefinition def = new RootBeanDefinition(CustomService.class);
        registry.registerBeanDefinition("customService", def);
    }
}
```

`ConfigurationClassPostProcessor` 在处理 `@Import` 时会按类型分别调用对应的处理逻辑，整个过程是递归的，导入的配置类里还可以继续 `@Import`。

###### 7. ImportSelector 和 ImportBeanDefinitionRegistrar 的区别是什么?

两者都是 `@Import` 的高级用法，但操作粒度不一样。

**ImportSelector**：返回要导入的**配置类**的类名数组，由框架去解析和实例化这些配置类。它是在**类**的层面工作，导入的是整个配置类。使用场景是根据环境条件动态启用或禁用某个功能模块，比如根据激活的 Profile 选择导入开发环境配置还是生产环境配置。

```java
public class ProfileBasedSelector implements ImportSelector {
    @Override
    public String[] selectImports(AnnotationMetadata metadata) {
        return isDevProfileActive()
            ? new String[]{DevConfig.class.getName()}
            : new String[]{ProdConfig.class.getName()};
    }
}
```

**ImportBeanDefinitionRegistrar**：直接操作 `BeanDefinitionRegistry`，把 Bean 定义注册进容器，跳过配置类解析阶段。它是在 **Bean** 的层面工作，控制粒度更细，可以编程式控制 Bean 的每个属性。使用场景是需要自定义 Bean 的构造逻辑、批量注册 Bean，或者有复杂的条件判断。

```java
public class CustomServiceRegistrar implements ImportBeanDefinitionRegistrar {
    @Override
    public void registerBeanDefinitions(AnnotationMetadata metadata, BeanDefinitionRegistry registry) {
        if (!registry.containsBeanDefinition("requiredBean")) return;
        BeanDefinitionBuilder builder = BeanDefinitionBuilder.rootBeanDefinition(CustomService.class);
        builder.addPropertyReference("dependency", "dependentBean");
        registry.registerBeanDefinition("customService", builder.getBeanDefinition());
    }
}
```

**简单对比**：`ImportSelector` 更轻量，适合模块级别的条件开关；`ImportBeanDefinitionRegistrar` 更强大，适合需要精细控制 Bean 定义的框架级扩展。MyBatis-Spring 的 `@MapperScan` 底层就是 `ImportBeanDefinitionRegistrar`，扫描指定包下的 Mapper 接口并批量注册 Bean 定义。

###### 8. 如何实现 BeanPostProcessor?

`BeanPostProcessor` 是 Spring 容器最重要的扩展点之一，让你能在每个 Bean 完成实例化和属性注入之后、初始化方法执行前后，插入自定义逻辑。

**生命周期位置**：

```
实例化（new）→ 属性注入 → postProcessBeforeInitialization
→ @PostConstruct / InitializingBean / init-method
→ postProcessAfterInitialization → Bean 投入使用
```

**基本实现**：

```java
@Component
public class CustomBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        // 初始化方法之前，可以对 Bean 做一些检查或前处理
        if (bean instanceof Validatable) {
            ((Validatable) bean).validate();
        }
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        // 初始化方法之后，可以返回一个代理对象替换原来的 Bean
        if (bean instanceof Cacheable) {
            return createCacheProxy(bean);
        }
        return bean;
    }
}
```

**典型应用**：Spring AOP 就是通过 `AbstractAutoProxyCreator`（一个 `BeanPostProcessor`）在 `postProcessAfterInitialization` 阶段把 Bean 替换成代理对象来实现的；`@Async`、`@Transactional` 的代理也是同样原理。你也可以用它来处理自定义注解——在 `postProcessBeforeInitialization` 里扫描 Bean 的字段，对带有特定注解的字段做特殊处理（比如字段值加密）。

**注意事项**：`BeanPostProcessor` 本身必须注册为 Spring Bean 才会生效；多个 `BeanPostProcessor` 可以用 `@Order` 控制执行顺序；避免在处理器里做耗时操作，因为它会对容器里所有 Bean 执行，会影响启动速度。

###### 9. 如何实现 BeanFactoryPostProcessor?

`BeanFactoryPostProcessor` 比 `BeanPostProcessor` 更底层——它在 Bean **定义**加载完成之后、Bean **实例化**开始之前执行，操作对象是 `BeanDefinition`（元数据），而不是 Bean 实例。

**执行时机**：Spring 容器 `refresh()` 过程中，`invokeBeanFactoryPostProcessors()` 会优先执行所有 `BeanFactoryPostProcessor`，这时候还没有任何 Bean 被实例化（除了 `BeanFactoryPostProcessor` 自身）。

**典型用法**：

```java
@Component
public class CustomBeanFactoryPostProcessor implements BeanFactoryPostProcessor {

    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
        // 修改已有 Bean 的定义（比如替换数据源 URL）
        BeanDefinition beanDef = beanFactory.getBeanDefinition("dataSource");
        beanDef.getPropertyValues().add("url", "jdbc:mysql://localhost:3306/mydb");

        // 动态注册新的 Bean 定义
        if (!beanFactory.containsBeanDefinition("myNewService")) {
            beanFactory.registerBeanDefinition("myNewService",
                new RootBeanDefinition(MyNewService.class));
        }
    }
}
```

**与 BeanPostProcessor 的核心区别**：

`BeanFactoryPostProcessor` 操作的是 Bean 定义（元数据层面），影响的是 Bean 在容器中的注册信息；`BeanPostProcessor` 操作的是 Bean 实例，影响的是对象本身的行为。前者是容器级别的扩展，后者是单个 Bean 级别的扩展。

**实际应用**：`PropertySourcesPlaceholderConfigurer` 就是一个 `BeanFactoryPostProcessor`，负责在 Bean 实例化前把 `${property}` 占位符替换成实际值；`ConfigurationClassPostProcessor` 也是它的子接口实现，负责处理 `@Configuration`、`@ComponentScan`、`@Import` 等注解。

###### 10. 如何实现自定义注解?

自定义注解是声明式编程的基础，配合 AOP 或 `BeanPostProcessor` 使用，可以把横切逻辑从业务代码里剥离出来，让代码更干净。

**定义注解**：

```java
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)   // 运行时保留，才能通过反射读取
@Documented
public @interface AuditLog {
    String action();
    String module() default "";
}
```

四个元注解：`@Target` 指定用在哪里（类/方法/字段）；`@Retention` 指定保留策略（`RUNTIME` 才能运行时读取）；`@Documented` 让它出现在 Javadoc 里；`@Inherited` 让子类继承父类上的注解。

**配合 AOP 处理注解**（推荐，业务逻辑和横切逻辑完全分离）：

```java
@Aspect
@Component
@Slf4j
public class AuditLogAspect {

    @AfterReturning("@annotation(auditLog)")
    public void auditSuccess(JoinPoint joinPoint, AuditLog auditLog) {
        log.info("操作成功 - 模块: {}, 动作: {}, 用户: {}",
            auditLog.module(), auditLog.action(), getCurrentUser());
    }

    @AfterThrowing(pointcut = "@annotation(auditLog)", throwing = "ex")
    public void auditFailure(JoinPoint joinPoint, AuditLog auditLog, Exception ex) {
        log.error("操作失败 - 模块: {}, 动作: {}, 错误: {}",
            auditLog.module(), auditLog.action(), ex.getMessage());
    }
}
```

业务代码里只需要打上注解，审计逻辑完全透明：

```java
@AuditLog(action = "DELETE", module = "USER")
public void deleteUser(Long userId) {
    userRepository.deleteById(userId);
}
```

这种模式特别适合做操作日志、权限校验、接口限流、分布式锁等横切场景，修改这些逻辑只需要改切面，业务代码一行不动。

###### 11. 如何使用 AOP 实现切面编程?

AOP（面向切面编程）把日志、事务、权限这类横切关注点从业务逻辑里抽出来，用切面统一管理，避免代码散落在各处。

**核心概念**：
- **切点（Pointcut）**：用表达式定义哪些方法需要被拦截
- **通知（Advice）**：在切点位置执行的逻辑，分 Before/After/Around 等类型
- **切面（Aspect）**：切点 + 通知的组合

**完整切面示例**：

```java
@Aspect
@Component
@Slf4j
public class ServiceMonitorAspect {

    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceLayer() {}

    @Before("serviceLayer()")
    public void logBefore(JoinPoint jp) {
        log.info("调用: {} 参数: {}", jp.getSignature().getName(),
            Arrays.toString(jp.getArgs()));
    }

    @Around("serviceLayer()")
    public Object measureTime(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return pjp.proceed();
        } finally {
            log.info("{} 耗时: {}ms", pjp.getSignature().getName(),
                System.currentTimeMillis() - start);
        }
    }

    @AfterThrowing(pointcut = "serviceLayer()", throwing = "ex")
    public void logException(JoinPoint jp, Exception ex) {
        log.error("{} 异常: {}", jp.getSignature().getName(), ex.getMessage());
    }
}
```

**常用切点表达式**：
- `execution(* com.example.service.*.*(..))` —— Service 层所有方法
- `@annotation(com.example.annotation.AuditLog)` —— 带特定注解的方法
- `@within(org.springframework.stereotype.Service)` —— 带 `@Service` 注解的类里所有方法
- `args(java.lang.String, ..)` —— 第一个参数是 String 的方法

**代理机制**：Spring AOP 在容器初始化时（`BeanPostProcessor` 后置处理阶段）把需要被拦截的 Bean 替换成代理对象。目标类实现了接口用 JDK 动态代理，没实现接口用 CGLIB 子类代理。Spring Boot 2.x 起默认强制用 CGLIB，避免因接口问题导致代理失效。

**AOP 失效的常见原因**：同类内部调用不走代理（因为是直接方法调用，不经过代理对象）；`@Async`、`@Transactional` 也有同样的问题，一定要通过外部 Bean 调用才生效。

###### 12. 如何实现动态代理?

动态代理是在运行时生成代理对象，让你能在不修改原始类代码的情况下，对方法调用进行拦截和增强。Spring AOP 的底层实现就是两种动态代理技术。

**JDK 动态代理**（基于接口）：

```java
public class TimingHandler implements InvocationHandler {
    private final Object target;

    public TimingHandler(Object target) { this.target = target; }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return method.invoke(target, args);
        } finally {
            System.out.printf("%s 耗时 %dms%n",
                method.getName(), System.currentTimeMillis() - start);
        }
    }

    // 创建代理
    public static <T> T createProxy(T target, Class<T> iface) {
        return iface.cast(Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            new Class[]{iface},
            new TimingHandler(target)));
    }
}

// 使用（目标类必须实现 UserService 接口）
UserService proxy = TimingHandler.createProxy(new UserServiceImpl(), UserService.class);
```

**CGLIB 动态代理**（基于继承，不需要接口）：

```java
public class TimingInterceptor implements MethodInterceptor {

    @Override
    public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return proxy.invokeSuper(obj, args);  // 注意用 invokeSuper，不要用 method.invoke
        } finally {
            System.out.printf("%s 耗时 %dms%n",
                method.getName(), System.currentTimeMillis() - start);
        }
    }

    public static <T> T createProxy(Class<T> targetClass) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(targetClass);
        enhancer.setCallback(new TimingInterceptor());
        return targetClass.cast(enhancer.create());
    }
}
```

**两种方式的区别**：JDK 代理要求目标类实现接口，生成的代理类是目标接口的实现；CGLIB 代理不需要接口，生成的是目标类的子类，所以 `final` 类和 `final` 方法无法被代理。性能上现代 JDK（9+）的动态代理性能已经与 CGLIB 相差不大，选择时更多考虑是否有接口约束。

###### 13. JDK 动态代理和 CGLIB 代理的区别是什么?

两种代理技术在实现原理、使用限制和性能特点上都有差异，面试里经常一起问。

**实现基础**：JDK 代理基于**接口**，通过 `java.lang.reflect.Proxy` 在运行时生成实现了目标接口的代理类；CGLIB 代理基于**继承**，通过 ASM 框架在运行时生成目标类的子类。

**使用限制**：JDK 代理要求目标对象必须实现至少一个接口，只能代理接口中声明的方法；CGLIB 可以代理普通类（不需要接口），但不能代理 `final` 类（无法继承）和 `final` 方法（无法覆盖）。

**性能**：从 JDK 8 开始两者性能差距已经很小；JDK 9+ 后 JDK 代理通过 `Lookup` 机制进一步优化，某些场景下反而比 CGLIB 快。早期说 CGLIB 更快是基于反射调用的对比，现在不要拿来当绝对结论。

**Spring 的选择策略**：`DefaultAopProxyFactory` 的选择逻辑是——目标对象实现了接口，且没有开启 `proxyTargetClass=true`，就用 JDK 代理；否则用 CGLIB。Spring Boot 2.x 起通过 `@EnableAspectJAutoProxy(proxyTargetClass = true)` 默认强制使用 CGLIB，减少因"忘了写接口"导致代理失效的问题。

**实践建议**：面向接口编程的场景用 JDK 代理没有问题；没有接口的遗留代码用 CGLIB；Spring Boot 默认的 CGLIB 策略适合大多数场景，不需要特意改。值得注意的是，`@Transactional`、`@Async` 这些注解都依赖代理，所以 `final` 类/方法上用这些注解会静默失效，是个常见的坑。

###### 14. 如何实现响应式编程（WebFlux）?

WebFlux 是 Spring 5 引入的响应式 Web 框架，基于 Project Reactor 的 `Mono`/`Flux`，整个请求处理链路完全非阻塞，底层默认跑在 Netty 上。

**核心概念**：
- `Mono<T>`：0 或 1 个异步结果，类比 `Optional` 的异步版
- `Flux<T>`：0 到 N 个异步元素组成的流
- **背压（Backpressure）**：下游消费跟不上时，通知上游减速，避免内存溢出

**响应式分层示例**：

```java
// Repository 层
public interface UserRepository extends ReactiveCrudRepository<User, String> {
    Flux<User> findByAgeGreaterThan(int age);
    Mono<User> findByUsername(String username);
}

// Service 层
@Service
public class UserService {
    public Mono<User> getUserById(String id) {
        return userRepository.findById(id)
            .switchIfEmpty(Mono.error(new UserNotFoundException(id)));
    }

    public Flux<User> streamUsers() {
        return userRepository.findAll()
            .onBackpressureBuffer(1000)   // 缓冲背压
            .delayElements(Duration.ofMillis(50));
    }
}

// Controller 层
@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping("/{id}")
    public Mono<User> getUser(@PathVariable String id) {
        return userService.getUserById(id);
    }

    // SSE 流式端点
    @GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<User> streamUsers() {
        return userService.streamUsers();
    }
}
```

**WebClient**（响应式 HTTP 客户端，替代 RestTemplate）：

```java
WebClient webClient = WebClient.builder().baseUrl("https://api.example.com").build();

Mono<User> user = webClient.get()
    .uri("/users/{id}", userId)
    .retrieve()
    .bodyToMono(User.class)
    .timeout(Duration.ofSeconds(5))
    .retryWhen(Retry.backoff(3, Duration.ofSeconds(1)));  // 指数退避重试
```

**适用场景**：WebFlux 在高并发、I/O 密集型场景（大量并发 HTTP 调用、实时数据推送、SSE）下有明显优势，能用少量线程处理大量并发连接。CPU 密集型任务、大量阻塞式三方库的场景，用传统 MVC 反而更合适。

###### 15. WebFlux 与传统 MVC 的区别是什么?

WebFlux 和 Spring MVC 代表两种不同的并发模型，核心差异在于线程如何与请求绑定。

**线程模型**：MVC 是经典的"每请求一线程"模型，每个 HTTP 请求占用一个线程，线程在等待数据库、HTTP 调用等 I/O 期间完全阻塞，白白消耗。并发量大时需要大量线程，内存和上下文切换开销大。WebFlux 采用事件循环模型，少量线程（通常是 CPU 核心数）处理所有请求，I/O 等待时线程不阻塞，继续处理其他请求，等结果回来后通过回调继续处理，资源利用率极高。

**编程模型**：MVC 是命令式同步编程，代码逻辑线性清晰，符合大多数开发者的直觉；WebFlux 是声明式响应式编程，通过操作符链式组合数据流，不熟悉 Reactor 的话学习曲线比较陡。

**错误处理**：MVC 用 `@ExceptionHandler` / `@ControllerAdvice` 全局处理异常，同步且直观；WebFlux 用 `onErrorReturn`、`onErrorResume` 等操作符在流里处理异常，也可以实现 `WebExceptionHandler` 全局处理。

**测试方式**：MVC 用 `MockMvc` 测试；WebFlux 用 `WebTestClient` 测试，`StepVerifier` 验证 Flux 的元素序列。

**怎么选**：团队没有响应式编程经验、项目以 CRUD 为主、大量依赖阻塞式三方库（JDBC、同步 Redis 客户端）——选 MVC；需要处理大量长连接（WebSocket、SSE）、I/O 密集型应用、需要实时数据流——选 WebFlux。两者在同一个项目里可以共存，通常的做法是大部分用 MVC，特定的高并发端点用 WebFlux 处理。
