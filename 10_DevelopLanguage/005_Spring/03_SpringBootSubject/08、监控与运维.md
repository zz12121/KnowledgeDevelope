###### 1. Spring Boot Actuator 是什么?

Spring Boot Actuator 是 Spring Boot 提供的生产级监控管理模块，通过一组内置的 RESTful 端点暴露应用内部状态，帮助开发和运维团队实时掌握应用的运行状况。

**四大核心价值：**

- **健康监控**：检查数据库、磁盘空间、Redis 等关键组件状态
- **指标收集**：JVM 内存、GC 情况、HTTP 请求统计等运行时指标
- **操作控制**：动态修改日志级别、优雅关闭应用等管理操作
- **生态集成**：通过 Micrometer 与 Prometheus、Grafana 等监控系统无缝对接

**底层实现：** Actuator 基于 Spring 的条件化配置和自动装配。`EndpointAutoConfiguration` 负责端点自动配置，每个端点对应一个 `Endpoint` 实现类，例如 `HealthEndpoint` 处理健康检查，`MetricsEndpoint` 处理指标收集。

###### 2. Actuator 有哪些常用端点?

Actuator 提供了丰富的内置端点，按用途分类如下：

**健康与状态类（生产环境必开）：**

- `/actuator/health`：应用及各依赖组件的健康状态，K8s 存活/就绪探针标配
- `/actuator/info`：应用基本信息（版本、Git commit 等）

**指标与监控类（推荐开放）：**

- `/actuator/metrics`：JVM 内存、GC、HTTP 请求等运行时指标
- `/actuator/prometheus`：Prometheus 格式的指标数据，配合 Grafana 使用

**调试类（受限或仅调试时开放）：**

- `/actuator/env`：环境变量和配置属性（含敏感信息，生产受限）
- `/actuator/loggers`：查看和动态修改日志级别
- `/actuator/threaddump`：线程堆栈快照，排查死锁和线程泄漏
- `/actuator/mappings`：URL 路径映射信息
- `/actuator/beans`：Spring 容器中所有 Bean

**配置示例：**

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,metrics,info,prometheus
      base-path: /monitor      # 自定义 Actuator 路径前缀
  endpoint:
    health:
      show-details: always     # 显示详细健康状态
      show-components: always
```

生产环境建议将 Actuator 端口与业务端口分离，并配置 Spring Security 限制访问。

###### 3. 如何自定义健康检查指示器?

实现 `HealthIndicator` 接口或继承 `AbstractHealthIndicator` 类，Spring Boot 启动时会自动注册并聚合到 `/actuator/health`。

**数据库连接健康检查：**

```java
@Component
public class DatabaseHealthIndicator extends AbstractHealthIndicator {

    private final DataSource dataSource;

    public DatabaseHealthIndicator(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    @Override
    protected void doHealthCheck(Health.Builder builder) throws Exception {
        if (isDatabaseConnected()) {
            builder.up()
                   .withDetail("database", "MySQL")
                   .withDetail("responseTime", checkResponseTime() + "ms");
        } else {
            builder.down()
                   .withDetail("error", "数据库连接失败");
        }
    }

    private boolean isDatabaseConnected() {
        try (Connection conn = dataSource.getConnection()) {
            return conn.isValid(5);  // 5s 超时
        } catch (SQLException e) {
            return false;
        }
    }
}
```

**第三方 API 健康检查：**

```java
@Component
public class ExternalApiHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        try {
            ResponseEntity<String> response = restTemplate.getForEntity(
                "https://api.external-service.com/health", String.class);

            if (response.getStatusCode().is2xxSuccessful()) {
                return Health.up().withDetail("service", "external-api").build();
            } else {
                return Health.down().withDetail("status", response.getStatusCode().value()).build();
            }
        } catch (Exception e) {
            return Health.outOfService().withDetail("service", "external-api").withException(e).build();
        }
    }
}
```

**健康状态聚合规则：** `SimpleStatusAggregator` 按严重程度排序（DOWN > OUT_OF_SERVICE > UP > UNKNOWN），返回最严重的状态。任一组件 DOWN，整体健康状态就是 DOWN。

###### 4. 如何集成 Prometheus 和 Grafana?

Prometheus + Grafana 是最主流的 Spring Boot 监控方案，Prometheus 负责指标采集和存储，Grafana 负责可视化展示。

**集成步骤：**

第一步，引入依赖：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

第二步，暴露 Prometheus 端点：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: prometheus,health,metrics
  metrics:
    tags:
      application: ${spring.application.name}
      environment: ${spring.profiles.active}
```

第三步，配置 Prometheus 抓取（prometheus.yml）：

```yaml
scrape_configs:
  - job_name: 'spring-boot-app'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['localhost:8080']
    scrape_interval: 15s
```

第四步，Grafana 配置：添加 Prometheus 数据源，导入仪表板模板（ID: 6756，Spring Boot 官方模板）。

**关键监控指标：**

- `jvm_memory_used_bytes`：JVM 内存使用量
- `jvm_gc_pause_seconds`：GC 停顿时间（`p99`、`p95` 尤其重要）
- `http_server_requests_seconds`：HTTP 请求延迟
- `hikaricp_connections_active`：活跃数据库连接数

**自定义业务指标（重要！）：**

```java
public OrderMetricsService(MeterRegistry registry) {
    orderCounter = Counter.builder("orders.total")
        .tag("type", "online")
        .register(registry);

    orderTimer = Timer.builder("orders.processing.time").register(registry);
}
```

###### 5. 如何实现应用性能监控（APM）?

APM（Application Performance Monitoring）关注应用的运行时性能数据，包括方法执行时间、调用次数、错误率等。Spring Boot 通过 Micrometer 提供统一的 APM 支持。

**全局公共 Tag 配置（所有指标都带上应用标识）：**

```java
@Bean
public MeterRegistryCustomizer<MeterRegistry> metricsCommonTags() {
    return registry -> registry.config().commonTags(
        "application", "user-service",
        "region", "us-east-1"
    );
}
```

**方法级性能监控（注解方式，需要注册 TimedAspect）：**

```java
@Bean
public TimedAspect timedAspect(MeterRegistry registry) {
    return new TimedAspect(registry);  // 开启 @Timed 注解支持
}

@Service
public class UserService {

    @Timed(value = "user.service.find.by.id", description = "根据ID查询用户")
    @Counted(value = "user.service.find.count")
    public User findUserById(Long id) {
        return userRepository.findById(id);
    }
}
```

**分布式追踪（Spring Cloud Sleuth）：**

和 Zipkin 配合使用，能追踪跨服务的完整请求链路，自动在日志中注入 traceId/spanId，排查微服务性能问题时非常有用。

###### 6. 如何实现日志管理（Logback/Log4j2）?

Spring Boot 默认使用 Logback 作为日志框架，可通过 `logback-spring.xml` 进行配置。

**Logback 滚动文件配置（生产推荐）：**

```xml
<configuration>
    <property name="LOG_PATTERN"
              value="%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n"/>

    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/application.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>logs/application.%d{yyyy-MM-dd}.%i.log.gz</fileNamePattern>
            <maxHistory>30</maxHistory>      <!-- 保留 30 天 -->
            <maxFileSize>100MB</maxFileSize>  <!-- 单文件最大 100MB -->
        </rollingPolicy>
        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
        </encoder>
    </appender>

    <root level="INFO">
        <appender-ref ref="FILE"/>
    </root>
</configuration>
```

**Log4j2 异步日志（高并发场景性能更好）：**

Log4j2 使用 Disruptor 无锁队列实现异步日志，在高并发下吞吐量是 Logback 的 5-10 倍。只需在 `pom.xml` 中排除 Logback，引入 `spring-boot-starter-log4j2` 即可切换。

###### 7. 如何实现日志分级输出?

**通过配置文件设置不同包的日志级别：**

```yaml
logging:
  level:
    com.example.service: INFO       # 业务代码 INFO 级别
    org.springframework.web: WARN   # Spring Web 框架只打 WARN 以上
    org.hibernate.SQL: DEBUG        # 开发时打印 SQL
  group:
    business: com.example.service,com.example.repository  # 定义日志组
```

**通过 Actuator 动态修改日志级别（无需重启，生产排查利器）：**

```bash
# 查看当前日志级别
curl http://localhost:8080/actuator/loggers/com.example.service

# 动态改为 DEBUG（遇到问题时临时开启）
curl -X POST -H "Content-Type: application/json" \
  -d '{"configuredLevel": "DEBUG"}' \
  http://localhost:8080/actuator/loggers/com.example.service

# 排查完成后改回 INFO
curl -X POST -H "Content-Type: application/json" \
  -d '{"configuredLevel": "INFO"}' \
  http://localhost:8080/actuator/loggers/com.example.service
```

这个能力在生产环境排查问题时非常有用，不需要改配置文件、不需要重启，立即生效。

###### 8. 如何实现日志集中收集（ELK）?

ELK（Elasticsearch + Logstash + Kibana）是最主流的集中式日志收集和分析方案，能统一收集多台服务器的日志并提供可视化查询。

**架构流程：** Spring Boot 通过 Logstash Appender 将日志以 JSON 格式发送给 Logstash → Logstash 过滤/转换 → 存入 Elasticsearch → Kibana 可视化查询。

**Spring Boot 集成 Logstash：**

引入 `logstash-logback-encoder` 依赖，然后在 Logback 配置中添加 TCP Appender：

```xml
<appender name="LOGSTASH" class="net.logstash.logback.appender.LogstashTcpSocketAppender">
    <destination>logstash:5000</destination>
    <encoder class="net.logstash.logback.encoder.LoggingEventCompositeJsonEncoder">
        <providers>
            <timestamp/>
            <logLevel/>
            <loggerName/>
            <threadName/>
            <mdc/>       <!-- MDC 上下文（traceId、userId 等）-->
            <message/>
            <stackTrace/>
        </providers>
    </encoder>
</appender>
```

**MDC 是关键：** 在请求开始时把 traceId、userId 等上下文写入 MDC（`MDC.put("traceId", traceId)`），日志自动携带这些信息，在 Kibana 里按 traceId 过滤就能看到完整链路日志。

###### 9. 如何实现应用优雅关闭?

优雅关闭确保服务停止时不会突然切断正在处理的请求，是生产环境发布、容器滚动更新的基础能力。

**Spring Boot 2.3+ 内置优雅关闭：**

```yaml
server:
  shutdown: graceful  # 等待当前请求处理完成
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 最长等待 30s，超时强制关闭
```

开启后，收到停止信号时，Spring Boot 会停止接收新请求（返回 503），等待已有请求处理完成。

**自定义关闭钩子（处理更多资源清理）：**

```java
@Component
public class GracefulShutdown implements ApplicationListener<ContextClosedEvent> {

    private final ThreadPoolTaskExecutor taskExecutor;

    @Override
    public void onApplicationEvent(ContextClosedEvent event) {
        // 按顺序执行清理
        stopAcceptingNewRequests();         // 1. 停止接收新请求

        // 2. 等待线程池中的任务完成（最多 30s）
        try {
            if (!taskExecutor.getThreadPoolExecutor().awaitTermination(30, TimeUnit.SECONDS)) {
                taskExecutor.getThreadPoolExecutor().shutdownNow();  // 超时强制关闭
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }

        closeDataSource();    // 3. 关闭数据库连接池
        cleanupResources();   // 4. 其他资源清理
    }
}
```

**K8s 环境注意：** 配置 `preStop` 钩子等待几秒（如 `sleep 5`），给 Service 负载均衡摘除 Pod 留出时间，避免关闭过程中还有流量打进来。

###### 10. 如何实现应用健康检查与自动重启?

K8s 通过探针机制实现应用健康检查和自动重启，Spring Boot Actuator 提供了标准的健康检查端点与之对接。

**K8s 探针类型：**

- **Liveness Probe（存活探针）**：检测应用是否需要重启。探针失败，K8s 重启容器
- **Readiness Probe（就绪探针）**：检测应用是否可以接收流量。探针失败，K8s 从 Service 摘除 Pod

**K8s Deployment 配置：**

```yaml
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 60  # 启动后等 60s 再开始探测
  periodSeconds: 10
  failureThreshold: 3       # 连续 3 次失败才重启

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 5
  successThreshold: 1       # 1 次成功即恢复流量
```

**Spring Boot 2.3+ 支持 Liveness/Readiness 分组：**

```yaml
management:
  endpoint:
    health:
      group:
        liveness:
          include: livenessState     # 存活状态（应用内部错误）
        readiness:
          include: readinessState,db,redis  # 就绪状态（依赖是否就绪）
```

**自定义就绪性控制（如依赖初始化完成才接收流量）：**

```java
@EventListener(ApplicationReadyEvent.class)
public void setReady() {
    AvailabilityChangeEvent.publish(eventPublisher, this, ReadinessState.ACCEPTING_TRAFFIC);
}
```
