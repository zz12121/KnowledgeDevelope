###### 1. Spring Boot 与 Spring Cloud 的关系是什么?

Spring Boot 和 Spring Cloud 是「基础与扩展、单体与集群」的关系，共同构成了现代 Java 微服务开发的核心生态。

**层次关系：**

- **Spring Boot** 聚焦于**单个微服务**的快速开发，通过自动配置、内嵌容器等特性简化应用搭建，是微服务的「砖块」
- **Spring Cloud** 构建在 Spring Boot 之上，解决**分布式系统的共性问题**，提供微服务治理的全套解决方案，是微服务的「建筑设计图」

**具体分工：** Spring Boot 负责单个服务的开发、打包、运行；Spring Cloud 负责服务注册发现、配置中心、熔断降级、链路追踪、API 网关等跨服务的基础设施能力。

**协作模式：** Spring Cloud 的各个组件通常以 Spring Boot Starter 形式提供，基于 Spring Boot 的自动配置和条件注解机制，两者使用相同的编程模型和配置方式，无缝集成。

###### 2. 如何实现服务注册与发现（Eureka/Nacos）?

服务注册发现是微服务架构的核心基础设施，解决的是「服务 A 怎么知道服务 B 在哪里」的问题。

**Eureka 方案：**

服务端只需一个注解：

```java
@SpringBootApplication
@EnableEurekaServer
public class EurekaServerApplication { ... }
```

客户端注册和发现：

```yaml
eureka:
  client:
    serviceUrl:
      defaultZone: http://localhost:8761/eureka/
  instance:
    preferIpAddress: true
    instanceId: ${spring.application.name}:${server.port}
```

**Nacos 方案（推荐，阿里开源，功能更强）：**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

Nacos 采用**增量拉取 + 长轮询**机制维护服务列表，实时性好。相比 Eureka，Nacos 还支持配置中心、服务分组、权重路由等更多功能。

###### 3. 如何实现服务间调用（Feign/RestTemplate）?

服务间调用需要解决负载均衡、故障容错、序列化等问题。

**RestTemplate + @LoadBalanced（传统方案）：**

```java
@Bean
@LoadBalanced  // 启用客户端负载均衡，RestTemplate 可以直接用服务名调用
public RestTemplate restTemplate() {
    return new RestTemplate();
}

// 使用服务名代替 IP:Port，由 Ribbon/LoadBalancer 自动解析
User user = restTemplate.getForObject("http://user-service/users/{id}", User.class, userId);
```

**OpenFeign 声明式调用（推荐，更优雅）：**

```java
@FeignClient(name = "order-service", path = "/api/orders")
public interface OrderServiceClient {

    @GetMapping("/user/{userId}")
    List<Order> getOrdersByUser(@PathVariable("userId") Long userId);

    @PostMapping
    Order createOrder(@RequestBody OrderCreateRequest request);
}
```

直接注入 `OrderServiceClient` 调用，就像调用本地方法一样，框架负责 HTTP 请求构建和负载均衡。

**Feign 工作原理：**

1. 为接口创建 JDK 动态代理
2. 解析注解生成 HTTP 请求模板
3. 通过 Ribbon/LoadBalancer 选择目标实例
4. 通过 HttpMessageConverter 处理序列化/反序列化

###### 4. 如何实现服务负载均衡（Ribbon/LoadBalancer）?

负载均衡决定请求如何分发到多个服务实例，分为客户端负载均衡和服务端负载均衡两种。Spring Cloud 采用客户端负载均衡，每个调用方自己决定调哪个实例。

**常见负载均衡策略：**

- **轮询**（默认）：依次分发，简单公平，适合性能相近的服务器
- **随机**：随机选择，简单，避免热点问题
- **加权轮询**：根据权重分发，适合异构服务器环境
- **最少连接**：优先选择当前连接数最少的实例，适合长连接服务
- **响应时间加权**：响应越快权重越高，自动适应实例性能差异

**Spring Cloud LoadBalancer（现代替代 Ribbon）：**

```java
@Bean
@LoadBalanced
public WebClient.Builder loadBalancedWebClientBuilder() {
    return WebClient.builder();
}

// Reactive 方式调用
Mono<Product> product = webClientBuilder.build()
        .get()
        .uri("http://product-service/products/{id}", id)
        .retrieve()
        .bodyToMono(Product.class);
```

###### 5. 如何实现服务熔断降级（Hystrix/Sentinel）?

熔断降级是微服务故障隔离的关键机制，防止单个服务故障扩散导致级联雪崩。

**熔断器三个状态（状态机模型）：**

- **CLOSED**：正常状态，请求正常放行
- **OPEN**：熔断状态，直接拒绝请求，立即返回降级结果
- **HALF_OPEN**：半开状态，试探性放行少量请求，成功则转回 CLOSED，失败则重新 OPEN

**Sentinel 配置（推荐，阿里开源，功能更强）：**

```java
@SentinelResource(
    value = "getUserInfo",
    blockHandler = "handleBlock",  // 流控/熔断时的处理方法
    fallback = "handleFallback"    // 业务异常的降级方法
)
public User getUserById(Long userId) {
    return userClient.getUser(userId);
}

public User handleBlock(Long userId, BlockException ex) {
    log.warn("请求被限流或熔断", ex);
    return User.defaultUser();
}
```

**动态规则配置：**

```java
FlowRule rule = new FlowRule("getUserInfo");
rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
rule.setCount(100);  // QPS 阈值
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_WARM_UP);  // 冷启动
FlowRuleManager.loadRules(Arrays.asList(rule));
```

###### 6. 如何实现 API 网关（Gateway/Zuul）?

API 网关是微服务的统一入口，负责路由转发、鉴权、限流、监控等横切关注点，让业务服务专注自身逻辑。

**Spring Cloud Gateway 核心概念：**

- **Route（路由）**：包含目标 URI、断言和过滤器的完整路由配置
- **Predicate（断言）**：决定请求是否匹配某条路由（路径、Header、Method 等）
- **Filter（过滤器）**：对请求和响应进行处理（鉴权、限流、日志、修改请求头等）

**路由配置示例：**

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service      # lb:// 表示客户端负载均衡
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1           # 去除路径前缀
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10
                redis-rate-limiter.burstCapacity: 20
```

**自定义全局过滤器（如 JWT 鉴权）：**

```java
@Component
public class AuthFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (!isValidToken(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() { return -1; }  // 值越小优先级越高
}
```

###### 7. 如何实现分布式配置中心?

分布式配置中心解决多服务、多环境下配置集中管理和动态刷新的问题，避免每次改配置都要重启服务。

**Spring Cloud Config 方案：**

服务端：

```java
@SpringBootApplication
@EnableConfigServer  // 启用配置服务器
public class ConfigServerApplication { ... }
```

客户端通过 `bootstrap.yml` 指定配置中心地址（优先于 `application.yml` 加载）：

```yaml
spring:
  cloud:
    config:
      uri: http://config-server:8888
      name: user-service
      profile: dev
      label: master  # Git 分支
```

动态刷新（无需重启）：

```java
@RefreshScope   // Bean 上加此注解，配置变更后自动重新绑定
@RestController
public class UserController {
    @Value("${app.feature.enabled:false}")
    private boolean featureEnabled;
}
```

触发刷新：`POST /actuator/refresh`（单节点）或 Spring Cloud Bus 广播到所有实例。

**Nacos 配置中心（推荐，更易用）：**

```yaml
spring:
  cloud:
    nacos:
      config:
        server-addr: localhost:8848
        file-extension: yaml
        shared-configs:
          - data-id: common.yaml
            refresh: true  # 开启动态刷新
```

Nacos 配置变更实时推送，无需手动触发 refresh 端点。

###### 8. 如何实现分布式链路追踪（Sleuth/Zipkin）?

链路追踪用于监控微服务调用链，定位性能瓶颈和错误节点，能直观看到「一个请求经过了哪些服务、每个服务耗时多少」。

**核心概念：**

- **Trace**：一次完整请求的全链路（贯穿所有服务）
- **Span**：链路中每个服务的处理单元（有开始时间、结束时间、名称）
- **TraceId**：标识整条链路的唯一 ID，在所有服务间传播
- **SpanId**：标识某个服务处理单元的 ID

**Sleuth + Zipkin 集成：**

```yaml
spring:
  sleuth:
    sampler:
      probability: 1.0  # 采样率，生产环境建议 0.1（10%）
    zipkin:
      base-url: http://zipkin-server:9411
```

引入依赖后，Sleuth 自动在 HTTP 请求头、消息队列消息中传播 TraceId/SpanId，无需改业务代码。

**自定义业务 Span（追踪关键业务操作）：**

```java
Span span = tracer.nextSpan().name("order-process").start();
try (Tracer.SpanInScope ws = tracer.withSpanInScope(span)) {
    // 业务处理
    inventoryService.reserveStock(order);
    paymentService.processPayment(order);
} finally {
    span.end();
}
```

Zipkin UI 提供服务依赖图、时序图和错误链路分析，是排查微服务性能问题的利器。

###### 9. 如何实现分布式锁?

分布式锁解决跨进程/跨服务的资源争用问题，防止多个实例同时操作同一资源。

**Redis 分布式锁（Redisson 实现，推荐）：**

```java
@Service
public class InventoryService {

    @Autowired
    private RedissonClient redissonClient;

    public boolean deductStock(String productId, int quantity) {
        RLock lock = redissonClient.getLock("lock:inventory:" + productId);
        try {
            boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);  // 等待 10s，锁超时 30s
            if (locked) {
                return doDeductStock(productId, quantity);
            }
            return false;
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();  // 必须判断是当前线程持有才解锁
            }
        }
    }
}
```

**Redisson 优于手动 SETNX 的地方：**

- **自动续期（WatchDog）**：后台线程自动延长锁超时，防止业务未完成锁就过期
- **可重入**：同一线程可以多次获取同一把锁
- **公平锁**：支持 `getFairLock()`，按请求顺序获取锁
- **联锁/红锁**：多节点锁，防止单 Redis 节点宕机

**ZooKeeper 分布式锁（一致性更强）：**

基于 ZK 临时顺序节点实现，天然公平，节点宕机后锁自动释放（临时节点），适合对强一致性要求高的场景。

###### 10. 如何实现消息队列集成（RabbitMQ/Kafka/RocketMQ）?

消息队列实现服务间异步通信和解耦，常见场景：订单创建后异步通知库存服务扣减库存。

**Spring Cloud Stream（统一编程模型，屏蔽底层 MQ 差异）：**

```java
// 生产者：发送消息
@Service
public class OrderEventPublisher {
    @Autowired
    private StreamBridge streamBridge;

    public void publishOrderCreated(Order order) {
        streamBridge.send("orderCreated-out-0", OrderEvent.from(order));
    }
}

// 消费者：处理消息
@Component
public class OrderEventConsumer {
    @Bean
    public Consumer<OrderEvent> orderCreated() {
        return event -> inventoryService.updateStock(event.getProductId(), -event.getQuantity());
    }
}
```

**绑定器配置（切换 MQ 只需改配置）：**

```yaml
spring:
  cloud:
    stream:
      bindings:
        orderCreated-out-0:
          destination: order-topic
          contentType: application/json
        orderCreated-in-0:
          destination: order-topic
          group: inventory-service  # 同 group 的消费者只有一个会处理
```

**消息可靠性三个关键环节：**

- **生产者侧**：开启 confirm 机制，确认消息已到达 Broker（`publisher-confirm-type: correlated`）
- **Broker 侧**：开启持久化 + 副本同步（Kafka `acks: all`）
- **消费者侧**：手动 ACK，业务处理成功后再确认

###### 11. 如何保证消息的可靠性传输?

消息可靠性是消息系统的基础要求，需要在「生产者 → Broker → 消费者」三个环节都做保障。

**端到端可靠性方案：**

**生产者侧（本地消息表方案）：**

```java
@Service
@Transactional
public class ReliableMessageService {

    public void saveOrderWithMessage(Order order) {
        // 业务数据和消息在同一事务内保存，要么都成功，要么都失败
        orderRepository.save(order);

        Message message = new Message();
        message.setTopic("order-created");
        message.setContent(JSON.toJSONString(order));
        message.setStatus("PENDING");
        messageRepository.save(message);  // 同事务
    }

    @Scheduled(fixedRate = 10000)
    public void retryPendingMessages() {
        // 定时扫描 PENDING 状态消息，重新发送
        messageRepository.findByStatus("PENDING").forEach(this::retrySendMessage);
    }
}
```

**Broker 侧：** RabbitMQ 开启持久化（队列 + 消息 durable）；Kafka 配置 `acks=all`、`replication.factor≥3`。

**消费者侧：** 关闭自动 ACK，业务处理成功后手动确认；处理失败进入死信队列，而不是无限重试。

###### 12. 如何处理消息重复消费?

在分布式环境中，消息重复消费几乎是不可避免的（网络重试、消费者重启等），需要在消费端做幂等性处理。

**方案一：Redis 原子操作（最常用）：**

```java
@Service
public class IdempotentConsumer {

    public boolean processOrder(OrderEvent event) {
        String key = "order:idempotent:" + event.getOrderId();

        // SETNX 原子操作，设置成功说明是第一次处理
        Boolean absent = redisTemplate.opsForValue()
            .setIfAbsent(key, "processed", Duration.ofMinutes(30));

        if (Boolean.TRUE.equals(absent)) {
            return doProcessOrder(event);  // 第一次，正常处理
        } else {
            log.warn("重复消息，跳过处理: {}", event.getOrderId());
            return true;  // 重复消息，直接返回成功
        }
    }
}
```

**方案二：数据库唯一约束（最可靠）：**

```java
@Table(name = "orders", uniqueConstraints = {
    @UniqueConstraint(columnNames = {"orderNo"})
})
```

插入时利用唯一键约束，重复消息会抛 `DuplicateKeyException`，捕获后忽略即可。

**方案三：乐观锁版本号：** 更新时带上版本号条件，版本不匹配则更新失败，通过版本号保证幂等性。

**实践建议：** 生产环境推荐 Redis + 数据库唯一约束双重保障，Redis 做快速拦截，数据库做最终防线。

###### 13. 如何实现服务限流?

限流保护系统免受过载请求影响，是稳定性的重要防线。限流可以在网关层和应用层两个位置实施。

**网关层限流（Spring Cloud Gateway + Redis 令牌桶）：**

```yaml
filters:
  - name: RequestRateLimiter
    args:
      redis-rate-limiter.replenishRate: 10   # 每秒补充 10 个令牌
      redis-rate-limiter.burstCapacity: 20   # 桶容量（允许突发流量）
      key-resolver: "#{@userKeyResolver}"    # 按 userId 限流
```

**应用层限流（Guava RateLimiter）：**

```java
@Service
public class RateLimitService {

    // 令牌桶算法，每秒生成 100 个令牌
    private final RateLimiter rateLimiter = RateLimiter.create(100.0);

    public void processRequest(Request request) {
        if (!rateLimiter.tryAcquire()) {
            throw new RateLimitException("请求过于频繁，请稍后重试");
        }
        handleRequest(request);
    }
}
```

**Sentinel 限流（功能最强大）：** 支持 QPS 限流、并发线程数限流、冷启动、匀速排队等多种控制策略，还可配置集群限流，是阿里开源的生产级限流组件。

**限流算法对比：**

- **计数器**：简单，但存在边界问题
- **滑动窗口**：解决计数器边界问题
- **令牌桶**：允许突发流量，超过桶容量则限流（RateLimiter 默认）
- **漏桶**：匀速流出，不允许突发，适合匀速处理场景

###### 14. 如何实现分布式 ID 生成?

分布式 ID 需要满足：全局唯一、趋势递增（对数据库索引友好）、高性能、高可用。

**Snowflake 算法（最经典）：**

64 位 Long 类型 ID，由三部分组成：

- 41 位时间戳（毫秒级，可用约 69 年）
- 10 位工作机器 ID（5 位数据中心 + 5 位机器 ID，支持 1024 个节点）
- 12 位序列号（每毫秒可生成 4096 个）

优点：不依赖外部存储，单机生成性能极高；趋势递增，对数据库索引友好。
缺点：依赖时钟，时钟回拨会导致 ID 重复（需要处理回拨问题）。

**数据库号段模式（美团 Leaf）：**

每次从数据库批量获取一段 ID（如 1-1000），在内存中分配，减少数据库访问。数据库中只记录当前 max_id 和 step，性能好，不依赖时钟。

**选型建议：**

- 单数据中心、对时钟依赖可接受 → Snowflake
- 需要更高可用性、不想处理时钟问题 → 号段模式（Leaf）
- 业务要求绝对有序 → 数据库自增（性能有限）

###### 15. 如何实现服务优雅下线?

服务优雅下线确保服务停止时不丢失正在处理的请求，是生产环境发布和容器调度的关键能力。

**Spring Boot 内置优雅关闭（2.3+ 版本）：**

```yaml
server:
  shutdown: graceful  # 等待当前请求处理完成再关闭
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 最长等待 30s，超时强制关闭
```

**完整的优雅下线流程：**

1. **接收关闭信号**（`SIGTERM` 或 `/actuator/shutdown`）
2. **从注册中心摘除**：向 Eureka/Nacos 发送下线请求，停止接收新流量（注意注册中心的缓存延迟）
3. **等待进行中的请求完成**：最长等待 30s（可配置），超时后强制关闭
4. **关闭线程池**：等待异步任务完成
5. **关闭数据库连接池**
6. **资源清理**

**注意事项：**

- 从注册中心下线到 Consumer 端感知有延迟（缓存刷新间隔），可能仍有少量请求打过来，需要靠 graceful shutdown 兜底
- K8s 环境使用 `preStop` 钩子等待一段时间，给注册中心感知下线留出时间
- `@PreDestroy` 注解可以在 Bean 销毁前执行清理逻辑
