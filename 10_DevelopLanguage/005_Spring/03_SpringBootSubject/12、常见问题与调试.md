###### 1. 如何解决循环依赖问题?

循环依赖是指 Bean A 依赖 Bean B、Bean B 反过来又依赖 Bean A，形成环路。Spring 通过**三级缓存**解决了大多数情况下的循环依赖，但构造器注入场景除外。

**三级缓存机制**：Spring 内部维护三个 Map 来处理 Bean 的创建状态：

```java
// 一级缓存：完整的 Bean 实例（已完成初始化）
Map<String, Object> singletonObjects;
// 二级缓存：早期 Bean 引用（实例化了但属性还没注入）
Map<String, Object> earlySingletonObjects;
// 三级缓存：ObjectFactory，用于生成早期引用（支持 AOP 代理）
Map<String, ObjectFactory<?>> singletonFactories;
```

创建 Bean A 时，先把它的工厂放进三级缓存，再去初始化属性——发现依赖 Bean B，就去创建 B。创建 B 时发现依赖 A，从三级缓存里拿到 A 的工厂生成早期引用，B 顺利完成创建并进入一级缓存，A 拿到 B 的引用后也完成初始化。

**解决方案**：

最推荐的方式是**Setter 注入**：

```java
@Service
public class UserService {
    private OrderService orderService;

    @Autowired
    public void setOrderService(OrderService orderService) {
        this.orderService = orderService;
    }
}
```

Setter 注入先完成实例化再做属性注入，给三级缓存有机会介入。**构造器注入**则在实例化阶段就要求依赖已经存在，三级缓存帮不上忙，所以会直接报错。

`@Lazy` 延迟注入也能解决：`@Autowired @Lazy private ServiceB serviceB`，让 B 在第一次真正调用时才初始化。但更彻底的做法是**重构设计**，把两个 Bean 的公共逻辑提取到第三个 Bean 里，消除真正的循环依赖。

从 **Spring Boot 2.6.0** 开始，循环依赖默认被禁止，启动时直接报错，强迫开发者重构。如果确实要临时兼容旧代码，可以加：

```yaml
spring:
  main:
    allow-circular-references: true
```

但这只是临时方案，根本上还是要改设计。

###### 2. 如何解决端口占用问题?

端口占用是开发和运维中最常见的启动故障，排查和解决都很简单。

**定位占用进程**：

```bash
# Linux/Mac
lsof -i :8080
netstat -tulpn | grep 8080

# Windows
netstat -ano | findstr 8080
taskkill /PID <PID> /F
```

找到 PID 后直接 `kill -9 <PID>` 干掉就行。

**配置层面的解决方案**：

开发阶段可以配 `server.port=0` 让 Spring Boot 随机选一个可用端口，微服务场景下也很常用；生产环境建议用环境变量控制端口，方便容器化部署：

```yaml
server:
  port: ${PORT:8080}   # 优先读环境变量 PORT，没有就用默认 8080
```

多环境配置也是标准做法：

```yaml
---
spring:
  profiles: dev
server:
  port: 8080
---
spring:
  profiles: test
server:
  port: 8081
```

如果是容器化部署，端口冲突问题基本上在容器编排层就解决了，本地开发更多是不小心有进程没关掉。

###### 3. 如何解决类加载冲突?

类加载冲突通常是项目里引入了两个版本的同一个库，运行时加载了"错误"的版本，导致 `NoSuchMethodError`、`ClassCastException` 等诡异错误。

**排查依赖冲突**：

```bash
# Maven 查看完整依赖树
mvn dependency:tree -Dverbose

# 只看某个库的冲突
mvn dependency:tree -Dincludes=com.google.guava:guava

# Gradle
gradle dependencies --configuration compileClasspath
```

在树形输出里找到标有 `(omitted for conflict with x.x.x)` 的依赖，说明有版本冲突。

**排除冲突版本**：找到引入旧版本的那个传递依赖，在父依赖上加 `<exclusion>` 排除掉：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>com.fasterxml.jackson.core</groupId>
            <artifactId>jackson-databind</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

或者在 `<dependencyManagement>` 里统一锁定版本，强制整个项目使用同一个版本。

**调试类加载**：如果不确定加载了哪个版本的类，可以打印类的来源：

```java
// 看这个类是从哪个 JAR 加载的
System.out.println(SomeClass.class.getProtectionDomain().getCodeSource().getLocation());
```

Spring Boot 的 Fat JAR 使用自定义类加载器（`LaunchedURLClassLoader`），应用代码和 BOOT-INF/lib 里的依赖都由它加载，跟外部容器的类加载器隔离，大多数时候不会互相干扰。

###### 4. 如何定位 Bean 注入失败问题?

Bean 注入失败时，Spring 的错误信息通常已经说得比较明确了，关键是要会读错误信息、会系统排查。

**常见错误类型**：
- `NoSuchBeanDefinitionException`：容器里根本没有这个 Bean，检查是不是忘了加注解、扫描路径没覆盖到、`@Conditional` 条件不满足
- `NoUniqueBeanDefinitionException`：同类型有多个 Bean，没指定用哪个，加 `@Primary` 或 `@Qualifier` 解决
- `BeanCreationException`：Bean 在初始化过程中出错，看 `caused by` 里面的根因

**排查思路**：

开启 `debug=true`（在 `application.properties` 里），Spring Boot 会打印详细的自动配置报告，列出哪些自动配置生效了、哪些没生效以及原因。也可以通过 Actuator 的 `/actuator/conditions` 端点在线查看：

```properties
debug=true
management.endpoints.web.exposure.include=conditions
```

**手动诊断**：通过 `ApplicationContext` 查看容器里实际注册了哪些 Bean：

```java
@PostConstruct
public void diagnose() {
    String[] beanNames = context.getBeanDefinitionNames();
    Arrays.sort(beanNames);
    Arrays.stream(beanNames).forEach(System.out::println);
}
```

**条件配置检查**：如果用了 `@ConditionalOnProperty`、`@ConditionalOnClass` 这类条件注解，要确认对应的配置属性值是否正确、依赖库是否在类路径上。用 `@SpringBootTest` 跑一个最小化的集成测试，能快速复现和定位问题。

###### 5. 如何查看自动配置报告?

自动配置报告是理解 Spring Boot 为你做了什么、没做什么的最直接方式，排查"为什么 Bean 没有创建"的时候特别有用。

**开启自动配置报告**：最简单的方式是在 `application.properties` 里加一行：

```properties
debug=true
```

启动后控制台会打印 `CONDITIONS EVALUATION REPORT`，分为两部分：

```
Positive matches（生效的配置）:
   DataSourceAutoConfiguration matched:
      - @ConditionalOnClass found required classes: javax.sql.DataSource
      - @ConditionalOnProperty (spring.datasource.url) found

Negative matches（未生效的配置）:
   RabbitAutoConfiguration:
      Did not match:
         - @ConditionalOnClass did not find required class: com.rabbitmq.client.Channel
```

看到这个，你就能清楚地知道某个自动配置生效了，原因是什么；哪些没生效，是缺了哪个类还是哪个配置属性。

**通过 Actuator 端点查看**：生产环境不适合开 debug 日志，可以改用 Actuator 端点：

```properties
management.endpoints.web.exposure.include=conditions
```

访问 `/actuator/conditions` 就能看到 JSON 格式的条件评估报告，随时查看，不影响日志量。

这个报告是解决"自动配置没生效"、"Bean 没有创建"等问题的第一个应该看的地方。

###### 6. 如何开启 Debug 日志?

Debug 日志是排查框架内部行为、定位问题的利器，但要注意粒度控制，不然日志量会大到让人崩溃。

**配置文件方式**（最常用）：

```yaml
logging:
  level:
    org.springframework: DEBUG           # Spring 框架整体
    org.springframework.web: DEBUG       # 只看 Web 层
    org.hibernate.SQL: DEBUG             # 打印 SQL 语句
    org.hibernate.type.descriptor.sql.BasicBinder: TRACE  # 打印 SQL 参数值
    com.example: DEBUG                   # 自己的代码
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n"
  file:
    name: logs/app.log
```

**Actuator 动态调整日志级别**（不重启应用）：

生产环境遇到问题时，不能重启应用，可以通过 Actuator 的 `loggers` 端点临时调整日志级别：

```bash
# 查看当前日志级别
GET /actuator/loggers/com.example.service

# 临时提升到 DEBUG
POST /actuator/loggers/com.example.service
Content-Type: application/json
{"configuredLevel": "DEBUG"}

# 问题排查完后恢复
POST /actuator/loggers/com.example.service
{"configuredLevel": "INFO"}
```

记得开放 `loggers` 端点：`management.endpoints.web.exposure.include=loggers`。

**技巧**：排查 SQL 问题时开 Hibernate 的 SQL 日志；排查请求处理问题时开 `org.springframework.web.servlet.DispatcherServlet`；排查自动配置时开 `debug=true`。不要全量开 `org.springframework: TRACE`，那个量真的很大。

###### 7. 如何进行远程调试?

远程调试允许你把本地 IDE 的断点接到远程运行的 JVM 上，对于排查测试环境或生产环境上才能复现的问题非常有用。

**JVM 端启动调试端口**：

```bash
# 标准方式，5005 是调试端口，suspend=n 表示不等待调试器连接就继续启动
java -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005 \
     -jar application.jar
```

如果想让 JVM 在调试器连上之前暂停，把 `suspend=n` 改成 `suspend=y`，适合排查启动阶段的问题。

**IDE 配置（IntelliJ IDEA）**：Run → Edit Configurations → Add → Remote JVM Debug，填入远程服务器 IP 和端口 5005，点 Debug 就能连上，然后正常打断点。

**Spring Boot DevTools 远程开发**（开发环境用）：

```yaml
spring:
  devtools:
    remote:
      secret: mydebugsecret
    restart:
      enabled: true
```

DevTools 远程模式支持代码修改后自动推送到远程并触发重启，比纯 JVM 调试更方便，但只建议在内网开发环境使用。

**安全提醒**：调试端口一定不能对公网开放，生产环境调试要走 SSH 隧道转发，用完立刻关闭。开着调试端口的生产服务器是重大安全隐患——任何人连上去都可以看变量、改程序执行流程。

###### 8. 如何解决 OOM 问题?

OOM（OutOfMemoryError）是 JVM 内存耗尽的直接体现，不同的 OOM 类型对应不同的原因，要先判断类型再对症下药。

**OOM 类型识别**：
- `java.lang.OutOfMemoryError: Java heap space`：堆内存不足，可能是内存泄漏，也可能是堆设置太小
- `java.lang.OutOfMemoryError: Metaspace`：元空间不足，通常是动态生成大量类（反射、CGLIB）
- `java.lang.OutOfMemoryError: unable to create new native thread`：线程数超限，线程池配置不当

**关键 JVM 参数**：

```bash
-XX:+HeapDumpOnOutOfMemoryError    # OOM 时自动生成堆转储
-XX:HeapDumpPath=/tmp/heapdump/    # 转储文件路径
-XX:OnOutOfMemoryError="kill -9 %p"  # OOM 后自动重启（Kubernetes 下）
```

**分析工具链**：

jmap 生成堆转储（如果没配自动转储）：
```bash
jmap -dump:format=b,file=heap.bin <pid>
```

Eclipse MAT 是分析堆转储最好用的工具，它的 "Leak Suspects" 功能能自动帮你找内存泄漏嫌疑对象；`jmap -histo <pid>` 可以快速看对象数量分布，不用生成完整堆转储。

**常见 OOM 原因**：大量字符串拼接（应用 `StringBuilder`）；大文件一次性加载进内存（应该流式处理）；无限增长的缓存（配置 LRU 淘汰策略）；连接池、线程池对象泄漏。

###### 9. 如何解决 CPU 占用过高问题?

CPU 飙高通常是死循环、频繁 Full GC、大量线程上下文切换或者某段算法效率太低导致的。

**排查流程**：

```bash
# 第一步：找出高 CPU 的 Java 进程
top

# 第二步：看进程内哪些线程 CPU 最高（-H 是线程模式）
top -H -p <PID>

# 第三步：把十进制线程 ID 转成十六进制
printf "%x\n" <线程ID>

# 第四步：抓线程堆栈
jstack <PID> > thread_dump.txt
# 在文件里搜索十六进制线程 ID，找到对应线程的调用栈
```

找到调用栈之后，看方法名基本就能判断是哪段业务代码在疯狂消耗 CPU。

**常见原因**：
- **死循环**：while 循环里条件判断有 bug，或者递归调用没有正确的终止条件
- **频繁 Full GC**：内存不足导致 GC 线程持续工作，CPU 全被 GC 吃掉。`jstat -gcutil <pid> 1000` 可以实时看 GC 频率
- **大量正则表达式**：正则匹配在回溯严重时 CPU 消耗极高，某些写法会导致指数级回溯
- **序列化/反序列化**：高并发下大量 JSON 序列化操作，注意复用 `ObjectMapper`

**监控预防**：

配置线程 CPU 时间监控，定期打印高 CPU 线程的堆栈信息，异常时自动告警。Arthas 工具箱的 `thread -n 3` 命令可以直接列出 CPU 占用最高的 3 个线程及其堆栈，不需要手动做进制转换，强烈推荐。

###### 10. 如何排查接口响应慢的问题?

接口慢要全链路看，可能是数据库、可能是外部调用、可能是自身逻辑复杂，得先定位慢在哪里。

**接入 APM 监控**：链路追踪是排查接口慢问题最有效的手段，Sleuth + Zipkin 或者 SkyWalking 都能做到为每个请求生成 TraceId，记录每个跨度（Span）的耗时，层层拆解到底哪段慢。

**本地定位方案**——用 AOP 监控接口耗时：

```java
@Aspect
@Component
@Slf4j
public class PerformanceAspect {

    @Around("@within(org.springframework.web.bind.annotation.RestController)")
    public Object monitor(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        String method = pjp.getSignature().toShortString();
        try {
            return pjp.proceed();
        } finally {
            long duration = System.currentTimeMillis() - start;
            if (duration > 1000) {
                log.warn("慢接口: {} 耗时 {}ms", method, duration);
            }
        }
    }
}
```

**Actuator 指标监控**：

```yaml
management:
  metrics:
    distribution:
      percentiles-histogram:
        http.server.requests: true
      sla:
        http.server.requests: 100ms,500ms,1s
```

配置好之后接 Prometheus + Grafana，能实时看到各接口的 P50/P95/P99 响应时间分布。

**常见慢接口原因**：
- SQL 没有索引，走全表扫描
- N+1 查询问题（List 里每个对象都单独查一次）
- 调用外部 HTTP 接口没有超时控制，下游超时了就一直等
- 对大数据量做了全量内存计算，应该分页或异步
- 锁竞争严重，多线程排队等锁

###### 11. 如何解决数据库连接池耗尽问题?

连接池耗尽时，新的数据库操作会一直等待，直到超时抛 `Unable to acquire JDBC Connection` 异常。这通常是连接泄漏或者连接数配置不够导致的。

**HikariCP 连接泄漏检测**：HikariCP 自带泄漏检测功能，超过指定时间还没归还的连接会打印警告日志：

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000       # 获取连接的最大等待时间
      idle-timeout: 600000            # 空闲连接超时
      max-lifetime: 1800000           # 连接的最大生命周期
      leak-detection-threshold: 60000  # 60秒未归还，打印泄漏警告
```

**连接泄漏的常见原因**：手动使用 `Connection` 对象时没有在 finally 块或 try-with-resources 里关闭；事务异常时连接没有正确释放；某些框架的非标准用法导致连接无法归还到池子里。

**连接池监控**：通过 HikariCP 的 MXBean 可以实时查看连接池状态：

```java
HikariPoolMXBean pool = hikariDataSource.getHikariPoolMXBean();
log.info("活跃连接: {}, 空闲连接: {}, 等待线程: {}",
    pool.getActiveConnections(), pool.getIdleConnections(), pool.getThreadsAwaitingConnection());
```

当活跃连接持续接近最大值且等待线程数不断增加，基本可以确定是连接泄漏。接 Actuator 的 `datasource` 端点和 Prometheus 指标，能在告警阈值触发时第一时间发现问题。

**紧急处置**：如果是线上连接池耗尽，可以临时调大 `maximum-pool-size`（通过动态配置或重启），同时快速定位泄漏代码，根治才是关键。

###### 12. 如何处理线程死锁问题?

死锁是多个线程互相等待对方持有的锁，谁也不肯先释放，形成永久等待。

**死锁检测**：JVM 自带了死锁检测能力，`ThreadMXBean` 的 `findDeadlockedThreads()` 方法能直接找出处于死锁状态的线程，可以做成定时任务定期检测：

```java
@Scheduled(fixedRate = 30000)
public void detectDeadlock() {
    ThreadMXBean threadBean = ManagementFactory.getThreadMXBean();
    long[] deadlockedIds = threadBean.findDeadlockedThreads();
    if (deadlockedIds != null) {
        ThreadInfo[] infos = threadBean.getThreadInfo(deadlockedIds);
        for (ThreadInfo info : infos) {
            log.error("死锁线程: {}, 等待锁: {}, 锁被持有者: {}",
                info.getThreadName(), info.getLockName(), info.getLockOwnerName());
        }
    }
}
```

也可以用 `jstack <pid>` 命令，输出里会明确标注 "Found one Java-level deadlock"。

**死锁预防**：

最关键的原则是**按固定顺序加锁**，所有代码都按同样的顺序申请多个锁，就不会出现 A 持有 lock1 等 lock2、B 持有 lock2 等 lock1 的情况。

使用 `ReentrantLock` 的 `tryLock(timeout)` 方法加超时限制，申请锁超时就放弃，避免永久等待：

```java
if (lock1.tryLock(500, TimeUnit.MILLISECONDS)) {
    try {
        if (lock2.tryLock(500, TimeUnit.MILLISECONDS)) {
            try {
                // 执行业务
            } finally { lock2.unlock(); }
        }
    } finally { lock1.unlock(); }
}
```

更好的做法是从架构上减少锁的使用：能用无锁数据结构（`ConcurrentHashMap`、`AtomicLong`）的就不用显式锁；能用数据库乐观锁（`@Version`）的就不用 Java 级别的锁；消息队列异步化可以彻底避免某些竞争场景。
