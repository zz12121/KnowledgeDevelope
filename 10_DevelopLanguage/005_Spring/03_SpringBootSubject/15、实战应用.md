###### 1. 如何设计一个高可用的微服务架构?

高可用微服务架构的核心是多层次的容错能力，从网络层到数据层每个环节都要有容错设计，单点故障不能导致系统整体不可用。

**服务治理与容错**：注册中心用 Eureka 或 Nacos，客户端本地缓存一份服务列表，即使注册中心短暂不可用，已有的服务发现信息仍然可以用。熔断器用 Hystrix 或 Resilience4j 包裹下游调用，错误率超过阈值（默认 50%）后进入熔断状态，直接返回降级结果，保护上游不被慢服务拖垮。负载均衡用 Ribbon（或 Spring Cloud LoadBalancer），支持轮询、权重、最少连接等策略，结合健康检查动态剔除不健康实例。

**数据高可用**：

分布式事务用 Saga 模式，每个服务执行本地事务，失败时通过补偿操作回滚：

```java
@Saga
public class OrderSaga {
    @StartSaga
    @Compensating(compensateMethod = "cancelOrder")
    public void createOrder(Order order) {
        // 1. 冻结库存
        // 2. 扣减账户
    }

    public void cancelOrder(Order order) {
        // 补偿：释放库存，退还账户余额
    }
}
```

数据库做主从复制，读流量走从库，写流量走主库，通过 `AbstractRoutingDataSource` 实现动态路由。Redis 集群用 CRC16 分片，配置 Sentinel 做故障自动转移，主节点宕机时自动选举新主。

**架构层高可用**：异地多活部署，通过 GSLB 按地域调度流量；关键服务做至少 3 节点部署，Kubernetes HPA 根据 CPU/内存指标自动扩缩容；接 APM 和告警系统，异常时秒级通知。

###### 2. 如何实现秒杀系统?

秒杀的核心挑战是短时间内极大并发量打在一个资源（库存）上，设计思路是**层层过滤，把请求挡在前面，减少打到后端的量**。

**流量削峰**：商品详情页静态化，放 CDN，大部分"看"的流量在 CDN 层就消化了，不用打到服务器。前置验证码或滑块挑战，让恶意刷子脚本的成本更高，也能把并发请求在时间轴上展开。网关层用 Sentinel 或 Nginx 做限流，超过系统承受能力的请求直接返回"活动火爆，请稍后"。

**库存扣减**：用 Redis + Lua 脚本做原子扣减，保证高并发下库存不超卖：

```java
String script =
    "if redis.call('exists', KEYS[1]) == 1 then " +
    "  local stock = tonumber(redis.call('get', KEYS[1])); " +
    "  if stock > 0 then redis.call('decr', KEYS[1]); return 1; end; " +
    "end; return 0;";
Long result = redisTemplate.execute(
    new DefaultRedisScript<>(script, Long.class),
    Arrays.asList("sec_kill:" + productId));
```

Redis 扣减成功后，把秒杀请求写入消息队列（RocketMQ），异步处理订单创建、支付等后续流程，数据库压力大幅降低。数据库层再用乐观锁兜底：`UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0`，最终防超卖。

**系统保护**：整条链路都要有限流和降级。Kubernetes HPA 配置好，活动开始前提前扩容；活动后流量回落，自动缩容释放资源。

###### 3. 如何实现分布式任务调度?

分布式任务调度要解决两个核心问题：**同一个任务只由一个节点执行**（避免重复执行），以及**节点宕机后任务能自动转移**（可靠性）。

**主流方案**：

XXL-Job 是国内用得最多的分布式任务调度框架，调度中心集中管理任务，执行器分布在各个服务实例上：

```java
@XxlJob("orderTimeoutJob")
public ReturnT<String> handleOrderTimeout(String param) {
    int shardIndex = XxlJobHelper.getShardIndex();   // 分片索引
    int shardTotal = XxlJobHelper.getShardTotal();   // 总分片数
    // 只处理 id % shardTotal == shardIndex 的订单，多节点不重复
    orderService.closeTimeoutOrders(shardIndex, shardTotal);
    return ReturnT.SUCCESS;
}
```

分片路由让每个节点只处理数据的一部分，既支持水平扩展，又保证不重复处理。

**幂等性保证**：任务执行前先用数据库乐观锁或 Redis SETNX 抢锁，抢到才执行：`UPDATE task SET status='running' WHERE id=? AND status='pending'`，更新成功的节点才去执行任务，天然避免重复执行。

**容错**：调度中心自身做集群部署，通过数据库锁协调主节点；执行器宕机时，调度中心会把任务路由到其他健康节点重新执行；失败任务支持自动重试，告警到钉钉/企业微信。

**Quartz**：如果已经在用 Quartz，Spring Boot 提供了 `spring-boot-starter-quartz`，通过数据库持久化触发记录，天然支持集群模式。但 Quartz 的集群控制台不够直观，适合已有 Quartz 经验的团队。

###### 4. 如何实现电商订单系统?

订单系统的核心是**状态机管理**和**数据一致性**。订单状态多、流转复杂，用状态机能让逻辑更清晰、可扩展性更好。

**订单状态流转**：

```java
public enum OrderStatus {
    PENDING {
        @Override
        public OrderStatus pay() { return PAID; }
    },
    PAID {
        @Override
        public OrderStatus deliver() { return DELIVERED; }
        @Override
        public OrderStatus refund() { return REFUNDING; }
    },
    DELIVERED {
        @Override
        public OrderStatus complete() { return COMPLETED; }
    },
    REFUNDING {
        @Override
        public OrderStatus refundSuccess() { return REFUNDED; }
    },
    COMPLETED, REFUNDED, CANCELLED;

    public OrderStatus pay() { throw new IllegalStateException("当前状态不允许支付"); }
    public OrderStatus deliver() { throw new IllegalStateException(); }
    // ...
}
```

每个状态只暴露它允许的操作，非法状态转换直接抛异常，不需要到处写 if-else 判断。

**数据分片**：大促场景下订单量极大，按用户 ID 分库分表，`userId % 64` 路由到具体表，ShardingSphere 可以透明做路由。历史订单定期归档到 ClickHouse，OLAP 查询不影响线上库。

**关键一致性设计**：支付成功后通过 MQ 异步触发库存扣减、积分发放、通知推送，主流程快速返回；用本地消息表或 Outbox 模式保证 MQ 消息至少发出去一次，消费端做幂等处理（用订单号 + 操作类型作为幂等键）。

###### 5. 如何实现支付系统集成?

支付系统的特殊性在于金钱的流动，对安全性、一致性和可审计性要求极高。

**支付网关设计**：在业务系统和支付渠道之间加一层统一的支付网关，屏蔽不同渠道（支付宝、微信、银行卡）的接入差异。根据渠道权重和健康状态做路由，某个渠道不可用时自动熔断切换：

```java
// TCC 分布式事务保证资金安全
@Compensable(confirmMethod = "confirmDeduct", cancelMethod = "cancelDeduct")
public void tryDeduct(Account account, BigDecimal amount) {
    accountService.freeze(account, amount);  // Try：冻结资金
}

public void confirmDeduct(Account account, BigDecimal amount) {
    accountService.deduct(account, amount);  // Confirm：实际扣款
}

public void cancelDeduct(Account account, BigDecimal amount) {
    accountService.unfreeze(account, amount);  // Cancel：解冻
}
```

**安全性**：与三方渠道通信必须做签名验证，防止请求被篡改。持卡人信息用 `AES/GCM/NoPadding` 加密存储，密钥托管在 HSM（硬件安全模块）里，不放在代码和配置文件里。

**对账系统**：每天定时从各支付渠道下载对账文件，与系统内部流水逐笔比对，发现差异立即告警，是保证资金安全的最后一道防线。用 EasyExcel 流式解析对账文件，避免大文件 OOM。

###### 6. 如何实现短信验证码功能?

短信验证码表面简单，但如果不做防护，很容易被恶意刷接口，既损失费用又造成安全风险。

**防攻击**：发送前必须做图形验证码或行为验证，确认是人工操作；用 Redis 做频率控制，同一手机号每分钟最多发一次，每天最多发 N 次：

```java
String rateLimitKey = "sms:ratelimit:" + phone;
Boolean firstRequest = redisTemplate.opsForValue()
    .setIfAbsent(rateLimitKey, "1", 1, TimeUnit.MINUTES);
if (!Boolean.TRUE.equals(firstRequest)) {
    throw new BusinessException("发送频率过高，请稍后重试");
}
```

**验证码存储**：生成的验证码存 Redis，设置 5 分钟过期，key 格式为 `sms:code:{phone}`。验证时取出比对，验证成功立即删除，防止重放攻击。

**供应商路由**：接入多家短信服务商，主供应商优先，不可用时自动降级到备用供应商，降低单点依赖风险。发送逻辑异步化，不阻塞接口响应：

```java
CompletableFuture.runAsync(() -> smsProviderRouter.send(phone, code));
```

**监控**：统计发送成功率、失败率，异常时告警，防止出现大量发送失败用户无感知的情况。

###### 7. 如何实现邮件发送功能?

邮件功能核心是三件事：内容模板化、发送可靠性（失败重试）、批量时不 OOM。

**模板化**：用 Thymeleaf 生成邮件 HTML，支持个性化内容和国际化：

```java
Context context = new Context();
context.setVariable("username", username);
context.setVariable("verifyLink", link);
String html = templateEngine.process("email/verify.html", context);

MimeMessage message = mailSender.createMimeMessage();
MimeMessageHelper helper = new MimeMessageHelper(message, true, "UTF-8");
helper.setTo(to);
helper.setSubject("邮箱验证");
helper.setText(html, true);  // 第二个参数 true 表示 HTML 格式
mailSender.send(message);
```

大文件附件用 `FileSystemResource` 而不是把文件全读进内存，避免 OOM：

```java
helper.addAttachment("report.pdf", new FileSystemResource(filePath));
```

**可靠性**：把邮件任务放入消息队列，异步消费。失败后做指数退避重试，比如 1分钟、2分钟、4分钟依次重试，最多重试 3 次，超过后转人工处理队列，同时告警。

**批量场景**：大批量邮件（比如活动通知）要做限速，避免被邮件服务商封禁 IP。用 `@Scheduled` 定时任务控制发送速率，或者用 RateLimiter 做流量控制。

###### 8. 如何实现文件存储（本地/OSS）?

文件存储要用抽象接口屏蔽底层实现，本地存储和云存储对外提供一样的接口，切换时业务代码不需要改。

**存储抽象**：定义统一接口，用 `@Profile` 区分环境：

```java
public interface FileService {
    String upload(InputStream inputStream, String key, long size);
    InputStream download(String key);
    void delete(String key);
}

@Service
@Profile("local")
public class LocalFileService implements FileService {
    @Value("${file.upload.path}")
    private String basePath;

    @Override
    public String upload(InputStream input, String key, long size) {
        Path target = Paths.get(basePath, key);
        Files.createDirectories(target.getParent());
        Files.copy(input, target, StandardCopyOption.REPLACE_EXISTING);
        return target.toString();
    }
}

@Service
@Profile("prod")
public class OssFileService implements FileService {
    @Override
    public String upload(InputStream input, String key, long size) {
        ossClient.putObject("my-bucket", key, input, buildObjectMetadata(size));
        return ossClient.generatePresignedUrl("my-bucket", key, expireDate).toString();
    }
}
```

**大文件处理**：超过 5MB 的文件用分块上传（Multipart Upload），支持断点续传，网络中断不用从头开始；用 `TransferManager` 自动处理分块逻辑。

**安全访问**：私有文件通过后端生成临时预签名 URL（有效期 15 分钟），用户直接下载不经过后端，减少带宽压力，权限还在后端控制。上传后触发 CDN 预热，让热点资源尽快扩散到边缘节点。

###### 9. 如何实现数据导入导出（Excel/CSV）?

大数据量的 Excel 导入导出最容易 OOM，关键在于流式处理，绝对不能把几十万行数据一次性加载进内存。

**EasyExcel 流式导入**：

```java
public class UserImportListener extends AnalysisEventListener<UserImportDTO> {
    private final List<UserImportDTO> buffer = new ArrayList<>(1000);
    private final UserService userService;

    @Override
    public void invoke(UserImportDTO data, AnalysisContext context) {
        // 先做数据校验
        validateRow(data, context.readRowHolder().getRowIndex());
        buffer.add(data);
        if (buffer.size() >= 1000) {
            userService.batchInsert(buffer);   // 每 1000 条批量写入
            buffer.clear();
        }
    }

    @Override
    public void doAfterAllAnalysed(AnalysisContext context) {
        if (!buffer.isEmpty()) {
            userService.batchInsert(buffer);   // 处理剩余
        }
    }
}

// 触发导入
EasyExcel.read(inputStream, UserImportDTO.class, new UserImportListener(userService)).sheet().doRead();
```

**EasyExcel 流式导出**：

```java
public void exportUsers(HttpServletResponse response) throws Exception {
    response.setContentType("application/vnd.openxmlformats-officedocument.spreadsheetml.sheet");
    response.setHeader("Content-Disposition", "attachment;filename=users.xlsx");

    try (ExcelWriter writer = EasyExcel.write(response.getOutputStream(), UserExportDTO.class).build()) {
        WriteSheet sheet = EasyExcel.writerSheet("用户数据").build();
        // 分批查询，分批写入，内存始终只有一批数据
        int page = 0;
        List<UserExportDTO> data;
        do {
            data = userService.findPage(page++, 1000);
            writer.write(data, sheet);
        } while (data.size() == 1000);
    }
}
```

**数据校验**：导入时用 Bean Validation 做字段校验（`@NotBlank`、`@Pattern`、`@Size`），记录每行的错误信息，最终生成一份"导入失败明细"文件返回给用户，让用户能定位和修正问题。

###### 10. 如何实现全文搜索功能?

全文搜索不能用数据库的 LIKE 查询，LIKE 的前模糊查询无法走索引，几十万数据就会很慢。Elasticsearch 的倒排索引专门为全文搜索而生，性能和功能都远超关系型数据库。

**数据模型设计**：

```java
@Document(indexName = "products")
public class ProductDocument {
    @Id
    private String id;

    @Field(type = FieldType.Text, analyzer = "ik_max_word", searchAnalyzer = "ik_smart")
    private String name;    // 中文分词：ik_max_word 建索引，ik_smart 查询

    @Field(type = FieldType.Keyword)
    private String category;  // 精确匹配字段，不分词

    @Field(type = FieldType.Double)
    private Double price;
}
```

**搜索服务**：

```java
@Service
public class ProductSearchService {

    public SearchHits<ProductDocument> search(String keyword, String category, Pageable pageable) {
        BoolQueryBuilder query = QueryBuilders.boolQuery()
            .must(QueryBuilders.matchQuery("name", keyword))     // 全文匹配
            .filter(QueryBuilders.termQuery("category", category));  // 精确过滤

        NativeSearchQuery searchQuery = new NativeSearchQueryBuilder()
            .withQuery(query)
            .withHighlightFields(new HighlightBuilder.Field("name")
                .preTags("<em>").postTags("</em>"))   // 关键词高亮
            .withPageable(pageable)
            .withSort(SortBuilders.scoreSort())       // 按相关性排序
            .build();

        return elasticsearchTemplate.search(searchQuery, ProductDocument.class);
    }
}
```

**数据同步**：商品数据变更时，通过 MQ 异步通知 Elasticsearch 更新索引，或者用 Canal 监听 MySQL binlog 增量同步。不要在业务事务里同步写 ES，会增加延迟和耦合。

**性能优化**：`Filter` 查询结果可以被 ES 缓存（bitset 格式），频繁的类目过滤走 `Filter` 而不是 `Must`，能大幅提升查询性能。冷索引用 `_forcemerge` 合并 segment，减少查询时的 IO 开销。
