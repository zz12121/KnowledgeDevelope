## 一、Nginx vs Apache

| 对比维度 | Nginx | Apache |
|---------|-------|--------|
| 并发模型 | 事件驱动，异步非阻塞（epoll） | 多进程/多线程（同步阻塞） |
| 高并发内存 | 极低（数十 MB 处理万级并发） | 较高（千级并发即需 GB 级内存） |
| 静态资源性能 | 极强（sendfile 零拷贝） | 一般 |
| 动态内容 | 需反向代理到后端 | 可内嵌 mod_php/mod_python（直接处理） |
| 配置灵活性 | 集中式配置，不支持 .htaccess | 支持 .htaccess（目录级配置，灵活但慢） |
| 热重载 | ✅ `nginx -s reload`（不中断服务） | ✅ `apachectl graceful` |
| 模块热加载 | ❌ 需重新编译 | ✅ DSO 动态模块 |
| 内置模块丰富度 | 适中 | 极丰富（mod_rewrite/mod_security 等） |
| Windows 支持 | 弱（不支持多 Worker） | 好 |
| 学习成本 | 较低 | 中等 |
| 社区活跃度 | 极高 | 高 |

**选择原则**：
- **Nginx**：新项目标配，高并发静态资源/反向代理/API 网关首选，VPS 内存受限环境
- **Apache**：WordPress/Drupal 等依赖 `.htaccess` 的 CMS；遗留 `mod_php` 应用；CGI/Perl 老项目

## 二、Nginx vs HAProxy

| 对比维度 | Nginx | HAProxy |
|---------|-------|---------|
| 主要定位 | Web 服务器 + HTTP 反向代理 | 专业高性能负载均衡器 |
| 支持协议 | HTTP/HTTPS/TCP（1.9+）/UDP（1.15+）/Stream | HTTP/HTTPS/TCP/UDP（更全面的四层支持） |
| 负载均衡算法 | 5 种（RR/Weight/IP Hash/Least Conn/Hash） | 8 种（含 Random/最少响应时间/URI/Source） |
| 主动健康检查 | 需商业版（Nginx Plus）或第三方模块 | **原生内置**，功能完善（HTTP/TCP/外部脚本） |
| 会话保持 | ip_hash（基础）/Lua 实现 Cookie Stick | **原生 Cookie Stick**，支持插入 Cookie |
| 监控统计 | stub_status（基础，5 个数字） | **内置强大 Stats 页面**（实时连接/流量/后端状态） |
| 动态配置 | ❌ 必须 reload | ✅ Runtime API 动态增删后端节点 |
| 静态资源服务 | ✅ 强（本职工作） | ❌ 不支持 |
| SSL 终止 | ✅ 完善 | ✅ 完善 |
| 开源版本功能 | 中等（高级功能需 Nginx Plus） | 完整（开源版已包含企业级功能） |

**选择原则**：
- **Nginx**：需要静态资源服务 + 反向代理 + 负载均衡的一站式方案
- **HAProxy**：纯粹的高性能负载均衡、精细健康检查、需要 Runtime API 动态管理后端
- **国内大厂常见架构**：HAProxy 做核心 LB（或 LVS），Nginx 做七层接入层

## 三、Nginx vs LVS

| 对比维度 | Nginx | LVS（Linux Virtual Server） |
|---------|-------|---------------------------|
| 工作层级 | **七层**（应用层，解析 HTTP） | **四层**（传输层，转发 TCP/UDP 包） |
| 工作位置 | 用户空间 | **内核空间**（IPVS 内核模块） |
| 性能上限 | 数十万 QPS | **百万级并发**（接近硬件转发速度） |
| 内容感知 | ✅（可按 URL/Cookie/Header 路由） | ❌（不解析应用层内容） |
| SSL 卸载 | ✅ | ❌ |
| 会话保持 | ✅（多种方式） | 基于 IP 的简单保持 |
| 健康检查 | 被动检查（主动需模块） | 需配合 **Keepalived** |
| 配置复杂度 | 简单，配置文件直观 | 复杂（需要网络拓扑配置） |
| 高可用 | 通常配合 Keepalived | **必须** 配合 Keepalived（VIP 漂移） |

**LVS 三种工作模式**：

| 模式 | 原理 | 特点 |
|------|------|------|
| NAT | 修改 IP 包的源/目 IP | 性能最低，但客户端/服务器无需改动 |
| TUN（IP 隧道） | IP 包封装转发，响应直接返回客户端 | 性能高，服务器需支持 IP 隧道 |
| DR（直接路由） | 修改 MAC 地址，不修改 IP | **性能最高**，服务器和 LVS 必须在同一物理网段 |

**企业级高可用架构**：
```
互联网流量
    │
    ▼
[DNS/GSLB 全局负载均衡]  ← 跨地域流量调度
    │
    ▼
[LVS + Keepalived] × 2   ← 四层 VIP 高可用（Active/Standby，<1s 故障切换）
    │                       内核空间转发，100 万 QPS，不解析内容
    ▼
[Nginx 集群] × N         ← 七层处理（SSL 卸载/限流/缓存/路由/WAF）
    │
    ▼
[后端应用集群]           ← Spring Boot / Go / Node.js 服务
```

## 四、Nginx vs Tengine

| 对比维度 | Nginx | Tengine（阿里巴巴） |
|---------|-------|-------------------|
| 主动健康检查 | 需第三方模块 | **内置**（`check` 指令） |
| 动态 upstream | 需商业版 | 支持（Lua 集成） |
| 请求合并（PIPE） | 不支持 | 支持（减少后端连接数） |
| Lua 集成 | 需单独安装 OpenResty | 内置 Lua 支持 |
| 官方版本跟进 | **最新版**（官方维护） | 可能落后 Nginx 数个版本 |
| 社区活跃度 | **极高** | 较低（阿里内部需求驱动） |

**结论**：一般项目选**标准 Nginx + OpenResty**；阿里云用户直接使用 SLB（底层 Tengine）。

## 五、Nginx vs Spring Cloud Gateway

| 对比维度 | Nginx | Spring Cloud Gateway |
|---------|-------|---------------------|
| 定位 | 通用 Web 服务器/反向代理 | **微服务 API 网关** |
| 语言 | C | Java（Spring Boot） |
| 性能 | 极高（C 语言，epoll） | 高（Reactor 异步非阻塞） |
| 服务注册集成 | ❌ 不支持（需手动配置 upstream） | ✅ **自动集成** Nacos/Eureka/Consul |
| 动态路由 | ❌ 需 reload 或付费版 | ✅ **动态路由**（Nacos 热更新） |
| 限流熔断 | 基础（limit_req/limit_conn） | ✅ 集成 Sentinel/Resilience4j |
| 鉴权扩展 | Lua 脚本（OpenResty） | ✅ Java Filter（与 Spring Security 深度集成） |
| 链路追踪 | 需手动传递 Header | ✅ 自动集成 SkyWalking/Zipkin |
| 监控 | stub_status（基础） | ✅ 集成 Micrometer + Prometheus |
| 适用场景 | 接入层网关、静态资源、通用代理 | 微服务内部网关 |

**推荐架构**：
```
Internet → Nginx（接入层：SSL卸载/WAF/CDN回源）→ Spring Cloud Gateway（业务网关：鉴权/路由/限流）→ 微服务
```

## 六、常用运维命令速查

```bash
# 服务管理
nginx                     # 启动
nginx -s reload           # 优雅重载配置（不中断服务）
nginx -s reopen           # 重新打开日志文件（日志切割后执行）
nginx -s quit             # 优雅停止（处理完当前请求再退出）
nginx -s stop             # 强制停止

# 配置管理
nginx -t                  # 检查配置语法
nginx -T                  # 检查并输出完整配置
nginx -V                  # 查看版本和编译参数

# 进程信号
kill -HUP   $(cat /var/run/nginx.pid)   # 重载配置（等同 nginx -s reload）
kill -USR1  $(cat /var/run/nginx.pid)   # 重新打开日志文件
kill -USR2  $(cat /var/run/nginx.pid)   # 热升级二进制
kill -WINCH $(cat /var/run/nginx.pid)   # 优雅停止所有 Worker

# 日志查看
tail -f /var/log/nginx/error.log        # 实时查看错误日志
tail -f /var/log/nginx/access.log       # 实时查看访问日志
grep "error\|crit\|alert\|emerg" /var/log/nginx/error.log  # 过滤严重错误

# 连接状态查看
curl http://127.0.0.1/nginx_status      # 查看连接统计（需配置 stub_status）
ss -s                                   # 系统级 socket 统计
ss -lntp | grep nginx                   # 查看 Nginx 监听端口

# 性能分析
strace -p $(pgrep nginx | head -1) -e trace=network  # 跟踪 Nginx 网络调用（慎用）
```

## 七、面试高频考点速记

### Nginx 为什么性能好？
**三个核心原因**：
1. **事件驱动**：基于 epoll 的 I/O 多路复用，单 Worker 处理万级并发，无线程切换开销
2. **零拷贝**：sendfile 直接将磁盘 page cache 发送到 socket，跳过用户空间拷贝
3. **内存池**：请求级内存池，O(1) 分配，请求结束整块释放，无内存碎片

### location 匹配口诀
```
精确(=) > 阻断(^~) > 正则(~/~*按顺序) > 普通前缀(最长)
```

### upstream 选哪个策略？
- **无状态微服务** → 轮询/加权轮询（默认选择）
- **Session 保持**（无分布式 Session）→ ip_hash
- **长连接 / 响应时间差异大** → least_conn
- **缓存穿透优化** → hash $request_uri consistent

### 502 和 504 的区别？
- **502**：Nginx 连不上后端（进程挂了、端口未监听）= 后端根本没响应
- **504**：Nginx 连上了后端，但超时（`proxy_read_timeout` 到期）= 后端响应太慢
