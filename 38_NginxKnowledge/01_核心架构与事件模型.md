## 一、Master-Worker 多进程架构

### 1.1 架构图

```
+──────────────────────────────────────────────────────+
│                   Nginx 进程模型                      │
│                                                      │
│  ┌─────────────────┐                                 │
│  │  Master Process │  ← root 权限启动，不处理请求      │
│  │  ─────────────  │    读/验证配置、管理 Worker        │
│  │  信号处理中心    │    处理 HUP/QUIT/USR1/USR2/WINCH │
│  └────────┬────────┘                                 │
│           │  fork                                    │
│    ┌──────┴──────┐                                   │
│    ▼             ▼                                   │
│ ┌──────┐      ┌──────┐  ← 低权限用户（nginx/nobody）  │
│ │Worker│      │Worker│    单线程 + epoll 事件循环      │
│ │ #1   │      │ #2   │    每个 Worker 独立处理请求     │
│ └──────┘      └──────┘                               │
│                                                      │
│  ┌──────────────────────────────┐                    │
│  │ Cache Manager / Cache Loader │ ← 可选，管理缓存文件 │
│  └──────────────────────────────┘                    │
+──────────────────────────────────────────────────────+
```

### 1.2 Master 进程职责

| 职责 | 说明 |
|------|------|
| 配置管理 | 读取、解析、验证 nginx.conf |
| Worker 管理 | fork Worker、监控存活、崩溃时自动重启 |
| 信号处理 | HUP（重载配置）、QUIT（优雅停止）、USR1（重开日志）、USR2（热升级二进制）、WINCH（优雅停止所有Worker） |
| 权限绑定 | 以 root 启动，绑定特权端口（80/443）后将权限降级给 Worker |

### 1.3 Worker 进程特性

- **单线程事件循环**：无锁竞争，无上下文切换开销
- **数量 = CPU 核数**（`worker_processes auto`）：充分利用多核，同时避免进程切换
- **进程间共享内存**：限流计数器（`limit_req_zone`）、缓存元数据通过共享内存同步
- **崩溃隔离**：单个 Worker 崩溃不影响其他 Worker 和 Master

## 二、事件驱动模型（epoll）

### 2.1 I/O 多路复用原理

```
Worker 进程事件循环:

  epoll_create()  ──→  创建事件监听器（内核 epoll 对象）
       │
  epoll_ctl()     ──→  注册感兴趣的 fd 事件（EPOLLIN/EPOLLOUT）
       │
  [循环]
       │
  epoll_wait()    ──→  阻塞等待就绪事件（有事件/超时时返回）
       │
  遍历就绪 fd    ──→  调用对应回调函数
  (非阻塞处理)        ├─ 新连接：accept() + 注册新 fd
                      ├─ 可读：读请求头/体
                      ├─ 可写：发送响应
                      └─ 代理：转发到上游
```

### 2.2 与多线程模型对比

| 维度 | 多进程/多线程（Apache prefork/worker） | 事件驱动（Nginx epoll） |
|------|--------------------------------------|----------------------|
| 并发 1 万连接 | 1 万线程，内存 > 10 GB | 数十个 Worker，内存 < 100 MB |
| 上下文切换 | 频繁，开销大（保存/恢复 CPU 寄存器） | 几乎没有 |
| CPU 利用率 | 低（大量线程阻塞等待 I/O） | 高（只处理就绪事件） |
| 代码复杂度 | 简单（同步顺序编程） | 较高（回调/状态机编程） |
| CPU 密集型 | 适合 | 不适合（会阻塞事件循环） |

### 2.3 epoll 的三种模式

| 模式 | 说明 | Nginx 使用 |
|------|------|-----------|
| `select/poll` | 每次扫描全部 fd，O(n) 复杂度 | 不使用（性能差） |
| `epoll LT`（水平触发） | fd 就绪且未读完，持续通知 | 默认模式 |
| `epoll ET`（边缘触发） | fd 状态变化时只通知一次，需循环读完 | 某些模块使用 |

## 三、连接处理机制

### 3.1 惊群问题（Thundering Herd）

**问题**：多个 Worker 共同监听同一端口，新连接到来时**所有 Worker 被同时唤醒**，但只有一个能成功 accept，其余白白唤醒浪费 CPU。

**Nginx 的解决方案**：

```nginx
# 旧版（< 1.9.1）：互斥锁轮流获取 accept 权
events {
    accept_mutex on;  # 开启互斥锁（默认 off，现代内核已不需要）
    accept_mutex_delay 500ms;  # 竞争失败后等待时间
}

# 新版（>= 1.9.1）：使用 EPOLLEXCLUSIVE 标志
# 内核保证只唤醒一个 Worker，彻底解决惊群
# Nginx 1.11.3+ 自动使用，无需配置
```

### 3.2 连接限制与文件描述符

```nginx
worker_processes  auto;          # Worker 数 = CPU 核数
worker_rlimit_nofile 65535;      # 每个 Worker 最大 fd 数

events {
    worker_connections 10240;    # 每个 Worker 最大并发连接
    use epoll;                   # Linux 上的事件模型（自动选择）
    multi_accept on;             # 一次 accept 尽可能多的连接
}

# 理论最大并发 = worker_processes × worker_connections
# 反向代理时每个请求占用 2 个连接（客户端 + 上游），实际减半
```

**系统级配置**（必须同步调整）：
```bash
# /etc/security/limits.conf
nginx  soft  nofile  65535
nginx  hard  nofile  65535

# /etc/sysctl.conf
net.core.somaxconn = 65535        # TCP 全连接队列长度
net.ipv4.tcp_max_syn_backlog = 65535  # TCP 半连接队列长度
```

## 四、Nginx 内存池

### 4.1 内存池设计

Nginx 的内存池（`ngx_pool_t`）是高性能的关键之一：

```
ngx_pool_t 结构:
┌─────────────────────────────────────┐
│  large: 大块内存链表（> pool_size）  │
│  current: 当前使用的小块内存         │
│  chain: 内存块链表                  │
└─────────────────────────────────────┘
  │
  ▼
┌──────────┐ → ┌──────────┐ → ...
│ ngx_pool │   │ ngx_pool │
│ _data_t  │   │ _data_t  │
│ [last]   │   │ [last]   │  ← 指向下一个可用地址
│ [end]    │   │ [end]    │  ← 块结束地址
└──────────┘   └──────────┘
```

**核心特性**：
- **分配快**：小内存直接移动指针，O(1) 分配，无锁
- **释放方便**：请求结束时整个内存池一次性销毁，无需逐个 free
- **减少碎片**：按请求生命周期分配，请求结束清空，不会积累碎片
- **大对象单独管理**：超过 `pool_size`（默认 4KB）的内存单独通过 malloc 分配，挂在 large 链表上

### 4.2 生命周期

| 内存池类型 | 生命周期 | 典型对象 |
|-----------|---------|---------|
| 连接池 | 连接存活期间 | 连接结构体、SSL 上下文 |
| 请求池 | 单次 HTTP 请求 | 请求头、URI、响应头 |
| 配置池 | Nginx 运行期间 | 配置结构体、server/location 配置 |

## 五、热升级与平滑重启

### 5.1 热重载配置（不停服）

```bash
nginx -s reload
# 等价于：kill -HUP $(cat /var/run/nginx.pid)
```

**流程**：
1. Master 收到 HUP 信号
2. Master 重新加载配置文件并验证
3. Master fork 新 Worker（使用新配置）
4. 新 Worker 开始接受新请求
5. Master 向旧 Worker 发送 QUIT 信号（优雅关闭）
6. 旧 Worker 处理完现有请求后退出

### 5.2 热升级二进制（零停机升级 Nginx 版本）

```bash
# 1. 替换 nginx 二进制文件（新版本）
cp new_nginx /usr/sbin/nginx

# 2. 发送 USR2 信号（启动新 Master + 新 Worker）
kill -USR2 $(cat /var/run/nginx.pid)
# 此时新旧 Master 同时运行，新旧 Worker 同时处理请求

# 3. 向旧 Master 发送 WINCH（优雅停止旧 Worker）
kill -WINCH $(cat /var/run/nginx.pid.oldbin)

# 4. 确认新版本无问题后，停止旧 Master
kill -QUIT $(cat /var/run/nginx.pid.oldbin)

# 回滚：向旧 Master 发送 HUP（重新拉起旧 Worker），再停新 Master
```

## 六、Nginx 与 Netty 事件模型对比

| 对比维度 | Nginx | Netty |
|---------|-------|-------|
| 语言 | C | Java |
| 线程模型 | Master-Worker 多进程，Worker 单线程 | 主从 Reactor，NioEventLoop 单线程 |
| I/O 多路复用 | epoll（Linux）/ kqueue（BSD） | epoll（Linux）/ kqueue（macOS）/ NIO |
| 内存管理 | C 内存池，极低开销 | PooledByteBuf，池化堆外内存 |
| 定位 | Web 服务器 / 反向代理 | 网络框架（Dubbo/RocketMQ 底层） |
| 性能极限 | 百万级 QPS（静态资源） | 十万级 QPS（业务逻辑处理） |
| 惊群处理 | EPOLLEXCLUSIVE / accept_mutex | 无问题（JVM 线程绑定 Selector） |
