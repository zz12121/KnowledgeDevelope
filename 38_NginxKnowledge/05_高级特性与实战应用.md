## 一、rewrite 与 URL 重写

### 1.1 rewrite 语法

```nginx
# rewrite regex replacement [flag];

# flag 含义：
# last      ─ 停止当前 rewrite 规则，重新发起请求（用新 URI 匹配 location）
# break     ─ 停止当前 rewrite 规则，继续处理当前 location（不重新匹配）
# redirect  ─ 返回 302 临时重定向
# permanent ─ 返回 301 永久重定向

server {
    # HTTP → HTTPS 跳转（推荐用 return，性能更好）
    listen 80;
    return 301 https://$host$request_uri;
    
    # 或用 rewrite
    rewrite ^ https://$host$request_uri permanent;
    
    # URL 规范化（去掉多余的斜杠）
    rewrite ^/(.*)//(.*)$ /$1/$2 permanent;
    
    # 旧 API 版本兼容
    rewrite ^/api/v1/(.*)$ /api/v2/$1 last;
    
    # 域名跳转
    if ($host = 'old-domain.com') {
        rewrite ^(.*)$ https://new-domain.com$1 permanent;
    }
}
```

### 1.2 return vs rewrite

| 维度 | `return` | `rewrite` |
|------|----------|-----------|
| 性能 | 直接返回，无需匹配 | 正则匹配，有开销 |
| 功能 | 只能固定状态码+URL | 支持正则捕获组、复杂规则 |
| 适用 | 简单跳转、固定响应 | 复杂 URL 改写 |
| 推荐场景 | HTTP→HTTPS、域名跳转 | URL 规范化、版本兼容 |

### 2.1 常用内置模块

| 模块 | 功能 | 典型指令 |

### 2.2 ngx_http_lua_module（OpenResty）

OpenResty = Nginx + LuaJIT + 一系列 Lua 库，可用 Lua 脚本实现复杂的业务逻辑：

```nginx
# OpenResty 请求处理阶段
# rewrite_by_lua_block    ─ URL 重写/跳转
# access_by_lua_block     ─ 鉴权/限流
# content_by_lua_block    ─ 生成响应内容
# header_filter_by_lua    ─ 修改响应头
# body_filter_by_lua      ─ 修改响应体

server {
    # 用 Lua 实现 JWT 鉴权
    access_by_lua_block {
        local jwt = require "resty.jwt"
        local auth_header = ngx.var.http_authorization
        
        if not auth_header then
            ngx.status = 401
            ngx.say('{"error": "No authorization header"}')
            return ngx.exit(401)
        end
        
        local token = auth_header:match("Bearer (.+)")
        local verified = jwt:verify("your-secret", token)
        
        if not verified or not verified.verified then
            ngx.status = 401
            ngx.say('{"error": "Invalid token"}')
            return ngx.exit(401)
        end
        
        -- 将用户 ID 传给后端
        ngx.req.set_header("X-User-Id", verified.payload.sub)
    }
    
    location /api/ {
        proxy_pass http://backend;
    }
    
    # 用 Lua 实现 Redis 限流
    location /rate_limited/ {
        access_by_lua_block {
            local redis = require "resty.redis"
            local red = redis:new()
            red:connect("127.0.0.1", 6379)
            
            local key = "rate:" .. ngx.var.remote_addr
            local count = red:incr(key)
            if count == 1 then
                red:expire(key, 1)  -- 1秒窗口
            end
            
            if count > 100 then  -- 100 QPS
                return ngx.exit(429)
            end
        }
        proxy_pass http://backend;
    }
}
```

### 2.3 编译安装第三方模块

```bash
# 查看当前编译参数
nginx -V 2>&1

# 下载相同版本的 Nginx 源码
wget http://nginx.org/download/nginx-1.24.0.tar.gz

# 编译安装（添加第三方模块）
./configure \
    --prefix=/etc/nginx \
    --with-http_ssl_module \
    --with-http_v2_module \
    --with-http_stub_status_module \
    --add-module=/path/to/nginx-module-vts \       # 第三方：流量监控
    --add-module=/path/to/nginx_upstream_check_module  # 第三方：主动健康检查
make && make install

# 验证
nginx -V
nginx -t && nginx -s reload
```

## 三、日志与监控

### 3.1 stub_status 状态监控

```nginx
server {
    listen 127.0.0.1:8080;  # 只监听本地（安全）
    
    location /nginx_status {
        stub_status;
        allow 127.0.0.1;
        deny all;
    }
}
```

**输出示例**：
```
Active connections: 291      ← 当前活动连接数
server accepts handled requests
 16630948 16630948 31070465  ← 总接受/处理连接数/处理请求数
Reading: 6 Writing: 179 Waiting: 106
         ^读请求头   ^发响应    ^Keep-Alive 等待
```

**关键指标解读**：
- `accepts - handled > 0`：有连接被丢弃（OS 队列满）
- `Reading` 高：客户端上传慢（大请求体/慢连接）
- `Waiting` 高：Keep-Alive 空闲连接多（正常）
- `Writing` 高：下行带宽不足 or 上游响应慢

### 3.2 日志切割

```bash
# 方法一：logrotate（推荐）
# /etc/logrotate.d/nginx
/var/log/nginx/*.log {
    daily                       # 每天切割
    missingok                   # 文件不存在不报错
    rotate 30                   # 保留 30 份
    compress                    # gzip 压缩
    delaycompress               # 最新的不压缩（还在使用中）
    notifempty                  # 空文件不切割
    create 640 nginx adm        # 新日志文件权限
    sharedscripts
    postrotate
        # 给 Nginx 发 USR1 信号，重新打开日志文件
        nginx -s reopen
        # 或：kill -USR1 $(cat /var/run/nginx.pid)
    endscript
}

# 方法二：脚本手动切割
DATE=$(date +%Y%m%d)
mv /var/log/nginx/access.log /var/log/nginx/access.${DATE}.log
nginx -s reopen  # 重新打开日志文件（不中断服务）
```

### 3.3 Prometheus + Grafana 集成

```nginx
# 使用 nginx-module-vts（流量统计模块）
http {
    vhost_traffic_status_zone;  # 开启统计
    
    server {
        location /metrics {
            vhost_traffic_status_display;
            vhost_traffic_status_display_format prometheus;  # 输出 Prometheus 格式
            allow 10.0.0.0/8;
            deny all;
        }
    }
}
```

## 四、故障排查

### 4.1 常见错误与排查

| 状态码 | 常见原因 | 排查方法 |
|--------|---------|---------|
| 400 | 请求头格式错误、`client_header_buffer_size` 太小 | 检查 `large_client_header_buffers` 配置 |
| 403 | `allow/deny` 规则、目录无读权限、`autoindex off` | 检查 Nginx 用户权限和访问规则 |
| 404 | `root` 路径错误、`try_files` 配置错误 | 确认文件实际路径，开启 `debug` 日志 |
| 413 | 请求体超过 `client_max_body_size` | 调大 `client_max_body_size` |
| 499 | 客户端主动断开（接口太慢，客户端超时） | 优化后端响应时间，检查网络延迟 |
| 502 | 后端无法连接（进程崩溃、端口未监听） | 检查后端服务、`error.log`、`proxy_connect_timeout` |
| 503 | upstream 所有节点不可用 or `limit_req/limit_conn` 触发 | 检查后端健康状态、限流配置 |
| 504 | 后端响应超时（`proxy_read_timeout` 到期） | 调大 `proxy_read_timeout` 或优化后端 |

### 4.2 配置检查与调试

```bash
nginx -t                    # 检查配置文件语法（必须在重载前执行）
nginx -T                    # 检查并打印完整配置（包含所有 include 文件）
nginx -s reload             # 优雅重载配置

# 开启 debug 日志（仅排查时用，生产环境关闭，会产生大量日志）
error_log /var/log/nginx/error.log debug;

# 查看实时错误
tail -f /var/log/nginx/error.log | grep -E "error|crit|alert|emerg"

# 查看特定请求的处理过程（加入 debug 模块后可用）
# debug_connection 192.168.1.100;  # 只对特定 IP 开启 debug
```

### 4.3 502 排查清单

```
502 Bad Gateway 排查步骤:

1. 后端是否存活？
   curl http://127.0.0.1:8080/health

2. Nginx 能连接到后端？
   telnet 127.0.0.1 8080

3. 检查 Nginx error.log
   grep "connect() failed\|upstream" /var/log/nginx/error.log

4. 检查 proxy_connect_timeout 是否过短
   proxy_connect_timeout 5s;（默认 60s，不建议过长）

5. 检查后端 backlog 队列
   ss -lntp | grep 8080
   # 查看 Send-Q（未处理连接队列）是否满

6. 检查 upstream max_fails/fail_timeout
   是否因为频繁失败被 Nginx 标记为不可用
```
