## 一、死锁（Deadlock）

### 1.1 死锁的四个必要条件

死锁发生必须同时满足以下四个条件（破坏任意一个即可避免死锁）：

```
1. 互斥条件   → 资源同一时刻只能被一个线程占用
2. 占有并等待 → 线程持有资源的同时，等待获取其他资源
3. 不可抢占   → 线程不能强制夺走其他线程持有的资源
4. 循环等待   → 线程间形成循环等待链（A等B，B等C，C等A）
```

### 1.2 死锁示例

```java
// 经典死锁：两个线程以相反顺序获取两把锁
Object lockA = new Object();
Object lockB = new Object();

Thread t1 = new Thread(() -> {
    synchronized (lockA) {
        System.out.println("T1 持有 lockA，等待 lockB...");
        try { Thread.sleep(100); } catch (InterruptedException e) {}
        synchronized (lockB) {  // 等待 T2 释放 lockB，但 T2 在等 lockA
            System.out.println("T1 获得 lockB");
        }
    }
});

Thread t2 = new Thread(() -> {
    synchronized (lockB) {
        System.out.println("T2 持有 lockB，等待 lockA...");
        try { Thread.sleep(100); } catch (InterruptedException e) {}
        synchronized (lockA) {  // 等待 T1 释放 lockA，但 T1 在等 lockB
            System.out.println("T2 获得 lockA");
        }
    }
});

t1.start();
t2.start();
// 两个线程永久阻塞 → 死锁！
```

### 1.3 死锁预防策略

```java
// 策略1：固定锁顺序（破坏循环等待）
// 所有线程按相同顺序获取锁
private static final Object LOCK_1 = new Object();
private static final Object LOCK_2 = new Object();

// 无论在哪里，总是先获取 LOCK_1，再获取 LOCK_2
void method() {
    synchronized (LOCK_1) {
        synchronized (LOCK_2) {
            // 操作
        }
    }
}

// 动态确定锁顺序（如转账，根据账户 id 决定获取顺序）
void transfer(Account from, Account to, int amount) {
    Account first  = from.id < to.id ? from : to;
    Account second = from.id < to.id ? to : from;
    synchronized (first) {
        synchronized (second) {
            from.balance -= amount;
            to.balance += amount;
        }
    }
}

// 策略2：使用 tryLock + 超时（破坏无限等待）
public boolean transfer(ReentrantLock lock1, ReentrantLock lock2) 
        throws InterruptedException {
    if (lock1.tryLock(500, TimeUnit.MILLISECONDS)) {
        try {
            if (lock2.tryLock(500, TimeUnit.MILLISECONDS)) {
                try {
                    // 执行操作
                    return true;
                } finally {
                    lock2.unlock();
                }
            }
        } finally {
            lock1.unlock();
        }
    }
    return false;  // 获取失败，稍后重试
}

// 策略3：使用 lockInterruptibly()（允许中断等锁的线程）
lock.lockInterruptibly();  // 等待期间可以被 interrupt() 中断
```

## 三、活锁（Livelock）与饥饿（Starvation）

### 3.1 活锁

```java
// 活锁：线程没有阻塞，但因为互相礼让导致无法推进
// 类比：两人在走廊相遇，互相让路，同方向移动，永远无法通过

// 解决：引入随机退避（random backoff）
void operation() throws InterruptedException {
    while (true) {
        if (tryAcquire()) {
            doWork();
            release();
            return;
        }
        // 随机等待，避免活锁
        Thread.sleep(new Random().nextInt(100));
    }
}
```

### 3.2 饥饿（Starvation）

```java
// 饥饿：低优先级线程长期无法获得 CPU 或锁资源
// 原因：
//   1. 公平锁 vs 非公平锁（非公平锁可能导致某些线程长期等待）
//   2. 线程优先级设置不当

// 解决：使用公平锁
ReentrantLock fairLock = new ReentrantLock(true);  // 公平锁，按等待顺序分配

// 注意：公平锁吞吐量低（每次都需要检查等待队列）
// 非公平锁性能更好，通常优先选择非公平锁
```

## 五、并发性能优化策略

### 5.1 减小锁粒度

```java
// 反例：整个方法加锁（粒度过大）
public synchronized void processOrder(Order order) {
    validateOrder(order);    // 纯计算，不需要锁
    saveToDatabase(order);   // 需要锁
    sendNotification(order); // 纯 IO，不需要锁
}

// 优化：只锁必要部分
public void processOrder(Order order) {
    validateOrder(order);    // 无锁
    synchronized (this) {
        saveToDatabase(order);  // 只锁数据库操作
    }
    sendNotification(order); // 无锁
}
```

### 5.2 锁分离（Lock Striping）

```java
// 对不同数据使用不同的锁，提升并发度
// ConcurrentHashMap 的核心思想（JDK 8 是对每个桶头节点加锁）

// 示例：分段锁（16个段，并发度最高16）
private static final int SEGMENTS = 16;
private final Object[] locks = new Object[SEGMENTS];
private final Map<String, Object>[] maps = new HashMap[SEGMENTS];

{
    for (int i = 0; i < SEGMENTS; i++) {
        locks[i] = new Object();
        maps[i] = new HashMap<>();
    }
}

private int segmentFor(String key) {
    return Math.abs(key.hashCode() % SEGMENTS);
}

public void put(String key, Object value) {
    int seg = segmentFor(key);
    synchronized (locks[seg]) {
        maps[seg].put(key, value);
    }
}
```

### 5.3 乐观锁 / 无锁化（CAS）

```java
// 用 CAS 替代悲观锁，适合低竞争场景
// AtomicInteger 替代 synchronized int
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();              // CAS 自增
counter.compareAndSet(expected, newVal); // CAS 操作

// LongAdder（高并发计数，比 AtomicLong 更好）
// 原理：分段计数，减少 CAS 竞争
LongAdder adder = new LongAdder();
adder.increment();
long total = adder.sum();

// 性能对比（高竞争场景，16线程）：
// synchronized counter: 最慢
// AtomicLong:           中
// LongAdder:            最快（5-10倍于 AtomicLong）
```

### 5.4 读写锁分离

```java
// 读多写少场景：ReadWriteLock 允许多个读线程并发
ReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock readLock = rwLock.readLock();
Lock writeLock = rwLock.writeLock();

// 缓存示例
private final Map<String, Object> cache = new HashMap<>();

public Object get(String key) {
    readLock.lock();   // 多个读线程可以同时获取读锁
    try {
        return cache.get(key);
    } finally {
        readLock.unlock();
    }
}

public void put(String key, Object value) {
    writeLock.lock();  // 写锁独占
    try {
        cache.put(key, value);
    } finally {
        writeLock.unlock();
    }
}

// 注意：写饥饿问题（大量读线程可能让写线程等很久）
// StampedLock（JDK 8）：提供乐观读，性能更好
StampedLock stampedLock = new StampedLock();
long stamp = stampedLock.tryOptimisticRead();  // 乐观读（无锁）
// ... 读操作
if (!stampedLock.validate(stamp)) {           // 验证是否有写操作
    stamp = stampedLock.readLock();            // 升级为悲观读锁
    try { /* 重新读 */ }
    finally { stampedLock.unlockRead(stamp); }
}
```

### 5.5 ThreadLocal 避免共享

```java
// 每个线程有自己的变量副本，彻底消除竞争
// 适合：DateFormat、数据库连接、用户上下文

// ⚠️ 注意内存泄漏：线程池中 ThreadLocal 使用后必须 remove()
ThreadLocal<SimpleDateFormat> dateFormat = 
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

try {
    String date = dateFormat.get().format(new Date());
    // 使用
} finally {
    dateFormat.remove();  // 线程池场景必须清理！
}
```

### 5.6 减少上下文切换

```java
// 1. 合理设置线程池大小
// CPU 密集型：线程数 ≈ CPU 核数 + 1
// IO 密集型：线程数 ≈ CPU 核数 × (1 + IO等待时间/CPU计算时间)

// 2. 协程/虚拟线程（JDK 21+）
// 虚拟线程阻塞时不占用 OS 线程，大幅减少上下文切换
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> blockingIO());  // 阻塞时不浪费 OS 线程
}

// 3. 自旋锁（短暂等待时比阻塞+唤醒更高效）
// JVM 内置锁会自动自旋（-XX:PreBlockSpin，默认10次）
// 自旋超时才真正挂起线程
```

## 六、常见性能陷阱

```java
// 陷阱1：synchronized(new Object())（每次创建新锁对象，锁无效）
synchronized (new Object()) { ... }  // ❌ 完全不同步

// 陷阱2：Double-Checked Locking 缺少 volatile
private static Singleton instance;  // ❌ 缺少 volatile
public static Singleton getInstance() {
    if (instance == null) {
        synchronized (Singleton.class) {
            if (instance == null) {
                instance = new Singleton();  // 指令重排可能导致问题
            }
        }
    }
    return instance;
}
// 正确：加 volatile
private static volatile Singleton instance;  // ✅

// 陷阱3：String intern 导致的意外锁共享
synchronized (key.intern()) { ... }  // ❌ 不同业务的相同 key 字符串共享同一把锁

// 陷阱4：锁对象变化（同步块内改变锁对象引用）
private Object lock = new Object();
void method() {
    synchronized (lock) {
        lock = new Object();  // ❌ 其他线程进入时锁已改变，同步失效
    }
}
```

## 七、面试要点速查

| 问题 | 要点 |
|------|------|
| 死锁四个必要条件 | 互斥 + 占有并等待 + 不可抢占 + 循环等待 |
| 如何避免死锁 | ① 固定锁顺序 ② tryLock 超时 ③ lockInterruptibly |
| 如何检测死锁 | jstack 线程转储；ThreadMXBean.findDeadlockedThreads() |
| 活锁和死锁区别 | 死锁：线程阻塞；活锁：线程一直运行但无法推进 |
| 如何解决饥饿 | 公平锁（ReentrantLock 传 true）；合理设置线程优先级 |
| 减小锁粒度 | 只锁必要的临界区，不要在 synchronized 里做无关操作 |
| LongAdder 为何比 AtomicLong 快 | 分段计数（Cell 数组），各段独立 CAS，最后汇总，减少竞争 |
| Double-Checked Locking 为何需要 volatile | 禁止指令重排：new 操作分3步（分配内存/初始化/赋值），volatile 保证赋值在初始化后 |
