## 一、JDK 8（2014，里程碑版本）

### 1.1 Lambda 表达式

```java
// 函数式接口：只有一个抽象方法的接口
@FunctionalInterface
interface Comparator<T> { int compare(T o1, T o2); }

// Lambda 语法：(参数) -> 表达式 或 (参数) -> { 代码块 }
List<String> list = Arrays.asList("banana", "apple", "cherry");

// 旧方式
Collections.sort(list, new Comparator<String>() {
    @Override
    public int compare(String a, String b) { return a.compareTo(b); }
});

// Lambda
list.sort((a, b) -> a.compareTo(b));

// 方法引用（Lambda 的简写）
list.sort(String::compareTo);            // 实例方法引用
list.forEach(System.out::println);       // 实例方法引用
list.stream().map(String::toUpperCase);  // 实例方法引用
Stream.of(1,2,3).map(Integer::valueOf);  // 静态方法引用
```

### 1.2 Stream API

```java
// 流：不存储数据，描述计算操作，惰性求值，可并行
List<Integer> result = list.stream()
    .filter(s -> s.startsWith("a"))   // 中间操作（惰性）
    .map(String::toUpperCase)          // 中间操作
    .sorted()                          // 中间操作
    .collect(Collectors.toList());     // 终止操作（触发计算）

// 常用收集器
Collectors.toList()
Collectors.toSet()
Collectors.toMap(User::getId, User::getName)
Collectors.groupingBy(User::getDept)            // 分组
Collectors.partitioningBy(u -> u.getAge() > 18) // 分区（true/false）
Collectors.counting()
Collectors.joining(", ", "[", "]")

// 数值流（避免装箱）
IntStream.range(0, 10).sum();
Arrays.stream(new int[]{1,2,3}).average();

// 并行流（底层 ForkJoin，适合 CPU 密集型无状态操作）
list.parallelStream().filter(...).collect(...);
```

### 1.3 Optional

```java
// 避免 NullPointerException 的容器类
Optional<String> opt = Optional.ofNullable(getValue());  // 可能为 null
Optional<String> present = Optional.of("value");          // 不能为 null
Optional<String> empty = Optional.empty();

// 常用方法
opt.isPresent()                          // 是否有值
opt.ifPresent(v -> System.out.println(v)) // 有值时执行
opt.orElse("default")                    // 无值时返回默认值
opt.orElseGet(() -> computeDefault())    // 无值时调用 Supplier
opt.orElseThrow(IllegalStateException::new) // 无值时抛异常
opt.map(String::toUpperCase)             // 变换值
opt.filter(s -> s.length() > 3)         // 过滤

// 链式调用（替代嵌套 null 检查）
String city = Optional.ofNullable(user)
    .map(User::getAddress)
    .map(Address::getCity)
    .orElse("未知");
```

## 五、JDK 14 ~ 16

```java
// JDK 14：Switch 表达式（正式版）
// 旧 switch（语句，容易漏 break）
int days;
switch (month) {
    case 1: case 3: case 5: days = 31; break;
    default: days = 30;
}

// 新 switch（表达式，有返回值，用 -> 消除 fall-through）
int days = switch (month) {
    case 1, 3, 5, 7, 8, 10, 12 -> 31;
    case 4, 6, 9, 11 -> 30;
    case 2 -> 28;
    default -> throw new IllegalArgumentException();
};

// yield 用于多行 case 块的返回值
int result = switch (op) {
    case "add" -> a + b;
    case "mul" -> {
        System.out.println("Multiplying");
        yield a * b;  // 多行 case 用 yield 返回
    }
    default -> 0;
};

// JDK 15：文本块（Text Blocks）
String json = """
        {
            "name": "张三",
            "age": 25
        }
        """;
// 自动处理缩进，\\n 换行，无需转义内部引号

// JDK 16：Record（不可变数据类）
// 编译器自动生成：构造器、getter、equals、hashCode、toString
public record Point(int x, int y) {
    // 紧凑构造器（可做参数校验）
    Point {
        if (x < 0 || y < 0) throw new IllegalArgumentException();
    }
    // 可添加自定义方法
    public double distanceTo(Point other) {
        return Math.hypot(x - other.x, y - other.y);
    }
}

Point p = new Point(3, 4);
p.x();          // getter（不是 getX()）
p.y();
```

## 七、JDK 21（LTS，2023）

### 6.1 虚拟线程（Virtual Threads）— 最重磅特性

```java
// 虚拟线程：JVM 管理的轻量级线程（不直接映射 OS 线程）
// 目标：让 BIO 代码也能达到 NIO 的并发吞吐量

// 创建虚拟线程
Thread vThread = Thread.ofVirtual().start(() -> {
    System.out.println("运行在虚拟线程中");
});

// 虚拟线程执行器（每个任务一个虚拟线程）
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 100_000).forEach(i -> {
        executor.submit(() -> handleRequest(i));  // 轻松10万并发
    });
}

// 阻塞不再是问题：虚拟线程阻塞时只挂起虚拟线程，底层 OS 线程继续执行其他任务
// Spring Boot 3.2+ 可通过配置一键启用虚拟线程
spring.threads.virtual.enabled=true
```

### 6.2 Switch 模式匹配（正式版）

        case Circle c    -> "圆形，半径 " + c.radius();
        case Rectangle r -> "矩形，宽 " + r.width() + " 高 " + r.height();

        case Integer i            -> "零";
        case String s             -> "字符串: " + s;
        case null                 -> "null";
        default                   -> "其他";
    };
}

## 十一、面试要点速查

| 问题 | 要点 |
|------|------|
| Stream 和 Collection 区别 | Stream 不存储数据、惰性求值、不可重用；Collection 存储数据 |
| 并行流注意事项 | 适合无状态 CPU 密集操作；有状态操作（排序）需注意；共享可变状态不安全 |
| Optional 能解决所有 NPE 吗 | 不能，Optional 本身也可能为 null，应作为返回值使用而非字段 |
| var 是动态类型吗 | 不是，是编译时类型推断，运行时是强类型 |
| Record 和普通类区别 | Record 不可变（字段 final）、自动生成 getter/equals/hashCode/toString、不能继承 |
| 虚拟线程原理 | JVM 管理的轻量级线程，M个虚拟线程复用N个OS线程（M>>N）；阻塞时挂起虚拟线程 |
| 密封类的作用 | 限制继承层次，使 switch 可以穷举所有子类型，提升代码安全性 |
| JDK 8 → 17 → 21 哪些是 LTS | 8、11、17、21 是长期支持版 |
| 文本块的优势 | 多行字符串无需转义、自动处理缩进、保留换行格式、提升代码可读性 |
| Switch 表达式 vs 语句 | 表达式有返回值、使用 -> 消除 fall-through、支持多 case 合并、编译器检查穷尽性 |
| instanceof 模式匹配 | JDK 16+ 支持，自动类型转换并赋值，避免强制类型转换代码 |
| var 使用限制 | 只能用于局部变量、必须有初始值、不能用于字段/参数/返回值、不是动态类型 |
| 虚拟线程适用场景 | 高并发 IO 密集型（Web 服务、数据库访问）、轻松支持百万级并发 |
| 虚拟线程注意事项 | 不要池化（创建成本低）、避免使用 ThreadLocal、同步块不会阻塞 OS 线程 |

### 进阶面试题

**Q：JDK 9 模块化系统（JPMS）解决了什么问题？**
> 1. 强封装：明确控制哪些包对外暴露
> 2. 可靠配置：编译时检查依赖，避免运行时 ClassNotFoundException
> 3. 可伸缩平台：可以构建仅包含所需模块的自定义运行时镜像
> 4. 改进安全性：内部实现细节不再暴露

**Q：为什么 JDK 11 移除了 Java EE 和 CORBA 模块？**
> 这些模块（java.xml.ws、java.xml.bind、java.corba 等）已经由 Jakarta EE 社区维护，Oracle 将其从 JDK 中移除以减小体积。需要使用时可通过 Maven 添加依赖：
> ```xml
> <dependency>
>     <groupId>jakarta.xml.bind</groupId>
>     <artifactId>jakarta.xml.bind-api</artifactId>
>     <version>4.0.0</version>
> </dependency>
> ```

**Q：G1 GC 在 JDK 9+ 有哪些改进？**
> 1. JDK 9：G1 成为默认垃圾收集器
> 2. JDK 10：并行 Full GC，使用多线程执行 Full GC，减少停顿时间
> 3. JDK 12：可中止的混合收集，如果超过停顿目标则中止收集
> 4. 持续优化：字符串去重（-XX:+UseStringDeduplication）、改进的回收策略

**Q：ZGC 和 Shenandoah 的区别？**
> | 特性 | ZGC | Shenandoah |
> |------|-----|------------|
> | 目标停顿时间 | < 10ms | < 10ms |
> | 并发整理 | ✅ | ✅ |
> | 染色指针 | ✅ | ❌（使用 Brooks 指针）|
> | 支持平台 | Linux/x64, Linux/AArch64, macOS, Windows | 更多平台 |
> | 最大堆 | 16TB | 受限于平台 |
> | 生产就绪 | JDK 15+ | JDK 15+ |

**Q：如何评估项目是否适合升级到 JDK 21？**
> 1. **依赖兼容性**：检查所有第三方库是否支持 JDK 21
> 2. **代码扫描**：使用 jdeps 分析依赖，检查是否使用了移除的 API
> 3. **测试覆盖**：确保有充分的自动化测试
> 4. **性能基准**：对比升级前后的性能表现
> 5. **虚拟线程收益**：如果应用是 IO 密集型，升级收益巨大
> 6. **渐进式升级**：建议 8 → 11 → 17 → 21 逐步升级，而非跨版本
