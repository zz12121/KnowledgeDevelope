# 异常处理机制

> **核心关键词**：Throwable、Error、Exception、checked/unchecked、try-with-resources、异常链、最佳实践

---

## 一、Java 异常体系

```
Throwable
├── Error（严重系统错误，程序一般不处理）
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   ├── VirtualMachineError
│   └── AssertionError
└── Exception（程序可处理的异常）
    ├── RuntimeException（非受检异常，Unchecked）
    │   ├── NullPointerException
    │   ├── IllegalArgumentException
    │   ├── IndexOutOfBoundsException（ArrayIndexOutOfBoundsException）
    │   ├── ClassCastException
    │   ├── ArithmeticException（除以零）
    │   ├── NumberFormatException
    │   ├── ConcurrentModificationException
    │   └── UnsupportedOperationException
    └── 非 RuntimeException（受检异常，Checked）
        ├── IOException（FileNotFoundException / SocketException）
        ├── SQLException
        ├── ClassNotFoundException
        ├── InterruptedException
        └── CloneNotSupportedException
```

### 1.1 受检异常 vs 非受检异常

| 类型 | 代表 | 编译器强制处理 | 适用场景 |
|------|------|-------------|---------|
| **受检异常（Checked）** | IOException, SQLException | ✅ 必须 catch 或 throws | 外部可恢复的情况（文件不存在、网络断开）|
| **非受检异常（Unchecked）** | RuntimeException 及子类 | ❌ 不强制 | 编程错误（空指针、数组越界）|
| **Error** | OOM, SOF | ❌ 不强制，也没意义处理 | JVM 级别严重问题 |

## 二、异常处理语法

### 2.1 try-catch-finally 基础

```java
try {
    int result = 10 / 0;  // 抛出 ArithmeticException
} catch (ArithmeticException e) {
    System.out.println("除法异常: " + e.getMessage());
} catch (Exception e) {
    // 父类异常放后面！
    System.out.println("通用异常: " + e.getMessage());
} finally {
    // 无论是否抛异常，finally 一定执行
    // 常用于：关闭流、释放锁、清理资源
    System.out.println("finally 执行");
}
```

**多异常捕获（JDK 7+）**：

```java
// 多个异常类型用 | 分隔，代码更简洁
try {
    // ...
} catch (IOException | SQLException e) {
    log.error("IO或SQL错误", e);
}
```

### 2.2 try-with-resources（JDK 7+）★★★

实现了 `AutoCloseable` 接口的资源，会在 try 块结束后自动关闭，彻底解决资源泄漏问题：

```java
// ❌ 旧写法：繁琐且容易忘记关闭
BufferedReader reader = null;
try {
    reader = new BufferedReader(new FileReader("file.txt"));
    String line = reader.readLine();
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (reader != null) {
        try { reader.close(); } catch (IOException e) { e.printStackTrace(); }
    }
}

// ✅ 新写法：简洁，自动关闭
try (BufferedReader reader = new BufferedReader(new FileReader("file.txt"))) {
    String line = reader.readLine();
    System.out.println(line);
} catch (IOException e) {
    log.error("读文件失败", e);
}
// reader 在 try 块结束后自动调用 close()

// 多个资源（按声明逆序关闭）
try (
    Connection conn = dataSource.getConnection();
    PreparedStatement ps = conn.prepareStatement(sql);
    ResultSet rs = ps.executeQuery()
) {
    // ...
}  // rs → ps → conn 依次关闭
```

**try-with-resources 的异常抑制**：

```java
// 如果 try 块和 close() 都抛出异常，以 try 块的异常为主
// close() 的异常被"抑制"，可通过 e.getSuppressed() 获取
try (SomeResource res = new SomeResource()) {
    throw new RuntimeException("try 里的异常");  // 主异常
    // res.close() 也抛异常时，被附加为 Suppressed 异常
} catch (RuntimeException e) {
    Throwable[] suppressed = e.getSuppressed();  // 获取被抑制的异常
}
```

### 2.3 throws 与 throw

```java
// throws：方法签名上声明可能抛出的受检异常
public void readFile(String path) throws IOException, FileNotFoundException {
    // ...
}

// throw：主动抛出异常
public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("年龄不合法: " + age);
    }
    this.age = age;
}
```

## 三、异常链（Exception Chaining）

将原始异常包装在新异常中，保留完整的异常传播路径：

```java
public User findUser(long id) {
    try {
        return userDao.findById(id);
    } catch (SQLException e) {
        // ❌ 错误：直接 e.printStackTrace()，丢失上下文
        // ❌ 错误：catch 后不处理直接吞掉（最危险）
        
        // ✅ 正确：包装为业务异常，保留原始异常
        throw new ServiceException("查询用户失败，id=" + id, e);
        //                                                      ↑ cause，保留原始异常链
    }
}

// 自定义业务异常
public class ServiceException extends RuntimeException {
    public ServiceException(String message, Throwable cause) {
        super(message, cause);
    }
}

// 调用栈打印时，会显示完整链：
// ServiceException: 查询用户失败，id=100
//   Caused by: java.sql.SQLException: ...
//     at ...
```

public class BusinessException extends RuntimeException {
    
    // 建议有错误码，便于国际化和监控
    private final String errorCode;
    
    public BusinessException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }
    
    public BusinessException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }
    
    public String getErrorCode() { return errorCode; }

## 五、finally 的陷阱

### 5.1 finally 中的 return 会覆盖 try/catch 中的 return

```java
int test() {
    try {
        return 1;
    } finally {
        return 2;  // ❌ 危险！finally 的 return 覆盖了 try 的 return 1
    }
}
// 返回 2，try 中的 return 1 被丢弃！
```

### 5.2 finally 中抛出异常会覆盖原异常

```java
void test() throws Exception {
    try {
        throw new Exception("原始异常");
    } finally {
        throw new Exception("finally异常");  // ❌ 原始异常被丢失！
    }
}
// 抛出的是 "finally异常"，"原始异常" 被覆盖了
```

**结论：finally 中不要有 return，不要抛出新异常**。

### 5.3 System.exit() 能跳过 finally 吗？

```java
try {
    System.exit(0);  // JVM 直接退出，finally 不会执行
} finally {
    System.out.println("这行不会执行");
}
```

## 六、异常处理最佳实践

```java
// ✅ 1. 捕获具体的异常类型，不要直接 catch Exception
catch (IOException e) { ... }  // 而不是 catch (Exception e)

// ✅ 2. 不要吞掉异常（空 catch 块）
catch (Exception e) {
    // ❌ 什么都不做 → 异常被静默吞掉，极难排查
}
catch (Exception e) {
    log.error("处理失败", e);  // ✅ 至少记录日志
    throw new RuntimeException("处理失败", e); // ✅ 或者重新抛出
}

// ✅ 3. 用 finally 或 try-with-resources 释放资源
try (Connection conn = ...) { ... }

// ✅ 4. 异常信息要有意义
throw new IllegalArgumentException("用户年龄必须在[0,150]范围内，当前值: " + age);
// ❌ 不要这样：throw new Exception("error");

// ✅ 5. 不要用异常控制正常流程
// ❌ 用 try-catch 代替 if 判断（性能差，语义模糊）
try {
    int value = Integer.parseInt(str);
} catch (NumberFormatException e) {
    // 当成 "str 不是数字" 的判断逻辑
}
// ✅ 用工具方法
if (NumberUtils.isCreatable(str)) { ... }

// ✅ 6. 受检异常的处理：能处理就处理，不能处理就向上传播（包装为非受检异常）
// 应用层一般不暴露 IOException/SQLException，统一包装为 ServiceException/RuntimeException
```

## 七、Spring 中的异常处理

```java
// Spring MVC 全局异常处理
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiResponse> handleBusiness(BusinessException e) {
        log.warn("业务异常: {}", e.getMessage());
        return ResponseEntity.badRequest()
            .body(ApiResponse.error(e.getErrorCode(), e.getMessage()));
    }
    
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse> handleAll(Exception e) {
        log.error("系统异常", e);
        return ResponseEntity.internalServerError()
            .body(ApiResponse.error("SYS_001", "系统繁忙，请稍后重试"));
    }
}
```

## 八、面试要点速查

| 问题 | 要点 |
|------|------|
| Error 和 Exception 的区别 | Error 是 JVM 级别严重错误，不可恢复；Exception 是程序级别，可以捕获处理 |
| 受检异常和非受检异常 | 受检异常（IOException等）编译器强制处理；非受检（RuntimeException）不强制 |
| try-with-resources 原理 | 实现 AutoCloseable，编译器在 try 结束后自动插入 close() 调用，异常时以 try 块异常为主 |
| finally 一定会执行吗 | 不一定：`System.exit()`、JVM 崩溃、无限循环不会执行 finally |
| finally 中 return 的问题 | finally 的 return 会覆盖 try/catch 中的 return，原返回值丢失 |
| 异常链的作用 | 保留原始异常信息，方便排查。通过 `new Exception("msg", cause)` 传入原始异常 |
| catch 多异常的语法 | `catch (IOException | SQLException e)`（JDK 7+）|
