# 字节码与执行引擎

> **核心关键词**：字节码指令、javap、操作数栈、JIT 编译、C1/C2 编译器、方法内联、OSR、逃逸分析

---

## 一、Java 字节码概述

```
Java 的"一次编写，到处运行"依赖字节码：
  .java → javac 编译 → .class（字节码）→ JVM 解释执行 / JIT 编译执行

字节码是平台无关的中间码，面向虚拟机设计：
  - 每条指令 1 字节的操作码（opcode），最多 256 种指令
  - 基于操作数栈（Stack-Based VM），非寄存器（Register-Based）
  - 强类型：大多数指令有类型前缀（i=int, l=long, f=float, d=double, a=引用）
```

## 二、字节码查看工具

```bash
# 反编译 class 文件（内置工具）
```

## 三、字节码指令分类

### 3.1 加载与存储指令

```
iload_0/1/2/3    → 将局部变量表第 0/1/2/3 个 int 变量压入操作数栈
iload <n>        → 将局部变量表第 n 个 int 变量压入操作数栈
aload_0          → 加载 this 引用（实例方法第0个变量是 this）
istore_1         → 将操作数栈顶 int 值存入局部变量表第1个位置
bipush 100       → 将 byte 范围整数压栈
sipush 1000      → 将 short 范围整数压栈
ldc "hello"      → 从常量池加载（字符串/float/long等）
```

### 3.2 运算指令

```
iadd / ladd / fadd / dadd   → 加法（int/long/float/double）
isub / imul / idiv / irem   → 减/乘/除/取余
ineg                        → 取反
ishl / ishr / iushr         → 左移/右移/无符号右移
iand / ior / ixor           → 位与/位或/位异或
```

### 3.3 类型转换

```
i2l  → int → long        i2f  → int → float
i2d  → int → double      l2i  → long → int
f2i  → float → int       d2i  → double → int
i2b  → int → byte        i2c  → int → char
```

### 3.4 对象操作

```
new #类         → 创建对象（不含初始化）
dup             → 复制栈顶值（new 后通常跟 dup，一份给 invokespecial，一份留用）
invokespecial   → 调用构造器、private 方法、父类方法
invokevirtual   → 调用实例方法（虚方法分派，多态核心）
invokestatic    → 调用静态方法
invokeinterface → 调用接口方法
invokedynamic   → 动态方法（Lambda 实现、MethodHandle）
getfield/putfield    → 访问实例字段
getstatic/putstatic  → 访问静态字段
checkcast       → 类型检查（强制转型）
instanceof      → 类型判断
```

### 3.5 控制流

```
ifeq / ifne / iflt / ifge / ifgt / ifle  → 与 0 比较跳转
if_icmpeq / if_icmplt ...               → 两个 int 比较跳转
goto <offset>   → 无条件跳转（实现 for/while）
tableswitch     → switch（连续整数，用跳转表，O(1)）
lookupswitch    → switch（稀疏整数，用查找表，O(log n)）
ireturn / lreturn / areturn / return    → 方法返回
athrow          → 抛出异常
```

## 四、字节码实例分析

```java
// Java 源码
public int add(int a, int b) {
    return a + b;
}
```

```
字节码（javap -c）：
  0: iload_1    // 将局部变量[1](参数a)压栈
  1: iload_2    // 将局部变量[2](参数b)压栈
  2: iadd       // 弹出两个 int，相加，结果压栈
  3: ireturn    // 弹出栈顶 int 作为返回值

局部变量表：
  [0] this   [1] a   [2] b
```

```java
// 更复杂的示例
public void example() {
    int a = 10;
    int b = 20;
    int c = a + b;
}
```

```
 0: bipush 10    // 常量 10 压栈
 2: istore_1     // 弹出 → 存入局部变量[1]（a）
 3: bipush 20    // 常量 20 压栈
 5: istore_2     // 弹出 → 存入局部变量[2]（b）
 6: iload_1      // 加载 a
 7: iload_2      // 加载 b
 8: iadd         // a + b
 9: istore_3     // 存入局部变量[3]（c）
10: return
```

## 五、执行引擎

### 5.1 解释执行 vs JIT 编译

```
解释执行（Interpreter）：
  逐条翻译字节码为机器码并执行
  优点：启动快，无需等待编译
  缺点：每次执行都翻译，性能低

JIT（Just-In-Time）编译：
  将热点代码（Hot Code）编译为本地机器码缓存
  后续执行直接运行机器码，接近 native 性能
  缺点：编译本身有开销（CPU + 内存），预热时间

HotSpot JVM 采用混合模式（Mixed Mode）：
  启动阶段：解释执行（快速启动）
  热点代码：JIT 编译（长期运行性能优）
```

### 5.2 热点代码探测

```
热点探测方式（HotSpot）：方法调用计数器 + 回边计数器

方法调用计数器（InvocationCounter）：
  记录方法被调用次数
  超过阈值（Client: 1500，Server: 10000，-XX:CompileThreshold）→ JIT 编译

回边计数器（BackEdgeCounter）：
  记录循环体执行次数（字节码中的向后跳转）
  用于 OSR（On-Stack Replacement）：循环体直接编译替换，不等方法返回

计数器衰减：JVM 会周期性地将计数器减半（半衰期），保证"近期"热点
```

### 5.3 C1 / C2 编译器（两层编译）

```
HotSpot 的分层编译（Tiered Compilation，JDK 7+ 默认）：

层级    编译器    说明
────────────────────────────────────────────────
Level 0  解释器    初始执行，收集性能数据（Profiling）
Level 1  C1       简单快速编译，无 Profiling 信息（稳定代码直接用）
Level 2  C1       有受限 Profiling 的 C1 编译
Level 3  C1       完整 Profiling 的 C1 编译（为 C2 收集数据）
Level 4  C2       最高优化（方法内联、逃逸分析、循环展开等）

典型路径：0 → 3 → 4（热点代码最终被 C2 深度优化）
         0 → 2 → 1（中等热点，用 C1 即可）
```

### 5.4 JIT 优化技术

```java
// 1. 方法内联（Method Inlining）—— 最重要的优化
// 将被调用方法的代码直接嵌入调用处，消除方法调用开销
// 同时为其他优化创造条件（如逃逸分析需要内联后才能生效）

// 原代码
int result = add(a, b);  // 方法调用

// 内联后（逻辑等价）
int result = a + b;      // 直接计算，消除调用开销

// 内联条件（热点 + 方法体小 + 非虚方法）：
-XX:MaxInlineSize=35      // 方法字节码 ≤ 35 bytes 才内联（默认）
-XX:FreqInlineSize=325    // 热点方法字节码 ≤ 325 bytes 才内联（默认）

// 2. 循环展开（Loop Unrolling）
// 减少循环迭代次数，降低跳转开销
for (int i = 0; i < 4; i++) sum += a[i];
// 展开后：
sum += a[0]; sum += a[1]; sum += a[2]; sum += a[3];

// 3. 消除冗余检查
// 自动消除数组边界检查（在安全范围内循环时）
// 消除空指针检查（已知非空的引用）

// 4. 逃逸分析（见对象内存布局章节）
// 栈上分配 + 标量替换 + 同步消除

// 5. 死码消除（Dead Code Elimination）
if (DEBUG) { log("..."); }  // DEBUG 为 final false → 编译时删除
```

### 5.5 invokedynamic 与 Lambda 底层原理

```java
// invokedynamic 是 Java 7 引入的指令，Lambda 正是基于它实现的

// Java Lambda 表达式：
Runnable r = () -> System.out.println("hello");

// 编译后并非生成匿名内部类，而是：
// 1. 生成一个私有静态方法（lambda$main$0）存储 Lambda 体
// 2. 在调用处生成 invokedynamic 指令
// 3. 首次执行时，通过 LambdaMetafactory.metafactory() 动态生成实现类
//    （底层用 ASM 动态生成字节码，或直接用 MethodHandle）
// 4. 后续调用复用已生成的实现类实例

// 优势：
//   比匿名内部类更高效（懒加载，不生成额外 .class 文件到磁盘）
//   JVM 可以按需选择最优实现方式
//   开销：首次调用有链接开销（约几毫秒），后续零开销

// MethodHandle（类似于函数指针）
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle mh = lookup.findStatic(Math.class, "abs",
    MethodType.methodType(int.class, int.class));
int result = (int) mh.invokeExact(-5);  // 调用 Math.abs(-5)，性能接近直接调用
```

### 5.6 GraalVM 与 Native Image

```
GraalVM：Oracle 开发的高性能多语言运行时
  - 可替换 HotSpot 的 JIT 编译器（Graal JIT，用 Java 写的编译器）
  - Native Image：将 Java 程序 AOT 编译为原生二进制

Native Image 编译流程：
  Java 源码 → javac → .class → native-image → 原生二进制

Native Image 特点：
  ✅ 启动时间：毫秒级（传统 JVM 秒级）
  ✅ 内存占用：小很多（无 JVM 元数据）
  ✅ 适合：Serverless、CLI 工具、容器化微服务
  ❌ 编译时间：分钟级（需要静态分析整个程序）
  ❌ 运行时反射、动态代理：需要配置 reflect-config.json
  ❌ 失去 JIT 动态优化：长期运行峰值性能略低于 HotSpot JIT

Spring Boot 3.x 官方支持 GraalVM Native Image（Spring Native）
Quarkus、Micronaut：原生支持 GraalVM，主打云原生低启动开销
```

## 六、面试要点速查

| 问题 | 要点 |
|------|------|
| JVM 是基于栈还是寄存器 | 操作数栈（Stack-Based），Dalvik/ART 是寄存器（Register-Based）|
| JIT 编译触发条件 | 方法调用次数 / 回边次数超过 CompileThreshold（默认 10000）|
| C1 和 C2 的区别 | C1：快速编译，轻量优化，适合客户端；C2：深度优化（内联/逃逸分析），适合服务端 |
| 分层编译的好处 | 兼顾启动速度（C1 快）和峰值性能（C2 强），JDK 7+ 默认开启 |
| 方法内联的条件 | 方法体小（≤ MaxInlineSize 字节）且调用频繁；虚方法需先去虚化 |
| OSR 是什么 | On-Stack Replacement：循环体正在执行时直接替换为 JIT 编译版本 |
| invokevirtual vs invokespecial | invokevirtual：虚方法（运行时多态分派）；invokespecial：构造器/private/super |
| invokedynamic 用在哪 | Lambda 表达式、方法引用、动态语言（Groovy/Scala）|
| Lambda 底层如何实现 | invokedynamic + LambdaMetafactory 在运行时动态生成实现类（非传统匿名内部类）|
| GraalVM Native Image 的优劣 | 优：毫秒启动、低内存；劣：AOT 失去运行时JIT优化，反射需额外配置 |
