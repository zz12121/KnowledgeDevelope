# 运行时数据区详解


> **核心关键词**：程序计数器、虚拟机栈、本地方法栈、堆、方法区、元空间、栈帧、OOM、StackOverflowError

---

## 一、JVM 内存区域总览

```
JVM 运行时数据区：
┌─────────────────────────────────────────────────┐
│                   线程共享区域                    │
│  ┌───────────────┐  ┌──────────────────────────┐ │
│  │    Java 堆     │  │  方法区（元空间 JDK 8+）   │ │
│  │（新生代+老年代）│  │（类信息、常量池、静态变量）│ │
│  └───────────────┘  └──────────────────────────┘ │
├─────────────────────────────────────────────────┤
│                   线程私有区域                    │
│  ┌──────────┐  ┌──────────────┐  ┌───────────┐  │
│  │ 程序计数器│  │   虚拟机栈    │  │ 本地方法栈 │  │
│  └──────────┘  └──────────────┘  └───────────┘  │
└─────────────────────────────────────────────────┘
直接内存（堆外内存，不在规范内但 JVM 会使用）
```

## 二、程序计数器（PC Register）

```
特点：
  - 线程私有，每个线程一个
  - 最小的一块内存区域
  - 记录当前线程正在执行的字节码指令地址
  - 执行 native 方法时值为 Undefined
  - 唯一不会发生 OOM 的区域

作用：
  - 线程切换后恢复到正确位置（上下文切换依赖 PC）
  - 分支、循环、跳转、异常处理都靠 PC 完成
```

## 三、虚拟机栈（JVM Stack）

### 3.1 结构与栈帧

```
线程私有，每个方法调用对应一个栈帧（Stack Frame）
栈帧入栈（方法调用） → 栈帧出栈（方法返回/异常）

栈帧结构：
  ┌─────────────────────────┐
  │    局部变量表             │  → 存放参数和局部变量（基本类型、引用）
  │    操作数栈              │  → 计算的临时工作区
  │    动态链接              │  → 指向常量池的方法引用（用于方法调用）
  │    方法返回地址           │  → 方法结束后返回调用者的位置
  └─────────────────────────┘
```

### 3.2 局部变量表

```java
// 局部变量表以 Slot（变量槽）为单位
// boolean/byte/char/short/int/float/reference/returnAddress → 1 个 Slot
// long/double → 2 个 Slot

void example(int a, long b, Object c) {
    // 局部变量表：
    // Slot 0: this（实例方法隐含参数）
    // Slot 1: a（int，1 slot）
    // Slot 2-3: b（long，2 slots）
    // Slot 4: c（reference，1 slot）
    boolean d = true;  // Slot 5
}

// 静态方法无 this，slot 0 从第一个参数开始
```

### 3.3 异常情况

```java
// StackOverflowError：栈深度超过限制
// 原因：无限递归，或递归层次过深
public int factorial(int n) {
    return n * factorial(n - 1);  // 忘记写终止条件 → StackOverflowError
}

// OOM（OutOfMemoryError）：栈内存耗尽（极少见，通常在创建大量线程时）

// JVM 参数
-Xss512k   // 设置每个线程的栈大小（默认 512k~1M）
// 栈越小，线程数越多（系统能支持更多并发线程）
// 栈越大，支持更深的递归
```

## 四、堆（Heap）

### 4.1 堆内存结构

```
堆（所有线程共享，GC 的主要管理区域）：

┌────────────────────────────────────────────────────────┐
│                        Java 堆                          │
│  ┌────────────────────────┐  ┌───────────────────────┐ │
│  │       新生代（1/3）      │  │     老年代（2/3）      │ │
│  │  ┌──────┐  ┌──┐  ┌──┐ │  │                       │ │
│  │  │ Eden │  │S0│  │S1│ │  │                       │ │
│  │  │ (8)  │  │(1)│ │(1)│ │  │                       │ │
│  │  └──────┘  └──┘  └──┘ │  │                       │ │
│  └────────────────────────┘  └───────────────────────┘ │
└────────────────────────────────────────────────────────┘
新生代比例：Eden:S0:S1 = 8:1:1（默认）
```

### 4.2 对象晋升机制

```
对象分配流程：
  1. 优先在 Eden 区分配
  2. Eden 满 → Minor GC（YGC）
     存活对象复制到 S0，age+1
  3. 下次 GC 时，Eden + S0 存活对象复制到 S1，age+1
     （Survivor 区交替使用）
  4. age 达到阈值（默认 15）→ 晋升老年代
  5. 大对象直接进入老年代（-XX:PretenureSizeThreshold）
  6. 老年代满 → Major GC / Full GC
```

### 4.3 堆相关 JVM 参数

```bash
-Xms512m          # 堆初始大小（建议与 Xmx 相同，避免动态扩容）
-Xmx2g            # 堆最大大小
-Xmn512m          # 新生代大小（或用 -XX:NewRatio=2 表示老年代/新生代=2）
-XX:SurvivorRatio=8  # Eden:S0:S1 = 8:1:1

# OOM 时自动 dump 堆快照
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/heapdump.hprof
```

## 五、方法区 / 元空间（Metaspace）

### 5.1 方法区存储内容

```
方法区（JDK 7 及之前是永久代 PermGen，JDK 8+ 改为堆外的元空间）：

存储内容：
  ├── 类信息（类名、父类、字段、方法描述）
  ├── 运行时常量池（字面量 + 符号引用）
  ├── 静态变量（JDK 7 起移到堆中）
  ├── JIT 编译后的代码缓存
  └── 方法字节码
```

### 5.2 永久代 → 元空间（JDK 8 的变化）

```
为什么废弃永久代（PermGen）：
  1. 永久代大小固定（-XX:MaxPermSize），容易 OOM
  2. 并发卸载类困难，GC 效率低
  3. HotSpot 与 JRockit 合并时 JRockit 没有永久代概念

元空间特点：
  1. 使用本地内存（native memory），不受 JVM 堆大小限制
  2. 默认无上限（受系统可用内存限制）
  3. 字符串常量池移到堆中（JDK 7 已移）
  4. 静态变量移到堆中（JDK 7 已移）
```

```bash
# 元空间相关参数
-XX:MetaspaceSize=256m      # 元空间初始大小（也是第一次 GC 的触发阈值）
-XX:MaxMetaspaceSize=512m   # 元空间最大大小（建议设置上限防止无限增长）

# 元空间 OOM 原因：
# 1. 动态生成大量类（CGLib、反射、JSP 编译）
# 2. 类加载器泄漏（类未被卸载）
```

### 5.3 运行时常量池

```java
// class 文件常量池 → 加载后 → 运行时常量池（在方法区/元空间）
// 存放字面量（字符串、数值）和符号引用（类名、方法名）

// 字符串常量池（String Pool / String Intern Table）
// JDK 6：在永久代
// JDK 7+：在堆中
String s1 = "hello";          // 字面量：在字符串常量池
String s2 = new String("hello"); // new：在堆中，不同对象
String s3 = s2.intern();      // intern：返回常量池中的引用

System.out.println(s1 == s2);  // false（一个常量池，一个堆）
System.out.println(s1 == s3);  // true（s3 指向常量池）
```

## 六、本地方法栈（Native Method Stack）

```
与虚拟机栈类似，但为 native 方法服务（C/C++ 实现的方法）
HotSpot JVM 将虚拟机栈和本地方法栈合二为一

调用 native 方法时，JVM 通过 JNI（Java Native Interface）调用本地代码
```

## 七、直接内存（Direct Memory）

```java
// 不属于 JVM 规范定义的运行时数据区，但 JVM 会使用
// 通过 ByteBuffer.allocateDirect() 在堆外分配
// JDK NIO、Netty 大量使用（零拷贝）

ByteBuffer directBuf = ByteBuffer.allocateDirect(1024 * 1024);

// 优点：读写性能更高（省去堆 ↔ OS 内存的拷贝）
// 缺点：分配/释放代价高，不受 GC 管理（通过 Cleaner/PhantomReference 释放）

// JVM 参数
-XX:MaxDirectMemorySize=256m  // 限制直接内存大小
// 不设置则默认等于 -Xmx 的值

// 直接内存 OOM：java.lang.OutOfMemoryError: Direct buffer memory
```

## 八、各区域 OOM 场景总结

| 区域 | OOM 类型 | 典型原因 |
|------|---------|---------|
| 堆 | `java.lang.OutOfMemoryError: Java heap space` | 对象太多/太大，内存泄漏 |
| 元空间 | `OOM: Metaspace` | 动态生成大量类，类加载器泄漏 |
| 直接内存 | `OOM: Direct buffer memory` | NIO 直接内存未释放 |
| 栈 | `StackOverflowError` | 无限递归 |
| 方法区（JDK7-） | `OOM: PermGen space` | 类太多（JDK 8+ 已无此问题）|

## 九、面试要点速查

| 问题 | 要点 |
|------|------|
| JVM 运行时数据区有哪些 | 程序计数器、虚拟机栈、本地方法栈（线程私有）+ 堆、方法区（线程共享）|
| 哪个区域不会 OOM | 程序计数器（唯一不会 OOM 的区域）|
| 栈帧包含哪些内容 | 局部变量表 + 操作数栈 + 动态链接 + 方法返回地址 |
| 永久代和元空间的区别 | PermGen：JVM 堆内，固定上限；Metaspace：本地内存，默认无上限 |
| 字符串常量池在哪 | JDK 6：永久代；JDK 7+：堆中 |
| 对象一定在堆上分配吗 | 不一定，逃逸分析可能让对象在栈上分配（标量替换）|
