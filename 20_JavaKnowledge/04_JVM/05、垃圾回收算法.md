# 垃圾回收算法

> **核心关键词**：引用计数、可达性分析、标记-清除、标记-整理、复制算法、分代收集、GC Roots

---

## 一、判断对象是否可回收

### 1.1 引用计数法（Reference Counting）

给每个对象维护一个引用计数器，引用数为 0 则可回收。

**优点**：实现简单，回收及时  
**致命缺陷**：无法解决**循环引用**问题

```java
// 循环引用导致内存泄漏（引用计数永远不为0）
class Node {
    Node next;
}

Node a = new Node();
Node b = new Node();
a.next = b;  // b 的引用计数 = 1
b.next = a;  // a 的引用计数 = 1

a = null;    // a 的引用计数 = 1（b.next 还指向它）
b = null;    // b 的引用计数 = 1（a.next 还指向它）
// 两个对象都不可达，但引用计数不为0，永远无法回收！
```

**Java 不使用引用计数法**（Python CPython 使用，但附加了循环检测）。

### 1.2 可达性分析（Reachability Analysis）

从 **GC Roots** 出发，沿引用链向下遍历，能被访问到的对象是**可达的**，不可达的对象可以被回收。

```
GC Roots
   │
   ├── A（可达）
   │   ├── B（可达）
   │   └── C（可达）
   └── D（可达）
   
   E → F（E 和 F 互相引用，但从 GC Roots 不可达 → 可回收）
```

**GC Roots 包括**：

| GC Roots 类型 | 说明 |
|-------------|------|
| 虚拟机栈中的引用 | 正在执行的方法中的局部变量 |
| 方法区静态字段引用 | `static` 修饰的对象引用 |
| 方法区常量引用 | `final static` 常量对象引用 |
| 本地方法栈（JNI）引用 | native 方法引用的对象 |
| JVM 内部引用 | 基础类型的 Class 对象、常驻异常对象（NPE/OOM）、系统类加载器 |
| 同步锁持有的对象 | 被 synchronized 锁定的对象 |

## 二、四种引用类型

```java
// 1. 强引用（Strong Reference）：绝不回收（即使 OOM）
Object obj = new Object();  // 最常见的引用

// 2. 软引用（Soft Reference）：内存不足时回收（适合缓存）
SoftReference<byte[]> softRef = new SoftReference<>(new byte[1024 * 1024]);
byte[] data = softRef.get();  // 可能为 null（已被回收）

// 3. 弱引用（Weak Reference）：只要 GC，就会回收
WeakReference<Object> weakRef = new WeakReference<>(new Object());
// GC 后 weakRef.get() 返回 null
// 应用：WeakHashMap、ThreadLocal 的 Entry

// 4. 虚引用（Phantom Reference）：随时回收，无法通过虚引用获取对象
// 唯一目的：对象被回收时收到通知（配合 ReferenceQueue）
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(new Object(), queue);
// 应用：堆外内存（DirectByteBuffer）的回收通知
```

## 三、核心 GC 算法

### 3.1 标记-清除算法（Mark-Sweep）

**阶段**：
1. **标记**：遍历所有可达对象，打上标记
2. **清除**：遍历堆，回收未标记对象

```
回收前：
[A][B][C][ ][ ][D][ ][E][F][ ]
 ✓  ✓      ✓        ✓

回收后（B C 不可达被回收）：
[A][ ][ ][ ][ ][D][ ][E][F][ ]
    ← 碎片 →
```

**优点**：实现简单，无需移动对象  
**缺点**：
- **内存碎片**：回收后产生大量不连续内存，可能导致大对象分配失败触发 GC
- **停顿（STW）**：标记和清除阶段需要暂停用户线程

### 3.2 标记-整理算法（Mark-Compact）

在标记-清除的基础上，多一步整理：将存活对象向一端移动，清理边界外的内存。

```
回收前：
[A][B][C][ ][ ][D][ ][E][F][ ]
 ✓  ✓      ✓        ✓

整理后：
[A][B][D][E][F][ ][ ][ ][ ][ ]
                ← 连续空闲空间 →
```

**优点**：无内存碎片，大对象分配简单（移动指针即可）  
**缺点**：
- 移动对象需要更新所有引用（开销大）
- 移动期间需要暂停用户线程

### 3.3 复制算法（Copying）

将内存分为两个等大的区域，每次只使用一半（From 区），GC 时将存活对象复制到另一半（To 区），然后清空 From 区。

```
From 区：[A][B][C][ ][ ][D][ ]
          ✓  ✓       ✓

GC 后：
From 区：[ ][ ][ ][ ][ ][ ][ ]  ← 全部清空
To 区：  [A][B][D][ ][ ][ ][ ]  ← 存活对象紧凑排列
         然后 From ↔ To 互换角色
```

**优点**：无碎片，分配速度快（只需移动指针）  
**缺点**：
- **内存利用率 50%**（一半空间永远不用）
- 对象存活率高时，复制开销大（适合新生代，对象死亡率高）

## 四、分代收集理论

### 4.1 分代假说

1. **弱分代假说**：大多数对象"朝生夕死"（新生代 GC 后通常 98% 以上对象死亡）
2. **强分代假说**：存活越久的对象越难以消亡
3. **跨代引用假说**：跨代引用相对于同代引用来说只占少数

### 4.2 分代收集策略

| 区域 | 对象特征 | GC 算法 | GC 类型 |
|------|---------|---------|---------|
| 新生代（Young Gen）| 朝生夕死，存活率低 | **复制算法** | Minor GC / Young GC |
| 老年代（Old Gen）| 长期存活，存活率高 | **标记-整理** 或 **标记-清除** | Major GC / Full GC |

### 4.3 对象晋升到老年代的条件

```java
// 1. 年龄阈值：Minor GC 后存活，年龄+1，超过阈值（默认15）晋升
// -XX:MaxTenuringThreshold=15

// 2. 大对象直接进老年代：超过 -XX:PretenureSizeThreshold 的对象
// 避免大对象在 Eden 和 Survivor 间复制

// 3. 动态年龄判断：
// Survivor 区中相同年龄所有对象大小之和 > Survivor 空间一半 → 
// 年龄 >= 该年龄的对象直接晋升（不必等到最大年龄）

// 4. Survivor 区空间不足：Minor GC 后存活对象 Survivor 放不下，晋升老年代
```

### 4.4 跨代引用处理：记忆集（Remembered Set）

老年代对象引用新生代对象时，Minor GC 需要将老年代对象也作为 GC Roots：

```
不能每次 Minor GC 都扫描整个老年代（太慢！）

解决方案：记忆集（Remembered Set）
- 记录老年代中所有指向新生代的引用
- Minor GC 时，将记忆集中的老年代对象加入 GC Roots
- HotSpot 使用"卡表（Card Table）"实现记忆集
  - 将堆分为 512 字节的"卡页"
  - 卡表：byte 数组，每个元素对应一个卡页
  - 卡页内有跨代引用时，对应卡表元素标记为"脏"
  - Minor GC 时只扫描脏卡页
```

## 五、Stop-The-World（STW）

### 5.1 为什么需要 STW？

GC 过程中，如果用户线程继续修改对象引用，会导致：
- 已标记的可达对象被引用删除 → **漏标**（对象本该活着被错误回收，**最危险**）
- 已清除的对象被重新引用 → 错误访问已回收内存

### 5.2 安全点（Safepoint）

JVM 不能在任意位置暂停线程，只能在**安全点**处停下（安全点是指令流中有特定特征的指令，如：方法调用、循环跳转、异常跳出等）。

```
用户线程 → 收到 GC 请求 → 跑到最近的安全点停下 → GC 开始
```

**安全区（Safe Region）**：对于阻塞/睡眠的线程，在安全区内任意位置都可以开始 GC。

## 六、三色标记算法

用于解决并发 GC（GC 线程与用户线程并发执行）时的对象漏标问题：

| 颜色 | 含义 |
|------|------|
| **白色** | 未被访问（初始状态；GC 结束后仍为白色=可回收）|
| **灰色** | 已被访问，但其引用的对象尚未全部访问（正在处理）|
| **黑色** | 已被访问，且所有引用对象也已访问（存活，GC 不会再处理）|

**漏标的条件（同时满足）**：
1. 赋值器**删除**了黑色对象到白色对象的引用（黑色不再引用白色）
2. 赋值器**新增**了黑色对象到白色对象的引用（白色挂在了黑色下面）

**解决方案**：
- **增量更新（Incremental Update）**：记录新增引用，GC 结束后重新扫描（CMS 使用）
- **原始快照（SATB，Snapshot At The Beginning）**：记录删除的引用，GC 结束后重新扫描（G1/ZGC 使用）

## 七、Minor GC vs Major GC vs Full GC

| 类型 | 回收区域 | 触发条件 | STW 时间 |
|------|---------|---------|---------|
| Minor GC / Young GC | 新生代 | Eden 区满 | 短（新生代小，存活对象少）|
| Major GC | 老年代 | 老年代满（或手动触发）| 较长 |
| Full GC | **整个堆 + 方法区** | 调用 `System.gc()`；老年代空间不足；方法区空间不足；Minor GC 晋升失败 | **最长** |

**Full GC 的触发场景（需重点排查）**：
```java
// 1. System.gc()（显式调用，生产环境禁用 -XX:+DisableExplicitGC）

// 2. 老年代空间不足
// - 大对象直接进老年代导致空间不足
// - Minor GC 后大量对象晋升，老年代容纳不下

// 3. 方法区（元空间）不足（大量动态类生成）

// 4. 内存担保失败（Minor GC 前检查老年代剩余空间 < 历次晋升平均大小）
```

## 八、面试要点速查

| 问题 | 要点 |
|------|------|
| Java 为什么不用引用计数 | 无法解决循环引用 |
| GC Roots 有哪些 | 虚拟机栈引用、静态字段引用、常量引用、JNI 引用、锁对象 |
| 三种基本 GC 算法 | 标记-清除（碎片）、标记-整理（移动对象，无碎片）、复制（利用率50%，适合新生代）|
| 为什么新生代用复制算法 | 新生代对象大多朝生夕死，存活率低，复制的对象少，效率高 |
| 对象晋升老年代的条件 | 年龄超阈值（默认15）/ 大对象 / 动态年龄 / Survivor 空间不足 |
| 什么是 STW | Stop-The-World，GC 时暂停所有用户线程，防止对象引用在 GC 期间变化 |
| 三色标记算法 | 白/灰/黑三色，解决并发 GC 漏标问题；增量更新（CMS）或 SATB（G1/ZGC）|
| Full GC 频繁怎么排查 | 查老年代大小/晋升量/大对象/内存泄漏；`jstat -gcutil` 分析各区使用率 |
