# 内存泄漏排查实战

> **核心关键词**：内存泄漏、堆转储（Heap Dump）、MAT、Arthas、jmap、jstat、GC日志分析、OOM 排查

---

## 一、Java 内存泄漏的本质

Java 有 GC，但仍会内存泄漏：**对象被 GC Root 持有引用，但业务上已不再使用**，导致 GC 无法回收。

```
常见 GC Root（不会被 GC 回收的对象）：
  - 线程栈中的本地变量
  - 静态变量引用的对象
  - JNI 引用的对象
  - 活跃线程本身

内存泄漏 = GC Root 路径存在 → 对象无法回收 → 堆内存持续增长 → 最终 OOM
```

## 二、常见内存泄漏场景

### 2.1 静态集合持续增长

```java
// ❌ 静态 Map 不断添加，无淘汰策略
public class CacheManager {
    private static final Map<String, Object> CACHE = new HashMap<>();
    
    public static void put(String key, Object value) {
        CACHE.put(key, value);  // 只增不减 → 内存持续增长
    }
}

// ✅ 使用有容量上限的缓存
// 方案1：LinkedHashMap + removeEldestEntry（LRU）
// 方案2：Guava Cache / Caffeine（支持过期/容量限制）
LoadingCache<String, Object> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build(key -> loadFromDB(key));
```

### 2.2 ThreadLocal 忘记 remove

```java
// ❌ 线程池中 ThreadLocal 未清理（见 ThreadLocal 章节）
pool.execute(() -> {
    threadLocal.set(new HugeObject());
    process();
    // 忘记 threadLocal.remove() → value 永久驻留
});

// ✅ try-finally 确保清理
pool.execute(() -> {
    try {
        threadLocal.set(new HugeObject());
        process();
    } finally {
        threadLocal.remove();
    }
});
```

### 2.3 监听器/回调未注销

```java
// ❌ 注册了监听器但没有注销（EventBus、Spring 事件等）
public class DataListener {
    @PostConstruct
    public void init() {
        eventBus.register(this);  // 注册
        // 但这个对象生命周期结束时，eventBus 仍持有引用
    }
    // 没有 @PreDestroy 调用 eventBus.unregister(this)
}

// ✅ 配对注销
@PreDestroy
public void destroy() {
    eventBus.unregister(this);
}
```

### 2.4 内部类持有外部类引用

```java
// ❌ 非静态内部类/匿名类持有外部类的隐式引用
public class OuterClass {
    private byte[] largeData = new byte[10 * 1024 * 1024];  // 10MB
    
    public Runnable createTask() {
        return new Runnable() {  // 非静态匿名类，隐式持有 OuterClass.this
            @Override
            public void run() { process(); }
        };
    }
}
// 如果 Runnable 被长期持有，OuterClass 就无法被 GC
// 即使业务上 OuterClass 已无用

// ✅ 使用静态内部类 + 弱引用
static class StaticTask implements Runnable {
    private final WeakReference<OuterClass> outerRef;
    StaticTask(OuterClass outer) { outerRef = new WeakReference<>(outer); }
    
    @Override
    public void run() {
        OuterClass outer = outerRef.get();
        if (outer != null) outer.process();
    }
}
```

### 2.5 连接/流未关闭

```java
// ❌ 资源未关闭
Connection conn = getConnection();
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery("SELECT ...");
// 异常时没有关闭 → 连接池连接耗尽，间接 OOM

// ✅ try-with-resources（JDK 7+，自动 close）
try (Connection conn = dataSource.getConnection();
     PreparedStatement stmt = conn.prepareStatement(sql);
     ResultSet rs = stmt.executeQuery()) {
    while (rs.next()) { ... }
}
```

### 2.6 大对象 / 大集合

```java
// ❌ 分页查询时不分页，一次性加载所有数据
List<Order> orders = orderRepository.findAll();  // 千万级数据 → OOM

// ✅ 分页 + 流式处理
orderRepository.findAll(PageRequest.of(0, 1000)).forEach(order -> process(order));

// ✅ Stream 流式查询（MyBatis Cursor / Spring Data Slice）
try (Cursor<Order> cursor = orderMapper.selectAllCursor()) {
    cursor.forEach(order -> process(order));
}
```

## 三、排查工具与命令

### 3.1 jstat — 实时监控 GC

```bash
# jstat -gcutil <pid> <interval_ms> <count>
jstat -gcutil 12345 1000 30

输出解读：
S0    S1     E      O      M     CCS    YGC  YGCT   FGC  FGCT    GCT
 0.00  4.87  73.12  85.23  96.45  94.12    15  0.234     2  1.567  1.801

S0/S1：Survivor 区使用率
E：Eden 区使用率
O：Old 区使用率（持续增高 → 内存泄漏嫌疑）
YGC/FGC：Young/Full GC 次数
YGCT/FGCT：GC 耗时（FGC 频繁 → 严重问题）

# 判断内存泄漏：
# 多次 Full GC 后，Old 区使用率仍持续上升 → 强烈内存泄漏信号
```

### 3.2 jmap — 堆转储与分析

```bash
# 查看堆内存摘要
jmap -heap 12345

# 生成堆转储文件（OOM 时也可配置自动生成）
jmap -dump:format=b,file=/tmp/heap-12345.hprof 12345
# ⚠️ 生产环境谨慎使用！生成 dump 期间会暂停 JVM（STW）

# 自动在 OOM 时 dump（推荐提前配置）
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/logs/heapdump.hprof

# 查看内存中对象统计（不需要完整 dump）
jmap -histo:live 12345 | head -30
输出：
num     #instances         #bytes  class name
--------------------------------------------
   1:         12345       50331648  [B  (byte 数组，通常是 String 或日志)
   2:          5678       45678912  java.lang.String
   3:           234       34567890  com.example.LargeObject
# byte 数组占用最高 → 可能是 String 积累，检查日志/缓存
```

### 3.3 MAT（Eclipse Memory Analyzer）— 堆分析

```
MAT 是分析 .hprof 文件的最强工具：

关键功能：
  1. Leak Suspects Report（泄漏嫌疑报告）
     MAT 自动分析，给出高嫌疑的 GC Root 路径
     
  2. Dominator Tree（支配树）
     展示哪些对象持有最多内存
     找到 "Retained Heap"（被某对象支配的总内存）最大的对象

  3. OQL（对象查询语言）
     SELECT * FROM java.lang.String s WHERE s.value.length > 1000
     找出所有超长 String

  4. 引用链（Reference Chain）
     追踪某个泄漏对象的 GC Root 路径，找到是谁持有了它

排查步骤：
  1. File → Open Heap Dump（打开 .hprof）
  2. Reports → Leak Suspects（自动报告）
  3. 查看 "Problem Suspect 1"，找到占内存最大的对象和持有链
  4. Dominator Tree → 按 Retained Heap 排序，找大对象
  5. 右键 → Path to GC Roots → exclude weak references（排除弱引用）
     → 找到强引用路径 → 定位泄漏代码
```

### 3.4 Arthas — 在线诊断（不停机）

```bash
# 下载并启动 Arthas（无需重启 JVM）
java -jar arthas-boot.jar
# 选择目标 JVM 进程

# 1. 查看堆内存使用情况
dashboard  # 实时刷新：内存/GC/线程/CPU

# 2. 查看内存区域详情
memory

# 3. 查看对象数量和大小（不需要 heap dump）
ognl "@java.lang.Runtime@getRuntime().totalMemory()"
jvm  # 显示 JVM 详细信息

# 4. 追踪方法调用（定位内存分配热点）
trace com.example.OrderService processOrder

# 5. 监控方法的入参/返回值（找到哪里创建了大量对象）
watch com.example.CacheManager put '{params,returnObj,throwExp}' -x 3

# 6. 强制 GC 后观察内存变化
ognl "@java.lang.System@gc()"
memory  # 再看内存，如果没降 → 内存泄漏

# 7. 分析类加载器（Metaspace 泄漏排查）
classloader -a  # 显示所有类加载器及加载类数量
```

## 四、OOM 类型与排查对策

| OOM 类型 | 典型原因 | 排查方法 |
|---------|---------|---------|
| `Java heap space` | 对象太多/内存泄漏/堆太小 | jmap dump → MAT 分析 Dominator Tree |
| `GC overhead limit exceeded` | GC 时间 > 98% 但回收不足 2% | 同上 + 检查 Full GC 后老年代仍高 |
| `Metaspace` | 动态类太多，类加载器泄漏 | `classloader -a` 统计类加载器 |
| `Direct buffer memory` | NIO/Netty 堆外内存未释放 | `-XX:MaxDirectMemorySize` 设限；netty `ByteBuf` 引用计数 |
| `unable to create new native thread` | 线程太多 | `jstack` 统计线程数，线程池配置 |
| `StackOverflow` | 无限递归 | `jstack` 看线程栈顶调用 |

## 五、内存泄漏排查完整流程

```
Step 1：监控告警（堆内存持续增长，Full GC 频繁）
  工具：Prometheus + Grafana / 公司监控平台
  指标：heap_used_bytes，jvm_gc_pause_seconds_count

Step 2：确认是内存泄漏
  jstat -gcutil <pid> 1000 30
  观察：多次 Full GC 后，Old 区使用率是否下降
  不下降 = 内存泄漏，下降 = 内存分配过多（调大堆或优化分配）

Step 3：生成 Heap Dump
  jmap -dump:live,format=b,file=/tmp/dump.hprof <pid>
  或等待 OOM 自动 dump（-XX:+HeapDumpOnOutOfMemoryError）

Step 4：MAT 分析
  打开 dump → Leak Suspects → 找到嫌疑对象
  查看 Retained Heap 最大的对象
  追踪 GC Root 路径

Step 5：定位代码
  根据 MAT 给出的类名和持有链，找到对应代码
  Review 为什么这个对象被持有

Step 6：修复验证
  修复后，部署到测试环境，长时间运行后观察内存是否平稳
```

## 六、面试要点速查

| 问题 | 要点 |
|------|------|
| Java 有 GC 为什么还会内存泄漏 | GC Root 仍持有引用，业务上无用的对象无法回收 |
| 常见内存泄漏场景 | 静态集合持续增长、ThreadLocal 未 remove、监听器未注销、非静态内部类、连接未关闭 |
| 如何判断是内存泄漏 | jstat 观察 Full GC 后 Old 区使用率，若不降则是内存泄漏 |
| OOM 如何排查 | 配置 HeapDumpOnOutOfMemoryError，用 MAT 分析 dump 文件，看 Leak Suspects |
| MAT 最重要的功能 | Leak Suspects（自动报告）和 Dominator Tree（找大对象），结合 Path to GC Roots |
| Arthas 如何排查内存问题 | dashboard/memory 实时查看；强制 GC 后看内存；watch/trace 找热点分配 |
| Metaspace OOM 原因 | 动态生成大量类（CGLib/反射）或类加载器泄漏（未卸载） |
