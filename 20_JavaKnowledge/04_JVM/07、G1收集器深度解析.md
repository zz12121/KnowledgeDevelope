# G1 收集器深度解析

> **核心关键词**：Region、Mixed GC、Remember Set、Card Table、SATB、停顿预测、并发标记、Humongous

---

## 一、G1 诞生背景

```
CMS 的痛点：
  1. 内存碎片（标记清除）→ 可能触发 Full GC（Serial Old 单线程清理）
  2. 无法预测停顿时间
  3. 老年代并发收集期间 CPU 资源占用高
  4. 对象晋升失败（Promotion Failure）时性能急剧下降

G1 目标：
  1. 可预测、可配置的停顿时间（-XX:MaxGCPauseMillis）
  2. 无内存碎片（整体整理）
  3. 适合大内存（6GB+）多核（4+）服务器
  4. 取代 CMS 成为默认 GC（JDK 9+ 默认 G1）
```

---

## 二、G1 核心设计：Region

```
传统 GC：物理上连续的新生代/老年代区域
G1：将堆划分为多个大小相等的 Region（1MB~32MB，2的幂），
    每个 Region 逻辑上扮演某种角色（Eden/Survivor/Old/Humongous）

┌──┬──┬──┬──┬──┬──┬──┬──┐
│E │E │S │O │O │H │H │E │   E=Eden, S=Survivor
├──┼──┼──┼──┼──┼──┼──┼──┤   O=Old, H=Humongous（大对象）
│O │O │E │E │S │O │E │O │   空白=Free Region
├──┼──┼──┼──┼──┼──┼──┼──┤
│H │E │O │O │O │E │O │E │
└──┴──┴──┴──┴──┴──┴──┴──┘

Region 灵活性：角色可以动态变化（GC 后 Eden Region 变 Free Region）

大对象（Humongous Object）：
  大小超过 Region 的 50% 的对象
  直接分配到一个或多个连续 Humongous Region
  在 Old GC 时才回收（JDK 8u60+ 在 Young GC 时也可回收）
```

## 三、G1 的 GC 模式

### 3.1 Young GC（年轻代回收）

```
触发条件：Eden Region 用完
STW（Stop-The-World）：是，但可预测时间

过程：
  1. 扫描 GC Roots + Remember Set（跨 Region 引用）
  2. 复制存活对象到 Survivor/Old Region
  3. 空 Region 加入空闲列表

目标：满足 -XX:MaxGCPauseMillis 的停顿目标（通过控制回收 Region 数量）
```

### 3.2 并发标记周期（Concurrent Marking Cycle）

```
触发条件：堆使用率达到 -XX:InitiatingHeapOccupancyPercent（默认 45%）

阶段（类似 CMS）：
  1. 初始标记（Initial Mark）  ← STW，很短，标记 GC Roots 直接可达对象
     （通常附带在 Young GC 上）
  2. 根 Region 扫描（Root Region Scanning）  ← 并发
  3. 并发标记（Concurrent Marking）  ← 并发，与用户线程同时进行
     使用 SATB（Snapshot At The Beginning）算法保证并发标记正确性
  4. 最终标记（Remark）  ← STW，处理 SATB 引用变更
  5. 清理（Cleanup）  ← STW（统计活跃对象）+ 并发（重置 Region）
```

### 3.3 Mixed GC（混合回收）——G1 特有

```
触发条件：并发标记完成后，根据各 Old Region 的垃圾密度，选择垃圾最多的 Old Region
STW：是，但控制停顿时间

Mixed GC = Young GC + 部分 Old Region 回收
  - 每次 Mixed GC 回收 1/8 的 Old Region（-XX:G1MixedGCCountTarget=8）
  - 经过多次 Mixed GC 完成一次完整的 Old 区清理

优势：按价值回收（垃圾多的 Region 先回收），高效利用停顿时间
```

### 3.4 Full GC（最后手段）

```
触发条件：
  1. 并发标记期间 to-space exhausted（Survivor/Old 空间不足）
  2. Mixed GC 还来不及回收就 OOM

行为：单线程（JDK 9 以前），JDK 10+ 并行 Full GC（ParallelFullGC）
→ G1 中 Full GC 是性能最差的情况，要尽量避免
```

## 四、G1 关键数据结构

### 4.1 Remember Set（RSet）

```
问题：Young GC 只回收 Young Region，但 Old Region 可能引用 Young Region 对象
→ 必须知道哪些 Old 对象引用了 Young 对象，否则不能确定 Young 对象是否存活

RSet：每个 Region 都有一个 Remember Set
  记录"哪些其他 Region 的对象引用了本 Region 中的对象"
  → Young GC 扫描 GC Roots + Young Region 的 RSet 即可，不用扫描全堆

实现：Card Table（卡表）+ RSet
  Card Table：将堆划分为 512字节 的卡（Card），引用关系发生变化时标记对应卡为"脏"
  RSet：存储哪些 Card 含有指向本 Region 的引用

写屏障（Write Barrier）：
  引用字段被修改时触发，更新 Card Table 和 RSet
```

### 4.2 SATB（Snapshot At The Beginning）

```
并发标记问题：
  用户线程在标记过程中可能修改引用关系
  → 某些存活对象可能被漏标（浮动垃圾问题）

SATB 解决方案：
  1. 在并发标记开始时，取引用关系的"快照"
  2. 写前屏障（Pre-Write Barrier）：在引用被修改前，将旧值记录到 SATB 队列
  3. 最终标记阶段处理 SATB 队列，保证旧引用不被遗漏

G1 的三色标记 + SATB：
  白色（未标记）→ 灰色（标记但子对象未扫描）→ 黑色（完全标记）
  SATB 保证：所有在标记开始时存活的对象最终都被标记为黑色
```

## 五、G1 停顿预测模型

```
G1 通过衰减均值（Decaying Average）统计历史 GC 数据：
  每个 Region 的回收耗时、存活率
  根据目标停顿时间 (-XX:MaxGCPauseMillis)，动态选择本次回收多少个 Region

核心原则：在允许的停顿时间内，尽量多回收垃圾（高价值优先）
```

## 六、G1 关键参数

```bash
# 最大停顿时间目标（软目标，GC 会尽力满足）
-XX:MaxGCPauseMillis=200       # 默认 200ms，不是越小越好

# 堆占用率达到多少时触发并发标记
-XX:InitiatingHeapOccupancyPercent=45   # 默认 45%

# Region 大小（不建议手动设置，JVM 会根据堆大小自动计算）
-XX:G1HeapRegionSize=4m        # 1m~32m，2的幂

# Mixed GC 参数
-XX:G1MixedGCCountTarget=8     # Mixed GC 分几次完成 Old 区回收
-XX:G1HeapWastePercent=5       # Old 区垃圾占比低于此值时停止 Mixed GC

# RSet 更新线程数
-XX:G1ConcRefinementThreads=4  # 默认等于 -XX:ParallelGCThreads

# 老年代存活率阈值（低于此值的 Old Region 才考虑 Mixed GC 回收）
-XX:G1MixedGCLiveThresholdPercent=85  # 默认 85%

# 新生代大小（G1 动态调整，一般不设置）
-XX:G1NewSizePercent=5         # 最小新生代比例
-XX:G1MaxNewSizePercent=60     # 最大新生代比例

# 日志
-XX:+PrintGCDetails -XX:+PrintGCDateStamps
-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m  # JDK 9+
```

## 七、G1 生产调优 Checklist

### 7.1 常见 G1 GC 日志解读

```
# JDK 9+ 统一日志格式
[0.345s][info][gc] GC(1) Pause Young (Normal) (G1 Evacuation Pause) 512M->256M(2048M) 18.234ms
[0.400s][info][gc] GC(2) Pause Young (Concurrent Start) (G1 Humongous Allocation) 768M->512M(2048M) 22.1ms
[1.234s][info][gc] GC(5) Pause Full (G1 Compaction Pause) 1900M->300M(2048M) 850.234ms  ← 危险！

关键指标：
  Evacuation Pause：正常 Young GC，STW 在 MaxGCPauseMillis 内
  Concurrent Start：触发了并发标记，通常说明 IHOP 被触发
  Humongous Allocation：大对象分配，注意关注 Humongous Region 使用率
  Full GC：生产环境应极力避免！
```

### 7.2 Evacuation Failure（晋升失败）

```
触发条件：
  Young GC 时，存活对象无法复制到 Survivor/Old Region（空间不足）

表现：
  GC 日志出现：(to-space exhausted) 或 (Evacuation Failure)
  性能急剧下降，可能退化为 Full GC

解决方案：
  1. 增大堆大小 (-Xmx)
  2. 增加 Old 区比例（降低 G1MaxNewSizePercent）
  3. 降低 IHOP（让并发标记更早触发，提前回收 Old）
  4. 优化代码减少对象分配（分配压力过大）
```

### 7.3 G1 调优决策树

```
1. Full GC 频繁？
   → 并发标记跟不上分配速度
   解决：降低 IHOP（从 45% 降到 35%），扩大堆，减少大对象

2. GC 停顿超过 MaxGCPauseMillis？
   → 回收的 Region 数量太多
   解决：增大 MaxGCPauseMillis 或减少 G1MaxNewSizePercent

3. GC 频率过高（Eden 很快被填满）？
   → 分配速率过高，增大新生代或优化代码减少短命对象

4. 内存使用率居高不下，但 GC 不触发？
   → IHOP 设太高，降低 IHOP

5. Humongous 对象频繁导致问题？
   → 增大 G1HeapRegionSize（减少对象被判定为 Humongous 的概率）
```

## 八、面试要点速查

```
1. 停顿时间不要设太小（< 50ms 往往适得其反，频繁 GC 反而吞吐低）
   推荐：100ms ~ 300ms

2. 避免 Full GC：
   - 避免大量大对象（Humongous Objects）
   - 增大堆大小或降低 IHOP（InitiatingHeapOccupancyPercent）
   - 增大 Region 大小（减少大对象判定）

3. 关注 Mixed GC 频率：
   - 并发标记太频繁 → IHOP 过低，适当调高
   - Full GC 频繁 → 可能 Mixed GC 跟不上分配速度

4. 打开 GC 日志分析：
   - 关注 GC 停顿时间、Evacuation Failure（晋升失败）
```

## 八、面试要点速查

| 问题 | 要点 |
|------|------|
| G1 与 CMS 的核心区别 | G1 基于 Region，可预测停顿时间，无内存碎片；CMS 只管老年代，会产生碎片 |
| G1 的 Region 有哪些类型 | Eden、Survivor、Old、Humongous（大对象）、Free |
| Mixed GC 是什么 | Young GC + 部分 Old Region 回收，G1 特有，垃圾多的 Region 优先回收 |
| RSet 的作用 | 记录跨 Region 的引用关系，Young GC 时无需扫描全堆 |
| SATB 解决什么问题 | 并发标记时引用变更导致的漏标问题（保证快照时存活的对象不被漏标）|
| Humongous 对象的影响 | 直接进 Old 区，可能导致 Full GC；建议减少大对象或增大 Region 大小 |
| G1 的默认停顿目标 | 200ms |
| G1 何时触发并发标记 | 堆占用率达到 IHOP（默认 45%）时 |
