# 对象的创建与内存布局

> **核心关键词**：对象创建流程、内存布局、对象头、Mark Word、指针压缩、对象访问定位、逃逸分析

---

## 一、对象创建流程（new 关键字背后）

```
new Object() 触发以下步骤：

1. 类加载检查
   → JVM 检查 new 指令对应的类是否已被加载、链接、初始化
   → 未加载则先执行类加载过程

2. 分配内存
   → 从堆中划出一块内存（大小在类加载时已确定）
   → 分配方式：指针碰撞（堆内存规整时）或空闲列表（不规整时）
   → 并发安全：CAS + 失败重试，或 TLAB（线程局部分配缓冲区）

3. 初始化为零值
   → 将分配的内存空间清零（int → 0, boolean → false, 引用 → null）
   → 保证字段不初始化也能用

4. 设置对象头
   → 存储类型指针（指向 Class 元数据）
   → 存储 Mark Word（哈希码、GC 年龄、锁状态等）

5. 执行 <init> 方法
   → 按代码顺序执行构造器（包括字段初始化和构造方法体）
   → 完成后才是"真正"的对象
```

## 二、内存分配方式

### 2.1 指针碰撞（Bump the Pointer）

```
适用于：堆内存规整（使用 Serial、ParNew 等带压缩整理的 GC）

│ 已用内存 │  空闲内存 │
          ↑
         指针（分界点）

分配：将指针向空闲方向移动对象大小的距离
优点：简单高效
```

### 2.2 空闲列表（Free List）

```
适用于：堆内存不规整（使用 CMS 等基于标记清除的 GC）

维护一张表记录哪些内存块是可用的
分配时从表中找一块足够大的内存，并更新表

缺点：分配效率低于指针碰撞
```

### 2.3 TLAB（Thread-Local Allocation Buffer）

```java
// 问题：多线程同时修改分配指针 → 并发冲突
// 解决1：CAS 原子操作（每次 new 都 CAS，高并发下开销大）
// 解决2：TLAB（推荐方案）

// TLAB：每个线程在 Eden 区预先分配一小块私有内存
// 线程内的对象分配直接在 TLAB 上操作（无需同步）
// TLAB 用完后再申请新的 TLAB（此时才需要同步）

// TLAB 大小：默认 Eden 区的 1%，可配置
-XX:+UseTLAB           // 开启 TLAB（默认开启）
-XX:TLABSize=512k      // 指定 TLAB 大小
-XX:+PrintTLAB         // 打印 TLAB 使用情况（调试用）
```

## 三、对象内存布局

HotSpot JVM 中，对象在堆内存中由三部分组成：

```
┌──────────────────────────────────┐
│          对象头（Header）         │
│  ┌─────────────────────────────┐ │
│  │  Mark Word（8 字节，64位JVM）│ │
│  ├─────────────────────────────┤ │
│  │  类型指针 Klass Pointer      │ │
│  │  （4字节开启压缩，8字节关闭） │ │
│  ├─────────────────────────────┤ │
│  │  数组长度（仅数组对象，4字节）│ │
│  └─────────────────────────────┘ │
├──────────────────────────────────┤
│          实例数据（Instance Data） │
│  字段值（父类字段在前，子类在后）  │
├──────────────────────────────────┤
│          对齐填充（Padding）       │
│  保证对象大小是 8 字节的整数倍     │
└──────────────────────────────────┘
```

### 3.1 Mark Word（核心）

Mark Word 在不同锁状态下内容不同（64位 JVM）：

```
锁状态         Mark Word 内容（64位）
无锁           25位：hashCode + 31位：未用 + 1位：是否偏向 + 2位：锁标志(01)
偏向锁         54位：线程ID + 2位：epoch + 1位：是否偏向(1) + 2位：锁标志(01)
轻量级锁       62位：Lock Record指针 + 2位：锁标志(00)
重量级锁       62位：Monitor对象指针 + 2位：锁标志(10)
GC 标记        62位：空 + 2位：锁标志(11)

包含信息：
  - HashCode（首次调用 hashCode() 时计算并存储）
  - GC 年龄（分代年龄，4位，最大15，即 -XX:MaxTenuringThreshold）
  - 锁状态（无锁/偏向锁/轻量级锁/重量级锁）
  - 线程ID（偏向锁时）
```

### 3.2 类型指针（Klass Pointer）

```java
// 指向对象所属类的元数据（在元空间/方法区）
// 通过类型指针，JVM 可以确定对象是哪个类的实例

// 指针压缩（Compressed Oops）
// 64位 JVM 默认开启：-XX:+UseCompressedOops
// 将 8字节 引用压缩为 4字节（堆 < 32GB 时有效）
// 原理：实际地址 = 压缩地址 × 8（地址对齐到 8 字节）

// 关闭指针压缩
-XX:-UseCompressedOops      // 关闭 oop 压缩
-XX:-UseCompressedClassPointers // 关闭类指针压缩
```

### 3.3 对象大小计算示例

```java
// 示例：计算 Integer 对象大小
// 环境：64位JVM，开启指针压缩

// Integer 对象内存布局：
//   Mark Word:    8字节
//   Klass Pointer:4字节（压缩）
//   int value:    4字节
//   合计:         16字节（已对齐，无需填充）

// 使用 JOL（Java Object Layout）工具查看实际布局
// Maven: org.openjdk.jol:jol-core:0.17
System.out.println(ClassLayout.parseInstance(new Integer(1)).toPrintable());
```

## 四、逃逸分析（Escape Analysis）

### 4.1 什么是逃逸

```java
// 对象逃逸：对象的引用被其他线程或方法访问
// 不逃逸（栈上分配或标量替换的前提）

// 逃逸示例
public StringBuilder escape() {
    StringBuilder sb = new StringBuilder();
    sb.append("hello");
    return sb;  // sb 逃逸到方法外，不能优化
}

// 不逃逸示例
public String noEscape() {
    StringBuilder sb = new StringBuilder();
    sb.append("hello");
    return sb.toString();  // sb 不会逃逸，可以优化
}
```

### 4.2 逃逸分析带来的优化

```java
// 1. 栈上分配（Stack Allocation）
// 不逃逸的对象直接在栈上分配，方法结束自动释放，无需 GC
// 需要开启：-XX:+DoEscapeAnalysis（JDK 6+ 默认开启）

// 2. 标量替换（Scalar Replacement）
// 将对象拆解为基本类型存储在栈帧的局部变量中
// 如果对象不逃逸且可以被拆解
Point point = new Point(1, 2);
int x = point.x;
int y = point.y;
// 经标量替换后，Point 对象可能根本不创建，直接用 x、y 两个 int 变量

// 3. 同步消除（Lock Elision）
// 不逃逸的对象上的同步操作可以被消除（只有一个线程访问）
public void method() {
    Vector<Integer> v = new Vector<>();  // Vector 的方法都是 synchronized
    v.add(1);  // 编译器发现 v 不逃逸 → 消除 synchronized 开销
    v.get(0);
}

// 验证逃逸分析效果
-XX:+DoEscapeAnalysis        // 开启逃逸分析（默认）
-XX:-DoEscapeAnalysis        // 关闭逃逸分析
-XX:+PrintEscapeAnalysis     // 打印逃逸分析结果
```

## 五、对象访问定位

JVM 通过栈上的引用（reference）访问堆上的对象，有两种主流方式：

### 5.1 句柄访问

```
栈帧引用 → 句柄池（到对象实例指针 + 到类型数据指针）→ 堆
                                                      → 方法区

优点：reference 稳定，对象移动时只需修改句柄，不改 reference
缺点：多一次间接寻址
```

### 5.2 直接指针（HotSpot 采用）

```
栈帧引用 → 堆中对象（含类型指针 → 方法区）

优点：速度快，少一次指针定位
缺点：对象移动时需要修改 reference（GC 整理时负担重）
```

## 六、面试要点速查

| 问题 | 要点 |
|------|------|
| 对象创建步骤 | 类加载检查 → 分配内存（指针碰撞/空闲列表）→ 初始化零值 → 设置对象头 → 执行 init |
| TLAB 作用 | 为每个线程预分配 Eden 私有内存，避免多线程分配竞争 |
| 对象头包含什么 | Mark Word（锁状态、hashCode、GC年龄）+ 类型指针 + 数组长度（数组才有）|
| Mark Word 大小 | 64位 JVM = 8字节；32位 JVM = 4字节 |
| 指针压缩的作用 | 将 8字节引用压缩为 4字节，堆 < 32GB 时生效，节省内存约 30% |
| 逃逸分析的优化 | 栈上分配 + 标量替换 + 同步消除 |
| HotSpot 用哪种对象访问方式 | 直接指针（速度更快）|
| GC 年龄最大值 | 15（Mark Word 中 4位记录，最大 1111b = 15）|
