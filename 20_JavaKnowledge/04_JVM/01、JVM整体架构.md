## 二、运行时数据区详解

### 2.1 程序计数器（PC Register）

- **线程私有**，每个线程独立一份
- 记录当前线程**正在执行的字节码指令的地址**（行号）
- 执行 native 方法时值为 undefined
- **唯一不会发生 OOM 的区域**

```java
// 理解程序计数器的作用：
// 线程A 执行到第 42 条字节码指令时被切换出去
// 线程B 执行...
// 线程A 恢复时，靠 PC 寄存器知道从第 42 条继续
```

### 2.2 虚拟机栈（JVM Stack）

- **线程私有**，生命周期与线程相同
- 每个**方法调用**对应一个**栈帧（Stack Frame）**，入栈执行，出栈返回

**栈帧结构**：
```
┌──────────────────────────┐
│         栈帧              │
│  ┌────────────────────┐  │
│  │  局部变量表         │  │ ← 基本类型值 + 对象引用
│  │  Local Variables   │  │
│  ├────────────────────┤  │
│  │  操作数栈           │  │ ← 方法执行中间结果的暂存区
│  │  Operand Stack     │  │
│  ├────────────────────┤  │
│  │  动态链接           │  │ ← 运行时将符号引用解析为直接引用
│  │  Dynamic Linking   │  │
│  ├────────────────────┤  │
│  │  方法返回地址       │  │ ← 调用方的 PC 值，用于方法返回
│  │  Return Address    │  │
│  └────────────────────┘  │
└──────────────────────────┘
```

**异常**：
- `StackOverflowError`：栈深度超限（递归过深）
- `OutOfMemoryError`：如果栈允许动态扩展且无法申请到足够内存

```java
// 触发 StackOverflowError 的例子
void infinite() { infinite(); }  // 无终止条件的递归

// 调整栈大小：-Xss256k
```

### 2.3 本地方法栈（Native Method Stack）

与虚拟机栈类似，但服务于 **native 方法**（C/C++ 实现的方法）。HotSpot JVM 将虚拟机栈和本地方法栈合二为一。

### 2.4 堆（Heap）

- **线程共享**，JVM 中最大的一块内存区域
- 所有**对象实例和数组**都在堆上分配（逃逸分析可能让对象在栈上分配）
- **GC 的主要管理区域**

**堆的分代结构（经典分代模型，G1之前）**：
```
┌─────────────────────────────────────────────┐
│                  堆 (Heap)                   │
│                                             │
│  ┌──────────────────────┐  ┌─────────────┐  │
│  │     新生代 (Young)    │  │  老年代     │  │
│  │                      │  │  (Old/Ten.) │  │
│  │  ┌──────┐  ┌──┐ ┌──┐ │  │             │  │
│  │  │ Eden │  │S0│ │S1│ │  │             │  │
│  │  │  80% │  │10│ │10│ │  │             │  │
│  │  └──────┘  └──┘ └──┘ │  │             │  │
│  └──────────────────────┘  └─────────────┘  │
│        比例约 1:2                             │
└─────────────────────────────────────────────┘
```

**相关 JVM 参数**：
```bash
-Xms512m          # 堆初始大小
-Xmx2g            # 堆最大大小
-Xmn512m          # 新生代大小
-XX:SurvivorRatio=8  # Eden:Survivor = 8:1:1（默认）
```

**OOM 场景**：
```java
// java.lang.OutOfMemoryError: Java heap space
List<byte[]> list = new ArrayList<>();
while (true) { list.add(new byte[1024 * 1024]); }  // 不断创建大对象
```

### 2.5 方法区（Method Area）/ 元空间（Metaspace）

- **线程共享**
- 存储**类信息、常量、静态变量、即时编译后的代码**

| JDK 版本 | 实现 | 内存位置 |
|---------|------|---------|
| JDK 7 及之前 | 永久代（PermGen）| JVM 堆内，大小固定 |
| JDK 8+ | 元空间（Metaspace）| **本地内存（Native Memory）**，默认无上限 |

**JDK 8 废弃永久代的原因**：
1. 永久代大小固定，容易 OOM（`java.lang.OutOfMemoryError: PermGen space`）
2. 难以调优（需要提前估算大小）
3. HotSpot JVM 与 JRockit 合并，JRockit 没有永久代概念

**相关参数（JDK 8+）**：
```bash
-XX:MetaspaceSize=256m     # 元空间初始大小（达到后触发 Full GC）
-XX:MaxMetaspaceSize=512m  # 元空间最大大小（设置上限防止无限增长）
```

**OOM 场景**：动态生成大量类（如 CGLib 代理、JSP 编译）：
```
java.lang.OutOfMemoryError: Metaspace
```

### 2.6 运行时常量池

方法区的一部分，存储编译期生成的：
- 字面量（数值、字符串）
- 符号引用（类名、字段名、方法名）

**字符串常量池**（String Pool）：JDK 7+ 移至**堆中**（不在方法区），存储字符串字面量和 `intern()` 的结果。

## 三、执行引擎

### 3.1 解释器（Interpreter）

逐条将字节码翻译为机器指令执行。
- **优点**：启动快，立即执行
- **缺点**：每次执行都需翻译，长期运行性能差

### 3.2 JIT 编译器（Just-In-Time Compiler）

监控代码执行频率，对"热点代码"（Hot Spot Code）进行**即时编译**，将其编译为本地机器码并缓存：

```
热点代码判定标准：
- 方法调用次数超过阈值（Client:-XX:CompileThreshold=1500，Server:10000）
- 循环体执行次数超过阈值（栈上替换 OSR）
```

**HotSpot JVM 的两个 JIT 编译器**：

| 编译器 | 编译速度 | 优化程度 | 适用场景 |
|--------|---------|---------|---------|
| C1（Client Compiler）| 快 | 低（基础优化）| 启动快的客户端应用 |
| C2（Server Compiler）| 慢 | **高**（激进优化）| 长期运行的服务端应用 |

**分层编译（Tiered Compilation，JDK 8 默认开启）**：
```
Level 0：解释执行
Level 1：C1 编译（无性能分析）
Level 2：C1 编译（有有限性能分析）
Level 3：C1 编译（有完整性能分析）
Level 4：C2 编译（充分优化）
```

### 3.3 JIT 主要优化技术

```java
// 1. 方法内联（Inlining）：将小方法调用替换为方法体，消除调用开销
// 原始代码
int square(int x) { return x * x; }
int result = square(5);
// 内联后
int result = 5 * 5;

// 2. 逃逸分析（Escape Analysis）
// 如果对象不会逃逸出方法，可以：
// - 栈上分配（Stack Allocation）：对象分配在栈帧，方法结束自动回收，不需 GC
// - 标量替换（Scalar Replacement）：拆分对象字段直接用局部变量
// - 消除同步（Lock Elision）：确认对象只被一个线程访问，消除 synchronized

// 3. 循环优化：循环展开、循环外提不变量等
```

## 四、类加载机制简述

详见 [[04_JVM/02、类加载机制]]，此处仅列流程：

```
Loading（加载）
→ Verification（验证）
→ Preparation（准备：静态变量分配内存，设默认值）
→ Resolution（解析：符号引用 → 直接引用）
→ Initialization（初始化：执行 <clinit>）
→ Using（使用）
→ Unloading（卸载）
```

**双亲委派模型**：
```
BootstrapClassLoader（C++ 实现，加载 rt.jar 等核心类库）
    └── ExtClassLoader（加载 jre/lib/ext/）
            └── AppClassLoader（加载 classpath）
                    └── 自定义 ClassLoader
```

## 五、直接内存（Direct Memory）

- 不属于 JVM 运行时数据区，是**堆外内存**（本地内存）
- NIO 的 `ByteBuffer.allocateDirect()` 使用直接内存，减少 Java 堆与操作系统之间的数据复制
- 由 JVM 的 `Unsafe` 类分配，`-XX:MaxDirectMemorySize` 控制上限
- **OOM 场景**：`java.lang.OutOfMemoryError: Direct buffer memory`

```java
// Netty 大量使用直接内存提升 I/O 性能
ByteBuffer buf = ByteBuffer.allocateDirect(1024 * 1024);
```

## 六、各区域 OOM 汇总

| 区域 | OOM 异常信息 | 常见原因 |
|------|------------|---------|
| 堆 | `Java heap space` | 对象太多，内存泄漏，大对象 |
| 元空间 | `Metaspace` | 动态生成大量类（CGLib/JSP）|
| 虚拟机栈 | `StackOverflowError` | 递归过深 |
| 直接内存 | `Direct buffer memory` | NIO/Netty 使用不当 |
| 虚拟机栈（扩展）| `OOM: unable to create native thread` | 线程数过多 |

## 七、面试要点速查

| 问题 | 要点 |
|------|------|
| JVM 内存区域有哪些 | PC寄存器、虚拟机栈、本地方法栈（线程私有）；堆、方法区（线程共享）|
| 哪些区域会 OOM | 堆、方法区/元空间、虚拟机栈（溢出）、直接内存 |
| PC 寄存器的作用 | 记录线程当前执行位置，线程切换后恢复执行 |
| 堆和栈的区别 | 堆：对象实例，线程共享，GC管理；栈：方法帧，线程私有，自动出栈 |
| 方法区 JDK8 变化 | 永久代 → 元空间，从 JVM 堆内迁到本地内存 |
| JIT 的作用 | 编译热点代码为机器码缓存，提升长期运行性能 |
| 分层编译 | JDK8 默认开启，C1（快速低优）+ C2（慢速高优）配合 |
| 逃逸分析的优化手段 | 栈上分配、标量替换、锁消除 |
