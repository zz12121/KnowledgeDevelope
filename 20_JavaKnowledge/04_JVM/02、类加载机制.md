# 类加载机制

> **核心关键词**：类加载流程、双亲委派、类加载器、破坏双亲委派、热加载

---

## 一、类的生命周期

```
加载(Loading)
   ↓
验证(Verification) ─┐
   ↓               │
准备(Preparation)  ├── 链接(Linking)
   ↓               │
解析(Resolution)  ─┘
   ↓
初始化(Initialization)
   ↓
使用(Using)
   ↓
卸载(Unloading)
```

---

## 二、各阶段详解

### 2.1 加载（Loading）

1. 通过类的**全限定名**找到对应的 `.class` 文件（或字节流）
2. 解析字节流，生成代表该类的 `java.lang.Class` 对象（存放在**方法区/元空间**）
3. 在内存中创建一个 `Class` 对象（位于**堆**中，作为访问方法区中类型数据的入口）

**字节流来源**（不限于本地文件）：
- 本地磁盘 `.class` 文件
- JAR/WAR 包中的 `.class` 文件
- 网络传输的字节流
- 运行时计算生成（动态代理、字节码增强框架 ASM/Javassist/ByteBuddy）
- 加密文件解密后的字节流

### 2.2 验证（Verification）

确保字节码符合 JVM 规范，防止恶意代码危害 JVM：

| 验证阶段 | 内容 |
|---------|------|
| 文件格式验证 | 魔数（0xCAFEBABE）、版本号是否支持 |
| 元数据验证 | 父类是否合法（是否继承了 final 类）、是否实现了接口的方法 |
| 字节码验证 | 操作数栈和局部变量的类型是否合法 |
| 符号引用验证 | 解析阶段前的检查，确保引用的类/字段/方法存在且可访问 |

### 2.3 准备（Preparation）

为类的**静态变量**分配内存，并设置**零值**（默认值）：

```java
public static int value = 123;
// 准备阶段：value = 0  ← 零值，不是 123
// 初始化阶段：value = 123  ← 执行赋值语句

public static final int CONST = 123;
// 准备阶段：CONST = 123  ← final 常量直接赋初值（编译期已确定）
```

**零值对照**：

| 类型 | 零值 |
|------|------|
| int/short/byte/char | 0 |
| long | 0L |
| float | 0.0f |
| double | 0.0d |
| boolean | false |
| 引用类型 | null |

### 2.4 解析（Resolution）

将常量池中的**符号引用**替换为**直接引用**：

```
符号引用：字符串形式的类名/方法名（如 "java/lang/String"）
直接引用：内存中的指针、偏移量等可以直接访问目标的引用

解析的目标：
- 类或接口的解析
- 字段解析
- 类方法解析
- 接口方法解析
```

> 解析可以在初始化阶段之前，也可以之后（JVM 规范允许懒解析，大多数实现采用懒解析）。

### 2.5 初始化（Initialization）

执行 `<clinit>()` 方法（类构造器），完成**静态变量赋值**和**静态代码块**的执行：

```java
class Demo {
    static int a = 10;         // ① 准备：a=0，初始化：a=10
    static String b;
    
    static {
        b = "hello";           // ② 静态代码块，按顺序合并到 <clinit>
        System.out.println("静态初始化");
    }
    
    static int c = a * 2;      // ③ c = 20（此时 a 已经是 10）
}

// 编译器合并生成的 <clinit>：
// a = 10
// b = "hello"
// System.out.println(...)
// c = a * 2
```

**触发初始化的六种场景（主动引用）**：
1. 使用 `new`、`getstatic`、`putstatic`、`invokestatic` 指令
2. 反射调用（`Class.forName("xxx")`）
3. 初始化子类时，父类未初始化 → 先初始化父类
4. JVM 启动时，初始化主类（含 `main` 方法的类）
5. JDK 7+ `MethodHandle` 相关
6. JDK 8+ 接口有 default 方法，实现类初始化时

**不触发初始化的"被动引用"**：

```java
// 1. 通过子类引用父类的静态字段（只初始化父类，不初始化子类）
System.out.println(Child.parentField);  // 只触发 Parent 初始化

// 2. 通过数组定义引用类（不触发 ClassName 初始化）
ClassName[] arr = new ClassName[10];    // 不触发 ClassName 初始化

// 3. 常量（编译期会被优化到调用类的常量池）
System.out.println(ClassName.CONST_VAL);  // 不触发 ClassName 初始化
```

## 三、类加载器

### 3.1 类加载器层次

```
BootstrapClassLoader（启动类加载器）
  用 C++ 实现，是 JVM 的一部分，不是 Java 类
  加载 $JAVA_HOME/lib/rt.jar 等核心类库
  如：java.lang.* java.util.* java.io.* 等
       ↑ 父类加载器
       
ExtClassLoader（扩展类加载器）
  加载 $JAVA_HOME/lib/ext/ 目录下的扩展库
  JDK9+ 改名为 PlatformClassLoader
       ↑ 父类加载器
       
AppClassLoader（应用类加载器/系统类加载器）
  加载用户类路径（ClassPath）下的类
  是我们编写的应用类默认的加载器
       ↑ 父类加载器
       
自定义 ClassLoader
  继承 ClassLoader，实现特殊的加载逻辑
```

> **注意**：这里的"父子关系"是**组合（包含）**关系，不是继承关系！每个加载器持有一个 `parent` 引用指向父加载器。

### 3.2 双亲委派模型

```
加载类 "com.example.Demo":

AppClassLoader.loadClass("com.example.Demo")
  → 先委派给 ExtClassLoader
  → ExtClassLoader 先委派给 BootstrapClassLoader
  → BootstrapClassLoader 在 rt.jar 中找不到 → 返回失败
  → ExtClassLoader 在 ext 目录中找不到 → 返回失败
  → AppClassLoader 自己在 classpath 中加载 → 成功
```

**源码实现**：

```java
// ClassLoader.loadClass()
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // 1. 先检查是否已经加载过
        Class<?> c = findLoadedClass(name);
        
        if (c == null) {
            try {
                // 2. 委派给父类加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    // parent 为 null，说明父加载器是 BootstrapClassLoader
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父加载器找不到，忽略异常
            }
            
            if (c == null) {
                // 3. 父加载器找不到，自己尝试加载
                c = findClass(name);  // 子类重写此方法实现自定义加载逻辑
            }
        }
        if (resolve) resolveClass(c);
        return c;
    }
}
```

### 3.3 双亲委派的优势

1. **安全性**：防止核心类被覆盖。用户自定义的 `java.lang.String` 永远不会被加载（BootstrapClassLoader 已加载了真正的 `String`）
2. **唯一性**：同一个类不会被加载两次（相同全限定名 + 相同类加载器 → 同一个类）
3. **隔离性**：Java 核心 API 与用户代码隔离

## 四、破坏双亲委派模型

### 4.1 第一次破坏：历史遗留

JDK 1.2 之前没有双亲委派。JDK 1.2 引入后，为兼容老代码，`ClassLoader` 的 `loadClass` 可以被重写（不推荐）。

### 4.2 第二次破坏：SPI 机制（线程上下文类加载器）

**问题**：JDBC 的 `java.sql.Driver` 接口在 rt.jar 中由 BootstrapClassLoader 加载，但其实现（如 `com.mysql.jdbc.Driver`）在用户 classpath 中，BootstrapClassLoader 无法加载用户代码。

**解决**：线程上下文类加载器（Thread Context ClassLoader）

```java
// 获取/设置线程上下文类加载器
ClassLoader contextClassLoader = Thread.currentThread().getContextClassLoader();
Thread.currentThread().setContextClassLoader(anotherLoader);

// JDBC SPI 核心加载逻辑（java.util.ServiceLoader）
ServiceLoader<Driver> loadedDrivers = ServiceLoader.load(Driver.class);
// ServiceLoader.load() 内部使用线程上下文类加载器（通常是 AppClassLoader）
// 这样 BootstrapClassLoader 加载的接口代码就能"向下"调用 AppClassLoader 加载的实现
```

### 4.3 第三次破坏：热加载（OSGi / Tomcat）

**Tomcat 为什么要破坏双亲委派？**

```
场景：Tomcat 中部署了两个 Web 应用：
  App1 依赖 Spring 3.x
  App2 依赖 Spring 4.x
  
  如果使用标准双亲委派，Spring 只会被加载一次（版本冲突！）

Tomcat 的类加载器层次：
  BootstrapClassLoader（JVM 核心）
       ↑
  ExtClassLoader
       ↑
  AppClassLoader（Tomcat 自身的类）
       ↑
  CommonClassLoader（Tomcat 和所有 WebApp 共享的类）
       ↑
  CatalinaClassLoader   WebAppClassLoader（每个 WebApp 独立一个）
  (Tomcat 私有)          ↑ 先自己加载，找不到再委派给上层
```

```java
// WebAppClassLoader 重写 loadClass，优先自己加载（破坏了双亲委派）
@Override
public Class<?> loadClass(String name, boolean resolve) {
    // 1. 先检查本地缓存
    Class<?> clazz = findLoadedClass(name);
    
    // 2. 核心类（java.*）仍然委派给 Bootstrap
    if (clazz == null) {
        if (name.startsWith("java.")) {
            clazz = j2seClassLoader.loadClass(name);  // 委派
        }
    }
    
    // 3. 其他类：先自己找（！不委派给父加载器）
    if (clazz == null) {
        clazz = findClass(name);  // 在 WebApp 的 WEB-INF/lib、WEB-INF/classes 中找
    }
    
    // 4. 找不到再委派给父加载器
    if (clazz == null) {
        clazz = super.loadClass(name, resolve);
    }
    // ...
}
```

## 五、自定义类加载器

### 5.1 标准写法（遵守双亲委派，只重写 findClass）

```java
public class MyClassLoader extends ClassLoader {
    
    private final String classDir;
    
    public MyClassLoader(String classDir, ClassLoader parent) {
        super(parent);  // 指定父加载器
        this.classDir = classDir;
    }
    
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 将全限定名转为文件路径
        String fileName = classDir + File.separator + 
                          name.replace('.', File.separatorChar) + ".class";
        
        try (FileInputStream in = new FileInputStream(fileName)) {
            byte[] bytes = in.readAllBytes();
            // 可以在这里对字节码解密
            return defineClass(name, bytes, 0, bytes.length);  // 核心方法
        } catch (IOException e) {
            throw new ClassNotFoundException(name, e);
        }
    }
}
```

### 5.2 应用场景

```java
// 1. 热加载：监控文件变更，重新创建类加载器加载新版本
// 注意：原 ClassLoader 加载的旧类不会立即 GC，需要等 ClassLoader 无引用

// 2. 加密类文件：字节码加密存储，加载时解密
// 防止反编译（但强度有限）

// 3. 类隔离：插件系统中每个插件用独立的 ClassLoader
// OSGi、JPMS（Java 9 模块系统）、SofaArk 等

// 4. 从非标准来源加载：数据库、网络、内存
```

## 六、类的唯一性

**同一个类的判断条件**：`全限定名 + 类加载器` 两者都相同，才是同一个类

```java
// 即使字节码完全相同，不同类加载器加载出来也是不同的类
ClassLoader loader1 = new MyClassLoader("/path/classes");
ClassLoader loader2 = new MyClassLoader("/path/classes");

Class<?> c1 = loader1.loadClass("com.example.Demo");
Class<?> c2 = loader2.loadClass("com.example.Demo");

System.out.println(c1 == c2);           // false！不同类加载器 → 不同的 Class 对象
System.out.println(c1.equals(c2));      // false
c1.isAssignableFrom(c2);               // false

// 实际后果：
Object obj2 = c2.newInstance();
c1.cast(obj2);  // → ClassCastException！虽然类名相同，但是不同的类型
```

## 七、面试要点速查

| 问题 | 要点 |
|------|------|
| 类加载的五个阶段 | 加载→验证→准备→解析→初始化（+使用+卸载）|
| 准备阶段做什么 | 为静态变量分配内存并设零值（final 常量直接赋初值）|
| 初始化阶段做什么 | 执行 `<clinit>()`：静态变量赋值 + 静态代码块 |
| 双亲委派的工作流程 | 先委派父加载器，父加载器找不到再自己加载 |
| 双亲委派的好处 | 核心类不被覆盖（安全）；同一个类不被重复加载（唯一）|
| 如何破坏双亲委派 | 重写 `loadClass()`（而不是 `findClass()`）|
| Tomcat 为什么破坏双亲委派 | 支持不同 WebApp 使用不同版本的同一个库（类隔离）|
| SPI 为什么需要线程上下文类加载器 | Bootstrap 加载的接口需要加载用户 classpath 中的实现，"向下"委托 |
| 类的唯一性由什么决定 | 全限定名 + 加载它的类加载器，两者都相同才是同一个类 |
