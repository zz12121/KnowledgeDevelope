# 垃圾收集器全览

> **核心关键词**：Serial、Parallel、CMS、G1、ZGC、Shenandoah、STW、吞吐量 vs 延迟

---

## 一、收集器演进概览

```
JDK 1.3：Serial GC（单线程，STW）
JDK 1.4：Parallel GC（多线程并行，提升吞吐量）
JDK 1.4.2：CMS（并发标记清除，减少 STW）
JDK 7u4：G1（Garbage First，低延迟高吞吐平衡）
JDK 11：ZGC（超低延迟，STW < 10ms）
JDK 12：Shenandoah（并发压缩，RedHat 贡献）
JDK 21：ZGC GA，G1 仍是默认收集器
```

---

## 二、Serial & Serial Old

### 2.1 特点

- **Serial**（新生代）：单线程复制算法，STW 期间只有一个 GC 线程
- **Serial Old**（老年代）：单线程标记-整理算法

```
运行期：[用户线程][用户线程][用户线程]
GC 时：      停！停！停！
             [GC 线程]（单线程清理）
GC 后：[用户线程][用户线程][用户线程]
```

**适用场景**：客户端应用、单核机器、内存小（几十 MB）、Client 模式 JVM

**PDF补充：串行回收器**

串行回收器也叫Serial收集器，是最古老收集器。它在JDK1.3之前是虚拟机新生代收集器的唯一选择。

它是**单线程**执行回收操作的。它的特点就是，在单核或内核少的计算机来说，有更好的性能表现。它的优点就是简单高效。

JVM 参数：
- 使用 `-XX:+UseSerialGC` 可以指定新生代和老年代都是Serial收集器
- 使用 `-XX:+PrintCommandLineFlags` 可以打印虚拟机参数，查看当前JVM使用什么垃圾回收器
- 新生代串行回收器采用：复制算法
- 老年代串行回收器采用：标记压缩算法

## 三、Parallel Scavenge & Parallel Old

### 3.1 特点

- **目标**：最大化**吞吐量**（Throughput = 用户代码运行时间 / 总时间）
- **多线程** GC，减少 STW 时间但仍有 STW
- JDK 8 的**默认收集器**

```
GC 时：      停！停！停！
             [GC线程1][GC线程2][GC线程3]（多线程并行清理，快）
```

**JVM 参数**：

```bash
-XX:+UseParallelGC              # 新生代 Parallel Scavenge
-XX:+UseParallelOldGC           # 老年代 Parallel Old（JDK 8 默认组合）
-XX:MaxGCPauseMillis=200        # 最大暂停时间目标（GC 自适应调整）
-XX:GCTimeRatio=99              # 吞吐量目标：99% 用于用户代码，1% 用于 GC
-XX:+UseAdaptiveSizePolicy      # 自适应调整堆大小（默认开启）
```

**适用场景**：批处理任务、科学计算、后台任务（对延迟不敏感，关注总吞吐）

**PDF补充：并行回收器概览**

将串行回收器并行化，与串行回收器有相同的回收策略、算法、参数。

**ParNew 回收器**：
ParNew是一个新生代的回收器，也是一个独占式的回收器，它与串行回收器唯一不同的，就是它采取并发方式执行GC。大家一定要注意一点，就是在cpu核数少的时候，它的性能很可能比串行回收器差。

- `-XX:ParallelGCThreads` 指定并行GC的线程个数，最好与CPU个数一致，否则会影响垃圾回收性能
- 默认情况下，当CPU数量<=8个的时候，并行线程数为8个。如果CPU数量>8，并行线程数量为：3+(5*cpu_nums/8)

**ParallelGC 回收器**：
ParallelGC也是新生代的回收器，也采用的复制算法执行GC回收任务。它与ParNew有一个不同点就是，它提供了一些设置系统吞吐量的参数用来控制GC行为。

- `-XX:MaxGCPauseMillis`：最大的垃圾收集暂停时间，它是一个大于0的整数，ParallelGC会根据设置的值来调整堆的大小和其他jvm参数，使其把GC停顿时间控制在MaxGCPauseMillis之内
- `-XX:GCTimeRatio`：设置吞吐量大小，可设置的值为0~100之间的整数（大于0且小于100）。比如说n=99，那么1/(1+99)=1%，也就是说，系统会花费小于1%的时间用于垃圾回收
- `-XX:+UseAdaptiveSizePolicy`：如果你不倾向手动设置上面的参数，可以采用把参数调整交给虚拟机自动设置

**ParallelOldGC 回收器**：
它跟ParallelGC相似，也是关注于吞吐量的收集器。从名字看，它比ParallelGC多了个Old，其实就表示它是一个应用于老年代的回收器。可以与ParallelGC搭配使用，即：ParallelGC（新生代收集器）+ ParallelOldGC（老年代收集器）。它采用标记压缩算法进行GC操作。

## 四、CMS（Concurrent Mark Sweep）

### 4.1 设计目标

**降低老年代 GC 的停顿时间**，让用户线程尽量与 GC 线程并发执行。

### 4.2 四阶段流程

```
阶段1：初始标记（Initial Mark）—— STW（很短）
  只标记 GC Roots 直接引用的对象（扫描范围小，停顿极短）

阶段2：并发标记（Concurrent Mark）—— 无 STW（最耗时）
  从 GC Roots 出发，遍历整个对象图（与用户线程并发）
  用户线程可能修改引用关系，需要后续重新扫描

阶段3：重新标记（Remark）—— STW（比初始标记长，比全量短）
  修正并发标记期间被用户线程修改的部分（增量更新）

阶段4：并发清除（Concurrent Sweep）—— 无 STW
  清除未被标记的对象（与用户线程并发，不移动对象）
```

**PDF补充：CMS 6个步骤详解**

CMS全称为Concurrent Mark Sweep，即：并发标记清除。它采用的是标记清除算法。也是多线程并发执行器，它的特点是会关心系统的停顿时间。分为如下6个步骤：

1. **初始标记（STW）**：标记根对象
2. **并发标记**：标记所有对象
3. **预清理**：清理前的准备以及控制停顿时间（可以采用 `-XX:-CMSPrecleaningEnabled` 关闭，不进行预清理）
4. **重新标记（STW）**：修正并发标记数据
5. **并发清理**：清理垃圾（真正的执行垃圾回收）
6. **并发重置**：重置状态等待下次CMS的触发

**为什么要有预清理？**
因为第4步重新标记是独占CPU的，如果YoungGC发生后，立即触发一次重新标记（新生代使用ParNew回收器），那么一次停顿时间可能很长，为了避免这种情况，预处理时，会刻意等待一次新生代GC的发生，然后根据历史数据预测下一次YoungGC的时间，在当前时间和预测时间取中间时刻执行重新标记操作，目的就是尽量避免YoungGC与重新标记重叠执行，从而减少一次停顿时间。

### 4.3 CMS 的问题

```java
// 1. 内存碎片：使用标记-清除算法，不整理内存，产生大量碎片
// 大对象分配失败 → 触发 Full GC（Serial Old 兜底，STW！很慢）
-XX:+UseCMSCompactAtFullCollection  // Full GC 时做内存整理
-XX:CMSFullGCsBeforeCompaction=5   // 每隔5次 Full GC 做一次整理

// 2. 浮动垃圾（Floating Garbage）
// 并发清除阶段，用户线程可能产生新的垃圾，要下次 GC 才能回收

// 3. 并发失败（Concurrent Mode Failure）
// CMS 并发期间老年代满了，必须退化为 Serial Old → 非常长的 STW！
// 原因：预留的剩余空间不够，或碎片过多
// 预防：增大堆，降低触发 CMS 的阈值（默认 68%）
-XX:CMSInitiatingOccupancyFraction=70  // 老年代使用 70% 时触发 CMS

// 4. CPU 资源争抢：并发 GC 线程与用户线程竞争 CPU
```

**JDK 14 正式移除 CMS**，被 G1 和 ZGC 替代。

**适用场景（历史）**：对停顿敏感的在线服务（JDK 8 时代，配合 4-8G 堆使用）

## 五、G1（Garbage First）

详见 [[04_JVM/07、G1收集器深度解析]]，此处列核心特点：

```
Region 分区：将堆划分为 N 个等大小的 Region（约 1-32MB）
  - 每个 Region 动态担任 Eden/Survivor/Old/Humongous 角色
  - 没有固定的物理分代

回收优先级：优先回收"垃圾最多"的 Region（Garbage First 的由来）

关键参数：
  -XX:+UseG1GC                    # 使用 G1（JDK 9+ 默认）
  -XX:MaxGCPauseMillis=200        # 目标最大停顿时间（默认 200ms）
  -XX:G1HeapRegionSize=16m        # Region 大小（1-32MB，2的幂）
  -XX:G1NewSizePercent=5          # 新生代最小占比（默认 5%）
  -XX:G1MaxNewSizePercent=60      # 新生代最大占比（默认 60%）
```

**适用场景**：JDK 9+ 推荐默认，大堆（6G+），平衡吞吐量与延迟。

**PDF补充：G1的垃圾收集概述**

G1全称Garbage First Garbage Collector。优先回收垃圾比例最高的区域。

G1收集器将堆划分为多个区域，每次收集部分区域来减少GC产生的停顿时间。

**G1 第1阶段：新生代GC**

新生代GC的主要工作就是回收eden区和survivor区。一旦eden区被占满，新生代GC就会启动。回收后，所有的eden区都应该被清空，而survivor区会被收集一部分数据，但是应该至少仍然存在一个survivor区。

**G1 第2阶段：并发标记周期**

- **初始标记（STW）**：标记从根节点直接可到达的对象。这阶段会伴随一次Young GC，会产生STW（全局停顿），应用程序会停止执行
- **根区域扫描**：由于Young GC的发生，所以初始标记后，eden被清空，存活对象放入Survivor区。然后本阶段，则扫描survivor区，标记可直达老年代的对象。本阶段应用程序可以并行执行。但是，根区域扫描不能和YoungGC同时执行（因为根区域扫描依赖survivor区的对象，而新生代GC会修改这个区域），因此如果恰巧在此时需要进行YoungGC，GC就需要等待根区域扫描结束后才能进行，如果发生这种情况，这次YoungGC的时间就会延长
- **并发标记**：用来再次扫描整个堆的存活对象，并做好标记。与CMS类似，该阶段可以被一次Young GC打断
- **重新标记（STW）**：本阶段也会发生STW，应用程序会停止执行。由于并发标记阶段中，应用程序也是并发执行的，所以本阶段，对标记结果进行最后的修正处理
- **独占清理（STW）**：本阶段也会发生STW，应用程序会停止执行。它用来计算各个区域的存活对象和GC回收比例，然后进行排序，从而识别出可以用来混合收集的区域。该阶段给出了需要被混合回收的区域并进行了标记，那么在混合收集阶段，是需要这些信息的
- **并发清理**：本阶段会去识别并清理那些完全空闲的区域

**G1 第3阶段：混合收集**

在第二步的并发标记周期过程中，虽然有部分对象被回收，但是总体回收比例还是比较低的。由于G1已经明确知道哪些区域含有比较多的垃圾比例，所以就可以针对比例较高的区域进行回收操作。

## 六、ZGC（Z Garbage Collector）

### 6.1 设计目标

**超低延迟**：STW 时间 < 1ms（JDK 16+），与堆大小无关。

### 6.2 核心技术

```
1. 染色指针（Colored Pointers）
   在 64 位指针中使用高位存储标记信息（4位用于 GC 状态）
   直接从指针就能知道对象的 GC 状态，无需访问对象

2. 读屏障（Load Barrier）
   每次读取对象引用时插入屏障代码，处理并发移动的对象
   (开销：每次引用读取多约 6% 的指令)

3. 并发移动（Concurrent Relocation）
   GC 线程与用户线程并发移动存活对象（几乎无 STW）
   通过读屏障将旧引用重定向到新地址
```

### 6.3 ZGC 参数

```bash
-XX:+UseZGC                     # JDK 15+ 正式可用，JDK 17+ 推荐
-XX:ZAllocationSpikeTolerance=5  # 分配波峰容忍度
-Xms16g -Xmx16g                 # 堆大小（建议初始=最大）
```

**适用场景**：对延迟极度敏感（金融交易、实时游戏服务器）；超大堆（几十 GB 到 TB 级）

## 七、收集器选择指南

```
堆 < 1G，单核，客户端：              → Serial + Serial Old
追求吞吐量，批处理，JDK 8：          → Parallel GC（默认）
在线服务，JDK 8，堆 4-8G，低延迟：  → CMS（已废弃）
通用，JDK 9+，平衡延迟与吞吐：      → G1（默认）
极低延迟（< 1ms），大堆，JDK 17+：  → ZGC
极低延迟，大堆，OpenJDK：           → Shenandoah
```

## 八、主要收集器对比

| 收集器 | 算法 | 区域 | 线程 | STW | 吞吐量 | 延迟 | 适用 |
|--------|------|------|------|-----|--------|------|------|
| Serial | 复制/标记整理 | 新+老 | 单线程 | 全程 | 差 | 高 | 客户端 |
| Parallel | 复制/标记整理 | 新+老 | 多线程并行 | 全程 | **高** | 较高 | 批处理 |
| CMS | 标记清除 | 老年代 | 并发 | 两次短暂 | 中 | 低 | 已废弃 |
| G1 | 复制+标记整理 | Region | 并发+并行 | 短 | 高 | **中低** | 通用 |
| ZGC | 并发复制 | Region | 几乎全并发 | **极短(<1ms)** | 中 | **极低** | 超低延迟 |

## 九、面试要点速查

| 问题 | 要点 |
|------|------|
| JDK 8 默认收集器 | Parallel Scavenge + Parallel Old（追求吞吐量）|
| JDK 9+ 默认收集器 | G1（平衡延迟与吞吐量）|
| CMS 的主要问题 | 内存碎片、浮动垃圾、并发失败退化、JDK 14 移除 |
| G1 和 CMS 的核心区别 | G1 基于 Region 可预测停顿，无碎片（复制算法）；CMS 标记清除有碎片 |
| ZGC 的核心技术 | 染色指针（指针存储GC状态）+ 读屏障（并发移动时重定向）|
| 如何选择 GC | 看堆大小、延迟要求、JDK 版本综合决定，压测验证 |
