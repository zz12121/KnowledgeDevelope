## 一、Spring 框架中的设计模式

Spring 是设计模式的教科书级实现，几乎每个核心模块都有经典模式的影子。

### 1.1 单例模式 — DefaultSingletonBeanRegistry

Spring 容器默认将 Bean 以单例形式管理，核心实现在 `DefaultSingletonBeanRegistry`：

```java
public class DefaultSingletonBeanRegistry {

    // 一级缓存：完整的单例对象
    private final Map<String, Object> singletonObjects =
            new ConcurrentHashMap<>(256);

    // 二级缓存：早期暴露对象（未完成属性注入，用于解决循环依赖）
    private final Map<String, Object> earlySingletonObjects =
            new ConcurrentHashMap<>(16);

    // 三级缓存：ObjectFactory，可产生代理对象
    private final Map<String, ObjectFactory<?>> singletonFactories =
            new HashMap<>(16);

    public Object getSingleton(String beanName) {
        // 先查一级缓存
        Object singletonObject = singletonObjects.get(beanName);
        if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
            // 查二级缓存
            singletonObject = earlySingletonObjects.get(beanName);
            if (singletonObject == null) {
                // 查三级缓存，触发 ObjectFactory 生成（可能产生代理）
                ObjectFactory<?> singletonFactory = singletonFactories.get(beanName);
                if (singletonFactory != null) {
                    singletonObject = singletonFactory.getObject();
                    earlySingletonObjects.put(beanName, singletonObject);
                    singletonFactories.remove(beanName);
                }
            }
        }
        return singletonObject;
    }
}
```

**与 GoF 单例的区别：**
- GoF 单例：类本身控制实例唯一性，`static` 字段保存
- Spring 单例：容器（IoC Container）控制唯一性，`ConcurrentHashMap` 保存
- Spring 单例的作用域是**容器级别**，同一个类可以在不同容器中有不同实例

### 1.3 模板方法模式 — JdbcTemplate & AbstractApplicationContext

**JdbcTemplate** 固定了 "获取连接 → 执行SQL → 处理结果 → 释放连接" 的骨架：

```java
public class JdbcTemplate extends JdbcAccessor implements JdbcOperations {

    // 模板方法，骨架固定，回调点可变
    public <T> T execute(PreparedStatementCreator psc,
                         PreparedStatementCallback<T> action) {
        Connection con = DataSourceUtils.getConnection(obtainDataSource()); // 固定步骤1
        PreparedStatement ps = null;
        try {
            ps = psc.createPreparedStatement(con);
            applyStatementSettings(ps);
            T result = action.doInPreparedStatement(ps); // 变化点：用户实现
            handleWarnings(ps);
            return result;
        } catch (SQLException ex) {
            // 固定步骤3：统一异常转换
            throw translateException("PreparedStatementCallback", getSql(psc), ex);
        } finally {
            JdbcUtils.closeStatement(ps);
            DataSourceUtils.releaseConnection(con, getDataSource()); // 固定步骤4
        }
    }
}

// 使用示例
@Repository
public class UserRepository {
    @Autowired
    private JdbcTemplate jdbcTemplate;

    public User findById(Long id) {
        return jdbcTemplate.queryForObject(
            "SELECT * FROM users WHERE id = ?",
            (rs, rowNum) -> new User(rs.getLong("id"), rs.getString("name")), // 回调点
            id
        );
    }
}
```

**AbstractApplicationContext.refresh()** 是更宏观的模板方法：

```java
public abstract class AbstractApplicationContext implements ApplicationContext {

    // final 保证骨架不可修改
    @Override
    public final void refresh() throws BeansException, IllegalStateException {
        prepareRefresh();                  // 1. 准备刷新
        ConfigurableListableBeanFactory beanFactory = obtainFreshBeanFactory(); // 2. 获取BF（子类实现）
        prepareBeanFactory(beanFactory);   // 3. 配置BF
        postProcessBeanFactory(beanFactory); // 4. 钩子（子类可重写）
        invokeBeanFactoryPostProcessors(beanFactory); // 5. 执行BF后处理器
        registerBeanPostProcessors(beanFactory);      // 6. 注册Bean后处理器
        initMessageSource();               // 7. 国际化
        initApplicationEventMulticaster(); // 8. 事件多播器
        onRefresh();                       // 9. 钩子（子类可重写，如启动Tomcat）
        registerListeners();               // 10. 注册监听器
        finishBeanFactoryInitialization(beanFactory); // 11. 实例化所有单例Bean
        finishRefresh();                   // 12. 完成刷新，发布ContextRefreshedEvent
    }

    // 抽象方法，子类必须实现
    protected abstract void refreshBeanFactory() throws BeansException;
}
```

### 1.4 代理模式 — AOP 核心实现

Spring AOP 根据目标类是否实现接口，自动选择代理策略：

```java
// AopProxyFactory 决策逻辑（DefaultAopProxyFactory 简化版）
public class DefaultAopProxyFactory implements AopProxyFactory {

    @Override
    public AopProxy createAopProxy(AdvisedSupport config) {
        if (!config.isOptimize() && !config.isProxyTargetClass()
                && !hasNoUserSuppliedProxyInterfaces(config)) {
            // 目标类实现了接口 → JDK 动态代理
            return new JdkDynamicAopProxy(config);
        } else {
            Class<?> targetClass = config.getTargetClass();
            if (targetClass.isInterface() || Proxy.isProxyClass(targetClass)) {
                return new JdkDynamicAopProxy(config);
            }
            // 没有实现接口 → CGLIB 代理
            return new ObjenesisCglibAopProxy(config);
        }
    }
}

// JDK 动态代理实现
final class JdkDynamicAopProxy implements AopProxy, InvocationHandler {

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 获取该方法的拦截器链
        List<Object> chain = this.advised.getInterceptorsAndDynamicInterceptionAdvice(
                method, targetClass);

        if (chain.isEmpty()) {
            // 无拦截器，直接调用目标方法
            return AopUtils.invokeJoinpointUsingReflection(target, method, args);
        } else {
            // 构建调用链，依次执行 Before/After/Around 通知
            MethodInvocation invocation = new ReflectiveMethodInvocation(
                    proxy, target, method, args, targetClass, chain);
            return invocation.proceed();
        }
    }
}
```

### 1.6 责任链模式 — Filter 链 & HandlerInterceptor

**Servlet Filter 链**（Spring Security 核心机制）：

```java
// 责任链节点基类
public abstract class OncePerRequestFilter implements Filter {

    @Override
    public final void doFilter(ServletRequest request, ServletResponse response,
                               FilterChain filterChain) {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        // 子类实现具体过滤逻辑
        doFilterInternal(httpRequest, (HttpServletResponse) response, filterChain);
    }

    protected abstract void doFilterInternal(HttpServletRequest request,
        HttpServletResponse response, FilterChain filterChain)
        throws ServletException, IOException;
}

// JWT 认证过滤器
@Component
@Order(1)
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain filterChain)
            throws ServletException, IOException {

        String token = extractToken(request);
        if (token != null && jwtUtils.validateToken(token)) {
            // 将认证信息放入 SecurityContext
            Authentication auth = jwtUtils.getAuthentication(token);
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        // 传递给下一个过滤器
        filterChain.doFilter(request, response);
    }
}

// Spring 自动按 @Order 排序注入过滤器链
@Configuration
public class FilterConfig {
    @Bean
    public FilterRegistrationBean<JwtAuthenticationFilter> jwtFilter(
            JwtAuthenticationFilter filter) {
        FilterRegistrationBean<JwtAuthenticationFilter> bean =
            new FilterRegistrationBean<>(filter);
        bean.setOrder(1);
        return bean;
    }
}
```

### 1.7 适配器模式 — HandlerAdapter

Spring MVC 的 `HandlerAdapter` 适配了不同类型的 Handler（`@Controller`、`HttpRequestHandler`、`Servlet` 等）：

```java
// 适配器接口
public interface HandlerAdapter {
    boolean supports(Object handler);
    ModelAndView handle(HttpServletRequest request,
                        HttpServletResponse response,
                        Object handler) throws Exception;
}

// 适配 @RequestMapping 注解的 Controller
public class RequestMappingHandlerAdapter implements HandlerAdapter {

    @Override
    public boolean supports(Object handler) {
        return handler instanceof HandlerMethod;
    }

    @Override
    public ModelAndView handle(HttpServletRequest request,
                               HttpServletResponse response,
                               Object handler) throws Exception {
        HandlerMethod handlerMethod = (HandlerMethod) handler;
        // 参数绑定、调用 Controller 方法、处理返回值
        return invokeHandlerMethod(request, response, handlerMethod);
    }
}

// DispatcherServlet 选择适配器（策略 + 适配器组合）
public class DispatcherServlet extends FrameworkServlet {
    private List<HandlerAdapter> handlerAdapters;

    protected HandlerAdapter getHandlerAdapter(Object handler) {
        for (HandlerAdapter adapter : this.handlerAdapters) {
            if (adapter.supports(handler)) {
                return adapter; // 找到能处理该 handler 的适配器
            }
        }
        throw new ServletException("No adapter for handler: " + handler);
    }
}
```

### 1.8 装饰器模式 — BeanWrapper & HttpServletRequestWrapper

**BeanWrapper** 为 Bean 对象提供属性访问能力（功能增强）：

```java

```

**HttpServletRequestWrapper** 装饰原始请求（读取一次body后可重复读）：

```java
public class CachedBodyHttpServletRequest extends HttpServletRequestWrapper {

    private final byte[] cachedBody;

    public CachedBodyHttpServletRequest(HttpServletRequest request) throws IOException {
        super(request);
        // 一次性读取并缓存 body
        InputStream inputStream = request.getInputStream();
        this.cachedBody = StreamUtils.copyToByteArray(inputStream);
    }

    @Override
    public ServletInputStream getInputStream() {
        // 每次调用都从缓存中返回新的流，支持重复读取
        return new CachedBodyServletInputStream(this.cachedBody);
    }

    @Override
    public BufferedReader getReader() {
        return new BufferedReader(new InputStreamReader(
            new ByteArrayInputStream(this.cachedBody)));
    }
}

// 在 Filter 中使用
@Component
public class RequestBodyCachingFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain) throws IOException, ServletException {
        // 用装饰后的 Request 替换原始 Request
        CachedBodyHttpServletRequest cachedRequest =
            new CachedBodyHttpServletRequest(request);
        chain.doFilter(cachedRequest, response); // 后续 Filter 可多次读取 body
    }
}
```

### 2.2 代理模式 — MapperProxy

MyBatis 为每个 Mapper 接口生成动态代理，这是它最核心的设计：

```java
// MapperProxy 实现了 InvocationHandler
public class MapperProxy<T> implements InvocationHandler, Serializable {

    private final SqlSession sqlSession;
    private final Class<T> mapperInterface;
    // 缓存方法对应的 MapperMethod（避免重复构建）
    private final Map<Method, MapperMethodInvoker> methodCache;

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // Object 的方法（toString、hashCode 等）直接调用
        if (Object.class.equals(method.getDeclaringClass())) {
            return method.invoke(this, args);
        }
        // 从缓存获取 MapperMethod，执行 SQL
        return cachedInvoker(method).invoke(proxy, method, args, sqlSession);
    }
}

// MapperProxyFactory 负责创建代理
public class MapperProxyFactory<T> {
    private final Class<T> mapperInterface;

    @SuppressWarnings("unchecked")
    public T newInstance(SqlSession sqlSession) {
        MapperProxy<T> mapperProxy = new MapperProxy<>(
            sqlSession, mapperInterface, methodCache);
        // JDK 动态代理生成接口实现类
        return (T) Proxy.newProxyInstance(
            mapperInterface.getClassLoader(),
            new Class[] { mapperInterface },
            mapperProxy
        );
    }
}
```

### 2.3 模板方法模式 — BaseExecutor

```java
// BaseExecutor 定义 SQL 执行的骨架
public abstract class BaseExecutor implements Executor {

    // 模板方法：查询骨架
    @Override
    public <E> List<E> query(MappedStatement ms, Object parameter,
                             RowBounds rowBounds, ResultHandler resultHandler,
```

### 2.4 装饰器模式 — CachingExecutor

```java
// CachingExecutor 装饰 Executor，为其添加二级缓存能力
public class CachingExecutor implements Executor {

    private final Executor delegate; // 被装饰的真实 Executor

    public CachingExecutor(Executor delegate) {
        this.delegate = delegate;
    }

    @Override
    public <E> List<E> query(MappedStatement ms, Object parameterObject,
                             RowBounds rowBounds, ResultHandler resultHandler,
                             CacheKey key, BoundSql boundSql) throws SQLException {

        Cache cache = ms.getCache(); // 获取二级缓存
        if (cache != null) {
            // 查询二级缓存
            flushCacheIfRequired(ms);
            if (ms.isUseCache() && resultHandler == null) {
                Object cachedResult = tcm.getObject(cache, key);
                if (cachedResult != null) {
                    return (List<E>) cachedResult; // 命中二级缓存，直接返回
                }
            }
        }
        // 未命中，委托给真实 Executor 执行
        List<E> list = delegate.query(ms, parameterObject, rowBounds,
                                      resultHandler, key, boundSql);
        if (cache != null && ms.isUseCache() && resultHandler == null) {
            tcm.putObject(cache, key, list); // 结果放入二级缓存
        }
        return list;
    }
}
```

### 2.5 组合模式 — 动态 SQL 的 SqlNode 树

```java
// 抽象组件：所有 SQL 节点的统一接口
public interface SqlNode {
    boolean apply(DynamicContext context);
}

// 叶子节点：静态 SQL 文本
public class StaticTextSqlNode implements SqlNode {
    private final String text;

    @Override
    public boolean apply(DynamicContext context) {
        context.appendSql(text); // 直接拼接文本
        return true;
    }
}

// 容器节点：<if> 标签
public class IfSqlNode implements SqlNode {
    private final SqlNode contents; // 子节点
    private final String test;      // 条件表达式

    @Override
    public boolean apply(DynamicContext context) {
        // 评估条件，满足则递归处理子节点
        if (evaluator.evaluateBoolean(test, context.getBindings())) {
            contents.apply(context);
            return true;
        }
        return false;
    }
}

// 容器节点：<foreach> 标签（更复杂的组合）
public class ForEachSqlNode implements SqlNode {
    private final SqlNode contents;
    // collection, item, index, open, close, separator...

    @Override
    public boolean apply(DynamicContext context) {
        // 迭代集合，对每个元素递归处理子节点
        Iterable<?> iterable = evaluator.evaluateIterable(collectionExpression, ...);
        for (Object item : iterable) {
            contents.apply(context); // 递归
        }
        return true;
    }
}

// MixedSqlNode：组合多个子节点（树的内部节点）
public class MixedSqlNode implements SqlNode {
    private final List<SqlNode> contents; // 子节点列表

    @Override
    public boolean apply(DynamicContext context) {
        // 依次处理所有子节点
        contents.forEach(node -> node.apply(context));
        return true;
    }
}
```

## 四、电商系统多模式综合实战

以下是一个**电商下单核心流程**的完整实现，综合运用了 5 种设计模式。

### 4.1 系统架构概览

```
用户下单请求
    ↓
【建造者】OrderRequest.build()    ← 构建复杂的订单请求对象
    ↓
【责任链】OrderValidationChain    ← 库存检查 → 风控检查 → 优惠券检查
    ↓
【模板方法】AbstractOrderProcessor ← 前置检查 → 扣库存 → 创建订单 → 后置处理
    ↓
【策略】PromotionStrategy         ← 根据活动类型选择优惠计算策略
    ↓
【观察者】OrderCreatedEvent        ← 通知：发短信、积分、推荐系统
```

### 4.2 完整代码实现

```java
// ==================== 建造者模式：构建订单请求 ====================
public class OrderRequest {
    // 必填字段
    private final Long userId;
    private final List<OrderItem> items;
    // 可选字段
    private final String couponCode;
    private final String promotionId;
    private final String deliveryAddress;
    private final String remark;

// 策略上下文：通过 Map 注入，按促销类型选择策略
@Component
public class PromotionContext {
    private final Map<String, PromotionStrategy> strategyMap;

    // Spring 自动将所有 PromotionStrategy 实现注入为 Map
    public PromotionContext(List<PromotionStrategy> strategies) {
        this.strategyMap = strategies.stream()
            .collect(Collectors.toMap(
                PromotionStrategy::getPromotionType,
                Function.identity()
            ));
    }

    public BigDecimal applyPromotion(OrderRequest request, BigDecimal originalPrice) {
        String promotionId = request.getPromotionId();
        if (promotionId == null) return originalPrice;

        PromotionStrategy strategy = strategyMap.get(promotionId);
        if (strategy == null) {
            throw new IllegalArgumentException("未知的促销类型: " + promotionId);
        }

        BigDecimal discount = strategy.calculateDiscount(request, originalPrice);
        return originalPrice.subtract(discount).max(BigDecimal.ZERO);
    }
}

// ==================== 模板方法：订单处理骨架 ====================
public abstract class AbstractOrderProcessor {

    @Autowired
    protected OrderValidationChain validationChain;
    @Autowired
    protected PromotionContext promotionContext;
    @Autowired
    protected ApplicationEventPublisher eventPublisher;

    // 模板方法：不可重写的处理骨架
    public final OrderResult process(OrderRequest request) {
        // 步骤1：前置校验（责任链）
        ValidationResult validation = validationChain.validate(request);
        if (!validation.isSuccess()) {
            return OrderResult.fail(validation.getMessage());
        }

        // 步骤2：计算价格（钩子：子类可重写）
        BigDecimal originalPrice = calculateOriginalPrice(request);
        BigDecimal finalPrice = promotionContext.applyPromotion(request, originalPrice);

        // 步骤3：执行订单创建（抽象方法，子类必须实现）
        Order order = doCreateOrder(request, finalPrice);

        // 步骤4：后置处理（钩子：子类可重写）
        afterOrderCreated(order, request);

        // 步骤5：发布事件（固定步骤，触发观察者）
        eventPublisher.publishEvent(new OrderCreatedEvent(this, order));

        return OrderResult.success(order.getOrderId());
    }

    // 钩子方法：默认实现，子类可重写
    protected BigDecimal calculateOriginalPrice(OrderRequest request) {
        return request.getItems().stream()
            .map(item -> item.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }

    // 抽象方法：子类实现不同的订单类型
    protected abstract Order doCreateOrder(OrderRequest request, BigDecimal finalPrice);

    // 钩子方法：子类可选扩展
    protected void afterOrderCreated(Order order, OrderRequest request) {
        // 默认空实现
    }
}
```

```java
// 断路器状态机（状态模式的典型应用）
public enum CircuitBreakerState {
    CLOSED,    // 关闭（正常）：请求正常通过
    OPEN,      // 打开（熔断）：请求直接失败，快速返回
    HALF_OPEN  // 半开（探测）：允许少量请求探测服务是否恢复
}

@Component
public class CircuitBreakerService {
    private CircuitBreakerState state = CircuitBreakerState.CLOSED;
    private final AtomicInteger failureCount = new AtomicInteger(0);
    private final AtomicInteger successCount = new AtomicInteger(0);
    private volatile long openTime;

    private static final int FAILURE_THRESHOLD = 5;     // 失败5次触发熔断
    private static final long TIMEOUT_DURATION = 10000; // 熔断持续10秒
    private static final int SUCCESS_THRESHOLD = 3;     // 半开状态成功3次恢复

    public <T> T execute(Supplier<T> request, Supplier<T> fallback) {
        switch (state) {
            case OPEN:
                if (System.currentTimeMillis() - openTime > TIMEOUT_DURATION) {
                    state = CircuitBreakerState.HALF_OPEN;
                    // 进入半开状态，尝试一个请求
                    return tryRequest(request, fallback);
                }
                return fallback.get(); // 熔断中，直接走降级

            case HALF_OPEN:
                return tryRequest(request, fallback);

            default: // CLOSED
                try {
                    T result = request.get();
                    reset();
                    return result;
                } catch (Exception e) {
                    recordFailure();
                    return fallback.get();
                }
        }
    }

    private <T> T tryRequest(Supplier<T> request, Supplier<T> fallback) {
        try {
            T result = request.get();
            if (successCount.incrementAndGet() >= SUCCESS_THRESHOLD) {
                state = CircuitBreakerState.CLOSED; // 恢复正常
                reset();
            }
            return result;
        } catch (Exception e) {
            state = CircuitBreakerState.OPEN; // 再次熔断
            openTime = System.currentTimeMillis();
            return fallback.get();
        }
    }

    private void recordFailure() {
        if (failureCount.incrementAndGet() >= FAILURE_THRESHOLD) {
            state = CircuitBreakerState.OPEN;
            openTime = System.currentTimeMillis();
        }
    }

    private void reset() {
        failureCount.set(0);
        successCount.set(0);
    }
}
```

```java
// Saga 协调者：使用命令模式封装每步操作（含补偿）
public interface SagaStep {
    void execute();          // 正向操作
    void compensate();       // 补偿操作（回滚）
    String getStepName();
}

@Component
public class OrderSaga {

    @Autowired private List<SagaStep> sagaSteps; // 按顺序注入所有步骤

    public void execute() {
        List<SagaStep> completedSteps = new ArrayList<>();

        for (SagaStep step : sagaSteps) {
            try {
                step.execute();
                completedSteps.add(step);
                log.info("Saga步骤 [{}] 执行成功", step.getStepName());
            } catch (Exception e) {
                log.error("Saga步骤 [{}] 执行失败，开始补偿", step.getStepName(), e);
                // 逆序执行已完成步骤的补偿操作
                Collections.reverse(completedSteps);
                completedSteps.forEach(completedStep -> {
                    try {
                        completedStep.compensate();
                        log.info("补偿步骤 [{}] 执行成功", completedStep.getStepName());
                    } catch (Exception compensateEx) {
                        log.error("补偿步骤 [{}] 失败，需要人工介入",
                            completedStep.getStepName(), compensateEx);
                    }
                });
                throw new SagaFailureException("Saga事务失败", e);
            }
        }
    }
}
```

## 六、设计模式选型原则

### 6.1 选型决策树

```
遇到设计问题时，按以下顺序思考：

1. 是否存在代码重复 / 扩展困难 / 高耦合？
   → 不是 → 不要引入模式，保持简单

2. 哪类问题？
   → 对象创建太复杂               → 创建型（单例/工厂/建造者/原型）
   → 类之间结构耦合太强           → 结构型（代理/装饰器/适配器/门面/桥接）
   → 算法/行为需要灵活切换        → 行为型（策略/模板方法/命令/责任链）
   → 对象状态变化触发行为变化     → 状态模式
   → 一个动作需要通知多个模块     → 观察者模式

3. 引入后是否值得？
   → 代码量增加 vs 可维护性提升
   → 是否在团队能力范围内理解和维护
   → 业务是否真的有多变点（多态点）
```

### 6.2 过度设计的边界

| 场景 | ❌ 过度设计 | ✅ 合理设计 |
|------|-----------|-----------|
| if-else 只有 2~3 个分支 | 引入策略模式 | 直接写 if-else |
| 对象创建只需 new + 几个属性 | 引入建造者模式 | 直接构造 |
| 调用链只有 1~2 个步骤 | 引入责任链 | 直接方法调用 |
| 只有一种算法实现 | 引入工厂方法 | 直接实例化 |
| 类之间没有变化方向 | 引入桥接/装饰器 | 直接继承或组合 |

**黄金判断标准**：**模式是为变化点而生的**。如果某个维度目前没有变化，也没有合理的扩展预期，不要提前引入模式。YAGNI（You Aren't Gonna Need It）。

### 6.3 渐进式重构方法

```
第一步：识别痛点（不是凭感觉，是看数据）
  → 哪个方法最难改？最容易出 Bug？每次需求都要动它？

第二步：最小化重构，先提取方法/类
  → 不要一上来就引入模式，先把代码组织好

第三步：观察变化方向
  → 经过 2~3 次需求后，变化的方向才会清晰
  → 根据变化方向选择对应的模式

第四步：引入模式，同步更新测试
  → 确保行为不变，测试覆盖重构前后

第五步：文档化
  → 注释说明为什么用这个模式，方便团队理解
```

```java
// 第一阶段：原始代码（需求增加时每次都要改这个方法）
public BigDecimal calculateDiscount(String type, BigDecimal price) {
    if ("FULL_REDUCE".equals(type)) {
        return price.compareTo(new BigDecimal("300")) >= 0
            ? new BigDecimal("50") : BigDecimal.ZERO;
    } else if ("PERCENT".equals(type)) {
        return price.multiply(new BigDecimal("0.1"));
    } else if ("NEW_USER".equals(type)) {
        return new BigDecimal("20");
    }
    // 每次新增促销类型都要改这里 ← 痛点
    return BigDecimal.ZERO;
}

// 第二阶段：提取接口，分离实现
public interface DiscountCalculator {
    boolean supports(String type);
    BigDecimal calculate(BigDecimal price);
}

// 第三阶段：Spring 注入，消除 if-else
@Component
public class DiscountService {
    private final Map<String, DiscountCalculator> calculatorMap;

    public DiscountService(List<DiscountCalculator> calculators) {
        this.calculatorMap = calculators.stream()
            .collect(Collectors.toMap(
                c -> c.getType(),  // 每个实现类声明自己支持的类型
                Function.identity()
            ));
    }

    // 新增促销类型：只需新增一个类，这里不需要改动 ← 开闭原则
    public BigDecimal calculate(String type, BigDecimal price) {
        return Optional.ofNullable(calculatorMap.get(type))
            .map(calc -> calc.calculate(price))
            .orElse(BigDecimal.ZERO);
    }
}
```

### 7.1 模式速查总表

| 分类 | 模式名称 | 核心意图 | 典型场景 |
|------|---------|---------|---------|

| | 工厂方法 | 子类决定创建哪种对象 | 通知服务（邮件/短信/Push） |
| | 抽象工厂 | 创建一组相关联的产品 | DAO层（MySQL/MongoDB切换） |
| | 建造者 | 逐步构建复杂对象，参数可选 | 查询条件、HTTP请求构建器 |
| | 原型 | 通过克隆创建对象 | 游戏角色、文档模板复制 |
| **结构型** | 代理 | 控制对象访问，功能增强 | AOP日志/缓存/事务，MyBatis Mapper |
| | 装饰器 | 动态叠加功能，不改原类 | I/O流、数据处理管道 |
| | 适配器 | 接口转换，兼容不同实现 | 第三方SDK适配、HandlerAdapter |
| | 门面 | 统一简化子系统调用 | 下单门面、支付门面 |
| | 桥接 | 分离变化的两个维度 | 消息类型 × 推送渠道 |
| | 组合 | 统一处理叶子和容器 | 组织架构树、菜单树、SqlNode |
| | 享元 | 共享细粒度对象，节省内存 | 字符样式池、Integer缓存、String常量池 |
| **行为型** | 模板方法 | 固定算法骨架，步骤可变 | JdbcTemplate、refresh()、导入框架 |
| | 策略 | 算法族可整体替换 | 促销计算、排序算法、支付方式 |
| | 观察者 | 一对多事件通知 | Spring ApplicationEvent、MQ消费 |
| | 责任链 | 请求沿链处理，职责分离 | 过滤器链、拦截器链、审批流 |
| | 命令 | 请求对象化，支持撤销排队 | 智能家居遥控、操作历史、MQ消息 |
| | 迭代器 | 统一遍历接口，隐藏内部结构 | 分页查询、数据库游标 |
| | 中介者 | 消除网状耦合，集中协调 | 聊天室、GUI事件总线 |
| | 备忘录 | 保存和恢复对象状态 | 文本编辑器撤销、游戏存档 |
| | 状态 | 对象行为随状态自动变化 | 订单状态机、审批流状态 |
| | 访问者 | 对稳定结构施加多种操作 | AST节点处理、编译器 |
| | 解释器 | 解析简单的规则语言 | 权限DSL、表达式引擎 |

### 7.2 模式之间的关联关系

```
【创建型 → 结构型】
  工厂方法 常与 模板方法 配合（父类定义创建逻辑，子类实现具体产品）
  抽象工厂 常与 单例 配合（工厂本身通常是单例）

【结构型内部】
  装饰器 和 代理 结构相似，区别在意图：
    - 代理：控制访问（权限、缓存、日志）
    - 装饰器：增强功能（功能叠加，用户主动选择）
  适配器 和 门面 都是"包装"，区别在对象数量：
    - 适配器：包装1个对象，转换接口
    - 门面：包装多个子系统，简化调用

【行为型内部】
  策略 和 模板方法 都是封装算法变化：
    - 模板方法：变化通过继承（is-a）
    - 策略：变化通过组合（has-a）推荐使用策略
  观察者 和 中介者 都处理对象间通信：
    - 观察者：一对多广播，观察者直接引用被观察者
    - 中介者：多对多，通过中间人解耦

【跨分类组合（最常见）】
  策略 + 工厂方法：用工厂创建策略对象
  模板方法 + 钩子：骨架固定，钩子点用策略填充
  责任链 + 命令：链上每个节点封装一个命令
  观察者 + 命令：将事件处理封装为命令，支持撤销
  装饰器 + 工厂：工厂负责构建装饰器链
```

---

> 设计模式不是目的，解决问题才是。好的代码首先是简单的，其次才是优雅的。在真实工程中，能用3行if-else解决的问题，不要用30行设计模式去解决——除非你确实预见了变化方向。
