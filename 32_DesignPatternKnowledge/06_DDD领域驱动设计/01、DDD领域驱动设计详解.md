# DDD 领域驱动设计

## 1. 为什么需要 DDD

### 日常迭代中的常见问题

**问题1：需求价值评估困难**
- 问题：针对每期迭代 PM 提出的大量业务需求，难道只能靠被动加班完成迭代任务吗？
- 解决办法：通过某种方式，评定业务价值的高低；根据业务价值实现需求的过滤和排序，从而把控迭代内的研发任务
- DDD 解决办法：
  1. 通过划分限界上下文和子域确定业务核心域
  2. 在核心领域下，由领域专家&研发人员共同确定业务需求真正的价值

**问题2：业务人员与研发人员的无尽"斗争"**
- 核心问题：业务人员与开发人员不同视角导致的争执
- 其他问题：
  1. 需求变味了：业务人员的需求转换为开发人员的理解，在代码实现过程中的转译
  2. 业务强势：需求不考虑技术底层实现，某些不重要的需求会导致原有架构破坏
  3. 开发强势：业务需求被扭曲变形或者被砍掉，无法体现业务价值
  4. 人员离职：业务知识和代码逻辑往往掌握在某些人手中，当其离职也随之被带走
- DDD 解决办法：
  1. 寻找领域专家，并构建领域团队
  2. 实现领域驱动开发，设计基于业务的合理的领域及对象模型，让领域对象"活起来"

**问题3：业务人员各抒己见分歧不断**
- 当多个业务人员之间存在分歧的时候，没有一个"地方"可以让他们坐下来统一分歧
- 核心问题：缺少一个组织结构上的"地点"，让领域专家之间达成共识
- 解决办法：构建领域团队，确定核心域

**问题4：软件的技术实现可能错误地改变软件原有的业务规则**
- 需求交流时没有问题，但开发过程中不确定因素导致最终交付产品与需求设想大相径庭
- 核心问题：技术人员对业务理解不深，没有理解领域专家所谓的业务价值
- 解决办法：领域专家与开发人员协同工作，创建领域内的通用语言提升沟通效率和降低理解差异性

### 我们期待的迭代模式

- 使领域专家和开发者一起工作，将他们组成一个密切协作的团队
- 开发出来的软件能够准确地传达业务规则（代码就好像是领域专家编写出来似的）
- 帮助业务人员自我提高，每个人都是学习者，同时又是知识的贡献者

## 2. DDD 概述

**领域驱动设计（DDD：Domain-Driven Design）** 是一种软件开发方法，它不是关于技术细节的，是用于构建集中化的业务知识体系。并且可以帮助我们设计出高质量的、能够准确表达业务意图的领域模型。

### 领域模型

- 领域模型是关于某个特定业务领域的软件模型
- 通常，领域模型通过对象模型来实现（对象 = 数据 + 行为），并且表达了准确的业务含义

### 领域专家

- 领域专家并不是一个职位，他可以是精通业务的任何人——销售、产品经理、软件开发人员......

### 战略设计和战术设计

DDD 同时提供了宏观的战略设计，也提供了需要细节的战术设计：

- **战略设计**：强调业务战略上的重点，如何按重要性分配工作，以及如何进行最佳整合
- **战术设计**：在打好战略设计的基础之后，使用精小的画笔在领域模型上描绘着每个细枝末节

## 3. 限界上下文与通用语言

### 限界上下文（Bounded Context）

- 限界上下文是语义和语境上的边界
- 它即是问题空间的一部分又会随着软件模型的进化转化为解决方案空间：
  - **问题空间**：在给定项目的约束条件下进行高级战略分析与设计的地方
  - **解决方案空间**：真正实施解决方案的地方，这些解决方案在问题空间讨论中被识别为核心域
- 当限界上下文被当做组织的关键战略举措进行开发时，即被称为**核心域（Core Domain）**
- 核心域表明了组织的业务主航道，发展核心域可以使组织在与其他组织的竞争中脱颖而出

### 领域通用语言

- 在 DDD 中，通用语言就是软件模型团队日常交流时使用的语言
- 软件模型的源代码就是这种语言的书面表达方式
- 我们可以将限界上下文理解为国家；将领域通用语言理解为这个国家的语言
- 限界上下文和通用语言是一对一的关系

### 限界上下文与团队&代码库

- 一个团队应该在一个限界上下文中工作
- 每个限界上下文应该拥有一个独立的源代码仓库
- 一个团队可能工作在多个限界上下文中，但是多个团队不应该在同一个限界上下文中共事

## 4. 子域（Subdomain）

### 什么是子域

- 子域代表的是一个单一的、有逻辑的领域模型
- 子域可以用来逻辑地拆分整个业务领域，便于认清业务空间重要性的主次
- 一个项目会被分割成为多个限界上下文，而这些限界上下文中有一个将会成为核心域，而其他限界上下文也会存在许多不同的子域

### 子域的类型

| 类型 | 说明 |
|------|------|
| **核心域（Core Domain）** | 唯一的、定义明确的领域模型，需要投入大量的资源去精心打磨。是组织中最重要的项目，是组织最需要在软件中倾斜其投资的方向 |
| **支撑子域（Supporting Subdomain）** | 建模场景提倡的是"定制开发"，也许会考虑采取外包的方式实现从而控制投入投资成本。核心域的成功离不开它 |
| **通用子域（Generic Subdomain）** | 解决方案可以采购现成的，也可以采用外包的方式，抑或是由内部团队实现，但不用为其分配与核心域同样优质的研发资源 |

## 5. 上下文映射

### 概述

- 多个限界上下文之间需要进行集成，这种集成关系在 DDD 中称为**上下文映射**
- 上下文映射代表着两种语言之间的转译过程

### 映射的种类

| 类型 | 说明 |
|------|------|
| **合作关系（Partnership）** | 两个团队各自负责一个限界上下文，但他们通过互相依赖的一套目标联合起来——即：一损俱损，一荣俱荣 |
| **共享内核（Shared Kernel）** | 两个（或更多）团队之间共享着一个小规模但却通用的模型，团队之间的沟通必须完全开放 |
| **客户-供应商（Consumer-Supplier）** | 供应商位于上游，客户位于下游。支配这种关系的是供应商，因为它必须提供客户需要的东西 |
| **跟随者（Conformist）** | 上游团队没有任何动机满足下游团队的具体需求，下游团队无法投入资源去翻译上游模型的通用语言，只能顺应上游的模型 |
| **防腐层（Anticorruption Layer）** | 下游团队在其通用语言和位于它上游的通用语言之间创建了一个翻译层，隔离了下游模型与上游模型 |
| **开放主机服务（Open Host Service）** | 定义一套协议或接口，让限界上下文可以被当做一组服务访问 |
| **已发布语言（Published Language）** | 一种有着丰富文档的信息交换语言，可以被许多消费方的限界上下文简单地使用和翻译 |
| **各行其道（Separate Way）** | 在限界上下文中创造属于自己的特殊解决方案，放弃针对这种特殊情况的集成 |

### 上下文映射集成方式

| 方式 | 说明 |
|------|------|
| **RPC** | 远程过程调用，现代 RPC 机制如 gRPC 和 Thrift 更友好支持多编程语言 |
| **RESTful HTTP** | 打破客户端和服务端编程语言上的隔阂，需要封装对外请求和响应的实体，与领域模型中的聚合分离开 |
| **消息机制** | 通过发送相关领域事件（Domain Event），由订阅的下游接收处理，是最健壮的集成方法之一 |

## 6. 实体、聚合、值对象

### 实体（Entity）

- 一个实体模型就是一个独立的事物
- 每个实体都拥有一个唯一的标识符，可以将它的个体性和所有其他类型相同或者不同的实体区分开
- 将实体与其他建模工具区分开的主要因素是它的唯一性——即：它的个体性

### 聚合（Aggregate）

- 聚合是由一个或多个实体组成，其中一个实体被称为**聚合根（AggregateRoot）**
- 聚合的组成还可能包括值对象

### 值对象（Value Object）

- 值对象是对一个不变的概念整体所建立的模型
- 和实体不一样，它没有唯一标识符，而是由值类型封装的属性对比来决定相等性
- 一个值对象不是事物，而是常常被用来描述、量化或者测量一个实体

### 聚合法则

| 法则 | 说明 |
|------|------|
| **法则一** | 在聚合边界内保护业务规则不变性 |
| **法则二** | 聚合要设计得小巧，内存占用空间和事务包含范围应该相对较小 |
| **法则三** | 只能通过标识符引用其他聚合，降低内存需求，提升持久化存储中加载的速度 |
| **法则四** | 使用最终一致性更新其他聚合，通过领域事件触发订阅方限界上下文继续后半部分操作 |

### 贫血模型 vs 充血模型

- **贫血模型**：只有公有访问器（Getter 和 Setter）之外没有包含任何真正的业务行为，业务逻辑被错放在服务或工具类中
- **充血模型**：业务逻辑放在领域模型中，领域对象"活起来"

## 7. 领域事件

### 概述

- 领域事件是一条记录，记录着在限界上下文中发生的对业务产生重要影响的事情
- 领域事件往往会在战术设计的过程中被概念化并演变成核心域的组成部分
- 如果因果关系没有以正确的方式实现，整个领域就会完全错误

### 设计原则

- 领域事件类型的命名应该体现出模型的通用语言
- 名称应该是对过去发生的事情的陈述，即**动词的过去式**，例如：ProductCreated、ReleaseScheduled
- 领域事件应该包含触发该领域事件产生的命令提供的所有属性信息

### 事件溯源（Event Sourcing）

- 对所有发生在聚合实例上的领域事件进行持久化，把它们当作对聚合实例变化的记录
- 一个聚合实例上发生的所有领域事件，按照它们原本发生的顺序，组成了该实例的事件流
- 在聚合上重新应用事件流，可以让它的状态从持久化存储中被重建到内存中

## 8. COLA 架构 & CQRS 架构

### CQRS（Command Query Responsibility Segregation）

- 命令查询职责分离
- 将读取和写入操作分离到不同的模型中
- 命令模型处理增删改，查询模型处理读取

### COLA（Clean Architecture and Object-Oriented Architecture）

- 整洁架构和面向对象架构
- 洋葱架构的落地实践
