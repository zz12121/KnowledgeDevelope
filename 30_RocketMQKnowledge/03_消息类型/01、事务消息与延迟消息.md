# 事务消息与延迟消息

## 一、事务消息核心原理

RocketMQ 事务消息通过**半消息（Half Message）+ 两阶段提交 + 事务回查**机制，解决本地事务与消息发送的原子性问题，实现分布式事务的最终一致性。

### 1.1 为什么需要事务消息？

直接发送消息存在两个问题：
- **先发消息再执行本地事务**：消息发出去了，本地事务执行失败，消费者却已消费——数据不一致
- **先执行本地事务再发消息**：本地事务成功了，消息发送失败——消费者收不到消息，数据丢失

事务消息通过引入**半消息**（对消费者不可见的中间状态）解决这两个问题。

### 1.2 两阶段提交流程

```
Producer                         Broker                        Consumer
   │                                │                              │
   │── 1. 发送半消息 ──────────────→│                              │
   │                                │ 存入 RMQ_SYS_TRANS_HALF_TOPIC│
   │←─ 2. 返回 SEND_OK ────────────│                              │
   │                                │                              │
   │── 3. 执行本地事务 ─────────────┤                              │
   │   (insert order, update balance)│                             │
   │                                │                              │
   │── 4a. Commit ─────────────────→│                              │
   │    OR                          │ 转存到目标 Topic              │
   │── 4b. Rollback ───────────────→│ 丢弃半消息                   │
   │                                │──────── 5. 投递消息 ────────→│
```

**关键细节**：
- 半消息存储在内部 Topic `RMQ_SYS_TRANS_HALF_TOPIC` 中，消费者订阅的是原始 Topic，所以看不到半消息
- Commit 时 Broker 将消息的 Topic 改回原始 Topic，重新写入 CommitLog，才对消费者可见
- Rollback 时直接标记删除，不写入消费者可见的 Topic

### 1.3 事务回查机制

当 Producer 发送 Commit/Rollback 后宕机，或网络分区导致 Broker 未收到确认，Broker 会启动定时回查：

```
Broker 的 TransactionalMessageCheckService 定时扫描
    ↓
找到超时未确认的半消息（默认 60s 超时）
    ↓
向 Producer 发起回查请求（checkLocalTransaction）
    ↓
Producer 查询本地事务状态：
    - 已提交 → 返回 COMMIT_MESSAGE
    - 已回滚 → 返回 ROLLBACK_MESSAGE
    - 不确定 → 返回 UNKNOW（等待下次回查，最多 5 次）
```

**回查配置**：
```properties
# broker.conf
transactionTimeout=60          # 事务超时时间（秒）
transactionCheckMax=5          # 最大回查次数
transactionCheckInterval=60    # 回查间隔（秒）
```

## 二、事务消息代码实现

### 2.1 原生 RocketMQ API

```java
TransactionMQProducer producer = new TransactionMQProducer("order-tx-group");
producer.setNamesrvAddr("localhost:9876");

// 回查线程池（用于处理 Broker 的回查请求）
ExecutorService checkPool = new ThreadPoolExecutor(2, 5, 100,
        TimeUnit.SECONDS, new ArrayBlockingQueue<>(2000));
producer.setExecutorService(checkPool);

// 事务监听器
producer.setTransactionListener(new TransactionListener() {

    /**
     * 第一阶段：半消息发送成功后，立即执行本地事务
     */
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        String orderId = (String) arg;
        try {
            // 执行本地业务：创建订单
            orderService.createOrder(orderId);
            // 本地事务成功 → Commit
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            log.error("本地事务执行失败, orderId={}", orderId, e);
            // 本地事务失败 → Rollback
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    /**
     * 第二阶段（回查）：Broker 未收到确认时，主动来查询本地事务状态
     */
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        String orderId = msg.getKeys();
        // 查询本地事务结果（如查数据库订单表）
        OrderStatus status = orderService.queryOrderStatus(orderId);
        if (status == OrderStatus.CREATED) {
            return LocalTransactionState.COMMIT_MESSAGE;
        } else if (status == OrderStatus.CANCELLED) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        } else {
            // 状态不明，等待下次回查
            return LocalTransactionState.UNKNOW;
        }
    }
});

producer.start();

// 发送事务消息
Message msg = new Message("OrderTopic", "TagA",
        "orderId=1001".getBytes(StandardCharsets.UTF_8));
msg.setKeys("order-1001"); // 设置业务 Key，方便回查和消息追踪

TransactionSendResult result = producer.sendMessageInTransaction(msg, "order-1001");
System.out.println("发送结果: " + result.getLocalTransactionState());
```

### 2.2 Spring Boot 集成

```java
@Service
public class OrderTransactionService implements RocketMQLocalTransactionListener {

    @Autowired
    private OrderRepository orderRepository;

    /**
     * 执行本地事务
     */
    @Override
    @Transactional
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            String orderId = new String((byte[]) msg.getPayload());
            orderRepository.save(new Order(orderId, OrderStatus.CREATED));
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }

    /**
     * 事务回查
     */
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        String orderId = new String((byte[]) msg.getPayload());
        Order order = orderRepository.findById(orderId).orElse(null);
        if (order != null && order.getStatus() == OrderStatus.CREATED) {
            return RocketMQLocalTransactionState.COMMIT;
        }
        return RocketMQLocalTransactionState.ROLLBACK;
    }
}
```

```java
@Component
public class OrderProducer {

}
```

## 三、分布式事务设计模式

### 3.1 跨服务场景

以「下单支付」场景为例，订单服务和支付服务需要保持数据一致：

```
订单服务                    RocketMQ                    支付服务
   │                           │                           │
   │── 1. 发送半消息 ──────────→│                           │
   │── 2. 创建订单（DB 写入）──→│                           │
   │── 3. Commit ─────────────→│                           │
   │                           │── 4. 投递订单消息 ────────→│
   │                           │                           │── 5. 执行扣款（幂等）
   │                           │                           │── 6. 回调订单服务更新状态
```

**设计要点**：
1. **回查方法要幂等**：可能被多次调用，查询结果需幂等
2. **回查与执行共享数据源**：如果查本地事务状态需要查数据库，要保证数据库可见
3. **消费侧需要幂等**：支付服务消费订单消息时，必须做幂等处理防止重复扣款

## 四、延迟消息原理

### 4.1 实现机制：主题切换 + 定时调度

RocketMQ 延迟消息的核心思路是**偷梁换柱**：
1. Broker 收到延迟消息，将原始 Topic 备份到消息属性中，然后把 Topic 改为内部主题 `SCHEDULE_TOPIC_XXXX`
2. `ScheduleMessageService` 为每个延迟级别启动一个定时任务，定期扫描是否有消息到期
3. 消息到期后，恢复原始 Topic，将消息重新写入 CommitLog，此时消费者才能看到

```
Producer 发送延迟消息
    ↓
Broker 接收：备份原 Topic → 改 Topic 为 SCHEDULE_TOPIC_XXXX → 存 CommitLog
    ↓
ScheduleMessageService 定时扫描（每 100ms）
    ↓
消息到期：恢复原 Topic → 重写 CommitLog → 构建 ConsumeQueue
    ↓
Consumer 正常消费
```

**关键源码片段**：
```java
// Broker 接收延迟消息时的处理（MessageStoreConfig）
if (msg.getDelayTimeLevel() > 0) {
    // 备份原始 Topic
    MessageAccessor.putProperty(msg, MessageConst.PROPERTY_REAL_TOPIC, msg.getTopic());
    MessageAccessor.putProperty(msg, MessageConst.PROPERTY_REAL_QUEUE_ID,
            String.valueOf(msg.getQueueId()));
    // 替换为内部延迟主题
    msg.setTopic(TopicValidator.RMQ_SYS_SCHEDULE_TOPIC);
    msg.setQueueId(ScheduleMessageService.delayLevel2QueueId(msg.getDelayTimeLevel()));
}
```

### 4.2 4.x 版本：18 个固定延迟级别

```java
// Broker 端固定配置，对应 delayLevel 1~18
// 1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h

// 使用方式：设置 delayLevel（1-18 对应上面的时间）
Message msg = new Message("DelayTopic", "TagA", "Hello".getBytes());
msg.setDelayTimeLevel(3); // 对应 10s 延迟
producer.send(msg);
```

### 4.3 5.x 版本：自定义延迟时间

```java
// RocketMQ 5.x 支持毫秒级自定义延迟
rocketMQTemplate.syncSendDelayTimeMills("DelayTopic:TagA", "Hello", 5000); // 延迟5秒

// 也支持设置具体的投递时间点
Message msg = new Message("DelayTopic", "TagA", "Hello".getBytes());
msg.setDeliverTimeMs(System.currentTimeMillis() + 60 * 1000); // 1分钟后投递
producer.send(msg);
```

## 五、延迟消息代码示例

```java
// ==================== 生产者：发送延迟消息 ====================
@Component
public class DelayMessageProducer {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    /**
     * 订单超时关闭：30分钟后检查订单支付状态
     */
    public void sendOrderTimeoutCheck(String orderId) {
        OrderCheckMessage checkMsg = new OrderCheckMessage(orderId, System.currentTimeMillis());

        // 4.x：使用固定级别（level=16 对应 30m）
        rocketMQTemplate.syncSend("OrderTimeoutTopic",
                MessageBuilder.withPayload(checkMsg).build(),
                3000,
                16); // delayLevel=16 → 30分钟

        log.info("发送订单超时检查消息, orderId={}, 将在30分钟后触发", orderId);
    }
}

// ==================== 消费者：处理到期消息 ====================
@Component
@RocketMQMessageListener(topic = "OrderTimeoutTopic", consumerGroup = "order-timeout-group")
public class OrderTimeoutConsumer implements RocketMQListener<OrderCheckMessage> {

    @Autowired
    private OrderService orderService;

    @Override
    public void onMessage(OrderCheckMessage message) {
        String orderId = message.getOrderId();
        Order order = orderService.findById(orderId);

        if (order == null) {
            log.warn("订单不存在: {}", orderId);
            return;
        }

        if (order.getStatus() == OrderStatus.PENDING_PAYMENT) {
            // 超时未支付，关闭订单
            orderService.closeOrder(orderId);
            log.info("订单超时关闭: {}", orderId);
        }
        // 已支付：无需处理
    }
}
```

## 六、定时消息 vs 延迟消息

在 RocketMQ 5.x 中二者本质相同，都通过 `ScheduleMessageService` 处理，差异只在使用语义：
- **延迟消息**：强调延迟一段时间间隔，用 `delayTimeLevel` 或 `syncSendDelayTimeMills`
- **定时消息**：强调在某个具体时间点投递，用 `setDeliverTimeMs(timestamp)` 设置绝对时间

两者均要求**应用服务器与 Broker 时钟同步**，否则投递时间不准确。

## 七、消息过滤：Tag vs SQL92

| 对比维度 | Tag 过滤 | SQL92 过滤 |
|---------|---------|-----------|
| 过滤粒度 | 单个字符串标签 | 多属性复杂条件表达式 |
| 性能开销 | 极低（哈希匹配，O(1)） | 较高（需解析 SQL 并计算） |
| 使用限制 | 一条消息只能设置一个 Tag | 需 Broker 开启 `enablePropertyFilter=true` |
| 适用场景 | 简单业务分类订阅 | 多属性精细过滤 |

```java
// Tag 过滤（推荐，性能最优）
consumer.subscribe("TopicTest", "TagA || TagB || TagC");

// SQL92 过滤（复杂条件场景）
consumer.subscribe("TopicTest",
    MessageSelector.bySql("(TAGS is not null and TAGS in ('TagA', 'TagB'))" +
                          " and (price between 10 and 100)"));

// 生产者设置自定义属性（供 SQL92 过滤使用）
Message msg = new Message("TopicTest", "TagA", "body".getBytes());
msg.putUserProperty("price", "50");
msg.putUserProperty("category", "electronics");
producer.send(msg);
```

**设计建议**：Tag 能满足需求优先用 Tag；SQL92 过滤要监控 Broker CPU，避免复杂表达式拖慢 Broker。
