# 消息发送与消费模式

## 一、三种发送方式

RocketMQ 提供三种消息发送方式，在**可靠性、性能和复杂度**上各有取舍：

| 发送方式 | 机制 | 可靠性 | 吞吐量 | 适用场景 |
|---------|------|--------|--------|---------|
| **同步发送** | 线程阻塞等待 Broker 响应 | 最高 | 中（~5k-10k TPS） | 金融交易、订单创建等强一致性场景 |
| **异步发送** | 立即返回，回调通知结果 | 高 | 高（~20k+ TPS） | 日志收集、通知类等高吞吐场景 |
| **单向发送** | 发后即忘，不关心结果 | 最低 | 极高（~50k+ TPS） | 心跳检测、监控上报等可容忍丢失的场景 |

```java
DefaultMQProducer producer = new DefaultMQProducer("ProducerGroup");
producer.setNamesrvAddr("localhost:9876");
producer.start();

Message msg = new Message("TopicTest", "TagA", "Hello RocketMQ".getBytes());

// 1. 同步发送 —— 阻塞等待结果
SendResult syncResult = producer.send(msg);
System.out.println("同步发送状态: " + syncResult.getSendStatus());

// 2. 异步发送 —— 立即返回，通过回调感知结果
producer.send(msg, new SendCallback() {
    @Override
    public void onSuccess(SendResult sendResult) {
        System.out.println("异步发送成功: " + sendResult.getMsgId());
    }
    @Override
    public void onException(Throwable e) {
        System.err.println("异步发送失败: " + e.getMessage());
        // 重试或告警
    }
});

// 3. 单向发送 —— 不等待响应，性能最高但可能丢消息
producer.sendOneway(msg);
```

**源码视角**：
- **同步发送**：`DefaultMQProducerImpl#sendSync` 内部通过 `CountDownLatch.await()` 阻塞，等待 Broker 响应
- **异步发送**：将发送请求封装为 Runnable 提交到异步线程池（默认有界队列容量 50,000），需注意堆积风险
- **单向发送**：仅执行 `channel.writeAndFlush()`，不注册回调，纯 Fire-and-Forget

## 二、发送重试机制

RocketMQ 内置两种重试模式：

```java
DefaultMQProducer producer = new DefaultMQProducer("ProducerGroup");
// 同步发送重试次数（默认2次，即最多3次尝试）
producer.setRetryTimesWhenSendFailed(2);
// 异步发送重试次数（默认2次）
producer.setRetryTimesWhenSendAsyncFailed(2);
// 开启Broker故障延迟规避，自动屏蔽慢节点
producer.setSendLatencyFaultEnable(true);
producer.start();
```

**重试核心逻辑**（`DefaultMQProducerImpl#sendDefaultImpl`）：
- 每次重试时 `selectOneMessageQueue` 会切换到不同 Broker，避免重复发到同一个故障节点
- 网络异常、Topic 不存在会触发重试；业务级错误（如权限不足）不触发
- **故障规避**：`MQFaultStrategy` 记录各 Broker 的响应延迟，将慢节点加入延迟规避表，后续重试跳过这些节点

## 三、批量发送

批量发送通过将多条消息合并为一次网络请求，显著降低网络开销：

```java
// 限制：同一 Topic、非事务/延迟消息、单批次 ≤ 4MB
List<Message> messages = Arrays.asList(
    new Message("BatchTopic", "TagA", "Msg1".getBytes()),
    new Message("BatchTopic", "TagA", "Msg2".getBytes()),
    new Message("BatchTopic", "TagA", "Msg3".getBytes())
);

// 超过4MB时使用消息拆分器
ListSplitter splitter = new ListSplitter(messages);
while (splitter.hasNext()) {
    producer.send(splitter.next());
}
```

**消息拆分器实现**：
```java
public class ListSplitter implements Iterator<List<Message>> {
    private static final int SIZE_LIMIT = 4 * 1024 * 1024; // 4MB
    private final List<Message> messages;
    private int currIndex;

    public ListSplitter(List<Message> messages) {
        this.messages = messages;
    }

    @Override
    public boolean hasNext() {
        return currIndex < messages.size();
    }

    @Override
    public List<Message> next() {
        int nextIndex = currIndex;
        int totalSize = 0;
        while (nextIndex < messages.size()) {
            Message msg = messages.get(nextIndex);
            // 估算消息大小（主题 + Body + 属性 + 固定开销）
            int size = msg.getTopic().length() + msg.getBody().length
                + msg.getProperties().toString().length() + 20;
            if (totalSize + size > SIZE_LIMIT) break;
            totalSize += size;
            nextIndex++;
        }
        List<Message> sub = messages.subList(currIndex, nextIndex);
        currIndex = nextIndex;
        return sub;
    }
}
```

## 四、Push vs Pull 消费模式

| 特性 | Push 模式 | Pull 模式 |
|------|-----------|-----------|
| 控制权 | Broker 主动推送 | 消费者主动拉取 |
| 实时性 | 高（基于长轮询） | 可控（取决于拉取频率） |
| 过载风险 | 消费者可能积压 | 消费者自主控制节奏 |
| 复杂度 | 低（API 封装完善） | 高（需手动管理位点） |
| 适用场景 | 绝大多数业务场景 | 流计算、批量处理等特殊场景 |

> **底层本质**：RocketMQ 的 Push 模式其实是 Pull 的封装。`DefaultMQPushConsumer` 内部通过 `PullMessageService` 线程不断地主动拉取消息，然后将消息投递给业务监听器，对外表现为 Push 语义。

```java
// Push 模式 —— 最常用
DefaultMQPushConsumer pushConsumer = new DefaultMQPushConsumer("ConsumerGroup");
pushConsumer.setNamesrvAddr("localhost:9876");
pushConsumer.subscribe("TopicTest", "*");
pushConsumer.registerMessageListener(new MessageListenerConcurrently() {
    @Override
    public ConsumeConcurrentlyStatus consumeMessage(List<MessageExt> msgs,
                                                     ConsumeConcurrentlyContext context) {
        for (MessageExt msg : msgs) {
            System.out.println("收到消息: " + new String(msg.getBody()));
        }
        return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
    }
});
pushConsumer.start();

// Pull 模式（LitePullConsumer）—— 消费者主动拉取
DefaultLitePullConsumer pullConsumer = new DefaultLitePullConsumer("PullGroup");
pullConsumer.setNamesrvAddr("localhost:9876");
pullConsumer.subscribe("TopicTest", "*");
pullConsumer.start();
// 主动拉取，自己控制消费节奏
List<MessageExt> msgs = pullConsumer.poll(Duration.ofSeconds(5));
msgs.forEach(msg -> System.out.println(new String(msg.getBody())));
pullConsumer.commitSync(); // 手动提交位点
```

## 五、并发消费 vs 顺序消费

### 5.1 并发消费

默认模式，使用线程池并发处理多条消息，吞吐量高：

```java
consumer.registerMessageListener(new MessageListenerConcurrently() {
    @Override
    public ConsumeConcurrentlyStatus consumeMessage(List<MessageExt> msgs,
                                                     ConsumeConcurrentlyContext context) {
        // 多线程并发处理，消息到达顺序不保证
        process(msgs);
        return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
    }
});
consumer.setConsumeThreadMin(16);
consumer.setConsumeThreadMax(64);
```

### 5.2 顺序消费

保证同一 MessageQueue 内的消息严格按 FIFO 顺序处理：

```java
// 生产者：使用 MessageQueueSelector 将同一订单的消息路由到同一队列
producer.send(msg, (mqs, message, arg) -> {
    long orderId = (long) arg;
    int index = (int)(orderId % mqs.size());
    return mqs.get(index);
}, orderId);

// 消费者：使用 MessageListenerOrderly
consumer.registerMessageListener(new MessageListenerOrderly() {
    @Override
    public ConsumeOrderlyStatus consumeMessage(List<MessageExt> msgs,
                                                ConsumeOrderlyContext context) {
        // 同一队列的消息串行执行，保证顺序
        for (MessageExt msg : msgs) {
            System.out.println("顺序消费: " + new String(msg.getBody()));
        }
        return ConsumeOrderlyStatus.SUCCESS;
    }
});
```

**顺序消费实现原理**：`ConsumeMessageOrderlyService` 为每个 `MessageQueue` 维护一把锁（`lockMap`），确保同一时刻只有一个线程处理该队列的消息。消息按位点顺序存入 `msgTreeMap`（TreeMap），严格保证 FIFO。

**注意事项**：
- 顺序消费遇到失败会阻塞队列，等待当前消息成功或超时才继续，吞吐量低于并发消费
- 需要确保发送侧的 `MessageQueueSelector` 和消费侧的 `MessageListenerOrderly` 配合使用

## 六、消费者负载均衡策略

RocketMQ 通过 `RebalanceService` 定时（默认每 20s）重新分配消费者与队列的映射关系：

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| **平均分配（默认）** | 连续分段分配，保证绝对均衡 | 通用场景 |
| **环形平均** | 轮询分配，适应队列数与消费者数差距大的情况 | 队列数远多于消费者数 |
| **一致性哈希** | 虚拟节点，最小化重平衡时的迁移量 | 频繁发布，需减少重分配影响 |
| **机房临近** | 优先分配同机房队列 | 跨机房部署 |

**平均分配核心逻辑**（`AllocateMessageQueueAveragely`）：
```java
public List<MessageQueue> allocate(String currentCID,
                                    List<MessageQueue> mqAll, List<String> cidAll) {
    int averageSize = mqAll.size() <= cidAll.size() ? 1 : mqAll.size() / cidAll.size();
    int index = cidAll.indexOf(currentCID);
    int startIndex = index * averageSize;
    int range = Math.min(averageSize, mqAll.size() - startIndex);
    return mqAll.subList(startIndex, startIndex + range);
}
```

## 七、集群消费 vs 广播消费

| 特性 | 集群消费（默认） | 广播消费 |
|------|-----------------|---------|
| 消息分发 | 组内每个实例消费部分消息 | 组内每个实例消费全部消息 |
| 进度存储 | Broker 端统一管理 | 消费者本地磁盘 |
| 重试机制 | 支持（最多16次） | 不支持，失败即失败 |
| 典型场景 | 业务解耦、订单处理 | 配置分发、本地缓存刷新 |

```java
// 广播消费配置
DefaultMQPushConsumer consumer = new DefaultMQPushConsumer("BroadcastGroup");
consumer.setMessageModel(MessageModel.BROADCASTING); // 默认是 CLUSTERING
consumer.subscribe("ConfigTopic", "*");
consumer.registerMessageListener(new MessageListenerConcurrently() {
    @Override
    public ConsumeConcurrentlyStatus consumeMessage(List<MessageExt> msgs,
                                                     ConsumeConcurrentlyContext context) {
        // 每个消费者实例都会收到这条消息
        refreshLocalCache(msgs);
        return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
    }
});
```

## 八、消费失败与重试

消费失败后消息进入重试 Topic（`%RETRY%ConsumerGroup`），按渐进式延迟重试（10s → 30s → 1min → 2min...），最多 **16 次**。超过次数进入死信队列 `%DLQ%ConsumerGroup`：

```java
consumer.registerMessageListener(new MessageListenerConcurrently() {
    @Override
    public ConsumeConcurrentlyStatus consumeMessage(List<MessageExt> msgs,
                                                     ConsumeConcurrentlyContext context) {
        for (MessageExt msg : msgs) {
            System.out.println("当前已重试次数: " + msg.getReconsumeTimes());
            try {
                process(msg);
                return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
            } catch (Exception e) {
                if (msg.getReconsumeTimes() >= 3) {
                    // 重试超过3次，记录日志，跳过（避免无限重试影响正常消息）
                    log.error("消息消费超过3次仍失败，进行人工干预: {}", msg.getMsgId());
                    return ConsumeConcurrentlyStatus.CONSUME_SUCCESS; // 标记成功，跳过
                }
                return ConsumeConcurrentlyStatus.RECONSUME_LATER; // 触发重试
            }
        }
        return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
    }
});
```

**死信队列监控**：
```bash
# 消费死信队列中未处理的消息
mqadmin consumeMessage -n localhost:9876 -t %DLQ%YourConsumerGroup -g YourConsumerGroup
```

## 九、消费进度（Offset）管理

| 消费模式 | 进度存储位置 | 特性 |
|---------|------------|------|
