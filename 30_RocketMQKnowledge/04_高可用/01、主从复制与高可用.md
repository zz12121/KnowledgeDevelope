# 主从复制、刷盘与高可用

## 一、刷盘机制

刷盘是指 Broker 将内存中的消息持久化到磁盘的过程，直接影响消息的可靠性和写入性能。

### 1.1 同步刷盘 vs 异步刷盘

| 维度 | 同步刷盘（SYNC_FLUSH） | 异步刷盘（ASYNC_FLUSH） |
|------|----------------------|----------------------|
| 机制 | 写入内存后立即阻塞等待刷盘完成再返回 | 写入内存后立即返回，后台线程定时批量刷盘 |
| 可靠性 | 高（数据落盘才确认） | 较低（宕机可能丢失内存中未刷盘的消息） |
| 吞吐量 | 低（每次写操作需等待磁盘 I/O） | 高（批量刷盘减少 I/O 次数） |
| 延迟 | 高（RT 增加） | 低 |
| 适用场景 | 金融交易、计费系统等高可靠场景 | 日志收集、监控数据等高吞吐场景 |

```properties
# broker.conf
flushDiskType=SYNC_FLUSH    # 同步刷盘
# flushDiskType=ASYNC_FLUSH  # 异步刷盘（默认）
```

### 1.2 同步刷盘源码原理

同步刷盘通过 `GroupCommitService` 处理：

```java
// GroupCommitService 核心逻辑
public void run() {
    while (!this.isStopped()) {
        this.waitForRunning(10);  // 等待最多10ms，收集一批请求
        this.doCommit();           // 批量刷盘
    }
}

private void doCommit() {
    // 交换读写队列，避免锁竞争
    swapRequests();
    // 遍历所有待刷盘的请求
    for (GroupCommitRequest req : this.requestsRead) {
        // 检查是否已经刷盘到该位置
        boolean flushOK = mappedFileQueue.getFlushedWhere() >= req.getNextOffset();
        if (!flushOK) {
            // 执行刷盘
            mappedFileQueue.flush(0);
            flushOK = mappedFileQueue.getFlushedWhere() >= req.getNextOffset();
        }
        // 通知等待中的发送线程
        req.wakeupCustomer(flushOK ? PutMessageStatus.PUT_OK : PutMessageStatus.FLUSH_DISK_TIMEOUT);
    }
}
```

### 1.3 异步刷盘与 TransientStorePool

开启 `transientStorePoolEnable=true` 时，写入先到**堆外内存（DirectByteBuffer）**，再由 `CommitRealTimeService` 异步提交到 PageCache，`FlushRealTimeService` 再从 PageCache 刷到磁盘，实现读写分离：

```
Producer 写入
    ↓
堆外内存（TransientStorePool）← 写入
    ↓ CommitRealTimeService（定期提交，默认500ms）
Page Cache
    ↓ FlushRealTimeService（定期刷盘）
磁盘
```

**优势**：缓解 PageCache 的写压力，避免 `broker busy` 问题；代价是多了一层内存拷贝。

## 二、主从复制机制

### 2.1 同步复制 vs 异步复制

| 维度 | 同步复制（SYNC_MASTER） | 异步复制（ASYNC_MASTER） |
|------|----------------------|----------------------|
| 数据一致性 | 强一致（主从均写入成功才返回） | 最终一致（主节点写入即返回） |
| 性能 | 较低（RT 增加，需等待从节点 ACK） | 高（无等待从节点的开销） |
| 故障恢复 | 主节点宕机后从节点数据完整 | 主节点宕机可能丢失未同步数据 |
| 适用场景 | 金融级高可靠场景 | 普通业务，高性能优先 |

```properties
# broker.conf
brokerRole=SYNC_MASTER   # 同步复制（高可靠）
# brokerRole=ASYNC_MASTER  # 异步复制（高性能，默认）
# brokerRole=SLAVE         # 从节点配置
```

**同步复制原理**：主节点通过 `HAConnection` 维护与从节点的长连接，收到消息后等待从节点返回 ACK 才响应生产者。超时未收到 ACK 则返回 `FLUSH_SLAVE_TIMEOUT` 状态。

### 2.2 主从架构高可用配置

```properties
# Master Broker (broker-a)
brokerClusterName=DefaultCluster
brokerName=broker-a
brokerId=0                    # 0 表示 Master
brokerRole=SYNC_MASTER
flushDiskType=ASYNC_FLUSH
storePathRootDir=/data/rocketmq/broker-a/store
namesrvAddr=namesrv1:9876;namesrv2:9876

# Slave Broker (broker-a-s)
brokerClusterName=DefaultCluster
brokerName=broker-a           # 与 Master 同名
brokerId=1                    # 非0 表示 Slave
brokerRole=SLAVE
flushDiskType=ASYNC_FLUSH
storePathRootDir=/data/rocketmq/broker-a-s/store
namesrvAddr=namesrv1:9876;namesrv2:9876
```

## 三、DLedger 模式（Raft 自动选主）

### 3.1 传统主从的痛点

传统主从架构的主要问题：**主节点宕机后需要人工干预才能切换到从节点**，存在停机时间，异步复制模式下还可能丢数据。

### 3.2 DLedger 解决方案

DLedger 基于 Raft 一致性协议，实现**自动故障转移 + 强一致数据复制**：

| 对比维度 | 主从模式 | DLedger 模式 |
|---------|---------|-------------|
| 故障转移 | 手动切换，停机时间长 | 自动选主，秒级恢复 |
| 数据一致性 | 异步复制，可能丢数据 | Raft 强一致，数据安全 |
| 架构复杂度 | 简单，易于部署 | 相对复杂，至少需要 3 节点 |
| 性能 | 高（无协调开销） | 较低（需日志复制和多数派仲裁） |
| 适用场景 | 可接受短暂停服 | 金融级场景，高可用+强一致 |

**DLedger 选举流程**：
1. 所有节点启动时状态为 `Follower`
2. `Follower` 在随机超时（150-300ms）内未收到 Leader 心跳，升为 `Candidate`，发起投票
3. `Candidate` 获得**超过半数**节点同意，升为 `Leader`
4. `Leader` 负责接收写请求，转发日志到 `Follower`，收到多数派 ACK 后提交

```properties
# DLedger 模式 broker.conf（至少部署3个节点）
enableDLegerCommitLog=true
dLegerGroup=RaftBrokerGroup
dLegerPeers=n0-127.0.0.1:40911;n1-127.0.0.1:40912;n2-127.0.0.1:40913
# 当前节点 ID
dLegerSelfId=n0
```

### 3.3 Broker 启动时的选择

```java
// Broker 启动时根据配置决定使用哪种 CommitLog 实现
if (messageStoreConfig.isEnableDLegerCommitLog()) {
    this.commitLog = new DLedgerCommitLog(this);  // Raft 强一致版本
} else {
    this.commitLog = new CommitLog(this);           // 传统主从版本
}
```

## 四、存储层零拷贝优化

### 4.1 mmap（内存映射文件）

mmap 是 RocketMQ 提升 CommitLog 读写性能的核心技术，通过 `FileChannel.map()` 将磁盘文件直接映射到进程虚拟内存：

```
传统 I/O 路径（4 次拷贝）：
磁盘 → 内核缓冲区 → 用户缓冲区 → 内核缓冲区 → 网络

mmap 路径（2 次拷贝）：
磁盘 ← → 内核缓冲区/Page Cache（与用户空间共享，省去一次拷贝）
```

**关键源码**（`MappedFile` 类）：
```java
// 将文件映射到内存
this.mappedByteBuffer = this.fileChannel.map(
    MapMode.READ_WRITE, 0, fileSize);

// 直接通过内存指针写入，无需系统调用
mappedByteBuffer.put(data);
```

**文件预热**（避免缺页中断）：
```java
public void warmMappedFile(FlushDiskType type, int pages) {
    // 用0填充，触发物理内存分配
    for (int i = 0, j = 0; i < fileSize; i += MappedFile.OS_PAGE_SIZE, j++) {
        byteBuffer.put(i, (byte) 0);
    }
    // 锁定内存，防止被交换到 SWAP
    LibC.INSTANCE.mlock(pointer, new NativeLong(fileSize));
    // 建议操作系统预加载文件数据
    LibC.INSTANCE.madvise(pointer, new NativeLong(fileSize), LibC.MADV_WILLNEED);
}
```

### 4.2 sendfile（零拷贝网络传输）

Consumer 消费消息时，Broker 通过 `sendfile` 将消息从 PageCache 直接传输到网络 Socket，跳过用户空间：

```
sendfile 路径（2 次拷贝，0 次上下文切换）：
磁盘/PageCache → 网络协议栈 → 网卡

源码实现（FileChannel.transferTo）：
fileChannel.transferTo(position, count, socketChannel);
// 底层调用 Linux sendfile64 系统调用
```

| 技术 | 拷贝次数 | 适用场景 |
|------|---------|---------|
| 传统 I/O | 4次 | — |
| mmap | 2次 | CommitLog/ConsumeQueue 读写 |
| sendfile | 2次（无用户态拷贝） | Consumer 消费消息时的网络传输 |

## 五、NameServer 高可用

### 5.1 NameServer 无状态设计原理

NameServer 节点之间**不互相通信**，不存在主从关系。每个 NameServer 维护完整的路由信息，通过 Broker 的心跳保持更新：

- **Broker → NameServer**：每隔 30s 发送心跳，注册路由信息
- **NameServer**：120s 未收到心跳，剔除该 Broker
- **Producer/Consumer**：每隔 30s 从 NameServer 拉取最新路由，本地缓存有效期 15s

**优势**：无主从协调开销，任意节点挂掉不影响整体（无单点），易于水平扩展。

### 5.2 NameServer 宕机影响

| 场景 | 影响 |
|------|------|
| 部分 NameServer 宕机 | 客户端自动切换到存活节点，无感知 |
| 全部 NameServer 宕机 | 已缓存路由的客户端短时间内仍可正常收发消息；新客户端无法启动 |
| Broker 挂掉 | NameServer 120s 后剔除，期间客户端可能仍路由到该 Broker |

```java
// 生产环境配置多个 NameServer 地址
producer.setNamesrvAddr("namesrv1:9876;namesrv2:9876;namesrv3:9876");
```

## 六、监控与运维

### 6.1 关键监控指标

| 指标 | 说明 | 告警阈值（参考） |
|------|------|----------------|
| 消息堆积量（Lag） | 未消费消息数 | 持续增长需关注 |
| TPS（发送/消费） | 每秒消息吞吐量 | 低于预期触发告警 |
| CommitLog 磁盘使用率 | 存储空间占用 | > 70% 告警 |
| Master-Slave 同步延迟 | 主从数据差值 | > 1000 条告警 |
| Broker RT | 消息发送响应时间 | > 500ms 告警 |

### 6.2 常用运维命令

```bash
# 查看集群状态
mqadmin clusterList -n localhost:9876

# 查看消费进度和堆积量
mqadmin consumerProgress -n localhost:9876 -g YourConsumerGroup

# 查看 Topic 路由信息
mqadmin topicRoute -n localhost:9876 -t YourTopic

# 查看 Broker 统计信息
mqadmin brokerStats -n localhost:9876 -b brokerAddr:10911

# 重置消费位点（处理堆积）
mqadmin resetOffsetByTime -n localhost:9876 -g YourConsumerGroup \
  -t YourTopic -s "2026-01-01#00:00:00:000"

# 手动触发消费死信队列中的消息
mqadmin consumeMessage -n localhost:9876 -t %DLQ%YourConsumerGroup \
  -g YourConsumerGroup -b brokerAddr:10911
```

## 七、最佳实践

### 7.1 可靠性最高配置（金融场景）

```properties
# broker.conf —— 金融级高可靠配置
flushDiskType=SYNC_FLUSH        # 同步刷盘
brokerRole=SYNC_MASTER          # 同步复制
# 或使用 DLedger 模式（推荐）：
enableDLegerCommitLog=true
```

### 7.2 高性能配置（日志场景）

```properties
# broker.conf —— 高吞吐配置
flushDiskType=ASYNC_FLUSH       # 异步刷盘
brokerRole=ASYNC_MASTER         # 异步复制
transientStorePoolEnable=true   # 堆外内存缓冲，缓解 PageCache 压力
diskMaxUsedSpaceRatio=90        # 适当提高磁盘利用率
```

# Broker JVM 配置（根据物理内存调整）
