## 一、文档模型与 BSON

### 1.1 MongoDB 文档结构

MongoDB 以 **BSON（Binary JSON）** 格式存储数据，每个文档是一个灵活的键值对集合：

```json
{
    "_id": ObjectId("65a1b2c3d4e5f6789012"),
    "name": "张三",
    "age": 28,
    "status": "active",
    "address": {
        "city": "北京",
        "district": "朝阳区",
        "coordinates": [116.4, 39.9]
    },
    "skills": ["Java", "MongoDB", "Redis"],
    "createdAt": ISODate("2024-01-01T00:00:00Z"),
    "score": NumberDecimal("99.95")
}
```

### 1.2 BSON vs JSON

| 维度 | JSON | BSON |
|------|------|------|
| 编码 | 文本（UTF-8） | **二进制** |
| 解析速度 | 较慢（文本扫描） | **极快**（二进制，有长度前缀可跳过字段） |
| 数据类型 | 6种（string/number/bool/null/array/object） | **丰富**（+Date/ObjectId/Binary/Decimal128/Int32/Int64） |
| 文件大小 | 纯文本，无额外字节 | 稍大（类型标记+长度前缀） |
| 快速字段跳过 | ❌（需逐字符扫描） | ✅（长度前缀直接跳过） |

### 1.3 ObjectId 结构

ObjectId 是 MongoDB 自动生成的 12 字节主键：
```
65a1b2c3  d4e5f6  7890  12
─────────  ──────  ────  ──
4字节       3字节   2字节  3字节
Unix时间戳  机器ID  进程ID  随机计数器
（按时间单调递增）
```

**特点**：全局唯一、包含创建时间（`ObjectId.getTimestamp()`）、单调递增（同一秒内随机）

## 二、Schema 设计模式

### 2.1 嵌入 vs 引用决策矩阵

| 关系类型 | 数据大小 | 访问频率 | 推荐方式 |
|---------|---------|---------|---------|
| 一对一 | 任意 | 总是一起访问 | **嵌入** |
| 一对少（< 100） | 小 | 一起访问 | **嵌入数组** |
| 一对多（100-1000） | 中 | 分开访问 | **引用** |
| 一对超多（> 1000） | 大 | 分开访问 | **引用**（必须） |
| 多对多 | 任意 | 视情况 | 双方存对方 _id 数组 |

### 2.2 常用 Schema 设计模式

**预计算模式（Computed Pattern）**：
```javascript
// 不好的做法：每次查询都要 COUNT
db.articles.aggregate([{ $group: {_id: "$articleId"} }, { $lookup: {...} }])

// 好的做法：预计算存储
{ _id: "article001", title: "...", commentCount: 128, likeCount: 350 }
// 新增评论时原子递增
db.articles.updateOne({ _id: "article001" }, { $inc: { commentCount: 1 } })
```

**桶模式（Bucket Pattern）** — 适合时序数据：
```javascript
// 不好：每条传感器数据一个文档（文档数极多，索引大）
{ sensorId: "S1", timestamp: T1, value: 25.6 }
{ sensorId: "S1", timestamp: T2, value: 25.8 }

// 好：按时间桶聚合（每小时一个文档）
{
    sensorId: "S1",
    hour: ISODate("2024-01-01T10:00:00Z"),
    count: 3600,
    sum: 92160,
    min: 24.5,
    max: 26.1,
    readings: [25.6, 25.8, ...]  // 或用 $push 积累
}
```

**多态模式（Polymorphic Pattern）** — MongoDB 最常用：
```javascript
// 不同类型产品，字段各不相同，存在同一集合
{ type: "phone",   brand: "Apple",  model: "iPhone 15", storage: "256GB", color: "黑" }
{ type: "clothes", brand: "Nike",   size: "XL", material: "棉" }
{ type: "book",    title: "MongoDB权威指南", author: "...", isbn: "..." }
```

## 三、事务与原子性

### 3.1 操作原子性层次

| 级别 | 原子性保证 | 示例 |
|------|-----------|------|
| 单字段更新 | ✅ 原子 | `$set` / `$inc` |
| 单文档操作 | ✅ 原子（天然） | 一次 `updateOne` 对单个文档的多字段修改 |
| 多文档操作（4.0+） | ✅ 原子（需显式事务） | 跨多个文档的转账操作 |
| 跨分片事务（4.2+） | ✅ 原子（性能损耗大） | 分片集群中的跨 Shard 事务 |

### 3.2 单文档原子操作符

```javascript
// 原子更新操作符（单文档内天然原子，无需事务）
$set        // 设置字段：{ $set: { status: "active", updatedAt: new Date() } }
$unset      // 删除字段：{ $unset: { tempField: "" } }
$inc        // 原子递增：{ $inc: { views: 1, version: 1 } }
$push       // 向数组追加：{ $push: { tags: "mongodb" } }
$addToSet   // 追加唯一元素：{ $addToSet: { subscribers: userId } }
$pull       // 移除数组元素：{ $pull: { tags: "outdated" } }
$pop        // 移除首/尾元素：{ $pop: { history: -1 } }（-1=首，1=尾）
$currentDate // 设为当前时间：{ $currentDate: { updatedAt: true } }

// findOneAndUpdate：原子"查找-修改-返回"
db.inventory.findOneAndUpdate(
    { sku: "apple", qty: { $gte: 10 } },   // 条件（库存充足才操作）
    { $inc: { qty: -10 } },                 // 扣减库存
    { returnDocument: "after", upsert: false }  // 返回更新后的文档
)
```

## 四、数据对比与选型

### 4.1 MongoDB vs MySQL 一句话总结

| 场景 | 选择 | 理由 |
|------|------|------|
| 内容管理（CMS/博客） | MongoDB | 不同类型内容字段各异 |
| 用户画像/行为数据 | MongoDB | 动态字段，嵌套结构 |
| 高并发写入（日志/事件） | MongoDB | 吞吐高，WiredTiger 压缩节省空间 |
| 附近的人/位置服务 | MongoDB | 原生 GeoJSON + 2dsphere 索引 |
| 金融转账/支付 | MySQL | 强 ACID 事务，行业标准 |
| 复杂报表（多表 JOIN） | MySQL/数仓 | SQL 灵活，关联查询强 |
| 用户账号+权限 | MySQL | 结构固定，关系清晰 |

### 4.2 MongoDB 常见误区

1. **误区：MongoDB 无 Schema 就可以随便存**  
   实际：乱 Schema 会导致查询无法优化（无法建索引），生产环境应通过 Schema Validation 约束

2. **误区：MongoDB 适合所有 NoSQL 场景**  
   实际：缓存用 Redis，超大规模时序用 InfluxDB/TDengine，MongoDB 是通用文档数据库

3. **误区：MongoDB 事务不可靠**  
   实际：4.0+ 多文档 ACID 事务和 MySQL 一样可靠，只是事务开销更大，不应作为默认编程模式
