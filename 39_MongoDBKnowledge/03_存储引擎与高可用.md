## 一、WiredTiger 存储引擎

### 核心特性

| 特性 | 说明 |
|------|------|
| 并发控制 | MVCC + 文档级锁，读写基本不阻塞 |
| 数据压缩 | Snappy（默认）/ zlib / zstd，减少 60-80% 存储 |
| Write Ahead Log | Journal（每 100ms fsync），崩溃恢复保障 |
| 内存管理 | WiredTiger Cache（默认 50% RAM - 1GB），自管理 |
| Checkpoint | 每 60s 将内存脏页写入数据文件 |

### 崩溃恢复流程

```
启动时：
1. 找最近的 Checkpoint（完整快照）
2. 从 Checkpoint 位置向后回放 Journal WAL 日志
3. 数据库恢复到崩溃前状态（最多丢失 100ms 的写操作）
```

## 二、副本集（Replica Set）

### 架构

```
Primary  ──Oplog同步──→  Secondary 1  （可提供只读）
   │     ──Oplog同步──→  Secondary 2  （参与选举）
   │                           ↑
   └─────────────  Arbiter（只投票，不存数据）
```

### 选举规则（Raft 变体）

1. Primary 心跳 10s 内无响应 → 触发选举
2. 候选节点发起投票请求
3. 优先选 **Oplog 最新**（lastOptime 最大）的节点
4. 需要 **多数派**（n/2 + 1）投票才能成为 Primary
5. 选举期间（10-30s）写操作不可用

### writeConcern 与 readConcern

```javascript
// 写入配置
{ w: "majority" }    // 多数派确认后才返回（最安全）
{ w: 1 }             // Primary 确认即返回（默认）
{ j: true }          // 写入 Journal 后才返回

// 读取配置
readConcern: "local"     // 读本地节点（默认，可能读到未复制的数据）
readConcern: "majority"  // 多数派确认的数据（强一致性）
readConcern: "snapshot"  // 事务用，一致性快照

// readPreference：读路由
primary            // 只读 Primary（默认）
primaryPreferred   // 优先 Primary，故障时读 Secondary
secondary          // 只读 Secondary（读写分离）
nearest            // 读延迟最低的节点
```

## 三、分片集群（Sharding）

### 架构组件

| 组件 | 数量 | 作用 |
|------|------|------|
| mongos | 多个（负载均衡） | 查询路由，对应用透明 |
| Config Server | 1个副本集（3节点） | 存储元数据（chunk → Shard 映射） |
| Shard | N个副本集 | 实际数据存储（每 Shard 是一个副本集） |

### 分片策略选择

| 场景 | 推荐策略 | 理由 |
|------|---------|------|
| 随机读写，需要均衡分布 | 哈希分片 | 写入自动均匀，无热点 |
| 需要范围查询 | 范围分片 | 相邻数据在同一 Shard，减少广播 |
| 数据本地化（地理合规） | Zone 分片 | 中国数据在中国 Shard |

### 好的片键 vs 坏的片键

```
✅ 好的片键：userId（哈希）— 高基数、均匀分布
✅ 好的片键：{region, userId}（复合）— 支持地区维度聚合

❌ 坏的片键：status（低基数）— 只有几个值，数据分布不均
❌ 坏的片键：createdAt（范围，单调递增）— 所有新数据都写最后一个 Shard（热点）
```

## 四、Oplog 与 Change Streams

### Oplog

- 存储位置：`local.oplog.rs`（固定大小 Capped Collection）
- 记录内容：所有写操作（幂等形式）
- Secondary 从 Oplog 同步数据
- Oplog 窗口期：决定 Secondary 最多可以落后 Primary 多久

### Change Streams（建立在 Oplog 上）

```javascript
// 监听集合变更
const stream = db.orders.watch([
    { $match: { operationType: { $in: ["insert", "update"] } } }
]);

stream.on("change", (event) => {
    // event.operationType: insert/update/delete/replace
    // event.fullDocument: 变更后的完整文档
    // event.resumeToken: 断点续传令牌
});
```

**应用场景**：实时同步到 Elasticsearch/Kafka/缓存失效/审计日志
