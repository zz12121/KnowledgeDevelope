# 线程模型与内存管理

## 1. EventLoop 线程模型详解

**EventLoop 的本质**：一个无限循环的单线程执行器，绑定一个 `Selector`，负责：

1. `select()`：等待 I/O 事件（可读/可写/连接建立）
2. `processSelectedKeys()`：处理就绪的 I/O 事件
3. `runAllTasks()`：执行任务队列中的异步任务和定时任务

**一个连接终身只属于一个 EventLoop（线程）：**

```java
// Channel 注册时绑定 EventLoop
channel.register(eventLoop, ops);
// 之后所有事件都在 eventLoop 对应的线程中处理
```

这个"终身绑定"的设计消除了 Channel 上的线程竞争，Handler 中的代码不需要任何同步措施（前提是不阻塞）。

**任务提交方式：**

非 EventLoop 线程需要操作 Channel 时，通过 `eventLoop.execute()` 提交任务：

```java
// 正确方式：提交到 EventLoop 线程执行
channel.eventLoop().execute(() -> {
    channel.writeAndFlush(msg);
});
```

如果当前已经是 EventLoop 线程（`eventLoop.inEventLoop()`），则直接执行，避免不必要的任务提交开销。

## 2. MpscQueue（无锁多生产者单消费者队列）

Netty 内部使用 `MpscQueue`（Multiple Producer Single Consumer Queue）作为 EventLoop 的任务队列，替代 `LinkedBlockingQueue`。

**为什么换掉 LinkedBlockingQueue：**

`LinkedBlockingQueue` 使用 `ReentrantLock` 保护队列操作，高并发下锁竞争严重。

**MpscQueue 的优化：**

基于 JCTools 库，利用 CAS + 无锁算法实现：
- 多个生产者（各 Netty 应用线程）并发入队：CAS 操作，无锁
- 单个消费者（EventLoop 线程）出队：完全无锁，直接操作指针

在生产者远多于消费者的场景（这正是 EventLoop 的工作模式）下性能显著优于有锁队列。

## 3. 阻塞操作的隔离方案

**在 EventLoop 线程中绝对禁止：**

- 同步数据库操作
- 同步 RPC 调用
- `Thread.sleep()`
- 任何可能阻塞的 I/O 操作

一旦 EventLoop 线程阻塞，这个线程负责的所有 Channel 都会停止响应，影响面极大。

**正确的隔离方案：**

**方案一：addLast 时指定业务线程池**

```java
EventExecutorGroup businessGroup = new DefaultEventExecutorGroup(16);

pipeline.addLast(new ProtocolDecoder());         // EventLoop 线程执行
pipeline.addLast(businessGroup, new BusinessHandler()); // 提交到业务线程池
```

**方案二：在 Handler 中手动提交**

```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    businessExecutor.submit(() -> {
        // 耗时操作在这里
        Object result = doSlowWork(msg);
        // 结果写回必须在 EventLoop 线程中执行
        ctx.channel().eventLoop().execute(() -> {
            ctx.writeAndFlush(result);
        });
    });
}
```

## 4. 内存管理：PoolArena 详解

**PoolArena 的职责**：管理一组 PoolChunk，对外提供线程安全的内存分配接口。

**多 Arena 设计**：默认 Arena 数量 = min(CPU核心数 × 2, workerGroup 线程数)，多个 Arena 分摊锁竞争，避免所有线程竞争同一把锁。

**线程 → Arena 的绑定**：每个线程第一次分配内存时，选择引用计数最少的 Arena 绑定，后续一直使用这个 Arena。

**三种大小的分配策略：**

```
size <= 512B   → PoolSubpage（位图管理，同规格的内存块共用一个 Subpage）
512B < size <= 8MB → PoolChunk（伙伴算法，二叉树管理）
size > 8MB     → 直接向 JVM 申请，不池化（Huge 分配）
```

**PoolSubpage 位图原理（以 16B 规格为例）：**

一个 PoolSubpage 是 8KB 的内存块，切成 8KB / 16B = 512 个 16B 的小块，用一个 long[] 位图标记每个小块是否已分配。分配时找第一个为 0 的 bit 位，返回对应内存；释放时将 bit 位重置为 0。

## 5. 内存泄漏检测（ResourceLeakDetector）

**开启方式：**

```bash
-Dio.netty.leakDetection.level=PARANOID    # 测试：每次分配都跟踪
-Dio.netty.leakDetection.level=ADVANCED    # 生产：采样（1/128）检测
-Dio.netty.leakDetection.level=SIMPLE      # 默认：采样检测，信息较少
-Dio.netty.leakDetection.level=DISABLED    # 关闭
```

**工作原理**：

在分配 ByteBuf 时，创建一个 `ResourceLeak` 对象，通过 `PhantomReference`（虚引用）跟踪 ByteBuf 的 GC。如果 ByteBuf 被 GC 了但还没被 `release()`，说明发生泄漏，会打印出访问堆栈（调用了 `record()`的位置），精确定位泄漏代码。

**PARANOID 级别**：每次访问都调用 `record()`，泄漏时有完整的访问历史，信息最全但开销最大。

## 6. 引用计数规则总结

| 场景 | 谁负责 release |
|---|---|
| 继承 `SimpleChannelInboundHandler` | 父类自动 release |
| 自定义入站 Handler，不转发 msg | 你负责 release |
| 自定义入站 Handler，`fireChannelRead(msg)` 传递 | 下游 Handler 负责 |
| 出站消息（write/writeAndFlush） | Netty 在 flush 完成后自动 release |
| 编码器中传入的 msg | Netty 自动 release（在 encode 结束后） |
| 手动创建的 ByteBuf，不加入 out 列表 | 你负责 release |
