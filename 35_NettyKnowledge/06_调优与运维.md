# 调优与运维

## 1. 关键配置参数速查表

| 参数 | ChannelOption | 推荐值 | 说明 |
|---|---|---|---|
| 全连接队列大小 | `SO_BACKLOG` | 1024（同步调整 somaxconn） | 高并发必须调大 |
| 禁用 Nagle 算法 | `TCP_NODELAY` | `true` | 低延迟服务必须开启 |
| TCP 层心跳保活 | `SO_KEEPALIVE` | `true`（配合应用层心跳） | 只是保底兜底 |
| 接收/发送缓冲区 | `SO_RCVBUF` / `SO_SNDBUF` | -1（系统自动调优） | 跨机房高带宽才手动设 |
| 端口复用 | `SO_REUSEADDR` | `true` | 重启服务避免端口占用 |
| ByteBuf 分配器 | `ALLOCATOR` | `PooledByteBufAllocator.DEFAULT` | 生产环境必须池化 |
| 写缓冲水位线 | `WRITE_BUFFER_WATER_MARK` | 32KB low / 64KB high | 背压控制 |
| 连接超时 | `CONNECT_TIMEOUT_MILLIS` | 3000-5000ms | 客户端连接超时 |

## 2. 线程池大小推荐

**BossGroup**：通常设 **1 个线程**，只有绑定多个端口才增加。

**WorkerGroup**：

```
默认：CPU核心数 × 2
纯 I/O 密集型（代理、转发）：CPU × 2 ~ CPU × 3
计算密集型（复杂编解码）：CPU × 1 ~ CPU × 2
```

**业务线程池（DefaultEventExecutorGroup）**：

```
阻塞 I/O 操作（DB 查询、HTTP 调用）：CPU × (1 + 等待时间 / 计算时间)
通常从 CPU × 2 开始，根据监控调整
```

**核心原则：理论是参考，压测是验证。**

监控目标：达到目标 QPS 时，CPU 在 70%-80%，EventLoop 任务队列无积压。

## 3. 性能调优清单

**必做（基础）：**

- [ ] `PooledByteBufAllocator.DEFAULT` 作为分配器
- [ ] `TCP_NODELAY=true`（低延迟服务）
- [ ] `SO_BACKLOG` 调大 + somaxconn 同步调整
- [ ] 业务 Handler 中无阻塞操作（严格隔离 I/O 线程）

**进阶：**

- [ ] 大文件传输用 `DefaultFileRegion`（零拷贝）
- [ ] 多段消息合并用 `CompositeByteBuf`（避免复制）
- [ ] 高并发写入场景：批量 `write()` + 最后一次 `flush()`
- [ ] 使用 `OpenSslProvider` 替代 JDK SSL 实现（SSL 加解密性能提升 2-3 倍）
- [ ] 调整 `AdaptiveRecvByteBufAllocator` 的初始/最小/最大值

**内存相关：**

- [ ] `-XX:MaxDirectMemorySize` 设置合理（推荐 = `SO_RCVBUF × 连接数 × 2` 估算）
- [ ] `-XX:+DisableExplicitGC` 禁止 `System.gc()` 触发 Full GC
- [ ] 开启内存泄漏检测（测试阶段：PARANOID，生产：ADVANCED）

## 4. 常见故障排查

**故障一：CPU 100%（空轮询 Bug）**

症状：服务无任何流量，但 CPU 飙满，日志出现大量空行。

排查：jstack 看 EventLoop 线程，是否在 `select()` 处无限循环而没有 I/O 事件。

修复：Netty 4.x 已有自动修复（空轮询 512 次重建 Selector），升级到最新 Netty 4.1.x 版本。

---

**故障二：OOM（直接内存耗尽）**

症状：`OutOfMemoryError: Direct buffer memory`

排查步骤：

1. 开启 `-Dio.netty.leakDetection.level=PARANOID`，看是否有 ByteBuf 泄漏日志
2. 检查 `ChannelOutboundBuffer` 是否积压（对方读取太慢 + 写入太快）
3. 检查是否有 Handler 保存了 ByteBuf 引用但没有 release

常见泄漏场景：
- 自定义 Handler 中没有调用 `ctx.fireChannelRead(msg)` 传递，也没有手动 `release()`
- 编解码器中创建了 ByteBuf 但没有加入 `out` 列表

---

**故障三：消息乱序或丢失**

症状：客户端收到的消息与发送顺序不一致，或部分消息丢失。

排查：

- 检查是否在非 EventLoop 线程直接调用了 `channel.write()`（线程安全问题）
- 检查 Pipeline 中某个 Handler 是否忘记调用 `ctx.fireChannelRead(msg)`（事件传播中断）
- 检查是否有多线程并发调用 `writeAndFlush`（应通过 `channel.eventLoop().execute()` 提交）

---

**故障四：连接大量 CLOSE_WAIT**

症状：`ss -s` 或 `netstat` 看到大量 CLOSE_WAIT 连接。

原因：对端发了 FIN，本端收到后没有主动关闭 Channel，导致连接停留在 CLOSE_WAIT 状态。

修复：在 `channelInactive` 或 `exceptionCaught` 中确保调用了 `ctx.close()`。

## 5. 监控指标关注点

**连接层：**

- 当前活跃连接数（`channelActive`/`channelInactive` 计数）
- 新建连接速率（突增可能是攻击或流量波动）
- CLOSE_WAIT / TIME_WAIT 连接数（内核 `ss -s`）

**流量层：**

- 每秒读/写字节数（`ChannelTrafficShapingHandler` 内置统计）
- `ChannelOutboundBuffer` 待发送字节数（积压预警）

**延迟层：**

- 请求处理耗时（P50/P99/P999）
- EventLoop 任务队列积压任务数

**内存层：**

- 直接内存使用量（`BufferPoolMXBean` 或 `PooledByteBufAllocator.metric()`）
- 堆内存 GC 频率与停顿时间

## 6. 测试与调试工具

**EmbeddedChannel**：单元测试 Handler 和编解码器的核心工具，在内存中模拟 Channel 环境，无需真实网络。

**LoggingHandler**：开发/测试环境添加到 Pipeline，实时打印数据流和事件，HEX_DUMP 格式可直观验证协议格式。**生产环境务必移除。**

**ResourceLeakDetector**：ByteBuf 内存泄漏检测，PARANOID 级别有完整调用栈。

**Wireshark / tcpdump**：网络层抓包，验证协议格式是否正确，判断问题是应用层还是网络层。

## 7. 优雅关闭流程

```java
// 1. 停止接受新连接
serverChannel.close().sync();

// 2. 关闭所有已建立的连接
allChannels.close().sync(); // ChannelGroup 批量关闭

// 3. 释放线程资源（quietPeriod 内无任务提交才真正关闭）
bossGroup.shutdownGracefully(100, 3000, TimeUnit.MILLISECONDS);
workerGroup.shutdownGracefully(100, 3000, TimeUnit.MILLISECONDS);
```

`shutdownGracefully(quietPeriod, timeout, unit)` 参数：
- `quietPeriod`：静默期，这段时间内没有新任务提交才认为可以关闭
- `timeout`：最长等待时间，超过后强制关闭
