# 协议与实战应用

## 1. TCP 粘包/拆包与四种解码器

TCP 是面向字节流的协议，应用层的"消息"边界在 TCP 层是不可见的，因此会出现粘包（多个包粘在一起）和拆包（一个包被拆成多个）问题。

**Netty 内置的四种解码器：**

| 解码器 | 适用场景 | 缺点 |
|---|---|---|
| `FixedLengthFrameDecoder` | 每条消息固定长度 | 不灵活，消息必须等长 |
| `LineBasedFrameDecoder` | 消息以 `\n` 或 `\r\n` 结尾 | 消息内容不能含换行符 |
| `DelimiterBasedFrameDecoder` | 自定义分隔符 | 消息内容不能含分隔符 |
| `LengthFieldBasedFrameDecoder` | 协议头含长度字段 | **最通用，生产首选** |

**`LengthFieldBasedFrameDecoder` 核心参数：**

```java
new LengthFieldBasedFrameDecoder(
    maxFrameLength,    // 最大帧长度，防止超大包打爆内存
    lengthFieldOffset, // 长度字段在帧中的偏移量
    lengthFieldLength, // 长度字段本身占几个字节
    lengthAdjustment,  // 长度字段值的调整量
    initialBytesToStrip // 解码后从帧头去掉几个字节
)
```

**ByteToMessageDecoder 处理半包：**

`ByteToMessageDecoder` 内部维护一个 `cumulation`（累积缓冲区），每次 `decode()` 被调用时，先把新数据追加到 cumulation，再尝试解码。如果数据不完整，调用 `markReaderIndex()` + `resetReaderIndex()` 回退，等下次数据到来再继续。

## 2. 序列化与编解码

**Java 原生序列化的缺点：**

- 性能差（序列化/反序列化慢）
- 序列化结果体积大
- 只支持 Java 语言，跨语言不友好
- 存在安全漏洞（反序列化攻击）

**常用序列化框架对比：**

| 框架 | 性能 | 跨语言 | 使用便捷度 | 特点 |
|---|---|---|---|---|
| **Protobuf** | 极高 | ✅ | 需要写 .proto | Google 出品，体积最小 |
| **Kryo** | 极高 | ❌（Java 专用） | 简单 | 线程不安全，需 ThreadLocal 隔离 |
| **Hessian** | 中等 | ✅ | 简单 | Dubbo 默认，兼容性好 |
| **Thrift** | 高 | ✅ | 需要写 IDL | Facebook 出品，含完整 RPC 框架 |
| **Avro** | 高 | ✅ | 需要 Schema | Hadoop 生态 |
| **Jackson / Gson** | 低 | ✅ | 极简单 | JSON，可读性好但体积大 |

**Kryo 线程安全问题（重要）：**

Kryo 不是线程安全的，不能共享实例。推荐用 `ThreadLocal` 隔离：

```java
private static final ThreadLocal<Kryo> KRYO_POOL = ThreadLocal.withInitial(() -> {
    Kryo kryo = new Kryo();
    kryo.setRegistrationRequired(false);
    return kryo;
});

public byte[] serialize(Object obj) {
    Kryo kryo = KRYO_POOL.get();
    // 使用 kryo 序列化
}
```

## 3. 心跳机制与长连接

**IdleStateHandler 三种空闲类型：**

```java
// 读空闲N秒：N秒内没有收到任何数据
new IdleStateHandler(readerIdleTime, 0, 0, TimeUnit.SECONDS)

// 写空闲N秒：N秒内没有发送任何数据
new IdleStateHandler(0, writerIdleTime, 0, TimeUnit.SECONDS)

// 读写空闲N秒：N秒内既没有读也没有写
new IdleStateHandler(0, 0, allIdleTime, TimeUnit.SECONDS)
```

**空闲事件处理：**

```java
@Override
public void userEventTriggered(ChannelHandlerContext ctx, Object evt) {
    if (evt instanceof IdleStateEvent) {
        IdleStateEvent e = (IdleStateEvent) evt;
        if (e.state() == IdleState.READER_IDLE) {
            // 读空闲：服务端关闭连接
            ctx.close();
        } else if (e.state() == IdleState.WRITER_IDLE) {
            // 写空闲：客户端发送心跳包
            ctx.writeAndFlush(HEARTBEAT_MESSAGE);
        }
    }
}
```

**断线重连（指数退避）：**

```java
public void reconnect(Bootstrap bootstrap, String host, int port) {
    bootstrap.connect(host, port).addListener(future -> {
        if (!future.isSuccess()) {
            int delay = Math.min(2 ^ retryCount * 100, 60000); // 指数退避，最多等 60 秒
            bootstrap.config().group().schedule(
                () -> reconnect(bootstrap, host, port), delay, TimeUnit.MILLISECONDS);
        }
    });
}
```

## 4. 自定义协议设计

**推荐的协议头结构：**

```
[魔数(4B)] [版本(1B)] [消息类型(1B)] [序列号(4B)] [数据长度(4B)] [数据体(可变)]
```

- **魔数**（如 0xCAFEBABE）：快速识别非法数据，收到非法魔数直接关闭连接
- **版本号**：协议升级时保持向后兼容
- **消息类型**：区分请求、响应、心跳、ACK 等
- **序列号（requestId）**：异步场景下匹配请求与响应
- **数据长度**：是解决粘包/拆包的关键，交给 `LengthFieldBasedFrameDecoder` 处理

**编解码器骨架：**

解码器（`ByteToMessageDecoder`）：先检查头部字段是否足够 → 读魔数验证 → 读长度字段 → 检查数据体是否完整 → 构建消息对象加入 `out` 列表。

编码器（`MessageToByteEncoder`）：按协议格式依次写入各字段。

## 5. HTTP / WebSocket / SSL 支持

**HTTP 服务器 Pipeline：**

```java
pipeline.addLast(new HttpServerCodec());          // HTTP 编解码
pipeline.addLast(new HttpObjectAggregator(65536)); // 聚合为 FullHttpRequest
pipeline.addLast(new HttpContentCompressor());     // 可选，gzip 压缩
pipeline.addLast(new MyHttpHandler());
```

**WebSocket 服务器 Pipeline（在 HTTP 基础上）：**

```java
pipeline.addLast(new HttpServerCodec());
pipeline.addLast(new HttpObjectAggregator(65536));
pipeline.addLast(new WebSocketServerProtocolHandler("/ws")); // 自动处理握手升级
pipeline.addLast(new MyWebSocketHandler());
```

握手完成后，`WebSocketServerProtocolHandler` 自动将 Pipeline 中的 HTTP 解码器替换为 WebSocket 帧解码器。

**SSL/TLS（必须 addFirst）：**

```java
SSLEngine engine = sslCtx.newEngine(ch.alloc());
pipeline.addFirst("ssl", new SslHandler(engine)); // 先解密！放第一个
```

## 6. Netty 在主流框架中的应用

**Dubbo**：使用 Netty 4 作为默认传输层，`InternalDecoder`/`InternalEncoder` 处理 Dubbo 协议的序列化；业务 Handler 调用上层 `ExchangeHandler` 完成反射调用；通过 `Dispatcher` 配置控制 I/O 线程与业务线程的隔离。详见 [[34_DubboKnowledge/03_通信与序列化]]。

**RocketMQ**：NameServer、Broker、Producer、Consumer 全部用 Netty 通信；`RemotingCommand` 作为统一消息格式；根据 code 字段路由到不同 `NettyRequestProcessor`；独立业务线程池保证 I/O 线程不阻塞。详见 [[30_RocketMQKnowledge]]。

**Elasticsearch**：`Netty4Transport` 实现节点间通信；不同类型操作（bulk/search/management）有独立线程池；解码后立即分发到业务线程，I/O 线程不做任何计算。
