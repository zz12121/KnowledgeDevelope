# 高性能设计

## 1. 零拷贝（三个层次）

Netty 的"零拷贝"并不是操作系统层面的完全零拷贝，而是在**三个不同层次**尽量减少不必要的内存拷贝。

**层次一：Direct Buffer（直接内存）**

使用直接内存作为 ByteBuf，避免堆内存 → 直接内存的一次额外拷贝：

```java
ByteBuf directBuf = ctx.alloc().directBuffer();
```

网络 I/O 操作（如 `FileChannel.read()`）需要直接内存参数，如果用堆内存则需要一次额外的内存拷贝。

**层次二：CompositeByteBuf（逻辑聚合）**

协议头 + 数据体合并发送，无需物理复制：

```java
CompositeByteBuf composite = ctx.alloc().compositeBuffer();
composite.addComponents(true, headerBuf, bodyBuf);
// 逻辑上是一个 ByteBuf，实际上底层是两块内存，没有复制
ctx.writeAndFlush(composite);
```

**层次三：FileRegion + sendfile（文件零拷贝）**

大文件传输时，通过 `DefaultFileRegion` 调用操作系统的 `sendfile` 系统调用：

```java
FileRegion region = new DefaultFileRegion(fileChannel, 0, fileLength);
channel.writeAndFlush(region);
```

数据路径：磁盘 → 内核 PageCache → DMA → 网卡缓冲区，全程不经过用户态内存，是真正的零拷贝。

## 2. FastThreadLocal

Netty 用 `FastThreadLocal` 替代 JDK `ThreadLocal`，专为 `FastThreadLocalThread`（Netty 的 EventLoop 线程）优化。

**JDK ThreadLocal 的问题**：底层是 `ThreadLocalMap`，用哈希散列存储，存在哈希冲突、线性探测开销。

**FastThreadLocal 的优化**：每个 `FastThreadLocal` 实例在创建时分配一个全局唯一的整数 index，线程本地存储是一个 Object 数组，访问时直接用 index 定位，O(1) 无冲突。

```java
// 访问效率对比
JDK ThreadLocal:   threadLocalMap.getEntry(key)  → 哈希 + 线性探测
FastThreadLocal:   array[index]                  → 直接下标访问
```

性能差异在 Netty 高频 I/O 操作（如 ByteBuf 分配、编解码缓存）中累积效果显著。

## 3. HashedWheelTimer（时间轮）

Netty 用 `HashedWheelTimer` 实现心跳超时、连接空闲检测等大量定时任务调度。

**为什么不用 `ScheduledThreadPoolExecutor`**：

它的底层是优先级堆，添加/删除任务都是 O(log n)；当有数万个定时任务时性能下降明显，且每次 tick 需要遍历堆顶。

**时间轮原理**：

```
    轮盘（256 个槽位，默认）
    [0][1][2]...[255]
     ↑
   当前 tick 指针，每 100ms 前进一格
   
每个槽位是一个链表，存放该时刻到期的任务
```

添加任务：根据 `delay` 计算落在哪个槽位（O(1)）
执行任务：指针走到该槽位时执行链表中的所有任务（O(1) 摊销）
取消任务：标记任务为 CANCELLED，执行时跳过（O(1)）

**核心参数**：

- `tickDuration`：每次 tick 的时间精度（默认 100ms）
- `ticksPerWheel`：轮盘槽位数（默认 512，建议是 2 的幂次方）

**适用场景**：大量超时相近、新增/取消频繁的定时任务。不适合需要精确毫秒级定时的场景（精度受 tickDuration 限制）。

## 4. 内存池（PooledByteBufAllocator）

Netty 借鉴 jemalloc 的思想实现了内存池，避免每次 I/O 都向操作系统申请/释放内存。

**三级结构：**

```
PoolArena（多个，减少多线程竞争）
    └─ PoolChunk（16MB 大块内存，伙伴算法管理）
           └─ PoolSubpage（256B 以下的小内存，位图管理）
```

**PoolThreadCache（线程本地缓存）：**

每个 EventLoop 线程有独立的 PoolThreadCache，存放最近释放的小对象，下次分配优先从这里取，避免进入 PoolArena 的加锁路径，是高频分配场景的性能关键。

**分配流程（以 256B 为例）：**

1. 检查当前线程的 `PoolThreadCache` → 命中则直接返回
2. 到对应 `PoolArena` 的 `tinySubpagePools`/`smallSubpagePools` 中查找
3. 没有则从 `PoolChunk` 中切割分配
4. 如果 PoolChunk 不够，向操作系统申请新的 PoolChunk

## 5. epoll 空轮询 Bug 修复

**Bug 描述**：JDK NIO 在 Linux 上的经典 Bug——在没有任何就绪事件时，`Selector.select()` 不阻塞直接返回，CPU 飙升到 100%。

**Netty 的修复策略**：

```java
// NioEventLoop.run() 中的计数逻辑（简化）
int selectCount = 0;
while (true) {
    int readyCount = selector.select(timeout);
    selectCount++;
    
    if (readyCount == 0 && selectCount >= SELECTOR_AUTO_REBUILD_THRESHOLD) {
        // 512 次空轮询，认为触发了 Bug
        rebuildSelector(); // 重建 Selector，将所有 Channel 重新注册
        selectCount = 0;
    }
    
    processSelectedKeys();
    runAllTasks();
}
```

只要空轮询次数超过阈值（默认 512 次），就主动重建 Selector，彻底规避这个 Bug。

## 6. IO 模型深度解析

> 本节补充自参考资料《把被说烂的BIO、NIO、AIO再从头到尾扯一遍》

### 6.1 五种 IO 模型

Unix 下有五种 IO 模型，理解它们是理解 Netty 高性能设计的基础：

| IO模型 | 等待数据 | 数据从内核拷到用户空间 | 特点 |
|--------|---------|---------------------|------|
| 阻塞IO（BIO） | 阻塞 | 阻塞 | 简单，线程完全被占用 |
| 非阻塞IO（NIO） | 非阻塞（轮询返回-1） | 阻塞 | 轮询浪费CPU |
| IO多路复用 | 非阻塞（select/epoll监听） | 阻塞 | 单线程监控多连接 |
| 信号驱动IO | 非阻塞（SIGIO信号） | 阻塞 | 实际应用少 |
| 异步IO（AIO） | 非阻塞 | 非阻塞（内核完成通知） | 真正异步 |

**同步vs异步的本质区分**：

```
同步IO：数据从内核拷贝到用户空间时，程序是阻塞的
        （BIO/NIO/多路复用/信号驱动 都是同步IO）
异步IO：程序发起操作后立即返回，内核完成所有工作后发信号通知
        （只有AIO是真正异步IO）
```

### 6.2 BIO 的问题与演进

```java
// BIO：一个连接一个线程，连接多了线程数爆炸
ServerSocket serverSocket = new ServerSocket(8080);
while (true) {
    Socket socket = serverSocket.accept();  // 阻塞等待连接
    new Thread(() -> {
        // 阻塞读写
        socket.getInputStream().read(buffer);
        // 处理
        socket.getOutputStream().write(response);
    }).start();
}
// 问题：10000个连接 = 10000个线程，内存爆炸，上下文切换开销极大
```

### 6.3 NIO 多路复用模型

```java
// NIO 核心：一个 Selector 管理多个 Channel
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.configureBlocking(false);  // 非阻塞
server.register(selector, SelectionKey.OP_ACCEPT);  // 注册Accept事件

while (true) {
    selector.select();  // 阻塞等待（但同时监控多个Channel）
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) {
            SocketChannel client = server.accept();
            client.configureBlocking(false);
            client.register(selector, SelectionKey.OP_READ);
        } else if (key.isReadable()) {
            // 读数据
        }
    }
}
// 优点：一个线程管理所有连接
// 底层：Linux → epoll，Mac → kqueue，Windows → IOCP
```

### 6.4 select vs poll vs epoll 深度对比

```
select 的流程：
1. 用户把 fd_set（最多1024个fd）从用户空间拷贝到内核
2. 内核遍历全部 fd，检查是否就绪
3. 把就绪的标记，把整个 fd_set 返回用户空间
4. 用户再遍历找到就绪的 fd
→ 每次调用 2 次拷贝 + O(n) 遍历，性能差

epoll 的流程：
1. epoll_create() 创建 epoll 实例，内核建红黑树和就绪链表
2. epoll_ctl() 注册 fd（一次拷贝），内核给 fd 注册回调
3. fd 就绪时，内核回调函数把 fd 加入就绪链表
4. epoll_wait() 直接从就绪链表取，O(1)
→ 注册时一次拷贝 + O(1) 取就绪，性能高
```

**水平触发（LT）vs 边缘触发（ET）**：

```
LT（Level Triggered，默认）：
  - 只要 fd 有数据可读，每次 epoll_wait 都通知
  - 没读完下次继续通知，不会漏数据
  - 编程简单，性能略低

ET（Edge Triggered）：
  - 只在 fd 状态变化时通知一次（从无数据变为有数据）
  - 必须一次性读完所有数据，否则下次不再通知
  - 性能更高，但编程难度大
  - 必须配合非阻塞 fd 使用
```

### 6.5 Netty 选择多路复用 IO 的理由

| 维度 | BIO | NIO（多路复用） | AIO |
|------|-----|---------------|-----|
| 线程模型 | 每连接一线程 | EventLoop（少量线程） | 线程池处理回调 |
| 连接数 | 受线程数限制 | 万级并发 | 万级并发 |
| 系统支持 | 全平台 | 全平台 | Windows完善，Linux不完善 |
| 编程复杂度 | 简单 | 中等 | 较复杂 |

> **Netty 选择 NIO（多路复用）而非 AIO 的原因**：
> Linux 的 AIO（libaio）实现并不成熟，实际在高并发场景下性能并不比 epoll 好。
> Netty 官方测试发现，在大多数场景下 epoll + Reactor 模型已经足够，所以 Netty 主推 NIO。
